# 第07章 IO模型 · 章节概览

## 为什么学习IO模型

输入/输出（I/O）是计算机系统中最频繁也最影响性能的操作。每一次网络请求、每一次磁盘读写、每一次终端交互，本质上都是 I/O 操作。理解 I/O 模型，就是在理解"程序如何与外部世界交换数据"这个根本问题。

从工程实践角度看，I/O 模型直接决定了：

- **服务器能承载多少并发连接** — 从 C10K 到 C10M 的核心瓶颈就在 I/O 层
- **系统的响应延迟** — 一个错误的 I/O 模型选择可能让延迟从微秒级退化到毫秒级
- **架构的可扩展性** — 选错模型意味着后续所有的架构优化都建立在错误的地基上

无论你是写一个简单的 TCP echo server，还是设计一个支撑百万连接的网关，I/O 模型都是绕不开的核心知识。

## 核心问题索引

阅读本章前，先问自己这四个问题。带着问题去读，效率最高：

**1. 五种I/O模型的本质区别是什么？**

不是"哪种更快"这种表面问题，而是要从两个维度精确分类：
- **数据准备阶段**：进程是否被阻塞？
- **数据拷贝阶段**：进程是否参与拷贝？

只有异步I/O（io_uring/AIO）在两个阶段都不阻塞用户进程。其余四种都是同步I/O——区别仅在于数据准备阶段的等待方式。

**2. epoll为什么比select/poll高效？**

| 对比维度 | select | poll | epoll |
|---------|--------|------|-------|
| 数据结构 | 位图(fd_set) | pollfd数组 | 红黑树+就绪链表 |
| fd上限 | 1024(FD_SETSIZE) | 无硬限制 | 无硬限制 |
| 就绪通知 | O(n)遍历全部fd | O(n)遍历全部fd | O(1)回调加入就绪链表 |
| 每次调用开销 | 拷贝fd_set到内核 | 拷贝pollfd数组到内核 | 无需重复拷贝 |

**3. io_uring带来了什么革命性变化？**

传统异步I/O（Linux AIO）的痛点：每次操作仍需系统调用，且仅支持O_DIRECT。io_uring通过**共享内存环形缓冲区**（SQ/CQ）实现：
- 提交队列（SQ）和完成队列（CQ）共享用户态/内核态内存
- 批量提交，一次系统调用提交多个I/O请求
- 内核态轮询（IORING_SETUP_SQPOLL），完全消除系统调用开销

**4. Reactor和Proactor如何指导服务器架构？**

| 模式 | 核心思想 | 代表实现 | 适用场景 |
|------|---------|---------|---------|
| Reactor | 事件就绪时通知用户态处理 | Nginx、Redis、Netty | Linux/高并发网络服务 |
| Proactor | 内核完成I/O后通知用户态结果 | Windows IOCP、io_uring | 追求极致性能的异步场景 |

## 核心数据结构速查

本章反复涉及的关键内核数据结构，建议在阅读过程中随时回查：

```c
// epoll的核心结构
struct eventpoll {
    struct list_head rdllist;           // 就绪链表：有事件的epitem
    struct rb_root_cached rbr;          // 红黑树：所有监控的epitem
    wait_queue_head_t wq;               // 等待队列：epoll_wait睡眠的进程
};

struct epitem {
    struct rb_node rbn;                 // 红黑树节点
    struct list_head rdllink;           // 就绪链表节点
    struct epoll_filefd ffd;            // 关联的fd+file
    struct epoll_event event;           // 注册的事件掩码
};

// io_uring的核心结构
struct io_ring_ctx {
    struct io_uring_sqe *sqes;          // 提交队列项数组
    struct io_uring_cqe *cqes;          // 完成队列项数组
    unsigned sq_entries;                 // 提交队列大小
    unsigned cq_entries;                 // 完成队列大小
    // ... 共享内存环形缓冲区
};
```
