# 练习方法

本章覆盖了IO系统的完整技术栈——从总线架构到中断机制，从DMA原理到IO调度，从存储协议到虚拟化。以下是五个递进式练习，帮助你将理论知识转化为实践能力。每个练习都基于真实的Linux环境，使用系统自带或常见开源工具。

---

## 练习一：IO系统全景认知（预计45分钟）

**目标**：建立对IO硬件架构和软件栈的完整认知，能识别系统中的IO设备并理解其拓扑关系。

### 步骤1：硬件拓扑探测（15分钟）

Linux提供了丰富的接口来查看IO硬件信息。通过这些命令，你可以"看到"系统的物理IO架构。

```bash
# 1. 查看PCIe设备列表及其总线拓扑
lspci -tv

# 预期输出示例：
# -[0000:00]-+-00.0  Intel Corporation Xeon E3-1200 v6 Host Bridge
#            +-01.0-[01]----00.0  NVIDIA Corporation GA106
#            +-14.0  Intel Corporation USB 3.1 xHCI Host Controller
#            +-17.0  Intel Corporation SATA Controller
#            +-1d.0-[02]----00.0  Samsung Electronics NVMe SSD Controller
#            \-1f.0  Intel Corporation LPC Controller/eSPI Controller

# 2. 查看每个设备的详细信息，重点关注NUMA节点和中断分配
lspci -vvs 02:00.0 | grep -E "NUMA|MSI|Region|Interrupt"

# 3. 查看NVMe设备列表
nvme list

# 4. 查看块设备信息
lsblk -f
```

**思考题**：在`lspci -tv`的输出中，识别哪些设备直接连接到CPU的PCIe根复合体，哪些经过了PCIe Switch或南桥。这会如何影响IO延迟？

### 步骤2：IO软件栈追踪（15分钟）

理解从应用层到硬件的完整IO路径：

```bash
# 1. 查看当前IO调度器
cat /sys/block/sda/queue/scheduler
# 输出示例：[mq-deadline] kyber bfq none

# 2. 查看设备的IO队列深度
cat /sys/block/nvme0n1/queue/nr_requests

# 3. 查看块设备的硬件特性
cat /sys/block/nvme0n1/queue/rotational    # 0=SSD, 1=HDD
cat /sys/block/nvme0n1/queue/maximum_hw_sectors_kb
cat /sys/block/nvme0n1/queue/physical_block_size
cat /sys/block/nvme0n1/queue/logical_block_size

# 4. 追踪一个文件读操作经过的内核路径
strace -e read,write -f cat /etc/hostname
# 观察系统调用层面的IO路径

# 5. 使用blktrace追踪完整的块IO路径
sudo blktrace -d /dev/nvme0n1 -o - | blkparse -i - | head -50
# 这会展示IO请求在IO调度层、驱动层的完整流转过程
```

**思考题**：`rotational`字段为0说明什么？它会影响IO调度器的行为吗？

### 步骤3：绘制个人系统的IO架构图（15分钟）

基于以上信息，手绘或用mermaid画出你的系统的IO架构图，包含：

- CPU与各IO设备的连接关系（通过哪条PCIe总线）
- 每个块设备的类型（NVMe/SATA/HDD）和挂载点
- 当前使用的IO调度器
- 中断分配情况（哪些CPU核心处理哪些设备的中断）

## 练习二：IO性能基线测量（预计60分钟）

**目标**：掌握IO性能测量的核心工具和方法论，能够建立系统的IO性能基线，为后续优化提供参照。

### 步骤1：安装和使用fio（15分钟）

fio（Flexible IO Tester）是业界标准的IO基准测试工具：

```bash
# 安装fio
sudo apt-get update &amp;&amp; sudo apt-get install -y fio

# 创建测试目录（注意：不要在SSD的系统分区上做大量写入测试）
mkdir -p /tmp/fio-test &amp;&amp; cd /tmp/fio-test

# 基础测试1：顺序读（模拟大文件拷贝）
fio --name=seq-read \
    --ioengine=libaio \
    --direct=1 \
    --bs=128k \
    --size=1G \
    --numjobs=1 \
    --rw=read \
    --runtime=30 \
    --group_reporting

# 基础测试2：随机4K读（模拟数据库OLTP场景）
fio --name=rand-read-4k \
    --ioengine=libaio \
    --direct=1 \
    --bs=4k \
    --size=1G \
    --numjobs=4 \
    --iodepth=32 \
    --rw=randread \
    --runtime=30 \
    --group_reporting

# 基础测试3：混合读写（模拟Web服务器场景，70%读30%写）
fio --name=mixed-rw \
    --ioengine=libaio \
    --direct=1 \
    --bs=4k \
    --size=512M \
    --numjobs=4 \
    --iodepth=16 \
    --rw=randrw \
    --rwmixread=70 \
    --runtime=30 \
    --group_reporting
```

**记录以下指标**：

| 测试场景 | IOPS | 带宽(MB/s) | 平均延迟(us) | P99延迟(us) |
|----------|------|------------|-------------|-------------|
| 顺序读 | | | | |
| 4K随机读 | | | | |
| 混合读写 | | | | |

### 步骤2：使用iostat进行实时监控（15分钟）

```bash
# 安装sysstat工具包
sudo apt-get install -y sysstat

# 实时监控IO状态（每秒刷新一次）
iostat -xdm 1

# 关键字段解读：
# rrqm/s, wrqm/s  : 读/写请求合并率（IO调度器合并效果）
# r/s, w/s         : 每秒读/写次数（IOPS）
# rkB/s, wkB/s     : 每秒读/写吞吐量
# await            : 平均IO等待时间（包含队列等待+设备服务时间）
# svctm            : 设备服务时间（不准确，已废弃）
# %util            : 设备利用率（接近100%说明设备已饱和）
```

**实操**：在终端1运行`iostat -xdm 1`，在终端2用fio发起负载，观察指标变化。

### 步骤3：使用pidstat定位IO热点进程（15分钟）

```bash
# 监控各进程的IO活动
pidstat -d 1

# 关键字段：
# kB_rd/s  : 每秒从磁盘读取的KB数
# kB_wr/s  : 每秒写入磁盘的KB数
# kB_ccwr/s: 每秒取消写入的KB数（写入后被截断的页面数）
```

### 步骤4：使用iotop定位IO热点（15分钟）

```bash
sudo apt-get install -y iotop

# 查看各进程的IO使用量
sudo iotop -oP

# 按IO使用量排序，找到最"吵"的进程
```

**思考题**：在你的系统上，哪个进程消耗了最多的IO资源？是正常的还是异常的？如果是数据库，它的IOPS和延迟是否合理？

## 练习三：IO调度器实验（预计45分钟）

**目标**：深入理解不同IO调度器的设计思想和行为差异，能够在不同场景下选择合适的调度器。

### 步骤1：理解当前调度器的参数（10分钟）

```bash
# 查看当前调度器
cat /sys/block/sda/queue/scheduler

# 查看调度器可调参数
ls /sys/block/sda/queue/iosched/

# 查看mq-deadline的关键参数
cat /sys/block/sda/queue/iosched/read_expire    # 读请求过期时间(ms)
cat /sys/block/sda/queue/iosched/write_expire   # 写请求过期时间(ms)
cat /sys/block/sda/queue/iosched/writes_starved # 读优先级权重
```

### 步骤2：对比不同调度器的性能（25分钟）

```bash
# 创建测试脚本
cat > /tmp/scheduler-test.sh << 'EOF'
#!/bin/bash
DEVICE=sda
TEST_DIR=/tmp/sched-test
mkdir -p $TEST_DIR

# 测试不同调度器
for sched in mq-deadline bfq kyber none; do
    echo "=== 测试调度器: $sched ==="
    
    # 切换调度器（仅对支持的设备有效）
    echo $sched | sudo tee /sys/block/$DEVICE/queue/scheduler 2>/dev/null
    
    # 清除缓存
    echo 3 | sudo tee /proc/sys/vm/drop_caches
    
    # 随机读测试
    echo "--- 4K随机读 ---"
    fio --name=test-rand-read \
        --filename=$TEST_DIR/testfile \
        --ioengine=libaio \
        --direct=1 \
        --bs=4k \
        --size=256M \
        --numjobs=4 \
        --iodepth=32 \
        --rw=randread \
        --runtime=10 \
        --group_reporting 2>/dev/null | grep -E "IOPS|lat.*avg"
    
    # 顺序写测试
    echo "--- 1M顺序写 ---"
    fio --name=test-seq-write \
        --filename=$TEST_DIR/testfile2 \
        --ioengine=libaio \
        --direct=1 \
        --bs=1m \
        --size=256M \
        --numjobs=1 \
        --rw=write \
        --runtime=10 \
        --group_reporting 2>/dev/null | grep -E "IOPS|bw.*MiB"
    
    echo ""
done
EOF
chmod +x /tmp/scheduler-test.sh
```

**注意**：只有多队列块设备支持mq-deadline/bfq/kyber/none四种调度器切换。单队列设备（旧HDD或SATA SSD）可能只有none和mq-deadline。

### 步骤3：分析调度器行为（10分钟）

```bash
# 观察IO请求的合并情况
# 使用blktrace抓取IO请求
sudo blktrace -d /dev/sda -o - | head -200 | blkparse -i - -d trace.bin
blkparse -i trace.bin -o trace.txt

# 分析结果中的关键字段：
# Q : 排队（Queued） - IO请求进入调度器
# M : 合并（Merged） - 被调度器合并为更大的请求
# D : 分发（Dispatch） - 从调度器发往驱动
# C : 完成（Complete） - IO操作完成
```

**分析要点**：
- 合并率（M/Q比例）高说明调度器有效减少了IO请求数
- 读写交替模式下，deadline调度器是否保证了读请求的优先级
- 不同调度器对顺序IO和随机IO的优化策略差异

**思考题**：对于OLTP数据库（高IOPS、小IO、读多写少），应该选择哪种调度器？为什么？对于视频编辑工作站（大文件顺序读写），又该选哪种？

## 练习四：IO问题排查（预计60分钟）

**目标**：掌握IO瓶颈的诊断流程，能够独立定位和解决IO性能问题。

### 步骤1：制造IO问题（15分钟）

```bash
# 场景1：模拟IO等待导致的性能下降
# 终端1：启动一个持续写入的进程
dd if=/dev/zero of=/tmp/io-heavy bs=1M count=2048 &amp;
DD_PID=$!

# 场景2：模拟高IOPS场景（随机小写入）
fio --name=rand-write \
    --ioengine=libaio \
    --direct=1 \
    --bs=4k \
    --size=512M \
    --numjobs=8 \
    --iodepth=64 \
    --rw=randwrite \
    --runtime=120 &amp;
FIO_PID=$!
```

### 步骤2：使用诊断工具定位问题（25分钟）

```bash
# 第一步：快速概览（vmstat + iostat）
vmstat 1 5
# 关注：b列（阻塞进程数）是否大于0，wa列（IO等待时间百分比）是否过高

iostat -xdm 1 5
# 关注：%util是否接近100%，await是否异常高

# 第二步：定位问题进程（iotop + pidstat）
sudo iotop -oP -d 1
pidstat -d 1 5

# 第三步：深入分析IO模式（blktrace）
# 终端A：启动blktrace
sudo blktrace -d /dev/nvme0n1 -w 30 -o io-trace

# 终端B：同时运行blktrace分析
sudo btrace /dev/nvme0n1

# 分析trace输出中的异常：
# - 读写比例是否异常
# - IO请求大小分布是否合理
# - 是否存在频繁的队列满（busy标记）

# 第四步：检查文件系统层面的问题
df -h                     # 磁盘空间是否充足
df -i                     # inode使用率
mount | grep noatime      # 检查挂载选项
cat /proc/fs/ext4/*/mb_groups  # ext4多块分配状态（如适用）
```

### 步骤3：实施修复并验证（20分钟）

```bash
# 场景A：某进程IO过多
# 找到问题进程后，降低其IO优先级
sudo ionice -c3 -p $FIO_PID    # 设置为idle级别IO
# 或限制其IO带宽
sudo ionice -c2 -n7 -p $FIO_PID  # best-effort, 最低优先级

# 场景B：写入过多导致系统卡顿
# 检查是否可以减少写入
sync &amp;&amp; echo 3 | sudo tee /proc/sys/vm/drop_caches
# 调整脏页参数
cat /proc/sys/vm/dirty_ratio           # 脏页比例上限
cat /proc/sys/vm/dirty_background_ratio # 脏页后台刷新比例

# 场景C：文件系统碎片化
# 检查ext4碎片率
e4defrag -c /dev/sda1   # 查看连续性评分

# 验证修复效果
iostat -xdm 1 10
pidstat -d 1 10
```

**排查思路总结**：

| 层级 | 诊断方法 | 关键指标 | 常见问题 |
|------|---------|---------|---------|
| 应用层 | iotop, pidstat | kB_rd/s, kB_wr/s | 单进程IO过大 |
| 文件系统层 | df, mount, tune2fs | 空间/inode使用率 | 碎片化、日志满 |
| 块IO层 | iostat, blktrace | %util, await, 合并率 | 调度不当、队列深 |
| 设备层 | smartctl, nvme-cli | 温度、错误计数 | 硬件故障、过热 |
| 硬件层 | lspci, dmidecode | 总线错误、掉速 | PCIe链路降级 |

## 练习五：IO性能调优实战（预计90分钟）

**目标**：掌握系统级IO调优方法，能够针对不同场景优化IO性能。

### 步骤1：文件系统调优（25分钟）

```bash
# 查看当前文件系统的挂载选项和默认参数
tune2fs -l /dev/sda1 | grep -E "Block size|Inode size|Reserved"

# 优化挂载选项（仅对数据分区操作，不要修改系统分区）
# 以下为参考示例，实际应用需要卸载分区或在fstab中配置

# 查看当前fstab配置
cat /etc/fstab | grep -v "^#" | grep -v "^$"

# 关键调优参数（在fstab中配置）：
# noatime      : 不更新文件访问时间，减少不必要的元数据写入
# commit=60    : 延长日志提交间隔（默认5秒），减少fsync频率
# barrier=0    : 禁用写屏障（仅在有UPS和电池备份缓存的RAID控制器上安全）

# 调整预读大小（read-ahead）
# 查看当前预读值（单位：512字节扇区）
cat /sys/block/sda/queue/read_ahead_kb
# 默认128KB，顺序读为主的工作负载可以增加到2048-4096
echo 2048 | sudo tee /sys/block/sda/queue/read_ahead_kb

# 调整队列深度
cat /sys/block/sda/queue/nr_requests
# 默认128，高IOPS场景可以适当增大
```

### 步骤2：内核IO参数调优（30分钟）

```bash
# --- 虚拟内存与脏页调优 ---

# 脏页刷盘策略
cat /proc/sys/vm/dirty_ratio              # 默认20%，超过此比例进程写入被阻塞
cat /proc/sys/vm/dirty_background_ratio   # 默认10%，超过此比例后台线程开始刷盘
cat /proc/sys/vm/dirty_expire_centisecs   # 默认3000(30秒)，脏页最长存活时间
cat /proc/sys/vm/dirty_writeback_centisecs # 默认500(5秒)，后台线程唤醒间隔

# 高写入场景调优：增大dirty_ratio避免频繁阻塞
sudo sysctl -w vm.dirty_ratio=40
sudo sysctl -w vm.dirty_background_ratio=20

# --- IO调度器调优 ---
# mq-deadline调优示例（对SSD）
cat /sys/block/nvme0n1/queue/iosched/read_expire    # 默认150ms
cat /sys/block/nvme0n1/queue/iosched/write_expire   # 默认1500ms
# SSD场景可以缩短过期时间以降低尾延迟
echo 50 | sudo tee /sys/block/nvme0n1/queue/iosched/read_expire

# --- 中断亲和性调优 ---
# 查看当前中断分布
cat /proc/interrupts | head -20

# 将NVMe中断绑定到特定CPU（减少跨NUMA访问）
# 查找NVMe设备的中断号
NVME_IRQ=$(grep nvme /proc/interrupts | awk '{print $1}' | tr -d ':' | head -1)
# 绑定到CPU 0
echo 1 | sudo tee /proc/irq/$NVME_IRQ/smp_affinity
```

### 步骤3：对比优化前后的性能（20分钟）

```bash
# 创建标准化测试脚本
cat > /tmp/benchmark-standard.sh << 'SCRIPT'
#!/bin/bash
echo "=== 性能基线测试 ==="
echo "测试时间: $(date)"
echo "内核版本: $(uname -r)"
echo "IO调度器: $(cat /sys/block/nvme0n1/queue/scheduler)"
echo ""

echo "--- 4K随机读 IOPS ---"
fio --name=4k-rand-read \
    --filename=/tmp/fio-bench \
    --ioengine=libaio \
    --direct=1 \
    --bs=4k \
    --size=1G \
    --numjobs=4 \
    --iodepth=32 \
    --rw=randread \
    --runtime=15 \
    --group_reporting 2>/dev/null | grep -E "IOPS.*randread|avg.*lat"

echo ""
echo "--- 4K随机写 IOPS ---"
fio --name=4k-rand-write \
    --filename=/tmp/fio-bench \
    --ioengine=libaio \
    --direct=1 \
    --bs=4k \
    --size=1G \
    --numjobs=4 \
    --iodepth=32 \
    --rw=randwrite \
    --runtime=15 \
    --group_reporting 2>/dev/null | grep -E "IOPS.*randwrite|avg.*lat"

echo ""
echo "--- 128K顺序读带宽 ---"
fio --name=seq-read \
    --filename=/tmp/fio-bench \
    --ioengine=libaio \
    --direct=1 \
    --bs=128k \
    --size=1G \
    --numjobs=1 \
    --rw=read \
    --runtime=15 \
    --group_reporting 2>/dev/null | grep -E "bw.*MiB|IOPS.*read"
SCRIPT
chmod +x /tmp/benchmark-standard.sh

# 运行基线测试（调优前）
/tmp/benchmark-standard.sh | tee /tmp/baseline-results.txt

# 执行调优操作（参考步骤1和步骤2）

# 再次运行测试（调优后）
/tmp/benchmark-standard.sh | tee /tmp/optimized-results.txt

# 对比结果
echo "=== 优化效果对比 ==="
diff /tmp/baseline-results.txt /tmp/optimized-results.txt
```

### 步骤4：将调优持久化（15分钟）

```bash
# 使用sysctl.conf持久化内核参数
cat >> /etc/sysctl.d/99-io-tuning.conf << 'EOF'
# IO性能调优参数
vm.dirty_ratio = 40
vm.dirty_background_ratio = 20
vm.dirty_expire_centisecs = 3000
vm.dirty_writeback_centisecs = 500
EOF

# 验证sysctl配置
sudo sysctl -p /etc/sysctl.d/99-io-tuning.conf

# 使用udev规则持久化调度器和队列参数
cat > /etc/udev/rules.d/60-io-scheduler.rules << 'EOF'
# NVMe设备使用kyber调度器，减少队列深度
ACTION=="add|change", KERNEL=="nvme[0-9]*n[0-9]*", ATTR{queue/scheduler}="kyber"
ACTION=="add|change", KERNEL=="nvme[0-9]*n[0-9]*", ATTR{queue/read_ahead_kb}="256"

# SATA/SAS设备使用mq-deadline调度器
ACTION=="add|change", KERNEL=="sd[a-z]", ATTR{queue/rotational}=="1", ATTR{queue/scheduler}="mq-deadline"
ACTION=="add|change", KERNEL=="sd[a-z]", ATTR{queue/rotational}=="1", ATTR{queue/read_ahead_kb}="2048"

# SATA SSD使用bfq调度器
ACTION=="add|change", KERNEL=="sd[a-z]", ATTR{queue/rotational}=="0", ATTR{queue/scheduler}="bfq"
EOF

# 加载udev规则
sudo udevadm control --reload-rules
```

## 综合挑战：设计一个IO密集型系统的最佳实践方案

**目标**：综合运用本章知识，为真实场景设计IO方案。

**场景**：你负责设计一个日均处理5000万次数据库查询的MySQL服务器的存储方案。

**要求**：
1. 选择存储硬件（NVMe SSD vs SATA SSD vs HDD RAID）
2. 选择文件系统和挂载参数（ext4 vs xfs vs btrfs）
3. 选择IO调度器和参数
4. 设计MySQL的innodb_buffer_pool_size、innodb_log_file_size等IO相关参数
5. 评估方案的IOPS和延迟是否满足业务需求

**评估标准**：
- [ ] 方案有明确的性能需求量化（IOPS、带宽、延迟）
- [ ] 硬件选型有成本效益分析
- [ ] 每个参数选择都有理论依据
- [ ] 方案考虑了故障容错和数据安全
- [ ] 有性能验证计划（使用fio或sysbench进行基准测试）
