# 14.1 索引实现的核心概念

索引是数据库系统中最重要的性能优化手段，但它的实现远非"在数据上建个查找表"这么简单。一个生产级索引需要同时解决查询性能、写入吞吐、并发安全、崩溃恢复四大难题，而不同的应用场景对这四个维度的优先级截然不同。本节将建立理解后续所有实现细节的概念框架。

## 索引的本质：空间换时间

从最根本的定义出发，索引是一种**辅助数据结构**，它维护数据的某种有序组织或映射关系，使查询操作可以跳过全表扫描，直接定位目标数据。

类比现实世界：一本书的目录就是索引。如果没有目录，要找"贝叶斯定理"在哪一页，你必须从第一页翻到最后一页（全表扫描，O(n)时间）。有了目录，你直接翻到对应页码（索引查找，O(log n)或O(1)时间）。目录本身占用了额外的纸张（存储开销），目录中的页码可能因为增删内容而需要更新（维护开销）。

数据库索引面临的核心约束与纸质目录有本质区别：

| 维度 | 纸质目录 | 数据库索引 |
|------|----------|------------|
| 更新频率 | 几乎不变 | 每秒可能数千次插入/删除 |
| 并发需求 | 无（单人使用） | 数百个查询同时进行 |
| 持久化 | 书丢了就丢了 | 必须保证崩溃后数据不丢失 |
| 空间限制 | 目录几页无所谓 | TB级数据下索引空间不可忽略 |
| 查询类型 | 只有"在哪儿" | 等值、范围、前缀、模糊、空间等 |

理解这些差异是理解索引实现复杂性的起点。

## 索引为何重要：量化分析

### 全表扫描的代价

假设一张表有 N = 10亿条记录，每条记录 200 字节，存储在 4KB 的磁盘页上：

总页数 = 10亿 × 200 / 4096 ≈ 4880万页

现代 SSD 的随机读延迟约为 0.1ms。如果做全表扫描：

时间 = 4880万 × 0.1ms ≈ 4880秒 ≈ 81分钟

即使使用顺序读优化（SSD 顺序读带宽约 2GB/s）：

总数据量 = 10亿 × 200 / 1024³ ≈ 186 GB
时间 = 186 GB / 2 GB/s ≈ 93秒

### 索引查找的代价

一个 B+tree 索引，假设扇出（fanout）为 500：

树高 = log₅₀₀(10亿) ≈ 4层
每次查询 = 4次随机读 = 4 × 0.1ms = 0.4ms

**从 93 秒到 0.4 毫秒**——这就是索引存在的意义。在 OLTP 系统中，单次查询从秒级降到亚毫秒级，系统的可用容量可能相差 1000 倍以上。

### 量化权衡公式

索引不是免费的午餐。建一个索引意味着：

读加速增益 = 全表扫描时间 / 索引查找时间
写放大代价 = 每次写操作需要同步更新的索引数
存储开销 = 索引占用空间 / 原始数据空间

一个典型的交易表可能有 5-10 个索引，每个索引占用原始数据 20%-100% 的空间。这意味着索引的总存储开销可能是原始数据的 1-5 倍。每次写入操作需要更新所有相关索引，写入延迟可能是无索引时的 5-10 倍。

## 索引的核心分类体系

理解索引的分类方式是选择正确索引策略的前提。可以从三个正交维度对索引进行分类。

### 按数据结构分类

这是最直观的分类方式，决定了索引的查询能力和性能特征：

| 索引类型 | 数据结构 | 等值查询 | 范围查询 | 排序 | 适用场景 |
|----------|----------|----------|----------|------|----------|
| **B+tree 索引** | 多路平衡搜索树 | O(log n) | O(log n + k) | 高效 | 通用场景，OLTP 主力 |
| **Hash 索引** | 哈希表 | O(1) 均摊 | 不支持 | 不支持 | 精确匹配，如缓存键 |
| **LSM-Tree** | 有序日志+合并 | O(log n) | O(log n + k) | 高效 | 写密集场景，如时序数据 |
| **倒排索引** | 词项→文档列表 | O(1) 到 O(k) | 不支持 | 不支持 | 全文搜索、搜索引擎 |
| **R-tree / GiST** | 空间树结构 | 不适用 | 支持空间范围 | 不适用 | 地理空间、多维数据 |
| **位图索引** | 位向量 | 极快(位运算) | 不直接支持 | 不适用 | 低基数列，数据仓库 |

其中 k 是结果集大小。O(log n + k) 表示定位第一个结果需要 O(log n)，扫描剩余结果需要 O(k)。

### 按存储组织分类

索引与原始数据的存储关系决定了查询路径的长短：

**聚簇索引（Clustered Index）**：索引的叶子节点直接存储完整的数据行。一张表最多只能有一个聚簇索引，因为数据行只能按一种顺序物理存储。InnoDB 的主键索引就是聚簇索引——通过主键查找时，不需要二次回表。

**非聚簇索引（Non-Clustered Index）**：索引的叶子节点存储的是指向数据行的指针（在 InnoDB 中是主键值）。查询时先通过索引找到主键，再通过主键索引找到完整数据行，这个过程称为**回表（Lookaside / Bookmark Lookup）**。

**覆盖索引（Covering Index）**：如果查询所需的所有列都包含在索引中，就不需要回表。例如索引 `(user_id, created_at, amount)` 可以直接回答 `SELECT created_at, amount FROM orders WHERE user_id = 123`，不需要访问原始数据行。

### 按功能特性分类

| 分类 | 说明 | 示例 |
|------|------|------|
| **单列索引** | 基于单个列创建 | `CREATE INDEX idx_name ON users(name)` |
| **复合索引（联合索引）** | 基于多个列创建，遵循最左前缀匹配 | `CREATE INDEX idx_multi ON users(name, age, city)` |
| **前缀索引** | 对字符串列的前 N 个字符建索引 | `CREATE INDEX idx_prefix ON users(name(10))` |
| **函数索引（表达式索引）** | 基于列的计算结果建索引 | `CREATE INDEX idx_func ON users(UPPER(name))` |
| **部分索引** | 只对满足条件的行建索引 | `CREATE INDEX idx_partial ON orders(status) WHERE status = 'active'` |
| **唯一索引** | 保证索引列的值唯一 | `CREATE UNIQUE INDEX idx_unique ON users(email)` |
| **多值索引** | 索引数组或 JSON 数组中的元素 | MySQL 8.0 的多值索引 |

## 索引的三大核心权衡

设计索引时，必须在三个互相矛盾的目标之间做取舍。理解这些权衡是避免"过度索引"或"索引不足"的关键。

### 权衡一：读性能 vs 写性能

索引加速读操作，但拖慢写操作。每次 INSERT 需要在所有相关索引中插入新条目；每次 UPDATE 需要更新被修改列对应的索引条目；每次 DELETE 需要从所有索引中移除条目。

以 PostgreSQL 为例，实测数据大致如下：

场景：100万行表，5个索引，单条 INSERT

无索引：  ~2,000 QPS（仅写 WAL + 堆表）
1个索引：  ~800 QPS（下降 60%）
5个索引：  ~300 QPS（下降 85%）

写放大的程度取决于索引类型。B+tree 索引的写放大通常在 2-5 倍，而 LSM-Tree 的写放大（由于 Compaction）可能达到 10-30 倍——但 LSM-Tree 将随机写转换为顺序写，在 SSD 上顺序写吞吐远高于随机写，所以总体写入性能可能反而更高。

### 权衡二：查询优化 vs 存储开销

每个索引都占用存储空间。在一个有 20 个索引的大型表中，索引的总空间可能超过数据本身。过多的索引不仅浪费磁盘，还会：
- 增加缓冲池（Buffer Pool）的压力，更多索引页面竞争有限的内存空间
- 延长备份和恢复的时间
- 增加 VACUUM（PostgreSQL）或 OPTIMIZE（MySQL）的维护时间

经验法则：索引的总大小通常不应超过数据表大小的 50%。超过这个比例需要审查每个索引是否真的在被使用。

### 权衡三：查询覆盖 vs 维护复杂度

一个设计精良的覆盖索引可以避免回表，大幅提升查询性能。但覆盖索引往往包含更多列，导致：
- 索引体积增大
- 写放大增加（更多列需要维护）
- 索引选择变得复杂（需要仔细分析查询模式）

例如，为了优化一条常用的查询：

```sql
SELECT name, email, created_at
FROM users
WHERE status = 'active'
ORDER BY created_at DESC
LIMIT 20;
```

一个理想的覆盖索引是 `(status, created_at DESC, name, email)`。但这意味着每次 INSERT 或 UPDATE 涉及 `status`、`created_at`、`name`、`email` 任何一个字段时，都需要更新此索引。

## 索引的 I/O 模型

理解索引性能的关键在于理解 I/O 模型。数据库索引的所有操作最终都是对磁盘页面的读写。

### 页面（Page）—— 索引的基本 I/O 单位

索引的每个节点对应磁盘上的一个页面（也称为块，Block）。现代数据库的页面大小通常为：

| 数据库 | 默认页面大小 | 说明 |
|--------|------------|------|
| PostgreSQL | 8 KB | 编译时固定，不可运行时修改 |
| InnoDB (MySQL) | 16 KB | 可通过 innodb_page_size 配置 |
| SQL Server | 8 KB | 可选 4KB/8KB/16KB/32KB/64KB |
| SQLite | 4 KB（默认） | 可在编译时调整，最大 65536 |

页面大小直接影响索引的扇出（fanout）和树高。以 B+tree 内部节点为例：

扇出 = 页面大小 / (键值大小 + 指针大小)

以 PostgreSQL 8KB 页、主键 INT(4字节)、指针(4字节) 为例：
扇出 = 8192 / (4 + 4) = 1024

树高 = log₁₀₂₄(10亿) ≈ 3层

### 页面访问模式

索引查询产生的页面访问可以分为两类：

**顺序访问（Sequential Access）**：读取连续的页面，充分利用操作系统的预读（Read-ahead）机制和磁盘的顺序读带宽。LSM-Tree 的 Compaction、范围扫描的叶子节点遍历都是顺序访问。

**随机访问（Random Access）**：读取不连续的页面，每次访问都是一次独立的磁盘寻道。B+tree 从根到叶的路径遍历、Hash 索引的桶查找都是随机访问。

SSD 相比 HDD 大幅降低了随机读延迟（HDD ~10ms vs SSD ~0.1ms），但顺序读的优势仍然存在（SSD 顺序带宽通常是随机读吞吐的 3-5 倍）。因此，减少页面访问次数（特别是随机访问）仍然是索引优化的核心目标。

### 缓冲池与页面替换

热数据的索引页面通常驻留在内存的缓冲池（Buffer Pool / Buffer Cache）中。查询的性能很大程度上取决于**缓存命中率**：

缓存命中率 = 命中缓存的页面访问次数 / 总页面访问次数

B+tree 4层查询：
  第1层(根): 100% 命中（常驻内存）
  第2层:     ~99% 命中（通常能全放在内存中）
  第3层:     ~95% 命中（取决于索引大小和内存）
  第4层(叶): ~90% 命中（取决于数据量和访问模式）
  综合命中率: ~94%

当缓存命中率低于 95% 时，查询性能会急剧下降——因为每次未命中都意味着一次磁盘随机读。

## 索引操作的基本语义

理解索引的搜索、插入、删除操作，是理解后续具体实现的基础。

### 等值查找（Point Lookup）

给定一个键值 K，找到所有键值等于 K 的记录。这是最基本的操作，所有索引类型都支持。

B+tree 的等值查找从根节点开始，每层用二分查找确定子节点方向，直到到达叶子节点。时间复杂度 O(log_m N)，其中 m 是扇出，N 是记录总数。

Hash 索引的等值查找通过哈希函数直接定位桶，时间复杂度 O(1) 均摊。

### 范围查找（Range Scan）

找到键值在 [K_low, K_high] 范围内的所有记录。B+tree 天然支持范围查找——先定位 K_low 所在的叶子节点，然后沿着叶子链表顺序扫描直到 K_high。时间复杂度 O(log_m N + k)，其中 k 是结果集大小。

Hash 索引不支持范围查找，因为它不维护键值的顺序。这是 Hash 索引最大的局限——在实际应用中，大量的查询都需要范围条件（如日期范围、价格区间）。

### 前缀匹配（Prefix Scan）

找到所有以给定前缀开头的记录。B+tree 通过键值的有序性天然支持前缀匹配。倒排索引也擅长前缀匹配，但机制完全不同——它通过词项的前缀在词典中查找。

### 最近邻/最近记录（Next/Previous）

找到键值排序后紧邻给定键的前一条或后一条记录。B+tree 通过叶子节点的双向链表高效支持。这在实现 ORDER BY + LIMIT 优化时至关重要。

### 插入与分裂

当新记录插入导致索引节点溢出时，需要进行节点分裂。B+tree 的分裂可能向上传播到根节点，最坏情况下树高增加 1。分裂操作的均摊代价为 O(1)——因为每次分裂将一个满节点变为两个半满节点，至少需要插入 m/2 个新键值才会触发下一次分裂。

### 删除与合并

当删除记录导致索引节点下溢时，需要从兄弟节点借键值（重分配）或与兄弟节点合并。合并操作可能向上传播，最坏情况下树高减少 1。在实际系统中，PostgreSQL 等采用"延迟合并"策略，不在删除时立即合并，而是在后续的清理操作中统一处理。

## 索引实现的四大核心挑战

从工程实现的角度，任何一个生产级索引都必须同时解决以下四个问题。这四个问题的解决方式直接决定了索引的实现方案。

### 挑战一：并发控制

在多线程/多连接环境下，多个操作可能同时修改同一个索引节点。简单的锁（如对整个索引加锁）会导致严重的性能瓶颈——想象 100 个查询排队等一把锁。

解决方案的演进：

粗粒度锁（整个索引） → 细粒度锁（节点级Latch） → Lock Coupling → Crabbing → 无锁方案
   ↓                      ↓                       ↓               ↓              ↓
 简单但慢              较好但仍有竞争          经典方案          更安全          极端场景

**Latch（闩锁）** 是索引内部使用的轻量级同步原语，保护单个页面的并发访问。与事务级的 Lock 不同，Latch 的生命周期极短（一次页面操作），不需要死锁检测机制。

**Lock Coupling（锁耦合/手锁协议）**：访问子节点时才获取子节点的 Latch，释放父节点的 Latch。这避免了长时间持有根节点锁导致的瓶颈，但需要小心处理并发分裂时的竞态条件。

**Crabbing（蟹行协议）**：访问子节点时获取子节点的 Latch，但只有在确认子节点安全（不会分裂或合并）时才释放父节点的 Latch。这比 Lock Coupling 更安全，但可能持有更多锁。

### 挑战二：崩溃恢复

数据库系统必须保证：即使在写入过程中发生崩溃（断电、内核 panic），索引数据也不会损坏。这要求索引实现遵循 **WAL（Write-Ahead Logging）** 协议：任何修改必须先记录到日志，然后才应用到索引页面。

以 B+tree 的节点分裂为例，正确的崩溃恢复流程是：

1. 写入 WAL 记录：记录"即将在页面 X 上执行分裂操作"
2. 分配新页面 Y
3. 将旧页面 X 的后半部分复制到新页面 Y
4. 更新父节点，添加指向 Y 的指针
5. 将修改后的页面刷入磁盘
6. 在 WAL 中写入"分裂完成"标记

如果崩溃发生在步骤 3 和步骤 5 之间，恢复时可以回滚到步骤 1 之前的状态。如果崩溃发生在步骤 5 之后，恢复时可以重放 WAL 完成分裂。

不同的索引结构有不同的恢复复杂度。B+tree 的恢复相对直接，因为每次修改影响的页面数有限。LSM-Tree 的 Compaction 涉及多个文件的原子替换，恢复逻辑更加复杂——这也是为什么 WAL 对 LSM-Tree 至关重要。

### 挑战三：空间管理

索引在生命周期中会经历不断的插入、删除和分裂、合并，导致页面出现碎片化。空间管理的核心问题是：

**空闲空间追踪**：如何高效地找到可以容纳新数据的空闲页面？常用的方案包括空闲空间映射（Free Space Map）、空闲链表等。

**碎片整理**：经过大量写入后，页面的填充率可能大幅下降（例如从 70% 降到 30%）。PostgreSQL 的 VACUUM 操作负责回收死元组空间并整理碎片；MySQL InnoDB 通过后台线程的碎片整理来维持页面填充率。

**预分配与水位线**：为了避免频繁的页面分配/释放操作，索引通常会预分配一批连续页面作为预留空间。PostgreSQL 使用"High Water Mark"机制——删除操作只标记页面可重用，不立即归还给操作系统，而是在积累到一定程度后批量释放。

### 挑战四：顺序扩展（Sequential Key Problem）

当索引键值是顺序递增的（如自增 ID、时间戳），每次插入都发生在 B+tree 最右侧的叶子节点。这导致两个问题：

**右偏树（Right-Skewed Tree）**：B+tree 退化为近似链表，右侧节点满而左侧节点空，空间利用率极低。

**热点竞争**：所有并发插入都争抢最右侧节点的 Latch，形成严重的锁竞争。

解决方案包括：
- PostgreSQL 的 **FastPath 优化**：对最右侧叶子节点使用专用的 fast-path Latch，避免根节点的锁竞争
- **右填充策略**：允许最右侧节点超过正常容量（高水位线），减少分裂频率
- **间隙锁（Gap Lock）**：在插入时锁住键值间隙，防止幻读但允许并发插入
- B-tree 的**延迟分裂**：在分裂条件上添加滞后区间，避免频繁的分裂-合并震荡

## 索引代价模型概述

数据库查询优化器在选择索引时，需要估算不同索引方案的代价。理解这个代价模型有助于设计高效的索引策略。

### 代价评估的核心变量

C(索引查找) = 页面访问次数 × 每次页面访问的代价
             = I/O 代价 + CPU 代价（比较操作）

在大多数场景下，I/O 代价远大于 CPU 代价，因此优化目标简化为**最小化页面访问次数**。

### 选择率（Selectivity）

选择率是查询返回的行数占总行数的比例。它是索引选择的关键因素：

选择率 = 满足条件的行数 / 总行数

例：WHERE age > 30，在100万行表中返回70万行
选择率 = 70% → 全表扫描更优（索引回表开销 > 顺序扫描开销）

例：WHERE id = 42，在100万行表中返回1行
选择率 = 0.0001% → 索引查找最优

**经验法则**：当选择率高于 10%-20% 时，全表扫描通常优于索引查找。但这个阈值受到缓存命中率、索引覆盖程度等因素影响。

### 成本公式

对于 B+tree 索引查找：

代价 = 树高（页面访问） + 结果集大小 × 回表代价

如果索引是覆盖索引：代价 = 树高（无需回表）
如果需要回表：代价 = 树高 + 结果行数 × 回表随机I/O代价

这意味着即使索引查找本身很快，如果结果集很大且回表代价高，查询优化器可能会选择全表扫描而不是索引。这就是为什么**覆盖索引**和**延迟关联（Deferred Join）** 在优化大结果集查询时如此重要。

## 本节要点回顾

本节建立了理解索引实现的概念框架，核心要点如下：

1. **索引是空间换时间的辅助数据结构**，但远比目录复杂——必须同时处理并发、崩溃恢复、空间管理等问题
2. **量化分析显示**，索引可以将查询从秒级降到亚毫秒级，但代价是写入放慢 2-5 倍、存储增加 20%-100%
3. **索引分类**可从三个正交维度展开：数据结构（B+tree/Hash/LSM/倒排）、存储组织（聚簇/非聚簇）、功能特性（单列/复合/前缀/函数）
4. **三大核心权衡**：读性能 vs 写性能、查询优化 vs 存储开销、覆盖索引 vs 维护复杂度
5. **I/O 模型**是理解性能的关键：减少随机页面访问次数是一切优化的出发点
6. **四大实现挑战**：并发控制（Latch 机制）、崩溃恢复（WAL 协议）、空间管理（碎片整理）、顺序键热点（右填充策略）
7. **代价模型**帮助查询优化器做出正确的索引选择——选择率和覆盖程度是两个最重要的因素

后续各节将深入每种索引结构的具体实现细节。理解这些核心概念后，你将能够：

- 看到一种索引结构时，快速判断它解决了哪些问题、留下了哪些权衡
- 在设计索引时，基于实际工作负载做出量化的决策
- 阅读数据库内核代码时，理解每个实现选择背后的设计意图
