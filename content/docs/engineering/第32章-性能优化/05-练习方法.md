## 练习一：用火焰图定位热点函数（入门 · 30分钟）

### 目标

掌握性能分析的第一步——用 `perf` + FlameGraph 定位程序的 CPU 热点。这是所有优化的起点：在你不知道慢在哪里之前，任何优化都是盲目的。

### 前置知识

- Linux 命令行基础
- 已阅读本章第1节"性能分析与基准测试"

### 环境准备

```bash
# Ubuntu/Debian
sudo apt-get update &amp;&amp; sudo apt-get install -y linux-tools-common linux-tools-$(uname -r) git

# 安装 FlameGraph 脚本（ Brendan Gregg 的工具集）

# 验证 perf 可用
perf stat ls > /dev/null 2>&amp;1 &amp;&amp; echo "perf OK" || echo "perf 需要 root 或调整 perf_event_paranoid"
```

### 步骤一：准备一个有热点的程序（5分钟）

创建一个故意有性能问题的 Python 脚本：

```python
# /tmp/perf_target.py
import hashlib
import json
import os

def cpu_intensive_hash(n):
    """CPU 密集：大量哈希计算 — 将成为主要热点"""
    result = b""
    for i in range(n):
        result = hashlib.sha256(result + str(i).encode()).digest()
    return result

def io_bound_read():
    """IO 密集：读取 /etc/passwd"""
    data = b""
    for _ in range(100):
        with open("/etc/passwd", "rb") as f:
            data = f.read()
    return data

def memory_intensive():
    """内存密集：大量列表操作"""
    data = list(range(1_000_000))
    _ = sorted(data, reverse=True)
    return len(data)

def main():
    # 各函数调用比例可调，模拟真实场景中的热点分布
    for _ in range(3):
        cpu_intensive_hash(50000)
    for _ in range(10):
        io_bound_read()
    for _ in range(5):
        memory_intensive()

if __name__ == "__main__":
    main()
```

### 步骤三：分析火焰图（10分钟）

打开火焰图 SVG 文件后，掌握以下阅读技巧：

| 火焰图特征 | 含义 | 优化方向 |
|-----------|------|---------|
| 宽底座 | 该函数自身消耗大量 CPU | 直接优化函数内部逻辑 |
| 长塔尖 | 调用链很深 | 考虑减少调用层级或缓存中间结果 |
| 平坦宽块 | 递归或循环热点 | 优化算法复杂度或减少迭代次数 |
| 窄柱子 | 非瓶颈 | 不要花时间优化这里 |

**在本例中**，你应该能看到 `cpu_intensive_hash` 占据了火焰图最宽的部分，`sha256` 的 `digest` 方法是底层热点。这验证了我们的分析：CPU 热点在哈希计算，不在 IO 或内存操作。

### 步骤一：O(n²) → O(n) 去重（10分钟）

```python
# /tmp/algo_bench.py
import timeit
import random

random.seed(42)
DATA_SIZE = 50000
data = [random.randint(0, DATA_SIZE * 2) for _ in range(DATA_SIZE)]

# === 方法 A: O(n²) 嵌套循环去重 ===
def dedup_slow(lst):
    result = []
    for item in lst:
        if item not in result:  # list 的 in 是 O(n) 扫描
            result.append(item)
    return result

# === 方法 B: O(n) 哈希集合去重 ===
def dedup_fast(lst):
    seen = set()
    result = []
    for item in lst:
        if item not in seen:  # set 的 in 是 O(1)
            seen.add(item)
            result.append(item)
    return result

# === 基准测试 ===
t_slow = timeit.timeit(lambda: dedup_slow(data), number=3)
t_fast = timeit.timeit(lambda: dedup_fast(data), number=3)

print(f"O(n²) 去重: {t_slow:.3f}s (3次平均 {t_slow/3:.3f}s/次)")
print(f"O(n)  去重: {t_fast:.3f}s (3次平均 {t_fast/3:.3f}s/次)")
print(f"加速比: {t_slow/t_fast:.1f}x")
print(f"正确性: {dedup_slow(data) == dedup_fast(data)}")
```

### 步骤二：O(n log n) → O(n) 用堆找 Top-K（15分钟）

```python
import heapq
import timeit

random.seed(42)
LARGE_DATA = [random.random() for _ in range(1_000_000)]
K = 100

# === 方法 A: 全量排序 O(n log n) ===
def topk_sort(data, k):
    return sorted(data, reverse=True)[:k]

# === 方法 B: 堆 O(n log k)，k << n 时优势明显 ===
def topk_heap(data, k):
    return heapq.nlargest(k, data)

t_sort = timeit.timeit(lambda: topk_sort(LARGE_DATA, K), number=5)
t_heap = timeit.timeit(lambda: topk_heap(LARGE_DATA, K), number=5)

print(f"排序取Top-K: {t_sort:.3f}s (5次平均 {t_sort/5:.3f}s)")
print(f"堆取Top-K:   {t_heap:.3f}s (5次平均 {t_heap/5:.3f}s)")
print(f"加速比: {t_sort/t_heap:.1f}x")
print(f"正确性: {topk_sort(LARGE_DATA, K) == topk_heap(LARGE_DATA, K)}")
```

### 步骤三：缓存友好的数据布局（20分钟）

理解"数据在内存中的排列方式"如何影响性能：

```python
import timeit

# === 场景：遍历对象的某个字段 ===

# 反面教材：数组 of 对象（指针追逐，cache miss 频繁）
class PointBad:
    __slots__ = ('x', 'y', 'z')  # __slots__ 减少内存开销但不解决布局问题
    def __init__(self, x, y, z):
        self.x = x
        self.y = y
        self.z = z

# 正面教材：结构体数组 SoA（连续内存，cache 友好）
class PointArray:
    def __init__(self, n):
        self.x = [0.0] * n
        self.y = [0.0] * n
        self.z = [0.0] * n

N = 100000

# 方式 A：AoS 遍历
points_bad = [PointBad(1.0, 2.0, 3.0) for _ in range(N)]

def sum_x_aos(points):
    total = 0.0
    for p in points:
        total += p.x  # 每次访问可能触发 cache miss
    return total

t_aos = timeit.timeit(lambda: sum_x_aos(points_bad), number=100)

# 方式 B：SoA 遍历
points_soa = PointArray(N)
for i in range(N):
    points_soa.x[i] = 1.0

def sum_x_soa(points):
    total = 0.0
    for val in points.x:  # 连续内存访问，prefetch 友好
        total += val
    return total

t_soa = timeit.timeit(lambda: sum_x_soa(points_soa), number=100)

print(f"AoS 遍历 X: {t_aos:.3f}s")
print(f"SoA 遍历 X: {t_soa:.3f}s")
print(f"加速比: {t_aos/t_soa:.1f}x")
print(f"注: C/C++/Rust 中差距更大（Python 对象开销掩盖了布局差异）")
```

> **为什么 Python 中差距不大？** Python 的 list 本身就是指针数组，每个元素都是 PyObject 指针，内存布局在两种方式下差异不大。真正的 SoA 优势体现在 C/C++/Rust 中，用 `struct` 数组对比指针数组，差距可达 3-10 倍。

### 步骤一：量化内存分配的代价（15分钟）

```python
# /tmp/mem_alloc_bench.py
import timeit
import sys

# 场景：频繁创建和销毁小对象（如 HTTP 请求/响应对象）

class Request:
    def __init__(self, method, path, headers, body):
        self.method = method
        self.path = path
        self.headers = headers
        self.body = body

METHODS = ["GET", "POST", "PUT", "DELETE"]
PATHS = ["/api/users", "/api/orders", "/api/products", "/api/auth/login"]

# === 方式 A：每次 new 一个对象（标准做法）===

def handle_requests_new(n):
    results = []
    for i in range(n):
        req = Request(
            METHODS[i % 4],
            PATHS[i % 4],
            {"Content-Type": "application/json"},
            f'{{"id": {i}}}'
        )
        results.append(req.method + req.path)
    return results

# === 方式 B：对象池复用 ===
class RequestPool:
    def __init__(self, size=1000):
        self._pool = [Request("", "", {}, "") for _ in range(size)]
        self._available = list(range(size))
    
    def acquire(self, method, path, headers, body):
        if self._available:
            idx = self._available.pop()
            req = self._pool[idx]
            req.method = method
            req.path = path
            req.headers = headers
            req.body = body
            return idx, req
        # 池耗尽时 fallback 到 new
        return -1, Request(method, path, headers, body)
    
    def release(self, idx):
        if idx >= 0:
            self._available.append(idx)

pool = RequestPool(2000)

def handle_requests_pool(n):
    results = []
    for i in range(n):
        idx, req = pool.acquire(
            METHODS[i % 4],
            PATHS[i % 4],
            {"Content-Type": "application/json"},
            f'{{"id": {i}}}'
        )
        results.append(req.method + req.path)
        pool.release(idx)
    return results

N = 100_000

t_new = timeit.timeit(lambda: handle_requests_new(N), number=3)
t_pool = timeit.timeit(lambda: handle_requests_pool(N), number=3)

print(f"每次 new:    {t_new:.3f}s ({N} 次分配)")
print(f"对象池复用:  {t_pool:.3f}s ({N} 次复用)")
print(f"加速比: {t_new/t_pool:.1f}x")
print(f"内存节省: {sys.getsizeof(Request('', '', {}, '')) * N / 1024 / 1024:.1f} MB → 近零分配")
```

### 步骤二：Arena 分配器模拟（20分钟）

Arena 分配器的核心思想：一次性申请大块内存，按需切分，整体释放。这在编译器、解析器、游戏引擎中广泛使用。

```python
# /tmp/arena_sim.py
import timeit
import ctypes

class Arena:
    """简化版 Arena 分配器模拟"""
    def __init__(self, block_size=65536):
        self.block_size = block_size
        self.buffer = bytearray(block_size)
        self.offset = 0
        self.alloc_count = 0
        self.alloc_bytes = 0
    
    def alloc(self, size):
        """从 arena 中分配 size 字节（bump pointer）"""
        # 对齐到 8 字节
        aligned_size = (size + 7) &amp; ~7
        
        if self.offset + aligned_size > self.block_size:
            raise MemoryError("Arena exhausted")
        
        start = self.offset
        self.offset += aligned_size
        self.alloc_count += 1
        self.alloc_bytes += aligned_size
        return start
    
    def reset(self):
        """一次性释放所有分配 — O(1)"""
        self.offset = 0
        self.alloc_count = 0
        self.alloc_bytes = 0
    
    def stats(self):
        return {
            "allocations": self.alloc_count,
            "bytes_used": self.alloc_bytes,
            "fragmentation": 1.0 - (self.alloc_bytes / max(self.offset, 1))
        }

# 对比: 逐个分配 vs Arena 分配
arena = Arena(block_size=1024 * 1024)  # 1MB Arena

def alloc_with_arena(n):
    """Arena 分配 + 统一释放"""
    arena.reset()
    for i in range(n):
        arena.alloc(64 + (i % 128))  # 变长分配
    return arena.stats()

def alloc_with_list(n):
    """标准 list 追加（模拟逐个 malloc）"""
    data = []
    for i in range(n):
        data.append(bytearray(64 + (i % 128)))
    # Python GC 负责释放
    return len(data)

N = 10000

t_arena = timeit.timeit(lambda: alloc_with_arena(N), number=100)
t_list = timeit.timeit(lambda: alloc_with_list(N), number=100)

print(f"Arena 分配: {t_arena:.3f}s ({N} 次, 100轮)")
print(f"逐个分配:  {t_list:.3f}s ({N} 次, 100轮)")
print(f"加速比: {t_list/t_arena:.1f}x")
print(f"Arena 统计: {alloc_with_arena(N)}")
print()
print("Arena 分配器优势:")
print("  1. 分配极快: 只需移动指针，无 malloc/free 开销")
print("  2. 零碎片: 连续分配，无外部碎片")
print("  3. 释放极快: reset() 是 O(1)，不需要逐个 free")
print("  4. 适用场景: 请求级生命周期（每个请求一个 arena，请求结束后整体释放）")
```

### 步骤三：使用 memory_profiler 诊断内存（25分钟）

```python
# 需要先安装: pip install memory_profiler
# /tmp/mem_diagnose.py
import sys

def process_data():
    """模拟一个有内存问题的函数"""
    # 问题 1: 不必要的大列表拷贝
    raw_data = list(range(10_000_000))  # ~80MB
    
    # 问题 2: 中间变量没有及时释放
    filtered = [x for x in raw_data if x % 2 == 0]  # ~40MB
    
    # 问题 3: 字符串拼接产生大量临时对象
    result = ""
    for i in filtered[:10000]:
        result += f"item_{i},"  # 每次 += 都创建新字符串
    
    # 修复方案:
    # 1. 用生成器替代列表
    # 2. 用 del + gc.collect() 及时释放
    # 3. 用 join() 或 io.StringIO 替代 +=

    del raw_data  # 显式释放大对象
    del filtered
    
    return len(result)

if __name__ == "__main__":
    process_data()
```

## 练习四：无锁队列 vs 有锁队列（高级 · 60分钟）

### 目标

通过实现并对比有锁和无锁队列在高并发下的性能差异，深入理解并发优化的核心理念：减少同步开销是提升并发性能的关键。

### 步骤一：有锁队列实现（Go，20分钟）

```go
// /tmp/lock_queue.go
package main

import (
    "fmt"
    "sync"
    "sync/atomic"
    "time"
)

// === 有锁队列 ===
type LockedQueue struct {
    mu    sync.Mutex
    items []int
}

func NewLockedQueue(cap int) *LockedQueue {
    return &amp;LockedQueue{items: make([]int, 0, cap)}
}

func (q *LockedQueue) Push(item int) {
    q.mu.Lock()
    q.items = append(q.items, item)
    q.mu.Unlock()
}

func (q *LockedQueue) Pop() (int, bool) {
    q.mu.Lock()
    defer q.mu.Unlock()
    if len(q.items) == 0 {
        return 0, false
    }
    item := q.items[0]
    q.items = q.items[1:]  // 注意: 这里有内存泄漏风险，实际应用应使用环形缓冲区
    return item, true
}

func (q *LockedQueue) Len() int {
    q.mu.Lock()
    defer q.mu.Unlock()
    return len(q.items)
}

func main() {
    const (
        OPS_PER_WORKER = 100000
        WORKERS        = 8
    )

    // === 测试有锁队列 ===
    lq := NewLockedQueue(OPS_PER_WORKER * WORKERS)
    var lockOps int64

    start := time.Now()
    var wg sync.WaitGroup
    for w := 0; w < WORKERS; w++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            var localOps int64
            for i := 0; i < OPS_PER_WORKER; i++ {
                lq.Push(id*OPS_PER_WORKER + i)
            }
            for lq.Len() > 0 {
                lq.Pop()
                localOps++
            }
            atomic.AddInt64(&amp;lockOps, localOps)
        }(w)
    }
    wg.Wait()
    lockDuration := time.Since(start)

    fmt.Printf("有锁队列:   %d ops in %v (%.0f ops/sec)\n",
        lockOps, lockDuration,
        float64(lockOps)/lockDuration.Seconds())

    // === 对比: 单 goroutine 无锁基准 ===
    start = time.Now()
    count := 0
    for i := 0; i < OPS_PER_WORKER*WORKERS; i++ {
        count++
    }
    singleDuration := time.Since(start)
    fmt.Printf("单线程基准: %d ops in %v (%.0f ops/sec)\n",
        count, singleDuration,
        float64(count)/singleDuration.Seconds())
    
    fmt.Printf("并发开销: %.1fx 减速 (锁竞争代价)\n",
        float64(lockDuration.Nanoseconds())/float64(singleDuration.Nanoseconds()))
}
```

### 步骤二：无锁队列实现（Go，20分钟）

```go
// /tmp/lockfree_queue.go
package main

import (
    "fmt"
    "runtime"
    "sync"
    "sync/atomic"
    "time"
    "unsafe"
)

// === 无锁队列 (Michael-Scott Queue 简化版) ===
type Node struct {
    value int
    next  unsafe.Pointer // *Node
}

type LockFreeQueue struct {
    head unsafe.Pointer // *Node
    tail unsafe.Pointer // *Node
}

func NewLockFreeQueue() *LockFreeQueue {
    // 哨兵节点: head 和 tail 都指向它
    sentinel := &amp;Node{value: 0, next: nil}
    return &amp;LockFreeQueue{
        head: unsafe.Pointer(sentinel),
        tail: unsafe.Pointer(sentinel),
    }
}

func (q *LockFreeQueue) Push(value int) {
    node := &amp;Node{value: value, next: nil}
    for {
        tail := (*Node)(atomic.LoadPointer(&amp;q.tail))
        next := (*Node)(atomic.LoadPointer(&amp;tail.next))
        
        // 检查 tail 是否已被其他 goroutine 移动
        if tail == (*Node)(atomic.LoadPointer(&amp;q.tail)) {
            if next == nil {
                // tail.next 为 nil，尝试 CAS 链接新节点
                if atomic.CompareAndSwapPointer(
                    &amp;tail.next,
                    unsafe.Pointer(next),
                    unsafe.Pointer(node),
                ) {
                    // 链接成功，尝试移动 tail 到新节点
                    atomic.CompareAndSwapPointer(
                        &amp;q.tail,
                        unsafe.Pointer(tail),
                        unsafe.Pointer(node),
                    )
                    return
                }
            } else {
                // tail 滞后了，帮忙推进
                atomic.CompareAndSwapPointer(
                    &amp;q.tail,
                    unsafe.Pointer(tail),
                    unsafe.Pointer(next),
                )
            }
        }
    }
}

func (q *LockFreeQueue) Pop() (int, bool) {
    for {
        head := (*Node)(atomic.LoadPointer(&amp;q.head))
        tail := (*Node)(atomic.LoadPointer(&amp;q.tail))
        next := (*Node)(atomic.LoadPointer(&amp;head.next))
        
        if head == (*Node)(atomic.LoadPointer(&amp;q.head)) {
            if head == tail {
                if next == nil {
                    return 0, false // 队列为空
                }
                // tail 落后了，帮忙推进
                atomic.CompareAndSwapPointer(
                    &amp;q.tail,
                    unsafe.Pointer(tail),
                    unsafe.Pointer(next),
                )
            } else {
                value := next.value
                // CAS 移动 head
                if atomic.CompareAndSwapPointer(
                    &amp;q.head,
                    unsafe.Pointer(head),
                    unsafe.Pointer(next),
                ) {
                    return value, true
                }
            }
        }
    }
}

func main() {
    runtime.GOMAXPROCS(runtime.NumCPU())
    
    const (
        OPS_PER_WORKER = 100000
        WORKERS        = 8
    )

    // === 测试无锁队列 ===
    lq := NewLockFreeQueue()
    var lockfreeOps int64

    start := time.Now()
    var wg sync.WaitGroup
    for w := 0; w < WORKERS; w++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            var localOps int64
            for i := 0; i < OPS_PER_WORKER; i++ {
                lq.Push(id*OPS_PER_WORKER + i)
            }
            for {
                if _, ok := lq.Pop(); ok {
                    localOps++
                } else {
                    break
                }
            }
            atomic.AddInt64(&amp;lockfreeOps, localOps)
        }(w)
    }
    wg.Wait()
    lockfreeDuration := time.Since(start)

    fmt.Printf("无锁队列:   %d ops in %v (%.0f ops/sec)\n",
        lockfreeOps, lockfreeDuration,
        float64(lockfreeOps)/lockfreeDuration.Seconds())
    
    // 对比有锁（可从练习一的结果对比）
    fmt.Println()
    fmt.Println("对比要点:")
    fmt.Println("  1. 无锁在高竞争场景下通常比有锁快 2-5 倍")
    fmt.Println("  2. 但无锁的实现复杂度远高于有锁")
    fmt.Println("  3. 无锁不等于无等待: CAS 重试在高竞争下可能浪费 CPU")
    fmt.Println("  4. 无锁队列的 pop 在空队列时会自旋，需要配合 backoff")
}
```

| 指标 | 有锁队列 | 无锁队列 | 差异分析 |
|------|---------|---------|---------|
| 总吞吐 (ops/sec) | ___ | ___ | |
| 锁竞争开销 | 高 — mutex 排队等待 | 低 — CAS 竞争 | |
| CPU 利用率 | 可能有线程阻塞 | 全部在用户态 | |

**关键洞察**：

1. **无锁并非总是更快**：在低竞争（< 4 个 goroutine）时，mutex 的开销很小（~20ns），无锁的 CAS 自旋反而更贵
2. **无锁的真正价值**：在高竞争（8+ goroutine）且操作极快（< 100ns）时，mutex 的排队等待成为瓶颈
3. **正确的选择**：先用有锁版本，profiling 显示锁竞争是瓶颈后再切换到无锁。Premature lock-free optimization 同样是万恶之源

var (

    users   = make(map[int]User)
    nextID  = 1

type User struct {
    ID    int    `json:"id"`
    Name  string `json:"name"`
    Email string `json:"email"`

func init() {

func getUserHandler(w http.ResponseWriter, r *http.Request) {
    idStr := r.URL.Query().Get("id")
    id, _ := strconv.Atoi(idStr)
    
    mu.Lock()  // 问题: 全局锁，所有请求排队
    defer mu.Unlock()
    
    user, ok := users[id]
    if !ok {
        http.Error(w, "not found", 404)
        return
    }
    
    // 问题 2: 每次都序列化整个响应
    data, _ := json.Marshal(user)
    w.Header().Set("Content-Type", "application/json")
    w.Write(data)
}

// === 性能问题 3: 慢查询模拟 ===
func searchHandler(w http.ResponseWriter, r *http.Request) {
    keyword := r.URL.Query().Get("q")
    
    mu.Lock()
    defer mu.Unlock()
    
    var results []User
    // 问题: 全表扫描 + 字符串匹配
    for _, user := range users {
        if strings.Contains(user.Name, keyword) || 
           strings.Contains(user.Email, keyword) {
            results = append(results, user)
        }
        time.Sleep(time.Microsecond) // 模拟慢查询
    }
    
    data, _ := json.Marshal(results)
    w.Header().Set("Content-Type", "application/json")
    w.Write(data)
}

func main() {
    http.HandleFunc("/user", getUserHandler)
    http.HandleFunc("/search", searchHandler)
    
    port := os.Getenv("PORT")
    if port == "" {
        port = "8080"
    }
    log.Printf("Server starting on :%s", port)

var rwmu sync.RWMutex

func getUserHandler(w http.ResponseWriter, r *http.Request) {
    idStr := r.URL.Query().Get("id")
    id, _ := strconv.Atoi(idStr)

// 更好的方案: 用 sync.Map 消除锁竞争
var syncUsers sync.Map

func init() {
    for i := 1; i <= 10000; i++ {
        syncUsers.Store(i, User{
            ID:    i,
            Name:  fmt.Sprintf("User_%d", i),
            Email: fmt.Sprintf("user%d@example.com", i),
        })
    }
}

func getUserHandlerV2(w http.ResponseWriter, r *http.Request) {
    idStr := r.URL.Query().Get("id")

    val, ok := syncUsers.Load(id)  // 无锁读

        return
    }

}

**修复 3：建立搜索索引（15分钟）**

```go
// 建立倒排索引: 关键词 -> 用户ID列表
var searchIndex map[string][]int
var indexMu sync.RWMutex

func buildIndex() {
    indexMu.Lock()
    defer indexMu.Unlock()
    
    searchIndex = make(map[string][]int)
    syncUsers.Range(func(key, value interface{}) bool {
        user := value.(User)
        // 对 name 和 email 的每个 token 建立索引
        for _, token := range tokenize(user.Name + " " + user.Email) {
            searchIndex[token] = append(searchIndex[token], user.ID)
        }
        return true
    })
}

func tokenize(s string) []string {
    s = strings.ToLower(s)
    words := strings.FieldsFunc(s, func(r rune) bool {
        return !((r >= 'a' &amp;&amp; r <= 'z') || (r >= '0' &amp;&amp; r <= '9'))
    })
    return words
}

func searchHandlerV2(w http.ResponseWriter, r *http.Request) {
    keyword := strings.ToLower(r.URL.Query().Get("q"))
    
    indexMu.RLock()
    ids, ok := searchIndex[keyword]
    indexMu.RUnlock()
    
    if !ok {
        w.Header().Set("Content-Type", "application/json")
        w.Write([]byte("[]"))
        return
    }
    
    var results []User
    for _, id := range ids {
        if val, ok := syncUsers.Load(id); ok {
            results = append(results, val.(User))
        }
    }
    
    data, _ := json.Marshal(results)
    w.Header().Set("Content-Type", "application/json")
    w.Write(data)
}
```
