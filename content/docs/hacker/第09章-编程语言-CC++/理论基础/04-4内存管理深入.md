## 4. 内存管理深入

内存管理是 C/C++ 安全攻防的核心战场。不同于 Java、Python 等带垃圾回收的语言，C/C++ 将内存分配与释放的全部责任交给程序员，由此产生了一系列经典漏洞——堆溢出、Use-After-Free、Double Free——至今仍占据 CVE 漏洞排行榜前列。本章从分配器底层机制讲起，逐层拆解 glibc malloc 的工作原理、C++ 内存语义，以及攻击者如何利用这些机制实现代码执行。

### 4.1 内存分配的两条路径：brk 与 mmap

glibc 的 `malloc` 底层并不直接管理物理内存，而是通过两个系统调用向内核申请虚拟地址空间：

```text
malloc 决策流程：

malloc(size)
    │
    ├─ size < 128KB（默认阈值，M_MMAP_THRESHOLD）
    │   └─ brk() / sbrk()
    │       • 扩展进程的堆顶（program break）
    │       • 向高地址连续增长
    │       • 释放后不立即归还操作系统（留在堆中供复用）
    │
    └─ size >= 128KB
        └─ mmap()
            • 在堆与栈之间的匿名映射区分配独立内存块
            • free 时直接 munmap 归还操作系统
            • 不经过 bin 管理，无碎片问题
            • 分配粒度为页（通常 4KB）
```

为什么攻击者需要关心这两条路径？因为 `brk` 分配的内存在堆中连续排列，攻击者可以通过溢出覆盖相邻 chunk 的元数据；而 `mmap` 分配的内存页对齐且独立，溢出通常只能影响自身。大多数堆利用技术都针对 `brk` 路径。

```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <sys/mman.h>

int main() {
    // brk 路径：小块内存，由 glibc bin 管理
    char *heap_buf = malloc(64);
    printf("brk 分配: %p\n", heap_buf);

    // mmap 路径：大块内存，独立映射区
    char *mmap_buf = malloc(256 * 1024);  // 256KB，超过默认阈值
    printf("mmap 分配: %p\n", mmap_buf);

    // 直接调用 mmap（绕过 malloc）
    void *direct = mmap(NULL, 4096, PROT_READ | PROT_WRITE,
                        MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);
    printf("直接 mmap: %p\n", direct);

    // 地址差异巨大：heap_buf 在低地址堆区，mmap_buf 和 direct 在高地址映射区
    free(heap_buf);
    free(mmap_buf);
    munmap(direct, 4096);
    return 0;
}
```

### 4.2 glibc malloc 的 chunk 结构

glibc 管理堆的基本单位是 **chunk**（内存块）。每个 chunk 除了用户数据区外，还有一个 16 字节（64 位系统）的头部元数据，记录大小和状态标志。理解 chunk 结构是所有堆利用的基础。

```text
已分配的 chunk（64 位系统）：
┌───────────────────────────────────────────────┐  ← 当前 chunk 起始
│  prev_size (8 字节)                            │  上一个 chunk 空闲时才有效
├───────────────────────────────────────────────┤
│  size (8 字节) + N|M|P 标志位                  │  P=1 表示前一个 chunk 正在使用
├───────────────────────────────────────────────┤  ← malloc() 返回的指针
│  user data ...                                │  用户实际使用的空间
│  (最小 2×SIZE_SZ = 16 字节，最大 无限制)       │
└───────────────────────────────────────────────┘  ← 下一个 chunk 起始

标志位含义：
  P (PREV_INUSE, bit 0)  = 1 表示前一个 chunk 正在使用
  M (IS_MMAPPED, bit 1)  = 1 表示该 chunk 由 mmap 分配
  N (NON_MAIN_ARENA, bit 2) = 1 表示不属于主 arena

size 字段始终是 2×SIZE_SZ（16）的倍数，低 3 位用于标志。
实际 size = size & ~0x7
```

空闲的 chunk 会被链入相应的 bin 链表，此时 user data 区域的前 16 字节被复用为链表指针：

```text
空闲的 chunk：
┌───────────────────────────────────────────────┐
│  prev_size                                    │
├───────────────────────────────────────────────┤
│  size + P|M|N 标志位                           │
├───────────────────────────────────────────────┤
│  fd (forward pointer, 8 字节)                 │  指向 bin 中下一个空闲 chunk
├───────────────────────────────────────────────┤
│  bk (backward pointer, 8 字节)                │  指向 bin 中上一个空闲 chunk
├───────────────────────────────────────────────┤
│  fd_nextsize / bk_nextsize（仅 large bin）    │
│  ... 剩余空间 ...                              │
└───────────────────────────────────────────────┘
```

### 4.3 Bin 机制详解

glibc 通过 **bin**（容器）链表管理所有空闲 chunk。不同大小和状态的 chunk 被分类存放在不同的 bin 中，以优化分配速度。

#### 4.3.1 Fast Bin

Fast bin 是为频繁分配释放的小 chunk 设计的快速缓存。

```text
Fast Bin 特性：
  • 大小范围：32~128 字节（64 位，每 16 字节一个 bin，共 10 个 bin）
  • 链表结构：单向链表，LIFO（后进先出）
  • 不合并：释放时不会与相邻空闲 chunk 合并
  • 标志位：空闲 chunk 的 P 位保持为 1（防止被合并）
  • 分配速度：O(1)，直接取链表头

┌──────┐    ┌──────┐    ┌──────┐
│ bin0 │───→│chunk3│───→│chunk2│───→│chunk1│───→ NULL
│(32B) │    │      │    │      │    │      │
└──────┘    └──────┘    └──────┘    └──────┘

攻击面：Double Free。由于 fast bin 不检查是否已经被释放，
同一 chunk 被 free 两次会形成环形链表，导致：
  chunk_A → chunk_A → chunk_A → ...
下次 malloc 两次会返回同一地址，实现任意地址写。
```

#### 4.3.2 Tcache Bin（glibc 2.26+）

glibc 2.26 引入了 **tcache**（Thread Local Cache），是现代 glibc 中最常被利用的机制。

```text
Tcache 特性：
  • 每个线程独立，无锁（thread-local）
  • 大小范围：24~1032 字节（64 位，每 16 字节一个 bin，共 64 个 bin）
  • 每个 bin 最多缓存 7 个 chunk（默认，TCACHE_MAX_BINS=64, TCACHE_FILL_COUNT=7）
  • 单向链表，LIFO
  • 不检查任何标志位（glibc 2.26~2.28）
  • 速度比 fast bin 更快

tcache_perthread_struct 结构：
┌────────────────────────────────┐
│  counts[64]  (每个 bin 的计数)  │  每个 2 字节，共 128 字节
├────────────────────────────────┤
│  entries[64] (每个 bin 的头指针)│  每个 8 字节，共 512 字节
└────────────────────────────────┘
总计 640 字节，位于堆的最开头

攻击面：
  1. Double Free（2.26~2.28 无 tcache key 检查，直接 free 两次即可）
  2. Tcache Poisoning：覆写 fd 指针，malloc 返回任意地址
  3. Tcache Stashing Unlink Attack：利用 tcache 与 small bin 交互
```

```c
// glibc 2.26~2.28 的 Double Free（无需绕过检查）
#include <stdio.h>
#include <stdlib.h>

int main() {
    char *a = malloc(0x20);   // 进入 tcache[4]（0x30 大小）
    char *b = malloc(0x20);

    free(a);                   // a 的 fd 指向 tcache 头部
    free(a);                   // 又 free 一次，形成 a→a 环

    // 两次 malloc 返回同一地址
    char *c = malloc(0x20);    // c == a
    char *d = malloc(0x20);    // d == a（同一地址！）

    // 此时 c 和 d 指向同一块内存
    // 写 c 就是写 d，实现堆地址复用
    printf("c = %p, d = %p\n", c, d);
    return 0;
}
```

#### 4.3.3 Unsorted Bin

不属于 fast bin 和 tcache 的 chunk 被释放后首先进入 unsorted bin。

```text
Unsorted Bin 特性：
  • 双向链表，FIFO
  • 所有大小混合存放（"未排序"）
  • malloc 时遍历 unsorted bin 寻找合适 chunk
  • 遍历过程中将不合适的 chunk 移入对应的 small bin 或 large bin
  • 只有 1 个 unsorted bin（全局唯一）

遍历逻辑：
  1. 从 bk 方向开始遍历（最早进入的 chunk）
  2. 如果 chunk 刚好满足请求大小，直接返回
  3. 如果太大，分裂为两部分
  4. 如果不合适，移入 small bin 或 large bin

攻击面：
  • Unsorted Bin Attack：覆写 bk 指针，利用遍历逻辑实现任意地址写
    （写入 main_arena+88 的值到目标地址）
  • 信息泄露：unsorted bin 的 fd/bk 指向 main_arena，
    如果能读取这些指针，可以泄露 libc 基地址
```

#### 4.3.4 Small Bin 与 Large Bin

```text
Small Bin：
  • 共 62 个 bin（64 位系统，64~512 字节，每 16 字节一个）
  • 双向链表，FIFO
  • 同一 bin 内所有 chunk 大小相同
  • malloc 时直接取链表尾部（bk 指向的 chunk）

Large Bin：
  • 共 63 个 bin，覆盖 > 512 字节的范围
  • 双向链表 + fd_nextsize/bk_nextsize（按大小排序）
  • 同一 bin 内 chunk 大小可以不同
  • malloc 时查找 best fit（最小满足请求的 chunk）

Large Bin Attack：
  覆写 large bin chunk 的 fd_nextsize/bk_nextsize，
  在 bin 内排序时向任意地址写入堆地址。
```

### 4.4 malloc 与 free 的完整流程

理解分配器的完整决策路径，才能找到可利用的边界条件。

```text
malloc(size) 完整流程：

1. 将 size 对齐到 16 字节倍数，加上 chunk header 的 16 字节
2. 检查 tcache 对应的 bin 是否有空闲 chunk
   └─ 有 → 直接返回（O(1)）
3. 检查 fast bin（仅当 size 属于 fast bin 范围）
   └─ 有 → 取出返回
4. 检查 small bin（仅当 size 属于 small bin 范围）
   └─ 有 → 取出返回
5. 合并 fast bin 中的 chunk（consolidate）→ 放入 unsorted bin
6. 遍历 unsorted bin
   ├─ 找到精确匹配 → 返回
   ├─ 找到更大的 → 分裂，剩余部分放回 unsorted bin
   └─ 不合适 → 移入 small/large bin，继续遍历
7. 检查 large bin
   └─ 找到 best fit → 分裂返回
8. 检查 top chunk（堆顶剩余空间）
   └─ 够大 → 从 top chunk 切割
9. 以上都失败 → brk/mmap 扩展堆 → 重试

free(ptr) 完整流程：

1. 检查 ptr 是否为 NULL（是则直接返回）
2. 获取 chunk 的 size 和标志位
3. 检查 IS_MMAPPED 标志
   └─ 是 → munmap 归还操作系统
4. 检查是否与下一个 chunk 合并（next chunk 的 P 位）
5. 检查是否与上一个 chunk 合并（当前 chunk 的 prev_size）
6. 检查 tcache 对应的 bin 是否未满（counts < 7）
   └─ 未满 → 放入 tcache（glibc 2.26+）
   └─ 已满 → 放入 unsorted bin
```

### 4.5 C++ 内存管理：new/delete 与 RAII

C++ 在 C 的 `malloc/free` 之上构建了更高级的内存管理语义。

#### 4.5.1 new/delete 的底层

```cpp
#include <iostream>
#include <cstdlib>

class Widget {
    int id;
    char data[64];
public:
    Widget(int i) : id(i) { std::cout << "Widget(" << id << ") 构造\n"; }
    ~Widget() { std::cout << "~Widget(" << id << ") 析构\n"; }
};

int main() {
    // new 做了两件事：
    // 1. 调用 operator new() → 底层调用 malloc() 分配内存
    // 2. 在分配的内存上调用构造函数
    Widget *w = new Widget(1);
    // 等价于：
    // void *mem = operator new(sizeof(Widget));  // 分配
    // Widget *w = new(mem) Widget(1);            // placement new

    // delete 做两件事（逆序）：
    // 1. 调用析构函数
    // 2. 调用 operator delete() → 底层调用 free() 释放内存
    delete w;
    // 等价于：
    // w->~Widget();            // 析构
    // operator delete(w);      // 释放

    // 数组形式
    Widget *arr = new Widget[3]{Widget(10), Widget(20), Widget(30)};
    delete[] arr;  // 必须用 delete[]，否则析构函数调用次数不对

    return 0;
}
```

#### 4.5.2 operator new 的可重载性

C++ 允许全局或逐类重载 `operator new`/`operator delete`，这既是强大的内存管理工具，也是攻击面。

```cpp
#include <cstdlib>
#include <iostream>

// 全局重载 operator new（影响所有 new 表达式）
void *operator new(size_t size) {
    std::cout << "[DEBUG] 分配 " << size << " 字节\n";
    void *p = malloc(size);
    if (!p) throw std::bad_alloc();
    return p;
}

void operator delete(void *p) noexcept {
    std::cout << "[DEBUG] 释放 " << p << "\n";
    free(p);
}

// placement new — 在已有内存上构造对象
// 常见用途：内存池、自定义分配器、STL 容器
char pool[4096];
void *placement_test() {
    // placement new 不分配内存，只调用构造函数
    Widget *w = new (pool) Widget(42);
    // 不能 delete w（内存不属于堆），必须手动析构：
    w->~Widget();
    return w;
}
```

#### 4.5.3 智能指针：现代 C++ 的内存安全基石

```cpp
#include <memory>
#include <iostream>

struct Resource {
    int id;
    Resource(int i) : id(i) { std::cout << "Resource(" << id << ") 创建\n"; }
    ~Resource() { std::cout << "Resource(" << id << ") 销毁\n"; }
};

void smart_pointer_demo() {
    // unique_ptr：独占所有权，零开销（和裸指针一样大）
    {
        auto p1 = std::make_unique<Resource>(1);
        // auto p2 = p1;  // 编译错误！不能复制
        auto p2 = std::move(p1);  // 可以移动，p1 变为 nullptr
        // 出作用域时 p2 自动销毁 Resource
    }

    // shared_ptr：共享所有权，引用计数
    {
        auto p1 = std::make_shared<Resource>(2);
        {
            auto p2 = p1;  // 引用计数 = 2
            std::cout << "引用计数: " << p1.use_count() << "\n";
        }  // p2 销毁，引用计数 = 1
        std::cout << "引用计数: " << p1.use_count() << "\n";
    }  // p1 销毁，引用计数 = 0，Resource 被释放

    // weak_ptr：打破循环引用
    // 当 A→B 且 B→A 时，用 weak_ptr 打破循环
}

// 安全隐患：shared_ptr 的引用计数不是原子的（除非用 std::atomic）
// 多线程环境下操作同一个 shared_ptr 需要加锁
```

```text
智能指针选择决策树：

需要动态分配对象？
  ├─ 否 → 直接用栈对象或值语义
  └─ 是 → 所有权关系？
      ├─ 单一所有者 → unique_ptr（首选，零开销）
      ├─ 多个所有者 → shared_ptr（引用计数有开销）
      └─ 观察但不拥有 → raw ptr 或 weak_ptr
          └─ 需要延迟检查对象是否存活 → weak_ptr
```

### 4.6 常见堆漏洞与利用原理

#### 4.6.1 堆溢出（Heap Overflow）

堆溢出是最基础的堆漏洞，写入超出分配的缓冲区边界，覆盖相邻 chunk 的元数据。

```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

int main() {
    // 分配两个相邻的 chunk
    char *a = malloc(0x20);  // chunk A
    char *b = malloc(0x20);  // chunk B，在 A 之后

    // 堆溢出：向 A 写入超过 0x20 字节的数据
    // 这会覆盖 B 的 chunk header（prev_size + size）
    // 以及 B 的用户数据
    memset(a, 'A', 0x40);  // 溢出 0x20 字节到 B 的区域

    // 攻击者利用场景：
    // 1. 覆盖 B 的 size 字段 → 后续 free(B) 时触发错误的合并
    // 2. 覆盖 B 的用户数据 → 如果 B 存储了函数指针，可以劫持控制流
    // 3. 覆盖相邻 chunk 的 fd/bk → 实现 unlink 攻击

    free(a);
    free(b);
    return 0;
}
```

#### 4.6.2 Use-After-Free（UAF）

释放内存后继续使用原指针，是浏览器和内核漏洞中最常见的类型之一。

```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

// 模拟一个简化的 UAF 场景
typedef struct {
    void (*callback)(void);  // 函数指针（攻击目标）
    char data[56];
} Handler;

void legitimate_action() {
    printf("执行合法操作\n");
}

void malicious_action() {
    printf("被劫持执行恶意代码！\n");
}

int main() {
    // 分配两个相同大小的 Handler
    Handler *h1 = (Handler *)malloc(sizeof(Handler));
    h1->callback = legitimate_action;

    // 释放 h1（但指针未置空）
    free(h1);

    // 分配新的 chunk，恰好复用了 h1 的内存
    // 在实际攻击中，攻击者会精心控制分配，确保复用
    Handler *h2 = (Handler *)malloc(sizeof(Handler));
    h2->callback = malicious_action;  // h2 和 h1 指向同一地址

    // 通过悬垂指针 h1 调用——实际执行的是 h2 设置的函数
    h1->callback();  // 输出：被劫持执行恶意代码！

    free(h2);
    return 0;
}
```

UAF 在真实攻击中的典型模式：

```text
UAF 利用链（浏览器场景）：

1. 目标对象 A 分配在堆上，包含虚函数表指针
2. JavaScript 代码触发 A 的释放（但 JS 侧仍持有引用）
3. 攻击者用精心构造的数据 B 占用 A 的内存
4. 通过 JS 引用调用 A 的虚函数 → 实际执行 B 中的伪造 vtable
5. 控制流劫持完成
```

#### 4.6.3 Double Free

同一 chunk 被释放两次，破坏 bin 的链表结构，导致后续 malloc 返回同一地址。

```c
#include <stdio.h>
#include <stdlib.h>

int main() {
    char *a = malloc(0x40);  // 进入 tcache[6] 或 fast bin[4]

    free(a);  // 第一次释放：a 进入 bin

    // 绕过 tcache key 检查（glibc 2.29+）的常见手法：
    // free 另一个大小不同的 chunk，使 tcache bin 暂时不指向 a
    // 然后再 free a

    free(a);  // 第二次释放：a 再次进入 bin，形成 a→a 环

    // 现在 bin 中：a→a→a→...（自引用环）
    // 两次 malloc 返回同一地址
    char *b = malloc(0x40);  // b == a
    char *c = malloc(0x40);  // c == a

    // b 和 c 是同一地址，写 b 就是写 c
    // 攻击者可以：覆写相邻 chunk 的 fd，实现任意地址分配
    // 例如让 c 指向 __malloc_hook，然后在其中写入 system 地址

    free(b);
    return 0;
}
```

#### 4.6.4 Off-by-One

单字节溢出看似微不足道，但在特定布局下可以实现 chunk header 篡改。

```c
// 经典 off-by-one：循环边界错误
void vulnerable(char *input) {
    char buf[64];
    for (int i = 0; i <= 64; i++) {  // 应该是 i < 64
        buf[i] = input[i];  // 第 65 次写入覆盖了下一个 chunk 的 size 低字节
    }
}

// null byte off-by-one（字符串操作常见）
void null_overflow(char *src) {
    char *buf = malloc(256);
    // 如果 src 恰好 256 字节，加上 \0 共 257 字节
    strncpy(buf, src, 256);
    buf[256] = '\0';  // 覆盖下一个 chunk 的 prev_size 低字节
}

// Off-by-one 利用原理（null byte poisoning）：
// 覆盖下一个 chunk 的 size 低字节为 0x00
// 例如 size 从 0x110 变为 0x100
// 导致后续 free 时 chunk 合并出错，创造出 overlapping chunk
```

### 4.7 内存调试与检测工具

#### 4.7.1 AddressSanitizer（ASAN）

ASAN 是 GCC/Clang 内置的内存错误检测工具，编译时插入检测代码，运行时捕获各类内存错误。

```bash
# 编译时启用 ASAN
gcc -fsanitize=address -g -O1 -o program program.c
clang -fsanitize=address -g -O1 -o program program.c

# 启用更严格的检查
gcc -fsanitize=address -fsanitize-address-use-after-scope \
    -fno-omit-frame-pointer -g -O1 -o program program.c

# ASAN 能检测的错误类型：
# • 堆溢出（heap-buffer-overflow）
# • 栈溢出（stack-buffer-overflow）
# • 全局变量溢出（global-buffer-overflow）
# • Use-After-Free（heap-use-after-free）
# • 栈上 Use-After-Return（-fsanitize-address-use-after-return=always）
# • Double-Free
# • 内存泄漏（需要设置 ASAN_OPTIONS=detect_leaks=1）

# 运行时配置
ASAN_OPTIONS="detect_leaks=1:halt_on_error=0:print_stats=1" ./program
```

#### 4.7.2 Valgrind（memcheck）

```bash
# 基本使用
valgrind --leak-check=full --show-leak-kinds=all \
         --track-origins=yes --verbose ./program

# 常见错误类型：
# • Invalid read/write：访问已释放或越界的内存
# • Conditional jump on uninitialized value：使用未初始化的值
# • Definitely lost / Indirectly lost / Possibly lost：内存泄漏
# • Mismatched free/delete：malloc 配 delete 或 new 配 free

# 注意：Valgrind 比 ASAN 慢 10~50 倍，但不需要重新编译
```

#### 4.7.3 mtrace：glibc 内置的 malloc 追踪

```c
#include <mcheck.h>  // 必须包含

int main() {
    mtrace();  // 开始追踪（需要设置 MALLOC_TRACE 环境变量）

    char *p = malloc(100);
    // ... 使用 p ...
    // 故意不 free(p) 来演示泄漏检测

    muntrace();  // 停止追踪
    return 0;
}
```

```bash
# 设置输出文件并运行
MALLOC_TRACE=./trace.log ./program

# 分析追踪日志
mtrace ./program ./trace.log
# 输出：
# Memory not freed:
#    Address     Size     Caller
# 0x0804a008     100  at 0x8048432
```

### 4.8 安全编码最佳实践

#### 4.8.1 内存分配的安全模式

```c
// 模式一：分配后立即检查
char *buf = malloc(size);
if (!buf) {
    perror("malloc");
    return -1;
}

// 模式二：使用 calloc 防止未初始化泄露
// calloc 会将内存清零，且在乘法溢出时返回 NULL
int *arr = calloc(n, sizeof(int));  // 如果 n * sizeof(int) 溢出，返回 NULL

// 模式三：realloc 的安全用法——不要直接覆盖原指针
char *tmp = realloc(old_ptr, new_size);
if (!tmp) {
    // realloc 失败时，old_ptr 仍然有效！
    free(old_ptr);  // 必须手动释放
    return -1;
}
old_ptr = tmp;  // 成功后才覆盖

// 模式四：释放后置空（防御 UAF）
free(ptr);
ptr = NULL;  // 后续 *ptr 会立即崩溃，而不是静默的 UAF
```

#### 4.8.2 防御性内存管理清单

```text
□ 所有 malloc/calloc/realloc 返回值都检查 NULL
□ realloc 不直接赋值给原指针（先用临时变量）
□ free 后立即将指针置 NULL
□ 避免使用 strncpy/strncat（容易 off-by-one），用 strlcpy/strlcpy 或 snprintf
□ 变长数组（VLA）能不用就不用（栈溢出风险）
□ 循环边界严格使用 < 而非 <=
□ 多线程环境下，堆访问需要互斥保护
□ 优先使用智能指针（C++）而非裸指针
□ 编译时开启 -Wall -Wextra -Werror -Wformat=2
□ CI 流程中集成 ASAN/UBSAN 测试
□ 定期用 Valgrind 做全量内存检查
```

### 4.9 C++ 内存安全的进阶话题

#### 4.9.1 自定义分配器与内存池

STL 容器支持自定义分配器，这在安全和性能敏感的场景下非常重要。

```cpp
#include <vector>
#include <memory>
#include <new>

// 简易内存池分配器
template <typename T, size_t PoolSize = 1024>
class PoolAllocator {
    alignas(T) char pool_[PoolSize * sizeof(T)];
    size_t offset_ = 0;
public:
    using value_type = T;

    T *allocate(size_t n) {
        if (offset_ + n > PoolSize)
            throw std::bad_alloc();
        T *p = reinterpret_cast<T *>(pool_ + offset_ * sizeof(T));
        offset_ += n;
        return p;
    }

    void deallocate(T *p, size_t n) noexcept {
        // 简化版不回收，生产环境需实现 free list
    }
};

// 使用方式
std::vector<int, PoolAllocator<int, 100>> secure_vec;
// 所有内存分配都在预分配的池中，不经过系统 malloc
// 好处：无碎片、可控布局、可以设置 guard page 检测溢出
```

#### 4.9.2 Placement new 与类型双关

```cpp
#include <new>
#include <cstring>
#include <cstdio>

struct Header {
    uint32_t magic;      // 魔数校验
    uint32_t size;       // 数据大小
    uint8_t  checksum;   // 校验和
    char     data[];     // 柔性数组成员
};

// 使用 placement new 在预分配缓冲区上构造对象
alignas(Header) char buffer[4096];

Header *create_packet(const void *payload, uint32_t len) {
    Header *h = new (buffer) Header();
    h->magic = 0xDEADBEEF;
    h->size = len;
    memcpy(h->data, payload, len);

    // 计算校验和
    h->checksum = 0;
    for (uint32_t i = 0; i < len; i++)
        h->checksum ^= static_cast<uint8_t>(h->data[i]);

    return h;
}
```
