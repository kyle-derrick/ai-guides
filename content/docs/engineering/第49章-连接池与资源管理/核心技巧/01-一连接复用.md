# 连接复用

连接复用是连接池技术的核心价值所在——它回答了一个根本问题：**如何让一条物理连接服务尽可能多的逻辑请求，同时保证数据隔离和系统稳定**。本节将从协议层到应用层，系统讲解连接复用的实现机制、最佳实践和常见陷阱。

## 1. 连接复用的本质

### 1.1 复用 vs 重建：性能差距有多大

在没有连接池的世界里，每个请求经历完整的连接生命周期：

无复用（每次新建）:
  TCP 握手 ████████░░░░░░░░  3.0ms
  TLS 握手 ████████░░░░░░░░  3.0ms（HTTPS）
  协议握手 ████░░░░░░░░░░░░  1.5ms（MySQL auth）
  业务查询 ████░░░░░░░░░░░░  3.0ms
  连接关闭 ██░░░░░░░░░░░░░░  1.0ms
  ──────────────────────────────────
  总计                        11.5ms（HTTPS）/ 8.5ms（TCP）

有复用（连接池）:
  从池获取 ░░░░░░░░░░░░░░░░  0.01ms
  业务查询 ████░░░░░░░░░░░░  3.0ms
  归还到池 ░░░░░░░░░░░░░░░░  0.01ms
  ──────────────────────────────────
  总计                        3.02ms

| 指标 | 无复用 | 有复用 | 提升倍数 |
|------|--------|--------|---------|
| 单次延迟 | 8.5-11.5ms | 3.02ms | 2.8-3.8x |
| QPS（4核机器） | ~2,000 | ~12,000 | 6x |
| 每万请求内存峰值 | ~640MB（TIME_WAIT） | ~3MB（空闲连接） | 213x |
| CPU 用户态利用率 | 35-45% | 8-15% | 3-5x |

连接复用不仅仅是"省掉几次握手"，它是高并发系统的**前提条件**。没有连接复用，任何上层优化都建立在沙塔之上。

### 1.2 复用的三个层次

连接复用并非只有一种形态，按照复用粒度从粗到细可以分为三个层次：

┌─────────────────────────────────────────────────────────┐
│ 第一层：连接级复用（Connection-level Reuse）             │
│   一条 TCP 连接服务多个串行请求                          │
│   典型：HTTP/1.1 Keep-Alive、MySQL 长连接                │
├─────────────────────────────────────────────────────────┤
│ 第二层：多路复用（Multiplexing）                         │
│   一条连接上并发传输多个请求流                            │
│   典型：HTTP/2 Stream、Redis Pipeline                   │
├─────────────────────────────────────────────────────────┤
│ 第三层：协议级复用（Protocol-level Reuse）               │
│   底层连接对上层完全透明，自动管理                        │
│   典型：Service Mesh sidecar、数据库代理                 │
└─────────────────────────────────────────────────────────┘

下面逐层展开。

## 2. 第一层：HTTP 连接复用

### 2.1 HTTP/1.1 Keep-Alive

HTTP/1.0 的每个请求都需要新建 TCP 连接，HTTP/1.1 默认启用 Keep-Alive，允许在同一个 TCP 连接上发送多个请求-响应对：

HTTP/1.1 Keep-Alive 流程:

客户端                  服务端
  │                       │
  │──── TCP 三次握手 ────→│
  │←─── SYN-ACK ─────────│
  │──── ACK ────────────→│
  │                       │
  │──── GET /api/1 ─────→│
  │←─── Response 1 ──────│
  │                       │  同一个 TCP 连接
  │──── GET /api/2 ─────→│
  │←─── Response 2 ──────│
  │                       │
  │──── GET /api/3 ─────→│
  │←─── Response 3 ──────│
  │                       │
  │──── Connection: close─│
  │←─── Response 4 ──────│
  │                       │
  │──── TCP FIN ────────→│

**服务端配置（Nginx）：**

```nginx
# /etc/nginx/nginx.conf
http {
    # Keep-Alive 持续时间（秒），超过后关闭连接
    keepalive_timeout 65;

    # 每个连接最多处理多少个请求后强制关闭
    # 防止长连接导致的内存泄漏
    keepalive_requests 1000;

    # Keep-Alive 连接缓冲区大小
    large_client_header_buffers 4 16k;

    # 与上游服务的 Keep-Alive 连接池
    upstream backend {
        server 10.0.0.1:8080;
        keepalive 32;           # 每个 worker 进程保持 32 个空闲长连接
        keepalive_requests 100; # 每个连接最多服务 100 个请求
        keepalive_timeout 60s;  # 空闲 60 秒后关闭
    }
}
```

**客户端配置（Python requests）：**

```python
import requests
from requests.adapters import HTTPAdapter
from urllib3.util.retry import Retry

# 创建持久化的 Session，底层自动复用连接
session = requests.Session()

# 配置连接池参数
adapter = HTTPAdapter(
    pool_connections=10,    # 缓存 10 个不同 host 的连接池
    pool_maxsize=20,        # 每个 host 最多保持 20 个连接
    max_retries=Retry(
        total=3,
        backoff_factor=0.3,
        status_forcelist=[500, 502, 503, 504],
    ),
)
session.mount('https://', adapter)
session.mount('http://', adapter)

# 后续所有请求自动复用同一 host 的 TCP 连接
for url in urls:
    resp = session.get(url)  # 首次请求新建连接，后续复用
```

**客户端配置（Java HttpClient）：**

```java
HttpClient client = HttpClient.newBuilder()
    .connectTimeout(Duration.ofSeconds(10))
    .executor(Executors.newFixedThreadPool(20))
    // Java 11+ HttpClient 内部自动维护连接池
    .build();

// 连续请求自动复用底层 TCP 连接
HttpRequest request1 = HttpRequest.newBuilder()
    .uri(URI.create("https://api.example.com/users"))
    .build();

HttpRequest request2 = HttpRequest.newBuilder()
    .uri(URI.create("https://api.example.com/orders"))
    .build();

client.sendAsync(request1, BodyHandlers.ofString());
client.sendAsync(request2, BodyHandlers.ofString());
// 两个请求共享同一条 TLS 连接到 api.example.com
```

### 2.2 HTTP/2 多路复用

HTTP/2 将连接复用提升到全新高度：在一条 TCP 连接上，多个请求可以**同时**传输，不再需要排队等待：

HTTP/1.1 队头阻塞（Head-of-Line Blocking）:
  连接 1: ──req1──resp1──req2──resp2──req3──resp3──
  连接 2: ──req4──resp4──req5──resp5──
  需要多条连接才能实现并发

HTTP/2 多路复用:
  唯一连接: ──req1──req4──req2──resp1──resp4──req5──resp2──req3──resp5──resp3──
             ↑ 所有请求交织传输，通过 Stream ID 区分
             ↑ 无队头阻塞，一条连接即可并行

| 对比维度 | HTTP/1.1 | HTTP/2 |
|---------|----------|--------|
| 并发模型 | 多条 TCP 连接 | 单条连接 + 多 Stream |
| 队头阻塞 | 应用层阻塞（前一个响应慢，后面的等） | TCP 层阻塞（丢包会阻塞所有 Stream） |
| 头部压缩 | 无（每次完整发送） | HPACK 压缩（增量 + 静态表 + Huffman） |
| 服务端推送 | 不支持 | 支持 Server Push |
| 连接开销 | 每条连接的握手+TLS开销 | 仅首条连接的开销 |
| 适合场景 | 简单 API、兼容性优先 | 高并发 Web、移动端弱网 |

**Nginx HTTP/2 配置：**

```nginx
server {
    listen 443 ssl http2;  # 启用 HTTP/2

    # HTTP/2 连接级配置
    http2_max_concurrent_streams 128;  # 单连接最大并发 Stream 数
    http2_recv_buffer_size 256k;       # 接收缓冲区

    # TLS 会话复用（减少 TLS 握手开销）
    ssl_session_cache shared:SSL:10m;  # 10MB 共享缓存，约可存 4 万个会话
    ssl_session_timeout 1d;            # 会话有效 1 天
    ssl_session_tickets on;            # 启用 Session Ticket（客户端持有密钥）
}
```

**客户端 HTTP/2（Go）：**

```go
package main

import (
    "net/http"
    "sync"
)

func main() {
    // Go 的 http.Client 默认启用 HTTP/2（对 HTTPS）
    // 且自动维护连接池和多路复用
    client := &amp;http.Client{
        Transport: &amp;http.Transport{
            MaxIdleConns:        100,
            MaxIdleConnsPerHost: 20,
            IdleConnTimeout:     90 * time.Second,
            // ForceAttemptHTTP2: true, // Go 1.13+ 默认 true
        },
    }

    // 并发请求自动在同一连接上多路复用
    var wg sync.WaitGroup
    for _, url := range urls {
        wg.Add(1)
        go func(u string) {
            defer wg.Done()
            resp, err := client.Get(u)
            if err != nil { return }
            defer resp.Body.Close()
        }(url)
    }
    wg.Wait()
}
```

## 3. 第二层：数据库连接复用

### 3.1 MySQL 连接复用

MySQL 协议是**一请求一响应**的同步模型，不支持真正的多路复用。因此 MySQL 连接复用的关键是：**在同一条连接上串行执行多个查询，避免频繁借出和归还**。

**连接复用的正确姿势（Java + HikariCP）：**

```java
// ❌ 错误：每次操作获取和归还连接
public User getUser(long id) {
    Connection conn = dataSource.getConnection();  // 借出
    try {
        // 单次查询
        return queryUser(conn, id);
    } finally {
        conn.close();  // 归还到池
    }
}

// ✅ 正确：一个业务操作内复用同一连接
public OrderDetail getOrderDetail(long orderId) {
    Connection conn = dataSource.getConnection();  // 借出一次
    try {
        Order order = queryOrder(conn, orderId);       // 复用
        List<Item> items = queryItems(conn, orderId);   // 复用
        User user = queryUser(conn, order.userId);      // 复用
        return new OrderDetail(order, items, user);     // 组装结果
    } finally {
        conn.close();  // 一次归还
    }
}
```

**连接复用的成本对比：**

| 场景 | 获取连接次数 | 网络往返 | 耗时 |
|------|-------------|---------|------|
| 每次查询独立获取 | 3 次 | 6 次 RTT | ~18ms |
| 业务内复用同一连接 | 1 次 | 2 次 RTT | ~6ms |

### 3.2 连接状态清理——复用的前提

连接能被安全复用的前提是：**归还到池的连接必须处于干净状态**。否则前一个请求的残留状态会影响下一个请求：

```java
// 连接归还前的状态清理清单
public void safeReturnConnection(Connection conn) {
    try {
        // 1. 检查是否有未提交的事务
        if (!conn.getAutoCommit()) {
            conn.rollback();       // 回滚未提交事务
            conn.setAutoCommit(true);  // 恢复自动提交
        }

        // 2. 检查是否有活跃的结果集
        // （try-with-resources 自动处理）

        // 3. 检查事务隔离级别是否被修改
        // （某些框架会在运行时修改隔离级别）
        conn.setTransactionIsolation(Connection.TRANSACTION_REPEATABLE_READ);

        // 4. 检查是否有临时表
        // （MySQL: DROP TEMPORARY TABLE IF EXISTS ...）

        // 5. 检查连接属性（如 last_insert_id 等）
    } catch (SQLException e) {
        // 清理失败，销毁连接而非归还
        invalidateConnection(conn);
    } finally {
        conn.close();  // 归还到池
    }
}
```

```python
# Python DBAPI 2.0 连接清理
def safe_return_connection(conn):
    try:
        if not conn.getautocommit():
            conn.rollback()
        conn.setautocommit(True)
        # 重置会话变量（如果修改过）
        # conn.cursor().execute("SET SESSION sql_mode = ...")
    except Exception:
        conn.close()  # 异常时直接关闭（销毁），不归还
        return
    finally:
        pool.put(conn)  # 归还到池
```

### 3.3 Redis 连接复用

Redis 是单线程模型，天然支持连接复用，但需要注意**Pipeline**和**MULTI/EXEC**的使用：

```python
import redis

# 基础复用：Session 级别连接池
r = redis.Redis(
    host='localhost',
    port=6379,
    max_connections=20,      # 连接池大小
    socket_timeout=5,        # 命令执行超时
    socket_connect_timeout=2,# 连接建立超时
    decode_responses=True,
    retry_on_timeout=True,
)

# ❌ 逐条命令：100 条命令 = 100 次网络往返
for key in keys:
    value = r.get(key)  # 每次独立请求

# ✅ Pipeline 批量：100 条命令 = 1 次网络往返
pipe = r.pipeline(transaction=False)  # 非事务 Pipeline
for key in keys:
    pipe.get(key)
values = pipe.execute()  # 一次性发送所有命令，一次性接收所有结果

# ✅ Lua 脚本：在服务端原子执行，一次网络往返
script = """
local result = {}
for i, key in ipairs(KEYS) do
    result[i] = redis.call('GET', key)
end
return result
"""
values = r.eval(script, len(keys), *keys)
```

**Pipeline 性能对比（100 个 GET 命令）：**

| 方式 | 网络往返次数 | 总耗时 | QPS |
|------|-------------|--------|-----|
| 逐条执行 | 100 | ~20ms | 5,000 |
| Pipeline | 1 | ~0.5ms | 200,000 |
| Lua 脚本 | 1 | ~0.3ms | 333,000 |

### 3.4 MongoDB 连接复用

MongoDB 驱动内置连接池，但连接管理有其独特性：

```python
from pymongo import MongoClient

# 连接池配置
client = MongoClient(
    'mongodb://host1:27017,host2:27017',
    maxPoolSize=100,          # 最大连接数
    minPoolSize=10,           # 最小保持连接数
    maxIdleTimeMS=60000,      # 空闲连接最大存活 60 秒
    waitQueueTimeoutMS=5000,  # 等待队列超时
    connectTimeoutMS=3000,    # 连接建立超时
    serverSelectionTimeoutMS=5000,  # 服务选择超时
    retryWrites=True,         # 自动重试写操作
)

# 底层自动复用连接
db = client.mydb
users = db.users
# find()、insert_one() 等操作自动借用和归还连接
```

MongoDB 4.2+ 支持**变更流（Change Stream）**，通过长时间持有单个连接来实现持续的数据变更通知，这是连接复用的一种特殊形态——**长连接专有化**。

## 4. 第三层：协议级复用

### 4.1 Service Mesh 透明复用

在 Istio/Linkerd 等 Service Mesh 架构中，sidecar 代理（Envoy）自动管理所有出站连接的池化和复用，应用代码完全不感知：

```yaml
# Istio VirtualService 连接池配置
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: my-service
spec:
  host: my-service
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100       # L4 连接池最大连接数
        connectTimeout: 30ms      # 连接建立超时
        tcpKeepalive:
          time: 7200s             # Keep-Alive 探测间隔
          interval: 75s
      http:
        h2UpgradePolicy: DEFAULT  # 是否升级到 HTTP/2
        http1MaxPendingRequests: 100
        http2MaxRequests: 1000    # HTTP/2 最大并发请求
        maxRequestsPerConnection: 0  # 0=不限制，每连接最大请求数
        maxRetries: 3
```

这种模式下，连接复用变成了一种**基础设施能力**，而非应用层的代码职责。

### 4.2 数据库代理层复用

PgBouncer（PostgreSQL）和 ProxySQL（MySQL）作为独立的连接代理层，在应用和数据库之间实现了连接池化：

```ini
# PgBouncer 配置 /etc/pgbouncer/pgbouncer.ini
[databases]
mydb = host=127.0.0.1 port=5432 dbname=mydb

[pgbouncer]
listen_addr = 0.0.0.0
listen_port = 6432

# 连接池模式——这是复用的核心配置
pool_mode = transaction  # 按事务复用（最常用）
# pool_mode = session     # 按会话复用（需要会话状态时）
# pool_mode = statement   # 按语句复用（最激进，不支持事务）

# 池大小
default_pool_size = 20    # 每个用户/数据库对的最大连接数
min_pool_size = 5
reserve_pool_size = 5     # 突发流量时的储备连接
max_client_conn = 1000    # 最大客户端连接数（可以远大于服务器连接数）

# 超时配置
server_idle_timeout = 600
client_idle_timeout = 0
query_timeout = 0
```

**三种复用模式对比：**

| 模式 | 复用粒度 | 典型场景 | 限制 |
|------|---------|---------|------|
| Session | 整个会话 | 使用 prepared statement、临时表、SET 变量 | 复用率低，接近 1:1 |
| Transaction | 事务内 | 标准 OLTP 应用，无会话状态依赖 | 最常用，复用率中等 |
| Statement | 单条语句 | 纯只读查询，无事务 | 最高效，但不支持多语句事务 |

```python
# 应用层使用 PgBouncer：无需修改代码
# 只需将连接地址从 :5432 改为 :6432
import psycopg2

conn = psycopg2.connect(
    host='pgbouncer-host',
    port=6432,      # PgBouncer 端口
    dbname='mydb',
    user='app_user',
    password='secret',
)
# 200 个应用连接可以被 PgBouncer 复用为 20 个后端连接
```

## 5. 连接复用的关键技术

### 5.1 连接有效性验证

复用一个可能已经失效的连接，会导致业务错误。验证策略的选择直接影响性能和可靠性：

```java
// 验证策略对比
public enum ValidationStrategy {
    NONE {
        // 不验证：最快，但可能拿到"半开连接"
        // 适用：网络极稳定、maxLifetime 设置合理
    },
    QUERY {
        // 执行轻量级查询验证
        // HikariCP: connectionTestQuery = "SELECT 1"
        // 适用：大多数数据库
    },
    AUTO_COMMIT {
        // 切换 autoCommit 触发 TCP 层验证
        // HikariCP: connectionTestQuery 留空时自动使用
        // 适用：不需要特定 SQL 的场景
    },
    NETWORK {
        // TCP keepalive 探测
        // 适用：验证网络连通性而非数据库状态
    };
}
```

**HikariCP 的优化验证机制：**

```java
HikariConfig config = new HikariConfig();
config.setJdbcUrl("jdbc:mysql://localhost:3306/mydb");

// 验证查询（留空则自动使用 autoCommit 切换方式）
config.setConnectionTestQuery("SELECT 1");

// 验证超时
config.setValidationTimeout(3000);  // 3 秒

// 连接有效性检测间隔（每 30 秒检测一次空闲连接）
// 这是后台线程的检测频率，不是每次借出时都检测
config.setKeepaliveTime(30000);

// 连接最大存活时间——超过必须重建，不管是否"看起来健康"
config.setMaxLifetime(1800000);  // 30 分钟

// 空闲超时——超过后销毁多余空闲连接
config.setIdleTimeout(600000);  // 10 分钟
```

### 5.2 连接预热

冷启动时连接池为空，首批请求需要等待连接创建。连接预热可以在应用启动时提前建立连接：

```python
class ConnectionPoolWarmer:
    """连接池预热器"""

    def __init__(self, pool, target_size):
        self.pool = pool
        self.target_size = target_size

    def warm_up(self):
        """在后台线程中预热连接池"""
        import threading

        def _warm():
            connections = []
            for i in range(self.target_size):
                try:
                    conn = self.pool.get_connection(timeout=5)
                    connections.append(conn)
                except Exception as e:
                    print(f"预热第 {i+1} 个连接失败: {e}")
                    break
            # 全部获取成功后一次性归还
            for conn in connections:
                self.pool.put(conn)
            print(f"连接池预热完成: {len(connections)}/{self.target_size}")

        thread = threading.Thread(target=_warm, daemon=True)
        thread.start()

# 使用
warmer = ConnectionPoolWarmer(pool, target_size=20)
warmer.warm_up()  # 应用启动时调用
```

```java
// Spring Boot 连接池预热
@Component
public class DataSourceWarmup implements ApplicationRunner {

    @Autowired
    private DataSource dataSource;

    @Override
    public void run(ApplicationArguments args) throws Exception {
        ExecutorService executor = Executors.newFixedThreadPool(5);
        CountDownLatch latch = new CountDownLatch(20);

        for (int i = 0; i < 20; i++) {
            executor.submit(() -> {
                try (Connection conn = dataSource.getConnection()) {
                    conn.isValid(3);  // 触发连接建立和验证
                } catch (SQLException e) {
                    // 记录日志但不阻止启动
                } finally {
                    latch.countDown();
                }
            });
        }

        latch.await(30, TimeUnit.SECONDS);  // 最多等 30 秒
        executor.shutdown();
        System.out.println("连接池预热完成");
    }
}
```

## 7. 常见问题与排查

### 7.1 半开连接（Half-Open Connection）

**症状**：获取连接后执行查询报 `Communications link failure` 或 `Connection reset`。

**原因**：客户端认为连接有效，但服务端已经关闭（服务端重启、网络中断、超时断开等）。

排查流程：
1. 检查 MySQL wait_timeout 设置
   SHOW VARIABLES LIKE 'wait_timeout';  -- 默认 28800 秒（8小时）

2. 确认连接池 maxLifetime < wait_timeout
   HikariCP: maxLifetime 应 < wait_timeout * 0.9

3. 检查网络层 TCP keepalive
   cat /proc/sys/net/ipv4/tcp_keepalive_time   -- 默认 7200 秒
   cat /proc/sys/net/ipv4/tcp_keepalive_intvl  -- 默认 75 秒
   cat /proc/sys/net/ipv4/tcp_keepalive_probes -- 默认 9 次

4. 缩短系统级 keepalive（可选）
   echo 600 | sudo tee /proc/sys/net/ipv4/tcp_keepalive_time

### 7.2 连接泄漏

**症状**：活跃连接数持续上升，最终达到 maxPoolSize 后新请求开始排队超时。

**排查步骤：**

```java
// 1. 开启 HikariCP 泄漏检测
config.setLeakDetectionThreshold(60000);  // 60 秒未归还则告警

// 2. 日志中会出现如下信息：
// WARNING: Connection leak detection triggered for connection abc123,
// stack trace follows
// java.lang.Exception: WARNING: Connection leak detection triggered
//     at com.zaxxer.hikari.HikariDataSource.getConnection(...)
//     at com.myapp.service.UserService.getUser(...)
//     at com.myapp.controller.UserController.handleRequest(...)

// 3. 根据调用栈定位泄漏点
// 常见原因：
// - 忘记在 finally 中 close
// - try-with-resources 用法错误
// - 异常路径提前 return 跳过了 close
```

```python
# Python 连接泄漏检测
import contextlib

@contextlib.contextmanager
def tracked_connection(pool):
    """带追踪的连接获取"""
    conn = pool.get_connection()
    import traceback
    creation_stack = ''.join(traceback.format_stack())
    try:
        yield conn
    finally:
        conn.close()
        # 检查是否超时未归还（通过 weakref 追踪）
        print(f"连接已归还，创建时调用栈:\n{creation_stack}")

# 使用
with tracked_connection(pool) as conn:
    cursor = conn.cursor()
    cursor.execute("SELECT 1")
```

### 7.3 连接风暴（Connection Storm）

**场景**：数据库重启后，所有应用实例同时检测到连接失效，同时尝试重建连接。

时间线：
T+0s   数据库重启完成
T+0.1s 10 个应用实例同时检测到连接断开
T+0.2s 10 个实例 × 20 连接/实例 = 200 个连接请求同时涌入
T+0.3s 数据库再次过载，部分连接创建失败
T+0.5s 应用重试，更多连接请求涌入
... （振荡）

**缓解措施：**

```java
// 1. 随机化重试间隔（避免同步重试）
int baseDelay = 1000;  // 基础延迟 1 秒
int jitter = ThreadLocalRandom.current().nextInt(0, baseDelay);
long delay = baseDelay + jitter;

// 2. 限流重建速率
Semaphore reconnectPermits = new Semaphore(5);  // 最多 5 个并发重建
void reconnect() {
    reconnectPermits.acquire();
    try {
        createNewConnection();
    } finally {
        reconnectPermits.release();
    }
}

// 3. 指数退避重试
int maxRetries = 5;
long delay = 1000;
for (int i = 0; i < maxRetries; i++) {
    try {
        Connection conn = dataSource.getConnection();
        if (conn.isValid(3)) return conn;
    } catch (SQLException e) {
        TimeUnit.MILLISECONDS.sleep(delay + random.nextInt(500));
        delay *= 2;  // 指数退避
    }
}
```

### 7.4 连接池饥饿

**症状**：连接池大小看起来够用，但高峰期仍然频繁超时。

**根因分析清单：**

1. 慢查询占用连接时间过长
   SHOW PROCESSLIST;  -- 查看执行时间超过 5s 的查询
   → 解决：优化慢查询 / 设置 query_timeout

2. 事务未及时提交
   一个业务操作中开始事务后做了远程调用（HTTP/RPC），事务被长时间持有
   → 解决：远程调用移到事务外部

3. 连接泄漏（见 7.2）

4. 连接池大小不足
   用 Little's Law 重新计算：
   L = λ × W
   如果每秒 1000 请求，平均耗时 50ms，则 L = 1000 × 0.05 = 50
   池大小至少需要 50 + 安全余量

5. 锁等待
   大量事务在同一行上竞争，连接虽然"活跃"但实际在等锁
   → 解决：优化索引 / 减小事务粒度

## 8. 最佳实践清单

### 8.1 连接复用的 Do's

✅ 始终使用连接池，不要手动管理连接生命周期
✅ 一个业务操作内复用同一连接，而不是每次查询都借还
✅ 配置合理的 maxLifetime，必须小于服务端的 wait_timeout
✅ 开启连接健康检查（至少在借出时检查）
✅ 设置合理的 connectionTimeout，快速失败比无限等待好
✅ 监控连接池指标，设置告警阈值
✅ 使用 try-with-resources / contextmanager 确保连接归还
✅ 应用启动时预热连接池

### 8.2 连接复用的 Don'ts

❌ 不要把连接存储在 ThreadLocal / 成员变量中长期持有
❌ 不要跨业务操作传递同一个连接（事务边界模糊）
❌ 不要禁用健康检查来"提升性能"
❌ 不要把 maxPoolSize 设得比数据库 max_connections 还大
❌ 不要在连接使用过程中修改隔离级别而不恢复
❌ 不要忽略连接泄漏检测日志
❌ 不要在无事务时使用 SELECT ... FOR UPDATE（锁住行但不释放）
❌ 不要忘记归还连接——用结构化方式（try-finally / contextmanager）

## 9. 本节小结

连接复用是连接池存在的根本理由。理解从协议层到应用层的三个复用层次，能够帮助你在不同场景下选择最合适的复用策略：

| 场景 | 推荐复用策略 | 关键配置 |
|------|------------|---------|
| HTTP API 服务 | HTTP/2 多路复用 + 连接池 | Keep-Alive timeout、max requests per connection |
| 关系型数据库 | 连接池 + 事务内复用 | maxLifetime < wait_timeout、健康检查 |
| Redis | Pipeline / Lua 脚本 | max_connections、socket_timeout |
| 微服务间调用 | gRPC + 连接池 | HTTP/2、max connection age |
| 跨服务透明复用 | Service Mesh sidecar | connectionPool.tcp.maxConnections |

核心原则始终如一：**用最少的物理连接，服务最多的逻辑请求，同时保证数据隔离和系统稳定**。
