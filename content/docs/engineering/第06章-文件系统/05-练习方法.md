## 文件系统练习方法

本章涵盖了 Linux 文件系统的理论基础（VFS 四大对象、ext4 布局与日志、Btrfs COW 快照、XFS 分配组、ZFS 存储池、F2FS 闪存优化）、核心技巧（选型决策、调优参数、IO 调度器）和实战案例。以下练习按照「理解→实操→排查→优化→设计」五层递进设计，帮助你将理论知识转化为系统工程师的实战能力。

每个练习标注了预计耗时和前置条件。建议按顺序完成，因为后序练习会用到前序练习搭建的环境。

### 练习一：VFS 抽象层与挂载管理（预计 40 分钟）

**目标**：理解 VFS 四大核心对象（super_block、inode、dentry、file）的作用，掌握文件系统的创建、挂载与卸载流程，能够在 `/proc` 和 `/sys` 中观察内核的文件系统状态。

**前置条件**：一台 Linux 虚拟机或容器（需要 root 权限），推荐 Debian/Ubuntu 系发行版。

#### 步骤 1：观察当前文件系统状态（10 分钟）

```bash
# 列出所有已挂载的文件系统及其类型
mount | head -20

# 查看 VFS 层的统计信息：打开的文件数、已用/未用 inode 等
cat /proc/fs/file-nr
# 输出格式：已分配inode数  未使用inode数  inode上限

# 查看内核支持的所有文件系统类型
cat /proc/filesystems

# 查看 ext4 相关的内核参数
sysctl -a | grep ext4
```

#### 步骤 2：创建并挂载 ext4 文件系统（15 分钟）

```bash
# 创建一个 512MB 的稀疏文件作为虚拟磁盘
dd if=/dev/zero of=/tmp/ext4_disk.img bs=1M count=0 seek=512

# 查看稀疏文件的实际磁盘占用（应远小于 512MB）
ls -lh /tmp/ext4_disk.img
du -sh /tmp/ext4_disk.img

# 关联块设备（使用 loop 设备）
sudo losetup -fP /tmp/ext4_disk.img
LOOP_DEV=$(losetup -j /tmp/ext4_disk.img | awk -F: '{print $1}')
echo "Loop device: $LOOP_DEV"

# 创建 ext4 文件系统，指定块大小为 4KB
sudo mkfs.ext4 -b 4096 -L "test_vol" $LOOP_DEV

# 查看文件系统元信息
sudo tune2fs -l $LOOP_DEV

# 创建挂载点并挂载
sudo mkdir -p /mnt/test_ext4
sudo mount -t ext4 $LOOP_DEV /mnt/test_ext4

# 验证挂载
df -Th /mnt/test_ext4
mount | grep test_ext4
```

#### 步骤 3：通过 debugfs 观察内核数据结构（15 分钟）

```bash
# 使用 debugfs 观察 inode 和块组结构（只读模式）
sudo debugfs -R "stats" $LOOP_DEV

# 查看根目录的 inode 信息
sudo debugfs -R "stat <2>" $LOOP_DEV
# inode 2 是 ext4 的根目录 inode

# 在挂载目录创建测试文件，观察 inode 分配
echo "Hello VFS" | sudo tee /mnt/test_ext4/hello.txt
sudo stat /mnt/test_ext4/hello.txt

# 对比 stat 输出中的关键字段：
# - File: 文件路径（dentry 解析结果）
# - Size/Blocks/IO Block: 物理布局信息
# - Inode: inode 号（VFS inode 对象标识）
# - Access/Modify/Change: 三种时间戳
```

**检查标准**：

- [ ] 能解释 `/proc/filesystems` 和 `mount` 输出的含义
- [ ] 能完整执行 ext4 虚拟磁盘的创建、格式化、挂载流程
- [ ] 能用 `stat` 和 `debugfs` 观察 inode 信息，理解 VFS 通过 inode 号定位数据块的机制
- [ ] 能区分文件的三个时间戳（atime/mtime/ctime）的触发条件

### 练习二：inode 管理与文件元数据操作（预计 50 分钟）

**目标**：深入理解 inode 的内部结构与资源管理，掌握硬链接/软链接的本质差异，能够诊断 inode 耗尽问题。

#### 步骤 1：硬链接与软链接的内核差异（15 分钟）

```bash
cd /mnt/test_ext4

# 创建原始文件
echo "Original content" > original.txt
INODE=$(stat -c "%i" original.txt)
echo "Original inode: $INODE"

# 创建硬链接
ln original.txt hardlink.txt
echo "Hardlink inode: $(stat -c "%i" hardlink.txt)"
# 预期：硬链接 inode 号与原文件相同

# 创建符号链接
ln -s original.txt symlink.txt
echo "Symlink inode: $(stat -c "%i" symlink.txt)"
# 预期：符号链接 inode 号不同，是独立的 inode

# 查看硬链接的链接计数
ls -li original.txt hardlink.txt
# 原始文件的 link count 应为 2

# 删除原文件后观察
rm original.txt
cat hardlink.txt    # 仍然可读——硬链接通过 inode 引用数据
cat symlink.txt     # 断链报错——符号链接依赖路径解析
```

#### 步骤 2：inode 耗尽模拟与诊断（20 分钟）

```bash
# 查看当前文件系统的 inode 总量和使用情况
df -i /mnt/test_ext4

# 创建大量小文件以耗尽 inode（每个空文件占用 1 个 inode）
# 小文件系统默认 inode 数有限：512MB ext4 约有 65536 个 inode
for i in $(seq 1 60000); do
    touch /mnt/test_ext4/fill_${i}.txt
done

# 再次查看 inode 使用情况
df -i /mnt/test_ext4
# 预期：IUse% 接近 100%

# 尝试创建新文件——应报错 "No space left on device"
touch /mnt/test_ext4/cannot_create.txt
# 注意：df 可能显示仍有磁盘空间（空间充足但 inode 耗尽）

# 清理：删除所有填充文件
find /mnt/test_ext4 -name "fill_*.txt" -delete
df -i /mnt/test_ext4
```

#### 步骤 3：用 stat 深入分析 inode 结构（15 分钟）

```bash
# 创建不同类型的文件，观察 inode 差异
mkdir /mnt/test_ext4/dir_type
mkfifo /mnt/test_ext4/my_pipe          # 管道文件
sudo mknod /mnt/test_ext4/my_char c 1 3  # 字符设备（null）
ln -s /tmp /mnt/test_ext4/my_link       # 符号链接

# 逐个观察 stat 输出中的 File type 字段
for f in /mnt/test_ext4/dir_type /mnt/test_ext4/my_pipe \
         /mnt/test_ext4/my_char /mnt/test_ext4/my_link; do
    echo "=== $f ==="
    stat "$f" | grep -E "File:|Inode:|Access:|Type:"
    echo
done
```

**检查标准**：

- [ ] 能解释硬链接与软链接在 inode 层面的本质区别（硬链接共享 inode，软链接是独立 inode 存储路径）
- [ ] 能使用 `df -i` 诊断 inode 耗尽，并理解其与磁盘空间耗尽的差异
- [ ] 能通过 `stat` 输出识别文件类型（普通文件、目录、符号链接、设备文件、管道等）
- [ ] 能解释为什么目录的硬链接数至少为 2（`.` 和父目录中的条目）

### 练习三：ext4 日志模式与数据一致性（预计 45 分钟）

**目标**：理解 ext4 三种日志模式（writeback、ordered、journal）的工作原理和性能差异，能够通过断电模拟验证日志模式对数据一致性的影响。

#### 步骤 1：配置不同日志模式（10 分钟）

```bash
# 查看当前日志模式
sudo tune2fs -l $LOOP_DEV | grep "Default mount options"

# 创建三个不同的文件系统，分别使用三种日志模式
for mode in writeback ordered journal; do
    sudo dd if=/dev/zero of=/tmp/ext4_${mode}.img bs=1M count=0 seek=128
    LOOP=$(sudo losetup -fP --show /tmp/ext4_${mode}.img)
    sudo mkfs.ext4 -b 4096 -O journal_data=$mode $LOOP 2>/dev/null
    sudo mkdir -p /mnt/ext4_${mode}
    sudo mount -o journal=$mode $LOOP /mnt/ext4_${mode}
    echo "Mounted $mode on /mnt/ext4_${mode}"
done
```

#### 步骤 2：理解日志写入流程（20 分钟）

```bash
# 使用 strace 跟踪写入操作的系统调用序列
sudo strace -e trace=openat,write,fsync,sync -f \
    bash -c 'echo "test data" > /mnt/ext4_ordered/testfile.txt &amp;&amp; sync'

# 对比三种日志模式的 strace 输出差异：
# - journal 模式：数据和元数据都先写入日志区
# - ordered 模式：只在日志区写元数据，数据直接写目标位置
# - writeback 模式：元数据和数据都直接写目标位置，无日志保障

# 观察日志区的磁盘活动
sudo debugfs -R "stat <8>" /tmp/ext4_ordered.img 2>/dev/null
# inode 8 是 ext4 的 journal inode
```

#### 步骤 3：使用 fio 模拟断电场景下的性能差异（15 分钟）

```bash
# 安装 fio（如果尚未安装）
sudo apt-get install -y fio 2>/dev/null || sudo yum install -y fio

# 对三种日志模式分别运行基准测试
for mode in writeback ordered journal; do
    echo "=== Testing $mode mode ==="
    fio --name=journal_test --directory=/mnt/ext4_${mode} \
        --rw=randwrite --bs=4k --size=64M --numjobs=4 \
        --runtime=10 --time_based --output-format=json \
        --output=/tmp/fio_${mode}.json

    # 提取关键指标
    python3 -c "
import json
with open('/tmp/fio_${mode}.json') as f:
    data = json.load(f)
    r = data['jobs'][0]['write']
    print(f'  Mode: ${mode}')
    print(f'  IOPS: {r[\"iops\"]:.0f}')
    print(f'  BW: {r[\"bw\"]/1024:.1f} MB/s')
    print(f'  Latency avg: {r[\"lat_ns\"][\"mean\"]/1000:.1f} us')
"
done
```

预期性能排序（随机小写入场景）：writeback > ordered > journal，其中 journal 模式因为双重写入（日志区 + 数据区）性能最低，但崩溃恢复能力最强。

**检查标准**：

- [ ] 能解释三种日志模式的写入路径差异（哪些数据经过日志区）
- [ ] 能解释为什么 ordered 模式是 ext4 的默认选择（性能与安全的平衡点）
- [ ] 能通过 fio 基准测试量化三种模式的性能差异
- [ ] 能说明日志模式在数据库、日志服务器等不同场景下的选型依据

### 练习四：文件系统性能诊断与调优（预计 60 分钟）

**目标**：掌握使用 iostat、blktrace、perf 等工具诊断文件系统性能瓶颈，能够识别并优化常见的 IO 性能问题。

#### 步骤 1：建立性能基线（15 分钟）

```bash
# 安装必要工具
sudo apt-get install -y sysstat blktrace fio 2>/dev/null

# 使用 fio 建立基线数据
# 顺序读测试
fio --name=seq_read --directory=/mnt/test_ext4 \
    --rw=read --bs=128k --size=256M --numjobs=1 \
    --runtime=10 --time_based --output-format=json \
    --output=/tmp/fio_seq_read.json

# 顺序写测试
fio --name=seq_write --directory=/mnt/test_ext4 \
    --rw=write --bs=128k --size=256M --numjobs=1 \
    --runtime=10 --time_based --output-format=json \
    --output=/tmp/fio_seq_write.json

# 随机读写混合测试（模拟数据库场景）
fio --name=rand_rw --directory=/mnt/test_ext4 \
    --rw=randrw --rwmixread=70 --bs=4k --size=64M \
    --numjobs=4 --runtime=10 --time_based \
    --output-format=json --output=/tmp/fio_rand_rw.json

# 使用 iostat 观察块设备级别的 IO 统计
iostat -xm 1 5
```

#### 步骤 2：使用 blktrace 追踪 IO 路径（20 分钟）

```bash
# 启动 blktrace 在后台追踪
sudo blktrace -d $(dirname $LOOP_DEV) -o - | \
    blkparse -i - > /tmp/blktrace_output.txt &amp;
BLKTRACE_PID=$!

# 运行工作负载
fio --name=trace_test --directory=/mnt/test_ext4 \
    --rw=randwrite --bs=4k --size=32M --numjobs=2 \
    --runtime=5 --time_based

# 停止追踪
sudo kill $BLKTRACE_PID
sleep 2

# 分析 IO 操作分布
cat /tmp/blktrace_output.txt | \
    awk '{print $6}' | sort | uniq -c | sort -rn | head
# 输出各 IO 操作类型（读、写、flush、fua 等）的频率

# 分析 IO 延迟分布
btt -i /tmp/blktrace_output.*.blktrace.0 2>/dev/null | \
    grep -A5 "D2C"
```

#### 步骤 3：实施针对性调优（25 分钟）

```bash
# 调优 1：调整预读窗口大小（适用于顺序读密集场景）
# 查看当前预读值（单位：512 字节扇区）
cat /sys/block/$(basename $LOOP_DEV)/queue/read_ahead_kb
# 默认通常为 128（即 64KB）

# 临时调整为 2048KB（适用于大文件顺序读）
echo 2048 | sudo tee /sys/block/$(basename $LOOP_DEV)/queue/read_ahead_kb

# 调优 2：调整 IO 调度器
cat /sys/block/$(basename $LOOP_DEV)/queue/scheduler
# 查看当前调度器和可用选项

# 对 SSD/虚拟设备推荐 none/mq-deadline
echo none | sudo tee /sys/block/$(basename $LOOP_DEV)/queue/scheduler

# 调优 3：调整 ext4 相关内核参数
# 禁用 delalloc（延迟分配）以避免断电丢数据风险
sudo mount -o remount,nodelalloc /mnt/test_ext4

# 重新运行 fio 对比调优前后的性能差异
```

**检查标准**：

- [ ] 能使用 fio 构建覆盖顺序读/写、随机读/写的完整基准测试矩阵
- [ ] 能使用 `iostat` 读懂 IOPS、吞吐量、平均延迟、IO 队列深度等关键指标
- [ ] 能使用 `blktrace` 追踪并分析具体的 IO 路径
- [ ] 能解释预读窗口、IO 调度器、delalloc 等参数对不同类型工作负载的影响

### 练习五：Btrfs 快照与子卷管理（预计 40 分钟）

**目标**：掌握 Btrfs 的快照机制和子卷管理，理解 COW（Copy-on-Write）的工作原理，能够使用快照实现数据保护和回滚。

#### 步骤 1：创建 Btrfs 文件系统和子卷（15 分钟）

```bash
# 创建 Btrfs 文件系统
sudo dd if=/dev/zero of=/tmp/btrfs_disk.img bs=1M count=0 seek=512
LOOP_BTRFS=$(sudo losetup -fP --show /tmp/btrfs_disk.img)
sudo mkfs.btrfs -L "btrfs_test" $LOOP_BTRFS

sudo mkdir -p /mnt/btrfs_test
sudo mount -t btrfs $LOOP_BTRFS /mnt/btrfs_test

# 创建子卷结构
sudo btrfs subvolume create /mnt/btrfs_test/@data
sudo btrfs subvolume create /mnt/btrfs_test/@snapshots

# 写入测试数据
for i in $(seq 1 100); do
    echo "Data block $i - $(date +%s%N)" | \
        sudo tee /mnt/btrfs_test/@data/file_${i}.txt > /dev/null
done

echo "Created $(ls /mnt/btrfs_test/@data/ | wc -l) files"

# 查看子卷列表
sudo btrfs subvolume list /mnt/btrfs_test
```

#### 步骤 2：创建快照并验证 COW 行为（15 分钟）

```bash
# 创建只读快照（用于备份）
sudo btrfs subvolume snapshot -r \
    /mnt/btrfs_test/@data \
    /mnt/btrfs_test/@snapshots/data_snap_01

# 查看快照创建前后磁盘使用量
sudo btrfs filesystem usage /mnt/btrfs_test

# 修改原始数据——COW 机制下，原始文件和快照都不会被覆盖
echo "Modified content $(date +%s)" | \
    sudo tee /mnt/btrfs_test/@data/file_1.txt

# 对比快照和当前数据的差异
diff <(cat /mnt/btrfs_test/@data/file_1.txt) \
     <(cat /mnt/btrfs_test/@snapshots/data_snap_01/file_1.txt)

# 再次查看磁盘使用——由于 COW，使用量应该增加
sudo btrfs filesystem usage /mnt/btrfs_test
```

#### 步骤 3：使用快照实现回滚（10 分钟）

```bash
# 模拟数据损坏
echo "CORRUPTED DATA" | sudo tee /mnt/btrfs_test/@data/file_1.txt
cat /mnt/btrfs_test/@data/file_1.txt

# 从快照恢复单个文件
sudo cp /mnt/btrfs_test/@snapshots/data_snap_01/file_1.txt \
    /mnt/btrfs_test/@data/file_1.txt
cat /mnt/btrfs_test/@data/file_1.txt
# 预期：恢复为原始内容

# 完整子卷回滚：删除损坏子卷，用快照替换
sudo btrfs subvolume delete /mnt/btrfs_test/@data
sudo btrfs subvolume snapshot \
    /mnt/btrfs_test/@snapshots/data_snap_01 \
    /mnt/btrfs_test/@data
```

**检查标准**：

- [ ] 能解释 Btrfs COW 的工作原理（不修改原数据块，写入新位置并更新指针）
- [ ] 能区分快照与普通复制的性能差异（快照创建几乎瞬时，因为只复制元数据）
- [ ] 能创建只读/可写快照，并实现单文件级和子卷级的数据恢复
- [ ] 能通过 `btrfs filesystem usage` 观察快照对磁盘空间的增量影响

### 练习六：文件系统故障排查综合实战（预计 60 分钟）

**目标**：综合运用前面的工具和知识，排查真实的文件系统故障场景，培养系统化的故障诊断思维。

#### 场景一：磁盘空间已满但 df 显示有余量（20 分钟）

```bash
# 场景描述：进程删除了大文件，但磁盘空间未释放
cd /mnt/test_ext4

# 创建一个大文件
sudo dd if=/dev/zero of=bigfile.img bs=1M count=200
df -h /mnt/test_ext4

# 后台进程正在写入此文件
sudo tail -f bigfile.img &amp; 2>/dev/null  # 保持文件描述符打开
TAIL_PID=$!

# 在另一个终端删除文件
sudo rm bigfile.img

# 观察：df 仍然显示空间被占用！
df -h /mnt/test_ext4

# 诊断：查看被删除但仍被进程持有的文件
sudo lsof | grep deleted | grep test_ext4
# 预期：看到 bigfile.img 显示为 (deleted)

# 解决方案 1：重启持有文件描述符的进程
sudo kill $TAIL_PID

# 解决方案 2（不重启进程）：截断文件内容释放空间
# 先找到 PID 和 FD
FD_INFO=$(sudo lsof 2>/dev/null | grep deleted | grep test_ext4 | awk '{print $2, $4}' | head -1)
PID=$(echo $FD_INFO | awk '{print $1}')
FD=$(echo $FD_INFO | awk '{print $2}' | grep -o '[0-9]*')
if [ -n "$PID" ] &amp;&amp; [ -n "$FD" ]; then
    echo "" | sudo tee /proc/$PID/fd/$FD > /dev/null 2>&amp;1
fi
```

#### 场景二：文件系统只读挂载的排查与修复（20 分钟）

```bash
# 模拟文件系统损坏导致只读挂载
# 先卸载
sudo umount /mnt/test_ext4

# 使用 dd 破坏部分超级块数据（模拟磁盘故障）
sudo dd if=/dev/zero of=$LOOP_DEV bs=1 count=16 seek=1024 conv=notrunc

# 尝试挂载——可能报错或被强制只读挂载
sudo mount -t ext4 $LOOP_DEV /mnt/test_ext4 2>&amp;1

# 如果挂载失败，使用 fsck 修复
sudo fsck.ext4 -y $LOOP_DEV
# -y 自动修复所有问题

# 修复后重新挂载
sudo mount -t ext4 $LOOP_DEV /mnt/test_ext4
dmesg | tail -20  # 查看内核日志中的文件系统消息
```

#### 场景三：识别和清理大文件与目录（20 分钟）

```bash
# 找出占用空间最大的目录（前 10 个）
sudo du -h --max-depth=1 /mnt/test_ext4 2>/dev/null | \
    sort -rh | head -10

# 找出最大的单个文件
sudo find /mnt/test_ext4 -type f -exec du -h {} + 2>/dev/null | \
    sort -rh | head -10

# 找出创建时间最早的 10 个文件（清理遗留数据）
find /mnt/test_ext4 -type f -printf '%T+ %p\n' 2>/dev/null | \
    sort | head -10

# 找出 inode 占用最多的目录
```

#### 挑战 1：自定义 FUSE 文件系统

编写一个简单的 FUSE 文件系统，实现一个只读的内存文件系统——将 `/proc/meminfo` 的内容以文件形式呈现：

```c
// fuse_example.c — 编译命令：
// gcc -o fuse_meminfo fuse_meminfo.c $(pkg-config --cflags --libs fuse)
#define FUSE_USE_VERSION 30
#include <fuse3/fuse.h>
#include <stdio.h>
#include <string.h>

static int do_getattr(const char *path, struct stat *st) {
    st->st_uid = getuid();
    st->st_gid = getgid();
    st->st_atime = time(NULL);
    st->st_mtime = time(NULL);
    if (strcmp(path, "/") == 0) {
        st->st_mode = S_IFDIR | 0755;
        st->st_nlink = 2;
    } else if (strcmp(path, "/meminfo") == 0) {
        st->st_mode = S_IFREG | 0644;
        st->st_nlink = 1;
        // 读取真实 meminfo 作为文件大小
        FILE *fp = fopen("/proc/meminfo", "r");
        if (fp) {
            char buf[4096];
            size_t n = fread(buf, 1, sizeof(buf)-1, fp);
            buf[n] = '\0';
            st->st_size = n;
            fclose(fp);
        }
    } else return -ENOENT;
    return 0;
}

static int do_readdir(const char *path, void *buf,
                       fuse_fill_dir_t filler) {
    filler(buf, ".", NULL, 0);
    filler(buf, "..", NULL, 0);
    if (strcmp(path, "/") == 0)
        filler(buf, "meminfo", NULL, 0);
    return 0;
}

static int do_read(const char *path, char *buf, size_t size,
                    off_t offset) {
    if (strcmp(path, "/meminfo") != 0) return -ENOENT;
    FILE *fp = fopen("/proc/meminfo", "r");
    if (!fp) return -ENOENT;
    fseek(fp, offset, SEEK_SET);
    size_t n = fread(buf, 1, size, fp);
    fclose(fp);
    return n;
}

static const struct fuse_operations ops = {
    .getattr  = do_getattr,
    .readdir  = do_readdir,
    .read     = do_read,
};

int main(int argc, char *argv[]) {
    return fuse_main(argc, argv, &amp;ops, NULL);
}
```
