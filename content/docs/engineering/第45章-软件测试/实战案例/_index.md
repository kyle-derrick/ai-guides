# 实战案例

理论和技巧的价值最终要在真实项目中得到检验。本节通过两个精心设计的实战案例，展示软件测试从微观到宏观的完整落地路径——从为单个微服务编写高覆盖率的单元测试，到在生产级分布式系统中实施混沌工程实验验证韧性。

这两个案例并非独立存在，而是构成了一条递进式的学习路径：单元测试守护代码逻辑的正确性，混沌工程验证系统整体的韧性。一个成熟的测试策略需要两者兼顾——既要保证"每个零件不出问题"，也要验证"整台机器在极端条件下依然可靠"。

## 案例一：为微服务编写单元测试

**场景**：电商系统的订单服务（Order Service），创建订单时需要协调用户服务、库存服务、支付服务和订单持久化四个依赖。

**核心挑战**：微服务架构中，每个下游服务都是独立部署的网络节点。单元测试不能依赖真实网络，必须通过测试替身（Test Double）替代所有外部依赖。同时，订单创建流程中的补偿逻辑（支付失败→释放库存、持久化失败→退款+释放库存）是测试的重点难点。

**案例覆盖的关键技术点**：

1. **依赖注入与接口设计**——通过 Go 接口 + 构造函数实现可测试性
2. **手工 Mock 实现**——为四个依赖分别编写 Mock，支持行为记录与参数断言
3. **框架 Mock（gomock）**——使用代码生成工具减少样板代码，适合大型项目
4. **表驱动测试**——将正常场景和异常场景组织在一个测试表中，覆盖以下维度：
   - 正常路径：单商品、多商品创建
   - 用户异常：用户不存在、用户服务不可用
   - 库存异常：库存不足、库存服务超时
   - 支付异常：余额不足、支付服务宕机
   - 持久化异常：数据库写入失败
   - 补偿逻辑验证：支付失败后库存是否正确释放
5. **上下文超时测试**——验证服务在超时场景下的行为
6. **并发安全测试**——使用 `t.Parallel()` 确保测试隔离

**适用读者**：已掌握基本单元测试概念，希望在微服务项目中建立系统化测试实践的开发者。

**详细内容**：[案例一：为微服务编写单元测试](01-案例一为微服务编写单元测试.md)

## 案例二：混沌工程实验

**场景**：典型电商平台"闪购系统"，包含用户服务、订单服务、支付服务、库存服务、通知服务五个微服务，部署在 Kubernetes 集群上，使用 MySQL、Redis、RabbitMQ 作为数据层。

**核心挑战**：微服务架构中故障传播路径复杂，传统的单元测试和集成测试只能验证"已知的已知"，而生产环境中的故障往往是"未知的未知"。混沌工程通过有计划地注入故障，在真实故障发生前暴露系统的脆弱点。

**案例覆盖的六个核心实验**：

| 实验 | 注入的故障 | 验证的稳态假设 | 关键工具 |
|------|----------|-------------|---------|
| 实验一：库存服务不可用 | Docker 容器停止 | 订单服务快速失败而非阻塞 | ChaosBlade docker stop |
| 实验二：Redis 主节点故障 | 网络延迟 + 容器冻结 | 库存查询自动切换从节点 | ChaosBlade network delay + docker pause |
| 实验三：MySQL 从库故障 | 磁盘 IO 压力 + 网络延迟 | 读请求自动路由主库 | ChaosBlade disk fill + network delay |
| 实验四：消息队列节点故障 | 进程终止 | 消息不丢失（镜像队列验证） | ChaosBlade process kill |
| 实验五：网络延迟突增 | 网络延迟 + 丢包 | 熔断器触发而非无限等待 | ChaosBlade network delay + loss |
| 实验六：节点 CPU 打满 | CPU 满载压力 | Kubernetes 自动调度到其他节点 | ChaosBlade cpu fullload |

**案例覆盖的关键技术点**：

1. **混沌工程方法论**——定义稳态假设→设计实验→执行实验→分析结果的科学流程
2. **工具选型**——对比 Chaos Mesh、Litmus Chaos、ChaosBlade、Gremlin 等主流工具的适用场景
3. **实验执行**——每个实验的完整步骤、命令、基线指标记录方法
4. **安全防护**——爆炸半径控制、自动熔断回滚、实验审批流程
5. **实验平台搭建**——从命令行实验到平台化的演进路径（管理→实验→观测→基础设施四层架构）
6. **指标观测**——基于 Prometheus + Grafana 的实验前后指标对比分析
7. **混沌工程成熟度模型**——从"工具试用"到"持续验证"的四个阶段

**适用读者**：已掌握微服务基础架构，希望在团队中引入混沌工程实践、提升系统韧性的工程师和架构师。

**详细内容**：[案例二：混沌工程实验](02-案例二混沌工程实验.md)

## 两个案例的协同关系

单元测试和混沌工程并非替代关系，而是互补关系。下表展示了它们在测试体系中的定位差异：

| 维度 | 案例一：单元测试 | 案例二：混沌工程 |
|------|----------------|----------------|
| 测试粒度 | 单个函数/方法 | 整个系统/集群 |
| 隔离程度 | 完全隔离（所有依赖被Mock） | 真实环境（或高度仿真环境） |
| 发现的问题 | 逻辑错误、边界条件、异常处理缺陷 | 系统级脆弱点、级联故障、配置错误 |
| 执行频率 | 每次代码提交（CI） | 定期执行（每周/每月） |
| 对系统的侵入 | 无侵入 | 控制范围内的侵入 |
| 失败成本 | 低（测试环境） | 高（可能影响生产） |
| 回报周期 | 即时（开发者本地） | 中长期（预防线上事故） |

一个成熟的工程团队应该同时投资两个层面：用单元测试保证代码变更不引入回归，用混沌工程持续验证系统在故障条件下的韧性。

## 学习建议

**初学者路径**：先精读案例一，理解单元测试的完整流程（接口设计→Mock实现→测试编写→边界覆盖），再通过案例二建立系统级测试的视野。案例一中的手工Mock技术和表驱动测试模式会在日常开发中频繁使用。

**有经验的开发者**：可以直接进入案例二，但建议先快速浏览案例一中关于补偿逻辑测试的部分——支付失败释放库存、持久化失败退款的测试模式，在分布式系统测试中具有普遍参考价值。

**团队负责人**：重点关注案例二中的实验平台架构和混沌工程成熟度模型，它们为团队从"偶尔做混沌实验"演进到"持续验证系统韧性"提供了清晰的路线图。

## 前置知识要求

在学习本节内容前，建议读者已掌握以下基础知识：

- **编程基础**：至少熟悉一门编程语言（案例一使用 Go，案例二涉及 Go 和 Java）
- **微服务架构**：理解服务间通信、依赖注入、API 设计等基本概念
- **Docker 与 Kubernetes**：案例二的实验在容器化环境中执行，需要了解 Pod、Deployment、Service 等 K8s 核心资源
- **基本的监控概念**：了解指标（Metrics）、日志（Logs）、追踪（Traces）三大支柱，案例二使用 Prometheus + Grafana 进行指标观测
- **第45章的理论基础和核心技巧**：建议先阅读本章的"理论基础"和"核心技巧"部分，掌握测试替身分类、Mock 策略、测试金字塔等前置知识
