# Linux内核性能评估的关键指标与度量方法

内核源码分析的终极目标之一是**理解和优化系统性能**。无论你是在排查数据库延迟飙升、Web服务吞吐量不足，还是在评估新内核版本的性能回归，都需要一套系统化的指标体系来量化内核行为。本节从CPU、内存、I/O、网络、调度五个维度建立内核性能的完整度量框架，给出每个指标的定义、来源、采集工具和优化方向，并提供可以直接在生产环境使用的监控脚本。

---

## 1 内核性能指标总览

Linux内核的性能指标可以从两个层面理解：**子系统级指标**（某个内核子系统内部的效率度量）和**系统级指标**（面向用户可感知的最终性能）。两者的关系如下：

用户可感知的性能（延迟、吞吐量）
        │
        ├── CPU维度：利用率、上下文切换、中断频率、缓存命中率
        ├── 内存维度：缺页率、SLUB分配延迟、OOM事件、NUMA本地访问率
        ├── I/O维度：IOPS、带宽、平均队列深度、I/O等待时间
        ├── 网络维度：收发包速率、丢包率、TCP重传率、软中断CPU占比
        └── 调度维度：调度延迟、运行队列长度、CFS带宽节流、唤醒延迟

| 维度 | 核心指标 | 对应内核子系统 | 典型采集工具 |
|------|---------|--------------|-------------|
| CPU | 利用率、上下文切换、缓存命中率 | `kernel/sched/`、`arch/x86/` | `perf stat`、`/proc/stat` |
| 内存 | 缺页率、OOM事件、NUMA命中率 | `mm/` | `vmstat`、`/proc/meminfo` |
| I/O | IOPS、吞吐量、队列深度 | `block/`、驱动层 | `iostat`、`/proc/diskstats` |
| 网络 | 收发速率、丢包率、重传率 | `net/` | `sar -n DEV`、`/proc/net/softnet_stat` |
| 调度 | 调度延迟、运行队列长度 | `kernel/sched/` | `perf sched`、`/proc/schedstat` |

## 2 CPU维度指标

### 2.1 CPU利用率

CPU利用率是最基础的性能指标，但其细分项往往比总利用率更有诊断价值。

**指标来源：/proc/stat**

# cat /proc/stat | head -1
cpu  12345 678 9012 345678 901 0 234 0 0 0
│    │     │   │    │      │   │ │
│    │     │   │    │      │   │ └── steal（虚拟化中被宿主机偷走的时间）
│    │     │   │    │      │   └── softirq
│    │     │   │    │      └── hardirq
│    │     │   │    └── idle（空闲时间）
│    │     │   └── system（内核态时间）
│    │     └── nice（低优先级用户态时间）
│    └── user（用户态时间）
└── cpu（所有CPU的汇总）

**关键细分指标：**

| 指标 | 含义 | 健康范围 | 异常信号 |
|------|------|---------|---------|
| `%usr` | 用户态CPU | 视负载而定 | 持续>90%说明应用瓶颈 |
| `%sys` | 内核态CPU | <30% | 持续>30%说明系统调用过频或锁竞争严重 |
| `%iowait` | 等待I/O的空闲时间 | <10% | 高值说明磁盘I/O是瓶颈 |
| `%irq` + `%softirq` | 硬中断+软中断CPU占比 | <10% | 高值说明网卡中断过多或中断未均衡分配 |
| `%steal` | 被宿主机偷走的时间 | 0% | >0%说明虚拟化超售或宿主机负载高 |
| `%idle` | 空闲时间 | >10% | 0%说明CPU饱和 |

**采集脚本（1秒间隔，采样10次取平均）：**

```bash
#!/bin/bash
# cpu_stat.sh - 采集CPU利用率细分指标
INTERVAL=1
COUNT=10

echo "=== CPU利用率统计（${COUNT}次采样，间隔${INTERVAL}秒）==="
mpstat ${INTERVAL} ${COUNT} | tail -1 | awk '
NR==1 {
    printf "%-10s %-10s %-10s %-10s %-10s %-10s %-10s\n", "usr%", "sys%", "iowait%", "irq%", "softirq%", "steal%", "idle%"
}
{
    printf "%-10.2f %-10.2f %-10.2f %-10.2f %-10.2f %-10.2f %-10.2f\n", $3, $5, $6, $11, $12, $13, $14
}'
```

### 2.2 上下文切换

上下文切换是内核调度行为的直接体现。过高的上下文切换意味着进程频繁被调度切换，每次切换都会带来CPU流水线清空、TLB失效、缓存冷启动等开销。

**指标来源：/proc/stat 和 /proc/[pid]/status**

# 系统级上下文切换
# grep ctxt /proc/stat
ctxt 2345678

# 进程级上下文切换
# grep voluntary /proc/[pid]/status
voluntary_ctxt_switches:  12345
nonvoluntary_ctxt_switches:  678

**两种上下文切换的区别：**

| 类型 | 原因 | 优化方向 |
|------|------|---------|
| 主动切换（voluntary） | 进程主动让出CPU（等待I/O、sleep、mutex_lock） | 减少阻塞操作、使用异步I/O |
| 非主动切换（nonvoluntary） | 时间片耗尽被抢占、被更高优先级进程抢占 | 增大时间片（nice -n）、调整cgroup配额 |

**采集命令：**

```bash
# 使用perf统计上下文切换
sudo perf stat -e context-switches -a -- sleep 10

# 实时监控上下文切换速率
vmstat 1 10 | awk '{print NR, "cs:", $12}'
```

**内核源码路径：**
- 上下文切换入口：`kernel/sched/core.c` → `__schedule()` → `context_switch()`
- 页表切换：`context_switch()` → `switch_mm_irqs_off()`
- 寄存器保存/恢复：`context_switch()` → `switch_to()`

### 2.3 缓存命中率

CPU缓存（L1/L2/L3）的命中率对性能有决定性影响。一次L1缓存命中的延迟约为1纳秒，而一次主存访问约为100纳秒——差距超过100倍。

**采集命令：**

```bash
# 使用perf采集缓存事件（需要CPU支持PMC）
sudo perf stat -e cache-references,cache-misses,L1-dcache-loads,L1-dcache-load-misses \
    -a -- sleep 10

# 输出示例：
#  12,345,678  cache-references
#    234,567  cache-misses    (1.90%)
#  45,678,901  L1-dcache-loads
#    567,890  L1-dcache-load-misses (1.24%)
```

**缓存层次与延迟对照：**

| 缓存层级 | 容量 | 命中延迟 | 失效后果 |
|---------|------|---------|---------|
| L1 dcache | 32-64 KB/核心 | ~1 ns | 回退到L2 |
| L2 cache | 256 KB-1 MB/核心 | ~4 ns | 回退到L3 |
| L3 cache | 数MB-数十MB（共享） | ~10 ns | 回退到主存 |
| 主存（DRAM） | 数GB-数TB | ~100 ns | 直接访问 |
| NUMA远端内存 | — | ~150-300 ns | 性能进一步劣化 |

**优化方向：**
- 减少缓存行伪共享（false sharing）：使用`____cacheline_aligned`宏确保关键数据结构独占缓存行
- 减少指针追踪：缓存友好的数据结构布局（如数组优于链表）
- NUMA感知：使用`numactl --membind`将内存分配在本地节点

## 3 内存维度指标

### 3.1 内存使用与缺页

**指标来源：/proc/meminfo 和 vmstat**

```bash
# 关键内存指标
free -h
# total    used    free    shared  buff/cache  available
# 62Gi     8.2Gi   52Gi    256Mi   1.8Gi       53Gi

# 缺页统计
vmstat 1 10
# procs -----------memory---------- ---swap----- io---- -system-- ------cpu-----
#  r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
#  1  0  0  54746840  16384 1884160    0    0     0     0  234  456  2  1 97  0  0
```

**关键内存指标详解：**

| 指标 | 内核来源 | 含义 | 优化方向 |
|------|---------|------|---------|
| **minor fault** | `/proc/vmstat` 中 `nr_inactive_anon` | 不需要磁盘I/O的缺页（COW、首次访问已映射页） | 合理的mmap预分配 |
| **major fault** | `/proc/vmstat` 中 `pgmajfault` | 需要磁盘I/O的缺页（从磁盘加载数据） | 增加页缓存、使用大页 |
| **OOM kill** | `dmesg | grep "Out of memory"` | 内核OOM killer终止进程 | 调整overcommit策略、增加物理内存 |
| **swap使用** | `/proc/meminfo` 中 `SwapTotal - SwapFree` | 交换空间使用量 | 减少内存压力、调整swappiness |
| **Slab使用** | `/proc/meminfo` 中 `Slab` | 内核对象缓存（SLUB） | 检查是否有内核内存泄漏（`slabtop`） |

**缺页中断的内核路径：**

用户访问未映射的虚拟地址
  → CPU触发缺页异常（page fault）
    → do_page_fault()           # arch/x86/mm/fault.c
      → find_vma()              # 查找VMA
      → handle_mm_fault()       # mm/memory.c
        ├── do_anonymous_page() # 匿名页：分配新物理页，清零，映射
        ├── do_wp_page()        # COW页：复制物理页，更新映射
        └ → filemap_fault()     # 文件页：从页缓存或磁盘加载
          → page_cache_readahead()  # 预读

### 3.2 NUMA内存分布

在NUMA（Non-Uniform Memory Access）架构上，内存访问延迟取决于物理内存与CPU的拓扑关系。本地内存访问约为100ns，远端内存访问可达200-300ns。

```bash
# 查看NUMA拓扑
numactl --hardware
# available: 2 nodes (0-1)
# node 0 cpus: 0 1 2 3 4 5 6 7
# node 0 size: 32768 MB
# node 1 cpus: 8 9 10 11 12 13 14 15
# node 1 size: 32768 MB

# 查看进程的NUMA内存分布
numastat -p $(pgrep mysqld)
# Per-node process memory usage (in MBs)
# Node 0     Node 1     Total
# 1234       567        1801

# 检查远端内存访问比例
numastat -m | grep "numa_miss\|numa_hit"
```

**NUMA性能影响：**

| 场景 | 延迟影响 | 优化策略 |
|------|---------|---------|
| 本地内存访问 | 基准（~100ns） | 无需优化 |
| 远端内存访问 | 1.5-3x延迟 | `numactl --cpunodebind=0 --membind=0` 绑定 |
| NUMA interleaving | 降低延迟方差 | `numactl --interleave=all`（适用于大内存池） |

### 3.3 Slab分配器统计

SLUB分配器管理内核小对象的分配，其统计信息可以揭示内核内存的使用模式和潜在泄漏。

```bash
# 交互式查看slab使用排名
slabtop --once --sort=c

# 输出示例（节选）：
#  OBJS ACTIVE  USE OBJ SIZE  SLABS OBJ/SLAB CACHE SIZE NAME
# 102400  98304  96% 0.04K   1600     64       640K dentry
#  81920  76800  93% 0.06K    800    102       640K inode_cache
#  51200  48000  93% 0.05K    500    102       400K ext4_inode_cache
#  40960  36000  87% 0.20K   1024     40       800K task_struct

# 查看特定slab的详细信息
slabtop --once --contains=task_struct -o
```

**关键slab对象与内核子系统的对应关系：**

| slab对象 | 内核子系统 | 典型场景 |
|---------|-----------|---------|
| `task_struct` | 进程管理 | 进程/线程数 |
| `dentry` / `inode_cache` | VFS/文件系统 | 打开的文件数 |
| `ext4_inode_cache` | ext4 | ext4文件系统的inode缓存 |
| `sk_buff` | 网络子系统 | 网络数据包 |
| `filp` | 文件系统 | 打开的文件描述符 |
| `radix_tree_node` | 页缓存 | 页缓存索引节点 |

## 4 I/O维度指标

### 4.1 块设备性能指标

**指标来源：/proc/diskstats 和 iostat**

```bash
# iostat 查看块设备性能
iostat -dx 1 10

# 输出示例：
# Device  r/s   w/s   rkB/s  wkB/s  rrqm/s  wrqm/s  %rrqm  %wrqm  r_await  w_await  aqu-sz  rareq-sz  wareq-sz  svctm  %util
# sda     120.5 345.2 4820.0 13808.0 12.5   189.3   9.4    35.4    0.82     1.23     0.56    40.0      40.0     2.1    45.2
# nvme0n1 890.3 1200.5 28489.6 48020.0 2.1    45.6    0.2    3.6     0.12     0.34     0.74    32.0      40.0     0.4    78.3
```

**关键I/O指标详解：**

| 指标 | 含义 | 健康范围 | 异常信号 |
|------|------|---------|---------|
| **r/s, w/s** | 每秒读/写请求数（IOPS） | HDD: <200, SSD: <100K | 接近设备极限说明I/O饱和 |
| **rkB/s, wkB/s** | 每秒读/写吞吐量 | 取决于设备 | 接近接口带宽说明瓶颈 |
| **r_await, w_await** | 读/写平均延迟（ms） | SSD: <1ms, HDD: <10ms | 延迟突增说明队列拥塞 |
| **aqu-sz** | 平均队列深度 | <设备并发数 | >设备并发数说明请求堆积 |
| **%util** | 设备利用率 | <80% | >90%说明设备饱和 |
| **rrqm/s, wrqm/s** | 合并的请求数 | 越高越好 | 低值说明I/O模式不连续 |

**注意**：`%util`在NVMe SSD上可能不准确（NVMe支持大量并发队列），此时应关注`r_await`和`w_await`。

### 4.2 I/O调度器与队列深度

Linux内核的I/O请求从应用层到硬件经历多层处理：

用户态 read()/write()
  → VFS → 页缓存（若未命中）
    → 通用块层（bio合并、排序）
      → I/O调度器（mq-deadline/bfq/none）
        → 多队列块层（blk-mq）
          → 设备驱动 → 硬件

**I/O调度器选择：**

| 调度器 | 适用场景 | 特点 |
|-------|---------|------|
| `none` | NVMe SSD、虚拟化环境 | 无调度开销，最低延迟 |
| `mq-deadline` | 数据库、混合读写 | 保证请求在deadline内完成 |
| `bfq` | 桌面交互、低速设备 | 公平性好，适合低并发 |
| `kyber` | 快速设备（NVMe） | 简单的双队列调度 |

```bash
# 查看当前调度器
cat /sys/block/sda/queue/scheduler
# [mq-deadline] kyber bfq none

# 切换调度器
echo mq-deadline > /sys/block/sda/queue/scheduler

# 查看队列深度
cat /sys/block/sda/queue/nr_requests
# 128
```

### 4.3 页缓存效率

页缓存（Page Cache）是Linux I/O性能的核心。高效的页缓存命中可以将磁盘I/O延迟从毫秒级降低到纳秒级。

```bash
# 查看页缓存状态
grep -E "Cached|Buffers|Dirty|Writeback" /proc/meminfo
# Cached:          1945600 kB
# Buffers:          16384 kB
# Dirty:            12288 kB
# Writeback:         4096 kB

# 查看脏页回写配置
sysctl vm.dirty_ratio              # 脏页占内存比例阈值（默认20%）
sysctl vm.dirty_background_ratio   # 后台回写触发阈值（默认10%）
sysctl vm.dirty_expire_centisecs   # 脏页过期时间（默认30秒）
sysctl vm.dirty_writeback_centisecs # 回写线程唤醒间隔（默认5秒）
```

**页缓存命中率计算：**

```bash
# 方法1：通过/proc/vmstat计算
cat /proc/vmstat | grep -E "pgpgin|pgpgout|pgmajfault"
# pgpgin 12345678    # 从磁盘读入的页数（千页）
# pgpgout 9876543    # 写出到磁盘的页数
# pgmajfault 4567    # 主缺页次数（需要磁盘I/O的缺页）

# 方法2：使用perf工具
sudo perf stat -e dTLB-loads,dTLB-load-misses,page-faults -a -- sleep 10
```

## 5 网络维度指标

### 5.1 收发包性能

**指标来源：/proc/net/dev 和 sar**

```bash
# 实时监控网络流量
sar -n DEV 1 10

# 输出示例：
# IFACE   rxpck/s  txpck/s  rxkB/s  txkB/s  rxcmp/s  txcmp/s  rxmcst/s  %ifutil
# eth0    15678.3  12345.6  23456.7 18901.2  0.0      0.0      0.0       45.2
# lo      567.8    567.8    12.3    12.3     0.0      0.0      0.0       0.0
```

### 5.2 网络错误与丢包

```bash
# 接口级错误统计
cat /proc/net/dev
# eth0: 123456789 0 1234 0 0 0 0 5678 987654321 0 9876 0 0 0 0 2345 0
#        │bytes     │errors  │drop   │bytes    │errors
#        │recv      │recv    │recv   │send     │send

# 软中断统计（网络包处理的关键路径）
cat /proc/net/softnet_stat
# 每行一个CPU，各列含义：
# 第1列：已处理的包数
# 第2列：因时间片用尽被丢弃的包数（应为0）
# 第3列：因softnet_data满被丢弃的包数（应为0，否则增加netdev_budget）

# socket级统计
cat /net/snmp | grep Tcp
# 关注：
# RetransSegs: TCP重传段数
# InErrs: 收到的错误段数
# OutRsts: 发出的RST段数
```

### 5.3 TCP连接性能

```bash
# TCP连接状态分布
ss -s
# Total: 2345
# TCP:   1890 (estab 1678, closed 56, orphaned 23, timewait 56)

# TCP重传率
netstat -s | grep -i retrans
#     2345 segments retransmitted
#     12 fast retransmits
#     89 timeouts

# 每个连接的重传情况
ss -ti | head -20
# ESTAB  0  0  192.168.1.1:45678  10.0.0.1:3306
#    cubic wscale:7,7 rto:204 rtt:1.2/0.3 ato:40 mss:1448
#    rcvmss:1448 advmss:1448 cwnd:10 ssthresh:7
#    bytes_sent:1234567 bytes_acked:1234567
#    retrans:0/0 rcv_rtt:1.2 rcv_space:14600
```

**TCP关键性能指标：**

| 指标 | 含义 | 健康值 | 异常信号 |
|------|------|--------|---------|
| RTT (rtt) | 往返时延 | <1ms(本地) / <50ms(跨城) | 波动大说明网络拥塞或路由不稳定 |
| Cwnd (cwnd) | 拥塞窗口 | 持续增长为正常 | 频繁减半说明丢包严重 |
| RTO (rto) | 重传超时 | 200-300ms | 增大说明检测到丢包 |
| 重传率 | 重传段/总段数 | <0.1% | >1%严重影响吞吐量 |
| Ssthresh | 慢启动阈值 | 持续增长为正常 | 频繁降至低值说明反复触发拥塞 |

### 5.4 网络软中断与NAPI

Linux网络收包使用NAPI（New API）机制，将中断驱动和轮询模式结合，避免高流量下中断风暴：

网卡收到数据包
  → 触发硬件中断（hardirq）
    → 禁用网卡中断
    → 触发softirq（NET_RX_SOFTIRQ）
      → napi_poll() → 驱动的poll函数
        → netif_receive_skb() → 协议栈处理
          → 处理完毕，重新启用网卡中断

```bash
# 检查软中断是否成为瓶颈
# 如果softnet_stat的第2列不为0，说明有包被丢弃
cat /proc/net/softnet_stat | awk '{print "CPU" NR-1, "dropped:", strtonum("0x"$2)}'

# 检查单CPU软中断负载不均
mpstat -I SCPU 1 5
# 如果某CPU的%soft远高于其他CPU，说明需要设置IRQ亲和性

# 设置中断亲和性（将网卡中断绑定到CPU）
echo 2 > /proc/irq/48/smp_affinity  # 绑定到CPU1
```

## 6 调度维度指标

### 6.1 调度延迟与运行队列

调度延迟是衡量进程从"准备好运行"到"真正获得CPU"的时间。这个延迟直接影响应用程序的响应速度。

```bash
# 方法1：使用perf sched
sudo perf sched record -a -- sleep 10
sudo perf sched latency --sort max
# Output:
#   Task                  Count   Max             Sum          Avg
#   mysqld/1234           5678    1234.56 us      8901.23 us   1.57 us
#   nginx/5678            12345   456.78 us       5678.90 us   0.46 us

# 方法2：使用bpftrace
sudo bpftrace -e '
tracepoint:sched:sched_wakeup { @[comm] = hist(args->target_cpu); }
tracepoint:sched:sched_switch { @[comm] = hist(args->next_cpu); }
interval:s:5 { exit(); }
'

# 方法3：运行队列长度
cat /proc/schedstat | head -5
# version 15
# timestamp 0
# This field is dumped by kernel 2.6.35+
# cpu0 1234567890 987654321 456789
#      │total     │total     │run_queue_length × 时间 的积分
#      │run       │schedule
#      │time(ns)  │count
```

**调度延迟的来源分解：**

进程唤醒（wakeup）
  │
  ├── 1. 唤醒延迟：从wakeup()到被放入运行队列 → 通常<1μs
  │
  ├── 2. 排队延迟：在运行队列中等待被调度
  │   ├── 当前CPU有更高优先级进程运行 → 等待其让出
  │   ├── CFS调度器选择下一个进程 → O(log n)红黑树查找
  │   └── 抢占检查 → need_resched标志
  │
  └── 3. 上下文切换开销：从旧进程切换到新进程 → 通常5-10μs
      ├── 保存旧进程寄存器状态
      ├── 切换页表（mm_switch）
      └── 恢复新进程寄存器状态

### 6.2 CFS带宽控制

CFS支持通过cgroup控制CPU带宽，防止某个进程组独占CPU：

```bash
# 查看cgroup CPU配额
cat /sys/fs/cgroup/cpu/cpu.cfs_quota_us    # -1表示无限制
cat /sys/fs/cgroup/cpu/cpu.cfs_period_us   # 默认100000（100ms）

# 设置限制：最多使用2个CPU核心（200ms/100ms = 2）
echo 200000 > /sys/fs/cgroup/cpu/cpu.cfs_quota_us

# 查看CFS带宽节流统计
cat /proc/schedstat | grep bandwidth
# 或者通过perf追踪：
sudo perf stat -e 'sched:sched_stat_runtime' -a -- sleep 10
```

### 6.3 实时调度器指标

对于实时应用（音频处理、工业控制），使用`SCHED_FIFO`或`SCHED_DEADLINE`调度策略：

```bash
# 查看进程调度策略
chrt -p $(pgrep -f "audio_process")
# pid 12345:
#     policy: SCHED_FIFO
#     priority: 50

# 查看实时进程的调度统计
cat /proc/[pid]/sched
# audio_process (12345, #threads: 4)
# se.sum_exec_runtime : 123456789.012345
# se.nr_migrations : 2345
# nr_switches : 67890
# nr_voluntary_switches : 12345
# nr_involuntary_switches : 567
```

**调度策略对比：**

| 策略 | 优先级范围 | 特点 | 适用场景 |
|------|-----------|------|---------|
| `SCHED_NORMAL` (CFS) | nice -20 ~ +19 | 时间片公平分配 | 普通应用 |
| `SCHED_FIFO` | 1-99 | 先来先服务，不被低优先级抢占 | 实时音频、控制 |
| `SCHED_RR` | 1-99 | 同优先级轮转 | 需要时间片的实时任务 |
| `SCHED_DEADLINE` | — | 最早截止时间优先 | 严格时限的硬实时任务 |

## 7 综合性能分析工具

### 7.1 perf：内核级性能分析瑞士军刀

perf是Linux内核自带的性能分析工具，直接访问内核的性能计数器（PMC）和tracepoint：

```bash
# 1. 系统级性能概览（最常用）
sudo perf stat -a -- sleep 10
# 输出包含：CPU周期、指令数、缓存命中率、上下文切换、页错误等

# 2. 热点函数分析（perf top / perf record + perf report）
sudo perf record -g -a -- sleep 30
sudo perf report --stdio
# 显示每个函数的CPU时间占比，快速定位热点

# 3. 火焰图生成（可视化热点）
sudo perf record -g -a -- sleep 30
sudo perf script | stackcollapse-perf.pl | flamegraph.pl > flame.svg

# 4. 调度延迟分析
sudo perf sched record -a -- sleep 10
sudo perf sched latency
sudo perf sched map  # 时间线可视化

# 5. 特定事件追踪
sudo perf trace -e open,read,write -p $(pgrep nginx)
# 等效于strace但开销更低
```

### 7.2 ftrace：内核函数追踪

ftrace是内核内置的函数级追踪框架，开销极低，适合生产环境：

```bash
# 1. 函数追踪
echo function > /sys/kernel/debug/tracing/current_tracer
echo 1 > /sys/kernel/debug/tracing/tracing_on
cat /sys/kernel/debug/tracing/trace_pipe
# ... 实时输出函数调用 ...

# 2. 函数调用图
echo function_graph > /sys/kernel/debug/tracing/current_tracer
echo 1 > /sys/kernel/debug/tracing/tracing_on
cat /sys/kernel/debug/tracing/trace_pipe

# 3. 延迟追踪（追踪从唤醒到运行的调度延迟）
echo 1 > /sys/kernel/debug/tracing/events/sched/sched_wakeup/enable
echo 1 > /sys/kernel/debug/tracing/events/sched/sched_switch/enable
echo 1 > /sys/kernel/debug/tracing/tracing_on
cat /sys/kernel/debug/tracing/trace_pipe
```

### 7.3 bpftrace/eBPF：可编程动态追踪

eBPF允许在内核中安全地运行自定义追踪逻辑，是现代内核性能分析的利器：

```bash
# 1. 追踪系统调用延迟分布
sudo bpftrace -e '
tracepoint:raw_syscalls:sys_enter { @start[tid] = nsecs; }
tracepoint:raw_syscalls:sys_exit /@start[tid]/ {
    @us = hist((nsecs - @start[tid]) / 1000);
    delete(@start[tid]);
}
interval:s:10 { exit(); }
'

# 2. 追踪I/O延迟分布
sudo bpftrace -e '
tracepoint:block:block_rq_issue { @start[args->dev, args->sector] = nsecs; }
tracepoint:block:block_rq_complete /@start[args->dev, args->sector]/ {
    @io_latency_us = hist((nsecs - @start[args->dev, args->sector]) / 1000);
    delete(@start[args->dev, args->sector]);
}
interval:s:10 { exit(); }
'

# 3. 追踪TCP重传
sudo bpftrace -e '
kprobe:tcp_retransmit_skb {
    printf("retransmit: %s -> %s\n",
        comm,
        ntop(((struct sock *)arg0)->__sk_common.skc_daddr));
}
'

# 4. BCC工具集（预写的eBPF工具）
/usr/share/bcc/tools/execsnoop    # 追踪新进程创建
/usr/share/bcc/tools/biolatency   # I/O延迟直方图
/usr/share/bcc/tools/cachestat    # 页缓存命中率
/usr/share/bcc/tools/tcplife      # TCP连接生命周期
```

### 7.4 /proc和/sys：运行时内核参数

`/proc`和`/sys`是内核暴露运行时状态和可调参数的接口：

```bash
# 内存相关参数
cat /proc/sys/vm/swappiness              # swap倾向（0-100，默认60）
cat /proc/sys/vm/dirty_ratio             # 脏页比例阈值
cat /proc/sys/vm/overcommit_memory       # 内存分配策略（0/1/2）
cat /proc/sys/vm/min_free_kbytes         # 最低空闲内存保障

# 网络相关参数
cat /proc/sys/net/core/somaxconn         # listen backlog最大值
cat /proc/sys/net/ipv4/tcp_max_syn_backlog  # SYN队列最大值
cat /proc/sys/net/core/netdev_max_backlog    # 设备接收队列最大值
cat /proc/sys/net/ipv4/tcp_tw_reuse      # TIME_WAIT复用策略

# 调度相关参数
cat /proc/sys/kernel/sched_min_granularity_ns  # CFS最小调度粒度
cat /proc/sys/kernel/sched_latency_ns         # CFS调度延迟目标
cat /proc/sys/kernel/sched_wakeup_granularity_ns  # 唤醒抢占粒度
```

## 8 性能基线建立与监控实践

### 8.1 建立性能基线

在优化之前，必须先建立可量化的性能基线：

```bash
#!/bin/bash
# perf_baseline.sh - 建立系统性能基线
REPORT_FILE="/tmp/perf_baseline_$(date +%Y%m%d_%H%M%S).txt"

echo "=== 性能基线报告 $(date) ===" > $REPORT_FILE

echo "" >> $REPORT_FILE
echo "### CPU利用率（10秒采样）" >> $REPORT_FILE
mpstat 1 10 >> $REPORT_FILE

echo "" >> $REPORT_FILE
echo "### 内存状态" >> $REPORT_FILE
free -h >> $REPORT_FILE
grep -E "Cached|Buffers|Dirty|Writeback|Slab|SReclaimable" /proc/meminfo >> $REPORT_FILE

echo "" >> $REPORT_FILE
echo "### I/O性能（10秒采样）" >> $REPORT_FILE
iostat -dx 1 10 >> $REPORT_FILE

echo "" >> $REPORT_FILE
echo "### 网络统计" >> $REPORT_FILE
sar -n DEV 1 5 >> $REPORT_FILE

echo "" >> $REPORT_FILE
echo "### 上下文切换" >> $REPORT_FILE
vmstat 1 5 >> $REPORT_FILE

echo "" >> $REPORT_FILE
echo "### 调度统计" >> $REPORT_FILE
cat /proc/schedstat >> $REPORT_FILE

echo "" >> $REPORT_FILE
echo "### Slab使用（Top 10）" >> $REPORT_FILE
slabtop --once --sort=c -o | head -25 >> $REPORT_FILE

echo "" >> $REPORT_FILE
echo "### TCP连接状态" >> $REPORT_FILE
ss -s >> $REPORT_FILE

echo "报告已保存到: $REPORT_FILE"
```

### 8.2 性能瓶颈定位流程

遇到性能问题时，按以下流程定位瓶颈：

性能问题报告
  │
  ├── 1. 快速概览
  │   ├── top/htop → CPU利用率是否饱和？哪个进程消耗最多？
  │   ├── free → 内存是否充足？swap是否在使用？
  │   ├── iostat → I/O是否饱和？延迟是否异常？
  │   └── ss -s → TCP连接数是否过多？
  │
  ├── 2. 细分定位
  │   ├── CPU高 → perf top → 热点函数在哪里？
  │   ├── 内存高 → slabtop → 哪个slab对象增长异常？
  │   ├── I/O高 → iostat -x → 读还是写？哪个设备？
  │   └── 网络慢 → ss -ti → 重传率？拥塞窗口？
  │
  ├── 3. 深入内核路径
  │   ├── ftrace → 追踪具体函数调用链
  │   ├── bpftrace → 编写自定义追踪逻辑
  │   └── perf record → 生成火焰图分析热点
  │
  └── 4. 验证与优化
      ├── 修改参数/代码
      ├── 重新测量基线
      └── 对比优化前后数据

### 8.3 常见性能问题与内核指标关联

| 现象 | 关键指标 | 可能的内核原因 | 排查工具 |
|------|---------|--------------|---------|
| 应用响应慢 | 调度延迟高、上下文切换多 | CPU竞争激烈、cgroup配额不足 | `perf sched latency` |
| 数据库查询慢 | %iowait高、I/O延迟高 | 页缓存不足、I/O调度不当 | `iostat -dx`、`cachestat` |
| 网络吞吐低 | 丢包率高、TCP重传多 | 网卡中断不均、ring buffer太小 | `sar -n DEV`、`ethtool -S` |
| OOM频繁 | 物理内存不足、slab增长 | 内核内存泄漏、缺少内存限制 | `slabtop`、`/proc/meminfo` |
| 系统卡顿 | %sys高、spinlock争用 | 内核锁竞争、中断处理过长 | `perf lock`、`perf top` |
| NUMA性能差 | 远端内存访问多 | 内存分配跨NUMA节点 | `numastat`、`numactl` |

## 9 误区与注意事项

### 误区一：只看CPU利用率判断性能

**纠正**：CPU利用率低于100%不代表没有性能问题。需要综合考虑：
- 进程是否在等待I/O（%iowait）
- 是否在等待锁（perf top中的spinlock热点）
- 是否被cgroup限制（`nr_throttled`非零）

### 误区二：/proc/stat的数据是精确的

**纠正**：/proc/stat的CPU时间统计基于内核tick中断，精度取决于HZ配置（通常250或1000）。对于短时间（<100ms）的性能事件，应使用perf的PMC（Performance Monitoring Counter）获取纳秒级精度。

### 误区三：%util高说明I/O是瓶颈

**纠正**：对于NVMe SSD，由于硬件支持大量并发队列，`%util`可能在实际吞吐量远未饱和时就显示100%。正确的做法是同时观察`await`（延迟）和吞吐量是否达到设备标称值。

### 误区四：优化单个指标就能提升整体性能

**纠正**：内核各子系统紧密耦合。例如，降低swappiness可能减少swap但增加OOM风险；增加网络ring buffer可能减少丢包但增加内存压力。优化时必须全局观察、权衡。

### 误区五：工具本身的开销可以忽略

**纠正**：某些监控工具本身引入显著开销。例如，持续运行`perf record -g`会使CPU利用率增加5-15%；高频率的vmstat轮询（<1秒间隔）在高负载系统上可能影响结果。生产环境建议：ftrace在需要时短暂开启；perf采样率根据场景调整（CPU采样1000-10000 Hz，I/O采样更低）。

## 10 本节小结

Linux内核性能评估需要一套系统化的指标体系，覆盖CPU、内存、I/O、网络、调度五个维度：

| 维度 | 核心指标 | 关键工具 | 优化杠杆 |
|------|---------|---------|---------|
| CPU | 利用率分布、上下文切换、缓存命中率 | `perf stat`、`mpstat` | 减少系统调用、优化锁、缓存友好设计 |
| 内存 | 缺页率、OOM、NUMA命中率、Slab | `vmstat`、`slabtop`、`numastat` | 增大页缓存、NUMA绑定、减少内核内存分配 |
| I/O | IOPS、延迟、队列深度、%util | `iostat`、`blktrace` | 选择I/O调度器、调整脏页策略、RAID优化 |
| 网络 | 收发速率、丢包率、TCP重传 | `sar`、`ss`、`ethtool` | 中断均衡、ring buffer、TCP调优 |
| 调度 | 调度延迟、运行队列、CFS带宽 | `perf sched`、`bpftrace` | nice值、cgroup配额、实时调度策略 |

**记住**：指标不是目的，理解指标背后的内核机制才是关键。当你看到一个高延迟指标时，不要急于"调参数"，而是追问：这个延迟发生在内核的哪个路径上？是CPU竞争、内存分配、磁盘I/O还是网络拥塞？沿着源码追踪到具体的内核函数，你才能做出精准的优化决策。
