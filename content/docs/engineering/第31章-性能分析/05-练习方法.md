# 31.5 练习方法

理论和技巧只有通过亲手实践才能内化为能力。本节提供三档递进练习（基础→进阶→高级），以及持续学习路径和推荐资源，帮助你系统性地掌握性能分析技能。

每个练习都设计了明确的目标、具体的步骤、可验证的检查标准和预估时间。建议严格按照顺序完成——前一个练习的产出往往是后一个练习的输入。

```mermaid
graph LR
    A["基础练习<br/>USE方法实践<br/>60秒系统体检"] --> B["进阶练习<br/>火焰图分析<br/>内存泄漏排查"]
    B --> C["高级练习<br/>端到端优化项目<br/>性能基线→瓶颈→优化→验证"]
    C --> D["持续学习路径<br/>推荐资源·社区·认证"]
```

## 基础练习（入门级，预计2-3小时）

基础练习的目标是建立"系统性检查"的肌肉记忆。不需要深厚的操作系统知识，只需要一台Linux机器和动手的意愿。

### 练习一：60秒系统体检（预计30分钟）

**目标**：使用Brendan Gregg的60秒检查清单，在1分钟内完成系统健康状态的快速评估，准确判断瓶颈所在资源。

**环境准备**：

```bash
# 确保基本工具已安装
# Ubuntu/Debian
sudo apt-get update &amp;&amp; sudo apt-get install -y sysstat linux-tools-common linux-tools-$(uname -r)

# CentOS/RHEL
sudo yum install -y sysstat

# 验证工具可用
which mpstat iostat vmstat sar ss
mpstat --version 2>&amp;1 | head -1
```

**步骤一：按顺序执行60秒清单（10分钟）**

逐条执行Brendan Gregg的经典检查清单。每条命令执行后，花10秒记录你观察到的关键数字：

```bash
# === 60秒系统体检清单 ===

# 1. 系统负载与运行时间（1秒）
uptime
# 记录：load average的三个值
# 判断：如果第1个值 > CPU核数，说明系统在排队

# 2. 内核错误消息（1秒）
dmesg -T | tail -20
# 记录：是否有OOM killer、硬件错误、IO错误
# 期望：应该为空或只有历史记录

# 3. 统计级概览（1秒）
vmstat 1 3
# 记录：r列（运行队列）、b列（阻塞进程）、si/so（swap IO）
# 关键判断：r列持续 > CPU核数 → CPU饱和

# 4. CPU时间分解（1秒）
mpstat -P ALL 1 3
# 记录：us（用户态）、sy（内核态）、wa（IO等待）、id（空闲）
# 关键判断：wa > 20% → IO是瓶颈

# 5. 内存使用（1秒）
free -h
# 记录：available列的值
# 关键判断：available < 总内存20% → 内存紧张

# 6. 磁盘IO（1秒）
iostat -xz 1 3
# 记录：%util、await、avgqu-sz
# 关键判断：%util接近100%且await > 10ms → IO瓶颈

# 7. 网络概览（1秒）
ss -s
# 记录：TCP连接状态分布
# 关键判断：TIME_WAIT大量积压 → 连接管理问题

# 8. 进程概览（1秒）
top -bn1 | head -20
# 记录：CPU和内存消耗最高的前3个进程
```

**步骤二：填写体检报告（10分钟）**

根据体检结果，填写以下模板：

### 练习二：USE方法逐资源排查（预计45分钟）

**目标**：对CPU、内存、磁盘、网络四种资源，分别执行USE三维度检查，掌握每种资源对应的诊断命令。

**步骤一：CPU的USE检查（10分钟）**

```bash
# === U（利用率）===
mpstat -P ALL 1 5
# 观察：每个核心的us + sy百分比
# 思考：us和sy的比例说明什么？us高意味着用户代码密集，sy高意味着系统调用密集

# === S（饱和度）===
vmstat 1 5
# 观察：r列（运行队列长度）
# 判断：r列持续 > CPU核数 = CPU饱和
# 也可以看 /proc/loadavg 或 top的load average

# === E（错误）===
# 硬件级错误需要perf stat
sudo perf stat -e cache-misses,branch-misses,dTLB-load-misses -a sleep 5
# 观察：各类硬件错误的绝对数值和百分比
# cache-miss-rate > 10% → 缓存友好性差
# branch-miss-rate > 5% → 分支预测不友好
```

**步骤二：内存的USE检查（10分钟）**

```bash
# === U（利用率）===
free -h
# 利用率 = used / total
# 但更关键的是 available（包含可回收的buffer/cache）

# === S（饱和度）===
# 观察swap活动
cat /proc/vmstat | grep -E "pswpin|pswpout"
# 记录数值
sleep 10
cat /proc/vmstat | grep -E "pswpin|pswpout"
# 两次差值 > 0 → 内存饱和，正在使用swap

# 也可以用 vmstat 1 观察 si/so 列

# === E（错误）===
dmesg -T | grep -i "oom\|out of memory\|killed process"
# OOM killer触发是内存错误的极端表现
```

**步骤三：磁盘IO的USE检查（10分钟）**

```bash
# === U（利用率）===
iostat -xz 1 5
# 观察：每个磁盘的 %util
# 注意：%util 对SSD可能不准确（SSD内部并行度高），对HDD更可靠

# === S（饱和度）===
# 同样用 iostat -xz 1 5
# 观察：avgqu-sz（平均队列长度）
# > 1 表示有IO请求在排队
# > 4 表示严重排队

# === E（错误）===
# 查看SMART错误
sudo smartctl -a /dev/sda 2>/dev/null | grep -E "Reallocated|Pending|Uncorrectable" || echo "smartctl不可用"
# 查看内核IO错误
dmesg -T | grep -i "io error\|buffer I/O\|ext4 error" | tail -5
```

**步骤四：网络的USE检查（10分钟）**

```bash
# === U（利用率）===
sar -n DEV 1 5 2>/dev/null || ifconfig eth0
# 观察：rxkB/s、txkB/s 与网卡总带宽的比值
# 1Gbps ≈ 125MB/s，10Gbps ≈ 1250MB/s

# === S（饱和度）===
ip -s link show $(ip route show default | awk '/default/ {print $5}' | head -1) 2>/dev/null || ss -s
# 观察：overruns（接收缓冲区溢出）、dropped（丢包）
# overruns > 0 → 网卡接收缓冲区满，网络饱和

# === E（错误）===
netstat -s 2>/dev/null | grep -i "error\|fail\|retransmit" || ss -s
# 观察：重传、校验和错误等
```

**步骤五：汇总USE检查结果（5分钟）**

将四种资源的USE检查结果汇总到一张表中：

```text
===== USE检查汇总表 =====

| 资源   | 利用率(U)          | 饱和度(S)           | 错误(E)         |
|--------|--------------------|--------------------|-----------------|
| CPU    | us:___% sy:___%   | runq:___           | cache:___%      |
| 内存   | used:___GB/___GB  | swap活动:□有 □无   | OOM:□有 □无     |
| 磁盘   | %util:___%        | avgqu-sz:___       | IO错误:□有 □无  |
| 网络   | 带宽:___%         | overruns:___       | retrans:___     |

异常项分析：
1. ___________________________________
2. ___________________________________
```

**检查标准**：

- [ ] 能独立完成四种资源的USE三维度检查
- [ ] 能正确解读每个命令的关键输出字段
- [ ] 能发现至少一个异常指标并给出初步判断
- [ ] USE汇总表填写完整，分析合理

### 练习三：RED方法服务级监控（预计30分钟）

**目标**：使用RED方法（Rate/Errors/Duration）对一个HTTP服务进行请求级别的性能评估。

**步骤一：准备测试环境（5分钟）**

```bash
# 启动一个简单的HTTP服务进行测试
# 方法A：用Python快速启动（如果没有Python HTTP服务）
python3 -c "
import http.server, socketserver, time, random, threading

class SlowHandler(http.server.BaseHTTPRequestHandler):
    def do_GET(self):
        # 模拟不同路径的延迟差异
        if '/api/fast' in self.path:
            time.sleep(0.01)  # 10ms
        elif '/api/slow' in self.path:
            time.sleep(0.1)   # 100ms
            if random.random() < 0.1:
                time.sleep(0.5)  # 10%概率出现500ms慢请求
                self.send_response(500)
                self.end_headers()
                self.wfile.write(b'Internal Server Error')
                return
        else:
            time.sleep(0.02)  # 20ms默认
        
        self.send_response(200)
        self.end_headers()
        self.wfile.write(b'OK')
    
    def log_message(self, format, *args):
        pass  # 关闭默认日志减少干扰

socketserver.TCPServer(('127.0.0.1', 8899), SlowHandler).serve_forever()
" &amp;
SERVER_PID=$!
sleep 1
echo "服务器已启动 PID=$SERVER_PID"
```

**步骤二：测量Rate、Errors、Duration（10分钟）**

```bash
# === Rate（请求速率）+ Duration（延迟分布）===
# 手动采集100个请求的延迟数据
echo "开始采集RED指标..."
for i in $(seq 1 100); do
  curl -s -o /dev/null -w "%{http_code} %{time_total}\n" http://127.0.0.1:8899/api/slow
done > /tmp/red_raw.txt

# 分析结果
echo "=== RED分析结果 ==="

# Rate = 总请求数 / 总耗时
TOTAL_TIME=$(awk '{sum+=$2} END{printf "%.2f", sum}' /tmp/red_raw.txt)
echo "请求速率(Rate): $(echo "scale=1; 100/$TOTAL_TIME" | bc) req/s"

# Errors
TOTAL_ERR=$(awk '$1 != 200 {count++} END{print count+0}' /tmp/red_raw.txt)
echo "错误数(Errors): $TOTAL_ERR / 100 ($((TOTAL_ERR))%)"

# Duration（延迟分布）
echo "延迟分布(Duration):"
sort -k2 -n /tmp/red_raw.txt | awk '
{
    vals[NR] = $2;
    sum += $2;
    if ($1 != 200) err_count++;
}
END {
    printf "  平均延迟: %.3f s\n", sum/NR
    printf "  P50: %.3f s\n", vals[int(NR*0.5)]
    printf "  P90: %.3f s\n", vals[int(NR*0.9)]
    printf "  P99: %.3f s\n", vals[int(NR*0.99)]
    printf "  最大: %.3f s\n", vals[NR]
}'
```

**步骤三：识别慢请求的特征（10分钟）**

```bash
# 分析不同HTTP状态码的延迟分布
echo "=== 按状态码分组的延迟分析 ==="
awk '{
    code=$1; time=$2;
    count[code]++;
    sum[code]+=time;
    if(!(code in max_t) || time>max_t[code]) max_t[code]=time;
    if(!(code in min_t) || time<min_t[code]) min_t[code]=time;
}
END {
    for(c in count) {
        printf "HTTP %s: 请求=%d次 平均=%.3fs 最小=%.3fs 最大=%.3fs\n", 
               c, count[c], sum[c]/count[c], min_t[c], max_t[c]
    }
}' /tmp/red_raw.txt | sort

# 健康服务的RED指标参考：
# Rate:   根据业务量级判断，波动<20%为正常
# Errors: < 0.1% 为优秀，0.1%-1% 需关注，> 1% 需排查
# Duration: P99应控制在SLA要求内，P99/P50比值 < 5为健康
```

**清理环境**：

```bash

rm -f /tmp/red_raw.txt
```

- [ ] 能识别出慢请求的特征（哪些路径慢、什么状态下慢）
- [ ] 能对比健康标准判断服务状态

### 练习四：火焰图分析实战（预计60分钟）

**目标**：使用perf采集CPU数据并生成火焰图，从火焰图中准确定位CPU热点函数，理解火焰图的阅读方法。

**环境准备**：

```bash
# 安装perf（如果尚未安装）
sudo apt-get install -y linux-tools-common linux-tools-$(uname -r)

# 安装FlameGraph工具集
cd /tmp
git clone --depth 1 https://github.com/brendangregg/FlameGraph.git 2>/dev/null
export PATH=$PATH:/tmp/FlameGraph
echo "FlameGraph安装完成: $(ls /tmp/FlameGraph/flamegraph.pl)"
```

**步骤一：准备一个有明显热点的测试程序（5分钟）**

```python
#!/usr/bin/env python3
"""flame_target.py - 一个有明确CPU热点的测试程序"""
import hashlib
import json
import os

def cpu_intensive_hash(data, iterations=10000):
    """CPU密集型：大量哈希计算"""
    result = data
    for _ in range(iterations):
        result = hashlib.sha256(result.encode()).hexdigest()
    return result

def cpu_intensive_sort(n=50000):
    """CPU密集型：大数组排序"""
    import random
    data = [random.random() for _ in range(n)]
    return sorted(data)

def io_simulation():
    """IO模拟：写临时文件"""
    path = "/tmp/flame_io_target.bin"
    with open(path, "wb") as f:
        f.write(os.urandom(1024 * 100))  # 写100KB
    with open(path, "rb") as f:
        f.read()
    os.unlink(path)

def main():
    """主循环：持续执行不同类型的负载"""
    count = 0
    while count < 100:
        if count % 3 == 0:
            cpu_intensive_hash("hello world")
        elif count % 3 == 1:
            cpu_intensive_sort()
        else:
            io_simulation()
        count += 1

if __name__ == "__main__":
    main()
```

**步骤二：使用perf采集CPU数据（10分钟）**

```bash
# 启动目标程序
python3 /tmp/flame_target.py &amp;
TARGET_PID=$!
sleep 2

# 使用perf记录CPU采样数据
# -F 999: 采样频率999Hz（开发环境可用较高频率）
# -g: 记录调用栈
# --call-graph dwarf: 使用DWARF调试信息记录调用栈（Python需要）
sudo perf record -p $TARGET_PID -F 999 -g --call-graph dwarf -o /tmp/perf.data -- sleep 10

echo "perf数据采集完成: $(ls -lh /tmp/perf.data)"

# 生成文本报告（查看top热点）
sudo perf report -i /tmp/perf.data --stdio --no-children 2>/dev/null | head -30

# 停止目标程序
kill $TARGET_PID 2>/dev/null
```

**步骤三：生成火焰图（5分钟）**

```bash
# 从perf数据生成折叠栈

# 检查perf script输出
echo "perf script行数: $(wc -l < /tmp/perf_script.txt)"

# 折叠调用栈
cat /tmp/perf_script.txt | stackcollapse-perf.pl > /tmp/perf.folded

# 生成SVG火焰图
```

**步骤五：生成perf stat硬件统计（5分钟）**

火焰图告诉你"哪个函数慢"，perf stat告诉你"为什么慢"：

```bash
# 对测试程序进行硬件级分析
sudo perf stat -d python3 /tmp/flame_target.py 2>&amp;1

# 重点关注：
# - IPC（insns per cycle）：> 1.0 优秀，< 0.5 说明经常停顿
# - cache-misses / cache-loads：缓存未命中率
# - branch-misses / branches：分支预测失败率
```

### 练习五：内存泄漏检测与分析（预计45分钟）

**目标**：使用多种方法检测内存泄漏，理解不同检测工具的适用场景。

**步骤一：编写一个有内存泄漏的程序（5分钟）**

```bash
cat > /tmp/memleak_target.c << 'CEOF'
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>

// 模拟内存泄漏：每次循环分配但不释放
void leak_memory() {
    static int alloc_count = 0;
    char *ptr = malloc(1024 * 100);  // 每次泄漏100KB
    if (ptr) {
        memset(ptr, 'A', 1024 * 100);
        alloc_count++;
        fprintf(stderr, "已分配 %d 次，当前泄漏约 %d KB\n", 
                alloc_count, alloc_count * 100);
    }
}

// 正常的内存使用（分配后释放）
void normal_usage() {
    char *ptr = malloc(1024);
    if (ptr) {
        memset(ptr, 'B', 1024);
        free(ptr);  // 正确释放
    }
}

int main() {
    int i = 0;
    while (i < 50) {  // 运行50轮
        leak_memory();     // 泄漏
        normal_usage();    // 正常使用
        i++;
        usleep(200000);    // 每200ms一轮
    }
    return 0;
}
CEOF

# 编译（带调试信息）
gcc -g -o /tmp/memleak_target /tmp/memleak_target.c
echo "内存泄漏测试程序编译完成"
```

**步骤二：运行时监控内存增长（10分钟）**
