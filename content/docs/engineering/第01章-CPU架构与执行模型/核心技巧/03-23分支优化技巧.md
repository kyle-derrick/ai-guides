## 2.3 分支优化技巧

现代 CPU 采用深度流水线设计（通常 14–22 级），**分支指令是流水线最大的威胁之一**。当 CPU 遇到一条条件跳转指令时，它必须预测跳转是否成立，以便提前取指和执行。一旦预测失败，整条流水线需要清空重来，浪费 10–20 个时钟周期。分支优化的本质目标就是：**减少分支数量、提高分支预测准确率、或者将条件分支转化为无分支的算术运算**。

本节从三个层次展开：理解分支预测硬件机制 → 掌握消除分支的编程技巧 → 在排序、查表等常见场景中灵活运用。

---

### 2.3.1 分支预测的硬件机制

要有效优化分支，首先必须理解 CPU 分支预测器的工作原理。

#### 2.3.1.1 分支预测器的演进

| 时代 | 预测器类型 | 核心思路 | 典型误预测率 |
|------|-----------|---------|-------------|
| 1990s | 1-bit 饱和计数器 | 上次跳就预测跳 | 高，对循环不友好 |
| 2000s | 2-bit 饱和计数器 | 连续两次才翻转，容忍循环尾部一次不跳 | 中等 |
| 2010s | TAGE（TAgged GEometric） | 基于历史长度的多级表，记录不同长度的跳转模式 | <5% |
| 2020s | 感知器预测器 | 用线性分类器结合多条历史路径 | <3% |
| 当代 | 混合预测器 | 多种预测器投票，取置信度最高的 | <2% |

现代 Intel/AMD 处理器使用混合预测器，结合全局历史、局部历史和分支地址特征，在大多数场景下误预测率低于 2%。但以下几种情况会让预测器"抓瞎"：

#### 2.3.1.2 让预测器失效的场景

**随机数据分支**：当分支条件取决于随机数时，任何预测器的准确率都会逼近 50%，与抛硬币无异。这是最致命的性能杀手。

**跨步不规律的数据访问**：即使数据本身不是随机的，如果访问顺序被打乱（如遍历链表而非数组），也会破坏 CPU 的预取和分支历史关联。

**间接跳转**：`switch` 语句、虚函数调用、函数指针等间接跳转的目标地址在运行时才确定，预测难度远高于条件分支。

**长依赖链中的条件判断**：当分支条件依赖于前面多条指令的计算结果时，预测器来不及在取指阶段完成预测。

### 2.3.2 用位运算消除分支

最直接的优化策略：**把 `if-else` 替换为无分支的算术或位运算**。现代编译器在很多场景下会自动完成这种转换，但理解原理有助于写出编译器难以自动优化的代码，并能审阅编译器的决策。

#### 2.3.2.1 经典无分支 max/min

```c
// ============ 有分支版本 ============
int max_branchy(int a, int b) {
    if (a > b) return a;    // 分支预测失败 → 流水线冲刷
    return b;
}

// ============ 无分支版本（位运算） ============
int max_branchless(int a, int b) {
    int diff = a - b;
    int mask = diff >> 31;  // 算术右移：a>=b 时 mask=0x00000000，a<b 时 mask=0xFFFFFFFF
    return a - (diff &amp; mask);
    // a >= b: return a - 0     = a  ✓
    // a <  b: return a - (a-b) = b  ✓
}

// ============ 无分支版本（CMOV 指令） ============
// 现代编译器通常会将三目运算符编译为 CMOV
int max_cmov(int a, int b) {
    return a > b ? a : b;  // GCC/Clang -O2 下生成 CMOV，无分支
}
```

**原理拆解**：

- `diff >> 31`：对 32 位整数做算术右移。若 `a - b >= 0`（即 `a >= b`），最高位为 0，右移后全 0；若 `a - b < 0`（即 `a < b`），最高位为 1，算术右移填充 1，得到 `0xFFFFFFFF`（即 -1 的补码）。
- `diff & mask`：当 `a >= b` 时结果为 0，当 `a < b` 时结果为 `a - b`。
- `a - (diff & mask)`：`a >= b` 时返回 `a`，`a < b` 时返回 `a - (a - b) = b`。

> **编译器自动优化提示**：在 GCC/Clang `-O2` 及以上，`return a > b ? a : b;` 通常直接编译为 `CMP` + `CMOV` 指令序列，完全消除分支。手动写位运算在现代编译器下往往不是更优解，但理解这一机制对于审查汇编输出和在受限环境（嵌入式、无分支要求的内核代码）中编程至关重要。

#### 2.3.2.2 无分支绝对值与 clamp

```c
// 绝对值（无分支）
int abs_branchless(int x) {
    int mask = x >> 31;
    return (x ^ mask) - mask;
    // x >= 0: mask=0,     return x ^ 0 - 0     = x  ✓
    // x <  0: mask=-1,    return x ^ (-1) - (-1) = ~x + 1 = |x|  ✓（补码取反加1即取绝对值）
}

// 将值限制在 [lo, hi] 范围内（无分支 clamp）
int clamp_branchless(int x, int lo, int hi) {
    // 先 clamp 下界，再 clamp 上界
    int t = x < lo ? lo : x;   // 编译器可能生成 CMOV
    return t > hi ? hi : t;
}

// 全无分支 clamp（纯算术）
int clamp_arithmetic(int x, int lo, int hi) {
    x = lo + (x - lo) &amp; ~((x - lo) >> 31);          // max(x, lo)
    x = hi - (hi - x) &amp; ~((hi - x) >> 31);          // min(x, hi)
    return x;
}
```

#### 2.3.2.3 无分支条件赋值的通用模式

```c
// 模式：condition ? a : b  的无分支写法
// 假设 condition 为 0 或 1
int select_branchless(int condition, int a, int b) {
    return a + (b - a) * condition;
    // condition=0: return a
    // condition=1: return b
}

// 对于 0/1 布尔值的取反
int not_branchless(int b) {
    return 1 - b;  // 或 b ^ 1
}
```

#### 2.3.2.4 位运算消除分支的适用边界

并非所有分支都适合用位运算消除。需要权衡的因素：

| 考量因素 | 适合位运算消除 | 不适合消除 |
|---------|--------------|-----------|
| 分支体大小 | 分支体只有一两个简单运算 | 分支体内有复杂逻辑、函数调用、内存访问 |
| 分支预测率 | 预测率 < 95%，误预测开销大 | 预测率 > 98%，硬件预测已经很准 |
| 数据并行性 | 向量化（SIMD）场景需要无分支 | 标量场景，编译器 CMOV 足够 |
| 代码可读性 | 内核代码、热路径关键函数 | 业务逻辑代码，可读性优先 |
| 二进制兼容性 | 可以接受所有路径都执行 | 某个分支有副作用（如可能除零、越界） |

> **性能验证法则**：不要凭直觉判断分支是否需要消除。用 `perf stat -e branches,branch-misses` 在实际数据上测量，如果误预测率已经低于 1%，通常不值得手动优化。

### 2.3.3 分支预测友好排序

这是一个被广泛讨论的经典案例，深刻揭示了**数据布局如何影响分支预测性能**。

#### 2.3.3.1 问题场景：随机数据的条件累加

```c
#include <stdlib.h>
#include <stdio.h>

int main() {
    const int N = 10000000;
    int* data = malloc(N * sizeof(int));

    // 填充 0~255 的随机数
    for (int i = 0; i < N; i++) {
        data[i] = rand() % 256;
    }

    // 分支累加
    long sum = 0;
    for (int i = 0; i < N; i++) {
        if (data[i] >= 128) {   // 随机数据：约50%命中
            sum += data[i];
        }
    }
    printf("sum = %ld\n", sum);
    return 0;
}
```

在随机数据上运行，分支预测准确率约为 50%。每次误预测浪费约 15 个周期（现代 Intel Skylake+），1000 万次迭代中约 500 万次误预测，额外开销约 **7500 万周期**。

#### 2.3.3.2 优化方案：排序后遍历

```c
// 排序后遍历：分支变为"先全不跳，再全跳"，预测率 > 95%
qsort(data, N, sizeof(int), compare_asc);
for (int i = 0; i < N; i++) {
    if (data[i] >= 128) {
        sum += data[i];
    }
}
```

排序后，数据变为：`[0,0,...,127,127,...,128,128,...,255,255,...]`。遍历时前半段全部不跳（稳定预测"不跳"），到 128 边界后跳一次，此后全部跳（稳定预测"跳"）。只有边界附近的 1–2 次误预测。

#### 2.3.3.3 更优方案：完全消除分支

```c
// 方案A：用三目运算（编译器生成 CMOV）
long sum = 0;
for (int i = 0; i < N; i++) {
    sum += (data[i] >= 128) ? data[i] : 0;  // 无分支
}

// 方案B：位掩码
long sum = 0;
for (int i = 0; i < N; i++) {
    int mask = -(data[i] >= 128);  // 条件为真 → 0xFFFFFFFF，假 → 0x00000000
    sum += data[i] &amp; mask;
}
```

#### 2.3.3.4 性能对比实测

以下数据来自 Intel i7-12700K（GCC 12, `-O3`），1000 万元素：

| 方案 | 耗时（ms） | 分支误预测率 | 加速比 |
|------|-----------|-------------|--------|
| 未排序 + if 分支 | ~85 | ~50% | 基准 |
| 排序后 + if 分支 | ~45 | <1% | 1.9× |
| CMOV / 三目运算 | ~12 | 0% | 7.1× |
| SIMD 向量化（AVX2） | ~3 | 0% | 28× |

> **关键洞察**：排序方案虽然有效，但排序本身的 O(N log N) 开销可能抵消收益。只有当**同一个数组会被多次遍历做条件过滤**时，排序才值得。如果只遍历一次，直接用无分支写法更优。

#### 2.3.3.5 实际工程应用：数据库过滤

在列式存储数据库（如 ClickHouse、Apache Parquet）中，类似的技术被广泛使用：

- **位图索引（Bitmap Index）**：为每列的每个取值维护一个位图，过滤操作变成位运算而非逐行分支判断。
- **Run-Length Encoding（RLE）**：排序后的列天然形成连续相同值的游程，过滤只需扫描游程边界，分支数量从 N 降到游程数。
- **向量化执行引擎**：用 SIMD 批量处理 8/16/32 个值，完全消除标量分支。

### 2.3.4 查表法替代分支

当分支条件是对某个小范围整数的分类或映射时，**查表法（Lookup Table）** 是最直接的消除分支方案。

#### 2.3.4.1 基本查表法

```c
// ============ 有分支版本 ============
int classify_branchy(int value) {
    if (value < 0)   return -1;  // 负数
    if (value == 0)  return 0;   // 零
    return 1;                     // 正数
}

// ============ 查表法 ============
// 预计算表：覆盖所有可能的输入值
static int class_table[256];  // 假设输入为 unsigned char 范围 [0, 255]

void init_class_table() {
    for (int i = 0; i < 256; i++) {
        if (i < 128)  class_table[i] = -1;  // [0, 127] → -1
        else          class_table[i] = 1;   // [128, 255] → 1
    }
    class_table[0] = 0;  // 特殊处理零
}

int classify_table(int value) {
    return class_table[(unsigned char)value];  // 一次数组访问，零分支
}
```

#### 2.3.4.2 查表法的适用条件

| 条件 | 说明 |
|------|------|
| 输入范围小且离散 | 最佳场景：8-bit（256 项）、10-bit（1024 项）。16-bit 以上（64K 项）会导致 L1 缓存未命中 |
| 计算代价高于查表 | if-else 中每个分支包含复杂计算时，查表预计算收益最大 |
| 查询频率极高 | 热路径中的每秒数百万次调用值得用空间换时间 |
| 表可以静态初始化 | 编译时已知的映射关系，表可写为 `static const` |

#### 2.3.4.3 实际案例：字符分类

C 标准库的 `<ctype.h>` 函数（`isalpha`、`isdigit`、`isspace` 等）在许多实现中就是查表法：

```c
// 伪代码：ctype.h 查表实现
static const unsigned char ctype_table[256] = {
    // 位掩码：bit0=space, bit1=digit, bit2=alpha, ...
    [0]   = CTRL,
    [' ']  = SPACE,
    ['\t'] = SPACE,
    ['0']  = DIGIT,
    ['1']  = DIGIT,
    // ... 完整映射
    ['a']  = ALPHA | LOWER,
    ['A']  = ALPHA | UPPER,
    // ...
};

int my_isdigit(int c) {
    return (ctype_table[(unsigned char)c] &amp; DIGIT) != 0;  // 无分支
}
```

#### 2.3.4.4 哈希表查表（大范围输入）

当输入范围较大但实际出现的值稀疏时，可以用哈希表：

```c
// 用完美哈希处理稀疏输入
// 场景：根据操作码（256种可能，实际使用30种）执行不同操作
typedef struct {
    int key;
    int value;
} Entry;

// 静态完美哈希表（编译时生成）
static Entry opcode_table[64];  // 64 个槽位，30 个有效条目
static const int HASH_MASK = 63;

void init_opcode_table() {
    memset(opcode_table, -1, sizeof(opcode_table));
    // 插入 opcode → action 映射
    for (int i = 0; i < num_opcodes; i++) {
        int h = opcodes[i].key &amp; HASH_MASK;
        while (opcode_table[h].key != -1) h = (h + 1) &amp; HASH_MASK;  // 线性探测
        opcode_table[h] = opcodes[i];
    }
}

int lookup_opcode(int opcode) {
    int h = opcode &amp; HASH_MASK;
    while (opcode_table[h].key != opcode &amp;&amp; opcode_table[h].key != -1) {
        h = (h + 1) &amp; HASH_MASK;
    }
    return opcode_table[h].value;  // 最多几次内存访问，无分支
}
```

#### 2.3.4.5 查表法的性能陷阱

**缓存友好性是查表法的生命线**。如果表太大导致频繁 L1/L2 缓存未命中，查表法反而比分支更慢。

```c
// 陷阱：16-bit 查表需要 64KB 表，超出 L1（通常 32KB）
// 对比：
//   L1 hit:     ~4 cycles
//   L2 hit:     ~12 cycles
//   L3 hit:     ~40 cycles
//   误预测分支:  ~15 cycles
//
// 如果查表导致 L2 miss，每次查表比误预测分支还慢！
```

**应对策略**：
- 表控制在 L1 缓存大小以内（32KB ≈ 32K 个 int 项）
- 对于大范围输入，考虑分层查表：先粗分类（少量分支），再在小范围内查表
- 将常用表项放在数组前部（利用时间和空间局部性）

### 2.3.5 CMOV 指令与编译器决策

#### 2.3.5.1 CMOV 是什么

`CMOV`（Conditional Move）是 x86 架构在 Pentium Pro (1995) 引入的条件移动指令。它在**不改变控制流**的情况下根据条件码寄存器的标志位复制操作数，是分支优化的硬件基础。

```nasm
; CMOV 示例：max(a, b)
; 寄存器: eax = a, ecx = b
    cmp  eax, ecx     ; 比较 a 和 b，设置标志位
    cmovl eax, ecx    ; 如果 a < ecx (SF≠OF)，则 eax = ecx
    ; 最终 eax = max(a, b)，无分支
```

#### 2.3.5.2 编译器何时选择 CMOV

```c
// 案例1：简单三目 → CMOV
int max(int a, int b) { return a > b ? a : b; }
// 编译器通常生成 CMOV ✓

// 案例2：分支体有副作用 → 保留分支
int safe_div(int a, int b, int fallback) {
    return (b != 0) ? a / b : fallback;
    // 除法可能异常，保留 if 更安全
    // 但编译器仍可能生成 CMOV（无副作用的算术运算）
}

// 案例3：分支体有内存访问 → 取决于优化级别
int pick(int cond, int* p, int fallback) {
    return cond ? *p : fallback;
    // *p 可能触发页错误 → 编译器保守地保留分支
}
```

#### 2.3.5.3 查看编译器是否生成 CMOV

```bash
# 查看汇编输出
gcc -O3 -S max.c -o max.s
grep cmov max.s   # 搜索 CMOV 指令

# 或用 Godbolt 在线编译器交互查看
# https://godbolt.org/
```

**注意 CMOV 的局限**：CMOV 不是万能的。当两个分支路径的执行时间差异极大（如一个轻量一个涉及内存分配）时，CMOV 会执行两条路径，浪费资源。编译器会根据代价模型决定是否使用 CMOV。

### 2.3.6 SIMD 向量化与无分支编程

SIMD（Single Instruction Multiple Data）是消除标量分支的终极手段：**当所有数据的条件判断被转化为掩码后，整批数据可以同时处理**。

#### 2.3.6.1 SIMD 分支消除思路

```c
// 传统标量：逐元素分支
long sum = 0;
for (int i = 0; i < n; i++) {
    if (data[i] >= 128)
        sum += data[i];
}

// SIMD 思路（伪代码）：
// 1. 加载 8 个 int 到向量寄存器
// 2. 与 128 比较 → 得到掩码（全 0 或全 1）
// 3. 用掩码 AND 保留需要的值，其余清零
// 4. 水平求和
// → 整个过程无分支，一次处理 8 个元素
```

#### 2.3.6.2 AVX2 实际示例

```c
#include <immintrin.h>

long sum_ge128_avx2(const int* data, int n) {
    __m256i threshold = _mm256_set1_epi32(128);
    __m256i sum_vec = _mm256_setzero_si256();

    int i = 0;
    for (; i + 7 < n; i += 8) {
        __m256i val = _mm256_loadu_si256((__m256i*)(data + i));
        __m256i mask = _mm256_cmpgt_epi32(val, threshold);  // val > 128 → 全1
        val = _mm256_and_si256(val, mask);                    // 不满足条件的清零
        sum_vec = _mm256_add_epi32(sum_vec, val);
    }

    // 水平求和
    int tmp[8];
    _mm256_storeu_si256((__m256i*)tmp, sum_vec);
    long sum = tmp[0]+tmp[1]+tmp[2]+tmp[3]+tmp[4]+tmp[5]+tmp[6]+tmp[7];

    // 处理剩余元素
    for (; i < n; i++) {
        if (data[i] >= 128) sum += data[i];
    }
    return sum;
}
```

**关键概念**：`_mm256_cmpgt_epi32` 返回的掩码不是布尔值，而是**每个位都置位的全 1 向量**（-1 的补码）。与原值做 AND 后，条件为真的保留原值，条件为假的变为 0。这就是 SIMD 世界中的"无分支条件过滤"。

### 2.3.7 实战中的分支优化策略总览

#### 2.3.7.1 优化决策树

遇到条件分支
  │
  ├─ 分支条件是否可预测（规律数据）？
  │   ├─ 是 → 保持分支，依赖硬件预测（可能已经足够快）
  │   └─ 否（随机/不可预测）→ 继续评估
  │
  ├─ 分支体是否简单（单条运算）？
  │   ├─ 是 → 用位运算/CMOV 消除
  │   └─ 否 → 继续评估
  │
  ├─ 输入范围是否小（< 1024）？
  │   ├─ 是 → 查表法
  │   └─ 否 → 继续评估
  │
  ├─ 是否处理大批量同构数据？
  │   ├─ 是 → SIMD 向量化
  │   └─ 否 → 继续评估
  │
  └─ 保持分支，专注提高预测率（数据排序、profile-guided optimization）

#### 2.3.7.2 常见误区

**误区 1：分支越多性能越差**
实际上，连续的、方向一致的分支（如循环中的 `if (i < N)`）预测率接近 100%。真正有害的是**数据依赖的、方向不确定的分支**。

**误区 2：无分支一定比有分支快**
当分支预测率 > 95% 时，有分支版本通常与 CMOV 版本性能相当甚至更快，因为 CMOV 要求两条路径都计算。编译器的代价模型会自动选择更优方案。

**误区 3：手动位运算一定优于编译器生成的代码**
现代编译器（GCC/Clang -O3）在简单的 max/min/abs 场景下会自动生成 CMOV 或无分支代码。手动位运算可能阻碍编译器的进一步优化（如自动向量化）。**先写清晰代码，再用 `-S` 检查汇编**。

**误区 4：查表法总是快的**
超过 L1 缓存容量的查表（>8K int 项）每次访问可能引发缓存未命中（12–40 cycles），反而比一个误预测分支（15 cycles）更慢。必须实测验证。

#### 2.3.7.3 性能分析工具

```bash
# 1. 查看分支预测率
perf stat -e branches,branch-misses ./your_program

# 2. 查看热点函数的分支详情
perf record -e branch-misses ./your_program
perf report

# 3. 查看特定函数的汇编
objdump -d ./your_program | grep -A 50 '<function_name>'

# 4. 使用 LLVM MCA 分析指令调度
llvm-mca -mcpu=skylake your_function.s

# 5. 用 cachegrind 查看缓存行为（查表法适用性）
valgrind --tool=cachegrind ./your_program
cg_annotate cachegrind.out.12345
```

### 2.3.8 本节小结

| 技术 | 核心原理 | 最佳适用场景 | 性能收益 |
|------|---------|-------------|---------|
| 位运算消除分支 | 用算术/位操作替代条件跳转 | 简单的 max/min/abs/clamp | 1.5–7×（取决于预测率） |
| 分支预测友好排序 | 改变数据顺序使分支方向可预测 | 多次遍历同一数组做条件过滤 | 2–3×（排序+多次遍历） |
| 查表法 | 预计算所有可能结果存入数组 | 小范围整数分类/映射 | 3–10×（取决于分支复杂度） |
| CMOV 条件移动 | 硬件支持的无条件移动 | 编译器自动选择的简单三目 | 与手写位运算相当 |
| SIMD 向量化 | 批量处理，掩码代替分支 | 大数组同构条件过滤 | 8–30×（取决于向量宽度） |

**核心原则**：分支优化是性能工程的重要组成部分，但不是全部。在动手优化之前，先用 `perf` 定位真正的瓶颈。一个误预测率 0.5% 的分支不值得花时间优化；而一个数据依赖的、预测率仅 40% 的分支，可能就是你程序慢 10 倍的根源。
