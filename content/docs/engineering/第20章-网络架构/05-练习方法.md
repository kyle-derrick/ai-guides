## 练习方法

本章内容覆盖了负载均衡、CDN、四层/七层代理、反向代理等网络架构核心主题。理论和配置读完了，但"看懂"和"会用"之间隔着一条鸿沟——只有通过动手实操才能真正内化这些知识。本节提供五组分级练习，从概念巩固到综合设计，逐步构建你的网络架构实战能力。

**练习分级说明：**

| 级别 | 练习 | 耗时 | 面向读者 | 核心目标 |
|------|------|------|----------|----------|
| L1 入门 | 练习一：核心概念梳理 | 30分钟 | 所有读者 | 夯实概念，建立知识框架 |
| L2 实操 | 练习二：Nginx 负载均衡实战 | 60分钟 | 初中级工程师 | 从零搭建可用的负载均衡 |
| L3 诊断 | 练习三：网络问题排查 | 45分钟 | 中级工程师 | 培养故障定位思维 |
| L4 调优 | 练习四：性能测试与优化 | 60分钟 | 中高级工程师 | 用数据驱动优化决策 |
| L5 架构 | 练习五：综合架构设计 | 90分钟 | 架构师/高级工程师 | 面对真实需求做架构决策 |

### 练习一：核心概念梳理（预计30分钟）

**目标**：巩固本章核心概念，能够清晰地解释负载均衡、代理层级、CDN的工作原理，并说出它们之间的协作关系。

**步骤**：

用户客户端
  → DNS（域名解析，可能返回多个IP实现最简负载均衡）
  → CDN 边缘节点（静态资源命中则直接返回）
  → L4 代理层（LVS / Nginx stream，TCP/UDP快速转发）
  → L7 代理层（Nginx http，SSL终结、URL路由、限流）
  → 后端应用服务器（多个实例，负载均衡算法选择实例）

- **场景C**：MySQL 主从集群的读写分离，应该选 L4 还是 L7 代理？为什么？（提示：SQL 协议不是 HTTP，L7 代理无法解析）

**3. 配置 Nginx 负载均衡（20分钟）**

创建自定义 Nginx 配置文件，实践三种不同的负载均衡策略：

```nginx
# /tmp/nginx-lb-demo.conf

# 工作进程数（建议设为CPU核心数）
worker_processes auto;

events {
    worker_connections 1024;
    use epoll;           # Linux 下使用 epoll 高性能事件模型
}

http {
    # ============================================
    # 策略一：加权轮询（默认策略的显式写法）
    # ============================================
    upstream weighted_rr {
        server 127.0.0.1:8081 weight=5 max_fails=3 fail_timeout=30s;
        server 127.0.0.1:8082 weight=3 max_fails=3 fail_timeout=30s;
        server 127.0.0.1:8083 weight=2 max_fails=3 fail_timeout=30s;
    }

    # ============================================
    # 策略二：最少连接
    # ============================================
    upstream least_conn_backend {
        least_conn;
        server 127.0.0.1:8081 max_fails=3 fail_timeout=30s;
        server 127.0.0.1:8082 max_fails=3 fail_timeout=30s;
        server 127.0.0.1:8083 max_fails=3 fail_timeout=30s;
    }

    # ============================================
    # 策略三：IP Hash（会话保持）
    # ============================================
    upstream ip_hash_backend {
        ip_hash;
        server 127.0.0.1:8081;
        server 127.0.0.1:8082;
        server 127.0.0.1:8083;
    }

    server {
        listen 8090;

        # 默认使用加权轮询
        location / {
            proxy_pass http://weighted_rr;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

            # 超时设置
            proxy_connect_timeout 5s;
            proxy_read_timeout 30s;

            # 故障重试：遇到错误或超时自动转发到下一个后端
            proxy_next_upstream error timeout http_502 http_503;
            proxy_next_upstream_tries 2;
        }

        # 最少连接策略
        location /least {
            proxy_pass http://least_conn_backend;
        }

        # IP Hash 策略
        location /session {
            proxy_pass http://ip_hash_backend;
        }

        # 状态监控页面
        location /nginx_status {
            stub_status on;
            allow 127.0.0.1;
            deny all;
        }
    }
}
```

```bash

# 验证配置语法

```

### 练习三：网络问题排查（预计45分钟）

**目标**：掌握网络架构常见问题的排查方法论——从"现象"到"根因"的系统化思维路径，而非盲目猜测。

**步骤**：

**1. 搭建问题复现环境（10分钟）**

```bash
# 创建一个故意有性能问题的 Nginx 配置
cat > /tmp/nginx-bad.conf << 'EOF'
worker_processes 1;       # 故意只用1个worker

events {
    worker_connections 64; # 故意设小连接数
}

http {
    upstream backend {
        server 127.0.0.1:8081;
    }

    server {
        listen 8091;
        
        # 故意缺少超时配置（使用默认值）
        location / {
            proxy_pass http://backend;
            # 缺少 proxy_set_header，后端无法获取客户端真实IP
            # 缺少 keepalive 配置，每次请求都新建连接
        }
    }
}
```

```bash
# 启动后端和有问题的 Nginx
python3 -c "
from http.server import HTTPServer, BaseHTTPRequestHandler
import time, os
class Handler(BaseHTTPRequestHandler):
    def do_GET(self):
        # 模拟后端处理时间

        self.send_response(200)
        self.end_headers()
        self.wfile.write(body.encode())
    def log_message(self, *args): pass
HTTPServer(('127.0.0.1', 8081), Handler).serve_forever()
" &amp;
BAD_PID=$!
sleep 1

nginx -c /tmp/nginx-bad.conf
```

**2. 系统化排查练习（25分钟）**

按以下排查清单逐项检查，找出配置中的所有问题：

```bash
# === 排查维度一：Nginx 进程与连接状态 ===

# 1. Nginx worker 进程数
ps aux | grep "nginx: worker"
# 问题发现：只有1个worker，无法利用多核CPU

# 2. 当前连接数

# 关注 TCP 连接总数和各状态分布

# 3. 连接状态详细分布
ss -ant | awk '{print $1}' | sort | uniq -c | sort -rn
# 关注 TIME_WAIT 数量是否过多（超过5000需要关注）

# === 排查维度二：端口与监听状态 ===

# 4. 确认 Nginx 监听端口
ss -tlnp | grep nginx
# 确认 8091 端口正常监听

# 5. 确认后端服务可达
curl -s http://127.0.0.1:8081/
# 确认后端正常响应

# === 排查维度三：性能压测暴露问题 ===

# 6. 用 curl 模拟并发请求
echo "=== 顺序请求测试 ==="

# 记录总耗时

# 7. 用更专业的工具压测（如果没有 ab/wrk，用 curl 的并发模拟）
echo "=== 并发请求测试 ==="
for i in $(seq 1 20); do
    curl -s http://127.0.0.1:8091/ > /dev/null &amp;
done
time wait
# 观察并发下的响应时间变化

# === 排查维度四：配置审查 ===

# 8. 导出当前配置进行审查
nginx -T -c /tmp/nginx-bad.conf | grep -A 20 "server {" | head -30
# 检查是否有遗漏的关键配置
```

**3. 问题汇总与修复（10分钟）**

将排查中发现的问题整理成表格，并逐一修复：

| 序号 | 问题 | 影响 | 修复方案 |
|------|------|------|----------|
| 1 | worker_processes=1 | 无法利用多核，高并发时CPU瓶颈 | 设为 auto 或匹配CPU核心数 |
| 2 | worker_connections=64 | 连接数上限过低，并发请求被拒绝 | 设为 1024 或更高 |
| 3 | 缺少 proxy_set_header | 后端无法获取客户端真实 IP | 添加 X-Real-IP 和 X-Forwarded-For |
| 4 | 缺少 keepalive 配置 | 每次请求新建 TCP 连接，延迟增加 | 添加 proxy_http_version 1.1 和 Connection "" |
| 5 | 缺少超时配置 | 使用默认超时，慢请求可能挂起 | 设置 proxy_connect_timeout 和 proxy_read_timeout |

修复后重新测试，对比修复前后的性能差异。

```bash

nginx -s stop -c /tmp/nginx-bad.conf 2>/dev/null
kill $BAD_PID 2>/dev/null
```

- [ ] 能按照系统化清单排查 Nginx 配置问题

**1. 需求理解（15分钟）**

### 进阶挑战（附加练习）

完成上述五组练习后，如果想进一步深入，可以尝试以下挑战：

**挑战A：Nginx stream 模块实战**

用 Nginx 的 `stream` 模块搭建一个 MySQL 读写分离代理——读请求分发到从库，写请求转发到主库。端口3306用于读，端口3307用于写。

```nginx
#     upstream mysql_read { ... }
#     upstream mysql_write { ... }
#     server { listen 3306; proxy_pass mysql_read; }
#     server { listen 3307; proxy_pass mysql_write; }
# }
```

**挑战B：CDN 缓存策略验证**

在 Nginx 上搭建一个自建 CDN 边缘节点，验证以下缓存行为：

- `Cache-Control: max-age=3600` 的实际缓存时间
- `proxy_cache_use_stale error timeout` 在后端故障时的行为
- `proxy_cache_lock on` 在并发回源时的效果（用 `ab -c 50` 并发请求同一未缓存URL，观察只有1次回源）
- 缓存命中率监控：解析 `X-Cache-Status` 头统计 HIT/MISS 比例

**挑战C：Keepalived 双机热备模拟**

在本地用两个 Nginx 实例模拟主备高可用架构（不需要真实 VIP，可以用端口区分），体验故障切换的过程：

- 主实例监听 8090，备用监听 8091
- 模拟主实例宕机后，流量切换到备用实例
- 思考：真实的 Keepalived + VIP 漂移机制与此有何区别？
