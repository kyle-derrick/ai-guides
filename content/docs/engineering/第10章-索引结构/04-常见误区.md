# 索引结构常见误区与避坑指南

> **本节定位**：前面的章节从理论、技巧到实战，系统讲解了索引结构的「正确姿势」。但实际工程中，开发者踩坑的概率远高于一次做对的概率。本节梳理索引设计、使用和运维中最常见的 **12 类典型误区**，每个误区都给出「错误现象→根因分析→正确做法→验证手段」的完整链路，帮助你在生产环境中少交学费。

---

## 误区总览

下表是 12 类常见误区的快速索引，方便按需查阅：

| 编号 | 误区 | 影响等级 | 典型后果 |
|:---:|------|:---:|------|
| 1 | 索引越多越好 | 🔴 高 | 写入变慢、存储膨胀 |
| 2 | 忽视最左前缀原则 | 🔴 高 | 索引失效、全表扫描 |
| 3 | 对小表盲目建索引 | 🟡 中 | 维护开销大于收益 |
| 4 | 在索引列上做函数运算 | 🔴 高 | 索引完全失效 |
| 5 | 忽略 LSM-Tree 的写放大 | 🔴 高 | 磁盘寿命缩短、写入毛刺 |
| 6 | 选错索引类型 | 🟡 中 | 查询性能远低于预期 |
| 7 | 忽视监控和慢查询分析 | 🔴 高 | 问题积累到崩溃才发现 |
| 8 | 过度调参，忽略实际负载 | 🟡 中 | 参数与场景不匹配 |
| 9 | 缺乏容错与降级机制 | 🔴 高 | 单点故障导致全局不可用 |
| 10 | 忽视索引的安全隐患 | 🔴 高 | 注入攻击、数据泄露 |

| 12 | 忽视维护窗口和碎片整理 | 🟡 中 | 性能缓慢退化 |

## 误区一：索引越多越好 —— 「堆索引」综合征

### 错误现象

初学者为了追求查询速度，给每个字段都建上索引，甚至在低基数字段（如性别 `gender`、状态 `status`）上也创建 B+Tree 索引。结果发现数据库写入性能急剧下降，磁盘空间快速膨胀。

### 根因分析

每个索引都是一棵独立的 B+Tree（InnoDB 聚簇索引除外）。这意味着：

1. **写放大**：每次 INSERT/UPDATE/DELETE 不仅要修改聚簇索引，还要修改所有二级索引。一个表有 10 个索引，一次写操作可能触发 10 次 B+Tree 节点分裂。
2. **存储开销**：每个二级索引叶子节点存储的是（索引列值 + 主键值）。假设主键是 BIGINT（8 字节），一个 4 字节的索引列加上主键就是 12 字节。如果表有 1 亿行，单个索引就占用约 1.2 GB。
3. **维护成本**：MySQL 在查询优化时需要评估每个索引的可用性，索引越多优化器的决策时间越长。

索引数量与写入性能的关系（示意）：

写入 QPS
  ↑
  │  ████
  │  ████████
  │  ████████████
  │  ████████████████
  │  ████████████████████
  │  ████████████████████████
  │  ████████████████████████████
  └──────────────────────────────→ 索引数量
  1    2    3    5    8   12   15
        ↑
     性能拐点（通常在 3-5 个索引后显著下降）

### 正确做法

**按需建索引，而非按字段建索引。** 具体执行步骤：

1. **收集慢查询日志**：开启 `slow_query_log`，收集 1-2 周的生产流量
2. **用 EXPLAIN 分析**：对每个慢查询执行 `EXPLAIN`，确认是否缺少合适的索引
3. **优先创建复合索引**：一个设计良好的复合索引往往能替代 2-3 个单列索引
4. **定期清理僵尸索引**：使用 `sys.schema_unused_indexes` 视图找出从未被使用的索引

```sql
-- 查找未使用的索引（MySQL 8.0+）
SELECT
    object_schema,
    object_name,
    index_name
FROM performance_schema.table_io_waits_summary_by_index_usage
WHERE index_name IS NOT NULL
  AND count_star = 0
  AND object_schema NOT IN ('mysql', 'performance_schema', 'sys')
ORDER BY object_schema, object_name;

-- 查找冗余索引
SELECT * FROM sys.schema_redundant_indexes;
```

**经验法则**：单表索引数量一般不超过 5-6 个。如果你发现需要 8 个以上索引，大概率是数据模型设计出了问题，应该考虑反范式化或分区策略。

## 误区二：忽视最左前缀原则 —— 复合索引的隐形陷阱

### 错误现象

创建了复合索引 `idx_user_time(user_id, created_at)`，但以下查询却走不了索引：

```sql
-- 这个查询无法利用 idx_user_time
SELECT * FROM orders WHERE created_at > '2025-01-01';
```

开发者困惑：明明 `created_at` 在索引里，为什么不用？

### 根因分析

B+Tree 复合索引的排序规则是**先按第一个字段排序，第一个字段相同时再按第二个字段排序**。因此，索引的物理顺序是：

索引结构示意（idx_user_time）：

(1001, 2025-01-01)
(1001, 2025-06-15)
(1002, 2025-03-20)
(1003, 2025-01-10)
(1003, 2025-07-01)
(1004, 2025-02-28)
...

→ 只有先确定 user_id，才能利用 created_at 的有序性
→ 跳过 user_id 直接查 created_at，整棵树对 created_at 来说是无序的

### 正确做法

遵循**最左前缀匹配原则**，调整索引设计或查询语句：

| 查询 | 能否用索引 | 说明 |
|------|:---:|------|
| `WHERE user_id = 1001` | ✅ | 匹配最左列 |
| `WHERE user_id = 1001 AND created_at > '2025-01-01'` | ✅ | 完全匹配两列 |
| `WHERE user_id = 1001 AND created_at > '2025-01-01' ORDER BY amount` | ⚠️ 部分 | order by 部分无法利用索引排序 |
| `WHERE created_at > '2025-01-01'` | ❌ | 跳过了最左列 |
| `WHERE user_id = 1001 OR created_at > '2025-01-01'` | ⚠️ 可能 | OR 可能导致索引合并 |

**如果你的业务大量依赖 `created_at` 范围查询**，正确方案是再建一个专门的索引：

```sql
-- 专门服务于时间范围查询
CREATE INDEX idx_created_at ON orders(created_at);

-- 或者如果两个字段都要独立查询
CREATE INDEX idx_user_time ON orders(user_id, created_at);
CREATE INDEX idx_time ON orders(created_at);
```

### 验证手段

```sql
-- 用 EXPLAIN 验证索引使用情况
EXPLAIN SELECT * FROM orders WHERE user_id = 1001 AND created_at > '2025-01-01';

-- 关注 key 列是否指向 idx_user_time
-- 关注 Extra 列是否出现 Using index（覆盖索引）或 Using where
```

## 误区三：对小表盲目建索引

### 错误现象

对只有几百行的配置表、字典表也精心设计索引，甚至建了多个复合索引。

### 根因分析

当表的数据量很小（比如几百行、几千行）时，MySQL 优化器**几乎总是选择全表扫描**，原因很简单：

- 全表扫描只需一次顺序 IO，读取所有数据页
- 走索引需要先查 B+Tree 找到主键，再回表查聚簇索引（至少 2 次随机 IO）
- 小表的全部数据可能就在 1-2 个数据页里，顺序扫描比索引查找更快

小表（1000 行，~200KB 数据）：
全表扫描 → 顺序读 13 个数据页（16KB/页）→ ~0.5ms
走索引   → B+Tree 查找 + 回表 × N 次随机 IO → ~2ms（反而更慢）

### 正确做法

1. **数据量 < 1 万行**：通常不需要二级索引（主键索引自动存在）
2. **数据量 1 万 - 10 万行**：根据查询模式谨慎添加
3. **数据量 > 10 万行**：开始认真考虑索引设计
4. **配置表/字典表**：考虑在应用层缓存，而非依赖数据库索引

**例外情况**：如果小表被高频 JOIN（作为驱动表），且 JOIN 条件字段没有索引，仍然需要为 JOIN 字段建索引。

## 误区四：在索引列上做函数运算 —— 隐式失效

### 错误现象

```sql
-- 以为能走索引，实际全表扫描
SELECT * FROM users WHERE YEAR(created_at) = 2025;
SELECT * FROM users WHERE LEFT(phone, 3) = '138';
SELECT * FROM orders WHERE amount + 100 > 500;
```

开发者看到 `created_at` 上有索引，认为范围查询一定走索引，结果 EXPLAIN 显示 `type: ALL`（全表扫描）。

### 根因分析

B+Tree 索引存储的是列的**原始值**，而不是函数运算后的值。当你对索引列施加函数时，MySQL 必须对每一行都执行函数计算，然后才能比较——这就等于全表扫描。

索引存储的值：  2024-01-15, 2024-03-20, 2025-01-10, ...
YEAR() 结果：   2024,       2024,       2025,       ...

MySQL 无法通过 B+Tree 的有序性快速定位 YEAR()=2025 的行
→ 必须逐行计算 YEAR(created_at) 并比较
→ 退化为全表扫描

### 正确做法

**将函数运算转化为对索引列的范围比较：**

```sql
-- ❌ 错误：函数包裹索引列
SELECT * FROM users WHERE YEAR(created_at) = 2025;

-- ✅ 正确：转化为范围条件
SELECT * FROM users WHERE created_at >= '2025-01-01' AND created_at < '2026-01-01';

-- ❌ 错误：函数包裹索引列
SELECT * FROM users WHERE LEFT(phone, 3) = '138';

-- ✅ 正确：LIKE 前缀匹配（可以走索引）
SELECT * FROM users WHERE phone LIKE '138%';

-- ❌ 错误：列参与计算
SELECT * FROM orders WHERE amount + 100 > 500;

-- ✅ 正确：将常量移到等式另一边
SELECT * FROM orders WHERE amount > 400;
```

**MySQL 8.0+ 的函数索引**：如果确实需要频繁按函数结果查询，可以创建函数索引：

```sql
-- MySQL 8.0+ 支持函数索引
CREATE INDEX idx_year_created ON users((YEAR(created_at)));
CREATE INDEX idx_left_phone ON users((LEFT(phone, 3)));

-- 此时 YEAR(created_at) = 2025 可以走索引
SELECT * FROM users WHERE YEAR(created_at) = 2025;  -- ✅ 走函数索引
```

## 误区五：忽视 LSM-Tree 的写放大问题

### 错误现象

在使用 RocksDB、LevelDB、TiKV 等 LSM-Tree 引擎的系统时，只关注写入吞吐量，忽视了写放大（Write Amplification）对磁盘的累积伤害。表现为：

- SSD 寿命提前耗尽（TBW 指标快速下降）
- Compaction 期间出现明显的写入延迟毛刺（P99 从 5ms 飙升到 500ms+）
- 磁盘 IO 利用率持续 100%

### 根因分析

LSM-Tree 的写放大来源于**多层 Compaction**。数据从 MemTable flush 到 L0 后，会被反复合并到更底层的 Level：

写放大计算公式（以 Size Ratio = 10 为例）：

Level 0:  写入 1 次数据
Level 1:  与 L0 合并，写放大 ~10 倍
Level 2:  与 L1 合并，写放大 ~10 倍
...
Level N:  总写放大 ≈ SizeRatio × Level数

典型场景：
  5 层 LSM-Tree，Size Ratio = 10
  理论最大写放大 ≈ 10 × 5 = 50 倍
  即写入 1MB 数据，磁盘实际写入 50MB

### 正确做法

**1. 调整 Compaction 策略**

```cpp
// RocksDB 关键配置
options.level0_file_num_compaction_trigger = 4;    // L0 文件数触发 compaction 的阈值
options.level0_slowdown_writes_trigger = 20;       // 开始限速
options.level0_stop_writes_trigger = 36;            // 完全停止写入
options.max_bytes_for_level_base = 256 * 1048576;  // L1 大小上限 256MB
options.max_bytes_for_level_multiplier = 10;        // 每层倍数

// Universal Compaction（适合写密集场景）
options.compaction_style = kCompactionStyleUniversal;
options.compaction_options_universal.size_ratio = 1;  // 更积极的合并
```

**2. 使用 FIFO Compaction（日志型数据）**

```cpp
// 适用于时序数据、日志数据，只保留最近的数据
options.compaction_style = kCompactionStyleFIFO;
options.max_table_files_size = 10ULL * 1024 * 1024 * 1024;  // 10GB 上限
// 超过上限直接删除最老的 SSTable，无写放大
```

**3. 监控 Compaction 健康度**

```bash
# 查看 RocksDB compaction 统计
# 在 LOG 文件中搜索 Compaction 相关行
grep "Compaction" LOG | tail -20

# 关键指标：
# - Compaction pending：待 compaction 的文件数
# - Write amplification ratio：实际写放大倍数
# - L0 文件数：超过 slowdown trigger 会限速
```

## 误区六：选错索引类型 —— B+Tree ≠ 万能

### 错误现象

在所有场景下都使用 B+Tree 索引，即使某些场景用 Hash 索引或倒排索引更合适。

### 根因分析

不同索引结构有不同的适用场景：

| 索引类型 | 最优场景 | 劣势场景 | 代表实现 |
|---------|---------|---------|---------|
| B+Tree | 范围查询、排序、前缀匹配 | 点查不如 Hash | InnoDB 二级索引 |
| Hash | 等值查询（=, IN） | 范围查询、排序 | MEMORY 引擎、Redis |
| 布隆过滤器 | 大集合的「不存在」判断 | 存在误判 | RocksDB、Cassandra |
| 倒排索引 | 全文检索、关键词搜索 | 精确匹配 | Elasticsearch、Lucene |
| 前缀树/Trie | 自动补全、词典查询 | 内存占用大 | 自定义实现 |

### 正确做法

根据查询模式选择索引：

```sql
-- 场景1：用户登录（精确匹配 user_id）
-- → Hash 索引更优（如果引擎支持）
CREATE TABLE sessions (
    token VARCHAR(128) PRIMARY KEY,
    user_id BIGINT NOT NULL
) ENGINE=MEMORY;

-- 场景2：按时间范围查询订单
-- → B+Tree 索引（利用有序性做范围扫描）
CREATE INDEX idx_order_time ON orders(created_at);

-- 场景3：商品全文搜索
-- → 倒排索引（用 Elasticsearch）
-- 不要在 MySQL LIKE '%keyword%' 上浪费时间
```

**MySQL 中的 Hash 索引使用**：InnoDB 不支持显式 Hash 索引，但支持**自适应 Hash 索引（AHI）**：

```sql
-- 查看 AHI 状态
SHOW ENGINE INNODB STATUS\G

-- 查看 AHI 命中率
SHOW STATUS LIKE 'Innodb_adaptive_hash%';
-- Innodb_adaptive_hash索引请求总数
-- Innodb_adaptive_hash索引命中数
-- 如果命中率 < 90%，考虑关闭 AHI
```

## 误区七：忽视监控和慢查询分析

### 错误现象

系统上线初期性能良好，几个月后逐渐变慢，直到某天突然崩溃才发现问题。排查时发现积累了大量的慢查询、锁等待、索引碎片，但之前没有任何监控手段。

### 根因分析

数据库性能退化是**渐进式**的——索引碎片逐渐增多、数据量缓慢增长、慢查询逐渐累积。没有监控就像开车不看仪表盘，直到爆缸才发现没机油了。

### 正确做法

**建立三层监控体系：**

**第一层：实时指标监控（Prometheus + Grafana）**

```yaml
# prometheus.yml 关键采集配置
- job_name: 'mysql'
  static_configs:
    - targets: ['db-host:9104']  # mysqld_exporter
  # 关键指标：
  # mysql_global_status_queries          — QPS
  # mysql_global_status_slow_queries     — 慢查询数
  # mysql_global_status_innodb_row_lock  — 行锁等待
  # mysql_global_status_threads_running  — 活跃线程数
```

**第二层：慢查询日志分析**

```sql
-- 开启慢查询日志
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;  -- 超过 1 秒记录
SET GLOBAL log_queries_not_using_indexes = ON;  -- 记录未使用索引的查询

-- 定期分析
-- 使用 pt-query-digest（Percona Toolkit）
pt-query-digest /var/log/mysql/slow.log > slow_report.txt
```

**第三层：索引健康巡检**

```sql
-- 每周执行的索引健康检查
-- 1. 索引碎片率
SELECT
    table_name,
    index_name,
    ROUND(stat_value * @@innodb_page_size / 1024 / 1024, 2) AS size_mb
FROM mysql.innodb_index_stats
WHERE stat_name = 'size'
ORDER BY stat_value DESC;

-- 2. 未使用的索引
SELECT * FROM sys.schema_unused_indexes;

-- 3. 冗余索引
SELECT * FROM sys.schema_redundant_indexes;
```

## 误区八：过度调参，忽略实际负载特征

### 错误现象

拿到一份「MySQL 最佳配置模板」直接应用到生产环境，或者把所有参数都调到最大值（如 `innodb_buffer_pool_size` 设为物理内存的 90%）。

### 根因分析

每个业务的负载特征不同——OLTP（高并发短事务）、OLAP（低并发长查询）、混合负载各有不同的参数最优解。盲目套用模板可能适得其反：

典型错误配置与影响：

innodb_buffer_pool_size = 90% 物理内存
  → 如果操作系统也需要内存，可能导致 swap
  → swap 一出现，数据库性能断崖式下降

innodb_flush_log_at_trx_commit = 0
  → 数据安全降级：宕机最多丢失 1 秒数据
  → 对金融/电商场景不可接受

innodb_log_file_size = 4GB
  → 对写密集场景太小 → 频繁 checkpoint → 写入抖动
  → 对读密集场景太大 → 恢复时间过长

### 正确做法

**根据实际负载调参，遵循「先测量再调整」原则：**

| 负载类型 | buffer_pool | log_file_size | flush_policy | 说明 |
|---------|:-----------:|:-------------:|:------------:|------|
| OLTP（读多写少） | 物理内存 60-70% | 1-2 GB | 1（完全持久化） | 保证数据安全 |
| OLTP（写多读少） | 物理内存 50-60% | 4-8 GB | 1 | 大日志减少 checkpoint |
| OLAP（分析查询） | 物理内存 70-80% | 2-4 GB | 1 | 缓存大量扫描结果 |
| 开发/测试 | 物理内存 50% | 512MB | 2（每秒刷盘） | 可接受少量数据丢失 |

```sql
-- 实时监控 buffer_pool 使用率（目标：> 95% 命中率）
SHOW STATUS LIKE 'Innodb_buffer_pool_read%';

-- 计算命中率
SELECT
    (1 - Innodb_buffer_pool_reads / Innodb_buffer_pool_read_requests) * 100
    AS hit_rate_pct
FROM (
    SELECT
        VARIABLE_VALUE AS Innodb_buffer_pool_reads
    FROM performance_schema.global_status
    WHERE VARIABLE_NAME = 'Innodb_buffer_pool_reads'
) a, (
    SELECT
        VARIABLE_VALUE AS Innodb_buffer_pool_read_requests
    FROM performance_schema.global_status
    WHERE VARIABLE_NAME = 'Innodb_buffer_pool_read_requests'
) b;
-- 命中率 < 95%：考虑增大 buffer_pool
-- 命中率 > 99%：buffer_pool 可能过大，可以适当缩小
```

## 误区九：缺乏容错与降级机制

### 错误现象

索引查询没有超时设置，没有重试策略，没有降级方案。当索引服务（如 Elasticsearch）或主库出现故障时，整个业务链条崩溃。

### 根因分析

在分布式系统中，任何组件都可能故障。索引相关的故障模式包括：

- **主库故障**：B+Tree 索引不可用，所有查询失败
- **从库延迟**：读写分离场景下，刚写入的数据在从库查不到（索引数据未同步）
- **ES 集群故障**：全文搜索不可用
- **Compaction 风暴**：LSM-Tree 后台 Compaction 抢占 IO，前台查询超时

### 正确做法

**1. 查询超时控制**

```sql
-- MySQL 级别超时
SET SESSION max_execution_time = 5000;  -- 5 秒超时（仅 SELECT）
SET SESSION lock_wait_timeout = 3;       -- 锁等待超时 3 秒
```

**2. 应用层熔断降级**

```java
// 使用 Resilience4j 熔断器
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    .failureRateThreshold(50)          // 失败率超过 50% 打开熔断
    .waitDurationInOpenState(Duration.ofSeconds(30))  // 熔断 30 秒
    .slidingWindowSize(10)             // 统计最近 10 次调用
    .build();

CircuitBreaker breaker = CircuitBreaker.of("es-search", config);

// 装饰搜索调用
Supplier<List<Product>> decoratedSearch = CircuitBreaker
    .decorateSupplier(breaker, () -> esClient.search(query));

// 熔断时降级到数据库模糊查询
try {
    return decoratedSearch.get();
} catch (CallNotPermittedException e) {
    return dbClient.fuzzySearch(keyword);  // 降级方案
}
```

**3. 读写分离延迟处理**

```java
// 关键业务写后立即读，强制走主库
public Order getOrderAfterCreate(Long orderId) {
    // 刚创建的订单立即查询，走主库避免复制延迟
    return orderRepository.findByIdPrimary(orderId);
}

// 普通查询走从库
public List<Order> searchOrders(Query query) {
    return orderRepository.findByQueryReplica(query);
}
```

## 误区十：忽视索引的安全隐患

### 错误现象

只关注索引的性能，不关注索引引入的安全风险。常见问题包括：

- 索引字段泄露敏感信息（如在索引统计信息中暴露邮箱、手机号）
- 盲注攻击利用索引报错信息探测数据库结构
- 未加密的索引文件被直接读取

### 根因分析

索引本质上是数据的**冗余副本**，它承载了与原始数据相同的信息量。如果原始数据需要加密或脱敏，索引数据同样需要。

### 正确做法

**1. 防止错误信息泄露**

```sql
-- 生产环境不要暴露详细的错误信息
-- 应用层统一捕获数据库异常，返回通用错误码
-- 不要把 MySQL 的原始错误信息返回给前端

-- ❌ 错误：暴露索引信息
catch (SQLException e) {
    response.setError("Index " + e.getMessage());
}

-- ✅ 正确：统一错误处理
catch (SQLException e) {
    log.error("Database error: {}", e.getMessage());  // 记录到日志
    response.setError("Service temporarily unavailable");
}
```

**2. 敏感字段索引的安全处理**

```sql
-- 手机号搜索场景：不要直接索引明文手机号
-- 方案一：索引哈希值（等值查询）
ALTER TABLE users ADD COLUMN phone_hash CHAR(32);
UPDATE users SET phone_hash = SHA2(phone, 256);
CREATE INDEX idx_phone_hash ON users(phone_hash);

-- 查询时
SELECT * FROM users WHERE phone_hash = SHA2('13800138000', 256);

-- 方案二：AES 加密 + 函数索引（MySQL 8.0+）
CREATE INDEX idx_encrypted_phone
    ON users((AES_ENCRYPT(phone, 'secret_key')));
```

**3. 审计索引访问**

```sql
-- 监控谁在查询索引相关的系统表
-- performance_schema 可以记录对元数据表的访问
SELECT * FROM performance_schema.events_statements_summary_by_digest
WHERE digest_text LIKE '%innodb_index_stats%'
ORDER BY count_star DESC;
```

## 误区十一：盲目照搬别人的索引方案

### 错误现象

看到知名项目的索引设计（如 Shopify 的 Rails 迁移、GitHub 的 MySQL 架构），直接复制到自己的项目中。结果发现性能反而更差。

### 根因分析

索引设计是**高度业务相关**的——别人的最优方案是基于他们特定的数据分布、查询模式、并发量和硬件配置。直接复制就像穿别人的鞋子，尺码大概率不合适。

具体差异包括：

| 维度 | 项目 A（电商） | 项目 B（社交） | 差异影响 |
|------|:---:|:---:|------|
| 数据分布 | 均匀分布 | 幂律分布（大V效应） | 二级索引效率差异大 |
| 查询模式 | 大量范围查询 | 大量等值查询 | B+Tree vs Hash |
| 写读比 | 1:10 | 10:1 | LSM-Tree 可能更优 |
| 数据量 | 千万级 | 十亿级 | 索引策略完全不同 |
| 延迟要求 | < 100ms | < 10ms | 索引设计精度不同 |

### 正确做法

**建立自己的索引设计流程：**

1. **分析实际查询**：收集 1-2 周的慢查询日志和通用查询日志
2. **统计字段基数**：用 `SELECT COUNT(DISTINCT col) / COUNT(*) FROM table` 评估字段选择性
3. **原型验证**：在测试环境用真实数据验证索引效果
4. **A/B 测试**：灰度发布对比索引变更前后的性能指标

```sql
-- 评估字段选择性（选择性越高，索引效果越好）
SELECT
    column_name,
    cardinality,
    ROUND(cardinality / (
        SELECT table_rows FROM information_schema.tables
        WHERE table_name = 'orders'
    ) * 100, 2) AS selectivity_pct
FROM information_schema.statistics
WHERE table_name = 'orders'
  AND index_name != 'PRIMARY'
ORDER BY selectivity_pct DESC;

-- selectivity > 10%：索引效果好
-- selectivity < 1%：索引效果差，考虑删除
```

**核心原则：没有万能方案，只有适合你业务的方案。**

## 误区十二：忽视维护窗口和碎片整理

### 错误现象

数据库运行几个月后，相同查询的执行时间明显变长，但索引和查询语句都没有变过。EXPLAIN 显示索引仍然在使用，但实际 IO 量大幅增加。

### 根因分析

**索引碎片**是长时间运行后不可避免的问题。当数据频繁更新、删除时，B+Tree 的叶子节点会出现大量空洞（page 内有空闲空间但不被复用），导致：

- **逻辑碎片**：数据在索引中的物理排列变得不连续，顺序扫描需要读取更多数据页
- **物理碎片**：磁盘上的数据文件产生空洞，文件系统层面的 IO 效率下降

碎片化前后的对比：

碎片化前（紧凑排列）：
[Page1: 100%满] [Page2: 100%满] [Page3: 100%满] → 读 3 页

碎片化后（大量空洞）：
[Page1: 60%满] [Page2: 40%满] [Page3: 70%满] [Page4: 30%满] → 读 4 页
数据量相同，但 IO 增加 33%

InnoDB 默认 page 分辨率：
当 page 空闲空间超过 innodb_page_fill_threshold（默认 50%）时
新插入的数据会被拆分到新 page → 加剧碎片化

### 正确做法

**1. 监控碎片率**

```sql
-- 查看表碎片率（information_schema）
SELECT
    table_name,
    ROUND(data_length / 1024 / 1024, 2) AS data_mb,
    ROUND(index_length / 1024 / 1024, 2) AS index_mb,
    ROUND(data_free / 1024 / 1024, 2) AS free_mb,
    ROUND(data_free / (data_length + 1) * 100, 1) AS frag_pct
FROM information_schema.tables
WHERE table_schema = 'your_database'
  AND data_length > 10 * 1024 * 1024  -- 只看大于 10MB 的表
ORDER BY frag_pct DESC;

-- 碎片率 > 20%：考虑整理
-- 碎片率 > 40%：应该立即整理
```

**2. 选择合适的碎片整理方式**

```sql
-- 方式一：OPTIMIZE TABLE（在线重建，锁表时间长）
-- 适用于可接受短暂维护窗口的场景
OPTIMIZE TABLE orders;
-- 会重建整张表和所有索引，碎片率降到最低
-- 缺点：需要额外磁盘空间（临时复制一份），执行期间有写锁

-- 方式二：ALTER TABLE ... ENGINE=InnoDB（等效重建）
ALTER TABLE orders ENGINE=InnoDB;

-- 方式三：在线 DDL（MySQL 5.6+，推荐）
-- 使用 pt-online-schema-change（Percona Toolkit）避免锁表
pt-online-schema-change --alter "ENGINE=InnoDB" \
    D=mydb,t=orders \
    --execute
```

**3. 建立定期维护计划**

```bash
#!/bin/bash
# weekly_maintenance.sh — 每周日凌晨执行

LOG_DIR="/var/log/db_maintenance"
mkdir -p $LOG_DIR

echo "=== 索引碎片检查 $(date) ===" > $LOG_DIR/maintenance.log

# 检查碎片率超过 20% 的表
mysql -u monitor -p"$MONITOR_PASS" mydb << 'EOF' >> $LOG_DIR/maintenance.log
SELECT
    table_name,
    ROUND(data_free / (data_length + 1) * 100, 1) AS frag_pct
FROM information_schema.tables
WHERE table_schema = 'mydb'
  AND data_free > 50 * 1024 * 1024  -- 空闲空间 > 50MB
  AND data_free / (data_length + 1) > 0.20
ORDER BY frag_pct DESC;
EOF

# 对高碎片表执行在线整理
TABLES_TO_OPTIMIZE=$(mysql -u monitor -p"$MONITOR_PASS" -N mydb << 'EOF'
SELECT table_name
FROM information_schema.tables
WHERE table_schema = 'mydb'
  AND data_free > 100 * 1024 * 1024
  AND data_free / (data_length + 1) > 0.30
EOF
)

for TABLE in $TABLES_TO_OPTIMIZE; do
    echo "整理表: $TABLE" >> $LOG_DIR/maintenance.log
    pt-online-schema-change --alter "ENGINE=InnoDB" \
        D=mydb,t=$TABLE --execute >> $LOG_DIR/maintenance.log 2>&amp;1
done
```

## 本节小结

索引结构的误区往往源于对底层原理的理解不足。核心要点：

1. **索引不是越多越好**，要根据实际查询模式按需设计
2. **复合索引遵循最左前缀原则**，理解 B+Tree 的排序规则是关键
3. **小表和函数运算**是索引失效的两大常见原因
4. **LSM-Tree 的写放大**需要通过 Compaction 策略控制
5. **监控是预防问题的最好手段**，不要等问题爆发再处理
6. **参数调优要因地制宜**，没有万能配置模板
7. **容错降级**是分布式系统的必备能力
8. **索引安全**和性能同等重要
9. **业务决定方案**，不要盲目复制别人的索引设计
10. **定期维护**是保持索引性能的关键

> **记住**：索引设计是一门平衡的艺术——在查询性能、写入性能、存储空间和维护成本之间找到适合你业务的平衡点。
