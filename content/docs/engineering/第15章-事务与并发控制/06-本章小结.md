# 本章小结

## 一、核心知识体系回顾

本章从实现者视角系统讲解了事务与并发控制的四大支柱：ACID特性的底层实现、两阶段锁协议（2PL）、死锁处理机制、多版本并发控制（MVCC），并扩展到乐观并发控制（OCC）和时间戳排序协议。以下是各主题的核心要点梳理。

### 1.1 ACID特性：不是四个独立机制，而是一个协作体系

ACID的四个属性在教科书中并列描述，但从实现角度看，它们的机制完全不同：

| 属性 | 实现机制 | 核心数据结构 | 工业标准 |
|------|----------|-------------|----------|
| 原子性（Atomicity） | Undo Log | 回滚段 + 链表 | ARIES Undo |
| 持久性（Durability） | Redo Log / WAL | 日志序列号（LSN） | ARIES Redo |
| 隔离性（Isolation） | 锁 + MVCC | 锁表 / 版本链 / Read View | 2PL / SI / SSI |
| 一致性（Consistency） | 以上三者合力 + 约束检查 | 主键/外键/CHECK/触发器 | 无单一实现 |

**关键认知**：一致性不是由某个机制单独保证的，而是原子性、隔离性和持久性三者共同作用的结果，再加上数据库自身的完整性约束检查。如果原子性或隔离性被打破，一致性就会受损。

**InnoDB的Undo Log实现**：记录每次修改前的旧值，同一事务的多条记录通过链表串联。回滚时从链表尾部逐条执行反向操作（INSERT→DELETE，UPDATE→反向UPDATE）。

**PostgreSQL的替代策略**：没有独立的Undo Log，UPDATE操作不原地修改，而是插入新元组并保留旧元组。回滚只需标记新元组无效。简化了回滚逻辑，但带来表膨胀（table bloat）问题，需要VACUUM定期清理。

**ARIES恢复算法**（Mohan et al., 1992）：分析→重做→撤销三阶段。重做阶段采用"无条件重做"策略——即使数据页已包含该修改也要重新应用，因为重做操作必须是幂等的。这大大简化了恢复逻辑。

### 1.2 两阶段锁协议（2PL）：可串行化的充分条件

2PL是保证可串行化的经典算法，由Eswaran等人于1976年提出。三个变体层层递进：

| 变体 | 规则 | 级联回滚 | 串行顺序 | 典型采用 |
|------|------|----------|----------|----------|
| Basic 2PL | 增长阶段只加锁，缩减阶段只解锁 | 存在 | 不确定 | 理论模型 |
| Strict 2PL | X锁在事务结束时释放，S锁可提前释放 | 消除 | 不确定 | 多数数据库 |
| Rigorous 2PL | 所有锁在事务结束时释放 | 消除 | 提交顺序 = 串行顺序 | **MySQL InnoDB** |

**定理**：任何遵循2PL的调度都是冲突可串行化的。证明通过反证法完成——假设优先图中存在环，可以推导出违反2PL规则的矛盾。

**Rigorous 2PL的简化优势**：事务的等价串行顺序就是事务提交的顺序。这大大简化了可串行化验证和并发控制的实现。

**2PL是死锁的根源**：因为事务在持有锁的同时请求新锁，循环等待不可避免。这是协议的固有特性，不是bug。

### 1.3 死锁处理：检测、预防、避免三管齐下

#### 检测：Wait-For Graph算法

系统维护有向图，节点是事务，边 Ti→Tj 表示 Ti 等待 Tj 释放锁。通过DFS检测环，检测到环则选择代价最小的事务回滚。MySQL InnoDB默认开启死锁检测（`innodb_deadlock_detect=ON`），在高并发（>1000并发事务）下可考虑关闭以降低CPU开销。

**MySQL死锁排查要点**：
- 开启 `innodb_print_all_deadlocks = ON`，记录所有死锁（默认只记录最后一次）
- 使用 `performance_schema.data_locks` 和 `data_lock_waits`（MySQL 8.0+）监控锁等待
- `SHOW ENGINE INNODB STATUS` 的 `LATEST DETECTED DEADLOCK` 部分

**PostgreSQL死锁排查要点**：
- `deadlock_timeout`（默认1s）控制死锁检测启动时机
- 使用 `pg_locks` + `pg_stat_activity` 监控锁等待关系
- PostgreSQL检测到死锁时自动回滚整个事务

**SQL Server死锁排查要点**：
- `lock monitor` 线程每5秒扫描一次
- 使用扩展事件（Extended Events）或 Trace Flag 1222 追踪死锁
- 使用 `sys.dm_tran_locks` 和 `sys.dm_exec_requests` 实时监控

#### 预防：Wait-Die vs Wound-Wait

| 策略 | 老事务请求 | 新事务请求 | 被回滚者 | 优势 |
|------|-----------|-----------|----------|------|
| Wait-Die | 等待 | 自行回滚 | 发起请求的事务 | 实现简单 |
| Wound-Wait | 强制回滚holder | 等待 | 持有锁的事务 | **老事务不被回滚，减少工作浪费** |

两者都基于时间戳全序关系保证无循环等待。Wound-Wait通常优于Wait-Die。

#### 四种常见死锁模式与解决方案

1. **交叉更新**：不同顺序更新同一组行 → 统一资源访问顺序（按ID升序）
2. **间隙锁（Gap Lock）**：RR隔离级别下间隙锁扩大冲突面 → 使用RC隔离级别或精确锁定
3. **唯一索引冲突**：并发插入同一唯一索引 → 使用 `INSERT ... ON DUPLICATE KEY UPDATE`
4. **外键约束**：外键自动加共享锁 → 避免高频跨表关联更新，或批量操作前预排序

### 1.4 MVCC：读不阻塞写，写不阻塞读

MVCC是现代数据库最核心的并发控制创新。每个事务在开始时获取快照，读操作基于快照数据，写操作创建新版本但不覆盖旧版本。

#### 快照隔离（Snapshot Isolation）

核心规则：First-Committer-Wins——如果两个并发事务修改同一数据，先提交的成功，后提交的回滚。消除了脏读和不可重复读，但**不能防止写偏斜（Write Skew）异常**。

**写偏斜的经典案例**：两个医生同时查看值班表，各自发现对方在值班，于是都申请休假，最终无人值班。两个事务各自基于快照读取，都认为满足"至少一人值班"约束，但提交后约束被违反。

#### 可串行化快照隔离（SSI）

SSI在快照隔离基础上增加写偏斜检测，通过检测rw-antidependency的危险结构实现。PostgreSQL从9.1版本开始实现SSI，是工业界首个基于MVCC的可串行化实现。

#### MySQL InnoDB vs PostgreSQL的MVCC实现

| 特性 | MySQL InnoDB | PostgreSQL |
|------|-------------|------------|
| 旧版本存储 | Undo Log（独立区域） | 数据页内（堆表） |
| 可见性判断 | Read View + Undo Log chain | 元组xmin/xmax + clog |
| 版本清理 | Purge线程异步清理Undo Log | VACUUM机制回收死元组 |
| 表膨胀问题 | 较轻（Undo Log独立管理） | 较重（需要定期VACUUM） |
| 可序列化 | 通过Next-Key Lock模拟 | SSI（真正的快照隔离可序列化） |

**版本链是性能关键**：链越长，遍历开销越大。长事务是版本链膨胀的最大元凶——即使旧版本早已"过期"，只要有活跃事务需要它，就不能被清理。

### 1.5 乐观并发控制（OCC）

OCC的核心假设是"大多数情况下事务之间不会发生冲突"。事务执行时不加锁，仅在提交时验证冲突。

**三阶段协议**：
1. **读取阶段**：在私有工作区执行操作，记录read_set和write_set
2. **验证阶段**：检查read_set是否被已提交事务修改（向后验证）或write_set是否影响活跃事务（向前验证）
3. **写入阶段**：验证通过后原子写入数据库

**应用层落地**：版本号机制——UPDATE时带上 `WHERE version = old_version`，影响行数为0则表示冲突，需重试。Java/Spring Boot通过 `@Version` 注解自动管理。

**适用场景**：低争用、读多写少的场景。高争用下频繁回滚反而不如悲观锁。

### 1.6 时间戳排序协议

基于事务时间戳全局排序来决定执行顺序。Thomas Write Rule允许跳过已过时的写操作而不必中止事务，减少了不必要的事务中止。多版本时间戳排序结合了时间戳排序和MVCC的优点。

## 二、关键公式与性能模型

| 概念 | 公式/模型 | 说明 | 应用场景 |
|------|-----------|------|----------|
| Little定律 | QPS = 并发数 / 平均延迟 | 系统吞吐量的理论上限 | 容量规划、性能评估 |
| 可用性 | SLA = 正常时间 / 总时间 | 99.9% = 8.76小时/年停机 | SLA制定、冗余设计 |
| 尾延迟 | P99 = 排序后第99百分位值 | 反映长尾用户体验 | 性能优化目标设定 |
| 容量规划 | QPS × 单次请求资源 = 总资源需求 | 资源评估基础 | 架构设计、资源分配 |
| 死锁检测开销 | O(N²)（最坏情况，N=并发事务数） | Wait-For Graph DFS遍历 | 高并发下的检测策略选择 |
| MVCC版本链开销 | O(L)（L=版本链长度） | 每次读操作遍历版本链 | 监控长事务、VACUUM策略 |
| 2PL锁冲突概率 | P(冲突) ≈ 1 - (1-p)^(N-1) | p=单行被修改概率，N=并发事务数 | 评估死锁风险、选择隔离级别 |

## 三、核心技巧总结

### 3.1 事务最佳实践

**设计阶段**：
- 事务尽量短小——长事务导致版本链膨胀、Undo Log积压、锁持有时间过长
- 按固定顺序访问资源——避免交叉更新导致的死锁
- 选择合适的隔离级别——读多写少用RC+MVCC，需要可串行化用RR+2PL或SSI

**实现阶段**：
- 利用索引避免锁升级——InnoDB行锁基于索引实现，无索引的WHERE条件会退化为表锁
- 批量操作合并为单个事务——减少日志刷盘次数
- 设置合理的锁等待超时——避免长时间阻塞

**监控阶段**：
- 监控活跃事务数量和持续时间——发现长事务及时处理
- 监控锁等待和死锁次数——识别热点资源
- 监控Undo Log空间使用——预防空间不足

### 3.2 死锁排查流程

1. 开启死锁日志记录
   MySQL: SET GLOBAL innodb_print_all_deadlocks = ON
   PostgreSQL: log_min_messages = LOG + deadlock_timeout = 1s

2. 监控锁等待关系
   MySQL: performance_schema.data_lock_waits
   PostgreSQL: pg_locks + pg_stat_activity

3. 解读死锁日志
   识别涉及的事务、SQL、锁类型、等待关系

4. 分析根因
   确定死锁模式（交叉更新/间隙锁/唯一索引/外键）

5. 实施修复
   统一资源顺序 / 调整隔离级别 / 优化索引 / 调整业务逻辑

6. 验证效果
   压力测试确认死锁消除

### 3.3 乐观锁实现要点

| 场景 | 推荐方案 | 注意事项 |
|------|----------|----------|
| 单机MySQL | version字段 + WHERE version = N | 重试次数限制（建议3次） |
| PostgreSQL | version + RETURNING子句 | 避免使用CTID（VACUUM后变化） |
| Redis | WATCH + MULTI | 注意WATCH期间key变更 |
| 分布式系统 | 雪花ID或向量时钟作为版本号 | 处理时钟偏移问题 |
| 高并发场景 | 乐观锁+异步队列削峰 | 监控重试率，过高则切换悲观锁 |

## 四、常见误区与纠正

| 误区 | 真相 | 纠正方法 |
|------|------|----------|
| "REPEATABLE READ下不会幻读" | SQL标准的RR不能防幻读，但InnoDB通过Next-Key Lock在RR下部分避免了幻读 | 理解InnoDB的间隙锁机制，必要时使用Serializable |
| "长事务只是性能差一点" | 长事务会阻塞版本链清理、占用Undo Log空间、增加死锁概率 | 设置事务超时，监控活跃事务时长 |
| "MVCC完全不需要锁" | MVCC读不加锁，但写操作仍需2PL加锁 | 理解MVCC+2PL的组合工作方式 |
| "死锁是bug" | 死锁是2PL协议的固有特性，不是实现缺陷 | 设计低死锁率的事务执行策略 |
| "乐观锁适用于所有场景" | 高争用场景下乐观锁频繁回滚，性能反而更差 | 根据冲突概率选择悲观锁或乐观锁 |
| "无索引查询只影响查询速度" | InnoDB无索引的UPDATE/DELETE会退化为表锁，大幅增加死锁概率 | 确保UPDATE/DELETE的WHERE条件命中索引 |
| "死锁检测开销可忽略" | 高并发下（>1000并发事务）Wait-For Graph检测消耗大量CPU | 高并发时可关闭死锁检测，改用锁等待超时 |
| "VACUUM/AutoVacuum不重要" | PostgreSQL不VACUUM会导致表膨胀、查询变慢、磁盘空间耗尽 | 监控表膨胀率，确保AutoVacuum正常运行 |

## 五、四种并发控制策略对比

| 维度 | 2PL（悲观锁） | MVCC | OCC（乐观锁） | 时间戳排序 |
|------|--------------|------|--------------|-----------|
| 核心思想 | 互斥访问 | 多版本读写分离 | 先操作后验证 | 时间戳全局排序 |
| 读操作开销 | 需要加S锁 | 快照读无锁 | 无锁 | 无锁 |
| 写操作开销 | 需要加X锁 | 创建新版本 | 提交时验证+写入 | 创建新版本 |
| 死锁风险 | 存在 | 写操作仍存在 | 不存在 | 不存在 |
| 冲突处理 | 阻塞等待 | First-Committer-Wins | 回滚重试 | 按时间戳顺序 |
| 适用场景 | 写密集、高争用 | 读多写少 | 低争用、读多写少 | 理论模型为主 |
| 工业采用 | InnoDB（Rigorous 2PL） | InnoDB + PostgreSQL | 应用层实现 | 较少直接采用 |
| 可串行化保证 | 2PL直接保证 | SI不能防写偏斜，SSI可以 | 验证策略决定 | Thomas Write Rule |

## 六、MySQL InnoDB的并发控制全景

InnoDB是2PL、MVCC和锁机制的集大成者，理解它的设计选择对实际工作极具价值：

MySQL InnoDB 并发控制架构
├── 隔离性实现
│   ├── 快照读（SELECT without FOR UPDATE） → MVCC Read View
│   └── 当前读（SELECT FOR UPDATE / UPDATE / DELETE） → 2PL加锁
│
├── 锁体系
│   ├── 行锁：基于索引实现（无索引→表锁）
│   │   ├── 记录锁（Record Lock）：锁定具体行
│   │   ├── 间隙锁（Gap Lock）：锁定索引间隙（RR级别）
│   │   └── 临键锁（Next-Key Lock）：记录锁 + 间隙锁
│   ├── 表锁：意向锁（IS/IX）+ 表级锁
│   └── 自增锁（AUTO-INC）：保护自增列
│
├── MVCC实现
│   ├── Read View：包含活跃事务ID列表
│   ├── Undo Log chain：版本链
│   └── Purge线程：异步清理过期版本
│
├── 死锁处理
│   ├── Wait-For Graph实时检测（innodb_deadlock_detect）
│   ├── 锁等待超时兜底（innodb_lock_wait_timeout=50s）
│   └── 牺牲者选择：回滚代价最小的事务
│
└── 隔离级别实现
    ├── READ UNCOMMITTED：不使用MVCC，读最新未提交数据
    ├── READ COMMITTED：每次读取创建新Read View
    ├── REPEATABLE READ：事务开始时创建Read View（默认）
    └── SERIALIZABLE：所有读操作自动加S锁

## 八、思考题

### 基础理解

1. **ACID的实现依赖关系**：为什么说一致性是由原子性、隔离性和持久性共同保证的？如果持久性被破坏（比如Redo Log丢失），一致性会受到怎样的影响？

2. **2PL与死锁的必然联系**：2PL协议为什么必然导致死锁风险？能否设计一个既保证可串行化又完全避免死锁的悲观并发控制协议？

3. **MVCC的读写模型**：InnoDB在READ COMMITTED和REPEATABLE READ下的Read View创建时机有何不同？这对查询结果有什么具体影响？

### 实际应用

4. **隔离级别选择**：在一个电商系统中，库存扣减（写密集）和商品列表查询（读密集）应该分别使用什么隔离级别？为什么？

## 九、核心概念速查表

| 术语 | 定义 | 关键点 |
|------|------|--------|
| ACID | 原子性、一致性、隔离性、持久性 | 一致性由其他三者共同保证 |
| 2PL | 两阶段锁协议：增长阶段+缩减阶段 | 保证冲突可串行化的充分条件 |
| Strict 2PL | X锁在事务结束时释放 | 消除级联回滚 |
| Rigorous 2PL | 所有锁在事务结束时释放 | 提交顺序=串行顺序，InnoDB采用 |
| MVCC | 多版本并发控制 | 读不阻塞写，写不阻塞读 |
| Read View | InnoDB的快照数据 | 包含活跃事务ID列表，决定可见性 |
| Snapshot Isolation | 快照隔离 | First-Committer-Wins，不防写偏斜 |
| SSI | 可串行化快照隔离 | 检测rw-antidependency，防写偏斜 |
| Undo Log | 撤销日志 | 记录修改前旧值，支持回滚和MVCC |
| Redo Log | 重做日志/WAL | 保证持久性，崩溃恢复的基础 |
| ARIES | 崩溃恢复算法 | 分析→重做→撤销三阶段 |
| LSN | 日志序列号 | 单调递增，标识日志记录位置 |
| Wait-For Graph | 等待图 | 事务为节点，等待关系为边，DFS检测环 |
| Wait-Die | 死锁预防策略 | 老事务等待，新事务回滚 |
| Wound-Wait | 死锁预防策略 | 老事务抢占，新事务等待 |
| Gap Lock | 间隙锁 | InnoDB RR级别防止幻读 |
| Next-Key Lock | 临键锁 | 记录锁+间隙锁，InnoDB默认行锁类型 |
| VACUUM | PostgreSQL版本清理 | 回收死元组，防止表膨胀 |
| Purge | InnoDB版本清理 | 异步清理过期Undo Log |
| OCC | 乐观并发控制 | 读取→验证→写入三阶段 |
| Write Skew | 写偏斜异常 | 快照隔离不能防止的异常 |
| Cascading Rollback | 级联回滚 | Basic 2PL的问题，Strict 2PL消除 |
