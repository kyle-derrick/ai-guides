## 八、文件系统管理

文件系统是操作系统管理持久化数据的核心机制。对于安全从业者而言，深入理解文件系统的底层原理不仅是系统管理的基本功，更是发现安全漏洞、进行取证分析、实施安全加固的关键能力。本节从磁盘硬件层出发，逐层向上讲解分区、文件系统创建、挂载管理、空间监控、文件查找等完整知识体系，并在每个环节融入安全视角。

### 8.1 存储设备与分区管理

#### 8.1.1 Linux存储设备命名规则

Linux将所有存储设备抽象为`/dev`目录下的设备文件，命名规则严格且可预测：

| 设备类型 | 命名格式 | 示例 | 说明 |
|----------|----------|------|------|
| SATA/SCSI硬盘 | `/dev/sd[a-z]` | `/dev/sda` | 按字母顺序分配，a为第一块盘 |
| NVMe SSD | `/dev/nvme[0-9]n[0-9]` | `/dev/nvme0n1` | 第0个控制器的第1个命名空间 |
| 分区 | 设备名+数字 | `/dev/sda1`、`/dev/nvme0n1p1` | NVMe分区加`p`前缀 |
| 软RAID | `/dev/md[0-9]` | `/dev/md0` | 软件RAID设备 |
| LVM逻辑卷 | `/dev/mapper/vg-lv` | `/dev/mapper/ubuntu--vg-root` | LVM逻辑卷设备 |
| 光驱 | `/dev/sr[0-9]` | `/dev/sr0` | SCSI光驱设备 |
| 虚拟磁盘 | `/dev/vd[a-z]` | `/dev/vda` | KVM虚拟机磁盘 |

```bash
# 查看所有块设备及其分区、挂载点、文件系统类型
lsblk
# NAME   MAJ:MIN RM   SIZE RO TYPE MOUNTPOINT
# sda      8:0    0   100G  0 disk
# ├─sda1   8:1    0     1G  0 part /boot
# ├─sda2   8:2    0    50G  0 part /
# └─sda3   8:3    0    49G  0 part
#   └─vg0-lv_data 253:0 0 49G  0 lvm  /data

# lsblk支持输出JSON格式，便于脚本解析
lsblk -J -o NAME,SIZE,TYPE,MOUNTPOINT,FSTYPE

# 查看设备的UUID和文件系统类型（比lsblk更详细）
blkid
# /dev/sda1: UUID="a1b2c3d4-..." TYPE="ext4" PARTUUID="..."
# /dev/sda2: UUID="e5f6a7b8-..." TYPE="ext4" PARTUUID="..."
```

**安全提示**：`blkid`输出的UUID是设备的唯一标识，在`/etc/fstab`中优先使用UUID而非设备名（如`/dev/sda1`），因为设备名在磁盘添加/移除时可能变化，而UUID保持不变。使用不稳定设备名挂载可能导致错误的分区被挂载到关键路径，进而引发安全风险。

#### 8.1.2 分区表格式：MBR vs GPT

分区表是磁盘上记录分区布局的元数据结构。理解两种主流格式的差异对于磁盘管理和安全分析都至关重要：

| 特性 | MBR (Master Boot Record) | GPT (GUID Partition Table) |
|------|--------------------------|---------------------------|
| 最大磁盘支持 | 2TB | 8ZiB（理论上限） |
| 最大分区数 | 4个主分区（或3主+1扩展） | 128个分区（默认） |
| 引导机制 | BIOS（MBR引导代码512字节） | UEFI（EFI System Partition） |
| 数据冗余 | 无，MBR损坏则分区表丢失 | 主备双份，末尾有备份 |
| 校验机制 | 无 | CRC32校验 |
| 安全风险 | 可被MBR病毒覆盖引导代码 | 相对更安全，但仍需保护ESP分区 |

```bash
# 查看分区表类型
sudo fdisk -l /dev/sda
# 输出中 "Disklabel type: dos" 表示MBR，"Disklabel type: gpt" 表示GPT

# 也可以用parted查看
sudo parted /dev/sda print
# Partition Table: gpt
```

#### 8.1.3 分区操作工具

**fdisk** — 经典MBR分区工具，也支持GPT：

```bash
# 交互式分区（进入fdisk交互shell）
sudo fdisk /dev/sdb

# fdisk常用交互命令：
# p    打印分区表
# n    新建分区
# d    删除分区
# t    更改分区类型（82=swap, 83=Linux, 8e=LVM）
# w    写入并退出
# q    不保存退出

# 非交互式：列出分区
sudo fdisk -l /dev/sdb
```

**gdisk** — GPT专用分区工具：

```bash
# 交互式操作GPT分区
sudo gdisk /dev/sdb
# 交互命令与fdisk类似，额外支持：
# i    查看分区详细信息（含GUID）
# c    更改分区名称
```

**parted** — 支持MBR和GPT，支持脚本化操作：

```bash
# 查看分区信息
sudo parted /dev/sdb print

# 脚本化创建GPT分区表并分区
sudo parted /dev/sdb --script \
    mklabel gpt \
    mkpart primary ext4 1MiB 50GiB \
    mkpart primary ext4 50GiB 100%

# 调整分区大小（在线调整，需文件系统支持）
sudo parted /dev/sdb resizepart 1 80GiB
```

**sfdisk** — 脚本化分区工具，适合自动化：

```bash
# 导出分区表为脚本格式
sudo sfdisk -d /dev/sdb > sdb-partition.dump

# 从脚本恢复分区表（批量部署场景）
sudo sfdisk /dev/sdb < sdb-partition.dump

# 批量创建分区
echo -e ',10G,L\n,,L' | sudo sfdisk /dev/sdb
```

#### 8.1.4 磁盘健康检测

磁盘硬件故障是数据丢失的首要原因。SMART（Self-Monitoring, Analysis and Reporting Technology）是现代磁盘内置的健康监测系统：

```bash
# 安装smartmontools
sudo apt install smartmontools   # Debian/Ubuntu
sudo dnf install smartmontools   # RHEL/Fedora

# 查看磁盘SMART概要信息
sudo smartctl -a /dev/sda

# 关注以下关键指标：
# SMART 5  - Reallocated_Sector_Ct（重映射扇区数，>0表示磁盘有坏道）
# SMART 187 - Reported_Uncorrect（不可修正错误，>0应警惕）
# SMART 188 - Command_Timeout（命令超时，快速增长表示磁盘即将故障）
# SMART 197 - Current_Pending_Sector（等待重映射的扇区）
# SMART 198 - Offline_Uncorrectable（离线不可修正扇区）
# SMART 9   - Power_On_Hours（通电时长，用于评估使用寿命）

# 快速健康检查
sudo smartctl -H /dev/sda
# PASSED = 健康
# FAILED! = 立即备份数据

# 启用SMART自动监控
sudo systemctl enable --now smartd

# 对NVMe SSD使用不同的参数
sudo smartctl -a /dev/nvme0n1
# 关注：Available Spare（备用空间百分比）、Media and Data Integrity Errors
```

**安全视角**：在应急响应中，检查磁盘SMART数据可以帮助判断硬件故障还是恶意软件导致的数据损坏。如果`Reallocated_Sector_Ct`或`Current_Pending_Sector`突然增长，可能是物理损坏；如果SMART正常但文件出现异常修改，则更可能是软件层面的问题（入侵、勒索软件等）。

### 8.2 文件系统类型

#### 8.2.1 主流Linux文件系统对比

选择合适的文件系统对性能、可靠性和安全特性都有深远影响：

| 文件系统 | 最大文件 | 最大卷 | 日志 | 透明压缩 | 快照 | 校验和 | 适用场景 |
|----------|----------|--------|------|----------|------|--------|----------|
| ext4 | 16TB | 1EB | 元数据+可选数据 | 不支持 | 不支持 | 不支持 | 通用场景，最成熟稳定 |
| XFS | 8EB | 8EB | 元数据 | 不支持 | 不支持 | 不支持 | 大文件、高并发I/O |
| Btrfs | 16EB | 16EB | 写时复制 | 支持(zlib/lzo/zstd) | 支持 | 支持 | 需要快照/压缩/校验场景 |
| ZFS | 16EB | 256ZiB | 写时复制 | 支持(lz4/zstd) | 支持 | 支持 | 存储服务器、数据密集型 |
| tmpfs | - | 内存大小 | - | - | - | - | 临时文件、/tmp |
| procfs | - | - | - | - | - | - | /proc虚拟文件系统 |
| sysfs | - | - | - | - | - | - | /sys内核对象 |
| overlay | - | - | - | - | - | - | 容器层叠文件系统 |

```bash
# 查看当前系统支持的文件系统类型
cat /proc/filesystems
# 也可用 lsblk -f 查看各分区的文件系统

# 查看已挂载文件系统的详细信息
df -T
# -T 显示文件系统类型
```

**ext4深度解析** — 最广泛使用的Linux文件系统：

- **日志机制**：ext4使用日志（journal）记录元数据变更，系统崩溃后可通过重放日志快速恢复一致性，避免漫长的一致性检查
- **延迟分配**：ext4将数据先缓存在内存中，延迟写入磁盘，减少碎片并提高性能
- **Extent映射**：用extent（连续块范围）替代传统的块映射，大幅提升大文件性能
- **安全特性**：支持`chattr +i`（不可变标志）、`chattr +a`（仅追加标志）等文件属性

```bash
# 查看ext4文件系统参数
sudo tune2fs -l /dev/sda1

# 关键参数说明：
# Filesystem features: has_journal ext_attr resize_inode dir_index ...
# has_journal = 启用日志
# dir_index = 使用哈希索引加速目录查找

# 查看/修改挂载选项
sudo tune2fs -o journal_data_writeback /dev/sda1  # 更改日志模式
sudo tune2fs -c 30 /dev/sda1                        # 每30次挂载后强制检查
sudo tune2fs -i 30d /dev/sda1                       # 每30天强制检查
```

**Btrfs** — 新一代文件系统的核心优势：

```bash
# 创建Btrfs文件系统
sudo mkfs.btrfs -L mydata /dev/sdb1

# 创建子卷（类似独立分区，但共享存储池）
sudo btrfs subvolume create /mnt/data/snapshots
sudo btrfs subvolume create /mnt/data/documents

# 快照（近乎瞬间完成，写时复制）
sudo btrfs subvolume snapshot /mnt/data/documents /mnt/data/snapshots/docs-backup-$(date +%Y%m%d)

# 透明压缩挂载
sudo mount -o compress=zstd:3 /dev/sdb1 /mnt/data

# 数据校验（Btrfs自动为所有数据和元数据计算校验和）
sudo btrfs scrub start /mnt/data
sudo btrfs scrub status /mnt/data

# 查看文件系统使用情况（比df更准确，因为Btrfs共享元数据）
sudo btrfs filesystem df /mnt/data
sudo btrfs filesystem usage /mnt/data
```

#### 8.2.2 虚拟文件系统

Linux中存在多种虚拟文件系统，它们不对应实际磁盘存储，而是内核数据结构的用户空间接口：

- **procfs（`/proc`）**：暴露进程信息、内核参数、硬件信息。安全人员通过读取`/proc`可以获取运行中进程的完整状态
- **sysfs（`/sys`）**：暴露内核对象（设备、驱动、模块）的层次化视图
- **tmpfs（`/tmp`、`/run`）**：基于内存的文件系统，重启后数据消失。适合存放临时数据，但要注意内存消耗
- **devpts（`/dev/pts`）**：伪终端设备
- **securityfs（`/sys/kernel/security`）**：安全框架（SELinux、IMA）的配置接口

```bash
# 查看所有已挂载的虚拟文件系统
mount -t tmpfs,proc,sysfs,devpts

# tmpfs可以限制大小，防止临时文件耗尽内存
sudo mount -t tmpfs -o size=2G,noexec,nosuid tmpfs /tmp
# noexec — 禁止执行该挂载点上的程序
# nosuid — 忽略SUID/SGID位
# 这两个选项显著提升了/tmp的安全性
```

### 8.3 创建文件系统（格式化）

#### 8.3.1 mkfs工具族

`mkfs`是创建文件系统的通用前端，实际调用具体文件系统的工具：

```bash
# 通用语法
sudo mkfs -t <类型> [选项] <设备>

# 创建ext4文件系统
sudo mkfs.ext4 /dev/sdb1
sudo mkfs.ext4 -L "mydata" -b 4096 -i 8192 /dev/sdb1
# -L 卷标
# -b 块大小（4096是默认值，对大多数场景最优）
# -i 每多少字节分配一个inode（8192=每8KB一个inode）

# 创建XFS文件系统
sudo mkfs.xfs -f -L "bigfiles" /dev/sdb1
# -f 强制覆盖已有文件系统

# 创建Btrfs文件系统
sudo mkfs.btrfs -L "snapshot-friendly" -d single -m dup /dev/sdb1
# -d single 数据不做镜像
# -m dup 元数据做镜像（提高可靠性）

# 创建swap分区
sudo mkswap /dev/sdb2
```

**inode数量规划**：inode在文件系统创建时确定数量，且创建后无法更改。如果inode用尽，即使磁盘空间剩余也无法创建新文件。对于存放大量小文件的场景（如邮件服务器、缓存服务器），创建时需要增大inode密度：

```bash
# 查看inode使用情况
df -i
# Filesystem      Inodes   IUsed   IFree IUse% Mounted on
# /dev/sda1      6553600  123456 6430144    2% /

# 创建时指定更密集的inode（每4KB一个inode，适合大量小文件）
sudo mkfs.ext4 -i 4096 /dev/sdb1
```

#### 8.3.2 文件系统参数调优

创建文件系统后，可以通过`tune2fs`（ext4）、`xfs_admin`（XFS）等工具调整参数：

```bash
# ext4文件系统调优
sudo tune2fs -l /dev/sda1                              # 查看所有参数
sudo tune2fs -m 1 /dev/sda1                            # 保留块比例改为1%（默认5%）
sudo tune2fs -L "new-label" /dev/sda1                  # 修改卷标
sudo tune2fs -U random /dev/sda1                       # 生成新的UUID

# XFS文件系统管理
sudo xfs_admin -l /dev/sdb1                            # 查看卷标
sudo xfs_admin -L "new-label" /dev/sdb1                # 修改卷标
sudo xfs_admin -U generate /dev/sdb1                   # 生成新UUID

# 安全相关的文件系统属性
# ext4的reserved blocks（保留块）
# 默认保留5%给root用户，确保系统在磁盘满时仍能正常运行
# 对于数据盘可以设为0或1%
```

### 8.4 挂载与卸载

#### 8.4.1 挂载的概念

在Linux中，文件系统必须"挂载"到目录树上的某个挂载点才能被访问。这是Linux与Windows的根本区别之一——Windows为每个分区分配盘符（C:、D:），而Linux将所有存储统一到单一的目录树下。

```text
挂载前：
/dev/sdb1  ← 独立的文件系统，有自己的根目录

挂载后：
/mnt/data  ← 挂载点成为 /dev/sdb1 的入口
/mnt/data/file1.txt  ← 实际存储在 /dev/sdb1 上
```

#### 8.4.2 mount与umount

```bash
# 基本挂载
sudo mount /dev/sdb1 /mnt/data

# 指定文件系统类型（通常内核可自动检测）
sudo mount -t ext4 /dev/sdb1 /mnt/data

# 以只读方式挂载（取证分析常用）
sudo mount -o ro /dev/sdb1 /mnt/data

# 安全相关的挂载选项
sudo mount -o nosuid,noexec,nodev /dev/sdb1 /mnt/data
# nosuid  — 忽略SUID/SGID位，防止提权
# noexec — 禁止执行挂载点上的二进制文件
# nodev  — 忽略设备文件，防止创建特殊设备

# 查看所有挂载选项
mount | grep sdb1
# /dev/sdb1 on /mnt/data type ext4 (rw,nosuid,noexec,nodev,relatime)

# 卸载
sudo umount /mnt/data
# 如果提示 "target is busy"，说明有进程在使用该挂载点
sudo umount -l /mnt/data    # 延迟卸载（lazy unmount），等到无引用时真正卸载
sudo fuser -vm /mnt/data    # 查看哪些进程在使用该挂载点
sudo lsof +D /mnt/data      # 另一种方式查看
```

#### 8.4.3 /etc/fstab — 持久化挂载配置

`/etc/fstab`定义了系统启动时自动挂载的文件系统，每行6个字段：

```text
# 设备                                   挂载点      文件系统  选项             dump fsck
UUID=a1b2c3d4-e5f6-7890-abcd-ef1234567890 /           ext4     errors=remount-ro 0     1
UUID=12345678-abcd-ef01-2345-678901234567 /boot       ext4     defaults          0     2
UUID=abcdef01-2345-6789-abcd-ef0123456789 /home       ext4     defaults,nosuid   0     2
UUID=87654321-dcba-fe10-9876-543210fedcba none        swap     sw                0     0
tmpfs                                     /tmp        tmpfs    defaults,noexec,nosuid,nodev 0 0
```

各字段含义：

| 字段 | 含义 | 常用值 |
|------|------|--------|
| 设备 | 设备路径或UUID | 推荐使用UUID（`blkid`查看） |
| 挂载点 | 目录路径 | swap分区填`none` |
| 文件系统 | 类型标识 | ext4, xfs, btrfs, swap, tmpfs, auto |
| 挂载选项 | 逗号分隔的选项列表 | defaults= rw,suid,dev,exec,auto,nouser,async |
| dump | dump备份标志 | 0=不备份, 1=每天备份 |
| fsck | 检查顺序 | 0=不检查, 1=根分区优先, 2=其他分区 |

**安全加固建议**：

```bash
# /tmp 分区安全挂载
tmpfs  /tmp  tmpfs  defaults,noexec,nosuid,nodev,size=2G  0  0

# /home 分区限制SUID
UUID=xxx  /home  ext4  defaults,nosuid  0  2

# /var/tmp 与 /tmp 类似
tmpfs  /var/tmp  tmpfs  defaults,noexec,nosuid,nodev,size=1G  0  0

# 只读挂载关键分区（如 /usr）
UUID=xxx  /usr  ext4  ro,nodev  0  2
```

```bash
# 修改fstab后验证配置是否正确（重要！错误的fstab可能导致系统无法启动）
sudo findmnt --verify
# 输出每一行的错误和警告

# 测试挂载（不实际写入fstab）
sudo mount -a
# 如果没有错误输出，说明fstab配置正确
```

#### 8.4.4 bind挂载与loop设备

**bind挂载**允许将一个目录挂载到另一个位置，创建一个"镜像"视图：

```bash
# 将 /home/user/documents 挂载到 /var/www/documents
sudo mount --bind /home/user/documents /var/www/documents

# 只读bind挂载（容器安全、沙箱场景）
sudo mount --bind /home/user/documents /var/www/documents
sudo mount -o remount,ro,bind /var/www/documents

# bind挂载在容器场景中广泛使用
# Docker的-v参数底层就是bind mount
```

**loop设备**允许将文件作为块设备挂载，常见场景是ISO镜像和加密容器：

```bash
# 挂载ISO镜像
sudo mount -o loop,ro ubuntu-24.04.iso /mnt/iso

# 创建加密的容器文件（安全存储敏感数据）
dd if=/dev/urandom of=~/secret.img bs=1M count=100        # 创建100MB文件
sudo losetup /dev/loop0 ~/secret.img                       # 关联到loop设备
sudo cryptsetup luksFormat /dev/loop0                      # LUKS加密
sudo cryptsetup luksOpen /dev/loop0 secret                  # 解密打开
sudo mkfs.ext4 /dev/mapper/secret                           # 创建文件系统
sudo mount /dev/mapper/secret /mnt/secret                   # 挂载使用

# 使用完毕后安全卸载
sudo umount /mnt/secret
sudo cryptsetup luksClose secret
sudo losetup -d /dev/loop0
```

### 8.5 磁盘空间监控

#### 8.5.1 df — 文件系统级别空间使用

```bash
# 基本用法
df -h
# Filesystem      Size  Used Avail Use% Mounted on
# /dev/sda2        50G   32G   15G  68% /
# /dev/sda1       976M  156M  754M  18% /boot
# tmpfs           3.9G     0  3.9G   0% /dev/shm

# 显示文件系统类型
df -Th

# 只显示特定文件系统类型
df -Th -t ext4
df -Th -x tmpfs        # 排除tmpfs

# 显示inode使用情况
df -iH

# 按可用空间排序（找出空间不足的分区）
df -h | sort -k4 -h
```

#### 8.5.2 du — 目录级别空间使用

```bash
# 查看目录总大小
du -sh /var/log
# 2.1G    /var/log

# 查看子目录大小并排序

# 120K    /var/log/apt
# 1.5G    /var/log/journal
# 200M    /var/log/syslog.1
# 2.1G    /var/log

# 查看当前目录下各子目录大小
du -sh */ | sort -h

# 排除特定目录

# 安全场景：找出占用空间最大的目录
```


### 8.6 文件查找与定位

#### 8.6.1 find — 最强大的文件搜索工具

`find`命令是Linux安全分析的核心工具之一，可以在文件系统上执行复杂的条件搜索：

```bash
# 基本语法
find [搜索路径] [条件] [动作]

# 按名称查找
find / -name "*.conf" -type f 2>/dev/null            # 查找所有.conf文件
find / -iname "passwd" -type f 2>/dev/null            # 不区分大小写

# 按类型查找
find / -type l 2>/dev/null                             # 查找所有符号链接
find / -type s 2>/dev/null                             # 查找所有套接字文件
find / -type p 2>/dev/null                             # 查找所有管道文件

# 按大小查找
find / -size +100M -type f 2>/dev/null                # 大于100MB的文件
find / -size 0 -type f 2>/dev/null                     # 空文件（可能隐藏恶意内容）

# 按时间查找
find /var/log -mtime -7 -type f                        # 最近7天修改的文件
find / -newer /tmp/reference_file -type f 2>/dev/null  # 比参考文件更新的文件
find / -atime +365 -type f 2>/dev/null                 # 超过1年未访问的文件

# 按权限查找（安全分析核心）
find / -perm -4000 -type f 2>/dev/null                 # 查找SUID文件
find / -perm -2000 -type f 2>/dev/null                 # 查找SGID文件
find / -perm -002 -type f 2>/dev/null                  # 全局可写的文件
find / -perm -o+w -type d 2>/dev/null                  # 其他用户可写的目录

# 按属主查找
find / -nouser -o -nogroup 2>/dev/null                 # 无主文件（可能被入侵者创建）
find / -user root -perm -o+w -type f 2>/dev/null       # root所有但其他用户可写的文件

# 组合条件
find / -type f -perm -4000 \( -user root \) -ls 2>/dev/null  # root拥有的SUID文件
find /tmp -type f -mtime -1 -name "*.sh" 2>/dev/null         # /tmp下最近1天的.sh文件
```

**安全扫描实战**：

```bash
# 1. 发现潜在的后门脚本
find /tmp /var/tmp /dev/shm -type f \( -name "*.sh" -o -name "*.py" -o -name "*.pl" \) \
    -mtime -7 2>/dev/null

# 2. 发现异常的cron文件
find /etc/cron* /var/spool/cron -type f 2>/dev/null -exec ls -la {} \;

# 3. 查找隐藏文件（以.开头）
find /home -name ".*" -type f 2>/dev/null
find /root -name ".*" -not -name ".bash*" -type f 2>/dev/null

# 4. 查找Web目录中的可执行文件（Webshell检测）
find /var/www -type f \( -name "*.php" -o -name "*.jsp" -o -name "*.asp" \) \
    -newer /var/www/index.html 2>/dev/null

# 5. 查找不常见的SUID程序
find / -perm -4000 -type f 2>/dev/null | while read f; do
    dpkg -S "$f" 2>/dev/null || echo "NOT IN PACKAGE: $f"
done
# 列出不在任何软件包中的SUID文件，这些可能是攻击者植入的
```

#### 8.6.2 find的-exec与xargs

```bash
# -exec 对每个结果执行命令
find /tmp -name "*.log" -type f -exec rm -f {} \;
# 注意：{} \; 会对每个文件启动一个新进程，效率低

# -exec ... + 批量传递参数（更高效）
find /tmp -name "*.log" -type f -exec rm -f {} +

# 配合xargs（处理大量文件时更高效）
find /tmp -name "*.log" -type f -print0 | xargs -0 rm -f
# -print0 和 -0 处理文件名中的特殊字符（空格、换行等）

# 安全的xargs用法：限制每次传递的参数数量
find / -name "*.bak" -type f -print0 | xargs -0 -n 10 rm -f
```

### 8.7 LVM逻辑卷管理

LVM（Logical Volume Manager）在物理磁盘之上添加了一个抽象层，提供灵活的存储管理能力。在服务器环境中，LVM几乎是标配。

#### 8.7.1 LVM架构

```text
物理磁盘 → 物理卷(PV) → 卷组(VG) → 逻辑卷(LV) → 文件系统 → 挂载点

┌─────────────────────────────────────────────────────┐
│                     卷组 (VG0)                        │
│  ┌──────────┐  ┌──────────┐  ┌────────────────────┐  │
│  │ 物理卷 PV1│  │ 物理卷 PV2│  │     逻辑卷 LV      │  │
│  │ /dev/sda2 │  │ /dev/sdb1 │  │  ┌──────┐ ┌─────┐│  │
│  │           │  │           │  │  │ LV1  │ │ LV2 ││  │
│  │           │  │           │  │  │ /root│ │/home││  │
│  └──────────┘  └──────────┘  │  └──────┘ └─────┘│  │
│                               └────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

#### 8.7.2 LVM基本操作

```bash
# === 创建阶段 ===

# 1. 创建物理卷（PV）
sudo pvcreate /dev/sdb /dev/sdc

# 2. 创建卷组（VG）
sudo vgcreate vg_data /dev/sdb /dev/sdc

# 3. 创建逻辑卷（LV）
sudo lvcreate -L 50G -n lv_web vg_data         # 固定大小50G
sudo lvcreate -l 100%FREE -n lv_data vg_data    # 使用剩余所有空间

# 4. 创建文件系统并挂载
sudo mkfs.ext4 /dev/vg_data/lv_web
sudo mount /dev/vg_data/lv_web /var/www

# === 管理阶段 ===

# 查看状态
sudo pvs         # 物理卷概要
sudo vgs         # 卷组概要
sudo lvs         # 逻辑卷概要
sudo pvdisplay   # 物理卷详细信息
sudo vgdisplay   # 卷组详细信息
sudo lvdisplay   # 逻辑卷详细信息

# 扩展逻辑卷（在线扩展，无需卸载）
sudo lvextend -L +20G /dev/vg_data/lv_web       # 增加20G
sudo resize2fs /dev/vg_data/lv_web               # ext4扩展文件系统
# XFS使用 xfs_growfs /var/www

# 缩小逻辑卷（需要先卸载、检查文件系统）
sudo umount /var/www
sudo e2fsck -f /dev/vg_data/lv_web               # 先检查
sudo resize2fs /dev/vg_data/lv_web 30G            # 缩小文件系统到30G
sudo lvreduce -L 30G /dev/vg_data/lv_web          # 缩小逻辑卷
sudo mount /dev/vg_data/lv_web /var/www
```

#### 8.7.3 LVM快照

LVM快照是数据备份和安全取证的重要工具：

```bash
# 创建快照（COW方式，只记录变化的部分）
sudo lvcreate -L 5G -s -n lv_web_snap /dev/vg_data/lv_web
# -L 快照空间大小（存放变化数据）
# -s 表示创建快照
# -n 快照名称

# 挂载快照（不影响原始卷）
sudo mkdir /mnt/snap
sudo mount /dev/vg_data/lv_web_snap /mnt/snap

# 使用场景：
# 1. 取证分析：在不干扰运行系统的情况下分析文件系统状态
# 2. 备份一致性：对快照进行备份，保证数据一致性
# 3. 安全实验：在快照中测试安全操作，不影响原始数据

# 恢复快照（将原始卷回滚到快照时刻的状态）
sudo umount /var/www
sudo lvconvert --merge /dev/vg_data/lv_web_snap
# merge完成后快照会消失，原始卷恢复到快照时的状态
```

### 8.8 Swap交换空间

交换空间是磁盘上用作虚拟内存扩展的区域。当物理内存不足时，内核将不活跃的内存页换出到交换空间。

#### 8.8.1 Swap分区与Swap文件

```bash
# 方法1：swap分区（推荐用于生产环境）
sudo mkswap /dev/sdb2
sudo swapon /dev/sdb2

# 方法2：swap文件（灵活，无需重新分区）
sudo fallocate -l 4G /swapfile                # 创建4GB文件
# 如果fallocate不可用：
# sudo dd if=/dev/zero of=/swapfile bs=1M count=4096
sudo chmod 600 /swapfile                       # 必须设置权限
sudo mkswap /swapfile                          # 格式化为swap
sudo swapon /swapfile                          # 启用

# 持久化配置（/etc/fstab）
# /swapfile  none  swap  sw  0  0

# 查看swap使用情况
swapon --show
free -h

# 调整swappiness（控制内核使用swap的倾向，0-100）
cat /proc/sys/vm/swappiness                    # 查看当前值（默认60）
sudo sysctl vm.swappiness=10                   # 临时修改
# /etc/sysctl.conf: vm.swappiness=10           # 永久修改
# 值越低，内核越倾向于使用物理内存
# 服务器场景通常设为10-30
```

#### 8.8.2 安全考量

Swap空间可能包含敏感数据的明文副本（密码、密钥等）。在安全敏感场景下：

```bash
# 方案1：加密swap
sudo cryptsetup open --type plain /dev/sdb2 crypt_swap
sudo mkswap /dev/mapper/crypt_swap
sudo swapon /dev/mapper/crypt_swap

# 方案2：临时关闭swap进行安全清理
sudo swapoff /swapfile
sudo dd if=/dev/urandom of=/swapfile bs=1M
sudo mkswap /swapfile
sudo swapon /swapfile

# 方案3：全盘加密（LUKS）自动覆盖swap
# 安装系统时选择加密方案，swap分区也会被加密
```

### 8.10 安全相关的文件系统操作

#### 8.10.1 文件属性（chattr/lsattr）

ext4文件系统提供了一些对安全加固非常有用的文件属性：

```bash
# 查看文件属性
lsattr /etc/passwd
# ----i--------e-- /etc/passwd

# 设置不可变标志（即使是root也无法修改或删除，必须先移除此标志）
sudo chattr +i /etc/passwd
sudo chattr +i /etc/shadow
sudo chattr +i /etc/group

# 设置仅追加标志（日志文件只能追加，不能修改或删除）
sudo chattr +a /var/log/auth.log
sudo chattr +a /var/log/syslog

# 安全加固：保护关键配置
sudo chattr +i /etc/ssh/sshd_config
sudo chattr +i /etc/sudoers

# 取消标志
sudo chattr -i /etc/passwd

# 批量加固脚本示例
for f in /etc/passwd /etc/shadow /etc/gshadow /etc/group; do
    sudo chattr +i "$f"
done
```

**安全提示**：攻击者在获取root权限后通常会尝试修改`/etc/passwd`和`/etc/shadow`来创建后门账户。设置`+i`标志可以在一定程度上增加攻击难度（但不是万能的，攻击者可以先`chattr -i`再修改，除非你也保护了`chattr`命令本身）。

#### 8.10.2 文件完整性监控

```bash
# 使用debsums验证已安装软件包的文件完整性
sudo apt install debsums
sudo debsums -c                      # 检查所有已安装包的文件
# 输出被修改的文件列表

# 使用rpm -V验证（RHEL系）
rpm -Va                              # 验证所有已安装RPM包
# 输出中：
# S = 大小变化  M = 权限变化  5 = MD5变化  D = 设备号变化
# L = 链接变化  U = 属主变化  G = 属组变化  T = 时间变化
```

### 8.11 常见错误与安全陷阱

#### 8.11.1 典型操作失误

| 错误操作 | 后果 | 预防措施 |
|----------|------|----------|
| 格式化错误的磁盘 | 数据永久丢失 | 操作前用`lsblk`和`blkid`确认设备 |
| fstab配置错误 | 系统无法启动 | 修改前备份fstab；用`findmnt --verify`验证 |
| 不检查SMART状态 | 磁盘故障导致数据丢失 | 定期`smartctl -H`检查 |
| 在挂载状态下格式化 | 数据损坏 | 先`umount`再操作 |
| 忘记`chmod 600 /swapfile` | swap文件权限过宽 | 创建swap文件后立即设置权限 |
| 扩容前不备份 | 扩容失败导致数据丢失 | 扩容前创建LVM快照 |

#### 8.11.2 安全陷阱

```bash
# 陷阱1：/tmp目录没有noexec挂载
# 攻击者可以在/tmp下编译并执行恶意程序
# 修复：
# /etc/fstab: tmpfs /tmp tmpfs defaults,noexec,nosuid,nodev 0 0

# 陷阱2：全局可写的目录缺乏Sticky Bit
# 没有sticky bit的可写目录中，用户可以删除其他用户的文件
ls -ld /tmp
# drwxrwxrwt 1 root root 4096 ...  /tmp   ← t表示sticky bit已设置
# 如果没有t：
sudo chmod +t /shared_directory

# 陷阱3：未监控的磁盘空间
# 磁盘满会导致日志停止记录、数据库崩溃、系统服务异常
# 定期监控：
df -h | awk '$5+0 > 80 {print "WARNING: " $6 " is " $5 " full"}'

# 陷阱4：未加密的swap可能泄露敏感数据
# 内存中的密码、密钥可能被换出到swap
# 修复：使用加密swap或关闭swap

# 陷阱5：SUID文件列表未定期审查
# 攻击者可能植入新的SUID程序实现持久化
# 建立基线并定期对比：
find / -perm -4000 -type f 2>/dev/null | sort > /tmp/suid-$(date +%Y%m%d).txt
diff /tmp/suid-baseline.txt /tmp/suid-$(date +%Y%m%d).txt
```
