# DDD实战案例：从领域发现到代码落地

本章通过三个递进的实战案例，展示如何将前两章介绍的DDD理论与技巧应用于真实业务场景。每个案例都遵循完整的工作流程：从领域对话与模型发现，到限界上下文划分、聚合设计，再到代码实现与演进。案例涵盖电商交易、银行金融和事件驱动微服务三大典型领域，难度逐步递增。

> **阅读建议**：案例一完整展示了DDD的全流程，适合入门理解；案例二聚焦金融领域的复杂业务规则与Saga模式；案例三深入CQRS和Event Sourcing的高级架构。建议按顺序阅读。

## 案例一：电商平台的DDD建模

### 1.1 业务背景与痛点

某中型B2C电商平台（日均订单量5万+），随着业务快速扩展，团队面临以下典型困境：

**代码层面**：
- `OrderService` 类膨胀到3000+行，承载了下单、支付回调、库存扣减、物流分配等全部逻辑
- 同一个"Product"概念在商品、库存、定价、推荐四个模块中含义不同，但被强制塞进一个数据库表
- 修改一个业务规则（如"满99包邮"）需要同时改动Controller、Service、DAO三层代码，牵一发而动全身

**团队层面**：
- 新成员需要3个月才能理解核心业务流程
- 开发与产品对"订单状态"的理解不一致——产品说"已支付"时，代码里对应的是 `status=2`，而"已发货"是 `status=5`，中间状态含义模糊
- 每次大促前的"压测-修bug"循环耗时耗力，根本原因是业务逻辑散布导致无法准确评估影响范围

**决策**：引入DDD，从电商核心交易链路开始重构。

### 1.2 领域发现：事件风暴工作坊

团队组织了一次为期两天的Event Storming工作坊，参与人员包括：2名产品经理、1名业务运营、3名后端开发、1名架构师。

> **工作坊准备**：提前收集了现有系统的操作手册、API文档和近半年的线上事故报告，作为领域知识的输入素材。

**第一步：列出领域事件（Domain Events）**

团队用橙色便利贴在墙上列出所有业务事件：

客户已注册 → 商品已上架 → 购物车已更新 → 订单已创建 →
库存已预留 → 订单已支付 → 支付已确认 → 库存已扣减 →
发货单已生成 → 商品已发货 → 客户已签收 → 订单已完成 →
退款已申请 → 退款已处理 → 优惠券已发放 → 评价已提交

**第二步：识别命令（Commands）与聚合（Aggregates）**

每个事件前面贴上触发它的命令，命令旁边贴上执行命令的聚合：

[注册客户] → 客户已注册           → 聚合：Customer
[上架商品] → 商品已上架           → 聚合：Product
[添加到购物车] → 购物车已更新      → 聚合：Cart
[创建订单] → 订单已创建           → 聚合：Order
[预留库存] → 库存已预留           → 聚合：Inventory
[支付订单] → 订单已支付           → 聚合：Payment
...

**第三步：识别限界上下文（Bounded Contexts）**

通过分析聚合之间的耦合关系和团队组织结构，团队识别出五个限界上下文：

| 限界上下文 | 核心聚合 | 职责范围 | 对应团队 |
|------------|---------|---------|---------|
| 商品目录（Catalog） | Product, Category, Brand | 商品信息管理、搜索、展示 | 商品团队 |
| 库存（Inventory） | Stock, Warehouse, StockReservation | 库存数量管理、预留、扣减 | 供应链团队 |
| 订单（Order） | Order, OrderLine, Cart | 订单生命周期管理 | 交易团队 |
| 支付（Payment） | Payment, Refund, PaymentMethod | 支付处理、退款 | 支付团队 |
| 客户（Customer） | Customer, Address, Coupon | 客户信息、地址、优惠券 | 用户团队 |

**关键洞察**：同一个"Product"在不同上下文中的模型完全不同：

商品目录上下文的Product：     库存上下文的Product：       订单上下文的Product：
├── name (String)           ├── sku (String)          ├── productId (ProductId)
├── description (Text)      ├── warehouseId (Id)      ├── name (String)
├── images (List<URL>)      ├── quantity (int)        ├── unitPrice (Money)
├── categoryId (Id)         ├── safetyStock (int)     └── quantity (int)
├── brandId (Id)            └── location (String)
└── status (enum)

> **为什么同一个概念要建不同的模型？** 因为不同上下文关注的属性不同。商品目录关注展示信息，库存关注数量和仓位，订单关注下单时刻的快照。强制统一模型会导致任何一个上下文的变更都影响其他上下文——这正是原系统的核心痛点。

### 1.3 上下文映射（Context Map）

团队绘制了上下文之间的集成关系：

                    ┌─────────────┐
                    │   客户上下文  │
                    │  (Customer)  │
                    └──────┬──────┘
                           │ Customer-Supplier
                           │ (提供客户信息)
                    ┌──────▼──────┐
┌─────────────┐    │   订单上下文  │    ┌─────────────┐
│  商品目录    │◄───│   (Order)    │───►│    支付      │
│  (Catalog)  │ACL │              │OHS │  (Payment)   │
└─────────────┘    └──────┬──────┘    └─────────────┘
                          │ Partnership
                   ┌──────▼──────┐
                   │   库存上下文  │
                   │ (Inventory)  │
                   └─────────────┘

集成模式说明：
- **订单 ↔ 库存**：Partnership（合作关系），双方共同定义接口，库存预留/释放与订单状态强关联
- **订单 → 商品目录**：Anticorruption Layer（防腐层），订单上下文通过ACL获取商品快照信息，避免被商品目录模型污染
- **订单 → 支付**：Open Host Service（开放主机服务），支付上下文提供标准化支付API
- **客户 → 订单**：Customer-Supplier（客户-供应商），客户上下文是上游，订单上下文消费客户信息

### 1.4 聚合设计详解：Order聚合

**业务不变量分析**：

通过领域对话，团队识别出订单聚合的以下不变量：
1. 订单总金额 = 所有订单行小计之和（强一致性）
2. 已取消的订单不能修改（强一致性）
3. 单笔订单行数不超过50（强一致性）
4. 订单状态流转遵循固定路径：CREATED → CONFIRMED → PAID → SHIPPED → COMPLETED

**错误设计（大聚合）**：

```java
// ❌ 反模式：大聚合——将所有相关概念塞进一个聚合
class Order {
    OrderId id;
    Customer customer;           // 直接引用Customer聚合——违反聚合间ID引用原则
    List<OrderLine> lines;
    Payment payment;             // 直接引用Payment聚合——支付有自己的事务边界
    Shipment shipment;           // 直接引用Shipment聚合——物流是独立领域
    Coupon coupon;               // 直接引用优惠券——优惠券有独立生命周期
    Address shippingAddress;     // 收货地址——这里放是对的（值对象）
}
```

这个设计的问题：
- Order聚合太大，包含了Payment、Shipment等不属于同一事务的内容
- 并发冲突严重：任何订单相关操作都会锁住整个聚合
- 支付失败回滚需要同时操作Order和Payment，违反"一事务一聚合"原则
- 修改支付逻辑可能意外影响订单逻辑，违反单一职责

**正确设计（小聚合 + 领域事件）**：

```java
// ✅ Order聚合根——只包含订单自身的业务不变量
public class Order {
    private final OrderId id;
    private CustomerId customerId;          // 通过ID引用，不是对象引用
    private List<OrderLine> lines;           // 聚合内部对象
    private Money totalAmount;               // 值对象
    private Address shippingAddress;         // 值对象（收货地址快照）
    private OrderStatus status;              // 值对象
    private int version;                     // 乐观锁版本号

    private final List<DomainEvent> domainEvents = new ArrayList<>();

    /**
     * 创建订单——领域行为方法
     */
    public static Order create(CustomerId customerId, Address address) {
        Order order = new Order();
        order.id = OrderId.generate();
        order.customerId = customerId;
        order.shippingAddress = address;     // 创建时快照地址
        order.status = OrderStatus.CREATED;
        order.lines = new ArrayList<>();
        order.totalAmount = Money.ZERO(Currency.CNY);

        // 发布领域事件
        order.domainEvents.add(new OrderCreated(order.id, customerId, Instant.now()));
        return order;
    }

    /**
     * 添加订单行——封装业务不变量
     */
    public void addLine(ProductId productId, String productName,
                        Money unitPrice, int quantity) {
        // 不变量1：只有CREATED状态可以添加
        if (status != OrderStatus.CREATED) {
            throw new OrderNotModifiableException(id, status);
        }
        // 不变量2：不超过50行
        if (lines.size() >= 50) {
            throw new TooManyOrderLinesException(id, lines.size());
        }

        OrderLine line = OrderLine.create(productId, productName, unitPrice, quantity);
        lines.add(line);
        recalculateTotal();

        domainEvents.add(new OrderLineAdded(id, productId, quantity));
    }

    /**
     * 确认订单——触发库存预留
     */
    public void confirm() {
        if (status != OrderStatus.CREATED) {
            throw new OrderNotModifiableException(id, status);
        }
        if (lines.isEmpty()) {
            throw new EmptyOrderException(id);
        }
        status = OrderStatus.CONFIRMED;
        domainEvents.add(new OrderConfirmed(id, totalAmount, lines.size()));
    }

    /**
     * 标记已支付——由支付上下文通过领域事件触发
     */
    public void markAsPaid(PaymentId paymentId) {
        if (status != OrderStatus.CONFIRMED) {
            throw new InvalidOrderStateException(id, status, OrderStatus.CONFIRMED);
        }
        status = OrderStatus.PAID;
        domainEvents.add(new OrderPaid(id, paymentId, totalAmount));
    }

    private void recalculateTotal() {
        this.totalAmount = lines.stream()
            .map(OrderLine::subtotal)
            .reduce(Money.ZERO(Currency.CNY), Money::add);
    }

    public List<DomainEvent> pullDomainEvents() {
        List<DomainEvent> events = new ArrayList<>(domainEvents);
        domainEvents.clear();
        return events;
    }
}
```

**OrderLine值对象**：

```java
// 值对象：不可变，自验证
public final class OrderLine {
    private final ProductId productId;
    private final String productName;        // 商品名称快照——下单时刻的名称
    private final Money unitPrice;            // 单价快照——下单时刻的价格
    private final int quantity;

    private OrderLine(ProductId productId, String productName,
                      Money unitPrice, int quantity) {
        if (quantity <= 0 || quantity > 999) {
            throw new InvalidQuantityException(quantity);
        }
        this.productId = productId;
        this.productName = productName;
        this.unitPrice = unitPrice;
        this.quantity = quantity;
    }

    public static OrderLine create(ProductId productId, String productName,
                                    Money unitPrice, int quantity) {
        return new OrderLine(productId, productName, unitPrice, quantity);
    }

    public Money subtotal() {
        return unitPrice.multiply(quantity);
    }

    // equals/hashCode基于所有属性——值对象的相等性由属性值决定
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof OrderLine)) return false;
        OrderLine that = (OrderLine) o;
        return quantity == that.quantity
            &amp;&amp; Objects.equals(productId, that.productId)
            &amp;&amp; Objects.equals(productName, that.productName)
            &amp;&amp; Objects.equals(unitPrice, that.unitPrice);
    }

    @Override
    public int hashCode() {
        return Objects.hash(productId, productName, unitPrice, quantity);
    }
}
```

> **设计要点：为什么快照价格和名称？** 商品名称和价格会随时间变化。如果订单直接引用商品的当前价格，历史订单的金额会"漂移"。OrderLine在创建时快照商品信息，确保订单数据的不变性——这是电商系统的基本要求。

**Payment聚合**（独立聚合，通过领域事件与Order协作）：

```java
// Payment聚合根——独立的事务边界
public class Payment {
    private final PaymentId id;
    private OrderId orderId;          // 通过ID引用Order
    private Money amount;
    private PaymentStatus status;
    private PaymentMethod method;
    private Instant createdAt;
    private Instant paidAt;

    private final List<DomainEvent> domainEvents = new ArrayList<>();

    public static Payment create(OrderId orderId, Money amount, PaymentMethod method) {
        Payment payment = new Payment();
        payment.id = PaymentId.generate();
        payment.orderId = orderId;
        payment.amount = amount;
        payment.method = method;
        payment.status = PaymentStatus.PENDING;
        payment.createdAt = Instant.now();
        return payment;
    }

    /**
     * 确认支付成功
     */
    public void confirmPayment(Instant paidAt) {
        if (status != PaymentStatus.PENDING) {
            throw new PaymentAlreadyProcessedException(id, status);
        }
        this.status = PaymentStatus.COMPLETED;
        this.paidAt = paidAt;
        // 发布事件——订单上下文监听此事件来更新订单状态
        domainEvents.add(new PaymentConfirmed(id, orderId, amount, paidAt));
    }

    /**
     * 退款
     */
    public void refund(Money refundAmount, String reason) {
        if (status != PaymentStatus.COMPLETED) {
            throw new PaymentNotRefundableException(id, status);
        }
        if (refundAmount.isGreaterThan(amount)) {
            throw new RefundExceedsPaymentException(refundAmount, amount);
        }
        this.status = PaymentStatus.REFUNDED;
        domainEvents.add(new PaymentRefunded(id, orderId, refundAmount, reason));
    }
}
```

### 1.5 领域事件驱动的跨聚合协作

Order和Payment是两个独立聚合，但需要协作完成"下单-支付"流程。通过领域事件实现最终一致性：

```java
// 订单确认应用服务
@Service
public class OrderApplicationService {
    private final OrderRepository orderRepo;
    private final InventoryService inventoryService;
    private final DomainEventPublisher eventPublisher;

    @Transactional
    public OrderId confirmOrder(OrderId orderId, List<OrderItemRequest> items) {
        // 1. 加载订单聚合
        Order order = orderRepo.findById(orderId);

        // 2. 调用领域行为
        for (OrderItemRequest item : items) {
            order.addLine(item.productId(), item.name(),
                         item.unitPrice(), item.quantity());
        }
        order.confirm();

        // 3. 持久化
        orderRepo.save(order);

        // 4. 发布领域事件（触发库存预留等后续流程）
        for (DomainEvent event : order.pullDomainEvents()) {
            eventPublisher.publish(event);
        }

        return orderId;
    }
}

// 库存上下文的事件监听器
@Component
public class OrderEventHandler {
    private final InventoryService inventoryService;

    @EventListener
    public void onOrderConfirmed(OrderConfirmed event) {
        // 收到订单确认事件，预留库存
        inventoryService.reserveStock(event.orderId(), event.quantity());
    }

    @EventListener
    public void onOrderCancelled(OrderCancelled event) {
        // 订单取消，释放预留库存
        inventoryService.releaseStock(event.orderId());
    }
}
```

### 1.6 防腐层（ACL）的应用

订单上下文需要商品信息（名称、价格），但不应依赖商品目录的内部模型。防腐层负责翻译：

```java
// 防腐层接口——定义在订单上下文中
public interface ProductQueryFacade {
    ProductSnapshot getProductInfo(ProductId productId);
}

// 值对象：订单上下文的商品快照
public final class ProductSnapshot {
    private final ProductId productId;
    private final String name;
    private final Money currentPrice;
    private final String imageUrl;

    // 只包含订单需要的信息，不暴露商品目录内部模型
}

// 防腐层实现——调用商品目录的API并翻译
@Component
public class CatalogAntiCorruptionLayer implements ProductQueryFacade {
    private final CatalogHttpClient catalogClient;  // 调用商品目录HTTP API

    @Override
    public ProductSnapshot getProductInfo(ProductId productId) {
        // 1. 调用上游API
        CatalogProductResponse upstream = catalogClient.getProduct(productId.value());

        // 2. 翻译为本上下文的模型
        return new ProductSnapshot(
            new ProductId(upstream.getId()),
            upstream.getTitle(),
            Money.of(upstream.getPrice(), Currency.CNY),
            upstream.getMainImageUrl()
        );
    }
}
```

> **ACL的防护价值**：假设商品目录团队将来把 `title` 字段改名为 `displayName`，或者把价格单位从"分"改为"元"，只有ACL这一个文件需要修改——订单上下文的代码完全不受影响。这就是防腐层的隔离作用。

### 1.7 仓储实现

```java
// Order仓储接口（定义在领域层）
public interface OrderRepository {
    Order findById(OrderId id);
    void save(Order order);
    List<Order> findByCustomerId(CustomerId customerId);
    List<Order> findByStatus(OrderStatus status);
}

// JPA实现（基础设施层）
@Repository
public class JpaOrderRepository implements OrderRepository {
    private final OrderJpaRepository jpaRepo;
    private final OrderMapper mapper;

    @Override
    public Order findById(OrderId id) {
        return jpaRepo.findById(id.value())
            .map(mapper::toDomain)
            .orElseThrow(() -> new OrderNotFoundException(id));
    }

    @Override
    public void save(Order order) {
        OrderJpaEntity entity = mapper.toEntity(order);
        jpaRepo.save(entity);
    }

    @Override
    public List<Order> findByCustomerId(CustomerId customerId) {
        return jpaRepo.findByCustomerId(customerId.value())
            .stream()
            .map(mapper::toDomain)
            .collect(Collectors.toList());
    }

    @Override
    public List<Order> findByStatus(OrderStatus status) {
        return jpaRepo.findByStatus(status.name())
            .stream()
            .map(mapper::toDomain)
            .collect(Collectors.toList());
    }
}
```

### 1.9 案例一常见陷阱

| 陷阱 | 表现 | 纠正方法 |
|------|------|---------|
| 聚合设计过大 | Order包含Payment、Shipment等 | 只保留同一事务边界内的对象，跨聚合用事件 |
| 忽略值对象 | 所有东西都建Entity，有独立ID | 不需要独立标识、按属性比较的概念用值对象 |
| 领域事件滥用 | 每个字段修改都发事件 | 只发有业务含义的事件（状态变更、关键行为） |
| 仓储泄漏实现 | Repository接口里写SQL | 接口定义在领域层，实现在基础设施层 |
| 快照遗漏 | 订单引用商品实时价格 | 下单时快照名称和价格到OrderLine |

## 案例二：银行转账系统的DDD建模

### 2.1 业务背景

银行核心系统需要支持账户间转账、跨行转账、批量代发工资等业务。传统设计将所有逻辑放在一个"转账服务"中，导致：
- 转账逻辑与账户管理、风控、清算等耦合
- 跨行转账与行内转账走同一代码路径，但实际上业务规则完全不同
- 对账和审计难以追溯每笔交易的完整上下文

### 2.2 领域发现：识别核心业务规则

团队通过与银行业务专家的多次领域对话，梳理出核心不变量：

**账户不变量**：
1. 可用余额 = 当前余额 - 冻结金额（强一致性）
2. 扣款前必须验证：账户状态、余额充足、日累计限额
3. 不同账户类型有不同的日转账限额

**转账不变量**：
1. 转账金额必须大于零且不超过限额
2. 行内转账：扣款和入账在同一事务中（或通过Saga保证最终一致性）
3. 跨行转账：资金先冻结，等清算回执后再实际扣款

**对账不变量**：
1. 每笔转账必须有唯一的审计追踪
2. 转账状态变更必须记录时间和原因

### 2.4 账户聚合设计

```java
// 账户聚合根
public class Account {
    private final AccountId id;
    private AccountType type;               // SAVINGS / CHECKING / CREDIT
    private CustomerId customerId;
    private Money balance;                  // 当前余额
    private Money availableBalance;         // 可用余额（扣除冻结金额）
    private Money frozenAmount;             // 冻结金额
    private AccountStatus status;           // ACTIVE / FROZEN / CLOSED
    private int dailyTransferCount;         // 当日转账次数
    private Money dailyTransferTotal;       // 当日转账累计金额
    private LocalDate lastTransferDate;     // 最后转账日期

    private final List<DomainEvent> domainEvents = new ArrayList<>();

    /**
     * 业务不变量：扣款前验证
     */
    public void debit(Money amount, String purpose) {
        // 不变量1：账户必须处于活跃状态
        if (status != AccountStatus.ACTIVE) {
            throw new AccountNotActiveException(id, status);
        }
        // 不变量2：余额充足（可用余额 >= 扣款金额）
        if (availableBalance.isLessThan(amount)) {
            throw new InsufficientBalanceException(id, availableBalance, amount);
        }
        // 不变量3：日累计转账限额
        if (dailyTransferTotal.add(amount).isGreaterThan(getDailyLimit())) {
            throw new DailyTransferLimitExceededException(id, dailyTransferTotal, amount);
        }

        balance = balance.subtract(amount);
        availableBalance = availableBalance.subtract(amount);
        dailyTransferCount++;
        dailyTransferTotal = dailyTransferTotal.add(amount);
        lastTransferDate = LocalDate.now();

        domainEvents.add(new AccountDebited(id, amount, purpose, Instant.now()));
    }

    /**
     * 业务不变量：入账
     */
    public void credit(Money amount, String purpose) {
        if (status == AccountStatus.CLOSED) {
            throw new AccountNotActiveException(id, status);
        }

        balance = balance.add(amount);
        availableBalance = availableBalance.add(amount);

        domainEvents.add(new AccountCredited(id, amount, purpose, Instant.now()));
    }

    /**
     * 冻结资金——跨行转账时先冻结，等清算确认后再实际扣款
     */
    public void freeze(Money amount) {
        if (status != AccountStatus.ACTIVE) {
            throw new AccountNotActiveException(id, status);
        }
        if (availableBalance.isLessThan(amount)) {
            throw new InsufficientBalanceException(id, availableBalance, amount);
        }
        availableBalance = availableBalance.subtract(amount);
        frozenAmount = frozenAmount.add(amount);

        domainEvents.add(new AccountFrozen(id, amount, Instant.now()));
    }

    /**
     * 解冻资金——转账失败或取消时恢复可用余额
     */
    public void unfreeze(Money amount) {
        if (frozenAmount.isLessThan(amount)) {
            throw new UnfreezeExceedsFrozenException(id, frozenAmount, amount);
        }
        availableBalance = availableBalance.add(amount);
        frozenAmount = frozenAmount.subtract(amount);

        domainEvents.add(new AccountUnfrozen(id, amount, Instant.now()));
    }

    /**
     * 冻结转扣款——冻结资金确认为实际扣款（清算回执到达后）
     */
    public void confirmFrozenDebit(Money amount, String clearingId) {
        if (frozenAmount.isLessThan(amount)) {
            throw new UnfreezeExceedsFrozenException(id, frozenAmount, amount);
        }
        balance = balance.subtract(amount);
        frozenAmount = frozenAmount.subtract(amount);
        dailyTransferCount++;
        dailyTransferTotal = dailyTransferTotal.add(amount);
        lastTransferDate = LocalDate.now();

        domainEvents.add(new AccountDebited(id, amount, "跨行转账-" + clearingId, Instant.now()));
    }

    /**
     * 日重置——每日凌晨清零当日转账计数
     */
    public void resetDailyCounters() {
        dailyTransferCount = 0;
        dailyTransferTotal = Money.ZERO(Currency.CNY);
        lastTransferDate = LocalDate.now();
    }

    private Money getDailyLimit() {
        return switch (type) {
            case SAVINGS -> Money.of(500000, Currency.CNY);    // 储蓄卡50万/日
            case CHECKING -> Money.of(1000000, Currency.CNY);  // 结算户100万/日
            case CREDIT -> Money.of(200000, Currency.CNY);     // 信用卡20万/日
        };
    }
}
```

> **设计要点：余额、可用余额与冻结金额的三字段设计**。在银行系统中，`balance`（当前余额）和`availableBalance`（可用余额）是两个不同概念。冻结资金时，`balance`不变但`availableBalance`减少；实际扣款时，`balance`才减少。这保证了冻结期间账户的"账面余额"不变，但"可操作余额"已减少，防止重复使用。

### 2.5 行内转账领域服务

```java
// 领域服务：行内转账（两个账户在同一事务中完成余额变更）
public class TransferService {
    private final AccountRepository accountRepo;
    private final TransactionRepository transactionRepo;
    private final DomainEventPublisher eventPublisher;

    /**
     * 行内转账——同一银行内两个账户的余额变更
     *
     * 设计要点：
     * 1. 同一事务中操作两个账户，保证强一致性
     * 2. 使用悲观锁（SELECT FOR UPDATE）防止并发扣款
     * 3. 交易记录提供完整审计信息
     */
    @Transactional
    public TransferResult transfer(AccountId fromId, AccountId toId,
                                    Money amount, String remark) {
        // 1. 加载两个账户（按固定顺序加锁，防止死锁）
        Account from = accountRepo.findByIdWithLock(fromId);
        Account to = accountRepo.findByIdWithLock(toId);

        // 确保固定的加锁顺序（按AccountId排序），避免死锁
        if (fromId.compareTo(toId) > 0) {
            Account temp = from;
            from = to;
            to = temp;
        }

        // 2. 基本验证
        if (fromId.equals(toId)) {
            throw new SameAccountTransferException(fromId);
        }

        // 3. 执行扣款和入账（会触发余额不足等异常，自动回滚）
        from.debit(amount, "转账支出-至" + toId);
        to.credit(amount, "转账收入-来自" + fromId);

        // 4. 创建交易记录（独立聚合，记录完整审计信息）
        Transaction tx = Transaction.create(
            TransactionType.TRANSFER,
            fromId, toId, amount, remark
        );
        transactionRepo.save(tx);

        // 5. 持久化账户变更
        accountRepo.save(from);
        accountRepo.save(to);

        // 6. 发布领域事件
        for (DomainEvent event : from.pullDomainEvents()) {
            eventPublisher.publish(event);
        }
        for (DomainEvent event : to.pullDomainEvents()) {
            eventPublisher.publish(event);
        }
        eventPublisher.publish(new TransferCompleted(tx.getId(), fromId, toId, amount));

        return TransferResult.success(tx.getId(), amount);
    }
}
```

> **为什么行内转账用单事务，跨行转账用Saga？** 行内转账的两个账户在同一数据库中，可以用单事务保证强一致性——扣款失败时整个事务回滚，不会出现"一边扣了一边没到"的情况。但跨行转账涉及外部银行系统，无法放在同一事务中，必须用Saga模式实现最终一致性。

### 2.6 跨行转账的Saga模式

跨行转账涉及外部银行系统，可能失败，需要Saga模式编排：

```java
/**
 * 跨行转出Saga——分步执行，每步可补偿
 *
 * 正向流程：冻结资金 → 发送跨行请求 → 等待清算回执 → 确认扣款
 * 补偿流程：解冻资金 ← 撤销请求 ← 等待超时 ← 冲正处理
 */
@Service
public class OutboundTransferSaga {
    private final AccountRepository accountRepo;
    private final TransferRequestRepository requestRepo;
    private final ClearingGateway clearingGateway;
    private final SagaLogRepository sagaLogRepo;

    /**
     * 执行跨行转账Saga
     */
    public SagaResult execute(TransferCommand command) {
        SagaLog saga = SagaLog.start("OUTBOUND_TRANSFER", command);

        try {
            // 步骤1：冻结资金
            freezeFunds(command.fromId(), command.amount(), saga);

            // 步骤2：发送跨行请求
            TransferRequestId requestId = sendOutboundRequest(
                command.fromId(), command.toId(),
                command.amount(), command.targetBank(), saga
            );

            // 步骤3：等待清算回执（异步，通过回调或轮询）
            // 此处返回PENDING状态，由后续回调触发confirmDebit
            return SagaResult.pending(saga.getId(), requestId);

        } catch (Exception e) {
            // 任何步骤失败，执行补偿
            compensate(saga, e.getMessage());
            return SagaResult.failed(saga.getId(), e.getMessage());
        }
    }

    /**
     * 步骤1：冻结资金
     */
    private void freezeFunds(AccountId fromId, Money amount, SagaLog saga) {
        Account from = accountRepo.findById(fromId);
        from.freeze(amount);                    // 将可用余额冻结
        accountRepo.save(from);
        saga.recordStep("FREEZE_FUNDS", "SUCCESS");
    }

    /**
     * 步骤2：发送跨行请求
     */
    private TransferRequestId sendOutboundRequest(AccountId fromId, AccountId toId,
                                                   Money amount, BankCode targetBank,
                                                   SagaLog saga) {
        TransferRequest request = TransferRequest.create(
            fromId, toId, amount, targetBank
        );
        requestRepo.save(request);

        // 调用央行大小额系统/网联
        clearingGateway.sendTransfer(request);

        saga.recordStep("SEND_OUTBOUND_REQUEST", "SUCCESS");
        return request.getId();
    }

    /**
     * 清算回执回调——确认扣款
     */
    @Transactional
    public void confirmDebit(SagaId sagaId, ClearingReceipt receipt) {
        SagaLog saga = sagaLogRepo.findById(sagaId);
        TransferCommand command = saga.getCommand();

        Account from = accountRepo.findById(command.fromId());
        from.confirmFrozenDebit(command.amount(), receipt.getClearingId());
        accountRepo.save(from);

        saga.recordStep("CONFIRM_DEBIT", "SUCCESS");
        saga.complete();
        sagaLogRepo.save(saga);
    }

    /**
     * 补偿流程——转账失败时，按逆序恢复已执行的步骤
     */
    private void compensate(SagaLog saga, String failureReason) {
        try {
            if (saga.hasCompletedStep("FREEZE_FUNDS")) {
                Account from = accountRepo.findById(saga.getCommand().fromId());
                from.unfreeze(saga.getCommand().amount());
                accountRepo.save(from);
                saga.recordStep("UNFREEZE_FUNDS", "SUCCESS");
            }
        } catch (Exception compensateError) {
            // 补偿失败需要人工介入——记录到告警系统
            saga.recordStep("COMPENSATION_FAILED", compensateError.getMessage());
            alertService.critical("跨行转账补偿失败，需要人工处理", saga.getId());
        }

        saga.recordCompensation(failureReason);
        sagaLogRepo.save(saga);
    }

    /**
     * 超时处理——跨行请求长时间未收到回执
     */
    @Scheduled(fixedDelay = 60000)
    public void handleTimeout() {
        List<SagaLog> pendingSagas = sagaLogRepo.findPendingTimeout(Duration.ofMinutes(30));
        for (SagaLog saga : pendingSagas) {
            // 尝试查询清算状态
            ClearingStatus status = clearingGateway.queryStatus(saga.getRequestId());
            if (status == ClearingStatus.FAILED) {
                compensate(saga, "清算超时失败");
            } else if (status == ClearingStatus.COMPLETED) {
                confirmDebit(saga.getId(), status.getReceipt());
            }
            // 仍在处理中的不处理，等待下次检查
        }
    }
}
```

> **Saga补偿的幂等性**：在分布式环境中，补偿操作可能被执行多次（网络重试、超时重发）。`unfreeze`操作必须是幂等的——如果已经解冻过，再次调用不能重复解冻。实现方式是在补偿操作前检查状态，或使用幂等键去重。

### 2.7 风控上下文的集成

```java
/**
 * 风控服务——作为交易上下文的上游，提供交易前的风险评估
 */
@Service
public class RiskControlService {

    /**
     * 交易前风控检查（同步调用）
     *
     * 检查规则：
     * 1. 大额交易（>5万）需要额外验证
     * 2. 异地交易检测
     * 3. 高频交易检测
     * 4. 黑名单检查
     */
    public RiskAssessment evaluate(TransactionContext context) {
        RiskLevel level = RiskLevel.LOW;
        List<String> reasons = new ArrayList<>();

        // 大额检查
        if (context.getAmount().isGreaterThan(Money.of(50000, Currency.CNY))) {
            level = RiskLevel.MEDIUM;
            reasons.add("大额交易: " + context.getAmount());
        }

        // 异地检测
        if (!context.getFromCity().equals(context.getToCity())) {
            level = RiskLevel.MEDIUM;
            reasons.add("异地转账: " + context.getFromCity() + " → " + context.getToCity());
        }

        // 高频检测
        if (context.getRecentTransferCount() > 10) {
            level = RiskLevel.HIGH;
            reasons.add("高频交易: 最近1小时" + context.getRecentTransferCount() + "笔");
        }

        // 黑名单
        if (blacklistRepo.exists(context.getToAccountId())) {
            return RiskAssessment.blocked("目标账户在黑名单中");
        }

        return new RiskAssessment(level, reasons);
    }
}
```

### 2.8 案例二常见陷阱

| 陷阱 | 表现 | 纠正方法 |
|------|------|---------|
| 事务边界错误 | 行内转账用Saga，跨行转账用单事务 | 同库用事务，跨系统用Saga |
| 死锁 | 两个事务互相等待对方的锁 | 按固定顺序（如AccountId排序）加锁 |
| 补偿不幂等 | 超时重试导致重复解冻 | 补偿前检查状态，或使用幂等键 |
| 忽略冻结状态 | 直接扣款导致并发超扣 | 区分冻结金额和可用余额 |
| 补偿失败无告警 | Saga补偿失败后静默丢失 | 记录到告警系统，人工介入 |

## 案例三：事件驱动微服务中的DDD

### 3.1 业务背景与痛点

在线教育平台，包含课程管理、学员管理、订单支付、学习进度、证书发放等子系统。

**系统痛点**：
- 课程详情页加载慢：一个页面需要查询课程信息、讲师信息、学员评价、学习进度等5张表，每次大促前的促销活动导致查询量激增
- 学习进度统计不准确：进度数据分散在多个服务中，汇总时经常出现不一致
- 证书发放逻辑与课程、订单强耦合，修改课程发布规则可能意外影响证书逻辑

**架构选型**：团队决定采用DDD + CQRS + Event Sourcing架构，核心理由：
1. 课程发布是核心业务流程，业务规则复杂（至少3章、每章有内容、定价合理）
2. 读写比例严重不对等（读:写 ≈ 100:1），CQRS可以独立优化查询
3. 学习进度需要完整的操作历史，Event Sourcing天然提供审计追踪

### 3.2 CQRS在课程上下文中的应用

**Command端**（写模型）：

```java
// 课程聚合根（Command侧）
public class Course {
    private final CourseId id;
    private CourseTitle title;
    private CourseDescription description;
    private CourseStatus status;              // DRAFT / PUBLISHED / ARCHIVED
    private List<Chapter> chapters;
    private Money price;
    private InstructorId instructorId;

    private final List<DomainEvent> events = new ArrayList<>();

    /**
     * 发布课程——业务规则：至少3个章节且每个章节至少有内容
     */
    public void publish() {
        if (status != CourseStatus.DRAFT) {
            throw new CourseNotDraftException(id, status);
        }
        if (chapters.size() < 3) {
            throw new InsufficientChaptersException(id, chapters.size(), 3);
        }
        boolean hasEmptyChapter = chapters.stream()
            .anyMatch(ch -> ch.getSections().isEmpty());
        if (hasEmptyChapter) {
            throw new EmptyChapterException(id);
        }
        if (price == null || price.isLessThanOrEqual(Money.ZERO(Currency.CNY))) {
            throw new InvalidCoursePriceException(id, price);
        }

        status = CourseStatus.PUBLISHED;
        events.add(new CoursePublished(id, title.value(), instructorId, price, chapters.size()));
    }

    /**
     * 更新章节——发布后不允许修改（重新创建草稿版本）
     */
    public void updateChapter(ChapterId chapterId, Chapter updated) {
        if (status != CourseStatus.DRAFT) {
            throw new CourseNotModifiableException(id, status);
        }
        int index = chapters.indexOf(new Chapter(chapterId));
        if (index == -1) {
            throw new ChapterNotFoundException(id, chapterId);
        }
        chapters.set(index, updated);
    }
}
```

**Query端**（读模型——投影）：


> **CQRS的读写分离**：Command端的写模型（Course聚合）关注业务规则验证，Query端的读模型（CourseListView）关注查询性能。两者通过领域事件连接：写操作产生事件，读模型消费事件更新视图。这种分离使得课程搜索可以用Elasticsearch，学员进度可以用Redis缓存，而写入仍然保证事务一致性。

### 3.3 Event Sourcing：以订单聚合为例

```java
/**
 * 支持事件溯源的订单聚合
 *
 * 核心思想：不直接存储当前状态，而是存储事件序列
 * 当前状态 = 初始状态 + 按顺序重放所有事件
 *
 * 优势：
 * 1. 完整的审计追踪——每一笔状态变更都有记录
 * 2. 时间旅行——可以重建任意历史时刻的状态
 * 3. 调试友好——出问题时可以精确回放事件定位原因
 *
 * 劣势：
 * 1. 查询需要重建状态，性能依赖事件数量
 * 2. 事件Schema演进需要兼容性处理
 * 3. 增加了系统复杂度
 */
public class OrderES {
    private OrderId id;
    private OrderStatus status;
    private Money totalAmount;
    private List<OrderLineSnapshot> lines;

    private final List<OrderEvent> uncommittedEvents = new ArrayList<>();

    // 无参构造——事件溯源重建用
    private OrderES() {}

    /**
     * 从事件历史重建聚合状态
     */
    public static OrderES reconstitute(OrderId id, List<OrderEvent> history) {
        OrderES order = new OrderES();
        order.id = id;
        for (OrderEvent event : history) {
            order.apply(event);     // 逐一应用事件，重建状态
        }
        return order;
    }

    /**
     * 创建订单
     */
    public static OrderES create(CustomerId customerId) {
        OrderES order = new OrderES();
        order.id = OrderId.generate();
        order.status = OrderStatus.CREATED;
        order.totalAmount = Money.ZERO(Currency.CNY);
        order.lines = new ArrayList<>();

        // 不直接修改状态，而是通过apply方法
        OrderCreated event = new OrderCreated(
            order.id, customerId, Instant.now()
        );
        order.raise(event);
        return order;
    }

    /**
     * 应用事件——修改状态（纯函数，无副作用）
     */
    private void apply(OrderEvent event) {
        if (event instanceof OrderCreated e) {
            this.status = OrderStatus.CREATED;
            this.totalAmount = Money.ZERO(Currency.CNY);
            this.lines = new ArrayList<>();
        } else if (event instanceof OrderLineAdded e) {
            this.lines.add(new OrderLineSnapshot(
                e.productId(), e.productName(), e.unitPrice(), e.quantity()
            ));
            recalculateTotal();
        } else if (event instanceof OrderConfirmed e) {
            this.status = OrderStatus.CONFIRMED;
        } else if (event instanceof OrderPaid e) {
            this.status = OrderStatus.PAID;
        } else if (event instanceof OrderCancelled e) {
            this.status = OrderStatus.CANCELLED;
        }
    }

    /**
     * 产生事件——先apply修改状态，再收集未提交事件
     */
    private void raise(OrderEvent event) {
        apply(event);
        uncommittedEvents.add(event);
    }

    public List<OrderEvent> pullEvents() {
        List<OrderEvent> events = new ArrayList<>(uncommittedEvents);
        uncommittedEvents.clear();
        return events;
    }

    private void recalculateTotal() {
        this.totalAmount = lines.stream()
            .map(l -> l.unitPrice().multiply(l.quantity()))
            .reduce(Money.ZERO(Currency.CNY), Money::add);
    }
}
```

**事件存储——持久化事件序列**：

```java
/**
 * 事件存储接口
 */
public interface EventStore {
    void appendEvents(String aggregateId, List<DomainEvent> events, int expectedVersion);
    List<DomainEvent> loadEvents(String aggregateId);
    List<DomainEvent> loadEventsAfter(String aggregateId, int afterVersion);
    int getCurrentVersion(String aggregateId);
}

/**
 * JPA事件存储实现
 */
@Repository
public class JpaEventStore implements EventStore {
    private final EventJpaRepository jpaRepo;

    @Override
    @Transactional
    public void appendEvents(String aggregateId, List<DomainEvent> events,
                              int expectedVersion) {
        // 乐观锁：检查版本号防止并发写入
        int currentVersion = getCurrentVersion(aggregateId);
        if (currentVersion != expectedVersion) {
            throw new ConcurrencyException(aggregateId, expectedVersion, currentVersion);
        }

        for (DomainEvent event : events) {
            EventJpaEntity entity = new EventJpaEntity();
            entity.setAggregateId(aggregateId);
            entity.setEventType(event.getClass().getSimpleName());
            entity.setEventData(serialize(event));
            entity.setVersion(++currentVersion);
            entity.setOccurredOn(event.occurredOn());
            jpaRepo.save(entity);
        }
    }

    @Override
    public List<DomainEvent> loadEvents(String aggregateId) {
        return jpaRepo.findByAggregateIdOrderByVersionAsc(aggregateId)
            .stream()
            .map(entity -> deserialize(entity.getEventType(), entity.getEventData()))
            .collect(Collectors.toList());
    }

    @Override
    public List<DomainEvent> loadEventsAfter(String aggregateId, int afterVersion) {
        return jpaRepo.findByAggregateIdAndVersionGreaterThanOrderByVersionAsc(
                aggregateId, afterVersion)
            .stream()
            .map(entity -> deserialize(entity.getEventType(), entity.getEventData()))
            .collect(Collectors.toList());
    }

    @Override
    public int getCurrentVersion(String aggregateId) {
        return jpaRepo.findMaxVersionByAggregateId(aggregateId).orElse(0);
    }
}
```

### 3.4 事件溯源的性能优化：快照

```java
/**
 * 快照机制——避免每次都从头重放全部事件
 *
 * 当事件数超过阈值时，保存当前状态快照
 * 重建时：先加载最近快照，再重放快照之后的增量事件
 *
 * 快照策略：
 * - 每100个事件创建一次快照（可配置）
 * - 快照异步创建，不阻塞写入
 * - 快照失败不影响正确性，只影响重建性能
 */
@Component
public class OrderAggregateRepository {
    private final EventStore eventStore;
    private final SnapshotStore snapshotStore;
    private static final int SNAPSHOT_INTERVAL = 100;

    public OrderES findById(OrderId id) {
        // 1. 尝试加载最近快照
        Optional<OrderSnapshot> snapshot = snapshotStore.getLatest(id);

        List<OrderEvent> events;
        OrderES order;

        if (snapshot.isPresent()) {
            // 从快照恢复基础状态
            order = OrderES.reconstitute(id, snapshot.get().state());
            // 只加载快照之后的增量事件
            events = eventStore.loadEventsAfter(id, snapshot.get().version());
        } else {
            // 无快照，从头重建
            events = eventStore.loadEvents(id);
            order = OrderES.reconstitute(id, events);
        }

        // 注意：reconstitute已经应用了所有事件
        // 如果有快照，这里只需应用增量事件
        if (snapshot.isPresent()) {
            for (OrderEvent event : events) {
                order.apply(event);
            }
        }

        return order;
    }

    public void save(OrderES order) {
        List<OrderEvent> events = order.pullEvents();
        if (events.isEmpty()) return;

        int currentVersion = eventStore.getCurrentVersion(order.getId().value());
        eventStore.appendEvents(order.getId().value(), events, currentVersion);

        // 检查是否需要创建快照
        int newVersion = currentVersion + events.size();
        if (newVersion % SNAPSHOT_INTERVAL == 0) {
            snapshotStore.save(new OrderSnapshot(order.getId(), newVersion, order));
        }
    }
}
```

> **事件溯源的Schema演进**：随着业务发展，事件的结构可能变化（如新增字段）。策略是：事件一旦发布就不可修改（immutable），新版本的事件用新的类型名（如 `OrderCreatedV2`），`apply`方法中处理新旧两种事件类型。Event Store中存储的是序列化的事件数据，旧事件仍然可以正确重放。

### 3.5 案例三常见陷阱

| 陷阱 | 表现 | 纠正方法 |
|------|------|---------|
| CQRS投影延迟 | 写入后查询看不到最新数据 | 接受最终一致性，或关键路径用同步投影 |
| 事件风暴 | 每个属性变更都发事件 | 只发有业务含义的事件 |
| 快照与事件不一致 | 快照创建后事件丢失 | 快照异步创建，失败不影响正确性 |
| 查询模型爆炸 | 每种查询建一个视图 | 从高频查询出发，逐步添加 |
| 事件Schema不兼容 | 旧事件无法重放 | 事件不可修改，新版本用新类型名 |

## DDD落地的渐进路径

对于大多数团队，DDD不需要一步到位。推荐的渐进路径：

阶段一：统一语言（1-2周）
├── 组织领域对话会议
├── 建立术语表
└── 统一代码命名

阶段二：限界上下文划分（2-4周）
├── 识别核心域/支撑域/通用域
├── 绘制上下文映射图
└── 确定团队边界

阶段三：聚合设计（4-8周）
├── 从核心域开始
├── 设计聚合和领域事件
└── 建立仓储接口

阶段四：代码重构（持续进行）
├── 按限界上下文拆分模块
├── 引入防腐层隔离遗留代码
└── 逐步迁移数据模型

阶段五：架构演进（视需要）
├── 引入CQRS优化查询
├── 引入事件溯源完善审计
└── 按上下文拆分微服务

### 从传统架构迁移到DDD的策略

对于已有系统，推荐"绞杀者模式"（Strangler Fig Pattern）渐进迁移：

```java
// 绞杀者模式：新旧系统并行，通过Feature Flag控制流量
public class OrderFacade {
    private final NewOrderService newService;      // 新的DDD实现
    private final LegacyOrderService legacyService; // 旧的贫血模型实现
    private final FeatureFlag featureFlag;

    public OrderId createOrder(CreateOrderRequest request) {
        if (featureFlag.isEnabled("ddd-order-creation")) {
            return newService.createOrder(request);   // 走新路径
        } else {
            return legacyService.createOrder(request); // 走旧路径
        }
    }
}
```

迁移五步法：
1. **识别核心域**：找到业务价值最高、变更最频繁的模块优先重构
2. **建立防腐层**：在新旧系统边界建立ACL，保护新模型不被旧代码污染
3. **并行运行**：新旧系统并行，通过对比验证新实现的正确性
4. **逐步切换**：验证通过后，逐步将流量切换到新实现（从1%→10%→50%→100%）
5. **清理旧代码**：确认新系统稳定后，移除旧代码

## 案例对比与决策指南

### 不同场景的架构选择

| 场景特征 | 推荐方案 | 关键理由 | 典型案例 |
|---------|---------|---------|---------|
| 业务逻辑简单，CRUD为主 | 贫血模型 + 传统分层 | 过度设计的DDD反而增加复杂度 | 内部管理后台 |
| 业务规则复杂，状态流转多 | 充血模型 + 聚合 | 业务不变量需要在领域层保护 | 电商订单、银行账户 |
| 查询需求远多于写入 | CQRS分离读写 | 读写模型可以独立优化 | 课程搜索、报表统计 |
| 需要完整审计追踪 | Event Sourcing | 事件天然提供审计日志 | 金融交易、操作日志 |
| 多团队协作，系统庞大 | 限界上下文 + 微服务 | 降低团队间的耦合 | 大型电商平台 |
| 高并发场景 | 小聚合 + 事件驱动 | 减少锁冲突，提升吞吐量 | 秒杀活动、实时转账 |

### 测试策略

DDD代码的测试与传统三层架构有明显区别：

```java
/**
 * 聚合的单元测试——测试业务不变量
 * 不需要Spring容器、不需要数据库，纯Java测试
 */
class OrderTest {

    @Test
    void should_create_order_with_created_status() {
        // Arrange
        CustomerId customerId = new CustomerId("C001");
        Address address = new Address("北京市", "朝阳区", "xx路1号");

        // Act
        Order order = Order.create(customerId, address);

        // Assert
        assertEquals(OrderStatus.CREATED, order.getStatus());
        assertEquals(customerId, order.getCustomerId());
        assertTrue(order.getDomainEvents().stream()
            .anyMatch(e -> e instanceof OrderCreated));
    }

    @Test
    void should_reject_adding_line_when_not_created_status() {
        // Arrange
        Order order = createAndConfirmOrder();

        // Act &amp; Assert
        assertThrows(OrderNotModifiableException.class, () ->
            order.addLine(new ProductId("P001"), "测试商品",
                         Money.of(100, Currency.CNY), 1)
        );
    }

    @Test
    void should_reject_order_with_more_than_50_lines() {
        // Arrange
        Order order = Order.create(new CustomerId("C001"), defaultAddress());

        // Act
        for (int i = 0; i < 50; i++) {
            order.addLine(new ProductId("P" + i), "商品" + i,
                         Money.of(10, Currency.CNY), 1);
        }

        // Assert
        assertThrows(TooManyOrderLinesException.class, () ->
            order.addLine(new ProductId("P51"), "第51个商品",
                         Money.of(10, Currency.CNY), 1)
        );
    }

    @Test
    void should_recalculate_total_when_line_added() {
        // Arrange
        Order order = Order.create(new CustomerId("C001"), defaultAddress());

        // Act
        order.addLine(new ProductId("P001"), "商品A",
                     Money.of(100, Currency.CNY), 2);  // 200
        order.addLine(new ProductId("P002"), "商品B",
                     Money.of(50, Currency.CNY), 3);   // 150

        // Assert
        assertEquals(Money.of(350, Currency.CNY), order.getTotalAmount());
    }
}
```

> **测试原则**：聚合的业务逻辑用单元测试（不需要基础设施），应用层的编排逻辑用集成测试（需要Spring容器），跨上下文的事件协作用端到端测试。单元测试应该覆盖所有不变量验证，这是DDD的核心价值——业务规则在聚合中被严格保护。

## 实战要点总结

| 原则 | 实操要点 | 反模式 |
|------|---------|--------|
| 限界上下文按业务能力划分 | 每个上下文对应一个团队负责的业务能力 | 按技术层划分（前端/后端/DB） |
| 聚合只包含业务不变量 | 找出"必须在同一事务中一致"的规则 | 将所有关联对象塞进一个聚合 |
| 聚合间通过ID引用 | `CustomerId customerId` 而非 `Customer customer` | 聚合间直接对象引用 |
| 领域事件实现最终一致性 | 一个事务一个聚合，跨聚合用事件 | 跨聚合使用分布式事务 |
| 防腐层保护下游模型 | 在边界处翻译，不让上游模型泄漏 | 直接依赖上游内部模型 |
| 仓储接口在领域层 | `interface OrderRepository` 定义在domain包 | 仓储接口在infrastructure层 |
| 代码即语言 | `order.confirm()` 而非 `order.setStatus(2)` | 技术性命名掩盖业务含义 |
| 值对象按属性比较 | Money、Address等无唯一标识的概念用值对象 | 给所有对象都分配数据库ID |
| 领域事件要有业务含义 | `OrderConfirmed` 而非 `OrderUpdated` | 每个属性变更都发事件 |
| 渐进式引入DDD | 从核心域开始，逐步扩展 | 一次性重写整个系统 |

> **最后一句话**：DDD不是银弹，它是一种思维方式。核心价值不在于用了多少设计模式，而在于团队对业务的深度理解和统一表达。如果一个简单的CRUD系统能很好地满足需求，不需要强行引入DDD。但当业务复杂度上升到一定程度时，DDD提供的结构化思维方式会让你的系统更加健壮和可维护。
