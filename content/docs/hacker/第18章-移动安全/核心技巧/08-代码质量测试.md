## 代码质量测试

代码质量测试是移动应用安全评估的核心环节，关注的是应用源码和配置层面的安全缺陷。与数据存储、网络通信等运行时测试不同，代码质量测试主要通过静态分析手段，在不执行应用的情况下发现安全隐患。这类漏洞往往源于开发者的安全意识不足或对平台安全机制的理解偏差，一旦被攻击者利用，可能导致应用被逆向破解、用户数据泄露甚至远程代码执行。

本节从 AndroidManifest.xml 安全配置、WebView 安全加固、组件导出控制、代码级安全缺陷、代码混淆分析五个维度展开，覆盖从配置到代码的完整攻击面。

### 18.8.1 AndroidManifest.xml 安全配置审计

AndroidManifest.xml 是 Android 应用的"身份证"，声明了应用的权限、组件、安全标志等关键信息。攻击者只需解包 APK 即可完整读取该文件，因此其中的每一项配置都直接影响应用的安全水位。

#### 18.8.1.1 debuggable 标志

`android:debuggable` 是 `<application>` 标签中的布尔属性，控制应用是否可被调试器附加。

**安全影响：**

当 `debuggable=true` 时，攻击者可以：

- 通过 JDWP（Java Debug Wire Protocol）附加调试器，任意读取和修改运行时变量
- 在任意方法入口设置断点，拦截加密密钥、Token 等敏感数据
- 通过 `adb shell run-as <package>` 访问应用私有目录（`/data/data/<package>/`）
- 绕过签名校验、License 验证等客户端保护逻辑

**检查方法：**

```bash
# 方法一：解包后直接查看 AndroidManifest.xml
$ apktool d target.apk -o decoded/
$ grep -i "debuggable" decoded/AndroidManifest.xml

# 方法二：使用 aapt 工具
$ aapt dump xmltree target.apk AndroidManifest.xml | grep debuggable

# 方法三：使用 apkanalyzer（Android SDK 内置）
$ apkanalyzer manifest print target.apk | grep debuggable
```

**典型漏洞配置：**

```xml
<!-- 危险：生产包开启了调试 -->
<application
    android:debuggable="true"
    android:allowBackup="true"
    ...>
```

**修复方案：**

```xml
<!-- 正确做法：显式关闭调试，不依赖默认值 -->
<application
    android:debuggable="false"
    ...>
```

在 `build.gradle` 中通过构建类型自动控制：

```groovy
android {
    buildTypes {
        release {
            debuggable false  // Release 构建显式关闭
            minifyEnabled true
            proguardFiles getDefaultProguardFile('proguard-android-optimize.txt'), 'proguard-rules.pro'
        }
        debug {
            debuggable true   // Debug 构建允许调试
        }
    }
}
```

> **测试要点：** 即使默认情况下 release 构建的 `debuggable` 为 `false`，仍需验证实际 APK 的配置值。某些开发者会在主工程或第三方 SDK 的 manifest 中误设该属性。使用 `aapt dump` 或 `apkanalyzer` 直接检查最终合并后的 manifest 才是可靠的做法。

#### 18.8.1.2 allowBackup 标志

`android:allowBackup` 控制应用数据是否可通过 `adb backup` 命令导出。

**安全影响：**

当 `allowBackup=true` 时，攻击者通过物理接触设备或 ADB 连接即可：

- 导出应用完整数据（`/data/data/<package>/` 的备份副本）
- 提取 SharedPreferences、SQLite 数据库、缓存文件中的敏感信息
- 在另一台设备上恢复备份，接管用户会话
- 绕过应用自身的数据保护机制（如加密密钥存储在 KeyStore 中，但备份可包含 KeyStore 元数据）

**攻击演示：**

```bash
# 备份目标应用数据
$ adb backup -f backup.ab -noapk com.target.app

# 使用 android-backup-extractor 解包
$ java -jar abe.jar unpack backup.ab backup.tar
$ tar xf backup.tar

# 分析备份数据
$ ls apps/com.target.app/
db/  sp/  shared_prefs/  cache/  files/

# 检查 SharedPreferences 中的敏感数据
$ grep -r "token\|session\|password" apps/com.target.app/sp/
```

**修复方案：**

```xml
<application
    android:allowBackup="false"
    android:fullBackupContent="false"
    android:dataExtractionRules="@xml/data_extraction_rules"
    ...>
```

Android 12+ 引入了更细粒度的备份控制：

```xml
<!-- res/xml/data_extraction_rules.xml -->
<?xml version="1.0" encoding="utf-8"?>
<data-extraction-rules>
    <cloud-backup>
        <exclude domain="database" path="sensitive.db"/>
        <exclude domain="sharedpref" path="auth_prefs.xml"/>
    </cloud-backup>
    <device-transfer>
        <exclude domain="database" path="sensitive.db"/>
    </device-transfer>
</data-extraction-rules>
```

> **测试要点：** 某些应用虽然设置了 `allowBackup="false"`，但未设置 `fullBackupContent` 或 `dataExtractionRules`，在特定 Android 版本上仍可能存在部分数据泄露。测试时应实际执行 `adb backup` 验证，而非仅检查配置。

#### 18.8.1.3 权限声明审计

应用申请的权限直接暴露了其功能范围和潜在攻击面。

**检查方法：**

```bash
# 列出所有声明的权限
$ aapt dump permissions target.apk

# 查看权限组和保护级别
$ aapt dump xmltree target.apk AndroidManifest.xml | grep -A2 "uses-permission"
```

**高风险权限清单：**

| 权限 | 风险等级 | 说明 |
|------|----------|------|
| `READ_SMS` | 🔴 高危 | 可读取短信验证码，劫持二次认证 |
| `READ_CONTACTS` | 🔴 高危 | 泄露用户社交关系 |
| `CAMERA` | 🟠 中高危 | 可被恶意调用进行偷拍 |
| `RECORD_AUDIO` | 🟠 中高危 | 环境监听风险 |
| `ACCESS_FINE_LOCATION` | 🟠 中高危 | 精确位置追踪 |
| `WRITE_EXTERNAL_STORAGE` | 🟡 中危 | 可写入恶意文件 |
| `SYSTEM_ALERT_WINDOW` | 🟡 中危 | 悬浮窗覆盖攻击（点击劫持） |
| `INSTALL_PACKAGES` | 🔴 高危 | 静默安装应用 |
| `BIND_ACCESSIBILITY_SERVICE` | 🔴 高危 | 无障碍服务可监控所有界面操作 |

**过度权限检测思路：**

对比应用实际使用的 API 调用与声明的权限，识别"申请但未使用"的冗余权限——这些权限扩大了攻击面却没有实际功能价值。

```bash
# 统计反编译代码中实际使用的权限相关 API
$ grep -rn "getContentResolver\|getSystemService\|SmsManager\|LocationManager" target_java/ --include="*.java" | wc -l
```

#### 18.8.1.4 组件导出安全

Android 四大组件（Activity、Service、BroadcastReceiver、ContentProvider）的 `exported` 属性决定了外部应用是否可以访问该组件。

**导出组件的风险：**

| 组件类型 | 风险场景 |
|----------|----------|
| Activity | 绕过登录界面直接访问内部功能页面 |
| Service | 调用特权服务执行未授权操作 |
| BroadcastReceiver | 注入伪造广播触发敏感逻辑 |
| ContentProvider | 读取/篡改应用数据库 |

**检查方法：**

```bash
# 查找所有显式导出的组件
$ grep -n 'exported="true"' decoded/AndroidManifest.xml

# 查找包含 intent-filter 的组件（Android 12 以下默认导出）
$ grep -B5 "intent-filter" decoded/AndroidManifest.xml | grep "activity\|service\|receiver"

# 使用 drozer 自动枚举
$ drozer console connect
dz> run app.package.attacksurface com.target.app
dz> run app.activity.info -a com.target.app
dz> run app.provider.info -a com.target.app
```

**典型漏洞与利用：**

```bash
# 利用导出的 Activity 绕过登录
$ adb shell am start -n com.target.app/.AdminActivity

# 利用导出的 ContentProvider 读取数据
$ adb shell content query --uri content://com.target.app.provider/users

# 利用导出的 Service 执行操作
$ adb shell am startservice -n com.target.app/.PaymentService --es action "transfer" --es amount "1000"
```

**修复方案：**

```xml
<!-- 非必要不导出 -->
<activity
    android:name=".InternalActivity"
    android:exported="false">
</activity>

<!-- 需要导出时添加权限保护 -->
<activity
    android:name=".ShareActivity"
    android:exported="true"
    android:permission="com.target.app.permission.USE_SHARE">
</activity>

<!-- ContentProvider 使用签名级权限 -->
<provider
    android:name=".UserDataProvider"
    android:exported="true"
    android:readPermission="com.target.app.permission.READ_USER_DATA"
    android:writePermission="com.target.app.permission.WRITE_USER_DATA"
    android:permission="com.target.app.permission.ACCESS_DATA">
</provider>
```

Android 12+ 强制要求显式声明 `exported`，不再依赖 intent-filter 隐式推断，这显著减少了意外导出的风险。

### 18.8.2 WebView 安全配置

WebView 是 Android 应用中最常见的攻击面之一。它本质上是一个嵌入式浏览器内核，如果配置不当，攻击者可以通过恶意网页或 URL scheme 实现 XSS、文件读取、代码执行等攻击。

#### 18.8.2.1 JavaScript 启用风险

```java
// 危险：启用了 JavaScript 且加载了不可信内容
webView.getSettings().setJavaScriptEnabled(true);
webView.loadUrl("https://user-content.example.com/profile/" + userId);
```

当 `setJavaScriptEnabled(true)` 与加载不可信内容结合时，攻击者可通过注入 JavaScript 实现：

- 窃取 Cookie 和 Token（`document.cookie`）
- 读取通过 `addJavascriptInterface` 暴露的原生方法
- 发起 CSRF 请求
- 操控 DOM 实现钓鱼

**安全配置基线：**

```java
WebSettings settings = webView.getSettings();

// 仅在必要时启用 JavaScript
settings.setJavaScriptEnabled(false);  // 默认关闭

// 如果必须启用 JavaScript，严格控制加载源
if (needsJavaScript) {
    settings.setJavaScriptEnabled(true);
    // 禁止文件访问
    settings.setAllowFileAccess(false);
    settings.setAllowFileAccessFromFileURLs(false);
    settings.setAllowUniversalAccessFromFileURLs(false);
    // 禁止内容访问
    settings.setAllowContentAccess(false);
}
```

#### 18.8.2.2 addJavascriptInterface 远程代码执行

`addJavascriptInterface` 在 Android 4.2 以下存在远程代码执行漏洞（CVE-2012-6636），攻击者通过反射调用 `Runtime.exec()` 执行任意命令。

```java
// 危险：暴露了过多的原生方法
webView.addJavascriptInterface(new Object() {
    @JavascriptInterface
    public String readFile(String path) {
        // 任意文件读取
        return new String(Files.readAllBytes(Paths.get(path)));
    }

    @JavascriptInterface
    public void executeCommand(String cmd) {
        // 远程命令执行
        Runtime.getRuntime().exec(cmd);
    }
}, "Android");
```

**攻击载荷：**

```javascript
// 通过反射调用 Runtime.exec()
function execute(cmd) {
    var runtime = window.Android.getClass()
        .forName('java.lang.Runtime')
        .getMethod('getRuntime', null)
        .invoke(null, null);
    runtime.exec(cmd);
}
execute('cat /etc/passwd');
```

**安全实践：**

```java
// 最小化暴露接口
webView.addJavascriptInterface(new WebAppInterface(this), "Bridge");

public class WebAppInterface {
    private Context context;

    WebAppInterface(Context c) { this.context = c; }

    @JavascriptInterface
    public void showToast(String message) {
        // 仅暴露无害功能，不接受路径/命令参数
        Toast.makeText(context, message, Toast.LENGTH_SHORT).show();
    }

    // 绝不暴露：文件操作、命令执行、数据库访问、网络请求
}
```

#### 18.8.2.3 SSL 错误处理

```java
// 致命错误：忽略 SSL 证书验证
webView.setWebViewClient(new WebViewClient() {
    @Override
    public void onReceivedSslError(WebView view, SslErrorHandler handler, SslError error) {
        handler.proceed();  // 接受所有证书，包括自签名和过期证书
    }
});
```

这使得应用完全暴露于中间人攻击（MITM）。攻击者可以截获所有 WebView 中的 HTTPS 通信，注入恶意 JavaScript 或替换页面内容。

**正确做法：**

```java
webView.setWebViewClient(new WebViewClient() {
    @Override
    public void onReceivedSslError(WebView view, SslErrorHandler handler, SslError error) {
        // 拒绝所有证书错误
        handler.cancel();

        // 可选：向用户展示错误信息
        new AlertDialog.Builder(view.getContext())
            .setMessage("SSL 证书验证失败，连接不安全")
            .setPositiveButton("确定", null)
            .show();
    }
});
```

#### 18.8.2.4 URL Scheme 劫持

自定义 URL Scheme（如 `myapp://`）是应用间通信的常见方式，但如果没有正确验证来源，攻击者可以通过恶意网页触发应用内的敏感操作。

```java
// 在 Activity 的 onCreate 中处理 URL Scheme
Intent intent = getIntent();
Uri uri = intent.getData();

if (uri != null && uri.getScheme().equals("myapp")) {
    String action = uri.getHost();
    String param = uri.getQueryParameter("url");

    // 危险：未验证来源，直接执行操作
    if ("open".equals(action)) {
        webView.loadUrl(param);  // 加载任意 URL
    }
}
```

**攻击方式：**

```html
<!-- 恶意网页触发应用加载钓鱼页面 -->
<a href="myapp://open?url=https://attacker.com/phishing.html">点击领取奖品</a>
```

**防御措施：**

```java
// 验证 URL 白名单
private static final List<String> ALLOWED_HOSTS = Arrays.asList(
    "www.example.com",
    "api.example.com"
);

private void handleDeepLink(Uri uri) {
    String url = uri.getQueryParameter("url");
    if (url != null) {
        Uri targetUri = Uri.parse(url);
        if (ALLOWED_HOSTS.contains(targetUri.getHost())) {
            webView.loadUrl(url);
        } else {
            Log.w(TAG, "Blocked untrusted URL: " + url);
        }
    }
}
```

### 18.8.3 代码级安全缺陷检测

配置层面的问题可以通过 Manifest 审计快速发现，但更多安全缺陷隐藏在业务代码中。以下是最常见的代码级安全问题及其检测方法。

#### 18.8.3.1 硬编码敏感信息

开发者为了方便，常将 API 密钥、加密密钥、后端地址等直接写死在代码中。攻击者通过反编译即可获取这些信息。

**常见模式：**

```java
// 硬编码 API 密钥
private static final String API_KEY = "sk-1234567890abcdef";

// 硬编码加密密钥
private static final byte[] AES_KEY = "0123456789abcdef".getBytes();

// 硬编码后端地址（可能泄露内部架构）
private static final String DEBUG_URL = "http://192.168.1.100:8080/api";

// 硬编码第三方服务凭据
private static final String AWS_SECRET = "YOUR_AWS_SECRET_KEY";
```

**自动化检测：**

```bash
# 使用 grep 扫描常见模式
$ grep -rn --include="*.java" \
    -E "(api_key|apikey|secret|password|token|private_key|AES|DES|RSA)\s*=\s*\"[^\"]+\"" \
    target_java/

# 检查 strings.xml 中的敏感字符串
$ grep -i "key\|secret\|password\|token" decoded/res/values/strings.xml

# 检查 assets 和 raw 资源中的配置文件
$ find decoded/assets/ decoded/res/raw/ -type f | xargs grep -l "key\|secret\|password"
```

**Semgrep 规则：**

```yaml
rules:
  - id: hardcoded-secret
    patterns:
      - pattern: |
          $TYPE $VAR = "...";
      - metavariable-regex:
          metavariable: $VAR
          regex: (?i)(api.?key|secret|password|token|private.?key|access.?key|auth.?token)
    message: "硬编码敏感信息: $VAR"
    severity: ERROR
    languages: [java]
```

**修复方案：**

```java
// 方案一：从 BuildConfig 读取（构建时注入，不进入版本控制）
private String apiKey = BuildConfig.API_KEY;

// build.gradle
android {
    defaultConfig {
        buildConfigField "String", "API_KEY", "\"${project.findProperty('API_KEY')}\""
    }
}

// 方案二：从 Android Keystore 读取加密密钥
private byte[] getEncryptionKey() {
    KeyStore keyStore = KeyStore.getInstance("AndroidKeyStore");
    keyStore.load(null);
    SecretKey key = (SecretKey) keyStore.getKey("my_alias", null);
    return key.getEncoded();
}

// 方案三：从服务器动态获取（适合高安全场景）
// 首次启动时通过安全通道获取密钥，存入 KeyStore
```

#### 18.8.3.2 不安全的日志输出

生产环境中残留的调试日志是信息泄露的常见来源。

**风险示例：**

```java
// 泄露用户凭据
Log.d("Auth", "Login attempt: user=" + username + " pass=" + password);

// 泄露 Token
Log.d("API", "Authorization header: " + bearerToken);

// 泄露加密密钥
Log.d("Crypto", "Using key: " + Base64.encodeToString(key, Base64.DEFAULT));

// 泄露用户隐私数据
Log.d("GPS", "Location: " + latitude + ", " + longitude);
```

**检测方法：**

```bash
# 搜索所有 Log 调用
$ grep -rn --include="*.java" \
    -E "Log\.(d|i|w|e|v|wtf)\(" \
    target_java/ | grep -i "password\|token\|key\|secret\|credential"

# 使用 lint 检测
$ ./gradlew lint --check Logging
```

**修复方案：**

```java
// 方案一：使用 ProGuard/R8 规则移除日志
// proguard-rules.pro
-assumenosideeffects class android.util.Log {
    public static int d(...);
    public static int v(...);
    public static int i(...);
}

// 方案二：封装日志工具类
public class SecureLog {
    private static final boolean DEBUG = BuildConfig.DEBUG;

    public static void d(String tag, String msg) {
        if (DEBUG) {
            Log.d(tag, msg);
        }
    }

    public static void e(String tag, String msg, Throwable t) {
        if (DEBUG) {
            Log.e(tag, msg, t);
        }
        // 生产环境：上报到崩溃监控系统（脱敏后）
        CrashReporter.logError(sanitize(msg), t);
    }

    private static String sanitize(String msg) {
        // 移除敏感信息模式
        return msg.replaceAll("(?i)(password|token|key|secret)=\\S+", "$1=***");
    }
}
```

#### 18.8.3.3 弱加密算法使用

使用过时或已知存在漏洞的加密算法，使得数据保护形同虚设。

**常见弱加密实例：**

```java
// DES 已被破解，密钥长度仅 56 位
Cipher cipher = Cipher.getInstance("DES/ECB/PKCS5Padding");

// ECB 模式不隐藏数据模式，相同的明文块产生相同的密文块
Cipher cipher = Cipher.getInstance("AES/ECB/PKCS5Padding");

// MD5/SHA1 已不适用于安全场景
MessageDigest md = MessageDigest.getInstance("MD5");
byte[] hash = md.digest(password.getBytes());

// 硬编码 IV（初始化向量）
byte[] iv = "0000000000000000".getBytes();
IvParameterSpec ivSpec = new IvParameterSpec(iv);
```

**安全替代方案：**

```java
// AES-256-GCM（推荐：认证加密）
KeyGenerator keyGen = KeyGenerator.getInstance("AES");
keyGen.init(256);
SecretKey secretKey = keyGen.generateKey();

Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
byte[] iv = new byte[12];  // GCM 推荐 12 字节 IV
SecureRandom.getInstanceStrong().nextBytes(iv);
GCMParameterSpec gcmSpec = new GCMParameterSpec(128, iv);
cipher.init(Cipher.ENCRYPT_MODE, secretKey, gcmSpec);

// 密码哈希使用 Argon2 或 bcrypt
// Android 端使用 PBKDF2 作为最低标准
SecretKeyFactory factory = SecretKeyFactory.getInstance("PBKDF2WithHmacSHA256");
KeySpec spec = new PBEKeySpec(password.toCharArray(), salt, 310000, 256);
SecretKey derivedKey = factory.generateSecret(spec);
```

**检测命令：**

```bash
# 搜索弱加密算法
$ grep -rn --include="*.java" \
    -E "(DES|RC4|MD5|SHA-1|ECB|DESede)" \
    target_java/

# 使用 MobSF 自动扫描加密问题
$ curl -X POST http://localhost:8000/api/v1/upload \
  -H "Authorization: <api_key>" \
  -F "file=@target.apk"
```

#### 18.8.3.4 ContentProvider SQL 注入

ContentProvider 对外暴露数据访问接口，如果查询参数未经过滤，可能被注入恶意 SQL。

**漏洞代码：**

```java
public class UserProvider extends ContentProvider {
    @Override
    public Cursor query(Uri uri, String[] projection, String selection,
                        String[] selectionArgs, String sortOrder) {
        // 危险：直接拼接 SQL 查询
        SQLiteDatabase db = dbHelper.getReadableDatabase();
        return db.query("users", projection, selection, selectionArgs,
                        null, null, sortOrder);
    }
}
```

**攻击方式：**

```bash
# SQL 注入读取任意表
$ adb shell content query --uri content://com.target.app.provider/users \
  --where "1=1 UNION SELECT sql FROM sqlite_master--"

# 读取其他应用数据
$ adb shell content query --uri content://com.target.app.provider/users \
  --where "1=1 UNION SELECT name,sql FROM sqlite_master WHERE type='table'--"
```

**防御措施：**

```java
@Override
public Cursor query(Uri uri, String[] projection, String selection,
                    String[] selectionArgs, String sortOrder) {
    // 使用参数化查询，禁止原始 SQL 拼接
    SQLiteQueryBuilder qb = new SQLiteQueryBuilder();
    qb.setTables("users");

    // 设置白名单列名
    qb.setProjectionMap(getColumnProjectionMap());

    // 限制只允许等值条件
    if (selection != null && selection.contains("=")) {
        // 使用参数化查询
        return qb.query(db, projection, selection, selectionArgs, null, null, sortOrder);
    }

    throw new IllegalArgumentException("Invalid query");
}
```

### 18.8.4 代码混淆与加固分析

代码混淆是防止逆向分析的第一道防线。测试时需要评估混淆的有效性，找出可被绕过的薄弱点。

#### 18.8.4.1 ProGuard/R8 混淆效果评估

```bash
# 检查类名和方法名是否有效混淆
$ jadx -d target_java/ target.apk

# 统计混淆后的类名比例
$ find target_java/ -name "*.java" | xargs grep -l "^public class [a-z]" | wc -l

# 检查是否保留了调试信息
$ apktool d target.apk -o decoded/
$ ls decoded/smali/  # 如果包名仍可读，说明混淆不足

# 检查是否有 mapping 文件泄露（可反向还原混淆）
$ find decoded/ -name "mapping.txt" -o -name "*.map"
```

**有效混淆的特征：**

| 指标 | 低效混淆 | 高效混淆 |
|------|----------|----------|
| 类名 | `com.target.app.LoginActivity` | `a.b.c.d` |
| 方法名 | `validatePassword()` | `a()` |
| 字符串 | 明文密钥、URL | 加密或拆分 |
| 控制流 | 线性执行 | 插入不透明谓词 |
| 调试信息 | SourceFile, LineNumberTable | 已移除 |

#### 18.8.4.2 字符串加密检测

#### 18.8.4.3 反调试与完整性校验

高质量的应用通常包含反调试和完整性校验机制，测试者需要评估这些保护的有效性：

```java
// 常见反调试检查
public class AntiTamper {
    // 检查调试器状态
    public static boolean isDebuggerConnected() {
        return Debug.isDebuggerConnected();
    }

    // 检查 TracerPid（检测 strace/ltrace）
    public static boolean isTraced() {
        try {
            BufferedReader reader = new BufferedReader(
                new FileReader("/proc/self/status"));
            String line;
            while ((line = reader.readLine()) != null) {
                if (line.startsWith("TracerPid:")) {
                    int pid = Integer.parseInt(line.split(":")[1].trim());
                    return pid != 0;
                }
            }
        } catch (Exception e) {
            return false;
        }
        return false;
    }

    // 签名校验
    public static boolean verifySignature(Context context) {
        try {
            PackageInfo info = context.getPackageManager()
                .getPackageInfo(context.getPackageName(), PackageManager.GET_SIGNATURES);
            String currentSig = info.signatures[0].toCharsString();
            String expectedSig = "EXPECTED_SIGNATURE_HASH";
            return currentSig.equals(expectedSig);
        } catch (Exception e) {
            return false;
        }
    }
}
```

**绕过反调试的常见方法：**

```bash
# 使用 Frida 绕过调试检测
$ frida -U -f com.target.app -l anti_debug_bypass.js

# anti_debug_bypass.js
Java.perform(function() {
    var Debug = Java.use("android.os.Debug");
    Debug.isDebuggerConnected.implementation = function() {
        return false;
    };

    // Hook 签名校验
    var AntiTamper = Java.use("com.target.app.AntiTamper");
    AntiTamper.verifySignature.implementation = function() {
        return true;
    };
});
```

### 18.8.5 代码质量测试工具链

将上述检查整合为自动化流水线，是提升测试效率的关键。

| 工具 | 用途 | 安装方式 |
|------|------|----------|
| **MobSF** | 综合静态分析（Manifest、代码、权限） | `docker pull opensecurity/mobile-security-framework-mobsf` |
| **QARK** | Android 代码审计（硬编码、不安全配置） | `pip install qark` |
| **Semgrep** | 自定义规则的代码扫描 | `pip install semgrep` |
| **APKiD** | 混淆/加固/编译器识别 | `pip install apkid` |
| **jadx** | 反编译 + 搜索 | `brew install jadx` |
| **drozer** | 组件导出安全测试 | `pip install drozer` |
| **Android Lint** | Android 官方静态检查 | Android Studio 内置 |
| **Checkmarx** | 商业级 SAST（支持 Android/iOS） | 商业授权 |

**集成化扫描脚本：**

```bash
#!/bin/bash
# mobile_code_audit.sh - 移动应用代码质量自动化审计
APK="$1"
REPORT_DIR="audit_$(date +%Y%m%d_%H%M%S)"
mkdir -p "$REPORT_DIR"

echo "[*] 目标: $APK"
echo "[*] 报告目录: $REPORT_DIR"

# 1. Manifest 安全配置检查
echo "[+] 检查 AndroidManifest.xml..."
apktool d "$APK" -o "$REPORT_DIR/decoded/" -f 2>/dev/null
grep -i "debuggable" "$REPORT_DIR/decoded/AndroidManifest.xml" > "$REPORT_DIR/debuggable.txt"
grep -i "allowBackup" "$REPORT_DIR/decoded/AndroidManifest.xml" > "$REPORT_DIR/allowbackup.txt"
grep 'exported="true"' "$REPORT_DIR/decoded/AndroidManifest.xml" > "$REPORT_DIR/exported.txt"

# 2. 反编译
echo "[+] 反编译..."
jadx -d "$REPORT_DIR/java/" "$APK" 2>/dev/null

# 3. 硬编码敏感信息扫描
echo "[+] 扫描硬编码敏感信息..."
grep -rn --include="*.java" \
    -E "(api_key|apikey|secret|password|token|private_key)\s*=\s*\"[^\"]+\"" \
    "$REPORT_DIR/java/" > "$REPORT_DIR/hardcoded_secrets.txt"

# 4. 弱加密算法扫描
echo "[+] 扫描弱加密算法..."
grep -rn --include="*.java" \
    -E "(DES|RC4|MD5|SHA-1|ECB)" \
    "$REPORT_DIR/java/" > "$REPORT_DIR/weak_crypto.txt"

# 5. 不安全日志扫描
echo "[+] 扫描敏感日志..."
grep -rn --include="*.java" \
    -E "Log\.(d|i|w|e|v)\(" \
    "$REPORT_DIR/java/" > "$REPORT_DIR/logging.txt"

# 6. WebView 安全配置
echo "[+] 扫描 WebView 配置..."
grep -rn --include="*.java" \
    -E "(setJavaScriptEnabled|addJavascriptInterface|setAllowFileAccess|onReceivedSslError)" \
    "$REPORT_DIR/java/" > "$REPORT_DIR/webview.txt"

# 7. 混淆效果评估
echo "[+] 评估代码混淆..."
TOTAL=$(find "$REPORT_DIR/java/" -name "*.java" | wc -l)
OBFUSCATED=$(find "$REPORT_DIR/java/" -name "*.java" -exec grep -l "^public class [a-z]" {} \; | wc -l)
echo "总类数: $TOTAL, 混淆类数: $OBFUSCATED" > "$REPORT_DIR/obfuscation.txt"

# 8. 汇总报告
echo ""
echo "========== 审计摘要 =========="
echo "debuggable:     $(wc -l < "$REPORT_DIR/debuggable.txt") 条"
echo "allowBackup:    $(wc -l < "$REPORT_DIR/allowbackup.txt") 条"
echo "导出组件:       $(wc -l < "$REPORT_DIR/exported.txt") 条"
echo "硬编码密钥:     $(wc -l < "$REPORT_DIR/hardcoded_secrets.txt") 条"
echo "弱加密算法:     $(wc -l < "$REPORT_DIR/weak_crypto.txt") 条"
echo "敏感日志:       $(wc -l < "$REPORT_DIR/logging.txt") 条"
echo "WebView配置:    $(wc -l < "$REPORT_DIR/webview.txt") 条"
echo "混淆率:         $OBFUSCATED / $TOTAL"
echo "================================"
echo "[*] 详细报告已保存至 $REPORT_DIR/"
```
