# 索引结构——理论基础

索引是数据库系统的"高速公路"。没有索引，查询只能在数据的荒原上跋涉；有了索引，系统可以精准地一步到位定位目标。理解索引的理论基础，是掌握数据库性能优化的第一步。

本节从三个维度构建索引知识的理论框架：**核心概念**建立对索引本质的认知，**技术演进**展示索引结构如何随硬件和需求变化而迭代，**关键指标**提供评估和选择索引结构的量化标准。三者环环相扣，构成从"是什么"到"怎么发展"再到"怎么衡量"的完整认知链路。

---

## 本节内容导览

### 第一部分：索引结构核心概念

索引的"道"——理解索引为什么存在、它的本质是什么、有哪些基本类型。

核心内容包括：

- **索引的本质**：用空间换时间的辅助数据结构，将查询复杂度从 O(n) 降低到 O(log n) 甚至 O(1)
- **索引的分类体系**：按数据结构（B+树 / LSM树 / 哈希 / 位图 / 倒排）、按存储方式（聚簇 / 非聚簇 / 覆盖索引）、按功能特性（唯一 / 组合 / 前缀 / 函数索引）三维分类
- **底层数据结构演进**：从二叉搜索树 → AVL/红黑树 → B树 → B+树 → LSM-Tree，每一步都是为了解决前一代的瓶颈
- **索引选择的决策框架**：根据查询模式、写入模式、数据规模选择最优索引类型
- **索引失效的常见原因**：函数作用、隐式类型转换、前导通配符、最左前缀违背等，附带 EXPLAIN 分析方法
- **索引设计的实战原则**：索引数量控制、选择性评估、联合索引列顺序、覆盖索引优化

> 适合读者：所有层级。即使是经验丰富的 DBA，决策框架和索引失效分析也值得回顾。

### 第二部分：索引结构的技术演进

索引的"史"——从 1950 年代的顺序文件到 2020 年代的学习型索引，七十年的技术迭代背后是存储硬件与查询需求的持续博弈。

演进脉络：

| 时代 | 存储介质 | 主导结构 | 核心瓶颈 |
|------|----------|----------|----------|
| 1960s | 磁鼓/早期磁盘 | 顺序文件、ISAM | 随机 I/O 极慢 |
| 1970-1980s | 机械磁盘 | B树/B+树 | 寻道时间 |
| 1980-1990s | 机械磁盘 | 哈希索引 | 哈希冲突、扩展困难 |
| 1996-2010s | 机械磁盘→SSD | LSM树 | 写放大、空间放大 |
| 2010s-至今 | SSD/NVMe | B+树 + LSM树并存 | CPU 成为瓶颈 |
| 2020s+ | 持久内存(CXL) | 自适应/学习型索引 | 软件栈延迟 |

每个阶段的关键突破：

1. **B树的发明（1970）**：多路平衡树将树高压缩到 3-4 层，查询 1 亿条记录只需 3 次 I/O
2. **LSM树的革命（1996-2012）**：将随机写转化为顺序写，写吞吐量提升 100 倍以上
3. **SSD 时代的重构（2010s）**：随机 I/O 不再是灾难，索引设计假设被重写
4. **学习型索引的探索（2018+）**：用机器学习模型替代 B+树，空间压缩 200 倍
5. **持久内存索引（2020s+）**：字节寻址带来全新的设计空间

> 适合读者：希望理解"为什么"的读者。了解演进历史能帮助你在技术选型时做出更有依据的判断。

### 第三部分：索引结构的关键指标

索引的"术"——用量化指标评估不同索引结构的性能表现，为技术选型提供数据支撑。

核心指标框架：

| 指标 | 含义 | 为什么重要 |
|------|------|------------|
| **点查询延迟** | 等值查找的 I/O 次数和 CPU 时间 | 决定 OLTP 查询响应时间 |
| **范围查询效率** | 范围扫描的 I/O 次数 | 影响报表和分析查询性能 |
| **写入放大** | 逻辑写入与物理写入的比值 | 直接影响 SSD 寿命和写吞吐量 |
| **空间放大** | 实际占用空间与逻辑数据量的比值 | 决定存储成本 |
| **读放大** | 一次查询需要读取的磁盘页数 | 影响查询延迟和 I/O 带宽占用 |

不同索引结构在这些指标上的表现：

         点查询    范围查询    写入    空间效率   并发性
B+树      ★★★★     ★★★★★    ★★      ★★★★     ★★★★
LSM树     ★★★      ★★★★     ★★★★★   ★★★      ★★★
哈希索引   ★★★★★    ✗        ★★★★    ★★★      ★★
位图索引   ★★       ★★★      ★★      ★★★★     ★★

> 适合读者：需要做技术选型或性能调优的工程师。这些指标是评估索引方案的通用语言。
