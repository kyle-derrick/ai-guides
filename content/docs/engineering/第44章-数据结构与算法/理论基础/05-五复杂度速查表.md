## 时间复杂度与空间复杂度速查表

### 1. 为什么需要复杂度速查表

在算法设计与工程实践中，时间复杂度和空间复杂度是评估程序效率的两大核心度量。面对具体问题时，开发者往往需要快速判断：某种数据结构的操作耗时如何？选择哪种算法能在时间和空间之间取得最佳平衡？复杂度速查表正是为此而生的工具——它将散落在教科书各处的复杂度信息集中呈现，让决策变得高效而准确。

本文覆盖常用数据结构与排序/搜索/图/字符串等经典算法的时间复杂度与空间复杂度，帮助读者在面试、系统设计、性能调优等场景下快速参考。

### 2. 复杂度基础概念速览

#### 2.1 大 O 表示法

大 O（Big-O）表示算法在最坏情况下的渐进时间或空间上界。核心规则：

- **忽略常数和低阶项**：O(2n + 1) 简化为 O(n)
- **关注增长趋势**：O(log n) < O(n) < O(n log n) < O(n²) < O(2ⁿ)
- **最坏情况分析**：大 O 描述的是输入规模趋近无穷时的上界

#### 2.2 常见复杂度排序（从快到慢）

| 符号 | 名称 | 典型代表 | 操作次数（n=10⁶） |
|------|------|----------|-------------------|
| O(1) | 常数 | 哈希表查找 | 1 |
| O(log n) | 对数 | 二分查找 | ≈20 |
| O(n) | 线性 | 遍历数组 | 10⁶ |
| O(n log n) | 线性对数 | 归并排序 | ≈2×10⁷ |
| O(n²) | 平方 | 冒泡排序 | 10¹² |
| O(n³) | 立方 | Floyd-Warshall | 10¹⁸ |
| O(2ⁿ) | 指数 | 子集枚举 | 不可计算 |
| O(n!) | 阶乘 | 全排列 | 不可计算 |

#### 2.3 空间复杂度

空间复杂度衡量算法运行时所需的额外内存空间（不计输入本身）。常见类别：

| 类别 | 含义 | 示例 |
|------|------|------|
| O(1) | 原地操作，固定额外空间 | 两指针交换、位运算 |
| O(n) | 额外空间与输入成线性关系 | 新建辅助数组、哈希表 |
| O(n²) | 额外空间与输入平方关系 | 邻接矩阵 |
| O(log n) | 递归栈深度为对数 | 二分查找递归、平衡树 |

### 9. 实际工程中的复杂度选择指南

#### 9.1 数据规模与复杂度适配

| 数据规模 n | 可接受的最高复杂度 | 推荐算法 |
|-----------|-------------------|----------|
| ≤ 20 | O(2ⁿ), O(n!) | 状态压缩 DP、全排列枚举 |
| ≤ 500 | O(n³) | Floyd-Warshall、矩阵链乘法 |
| ≤ 5,000 | O(n²) | 动态规划、Dijkstra 邻接矩阵 |
| ≤ 10⁶ | O(n log n) | 归并排序、快速排序、线段树 |
| ≤ 10⁸ | O(n) | 线性扫描、哈希表、前缀和 |
| ≤ 10⁹ | O(log n) | 二分查找、快速幂 |
| ≤ 10¹⁸ | O(1) | 数学公式、位运算 |

#### 9.2 数据结构选型决策树

需要快速查找？
├── 数据有序且需要范围查询 → 平衡BST (红黑树 / AVL)
├── 只需精确查找 → 哈希表 O(1)
├── 需要前缀匹配 → Trie 字典树
└── 内存受限 → 位图 / 布隆过滤器

需要频繁插入删除？
├── 两端操作 → 双端队列
├── 优先级驱动 → 堆 (优先队列)
├── 中间频繁操作 → 链表
└── 随机位置 → 动态数组（均摊 O(1) 尾操作）

需要范围查询？
├── 一维区间 → 线段树 / 树状数组
├── 二维区间 → 二维线段树 / 二维前缀和
└── 动态区间 → 树状数组（单点修改+前缀查询）

#### 9.3 常见误区与陷阱

**误区一：忽视常数因子**  
O(n log n) 的归并排序在小数据上可能不如 O(n²) 的插入排序快，因为归并的递归开销和内存分配代价更高。工程中常用混合排序（如 Introsort：快排 + 堆排 + 插入排序）来兼顾理论与实践。

**误区二：混淆平均与最坏**  
快速排序平均 O(n log n) 但最坏 O(n²)。在实时系统或安全敏感场景（如防止对手构造恶意输入），应选择最坏 O(n log n) 的归并排序或堆排序。

**误区三：忽略空间复杂度**  
递归实现的 DFS 可能达到 O(V) 的栈空间深度，在大图上可能栈溢出。此时需改为显式栈的迭代实现。

**误区四：哈希表并非总是 O(1)**  
自定义哈希函数质量差、或负载因子过高时，碰撞增多导致退化为 O(n)。工程中应控制负载因子在 0.75 以下，并选用高质量哈希函数。

**误区五：只看单次操作复杂度**  
均摊分析和摊销分析关注的是序列操作的总代价。动态数组尾部插入均摊 O(1) 意味着连续 n 次插入总代价 O(n)，但单次可能触发 O(n) 的扩容。

#### 9.4 测量真实性能的要点

理论复杂度描述的是渐进行为，实际性能还受以下因素影响：

- **缓存友好性**：数组比链表对 CPU 缓存更友好，连续内存访问命中缓存的概率更高
- **分支预测**：条件分支少的算法在现代 CPU 上流水线效率更高
- **内存分配**：频繁的 `new`/`malloc` 会带来系统调用开销
- **I/O 开销**：外部排序（归并）在磁盘上可能优于内存排序
- **并行化潜力**：归并排序天然适合并行化，而快速排序的并行版本更复杂

### 11. 总结

复杂度分析是算法设计的"第一性原理"。掌握这张速查表，能够在面对具体问题时：

1. **快速评估**——在几种候选方案中挑选理论复杂度最优的
2. **合理预估**——根据数据规模判断算法是否可行
3. **权衡取舍**——在时间与空间、实现复杂度与性能之间做 trade-off
4. **避免陷阱**——理解平均与最坏、均摊与单次、理论与实际的差异

记住：复杂度是工具而非教条。好的工程师不仅知道 O(n log n) 比 O(n²) 快，还知道何时 O(n²) 反而是更好的选择——在数据足够小、实现足够简单、可读性更重要的时候。理论指导实践，实践检验理论，二者相辅相成。
