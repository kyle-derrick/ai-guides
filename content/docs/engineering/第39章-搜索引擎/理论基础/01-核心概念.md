搜索引擎是现代互联网基础设施的核心组件，从Google的网页搜索到Elasticsearch驱动的电商商品检索，从手机输入法的联想词到法律文书的全文检索，搜索引擎技术已经渗透到几乎所有软件系统中。理解搜索引擎的核心原理，不仅是后端工程师的基本功，也是构建任何信息密集型系统的基石。

本章从倒排索引的底层数据结构出发，系统性地讲解搜索引擎的核心理论，涵盖从经典信息检索到现代向量检索的完整知识体系。

## 1. 倒排索引（Inverted Index）

倒排索引是搜索引擎最核心的数据结构，几乎所有搜索引擎——无论是全文检索引擎还是日志分析平台——都建立在倒排索引之上。其基本思想与书籍的索引类似：不是从文档出发查找包含哪些词，而是从词出发查找出现在哪些文档中。

### 1.1 正排索引 vs 倒排索引

| 对比维度 | 正排索引（Forward Index） | 倒排索引（Inverted Index） |
|----------|--------------------------|---------------------------|
| 查询方向 | 文档 → 词条列表 | 词条 → 文档列表 |
| 查询效率 | O(N) 全扫描 | O(1) 词典查找 + 列表扫描 |
| 适用场景 | 文档聚类、推荐系统 | 关键词搜索、全文检索 |
| 空间开销 | 较低 | 较高（需维护词典+列表） |
| 更新代价 | 低（直接修改文档） | 高（需更新多个列表） |

正排索引（Forward Index）：
  文档1 → ["搜索引擎", "是", "核心", "技术"]
  文档2 → ["倒排索引", "是", "搜索引擎", "基础"]
  文档3 → ["分词", "影响", "搜索引擎", "质量"]

倒排索引（Inverted Index）：
  "搜索引擎" → [(文档1, 词频2, 位置[1,6]), (文档2, 词频1, 位置[3]), (文档3, 词频1, 位置[3])]
  "倒排索引" → [(文档2, 词频1, 位置[0])]
  "分词"     → [(文档3, 词频1, 位置[0])]
  "核心"     → [(文档1, 词频1, 位置[2])]
  "是"       → [(文档1, 词频1, 位置[1]), (文档2, 词频1, 位置[1])]

### 1.2 倒排索引的内部结构

一个完整的倒排索引由两大部分组成：

**词典（Dictionary）**：存储所有出现过的词条（Term），是倒排索引的查找入口。词典的实现方案直接影响查询性能：

| 实现方式 | 查找时间 | 空间开销 | 适用场景 |
|---------|---------|---------|---------|
| 哈希表 | O(1) | 高 | 追求极速查找，内存充足 |
| B+树 | O(log N) | 中 | 需要前缀查询和范围查询 |
| Trie树 | O(L)，L为词长 | 中 | 前缀匹配、自动补全 |
| FST（有限状态转换器） | O(L) | 极低 | Lucene默认方案，兼顾速度和压缩 |

**倒排列表（Posting List）**：对每个词条，记录包含该词条的所有文档ID及其出现位置、频率等信息。倒排列表通常按文档ID升序排列，这一特性对后续的压缩和查询合并至关重要。

```python
# 倒排列表的基本结构
from dataclasses import dataclass, field
from typing import List, Dict

@dataclass
class Posting:
    doc_id: int              # 文档ID
    term_freq: int           # 词频（该词条在文档中出现的次数）
    positions: List[int]     # 词条在文档中出现的位置列表
    field_mask: int = 0      # 字段位掩码（标题=1, 正文=2, URL=4...）

@dataclass
class TermDictionary:
    """词典与倒排列表的组合结构"""
    # 词典：term → posting_list
    dictionary: Dict[str, List[Posting]] = field(default_factory=dict)
    # 词条总数
    total_terms: int = 0
    # 文档总数
    total_docs: int = 0
    
    def postings(self, term: str) -> List[Posting]:
        """获取某个词条的倒排列表"""
        return self.dictionary.get(term, [])
```

### 1.3 构建过程

构建倒排索引是一个多步骤的流水线，每个步骤都对最终的搜索质量产生影响：

**步骤一：文档预处理**
- 去除HTML标签、JavaScript/CSS代码、特殊字符
- 统一编码（UTF-8规范化）
- 过滤停用词（如英文的 "the"、"is"，中文的 "的"、"了"）

**步骤二：分词（Tokenization）**
- 将文档文本切分为词条序列
- 中文分词需要专用的分词器，详见第2节

**步骤三：归一化（Normalization）**
- 大小写转换：`"Search"` → `"search"`
- 词干提取：`"running"` → `"run"`
- 同义词扩展：`"手机"` → `"手机" OR "移动电话" OR "智能手机"`

**步骤四：索引写入**
- 将（词条, 文档ID, 位置信息, 字段信息）写入倒排列表
- 更新词典统计信息（文档频率DF等）

**索引构建的性能优化**：
- **批量写入**：积累一定量的文档后一次性写入，减少磁盘IO
- **多段合并**：类似LSM-Tree的思想，先写入小段（Segment），后台定期合并
- **并行构建**：将文档集分片，多线程/多机器并行构建后合并

### 1.4 倒排列表压缩技术

倒排列表可能非常庞大（一个十亿级网页的索引中，常见词的倒排列表可达数GB），压缩技术直接影响系统的存储成本和查询性能。压缩的核心思想是利用倒排列表的有序性和数值分布特征：

**变长编码（Variable Byte Encoding）**：用1-4个字节编码不同大小的整数。小数字用少字节，大数字用多字节，每个字节的最高位作为延续标志位。

数字 5    → 00000101          （1字节）
数字 200  → 00000001 01100100  （2字节）
数字 1500 → 00000000 00001011 11001100 （3字节）

**差值编码（Delta Encoding）**：由于倒排列表按文档ID升序排列，相邻ID之间的差值远小于ID本身，编码更小的数字需要更少的字节。实际应用中通常先做差值编码，再做变长编码。

**PForDelta**：利用文档ID差值分布的局部性——大部分差值集中在较小的范围内，只有少数异常值。将这些"异常值"单独存储，主体部分用定长位压缩，查询时通过位操作快速解码。

**Roaring Bitmap**：对稠密区域（连续文档ID）使用位图存储，对稀疏区域使用有序数组，自动选择最优编码策略。Elasticsearch和Apache Spark均已采用Roaring Bitmap。

| 压缩方法 | 压缩比 | 解压速度 | 适用场景 |
|---------|-------|---------|---------|
| 变长编码 | 中 | 快 | 通用场景 |
| 差值+变长 | 高 | 快 | 文档ID分布均匀 |
| PForDelta | 极高 | 极快 | 大规模搜索引擎 |
| Roaring Bitmap | 高 | 极快 | 集合运算（交集/并集） |

### 1.5 布尔查询与列表合并

当用户输入多个关键词时（如 `"搜索引擎" AND "倒排索引"`），搜索引擎需要对多个倒排列表进行合并操作。列表已经按文档ID排序，因此合并操作可以在线性时间内完成：

倒排列表A: "搜索引擎" → [1, 3, 5, 7, 9]
倒排列表B: "倒排索引" → [2, 3, 7, 11]

AND（求交集）: [3, 7]                    # 两个列表都包含的文档
OR （求并集）: [1, 2, 3, 5, 7, 9, 11]   # 至少一个列表包含的文档
NOT（求差集）: [1, 5, 9]                 # 在A中但不在B中的文档

**跳表指针（Skip Pointers）加速**：在倒排列表上每隔固定间隔（如每128个元素）设置一个跳表指针。当两个列表长度差异很大时（如短列表长度为M，长列表长度为N），利用跳表指针可以将时间复杂度从O(M+N)降低到接近O(M×log(N/M)+M)。

```python
def intersect_with_skips(list_a, list_b, skip_interval=128):
    """带跳表指针的交集运算"""
    result = []
    i, j = 0, 0
    len_a, len_b = len(list_a), len(list_b)
    
    while i < len_a and j < len_b:
        if list_a[i] == list_b[j]:
            result.append(list_a[i])
            i += 1
            j += 1
        elif list_a[i] < list_b[j]:
            # 利用跳表指针快速跳过
            if i + skip_interval < len_a and list_a[i + skip_interval] <= list_b[j]:
                i += skip_interval
            else:
                i += 1
        else:
            if j + skip_interval < len_b and list_b[j + skip_interval] <= list_a[i]:
                j += skip_interval
            else:
                j += 1
    return result
```

## 2. 分词（Tokenization）

分词是将连续文本切分为独立词条的过程，是影响搜索质量的关键环节。分词质量直接决定了召回率——分错的词无法被正确匹配到。

### 2.1 英文分词

英文分词相对简单，主要基于空格和标点进行切分，但完整的英文分词流程远不止于此：

**词干提取（Stemming）**：将词汇还原为词干形式，基于规则的启发式算法，速度快但可能过度还原。典型算法是Porter Stemmer，通过一系列后缀替换规则（如 `-ing` → 删除，`-ational` → `-ate`）实现。

"running" → "run"
"studies" → "studi"    （注意：Porter可能还原过度）
"organization" → "organ"

**词形还原（Lemmatization）**：基于词典和语法分析的精确还原，考虑了词性（名词/动词/形容词），还原结果更准确但计算成本更高。

"better" → "good"      （词形还原能识别不规则变形）
"running" → "run"
"went"   → "go"

**停用词过滤**：去除高频低信息量词汇。英文中最常见的停用词（the, is, at, which, on）在文档中出现频率极高，区分度极低，保留它们只会增加索引体积而无搜索收益。但在短语搜索中（如 "to be or not to be"），停用词反而有意义。

**N-gram分词**：生成连续字符组合以支持部分匹配。如 `"搜索引擎"` 的Bigram为 `["搜索", "索引", "引擎"]`，Trigram为 `["搜索引", "索引擎"]`。N-gram的主要用途是解决子串匹配和拼写纠错。

### 2.2 中文分词

中文没有天然的分隔符，分词是中文搜索的核心挑战。同一个句子可能有多种合理切分方式，例如：

"南京市长江大桥"
切法A: "南京市/长江大桥"  （南京市长江大桥）
切法B: "南京/市长/江大桥"  （南京市长/江大桥）

**基于词典的方法**：

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 最大正向匹配（FMM） | 从左到右，每次匹配词典中最长的词 | 实现简单，速度快 | 无法处理歧义 |
| 最大逆向匹配（BMM） | 从右到左，每次匹配词典中最长的词 | 对某些歧义更准 | 同样有盲区 |
| 双向最大匹配 | 同时跑FMM和BMM，选择切分词数少的 | 一定程度解决歧义 | 计算量翻倍 |

```python
def forward_max_match(text, dictionary, max_len=5):
    """最大正向匹配分词"""
    result = []
    while text:
        for length in range(min(max_len, len(text)), 0, -1):
            word = text[:length]
            if word in dictionary or length == 1:
                result.append(word)
                text = text[length:]
                break
    return result

# 使用示例
dict_words = {"搜索引擎", "核心", "技术", "索引", "倒排"}
# "搜索引擎核心技术" → ["搜索引擎", "核心", "技术"]
```

**基于统计的方法**：

隐马尔可夫模型（HMM）将分词建模为序列标注问题——每个字符被标注为B（词首）、M（词中）、E（词尾）或S（单字词）。通过Viterbi算法求解最优标注序列。HMM的优势在于能发现词典中未收录的新词（OOV问题）。

条件随机场（CRF）在HMM基础上引入更丰富的特征模板（如前后字的组合特征），准确率更高，是统计分词方法的巅峰。

**基于深度学习的方法**：

BiLSTM+CRF是当前主流的分词模型：BiLSTM负责学习上下文特征表示，CRF层负责学习标签之间的转移约束（如B后面不能直接跟E）。BERT-based分词器利用预训练语言模型的上下文表示，分词准确率进一步提升，但推理速度较慢。

### 2.3 分词质量评估

分词质量直接决定搜索效果，评估指标包括：

- **准确率（Precision）**：正确切分的词数 / 切分出的总词数
- **召回率（Recall）**：正确切分的词数 / 标注中的总词数
- **F1值**：2 × Precision × Recall / (Precision + Recall)
- **分词速度**：每秒处理字符数，影响索引构建效率

工业实践中，分词器的选择需要在准确率和速度之间权衡：在线搜索场景对延迟敏感，可能选择速度更快的词典分词+HMM；离线分析场景可以容忍更高延迟，可以使用CRF或BERT分词器。

### 2.4 主流分词器对比

| 分词器 | 语言 | 算法 | 速度 | 准确率 | 特点 |
|--------|------|------|------|--------|------|
| IK Analyzer | 中文 | 词典+HMM | 快 | 中 | Elasticsearch最常用 |
| jieba | 中文 | 词典+HMM | 快 | 中高 | Python生态首选 |
| HanLP | 中文 | CRF/DNN | 中 | 高 | 功能全面 |
| Jieba分词 | 中文 | 词典+HMM | 快 | 中 | 社区活跃 |

| Elasticsearch内置 | 多语言 | Unicode标准 | 快 | 中 | 开箱即用 |

## 3. BM25排序算法

BM25（Best Matching 25）是信息检索领域最经典的排序算法，由Stephen Robertson等人在1994年的TREC（Text REtrieval Conference）评测中提出。尽管已经过去三十年，BM25仍然是Elasticsearch、Solr等主流搜索引擎的默认排序算法，其核心思想至今仍在被深度学习排序模型作为基线特征使用。

### 3.1 TF-IDF基础

BM25是对TF-IDF的改进，理解TF-IDF是理解BM25的前提：

**TF（词频，Term Frequency）**：词条在文档中出现的频率。直觉上，一个词在文档中出现越多，文档与该词的相关性越强。但简单的线性增长不合理——一个词出现10次并不意味着相关性是出现1次的10倍，因此需要进行对数平滑或饱和处理。

**IDF（逆文档频率，Inverse Document Frequency）**：词条在整个文档集中的稀有程度。一个词出现在越多文档中（如"的"、"是"），区分度越低；只出现在少数文档中的词（如"倒排索引"），区分度越高。

TF(t, d) = 词t在文档d中出现的次数（或经平滑处理的版本）
IDF(t) = log(N / df(t))
  N = 文档总数
  df(t) = 包含词t的文档数
TF-IDF(t, d) = TF(t, d) × IDF(t)

**TF-IDF的局限性**：
1. TF线性增长，没有饱和上限——长文档天然TF更高，但不意味着更相关
2. 未考虑文档长度差异——一篇10000字的文章和一篇100字的摘要，即使TF相同，相关性感知也不同
3. IDF公式在df接近N时可能出现负值

### 3.2 BM25公式详解

BM25通过引入两个关键参数解决了TF-IDF的局限性：

BM25(Q, D) = Σ IDF(qi) × [f(qi, D) × (k1 + 1)] / [f(qi, D) + k1 × (1 - b + b × |D|/avgdl)]

其中：
- `f(qi, D)`：查询词qi在文档D中的词频（TF）
- `|D|`：文档D的长度（词数）
- `avgdl`：文档集中所有文档的平均长度
- `k1`：词频饱和参数（通常取1.2-2.0），控制TF的饱和速度
- `b`：文档长度归一化参数（通常取0.75），控制文档长度的影响程度

**参数的物理含义**：

`k1` 控制TF的饱和行为。当 `k1=0` 时，BM25退化为纯IDF模型（完全忽略词频）；`k1` 越大，TF分量的影响越持久（饱和越慢）。下表展示不同k1值下TF分量的变化：

| 词频TF | k1=0.5 | k1=1.0 | k1=1.5 | k1=2.0 |
|--------|--------|--------|--------|--------|
| 1 | 1.00 | 1.00 | 1.00 | 1.00 |
| 2 | 1.33 | 1.50 | 1.60 | 1.67 |
| 5 | 1.54 | 1.88 | 2.00 | 2.08 |
| 10 | 1.60 | 1.96 | 2.08 | 2.14 |
| 50 | 1.64 | 2.00 | 2.12 | 2.17 |
| 100 | 1.65 | 2.00 | 2.13 | 2.18 |

可以看到，TF=50和TF=100时分量差异极小——BM25通过饱和机制有效抑制了长文档或重复堆砌关键词的作弊行为。

`b` 控制文档长度归一化的强度。当 `b=0` 时完全忽略文档长度差异；`b=1` 时完全归一化。0.75是经验最优值，在大多数场景下表现良好。当一篇文档长度远大于平均长度时，其得分会被显著惩罚。

### 3.3 BM25F：多字段扩展

实际搜索中，文档由多个字段组成（标题、正文、URL、标签等），每个字段的重要性不同。BM25F对不同字段赋予不同权重：

BM25F(Q, D) = Σ IDF(qi) × weighted_tf(qi, D)
weighted_tf = Σ(wf × tf(f)) / (1 + b_f × (len(f)/avglen(f) - 1))

其中 `wf` 是字段权重（标题权重通常设为10，正文设为1，URL设为0.5）。这样，标题中出现的查询词比正文中出现的查询词获得更高的分数。

### 3.4 Python实现

```python
import math
from collections import Counter

def bm25_score(query_terms, doc_terms, corpus_docs, k1=1.5, b=0.75):
    """计算BM25评分"""
    N = len(corpus_docs)
    avgdl = sum(len(d) for d in corpus_docs) / N
    score = 0.0
    doc_len = len(doc_terms)
    
    for term in query_terms:
        # 计算文档频率DF
        df = sum(1 for d in corpus_docs if term in d)
        if df == 0:
            continue
        
        # BM25的IDF公式（Robertson-Sparck Jones）
        idf = math.log((N - df + 0.5) / (df + 0.5) + 1)
        
        # TF分量（带饱和+长度归一化）
        tf = doc_terms.count(term)
        tf_norm = (tf * (k1 + 1)) / (tf + k1 * (1 - b + b * doc_len / avgdl))
        
        score += idf * tf_norm
    
    return score
```

### 3.5 BM25的局限性

尽管BM25是强大的基线算法，但它有明确的局限：

1. **无法捕捉语义**：BM25是纯词汇匹配，"汽车"和"轿车"被视为完全不同的词
2. **词袋模型假设**：忽略词序，"dog bites man"和"man bites dog"得分相同
3. **无法处理同义词扩展**：需要外部词典或查询改写才能处理
4. **对长查询效果差**：查询词过多时，噪音词会稀释相关信号

这些局限性正是后续章节中学习向量检索和语义搜索的动机。

## 4. Elasticsearch架构

Elasticsearch（简称ES）是基于Apache Lucene构建的分布式全文搜索引擎，其架构设计体现了搜索引擎工程化的核心思想。Lucene负责单机级别的索引和搜索，Elasticsearch在此基础上构建了分布式能力。

### 4.1 核心概念映射

| Elasticsearch | Lucene | 关系数据库 |
|--------------|--------|-----------|
| Index（索引） | 多个Segment的集合 | Database |
| Document（文档） | 一条记录 | Row |
| Field（字段） | 一个字段 | Column |
| Shard（分片） | 一个独立的Lucene实例 | Partition |
| Mapping（映射） | Schema定义 | Table Schema |

### 4.2 分布式架构

Elasticsearch通过分片（Shard）和副本（Replica）实现水平扩展：

┌─────────────────────────────────────────────────┐
│              Elasticsearch Cluster               │
│                                                  │
│  ┌──────────────┐  ┌──────────────┐              │
│  │   Node 1     │  │   Node 2     │              │
│  │  (Master)    │  │  (Data)      │              │
│  │ ┌──────────┐ │  │ ┌──────────┐ │              │
│  │ │ Shard P0 │ │  │ │ Shard P1 │ │              │
│  │ │ Shard R2 │ │  │ │ Shard R0 │ │              │
│  │ └──────────┘ │  │ └──────────┘ │              │
│  └──────────────┘  └──────────────┘              │
│                                                  │
│  ┌──────────────┐                                │
│  │   Node 3     │                                │
│  │  (Data)      │                                │
│  │ ┌──────────┐ │                                │
│  │ │ Shard P2 │ │                                │
│  │ │ Shard R1 │ │                                │
│  │ └──────────┘ │                                │
│  └──────────────┘                                │
└─────────────────────────────────────────────────┘

P0/P1/P2 = 主分片（Primary Shard），负责写入
R0/R1/R2 = 副本分片（Replica Shard），负责读和容灾

- **分片（Shard）**：索引被水平拆分为多个分片，每个分片是一个独立的Lucene实例。分片数在索引创建时确定，后续不可更改（但可以Reindex）。
- **副本（Replica）**：每个主分片的完整拷贝，提供高可用（主分片所在节点宕机时自动提升副本为主分片）和读扩展（搜索请求可由副本处理）。
- **路由算法**：写入/查询时通过 `hash(_id) % num_primary_shards` 确定目标分片，确保同一文档始终落在同一分片。

### 4.3 写入流程

Elasticsearch的写入是一个多级缓冲+持久化的流程：

客户端写入请求
      │
      ▼
协调节点（路由到目标分片）
      │
      ▼
主分片执行写入
  ├── 写入内存Buffer（不可搜索）
  ├── 写入Translog（事务日志，保证持久化）
  │
  ├── 每1秒执行Refresh ──→ 新Segment生成（可搜索！）
  │                        此时数据在内存中，尚未持久化
  │
  └── 每30分钟/Translog达阈值执行Flush
       ├── 将内存Segment写入磁盘（fsync）
       ├── 清空Translog
       └── 数据真正持久化
      │
      ▼
副本分片同步（同步/异步）

**关键时间参数**：
- `refresh_interval`：默认1秒，决定了搜索的近实时延迟。可设为 `-1` 关闭自动Refresh以提高批量写入性能。
- `translog.durability`：`request`（每次写入都同步translog，安全但慢）或 `async`（异步，快但可能丢最近几秒数据）。
- `index.translog.flush_threshold_size`：默认512MB，translog达到此大小时触发Flush。

### 4.4 搜索流程（Query Then Fetch）

客户端搜索请求
      │
      ▼
协调节点广播到所有相关分片
      │
      ▼
每个分片执行本地搜索
  ├── 构建查询计划
  ├── 在倒排索引中检索匹配文档
  ├── 用BM25等算法计算相关性分数
  └── 返回 Top-K 的文档ID + 分数
      │
      ▼
协调节点合并所有分片结果
  ├── 全局排序（取Top-N）
  └── 确定需要完整文档的分片
      │
      ▼
向相关分片请求完整文档内容（Fetch阶段）
      │
      ▼
返回最终结果给客户端

**搜索过程的性能优化**：
- **Query阶段只返回文档ID和分数**：减少网络传输量
- **分片内提前终止**：如果某个分片的第N个分数已经远低于全局阈值，可以跳过剩余文档
- **SearchType**：`query_then_fetch`（默认，两阶段）和 `query_fetch`（一次性，适合小结果集）

### 4.5 Segment与不可变性

Lucene的核心数据结构是Segment——每个Segment维护独立的倒排索引、存储结构和元数据。Segment一旦写入磁盘就不可修改（immutable），这一设计带来显著优势：

- **无需加锁**：读操作不需要考虑写入冲突
- **缓存友好**：OS可以安全地缓存整个文件
- **高效的垃圾回收**：删除文档只需在位图中标记，真正清理在合并时进行

**段合并（Segment Merging）**：随着时间推移，系统中会积累大量小Segment，搜索时需要遍历所有Segment，性能下降。后台的Merge线程会定期将多个小Segment合并为大Segment，同时物理删除被标记为删除的文档。合并策略（TieredMergePolicy）会权衡合并收益和IO开销。

## 5. 搜索排序与相关性

现代搜索引擎的排序远不止BM25一个因素，而是融合了文本相关性、文档质量、用户行为、时效性等多维度信号的综合排序系统。

### 5.1 多维度排序因子

| 因子类别 | 具体信号 | 权重分配 | 适用场景 |
|---------|---------|---------|---------|
| 文本相关性 | BM25分数、短语匹配、同义词匹配 | 高（基础信号） | 所有搜索场景 |
| 字段权重 | 标题匹配 > 正文匹配 > URL匹配 | 中高 | 全文搜索 |
| 文档质量 | PageRank、域名权威度、内容原创度 | 中 | 网页搜索 |
| 新鲜度 | 文档发布时间/最后修改时间 | 中（场景相关） | 新闻搜索 |
| 用户行为 | 点击率、停留时长、跳出率 | 中 | 商业搜索引擎 |
| 个性化 | 用户画像、搜索历史、地理位置 | 低-中 | 个性化搜索 |

### 5.2 Learning to Rank（LTR）

当排序信号越来越多时，手动调权重变得不可行，机器学习排序（LTR）应运而生。LTR是现代搜索引擎的标配技术，分为三大范式：

**Pointwise**：将排序问题转化为回归/分类问题。每个（查询, 文档）对独立预测一个相关性分数，不考虑文档间的相对关系。常用模型：线性回归、随机森林。

**Pairwise**：学习文档对的偏序关系——给定两个文档，模型判断哪个更相关。常用模型：RankSVM、RankNet、RankBoost。目标是最小化逆序对数量。

**Listwise**：直接优化整个排序列表的质量指标（如NDCG）。这是最直接对应搜索目标的范式，常用模型：LambdaMART、ListNet、AdaRank。LambdaMART是工业界使用最广泛的LTR模型。

```python
import lightgbm as lgb

def train_lambdamart(train_data, feature_cols, label_col):
    """LambdaMART训练示例"""
    # 构建训练数据
    X_train = train_data[feature_cols]
    y_train = train_data[label_col]
    group_sizes = train_data.groupby('query_id').size().tolist()
    
    # 创建LightGBM数据集
    lgb_train = lgb.Dataset(X_train, label=y_train, group=group_sizes)
    
    # LambdaMART参数
    params = {
        'objective': 'lambdarank',
        'metric': 'ndcg',
        'ndcg_eval_at': [1, 3, 5, 10],
        'learning_rate': 0.05,
        'num_leaves': 64,
        'min_data_in_leaf': 50,
        'feature_fraction': 0.8,
        'bagging_fraction': 0.8,
        'verbose': -1,
    }
    
    model = lgb.train(params, lgb_train, num_boost_round=500)
    return model
```

### 5.3 排序质量评估指标

搜索引擎的排序质量需要客观指标来衡量：

**MRR（Mean Reciprocal Rank）**：第一个正确结果排名的倒数的均值。适合"只要找到一个就够了"的场景（如导航型搜索）。

MRR = (1/|Q|) × Σ(1/rank_i)
例：查询1结果排第2，查询2排第1，查询3排第4
MRR = (1/3) × (1/2 + 1/1 + 1/4) = 0.583

**MAP（Mean Average Precision）**：所有相关文档的准确率的均值。考虑了多个相关文档的排序质量。

**NDCG（Normalized Discounted Cumulative Gain）**：最全面的排序评估指标，支持分级相关性标注（0=不相关，1=部分相关，2=高度相关），并用对数折扣因子惩罚排在后面的高相关文档。

DCG@k = Σ(i=1 to k) (2^rel_i - 1) / log2(i + 1)
NDCG@k = DCG@k / IDCG@k
（IDCG是理想排序下的DCG值，作为归一化基准）

**在线指标**：离线评估（NDCG等）与在线效果之间可能存在gap，因此还需要关注：
- 点击率（CTR）：搜索结果被点击的比例
- 无结果率：返回空结果的查询占比
- 查询改良率：用户在搜索后修改查询词的比例
- 首次点击位置：用户点击的第一个结果的排名

## 6. 查询理解与自动补全

搜索引擎不仅要匹配文档，还要理解用户的真实意图。用户输入的查询词往往简短、模糊、有拼写错误，查询理解是提升搜索体验的关键环节。

### 6.1 查询理解全流程

用户输入原始查询
      │
      ▼
拼写纠错（Spell Correction）
  "elasticsearch教成" → "elasticsearch教程"
      │
      ▼
查询改写（Query Rewriting）
  同义词扩展: "手机" → "手机 OR 移动电话 OR 智能手机"
  缩写展开:   "US" → "美国 OR United States"
      │
      ▼
意图识别（Intent Classification）
  导航型（找特定网站）/ 信息型（查资料）/ 交易型（买东西）
      │
      ▼
实体识别（NER）
  "北京到上海的机票" → [北京/城市], [上海/城市], [机票/商品]
      │
      ▼
查询扩展（Query Expansion）
  基于用户行为日志扩展相关查询词

### 6.2 拼写纠错

拼写纠错的核心思路是：给定一个错误词，从候选词中选出最可能的正确词。

**编辑距离（Levenshtein Distance）**：计算将一个字符串变换为另一个字符串所需的最少单字符编辑操作次数（插入、删除、替换）。

```python
def levenshtein_distance(s1, s2):
    """计算编辑距离"""
    m, n = len(s1), len(s2)
    dp = [[0] * (n + 1) for _ in range(m + 1)]
    for i in range(m + 1): dp[i][0] = i
    for j in range(n + 1): dp[0][j] = j
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            cost = 0 if s1[i-1] == s2[j-1] else 1
            dp[i][j] = min(dp[i-1][j] + 1,       # 删除
                          dp[i][j-1] + 1,         # 插入
                          dp[i-1][j-1] + cost)    # 替换
    return dp[m][n]

# levenshtein_distance("search", "serch") → 2
# levenshtein_distance("elasticsearch", "elasticsearh") → 2
```

**N-gram候选生成**：生成错误词的所有N-gram变体（删除一个字符、替换一个字符、插入一个字符），在搜索索引中查找匹配的正确词。

**语言模型打分**：对候选词用语言模型评估"这个查询词是否常见"，结合编辑距离和词频，选择得分最高的候选。

Elasticsearch中的Fuzzy查询支持配置 `fuzziness`（最大编辑距离，通常为1-2）和 `prefix_length`（前缀保持不变的字符数），结合前缀匹配实现高效的模糊搜索。

### 6.3 自动补全

自动补全的核心目标是在用户输入过程中实时返回最相关的补全建议。核心数据结构是Trie树（前缀树）的变体：

```python
from collections import defaultdict
import heapq

class TrieNode:
    def __init__(self):
        self.children = defaultdict(TrieNode)
        self.is_end = False
        self.frequency = 0

class AutoComplete:
    def __init__(self):
        self.root = TrieNode()
    
    def insert(self, query, frequency):
        """插入一条查询及其热度"""
        node = self.root
        for char in query:
            node = node.children[char]
        node.is_end = True
        node.frequency = frequency
    
    def suggest(self, prefix, top_k=10):
        """获取前缀为prefix的Top-K建议"""
        node = self.root
        for char in prefix:
            if char not in node.children:
                return []
            node = node.children[char]
        
        # DFS收集所有匹配项
        results = []
        self._dfs(node, prefix, results)
        # 按热度排序，返回Top-K
        results.sort(key=lambda x: -x[1])
        return [(q, f) for q, f in results[:top_k]]
    
    def _dfs(self, node, current, results):
        if node.is_end:
            results.append((current, node.frequency))
        for char, child in node.children.items():
            self._dfs(child, current + char, results)
```

**工业级自动补全的优化**：
- **分层Trie**：在每个节点预计算Top-K建议，避免DFS遍历
- **时间衰减**：近期热门查询权重更高，防止过期热词
- **地域感知**：不同地区的用户看到不同的补全建议
- **安全过滤**：过滤敏感词、违规内容
- **实时更新**：将用户日志实时聚合到Trie树中

## 7. 现代搜索引擎的演进方向

经典搜索引擎基于关键词匹配（BM25），但在语义理解方面存在根本局限。现代搜索引擎正在向混合检索方向演进。

### 7.1 向量检索与语义搜索

向量检索将文本编码为稠密向量（Embedding），通过向量相似度（余弦相似度、内积、欧氏距离）实现语义层面的匹配。

"苹果手机" → Embedding模型 → [0.12, -0.34, 0.56, ...]（768维向量）
"iPhone"   → Embedding模型 → [0.13, -0.32, 0.58, ...]（768维向量）

两者的余弦相似度 = 0.95（高度语义相关）
但关键词匹配的BM25分数 = 0（完全没有词汇重叠）

向量检索的索引结构（如HNSW、IVF）与倒排索引截然不同，详细内容将在后续章节展开。

### 7.2 混合检索

混合检索结合了关键词检索（BM25）和向量检索（Dense Retrieval）的优势：

- **BM25擅长**：精确匹配、专有名词、产品型号
- **向量检索擅长**：语义理解、同义词、跨语言匹配
- **混合策略**：加权融合（RRF、线性加权）、级联检索（先粗排后精排）

```python
def reciprocal_rank_fusion(rankings, k=60):
    """RRF: Reciprocal Rank Fusion混合排序"""
    doc_scores = defaultdict(float)
    for ranking in rankings:
        for rank, doc_id in enumerate(ranking):
            doc_scores[doc_id] += 1.0 / (k + rank + 1)
    return sorted(doc_scores.items(), key=lambda x: -x[1])
```
