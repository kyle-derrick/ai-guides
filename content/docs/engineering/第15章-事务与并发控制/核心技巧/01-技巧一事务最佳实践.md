## 技巧一：事务最佳实践

事务（Transaction）是数据库系统中最核心的抽象之一。一条看似简单的 `BEGIN ... COMMIT` 背后，涉及锁机制、日志协议、隔离级别、崩溃恢复等一系列复杂工程决策。事务用得好，数据一致性和系统性能兼得；用得轻率，则可能引发死锁、数据污染甚至服务不可用。

本节从实战角度出发，系统梳理事务设计的原则、模式、陷阱与优化手段，帮助你在生产环境中做出可靠且高效的事务决策。

---

### 1.1 事务的核心原则：ACID 再理解

ACID 是事务的四大基石，但在实际工程中，每个属性都有需要深入理解的细节。

| 属性 | 定义 | 工程含义 | 常见误解 |
|------|------|----------|----------|
| **原子性（Atomicity）** | 事务要么全部成功，要么全部回滚 | 需要 undo log 支持；长事务会占用大量回滚段 | 以为 `COMMIT` 只是"保存"，实际上它触发持久化 |
| **一致性（Consistency）** | 事务前后数据库从一个一致状态转换到另一个 | 依赖应用层约束（外键、触发器、校验逻辑） | 把一致性完全交给数据库，应用层不做校验 |
| **隔离性（Isolation）** | 并发事务之间互不干扰 | 由隔离级别和锁/多版本机制实现 | 默认 `REPEATABLE READ` 就万无一失 |
| **持久性（Durability）** | 已提交的数据不因崩溃丢失 | 依赖 WAL（Write-Ahead Log）和 fsync | 以为 `COMMIT` 返回成功就一定持久化了（需关注 sync 策略） |

**关键洞察：** ACID 不是非黑即白的开关。不同的隔离级别在性能与正确性之间做不同的权衡。理解这个权衡，比背诵 ACID 定义重要得多。

### 1.2 选择正确的隔离级别

#### 隔离级别全景对比

```mermaid
graph LR
    A[READ UNCOMMITTED] -->|禁止脏读| B[READ COMMITTED]
    B -->|禁止不可重复读| C[REPEATABLE READ]
    C -->|禁止幻读| D[SERIALIZABLE]
```

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 锁开销 | 适用场景 |
|----------|------|-----------|------|--------|----------|
| READ UNCOMMITTED | ✓ | ✓ | ✓ | 最低 | 仅用于粗略统计、对精度无要求的场景 |
| READ COMMITTED | ✗ | ✓ | ✓ | 较低 | **大多数 OLTP 系统的默认选择** |
| REPEATABLE READ | ✗ | ✗ | ✓* | 中等 | 需要多次读取一致性的报表查询 |
| SERIALIZABLE | ✗ | ✗ | ✗ | 最高 | 金融转账、库存扣减等强一致性场景 |

> **注意：** MySQL InnoDB 的 REPEATABLE READ 通过间隙锁（Gap Lock）在很大程度上避免了幻读，但这不是标准 SQL 的要求，而是 InnoDB 的实现增强。

#### 选择建议

```text
决策路径：
1. 先问：应用层能否容忍短暂的不一致？
   → 能 → READ COMMITTED（性能最优）
   → 不能 → 继续

2. 再问：是否需要多次读取结果完全一致？
   → 是 → REPEATABLE READ
   → 否 → READ COMMITTED

3. 最后：是否存在不可重试的外部操作（如调用第三方 API）？
   → 是 → SERIALIZABLE 或 应用层乐观锁
   → 否 → 根据并发量权衡
```

**实战建议：** 不要盲目使用最高隔离级别。SERIALIZABLE 在高并发场景下会导致大量锁等待和超时。优先考虑 READ COMMITTED + 应用层校验的组合。

### 1.3 事务设计的七大原则

#### 原则一：事务尽可能短小

长事务是数据库性能的头号杀手。它会长时间持有锁，阻塞其他事务，消耗内存（undo log），甚至导致回滚段溢出。

**反面案例：** 在事务中调用外部 API

```python
# ❌ 错误示范：事务中包含外部 HTTP 调用
def transfer_money(from_id, to_id, amount):
    with db.transaction():
        debit(from_id, amount)
        # 外部 API 可能需要几秒甚至超时
        response = call_payment_gateway(from_id, amount)
        if response.ok:
            credit(to_id, amount)
        # 如果这里失败，整个事务回滚，但外部调用已执行
```

**正面案例：** 将外部操作移到事务外

```python
# ✅ 正确做法：外部操作在事务外，事务只管数据库
def transfer_money(from_id, to_id, amount):
    # 1. 先调用外部（事务外）
    response = call_payment_gateway(from_id, amount)
    if not response.ok:
        raise PaymentError("支付网关拒绝")

    # 2. 再执行数据库事务（极短）
    with db.transaction():
        debit(from_id, amount)
        credit(to_id, amount)
        record_ledger(from_id, to_id, amount, response.tx_id)
```

#### 原则二：减少事务内的工作量

```text
好的事务结构：
  BEGIN
    ① 读取必要的数据（最小范围）
    ② 执行业务判断
    ③ 写入变更（最小范围）
  COMMIT

坏的事务结构：
  BEGIN
    ① 大量查询（可能触发锁升级）
    ② 复杂计算（消耗内存和 CPU）
    ③ 等待用户输入 / 外部调用
    ④ 大批量写入
  COMMIT
```

#### 原则三：使用显式事务，避免隐式自动提交

大多数数据库驱动默认开启自动提交（autocommit），每条 SQL 自动成为一个独立事务。这在简单场景下没问题，但在需要多步操作原子性时，必须显式控制事务边界。

```python
# PostgreSQL（psycopg2）
conn.autocommit = False  # 关闭自动提交
try:
    cur.execute("UPDATE accounts SET balance = balance - 100 WHERE id = 1")
    cur.execute("UPDATE accounts SET balance = balance + 100 WHERE id = 2")
    conn.commit()
except Exception:
    conn.rollback()
    raise
finally:
    conn.autocommit = True
```

```java
// Spring Boot（声明式事务）
@Transactional(rollbackFor = Exception.class)
public void transfer(Long fromId, Long toId, BigDecimal amount) {
    accountRepository.debit(fromId, amount);
    accountRepository.credit(toId, amount);
    ledgerRepository.record(fromId, toId, amount);
}
```

#### 原则四：合理设置事务超时

长时间运行的事务应该有超时机制，避免无限期阻塞。

```sql
-- MySQL 设置事务超时（秒）
SET SESSION innodb_lock_wait_timeout = 10;

-- PostgreSQL 设置语句超时
SET statement_timeout = '10s';
```

#### 原则五：异常处理必须完善

事务中的异常处理不是可选的，而是必须的。未捕获的异常会导致连接泄漏或事务挂起。

```python
def safe_transfer(conn, from_id, to_id, amount):
    cur = conn.cursor()
    try:
        cur.execute("BEGIN")
        cur.execute("UPDATE accounts SET balance = balance - %s WHERE id = %s", (amount, from_id))
        cur.execute("UPDATE accounts SET balance = balance + %s WHERE id = %s", (amount, to_id))
        cur.execute("INSERT INTO ledger (from_id, to_id, amount) VALUES (%s, %s, %s)",
                     (from_id, to_id, amount))
        conn.commit()
    except Exception as e:
        conn.rollback()
        raise TransactionError(f"转账失败: {e}") from e
    finally:
        cur.close()  # 确保游标释放
```

#### 原则六：避免在事务中读取大结果集

在事务中执行 `SELECT * FROM large_table` 会持有快照或锁，增加内存压力和锁竞争。应使用分页或游标迭代。

```sql
-- ❌ 一次性加载全部数据
BEGIN;
SELECT * FROM orders WHERE status = 'pending';  -- 可能返回百万行
-- ... 处理 ...

-- ✅ 分批处理
BEGIN;
SELECT * FROM orders WHERE status = 'pending' LIMIT 1000 OFFSET 0;
-- 处理第一批
-- 分批提交或使用游标
```

#### 原则七：善用 SAVEPOINT 处理局部回滚

当事务中包含多个独立子操作，其中一个失败时不需要回滚整个事务时，使用 SAVEPOINT。

```sql
BEGIN;

INSERT INTO orders (user_id, total) VALUES (1, 299.00);
-- 保存点
SAVEPOINT after_order;

INSERT INTO order_items (order_id, product_id, qty) VALUES (1, 101, 2);
-- 如果优惠券码无效，只需回滚优惠部分
SAVEPOINT after_item;

INSERT INTO coupons (code, order_id) VALUES ('INVALID', 1);
-- 发现优惠券无效
ROLLBACK TO after_item;

-- 订单本身仍然有效
COMMIT;
```

### 1.4 常见事务模式与实战模板

#### 模式一：乐观锁（Optimistic Locking）

适用于读多写少、冲突概率低的场景。不加锁，通过版本号或条件判断检测冲突。

```sql
-- 方案 A：版本号
UPDATE products
SET stock = stock - 1, version = version + 1
WHERE id = 100 AND version = 5;
-- 影响行数为 0 说明被其他事务修改了

-- 方案 B：条件更新（CAS）
UPDATE accounts
SET balance = balance - 100
WHERE id = 1 AND balance >= 100;
-- 影响行数为 0 说明余额不足或已被修改
```

```python
# 应用层重试机制
MAX_RETRIES = 3

def optimistic_update(product_id, quantity):
    for attempt in range(MAX_RETRIES):
        product = db.query("SELECT stock, version FROM products WHERE id = %s", product_id)
        if product.stock < quantity:
            raise InsufficientStock()

        rows = db.execute(
            "UPDATE products SET stock = stock - %s, version = version + 1 "
            "WHERE id = %s AND version = %s",
            (quantity, product_id, product.version)
        )
        if rows > 0:
            return True  # 成功
        # 版本冲突，重试
        time.sleep(0.01 * (2 ** attempt))  # 指数退避
    raise OptimisticLockError("更新失败，冲突次数过多")
```

#### 模式二：悲观锁（Pessimistic Locking）

适用于写多、冲突概率高的场景。先加锁，再操作。

```sql
-- SELECT ... FOR UPDATE（行锁）
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;
-- 此时其他事务对 id=1 的 FOR UPDATE 会被阻塞
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;

-- SELECT ... FOR UPDATE NOWAIT（立即报错，不等待）
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE NOWAIT;
-- 如果 id=1 已被锁定，立即报错 ORA-00054 或类似错误
```

```sql
-- SELECT ... FOR UPDATE SKIP LOCKED（跳过已锁定的行）
-- 适用于队列消费场景
BEGIN;
SELECT * FROM job_queue
WHERE status = 'pending'
ORDER BY created_at
LIMIT 10
FOR UPDATE SKIP LOCKED;
-- 多个消费者可以并发处理不同行，不冲突
```

#### 模式三：补偿事务（Saga Pattern）

适用于分布式系统或跨服务事务。将大事务拆分为多个本地事务，每个都有对应的补偿操作。

```text
正向流程：
  创建订单 → 扣减库存 → 扣款 → 发货

补偿流程（任一步失败时）：
  失败于扣款     → 恢复库存 → 取消订单
  失败于发货     → 退款 → 恢复库存 → 取消订单
```

```python
class OrderSaga:
    def __init__(self):
        self.compensations = []

    def execute(self, order):
        try:
            # 步骤 1：创建订单
            order_id = self.create_order(order)
            self.compensations.append(lambda: self.cancel_order(order_id))

            # 步骤 2：扣减库存
            self.reserve_inventory(order.items)
            self.compensations.append(lambda: self.release_inventory(order.items))

            # 步骤 3：扣款
            self.charge_payment(order.user_id, order.total)
            self.compensations.append(lambda: self.refund_payment(order.user_id, order.total))

            # 步骤 4：发货（异步）
            self.schedule_delivery(order_id)

        except Exception as e:
            self.compensate()
            raise SagaFailed(f"订单创建失败，已回滚: {e}")

    def compensate(self):
        # 按逆序执行补偿操作
        for action in reversed(self.compensations):
            try:
                action()
            except Exception as ce:
                # 记录补偿失败，需要人工介入
                logger.critical(f"补偿操作失败: {ce}")
                self.alert_operations(ce)
```

#### 模式四：批量操作的事务拆分

大批量数据操作如果放在一个大事务中，会导致 undo log 膨胀、锁竞争加剧、恢复时间变长。应拆分为小批次。

```python
def batch_update(cursor, records, batch_size=500):
    """分批提交的大批量更新"""
    total = len(records)
    for i in range(0, total, batch_size):
        batch = records[i:i + batch_size]
        cursor.execute("BEGIN")
        for record in batch:
            cursor.execute(
                "UPDATE products SET price = %s WHERE id = %s",
                (record['price'], record['id'])
            )
        cursor.connection.commit()
        # 每批提交后释放锁和内存
```

### 1.5 事务陷阱与反模式

#### 陷阱一：事务中的幂等性缺失

网络超时或重试时，同一个事务可能被执行两次。如果没有幂等设计，就会造成重复扣款、重复发货等问题。

```sql
-- ❌ 非幂等：重试会导致重复扣款
UPDATE accounts SET balance = balance - 100 WHERE id = 1;

-- ✅ 幂等：使用唯一事务 ID 去重
INSERT INTO transactions (id, from_id, amount, created_at)
VALUES ('tx-20240101-abc', 1, 100, NOW())
ON CONFLICT (id) DO NOTHING;
-- 如果 id 已存在，不执行插入，也不会重复扣款
```

#### 陷阱二：跨连接的事务

事务绑定在数据库连接上。如果在事务执行过程中切换了连接，事务上下文就会丢失。

```python
# ❌ 错误：事务在连接 A 上，后续操作可能用了连接 B
with get_connection() as conn:
    conn.begin()
    conn.execute("INSERT INTO ...")
    # 某个 ORM 操作可能获取了另一个连接
    another_model.save()  # 这不在同一个事务中！
```

#### 陷阱三：DDL 语句隐式提交

在 MySQL 中，`ALTER TABLE`、`CREATE INDEX`、`DROP TABLE` 等 DDL 语句会隐式提交当前事务。

```sql
BEGIN;
INSERT INTO orders (user_id, total) VALUES (1, 99.00);
-- 以下 DDL 会隐式 COMMIT 上面的 INSERT
ALTER TABLE orders ADD COLUMN note VARCHAR(255);
-- 此时事务已经提交，后续的 ROLLBACK 无法撤销 INSERT
INSERT INTO orders (user_id, total) VALUES (2, 199.00);
ROLLBACK;
-- 只有第二条 INSERT 被回滚，第一条已经持久化
```

#### 陷阱四：连接池中的事务泄漏

从连接池获取的连接，如果未正确归还（未 commit/rollback），会导致事务挂起、连接耗尽。

```python
# ❌ 连接泄漏：异常路径未释放连接
def risky_operation():
    conn = pool.get_connection()
    conn.execute("BEGIN")
    conn.execute("INSERT INTO ...")
    raise SomeError()  # 异常！连接未归还

# ✅ 使用 context manager 确保归还
def safe_operation():
    with pool.connection() as conn:
        with conn.transaction():
            conn.execute("INSERT INTO ...")
        # 无论成功还是异常，连接都会归还
```

#### 陷阱五：N+1 查询中的事务范围过大

ORM 懒加载容易导致意外的大事务范围。

```python
# ❌ 事务范围被 ORM 懒加载撑大
@app.route('/order/<id>')
def get_order(id):
    order = Order.query.get(id)
    # 每个 item.product 的访问都可能触发新的查询
    # 如果整个请求在一个事务中，事务会持续到请求结束
    return render_template('order.html', order=order)

# ✅ 在只读场景关闭事务，或使用 eager loading
@app.route('/order/<id>')
def get_order(id):
    with db.session.no_autoflush:
        order = Order.query.options(
            joinedload(Order.items).joinedload(Item.product)
        ).get(id)
    return render_template('order.html', order=order)
```

### 1.6 性能优化实战

#### 1.6.1 减少锁竞争

```text
策略                        效果                          适用场景
─────────────────────────────────────────────────────────────
缩小事务范围                减少锁持有时间                所有场景
按固定顺序访问行            避免死锁                      批量更新
使用 READ COMMITTED         读不加锁                      OLTP 读多写少
索引优化                    锁精确到行而非表范围扫描      所有写操作
批量合并                    减少事务次数                  高频小写入
```

#### 1.6.2 连接池配置建议

```yaml
# 连接池关键参数
pool:
  min_size: 5              # 最小连接数，避免冷启动
  max_size: 20             # 最大连接数，不宜超过 CPU 核心数 * 2
  max_idle_time: 300       # 空闲连接超时（秒）
  connection_timeout: 10   # 获取连接超时（秒）
  validation_query: "SELECT 1"  # 连接健康检查
```

**为什么 max_size 有上限？** 每个连接占用内存（MySQL 约 256KB-1MB），且过多连接会导致 CPU 在上下文切换上浪费时间。经验公式：`max_connections = CPU核心数 * 2 + 磁盘数`。

#### 1.6.3 监控与诊断

```sql
-- MySQL：查看当前活跃事务
SELECT * FROM information_schema.INNODB_TRX
WHERE trx_state = 'RUNNING'
ORDER BY trx_started;

-- 查看锁等待
SELECT * FROM information_schema.INNODB_LOCK_WAITS;

-- 查看长事务（运行超过 60 秒的）
SELECT trx_id, trx_state, trx_started,
       TIMESTAMPDIFF(SECOND, trx_started, NOW()) AS duration_sec,
       trx_query
FROM information_schema.INNODB_TRX
WHERE TIMESTAMPDIFF(SECOND, trx_started, NOW()) > 60;
```

```sql
-- PostgreSQL：查看活跃事务
SELECT pid, state, query, query_start,
       now() - query_start AS duration
FROM pg_stat_activity
WHERE state != 'idle'
  AND query_start < now() - interval '30 seconds';

-- 查看锁冲突
SELECT blocked.pid AS blocked_pid,
       blocked.query AS blocked_query,
       blocking.pid AS blocking_pid,
       blocking.query AS blocking_query
FROM pg_stat_activity blocked
JOIN pg_locks blocked_locks ON blocked.pid = blocked_locks.pid
JOIN pg_locks blocking_locks ON blocking_locks.locktype = blocked_locks.locktype
  AND blocking_locks.relation = blocked_locks.relation
  AND blocking_locks.pid != blocked_locks.pid
JOIN pg_stat_activity blocking ON blocking_locks.pid = blocking.pid
WHERE NOT blocked_locks.granted;
```

```text
监控指标仪表盘建议：
  ① 活跃事务数量          → 异常升高说明有长事务或死锁
  ② 平均事务持续时间      → OLTP 场景应 < 500ms
  ③ 锁等待次数和时长      → 持续升高说明锁竞争严重
  ④ 死锁计数              → 增量突增需要排查
  ⑤ 回滚比例              > 5% 说明应用逻辑有问题
  ⑥ 连接池使用率          > 80% 需要扩容或优化
```

### 1.7 分布式事务：从原理到取舍

当事务需要跨越多个数据库或服务时，单机 ACID 无法直接保证。以下是主流方案的对比：

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|----------|
| 2PC（两阶段提交） | 强一致 | 低（阻塞式） | 中 | 同构数据库，内部系统 |
| 3PC | 强一致 | 略高于 2PC | 高 | 理论方案，实际少用 |
| TCC（Try-Confirm-Cancel） | 最终一致 | 高 | 高 | 金融核心链路 |
| Saga | 最终一致 | 高 | 中 | 跨服务业务流程 |
| 本地消息表 | 最终一致 | 高 | 低 | 异步通知、日志同步 |
| RocketMQ 事务消息 | 最终一致 | 高 | 低 | 消息驱动架构 |

**实践建议：** 大多数业务场景不需要分布式事务。优先考虑：能否通过合理拆分服务边界，将分布式事务转化为本地事务？如果不能，Saga 或 TCC 通常比 2PC 更实用。

### 1.8 测试事务代码

事务相关的代码必须经过充分测试，包括正常路径和异常路径。

```python
import pytest

class TestTransfer:
    def test_successful_transfer(self, db):
        """正常转账：两个账户余额正确变化"""
        with db.transaction():
            transfer(db, from_id=1, to_id=2, amount=100)
        assert db.query("SELECT balance FROM accounts WHERE id = 1") == 900
        assert db.query("SELECT balance FROM accounts WHERE id = 2") == 1100

    def test_insufficient_balance_rollback(self, db):
        """余额不足：整个事务回滚，两个账户不变"""
        with pytest.raises(InsufficientBalance):
            with db.transaction():
                transfer(db, from_id=1, to_id=2, amount=9999)
        assert db.query("SELECT balance FROM accounts WHERE id = 1") == 1000
        assert db.query("SELECT balance FROM accounts WHERE id = 2") == 1000

    def test_concurrent_transfer_no_deadlock(self, db):
        """并发转账不会死锁"""
        import concurrent.futures
        with concurrent.futures.ThreadPoolExecutor(max_workers=5) as executor:
            futures = [
                executor.submit(transfer, db, 1, 2, 10),
                executor.submit(transfer, db, 2, 1, 20),
                executor.submit(transfer, db, 1, 3, 30),
            ]
            # 所有操作应在超时内完成（无死锁）
            for f in concurrent.futures.as_completed(futures, timeout=5):
                f.result()

    def test_idempotent_transfer(self, db):
        """幂等性：同一笔转账执行两次不会重复扣款"""
        tx_id = "test-tx-001"
        with db.transaction():
            transfer(db, from_id=1, to_id=2, amount=100, tx_id=tx_id)
        # 重复执行
        with db.transaction():
            transfer(db, from_id=1, to_id=2, amount=100, tx_id=tx_id)
        # 余额只变化一次
        assert db.query("SELECT balance FROM accounts WHERE id = 1") == 900
```

### 1.9 速查清单

在每次设计或审查事务相关代码时，对照以下清单：

```text
□ 事务范围是否尽可能小？
□ 事务中是否有外部调用（HTTP/RPC/文件 IO）？
□ 异常处理是否覆盖所有路径（包括 finally）？
□ 连接是否能正确归还到连接池？
□ 隔离级别是否与业务需求匹配？
□ 是否存在隐式自动提交被忽略？
□ DDL 语句是否在事务外执行？
□ 批量操作是否做了分批提交？
□ 并发场景是否有死锁风险？（检查访问顺序）
□ 关键操作是否有幂等设计？
□ 长事务是否有超时保护？
□ 监控告警是否覆盖了长事务和死锁？
```

### 1.10 小结

事务最佳实践的核心可以归结为一句话：**让事务尽可能短、尽可能小、尽可能幂等、尽可能可观测**。

- **短**：不在事务中做网络调用、复杂计算或等待人工操作
- **小**：只包含必要的数据操作，不扩大锁范围
- **幂等**：任何事务都能安全重试，不产生副作用
- **可观测**：有监控、有日志、有告警，出问题能快速定位

掌握这些原则后，面对具体场景时就能做出正确的技术选型——是用乐观锁还是悲观锁、是本地事务还是分布式事务、是 RC 还是 RR。事务不是银弹，但理解它的边界和代价，是每个后端工程师的必修课。
