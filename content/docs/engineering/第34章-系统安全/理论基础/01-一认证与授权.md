## 一、认证与授权

认证与授权是系统安全的两块基石。认证解决"你是谁"的问题，授权解决"你能做什么"的问题。两者紧密配合，共同构成了访问控制的完整体系。一个安全的系统，必须在认证和授权两个层面都做到位——认证薄弱意味着身份容易伪造，授权薄弱意味着合法用户可以越权操作。

在实际安全事件中，认证与授权缺陷长期占据 OWASP Top 10 的核心位置。2021 年 OWASP Top 10 中，"Broken Authentication"和"Broken Access Control"合计影响了超过 40% 的 Web 应用漏洞。理解这两个领域的原理和最佳实践，是每一位系统工程师的必修课。

---

### 1.1 认证（Authentication）— 你是谁？

认证是验证用户身份的过程。在现代系统中，认证不仅仅是"输密码登录"这么简单，它涵盖了从身份证明到会话管理的完整生命周期。

#### 1.1.1 认证因素分类

所有认证手段本质上都在验证三类或更多类因素。安全性随因素数量和类型的增加而提升。

| 因素类型 | 具体示例 | 安全性 | 适用场景 |
|----------|---------|--------|----------|
| **知识因素**（Something You Know） | 密码、PIN码、安全问题答案、助记词 | 低 | 通用，作为基础认证层 |
| **持有因素**（Something You Have） | 手机（短信/推送）、硬件令牌（YubiKey）、智能卡、USB Key | 中 | 高安全需求的二次验证 |
| **固有因素**（Something You Are） | 指纹、面部识别、虹膜、声纹、掌纹 | 高 | 生物特征唯一性绑定 |
| **行为因素**（Something You Do） | 打字节奏、鼠标移动轨迹、步态、触屏压力 | 中-低 | 连续认证、风控辅助 |
| **位置因素**（Somewhere You Are） | GPS定位、IP地址段、WiFi接入点 | 低 | 风控辅助，非独立认证 |
| **设备因素**（Something You Own） | 设备指纹、TPM芯片、安全飞书 | 中 | 设备信任绑定 |

> **关键原则**：单因素认证的安全性取决于最弱的那个因素。密码再复杂，也比不上"密码+手机验证码"的双因素组合。

#### 1.1.2 多因素认证（MFA）

多因素认证要求用户同时提供两种或以上不同类型的认证因素。注意，"多因素"强调的是**不同类型**的因素——输入两个密码不算多因素，密码+指纹才算。

**典型 MFA 登录流程：**

第一步：输入用户名 + 密码（知识因素）
        ↓ 验证通过
第二步：输入手机短信验证码 / 硬件令牌OTP（持有因素）
        ↓ 验证通过
第三步（可选）：指纹或面容识别（固有因素）
        ↓ 验证通过
        → 签发会话令牌，授权访问

**MFA 方案对比：**

| 方案 | 安全性 | 用户体验 | 成本 | 防钓鱼能力 |
|------|--------|---------|------|-----------|
| 短信验证码（SMS OTP） | 中 | 好 | 低 | 弱（可被SIM Swap攻击） |
| TOTP（Google Authenticator等） | 中-高 | 较好 | 低 | 中（无绑定，可被社会工程） |
| 推送通知（Push Notification） | 中-高 | 好 | 中 | 中（需验证来源） |
| 硬件安全密钥（YubiKey/FIDO2） | 高 | 较好 | 高 | 强（绑定域名，抗钓鱼） |
| 生物识别（指纹/面容） | 高 | 好 | 中 | 强（不可转移） |
| Passkeys（WebAuthn） | 高 | 好 | 低 | 强（抗钓鱼+便捷） |

**2024年真实案例——Microsoft MFA迁移成效**：微软在其内部 35 万+ 员工中全面推行 MFA 后，成功实现了"零密码泄露"——攻击者即使获取了员工密码，也无法通过第二因素验证。2019 年的 Twitter 员工社工事件中，攻击者通过电话社工获取了一次性密码，正是因为 Twitter 当时仅依赖 SMS OTP 而未采用 FIDO2 硬件密钥，这直接推动了 Twitter（现 X）向硬件安全密钥的迁移。

#### 1.1.3 认证协议与机制

**基于密码的认证：**

最传统也最常见的认证方式。核心安全要求是**密码存储不能是明文**。

密码存储安全等级（从低到高）：
  MD5 哈希           → 已不安全，可被彩虹表攻击（<1秒碰撞）
  SHA-256 哈希       → 仍不安全，无盐值保护
  加盐SHA-256        → 有改善但无拉伸，GPU暴力破解仍可行（~10亿次/秒）
  bcrypt             → 自带盐值+拉伸，推荐选择 cost=12+（~4次/秒）
  scrypt             → 内存硬度高，抗ASIC/GPU攻击
  Argon2id（推荐）   → 2015年密码哈希竞赛冠军，抗GPU+侧信道（~1次/秒）

**为什么需要拉伸（Key Stretching）**？密码哈希的核心安全目标不是"不可逆"（所有哈希都不可逆），而是"计算成本足够高"。攻击者拿到数据库后可以离线尝试，拉伸的作用就是让每次尝试都消耗大量计算资源，使暴力破解在经济上不可行。

**Argon2id 推荐参数：**

```python
# Python argon2-cffi 示例
from argon2 import PasswordHasher

ph = PasswordHasher(
    time_cost=3,        # 迭代次数
    memory_cost=65536,  # 64MB 内存消耗
    parallelism=4,      # 4个并行线程
    hash_len=32,        # 32字节输出
    salt_len=16,        # 16字节盐值
)

# 哈希密码（存储到数据库）
hashed = ph.hash("user_password")

# 验证密码
try:
    ph.verify(hashed, "user_password")
    print("认证成功")
except argon2.exceptions.VerifyMismatchError:
    print("认证失败")
```

> **参数调优原则**：Argon2id 的参数应使得单次验证在你的服务器上耗时约 250-500ms（不影响用户体验），同时内存消耗足够大以抵御 GPU 攻击。可使用 `argon2-cffi` 的 `--force-zero-memory` 参数测试无内存限制下的安全性。

**基于令牌（Token）的认证：**

令牌认证是 Web 和 API 场景下的主流方案。用户登录成功后，服务器签发一个令牌，客户端在后续请求中携带该令牌证明身份。

┌──────────┐         ┌──────────┐
│  客户端   │ ──登录──→│  服务器   │
│          │←─令牌────│          │
│          │          │          │
│          │─请求+令牌→│          │
│          │←─响应─────│          │
└──────────┘         └──────────┘

**JWT（JSON Web Token）结构：**

Header.Payload.Signature

Header:  {"alg":"RS256","typ":"JWT"}
Payload: {"sub":"user123","iat":1719360000,"exp":1719363600,"role":"editor","scope":"read write"}
Signature: RS256(base64(header) + "." + base64(payload), private_key)

**JWT 攻击向量与防御（常被忽视）：**

| 攻击方式 | 原理 | 防御措施 |
|---------|------|---------|
| **alg:none 攻击** | 将 Header 的 alg 改为 "none"，跳过签名验证 | 服务端强制校验算法白名单，拒绝 none |
| **密钥混淆攻击** | 用 HS256 时传入公钥作为 HMAC 密钥，伪造合法签名 | 严格区分对称/非对称算法使用场景 |
| **过期绕过** | 删掉 exp 字段重新编码 | 服务端必须强制校验 exp，不允许缺失 |
| **算法降级攻击** | 将 RS256 降级为 HS256，用公钥伪造签名 | 签发和验证时固定算法，拒绝降级 |

**JWT 安全实践：**
- 使用 RS256 或 ES256（非对称签名），避免 HS256（对称签名，密钥可能泄露）
- 设置合理的过期时间（Access Token 15-30分钟，Refresh Token 7-30天）
- 不要在 Payload 中存放敏感信息（JWT 只签名不加密，任何人可读取）
- 使用 `jti`（JWT ID）支持令牌吊销
- 在生产环境中使用 `pyjwt` 或 `python-jose`，避免自行实现签名逻辑

**令牌吊销策略：**

JWT 天然是无状态的，一旦签发就无法撤回。这在需要"登出"或"吊销令牌"时成为问题。常用解决策略：

方案1：黑名单（Blacklist）
  将被吊销的jti存入Redis，设置与Token过期时间一致的TTL
  每次验证时检查 jti 是否在黑名单中
  优点：实现简单，吊销即时生效
  缺点：引入状态，需要查询Redis（增加延迟约1-2ms）

方案2：令牌轮换（Rotation）
  短生命周期 Access Token（5-15分钟）+ 长生命周期 Refresh Token
  Refresh Token 使用后立即作废，签发新的一对
  被盗 Token 窗口期极短
  优点：无状态，不依赖外部存储
  缺点：吊销窗口为 Token 有效期

方案3：令牌绑定（Token Binding）
  将 Token 与客户端设备指纹或 TLS 连接绑定
  即使 Token 被盗，在其他设备上也无法使用
  优点：安全等级最高
  缺点：实现复杂，影响用户体验

推荐组合：方案1 + 方案2（黑名单用于紧急吊销，轮换用于日常安全）

**会话（Session）认证：**

传统 Web 应用常用 Session Cookie 方案。

会话管理安全清单：
  ✓ Session ID 使用密码学安全的随机数（≥128位熵）
  ✓ 设置 Cookie 属性：HttpOnly + Secure + SameSite=Strict
  ✓ 会话超时设置：闲置超时（15-30分钟）+ 绝对超时（8-24小时）
  ✓ 登录成功后轮换 Session ID（防固定会话攻击）
  ✓ 服务端存储 Session 白名单，支持强制下线
  ✓ 绑定客户端特征（User-Agent、IP段）防会话劫持

**Session Fixation（会话固定）攻击详解：**

攻击流程：
1. 攻击者访问目标网站，获取一个Session ID: abc123
2. 攻击者诱导受害者访问：https://target.com/login?session=abc123
3. 受害者登录成功，服务器将 abc123 与受害者身份绑定
4. 攻击者使用 abc123 访问网站，直接获得受害者身份

防御措施：
  ✅ 登录成功后立即轮换 Session ID（新Session替换旧Session）
  ✅ 拒绝URL中传入的Session ID参数
  ✅ Session ID仅通过Set-Cookie下发

**基于 API Key 的认证：**

在微服务和开放平台场景中，API Key 是最基础的认证方式。

API Key 使用规范：
  ✓ 生成：使用密码学安全的随机数生成器（≥256位熵）
  ✓ 传输：仅通过 HTTPS 传输，禁止明文传输
  ✓ 存储：服务端用 bcrypt/Argon2id 哈希存储，不明文保存
  ✓ 展示：仅在创建时完整展示一次，之后只显示前缀（sk_live_****...）
  ✓ 权限：每个 API Key 绑定最小权限范围
  ✓ 轮换：支持多 Key 同时存在，旧 Key 可独立吊销
  ✓ 限制：绑定 IP 白名单或调用频率限制

# 生成 API Key 示例
import secrets
api_key = f"sk_live_{secrets.token_urlsafe(32)}"
# → sk_live_****

#### 1.1.4 单点登录（SSO）

单点登录让用户只需认证一次，即可访问多个关联系统。核心价值是提升用户体验和降低密码疲劳。

| 协议 | 特点 | 适用场景 |
|------|------|---------|
| **SAML 2.0** | XML-based，企业级标准，支持身份联邦 | 企业内部系统、政府/教育机构 |
| **OpenID Connect** | 基于OAuth 2.0，JSON/Web，移动端友好 | 现代Web/移动应用 |
| **Kerberos** | 基于票据，域内认证，微软AD深度集成 | Windows域环境、大型企业内网 |
| **CAS** | 中央认证服务，简单轻量，协议简洁 | 中小企业、学术机构 |

**SAML 2.0 认证流程（典型）：**

用户 (User)         服务提供商 (SP)         身份提供商 (IdP)
    │                    │                       │
    │──1.访问SP资源────→│                       │
    │                    │                       │
    │←─2.重定向到IdP────│                       │
    │──3.发送AuthnRequest──→│                    │
    │                    │     4.用户登录认证     │
    │                    │                       │
    │←─5.SAML Response（含用户属性）────────────│
    │──6.携带Assertion访问SP──→│                 │
    │                    │  7.验证签名+授权      │
    │←─8.返回资源────────│                       │

**SAML 关键概念：**
- **AuthnRequest**：SP 发给 IdP 的认证请求
- **SAML Response**：IdP 返回的认证结果，包含用户属性（NameID、角色等）
- **Assertion**：XML 格式的断言，包含认证声明和属性声明
- **签名验证**：SP 用 IdP 的公钥验证 Response 签名，防止伪造

**Kerberos 认证流程：**

客户端             KDC（密钥分发中心）         目标服务
    │                    │                     │
    │──1.AS-REQ──────→│                     │
    │←─2.AS-REP──────│（返回TGT票据）          │
    │                    │                     │
    │──3.TGS-REQ─────→│（用TGT请求服务票据）   │
    │←─4.TGS-REP─────│（返回服务票据）         │
    │                    │                     │
    │──5.AP-REQ──────────────────────────→│
    │                    │   6.验证票据         │
    │←─7.服务响应─────────────────────────│

#### 1.1.5 无密码认证（Passwordless）

无密码认证是近年的重要趋势。用户不再需要记忆密码，而是通过设备、生物特征或密钥进行身份验证。

Passkeys（WebAuthn/FIDO2）工作原理：

1. 注册阶段：
   用户设备生成密钥对 → 私钥留在设备安全芯片（TPM/Secure Enclave）
                       → 公钥注册到服务器

2. 认证阶段：
   服务器发送随机挑战 → 设备用私钥签名 → 服务器用公钥验证
   → 认证成功，签发令牌

核心优势：
  - 无密码可泄露 → 抗钓鱼、抗撞库
  - 设备私钥不可导出 → 无法远程窃取
  - 绑定域名 → 假冒网站无法获取认证
  - 多设备同步 → iCloud/Google账户跨设备同步私钥

**Passkeys 与传统方案的对比：**

| 特性 | 密码 | SMS OTP | TOTP | Passkeys |
|------|------|---------|------|----------|
| 抗钓鱼 | ❌ | ❌ | ⚠️ | ✅ |
| 抗暴力破解 | ⚠️（取决于强度） | ✅ | ✅ | ✅ |
| 抗撞库 | ❌ | ✅ | ✅ | ✅ |
| 用户记忆负担 | 高 | 低 | 低 | 无 |
| 部署成本 | 低 | 中 | 低 | 低-中 |
| 跨平台支持 | ✅ | ✅ | ✅ | ✅（2024年） |

#### 1.1.6 认证安全威胁与防御

| 威胁类型 | 攻击方式 | 防御措施 |
|----------|---------|---------|
| **暴力破解** | 穷举密码组合 | 账户锁定策略 + 验证码 + 速率限制 |
| **撞库攻击** | 使用泄露的用户名/密码批量尝试 | MFA + 异常登录检测 + 密码泄露库比对（HaveIBeenPwned API） |
| **凭证填充** | 自动化工具批量使用泄露凭证 | 速率限制 + 行为分析 + 设备指纹 |
| **会话劫持** | 窃取Session Cookie | HttpOnly + Secure + SameSite Cookie |
| **中间人攻击** | 拦截认证凭据 | HTTPS + 证书固定（Certificate Pinning） |
| **重放攻击** | 重放截获的认证消息 | 时间戳 + Nonce + HMAC签名 |
| **社会工程** | 钓鱼网站诱导输入密码 | FIDO2/Passkeys + 浏览器内置钓鱼检测 |
| **凭证泄露** | 内部人员或数据库泄露 | 哈希存储 + 最小权限 + 数据库加密 |

**速率限制实现方案：**

```python
import time
from collections import defaultdict

class RateLimiter:
    """基于滑动窗口的速率限制器"""

    def __init__(self):
        self.attempts = defaultdict(list)  # key → [timestamp, ...]

    def is_allowed(self, key: str, max_attempts: int = 5,
                   window_seconds: int = 300) -> bool:
        """
        检查是否允许请求
        key: 限流标识（如 IP地址、用户ID）
        max_attempts: 窗口内最大尝试次数
        window_seconds: 时间窗口（秒），默认5分钟
        """
        now = time.time()
        # 清理过期记录
        self.attempts[key] = [
            t for t in self.attempts[key]
            if now - t < window_seconds
        ]
        if len(self.attempts[key]) >= max_attempts:
            return False
        self.attempts[key].append(now)
        return True

# 使用示例
limiter = RateLimiter()

def login(username, password):
    key = f"login:{username}"
    if not limiter.is_allowed(key, max_attempts=5, window_seconds=300):
        return {"error": "登录尝试过多，请5分钟后重试"}, 429

    # 验证密码...
    return {"message": "登录成功"}, 200
```

> **生产级方案**：上述代码是教学示例。生产环境建议使用 Redis + Lua 脚本实现分布式速率限制，或直接采用 Nginx 的 `limit_req_zone` 模块、API 网关（如 Kong、APISIX）的内置限流功能。

**HaveIBeenPwned 密码泄露库比对：**

```python
import hashlib
import requests

def check_password_breach(password: str) -> bool:
    """使用 HaveIBeenPwned API 检查密码是否已泄露"""
    sha1_hash = hashlib.sha1(password.encode()).hexdigest().upper()
    prefix = sha1_hash[:5]
    suffix = sha1_hash[5:]

    response = requests.get(f"https://api.pwnedpasswords.com/range/{prefix}")
    for line in response.text.splitlines():
        hash_suffix, count = line.split(":")
        if hash_suffix == suffix:
            return True  # 密码已被泄露
    return False

# 使用示例
if check_password_breach("password123"):
    print("⚠️ 此密码已出现在数据泄露中，请更换！")
```

### 1.2 授权（Authorization）— 你能做什么？

授权在认证之后进行，决定已认证的用户对哪些资源可以执行哪些操作。授权设计的核心目标是**最小权限原则**——每个用户只能访问完成其工作所必需的最少资源。

#### 1.2.1 授权模型全景

授权模型演进：
  DAC（自主访问控制）    ← 早期Unix文件权限
       ↓ 被更强管控取代
  MAC（强制访问控制）    ← 军事/政府级别，标签+策略
       ↓ 灵活性不足
  RBAC（基于角色）       ← 企业应用主流，角色=权限集合
       ↓ 动态场景不足
  ABAC（基于属性）       ← 细粒度动态策略，云原生
       ↓ 复杂度高
  ReBAC（基于关系）      ← 社交/协作场景，Zanzibar/SpiceDB

#### 1.2.2 DAC（Discretionary Access Control，自主访问控制）

资源的所有者自主决定谁可以访问。Linux 文件系统权限是 DAC 的典型代表。

```bash
# Linux 文件权限示例
-rwxr-xr-- 1 alice devs 4096 Jun 26 10:00 report.pdf
│││ │││ │││ │      │
│││ │││ │││ │      └─ 所属组
│││ │││ │││ └─ 所有者
│││ │││ │││
│││ │││ ││└─ 其他人: r-- (只读)
│││ │││ │└─ 同组用户: r-x (读+执行)
│││ │││ └─ 所有者: rwx (读+写+执行)

# 修改权限
chmod 750 report.pdf          # 所有者rwx, 同组r-x, 其他人无权限
chown alice:admins report.pdf # 改变所有者和组

# 高级场景：ACL（访问控制列表）
setfacl -m u:bob:rw report.pdf    # 给bob单独的读写权限
getfacl report.pdf                # 查看完整ACL
```

**DAC 的局限**：所有者可以自行转移权限，无法实现全局强制策略，容易因配置疏忽导致越权。典型风险场景：开发者将数据库文件权限设为 777，导致任意用户可读写。

#### 1.2.3 MAC（Mandatory Access Control，强制访问控制）

由系统管理员或安全策略统一管控，资源所有者无权自行授权。典型实现包括 SELinux 和 AppArmor。

SELinux 策略示例：
  类型：servlet_t → 只能访问 httpd_sys_content_t 类型的文件
  不管文件的所有者是谁，即使 root 进程也不能越权访问未标记的资源

实际影响：
  - Nginx 进程只能读取 /usr/share/nginx/ 下标记为 httpd_sys_content_t 的文件
  - 即使攻击者拿到 Nginx 进程权限，也无法读取 /etc/shadow

# SELinux 常用命令
sestatus                    # 查看SELinux状态
getenforce                  # 查看当前模式（Enforcing/Permissive/Disabled）
ls -Z /etc/shadow           # 查看文件的SELinux标签
sudo ausearch -m AVC        # 查看访问拒绝日志

**适用场景**：军事/政府机密系统、金融核心系统、高安全等级的合规要求。

#### 1.2.4 RBAC（Role-Based Access Control，基于角色的访问控制）

RBAC 是目前企业应用中最广泛使用的授权模型。核心思想是**将权限分配给角色，再将角色分配给用户**，实现用户与权限的解耦。

**RBAC 核心组件：**

用户 ──分配──→ 角色 ──包含──→ 权限 ──应用于──→ 资源+操作

示例：
  Alice ──→ Admin ──→ [用户管理CRUD, 配置读写, 日志查看]
  Bob   ──→ Editor──→ [文章CRUD, 评论审核]
  Carol ──→ Viewer──→ [文章只读]

**RBAC 层级模型（NIST标准）：**

| 模型层级 | 名称 | 说明 |
|---------|------|------|
| RBAC0 | 基础模型 | 用户-角色-权限的基本映射 |
| RBAC1 | 层级RBAC | 支持角色继承（Admin继承Editor权限） |
| RBAC2 | 约束RBAC | 支持职责分离（互斥角色不能同时分配） |
| RBAC3 | 统一RBAC | RBAC1+RBAC2的组合 |

**RBAC 数据库 Schema 设计（生产级）：**

```sql
-- 用户表
CREATE TABLE users (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    username    VARCHAR(64) UNIQUE NOT NULL,
    email       VARCHAR(255) UNIQUE NOT NULL,
    password    VARCHAR(255) NOT NULL,  -- Argon2id哈希
    status      ENUM('active','disabled','locked') DEFAULT 'active',
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- 角色表
CREATE TABLE roles (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    name        VARCHAR(64) UNIQUE NOT NULL,
    description TEXT,
    is_system   BOOLEAN DEFAULT FALSE,  -- 系统内置角色不可删除
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 权限表
CREATE TABLE permissions (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    code        VARCHAR(128) UNIQUE NOT NULL,  -- 如 "article:create"
    resource    VARCHAR(64) NOT NULL,           -- 如 "article"
    action      VARCHAR(32) NOT NULL,           -- 如 "create"
    description TEXT,
    UNIQUE KEY uk_resource_action (resource, action)
);

-- 用户-角色关联（多对多）
CREATE TABLE user_roles (
    user_id     BIGINT NOT NULL,
    role_id     BIGINT NOT NULL,
    granted_by  BIGINT,           -- 谁授予的
    granted_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    expires_at  TIMESTAMP NULL,   -- 支持临时角色授权
    PRIMARY KEY (user_id, role_id),
    FOREIGN KEY (user_id) REFERENCES users(id),
    FOREIGN KEY (role_id) REFERENCES roles(id)
);

-- 角色-权限关联（多对多）
CREATE TABLE role_permissions (
    role_id       BIGINT NOT NULL,
    permission_id BIGINT NOT NULL,
    PRIMARY KEY (role_id, permission_id),
    FOREIGN KEY (role_id) REFERENCES roles(id),
    FOREIGN KEY (permission_id) REFERENCES permissions(id)
);

-- 角色继承关系（RBAC1层级）
CREATE TABLE role_hierarchy (
    parent_role_id BIGINT NOT NULL,
    child_role_id  BIGINT NOT NULL,
    PRIMARY KEY (parent_role_id, child_role_id),
    FOREIGN KEY (parent_role_id) REFERENCES roles(id),
    FOREIGN KEY (child_role_id) REFERENCES roles(id)
);

-- 互斥角色约束（RBAC2职责分离）
CREATE TABLE mutual_exclusion (
    id            BIGINT PRIMARY KEY AUTO_INCREMENT,
    role_a_id     BIGINT NOT NULL,
    role_b_id     BIGINT NOT NULL,
    constraint_type ENUM('static','dynamic') DEFAULT 'static',
    -- static: 不能同时拥有两个角色
    -- dynamic: 不能同时激活两个角色
    FOREIGN KEY (role_a_id) REFERENCES roles(id),
    FOREIGN KEY (role_b_id) REFERENCES roles(id)
);
```

**Python 实现示例：**

```python
from dataclasses import dataclass, field
from typing import Dict, Set, List

@dataclass
class RBACSystem:
    """RBAC 授权系统"""
    # 角色 → 权限集合
    roles: Dict[str, Set[str]] = field(default_factory=dict)
    # 角色继承关系：子角色 → 父角色
    role_parents: Dict[str, str] = field(default_factory=dict)
    # 用户 → 角色集合（一个用户可有多个角色）
    user_roles: Dict[str, Set[str]] = field(default_factory=dict)
    # 互斥角色对（职责分离约束）
    mutual_exclusion: List[Set[str]] = field(default_factory=list)

    def add_role(self, role: str, permissions: Set[str], parent: str = None):
        """添加角色及其权限"""
        self.roles[role] = permissions
        if parent:
            self.role_parents[role] = parent

    def assign_role(self, user: str, role: str) -> bool:
        """分配角色给用户（含互斥检查）"""
        # 检查互斥约束
        current_roles = self.user_roles.get(user, set())
        for me_pair in self.mutual_exclusion:
            if role in me_pair and me_pair &amp; current_roles:
                print(f"互斥冲突: {role} 与 {current_roles &amp; me_pair} 不能共存")
                return False
        self.user_roles.setdefault(user, set()).add(role)
        return True

    def get_effective_permissions(self, user: str) -> Set[str]:
        """获取用户的有效权限（含角色继承）"""
        permissions = set()
        for role in self.user_roles.get(user, set()):
            # 收集当前角色的权限
            permissions |= self.roles.get(role, set())
            # 向上遍历父角色
            current = role
            while current in self.role_parents:
                parent = self.role_parents[current]
                permissions |= self.roles.get(parent, set())
                current = parent
        return permissions

    def check_permission(self, user: str, permission: str) -> bool:
        """检查用户是否有指定权限"""
        return permission in self.get_effective_permissions(user)


# ---- 使用示例 ----
rbac = RBACSystem()

# 定义角色层级
rbac.add_role("viewer", {"read"})
rbac.add_role("editor", {"read", "write", "edit"}, parent="viewer")
rbac.add_role("admin", {"read", "write", "edit", "delete", "manage_users", "config"}, parent="editor")

# 职责分离：财务审批和代码部署不能是同一个人
rbac.mutual_exclusion = [{"finance_approve"}, {"code_deploy"}]

# 分配角色
rbac.assign_role("alice", "admin")
rbac.assign_role("bob", "editor")
rbac.assign_role("carol", "viewer")

# 权限检查
print(rbac.check_permission("alice", "delete"))       # True (admin继承了delete)
print(rbac.check_permission("bob", "delete"))         # False (editor没有delete)
print(rbac.check_permission("carol", "read"))         # True (viewer有read)
print(f"Bob的全部权限: {rbac.get_effective_permissions('bob')}")
# → {'read', 'write', 'edit'}（含继承的viewer权限）
```

**RBAC 设计最佳实践：**

1. **角色命名规范**：使用业务语义而非技术语义（`order_manager` 优于 `role_3`）
2. **最少角色原则**：角色数量过多说明粒度拆分过度，管理成本上升
3. **默认无权限**：新角色默认不包含任何权限，必须显式授予
4. **定期审计**：每季度审查角色定义和用户-角色映射，清理僵尸角色
5. **职责分离**：关键操作需要多人协作（如大额转账需两人审批）
6. **层级深度限制**：角色继承层级建议不超过4层，过深则难以理解和维护
7. **临时角色支持**：为角色设置有效期，避免"永远在线"的过度授权

#### 1.2.5 ABAC（Attribute-Based Access Control，基于属性的访问控制）

ABAC 通过评估**属性**（而非固定角色）来做授权决策。属性可以来自用户、资源、环境或操作本身。ABAC 适合动态、细粒度的授权场景。

**ABAC 核心架构（XACML标准）：**

请求流向：
  PEP (策略执行点)        PDP (策略决策点)        PIP (策略信息点)
  ┌──────────────┐       ┌──────────────┐       ┌──────────────┐
  │ 拦截用户请求  │──请求──→│ 评估策略规则  │──查询──→│ 提供属性数据  │
  │ 执行PDP决策   │←─决策──│ 返回允许/拒绝 │←─属性──│ 用户/资源/环境│
  └──────────────┘       └──────────────┘       └──────────────┘
         │
         │ 策略存储
         ↓
  ┌──────────────┐
  │  PAP (策略管理点) │
  │  策略的增删改查  │
  └──────────────┘

**ABAC 策略评估逻辑：**

```python
from dataclasses import dataclass
from typing import Callable, List
from datetime import datetime

@dataclass
class Subject:
    """主体属性（用户）"""
    role: str
    department: str
    clearance_level: int
    ip_address: str

@dataclass
class Resource:
    """资源属性"""
    type: str          # finance, hr, code
    sensitivity: int   # 1-5级敏感度
    owner_dept: str
    classification: str  # public, internal, confidential, secret

@dataclass
class Environment:
    """环境属性"""
    time: datetime
    location: str
    device_type: str   # corporate, personal, mobile
    network: str       # internal, vpn, public

@dataclass
class Action:
    """操作属性"""
    verb: str          # read, write, delete, approve
    risk_level: int    # 1-5

class ABACEngine:
    def __init__(self):
        self.policies: List[Callable] = []

    def add_policy(self, policy_fn: Callable):
        self.policies.append(policy_fn)

    def evaluate(self, subject: Subject, resource: Resource,
                 action: Action, environment: Environment) -> dict:
        results = []
        for policy in self.policies:
            result = policy(subject, resource, action, environment)
            results.append(result)

        # 默认拒绝策略：任何一个策略返回拒绝即拒绝
        if any(r["decision"] == "deny" for r in results):
            return {"decision": "deny", "reasons": [r for r in results if r["decision"] == "deny"]}
        # 全部允许才允许
        if all(r["decision"] == "permit" for r in results):
            return {"decision": "permit", "reasons": results}
        return {"decision": "deny", "reasons": [{"reason": "no matching policy"}]}


# ---- 策略定义示例 ----
def policy_working_hours(s, r, a, e):
    """策略1：非工作时间禁止写操作"""
    hour = e.time.hour
    if a.verb in ("write", "delete", "approve") and not (9 <= hour < 18):
        return {"decision": "deny", "reason": "非工作时间禁止写操作"}
    return {"decision": "permit", "reason": "工作时间检查通过"}

def policy_clearance(s, r, a, e):
    """策略2：用户安全等级必须≥资源敏感度"""
    if s.clearance_level < r.sensitivity:
        return {"decision": "deny", "reason": f"安全等级{s.clearance_level}<{r.sensitivity}"}
    return {"decision": "permit", "reason": "安全等级检查通过"}

def policy_network(s, r, a, e):
    """策略3：从公网访问机密资源需VPN"""
    if r.classification in ("confidential", "secret") and e.network == "public":
        return {"decision": "deny", "reason": "公网无法访问机密资源"}
    return {"decision": "permit", "reason": "网络检查通过"}

def policy_department(s, r, a, e):
    """策略4：只能操作本部门资源（管理员例外）"""
    if s.role != "admin" and r.owner_dept != s.department:
        return {"decision": "deny", "reason": "无权操作其他部门资源"}
    return {"decision": "permit", "reason": "部门检查通过"}


# ---- 引擎初始化 ----
engine = ABACEngine()
engine.add_policy(policy_working_hours)
engine.add_policy(policy_clearance)
engine.add_policy(policy_network)
engine.add_policy(policy_department)

# ---- 测试场景 ----
import datetime

alice = Subject(role="admin", department="engineering", clearance_level=5, ip_address="10.0.1.5")
finance_report = Resource(type="finance", sensitivity=4, owner_dept="finance", classification="confidential")
write_action = Action(verb="write", risk_level=3)

# 场景1：工作时间内网访问 → 应该允许
env1 = Environment(time=datetime.datetime(2026, 6, 26, 14, 0), location="office",
                    device_type="corporate", network="internal")
result1 = engine.evaluate(alice, finance_report, write_action, env1)
print(f"场景1: {result1['decision']}")  # permit

# 场景2：公网访问机密资源 → 应该拒绝
env2 = Environment(time=datetime.datetime(2026, 6, 26, 14, 0), location="home",
                    device_type="personal", network="public")
result2 = engine.evaluate(alice, finance_report, write_action, env2)
print(f"场景2: {result2['decision']}")  # deny（公网+机密）
```

**ABAC 与 RBAC 对比：**

| 维度 | RBAC | ABAC |
|------|------|------|
| 决策依据 | 角色 | 多维属性 |
| 灵活性 | 中（角色固定） | 高（属性动态变化） |
| 实现复杂度 | 低 | 高 |
| 管理成本 | 低（角色少） | 高（策略多） |
| 适用场景 | 企业内部权限管理 | 云原生、多租户、合规场景 |
| 粒度 | 粗到中 | 细到极细 |
| 审计友好 | 好（角色清晰） | 中（策略链复杂） |
| 性能 | 高（查询快） | 中（需评估多条策略） |

> **实践建议**：大多数场景 RBAC 已够用。在 RBAC 基础上叠加少量 ABAC 策略（如时间、网络环境限制）是最佳实践——既保持 RBAC 的简单性，又获得 ABAC 的灵活性。

#### 1.2.6 ReBAC（Relationship-Based Access Control，基于关系的访问控制）

ReBAC 以实体之间的**关系**作为授权依据。Google 的 Zanzibar 论文（2019）是该模型的奠基之作。

核心思想：

  "Alice 可以编辑文档 X"
  不是直接分配权限，而是因为：
  Alice --是文档X的--→ Editor
  Editor --被授予--→ [read, write]

  关系图谱：
  alice --member_of--> team_engineering
  team_engineering --owns--> project_alpha
  project_alpha --contains--> document_x
  document_x --allows--> [read, write] via Editor role

  关系查询（Zanzibar风格）：
  Check(document_x, "edit", alice) → 遍历关系图谱 → true/false

**Zanzibar 风格的关系定义语言（Zookie）：**

# 定义关系类型
definition user {}
definition team {}
definition document {
    relation owner: user
    relation editor: user | team#member
    relation viewer: user | team#member
    permission edit = editor + owner
    permission view = viewer + edit
}

# 定义关系元组
user:alice#owner@document:doc_123
team:eng#member@user:alice
document:doc_123#editor@team:eng

# 检查权限
Check(doc_123, "edit", alice) → true

**适用场景**：Google Drive 式的协文档权限、社交平台的好友可见性、GitHub 的仓库访问控制。

开源实现：OpenFGA（Auth0/Fiorebase）、SpiceDB（AuthZed）。

#### 1.2.7 授权安全漏洞与防御

| 漏洞类型 | 描述 | 防御措施 |
|----------|------|---------|
| **越权访问（IDOR）** | 用户通过修改URL/参数访问他人资源 | 服务端强制校验资源所有权，使用UUID替代自增ID |
| **权限提升** | 普通用户获取管理员权限 | 最小权限原则 + 角色变更审计 + 输入验证 |
| **强制浏览** | 直接访问未授权的页面URL | 中间件拦截 + 权限检查前置 |
| **API越权** | 未校验当前用户是否有权操作目标资源 | 每个API端点实现资源级别的权限校验 |
| **批量赋权** | 管理员误将过多角色分配给用户 | RBAC约束（互斥角色）+ 定期权限审计 |
| **间接对象引用** | 通过可预测的资源标识访问未授权数据 | 使用不可预测的标识符（UUID/ULID） |

**IDOR 防御代码示例：**

```python
# 错误做法 ❌ — 使用自增ID，可被枚举
@app.get("/api/orders/<int:order_id>")
def get_order(order_id):
    return db.query(Order).get(order_id)  # 任何用户都能看到任何订单

# 正确做法 ✅ — UUID + 所有权校验
@app.get("/api/orders/<uuid:order_id>")
@require_auth
def get_order(order_id):
    order = db.query(Order).filter_by(id=order_id).first()
    if not order or order.user_id != current_user.id:
        abort(403)  # 无权访问
    return order
```

**越权测试检查清单：**

在开发和测试阶段，系统性地检查以下越权场景：

□ 修改请求中的用户ID，查看是否能访问他人数据
□ 修改资源ID（订单号、文章ID等），查看是否能越权读取
□ 修改请求参数中的角色字段，查看是否能提升权限
□ 检查API端点是否对每个请求都校验了资源所有权
□ 测试删除/修改操作是否校验了当前用户身份
□ 验证批量操作接口是否对每条记录都做了权限检查
□ 检查文件上传/下载接口是否限制了路径遍历

### 1.3 认证与授权的协作架构

在实际系统中，认证和授权通常组合使用。理解它们的协作关系对于正确实现安全机制至关重要。

#### 1.3.1 完整的请求安全链

用户请求
  │
  ↓
┌─────────────────────────┐
│ 1. TLS 终止（传输安全）   │  ← HTTPS，防止中间人窃听
└─────────────────────────┘
  │
  ↓
┌─────────────────────────┐
│ 2. 认证（Authentication）│  ← 验证身份：JWT/Session/Cookie
│    • 无有效凭证 → 401    │
└─────────────────────────┘
  │ 认证通过
  ↓
┌─────────────────────────┐
│ 3. 授权（Authorization） │  ← 检查权限：RBAC/ABAC
│    • 权限不足 → 403      │
└─────────────────────────┘
  │ 授权通过
  ↓
┌─────────────────────────┐
│ 4. 输入验证              │  ← 防注入、XSS、参数篡改
└─────────────────────────┘
  │

#### 1.3.2 mTLS（双向TLS）认证

在微服务间通信中，mTLS 是比 JWT 更安全的认证方式——客户端和服务端互相验证证书。

mTLS 握手流程：
  客户端                              服务端
    │──1. ClientHello──────────────→│
    │←─2. ServerHello + 服务端证书──│
    │──3. 客户端证书──────────────→│  ← 关键区别：客户端也出示证书
    │←─4. 双方验证对方证书──────────│
    │──5. 建立加密通道──────────────→│

证书来源：
  • 内部CA签发（如 Vault PKI）
  • Let's Encrypt（外部服务）
  • 云厂商托管（AWS Private CA, GCP CAS）

**mTLS 与 JWT 的选择：**

| 维度 | mTLS | JWT |
|------|------|-----|
| 认证方式 | 证书（X.509） | 令牌（自签发） |
| 适用场景 | 服务间通信 | 用户→服务、服务→API |
| 前置条件 | 需要PKI基础设施 | 只需密钥管理 |
| 管理成本 | 高（证书轮换） | 低（Token自过期） |
| 性能 | 略高（证书验证） | 低（签名验证快） |
| 安全等级 | 高（硬件级） | 中-高（取决于实现） |

#### 1.3.3 JWT + RBAC 的生产级实现

```python
import jwt
import time
from functools import wraps
from flask import Flask, request, jsonify, g

app = Flask(__name__)
SECRET_KEY = "your-rsa-private-key"  # 实际使用应从环境变量/密钥管理服务获取

# ---- 认证中间件 ----
def require_auth(f):
    @wraps(f)
    def decorated(*args, **kwargs):
        token = request.headers.get("Authorization", "").replace("Bearer ", "")
        if not token:
            return jsonify({"error": "Missing token"}), 401
        try:
            payload = jwt.decode(token, SECRET_KEY, algorithms=["RS256"])
            # 检查过期
            if payload.get("exp", 0) < time.time():
                return jsonify({"error": "Token expired"}), 401
            g.current_user = payload
        except jwt.InvalidTokenError:
            return jsonify({"error": "Invalid token"}), 401
        return f(*args, **kwargs)
    return decorated

# ---- 授权装饰器 ----
def require_permission(permission: str):
    def decorator(f):
        @wraps(f)
        @require_auth  # 先认证
        def decorated(*args, **kwargs):
            user_permissions = g.current_user.get("permissions", [])
            if permission not in user_permissions:
                return jsonify({"error": f"Missing permission: {permission}"}), 403
            return f(*args, **kwargs)
        return decorated
    return decorator

# ---- 路由示例 ----
@app.route("/api/articles", methods=["GET"])
@require_permission("article:read")
def list_articles():
    return jsonify({"articles": []})

@app.route("/api/articles", methods=["POST"])
@require_permission("article:create")
def create_article():
    return jsonify({"message": "Created"}), 201

@app.route("/api/users", methods=["DELETE"])
@require_permission("user:delete")
def delete_user():
    return jsonify({"message": "Deleted"})
```

#### 1.3.4 Token 设计最佳实践

Access Token（短期令牌，15-30分钟）：
  用途：API请求认证
  包含：用户ID、角色、权限列表、过期时间
  存储：内存或安全存储（非localStorage）

Refresh Token（长期令牌，7-30天）：
  用途：刷新Access Token
  包含：用户ID、会话ID
  存储：HttpOnly Cookie 或安全存储
  特性：支持吊销、一次性使用（刷新后轮换）

令牌轮换流程：
  Access Token过期
    → 客户端用Refresh Token请求新Access Token
    → 服务器验证Refresh Token有效性
    → 签发新的Access Token + 新Refresh Token（旧的作废）
    → 客户端更新两个令牌

**前端 Token 存储安全对比：**

| 存储位置 | 安全性 | 风险 |
|---------|--------|------|
| localStorage | 低 | XSS可读取 |
| sessionStorage | 低 | XSS可读取，关闭标签页即失 |
| HttpOnly Cookie | 高 | XSS无法读取，但需防CSRF |
| 内存变量（闭包） | 中 | 刷新页面丢失，需重新认证 |
| 浏览器安全存储（Web Crypto API） | 高 | 需要适配，API较新 |

**推荐方案**：Access Token 存内存，Refresh Token 存 HttpOnly Cookie，配合 CSRF Token 防护。

#### 1.3.5 Cookie 安全属性详解

Set-Cookie 的安全属性：

HttpOnly = true
  → JavaScript 无法读取 Cookie（防 XSS 窃取）
  → document.cookie 不包含该 Cookie

Secure = true
  → 仅通过 HTTPS 传输
  → HTTP 请求不会携带该 Cookie

SameSite = Strict / Lax / None
  → 控制跨站请求是否携带 Cookie
  → Strict: 跨站请求完全不带 Cookie（最严格）
  → Lax: GET 导航带 Cookie，POST/iframe 不带（推荐默认）
  → None: 所有跨站请求都带 Cookie（需配合 Secure=true）

Domain = .example.com
  → 控制 Cookie 的作用域
  → 不设置则默认为当前域（不含子域）

Path = /api
  → Cookie 仅在指定路径下发送

Max-Age = 3600
  → Cookie 的最大存活时间（秒）

# 完整的安全 Cookie 示例
Set-Cookie: session_id=abc123; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=3600

### 1.4 常见误区

| 误区 | 为什么是错的 | 正确做法 |
|------|-------------|---------|
| "加密就是安全" | 加密只保护传输/存储，不解决身份验证和权限控制 | 认证+授权+加密 三者缺一不可 |
| "前端隐藏按钮就够了" | 前端控制可被绕过，安全检查必须在服务端 | 所有API端点都做服务端权限校验 |
| "密码复杂就够了" | 再复杂的密码也可能被泄露，单因素认证天生脆弱 | 强制MFA + 密码长度≥12位 |
| "JWT不可伪造很安全" | JWT签名不加密，Payload任何人都能读取 | 不在JWT中放敏感信息，使用HTTPS |
| "RBAC够用了" | RBAC无法处理动态上下文（时间、位置、设备） | 敏感操作叠加ABAC策略 |
| "默认允许比较方便" | 默认允许意味着任何未明确禁止的操作都可执行 | 默认拒绝，显式授予 |
| "管理员不受限制" | 管理员越权是最大的安全风险之一 | 管理员也要最小权限 + 操作审计 |
| "密码加密存储就行" | MD5/SHA已被破解，彩虹表可在秒级碰撞 | 使用Argon2id/bcrypt + 盐值 |
| "验证码能防所有机器人" | 现代AI和打码平台可绕过大部分验证码 | 验证码仅作辅助，核心靠速率限制 |
| "内网不需要认证" | 零信任架构下内网也不可信 | 所有网络位置都执行相同安全策略 |

**真实案例——Equifax 数据泄露（2017）**：攻击者利用未修补的 Apache Struts 漏洞（CVE-2017-5638）进入内网后，发现数据库中存储了 1.47 亿人的明文密码和敏感信息。如果 Equifax 使用了 Argon2id 哈希存储密码、实施了网络分段、启用了 mTLS 服务间认证，攻击者即使进入内网也难以窃取完整数据。

### 1.5 进阶内容

#### 1.5.1 零信任架构中的认证与授权

传统安全模型基于"内网可信"假设，零信任架构则假设**任何网络位置都不可信**。

传统模型 vs 零信任：
传统（城堡模型）：
  外部 ──防火墙──→ 内部（信任区）
  内部用户默认信任

零信任模型：
  每次请求都需要：
  1. 验证身份（强认证，MFA必须）
  2. 验证设备健康状态（补丁、防病毒、加密）
  3. 验证上下文（时间、位置、行为异常）
  4. 最小权限访问（Just-In-Time + Just-Enough-Access）
  5. 持续监控（会话中持续评估风险）

**零信任核心原则：**
- **永不信任，始终验证**：不因网络位置而降低认证要求
- **最小权限**：只授予完成当前任务所需的最小权限集
- **假设被入侵**：默认攻击者已在内网，必须限制爆炸半径

**零信任技术栈示例：**

身份层：    Azure AD / Okta / Keycloak（统一身份）
设备层：    MDM/EDR（设备健康检查）
网络层：    BeyondCorp / Cloudflare Access（零信任网关）
应用层：    SPIFFE/SPIRE（服务身份认证）
数据层：    Vault（动态凭证，自动轮换）

> 零信任架构的详细实现方案请参阅 [05-零信任架构](../05-五零信任架构.md)。

#### 1.5.2 OAuth 2.0 与 OpenID Connect 的关系

OAuth 2.0 本质是**授权**框架（不是认证），OpenID Connect 在 OAuth 2.0 之上增加了**身份认证**层。

OAuth 2.0（授权）：
  A网站用户 → 授权A网站访问其B网盘文件
  核心概念：Access Token, Scope, Authorization Grant

OpenID Connect（认证 + 授权）：
  A网站 → 获取用户身份信息（姓名、邮箱、头像）
  核心概念：ID Token（JWT）, UserInfo Endpoint, Claims

> 详细的 OAuth 2.0 和 OIDC 实现请参阅下一节 [02-OAuth 2.0](../02-二OAuth20.md)。

#### 1.5.3 安全审计与合规

认证授权审计检查清单：

身份管理：
  □ 所有生产账号都有明确的所有者
  □ 离职员工的账号在24小时内禁用
  □ 共享账号被禁止或严格管控
  □ 特权账号使用独立凭据，不复用普通账号
  □ 服务账号的密码/密钥定期轮换（≤90天）

密码策略：
  □ 密码最少12位，强制包含大小写+数字+特殊字符
  □ 密码不可重复使用最近10次
  □ 密码存储使用Argon2id或bcrypt
  □ 密码泄露库检查（登录时比对HaveIBeenPwned）
  □ 禁止使用常见弱密码（top 10000黑名单）

MFA覆盖：
  □ 所有管理员账号强制MFA
  □ 特权操作要求二次验证
  □ 支持FIDO2/Passkeys作为MFA选项
  □ 禁止仅依赖SMS OTP（SIM Swap风险）

会话管理：
  □ 会话闲置超时≤30分钟
  □ 绝对超时≤24小时
  □ 支持强制下线
  □ Session ID在登录后轮换

日志审计：
  □ 所有认证事件（成功/失败）记录日志
  □ 特权操作记录完整审计轨迹
  □ 日志不可被普通用户删除
  □ 失败登录告警（5次失败后通知管理员）
  □ 日志保留时间≥180天
  □ 审计日志写入独立存储，与业务日志分离

### 1.6 本节小结

| 维度 | 核心要点 |
|------|---------|
| 认证因素 | 知识+持有+固有+行为，多因素显著提升安全性 |
| MFA方案 | Passkeys/FIDO2 > 硬件密钥 > TOTP > 短信验证码 |
| 密码存储 | Argon2id > bcrypt > scrypt，绝不用MD5/SHA |
| 授权模型 | RBAC适合企业通用，ABAC适合动态细粒度，ReBAC适合关系场景 |
| 核心原则 | 最小权限、默认拒绝、职责分离、持续审计 |
| 安全架构 | 认证→授权→输入验证→业务逻辑，层层递进 |
| 进阶方向 | 零信任架构、OAuth 2.0/OIDC、安全合规审计 |

认证与授权是系统安全的第一道防线。设计得当，它们能挡住绝大多数攻击；设计不当，后续所有安全措施都将事倍功半。下一节将深入 OAuth 2.0 协议的具体实现与最佳实践。
