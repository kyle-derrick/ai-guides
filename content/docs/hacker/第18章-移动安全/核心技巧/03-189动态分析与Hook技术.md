## 18.9 动态分析与Hook技术

动态分析是移动安全研究的核心方法之一——在应用运行时观察、修改、拦截其行为，从而发现静态分析无法揭示的安全问题。与静态分析"看代码"不同，动态分析"跑代码"，能够捕获运行时才确定的值（如加密密钥、Token、动态加载的类）、观察真实的数据流路径、验证漏洞的可利用性。

Hook 技术是动态分析的核心手段：在目标函数执行前后插入自定义代码，实现参数监控、返回值篡改、逻辑绕过等效果。本节从原理到实战，系统讲解主流 Hook 框架、高级技巧和对抗方法。

### 18.9.1 动态分析基础概念

#### 18.9.1.1 动态分析 vs 静态分析

| 维度 | 静态分析 | 动态分析 |
|------|----------|----------|
| 分析时机 | 不运行应用 | 运行应用时 |
| 覆盖范围 | 理论上全覆盖（所有代码路径） | 仅执行到的路径 |
| 优势 | 发现隐藏代码、硬编码密钥、逻辑缺陷 | 获取运行时值、观察真实数据流、验证漏洞 |
| 局限 | 无法获取运行时状态、混淆代码难分析 | 代码覆盖率依赖触发路径、可能被反调试检测 |
| 典型工具 | jadx、apktool、Ghidra | Frida、Xposed、LLDB |
| 适用场景 | 初步了解应用结构、查找可疑代码 | 验证漏洞、绕过保护、提取敏感数据 |

实战中两者互补：先用静态分析定位目标函数和攻击面，再用动态分析验证和利用。

#### 18.9.1.2 动态分析的核心能力

**数据流追踪**：应用运行时的数据（用户输入、网络响应、加密结果）在代码中的完整流动路径。静态分析只能猜测数据流向，动态分析可以在每个节点拦截并记录实际值。

**运行时修改**：改变函数参数、返回值、内存中的变量值，观察应用行为变化。例如将 `isPremium=false` 改为 `true`，验证付费功能是否真的依赖服务端校验。

**行为监控**：记录应用的系统调用、网络请求、文件读写、数据库操作等外部行为，构建完整的运行时画像。

**调用链分析**：追踪一个请求从前端 UI 到后端网络通信的完整调用栈，理解业务逻辑的实现细节。

#### 18.9.1.3 Hook 的基本原理

Hook 本质是在目标函数的入口或出口处"劫持"执行流：

```text
正常执行流：
  调用者 → 目标函数 → 返回结果

Hook 后执行流：
  调用者 → Hook入口（自定义逻辑） → 原始函数（可选调用） → Hook出口（自定义逻辑） → 返回结果
```

实现层面，Hook 的核心机制有三种：

**函数替换（Detours）**：将目标函数的前几条指令替换为跳转指令，跳转到 Hook 函数。Hook 函数执行完毕后可以选择是否调用原始函数。这是最通用的方式，Frida 的 `Interceptor` 就使用这种方式。

**方法表替换（VTable/PLT Hook）**：修改虚函数表或 PLT（Procedure Linkage Table）中的函数指针，使其指向 Hook 函数。适用于面向对象语言的虚方法调用，Xposed 的 Java 层 Hook 基于此原理。

**断点替换（Breakpoint Hook）**：在目标地址设置软件断点（如 x86 的 `INT 3`），断点触发时执行自定义逻辑。LLDB 调试器使用这种方式，性能开销较大但灵活性高。

### 18.9.2 Frida 框架体系

Frida 是目前最流行的动态插桩框架，由 Ole André Vadla Ravnås 开发，具有跨平台（Android/iOS/Windows/macOS/Linux）、多语言绑定（JavaScript/Python/C/Rust）、无需修改目标应用等优势。

#### 18.9.2.1 架构原理

Frida 采用 Client-Server 架构：

```text
┌─────────────────────────────────────────────┐
│  控制端 (Host)                                │
│  ┌───────────┐  ┌──────────────┐             │
│  │ frida-cli │  │ frida-python │             │
│  └─────┬─────┘  └──────┬───────┘             │
│        └───────┬────────┘                     │
│           frida-core (GumJS)                 │
│                │                              │
│          USB/TCP 通信                          │
├─────────────────────────────────────────────┤
│  目标端 (Device)                              │
│  ┌──────────────┐  ┌──────────────────────┐  │
│  │ frida-server │  │ frida-gadget (.so)   │  │
│  │ (独立进程)    │  │ (注入目标进程)        │  │
│  └──────┬───────┘  └──────────┬───────────┘  │
│         └──────────┬─────────┘               │
│              ptrace 注入                      │
│         ┌─────────────────┐                  │
│         │   目标应用进程    │                  │
│         │  ┌───────────┐  │                  │
│         │  │ Frida Agent│  │                  │
│         │  │ (JS Engine)│  │                  │
│         │  └───────────┘  │                  │
│         └─────────────────┘                  │
└─────────────────────────────────────────────┘
```

**frida-server**：运行在目标设备上的守护进程，以 root 权限运行，负责注入 Agent 到目标进程。Android 上需要匹配设备架构（arm/arm64/x86/x86_64）。

**frida-gadget**：一个共享库（.so/.dylib），可以打包进 APK/IPA 中，目标应用启动时自动加载。适用于无法 root 的设备，或需要在应用启动最早期就介入的场景。

**frida-agent**：注入到目标进程中的 JavaScript 运行时环境（基于 V8/QuickJS），执行用户编写的 Hook 脚本。通过 GumJS 引擎与目标进程的内存空间直接交互。

**frida-core**：控制端的核心库，负责与 frida-server 通信、管理脚本生命周期。Python 绑定和 CLI 工具都基于它。

#### 18.9.2.2 环境搭建

**安装 frida 工具链（控制端）**：

```bash
# 安装 frida-tools（包含 frida CLI、frida-ps、frida-trace 等）
pip install frida-tools

# 如果需要 Python 绑定
pip install frida

# 验证安装
frida --version
```

**Android 设备端部署**：

```bash
# 1. 从 GitHub Releases 下载对应架构的 frida-server
#    https://github.com/frida/frida/releases
#    文件名格式: frida-server-{version}-android-{arch}.xz
FRIDA_VERSION=$(frida --version)
wget "https://github.com/frida/frida/releases/download/${FRIDA_VERSION}/frida-server-${FRIDA_VERSION}-android-arm64.xz"

# 2. 解压并推送到设备
xz -d frida-server-*.xz
adb push frida-server-* /data/local/tmp/frida-server
adb shell "chmod 755 /data/local/tmp/frida-server"

# 3. 启动 frida-server（需要 root）
adb shell "su -c /data/local/tmp/frida-server -l 0.0.0.0:27042 &"

# 4. 验证连接
frida-ps -U          # 通过 USB 列出进程
frida-ps -R          # 通过远程（TCP）列出进程
frida-ps -U | grep target   # 过滤目标应用
```

**常见部署问题排查**：

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| `Failed to enumerate processes: unable to connect` | frida-server 未启动或架构不匹配 | 确认 `adb shell uname -m` 返回的架构与下载的 frida-server 一致 |
| `Unable to connect to remote frida-server: connection refused` | 端口未监听或防火墙 | 检查 `adb shell netstat -tlnp | grep 27042` |
| `Failed to attach: process not found` | 应用进程名错误 | 用 `frida-ps -Ua` 列出所有应用（显示包名） |
| `Failed to spawn: unable to find process` | SELinux 阻止 | `adb shell "su -c setenforce 0"` 临时关闭 |
| `frida-server 版本不匹配` | 控制端和设备端版本不同 | 确保 `pip show frida` 版本与 frida-server 版本一致 |

#### 18.9.2.3 两种运行模式：Spawn vs Attach

**Spawn 模式**（`-f` 标志）：Frida 先杀死目标进程再重新启动它，Hook 脚本在应用最早期执行。适用于需要 Hook 应用启动阶段代码（如 `Application.onCreate`）的场景。

```bash
# Spawn 模式：重启应用并注入
frida -U -f com.target.app -l hook-script.js --no-pause
```

`--no-pause` 表示注入后立即恢复应用执行（不暂停在入口点）。不加 `--no-pause` 时应用会暂停，你可以在应用启动前手动设置更多 Hook，然后调用 `%resume` 恢复执行。

**Attach 模式**（默认）：连接到已在运行的目标进程，不重启应用。适用于应用已经启动，你只需要在运行时监控或修改行为。

```bash
# Attach 模式：附加到已运行的进程
frida -U com.target.app -l hook-script.js
# 或者用 PID
frida -U -p 12345 -l hook-script.js
```

**选择建议**：

| 场景 | 推荐模式 | 原因 |
|------|----------|------|
| Hook `Application.onCreate`、启动初始化 | Spawn | 需要在最早期介入 |
| Hook 登录流程、业务逻辑 | Attach | 应用已启动，无需重启 |
| 绕过启动时的 Root/Debug 检测 | Spawn | 检测代码在启动时执行 |
| 分析已运行的服务进程 | Attach | 服务可能不应被重启 |
| 反复调试同一段代码 | Attach | 保持应用状态，提高效率 |

### 18.9.3 Java 层 Hook 实战

Android 应用的业务逻辑主要用 Java/Kotlin 编写，Java 层 Hook 是最常见的攻击面。

#### 18.9.3.1 基础语法

Frida 使用 `Java.perform()` 确保在 Java VM 线程上执行代码（JNI 要求），在其中通过 `Java.use()` 获取类的引用，然后替换方法的 `implementation` 属性。

```javascript
Java.perform(function() {
    // 获取类引用
    var MyClass = Java.use('com.target.app.MyClass');

    // Hook 方法 —— 重载方法需要指定参数类型
    MyClass.myMethod.implementation = function(arg1, arg2) {
        // 打印原始参数
        console.log('[Hook] myMethod called: arg1=' + arg1 + ', arg2=' + arg2);

        // 调用原始方法（可选，可以修改参数或跳过调用）
        var result = this.myMethod(arg1, arg2);

        // 打印返回值
        console.log('[Hook] myMethod returned: ' + result);

        // 可以修改返回值
        return result;
    };
});
```

**关键要点**：

- `Java.use()` 返回的是类的包装器，不创建新实例
- `implementation` 被赋值后，每次调用目标方法都会执行你的函数
- `this` 关键字指向调用方法的对象实例，可以访问实例字段
- 如果目标方法有多个重载版本，必须用 `.overload(...)` 指定参数类型

#### 18.9.3.2 处理重载方法

Java 方法可以重载（同名不同参数），Frida 需要明确指定要 Hook 哪个版本：

```javascript
Java.perform(function() {
    var MyClass = Java.use('com.target.app.MyClass');

    // Hook 无参版本
    MyClass.myMethod.overload().implementation = function() {
        console.log('[Hook] myMethod() called');
        return this.myMethod();
    };

    // Hook String 参数版本
    MyClass.myMethod.overload('java.lang.String').implementation = function(str) {
        console.log('[Hook] myMethod(String) called: ' + str);
        return this.myMethod(str);
    };

    // Hook int 参数版本
    MyClass.myMethod.overload('int').implementation = function(num) {
        console.log('[Hook] myMethod(int) called: ' + num);
        return this.myMethod(num);
    };

    // Hook 多参数版本
    MyClass.myMethod.overload('java.lang.String', 'int', 'boolean')
        .implementation = function(str, num, flag) {
        console.log('[Hook] myMethod(String, int, boolean)');
        return this.myMethod(str, num, flag);
    };

    // Hook 数组参数 —— 用 '[B' 表示 byte[]
    MyClass.processData.overload('[B').implementation = function(data) {
        console.log('[Hook] process data length: ' + data.length);
        return this.processData(data);
    };

    // Hook 可变参数 —— 实际类型是 Object[]
    MyClass.handle.overload('[Ljava.lang.Object;').implementation = function(args) {
        console.log('[Hook] handle called with ' + args.length + ' args');
        return this.handle(args);
    };
});
```

**类型映射速查**：

| Java 类型 | Frida 类型字符串 |
|-----------|-----------------|
| `byte[]` | `'[B'` |
| `int[]` | `'[I'` |
| `String[]` | `'[Ljava.lang.String;'` |
| `byte[][]` | `'[[B'` |
| `java.util.List` | `'java.util.List'` |
| `java.util.Map` | `'java.util.Map'` |
| `android.os.Bundle` | `'android.os.Bundle'` |

如果不确定参数类型，可以用 Objection 的 `android hooking list class_methods <class>` 命令查看所有重载签名。

#### 18.9.3.3 访问和修改实例字段

```javascript
Java.perform(function() {
    var LoginActivity = Java.use('com.target.app.LoginActivity');

    // Hook 登录方法
    LoginActivity.authenticate.overload('java.lang.String', 'java.lang.String')
        .implementation = function(username, password) {
        console.log('[Login] Username: ' + username);
        console.log('[Login] Password: ' + password);

        // 访问实例字段
        console.log('[Login] mBaseUrl: ' + this.mBaseUrl.value);
        console.log('[Login] mIsDebug: ' + this.mIsDebug.value);

        // 修改实例字段
        this.mIsDebug.value = true;

        // 调用原始方法
        var result = this.authenticate(username, password);
        console.log('[Login] Result: ' + result);
        return result;
    };
});
```

注意：Frida 中访问实例字段需要通过 `.value` 属性。对于静态字段，同样使用 `.value`，但不需要实例：

```javascript
var Config = Java.use('com.target.app.Config');
console.log('[Config] API_KEY: ' + Config.sApiKey.value);
Config.sDebugMode.value = true;  // 修改静态字段
```

#### 18.9.3.4 主动调用（主动执行目标方法）

Frida 不仅能被动拦截方法调用，还能主动调用目标进程中的任意方法：

```javascript
Java.perform(function() {
    // 主动调用静态方法
    var CryptoUtil = Java.use('com.target.app.CryptoUtil');
    var encrypted = CryptoUtil.encrypt('hello', 'secret_key');
    console.log('[CryptoUtil] Encrypted: ' + encrypted);

    // 主动调用实例方法 —— 需要先获取实例
    // 方式1: 从 Activity 获取
    Java.choose('com.target.app.MainActivity', {
        onMatch: function(instance) {
            console.log('[MainActivity] Found instance');
            var token = instance.getToken();
            console.log('[MainActivity] Token: ' + token);
        },
        onComplete: function() {
            console.log('[MainActivity] Search complete');
        }
    });

    // 方式2: 构造新实例
    var User = Java.use('com.target.app.model.User');
    var user = User.$new('admin', 'password123');
    console.log('[User] Hash: ' + user.getPasswordHash());
});
```

`Java.choose()` 会在 Java 堆中搜索指定类的活跃实例，类似于在内存中查找对象。对于 Activity、Service 等有生命周期的组件，这是获取实例最可靠的方式。

### 18.9.4 SSL Pinning 绕过

SSL Pinning（证书锁定）是移动应用最常见的安全加固措施之一。应用在代码中预埋服务端证书或公钥，运行时校验服务端返回的证书是否匹配，防止中间人攻击。绕过 SSL Pinning 是进行流量分析的前提。

#### 18.9.4.1 SSL Pinning 实现方式

理解目标应用使用哪种 Pinning 方式，才能选择正确的绕过策略：

| 实现方式 | 原理 | 常见库 |
|----------|------|--------|
| TrustManager 自定义 | 重写 `X509TrustManager.checkServerTrusted()` | 自定义代码 |
| CertificatePinner | 基于域名的证书指纹校验 | OkHttp3 |
| Network Security Config | Android 7+ XML 配置文件声明信任的证书 | 系统框架 |
| 公钥锁定 | 对比证书公钥的 SHA-256 哈希 | TrustKit, OkHttp |
| WebView SSL | `WebViewClient.onReceivedSslError()` | WebView |

#### 18.9.4.2 通用 Frida SSL Pinning 绕过脚本

```javascript
// ssl-pinning-bypass.js —— 覆盖主流 SSL Pinning 实现
Java.perform(function() {
    console.log('[*] SSL Pinning Bypass loaded');

    // === 1. 绕过自定义 TrustManager ===
    try {
        var X509TrustManager = Java.use('javax.net.ssl.X509TrustManager');
        var SSLContext = Java.use('javax.net.ssl.SSLContext');

        // 创建一个空的 TrustManager（不校验证书）
        var TrustManager = Java.registerClass({
            name: 'com.frida.BypassTrustManager',
            implements: [X509TrustManager],
            methods: {
                checkClientTrusted: function(chain, authType) {},
                checkServerTrusted: function(chain, authType) {},
                getAcceptedIssuers: function() { return []; }
            }
        });

        // 替换 SSLContext 默认的 TrustManager
        var TrustManagers = [TrustManager.$new()];
        var SSLContextInit = SSLContext.init.overload(
            '[Ljavax.net.ssl.KeyManager;',
            '[Ljavax.net.ssl.TrustManager;',
            'java.security.SecureRandom'
        );
        SSLContextInit.implementation = function(keyManager, trustManager, secureRandom) {
            console.log('[+] SSLContext.init called — injecting bypass TrustManager');
            SSLContextInit.call(this, keyManager, TrustManagers, secureRandom);
        };
    } catch(e) {
        console.log('[-] TrustManager bypass error: ' + e);
    }

    // === 2. 绕过 OkHttp3 CertificatePinner ===
    try {
        var CertificatePinner = Java.use('okhttp3.CertificatePinner');
        CertificatePinner.check.overload('java.lang.String', 'java.util.List')
            .implementation = function(hostname, peerCertificates) {
            console.log('[+] Bypassing OkHttp3 CertificatePinner for: ' + hostname);
            // 直接返回，不执行原始的 pin 校验
        };
    } catch(e) {
        console.log('[-] OkHttp3 CertificatePinner not found (app may not use OkHttp3)');
    }

    // === 3. 绕过 OkHttp3 CertificatePinner (新版本 API) ===
    try {
        var CertificatePinner2 = Java.use('okhttp3.CertificatePinner');
        CertificatePinner2.check$okhttp.overload('java.lang.String', 'kotlin.jvm.functions.Function0')
            .implementation = function(hostname, cleanedPeerCertificatesFn) {
            console.log('[+] Bypassing OkHttp3 CertificatePinner (v4+) for: ' + hostname);
        };
    } catch(e) {}

    // === 4. 绕过 WebView SSL 错误处理 ===
    try {
        var WebViewClient = Java.use('android.webkit.WebViewClient');
        WebViewClient.onReceivedSslError.implementation = function(view, handler, error) {
            console.log('[+] Bypassing WebView SSL error');
            handler.proceed();  // 忽略 SSL 错误，继续加载
        };
    } catch(e) {
        console.log('[-] WebView SSL bypass error: ' + e);
    }

    // === 5. 绕过 Network Security Config 的证书校验 ===
    try {
        var Pam = Java.use('android.security.net.config.NetworkSecurityTrustManager');
        Pam.checkServerTrusted.overload(
            '[Ljava.security.cert.X509Certificate;', 'java.lang.String'
        ).implementation = function(chain, authType) {
            console.log('[+] Bypassing NetworkSecurityConfig trust check');
        };
    } catch(e) {}

    console.log('[*] SSL Pinning Bypass script loaded successfully');
});
```


Objection 内置的 `sslpinning disable` 命令包含了主流 Pinning 方式的绕过逻辑，适合快速验证。但如果应用使用了非标准的 Pinning 实现，仍需编写自定义脚本。

#### 18.9.4.4 结合抓包工具验证

绕过 Pinning 后，必须结合抓包工具验证流量是否成功解密：

```bash
# 1. 启动 mitmproxy
mitmproxy --listen-port 8080

# 2. 设备设置代理指向 mitmproxy
adb shell settings put global http_proxy <host-ip>:8080

# 3. 设备安装 mitmproxy CA 证书
#    访问 mitm.it 下载并安装

# 4. 启动应用并注入绕过脚本
frida -U -f com.target.app -l ssl-pinning-bypass.js --no-pause

# 5. 在 mitmproxy 中观察解密的 HTTPS 流量
```

如果仍然看不到流量，可能原因：
- 应用使用了自定义的网络库（不基于标准 Java/OkHttp）
- 应用使用了基于 native 层的 SSL（如 BoringSSL），需要同时 Hook native 函数
- 代理配置未生效（应用可能硬编码了直连 IP）

### 18.9.5 Native 层 Hook

当 Java 层 Hook 无法满足需求时（如应用将核心逻辑放在 native 层、使用 JNI 调用 C/C++ 代码），需要直接 Hook native 函数。

#### 18.9.5.1 Frida Native Hook 基础

```javascript
// Hook native 函数 —— 以 Hook libc 的 open() 为例
Interceptor.attach(Module.findExportByName('libc.so', 'open'), {
    onEnter: function(args) {
        // args[0] = const char* pathname
        var pathname = args[0].readUtf8String();
        console.log('[open] ' + pathname);

        // 保存参数供 onLeave 使用
        this.pathname = pathname;
    },
    onLeave: function(retval) {
        // retval 是返回值（文件描述符）
        if (this.pathname.indexOf('flag') !== -1) {
            console.log('[open] fd=' + retval + ' for ' + this.pathname);
        }
    }
});
```

**`Interceptor.attach` 的两个回调**：

- `onEnter`：函数入口，可以读取/修改参数。`this` 是一个上下文对象，用于在 `onEnter` 和 `onLeave` 之间传递数据。
- `onLeave`：函数出口，可以读取/修改返回值。通过 `retval.replace(new_value)` 替换返回值。

**`NativePointer` 操作**：

```javascript
// 读取内存
var ptr = new NativePointer('0x12345678');
var byte = ptr.readU8();           // 读 1 字节
var word = ptr.readU16();          // 读 2 字节
var dword = ptr.readU32();         // 读 4 字节
var str = ptr.readUtf8String();    // 读 C 字符串
var buf = ptr.readByteArray(16);   // 读 16 字节

// 写入内存
ptr.writeU8(0x90);                 // 写 NOP
ptr.writeUtf8String('hello');      // 写字符串

// 指针运算
var next = ptr.add(4);             // 偏移 +4
```

#### 18.9.5.2 JNI 函数 Hook

Android 应用通过 JNI 调用 native 代码。Hook JNI 函数可以监控 Java 与 native 层的交互：

```javascript
// Hook JNI RegisterNatives —— 发现动态注册的 native 方法
Interceptor.attach(Module.findExportByName('libart.so', 'RegisterNatives'), {
    onEnter: function(args) {
        // args[1] = JNINativeMethod* methods
        // args[2] = nMethods
        var methodsPtr = args[1];
        var nMethods = args[2].toInt32();
        console.log('[RegisterNatives] Registering ' + nMethods + ' methods');

        for (var i = 0; i < nMethods; i++) {
            var namePtr = methodsPtr.add(i * Process.pointerSize * 3).readPointer();
            var sigPtr = methodsPtr.add(i * Process.pointerSize * 3 + Process.pointerSize).readPointer();
            var fnPtr = methodsPtr.add(i * Process.pointerSize * 3 + Process.pointerSize * 2).readPointer();

            var name = namePtr.readUtf8String();
            var sig = sigPtr.readUtf8String();
            var module = Process.findModuleByAddress(fnPtr);

            console.log('  Method: ' + name + ' ' + sig);
            console.log('  Native: ' + fnPtr + ' (' + (module ? module.name : 'unknown') + ')');
        }
    }
});
```

### 18.9.6 加密操作监控

监控应用的加密操作是安全审计的核心环节——提取密钥、理解加密方案、发现弱加密或硬编码密钥。

#### 18.9.6.1 监控 Java 加密 API

```javascript
Java.perform(function() {
    // === 监控 javax.crypto.Cipher ===
    var Cipher = Java.use('javax.crypto.Cipher');

    // 监控加密初始化
    Cipher.init.overload('int', 'java.security.Key')
        .implementation = function(mode, key) {
        var modeStr = (mode === 1) ? 'ENCRYPT' : 'DECRYPT';
        console.log('\n[Cipher.init] Mode: ' + modeStr);
        console.log('[Cipher.init] Algorithm: ' + this.getAlgorithm());

        // 提取密钥字节
        var keyBytes = key.getEncoded();
        console.log('[Cipher.init] Key: ' + bytesToHex(keyBytes));

        this.init(mode, key);
    };

    // 监控 doFinal（加密/解密执行）
    Cipher.doFinal.overload('[B').implementation = function(input) {
        console.log('[Cipher.doFinal] Input (' + input.length + ' bytes): ' + bytesToHex(input));
        var output = this.doFinal(input);
        console.log('[Cipher.doFinal] Output (' + output.length + ' bytes): ' + bytesToHex(output));
        return output;
    };

    // === 监控 MessageDigest（哈希） ===
    var MessageDigest = Java.use('java.security.MessageDigest');

    MessageDigest.digest.overload('[B').implementation = function(input) {
        console.log('[MessageDigest] Algorithm: ' + this.getAlgorithm());
        console.log('[MessageDigest] Input: ' + bytesToHex(input));
        var hash = this.digest(input);
        console.log('[MessageDigest] Hash: ' + bytesToHex(hash));
        return hash;
    };

    // === 监控 SecretKeySpec（密钥构造） ===
    var SecretKeySpec = Java.use('javax.crypto.spec.SecretKeySpec');
    SecretKeySpec.$init.overload('[B', 'java.lang.String')
        .implementation = function(keyBytes, algorithm) {
        console.log('[SecretKeySpec] Algorithm: ' + algorithm);
        console.log('[SecretKeySpec] Key: ' + bytesToHex(keyBytes));
        this.$init(keyBytes, algorithm);
    };

    // === 监控 SharedPreferences 存储 ===
    var SharedPreferencesImpl = Java.use('android.app.SharedPreferencesImpl');
    SharedPreferencesImpl.getString.implementation = function(key, defValue) {
        var value = this.getString(key, defValue);
        console.log('[SharedPrefs] getString("' + key + '") = "' + value + '"');
        return value;
    };

    // === 辅助函数：字节数组转十六进制 ===
    function bytesToHex(bytes) {
        if (!bytes) return 'null';
        var hex = [];
        for (var i = 0; i < bytes.length; i++) {
            hex.push(('0' + (bytes[i] & 0xFF).toString(16)).slice(-2));
        }
        return hex.join('');
    }
});
```

### 18.9.7 Xposed 框架

Xposed 是 Android 平台上另一个重要的 Hook 框架，通过修改 Android 系统的 `app_process` 来实现全局 Hook。与 Frida 不同，Xposed 以模块化插件形式运行，适合长期使用的功能扩展。

#### 18.9.7.1 Xposed vs Frida 对比

| 维度 | Frida | Xposed |
|------|-------|--------|
| 运行方式 | 注入到目标进程 | 系统级 Hook（修改 app_process） |
| 是否需要 Root | 需要（frida-server） | 需要（修改系统） |
| 是否需要重启 | 不需要 | 安装模块后需要重启 |
| Hook 层级 | Java + Native | 主要 Java 层 |
| 脚本语言 | JavaScript | Java（编译为 APK） |
| 适用场景 | 临时分析、渗透测试 | 长期功能扩展、逆向研究 |
| 生态系统 | frida-tools、frida-trace | LSPosed、EdXposed |
| 调试便利性 | 实时修改脚本，即时生效 | 修改后需重新编译安装 |
| 隐蔽性 | 进程级，可被检测 | 系统级，检测难度较高 |

#### 18.9.7.2 Xposed 模块开发基础

```java
// Xposed 模块入口 —— 实现 IXposedHookLoadPackage 接口
package com.example.hookmodule;

import de.robv.android.xposed.IXposedHookLoadPackage;
import de.robv.android.xposed.XC_MethodHook;
import de.robv.android.xposed.XposedBridge;
import de.robv.android.xposed.XposedHelpers;
import de.robv.android.xposed.callbacks.XC_LoadPackage;

public class MainHook implements IXposedHookLoadPackage {

    @Override
    public void handleLoadPackage(XC_LoadPackage.LoadPackageParam lpparam)
            throws Throwable {

        // 只处理目标应用
        if (!lpparam.packageName.equals("com.target.app")) return;

        XposedBridge.log("[Hook] Target app loaded: " + lpparam.packageName);

        // Hook 方法 —— 前置回调
        XposedHelpers.findAndHookMethod(
            "com.target.app.LoginActivity",  // 类名
            lpparam.classLoader,              // 类加载器
            "authenticate",                    // 方法名
            String.class, String.class,        // 参数类型
            new XC_MethodHook() {
                @Override
                protected void beforeHookedMethod(MethodHookParam param)
                        throws Throwable {
                    // param.args 是参数数组
                    String username = (String) param.args[0];
                    String password = (String) param.args[1];
                    XposedBridge.log("[Login] Username: " + username);
                    XposedBridge.log("[Login] Password: " + password);

                    // 修改参数
                    // param.args[0] = "admin";
                }

                @Override
                protected void afterHookedMethod(MethodHookParam param)
                        throws Throwable {
                    // param.getResult() 获取返回值
                    boolean result = (boolean) param.getResult();
                    XposedBridge.log("[Login] Result: " + result);

                    // 修改返回值
                    // param.setResult(true);
                }
            }
        );
    }
}
```

模块的 `assets/xposed_init` 文件中声明入口类：
```text

```

`build.gradle` 中添加 Xposed 依赖：


#### 18.9.7.3 现代 Xposed 生态：LSPosed

LSPosed 是目前最活跃的 Xposed 框架实现，支持 Android 8-14，基于 Magisk + Zygisk 运行：

1. 安装 Magisk（设备 Root）
2. 在 Magisk 中启用 Zygisk
3. 安装 LSPosed 模块（.zip 通过 Magisk 刷入）
4. 重启设备
5. 在 LSPosed 管理器中激活模块并指定目标应用（作用域）

LSPosed 的核心改进：
- **作用域管理**：指定模块只 Hook 哪些应用，避免全局 Hook 的性能和兼容性问题
- **模块仓库**：内置模块下载功能
- **日志系统**：详细的 Hook 日志，方便调试
- **兼容性**：支持 ART 内联 Hook、隐藏 API 绕过

### 18.9.9 frida-trace 自动化追踪

### 18.9.10 反 Hook 与反检测对抗

高级应用会检测并对抗 Hook 框架。理解这些检测手段是绕过它们的前提。

#### 18.9.10.1 Frida 检测方法

| 检测技术 | 原理 | 绕过策略 |
|----------|------|----------|
| 进程检测 | 检查 `/proc/self/maps` 中是否有 `frida-agent` | Hook `fopen`/`fgets` 过滤 maps 输出 |
| 端口检测 | 连接 `127.0.0.1:27042` 检查 frida-server | 修改 frida-server 监听端口 |
| 文件检测 | 检查 `/data/local/tmp/frida-server` 是否存在 | 重命名或删除文件 |
| 内存扫描 | 扫描内存中 frida 的特征字节串 | Hook `read`/`openat` 过滤 maps 内容 |
| D-Bus 检测 | frida 使用 D-Bus 协议通信，有固定特征 | 使用 `--disable-preload` 或修改 Gadget |
| pthread 检测 | 检查 frida 创建的线程名（如 `gmain`、`gdbus`） | Hook `pthread_create` 修改线程名 |
| 代码完整性 | 校验关键函数的前几字节是否被修改（Hook 跳转指令） | 使用硬件断点或 inline hook 的变种 |

#### 18.9.10.2 绕过 Frida 检测的通用脚本

```javascript
// anti-frida-detection.js —— 绕过常见 Frida 检测

// === 1. 绕过 /proc/self/maps 检测 ===
var openPtr = Module.findExportByName('libc.so', 'openat');
Interceptor.attach(openPtr, {
    onEnter: function(args) {
        var pathname = args[1].readUtf8String();
        this.isMaps = false;
        if (pathname && pathname.indexOf('/proc/') !== -1 &&
            pathname.indexOf('maps') !== -1) {
            this.isMaps = true;
        }
    },
    onLeave: function(retval) {
        // 如果需要更精细的控制，可以在 onLeave 中修改读取内容
    }
});

// 绕过 fgets 读取 maps 的检测
var fgetsPtr = Module.findExportByName('libc.so', 'fgets');
Interceptor.attach(fgetsPtr, {
    onEnter: function(args) {
        this.bufSize = args[1].toInt32();
        this.buf = args[0];
    },
    onLeave: function(retval) {
        if (!retval.isNull()) {
            var line = this.buf.readUtf8String();
            if (line && (line.indexOf('frida') !== -1 ||
                        line.indexOf('gadget') !== -1 ||
                        line.indexOf('linjector') !== -1)) {
                // 清除包含 frida 特征的行
                this.buf.writeUtf8String('\0');
            }
        }
    }
});

// === 2. 绕过端口检测 ===
var connectPtr = Module.findExportByName('libc.so', 'connect');
Interceptor.attach(connectPtr, {
    onEnter: function(args) {
        var sockaddr = args[1];
        var family = sockaddr.readU16();
        if (family === 2) {  // AF_INET
            var port = (sockaddr.add(2).readU8() << 8) | sockaddr.add(3).readU8();
            if (port === 27042 || port === 27043) {
                // 修改端口为无效值，使连接失败
                sockaddr.add(2).writeU8(0);
                sockaddr.add(3).writeU8(0);
                console.log('[AntiDetect] Blocked port ' + port + ' detection');
            }
        }
    }
});

// === 3. 绕过 pthread 线程名检测 ===
var pthreadCreatePtr = Module.findExportByName('libc.so', 'pthread_create');
Interceptor.attach(pthreadCreatePtr, {
    onEnter: function(args) {
        this.threadPtr = args[0];
    },
    onLeave: function(retval) {
        // 线程创建后，线程名会被设置为 gmain/gdbus 等
        // 可以在这里延迟修改线程名
    }
});

// 重命名 frida 相关线程
var pthreadSetname = Module.findExportByName('libc.so', 'pthread_setname_np');
if (pthreadSetname) {
    Interceptor.attach(pthreadSetname, {
        onEnter: function(args) {
            var name = args[1].readUtf8String();
            if (name && (name.indexOf('gmain') !== -1 ||
                        name.indexOf('gdbus') !== -1 ||
                        name.indexOf('gum-js-loop') !== -1)) {
```

```bash
#    重命名为 libfrida-gadget.so，放入 target_dir/lib/arm64-v8a/
```

### 18.9.13 常见误区与最佳实践

#### 18.9.13.1 常见错误

**误区1：只 Hook Java 层，忽略 Native 层**

很多安全加固方案会将核心逻辑（加密、认证、校验）放到 C/C++ 层通过 JNI 调用。只在 Java 层 Hook 会遗漏大量攻击面。正确做法是先用 `android hooking list classes` 和 `env --modules` 了解应用加载了哪些 native 库，再针对性地 Hook native 函数。

**误区2：忽视线程安全**

Android 应用是多线程的，Hook 的函数可能同时被多个线程调用。在 Hook 函数中修改共享变量需要考虑竞态条件。使用 `synchronized` 或线程局部变量存储每线程的上下文数据。

**误区3：Hook 点选择不当**

在高层函数 Hook（如 `Cipher.doFinal`）可以看到加密结果，但看不到底层使用的具体算法参数。在太底层的函数 Hook（如单条 ARM 指令）会产生海量日志，淹没真正有用的信息。应该根据分析目标选择合适层级的 Hook 点。

**误区4：不保存 Hook 日志**

动态分析的输出是临时的——进程退出后所有 console.log 都丢失。应该将日志写入文件：

```javascript
// 将日志写入文件
var logFile = new File('/data/local/tmp/hook.log', 'a');
function log(msg) {
    logFile.write(msg + '\n');
    logFile.flush();
    console.log(msg);  // 同时输出到控制台
}
```

或者使用 `frida -l script.js -o output.log` 将所有输出重定向到文件。

**误区5：忽略反调试和完整性校验**

应用可能检测到 Frida 的存在后修改自身行为（如进入"安全模式"、禁用网络请求、清除敏感数据）。Hook 脚本应该优先绕过这些检测机制，否则后续的所有分析都基于被篡改的行为，结论不可靠。

#### 18.9.13.2 最佳实践

1. **先观察后干预**：先用 `frida-trace` 或 Objection 的 watch 命令观察应用正常行为，理解数据流后再决定修改点。

2. **Hook 脚本模块化**：将 SSL 绕过、Root 检测绕过、加密监控等功能拆分为独立脚本，按需组合加载。

3. **使用 `%resume` 调试启动阶段**：Spawn 模式加 `--no-pause` 可以在应用启动前注入多个脚本，但有时需要更精细的控制——省略 `--no-pause`，在暂停状态下设置好所有 Hook 后手动 `%resume`。

4. **版本化管理脚本**：将 Hook 脚本纳入版本控制（Git），记录每个版本的作用和适用场景。

5. **结合多种工具**：Frida 负责 Hook，mitmproxy/Burp Suite 负责抓包，jadx 负责静态分析，三者配合才能全面理解应用行为。
