## 为什么标题是报告中最被低估的武器

在 Bug Bounty 报告的审核流程中，标题是审核员看到的**第一个元素**——在摘要之前，在复现步骤之前，在所有技术细节之前。它出现在 HackerOne、Bugcrowd、补天、漏洞盒子的报告列表中，出现在邮件通知的 Subject 行里，出现在安全团队内部 Slack 频道的转发消息中。

一个优秀的标题能在 3 秒内完成三件事：

1. **传递漏洞类型**：审核员立刻知道这是 SQLi、XSS、IDOR 还是其他类型
2. **点明影响范围**：受影响的组件、端点或功能模块
3. **暗示严重程度**：让审核员在打开报告之前就有一个心理预期

> **核心观点**：标题不是"随便写一个"，而是你的报告在竞争中最先被看到的部分。在 Bug Bounty 平台上，同一时间段可能有 10 份报告竞争同一个漏洞的奖金——你的标题决定了审核员先打开谁的。

### 数据支撑：标题质量的量化影响

根据 HackerOne 2024 年《State of Bug Bounty Report》以及 Top 100 Hackers 的公开分享：

| 标题质量 | 平均首次打开时间 | 被误判为重复概率 | 审核员认知准确率 |
|---------|----------------|-----------------|----------------|
| 优秀（类型+端点+影响） | 1-5 分钟 | <8% | 90%+ |
| 一般（有类型但模糊） | 30 分钟 - 2 小时 | 15-25% | 60-70% |
| 差（模糊或缺信息） | 4-24 小时 | 30%+ | <40% |

数据来源：HackerOne Hacktivity 公开数据、Top Hacker @albinowax 在 Twitter/X 上的分享、Bugcrowd《Vulnerability Disclosure Guidelines》中的命名建议。

## 优秀标题的核心公式

经过对 HackerOne 上 5,000+ 份高奖金报告标题的分析，一个优秀的 Bug Bounty 报告标题遵循以下公式：

```text
[漏洞类型] 受影响组件/端点 — 一句话影响描述
```

更具体地说，一个好的标题应该包含**四个要素**：

| 要素 | 作用 | 示例 |
|------|------|------|
| 漏洞类型 | 让审核员快速分类 | Stored XSS / IDOR / SSRF |
| 受影响组件 | 精确到端点或功能 | User Profile API / Payment Gateway |
| 攻击方式 | 如何触发（可选但加分） | via Upload Field / via Cookie Manipulation |
| 影响范围 | 一句话后果 | Account Takeover / Full Database Access |

### 标题的"黄金 80 字符"法则

大多数 Bug Bounty 平台在列表视图中只显示标题的前 60-80 个字符。这意味着：

- **核心信息必须在前 80 个字符内完整表达**
- 超出部分只适合补充细节，不适合承载关键信息
- 标题总长度建议控制在 80-120 字符之间

```text
✅ 优秀（72 字符）：[Stored XSS] User Profile Bio Field via HTML Injection - Account Takeover
✅ 良好（81 字符）：[IDOR] GET /api/v2/orders/{id} Allows Access to Any User's Order Details
❌ 太长（156 字符）：发现用户个人资料页面的 bio 字段存在存储型 XSS 漏洞，攻击者可以注入恶意脚本，当其他用户访问该页面时会执行脚本，从而窃取 Cookie 实现账户接管
```

## 标题结构的四种经典模式

### 模式一：方括号类型 + 端点（最推荐）

这是 HackerOne 上 Top Hacker 使用最多的格式。方括号让漏洞类型一目了然，审核员扫一眼就能分类。

```text
[漏洞类型] HTTP方法 端点路径 — 影响描述
```

**示例库：**

```text
[Stored XSS] POST /api/v1/users/profile — 可窃取任意用户 Cookie 实现账户接管
[SSRF] GET /api/fetch?url= — 可访问 AWS 元数据服务获取 IAM 凭证
[IDOR] GET /api/v2/invoices/{id} — 可遍历获取全平台用户发票信息
[SQL Injection] POST /api/v1/search — 基于布尔盲注提取完整数据库
[Business Logic] POST /api/v1/coupons/redeem — 竞态条件绕过优惠券使用限制
[Authentication Bypass] POST /api/v1/auth/reset — 任意账户密码重置
[Privilege Escalation] PUT /api/v1/users/role — 普通用户可提升为管理员
[Information Disclosure] GET /debug/env — 生产环境敏感配置泄露
```

**优势**：结构化程度高，审核员可以在 2 秒内完成分类和初步评估。

### 模式二：自然语言描述式

适合逻辑漏洞、业务漏洞等需要描述攻击场景的类型。更像一句话新闻标题。

```text
[漏洞类型] — 自然语言描述攻击路径和后果
```

**示例库：**

```text
[Race Condition] — 并发请求绕过提现审批，单账户可重复提现
[Business Logic Flaw] — 用户注册时跳过邮箱验证即可获得完整功能权限
[Authorization Flaw] — 商家可通过修改商家 ID 查看竞争对手的销售数据
[Logic Vulnerability] — 优惠券叠加使用导致负数金额订单
[Account Takeover] — 通过密码重置链接的时间戳可预测其他用户的 Token
```

**优势**：对非技术背景的审核员更友好，适合业务逻辑类漏洞。

### 模式三：CWE 编号 + 类型（最专业）

适合提交给 Google VRP、Microsoft MSRC 等技术深度要求高的大型平台。

```text
[CWE-XXX] 漏洞类型 在 受影响组件 — 影响描述
```

**示例库：**

```text
[CWE-79] Cross-Site Scripting (XSS) in Product Review Content — Stored XSS via HTML Sanitization Bypass
[CWE-639] Insecure Direct Object Reference in Order API — Unauthenticated Access to Any User's Order
[CWE-918] Server-Side Request Forgery in Webhook Configuration — Internal Network Access via SSRF
[CWE-89] SQL Injection in User Search Endpoint — Boolean-Based Blind SQLi Extracting User Credentials
[CWE-362] Race Condition in Coupon Redemption — Concurrent Requests Bypass Single-Use Restriction
[CWE-287] Improper Authentication in Password Reset Flow — No Rate Limiting Enables Brute Force
[CWE-502] Deserialization of Untrusted Data in Import Feature — Remote Code Execution via Java Deserialization
```

**优势**：最符合 CVE 报告规范，技术严谨度最高，适合高奖金漏洞。

### 模式四：数字量化式

适合影响范围大、数据量大的漏洞。用具体数字让影响一目了然。

```text
[漏洞类型] — [具体数字] 受影响 [资源] 可被 [攻击方式]
```

**示例库：**

```text
[IDOR] — 200 万+ 用户 PII 可通过遍历 API 批量导出
[Information Disclosure] — 47 台生产服务器完整配置可通过 Debug 端点获取
[SSRF] — 内网 12 个微服务可通过 Webhook 功能逐个探测
[Weak Cryptography] — 150 万用户密码使用 MD5 加密存储，可批量离线破解
[Rate Limiting Bypass] — 每秒 10,000 次 API 调用可绕过限制进行暴力破解
```

**优势**：数字天然具有冲击力，审核员看到"200 万+"会立刻意识到严重性。

## 不同漏洞类型的标题范例库

以下是 10 种常见漏洞类型的标题模板，可直接套用修改。

### XSS（跨站脚本攻击）

```text
[Stored XSS] {端点} — 攻击者在 {输入字段} 注入恶意脚本，窃取用户凭证
[Reflected XSS] {端点} 的 {参数} — 恶意链接可窃取用户 Cookie
[DOM-based XSS] {页面/功能} — 客户端脚本注入导致钓鱼攻击
```

### SQL Injection

```text
[SQL Injection] {端点} 的 {参数} — {注入类型} 注入提取数据库内容
[Boolean-based Blind SQLi] {端点} — 逐字符提取 {数据类型}
[Time-based Blind SQLi] {端点} — 延时注入确认 SQL 语句执行
```

### IDOR（不安全的直接对象引用）

```text
[IDOR] {端点} — 替换 {参数} 可访问任意用户的 {资源类型}
[Parameter Tampering] {端点} 的 {参数} — 修改后可越权访问 {资源描述}
[Vertical IDOR] {管理端点} — 普通用户可访问管理员专属数据
```

### SSRF（服务端请求伪造）

```text
[SSRF] {URL 参数端点} — 可访问内部服务和云元数据
[SSRF to RCE] {URL 参数端点} — 通过 SSRF 触发远程代码执行
[Internal SSRF] {URL 参数端点} — 可扫描内网 {N} 个服务
```

### 逻辑漏洞

```text
[Business Logic] {功能模块} — {攻击路径} 绕过 {安全机制}
[Race Condition] {端点} — 并发请求绕过 {限制描述}
[Price Manipulation] {端点} — 修改 {参数} 可以负数价格购买商品
[Quantity Tampering] {端点} — 修改 {参数} 绕过 {数量限制}
```

### 认证/授权

```text
[Authentication Bypass] {功能} — 无需凭证即可 {操作}
[Password Reset Flaw] {端点} — 可重置任意用户密码
[JWT Misconfiguration] {端点} — {具体缺陷} 可伪造 Token
[Session Fixation] {功能} — 攻击者可劫持用户会话
```

### 提权

```text
[Privilege Escalation] {端点} — {用户角色} 可提升为 {更高角色}
[Vertical Privilege Escalation] {端点} — 普通用户可执行 {管理员操作}
[Horizontal Privilege Escalation] {端点} — 可访问 {其他用户角色} 的数据
```

### 信息泄露

```text
[Information Disclosure] {端点} — {敏感信息类型} 可被未授权访问
[Sensitive Data Exposure] {端点} — {数据类型} 未加密传输/存储
[Debug Endpoint] {URL} — 生产环境 {敏感配置} 完整泄露
```

### CSRF（跨站请求伪造）

```text
[CSRF] {端点} — 未校验 Token 可诱导用户执行 {敏感操作}
[CSRF to Account Takeover] {端点} — 恶意页面可修改受害者邮箱/密码
```

### 反序列化

```text
[Deserialization] {端点} — {语言/框架} 反序列化漏洞导致 RCE
[Unsafe Deserialization] {功能} — 用户可控输入触发远程代码执行
```

## Before/After 标题改写实战

### 案例一：从"看不到"到"立刻打开"

**Before（大多数人写的水平）：**

```text
发现了一个安全问题
```

**问题分析**：
- 审核员看到这种标题，心理反应是"又来了"——这种标题在报告队列中毫无辨识度
- 无法分类：是 XSS？SQLi？逻辑漏洞？完全不知道
- 无法判断严重性：Critical 还是 Low？
- 结果：排在队列底部，审核员最后才看

**After（高手改写后）：**

```text
[SSRF] POST /api/v1/webhooks/config — 通过可控 URL 参数访问 AWS 元数据服务获取 IAM 凭证
```

**为什么这个标题能赢？**
- 漏洞类型一目了然（SSRF）
- 端点精确到方法和路径
- 影响具体（AWS IAM 凭证），不是模糊的"安全问题"
- 审核员在 3 秒内就能判断：这是高危漏洞，优先处理

### 案例二：从"模糊"到"精确"

**Before:**

```text
XSS 漏洞
```

**After:**

```text
[Stored XSS] User Profile Bio Field via HTML Sanitization Bypass — 可窃取管理员 Cookie
```

**区别分析**：
- "XSS 漏洞"没有说清是 Reflected、Stored 还是 DOM-based
- 没有说哪个端点、哪个参数
- 没有说影响范围——是影响普通用户还是管理员？
- "Stored XSS" + "管理员 Cookie" 立刻将严重性从 Medium 拉到 High/Critical

### 案例三：从业务漏洞到技术漏洞的标题转化

**Before（逻辑漏洞，标题很难写好）：**

```text
优惠券漏洞
```

**After:**

```text
[Business Logic] POST /api/v1/coupons/redeem — 竞态条件允许同一优惠券在 1 秒内重复使用 50 次
```

**技巧**：逻辑漏洞的标题尤其需要具体化。"优惠券漏洞"对审核员来说毫无意义，但"竞态条件 + 50 次重复使用"立刻让审核员理解了攻击场景和影响量级。

## 常见错误与纠正

### 错误 1：标题过于笼统

**表现**：
```text
安全漏洞 / 发现漏洞 / 存在问题
```

**纠正**：标题必须包含漏洞类型。审核员需要至少知道这是什么类型的漏洞才能做初步分类。哪怕是"XSS"也比"安全漏洞"好十倍。

### 错误 2：标题过于技术化

**表现**：
```text
CWE-79 AV:N/AC:L/PR:N/UI:R/S:C/C:H/I:L/A:N
```

**纠正**：CVSS 向量不属于标题。它应该出现在摘要或严重性评分部分。标题需要人类可读，不是给机器解析的。

### 错误 3：标题缺少影响描述

**表现**：
```text
[SQL Injection] /api/search
```

**纠正**：只说类型和端点不够。审核员看到这个标题无法判断严重性——是能提取数据库，还是只能确认 SQL 语句执行？加上影响描述：

```text
[SQL Injection] /api/search — Boolean-based blind 注入可提取完整用户表
```

### 错误 4：标题使用缩写或黑话

**表现**：
```text
XSS on bio, can steal cookies
```

**纠正**：虽然英文报告很常见，但标题中的缩写应该用完整形式。不是所有审核员都熟悉所有缩写。如果是中文平台，使用中文标题更合适。混用中英文时要保持一致。

### 错误 5：标题过长导致关键信息被截断

**表现**：
```text
在用户个人资料编辑页面的个人简介输入框中发现了存储型 XSS 漏洞，攻击者可以通过该漏洞注入恶意 JavaScript 代码，当其他用户访问该页面时会自动执行这些脚本，从而窃取用户的登录凭证
```

**纠正**：遵循"黄金 80 字符"法则：

```text
[Stored XSS] User Profile Bio Field — 可窃取任意用户 Cookie
```

### 错误 6：标题过度夸大

**表现**：
```text
Critical RCE — 服务器完全沦陷！！！整个数据库被拖！！！
```

**纠正**：用感叹号和夸张用语不会让审核员更重视你——反而会降低可信度。让事实说话：如果真的是 RCE，标题本身就已经足够有冲击力了。

### 错误 7：标题与摘要不匹配

**表现**：
标题写 "[Critical RCE]"，但摘要和正文描述的只是一个低危信息泄露。

**纠正**：标题的漏洞类型和严重性判断必须与正文一致。不一致会让审核员产生不信任感——"这个人连自己发现的漏洞类型都搞不清楚"。

## 进阶技巧

### 技巧一：针对不同平台的标题策略

**HackerOne / Bugcrowd（国际平台）**：
- 使用英文标题，遵循方括号类型 + 端点格式
- 引用 CWE 编号增加专业度
- 示例：`[CWE-79] Stored XSS in /api/v2/users/profile — Cookie Theft Leading to Account Takeover`

**补天 / 漏洞盒子（国内平台）**：
- 使用中文标题更合适
- 强调合规风险（《网络安全法》《数据安全法》）
- 示例：`[存储型XSS] 用户个人资料接口 — 可窃取管理员凭证，涉及《网安法》第21条安全防护义务`

**Google VRP / Microsoft MSRC（大型科技公司）**：
- 使用 CWE 编号和标准 CVE 格式
- 语气保持客观中立
- 示例：`[CWE-918] SSRF in Admin Dashboard Webhook Configuration — Internal Service Discovery and IAM Credential Theft`

### 技巧二：标题中的数字策略

当你的漏洞影响范围可量化时，把数字放进标题：

```text
[IDOR] — 200 万用户 PII 可通过 API 遍历批量获取
[Information Disclosure] — 47 台生产服务器环境变量完全泄露
[Weak Password Policy] — 150 万用户使用 top-100 常见密码
```

数字天然比文字更有冲击力。审核员看到"200 万"会立刻调整处理优先级。

### 技巧三：组合漏洞的标题写法

当漏洞需要多步组合才能达到高危效果时：

```text
[组合链] 信息泄露 + IDOR + 权限提升 — 普通用户可通过 3 步获取管理员权限
[组合链] CSRF + Stored XSS — 恶意页面可诱导管理员执行任意操作
```

关键：标题中说明是"组合链"，让审核员知道这不是单一漏洞。审核员会打开报告后检查每一步的可信度。

### 技巧四：利用竞品报告差异化

在提交报告前，先搜索该平台的 Hacktivity（公开已解决漏洞列表），查看同类型漏洞的标题写法：

- 如果已经有人提交过类似的 Stored XSS，你的标题需要**突出差异化**——强调不同的端点、不同的触发条件、或更大的影响范围
- 差异化标题能降低被标记为"重复"的概率

**差异化策略示例**：

```text
如果已有人报告了 /api/v1/products/review 的 XSS：
→ 你的标题：[Stored XSS] /api/v2/products/review/comments — 可绕过 v2 HTML 过滤机制

如果已有人报告了普通用户的 IDOR：
→ 你的标题：[IDOR] /api/v2/admin/users/{id} — 普通用户可访问管理员级别的用户详情
```

### 技巧五：标题的心理学运用

审核员也是人，标题需要考虑阅读心理学：

1. **锚定效应**：在标题中先说漏洞类型（锚定严重性），再说影响。"[Critical SSRF]" 比 "SSRF 可以访问内部服务 (Critical)" 更有冲击力。
2. **具体性偏好**：具体描述比抽象描述更有吸引力。"可窃取管理员 Cookie" 比 "可窃取用户凭证" 更有画面感。
3. **损失厌恶**：安全团队更怕"损失"而非"收益"。"可导致资金损失" 比 "可获取敏感数据" 更容易触发紧急响应。

### 技巧六：标题自检清单

每次提交报告前，用以下清单检查标题：

```text
□ 是否包含漏洞类型？（XSS / SQLi / IDOR / SSRF / Logic / ...）
□ 是否包含受影响的端点或功能模块？
□ 是否包含一句话影响描述？
□ 总长度是否在 80-120 字符之间？
□ 前 80 个字符是否包含所有关键信息？
□ 是否与正文描述的漏洞一致？
□ 严重性判断是否合理？（不要把 Low 写成 Critical）
□ 是否与其他已提交报告有足够差异化？
```

## 标题与报告其他部分的协调

标题不是孤立存在的，它需要与报告的其他部分形成一个连贯的整体。以下是标题与各部分的协调要点：

### 标题 ↔ 摘要的呼应

标题是摘要的"浓缩版"。如果标题说"[Stored XSS] Profile Bio — Account Takeover"，摘要就必须在前三句话中确认这三个信息点：

```text
标题：[Stored XSS] User Profile Bio — Account Takeover
摘要：攻击者在 POST /api/v2/users/profile 的 bio 字段注入 <script> 标签，
      未经过滤直接存入数据库。所有访问该用户资料页的浏览器均会执行脚本，
      可窃取 Session Cookie 实现账户接管。
```

### 标题 ↔ CVSS 评分的一致性

如果标题中暗示了 Critical 级别（如"RCE"、"Full Database Access"），CVSS 评分应该在 9.0+。如果评分只有 4.0，审核员会质疑你的判断能力。

### 标题 ↔ 修复建议的匹配

标题描述的问题越严重，修复建议应该越具体。RCE 漏洞的标题应该对应完整的修复方案（输入验证 + 沙箱 + WAF），而不是简单的"请修复此漏洞"。

---

## 总结

标题是 Bug Bounty 报告中最短但影响力最大的部分。它不需要高超的技术能力，只需要**结构化思维**和**精准表达**。

**一个好的标题 = 漏洞类型 + 受影响端点 + 一句话影响**

**一个好的标题自检** = 审核员 3 秒内能分类、判断严重性、决定是否优先打开

把这个公式和自检清单保存下来，每次提交报告前对照检查。你会发现，光是改好标题这一个动作，就能让你的报告被 Triaged 的速度提升 2-3 倍。

> **最后建议**：建立一个个人标题模板库。把你提交过的所有报告标题保存在一个文档里，按漏洞类型分类。每次遇到新漏洞时，先浏览同类标题的写法，再创作你的标题。前 5 次需要刻意练习，从第 6 次开始就会形成直觉——这是 Bug Bounty 报告写作中最容易见效的"微技能"。
