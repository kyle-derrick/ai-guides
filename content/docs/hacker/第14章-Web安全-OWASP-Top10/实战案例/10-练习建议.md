## 14.42 练习建议

本章通过8个真实案例展示了OWASP Top 10各类风险在实际环境中的表现形式。理论学习和案例分析只是起点，真正的安全能力必须通过系统化的动手练习来建立。本节将所有案例转化为可执行的练习计划，帮助你从"知道漏洞"进阶到"能发现、能利用、能防御"。

### 14.42.1 案例-练习对照表

每个案例都对应特定的技能维度，练习时应明确目标：

| 案例 | OWASP分类 | 核心技能 | 推荐靶场 | 难度 |
|------|-----------|----------|----------|------|
| 案例一：电商平台SQL注入 | A03 注入 | SQL注入检测、数据提取、WAF绕过 | DVWA、SQLi-labs、PortSwigger SQL Injection | ★★☆ |
| 案例二：社交平台XSS蠕虫 | A03 注入 | XSS发现、Payload构造、DOM分析 | DVWA、XSS Hunter、PortSwigger XSS | ★★★ |
| 案例三：云服务SSRF攻击链 | A10 SSRF | URL探测、协议利用、元数据访问 | SSRF Labs、HackTheBox | ★★★ |
| 案例四：金融系统业务逻辑漏洞 | A01+A04 | 流程分析、参数篡改、竞态条件 | Juice Shop、PortSwigger Business Logic | ★★★★ |
| 案例五：SolarWinds供应链攻击 | A08 完整性失效 | 依赖审计、签名验证、CI/CD安全 | Snyk演示项目、GitHub Advisory DB | ★★★★ |
| 案例六：GraphQL内省查询泄露 | A01+A05 | Schema发现、查询构造、权限测试 | DVGA、GraphQL Voyager | ★★★ |
| 案例七：OAuth重定向URI攻击 | A07 认证失效 | OAuth流程分析、URI劫持、Token窃取 | PortSwigger OAuth、OAuth安全测试环境 | ★★★★ |
| 案例八：WebSocket跨站劫持 | A01+A07 | WebSocket握手分析、CSWSH利用 | PortSwigger WebSocket、自建靶场 | ★★★ |

### 14.42.2 分阶段练习路径

#### 第一阶段：复现案例（2-3周）

目标是完全复现每个案例中的攻击过程，理解每一步的技术细节。

**练习流程**（以案例一SQL注入为例）：

```text
1. 阅读案例 → 理解攻击场景和影响
2. 搭建靶场环境（DVWA/SQLi-labs）
3. 手动复现每一步操作
4. 记录payload、响应、结果
5. 对比案例中的手法，理解差异
6. 总结：哪些防御措施可以阻断此攻击
```

**每日练习计划**：

| 天数 | 内容 | 时长 | 产出 |
|------|------|------|------|
| 第1-2天 | 案例一：SQL注入 — UNION注入、盲注、时间盲注复现 | 2-3h | 注入检测清单 |
| 第3-4天 | 案例二：XSS — 反射型、存储型、DOM型复现 | 2-3h | XSS Payload库 |
| 第5-6天 | 案例三：SSRF — 内网探测、协议利用、云元数据访问 | 2-3h | SSRF检测流程 |
| 第7-8天 | 案例四：业务逻辑 — 金额篡改、流程绕过、竞态条件 | 2-3h | 逻辑漏洞检查表 |
| 第9-10天 | 案例五：供应链 — 依赖审计、Typosquatting分析 | 1-2h | 依赖安全报告模板 |
| 第11-12天 | 案例六：GraphQL — 内省查询、批量查询、深度嵌套 | 2-3h | GraphQL测试脚本 |
| 第13-14天 | 案例七+八：OAuth + WebSocket — 认证流程劫持 | 2-3h | 认证安全测试清单 |

#### 第二阶段：扩展练习（3-4周）

在复现基础上，拓展到案例未覆盖的变体和绕过技巧。

**案例一扩展：SQL注入深度练习**

```bash
# 1. WAF绕过练习 — 使用SQLi-labs高级关卡
# 练习以下绕过技术：
# - 大小写混合：SeLeCt, UnIoN
# - 注释绕过：/*!UNION*/ /*!SELECT*/
# - 编码绕过：URL编码、双重编码、Unicode编码
# - 内联注释：/*!50000UNION*/ /*!50000SELECT*/
# - 空格替代：%09(Tab), %0A(换行), /**/(), 括号

# 2. 不同数据库的注入差异
# MySQL:  LIMIT, GROUP_CONCAT, INFORMATION_SCHEMA
# PostgreSQL: STRING_AGG, pg_catalog, LIMIT ALL
# MSSQL: TOP, FOR XML PATH, sysobjects
# Oracle: ROWNUM, WM_CONCAT, ALL_TABLES

# 3. 使用SQLMap自动化验证手动发现
sqlmap -u "http://target/search?id=1" --batch --dbs --level=3 --risk=2
```

**案例二扩展：XSS高级Payload构造**

```javascript
// 1. CSP绕过练习（在配置了不同CSP策略的靶场上练习）
// 练习目标：绕过以下CSP策略
// - script-src 'self'（寻找JSONP端点）
// - script-src 'unsafe-inline'（直接注入）
// - script-src *.googleapis.com（利用CDN）

// 2. 过滤器绕过技术
// 大小写：<ScRiPt>alert(1)</sCrIpT>
// 编码：<img src=x onerror=&#97;&#108;&#101;&#114;&#116;(1)>
// 事件属性：<details open ontoggle=alert(1)>
// SVG利用：<svg/onload=alert(1)>
// JavaScript伪协议：<a href="javascript:alert(1)">click</a>

// 3. DOM XSS练习
// 寻找危险的source和sink
// Sources: location.hash, document.referrer, window.name
// Sinks: innerHTML, eval(), document.write(), setTimeout()
```

**案例三扩展：SSRF协议利用清单**

```text
# 在靶场环境中逐一测试以下协议：
file:///etc/passwd                    # 本地文件读取
dict://127.0.0.1:6379/info           # Redis命令执行

http://metadata.google.internal/      # GCP元数据
http://169.254.169.254/metadata/      # Azure元数据

# 绕过技巧练习：
# - IP十进制表示：http://2130706433 (= 127.0.0.1)

# - 302重定向：在可控服务器上设置跳转到内网地址
# - URL解析差异：http://127.0.0.1@evil.com
```

#### 第三阶段：综合挑战（2-3周）

将多个案例中的技术组合运用，模拟真实渗透测试场景。

**挑战一：模拟电商平台全面渗透**

```text
场景：你获得了一个电商网站的测试授权，需要进行完整的Web安全评估。

步骤：
1. 信息收集 → 识别技术栈、子域名、API端点
2. 认证测试 → 暴力破解、Session固定、JWT弱点
3. 注入测试 → SQL注入、NoSQL注入、命令注入
4. XSS测试 → 反射型、存储型、DOM型
5. 访问控制 → IDOR、权限提升、API越权
6. 业务逻辑 → 价格篡改、优惠券重用、竞态条件
7. 报告撰写 → 漏洞评级、复现步骤、修复建议
```

**挑战二：GraphQL API安全评估**

```text
场景：目标使用GraphQL作为主要API层，需要评估其安全性。

步骤：
1. 端点发现 → /graphql, /graphiql, /v1/graphql
2. 内省查询 → 获取完整Schema
3. 查询深度攻击 → 嵌套查询导致DoS
4. 批量查询 → 单次请求执行多个操作
5. 授权绕过 → 修改查询访问未授权字段
6. 注入测试 → GraphQL参数中的SQL/NoSQL注入
```

**挑战三：认证系统深度测试**

```text
场景：评估一个使用OAuth 2.0 + JWT的认证系统。

步骤：
1. OAuth流程分析 → 识别grant type、重定向URI
2. Token安全 → JWT算法篡改、密钥强度、过期验证
3. 会话管理 → Session固定、并发登录、注销清理
4. 多因素认证 → 绕过尝试、备份码逻辑
5. 密码策略 → 复杂度要求、历史检查、锁定机制
```

#### 效率提升技巧

**善用Burp Suite的工作流**：

```text
1. Proxy → 拦截和修改请求（手动测试的核心）
2. Repeater → 反复发送修改后的请求（调试Payload）
3. Intruder → 自动化模糊测试（暴力枚举、参数遍历）
4. Scanner → 自动扫描（Community版功能有限）
5. Decoder → 编码解码（URL、Base64、HTML实体）
6. Comparer → 对比请求/响应差异（判断布尔盲注结果）
```

**手动测试优于盲目扫描**：

自动化工具适合发现已知模式的漏洞，但业务逻辑漏洞、复杂的认证绕过、竞态条件等都需要手动分析。建议先手动测试关键功能，再用自动化工具补充覆盖。

### 14.42.7 常见练习误区

**误区一：只用自动化工具扫描**

很多初学者安装SQLMap后就开始扫描目标，发现不了漏洞就觉得"这个应用很安全"。自动化工具只能发现模式化的漏洞，需要先手动理解应用逻辑，再用工具辅助验证。

**误区二：只练习攻击不学防御**

理解防御机制不仅帮助你成为更好的开发者，也能帮助你发现更深的漏洞。例如，了解WAF的工作原理后，你才能设计出更有效的绕过方法。

**误区三：Payload收集癖**

背诵大量Payload不如理解漏洞原理。当你真正理解了SQL注入的本质是"将用户输入混入SQL语句结构"，你就能针对任何场景构造出有效的Payload。

**误区四：忽略信息收集**

直接开始漏洞测试而跳过信息收集是最常见的错误。技术栈识别、API文档发现、子域名枚举、JS文件分析往往比直接测试能发现更多有价值的信息。

**误区五：练习环境过于简单**

只在DVWA的Low难度上练习SQL注入，和在真实应用中发现SQL注入是完全不同的事。真实应用有多层防御（WAF、参数化查询、输入过滤），需要系统性地提升难度。
