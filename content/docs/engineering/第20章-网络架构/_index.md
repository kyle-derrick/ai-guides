# 第20章 网络架构

## 章节定位

网络架构是现代分布式系统的骨架。从单体应用到微服务，从本地部署到全球分布，网络架构的设计直接决定了系统的可用性、扩展性和安全性。本章系统梳理网络架构的核心概念、设计模式和工程实践，帮助读者建立从负载均衡到服务网格的完整知识体系。

**本章在全书中的位置**：第18章（TCP/IP协议栈）和第19章（应用层协议）提供了网络通信的底层基础，本章在此之上构建系统级的网络架构设计能力。后续第21章（分布式理论）和第22章（分布式共识）则依赖本章的网络基础设施进行分布式系统设计。

## 核心问题

本章围绕以下五个核心问题展开：

1. **流量分发**：如何将流量高效、均匀地分发到后端服务，同时应对服务器性能差异和动态扩缩容？
2. **延迟优化**：如何构建高可用、低延迟的网络拓扑，从全局视角优化请求路径？
3. **服务间通信**：如何在微服务架构中管理服务间通信，实现安全、可观测、可治理的网络？
4. **安全防护**：如何在不牺牲性能的前提下保障网络层面的安全性？
5. **架构权衡**：如何在性能、可靠性和复杂度之间做出最优权衡？

## 术法道贯通

- **术**：掌握Nginx配置、Envoy Filter编写、Istio VirtualService定义、Redis分布式限流Lua脚本、Kubernetes NetworkPolicy等具体技术
- **法**：理解负载均衡算法的选择原则、熔断限流的组合策略、服务网格的演进路径、零信任网络的实施阶段
- **道**：认识网络架构的本质——在可靠性、性能和复杂度之间寻找最优平衡。没有银弹，每个决策都是权衡（Trade-off）

### 常见误区

> 📖 详见 [常见误区](04-常见误区.md)（八大误区深度剖析，27KB）

八个高频架构误区，每个都按照"错误认知→根因分析→正确做法"的结构展开：

1. **没有系统性监控** — 只靠用户投诉发现故障 → 建立多层次监控体系
2. **过早优化** — 没有性能数据就优化 → 先度量再优化
3. **负载均衡策略选错** — 盲目选择 → 根据业务特征选择算法
4. **TCP参数配置错误** — backlog/tw_reuse/keepalive/缓冲区 → 生产级sysctl参数配置
5. **缺少超时重试熔断** — 一个服务故障拖垮整个系统 → 分层容错机制
6. **DNS配置被忽视** — TTL设置不当导致故障切换慢 → DNS最佳实践
7. **网络安全过松或过紧** — 要么完全开放要么影响正常业务 → 渐进式安全策略
8. **缺乏可观测性和混沌工程** — 无法观测→无法运维 → 可观测性三支柱+混沌工程

每个误区都附带具体的Nginx/sysctl/YAML/Python配置示例和最佳实践参数表。

## 关键设计原则

本章提炼出六条贯穿始终的设计原则：

1. **分层设计**：L4+L7各司其职，每层只做自己最擅长的事。L4追求极致性能（内核态转发，百万级PPS），L7追求极致灵活性（应用层解析，内容路由）。
2. **冗余消除**：每个组件都要考虑高可用，消除单点故障。负载均衡器用Keepalived VIP，服务用多副本，链路用ECMP多路径。
3. **渐进式架构**：从简单开始，按需演进。DNS轮询→Nginx反向代理→L4+L7混合→服务网格，每个阶段解决当前规模的问题。
4. **可观测性优先**：监控、追踪、日志缺一不可。Golden Metrics（QPS、错误率、P99延迟、流量）是最小可观测集。
5. **安全纵深**：多层防护，不依赖单一机制。WAF+DDoS防护+mTLS+微隔离+零信任，纵深防御。
6. **简单优于复杂**：KISS原则，但不能以牺牲必要功能为代价。服务数量<10不需要服务网格，<50可选评估。

## 关键公式与指标

| 公式/指标 | 表达式 | 应用场景 |
|-----------|--------|---------|
| TCP吞吐量 | 吞吐量 ≈ 窗口大小 / RTT | 评估网络传输能力 |
| SLA换算 | 99.9%=8.76h/年, 99.99%=52.6min/年, 99.999%=5.26min/年 | 可用性目标设定 |
| 延迟组成 | 总延迟 = 传输延迟 + 传播延迟 + 排队延迟 + 处理延迟 | 延迟优化定位 |
| 一致性哈希偏移 | 节点增减影响 ≈ 1/n 数据量 | 扩缩容影响评估 |
| 收敛比 | 下行端口 / 上行端口（如5:1） | 数据中心网络容量规划 |
| Little's Law | L = λ × W（并发数 = 到达率 × 平均处理时间） | 系统容量规划 |
