## 3. 指针深入理解

指针是 C/C++ 最强大也最危险的特性。在安全领域，绝大多数内存破坏漏洞——缓冲区溢出、Use-After-Free、格式化字符串、控制流劫持——都与指针直接相关。理解指针的底层机制，是掌握漏洞利用与防御的必经之路。

### 3.1 指针的本质：一个存储地址的变量

指针并非什么神秘概念，它就是一个普通的变量，只不过存储的值是另一个变量的内存地址。在 64 位系统中，指针固定占 8 字节（无论指向什么类型）；在 32 位系统中占 4 字节。

```c
#include <stdio.h>

int main() {
    int x = 42;
    int *p = &x;        // p 存储 x 的地址
    int **pp = &p;      // pp 存储 p 的地址（指针的指针）

    printf("x 的值:     %d\n", x);            // 42
    printf("x 的地址:   %p\n", (void*)&x);    // 如 0x7ffd12345678
    printf("p 的值:     %p\n", (void*)p);     // 同 x 的地址
    printf("*p 的值:    %d\n", *p);           // 42（解引用）
    printf("pp 的值:    %p\n", (void*)pp);    // p 的地址
    printf("*pp 的值:   %p\n", (void*)*pp);   // x 的地址
    printf("**pp 的值:  %d\n", **pp);         // 42

    // 查看指针本身的大小
    printf("sizeof(p):  %zu\n", sizeof(p));   // 8（64位）
    printf("sizeof(pp): %zu\n", sizeof(pp));  // 8（无论指向几级）

    return 0;
}
```

**关键认知：** 指针的值就是地址，`*` 运算符的含义是"去这个地址取值"。`&` 运算符的含义是"取这个变量的地址"。这构成了指针操作的全部基础。

#### 3.1.1 指针类型的作用

指针类型并非决定"能不能存地址"——任何指针都能存任何地址。类型的作用是：

- **决定解引用时读取多少字节**：`int*` 解引用读 4 字节，`char*` 读 1 字节
- **决定指针运算的步长**：`p+1` 对 `int*` 前进 4 字节，对 `char*` 前进 1 字节
- **编译器类型检查**：防止误操作（但可以强制转换绕过）

```c
int val = 0x41424344;   // 内存中存储为 44 43 42 41（小端序）

int *ip = &val;
char *cp = (char*)&val;

printf("*ip = 0x%x\n", *ip);     // 0x41424344（读取4字节）
printf("*cp = 0x%x\n", *cp);     // 0x44（只读取1字节，最低位字节）
```

从安全角度看，类型混淆（Type Confusion）漏洞正是利用了这一机制：同一块内存被不同类型的指针解释，导致程序行为偏离预期。

#### 3.1.2 void 指针

`void*` 是通用指针，可以接收任何类型的地址，但不能直接解引用（编译器不知道读取多少字节）。常用于泛型接口：

```c
void *vp = malloc(64);    // malloc 返回 void*
// *vp;  // 编译错误：不能对 void* 解引用

// 使用时必须转换为具体类型
int *ip = (int*)vp;
*ip = 100;  // 正确
```

### 3.2 指针运算

指针运算是 C 语言高效操作内存的核心机制，也是缓冲区溢出漏洞的技术基础。

```c
int arr[] = {10, 20, 30, 40, 50};
int *p = arr;  // arr 等价于 &arr[0]

// 指针算术：p + n 前进 n * sizeof(元素类型) 字节
p + 1;    // 指向 arr[1]，地址增加 sizeof(int) = 4 字节
p + 2;    // 指向 arr[2]

// 数组遍历：指针方式 vs 下标方式
for (int i = 0; i < 5; i++) {
    printf("arr[%d] = %d, addr = %p\n", i, *(p + i), (void*)(p + i));
}
// *(p + i) 等价于 arr[i]，编译器生成的指令完全相同

// 指针相减：结果是元素个数，不是字节数
int *start = &arr[0];
int *end = &arr[4];
ptrdiff_t diff = end - start;  // diff = 4（元素个数）

// 指针比较：比较地址大小，用于判断相对位置
if (end > start) {
    printf("end 在 start 之后\n");
}
```

**安全要点：** 指针本身不做边界检查。`p + 100` 可以指向数组之外的任意地址，这是缓冲区溢出的根本原因。C 语言信任程序员不会越界——而攻击者恰恰利用了这一点。

#### 3.2.1 指针与数组的关系

数组名在大多数上下文中退化为指向首元素的指针，但两者并不等价：

```c
int arr[5] = {1, 2, 3, 4, 5};
int *p = arr;

// 相同点
arr[2] == *(p + 2);   // 都是 3

// 不同点
sizeof(arr);   // 20（整个数组的大小：5 * 4）
sizeof(p);     // 8（指针本身的大小）

// arr 不可重新赋值（数组名是常量）
// arr = p;    // 编译错误
p = arr + 3;   // 指针可以自由移动
```

#### 3.2.2 数组指针与指针数组

这两个概念名字相似但含义完全不同，混淆它们是常见错误：

```c
// 指针数组：数组的每个元素都是指针
int *ptr_arr[5];       // 5 个 int* 指针组成的数组
ptr_arr[0] = &x;
ptr_arr[1] = &y;

// 数组指针：指向一个数组的指针
int (*arr_ptr)[5];     // 指向含 5 个 int 的数组的指针
int matrix[3][5];
arr_ptr = matrix;      // 指向 matrix 的第一行
arr_ptr++;             // 跳过一行（5 * 4 = 20 字节）
```

在多维数组操作中，数组指针用于按行遍历，指针数组则常用于实现不规则的指针表——后者在内核 exploit 中常见，因为函数指针表（如 vtable）本质上就是指针数组。

### 3.3 const 与指针

`const` 修饰指针时的位置决定了它限制的是"指针本身"还是"指向的值"：

```c
int x = 10, y = 20;

// 1. 指向常量的指针（不能通过该指针修改值，但指针本身可以改指向）
const int *p1 = &x;
// *p1 = 100;  // 编译错误：不能通过 p1 修改
p1 = &y;       // 合法：p1 可以指向别处

// 2. 常量指针（指针本身不可改指向，但可以通过指针修改值）
int *const p2 = &x;
*p2 = 100;     // 合法：可以修改 x
// p2 = &y;   // 编译错误：p2 不可改指向

// 3. 指向常量的常量指针（都不可改）
const int *const p3 = &x;
// *p3 = 100;  // 错误
// p3 = &y;    // 错误
```

记忆技巧：**`const` 修饰它左边的东西，如果左边没有东西则修饰右边的**。`const int *` 中 `const` 修饰 `int`（值不可改）；`int *const` 中 `const` 修饰 `*p`（指针不可改）。

**安全意义：** 正确使用 `const` 可以防止意外修改关键数据。在安全编码规范中（如 CERT C），所有不需要修改的参数都应声明为 `const` 指针，这既是文档也是防护。

### 3.4 函数指针

函数指针是 C 语言实现多态、回调、插件系统的核心机制，同时也是控制流劫持攻击的主要目标。

#### 3.4.1 基本用法

```c
#include <stdio.h>

void greet()      { printf("Hello!\n"); }
void farewell()   { printf("Goodbye!\n"); }

// 函数指针类型定义
typedef void (*func_t)();

int main() {
    func_t fp = greet;     // 函数名本身就是函数指针
    fp();                  // 调用 greet
    fp = farewell;
    fp();                  // 调用 farewell

    // 函数指针数组——调度表
    func_t dispatch[2] = {greet, farewell};
    for (int i = 0; i < 2; i++) {
        dispatch[i]();
    }

    return 0;
}
```

#### 3.4.2 函数指针与安全：控制流劫持

在漏洞利用中，攻击者通过溢出覆盖函数指针的值，将控制流重定向到任意地址。这就是经典的 **控制流劫持（Control Flow Hijacking）**。

```c
#include <stdio.h>
#include <string.h>

void normal_func() {
    printf("正常执行流程\n");
}

void shell() {
    printf("攻击者获得控制权！执行 shellcode...\n");
}

int main() {
    void (*func_ptr)() = normal_func;
    char buf[16];

    // 漏洞：未限制输入长度
    gets(buf);  // 如果输入超过16字节，会覆盖 func_ptr

    func_ptr();  // 调用函数指针
    return 0;
}
```

攻击者输入精心构造的 payload，使 `buf` 溢出后覆盖 `func_ptr` 为 `shell` 的地址，程序就会执行 `shell()` 而非 `normal_func()`。

这种攻击模式延伸到以下实际场景：

| 场景 | 攻击目标 | 原理 |
|------|---------|------|
| C++ 虚函数表劫持 | vtable 指针 | 对象的 vptr 被覆盖，指向伪造的 vtable |
| GOT 覆写 | 全局偏移表 | 覆盖库函数的 GOT 条目，下次调用时跳转到攻击者地址 |
| 覆盖回调函数指针 | 结构体中的函数指针 | 如 signal handler、atexit 注册的函数 |
| ROP（返回地址覆盖） | 栈上的返回地址 | 覆盖返回地址，串联已有的代码片段（gadget） |

### 3.5 危险指针模式

以下模式是真实漏洞中最常见的根因。掌握它们不仅是"写出安全代码"的前提，更是"读懂漏洞 PoC"的基础。

#### 3.5.1 野指针（Dangling Pointer）与 Use-After-Free

内存释放后，指针并未自动置空。此时指针仍然保存着已释放内存的地址——这就是野指针。如果继续使用，就是 Use-After-Free（UAF）漏洞。

```c
char *p = malloc(64);
strcpy(p, "important data");

free(p);          // 内存已归还给堆管理器
// p 仍然保存旧地址——野指针！

// UAF：使用已释放的内存
printf("%s\n", p);  // 可能正常输出（因为内存还没被重新分配）
                     // 也可能崩溃或输出垃圾数据

// 真实攻击场景：
// 1. free(p) 之后，malloc 分配新对象到同一地址
// 2. 攻击者控制新对象的内容
// 3. 通过野指针 p 访问，实际访问的是攻击者控制的数据
```

**防御：** 释放后立即置空——`free(p); p = NULL;`。虽然不能阻止所有 UAF，但至少让解引用空指针时立即崩溃（SIGSEGV），而非静默地访问错误数据。

#### 3.5.2 空指针解引用

```c
int *p = NULL;
// *p = 42;     // 段错误（SIGSEGV）——访问地址 0x0

// 安全编码：使用前检查
if (p != NULL) {
    *p = 42;    // 安全
}
```

空指针解引用本身是崩溃类漏洞，但在特定条件下可被利用：如果内核中某个空指针可被用户态映射到地址 0 附近（`mmap(NULL, ...)`），攻击者就能控制"解引用空指针"时读取的内容，实现权限提升。这就是 **空指针解引用利用（Null Pointer Dereference Exploitation）**。

#### 3.5.3 类型混淆

```c
void *p = malloc(64);
int *ip = (int*)p;
char *cp = (char*)p;
// 同一块内存，不同类型解释

// 更危险的场景：结构体类型混淆
struct Animal { int type; void (*speak)(); };
struct Dog    { int type; void (*bark)(); };

struct Animal *a = (struct Animal*)malloc(sizeof(struct Animal));
a->type = 1;
a->speak = normal_speak;

// 如果攻击者能控制 a->type，让程序误把 Animal 当 Dog 处理
// Dog 的 bark 指针可能指向完全不同的位置
```

C++ 中的类型混淆更为严重，因为涉及虚函数表。当对象的动态类型与静态类型不一致时，虚函数调用可能跳转到错误的地址。浏览器漏洞（如 V8、SpiderMonkey）中大量利用的就是 C++ 类型混淆。

#### 3.5.4 整数溢出导致的指针错误

```c
// 计算缓冲区大小时整数溢出
size_t count = 0xFFFFFFFF;
size_t size = count * sizeof(int);  // 溢出！结果可能是 0xFFFFFFFC（小值）

int *buf = malloc(size);  // 分配了远小于预期的内存
// 后续按 count 遍历 buf → 堆溢出
```

### 3.6 指针在内存中的布局

理解指针与内存布局的关系，对漏洞利用至关重要。程序运行时，指针分布在各个内存段中：

```text
高地址  +------------------+
        |       栈         |  ← 局部变量中的指针、函数返回地址
        |   (向下增长)      |     栈溢出可覆盖这些指针
        +------------------+
        |                  |
        |       堆         |  ← malloc/new 分配的对象
        |   (向上增长)      |     堆溢出、UAF 发生在这里
        +------------------+
        |     BSS 段       |  ← 未初始化的全局指针（自动置零）
        +------------------+
        |     数据段       |  ← 已初始化的全局函数指针、GOT 表
        +------------------+
        |     代码段       |  ← 函数代码（含函数地址）
低地址  +------------------+
```

- **栈上的指针**：局部变量中的函数指针，被栈缓冲区溢出覆盖后直接劫持控制流
- **堆上的指针**：对象内部的 vptr、回调函数指针，UAF 或堆溢出可篡改
- **数据段/GOT**：全局函数地址表，GOT overwrite 可劫持任意库函数调用
- **代码段**：函数地址本身不可修改（只读），但 ROP 利用的是其中已有的指令片段

### 3.7 调试指针：用 GDB 观察指针行为

实际调试时，GDB 是观察指针状态最直接的工具：

```bash
# 编译时带调试信息
gcc -g -o test test.c

# 启动 GDB
gdb ./test
```

```gdb
# 常用指针调试命令
(gdb) break main          # 在 main 设断点
(gdb) run                 # 运行程序
(gdb) print p             # 打印指针的值（地址）
(gdb) print *p            # 解引用：打印指针指向的值
(gdb) print &x            # 打印变量 x 的地址
(gdb) x/10xw p            # 以十六进制查看 p 指向的 10 个 4 字节数据
(gdb) x/s p               # 把 p 指向的内容当字符串打印
(gdb) info registers      # 查看寄存器（含栈指针 rsp/rbp）
(gdb) info locals         # 查看所有局部变量（含指针）
(gdb) backtrace           # 查看调用栈（栈帧中的返回地址）
```

**安全研究常用技巧：** 使用 `x/i $pc` 查看当前执行的指令，`x/20xg $rsp` 查看栈内容（寻找返回地址、canary 值），`disassemble main` 反汇编函数。

### 3.8 常见误区与纠正

| 误区 | 正确理解 |
|------|---------|
| `free(p)` 后 `p` 自动变为 `NULL` | 不会。`p` 仍然保存旧地址，必须手动 `p = NULL` |
| `sizeof(指针)` 取决于指向的类型 | 不对。`sizeof(任何指针)` 在同平台上固定（64位=8字节） |
| `void*` 可以直接解引用 | 不行。`void` 没有大小，编译器不知道读多少字节 |
| 数组名就是指针 | 不完全对。数组名在 `sizeof` 和 `&` 操作中不退化为指针 |
| 指针加 1 就是地址加 1 | 不对。`p+1` 地址加 `sizeof(*p)` 个字节 |
| `const int *p` 不能修改 `p` | 不对。不能修改的是 `*p`（指向的值），`p` 本身可以改指向 |
| 函数指针和数据指针大小一定相同 | 大多数情况相同，但 C 标准不保证（某些嵌入式架构不同） |
