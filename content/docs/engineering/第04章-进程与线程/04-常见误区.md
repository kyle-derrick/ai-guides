进程与线程编程是系统开发中最容易出错的领域之一。不同于单线程顺序编程的直觉友好性，并发编程的错误往往具有非确定性——同样的代码可能运行百万次才触发一次bug，而且一旦触发，几乎无法通过"再试一次"来复现。

本节系统梳理进程与线程编程中最常见、最危险的误区，每个误区都包含：**错误现象 → 根因分析 → 代码示例 → 修复方案 → 诊断工具**。掌握这些反模式，是写出可靠并发程序的关键一步。

## 误区一：竞态条件——并发编程的头号杀手

### 错误现象

多个线程同时修改共享变量，导致程序出现不可预测的行为：计数器值偏移、数据结构损坏、偶发的空指针异常。症状时有时无，极难复现。

### 根因分析

竞态条件（Race Condition）发生在多个执行流对共享资源的访问没有适当的同步，且至少有一个是写操作时。操作系统对线程调度的不确定性，使得两个线程的指令交错顺序每次都可能不同，导致最终结果取决于"谁先执行"。

竞态条件发生时的指令交错示例：

线程A执行 counter++              线程B执行 counter++
─────────────────────            ─────────────────────
1. 读取 counter (=100)          
                                 2. 读取 counter (=100)
3. counter = 100 + 1 = 101      
                                 4. counter = 100 + 1 = 101
5. 写回 counter (=101)          
                                 6. 写回 counter (=101)

期望结果: counter = 102
实际结果: counter = 101    ← 丢失了一次更新

### 代码示例：典型的数据竞争

```c
#include <pthread.h>
#include <stdio.h>

int counter = 0;  // 共享变量，无任何保护

void *increment(void *arg) {
    for (int i = 0; i < 1000000; i++) {
        counter++;  // 非原子操作：读-改-写三步
    }
    return NULL;
}

int main() {
    pthread_t t1, t2;
    pthread_create(&amp;t1, NULL, increment, NULL);
    pthread_create(&amp;t2, NULL, increment, NULL);
    pthread_join(t1, NULL);
    pthread_join(t2, NULL);
    printf("Expected: 2000000, Actual: %d\n", counter);
    // 典型输出: 1423857, 1789234, ... 每次都不同
    return 0;
}
```

`counter++` 看似一条语句，实际上编译为三条指令：从内存加载到寄存器、寄存器加1、写回内存。两个线程在这三步之间可以任意交错，导致"丢失更新"（Lost Update）。

### 修复方案

```c
#include <pthread.h>
#include <stdio.h>

int counter = 0;
pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER;

void *increment(void *arg) {
    for (int i = 0; i < 1000000; i++) {
        pthread_mutex_lock(&amp;mutex);
        counter++;
        pthread_mutex_unlock(&amp;mutex);
    }
    return NULL;
}
// 输出: Expected: 2000000, Actual: 2000000  ✓
```

对于计数器这类简单场景，也可以使用原子操作（无锁方案）：

```c
#include <stdatomic.h>

atomic_int counter = ATOMIC_VAR_INIT(0);

void *increment(void *arg) {
    for (int i = 0; i < 1000000; i++) {
        atomic_fetch_add(&amp;counter, 1);  // 原子递增
    }
    return NULL;
}
```

### 诊断工具

```bash
# GCC/Clang 的 ThreadSanitizer（最推荐）
gcc -fsanitize=thread -g -o test test.c
./test
# 输出: WARNING: ThreadSanitizer: data race

# Valgrind 的 DRD/Helgrind 工具
valgrind --tool=helgrind ./test
valgrind --tool=drd ./test

# Go 的 race detector
go run -race main.go

# Java 的 -XX:+EnableThreadSMRStatistics（JDK 9+）
java -XX:+EnableThreadSMRStatistics -jar app.jar
```

> **关键要点**：数据竞争是未定义行为（UB），编译器和CPU都可能对其做激进优化。GCC的 `-O2` 下，编译器可能将循环中的 `counter++` 优化到寄存器中，进一步加剧问题。永远不要"测试着看起来没出问题"就认为没有竞态。

## 误区二：死锁——四条件同时满足的致命陷阱

### 错误现象

程序运行一段时间后完全卡住，不再响应任何请求，CPU使用率骤降，但进程不会退出。使用 `gdb` 附加可以看到线程都阻塞在各自的锁上。

### 根因分析

死锁（Deadlock）发生的四个必要条件（Coffman条件）：

1. **互斥**（Mutual Exclusion）：资源一次只能被一个线程持有
2. **占有并等待**（Hold and Wait）：线程持有至少一个资源，同时等待其他资源
3. **不可抢占**（No Preemption）：已获得的资源不能被强制剥夺
4. **循环等待**（Circular Wait）：存在一个线程等待链 A→B→C→A

典型的双锁死锁场景：

线程1: lock(A) → lock(B)     线程2: lock(B) → lock(A)
         │                              │
    持有A，等待B                   持有B，等待A
         │                              │
         └──────── 死锁 ───────────────┘

### 代码示例：银行转账死锁

```c
#include <pthread.h>

pthread_mutex_t account_A = PTHREAD_MUTEX_INITIALIZER;
pthread_mutex_t account_B = PTHREAD_MUTEX_INITIALIZER;

// 线程1: A → B 转账
void *transfer_A_to_B(void *arg) {
    pthread_mutex_lock(&amp;account_A);   // 锁定A
    // 此时线程2可能已锁定了B
    pthread_mutex_lock(&amp;account_B);   // 等待B ← 卡住！
    // 执行转账...
    pthread_mutex_unlock(&amp;account_B);
    pthread_mutex_unlock(&amp;account_A);
    return NULL;
}

// 线程2: B → A 转账
void *transfer_B_to_A(void *arg) {
    pthread_mutex_lock(&amp;account_B);   // 锁定B
    // 此时线程1可能已锁定了A
    pthread_mutex_lock(&amp;account_A);   // 等待A ← 卡住！
    // 执行转账...
    pthread_mutex_unlock(&amp;account_A);
    pthread_mutex_unlock(&amp;account_B);
    return NULL;
}
```

### 修复方案

**方案一：固定加锁顺序（最根本的预防手段）**

所有线程都按相同顺序获取锁（如按账户ID排序）：

```c
// 所有转账操作都先锁ID较小的账户
void transfer(pthread_mutex_t *lock1, pthread_mutex_t *lock2) {
    if (lock1 > lock2) {
        pthread_mutex_t *tmp = lock1;
        lock1 = lock2;
        lock2 = tmp;
    }
    pthread_mutex_lock(lock1);
    pthread_mutex_lock(lock2);
    // 执行操作...
    pthread_mutex_unlock(lock2);
    pthread_mutex_unlock(lock1);
}
```

**方案二：使用 trylock + 超时 + 退避**

```c
#include <errno.h>

int safe_lock(pthread_mutex_t *lock1, pthread_mutex_t *lock2,
              int max_retries) {
    for (int i = 0; i < max_retries; i++) {
        if (pthread_mutex_trylock(lock1) == 0) {
            if (pthread_mutex_trylock(lock2) == 0) {
                return 0;  // 两把锁都获取成功
            }
            pthread_mutex_unlock(lock1);  // 获取第二把失败，释放第一把
        }
        // 指数退避，避免活锁
        usleep((1 << i) * 100);  // 100μs, 200μs, 400μs, ...
    }
    return EBUSY;  // 超时放弃
}
```

**方案三：死锁检测（调试阶段使用）**

```bash
# Linux 内核级死锁检测
sudo echo 1 > /proc/sys/kernel/hung_task_timeout_secs

# GDB 检查线程状态
gdb -p <pid>
(gdb) info threads
(gdb) thread 1
(gdb) bt  # 查看线程1的调用栈
```

### 预防检查表

| 检查项 | 方法 |
|--------|------|
| 是否所有路径都按相同顺序获取锁？ | Code Review + 锁顺序审计 |
| 是否存在嵌套锁获取？ | 用静态分析工具（如 `lockdep`） |
| 锁持有时间是否过短？ | 压测 + 锁持有时间监控 |
| 是否可以用无锁方案替代？ | 评估是否可用原子操作/CAS |

## 误区三：线程泄漏——悄无声息的资源耗尽

### 错误现象

服务器运行一段时间后，线程数持续增长，最终触发系统限制（`ulimit -u`）导致 `pthread_create` 失败，新请求无法处理。使用 `ps -eLf | grep <pid> | wc -l` 可以看到线程数不断上升。

### 根因分析

线程泄漏通常由以下原因导致：

1. **创建了线程但没有 join 或 detach**：线程退出后，其资源（线程ID、内核线程资源）不会被自动回收，变成"僵尸线程"
2. **在循环中创建线程但没有等待完成**：每次都创建新线程而不复用
3. **线程函数中有异常退出路径**：`return` 或 `pthread_exit` 之前没有正确清理

线程泄漏的生命周期：

创建线程 → 执行任务 → 任务完成 → 线程退出
                                    ↓
                        如果没有 join/detach：
                        线程资源永远不释放
                        新的 pthread_create() 会分配新线程
                                    ↓
                        线程数持续累积 → 触发系统限制 → 服务崩溃

### 代码示例：Web服务器的线程泄漏

```c
// 错误代码：每请求创建线程，但从不回收
void handle_connection(int client_fd) {
    pthread_t thread;
    // 创建线程处理请求
    pthread_create(&amp;thread, NULL, process_request, &amp;client_fd);
    // 问题：没有 join 或 detach！
    // 线程退出后变成僵尸线程，资源永不释放
}

// 正确做法一：detach（不需要获取线程退出状态时使用）
void handle_connection_fixed(int client_fd) {
    pthread_t thread;
    pthread_attr_t attr;
    pthread_attr_init(&amp;attr);
    pthread_attr_setdetachstate(&amp;attr, PTHREAD_CREATE_DETACHED);
    pthread_create(&amp;thread, &amp;attr, process_request, &amp;client_fd);
    pthread_attr_destroy(&amp;attr);
}

// 正确做法二：join（需要获取线程退出状态时使用）
void handle_connection_join(int client_fd) {
    pthread_t thread;
    pthread_create(&amp;thread, NULL, process_request, &amp;client_fd);
    pthread_join(thread, NULL);  // 等待线程完成并回收资源
}

// 最佳实践：使用线程池
// 见本章核心技巧部分的线程池实现
```

### 诊断与监控

```bash
# 查看进程的线程数
ps -eLf | grep <pid> | wc -l

# 持续监控线程数变化
while true; do
    echo "$(date): $(ps -eLf | grep <pid> | wc -l) threads"
    sleep 5
done

# 使用 /proc 文件系统
cat /proc/<pid>/status | grep Threads

# 设置线程数告警（超过阈值自动报警）
THRESHOLD=500
THREADS=$(ps -eLf | grep <pid> | wc -l)
[ "$THREADS" -gt "$THRESHOLD" ] &amp;&amp; echo "ALERT: $THREADS threads!"

# 查看线程栈信息
ls /proc/<pid>/task/  # 每个子目录是一个线程
cat /proc/<pid>/task/<tid>/status
```

### 预防措施

```c
// 在程序启动时设置线程数上限
#include <sys/resource.h>

struct rlimit rl;
rl.rlim_cur = 256;   // 软限制：最多256个线程
rl.rlim_max = 512;   // 硬限制
setrlimit(RLIMIT_NPROC, &amp;rl);

// 在线程创建失败时优雅处理
if (pthread_create(&amp;thread, NULL, worker, arg) != 0) {
    perror("pthread_create failed");
    // 记录日志、告警、拒绝新请求，而不是继续崩溃
    log_error("Thread creation failed, thread pool exhausted");
    reject_new_requests();
}
```

## 误区四：虚假唤醒——条件变量的隐藏陷阱

### 错误现象

使用条件变量的生产者-消费者模型中，消费者偶尔从条件变量唤醒后访问到空数据或未初始化数据，导致段错误或逻辑错误。问题只在高并发下偶尔出现。

### 根因分析

条件变量的 `pthread_cond_wait` 可能在**没有被显式 signal/broadcast 的情况下返回**，这就是"虚假唤醒"（Spurious Wakeup）。Linux 的 `futex` 实现中，虚假唤醒是合法行为，可能由以下原因触发：

- 信号处理（被信号中断后恢复执行）
- 内核调度器的实现细节
- 多消费者场景下的竞争

> POSIX 标准明确允许虚假唤醒：*"Spurious wakeups from pthread_cond_wait() may occur."*

### 代码示例

```c
// 错误：用 if 检查条件
pthread_mutex_lock(&amp;mutex);
if (buffer_empty()) {           // ← 如果是虚假唤醒，这里跳过了
    pthread_cond_wait(&amp;cond, &amp;mutex);
}
// 访问 buffer → 可能是空的！段错误！
process(buffer);
pthread_mutex_unlock(&amp;mutex);

// 正确：用 while 循环检查条件
pthread_mutex_lock(&amp;mutex);
while (buffer_empty()) {        // ← 虚假唤醒时会重新检查条件
    pthread_cond_wait(&amp;cond, &amp;mutex);
}
// 走到这里保证 buffer 非空
process(buffer);
pthread_mutex_unlock(&amp;mutex);
```

### 完整的生产者-消费者模板

```c
#include <pthread.h>

#define BUFFER_SIZE 256

typedef struct {
    int data[BUFFER_SIZE];
    int head, tail, count;
    pthread_mutex_t mutex;
    pthread_cond_t not_empty;
    pthread_cond_t not_full;
} bounded_buffer_t;

void buffer_init(bounded_buffer_t *buf) {
    buf->head = buf->tail = buf->count = 0;
    pthread_mutex_init(&amp;buf->mutex, NULL);
    pthread_cond_init(&amp;buf->not_empty, NULL);
    pthread_cond_init(&amp;buf->not_full, NULL);
}

void buffer_put(bounded_buffer_t *buf, int value) {
    pthread_mutex_lock(&amp;buf->mutex);
    while (buf->count >= BUFFER_SIZE) {     // while，不是 if！
        pthread_cond_wait(&amp;buf->not_full, &amp;buf->mutex);
    }
    buf->data[buf->tail] = value;
    buf->tail = (buf->tail + 1) % BUFFER_SIZE;
    buf->count++;
    pthread_cond_signal(&amp;buf->not_empty);   // 通知消费者
    pthread_mutex_unlock(&amp;buf->mutex);
}

int buffer_get(bounded_buffer_t *buf) {
    pthread_mutex_lock(&amp;buf->mutex);
    while (buf->count <= 0) {               // while，不是 if！
        pthread_cond_wait(&amp;buf->not_empty, &amp;buf->mutex);
    }
    int value = buf->data[buf->head];
    buf->head = (buf->head + 1) % BUFFER_SIZE;
    buf->count--;
    pthread_cond_signal(&amp;buf->not_full);    // 通知生产者
    pthread_mutex_unlock(&amp;buf->mutex);
    return value;
}
```

### 记忆口诀

> **条件变量，永远用 while，不要用 if。**
>
> 这是并发编程中最重要的规则之一。无论你多确定"这次一定是被正确 signal 的"，都必须用 `while` 循环。

## 误区五：优先级反转——低优先级线程阻塞高优先级

### 错误现象

高优先级的实时线程被低优先级线程间接阻塞，导致系统响应延迟远超预期。在实时系统中，这可能造成任务超时（deadline miss），后果严重。

### 根因分析

优先级反转（Priority Inversion）的经典场景：

优先级反转时间线：

时间 ──────────────────────────────────────►

线程H（高优先级）:  [等待锁M]──────────────[运行]  ← 被L间接阻塞！
线程M（中优先级）:  ──[运行]──────────────[运行]──  ← 抢占了L
线程L（低优先级）:  [持有锁M]──[被M抢占]────[释放M]  
                         ↑
                    L持有H需要的锁M，但被更高优先级的M抢占了
                    → H等价于被M阻塞（优先级反转）

这种问题在1997年火星探路者号（Mars Pathfinder）任务中真实发生过，导致系统反复重启。

### 修复方案：优先级继承

```c
// 使用 PTHREAD_PRIO_INHERIT 属性创建互斥锁
pthread_mutexattr_t attr;
pthread_mutexattr_init(&amp;attr);
pthread_mutexattr_setprotocol(&amp;attr, PTHREAD_PRIO_INHERIT);

pthread_mutex_t mutex;
pthread_mutex_init(&amp;mutex, &amp;attr);

// 当高优先级线程H等待这把锁时：
// L的临时优先级会被提升到H的优先级
// → L不会被M抢占 → L尽快释放锁 → H尽快获得锁
// → 优先级反转被消除
```

优先级继承修复后的时间线：

线程H（高优先级）:  [等待锁M]────[运行]  
线程M（中优先级）:  ────────────[运行]──     ← 无法抢占已继承高优先级的L
线程L（低优先级）:  [持有锁M]──[释放M]──     ← 优先级被提升到H的级别
                              ↑
                    优先级继承：L临时获得H的优先级
                    → M无法抢占L → L尽快释放锁

### 三种优先级协议对比

| 协议 | 机制 | 优点 | 缺点 |
|------|------|------|------|
| 无保护（原始问题） | 无 | 无 | 高优先级线程可能被无限期阻塞 |
| 优先级继承（PI） | 等待者提升持有者优先级 | 解决基本反转 | 可能发生链式反转 |
| 优先级天花板（PC） | 锁创建时绑定最高使用者优先级 | 防止链式反转 | 需要预先分析，不灵活 |

## 误区六：僵尸进程积累——fork后不wait的代价

### 错误现象

系统中出现大量 `Z` 状态进程，`ps aux | grep Z` 可以看到。僵尸进程本身不占CPU和内存，但占用进程表项。当进程表项耗尽时，系统无法创建新进程，所有 `fork` 调用都会失败。

### 根因分析

进程退出后，内核会保留其 `task_struct` 中的信息（退出码、资源使用统计），等待父进程通过 `wait()/waitpid()` 读取。在父进程调用 `wait()` 之前，子进程处于僵尸状态。如果父进程从来不调用 `wait()`，僵尸进程会一直存在。

```bash
# 查看僵尸进程数量
ps -eo stat | grep -c Z

# 查看系统总进程数限制
cat /proc/sys/kernel/pid_max      # 最大PID值（默认32768）
cat /proc/sys/kernel/threads-max  # 最大线程数

# 查看僵尸进程详情
ps -eo pid,ppid,stat,comm | grep Z
```

### 代码示例与修复

```c
// 错误：fork后不处理子进程退出
void spawn_worker() {
    pid_t pid = fork();
    if (pid == 0) {
        // 子进程
        do_work();
        exit(0);
    }
    // 父进程直接返回，不等待子进程
    // → 子进程退出后变成僵尸
}

// 修复方案一：注册 SIGCHLD 处理函数（推荐，异步回收）
#include <signal.h>

void sigchld_handler(int sig) {
    // 使用 WNOHANG 循环回收所有已退出的子进程
    while (waitpid(-1, NULL, WNOHANG) > 0) {
        // 回收成功
    }
}

void setup_signal_handler() {
    struct sigaction sa;
    sa.sa_handler = sigchld_handler;
    sigemptyset(&amp;sa.sa_mask);
    sa.sa_flags = SA_RESTART | SA_NOCLDSTOP;
    sigaction(SIGCHLD, &amp;sa, NULL);
}

// 修复方案二：忽略 SIGCHLD（Linux特有，最简单）
// 在fork之前调用一次即可：
signal(SIGCHLD, SIG_IGN);
// Linux下子进程退出时内核自动回收，不会产生僵尸

// 修复方案三：显式等待（适合需要获取子进程退出状态的场景）
pid_t pid = fork();
if (pid == 0) {
    do_work();
    exit(42);
} else {
    int status;
    waitpid(pid, &amp;status, 0);  // 阻塞等待
    if (WIFEXITED(status)) {
        printf("Worker exited with code %d\n", WEXITSTATUS(status));
    }
}
```

### 修复方案对比

| 方案 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| `SIGCHLD` + `waitpid` | 需要获取退出状态 | 可靠、可控 | 需要编写信号处理函数 |
| `signal(SIGCHLD, SIG_IGN)` | 不关心退出状态 | 最简单（一行代码） | Linux特有，无法获取退出状态 |
| 显式 `waitpid` | 需要同步等待子进程 | 可靠、可获取状态 | 阻塞父进程 |

## 误区七：fork与多线程混用——最常见的隐蔽炸弹

### 错误现象

在一个已经创建了多线程的进程中调用 `fork()`，子进程中的线程同步原语（互斥锁）处于不一致状态，导致死锁或数据损坏。问题可能在子进程中永远不会被触发（如果子进程立即exec），但在fork后继续执行的场景下是致命的。

### 根因分析

`fork()` 只会复制**调用fork的那个线程**，其他线程不会被复制。但是，所有线程持有的锁的状态会被原样保留——包括那些处于"已锁定"状态的互斥锁。在子进程中，这些锁永远不会被解锁（因为持有它们的线程不存在了），导致任何试图获取这些锁的操作都会死锁。

fork与多线程的问题：

父进程:                    fork()后子进程:
┌──────────────────┐      ┌──────────────────┐
│ 主线程: 持有锁M   │  ──► │ 唯一线程: 想获取M  │ → 死锁！
│ 工作线程1: 运行中  │      │ (工作线程1不存在)  │
│ 工作线程2: 持有锁N │      │                   │
└──────────────────┘      └──────────────────┘

### 代码示例

```c
#include <pthread.h>
#include <stdio.h>
#include <unistd.h>
#include <stdlib.h>
#include <signal.h>

pthread_mutex_t global_lock = PTHREAD_MUTEX_INITIALIZER;

void *worker(void *arg) {
    pthread_mutex_lock(&amp;global_lock);
    // 长时间操作...
    sleep(10);  // 在此期间主线程可能fork
    pthread_mutex_unlock(&amp;global_lock);
    return NULL;
}

int main() {
    pthread_t t1;
    pthread_create(&amp;t1, NULL, worker, NULL);
    sleep(1);  // 等待子线程获取锁

    pid_t pid = fork();
    if (pid == 0) {
        // 子进程：只有一个线程（调用fork的线程）
        // 但 global_lock 可能已被工作线程锁定
        pthread_mutex_lock(&amp;global_lock);  // ← 可能永远阻塞！
        printf("This line may never print\n");
        pthread_mutex_unlock(&amp;global_lock);
        exit(0);
    }

    pthread_join(t1, NULL);
    wait(NULL);
    return 0;
}
```

### 修复方案

**方案一：fork前加锁，fork后在子进程中解锁（不推荐，容易出错）**

```c
pthread_mutex_lock(&amp;global_lock);
pid_t pid = fork();
if (pid == 0) {
    pthread_mutex_unlock(&amp;global_lock);
    // ...
}
pthread_mutex_unlock(&amp;global_lock);
```

**方案二：使用 pthread_atfork 注册 fork 处理函数（推荐）**

```c
pthread_mutex_t lock1 = PTHREAD_MUTEX_INITIALIZER;
pthread_mutex_t lock2 = PTHREAD_MUTEX_INITIALIZER;

void prepare(void) {
    // fork 前：获取所有需要保护的锁
    pthread_mutex_lock(&amp;lock1);
    pthread_mutex_lock(&amp;lock2);
}

void parent(void) {
    // fork 后（父进程）：释放锁
    pthread_mutex_unlock(&amp;lock2);
    pthread_mutex_unlock(&amp;lock1);
}

void child(void) {
    // fork 后（子进程）：释放锁
    pthread_mutex_unlock(&amp;lock2);
    pthread_mutex_unlock(&amp;lock1);
}

int main() {
    pthread_atfork(prepare, parent, child);
    // ...
}
```

**方案三：fork后在子进程中立即 exec（最安全）**

```c
pid_t pid = fork();
if (pid == 0) {
    // 子进程：立即替换为新程序
    // exec会重置所有锁和同步状态
    execl("/bin/sh", "sh", "-c", "some_command", NULL);
    perror("exec");
    exit(1);
}
```

> **设计原则**：在多线程程序中，`fork()` 之后应该只做两件事：调用 `exec()` 或调用 `exit()`。不要在子进程中执行其他逻辑。

## 误区八：上下文切换抖动——线程过多的反效果

### 错误现象

创建了大量线程（远超CPU核心数），期望获得线性性能提升，但实际性能反而下降。CPU使用率很高（`top` 显示90%+），但有效计算量很低——大量时间花在了线程切换上。

### 根因分析

每次上下文切换的直接开销约为 3-15μs（寄存器保存恢复、TLB刷新），但更大的间接开销来自**CPU缓存失效**。当线程A被切走、线程B开始运行时，B的代码和数据不在L1/L2缓存中，需要从主存重新加载，这个过程可能消耗数十到数百微秒。

上下文切换的隐藏成本：

直接成本（可见）：
├── 保存寄存器: ~1μs
├── 调度器决策: ~1μs  
└── 恢复寄存器: ~1μs
    小计: ~3μs

间接成本（隐藏，但更大）：
├── L1缓存失效: ~1-5μs  × 核心数
├── L2缓存失效: ~5-20μs × 核心数
├── TLB刷新: ~5-50μs（无PCID时）
└── 缓存行逐出: ~10-100μs
    小计: ~20-170μs  ← 这才是大头！

当线程数 >> CPU核心数时:
CPU时间 = 有效计算 + 上下文切换开销
        = 有效计算 + (线程数/核心数) × 切换开销
        → 线程越多，有效计算占比越低

### 代码示例：线程过多导致性能下降

```c
#include <pthread.h>
#include <stdio.h>
#include <time.h>

#define WORK_PER_THREAD 10000000

void *cpu_work(void *arg) {
    volatile long sum = 0;
    for (long i = 0; i < WORK_PER_THREAD; i++) {
        sum += i;
    }
    return NULL;
}

// 测试不同线程数的性能
// 假设8核CPU：
// 8个线程:  ~1.0x  (最佳，每个核心一个线程)
// 16个线程: ~0.9x  (线程切换开始影响)
// 64个线程: ~0.5x  (大量切换，性能减半)
// 256个线程: ~0.2x (系统基本在做切换，不做计算)
```

### 线程数选择指南

| 工作负载类型 | 推荐线程数 | 原因 |
|------------|-----------|------|
| CPU密集型（纯计算） | CPU核心数 | 没有IO等待，更多线程只会增加切换 |
| IO密集型（网络/磁盘） | 核心数 × (1 + IO等待比例/CPU计算比例) | IO等待期间CPU空闲，可以服务其他线程 |
| 混合型 | 核心数 × 2 ~ 4 | 适度超额订阅，平衡切换和利用率 |

```bash
# 查看CPU核心数
nproc                          # 逻辑核心数
lscpu | grep "Core(s) per socket"  # 物理核心数

# 查看当前上下文切换频率
vmstat 1
# 关注 cs（context switch）列
# 正常: 数千/秒
# 抖动: 数十万/秒以上

# 使用 pidstat 查看单个进程的上下文切换
pidstat -w -p <pid> 1
# cswch/s: 自愿切换（IO等待等主动让出）
# nvcswch/s: 非自愿切换（时间片用完被抢占）
# 非自愿切换过高说明线程过多或CPU不足
```

### 修复方案

```c
// 1. 使用线程池，限制并发线程数
#define MAX_WORKERS 8  // 等于CPU核心数

pthread_t workers[MAX_WORKERS];
for (int i = 0; i < MAX_WORKERS; i++) {
    pthread_create(&amp;workers[i], NULL, worker_loop, NULL);
}

// 2. 对于IO密集型任务，使用事件驱动或协程
// Go 的 goroutine / Kotlin 的 coroutine / Java 的 Virtual Thread
// 比系统线程轻量100-1000倍

// 3. 设置CPU亲和性，减少缓存失效
cpu_set_t cpuset;
CPU_ZERO(&amp;cpuset);
CPU_SET(i % num_cpus, &amp;cpuset);  // 绑定到固定核心
pthread_setaffinity_np(thread, sizeof(cpuset), &amp;cpuset);
```

## 误区九：信号与多线程的交互——最容易被忽视的雷区

### 错误现象

多线程程序收到信号后行为异常：程序突然终止（没有被 catch）、在不该执行的地方执行了信号处理函数、或者信号处理函数中的操作导致死锁。

### 根因分析

在多线程程序中，信号的处理规则与单线程程序有本质区别：

1. **信号发送目标**：发送给进程的信号只会被**一个线程**处理，具体是哪个线程不确定
2. **信号处理函数**：是进程级别的，所有线程共享同一个信号处理函数
3. **异步信号安全**：信号处理函数中只能调用异步信号安全（async-signal-safe）的函数

多线程信号处理的陷阱：

信号 SIGINT 到达
    ↓
内核选择线程T2来处理（不是主线程！）
    ↓
T2 正在执行 pthread_mutex_lock()
    ↓
信号处理函数被调用，其中又调用了 printf()（非async-signal-safe）
    ↓
printf 内部使用了静态缓冲区和锁 → 可能死锁或数据损坏

### 代码示例与修复

```c
#include <signal.h>
#include <stdio.h>
#include <unistd.h>
#include <string.h>

// 错误：信号处理函数中调用非async-signal-safe函数
void bad_handler(int sig) {
    printf("Caught signal %d\n", sig);     // ← 非安全！
    fflush(stdout);                          // ← 非安全！
    // malloc/free, printf, pthread_mutex_lock 等都不是async-signal-safe
}

// 正确：使用 sig_atomic_t 和 write（async-signal-safe）
volatile sig_atomic_t got_signal = 0;

void good_handler(int sig) {
    got_signal = 1;  // 只设置标志，不做复杂操作
    // write() 是 async-signal-safe 的
    const char msg[] = "Signal received\n";
    write(STDERR_FILENO, msg, sizeof(msg) - 1);
}

int main() {
    struct sigaction sa;
    sa.sa_handler = good_handler;
    sigemptyset(&amp;sa.sa_mask);
    sa.sa_flags = 0;  // 不使用 SA_RESTART（根据需要决定）
    sigaction(SIGINT, &amp;sa, NULL);

    // 在工作线程中检查信号标志
    while (!got_signal) {
        do_work();
    }
    cleanup_and_exit();
    return 0;
}
```

### 多线程信号最佳实践

| 场景 | 推荐方案 |
|------|----------|
| 需要通知线程退出 | 使用 pipe + `pthread_cancel` 或 atomic flag，不用信号 |
| 需要定时器 | 使用 `timer_create` + `SIGEV_THREAD`，避免信号竞争 |
| 必须使用信号 | 用 `sigprocmask` 统一在线程启动时屏蔽，由专用线程处理 |
| 处理 SIGSEGV 等 | 使用 `sigaltstack` 防止栈溢出时信号处理函数也栈溢出 |

## 误区十：锁的使用粒度不当——过粗或过细都有害

### 错误现象

**粒度过粗**：一把大锁保护所有数据，多线程退化为串行执行，CPU利用率很低，性能不随线程数增长。

**粒度过细**：大量的细粒度锁，代码复杂度飙升，频繁的加锁解锁本身成为性能瓶颈，且极易引入死锁。

### 根因分析

锁粒度的权衡：

粗粒度锁（一把大锁）:
┌─────────────────────────────┐
│  整个数据结构用一把锁保护     │
│  线程1: [lock][操作A][unlock]│
│  线程2:       [lock][操作B][unlock]│  ← 串行化
│  线程3:                  [lock][操作C][unlock]│
└─────────────────────────────┘
  优点: 简单、不会死锁
  缺点: 并发度为零

细粒度锁（每字段一把锁）:
┌─────────────────────────────┐
│  field_a (lock_a)            │
│  field_b (lock_b)            │
│  field_c (lock_c)            │
│  线程1: [lock_a][操作A][unlock_a]│
│  线程2: [lock_b][操作B][unlock_b]│  ← 真正并发
│  线程3: [lock_c][操作C][unlock_c]│
└─────────────────────────────┘
  优点: 高并发
  缺点: 复杂、可能死锁、维护困难

### 修复方案：分层锁策略

```c
// 示例：并发哈希表的分段锁设计
#define NUM_SEGMENTS 16

typedef struct {
    pthread_mutex_t lock;
    entry_t *head;
} segment_t;

typedef struct {
    segment_t segments[NUM_SEGMENTS];
} concurrent_map_t;

// 哈希到对应的分段，只锁该分段
void map_put(concurrent_map_t *map, int key, int value) {
    int seg_idx = hash(key) % NUM_SEGMENTS;
    segment_t *seg = &amp;map->segments[seg_idx];
    
    pthread_mutex_lock(&amp;seg->lock);
    // 只锁了 1/16 的数据，其他15个分段可以并行访问
    insert_entry(seg, key, value);
    pthread_mutex_unlock(&amp;seg->lock);
}

// 分段数选择: 通常取 CPU核心数 × 4，确保锁竞争概率低
// Java 的 ConcurrentHashMap 默认 16 个分段（JDK7）/ CAS + synchronized（JDK8+）
```

### 锁粒度选择指南

| 场景 | 推荐策略 |
|------|----------|
| 共享计数器（如统计） | 原子操作（`atomic_*`），不用锁 |
| 共享配置数据（读多写少） | 读写锁（`rwlock`）或 RCU |
| 哈希表/链表 | 分段锁或无锁数据结构 |
| 全局状态管理 | 单一互斥锁 + 简化状态转移 |
| 金融交易系统 | 细粒度锁 + 死锁检测 |

## 误区十一：线程栈溢出——默认栈大小的隐藏限制

### 错误现象

程序在处理少量请求时正常，但在高并发或大数据量时突然崩溃（`SIGSEGV`），`dmesg` 中出现 "out of memory" 或 "stack overflow" 信息。

### 根因分析

Linux 上每个线程的默认栈大小通常是 8MB（可通过 `ulimit -s` 查看）。对于创建大量线程的场景，栈内存可能成为瓶颈：

栈内存消耗分析：

100个线程 × 8MB默认栈 = 800MB 仅用于栈！
加上每个线程的内核数据结构（约1.7KB task_struct + 内核栈16KB）
实际内存消耗可能超过 1GB，而真正需要栈空间的可能只有几百字节

更危险的是：如果函数递归层次太深或局部数组太大，
可能在 8MB 栈内就发生溢出

### 代码示例：栈溢出的典型场景

```c
#include <pthread.h>

// 错误：大数组放在栈上
void process_data() {
    char buffer[4 * 1024 * 1024];  // 4MB局部变量！
    // 如果多个函数都有类似的栈分配，8MB栈很快用尽
    // → SIGSEGV
}

// 修复方案一：动态分配
void process_data_fixed() {
    char *buffer = malloc(4 * 1024 * 1024);
    if (!buffer) return;
    // 使用 buffer...
    free(buffer);
}

// 修复方案二：创建线程时指定小栈
void create_light_thread() {
    pthread_t thread;
    pthread_attr_t attr;
    pthread_attr_init(&amp;attr);
    pthread_attr_setstacksize(&amp;attr, 256 * 1024);  // 256KB足够大多数场景
    pthread_create(&amp;thread, &amp;attr, worker, NULL);
    pthread_attr_destroy(&amp;attr);
}

// 修复方案三：使用 mmap 分配的栈（Go goroutine的方式）
// 初始2KB，按需增长到最大1GB
```

### 栈大小调优参考

| 场景 | 建议栈大小 | 说明 |
|------|-----------|------|
| 简单工作线程 | 128KB-256KB | 大多数请求处理不需要大栈 |
| 递归算法（如深度优先搜索） | 4MB-8MB | 根据递归深度调整 |
| 协程/goroutine | 2KB起，动态增长 | 运行时管理，用户无需关心 |
| 系统默认 | 8MB | 浪费内存，不适合大量线程 |

```bash
# 查看当前线程栈大小限制
ulimit -s     # 默认栈大小（KB）
ulimit -a     # 所有资源限制

# 查看指定进程的线程栈信息
cat /proc/<pid>/maps | grep stack
# 输出示例:
# 7f1234000000-7f1234800000 rw-p 00000000 00:00 0  [stack:12345]
# 8MB = 0x800000 bytes

# 检测栈使用量
# 使用 AddressSanitizer
gcc -fsanitize=address -g -o test test.c
```

## 误区十二：无锁编程的正确性陷阱——CAS不是万能药

### 错误现象

使用 CAS（Compare-And-Swap）等原子操作实现"无锁"数据结构，但在高并发下出现数据不一致、ABA问题或活锁。调试极其困难，因为问题只在特定的指令交错下出现。

### 根因分析

无锁编程看似简单（不需要锁），但实际上需要处理多个微妙的问题：

**ABA问题**：CAS只比较值，不比较"是否被修改过"。如果值从A变为B再变回A，CAS会认为"没有被修改过"，但中间状态可能已经破坏了不变量。

ABA问题示例：

线程1: 读取栈顶 A → 准备 CAS(A, C)
       （被调度器挂起）
线程2: pop A → push B → pop B → push A  （栈顶又变成了A）
线程1: CAS(A, C) 成功！但栈的内部状态已经不一致了
       → 栈中可能还有指向已释放内存的指针

**活锁**：CAS失败后重试，但如果竞争线程持续成功，重试线程可能永远无法成功（虽然概率低，但理论上可能）。

### 代码示例：无锁栈的ABA问题

```c
#include <stdatomic.h>
#include <stdlib.h>

typedef struct Node {
    int data;
    struct Node *next;
} Node;

typedef struct {
    _Atomic(Node *) top;
} LockFreeStack;

// 错误的无锁push（ABA问题）
void push_wrong(LockFreeStack *stack, int value) {
    Node *new_node = malloc(sizeof(Node));
    new_node->data = value;
    
    Node *old_top;
    do {
        old_top = atomic_load(&amp;stack->top);
        new_node->next = old_top;
    } while (!atomic_compare_exchange_weak(&amp;stack->top, &amp;old_top, new_node));
    // 如果在 CAS 前，old_top 被 pop 后又 push 回来（ABA），
    // CAS 会成功但 new_node->next 指向错误的节点
}

// 修复方案一：使用带版本号的双字CAS（DCAS）
typedef struct {
    Node *ptr;
    uintptr_t version;  // 单调递增的版本号
} TaggedPointer;

// 修复方案二：使用 hazard pointer 或 epoch-based 回收
// 防止已释放的内存被重新分配

// 修复方案三：使用RCU（Read-Copy-Update），Linux内核中广泛使用
```

### 无锁编程检查表

| 检查项 | 说明 |
|--------|------|
| CAS失败时是否正确重试？ | 必须用循环，且每次重试都重新读取最新值 |
| 是否存在ABA问题？ | 值可能从A→B→A的场景，需要用版本号或hazard pointer |
| 内存回收是否安全？ | 被替换的节点可能还有其他线程在读取 |
| 是否有内存序要求？ | `memory_order_relaxed` 可能导致其他线程看到不一致的状态 |
| 是否真正比锁快？ | 小数据结构+低竞争时，锁可能更快 |

> **务实建议**：除非你在写内核代码或极高性能的库，否则优先使用锁。锁是经过数十年验证的正确抽象，现代锁实现（如 futex）在无竞争时几乎没有开销。无锁编程的正确性验证成本远高于锁。

## 误区总结与自检清单

### 误区速查表

| 误区 | 核心问题 | 最简修复 |
|------|----------|----------|
| 竞态条件 | 多线程无保护访问共享数据 | 用 mutex 或 atomic 保护 |
| 死锁 | 两把锁以不同顺序获取 | 固定加锁顺序 |
| 线程泄漏 | 创建线程后不 join/detach | 使用线程池或 detach |
| 虚假唤醒 | 条件变量用 if 检查 | 永远用 while 循环 |
| 优先级反转 | 低优先级线程持锁阻塞高优先级 | 优先级继承 mutex |
| 僵尸进程 | fork后不wait子进程 | SIGCHLD处理或SIG_IGN |
| fork+多线程 | fork只复制一个线程 | fork后立即exec |
| 上下文切换抖动 | 线程数远超核心数 | 用线程池限制并发数 |
| 信号+多线程 | 信号处理函数中调用非安全函数 | 只设标志，用write |
| 锁粒度不当 | 太粗串行、太细复杂 | 分段锁或读写锁 |
| 线程栈溢出 | 默认8MB不够用 | 控制栈大小，大对象放堆 |
| 无锁CAS | ABA问题和内存回收 | 用hazard pointer或干脆用锁 |

### 编码前自检

□ 共享变量是否都有同步保护？
□ 加锁顺序在所有代码路径中是否一致？
□ 条件变量等待是否使用 while 循环？
□ 线程是否都被 join 或 detach？
□ fork是否在多线程环境中调用？如果是，是否立即exec？
□ 信号处理函数是否只使用 async-signal-safe 函数？
□ 线程栈大小是否匹配实际需求？
□ 线程数是否限制在合理范围内？
□ 锁持有时间是否尽量短？
□ 是否考虑了优先级继承（实时场景）？

### 运行时监控清单

```bash
# 持续监控脚本（可集成到 Prometheus/Grafana）
#!/bin/bash
PID=$1

while true; do
    # 线程数
    THREADS=$(ls /proc/$PID/task | wc -l)
    
    # 上下文切换率
    CS=$(grep ctxt /proc/$PID/status | awk '{print $2}')
    
    # 文件描述符数
    FDS=$(ls /proc/$PID/fd | wc -l)
    
    # 僵尸子进程数
    ZOMBIES=$(ps --ppid $PID -o stat= | grep -c Z)
    
    echo "$(date) threads=$THREADS cs=$CS fds=$FDS zombies=$ZOMBIES"
    sleep 5
done
```

掌握这些误区不是为了记住"不要做什么"，而是为了理解**为什么**并发编程如此容易出错。每一个反模式背后，都是操作系统对资源管理、调度策略和同步机制的深层设计决策。理解了原理，才能在遇到新问题时做出正确的判断。
