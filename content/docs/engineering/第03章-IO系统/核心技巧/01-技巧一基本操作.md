# 技巧一：IO操作基本模式

> 选择正确的IO模式，等于选对了优化的起点。IO模式不是"能跑就行"的实现细节，而是决定系统并发上限、延迟下限和CPU效率的根本架构选择。

## 本技巧定位

在 [理论基础](../理论基础/01-一什么是IO系统.md) 中，我们从硬件层面理解了CPU、内存与外部设备之间的速度鸿沟——NVMe SSD的延迟是L1 Cache的8万倍，HDD的延迟是NVMe的500倍。在 [技巧全景总览](../核心技巧/_index.md) 中，我们建立了IO优化的方法论框架。本技巧聚焦于框架中的"道"——**IO操作模式的选择**，这是所有优化手段的根基。

IO操作模式回答的核心问题是：**当应用程序发起一次IO请求后，CPU在等待IO完成期间应该做什么？** 是傻等（阻塞）、反复问（轮询）、坐等叫号（多路复用）、还是留下电话号码等回电（异步）？不同的答案对应不同的模式，适用于不同的场景。

---

## 一、六种IO模式的本质差异

Linux系统提供了从简单到复杂的六种IO模式，它们的核心区别在于**系统调用行为**和**CPU参与程度**。

### 1.1 从餐厅点餐理解IO模式

用一个生活化的类比建立直觉：

场景：你在餐厅点了一道菜，需要等后厨做好。

阻塞IO（Blocking IO）：
  你点了菜，站在出餐口一动不动等着。
  做好了你才离开。
  → 简单直接，但你什么都干不了。

非阻塞IO（Non-blocking IO）：
  你点了菜，每隔30秒去问一次"好了没？"
  没好就回来继续玩手机，过一会儿再去问。
  → 你可以做别的事，但反复问很累。

select/poll（IO多路复用，水平触发）：
  你点了3道菜，坐在座位上等。
  服务员一次告诉你哪几道菜好了。
  但没好的菜下次还要继续等。
  → 可以同时等多道菜，但每次都要重新注册。

epoll（IO多路复用，就绪通知）：
  你点了100道菜，服务员只在某道菜做好时主动通知你。
  不需要你反复询问。
  → 高效，适合大量并发请求。

signal-driven IO（信号驱动IO）：
  你点了菜，留下手机号。
  菜好了厨房发短信通知你，你直接去取。
  → 被动通知，但信号本身有开销。

io_uring（真正的异步IO）：
  你把10道菜的订单一次性交给前台，前台统一处理。
  所有菜做好后一起通知你。
  → 批量提交，零轮询开销，目前最强。

### 1.2 系统调用行为对比

| 模式 | 系统调用方式 | 等待期间CPU行为 | 返回时机 | 并发能力 |
|------|-------------|----------------|----------|----------|
| 阻塞IO | read/write | 进程挂起（睡眠） | 数据就绪后 | 单任务 |
| 非阻塞IO | read(O_NONBLOCK) | 立即返回EAGAIN，应用自行轮询 | 立即 | 单任务轮询 |
| select | select() | 进程阻塞在select调用上 | 至少一个fd就绪 | fd<1024 |
| poll | poll() | 进程阻塞在poll调用上 | 至少一个fd就绪 | 无硬上限 |
| epoll | epoll_wait() | 进程阻塞在epoll_wait上 | 就绪事件加入就绪链表 | 百万级 |
| io_uring | 共享内存SQ/CQ | 完全不阻塞（轮询模式） | 内核写入CQ | 百万级 |

### 1.3 关键概念澄清：同步 vs 异步

初学者最容易混淆的概念。判断标准只有一个：**应用程序是否需要自己去检查IO是否完成**。

同步IO：
  应用程序主动发起IO → 等待/轮询 → 自己取回结果
  （阻塞IO、非阻塞IO、select/poll/epoll 都是同步IO）

异步IO：
  应用程序发起IO → 内核完成后自动通知/回调
  （只有 io_uring、POSIX AIO 算真正的异步IO）

一个常见的错误认知是"epoll是异步的"。实际上epoll只是解决了"同时监控多个IO"的问题，每次IO操作本身仍然是同步的——你仍然需要调用`read()`去取数据，`read()`仍然可能阻塞或返回EAGAIN。真正的异步IO是：你提交一个`read`请求后，内核在数据就绪后**直接把数据放到你指定的缓冲区**，你根本不需要再调用`read()`。

## 二、内核视角：每种模式到底发生了什么

理解IO模式的关键在于理解**系统调用**在内核中到底做了什么。我们以`read()`系统调用为例，分析每种模式的内核路径。

### 2.1 阻塞IO的内核路径

用户调用 read(fd, buf, count)
    ↓
内核 sys_read()
    ↓
VFS层: 检查文件描述符有效性
    ↓
文件系统层: 查找数据所在磁盘块
    ├── 页缓存命中 → 直接拷贝到用户空间 → 返回 ✓
    └── 页缓存未命中 → 构造bio请求 → 提交到块层
                                          ↓
                              数据从磁盘读入页缓存
                                          ↓
                              拷贝到用户空间 → 返回 ✓

关键点：在"数据从磁盘读入页缓存"期间，
       进程被标记为 TASK_INTERRUPTIBLE，从运行队列移除
       CPU调度器会切换到其他进程执行
       当IO完成后，中断处理程序将进程唤醒

**阻塞IO的优缺点：**
- 优点：实现最简单；内核自动管理等待/唤醒，无额外CPU开销
- 缺点：每个阻塞的线程都占用一个OS线程（默认栈大小8MB），数千并发就需要数千线程，上下文切换开销成为瓶颈

### 2.2 非阻塞IO的内核路径

用户调用 read(fd, buf, count)   // fd以 O_NONBLOCK 打开
    ↓
内核 sys_read()
    ↓
检查数据是否就绪
    ├── 数据就绪 → 拷贝到用户空间 → 返回字节数 ✓
    └── 数据未就绪 → 立即返回 -1, errno = EAGAIN ✓
                       （不阻塞，不睡眠）

应用层需要自行循环调用 read()，直到返回正值

**非阻塞IO的优缺点：**
- 优点：进程永远不会被阻塞，始终可以做其他事
- 缺点：应用必须不断轮询（busy-wait），大量CPU时间浪费在无效的系统调用上

非阻塞IO极少单独使用，它通常配合select/poll/epoll使用——将fd设置为非阻塞后，由epoll通知你哪个fd就绪了，你再用非阻塞read去取数据。

### 2.3 select的内核路径

用户调用 select(n, &read_fds, &write_fds, &except_fds, &timeout)
    ↓
内核 sys_select()
    ↓
1. 将用户传入的fd集合从用户空间拷贝到内核空间（每次调用都拷贝！）
    ↓
2. 遍历所有被监控的fd（O(n)）：
   对每个fd：
     检查其内核数据结构的就绪状态
     如果就绪 → 标记到输出fd_set中
    ↓
3. 如果没有任何fd就绪且未超时 → 进程睡眠
    ↓
4. 当某个fd就绪时 → 唤醒进程
    ↓
5. 再次遍历所有fd（O(n)），找出真正就绪的
    ↓
6. 将结果fd_set从内核空间拷贝回用户空间
    ↓
7. 用户遍历fd_set，对就绪的fd调用read/write

**select的性能瓶颈：**
- 每次调用都要拷贝fd集合（用户态→内核态→用户态），3次内存拷贝
- 内核需要O(n)遍历所有fd，即使只有1个fd就绪
- fd数量有硬上限`FD_SETSIZE`（通常1024）

### 2.4 poll的内核路径

poll与select在内核中的执行路径几乎相同，区别仅在于数据结构：

select使用fd_set（位图）：FD_SET(fd, &fds) → fd_set是固定大小的位数组
poll使用pollfd数组：     pfd[i].fd = fd; pfd[i].events = POLLIN;

两者在内核中都是O(n)遍历，都有相同的性能瓶颈
poll只是突破了FD_SETSIZE的1024限制

### 2.5 epoll的内核路径

epoll的设计彻底改变了性能特征：

// 第一步：创建epoll实例（仅一次）
epfd = epoll_create1(0)
  → 内核创建 eventpoll 结构体
  → 包含一个红黑树（存储所有监控的fd）和一个就绪链表

// 第二步：注册fd（仅一次，后续不变）
epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &ev)
  → 将fd插入红黑树 O(log n)
  → 为该fd注册回调函数：当fd就绪时，回调将fd加入就绪链表

// 第三步：等待事件（可重复调用）
n = epoll_wait(epfd, events, maxevents, timeout)
  → 检查就绪链表是否非空
  → 非空 → 直接取出就绪事件返回 O(1)
  → 为空 → 进程睡眠，等待回调唤醒

**epoll的性能优势本质：**

select/poll: 每次调用 → 遍历全部fd → 返回就绪的    O(n)
epoll:       注册一次 → 就绪时回调加入链表 → 直接取   O(1)

形象比喻：
select: 你去快递站，工作人员翻遍所有货架找你的包裹
epoll:  你的包裹到了，工作人员直接放在"待取"架上，你去拿就行

### 2.6 io_uring的内核路径

io_uring是Linux IO的范式级革新，通过共享内存消除了系统调用开销：

用户空间                         内核空间
┌──────────────┐                ┌──────────────┐
│ 应用程序      │                │ io_uring内核  │
│              │                │              │
│ 1.写SQE到SQ  │ ──共享内存──→  │ 2.轮询SQ取请求│
│   (无系统调用)│                │   (内核轮询)  │
│              │                │              │
│ 3.读取CQ结果  │ ←──共享内存──  │ 4.完成写CQ   │
│   (无系统调用)│                │              │
└──────────────┘                └──────────────┘

SQ = Submission Queue（提交队列）：应用写入IO请求
CQ = Completion Queue（完成队列）：内核写入IO结果
SQE = Submission Queue Entry（单个提交项）
CQE = Completion Queue Entry（单个完成项）

关键创新：SQ和CQ都是内核和用户空间共享的内存区域
  → 提交请求：只需在SQ中写入一个条目，无需系统调用
  → 获取结果：只需读取CQ中的条目，无需系统调用
  → 批量操作：一次系统调用可以提交/收割多个请求

**io_uring的三种工作模式：**

| 模式 | 提交方式 | 完成通知 | 延迟 | CPU开销 | 适用场景 |
|------|---------|---------|------|---------|---------|
| 中断模式 | io_uring_enter()系统调用 | 中断 | ~20μs | 低 | 通用场景 |
| SQ轮询 | 内存写入SQ | 应用轮询CQ | ~5μs | 中 | 高性能网络 |
| 内核轮询(SQPOLL) | 内存写入SQ | 内核线程轮询SQ | ~2μs | 高 | 超低延迟存储 |

## 三、实战代码示例

### 3.1 C语言：阻塞IO — 文件逐块读取

```c
#include <fcntl.h>
#include <unistd.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <errno.h>
#include <time.h>

/**
 * 阻塞IO示例：逐块读取文件
 * 每次read()在数据就绪前不返回
 * 适用于简单的文件处理场景
 */
int main(int argc, char *argv[]) {
    const char *path = argc > 1 ? argv[1] : "/tmp/testfile";
    char buf[4096];
    ssize_t total = 0, n;
    int fd;

    // 打开文件
    fd = open(path, O_RDONLY);
    if (fd < 0) {
        perror("open");
        return 1;
    }

    // 阻塞读取 — 每次read在数据就绪前不返回
    struct timespec start, end;
    clock_gettime(CLOCK_MONOTONIC, &amp;start);

    while ((n = read(fd, buf, sizeof(buf))) > 0) {
        total += n;
        // 实际应用中在这里处理数据
        // process_data(buf, n);
    }

    clock_gettime(CLOCK_MONOTONIC, &amp;end);

    if (n < 0) {
        perror("read");
        close(fd);
        return 1;
    }

    double ms = (end.tv_sec - start.tv_sec) * 1000.0 +
                (end.tv_nsec - start.tv_nsec) / 1e6;
    printf("读取 %zd 字节，耗时 %.1f ms (%.1f MB/s)\n",
           total, ms, total / 1024.0 / 1024.0 / (ms / 1000));

    close(fd);
    return 0;
}
```

### 3.2 C语言：非阻塞IO — 手动轮询

```c
#include <fcntl.h>
#include <unistd.h>
#include <stdio.h>
#include <errno.h>
#include <time.h>

/**
 * 非阻塞IO示例：设置O_NONBLOCK后手动轮询
 * read()在数据未就绪时立即返回EAGAIN
 * 应用需要自行处理重试逻辑
 */
int main(void) {
    int fds[2];
    // 创建管道对（用于模拟IO就绪场景）
    if (pipe(fds) < 0) { perror("pipe"); return 1; }

    // 设置非阻塞
    fcntl(fds[0], F_SETFL, O_NONBLOCK);

    // 管道为空，读取应返回 EAGAIN
    char buf[1024];
    ssize_t n = read(fds[0], buf, sizeof(buf));
    if (n < 0 &amp;&amp; errno == EAGAIN) {
        printf("数据未就绪，read()立即返回 EAGAIN\n");
    }

    // 写入数据使管道可读
    const char *msg = "Hello from non-blocking IO!";
    write(fds[1], msg, strlen(msg));

    // 现在可以读取了
    n = read(fds[0], buf, sizeof(buf));
    if (n > 0) {
        buf[n] = '\0';
        printf("读取到: %s\n", buf);
    }

    // 典型的非阻塞轮询模式（生产环境中应避免busy-wait）
    /*
    while (1) {
        n = read(fd, buf, sizeof(buf));
        if (n > 0) {
            process_data(buf, n);
        } else if (n < 0 &amp;&amp; errno == EAGAIN) {
            // 数据还没好，做点别的事
            do_other_work();
        } else {
            perror("read");
            break;
        }
    }
    */

    close(fds[0]);
    close(fds[1]);
    return 0;
}
```

### 3.3 C语言：select — 单线程监控多个IO

```c
#include <sys/select.h>
#include <sys/time.h>
#include <unistd.h>
#include <stdio.h>
#include <string.h>

/**
 * select示例：单线程同时监控多个fd
 * 适用于连接数较少（<1024）的场景
 * 缺点：每次调用都要遍历全部fd，O(n)复杂度
 */
int main(void) {
    int pipe_a[2], pipe_b[2];
    pipe(pipe_a);
    pipe(pipe_b);

    // 预写入数据（模拟异步IO就绪）
    write(pipe_a[1], "data from A", 11);
    write(pipe_b[1], "data from B", 11);

    fd_set read_fds;
    int maxfd = (pipe_a[0] > pipe_b[0] ? pipe_a[0] : pipe_b[0]) + 1;

    // select等待（最多等5秒）
    struct timeval tv = { .tv_sec = 5, .tv_usec = 0 };
    FD_ZERO(&amp;read_fds);
    FD_SET(pipe_a[0], &amp;read_fds);
    FD_SET(pipe_b[0], &amp;read_fds);

    int ready = select(maxfd, &amp;read_fds, NULL, NULL, &amp;tv);
    if (ready < 0) {
        perror("select");
        return 1;
    }
    if (ready == 0) {
        printf("超时，无fd就绪\n");
        return 0;
    }

    printf("%d 个fd就绪:\n", ready);
    char buf[1024];

    // 检查哪个fd就绪
    if (FD_ISSET(pipe_a[0], &amp;read_fds)) {
        ssize_t n = read(pipe_a[0], buf, sizeof(buf));
        buf[n] = '\0';
        printf("  pipe_a: %s\n", buf);
    }
    if (FD_ISSET(pipe_b[0], &amp;read_fds)) {
        ssize_t n = read(pipe_b[0], buf, sizeof(buf));
        buf[n] = '\0';
        printf("  pipe_b: %s\n", buf);
    }

    close(pipe_a[0]); close(pipe_a[1]);
    close(pipe_b[0]); close(pipe_b[1]);
    return 0;
}
```

### 3.4 C语言：epoll — 高并发事件驱动

```c
#include <sys/epoll.h>
#include <unistd.h>
#include <stdio.h>
#include <string.h>

#define MAX_EVENTS 10

/**
 * epoll示例：事件驱动的IO多路复用
 * 注册一次，后续只返回就绪的fd
 * 时间复杂度：O(1)就绪通知，适合海量并发连接
 */
int main(void) {
    int pipe_a[2], pipe_b[2];
    pipe(pipe_a);
    pipe(pipe_b);
    write(pipe_a[1], "msg_a", 5);
    write(pipe_b[1], "msg_b", 5);

    // 1. 创建epoll实例
    int epfd = epoll_create1(0);
    if (epfd < 0) { perror("epoll_create1"); return 1; }

    // 2. 注册感兴趣的fd（一次性操作）
    struct epoll_event ev;
    ev.events = EPOLLIN;  // 监听可读事件
    ev.data.fd = pipe_a[0];
    epoll_ctl(epfd, EPOLL_CTL_ADD, pipe_a[0], &amp;ev);

    ev.data.fd = pipe_b[0];
    epoll_ctl(epfd, EPOLL_CTL_ADD, pipe_b[0], &amp;ev);

    // 3. 等待事件（只返回就绪的fd，不遍历全部）
    struct epoll_event events[MAX_EVENTS];
    int nfds = epoll_wait(epfd, events, MAX_EVENTS, 5000);

    if (nfds < 0) { perror("epoll_wait"); return 1; }
    if (nfds == 0) { printf("超时\n"); return 0; }

    printf("%d 个事件就绪:\n", nfds);
    char buf[1024];

    for (int i = 0; i < nfds; i++) {
        ssize_t n = read(events[i].data.fd, buf, sizeof(buf));
        buf[n] = '\0';
        printf("  fd=%d: %s\n", events[i].data.fd, buf);
    }

    close(epfd);
    close(pipe_a[0]); close(pipe_a[1]);
    close(pipe_b[0]); close(pipe_b[1]);
    return 0;
}
```

### 3.5 Python：epoll事件驱动服务器骨架

```python
#!/usr/bin/env python3
"""
Python epoll示例：构建事件驱动的IO模型
适用于需要同时处理大量连接的网络服务器场景
"""
import select
import socket
import os

def epoll_server_demo():
    """演示epoll的核心用法"""
    # 创建一个监听socket
    server = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    server.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
    server.bind(('127.0.0.1', 9999))
    server.listen(128)
    server.setblocking(False)  # 非阻塞模式

    # 创建epoll实例
    ep = select.epoll()
    # 将server socket注册到epoll
    ep.register(server.fileno(), select.EPOLLIN | select.EPOLLET)

    # fd → socket的映射
    fd_to_socket = {server.fileno(): server}
    print("epoll服务器启动，等待连接...")

    try:
        while True:
            # 等待事件（阻塞，直到有IO就绪）
            events = ep.poll(timeout=1.0)

            for fd, event in events:
                sock = fd_to_socket[fd]

                if fd == server.fileno():
                    # 新连接到来
                    client, addr = server.accept()
                    print(f"新连接: {addr}")
                    client.setblocking(False)
                    ep.register(client.fileno(), select.EPOLLIN | select.EPOLLET)
                    fd_to_socket[client.fileno()] = client

                elif event &amp; select.EPOLLIN:
                    # 可读事件
                    data = sock.recv(1024)
                    if data:
                        print(f"收到: {data.decode()}")
                        # Echo回客户端
                        sock.sendall(data)
                    else:
                        # 连接关闭
                        print(f"fd={fd} 断开")
                        ep.unregister(fd)
                        sock.close()
                        del fd_to_socket[fd]

    except KeyboardInterrupt:
        print("服务器关闭")
    finally:
        ep.close()
        server.close()

if __name__ == "__main__":
    epoll_server_demo()
```

### 3.6 Go语言：goroutine天然封装的IO模式

Go语言的独特之处在于：**goroutine将阻塞IO封装成了看似异步的并发模型**。你写的代码看起来是同步阻塞的，但Go runtime在底层使用了epoll（Linux）/kqueue（macOS）实现了高效的IO多路复用。

```go
package main

import (
	"bufio"
	"fmt"
	"io"
	"os"
	"strings"
	"sync"
	"time"
)

func main() {
	// ===== 模式1：标准IO（Go runtime自动使用epoll） =====
	fmt.Println("=== Go标准IO ===")

	// 文件读取 — 底层使用缓冲IO + epoll
	f, err := os.Open("/etc/passwd")
	if err != nil {
		fmt.Printf("打开失败: %v\n", err)
		return
	}
	defer f.Close()

	// io.ReadAll 内部自动缓冲读取
	data, err := io.ReadAll(f)
	fmt.Printf("读取 %d 字节\n", len(data))

	// 字符串IO（内存中的IO操作）
	reader := strings.NewReader("Hello, IO System!")
	buf := make([]byte, 5)
	n, _ := reader.Read(buf)
	fmt.Printf("读取: %s (%d bytes)\n", buf[:n], n)

	// ===== 模式2：并发IO（goroutine + channel） =====
	fmt.Println("\n=== 并发IO ===")

	// Go的并发模型：每个goroutine一个"伪阻塞"IO
	// 实际上Go runtime会将阻塞的goroutine挂起，不占用OS线程
	var wg sync.WaitGroup
	for i := 0; i < 5; i++ {
		wg.Add(1)
		go func(id int) {
			defer wg.Done()
			// 模拟IO操作
			time.Sleep(time.Duration(id) * 100 * time.Millisecond)
			fmt.Printf("goroutine %d 完成IO\n", id)
		}(i)
	}
	wg.Wait()

	// ===== 模式3：bufio带缓冲IO =====
	fmt.Println("\n=== 带缓冲IO ===")

	// bufio.Reader 在用户态维护缓冲区，减少系统调用次数
	f2, _ := os.Create("/tmp/go_io_test")
	w := bufio.NewWriterSize(f2, 256*1024) // 256KB缓冲

	for i := 0; i < 1000; i++ {
		fmt.Fprintf(w, "line %d: %s\n", i, strings.Repeat("x", 100))
	}
	w.Flush() // 手动刷新缓冲区
	f2.Close()

	fmt.Println("带缓冲写入完成")

	// ===== Go的IO模式选择指南 =====
	fmt.Println(`

  ┌─────────────────────────────────────────────────────────┐
  │ 场景              推荐方式            底层机制          │
  ├─────────────────────────────────────────────────────────┤
  │ 网络服务器        goroutine+net.Conn  epoll/kqueue      │

  │ 日志写入          bufio.Writer        用户态缓冲        │
  │ 数据库连接池      goroutine pool      epoll+连接复用    │
  │ 超低延迟存储      调用C库(io_uring)  cgo桥接           │

关键理解: Go的"阻塞IO" ≠ 传统意义的阻塞
  goroutine阻塞时，Go runtime会将其挂起（类似绿色线程）
  OS线程不会被占用，继续调度其他goroutine
  但对于CPU密集型任务，仍需注意线程数限制
	`)
```

### 3.7 C语言：io_uring — 下一代异步IO

```c
// 需要 liburing 库
// apt install liburing-dev
// gcc -o uring_demo uring_demo.c -luring
#include <liburing.h>
#include <fcntl.h>
#include <string.h>
#include <stdio.h>
#include <stdlib.h>

#define QUEUE_DEPTH 4

int main(void) {
    struct io_uring ring;

    // 1. 初始化io_uring实例（队列深度为4）
    // flags=0 表示使用默认的中断模式
    // 可用 IORING_SETUP_SQPOLL 开启SQ轮询模式
    io_uring_queue_init(QUEUE_DEPTH, &amp;ring, 0);

    // 2. 打开文件
    int fd = open("/tmp/testfile", O_RDONLY | O_DIRECT);
    if (fd < 0) {
        perror("open (需要O_DIRECT对齐)");
        io_uring_queue_exit(&amp;ring);
        return 1;
    }

    // 3. 准备对齐的缓冲区（Direct IO要求4096字节对齐）
    char *buf;
    posix_memalign((void**)&amp;buf, 4096, 4096);

    // 4. 获取SQE（Submission Queue Entry）
    struct io_uring_sqe *sqe = io_uring_get_sqe(&amp;ring);
    // 准备读请求：从fd读取4096字节到buf，偏移0
    io_uring_prep_read(sqe, fd, buf, 4096, 0);
    // 可选：设置用户数据，用于在完成时标识请求
    io_uring_sqe_set_data(sqe, (void*)0x1234);

    // 5. 提交请求到内核
    io_uring_submit(&amp;ring);  // 这是一次系统调用

    // 6. 等待完成队列（CQ）有结果
    struct io_uring_cqe *cqe;
    io_uring_wait_cqe(&amp;ring, &amp;cqe);

    // 7. 处理结果
    if (cqe->res > 0) {
        printf("读取了 %d 字节:\n", cqe->res);
        // 打印前20字节
        int print_len = cqe->res > 20 ? 20 : cqe->res;
        printf("  前20字节: %.*s\n", print_len, buf);
    } else {
        printf("IO失败: %d\n", cqe->res);
    }

    // 8. 标记CQE已消费（必须调用，否则CQ会满）
    io_uring_cqe_seen(&amp;ring, cqe);

    // 批量提交示例：一次提交多个请求
    printf("\n=== 批量提交示例 ===\n");
    for (int i = 0; i < QUEUE_DEPTH; i++) {
        sqe = io_uring_get_sqe(&amp;ring);
        io_uring_prep_read(sqe, fd, buf, 4096, i * 4096);
        io_uring_sqe_set_data(sqe, (void*)(long)i);
    }
    // 一次系统调用提交4个请求
    int submitted = io_uring_submit(&amp;ring);
    printf("提交了 %d 个请求\n", submitted);

    // 收割完成事件
    for (int i = 0; i < submitted; i++) {
        io_uring_wait_cqe(&amp;ring, &amp;cqe);
        int idx = (int)(long)io_uring_cqe_get_data(cqe);
        printf("请求 %d 完成: %d 字节\n", idx, cqe->res);
        io_uring_cqe_seen(&amp;ring, cqe);
    }

    // 清理
    close(fd);
    io_uring_queue_exit(&amp;ring);
    free(buf);
    return 0;
}
```

## 四、选择决策框架

面对一个具体的IO场景，如何选择最合适的模式？以下决策路径基于**并发数**、**延迟要求**和**IO类型**三个维度：

### 4.1 选择决策树

需要处理多少个并发IO？
│
├── 1个 → 阻塞IO
│   最简单，适合单文件处理、脚本任务
│   如果需要并发 → 开goroutine（Go）或多线程（C/Python）
│
├── 2 ~ 数百个 → select/poll
│   跨平台兼容（Linux/macOS/Windows都支持）
│   fd数量可控时性能足够
│
├── 数百 ~ 数万个 → epoll（Linux）/ kqueue（macOS/BSD）
│   事件驱动，O(1)就绪通知
│   Nginx/Redis/Node.js的核心机制
│
├── 数万 ~ 百万级 → epoll + worker线程池
│   主线程用epoll接收连接
│   worker线程处理具体IO
│
└── 超低延迟存储IO（<10μs） → io_uring
    零系统调用开销
    支持批量提交和轮询模式
    需要Linux 5.1+内核

### 4.2 场景匹配速查表

| 应用场景 | 推荐模式 | 理由 |
|----------|---------|------|
| 简单脚本读文件 | 阻塞IO | 最简单，无需额外复杂度 |
| Web服务器(Nginx) | epoll | 高并发，事件驱动 |
| Redis | epoll | 单线程事件循环，低延迟 |
| 数据库(MySQL) | 多线程+阻塞IO | 每个连接一个线程，简单可靠 |
| 数据库(PostgreSQL) | epoll | 多进程架构，事件驱动 |
| 消息队列(Kafka) | epoll + sendfile | 高吞吐网络IO |
| 视频流媒体服务器 | epoll + 异步IO | 大量并发流 |
| 实时交易系统 | io_uring + 轮询 | 超低延迟要求 |
| 分布式存储(Ceph) | io_uring | 高性能存储IO |
| CLI工具/批处理脚本 | 阻塞IO | 无需并发 |

### 4.3 系统调用开销量化对比

以一个典型的高并发场景为例：**1000个并发连接，每秒10万次请求**。

| 模式 | 系统调用次数/秒 | 上下文切换次数/秒 | 内核CPU时间占比 | 吞吐能力上限 |
|------|----------------|-------------------|----------------|-------------|
| 多线程阻塞 | ~100,000 | ~200,000 | 40-60% | ~5万QPS |
| 单线程select | ~100,000 | ~100,000 | 30-50% | ~8万QPS |
| 单线程epoll(LT) | ~100,000 | ~100,000 | 15-25% | ~20万QPS |
| 单线程epoll(ET) | ~100,000 | ~100,000 | 10-20% | ~30万QPS |
| io_uring(批处理) | ~1,000 | ~2,000 | 3-8% | ~100万+QPS |

> **注**：上表中epoll的系统调用次数看似与select相同，但关键差异在于：select每次都遍历全部fd（O(n)），而epoll内核只返回就绪的fd（O(1)），因此epoll每次系统调用的实际CPU消耗远低于select。io_uring通过批量提交将系统调用频率降低了1-2个数量级。

## 五、epoll的两种触发模式

epoll支持两种触发模式，对应用编程模型有根本性影响：

### 5.1 水平触发（Level Triggered, LT）

默认模式。只要fd处于"可读"状态，每次epoll_wait都会返回该fd。

epoll_wait() → fd可读 → 通知
epoll_wait() → fd仍可读 → 再次通知
epoll_wait() → 数据读完 → 不再通知

特点：
  ✅ 安全：即使你忘记读完数据，下次还会提醒
  ❌ 低效：每次都要处理，即使你还没来得及读完
  类似select/poll的语义

### 5.2 边沿触发（Edge Triggered, ET）

只在fd状态变化时通知一次。

epoll_wait() → fd从不可读变为可读 → 通知（仅一次！）
epoll_wait() → fd仍可读但状态未变 → 不通知
必须一次性读完所有数据，否则后续数据不会触发通知

特点：
  ✅ 高效：状态变化才通知，减少无效事件
  ❌ 危险：必须配合非阻塞IO，必须循环读到EAGAIN
  大多数高性能服务器（Nginx、Redis）使用ET模式

### 5.3 ET模式的正确使用模式

```c
// ET模式：必须配合非阻塞IO + 循环读取到EAGAIN
int fd = socket(AF_INET, SOCK_STREAM, 0);
fcntl(fd, F_SETFL, O_NONBLOCK);  // 必须设置非阻塞！

struct epoll_event ev;
ev.events = EPOLLIN | EPOLLET;  // ET模式
ev.data.fd = fd;
epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &amp;ev);

// 当epoll_wait返回该fd可读时：
char buf[4096];
while (1) {
    ssize_t n = read(fd, buf, sizeof(buf));
    if (n > 0) {
        process_data(buf, n);
    } else if (n < 0 &amp;&amp; errno == EAGAIN) {
        break;  // 数据读完了，退出循环
    } else {
        perror("read");
        break;
    }
}
```

## 六、常见错误与陷阱

### 错误1：select的fd_set未重新初始化

```c
// ❌ 错误：select会修改fd_set，下次循环必须重新设置
fd_set read_fds;
FD_ZERO(&amp;read_fds);
FD_SET(fd1, &amp;read_fds);
FD_SET(fd2, &amp;read_fds);

while (1) {
    select(maxfd, &amp;read_fds, NULL, NULL, NULL);
    // ❌ 第二次循环时read_fds已经被select修改了！
    // 只有第一个就绪的fd会被检查到
    if (FD_ISSET(fd1, &amp;read_fds)) handle(fd1);
    if (FD_ISSET(fd2, &amp;read_fds)) handle(fd2);
}

// ✅ 正确：每次循环前重新设置fd_set
while (1) {
    FD_ZERO(&amp;read_fds);
    FD_SET(fd1, &amp;read_fds);
    FD_SET(fd2, &amp;read_fds);
    select(maxfd, &amp;read_fds, NULL, NULL, NULL);
    if (FD_ISSET(fd1, &amp;read_fds)) handle(fd1);
    if (FD_ISSET(fd2, &amp;read_fds)) handle(fd2);
}
```

### 错误2：ET模式未读取所有数据

```c
// ❌ 错误：ET模式下只读一次，可能丢失数据
// epoll_wait返回fd可读
char buf[1024];
read(fd, buf, sizeof(buf));  // 只读了一部分！
// 剩余数据不会再次触发通知，永远丢失

// ✅ 正确：循环读取到EAGAIN
while (1) {
    ssize_t n = read(fd, buf, sizeof(buf));
    if (n < 0 &amp;&amp; errno == EAGAIN) break;  // 读完了
    if (n > 0) process(buf, n);
}
```

### 错误3：Go中在goroutine内执行阻塞系统调用

```go
// ❌ 错误：在goroutine中执行真正的阻塞系统调用
// Go的net包使用epoll，但os.File.Read可能真正阻塞OS线程
f, _ := os.Open("/dev/sda")  // 块设备读取可能很慢
buf := make([]byte, 4096)
f.Read(buf)  // 可能阻塞整个OS线程数秒！

// ✅ 正确：使用Go的标准IO模式
// 方法1：os.ReadFile（一次性读取，简单场景）
data, _ := os.ReadFile("/tmp/data")

// 方法2：goroutine + channel（并发场景）
go func() {
    data, _ := os.ReadFile("/tmp/data")
    resultCh <- data
}()

// 方法3：对于块设备，考虑使用io.Copy或bufio
```

### 错误4：忘记设置O_NONBLOCK导致epoll+read组合阻塞

```c
// ❌ 错误：fd未设置非阻塞，epoll通知后read仍然可能阻塞
int fd = open("/dev/input/event0", O_RDONLY);
// 没有设置O_NONBLOCK！

struct epoll_event ev = { .events = EPOLLIN, .data.fd = fd };
epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &amp;ev);

// 即使epoll通知了EPOLLIN，read仍然可能因为竞争条件而阻塞
read(fd, buf, sizeof(buf));  // 可能阻塞！

// ✅ 正确：非阻塞fd + epoll 是标准搭配
int fd = open("/dev/input/event0", O_RDONLY | O_NONBLOCK);
```

### 错误5：io_uring忘记标记CQE已消费

```c
// ❌ 错误：消费CQ但不标记已读
struct io_uring_cqe *cqe;
io_uring_wait_cqe(&amp;ring, &amp;cqe);
// 处理cqe...
// 忘记调用 io_uring_cqe_seen()
// → CQ满后无法接受新的完成事件，IO完全停滞

// ✅ 正确：每次处理后标记
io_uring_wait_cqe(&amp;ring, &amp;cqe);
process_completion(cqe);
io_uring_cqe_seen(&amp;ring, cqe);  // 必须调用！
```

## 八、演进脉络：为什么需要这些模式

IO模式的演进不是"新替代旧"，而是**每一层都是为了解决上一层解决不了的问题**：

1970s  阻塞IO
       问题: 一个线程只能处理一个连接
       ↓ 催生
1983   select/poll（IO多路复用）
       问题: O(n)遍历，fd数量受限
       ↓ 催生
2002   epoll（O(1)事件通知）
       问题: 仍然是同步IO，每次read/write需要系统调用
       ↓ 催生
2002   Linux AIO（尝试异步IO）
       问题: 实现有缺陷，只支持O_DIRECT
       ↓ 催生
2019   io_uring（真正的异步IO框架）
       解决: 共享内存+批量提交，系统调用开销趋近于零

每一代技术都没有完全取代前一代：
- 阻塞IO至今仍是大多数应用的默认选择
- select因其跨平台性仍然广泛使用
- epoll是当前Linux网络服务器的主流方案
- io_uring是未来的方向，但生态成熟度还在追赶中

理解这个演进脉络，比记住每个模式的API更重要——它告诉你**每种模式存在的理由和适用边界**。

## 九、本技巧小结

| 要点 | 核心内容 |
|------|---------|
| 六种模式 | 阻塞IO → 非阻塞IO → select → poll → epoll → io_uring |
| 同步vs异步 | 应用是否需要自己检查IO完成（epoll仍是同步IO！） |
| 选择依据 | 并发数 × 延迟要求 × IO类型 = 最优模式 |
| epoll本质 | O(1)就绪通知 + 边沿触发/水平触发 |
| io_uring本质 | 共享内存SQ/CQ + 批量提交 + 零系统调用 |
| ET模式 | 必须配合非阻塞IO，必须循环读到EAGAIN |
| Go的特殊性 | goroutine将阻塞IO封装为并发模型，底层用epoll |
| 演进逻辑 | 每一代解决上一代的瓶颈，但不完全替代 |
