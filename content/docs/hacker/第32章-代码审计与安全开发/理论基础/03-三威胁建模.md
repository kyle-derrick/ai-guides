## 三、威胁建模

### 3.1 威胁建模概述

威胁建模（Threat Modeling）是一种系统性的安全分析方法，核心目标是在软件设计阶段而非部署阶段识别安全风险。它源于微软SDL框架中的安全设计阶段，至今已成为行业公认的最佳实践。根据BSIMM（Building Security In Maturity Model）对全球130+企业的调研数据，采用威胁建模的组织在漏洞修复成本上比事后修补低60-100倍——设计阶段发现的缺陷修复成本约为$10，而发布后发现的同类缺陷修复成本高达$100-1000。

威胁建模回答四个核心问题：

| 核心问题 | 活动内容 | 产出物 |
|----------|----------|--------|
| **我们在做什么？** | 绘制数据流图（DFD），梳理系统架构 | 系统架构图、信任边界标注 |
| **什么可能出错？** | 系统性识别威胁，分析攻击路径 | 威胁清单、攻击树 |
| **我们要怎么应对？** | 制定缓解措施和安全控制 | 缓解方案、安全需求 |
| **我们做得够好吗？** | 验证缓解措施的有效性 | 风险评估报告、验证记录 |

**威胁建模 vs 其他安全活动**

| 对比维度 | 威胁建模 | 渗透测试 | 代码审计 |
|----------|----------|----------|----------|
| 执行时机 | 设计阶段（左移） | 部署前后 | 开发阶段 |
| 分析对象 | 架构与设计 | 运行时系统 | 源代码 |
| 发现问题类型 | 架构级缺陷 | 可利用漏洞 | 编码缺陷 |
| 修复成本 | 最低 | 较高 | 中等 |
| 覆盖范围 | 全系统视角 | 特定攻击面 | 代码逻辑 |
| 可自动化程度 | 低（需要人工判断） | 中 | 高 |

威胁建模的独特价值在于：它是唯一在**设计阶段**就能发现**架构级安全缺陷**的方法。渗透测试和代码审计虽然重要，但它们发现的问题往往已经进入编码或部署阶段，修复成本远高于设计阶段。

### 3.2 数据流图（DFD）

数据流图（Data Flow Diagram）是威胁建模的基础工具，用于描述系统的数据流动、处理过程和存储方式。DFD由Larry Constantine在1970年代提出，后被安全领域广泛采用作为威胁建模的起点。

#### 3.2.1 DFD基本元素

DFD包含四种基本元素：

```text
┌─────────────────┐
│   外部实体        │  External Entity：系统边界外的参与者
│   （矩形）        │  例如：用户、第三方API、外部系统
└─────────────────┘

    ───────→          Data Flow：数据流动方向
                       例如：HTTP请求、数据库查询

    ○─────────○       Process：数据的处理和转换
    （圆形）           例如：认证服务、支付处理

    ═══════════       Data Store：数据的持久化存储
    （平行线）         例如：数据库、文件系统、缓存
```

#### 3.2.2 Web应用DFD示例

以一个典型的电商Web应用为例：

```text
                         信任边界1（网络边界）
                              ║
┌──────────┐    HTTP/HTTPS    ║    ┌──────────────┐     SQL      ════════
│  用户     │ ──────────────→  ║ →  │  Web服务器   │ ──────────→  │ 数据库  │
│ (外部实体) │ ←──────────────  ║ ←  │  (处理过程)  │ ←──────────  │(数据存储)│
└──────────┘    HTML/JSON     ║    └──────┬───────┘     结果      ════════
                              ║           │
                              ║           │ API调用
                              ║           ↓
                              ║    ┌──────────────┐
                              ║ ←  │  支付网关    │
                              ║    │ (外部实体)    │
                              ║    └──────────────┘
```

这个DFD揭示了三个关键的信任边界：
- 用户与Web服务器之间（网络边界）
- Web服务器与数据库之间（应用边界）
- Web服务器与支付网关之间（第三方边界）

#### 3.2.3 信任边界标注

信任边界（Trust Boundary）是DFD中最重要的安全标注，它标识了数据从一个信任级别流向另一个信任级别的位置。所有穿越信任边界的数据流都是潜在的攻击面，必须进行严格的输入验证和安全控制。

```text
┌─────────────┐     ║  信任边界    ║     ┌─────────────┐
│  不可信区域   │ ──→ ║            ║ ──→  │  可信区域    │
│              │     ║  验证/过滤层 ║     │              │
│  外部输入     │     ║            ║     │  内部处理     │
│  未认证数据   │     ║  身份验证   ║     │  已验证数据   │
└─────────────┘     ║  权限检查   ║     └─────────────┘
                    ║  数据校验   ║
```

**信任边界标注的最佳实践：**

1. **不要信任任何穿越边界的数据**——即使来自"内部"系统的数据
2. **边界处必须实施验证**——输入验证、身份认证、权限检查
3. **记录所有边界穿越事件**——便于事后审计和攻击溯源
4. **最小化穿越边界的接口**——减少攻击面

#### 3.2.4 系统级DFD绘制方法

绘制系统级DFD的步骤：

1. **识别所有外部实体**——列出与系统交互的所有外部系统、用户、第三方服务
2. **识别所有处理过程**——列出系统中所有数据处理节点（微服务、API网关、消息队列等）
3. **识别所有数据存储**——列出所有数据库、文件存储、缓存、配置存储
4. **绘制数据流**——用箭头连接所有数据流动路径，标注数据类型
5. **标注信任边界**——用虚线框标出所有信任级别变化的位置

### 3.3 STRIDE威胁模型

STRIDE是微软在1990年代提出的经典威胁分类模型，名称由六种威胁类型的英文首字母组成。至今仍是业界最广泛使用的威胁分类框架之一。

#### 3.3.1 STRIDE六类威胁详解

| 威胁类型 | 英文全称 | 破坏的安全属性 | 描述 | 典型漏洞示例 |
|----------|----------|----------------|------|--------------|
| **S** | Spoofing（欺骗） | 认证性 | 冒充其他用户或系统的身份 | 弱密码认证、会话固定、JWT密钥泄露 |
| **T** | Tampering（篡改） | 完整性 | 未授权修改数据或代码 | SQL注入、参数篡改、文件上传漏洞 |
| **R** | Repudiation（抵赖） | 不可否认性 | 用户否认执行过某操作 | 缺少审计日志、日志可被篡改 |
| **I** | Information Disclosure（信息泄露） | 保密性 | 未授权访问敏感信息 | 明文传输、目录遍历、错误信息泄露 |
| **D** | Denial of Service（拒绝服务） | 可用性 | 使系统无法正常提供服务 | 资源耗尽、死锁、无限循环 |
| **E** | Elevation of Privilege（权限提升） | 授权性 | 获取超出预期的权限级别 | 越权访问、SQL注入提权、反序列化 |

#### 3.3.2 STRIDE应用方法

**STEP 1：对每个DFD元素应用STRIDE**

不是所有威胁类型都适用于所有DFD元素。以下是STRIDE与DFD元素的映射关系：

| DFD元素 | S | T | R | I | D | E |
|---------|---|---|---|---|---|---|
| 外部实体 | ✓ | - | - | ✓ | - | - |
| 处理过程 | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |
| 数据流 | ✓ | ✓ | - | ✓ | ✓ | - |
| 数据存储 | - | ✓ | ✓ | ✓ | ✓ | - |

- **外部实体**：主要关注欺骗（冒充身份）和信息泄露
- **处理过程**：所有威胁类型都适用（因为处理过程是系统的"大脑"）
- **数据流**：关注篡改（数据被修改）、欺骗（伪造数据流）、拒绝服务（数据流阻断）
- **数据存储**：关注篡改（数据被修改）、抵赖（操作无记录）、信息泄露

**STEP 2：逐元素分析威胁**

以"用户登录模块"为例进行完整的STRIDE分析：

```text
威胁分析：用户登录模块
│
├── S（欺骗）
│   ├── 威胁描述：攻击者使用窃取的凭证冒充合法用户
│   ├── 攻击场景：
│   │   ├── 场景1：暴力破解弱密码（自动化工具尝试常见密码组合）
│   │   ├── 场景2：凭证填充（使用其他网站泄露的用户名密码组合）
│   │   ├── 场景3：会话固定（诱导用户使用攻击者预设的会话ID）
│   │   └── 场景4：钓鱼攻击（伪造登录页面获取用户凭证）
│   ├── 缓解措施：
│   │   ├── 实施多因素认证（MFA）
│   │   ├── 账户锁定策略（5次失败后锁定15分钟）
│   │   ├── 强密码策略（最小12位，包含大小写+数字+特殊字符）
│   │   ├── 登录失败延迟递增（1s→2s→4s→8s）
│   │   └── 使用安全的会话管理（服务端生成随机会话ID）
│   └── 残余风险：社会工程学攻击、内部威胁
│
├── T（篡改）
│   ├── 威胁描述：篡改登录请求或认证过程中的数据
│   ├── 攻击场景：
│   │   ├── 场景1：修改HTTP请求中的用户名参数（水平越权）
│   │   ├── 场景2：修改认证响应（伪造认证结果）
│   │   └── 场景3：篡改密码重置链接中的参数
│   ├── 缓解措施：
│   │   ├── 全站HTTPS（强制TLS 1.2+）
│   │   ├── 服务端参数验证（不信任客户端任何输入）
│   │   ├── 请求签名（HMAC验证请求完整性）
│   │   └── CSRF Token保护
│   └── 残余风险：证书伪造、中间人攻击
│
├── R（抵赖）
│   ├── 威胁描述：用户否认执行了某次登录或敏感操作
│   ├── 攻击场景：
│   │   ├── 场景1：用户否认修改了账户信息
│   │   ├── 场景2：管理员否认执行了权限变更
│   │   └── 场景3：交易双方对交易内容产生争议
│   ├── 缓解措施：
│   │   ├── 详细的审计日志（who+when+what+fromWhere+result）
│   │   ├── 日志防篡改（WORM存储、区块链哈希校验）
│   │   ├── 数字签名（关键操作需用户签名确认）
│   │   └── 时间戳服务（可信时间源）
│   └── 残余风险：日志系统被入侵、内部人员删除日志
│
├── I（信息泄露）
│   ├── 威胁描述：未授权访问敏感信息（用户凭证、业务数据）
│   ├── 攻击场景：
│   │   ├── 场景1：错误信息泄露数据库结构（SQL报错返回表名字段）
│   │   ├── 场景2：明文传输密码（HTTP而非HTTPS）
│   │   ├── 场景3：密码重置链接包含可预测的token
│   │   └── 场景4：调试接口未关闭（/debug、/actuator）
│   ├── 缓解措施：
│   │   ├── 密码使用bcrypt/scrypt/Argon2哈希存储
│   │   ├── 统一错误页面（不暴露系统细节）
│   │   ├── HttpOnly + Secure + SameSite Cookie
│   │   ├── 敏感数据加密存储（AES-256-GCM）
│   │   └── 关闭调试端点和详细错误输出
│   └── 残余风险：侧信道攻击、内存取证
│
├── D（拒绝服务）
│   ├── 威胁描述：大量登录请求或资源耗尽导致服务不可用
│   ├── 攻击场景：
│   │   ├── 场景1：暴力破解触发大量数据库查询
│   │   ├── 场景2：发送超大登录表单消耗内存
│   │   ├── 场景3：慢速连接占用服务器连接池
│   │   └── 场景4：密码重置接口被滥用发送大量邮件
│   ├── 缓解措施：
│   │   ├── 速率限制（IP级+账号级+全局级）
│   │   ├── 验证码（登录失败3次后触发）
│   │   ├── WAF规则（拦截异常请求模式）
│   │   ├── 连接超时设置（防止慢速攻击）
│   │   └── 资源配额（限制单用户每小时请求次数）
│   └── 残余风险：分布式DDoS攻击
│
└── E（权限提升）
    ├── 威绕认证获得管理员权限
    ├── 攻击场景：
    │   ├── 场景1：修改登录响应中的用户角色字段
    │   ├── 场景2：利用参数注入修改SQL查询逻辑
    │   ├── 场景3：通过会话劫持获取管理员会话
    │   └── 场景4：利用反序列化漏洞执行任意代码
    ├── 缓解措施：
    │   ├── 服务端权威验证（角色/权限由服务端决定，不由客户端传递）
    │   ├── 最小权限原则（默认最低权限，按需升级）
    │   ├── 输入参数化（防止注入攻击）
    │   └── 会话安全（绑定IP/设备指纹、定期轮换）
    └── 残余风险：零日漏洞、供应链攻击
```

#### 3.3.3 STRIDE的实际执行技巧

**技巧1：创建STRIDE分析矩阵**

对于大型系统，使用矩阵形式系统化地分析每个组件：

| 系统组件 | S(欺骗) | T(篡改) | R(抵赖) | I(泄露) | D(拒绝) | E(提权) |
|----------|---------|---------|---------|---------|---------|---------|
| 登录API | 高 | 中 | 中 | 高 | 高 | 高 |
| 用户信息API | 中 | 中 | 低 | 高 | 中 | 中 |
| 支付接口 | 高 | 高 | 高 | 高 | 中 | 高 |
| 日志系统 | 低 | 高 | 高 | 中 | 低 | 低 |
| 配置管理 | 中 | 高 | 中 | 高 | 中 | 高 |

**技巧2：利用攻击树辅助STRIDE**

对于高风险威胁，可以使用攻击树进一步分析攻击路径：

```text
目标：获取管理员权限（STRIDE-E）
│
├── 分支1：绕过认证
│   ├── 弱密码暴力破解
│   ├── 会话固定攻击
│   └── JWT密钥泄露
│
├── 分支2：权限提升
│   ├── 水平越权→垂直越权
│   │   └── 修改请求参数中的角色ID
│   ├── 通过SQL注入修改用户角色
│   └── 利用反序列化漏洞执行代码
│
└── 分支3：获取高权限凭证
    ├── 从配置文件读取硬编码密码
    ├── 从日志中获取明文凭证
    └── 从备份文件中恢复凭证
```

### 3.4 DREAD风险评估模型

DREAD是用于对STRIDE识别出的威胁进行量化风险评估的经典模型。它通过五个维度评估每个威胁的风险等级，帮助安全团队确定修复优先级。

#### 3.4.1 DREAD五维度详解

| 维度 | 英文全称 | 评估标准 | 低分(1-3) | 中分(4-6) | 高分(7-10) |
|------|----------|----------|-----------|-----------|------------|
| **D** | Damage Potential | 漏洞被利用后造成的损害程度 | 仅影响单个用户 | 影响部分用户或数据泄露 | 影响全部用户或系统崩溃 |
| **R** | Reproducibility | 攻击的可重复程度 | 需要特殊条件才能触发 | 在特定条件下可重复 | 100%可重复，无需特殊条件 |
| **E** | Exploitability | 攻击的难度 | 需要高级技术能力和定制工具 | 使用现有工具可实现 | 使用浏览器/命令行即可 |
| **A** | Affected Users | 受影响的用户数量 | 仅影响内部用户 | 影响部分外部用户 | 影响所有用户 |
| **D** | Discoverability | 漏洞被发现的容易程度 | 几乎不可能被发现 | 需要一定分析才能发现 | 公开信息，任何人可发现 |

#### 3.4.2 风险评分计算

```text
风险分数 = (D + R + E + A + D) / 5

风险等级划分：
├── 高危（7.0 - 10.0）：必须立即修复，阻断发布
├── 中危（4.0 - 6.9）：计划修复，下一迭代处理
└── 低危（1.0 - 3.9）：记录在案，视情况修复
```

**评分示例：SQL注入漏洞**

| 维度 | 评分 | 理由 |
|------|------|------|
| Damage Potential | 10 | 可读取/修改/删除全部数据库，可能导致数据泄露 |
| Reproducibility | 10 | 100%可重复，参数原样发送即可触发 |
| Exploitability | 8 | 使用sqlmap等工具即可利用，门槛较低 |
| Affected Users | 10 | 影响所有使用该功能的用户 |
| Discoverability | 9 | 通过自动化扫描工具容易发现 |
| **总分** | **9.4** | **高危——必须立即修复** |

**评分示例：存储型XSS漏洞**

| 维度 | 评分 | 理由 |
|------|------|------|
| Damage Potential | 7 | 可窃取用户Cookie、执行钓鱼攻击，但不能直接修改服务器数据 |
| Reproducibility | 9 | 构造特定payload后100%可触发 |
| Exploitability | 7 | 需要了解XSS payload构造，但有成熟工具辅助 |
| Affected Users | 8 | 所有查看该页面的用户都会受影响 |
| Discoverability | 8 | 通过自动化扫描工具可发现 |
| **总分** | **7.8** | **高危——需要尽快修复** |

**评分示例：CSRF漏洞**

| 维度 | 评分 | 理由 |
|------|------|------|
| Damage Potential | 5 | 可伪造用户执行操作，但需用户已登录 |
| Reproducibility | 8 | 需要用户点击恶意链接，但技术上可重复 |
| Exploitability | 6 | 需要构造特定请求并诱导用户点击 |
| Affected Users | 7 | 所有登录用户都可能受影响 |
| Discoverability | 6 | 需要手动测试发现，自动化发现较难 |
| **总分** | **6.4** | **中危——计划修复** |

#### 3.4.3 DREAD的局限性与改进

DREAD模型存在以下已知问题：

1. **主观性强**——不同评估者可能给出差异较大的评分
2. **不考虑攻击概率**——只关注漏洞本身特性，不考虑攻击者意愿
3. **过于简化**——五个维度无法覆盖所有风险因素

**改进方案：CVSS向量评分法**

对于需要更精确评估的场景，建议使用CVSS（Common Vulnerability Scoring System）v3.1：

```text
CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H

指标含义：
AV:N = 攻击向量：网络（Network）
AC:L = 攻击复杂度：低（Low）
PR:N = 所需权限：无（None）
UI:N = 用户交互：无（None）
S:U  = 影响范围：不变（Unchanged）
C:H  = 保密性影响：高（High）
I:H  = 完整性影响：高（High）
A:H  = 可用性影响：高（High）

评分结果：9.8（Critical）
```

### 3.5 其他威胁建模方法

除了STRIDE+DREAD，业界还有多种成熟的威胁建模方法，各有侧重。

#### 3.5.1 PASTA威胁建模方法

PASTA（Process for Attack Simulation and Threat Analysis）由Tony UcedaVelez和Marco Morana在2012年提出，是一种以攻击者视角为中心的七阶段威胁建模方法。相比STRIDE的防御者视角，PASTA从攻击者角度出发，更贴近实际威胁态势。

```text
┌─────────────────────────────────────────────────────┐
│               PASTA 七阶段流程                        │
├─────────────────────────────────────────────────────┤
│                                                     │
│  阶段1：定义业务目标                                  │
│  ├── 明确业务需求和安全目标                           │
│  ├── 识别关键业务资产和合规要求                       │
│  └── 产出：业务目标文档、风险承受度                    │
│                                                     │
│  阶段2：定义技术范围                                  │
│  ├── 绘制系统架构图（DFD）                           │
│  ├── 识别所有技术组件和数据流                         │
│  └── 产出：技术架构文档、技术资产清单                  │
│                                                     │
│  阶段3：应用威胁情报                                  │
│  ├── 收集相关威胁情报（CVE、ATT&CK、行业报告）        │
│  ├── 分析历史攻击事件和TTP（战术、技术、过程）         │
│  └── 产出：威胁情报报告、攻击面分析                   │
│                                                     │
│  阶段4：威胁分析                                     │
│  ├── 使用攻击树分析攻击路径                          │
│  ├── 枚举攻击场景和攻击向量                          │
│  └── 产出：威胁清单、攻击树模型                      │
│                                                     │
│  阶段5：漏洞与弱点分析                                │
│  ├── 对每个威胁点进行漏洞分析                        │
│  ├── 使用SAST/DAST工具验证漏洞存在性                 │
│  └── 产出：漏洞清单、漏洞验证报告                    │
│                                                     │
│  阶段6：攻击建模                                     │
│  ├── 构建攻击模型，验证攻击可行性                    │
│  ├── 使用MITRE ATT&CK框架映射攻击技术               │
│  └── 产出：攻击模型、攻击路径验证结果                 │
│                                                     │
│  阶段7：风险与影响分析                                │
│  ├── 综合评估风险等级                                │
│  ├── 制定缓解措施和优先级                            │
│  └── 产出：风险评估报告、缓解方案                     │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**PASTA与STRIDE的关键差异：**

| 对比维度 | STRIDE | PASTA |
|----------|--------|-------|
| 视角 | 防御者视角 | 攻击者视角 |
| 适用规模 | 小到中型系统 | 中到大型企业系统 |
| 阶段数 | 无固定阶段 | 7个阶段 |
| 威胁分类 | 6类（STRIDE） | 无固定分类（开放） |
| 风险评估 | DREAD | 自定义（可结合CVSS） |
| 复杂度 | 低 | 高 |
| 学习曲线 | 短 | 长 |

#### 3.5.2 LINDDUN隐私威胁建模

LINDDUN是专门针对隐私威胁的建模框架，适用于GDPR/CCPA等隐私合规场景：

| 威胁类型 | 全称 | 隐私属性 | 描述 |
|----------|------|----------|------|
| **L** | Linkability（关联性） | 匿名性 | 将不同来源的数据关联到同一用户 |
| **I** | Identifiability（可识别性） | 假名性 | 从数据中识别出具体个人 |
| **N** | Non-repudiation（不可否认性） | 不可否认性 | 用户行为可被追踪并无法否认 |
| **D** | Detectability（可检测性） | 隐私存在性 | 攻击者可检测到用户拥有特定数据 |
| **D** | Disclosure of information（信息泄露） | 数据保密性 | 未授权访问个人隐私数据 |
| **U** | Unawareness（不知情性） | 知情权 | 数据主体不知道数据被收集/使用 |
| **N** | Non-compliance（不合规性） | 合规性 | 数据处理违反隐私法规 |

#### 3.5.3 Attack Tree（攻击树）

攻击树是由Bruce Schneier提出的威胁建模方法，以树状结构描述攻击者达成目标的多种路径：

```text
                    目标：窃取用户数据库
                           │
            ┌──────────────┼──────────────┐
            │              │              │
      攻击Web应用     攻击基础设施     攻击人员
            │              │              │
     ┌──────┼──────┐       │         ┌────┼────┐
     │      │      │       │         │         │
   SQL注入 XSS  文件上传  物理入侵  社会工程  内部威胁
     │      │      │       │         │         │
   [高]   [中]   [中]    [低]      [中]      [中]
   
   评分说明：[高]=高风险 [中]=中风险 [低]=低风险
```

攻击树的优势：
- 可视化展示多种攻击路径
- 便于识别最关键的防御点（阻断"根节点"或"中间节点"）
- 支持定量分析（计算每个攻击路径的成本/成功率）

#### 3.5.4 VAST威胁建模

VAST（Visual Agile and Simple Threat modeling）是ThreatModeler公司提出的敏捷威胁建模方法，核心特点：
- **应用级威胁模型**：针对单个应用的详细分析
- **运营级威胁模型**：针对整个IT基础设施的宏观分析
- 支持敏捷开发流程，可集成到CI/CD
- 自动化程度高，适合大型组织

#### 3.5.5 方法选择指南

```text
选择威胁建模方法的决策树：

你的系统是什么规模？
├── 小型单体应用 → STRIDE + DREAD
├── 中型Web应用 → STRIDE + Attack Tree
├── 大型企业系统 → PASTA + CVSS
├── 隐私敏感系统 → LINDDUN + STRIDE
└── 敏捷DevSecOps → VAST 或 Pytm（代码化建模）

你的团队安全经验如何？
├── 初学者 → STRIDE（学习曲线最短）
├── 有一定经验 → STRIDE + Attack Tree
├── 专业安全团队 → PASTA
└── 隐私合规团队 → LINDDUN
```

#### 3.6.3 实战：为电商平台做威胁建模

以一个中型电商平台为例，演示完整的威胁建模过程：

**系统概述：**
- 用户端：Web应用 + 移动App
- 后端：微服务架构（用户服务、商品服务、订单服务、支付服务）
- 数据存储：MySQL（主库）、Redis（缓存）、OSS（文件存储）
- 第三方：支付网关（微信支付/支付宝）、短信服务、物流API

**步骤一：绘制系统级DFD**

```text
┌─────────┐                                    ┌─────────┐
│ 用户     │ ──HTTPS──→  ┌──────────┐           │ 支付网关 │
│(外部实体)│ ←──HTTPS──  │ API网关   │ ←───────→ │(外部实体)│

               │        └────┬─────┘
┌─────────┐    │             │
│ 移动App  │ ───┘             │ 内部RPC
│(外部实体)│ ←────            ↓
└─────────┘         ┌────────────────┐
                    │   微服务集群     │
                    │ ┌────┬────┬───┐ │
                    │ │用户│商品│订单│ │    ════════════
                    │ │服务│服务│服务│ │──→ │  MySQL    │
                    │ └────┴────┴───┘ │    │ (主库)    │
                    └────────┬────────┘    ════════════
                             │
                    ┌────────┴────────┐    ┌──────────┐
                    │   支付服务       │───→│ 短信服务  │
                    │  (处理过程)      │    │(外部实体) │
                    └─────────────────┘    └──────────┘
```

**步骤二：识别信任边界**

```text
信任边界1：用户/外部 → API网关（网络边界）
信任边界2：API网关 → 微服务集群（应用边界）
信任边界3：微服务 → 数据库（数据边界）
信任边界4：微服务 → 第三方服务（第三方边界）
```

**步骤三：逐组件STRIDE分析**

| 组件 | 主要威胁 | 风险等级 | 缓解措施 |
|------|----------|----------|----------|
| API网关 | S(欺骗)、D(拒绝)、E(提权) | 高 | JWT验证、速率限制、最小权限 |
| 用户服务 | T(篡改)、I(泄露)、E(提权) | 高 | 参数化查询、加密存储、RBAC |
| 支付服务 | S(欺骗)、T(篡改)、R(抵赖) | 极高 | 签名验证、幂等设计、完整日志 |
| 商品服务 | I(泄露)、D(拒绝) | 中 | 数据脱敏、缓存防雪崩 |
| 订单服务 | T(篡改)、R(抵赖) | 高 | 事务一致性、审计日志 |
| MySQL | I(泄露)、T(篡改) | 高 | 加密传输、权限隔离、备份加密 |

#### 3.6.4 威胁建模的常见误区

| 误区 | 问题描述 | 正确做法 |
|------|----------|----------|
| **只做一次** | 项目启动时做一次就再也不更新 | 每次架构变更、新功能上线都应重新评估 |
| **过度建模** | 对所有模块都做详细建模，耗费大量时间 | 聚焦高风险和关键业务模块，低风险模块简化处理 |
| **忽视运维视角** | 只关注代码层面，忽略部署和运维安全 | 运维人员必须参与，关注配置安全、日志安全、监控 |
| **纸上谈兵** | 产出文档但不验证缓解措施是否有效 | 通过SAST/DAST/渗透测试验证每个缓解措施 |
| **孤立进行** | 安全团队独立建模，不与开发团队协作 | 跨职能团队协作，确保技术和业务视角融合 |
| **忽视残余风险** | 假设缓解措施100%有效 | 明确记录残余风险，持续监控 |

#### 3.7.2 Pytm：代码化威胁建模

Pytm是目前最适合DevSecOps场景的威胁建模工具，它允许用Python代码定义系统架构，自动生成DFD和STRIDE威胁分析报告。

**安装与基本使用：**

```bash
# 安装pytm
pip install pytm

# 运行威胁建模（生成DFD、威胁报告）
python model.py --dfd > dfd.dot
dot -Tpng dfd.dot -o dfd.png

# 生成STRIDE威胁分析
python model.py --seq  # 生成序列图
python model.py --report threats.md  # 生成威胁报告
```

**完整示例：电商系统威胁模型**

```python
from pytm import (
    TM, Server, Dataflow, Boundary, Actor,
    Database, ExternalEntity, Datastore
)

# 初始化威胁模型
tm = TM("电商平台威胁模型")
tm.isOrdered = True
tm.description = "典型电商平台——用户、商品、订单、支付"

# 定义边界
internet = Boundary("Internet")
dmz = Boundary("DMZ")
internal = Boundary("Internal")
database_zone = Boundary("Database Zone")

# 定义组件
user = Actor("用户", inBoundary=internet)
app = Server("Web应用", inBoundary=dmz)
api_gateway = Server("API网关", inBoundary=dmz)
user_service = Server("用户服务", inBoundary=internal)
order_service = Server("订单服务", inBoundary=internal)
payment_service = Server("支付服务", inBoundary=internal)
db = Database("MySQL主库", inBoundary=database_zone)
cache = Datastore("Redis缓存", inBoundary=database_zone)
payment_gateway = ExternalEntity("支付网关", inBoundary=internet)
sms_service = ExternalEntity("短信服务", inBoundary=internet)

# 定义数据流（自动触发STRIDE分析）
req = Dataflow(user, app, "HTTPS请求")
auth = Dataflow(app, user_service, "认证请求(内部RPC)")
auth_resp = Dataflow(user_service, app, "认证结果")
order_req = Dataflow(app, order_service, "订单请求")
pay_req = Dataflow(order_service, payment_service, "支付请求")
pay_resp = Dataflow(payment_service, payment_gateway, "支付调用")
pay_result = Dataflow(payment_gateway, payment_service, "支付结果")
db_query = Dataflow(user_service, db, "用户查询")
db_resp = Dataflow(db, user_service, "查询结果")
cache_read = Dataflow(app, cache, "缓存读取")

# 运行分析
if __name__ == "__main__":
    tm.process()
```

**Pytm生成的STRIDE分析输出示例：**

```text
威胁: 用户 → Web应用 (HTTPS请求)
  S: 可能被欺骗 → 缓解: HTTPS + 认证机制
  T: 数据可能被篡改 → 缓解: TLS加密 + 输入验证
  I: 信息可能泄露 → 缓解: HTTPS + 敏感字段脱敏
  D: 服务可能被拒绝 → 缓解: 速率限制 + WAF

威胁: 支付服务 → 支付网关 (支付调用)
  S: 可能被欺骗 → 缓解: API密钥 + 签名验证
  T: 数据可能被篡改 → 缓解: 请求签名 + 幂等设计
  R: 操作可能被抵赖 → 缓解: 完整审计日志 + 时间戳
  I: 信息可能泄露 → 缓解: HTTPS + 数据加密
```

#### 3.7.3 OWASP Threat Dragon

OWASP Threat Dragon是一款开源的Web/桌面威胁建模工具，支持STRIDE/DREAD模型，输出Markdown格式的威胁报告。

**核心优势：**
- 完全开源（Apache 2.0协议）
- 支持Web版和桌面版（Electron）
- 自动应用STRIDE模型
- 输出结构化的Markdown威胁报告
- 支持GitHub集成（直接存储模型文件）

### 3.8 威胁建模在不同架构中的应用

#### 3.8.1 微服务架构的威胁建模

微服务架构相比单体架构，信任边界更多，攻击面更广。威胁建模需要额外关注：

```text
微服务特有威胁：
├── 服务间通信安全
│   ├── 服务间未加密通信（明文传输）
│   ├── 服务身份伪造（未实施mTLS）
│   └── 消息队列数据泄露
│
├── API网关安全
│   ├── 路由规则绕过（访问未授权服务）
│   ├── 速率限制缺失（DDoS攻击）
│   └── 认证令牌泄露（JWT重放攻击）
│
├── 服务发现安全
│   ├── 服务注册信息篡改
│   ├── 未授权服务注册（恶意服务接入）
│   └── 服务发现中心单点故障
│
├── 分布式事务安全
│   ├── 数据一致性破坏
│   ├── 幂等性缺失（重复扣款）
│   └── 补偿事务逻辑漏洞
│
└── 容器与编排安全
    ├── 容器逃逸（共享内核漏洞）
    ├── Kubernetes RBAC配置错误
    └── 镜像供应链攻击
```

**微服务DFD的特殊考虑：**

```text
                        信任边界1
                           ║
┌────────┐  ──→  ╔═══════════════════╗  ──→  ┌────────┐
│ 外部   │       ║   API Gateway     ║       │ 用户   │
│ 用户   │ ←──   ║   (认证+路由)      ║ ←──   │ 服务   │
└────────┘       ╚═══════╤═══════════╝       └───┬────┘
                         ║  信任边界2              │
                         ↓                        ↓
                  ┌──────────────┐         ┌──────────────┐
                  │  服务注册中心  │←──────→ │  消息队列     │
                  │  (Consul/Nacos)│         │ (RabbitMQ)   │
                  └──────────────┘         └──────────────┘
                         ↑                        ↑
                         ║  信任边界3              ║
                         ↓                        ↓
                  ┌──────────────┐         ┌──────────────┐
                  │  MySQL集群    │         │  Redis集群    │
                  └──────────────┘         └──────────────┘
```

#### 3.8.2 云原生架构的威胁建模

云原生架构（Kubernetes + 容器）需要额外关注：

| 威胁层面 | 具体威胁 | 缓解措施 |
|----------|----------|----------|
| 容器镜像 | 使用含漏洞的基础镜像 | 镜像扫描（Trivy）、最小化基础镜像（distroless） |
| 容器运行时 | 容器逃逸、特权容器 | Pod Security Policy、非root用户、只读文件系统 |
| Kubernetes | RBAC权限过大、etcd泄露 | 最小权限RBAC、etcd加密、网络策略 |
| 服务网格 | mTLS配置错误、sidecar漏洞 | 自动mTLS、定期更新sidecar |
| 云服务 | IAM配置错误、S3公开访问 | IAM最小权限、存储桶策略审查 |
| CI/CD | 密钥泄露、构建环境污染 | 密钥管理（Vault）、构建环境隔离 |

#### 3.8.3 IoT系统的威胁建模

IoT系统因其物理属性和资源受限特性，威胁建模需要特别关注：

```text
IoT系统威胁建模要点：
├── 设备层
│   ├── 固件漏洞（未及时更新）
│   ├── 物理接口暴露（UART、JTAG调试接口）
│   ├── 预置凭证未修改（默认密码）
│   └── 侧信道攻击（功耗分析、电磁辐射）
│
├── 通信层
│   ├── 明文通信（MQTT未加密）
│   ├── 弱加密协议（WEP、DES）
│   ├── 中间人攻击（证书固定缺失）
│   └── 重放攻击（无时间戳/nonce）
│
├── 云平台层
│   ├── 设备身份伪造
│   ├── 数据泄露（传感器数据含隐私信息）
│   ├── DDoS（僵尸网络控制）
│   └── API安全
│
└── 移动应用层
    ├── 不安全的数据存储
    ├── 未验证的输入
    └── 不安全的认证机制
```

### 3.9 威胁情报与持续建模

#### 3.9.1 威胁情报集成

威胁建模不应是一次性活动，而应与威胁情报持续集成：

```text
威胁情报来源：
├── 公开情报源
│   ├── MITRE ATT&CK（攻击技术库）
│   ├── NVD/CVE（漏洞数据库）
│   ├── OWASP Top 10（Web安全风险）
│   └── CIS Benchmarks（安全配置基线）
│
├── 行业情报
│   ├── 行业ISAC（信息共享与分析中心）
│   ├── CERT/CC安全公告
│   └── 安全厂商威胁报告
│
├── 商业情报
│   ├── Recorded Future
│   ├── Mandiant Threat Intelligence
│   └── CrowdStrike Intelligence
│
└── 内部情报
    ├── 历史安全事件
    ├── 内部渗透测试结果
    └── 安全审计发现
```

#### 3.9.2 持续威胁建模

将威胁建模融入DevSecOps流水线：

```yaml
# 持续威胁建模的CI/CD集成示例（GitHub Actions）
name: Continuous Threat Modeling

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  threat-model:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # 代码化威胁建模
      - name: Run Pytm Threat Model
        run: |
          pip install pytm
          python threat_model.py --dfd > dfd.dot
          python threat_model.py --report threats.md

      # 自动对比变更
      - name: Compare with Previous Model
        run: |
          # 对比当前威胁模型与上次提交的差异
          git diff HEAD~1 -- threat_model.py
          # 如果关键威胁发生变化，触发告警
          python compare_threats.py --prev threats_prev.md --curr threats.md

      # SAST验证缓解措施
      - name: Semgrep SAST Scan
        uses: semgrep/semgrep-action@v1
        with:
          config: >-
            p/owasp-top-ten
            p/python
            p/security-audit

      # 依赖漏洞扫描
      - name: Trivy Vulnerability Scan
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: 'fs'
          scan-ref: '.'
          severity: 'CRITICAL,HIGH'
```

### 3.10 威胁建模最佳实践总结

| 实践 | 说明 | 优先级 |
|------|------|--------|
| **早期介入** | 在设计阶段（而非编码阶段）进行威胁建模 | 必须 |
| **聚焦关键资产** | 优先建模高价值、高风险模块 | 必须 |
| **跨职能团队** | 开发、安全、运维、产品共同参与 | 必须 |
| **持续更新** | 架构变更时同步更新威胁模型 | 必须 |
| **验证缓解** | 使用SAST/DAST/渗透测试验证缓解措施 | 必须 |
| **记录决策** | 记录风险接受决策和残余风险 | 推荐 |
| **自动化集成** | 将威胁建模集成到CI/CD流水线 | 推荐 |
| **威胁情报** | 结合MITRE ATT&CK等框架更新威胁库 | 推荐 |
| **培训推广** | 定期对开发团队进行威胁建模培训 | 建议 |

**关键认知：** 威胁建模不是安全团队的独角戏，而是整个开发团队的共同责任。最有效的威胁建模是嵌入到日常开发流程中的——每次设计评审、每次架构决策、每次新功能上线都应该包含安全思考。这不是额外的负担，而是构建可靠系统的内在需求。
