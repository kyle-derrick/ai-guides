## 概述与定位

锁机制是数据库并发控制中最古老、最基础的原语。在事务与并发控制的知识体系中，锁机制位于 ACID 特性之后、MVCC 和 2PL 之前——理解锁是理解后续所有并发控制方案的前提。

为什么需要锁？考虑一个简单的银行转账场景：事务 A 读取账户余额为 1000 元，事务 B 同时也读取余额为 1000 元。A 转出 500 元后写入余额 500 元，B 转出 300 元后写入余额 700 元。最终余额为 700 元——但实际应该只剩 200 元。这就是经典的**丢失更新（Lost Update）**问题。锁机制通过在事务访问数据前加锁，保证同一时刻只有一个事务能修改特定数据，从根本上消除这类并发异常。

本节将系统性地讲解锁的类型体系、兼容性矩阵、粒度层次、锁管理器的内部实现，以及锁与索引的关系——这是所有数据库开发者必须掌握的核心知识。

## 锁的类型体系

锁不是只有"共享锁"和"排他锁"两种。完整的锁类型体系包括行级锁、表级锁、间隙锁等多个层次，每种锁都有明确的语义和适用场景。

### 共享锁（Shared Lock，S Lock）

也叫读锁。多个事务可以同时持有共享锁，用于读操作。共享锁之间不互斥——这允许多个事务同时读取同一数据而不互相阻塞。

```sql
-- MySQL: 手动加共享锁
SELECT * FROM accounts WHERE id = 1 LOCK IN SHARE MODE;

-- MySQL 8.0+ 新语法
SELECT * FROM accounts WHERE id = 1 FOR SHARE;
```

**关键语义**：持有共享锁的事务可以读取数据，但不能修改。如果需要修改，必须先升级为排他锁。共享锁的兼容规则是：S 锁与 S 锁兼容，S 锁与 X 锁冲突。

### 排他锁（Exclusive Lock，X Lock）

也叫写锁。同一时刻只能有一个事务持有排他锁，用于写操作（INSERT、UPDATE、DELETE）或显式的 SELECT ... FOR UPDATE。排他锁与任何锁都冲突——无论是共享锁还是另一个排他锁。

```sql
-- MySQL: 显式加排他锁
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;

-- UPDATE/DELETE 操作自动加排他锁
UPDATE accounts SET balance = balance - 500 WHERE id = 1;
DELETE FROM orders WHERE id = 100;
```

**关键语义**：持有排他锁的事务可以读取和修改数据。在锁持有期间，其他事务既不能读（加共享锁）也不能写（加排他锁）。排他锁是保证数据一致性的最强约束。

### 锁兼容矩阵

两个事务对同一数据项请求锁时，是否允许取决于锁类型的组合。这个矩阵是理解所有锁协议的基石：

| 已持有 \ 请求 | S Lock | X Lock |
|:---:|:---:|:---:|
| **S Lock** | ✅ 兼容 | ❌ 冲突 |
| **X Lock** | ❌ 冲突 | ❌ 冲突 |
| **无锁** | ✅ 兼容 | ✅ 兼容 |

读写互斥是这个矩阵的核心规则：共享锁之间可以共存（多读不互斥），但排他锁与任何锁都互斥（写时禁止一切并发访问）。

**为什么是这个设计？** 如果允许多个写事务同时修改同一数据，最终结果取决于执行顺序，数据库无法保证一致性。如果允许读事务在写事务修改过程中读取，会读到不一致的中间状态（脏读）。因此，写-写互斥和读-写互斥是数据库一致性的基本保障。

### 意向锁（Intention Lock）

意向锁是很多人忽视的细节，但它们对数据库性能至关重要。意向锁是**表级锁**，用于标记事务接下来要在表中的某些行上加什么类型的锁。

| 意向锁类型 | 含义 | 兼容性 |
|:---:|:---:|:---:|
| 意向共享锁（IS） | 事务意图对某些行加 S 锁 | IS 与 IX 兼容 |
| 意向排他锁（IX） | 事务意图对某些行加 X 锁 | IS 与 IX 兼容 |

**意向锁存在的意义**：解决"行锁和表锁的冲突判断"问题。当事务要给整张表加 X 锁时（例如 ALTER TABLE），如果没有意向锁，数据库必须逐行检查是否有行级锁——这是 O(n) 操作。有了意向锁，只需检查表级是否有 IS 或 IX 锁——这是 O(1) 操作。

```sql
-- 事务 A：给某行加行级 X 锁
-- InnoDB 自动操作流程：
-- 1. 先在表级别获取 IX 锁
-- 2. 再在行级别获取 X 锁
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;  -- 自动获取 IX + 行X锁
-- ... 处理 ...
COMMIT;

-- 事务 B：尝试给整张表加表级 X 锁
-- InnoDB 自动操作流程：
-- 1. 检查表级是否有 IS 或 IX 锁 → 发现有 IX
-- 2. 直接判断冲突，不需要逐行扫描
ALTER TABLE accounts ADD INDEX idx_name(name);  -- 与事务 A 的 IX 冲突，等待
```

**意向锁兼容矩阵**：

| 已持有 \ 请求 | IS | IX | S | X |
|:---:|:---:|:---:|:---:|:---:|
| **IS** | ✅ | ✅ | ✅ | ❌ |
| **IX** | ✅ | ✅ | ❌ | ❌ |
| **S** | ✅ | ❌ | ✅ | ❌ |
| **X** | ❌ | ❌ | ❌ | ❌ |

注意：IS 和 IX 之间互相兼容——这允许一个事务对表中的某一行加 S 锁，同时另一个事务对同一表中的另一行加 X 锁。

### 自增锁（AUTO-INC Lock）

自增锁是一种特殊的表级锁，用于保护自增列（AUTO_INCREMENT）的值分配。在 InnoDB 中，自增锁的行为由 `innodb_autoinc_lock_mode` 参数控制：

| 锁模式 | 值 | 行为 | 并发性 | 安全性 |
|:---:|:---:|:---:|:---:|:---:|
| 传统模式 | 0 | 所有 INSERT 都加自增锁 | 低 | 最高（可预测） |
| 连续模式 | 1 | 简单 INSERT 不加锁，批量 INSERT 加自增锁 | 中 | 高 |
| 交错模式 | 2 | 所有 INSERT 都不加自增锁 | 最高 | 低（日志复制不安全） |

**为什么自增锁存在？** 考虑两个并发 INSERT 语句。如果没有自增锁，两个事务可能获取相同的自增值，导致主键冲突或数据覆盖。传统模式通过加锁保证自增值的连续分配，但代价是所有 INSERT 操作串行化。

### 间隙锁（Gap Lock）与临键锁（Next-Key Lock）

间隙锁和临键锁是 InnoDB 在 REPEATABLE READ 隔离级别下防止幻读的核心机制。这两个概念比较复杂，值得深入理解。

**间隙锁（Gap Lock）**：锁定索引记录之间的"间隙"——即索引值之间的范围，但不包括索引记录本身。间隙锁的唯一目的是阻止其他事务在间隙中插入新记录。

```sql
-- 假设 accounts 表中 id 为 1, 5, 10
-- 事务执行：
BEGIN;
SELECT * FROM accounts WHERE id = 7 FOR UPDATE;
-- InnoDB 会在 (5, 10) 这个间隙上加间隙锁
-- 其他事务无法在这个范围内插入新记录
-- 但可以修改 id=5 或 id=10 的现有记录
```

**临键锁（Next-Key Lock）**：是记录锁（Record Lock）和间隙锁的组合，锁定一个左开右闭的区间 `(a, b]`。InnoDB 在 REPEATABLE READ 隔离级别下的行锁默认就是临键锁。

```sql
-- 假设 accounts 表中 id 为 1, 5, 10
-- 事务执行：
BEGIN;
SELECT * FROM accounts WHERE id = 5 FOR UPDATE;
-- InnoDB 在 id=5 上加临键锁，锁定区间 (1, 5]
-- 这个锁同时锁住了 id=5 这条记录，以及 id=1 到 id=5 之间的间隙
```

**间隙锁的兼容规则**：间隙锁之间不冲突——两个事务可以在同一个间隙上加间隙锁。间隙锁的唯一冲突对象是"在间隙中插入记录"的操作（INSERT Intention Lock）。

**什么时候退化为记录锁？** 当查询命中的索引记录是唯一索引上的等值查询时，临键锁会退化为记录锁——因为唯一性已经保证了不会有其他记录落在同一个间隙中。

### 插入意向锁（Insert Intention Lock）

插入意向锁是间隙锁的一种特殊变种。当事务向间隙中插入新记录时，在插入操作等待间隙锁释放时，会获取插入意向锁。

```sql
-- 事务 A：持有间隙锁 (5, 10)
BEGIN;
SELECT * FROM accounts WHERE id BETWEEN 5 AND 10 FOR UPDATE;

-- 事务 B：尝试插入 id=7，需要等待间隙锁释放
BEGIN;
INSERT INTO accounts(id, balance) VALUES(7, 1000);
-- 此时事务 B 获取插入意向锁，进入等待状态
```

插入意向锁与普通的间隙锁不冲突——这意味着多个事务可以在同一个间隙的不同位置并发插入（只要间隙锁的范围不重叠）。这是 InnoDB 实现高并发插入的关键设计。

## 锁粒度层次

锁的粒度决定了锁影响的数据范围。粒度越粗，并发度越低但管理开销越小；粒度越细，并发度越高但管理开销越大。

### 粒度层次结构

InnoDB 的锁粒度从粗到细形成一个层次结构：

表锁（Table Lock）
 └── 意向锁（Intention Lock）—— IS/IX
      └── 行锁（Row Lock）—— S/X
           └── 记录锁（Record Lock）—— 锁定单条索引记录
           └── 间隙锁（Gap Lock）—— 锁定索引间隙
           └── 临键锁（Next-Key Lock）—— 记录锁 + 间隙锁
                └── 插入意向锁（Insert Intention Lock）

### 表锁（Table Lock）

表锁锁定整张表，是粒度最粗的锁。MySQL 的 MyISAM 引擎只支持表锁，InnoDB 同时支持表锁和行锁。

```sql
-- 手动加表锁
LOCK TABLES accounts READ;   -- 表级共享锁
LOCK TABLES accounts WRITE;  -- 表级排他锁

-- 以下操作会自动加表级锁
ALTER TABLE accounts ADD COLUMN age INT;  -- 表级 X 锁
ANALYZE TABLE accounts;                    -- 表级 S 锁
```

**表锁的性能影响**：一旦某个事务持有表级排他锁，所有其他事务对这张表的读写都会被阻塞。在高并发场景下，表锁会导致严重的性能瓶颈——这也是 InnoDB 引入行级锁的根本原因。

### 行锁（Row Lock）

行锁锁定单条索引记录，是 InnoDB 最常用的锁粒度。行锁的并发度远高于表锁，因为不同行之间的锁互不影响。

**关键细节：行锁基于索引实现**。InnoDB 的行锁是加在索引上的，而不是加在数据行上。这意味着：

- 如果查询使用了索引，行锁只锁定索引命中的行
- 如果查询没有使用索引（全表扫描），行锁会锁定所有行——退化为表锁
- 这是一个常见且致命的性能陷阱

```sql
-- 场景：accounts 表有主键索引 id，无其他索引

-- 使用索引 → 行锁生效
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;  -- 只锁 id=1 这一行

-- 不使用索引 → 行锁退化为表锁
SELECT * FROM accounts WHERE balance > 1000 FOR UPDATE;  -- 锁定所有行！
```

### 锁粒度的选择策略

| 粒度 | 并发度 | 管理开销 | 适用场景 |
|:---:|:---:|:---:|:---:|
| 表锁 | 低 | 低 | DDL 操作、全表扫描、读密集型 |
| 行锁 | 高 | 高 | OLTP、高并发读写 |
| 页锁 | 中 | 中 | PostgreSQL（默认页级锁） |

**实际选择原则**：InnoDB 的设计哲学是"尽可能用行锁"。只有在以下情况才会使用表锁：
1. DDL 操作（ALTER TABLE、DROP TABLE）
2. 显式 LOCK TABLES 语句
3. 查询没有命中任何索引时行锁退化

## 锁管理器的实现

锁管理器（Lock Manager）是数据库内核中的核心组件，负责锁的授予、等待和释放。理解锁管理器的内部实现，有助于深入理解锁等待、死锁检测等行为。

### 数据结构

锁管理器通常以哈希表实现，核心结构如下：

```python
class LockManager:
    """简化的锁管理器实现"""
    
    def __init__(self):
        # 哈希表：key=资源标识，value=锁请求队列
        self.lock_table = {}
    
    def request_lock(self, transaction_id, resource_id, lock_type):
        """请求锁"""
        if resource_id not in self.lock_table:
            self.lock_table[resource_id] = LockQueue()
        
        queue = self.lock_table[resource_id]
        
        # 检查是否与已有锁冲突
        if queue.can_grant(lock_type):
            queue.grant(transaction_id, lock_type)
            return LockResult.GRANTED
        else:
            # 加入等待队列
            queue.enqueue(transaction_id, lock_type)
            return LockResult.WAITING
    
    def release_lock(self, transaction_id, resource_id):
        """释放锁"""
        queue = self.lock_table[resource_id]
        queue.release(transaction_id)
        # 检查等待队列，决定下一个授予的对象
        queue.process_wait_queue()
```

**锁请求队列**的工作流程：
1. 新锁请求到达时，检查与已有锁的兼容性
2. 如果兼容，直接授予
3. 如果冲突，请求者进入等待队列
4. 当持有者释放锁时，管理器按照等待队列中的顺序（或优先级策略）决定下一个授予的对象
5. 重复步骤 2-4，直到队列为空

### 锁授予策略

当多个事务同时等待同一资源上的锁时，管理器需要决定按什么顺序授予。常用策略有：

| 策略 | 描述 | 优点 | 缺点 |
|:---:|:---:|:---:|:---:|
| FIFO | 先到先服务 | 公平，无饥饿 | 可能导致死锁 |
| 优先级 | 按事务优先级授予 | 关键事务优先 | 低优先级事务可能饥饿 |
| 超时 | 等待超时后回滚 | 避免长时间阻塞 | 可能误杀正常事务 |

MySQL InnoDB 采用 FIFO 策略，配合死锁检测机制（Wait-For Graph）来处理循环等待。

### InnoDB 锁管理器的内部实现

InnoDB 的锁管理器用两个哈希表管理锁：

- **Lock System Hash Table**：记录每个事务持有哪些锁
- **Lock Wait Hash Table**：记录哪些事务在等待哪些锁

```sql
-- 查看 InnoDB 的锁信息（MySQL 8.0+）
-- 当前持有和等待的锁
SELECT * FROM performance_schema.data_locks;
SELECT * FROM performance_schema.data_lock_waits;

-- 锁等待的可视化查询
SELECT
    r.trx_id AS waiting_trx,
    r.trx_mysql_thread_id AS waiting_thread,
    r.trx_query AS waiting_query,
    b.trx_id AS blocking_trx,
    b.trx_mysql_thread_id AS blocking_thread,
    b.trx_query AS blocking_query
FROM information_schema.innodb_lock_waits w
JOIN information_schema.innodb_trx b ON b.trx_id = w.blocking_trx_id
JOIN information_schema.innodb_trx r ON r.trx_id = w.requesting_trx_id;
```

## 锁与索引的关系

锁与索引的关系是 InnoDB 中最容易被忽视、也最容易引发性能问题的知识点。

### 行锁基于索引实现

InnoDB 的行锁不是加在物理行上的，而是加在索引上的。这意味着：

**场景一：使用主键索引 → 精确行锁**
```sql
-- accounts 表主键为 id
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;
-- 锁定：id=1 对应的主键索引记录
-- 影响范围：1 行
```

**场景二：使用二级索引 → 两步锁定**
```sql
-- accounts 表有二级索引 idx_name(name)
BEGIN;
SELECT * FROM accounts WHERE name = 'Alice' FOR UPDATE;
-- 第一步：在 idx_name 索引上锁定 name='Alice' 的记录
-- 第二步：在主键索引上锁定对应的行
-- 影响范围：2 个索引记录
```

**场景三：没有索引 → 全表扫描 → 锁定所有行**
```sql
-- accounts 表无 balance 索引
BEGIN;
SELECT * FROM accounts WHERE balance > 1000 FOR UPDATE;
-- InnoDB 无法使用索引定位，只能全表扫描
-- 扫描过程中锁定所有行 → 等效于表锁
-- 影响范围：全表所有行
```

### 索引选择对锁的影响

这个特性直接决定了索引设计的重要性：

| 查询条件 | 有索引 | 无索引 | 锁影响差异 |
|:---:|:---:|:---:|:---:|
| `WHERE id = 1` | 行锁（1行） | N/A（主键必有索引） | — |
| `WHERE name = 'Alice'` | 行锁（少量行） | 表锁（全表） | 并发度下降 100 倍+ |
| `WHERE age > 25` | 行锁（范围行） | 表锁（全表） | 并发度下降 100 倍+ |
| `WHERE status = 'active'` | 行锁（少量行） | 表锁（全表） | 并发度下降 100 倍+ |

**最佳实践**：在设计表结构时，必须考虑哪些列会被用于 WHERE 条件中的等值或范围查询，为这些列创建合适的索引。没有索引的列上的行锁操作会退化为表锁，在高并发场景下是致命的。

### 间隙锁与索引的关系

间隙锁也基于索引实现。只有在使用索引查询时，InnoDB 才会在索引间隙上加锁。如果查询没有命中任何索引，间隙锁也不会生效。

```sql
-- accounts 表有主键 id，无其他索引

-- 使用索引 → 间隙锁生效
BEGIN;
SELECT * FROM accounts WHERE id = 7 FOR UPDATE;
-- id=7 不存在，在间隙 (5, 10) 上加间隙锁
-- 其他事务无法插入 id 在 (5, 10) 之间的记录

-- 不使用索引 → 无间隙锁
BEGIN;
SELECT * FROM accounts WHERE balance = 0 FOR UPDATE;
-- 全表扫描，行锁退化为表锁，但不会加间隙锁
-- 其他事务可以插入新记录
```

## 锁的生命周期与释放

理解锁何时获取、何时释放，对于优化事务性能和排查锁冲突至关重要。

### 锁的获取时机

| 操作 | 锁类型 | 获取时机 |
|:---:|:---:|:---:|
| `SELECT ... FOR UPDATE` | 行级 X 锁 | 语句执行时 |
| `SELECT ... FOR SHARE` | 行级 S 锁 | 语句执行时 |
| `UPDATE` | 行级 X 锁 | 语句执行时（扫描到行时） |
| `DELETE` | 行级 X 锁 | 语句执行时（扫描到行时） |
| `INSERT` | 行级 X 锁 + 插入意向锁 | 语句执行时 |
| `SELECT ... LOCK IN SHARE MODE` | 行级 S 锁 | 语句执行时 |
| 普通 `SELECT` | 无锁（MVCC 快照读） | — |

### 锁的释放时机

锁的释放时机取决于隔离级别和 2PL 的变体：

| 隔离级别 | S 锁释放 | X 锁释放 |
|:---:|:---:|:---:|
| READ UNCOMMITTED | 语句结束时 | 语句结束时 |
| READ COMMITTED | 语句结束时 | 事务结束时 |
| REPEATABLE READ | 事务结束时 | 事务结束时 |
| SERIALIZABLE | 事务结束时 | 事务结束时 |

**READ COMMITTED vs REPEATABLE READ 的锁差异**：在 READ COMMITTED 隔离级别下，UPDATE 操作在语句执行完成后就释放不满足 WHERE 条件的行上的锁；而在 REPEATABLE READ 下，这些锁会持有到事务结束。这意味着 REPEATABLE READ 的锁持有时间更长，并发冲突的概率更大。

### 长事务与锁持有

长事务是锁冲突的最大元凶。事务持有锁的时间越长，其他事务等待锁的概率越大，系统整体性能越差。

```sql
-- ❌ 反面教材：长事务中包含大量操作
BEGIN;
-- 1. 读取数据（加锁）
SELECT * FROM orders WHERE status = 'pending' FOR UPDATE;
-- 2. 调用外部 API（可能耗时数秒）
-- application code: call external_api(...)
-- 3. 处理结果
UPDATE orders SET status = 'processed' WHERE id = 1;
-- 4. 大批量操作
UPDATE inventory SET stock = stock - 1 WHERE product_id IN (...);
-- 5. 最终提交
COMMIT;  -- 整个事务可能持续数十秒！
```

```sql
-- ✅ 正确做法：将非数据库操作移到事务外
-- 1. 先做外部调用
result = call external_api(order_id);
-- 2. 最后才开事务，快速完成
BEGIN;
UPDATE orders SET status = 'processed' WHERE id = 1;
COMMIT;  -- 事务可能只需要几毫秒
```

## 锁升级与降级

### 锁升级（Lock Escalation）

锁升级是指数据库将细粒度的锁自动转换为粗粒度的锁，以减少锁管理的内存开销。

大量行锁 → 升级为页锁 → 升级为表锁

**MySQL InnoDB 的策略**：InnoDB **不支持**锁升级。InnoDB 使用位图（bitmap）管理行锁，内存开销很小，不需要通过锁升级来节省资源。这是 InnoDB 相对于 SQL Server 的一个设计优势——SQL Server 在单个事务持有超过 5000 个锁时会自动升级为表锁。

**SQL Server 的锁升级**：
-- SQL Server 的锁升级行为
-- 当单个事务的锁数量超过阈值时，自动升级
-- 例如：UPDATE 10000 行 → 从 10000 个行锁升级为 1 个表锁

### 锁降级

锁降级是指将高级别的锁转换为低级别的锁，例如将排他锁降级为共享锁。MySQL InnoDB 支持在特定场景下的锁降级——在 READ COMMITTED 隔离级别下，不满足 WHERE 条件的行在扫描后立即释放锁（不再持有到事务结束）。

## 不同数据库的锁实现对比

| 特性 | MySQL InnoDB | PostgreSQL | SQL Server | Oracle |
|:---:|:---:|:---:|:---:|:---:|
| 行锁实现 | 基于索引 | 基于元组 | 行级锁表 | 基于数据块 |
| 间隙锁 | 支持（RR 级别） | 不支持 | 不支持 | 不支持 |
| 锁升级 | 不支持 | 不支持 | 自动升级 | 不支持 |
| 死锁检测 | Wait-For Graph | Wait-For Graph | 超时 + 检测 | 超时 |
| 默认隔离级别 | REPEATABLE READ | READ COMMITTED | READ COMMITTED | READ COMMITTED |
| MVCC 与锁 | 混合使用 | MVCC 优先 | 行版本控制 | MVCC |

**PostgreSQL 的独特设计**：PostgreSQL 默认使用 MVCC 读取数据，普通 SELECT 完全不加锁。只有在 SELECT FOR UPDATE/SHARE 时才加行级锁。PostgreSQL 不支持间隙锁，因此在 SERIALIZABLE 隔离级别下使用 SSI（可串行化快照隔离）而非间隙锁来防止幻读。

**SQL Server 的锁升级**：SQL Server 是唯一支持自动锁升级的主流数据库。当单个事务的锁数量超过阈值（默认 5000 个锁），会自动将行锁升级为表锁。这减少了内存开销，但可能导致严重的并发瓶颈。

## 常见误区与纠正

| 误区 | 事实 | 纠正方法 |
|:---:|:---:|:---:|
| "加了锁就一定安全" | 锁的粒度和范围同样重要；没有索引的列上加行锁会退化为表锁 | 确保查询条件命中索引 |
| "InnoDB 默认使用表锁" | InnoDB 默认使用行锁，只在特定场景下使用表锁 | 理解锁粒度层次 |
| "共享锁可以防止所有并发问题" | 共享锁只防止写-读冲突，不防止读-读不一致（需要 MVCC） | 根据需求选择合适的锁类型 |
| "SELECT 不需要关注锁" | 普通 SELECT 在 InnoDB 中通过 MVCC 读取，但 FOR UPDATE 的 SELECT 会参与锁竞争 | 区分快照读和当前读 |
| "事务越长越安全" | 事务越长，锁持有时间越长，并发度和性能越差 | 保持事务尽可能短小 |
| "间隙锁会影响所有插入操作" | 间隙锁只阻止在间隙范围内插入记录，不影响间隙外的插入 | 理解间隙锁的范围语义 |
| "隔离级别越高越好" | SERIALIZABLE 的锁开销最大，应根据业务需求选择适当的隔离级别 | 从最低可用隔离级别开始 |
| "死锁是 Bug" | 死锁是 2PL 的固有产物，不是程序错误，需要通过锁顺序或检测机制处理 | 设计时考虑死锁预防 |

## 实际应用：锁行为分析

### 完整的锁行为追踪

```sql
-- 环境准备
CREATE TABLE accounts (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    balance DECIMAL(10,2)
);
CREATE INDEX idx_name ON accounts(name);
INSERT INTO accounts VALUES (1, 'Alice', 1000), (2, 'Bob', 2000), (3, 'Carol', 500);

-- 事务 A（在终端 1 执行）
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;
-- 获取的锁：
--   1. 表级 IX 锁（意向排他锁）
--   2. 主键索引 id=1 上的 Record Lock（X 锁）

-- 事务 B（在终端 2 执行）
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;
-- 获取的锁：
--   1. 表级 IX 锁（与事务 A 兼容）
--   2. 主键索引 id=1 上的 Record Lock（X 锁）→ 与事务 A 冲突 → 等待

-- 事务 A 提交
COMMIT;
-- 事务 A 释放所有锁
-- 事务 B 获得锁，继续执行

-- 事务 B 提交
COMMIT;
```

### 验证间隙锁行为

```sql
-- 环境准备
CREATE TABLE scores (
    id INT PRIMARY KEY,
    score INT
);
INSERT INTO scores VALUES (1, 80), (5, 90), (10, 70);

-- 事务 A：查询不存在的值，触发间隙锁
BEGIN;
SELECT * FROM scores WHERE id = 7 FOR UPDATE;
-- id=7 不存在
-- 获取的锁：Gap Lock on (5, 10) — 左开右开区间

-- 事务 B：尝试在间隙内插入
BEGIN;
INSERT INTO scores(id, score) VALUES(8, 85);
-- 阻塞！等待事务 A 的间隙锁释放

-- 事务 C：在间隙外插入
BEGIN;
INSERT INTO scores(id, score) VALUES(3, 75);
-- 成功！id=3 不在间隙 (5, 10) 内

-- 事务 A 提交
COMMIT;
-- 事务 B 获得锁，插入成功
```

### 验证无索引导致的表锁退化

```sql
-- 环境准备
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    email VARCHAR(100)
);
-- 注意：没有 email 索引
INSERT INTO users VALUES (1, 'Alice', 'alice@example.com'),
                          (2, 'Bob', 'bob@example.com'),
                          (3, 'Carol', 'carol@example.com');

-- 事务 A：在无索引列上加锁
BEGIN;
SELECT * FROM users WHERE email = 'alice@example.com' FOR UPDATE;
-- InnoDB 需要全表扫描来找到 email = 'alice@example.com' 的行
-- 扫描过程中锁定所有行 → 等效于表锁

-- 事务 B：更新其他行
BEGIN;
UPDATE users SET name = 'Robert' WHERE id = 2;
-- 阻塞！因为事务 A 锁定了所有行（表锁退化）

-- 对比：如果 email 有索引
CREATE INDEX idx_email ON users(email);
-- 重新执行事务 A：只锁定 email='alice@example.com' 对应的行
-- 事务 B 更新 id=2 不会被阻塞
```

## 监控与诊断

### MySQL InnoDB 锁监控

```sql
-- 方法一：查看当前锁（MySQL 8.0+）
SELECT 
    lock_id, lock_type, table_name, index_name,
    lock_mode, lock_status, lock_data
FROM performance_schema.data_locks;

-- 方法二：查看锁等待关系
SELECT 
    requesting_tx.trx_id AS requesting_trx,
    requesting_tx.trx_query AS requesting_query,
    blocking_tx.trx_id AS blocking_trx,
    blocking_tx.trx_query AS blocking_query
FROM performance_schema.data_lock_waits w
JOIN information_schema.innodb_trx requesting_tx 
    ON requesting_tx.trx_id = w.REQUESTING_ENGINE_TRANSACTION_ID
JOIN information_schema.innodb_trx blocking_tx 
    ON blocking_tx.trx_id = w.BLOCKING_ENGINE_TRANSACTION_ID;

-- 方法三：通过 InnoDB Status 查看锁信息
SHOW ENGINE INNODB STATUS\G
-- 关注以下部分：
-- SEMAPHORES：等待的信号量
-- TRANSACTIONS：活跃事务和锁信息
-- LATEST DETECTED DEADLOCK：最近的死锁信息
```

### PostgreSQL 锁监控

```sql
-- 查看当前所有锁
SELECT 
    l.pid, l.locktype, l.mode, l.granted,
    a.query, a.state
FROM pg_catalog.pg_locks l
JOIN pg_catalog.pg_stat_activity a ON l.pid = a.pid
WHERE l.pid != pg_backend_pid()
ORDER BY l.pid;

-- 查看锁等待关系
SELECT 
    blocked.pid AS blocked_pid,
    blocked.query AS blocked_query,
    blocking.pid AS blocking_pid,
    blocking.query AS blocking_query
FROM pg_catalog.pg_locks AS blocked
JOIN pg_catalog.pg_stat_activity AS blocked_activity 
    ON blocked.pid = blocked_activity.pid
JOIN pg_catalog.pg_locks AS blocking 
    ON blocking.locktype = blocked.locktype
    AND blocking.relation = blocked.relation
    AND blocking.pid != blocked.pid
JOIN pg_catalog.pg_stat_activity AS blocking_activity 
    ON blocking.pid = blocking_activity.pid
WHERE NOT blocked.granted;
```

### 关键监控指标

| 指标 | 含义 | 告警阈值 | 优化方向 |
|:---:|:---:|:---:|:---:|
| 锁等待时间 | 事务等待锁的平均时长 | > 100ms | 优化事务设计、添加索引 |
| 锁等待次数 | 单位时间内发生的锁等待次数 | > 100次/min | 减少锁粒度、优化事务顺序 |
| 死锁频率 | 单位时间内的死锁次数 | > 5次/h | 统一锁获取顺序 |
| 锁持有时间 | 单个锁的平均持有时长 | > 1s | 缩短事务长度 |
| 当前锁数量 | 系统中同时持有的锁总数 | > 10000 | 减少事务并发数 |

## 总结

锁机制是数据库并发控制的基石，贯穿于所有事务操作的始终。理解锁机制的核心要点：

**类型体系**：共享锁（S）用于读，排他锁（X）用于写，意向锁（IS/IX）用于表级标记，间隙锁（Gap Lock）防止幻读，临键锁（Next-Key Lock）是 InnoDB 的默认行锁。每种锁都有明确的语义和兼容规则。

**粒度层次**：从表锁到行锁到间隙锁，粒度越细并发度越高但管理开销越大。InnoDB 的行锁基于索引实现——没有索引的操作会退化为表锁，这是最常见的性能陷阱。

**锁管理器**：以哈希表为核心数据结构，维护资源到锁请求队列的映射。采用 FIFO 授予策略，配合死锁检测机制处理循环等待。

**索引决定锁行为**：查询条件是否命中索引，直接决定了锁的范围和类型。设计表结构时必须考虑查询模式，为高频查询条件创建合适的索引。

**事务设计原则**：事务尽可能短小、减少锁的范围、统一锁获取顺序——这些比选择哪种锁策略更重要。锁机制的最终目标是在保证一致性的同时最大化并发度。
