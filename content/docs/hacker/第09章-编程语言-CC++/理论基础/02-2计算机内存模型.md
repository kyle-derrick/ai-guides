## 2. 计算机内存模型

内存是程序运行的物质基础。每一个变量的存储、每一次函数调用的上下文切换、每一个动态对象的生命周期，都发生在内存中。对于安全研究者而言，理解内存模型不仅是编程基本功，更是发现漏洞、构造利用链的根基——栈溢出、堆利用、信息泄露、ROP链构造，无一不建立在对内存布局的精确理解之上。

本章从硬件到软件、从物理到逻辑，系统性地讲解计算机内存模型的方方面面。先建立全局视角，再逐一深入每个内存区域的机制、保护措施和安全含义。

### 2.1 虚拟内存基础

#### 2.1.1 为什么需要虚拟内存

早期计算机直接使用物理地址，程序直接访问物理内存。这带来三个致命问题：

1. **地址冲突**：多个程序可能使用相同的物理地址，互相覆盖数据
2. **安全隔离缺失**：任何程序都能读写全部内存，无法防止恶意访问
3. **内存不足**：物理内存有限，无法同时加载所有程序

虚拟内存通过在程序和物理内存之间增加一层抽象来解决这些问题。每个程序看到的是一个独立的、连续的地址空间（虚拟地址空间），由操作系统和硬件协作将虚拟地址映射到物理地址。

#### 2.1.2 分页机制

虚拟内存的核心是**分页（Paging）**。虚拟地址空间被划分为固定大小的**页（Page）**，物理内存被划分为同样大小的**页帧（Page Frame）**。操作系统维护一张**页表（Page Table）**，记录每个虚拟页到物理页帧的映射关系。

在 x86-64 架构下，页面大小通常为 4KB（2^12 字节），虚拟地址空间为 48 位（256TB）。页表采用多级结构：

```text
虚拟地址 48位的拆分：
┌─────────┬─────────┬─────────┬─────────┬──────────┐
│ PML4(9) │ PDPT(9) │  PD(9)  │  PT(9)  │ Offset(12)│
└─────────┴─────────┴─────────┴─────────┴──────────┘
  第4级索引   第3级索引   第2级索引   第1级索引   页内偏移

四级页表转换过程：
CR3寄存器 → PML4表 → PDPT表 → PD表 → PT表 → 物理页帧
```

#### 2.1.3 TLB 与缓存

每次内存访问都查询四级页表会带来严重的性能开销。硬件通过 **TLB（Translation Lookaside Buffer）** 缓存最近使用的页表映射。TLB 是一种高速全相联缓存，典型容量为 64-1536 条目，命中延迟仅 1 个时钟周期。

```text
CPU访问内存的流程：
                ┌─────── 命中 ────→ 使用物理地址访问内存
 CPU ─→ TLB ──┤
                └─────── 未命中 ──→ 查页表 ─→ 更新TLB ─→ 访问内存
```

对安全研究而言，TLB 有两个重要含义：
- **ASLR 的粒度限制**：由于 TLB 按页缓存映射，ASLR 的随机化粒度通常为 4KB（页大小），而非单个字节
- **侧信道攻击**：通过观察 TLB 命中/未命中模式（如 Flush+Reload 攻击），可以推断其他进程的内存访问模式

#### 2.1.4 物理地址扩展与大页

- **PAE（Physical Address Extension）**：32 位系统通过 PAE 可访问超过 4GB 的物理内存，页表从两级扩展到三级，物理地址从 32 位扩展到 36 位
- **大页（Huge Pages）**：x86-64 支持 2MB（21 位偏移）和 1GB（30 位偏移）的大页，减少 TLB 缺失，常用于数据库等内存密集型应用。安全含义：大页不能使用 mprotect 修改权限，可能影响某些利用技术

### 2.2 进程内存布局

#### 2.2.1 Linux x86-64 进程地址空间全貌

Linux 下每个进程拥有独立的 48 位虚拟地址空间（理论 256TB，实际使用范围受内核配置限制）。下图展示了一个典型 C 程序的完整内存布局：

│   │ 函数栈帧 N           │     │
│   ├──────────────────────┤     │
│   │ 函数栈帧 N-1         │     │    ← RBP/RSP 在此工作
│   ├──────────────────────┤     │
│   │        ...           │     │
│   └──────────────────────┘     │
│                                │
│         空闲区域               │ ← 栈与堆之间的动态增长空间
│                                │
│   ┌──────────────────────┐     │
│   │        ...           │     │
│   ├──────────────────────┤     │

│   ├──────────────────────┤     │
│   │        ...           │     │

│   │        ...           │     │
│   └──────────────────────┘     │

├────────────────────────────────┤

#### 2.2.2 各段的属性与安全含义

| 段名 | 内容 | 读 | 写 | 执行 | 安全含义 |
|------|------|----|----|------|----------|
| `.text` | 机器指令 | ✓ | ✗ | ✓ | 只读防止代码篡改，可执行是 ROP 的前提 |
| `.rodata` | 常量数据 | ✓ | ✗ | ✗ | 字符串常量、跳转表，信息泄露源 |
| `.data` | 已初始化全局变量 | ✓ | ✓ | ✗ | GOT/PLT 表在这里，是 GOT overwrite 的目标 |
| `.bss` | 未初始化全局变量 | ✓ | ✓ | ✗ | 未初始化变量可泄露堆栈残留数据 |
| `Heap` | 动态分配 | ✓ | ✓ | ✗ | UAF、double free、堆溢出的主战场 |
| `Stack` | 函数局部变量 | ✓ | ✓ | ✗* | 缓冲区溢出、栈破坏的主战场 |
| `mmap` | 映射区域 | 视情况 | 视情况 | 视情况 | 共享库、RWX 区域（JIT 场景） |

*栈默认不可执行，但某些场景（如 execstack）可以开启。

### 2.3 栈（Stack）详解

栈是安全研究中最关键的内存区域之一。绝大多数经典漏洞（栈溢出、格式化字符串、ROP）都与栈有关。

#### 2.3.1 栈的基本特性

| 特性 | 说明 |
|------|------|
| 管理方式 | 自动管理——函数进入时分配，退出时释放 |
| 增长方向 | 从高地址向低地址增长（x86/x86-64/ARM） |
| 数据结构 | LIFO（后进先出） |
| 分配速度 | 极快——只需移动栈指针（一条 `sub rsp, N`） |
| 默认大小 | Linux 下通常 8MB（可通过 `ulimit -s` 查看） |
| 存储内容 | 局部变量、函数参数、返回地址、保存的寄存器、栈对齐填充 |

#### 2.3.2 栈帧结构详解

每次函数调用都会在栈上创建一个**栈帧（Stack Frame）**。以 x86-64 为例，一个完整的栈帧结构如下：

```text
高地址（调用者栈帧）
┌──────────────────────────────────┐
│  第7+个参数（如有）               │ ← [RBP + 32], [RBP + 40]...
│  （x86-64 前6个参数用寄存器传递） │
├──────────────────────────────────┤
│  返回地址 (8字节)                 │ ← [RBP + 8] — 栈溢出的首要目标
├──────────────────────────────────┤
│  保存的 RBP (8字节)               │ ← [RBP] — 栈帧基址指针
├──────────────────────────────────┤  ← RBP 指向这里
│  局部变量区                       │
│  ┌────────────────────────────┐  │
│  │ buffer[64] (64字节)        │  │ ← [RBP - 64] — 溢出源
│  ├────────────────────────────┤  │
│  │ canary (8字节)             │  │ ← [RBP - 8] — 栈保护
│  └────────────────────────────┘  │
├──────────────────────────────────┤
│  可能的对齐填充                   │
└──────────────────────────────────┘ ← RSP 指向这里（栈顶）
低地址
```

#### 2.3.3 函数调用全过程

以一段简单代码为例，完整追踪函数调用的每一步：

```c
long add(long a, long b) {
    long result = a + b;
    return result;
}

int main() {
    long x = add(1, 2);
    return 0;
}
```

对应的汇编（gcc -O0 -fno-stack-protector，为教学简化）：

```asm
; ===== main 函数 =====
main:
    push rbp                   ; 保存 main 的栈帧基址
    mov  rbp, rsp              ; 建立 main 的栈帧
    sub  rsp, 0x10             ; 分配局部变量空间

    ; 调用 add(1, 2)
    mov  esi, 2                ; 第2个参数 → rsi
    mov  edi, 1                ; 第1个参数 → rdi
    call add                   ; ① 将返回地址压栈 ② 跳转到 add
    mov  [rbp-8], rax          ; 保存返回值到局部变量 x

    mov  eax, 0                ; 返回值
    leave                      ; 等价于 mov rsp,rbp + pop rbp
    ret                        ; 弹出返回地址，跳转回去

; ===== add 函数 =====
add:
    push rbp                   ; ① 保存调用者的 RBP
    mov  rbp, rsp              ; ② 建立新栈帧基址
    sub  rsp, 0x10             ; ③ 为局部变量分配空间

    mov  [rbp-0x10], edi       ; 保存参数 a
    mov  [rbp-0x0c], esi       ; 保存参数 b
    mov  rdx, [rbp-0x10]
    add  rdx, [rbp-0x0c]
    mov  [rbp-0x08], rdx       ; result = a + b

    mov  rax, [rbp-0x08]       ; 返回值放入 rax
    leave                      ; ④ 恢复 RSP 和 RBP
    ret                        ; ⑤ 弹出返回地址，跳回调用者
```

```text
执行 call add 时的栈状态变化：

执行前（main 栈帧）：        执行后（进入 add）：
RSP ──→ ┌──────────┐         RSP ──→ ┌──────────┐
         │ 局部变量  │                  │ 返回地址   │ ← call 自动压入
         │ ...      │         RBP ──→ ├──────────┤
         │ 保存RBP  │         (旧RBP)  │ 保存的RBP │ ← push rbp
         └──────────┘                  │ 局部变量   │ ← sub rsp, 0x10
                                       └──────────┘
```

#### 2.3.4 x86-64 调用约定

调用约定规定了函数参数如何传递、返回值如何获取、哪些寄存器由谁负责保存。x86-64 Linux 遵循 **System V AMD64 ABI**：

```text
参数传递规则（System V AMD64 ABI）：
┌──────────┬────────────┬──────────────┐
│ 参数顺序  │ 整数/指针   │ 浮点数        │
├──────────┼────────────┼──────────────┤
│ 第1个     │ RDI        │ XMM0         │
│ 第2个     │ RSI        │ XMM1         │
│ 第3个     │ RDX        │ XMM2         │
│ 第4个     │ RCX        │ XMM3         │
│ 第5个     │ R8         │ XMM4         │
│ 第6个     │ R9         │ XMM5         │
│ 第7+个    │ 栈传递      │ 栈传递        │
└──────────┴────────────┴──────────────┘

返回值：
  整数/指针 → RAX（64位）；128位用 RDX:RAX
  浮点      → XMM0

寄存器保存责任：
  调用者保存（caller-saved）：RAX, RCX, RDX, RSI, RDI, R8-R11, XMM0-XMM15
  被调用者保存（callee-saved）：RBX, RBP, R12-R15, RSP
```

Windows x64 调用约定的差异：

| 差异点 | System V (Linux/macOS) | Microsoft x64 (Windows) |
|--------|----------------------|------------------------|
| 前4个整数参数 | RDI, RSI, RDX, RCX | RCX, RDX, R8, R9 |
| 栈空间分配 | 仅分配第7+参数 | 始终为前4个参数保留影子空间 |
| 红区 | 128字节（RSP以下） | 无红区 |
| XMM保存 | XMM0-XMM15 均 caller-saved | XMM6-XMM15 callee-saved |

#### 2.3.5 栈溢出原理

当局部变量（尤其是数组/缓冲区）的写入超出分配的空间时，就会覆盖栈帧中的相邻数据。最关键的覆盖目标是**返回地址**：

```c
#include <stdio.h>
#include <string.h>

void vulnerable() {
    char buffer[64];
    gets(buffer);  // 危险！没有边界检查
}

void target() {
    printf("被劫持了！\n");
}

int main() {
    vulnerable();
    printf("正常返回\n");
    return 0;
}
```

```text
栈溢出示意图：

正常调用 vulnerable() 的栈帧：
┌─────────────────┐ 高地址
│  返回地址(main)  │ ← 正常情况下返回到 main
├─────────────────┤
│  保存的 RBP      │
├─────────────────┤
│  buffer[64]      │ ← gets() 写入起点
└─────────────────┘ 低地址

输入超过 72 字节（64+8）后的栈帧：
┌─────────────────┐ 高地址
│  target() 地址   │ ← 被攻击者覆盖！
├─────────────────┤
│  覆盖数据AAAA    │ ← 覆盖了保存的 RBP
├─────────────────┤
│  AAAAAAAAAA...   │ ← 溢出 buffer
└─────────────────┘

ret 指令执行时，弹出的不是 main 的地址，而是 target() 的地址，
程序跳转到 target() 执行。
```

#### 2.3.6 栈保护机制

现代系统部署了多层栈保护机制：

**Stack Canary（栈金丝雀）**

在返回地址之前插入一个随机值（canary），函数返回前检查该值是否被修改：

```c
// 编译器生成的带 canary 的函数伪代码
void vulnerable() {
    long canary = __stack_chk_guard;  // 从 TLS 读取 canary 值
    char buffer[64];
    gets(buffer);                     // 溢出会覆盖 canary
    if (canary != __stack_chk_guard)  // 检查 canary 是否被修改
        __stack_chk_fail();           // 检测到栈破坏，终止程序
}
```

```asm
; 带 canary 的函数序言/尾声
vulnerable:
    push rbp
    mov  rbp, rsp
    sub  rsp, 0x50
    mov  rax, fs:[0x28]        ; 从 TLS 读取 canary
    mov  [rbp-8], rax          ; 放在局部变量区顶部
    ; ... 函数体 ...
    mov  rax, [rbp-8]
    xor  rax, fs:[0x28]        ; 与原始值比较
    jne  __stack_chk_fail      ; 不匹配则终止
    leave
    ret
```

Canary 的绕过思路（了解原理，防御更重要）：
- 逐字节爆破（fork 场景下子进程 canary 相同）
- 信息泄露（printf 泄露 canary 值）
- 覆盖 canary 之前的数据而不触碰 canary

**NX/DEP（不可执行栈）**

标记栈内存为不可执行（`RW-` 而非 `RWX`），阻止在栈上执行 shellcode：

```bash
# 检查 NX 状态
readelf -l ./target | grep GNU_STACK
# GNU_STACK  RW-  → NX 已开启（不可执行）
# GNU_STACK  RWX  → NX 未开启（可执行，危险）

# GCC 默认开启 NX
gcc -o target target.c              # 默认 NX 开启
gcc -z execstack -o target target.c # 关闭 NX（不推荐）
```

**ASLR（地址空间布局随机化）**

每次运行程序时，栈、堆、共享库的基地址随机化，使攻击者无法预测目标地址：

```bash
# 查看 ASLR 状态
cat /proc/sys/kernel/randomize_va_space
# 0 = 关闭
# 1 = 部分开启（栈、共享库、VDSO）
# 2 = 完全开启（堆也随机化）

# 临时设置
echo 2 > /proc/sys/kernel/randomize_va_space

# 验证随机化效果
for i in $(seq 1 5); do cat /proc/self/maps | grep stack; done
# 每次运行栈地址都不同
```

### 2.4 堆（Heap）详解

堆是动态内存分配的主要区域，也是现代漏洞利用最复杂、最精妙的战场。

#### 2.4.1 堆的基本特性

| 特性 | 说明 |
|------|------|
| 管理方式 | 手动管理——malloc/free 或 new/delete |
| 增长方向 | 从低地址向高地址增长（通过 brk/sbrk 系统调用扩展） |
| 分配粒度 | 16 字节对齐（glibc） |
| 默认大小 | 初始很小，按需增长 |
| 存储内容 | 动态分配的对象、缓冲区、字符串 |
| 碎片问题 | 频繁分配释放导致内存碎片化 |

#### 2.4.2 Linux 内存分配器概览

Linux 下有多种堆分配器实现，安全特性各不相同：

| 分配器 | 使用场景 | 特点 |
|--------|----------|------|
| glibc ptmalloc | 大多数 Linux 发行版默认 | 分区锁、per-thread arena、复杂 bin 系统 |
| jemalloc | FreeBSD、Android、Firefox | 大小类分级、arena 隔离、较少碎片 |
| tcmalloc | Google 基础设施 | 线程缓存、高性能、低锁争用 |
| musl libc allocator | Alpine Linux 等 | 简单、安全性较好 |
| scudo | Android 11+、LLVM | 专为安全设计，元数据校验 |

#### 2.4.3 glibc ptmalloc 核心机制

**Chunk 结构**

glibc 管理的堆内存以 **chunk** 为基本单位。每个 chunk 有一个 16 字节（x86-64）的头部元数据：

```text
已分配的 chunk：
┌──────────────────────────────────┐ ← chunk 指针（malloc 返回值 - 16）
│  prev_size (8字节)               │ ← 前一个 chunk 如果空闲，存储其大小
├──────────────────────────────────┤
│  size (8字节) + flags (低3位)    │ ← chunk 大小（含头部，8字节对齐）
│  [A|M|P]                         │   A=arena, M=mmap'd, P=prev_inuse
├──────────────────────────────────┤ ← malloc 返回的指针指向这里
│  用户数据区域                     │
│  (fd/bk 指针当 chunk 空闲时       │
│   会覆盖此区域)                   │
│                                  │
└──────────────────────────────────┘

空闲的 chunk：
┌──────────────────────────────────┐
│  prev_size                       │
├──────────────────────────────────┤
│  size + flags                    │
├──────────────────────────────────┤
│  fd (forward pointer, 8字节)     │ → 指向 bin 中下一个空闲 chunk
├──────────────────────────────────┤
│  bk (backward pointer, 8字节)    │ → 指向 bin 中上一个空闲 chunk
├──────────────────────────────────┤
│  fd_nextsize (large bin 才有)    │
├──────────────────────────────────┤
│  bk_nextsize                     │
└──────────────────────────────────┘
```

**Bin 系统**

释放的 chunk 根据大小被放入不同的 bin（链表结构）进行管理：

```text
┌─────────────────────────────────────────────────────────────┐
│  tcache bins (per-thread, 64个单链表, 每个最多7个chunk)       │
│  大小范围：0x20 ~ 0x410 (x86-64)                            │
│  LIFO 结构，无合并检查，速度最快                               │
│  安全风险：double free 检测较弱                                │
├─────────────────────────────────────────────────────────────┤
│  fast bins (10个单链表, 小chunk, 不合并)                      │
│  大小范围：0x20 ~ 0xb0 (x86-64)                              │
│  LIFO 结构，释放时不检查相邻 chunk 是否空闲                     │
│  适用场景：频繁分配释放小对象                                    │
├─────────────────────────────────────────────────────────────┤
│  unsorted bin (1个双向链表)                                   │
│  所有非 fast/tcache 的 chunk 释放后先进入 unsorted bin        │
│  malloc 时遍历 unsorted bin，按大小分类放入 small/large bin    │
│  类似 LRU 缓存机制                                           │
├─────────────────────────────────────────────────────────────┤
│  small bins (62个双向链表, ≤ 0x400字节)                       │
│  FIFO 结构，同一 bin 中所有 chunk 大小相同                     │
│  精确匹配，不会切割                                           │
├─────────────────────────────────────────────────────────────┤
│  large bins (63个双向链表, > 0x400字节)                       │
│  双向链表 + fd_nextsize/bk_nextsize 排序                     │
│  同一 bin 中 chunk 大小不同，可能切割                          │
├─────────────────────────────────────────────────────────────┤
│  top chunk (堆顶剩余空间)                                     │
│  如果所有 bin 都没有合适的 chunk，就从 top chunk 切割          │
│  top chunk 不足时，通过 brk/mmap 扩展堆                        │
└─────────────────────────────────────────────────────────────┘
```

**malloc 分配流程简化**

```text
malloc(size):
  1. 计算实际需要的 chunk 大小（含头部，16字节对齐）
  2. size ≤ 0x410 ? 检查 tcache → 命中则直接返回
  3. size ≤ 0xb0 ? 检查 fastbin → 命中则返回
  4. 检查 small bin → 命中则返回
  5. 检查 unsorted bin:
     a. 遍历 unsorted bin，找到精确匹配则返回
     b. 不匹配的 chunk 分类放入 small/large bin
     c. 遍历过程中找到 best fit 则切割并返回
  6. 检查 large bin → 找到 best fit 则切割
  7. 从 top chunk 切割
  8. top chunk 不够 → 调用 sbrk/brk 扩展堆，然后切割
  9. 都不够 → 调用 mmap 分配独立的匿名映射区域
```

**free 释放流程简化**

```text
free(ptr):
  1. 检查指针有效性
  2. chunk_size ≤ 0x410 且 tcache 对应 bin 未满( <7)?
     → 放入 tcache（不合并）
  3. 检查是否可以与相邻空闲 chunk 合并:
     a. 下一个 chunk 是空闲的？合并
     b. 上一个 chunk 是空闲的？合并
  4. 合并后的 chunk 放入 unsorted bin
  5. 如果合并后触及 top chunk，合并为新的 top chunk
```

#### 2.4.4 常见堆漏洞类型

```c
// 1. 堆溢出（Heap Overflow）
char *buf = malloc(0x20);
gets(buf);  // 可以写入超过 0x20 字节，覆盖相邻 chunk 的元数据

// 2. Use-After-Free（UAF）
char *p = malloc(0x20);
free(p);       // p 指向的内存被释放
// p 仍然可用（悬垂指针），但内存可能已被重新分配
strcpy(p, "controlled data");  // 写入已被重新分配的 chunk

// 3. Double Free
char *p = malloc(0x20);
free(p);
free(p);  // 同一个 chunk 被释放两次，导致 bin 链表环路

// 4. Off-By-One
char buf[8];
buf[8] = '\x00';  // 多写了一个字节，可能覆盖 prev_inuse 标志

// 5. Uninitialized Memory
char *p = malloc(0x100);
free(p);
char *q = malloc(0x100);  // 可能返回同一个地址
// q 中可能残留 p 写入的数据（信息泄露）
```

#### 2.4.5 堆保护机制

glibc 从 2.26+ 引入了多种安全检查：

| 保护机制 | 版本 | 说明 |
|----------|------|------|
| tcache double free 检测 | 2.29+ | 检测 tcache 中的 double free（检查 key 字段） |
| tcache perthread_struct 保护 | 2.32+ | tcache 元数据使用 XOR 混淆 |
| fastbin double free 检测 | 2.25+ | 检测连续两个相同 chunk 在 fastbin |
| unlink 安全检查 | 2.3.4+ | 检查 fd->bk == p && bk->fd == p |
| size 检查 | 各版本 | 检查 chunk size 是否合法 |
| ASLR | 内核 | 堆地址随机化 |

### 2.5 全局区：.data、.bss、.rodata

#### 2.5.1 三个段的区分

```c
// .rodata 段：只读数据（字符串常量、const 修饰的全局变量）
const int VERSION = 1;              // 编译器可能放 .rodata
char *msg = "Hello";               // "Hello" 在 .rodata，msg 指针在 .data
#define WELCOME "Welcome"          // 宏展开后字符串在 .rodata

// .data 段：已初始化的非零全局/静态变量
int global_init = 42;              // 在 .data 段
static int static_init = 100;      // 在 .data 段
int *ptr = &global_init;           // 指针本身在 .data 段

// .bss 段：未初始化或零初始化的全局/静态变量
int global_uninit;                 // 在 .bss 段，运行时初始化为 0
static int static_uninit;          // 在 .bss 段
char buffer[4096];                 // 在 .bss 段
int global_zero = 0;               // 零初始化，也在 .bss 段
```

#### 2.5.2 安全含义

**GOT/PLT 表**：`.data` 段中最重要的安全相关结构是 **GOT（Global Offset Table）**。它存储了动态链接函数的实际地址，攻击者如果能覆写 GOT 表项，就能劫持任意函数调用：

```c
调用 printf 的 PLT/GOT 机制：
printf@plt:
    jmp *printf@got       ; 从 GOT 取出 printf 的真实地址并跳转
    push index             ; 如果 GOT 未填充，触发动态链接
    jmp resolver

攻击思路（GOT overwrite）：
1. 找到 printf@plt 在 GOT 中的表项地址
2. 通过某个漏洞将 GOT 中 printf 的地址改为 system
3. 之后程序调用 printf("ls") 时，实际执行的是 system("ls")
```

**RELRO 保护**：

```bash
# Partial RELRO（默认）
gcc -o target target.c
# 只有 .got.plt 部分只读，GOT 仍然可写

# Full RELRO（推荐）
gcc -Wl,-z,relro,-z,now -o target target.c
# 所有 GOT 表项在程序启动时立即解析，之后标记为只读
# 代价：启动稍慢（需要解析所有符号）

# 检查 RELRO 状态
readelf -l ./target | grep GNU_RELRO
checksec ./target       # peda/pwntools
```

#### 2.5.3 BSS 段的安全风险

未初始化的全局变量可能导致信息泄露。BSS 段在程序加载时被零初始化，但在某些场景下（如 `execve` 后的残留、共享内存），可能包含上一个进程的数据。

```c
// 信息泄露示例
char global_buffer[64];  // BSS 段
void secret() {
    char local_secret[] = "password123";
    memcpy(global_buffer, local_secret, 12);
}
void cleanup() {
    memset(global_buffer, 0, 64);  // 正确清理
}
// 如果 secret() 执行后 cleanup() 未被调用
// 其他代码路径可能读到 global_buffer 中的残留数据
```

### 2.6 内存映射（mmap）与共享库

#### 2.6.1 mmap 系统调用

`mmap` 是比 `malloc` 更底层的内存分配机制。`malloc` 内部在分配大块内存（通常 > 128KB）时会使用 `mmap`：

```c
#include <sys/mman.h>

// 基本用法：分配匿名内存
void *ptr = mmap(
    NULL,                // 内核自动选择地址
    4096,                // 大小（页对齐）
    PROT_READ | PROT_WRITE,  // 权限：可读可写
    MAP_PRIVATE | MAP_ANONYMOUS,  // 私有匿名映射
    -1,                  // 无文件描述符
    0                    // 偏移量
);

// 映射文件
int fd = open("data.bin", O_RDONLY);
void *file_mem = mmap(NULL, file_size, PROT_READ, MAP_PRIVATE, fd, 0);

// 设置执行权限（用于 JIT 或 shellcode）
void *exec_mem = mmap(NULL, 4096,
    PROT_READ | PROT_WRITE | PROT_EXEC,  // RWX
    MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);

// 释放映射
munmap(ptr, 4096);
```

#### 2.6.2 共享库加载

动态链接的共享库（`.so` 文件）通过 mmap 映射到进程地址空间：

```text
共享库在进程内存中的布局（以 libc 为例）：

7f1a2c000000-7f1a2c021000 r-xp ... /libc.so.6  ← .text（代码，只读可执行）
7f1a2c221000-7f1a2c222000 r--p ... /libc.so.6  ← .rodata（只读数据）
7f1a2c222000-7f1a2c224000 rw-p ... /libc.so.6  ← .data/.bss（可读写）

安全含义：
- libc 的地址是 ASLR 随机化的
- 如果泄露了 libc 中某个函数的地址，就能计算出 libc 基址
- 得到 libc 基址后，可以计算 system()、"/bin/sh" 等地址
- 这是 ret2libc 攻击的基础
```

#### 2.6.3 ASLR 随机化效果

不同内存区域的 ASLR 随机化精度不同：

| 区域 | 随机化熵（位数） | 说明 |
|------|-----------------|------|
| 栈 | ~19位（Linux x86-64） | 每次运行随机偏移 |
| 堆 | ~13位 | brk 分配的堆 |
| mmap 区域 | ~28位 | 共享库、mmap 分配 |
| 主程序基址 | 无（PIE 关闭）/ ~28位（PIE 开启） | 位置无关可执行文件 |

```bash
# 编译为 PIE 可执行文件（默认开启）
gcc -o target target.c -pie -fPIE

# 关闭 PIE（便于调试，但安全性降低）
gcc -o target target.c -no-pie

# 检查 PIE 状态
file ./target
# "ELF 64-bit LSB shared object" → PIE
# "ELF 64-bit LSB executable" → 非 PIE
```

### 2.8 C/C++ 内存模型（并发视角）

C11/C++11 引入了正式的**内存模型（Memory Model）**，定义了多线程环境下内存访问的可见性和顺序保证。这与前面讨论的物理/虚拟内存模型是不同层面的概念——前者解决的是"多线程如何安全地共享内存"，后者解决的是"内存如何组织和保护"。

#### 2.8.1 数据竞争与未定义行为

当多个线程同时访问同一内存地址，且至少一个是写操作，且没有使用同步机制时，就会发生**数据竞争（Data Race）**。数据竞争在 C/C++ 中是**未定义行为（UB）**——编译器可以做出任意假设，生成不可预期的代码。

```c
// 有数据竞争的代码（UB）
int counter = 0;  // 共享变量

void *thread_func(void *arg) {
    for (int i = 0; i < 1000000; i++) {
        counter++;  // 非原子操作：读-改-写
    }
    return NULL;
}
// 两个线程同时执行 → 结果不确定，可能是 1000000 到 2000000 之间的任意值
```

#### 2.8.2 原子操作与内存序

C11 提供了 `_Atomic` 类型和 `<stdatomic.h>` 头文件，支持原子操作和指定内存序：

```c
#include <stdatomic.h>

atomic_int counter = ATOMIC_VAR_INIT(0);

// 六种内存序（从弱到强）
atomic_load_explicit(&counter, memory_order_relaxed);    // 无顺序保证，最快
atomic_load_explicit(&counter, memory_order_acquire);    // 读屏障
atomic_load_explicit(&counter, memory_order_consume);    // 数据依赖顺序
atomic_store_explicit(&counter, 1, memory_order_release); // 写屏障
atomic_store_explicit(&counter, 1, memory_order_acq_rel); // 读写屏障
atomic_fetch_add_explicit(&counter, 1, memory_order_seq_cst); // 顺序一致性，最慢但最安全

// 简写（默认 memory_order_seq_cst）
atomic_int counter;
atomic_store(&counter, 1);
int val = atomic_load(&counter);
atomic_fetch_add(&counter, 1);
```

**内存序含义**：

| 内存序 | 保证 | 典型使用场景 |
|--------|------|-------------|
| `relaxed` | 仅原子性，无顺序 | 计数器、统计 |
| `acquire` | 读操作之后的读写不会重排到此之前 | 加锁 |
| `release` | 写操作之前的读写不会重排到此之后 | 解锁 |
| `acq_rel` | 同时具有 acquire 和 release | read-modify-write |
| `seq_cst` | 全局顺序一致 | 默认选择，最安全 |

#### 2.8.3 happens-before 关系

C11 内存模型的核心是 **happens-before** 关系——定义了操作之间的可见性保证：

```c
// 线程 1
data = 42;                              // (A)
atomic_store_explicit(&flag, 1, memory_order_release);  // (B)

// 线程 2
while (atomic_load_explicit(&flag, memory_order_acquire) != 1)  // (C)
    ;
printf("%d\n", data);                    // (D) 保证读到 42

// happens-before 关系：
// A sequenced-before B (同一线程)
// B synchronizes-with C (release-acquire 对)
// C sequenced-before D (同一线程)
// 因此 A happens-before D，data 的写入对 D 可见
```

### 2.9 缓存层次结构与内存性能

#### 2.9.1 缓存层次

CPU 缓存不在安全研究的核心范围内，但理解缓存有助于理解侧信道攻击（如 Spectre/Meltdown）和性能优化：

```text
CPU ─→ L1 Cache (32KB, ~1ns, per-core)
     ─→ L2 Cache (256KB, ~4ns, per-core)
     ─→ L3 Cache (8-32MB, ~12ns, shared)
     ─→ 主内存 (4-64GB, ~100ns)
     ─→ SSD (~100μs)
     ─→ HDD (~10ms)
```

缓存行（Cache Line）大小通常为 64 字节。当 CPU 访问一个地址时，整个缓存行被加载。这意味着：
- **空间局部性**：访问数组中相邻元素几乎无额外开销
- **伪共享（False Sharing）**：不同线程修改同一缓存行中的不同变量会导致缓存失效风暴

#### 2.9.2 侧信道攻击简介

缓存的时序差异可以被利用来推断其他程序的内存访问模式：

- **Flush+Reload**：攻击者刷新目标缓存行，等待目标执行后重新加载，通过加载时间推断目标是否访问了该缓存行
- **Prime+Probe**：攻击者填充缓存集，等待目标执行后检查哪些缓存行被驱逐
- **Spectre**：利用分支预测器的推测执行绕过边界检查，通过缓存侧信道泄露数据

### 2.11 常见误区与纠正

| 误区 | 正确理解 |
|------|----------|
| "栈向低地址增长所以 buffer 向低地址溢出" | 栈的增长方向和 buffer 的溢出方向是两回事。buffer 在栈上从低地址向高地址写入（数组特性），溢出会覆盖高地址的返回地址 |
| "malloc 返回的内存一定是从堆分配的" | 大块分配（默认 > 128KB）使用 mmap，在堆和栈之间的映射区域分配，不在传统堆区 |
| "free 后指针变为 NULL" | free 不会修改指针变量的值，指针仍然指向已释放的内存（悬垂指针），这就是 UAF 的根源 |
| "ASLR 让所有漏洞无法利用" | ASLR 只是增加了难度。信息泄露（如格式化字符串、部分溢出读取）可以绕过 ASLR。fork 场景下子进程继承父进程的 ASLR 布局 |
| "栈保护 canary 完全安全" | canary 的第一个字节是 `\x00`（终止符），对基于字符串的溢出（如 gets）有效，但对逐字节覆盖或二进制溢出无效。fork 后子进程继承父进程的 canary |
| "NX 保护意味着不能执行任意代码" | NX 阻止在栈/堆上执行代码，但攻击者可以用 ROP（Return-Oriented Programming）复用已有代码片段，不需要注入新代码 |
| "volatile 可以替代原子操作" | volatile 只阻止编译器优化（不缓存到寄存器），不提供原子性，不建立 happens-before 关系，不能用于线程同步 |
| "C/C++ 内存模型和虚拟内存是同一回事" | 虚拟内存是操作系统/硬件层的地址抽象；C/C++ 内存模型是语言标准定义的并发访问语义。两者在不同层面解决不同问题 |
| "const 变量一定不可修改" | const 修饰的变量可以通过指针强转绕过（未定义行为），但编译器可能将其放在只读段（.rodata），此时修改会导致段错误 |

### 2.12 总结与知识图谱

```text
计算机内存模型知识体系：

硬件层
├── 物理内存 → 虚拟地址空间 → 分页机制 → 页表 → TLB
├── 缓存层次 → L1/L2/L3 → 缓存行 → 侧信道
└── 内存保护 → NX bit → SMAP/SMEP → CET

操作系统层
├── 虚拟内存管理 → mmap/brk → 页面置换
├── ASLR → 栈/堆/库/PIE 随机化
└── 进程地址空间 → 内核空间/用户空间

语言层（C/C++）
├── 进程内存布局
│   ├── .text → 代码段（只读可执行）
│   ├── .rodata → 只读数据
│   ├── .data → 已初始化全局变量
│   ├── .bss → 未初始化全局变量
│   ├── Heap → 动态分配（malloc/free）
│   ├── mmap → 映射区域（共享库、大块分配）
│   └── Stack → 函数调用栈帧
├── 栈机制 → 调用约定 → 栈帧 → canary
├── 堆机制 → bin 系统 → chunk → 分配器
├── 内存保护 → NX → RELRO → FORTIFY
└── 并发内存模型 → 原子操作 → 内存序 → happens-before

安全视角
├── 栈漏洞 → 缓冲区溢出 → ROP → ret2libc
├── 堆漏洞 → UAF → double free → 堆喷射
├── 信息泄露 → 格式化字符串 → 部分溢出
└── 绕过技术 → 泄露 canary → Bypass ASLR → JIT spray
```

理解内存模型是安全研究的基石。从物理内存的分页机制到虚拟地址空间的布局，从栈帧的精确结构到堆分配器的 bin 链表，从编译器的保护选项到硬件的 NX/CFI 特性——每一层都是攻防博弈的战场。掌握这些知识，才能在漏洞研究中做到知其然、知其所以然。
