## 5. 字符串处理——漏洞之源

字符串处理是 C/C++ 安全漏洞的第一大来源。从 Morris 蠕虫（1988）到现代浏览器远程代码执行，无数重大安全事件的根因都可以追溯到 C 语言字符串处理的先天缺陷。本章从内存模型出发，系统剖析字符串漏洞的成因、分类、利用与防御。

### 5.1 C 字符串的内存本质

#### 5.1.1 null 终止符模型

C 语言没有内建的字符串类型。所谓"字符串"，不过是一段以 `\0`（ASCII 值 0x00）结尾的连续字节。这个设计决策来自 1970 年代的 PDP-11 时代——Dennis Ritchie 选择用一个哨兵值标记字符串结束，而不是在每个字符串前面存储长度，原因是当时的内存极为宝贵，多存一个字节的长度字段都是浪费。

```c
// 三种等价的字符串初始化方式
char str1[] = "hello";                          // 编译器自动追加 \0，sizeof = 6
char str2[] = {'h', 'e', 'l', 'l', 'o', '\0'}; // 手动构造，与 str1 完全相同
char str3[10] = "hello";                        // 前 6 字节填入，剩余 4 字节为 0

// strlen vs sizeof 的本质区别
strlen(str1);   // 5  —— 从首地址向后扫描直到遇到 \0
sizeof(str1);   // 6  —— 编译期确定的数组大小，包含 \0
sizeof(str3);   // 10 —— 数组总大小，与内容无关
```

这个模型的核心问题是：**字符串的"长度"不是一个存储在内存中的元数据，而是每次需要时通过遍历计算出来的运行时值**。`strlen` 的时间复杂度是 O(n)，它必须逐字节扫描直到找到 `\0`。如果 `\0` 不存在（比如 `char buf[5] = "hello";` 中没有空间放终止符），`strlen` 会继续向后扫描，越过数组边界，进入不属于该字符串的内存区域——这就是缓冲区越界读取的起点。

#### 5.1.2 栈上的字符串布局

理解字符串漏洞必须理解栈帧布局。以下是一个典型的 x86-64 函数调用栈帧：

```text
高地址
┌─────────────────────────┐
│     函数参数 (溢出方向↑)  │  ← 调用者栈帧
├─────────────────────────┤
│     返回地址 (RIP)        │  ← 8 字节，控制执行流的关键目标
├─────────────────────────┤
│     保存的 RBP            │  ← 8 字节，栈帧指针
├─────────────────────────┤
│     局部变量区             │
│  ┌───────────────────┐  │
│  │  buffer[64]       │  │  ← 字符串缓冲区（低地址 ← 溢出方向）
│  └───────────────────┘  │
│     canary (如果有)       │  ← 栈保护哨兵值
├─────────────────────────┤
│     其他局部变量           │
└─────────────────────────┘
低地址
```

关键要点：栈是从高地址向低地址增长的，而 `strcpy` 等函数是从低地址向高地址写入的。这意味着当缓冲区溢出时，写入会向高地址蔓延，依次覆盖保存的 RBP、返回地址，以及其他更高地址的局部变量。

#### 5.1.3 堆上的字符串布局

堆上分配的字符串同样脆弱，但溢出后果不同：

```c
char *a = malloc(32);   // 分配 32 字节
char *b = malloc(32);   // 紧随其后分配
strcpy(a, very_long_input);  // 溢出会覆盖 b 的内容，甚至堆管理元数据
```

堆溢出可以覆盖相邻堆块的元数据（chunk header），攻击者通过精心构造的溢出数据可以实现堆风水（Heap Feng Shui）攻击，最终控制程序执行流。堆溢出比栈溢出更难利用，但危害同样严重。

### 5.2 危险函数全景

#### 5.2.1 危险函数与安全替代对照表

| 危险函数 | 风险类型 | 安全替代 | 替代函数的注意事项 |
|---------|---------|---------|-----------------|
| `strcpy(dst, src)` | 无长度限制，溢出 | `strncpy(dst, src, n)` | 不保证 `\0` 终止，需手动添加 |
| `strcat(dst, src)` | 无长度限制，溢出 | `strncat(dst, src, n)` | n 是追加的最大字符数，不含已有的 |
| `sprintf(buf, fmt, ...)` | 无长度限制 | `snprintf(buf, size, fmt, ...)` | 返回"应写入"的长度，非实际写入 |
| `gets(buf)` | 无长度限制，已从 C11 移除 | `fgets(buf, size, stdin)` | 会保留换行符，需手动去除 |
| `scanf("%s", buf)` | 无长度限制 | `scanf("%63s", buf)` 或 `fgets` | 字段宽度限定符是唯一防线 |
| `vsprintf(buf, fmt, ap)` | 无长度限制 | `vsnprintf(buf, size, fmt, ap)` | 同 snprintf |
| `realpath(path, resolved)` | resolved 可能不够大 | 传 NULL 让系统分配 | POSIX 2008+ 支持 |

#### 5.2.2 `strncpy` 的隐藏陷阱

`strncpy` 并非万无一失的安全替代。它有两个容易被忽略的行为：

```c
char buf[8];
strncpy(buf, "abcdefghijklmnop", sizeof(buf));
// buf 的内容：{'a','b','c','d','e','f','g','h'}
// 没有 \0 终止符！strncpy 在 src 长度 >= n 时不会追加 \0

// 正确用法：始终手动终止
strncpy(buf, src, sizeof(buf) - 1);
buf[sizeof(buf) - 1] = '\0';
```

第二个问题是性能：当 `src` 长度远小于 `n` 时，`strncpy` 会用 `\0` 填充剩余空间至 `n` 字节，这在热路径上可能成为性能瓶颈。

#### 5.2.3 `gets` 的彻底死刑

`gets` 函数没有任何方式限制读取长度，它在 C99 中被标记为 deprecated，在 C11 的 Annex K 中被正式移除。任何使用 `gets` 的代码都应当立即重构。如果你在遗留代码中看到 `gets`，这是一个高优先级的安全审计发现。

### 5.3 缓冲区溢出的完整分类

#### 5.3.1 栈缓冲区溢出（Stack Buffer Overflow）

这是最经典、最广为人知的溢出类型。以下是一个完整的漏洞示例与利用分析：

```c
#include <stdio.h>
#include <string.h>

void secret_function() {
    printf("You should never reach here!\n");
    // 实际攻击中，这里可以是 shellcode 或 system("/bin/sh")
}

void vulnerable(char *input) {
    char buffer[64];
    strcpy(buffer, input);  // 危险：无边界检查
}

int main(int argc, char **argv) {
    if (argc > 1) {
        vulnerable(argv[1]);
    }
    return 0;
}
```

**利用原理**：当 `input` 长度超过 64 字节时，溢出的数据会依次覆盖：

```text
偏移量    内容              覆盖目标
0-63      "AAAA...AAAA"     buffer[64] 本身
64-71     "AAAAAAAA"         保存的 RBP（栈帧指针）
72-79     <secret_function>  返回地址 → 劫持执行流
```

**精确定位偏移量的方法**：

```bash
# 方法一：模式字符串法（推荐）
# 生成一个独特模式，溢出后检查 EIP/RIP 中的模式片段
# 使用 pwntools 或 Metasploit 的 pattern_create/pattern_offset
python3 -c "from pwn import *; print(cyclic(200).decode())"
# 运行程序，gdb 中查看 RIP 的值，然后用 cyclic_find 定位偏移

# 方法二：二分法手动测试
for i in $(seq 60 2 100); do
    python3 -c "print('A'*$i)" | ./vulnerable
done
# 观察何时出现 Segmentation fault 以及崩溃地址
```

#### 5.3.2 堆缓冲区溢出（Heap Buffer Overflow）

```c
#include <stdlib.h>
#include <string.h>

int main() {
    char *username = malloc(16);   // 用户名缓冲区
    char *auth_flag = malloc(8);   // 认证标志

    strcpy(auth_flag, "DENIED");  // 初始化为拒绝
    gets(username);                // 用户输入溢出 username

    // 如果 username 溢出覆盖了 auth_flag 的内容
    // 攻击者可以将 auth_flag 改为 "GRANTED"
    if (strcmp(auth_flag, "GRANTED") == 0) {
        printf("Access granted!\n");
    }
    return 0;
}
```

堆溢出的关键挑战在于：堆块的布局不像栈那样可预测。不同分配器（glibc malloc、jemalloc、tcmalloc）有不同的元数据格式和空闲块管理策略。攻击者需要理解目标分配器的行为才能稳定利用堆溢出漏洞。

#### 5.3.3 Off-by-One 溢出

Off-by-one 是最常见的边界错误之一，差一个字节就足以改变程序行为：

```c
void off_by_one(char *input) {
    char buffer[256];
    int i;
    for (i = 0; i <= 256; i++) {  // BUG: <= 应该是 <
        buffer[i] = input[i];
    }
}
```

在栈上，off-by-one 溢出的一个字节可能覆盖保存的 RBP 的最低有效字节（LSB）。由于 x86-64 使用小端序，修改 LSB 可以让栈帧指针偏移一个字节，进而导致后续的栈帧引用指向错误位置。攻击者可以利用这一点进行栈伪造（stack pivoting）。

#### 5.3.4 整数溢出导致的字符串溢出

```c
void process_data(unsigned int len, char *data) {
    // len 来自不可信的网络输入
    char *buf = malloc(len + 1);  // +1 预留 \0
    // 当 len = 0xFFFFFFFF 时，len + 1 = 0（整数溢出）
    // malloc(0) 可能返回一个极小的堆块
    memcpy(buf, data, len);       // 用 0xFFFFFFFF 字节写入极小缓冲区
    // → 堆溢出
}
```

此类漏洞的根因是长度字段本身不可信。防御时必须在算术运算前检查溢出边界：

```c
// 安全的长度检查
if (len > SIZE_MAX - 1) {
    return ERROR;
}
char *buf = malloc(len + 1);
```

### 5.4 格式化字符串漏洞

#### 5.4.1 原理

格式化字符串漏洞（Format String Vulnerability）是另一类与字符串处理直接相关的安全问题。当用户输入被直接用作 `printf` 的格式字符串时，攻击者可以读取和写入任意内存：

```c
// 危险代码
char input[256];
fgets(input, sizeof(input), stdin);
printf(input);  // BUG：用户输入直接作为格式字符串

// 攻击者输入 "%x.%x.%x.%x" 可以泄露栈上的数据
// 攻击者输入 "%08x.%08x.%08x" 可以逐个读取栈帧内容
```

#### 5.4.2 格式化字符串的读与写

```text
读取内存：  %s      —— 将栈上的值作为地址，读取该地址处的字符串
           %x / %p —— 读取栈上的值并以十六进制输出
           %n$s    —— 直接读取第 n 个参数（位置参数）

写入内存：  %n      —— 将已输出的字符数写入指定地址
           %hn     —— 写入 2 字节（short）
           %hhn    —— 写入 1 字节（char）
```

**任意写入原理**：`%n` 会将 `printf` 到目前为止输出的字符总数写入到对应的指针参数。攻击者通过控制格式字符串的宽度修饰符（如 `%1234x`），可以精确控制写入的数值。结合位置参数（如 `%7$n`），可以精确控制写入的目标地址。通过多次 `%hhn` 写入（每次写 1 字节），可以在任意地址写入任意 4 字节或 8 字节值。

```text
# 利用示例：将 GOT 表中的 printf 地址改写为 system 地址
# 从而实现：下一次调用 printf(input) 实际执行 system(input)
# 格式字符串载荷（简化示意）：
# [GOT_printf][GOT_printf+1][GOT_printf+2][GOT_printf+3]
# %[offset1]$hhn  %[offset2]$hhn  %[offset3]$hhn  %[offset4]$hhn
```

#### 5.4.3 防御

```c
// 绝对正确的方式：永远不要将用户输入作为格式字符串
printf("%s", user_input);  // 安全：user_input 只是数据，不会被解析

// 编译器可以帮忙：GCC/Clang 的 -Wformat-security 会警告
// 但不能完全依赖编译器警告
```

### 5.5 多字节字符与编码陷阱

#### 5.5.1 UTF-8 的安全隐患

UTF-8 使用变长编码（1-4 字节/字符），这与 C 标准库假设"一个字符 = 一个字节"的隐含前提产生了根本冲突：

```c
char buf[8];
const char *chinese = "你好世界";  // 每个汉字 3 字节，总共 12 字节 + \0 = 13 字节

strncpy(buf, chinese, sizeof(buf));
// buf: {'\xe4', '\xbd', '\xa0', '\xe5', '\xa5', '\xbd', '\xe4', '\xb8'}
// 没有 \0 终止符（strncpy 的已知行为）
// 更严重的是：第二个汉字"好"的 3 字节被截断，buf 中是不完整的 UTF-8 序列
```

截断 UTF-8 序列会导致两个问题：

1. **数据损坏**：不完整的多字节序列无法正确解码，显示为乱码（replacement character U+FFFD）。
2. **安全漏洞**：某些程序在处理不完整 UTF-8 时存在解析差异（parsing differential），导致不同组件对同一段字节序列产生不同的解释。这在 HTTP 头解析、URL 解析、文件路径解析等场景中可能被利用。

#### 5.5.2 宽字符函数的安全问题

```c
#include <wchar.h>

wchar_t wbuf[64];
// wcsncpy 与 strncpy 有相同的 \0 终止问题
// 但单位是 wchar_t（通常 4 字节），不是字节

// 危险的转换
char input[] = "AAAA...很长...";
wchar_t wbuf[64];
mbstowcs(wbuf, input, 64);  // 如果 input 的多字节序列超过 64 个宽字符
                              // 行为未定义（C 标准说"不确定"）
```

#### 5.5.3 路径遍历中的编码差异

Web 应用和操作系统对 URL/路径编码的处理差异是经典漏洞来源：

```text
# 不同组件对同一路径的不同解释
路径: /..%2f..%2f..%2fetc%2fpasswd

Web 服务器:    /../../../etc/passwd  → 拒绝（检测到 ../）
后端应用:      /..%2f..%2f..%2fetc%2fpasswd  → 接受（未检测 ../）
文件系统层:    /../../../etc/passwd  → 读取成功

# Unicode 规范化攻击
输入: /admin（正常的 a）
变体: /аdmin（西里尔字母 а = U+0430）

WAF 检查: /admin → 允许（匹配白名单）
后端规范化: аdmin → admin → 进入管理后台
```

### 5.6 现代防御体系

#### 5.6.1 编译期防御

```bash
# 栈保护（Stack Canary）
gcc -fstack-protector-all -o prog prog.c
# 在局部变量和返回地址之间插入一个随机值（canary）
# 函数返回前检查 canary 是否被篡改，若被篡改则立即终止

# 格式化字符串保护
gcc -Wformat -Wformat-security -Werror=format-security -o prog prog.c
# 将格式化字符串相关的警告升级为编译错误

# Fortify Source
gcc -D_FORTIFY_SOURCE=2 -O2 -o prog prog.c
# 将 strcpy → __strcpy_chk、sprintf → __sprintf_chk 等
# 运行时检查缓冲区大小（需要编译器和 glibc 配合）
```

#### 5.6.2 运行时防御

```text
┌─────────────────────────────────────────────────────────────────┐
│                        现代防御层次                              │
├──────────┬──────────────────────────────────────────────────────┤
│ 防御层    │ 说明                                                │
├──────────┼──────────────────────────────────────────────────────┤
│ ASLR     │ 地址空间布局随机化。每次运行时栈、堆、库的基地址不同  │
│          │ → 攻击者无法预测目标地址                              │
├──────────┼──────────────────────────────────────────────────────┤
│ DEP/NX   │ 数据执行保护。栈和堆标记为不可执行                   │
│          │ → 注入的 shellcode 无法直接执行                      │
├──────────┼──────────────────────────────────────────────────────┤
│ PIE      │ 位置无关可执行文件。代码段地址也随机化               │
│          │ → ROP 链的 gadget 地址每次不同                       │
├──────────┼──────────────────────────────────────────────────────┤
│ Stack    │ 栈保护哨兵值。插入随机 canary                       │
│ Canary   │ → 栈溢出覆盖 canary 时程序终止                       │
├──────────┼──────────────────────────────────────────────────────┤
│ CFI      │ 控制流完整性。间接跳转目标必须合法                   │
│ (Clang)  │ → 阻止 ROP/JOP 等代码重用攻击                       │
├──────────┼──────────────────────────────────────────────────────┤
│ Safe     │ glibc 2.23+ 有 tcache、unlink 检查等                │
│ Unlink   │ → 增加堆利用难度                                     │
└──────────┴──────────────────────────────────────────────────────┘
```

#### 5.6.3 安全的字符串处理实践

```c
// 方法一：使用长度受限的函数 + 强制终止
int safe_copy(char *dst, size_t dst_size, const char *src) {
    if (dst == NULL || src == NULL || dst_size == 0) {
        return -1;
    }
    strncpy(dst, src, dst_size - 1);
    dst[dst_size - 1] = '\0';
    return 0;
}

// 方法二：使用 C11 附录 K 的安全函数（需编译器支持）
#define __STDC_WANT_LIB_EXT1__ 1
#include <string.h>
char buf[64];
errno_t err = strcpy_s(buf, sizeof(buf), src);
if (err != 0) {
    // 处理错误
}

// 方法三：使用动态缓冲区（如 asprintf，GNU 扩展）
char *result;
if (asprintf(&result, "Hello, %s!", name) == -1) {
    // 处理错误
}
// 使用完毕后 free(result)

// 方法四：C++ 中使用 std::string
std::string safe_str = user_input;  // 自动管理内存和长度
// 但仍然需要注意：
// - std::string 可以包含 \0（中间的 null 不截断）
// - c_str() 返回的指针生命周期绑定到 string 对象
// - 在需要 C 接口时仍需小心边界
```

### 5.7 真实世界的重大漏洞

#### 5.7.1 经典案例一览

| 漏洞 | 年份 | 根因 | 影响 |
|------|------|------|------|
| Morris 蠕虫（fingerd gets） | 1988 | `gets()` 无边界检查 | 感染 6000+ 台计算机（当时互联网的 10%） |
| Code Red（IIS 索引服务） | 2001 | 栈缓冲区溢出 | 感染 359000+ 台服务器 |
| Slammer（SQL Server） | 2003 | 栈溢出（UDP 包） | 10 分钟内感染 75000 台服务器 |
| OpenSSL Heartbleed | 2014 | 缺少长度校验的 `memcpy` | 泄露服务器私钥和用户数据 |
| glibc Ghost（gethostbyname） | 2015 | `__nss_hostname_digits_dots` 中的堆溢出 | 影响几乎所有 Linux 系统 |
| sudo CVE-2021-3156 | 2021 | Off-by-one 堆溢出 | 任何本地用户可提升至 root |

#### 5.7.2 深入分析：Heartbleed（CVE-2014-0160）

Heartbleed 的代码根因极为简单，却造成了巨大影响：

```c
// 简化的漏洞代码
/* 读取心跳请求中的 payload_length */
unsigned short payload_length = ntohs(*(unsigned short *)p);
p += 2;

// BUG：没有检查 payload_length 是否与实际收到的数据匹配
// 攻击者可以声称 payload_length = 65535，而实际只发送了 1 字节

// 响应时，服务器从内存中读取 payload_length 字节
memcpy(response, p, payload_length);  // 越界读取！
```

这个漏洞的修复只有两行代码——加一个长度检查。但因为 OpenSSL 代码库缺乏明确的长度记录（实际收到的字节数没有被追踪），这个简单的检查遗漏了数年。

#### 5.7.3 深入分析：sudo Baron Samedit（CVE-2021-3156）


### 5.8 防御检查清单

在代码审计中，以下是字符串处理相关的高优先级检查项：

```text
□ 所有字符串复制操作是否使用了长度受限函数？
□ strncpy 调用后是否手动添加了 \0 终止符？
□ 是否存在用户输入直接作为 printf 格式字符串的情况？
□ 所有长度计算是否在整数运算前检查了溢出边界？
□ 多字节字符串（UTF-8）的截断是否得到了正确处理？
□ 网络协议解析中，length 字段是否经过了合法性验证？
□ 是否启用了编译器安全选项（-fstack-protector-all, -D_FORTIFY_SOURCE=2）？
□ 是否在代码审查中将 gets/strcpy/strcat/sprintf 列为禁用函数？
□ 动态分配的字符串缓冲区是否有明确的所有权和生命周期管理？
□ 单元测试是否覆盖了边界长度、空字符串、超长输入等边界情况？
```

### 5.9 与安全语言的对比

为什么现代语言在字符串处理上更安全？关键区别在于：

| 特性 | C/C++ | Rust | Go | Python |
|------|-------|------|-----|--------|
| 字符串长度 | 运行时遍历计算 | 编译期存储为字段 | 内建 len 字段 | 内建 len 方法 |
| 边界检查 | 无 | 编译期 + 运行时 | 运行时 | 运行时 |
| 内存所有权 | 手动管理 | 所有权系统强制 | GC 自动管理 | 引用计数 + GC |
| 编码处理 | 无内建支持 | UTF-8 强制 | UTF-8 内建 | Unicode 内建 |
| 空指针/溢出 | 未定义行为 | panic（可恢复） | panic（可恢复） | 异常 |

Rust 的 `String` 类型在编译期就通过所有权系统防止了大多数 use-after-free 和 double-free 问题，在运行时通过边界检查防止了缓冲区溢出。这不是说 Rust 完全安全（`unsafe` 块可以绕过所有检查），但它将安全默认值从"不安全"翻转为"安全"，使得安全成为路径依赖的自然结果，而非需要开发者额外努力的目标。

### 5.10 进阶话题

#### 5.10.1 Fuzzing 字符串处理代码

模糊测试是发现字符串处理漏洞最有效的方法之一：

```c
// 使用 libFuzzer 测试字符串处理函数
#include <stdint.h>
#include <string.h>

int LLVMFuzzerTestOneInput(const uint8_t *data, size_t size) {
    char buf[256];
    // 将 fuzzer 输入传入被测函数
    // fuzzer 会自动生成边界用例、超长输入、特殊字符等
    if (size < sizeof(buf)) {
        memcpy(buf, data, size);
        buf[size] = '\0';
        process_string(buf);  // 被测函数
    }
    return 0;
}
```

```bash
# 编译与运行
```

字符串处理漏洞的根因在于 C 语言的设计哲学：信任程序员，最小化运行时开销。这在 1970 年代是合理的工程取舍，但在今天的网络威胁环境下，这个取舍的代价是无数的安全漏洞。理解这些漏洞的成因，是从"会写 C 代码"到"能写安全的 C 代码"的关键跨越。
