# 第05章 计算机网络基础 - 练习方法

网络协议的学习如果只停留在书本和课堂，就像学游泳只看视频不下水——你以为自己懂了，真正上手时才发现什么都抓不住。本章提供一套从零到实战的完整练习体系，每一项练习都配有明确的目标、可执行的步骤、预期输出和自检标准。按照这套方法练习，你不仅能"知道"协议，还能"看见"协议、"操控"协议。

## 一、实验环境搭建

### 1.1 物理环境 vs 虚拟环境

在开始之前，你需要决定在哪里搭建实验环境。两种方案各有优劣：

| 维度 | 物理环境 | 虚拟环境 |
|------|---------|---------|
| 成本 | 需要多台设备或二手硬件 | 一台性能足够的电脑即可 |
| 灵活性 | 拓扑变更需要重新接线 | 几分钟内重建整个网络 |
| 真实度 | 与生产环境完全一致 | 虚拟网卡行为可能有细微差异 |
| 快照/回滚 | 不支持 | 支持，实验失败秒级恢复 |
| 推荐场景 | 无线网络实验、硬件级攻击 | 其他所有场景 |

**建议**：日常练习使用虚拟环境，效率远高于物理环境。仅在涉及无线网络（802.11）、硬件协议（如串口、CAN总线）或需要真实交换机行为时才搭建物理环境。

### 1.2 虚拟网络实验室方案

#### 方案一：VirtualBox/VMware + 仅主机网络（推荐入门）

这是最简单、最稳定的入门方案。

**第一步：创建虚拟网络**

在 VirtualBox 中，进入「管理 → 主机网络管理器 → 创建」，创建一个仅主机网络（Host-Only Adapter）。默认会生成一个 `vboxnet0`（Linux/macOS）或 `VirtualBox Host-Only Ethernet Adapter`（Windows）。

```text
网络拓扑示意：

┌─────────────────────────────────────────────┐
│           VirtualBox Host-Only Network       │
│              192.168.56.0/24                 │
│                                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │  Kali     │  │ Ubuntu   │  │ pfSense  │   │
│  │ .56.10    │  │ .56.100  │  │ .56.1    │   │
│  │ (攻击机)  │  │ (目标机) │  │ (网关)   │   │
│  └──────────┘  └──────────┘  └──────────┘   │
└─────────────────────────────────────────────┘
```

**第二步：配置虚拟机**

创建三台虚拟机，按以下规格分配资源：

| 虚拟机 | 角色 | 推荐系统 | CPU | 内存 | 磁盘 |
|--------|------|---------|-----|------|------|
| 攻击机 | 渗透测试工具集 | Kali Linux 2024+ | 2核 | 4GB | 40GB |
| 目标机 | 漏洞靶机 | Metasploitable 3 | 1核 | 2GB | 20GB |
| 网关/防火墙 | 流量控制与监控 | pfSense CE | 1核 | 1GB | 10GB |

**第三步：配置网络**

每台虚拟机添加一块网卡，绑定到刚才创建的仅主机网络。启动后手动配置 IP：

```bash
# Kali 攻击机
sudo ip addr add 192.168.56.10/24 dev eth0
sudo ip link set eth0 up

# Ubuntu 目标机
sudo ip addr add 192.168.56.100/24 dev ens33
sudo ip link set ens33 up

# 验证连通性
ping -c 3 192.168.56.100    # 从Kali ping目标机
ping -c 3 192.168.56.10     # 从目标机ping Kali
```

**预期输出**：三台机器之间互相 ping 通，丢包率 0%。如果 ping 不通，检查防火墙规则（`sudo ufw disable`）和虚拟网络配置。

#### 方案二：GNS3/EVE-NG 网络模拟器（推荐进阶）

当你需要模拟包含多台路由器、交换机、防火墙的复杂企业网络拓扑时，GNS3 和 EVE-NG 是首选工具。

**GNS3 搭建要点**：

1. 安装 GNS3（支持 Windows/macOS/Linux）
2. 下载 GNS3 VM（VMware/VirtualBox 格式），这是实际运行网络设备模拟器的后端
3. 导入 Cisco IOS 镜像（需自行准备授权镜像）或使用内置的 IOSvL2/vIOS
4. 拖拽设备到画布，连线，启动设备
5. 通过 Console 端口配置每台设备

**EVE-NG 搭建要点**：

1. 下载 EVE-NG Community Edition OVA 文件
2. 导入 VMware/VirtualBox，分配至少 8GB 内存和 4 核 CPU
3. 浏览器访问 EVE-NG Web 界面
4. 上传设备镜像到 `/opt/unetlab/addons/` 对应目录
5. 创建 Lab，拖拽设备，启动并配置

```text
复杂拓扑示例（EVE-NG）：

                    ┌─────────┐
                    │ Internet │
                    │  (模拟)  │
                    └────┬────┘
                         │
                    ┌────┴────┐
                    │ 防火墙  │
                    │ pfSense │
                    └────┬────┘
                         │
              ┌──────────┼──────────┐
              │          │          │
         ┌────┴───┐ ┌────┴───┐ ┌────┴───┐
         │ VLAN 10│ │ VLAN 20│ │ VLAN 30│
         │  Web   │ │  DB    │ │  管理  │
         │10.0.10 │ │10.0.20 │ │10.0.30 │
         └────────┘ └────────┘ └────────┘
```

#### 方案三：Docker 容器网络（推荐快速实验）

Docker 容器启动速度快、资源占用少，非常适合快速验证网络概念和进行批量实验。

**基础搭建**：

```bash
# 创建隔离的 Docker 网络
docker network create --subnet=172.20.0.0/24 --gateway=172.20.0.1 hacknet

# 启动目标容器（使用已知漏洞的镜像）
docker run -d --name victim --net hacknet --ip 172.20.0.100 \
  vulnerables/web-dvwa

# 启动攻击容器
docker run -d --name attacker --net hacknet --ip 172.20.0.10 \
  kalilinux/kali-rolling sleep infinity

# 进入攻击容器
docker exec -it attacker bash

# 验证连通性
ping -c 3 172.20.0.100
```

**多网段 Docker 拓扑**：

```bash
# 创建两个隔离网络
docker network create --subnet=172.20.0.0/24 net-frontend
docker network create --subnet=172.21.0.0/24 net-backend

# 启动 Web 服务器（同时连接两个网络）
docker run -d --name web --net net-frontend --ip 172.20.0.10 \
  vulnerables/web-dvwa
docker network connect --ip 172.21.0.10 net-backend web

# 启动数据库（仅在后端网络）
docker run -d --name db --net net-backend --ip 172.21.0.100 \
  mysql:5.7

# 启动攻击机（仅在前端网络）
docker run -d --name attacker --net net-frontend --ip 172.20.0.2 \
  kalilinux/kali-rolling sleep infinity

# 验证：攻击机可以直接访问 Web，但不能直接访问 DB
docker exec attacker ping -c 1 172.20.0.10   # 应该通
docker exec attacker ping -c 1 172.21.0.100  # 应该不通
```

**Docker 常用漏洞靶机镜像**：

| 镜像名 | 用途 | 拉取命令 |
|--------|------|---------|
| `vulnerables/web-dvwa` | Web 漏洞练习 | `docker pull vulnerables/web-dvwa` |
| `vulnerables/web-juice-shop` | OWASP Juice Shop | `docker pull bkimminich/juice-shop` |
| `sickp/olddebian` | 老版本 Debian（含已知漏洞） | `docker pull sickp/olddebian` |
| `mikesplain/openvas` | 漏洞扫描器 | `docker pull mikesplain/openvas` |
| `citellus/metasploitable` | Metasploitable 容器化版 | `docker pull citellus/metasploitable` |

### 1.3 靶机环境选择指南

不同水平的学习者应该选择不同难度的靶机：

**入门级（零基础~1个月）**：

- **DVWA (Damn Vulnerable Web Application)**：最经典的 Web 漏洞练习平台，包含 SQL 注入、XSS、文件上传等常见漏洞，每个漏洞都有 Low/Medium/High/Impossible 四个难度等级
- **Mutillidae**：OWASP 推荐的 Web 漏洞练习平台，比 DVWA 更全面
- **Metasploitable 2**：Ubuntu 8.04 为基础，预装了大量含漏洞的服务（vsftpd 2.3.4、Apache Tomcat、Samba 等），非常适合练习网络层攻击

**中级（1~3个月）**：

- **Metasploitable 3**：基于 Windows 和 Ubuntu 的更新版本靶机，漏洞更接近真实环境
- **HackTheBox Starting Point**：HTB 为初学者设计的入门路径，有引导式提示
- **TryHackMe**：房间（Room）形式的学习路径，从基础到高级逐步递进
- **VulnHub 靶机推荐**：Kioptrix（1-4）、DC-1、Brainpan

**高级（3个月以上）**：

- **HackTheBox Active Machines**：真实难度的靶机，无提示
- **Proving Grounds (Offensive Security)**：OffSec 官方练习平台
- **PentesterLab**：专注于 Web 安全的进阶练习
- **CyberDefenders**：蓝队方向的事件分析练习

**靶机环境快速对照表**：

| 平台 | 难度 | 方向 | 是否免费 | 特点 |
|------|------|------|---------|------|
| DVWA | ★☆☆☆☆ | Web | 免费 | 经典入门，难度可调 |
| Metasploitable 2 | ★★☆☆☆ | 网络+服务 | 免费 | 服务漏洞丰富 |
| TryHackMe | ★★☆☆☆ | 综合 | 部分免费 | 引导式学习 |
| HackTheBox | ★★★☆☆ | 综合 | 部分免费 | 社区活跃 |
| Proving Grounds | ★★★★☆ | 综合 | 付费 | OffSec 官方 |
| CyberDefenders | ★★★★☆ | 蓝队 | 部分免费 | 事件分析 |

## 二、抓包分析练习

### 2.1 工具准备与基础操作

在开始抓包练习之前，确保你已经安装并了解以下工具：

**Wireshark（GUI）**：

```bash
# Debian/Ubuntu 安装
sudo apt install wireshark
# 安装过程中选择 "Yes" 允许非 root 用户抓包
sudo usermod -aG wireshark $USER
# 重新登录后生效
```

**tcpdump（命令行）**：

```bash
# 通常已预装，如果没有：
sudo apt install tcpdump

# 基础抓包：抓取 eth0 接口的前 100 个包
sudo tcpdump -i eth0 -c 100

# 保存到文件（后续可用 Wireshark 打开分析）
sudo tcpdump -i eth0 -w /tmp/capture.pcap -c 1000

# 读取 pcap 文件
tcpdump -r /tmp/capture.pcap

# 只抓 TCP 80 端口的流量
sudo tcpdump -i eth0 tcp port 80 -c 50

# 抓取特定主机的流量
sudo tcpdump -i eth0 host 192.168.1.100

# 抓取 DNS 流量
sudo tcpdump -i eth0 port 53 -c 20
```

**tshark（Wireshark 的命令行版本）**：

```bash
# 安装
sudo apt install tshark

# 实时抓包并显示 HTTP 请求的 Host 和 URI
sudo tshark -i eth0 -Y "http.request" -T fields \
  -e http.host -e http.request.uri

# 抓取 DNS 查询并显示查询的域名
sudo tshark -i eth0 -Y "dns.qry.name" -T fields \
  -e dns.qry.name -e dns.qry.type

# 从 pcap 文件中统计会话
tshark -r capture.pcap -z conv,tcp

# 提取 HTTP 传输的文件
tshark -r capture.pcap --export-objects http,./exported_files/
```

### 2.2 基础练习：协议识别与报文结构

**练习一：HTTP vs HTTPS 对比**

目标：直观理解加密与明文通信的区别。

步骤：
1. 打开一个终端窗口，启动抓包：
   ```bash
   sudo tcpdump -i eth0 -A tcp port 80 or tcp port 443 | head -100
   ```
2. 打开另一个终端，先访问 HTTP 站点：
   ```bash
   curl http://httpbin.org/get
   ```
3. 观察 tcpdump 输出中可以清晰看到 HTTP 请求头（`GET /get HTTP/1.1`）和响应体（JSON 数据）
4. 再访问 HTTPS 站点：
   ```bash
   curl https://httpbin.org/get
   ```
5. 观察 tcpdump 输出全部是乱码（TLS 加密数据）

**预期输出对比**：

```text
# HTTP 流量（明文，可直接读取）
GET /get HTTP/1.1
Host: httpbin.org
User-Agent: curl/7.88.1
Accept: *

HTTP/1.1 200 OK
Content-Type: application/json
{"origin":"192.168.1.50"}

# HTTPS 流量（加密，不可读）
...  17 03 03 01 2a 00 00 00 00 00 ...  (二进制乱码)
```

**关键认知**：HTTP 流量中用户名、密码、Cookie、Token 全部明文传输——这就是为什么在公共 WiFi 上使用 HTTP 极其危险，也是中间人攻击能窃取敏感信息的根本原因。

**练习二：TCP 三次握手与四次挥手**

目标：亲眼看到 TCP 连接的建立和断开过程。

步骤：
1. 启动 Wireshark，选择网卡开始抓包
2. 在浏览器中访问 `http://httpbin.org/get`（使用 HTTP 以避免 TLS 的干扰）
3. 停止抓包，在显示过滤器中输入：`tcp.stream eq 0`
4. 右键第一个 SYN 包 → Follow → TCP Stream

你应该看到类似这样的数据包序列：

```text
序号  源            目标          标志    说明
1     192.168.1.50  93.184.216.34 [SYN]        客户端发起连接
2     93.184.216.34 192.168.1.50  [SYN, ACK]   服务器确认并发起
3     192.168.1.50  93.184.216.34 [ACK]        客户端确认，连接建立
  ... 数据传输 ...
4     192.168.1.50  93.184.216.34 [FIN, ACK]   客户端请求关闭
5     93.184.216.34 192.168.1.50  [ACK]        服务器确认收到
6     93.184.216.34 192.168.1.50  [FIN, ACK]   服务器也请求关闭
7     192.168.1.50  93.184.216.34 [ACK]        客户端确认，连接关闭
```

**在 Wireshark 中验证**：点击任意 TCP 包，展开 `Transmission Control Protocol` 字段，重点观察：
- `Flags` 字段：SYN、ACK、FIN、RST 各标志位的状态
- `Sequence number`：三次握手中的初始序列号（ISN）
- `Window size`：滑动窗口大小，反映接收方的缓冲区能力
- `TCP Option`：MSS、Window Scale、SACK Permitted 等选项

**进阶：使用 Wireshark 的 Flow Graph**

1. 菜单：Statistics → Flow Graph
2. 选择 "Limit to display filter"
3. 你会看到一个时序图，清晰展示每个数据包在客户端和服务器之间的往返

**练习三：DNS 查询全流程**

目标：理解从域名到 IP 的完整解析过程。

步骤：
1. 先清除本地 DNS 缓存：
   ```bash
   # Linux
   sudo systemd-resolve --flush-caches
   # 或者
   sudo resolvectl flush-caches

   # macOS
   sudo dscacheutil -flushcache; sudo killall -HUP mDNSResponder

   # Windows
   ipconfig /flushdns
   ```
2. 启动抓包：
   ```bash
   sudo tcpdump -i eth0 -n port 53 -c 20
   ```
3. 在另一个终端发起 DNS 查询：
   ```bash
   dig www.example.com A
   ```
4. 观察 tcpdump 输出

**预期输出分析**：

```text
# 查询包（你的机器 → DNS 服务器）
192.168.1.50.54321 > 8.8.8.8.53: 12345+ A? www.example.com. (33)

# 响应包（DNS 服务器 → 你的机器）
8.8.8.8.53 > 192.168.1.50.54321: 12345 1/0/0 A 93.184.216.34 (49)
```

**进阶练习**：查询不同类型的 DNS 记录，理解每种记录的作用：

```bash
# A 记录：域名 → IPv4
dig example.com A

# AAAA 记录：域名 → IPv6
dig example.com AAAA

# MX 记录：邮件服务器
dig example.com MX

# NS 记录：域名服务器
dig example.com NS

# TXT 记录：SPF/DKIM 等验证记录
dig example.com TXT

# CNAME 记录：别名
dig www.example.com CNAME

# ANY 记录：获取所有记录（很多DNS服务器已禁用）
dig example.com ANY

# 反向查询：IP → 域名
dig -x 93.184.216.34
```

使用 Wireshark 打开 DNS 流量，展开 Domain Name System 字段，观察：
- `Transaction ID`：请求和响应的匹配标识
- `Flags`：QR（查询/响应）、RD（期望递归）、RA（支持递归）等
- `Queries`：查询的域名和类型
- `Answers`：解析结果，包含 TTL（缓存时间）

**练习四：ARP 协议解析**

目标：理解局域网内 MAC 地址解析过程。

```bash
# 先清除 ARP 缓存
sudo ip neigh flush all

# 抓取 ARP 流量
sudo tcpdump -i eth0 -n arp -c 10

# 触发 ARP 请求（ping 局域网内一个 IP）
ping -c 1 192.168.1.1
```

**预期输出**：

```text
# ARP 请求（广播：谁是 192.168.1.1？）
ARP, Request who-has 192.168.1.1 tell 192.168.1.50, length 28

# ARP 响应（单播：我是 192.168.1.1，我的 MAC 是 aa:bb:cc:dd:ee:ff）
ARP, Reply 192.168.1.1 is-at aa:bb:cc:dd:ee:ff, length 28
```

在 Wireshark 中观察 ARP 包的完整结构：
- `Hardware type`：以太网（0x0001）
- `Protocol type`：IPv4（0x0800）
- `Opcode`：1=请求，2=响应
- `Sender MAC/IP`：发送方的 MAC 和 IP
- `Target MAC/IP`：目标的 MAC（请求时为全0）和 IP

### 2.3 中级练习：会话重组与文件提取

**练习一：HTTP 会话重组**

目标：从抓包数据中还原完整的 HTTP 通信内容。

步骤：
1. 先用 Python 启动一个简单的 HTTP 服务器（提供可下载的测试文件）：
   ```bash
   # 创建测试文件
   echo "This is a secret test file for packet capture practice." > /tmp/secret.txt

   # 启动 HTTP 服务器
   cd /tmp && python3 -m http.server 8080 &
   ```
2. 启动 Wireshark 抓包
3. 从另一台机器（或另一个终端）下载文件：
   ```bash
   curl http://192.168.56.100:8080/secret.txt
   ```
4. 停止抓包，在 Wireshark 中右键任意 HTTP 包 → Follow → TCP Stream
5. 你能看到完整的 HTTP 请求头、响应头和响应体（文件内容）

**练习二：从 pcap 中提取文件**

目标：学会从抓包数据中恢复传输的文件。

```bash
# 方法一：使用 tshark 提取 HTTP 对象
tshark -r capture.pcap --export-objects http,./extracted_files/
ls ./extracted_files/
# 你会看到下载的文件

# 方法二：使用 NetworkMiner（GUI 工具）
# 安装：sudo apt install networkminer
# 打开 pcap 文件，自动提取文件、图片、凭据等

# 方法三：使用 Wireshark 内置功能
# File → Export Objects → HTTP → 选择文件 → Save
```

**练习三：FTP 明文凭据提取**

目标：从 FTP 流量中提取用户名和密码。

```bash
# 启动 FTP 服务器（Docker 方式）
docker run -d --name ftpd -p 21:21 -p 30000-30010:30000-30010 \
  -e FTP_USER=testuser -e FTP_PASS=SuperSecret123 stilliard/pure-ftpd

# 启动抓包
sudo tcpdump -i lo -w /tmp/ftp_capture.pcap tcp port 21 &

# 用 FTP 客户端连接
ftp localhost
# 输入用户名 testuser，密码 SuperSecret123
# 执行几个命令：ls, pwd, quit

# 分析抓包
tshark -r /tmp/ftp_capture.pcap -Y "ftp.request.command == USER || ftp.request.command == PASS" \
  -T fields -e ftp.request.arg
```

**预期输出**：

```text
testuser
SuperSecret123
```

**关键认知**：FTP、Telnet、HTTP、POP3、IMAP（非 SSL/TLS 版本）等协议的凭据全部明文传输。这就是为什么现代系统都要求使用 SFTP、SSH、HTTPS、IMAPS 等加密替代方案。

### 2.4 高级练习：异常流量识别

**练习一：ARP 欺骗流量识别**

目标：学会在抓包数据中识别 ARP 欺骗攻击的特征。

步骤：
1. 在正常网络中抓取 5 分钟的 ARP 流量作为基线：
   ```bash
   sudo tcpdump -i eth0 -n arp -c 100 -w /tmp/arp_baseline.pcap
   ```
2. 在 Kali 上使用 arpspoof 发起 ARP 欺骗（见第四节详细步骤）
3. 同时在目标机上抓取 ARP 流量：
   ```bash
   sudo tcpdump -i eth0 -n arp -w /tmp/arp_attack.pcap
   ```
4. 用 Wireshark 对比两个 pcap 文件

**ARP 欺骗的典型特征**：

```text
异常特征 1：短时间内出现大量 ARP 响应（没有对应的请求）
异常特征 2：同一个 IP 对应的 MAC 地址发生变化
异常特征 3：MAC 地址出现非厂商前缀（如 00:0c:29 开头是 VMware 虚拟网卡）
异常特征 4：ARP 包的发送频率远高于正常水平
```

在 Wireshark 中使用以下过滤器快速定位异常：

```text
# 检测 ARP 缓存翻转（同一 IP 对应多个 MAC）
arp.duplicate-address-detected

# 查看所有 ARP 响应
arp.opcode == 2

# 检测 ARP 响应中的 MAC 与已知网关 MAC 不符
arp.src.hw_mac != aa:bb:cc:dd:ee:ff && arp.opcode == 2
```

**练习二：端口扫描流量识别**

目标：学会识别 Nmap 扫描产生的流量特征。

```bash
# 在靶机上启动抓包
sudo tcpdump -i eth0 -n -w /tmp/scan_detect.pcap &

# 从 Kali 上执行不同类型的扫描

nmap -sT 192.168.56.100    # 全连接扫描

# 分析抓包

```

**各类扫描的流量特征**：

| 扫描类型 | 流量特征 | Wireshark 过滤器 |
|---------|---------|-----------------|
| SYN 扫描 | 大量 SYN 包，收到 SYN/ACK 后立即 RST | `tcp.flags.syn==1 && tcp.flags.ack==0 && tcp.flags.reset==0` |
| 全连接扫描 | 完整三次握手后正常 RST 关闭 | `tcp.flags.syn==1 && tcp.flags.ack==0`，连接后有 RST |
| UDP 扫描 | 大量 UDP 包发往不同端口，收到 ICMP Port Unreachable | `icmp.type==3 && icmp.code==3` |
| XMAS 扫描 | FIN+PSH+URG 标志同时设置的包 | `tcp.flags.fin==1 && tcp.flags.push==1 && tcp.flags.urg==1` |
| NULL 扫描 | 没有任何标志位的 TCP 包 | `tcp.flags==0x000` |

**练习三：DNS 隧道检测**

目标：识别利用 DNS 协议进行数据外泄的异常行为。

DNS 隧道的原理：攻击者将数据编码后嵌入 DNS 查询的域名中（如 `aGVsbG8gd29ybGQ.evil.com`），由于 DNS 流量通常不被防火墙拦截，数据可以通过 DNS 查询悄悄传出。

```bash
# 正常 DNS 查询（域名短，类型单一）
dig www.google.com A
# 域名长度：14 字节

# DNS 隧道工具（如 iodine、dnscat2）产生的查询
# 域名长度可达 253 字节，包含大量随机编码字符
# 如：aGVsbG8gd29ybGQgdGhpcyBpcyBhIHRlc3Q.evil.com
```

**DNS 隧道的检测特征**：

1. **异常长的域名**：正常域名通常不超过 50 字节，隧道域名可能接近 253 字节上限
2. **高频率的 DNS 查询**：正常用户每秒 1-5 次，隧道可能每秒几十到几百次
3. **大量 TXT 或 NULL 记录查询**：隧道工具常用这些类型传输更多数据
4. **域名中包含 Base64/Hex 编码特征**：连续的字母数字组合，无实际语义
5. **单个域名下的子域名数量异常多**：如 `a.evil.com`、`ab.evil.com`、`abc.evil.com`...

在 Wireshark 中使用以下过滤器：

```text
# 查找超长域名的 DNS 查询
dns.qry.name.len > 50

# 查找 TXT 记录查询
dns.qry.type == 16
```

### 2.5 tshark 自动化分析脚本

**批量分析 pcap 文件**：

```bash
#!/bin/bash
# analyze_pcap.sh - 批量分析 pcap 文件
# 用法：./analyze_pcap.sh /path/to/pcap/dir

PCAP_DIR="$1"
REPORT="/tmp/pcap_analysis_$(date +%Y%m%d_%H%M%S).txt"

echo "=== PCAP 批量分析报告 ===" > "$REPORT"
echo "分析时间：$(date)" >> "$REPORT"
echo "" >> "$REPORT"

for pcap in "$PCAP_DIR"/*.pcap; do
    [ -f "$pcap" ] || continue
    echo "======================================" >> "$REPORT"
    echo "文件：$pcap" >> "$REPORT"
    echo "======================================" >> "$REPORT"

    # 基本统计
    echo "--- 基本统计 ---" >> "$REPORT"
    capinfos "$pcap" >> "$REPORT" 2>/dev/null

    # 协议分层
    echo "" >> "$REPORT"
    echo "--- 协议分布 ---" >> "$REPORT"
    tshark -r "$pcap" -z io,phs -q >> "$REPORT" 2>/dev/null

    # 会话统计
    echo "" >> "$REPORT"
    echo "--- TCP 会话（前10） ---" >> "$REPORT"
    tshark -r "$pcap" -z conv,tcp -q 2>/dev/null | head -15 >> "$REPORT"

    # DNS 查询
    echo "" >> "$REPORT"
    echo "--- DNS 查询 ---" >> "$REPORT"
    tshark -r "$pcap" -Y "dns.qry.name" -T fields \
      -e frame.time -e dns.qry.name -e dns.qry.type 2>/dev/null | \
      sort -u | head -20 >> "$REPORT"

    # HTTP 请求
    echo "" >> "$REPORT"
    echo "--- HTTP 请求 ---" >> "$REPORT"
    tshark -r "$pcap" -Y "http.request" -T fields \
      -e frame.time -e http.request.method -e http.host -e http.request.uri 2>/dev/null | \
      head -20 >> "$REPORT"

    # 异常检测
    echo "" >> "$REPORT"
    echo "--- 异常检测 ---" >> "$REPORT"

    # RST 包数量（可能的端口扫描）
    RST_COUNT=$(tshark -r "$pcap" -Y "tcp.flags.reset==1" 2>/dev/null | wc -l)
    echo "RST 包数量：$RST_COUNT" >> "$REPORT"

    # ICMP 不可达数量
    ICMP_UNREACH=$(tshark -r "$pcap" -Y "icmp.type==3" 2>/dev/null | wc -l)
    echo "ICMP 不可达：$ICMP_UNREACH" >> "$REPORT"

    echo "" >> "$REPORT"
done

echo "分析完成，报告保存在：$REPORT"
cat "$REPORT"
```

**使用 Python + Scapy 进行自动化分析**：

```python
#!/usr/bin/env python3
"""pcap_analyzer.py - 使用 Scapy 分析网络流量"""
from scapy.all import *
from collections import Counter
import sys

def analyze_pcap(pcap_file):
    packets = rdpcap(pcap_file)
    print(f"[*] 总数据包数: {len(packets)}")

    # 协议分布
    proto_counter = Counter()
    for pkt in packets:
        if pkt.haslayer(TCP):
            proto_counter['TCP'] += 1
        elif pkt.haslayer(UDP):
            proto_counter['UDP'] += 1
        elif pkt.haslayer(ICMP):
            proto_counter['ICMP'] += 1
        elif pkt.haslayer(ARP):
            proto_counter['ARP'] += 1

    print("\n[*] 协议分布:")
    for proto, count in proto_counter.most_common():
        print(f"    {proto}: {count} ({count/len(packets)*100:.1f}%)")

    # IP 通信对
    ip_pairs = Counter()
    for pkt in packets:
        if pkt.haslayer(IP):
            src = pkt[IP].src
            dst = pkt[IP].dst
            ip_pairs[(src, dst)] += 1

    print("\n[*] Top 10 IP 通信对:")
    for (src, dst), count in ip_pairs.most_common(10):
        print(f"    {src} -> {dst}: {count} 包")

    # 端口统计
    port_counter = Counter()
    for pkt in packets:
        if pkt.haslayer(TCP):
            port_counter[pkt[TCP].dport] += 1
        elif pkt.haslayer(UDP):
            port_counter[pkt[UDP].dport] += 1

    print("\n[*] Top 10 目标端口:")
    for port, count in port_counter.most_common(10):
        service = {21:'FTP', 22:'SSH', 23:'Telnet', 25:'SMTP',
                   53:'DNS', 80:'HTTP', 443:'HTTPS', 3389:'RDP'
                  }.get(port, 'Unknown')
        print(f"    端口 {port} ({service}): {count} 包")

    # ARP 异常检测
    arp_table = {}
    arp_anomalies = []
    for pkt in packets:
        if pkt.haslayer(ARP) and pkt[ARP].op == 2:  # ARP Reply
            ip = pkt[ARP].psrc
            mac = pkt[ARP].hwsrc
            if ip in arp_table and arp_table[ip] != mac:
                arp_anomalies.append(
                    f"    ARP 翻转: {ip} MAC 从 {arp_table[ip]} 变为 {mac}"
                )
            arp_table[ip] = mac

    if arp_anomalies:
        print("\n[!] ARP 异常检测:")
        for a in arp_anomalies:
            print(a)
    else:
        print("\n[+] ARP 检测：未发现异常")

    # SYN 扫描检测
    syn_count = sum(1 for pkt in packets
                    if pkt.haslayer(TCP) and pkt[TCP].flags == 'S')
    rst_count = sum(1 for pkt in packets
                    if pkt.haslayer(TCP) and pkt[TCP].flags & 0x04)

    print(f"\n[*] SYN 包数量: {syn_count}")
    print(f"[*] RST 包数量: {rst_count}")
    if rst_count > syn_count * 0.3 and rst_count > 50:
        print("[!] 疑似端口扫描：RST 比例异常高")

if __name__ == '__main__':
    if len(sys.argv) != 2:
        print(f"用法: {sys.argv[0]} <pcap文件>")
        sys.exit(1)
    analyze_pcap(sys.argv[1])
```

运行方式：

### 3.2 扫描技术深度对比

**练习目标**：通过实际对比不同扫描技术的结果，理解它们的原理差异和适用场景。

在同一个目标上执行以下扫描，并记录结果：

```bash
# TCP SYN 扫描（半开扫描，默认需要 root）
sudo nmap -sS 192.168.56.100 -oN /tmp/scan_syn.txt

# TCP Connect 扫描（完整三次握手）
nmap -sT 192.168.56.100 -oN /tmp/scan_connect.txt

# TCP ACK 扫描（探测防火墙规则）
sudo nmap -sA 192.168.56.100 -oN /tmp/scan_ack.txt

# TCP Window 扫描
sudo nmap -sW 192.168.56.100 -oN /tmp/scan_window.txt

# TCP NULL 扫描（无标志位）
sudo nmap -sN 192.168.56.100 -oN /tmp/scan_null.txt

# TCP FIN 扫描
sudo nmap -sF 192.168.56.100 -oN /tmp/scan_fin.txt

# TCP XMAS 扫描（FIN+PSH+URG）
sudo nmap -sX 192.168.56.100 -oN /tmp/scan_xmas.txt

# UDP 扫描
sudo nmap -sU 192.168.56.100 -oN /tmp/scan_udp.txt

# IDLE 扫描（需要一个僵尸主机）
sudo nmap -sI zombie_host:port 192.168.56.100 -oN /tmp/scan_idle.txt
```

**各扫描技术对比记录表**（自己填写）：

| 扫描类型 | 发现开放端口数 | 发现关闭端口数 | 扫描速度 | 隐蔽性 | 是否需要 root |
|---------|--------------|--------------|---------|--------|-------------|
| SYN (-sS) | | | | | 是 |
| Connect (-sT) | | | | | 否 |
| ACK (-sA) | | | | | 是 |
| NULL (-sN) | | | | | 是 |
| FIN (-sF) | | | | | 是 |
| XMAS (-sX) | | | | | 是 |
| UDP (-sU) | | | | | 是 |

**每种扫描的工作原理**：

- **SYN 扫描**：发送 SYN 包，收到 SYN/ACK 表示端口开放（然后发 RST 关闭连接，不完成握手），收到 RST 表示端口关闭。最常用、最可靠、最隐蔽。
- **Connect 扫描**：使用系统调用 `connect()` 完成完整三次握手。不需要 root 权限，但会在目标日志中留下完整连接记录。
- **NULL/FIN/XMAS 扫描**：利用 RFC 793 的实现差异——合规的 TCP 实现对关闭端口返回 RST，对开放端口不响应。对 Windows 系统无效（Windows 不遵守此 RFC）。
- **ACK 扫描**：专门用于探测防火墙规则。有状态防火墙会丢弃未匹配的 ACK 包（端口被过滤），无状态防火墙或无防火墙会返回 RST（端口未被过滤）。
- **IDLE 扫描**：利用僵尸主机的 IP ID 值变化来推断目标端口状态，完全不暴露扫描者的真实 IP。

**练习目标**：不仅能在 Wireshark 中看到 TCP 状态转换，还能理解每个状态转换的原因和安全含义。

**练习一：正常连接的完整生命周期**

```bash

s.send(b'Hello from client')

s.close()
"
```

**你应该在 `watch` 终端观察到以下状态变化**：

```text
服务器端：
  LISTEN     → 等待连接
  SYN-RECV   → 收到 SYN，发送 SYN/ACK
  ESTABLISHED → 收到 ACK，连接建立
  CLOSE-WAIT → 收到 FIN，应用层还没关闭
  LAST-ACK   → 应用层关闭，发送 FIN
  (消失)     → 收到 ACK，连接完全关闭

客户端：
  SYN-SENT   → 发送 SYN
  ESTABLISHED → 收到 SYN/ACK，发送 ACK
  FIN-WAIT-1 → 应用层关闭，发送 FIN
  FIN-WAIT-2 → 收到对方的 ACK
  TIME-WAIT  → 收到对方的 FIN，发送 ACK
  (消失)     → 2MSL 超时后关闭
```

**安全含义**：
- **SYN-RECV 状态过多**：可能是 SYN Flood 攻击
- **TIME-WAIT 状态过多**：可能是高并发短连接导致的端口耗尽
- **CLOSE-WAIT 状态过多**：应用程序没有正确关闭连接（资源泄漏）

### 6.2 高级数据包构造

```python
#!/usr/bin/env python3
"""scapy_advanced.py - Scapy 高级练习"""
from scapy.all import *
import random

# SYN 端口扫描
def syn_scan(target, ports):
    """SYN 扫描指定目标的指定端口"""
    results = {}
    for port in ports:
        pkt = IP(dst=target) / TCP(dport=port, flags="S")
        reply = sr1(pkt, timeout=1, verbose=0)
        if reply is None:
            results[port] = "filtered"
        elif reply[TCP].flags == 0x12:  # SYN/ACK
            # 发送 RST 关闭连接
            send(IP(dst=target) / TCP(dport=port, flags="R"), verbose=0)
            results[port] = "open"
        elif reply[TCP].flags == 0x14:  # RST/ACK
            results[port] = "closed"
        else:
            results[port] = f"unknown ({reply[TCP].flags})"
    return results

# 执行扫描
ports = [21, 22, 23, 25, 80, 443, 3306, 5432, 8080]
results = syn_scan("192.168.56.100", ports)
for port, status in sorted(results.items()):
    print(f"  端口 {port}: {status}")
```

```python
#!/usr/bin/env python3
"""traceroute_scapy.py - 使用 Scapy 实现 Traceroute"""
from scapy.all import *

def traceroute(target, max_hops=30):
    """使用 Scapy 执行 traceroute"""
    print(f"Traceroute to {target}")
    for ttl in range(1, max_hops + 1):
        pkt = IP(dst=target, ttl=ttl) / ICMP()
        reply = sr1(pkt, timeout=2, verbose=0)
        if reply is None:
            print(f"  {ttl}: * * *")
        elif reply[ICMP].type == 0:  # Echo Reply（到达目标）
            print(f"  {ttl}: {reply[IP].src} (到达目标)")
            break
        else:
            print(f"  {ttl}: {reply[IP].src} (TTL={reply[IP].ttl})")

traceroute("8.8.8.8")
```

```python
from collections import defaultdict

# 统计每个 IP 的数据包数量
```

```python
import socket
import threading

def handle_client(conn, addr):
    """处理客户端连接"""

        pass
    finally:
        conn.close()
        print(f"[-] 连接关闭: {addr}")

def start_server(host='0.0.0.0', port=9999):
    """启动 TCP 服务器"""
    server = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    server.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
    server.bind((host, port))
    server.listen(5)
    print(f"[*] 服务器监听 {host}:{port}")

    try:
        while True:
            conn, addr = server.accept()
            thread = threading.Thread(target=handle_client, args=(conn, addr))
            thread.daemon = True
            thread.start()
    except KeyboardInterrupt:
        print("\n[*] 服务器关闭")
        server.close()

if __name__ == '__main__':
    start_server()
```

```python
#!/usr/bin/env python3
"""port_scanner.py - 纯 Python 端口扫描器"""
import socket
import threading
from queue import Queue

def scan_port(target, port, results):
    """扫描单个端口"""
    try:
        s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
        s.settimeout(1)
        result = s.connect_ex((target, port))
        if result == 0:
            try:
                service = socket.getservbyport(port)
            except OSError:
                service = "unknown"
            results.append((port, "open", service))
        s.close()
    except Exception:
        pass

def threaded_scan(target, ports, threads=100):
    """多线程端口扫描"""
    results = []
    port_queue = Queue()

    for port in ports:
        port_queue.put(port)

    def worker():
        while not port_queue.empty():
            try:
                port = port_queue.get_nowait()
                scan_port(target, port, results)
            except Exception:
                break

    thread_list = []
    for _ in range(min(threads, len(ports))):
        t = threading.Thread(target=worker)
        t.daemon = True
        t.start()
        thread_list.append(t)

    for t in thread_list:
        t.join()

    return sorted(results, key=lambda x: x[0])

if __name__ == '__main__':
    target = "192.168.56.100"
    ports = range(1, 1025)
    print(f"扫描 {target} 的端口 1-1024...")

    results = threaded_scan(target, ports)
    print(f"\n发现 {len(results)} 个开放端口：")
    for port, state, service in results:
        print(f"  {port}/{state} - {service}")
```

```python
"""banner_grabber.py - 服务 Banner 抓取"""
import socket

def grab_banner(ip, port, timeout=3):
    """抓取服务 Banner 信息"""
    try:
        s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
        s.settimeout(timeout)
        s.connect((ip, port))

        # 对于 HTTP 服务，发送请求
        if port in (80, 8080, 8000):
            s.send(b"HEAD / HTTP/1.1\r\nHost: " +
                   ip.encode() + b"\r\n\r\n")
        # 对于 SMTP，直接接收 Banner
        # 对于其他服务，等待服务主动发送 Banner

        banner = s.recv(4096)
        s.close()
        return banner.decode('utf-8', errors='replace').strip()
    except Exception as e:
        return f"Error: {e}"

    if banner and "Error" not in banner:
        print(f"\n  端口 {port}:")
        # 只显示前 3 行
```
