# 实战案例

## 一、本节定位

实战案例是理论走向生产的桥梁。在前两节中，你已经掌握了微服务、无服务器、服务网格、事件驱动等理论范式（理论基础），以及容器化、微服务拆分、服务编排等工程方法（核心技巧）。本节将这些知识整合到两个真实业务场景中，让你看到一个完整的云原生项目是如何从规划、搭建到运维落地的。

**为什么实战案例重要？**

- 理论知识告诉你"应该怎么做"，但真实项目中充满了权衡和妥协——为什么这里选 Calico 而不是 Cilium？为什么 HPA 的缩容窗口要比扩容窗口长？这些决策背后的逻辑只有在具体场景中才能理解
- 生产环境的复杂性远超教科书：多语言技术栈共存、大促流量洪峰、数据库连接池耗尽、缓存穿透——每一个都是理论课不会教的坑
- 案例提供可复用的模板：Deployment、HPA、Ingress、ConfigMap/Secret 的完整 YAML 可以直接作为你项目的起点

## 二、案例总览

本节包含两个案例，分别覆盖云原生架构中最核心的两个实战场景：

| 维度 | 案例一：Kubernetes 部署实战 | 案例二：性能诊断与优化 |
|------|---------------------------|----------------------|
| **核心主题** | 电商平台从 VM 迁移到 K8s，实现弹性伸缩 | 大促期间性能瓶颈的排查与根治 |
| **规模** | 日活 50 万，峰值 QPS 3 万 | 影响 100 万用户，持续 30 分钟 |
| **技术栈** | Kubernetes + Helm + Calico + Prometheus | Linux 性能工具链 + JVM + MySQL + Redis |
| **关键收获** | 集群规划、Deployment 编排、HPA 自动扩缩、ConfigMap/Secret 管理 | 监控数据分析、应用层排查、数据库调优、缓存策略 |
| **前置知识** | Docker 基础（第40章）、Helm 基本概念 | Linux 性能工具、JVM 调优基础、SQL 优化 |
| **预计阅读** | 60-90 分钟 | 30-45 分钟 |

## 三、案例一：电商平台 Kubernetes 部署与弹性伸缩实战

**文件位置：** [案例一：Kubernetes 实战](01-案例一Kubernetes实战.md)

### 3.1 场景概述

某中型电商平台（日活约 50 万，峰值 QPS 约 3 万）面临传统 VM 部署的典型困境：扩容需要采购和分配虚拟机，耗时 2-4 小时；资源按峰值预留，日常利用率仅 20-30%；发布无回滚能力，环境一致性差。本案例完整还原从 VM 迁移到 Kubernetes 集群的全过程。

### 3.2 技术架构

平台采用微服务架构，核心服务包括：

- **gateway**（Go）：API 网关、鉴权、限流
- **user-service**（Java）：用户注册、登录、个人信息
- **product-service**（Java）：商品查询、搜索、详情
- **order-service**（Java）：下单、支付回调、订单查询
- **inventory-service**（Go）：库存扣减、预占、回滚
- **cart-service**（Node.js）：购物车增删改查
- **notification-service**（Python）：短信、邮件、推送通知
- **MySQL 主从**：有状态服务，StatefulSet 部署
- **Redis Cluster**：缓存层，6 节点集群

### 3.3 核心内容

案例覆盖了 Kubernetes 生产部署的完整链路：

**1. 集群规划与搭建**
- 混合规格节点池设计（app-pool / data-pool / monitor-pool / system-pool）
- 网络方案选型对比（Calico vs Flannel vs Cilium），本案例选择 Calico
- 存储方案：SSD 云盘用于有状态服务，NFS 用于应用层共享存储

**2. 应用容器化**
- 多阶段构建 Dockerfile（Maven 依赖层与运行层分离，镜像从 800MB 压缩到 120MB）
- 非 root 运行、容器感知 JVM 参数、HEALTHCHECK 配置
- 五种探针的协同：startupProbe / livenessProbe / readinessProbe

**3. Deployment 编排**
- 滚动更新策略（maxUnavailable / maxSurge）
- 跨可用区拓扑约束（topologySpreadConstraints）
- 安全上下文、资源 requests/limits、ConfigMap 挂载、Secret 注入
- 优雅终止配置（terminationGracePeriodSeconds）

**4. Service 与 Ingress**
- ClusterIP 内部通信 + Nginx Ingress 统一对外暴露
- SSL 终止、限流、超时、金丝雀发布（canary-weight: 10%）

**5. 自动弹性伸缩（HPA）**
- 基于 CPU 利用率的自动扩缩
- 基于自定义指标（每秒订单数）的业务级扩缩
- 基于外部指标（消息队列积压量）的事件驱动扩缩
- 扩容稳定窗口 30 秒 vs 缩容稳定窗口 5 分钟的设计权衡

**6. ConfigMap 与 Secret 管理**
- 非敏感配置通过 ConfigMap 注入，支持热更新
- 敏感信息通过 Secret 管理，KMS 加密存储
- 热更新机制：ConfigMap 更新后如何让 Pod 感知（重挂载 vs 应用刷新）

**7. Helm Chart 模板化部署**
- 多环境管理（dev / staging / prod）
- 模板化变量注入
- Chart 版本管理与回滚

**8. 大促弹性伸缩实战**
- 大促前：预扩容、资源预留、压力测试
- 大促中：实时监控、自动扩缩、手动干预预案
- 大促后：缩容、成本核算、复盘

### 3.4 学习要点

通过本案例，你将掌握：
- Kubernetes 集群的生产级规划方法
- Deployment、Service、Ingress、HPA、PDB 等核心资源的协作关系
- 多指标 HPA 的配置技巧（CPU + 自定义指标 + 外部指标）
- ConfigMap/Secret 的安全配置与热更新
- Helm Chart 管理多环境部署的完整流程
- 大促场景下的弹性伸缩与故障恢复能力

## 四、案例二：大促期间性能诊断与优化实战

**文件位置：** [案例二：诊断命令](02-案例二Istio实战.md)

### 4.1 场景概述

某电商平台在大促预热期间遭遇严重性能问题：接口响应时间从平均 50ms 飙升至 500ms，部分请求超时，服务器 CPU 持续 90% 以上，数据库连接池耗尽。本次事件影响约 100 万用户，持续约 30 分钟，预估业务损失数十万元。

### 4.2 核心内容

本案例展示了从问题发现到根因定位再到方案实施的完整排查链路：

**1. 监控数据分析**
- 系统负载：`uptime` 显示 load average 达 25.50
- CPU 状态：`top -c` 显示 95% 使用率
- 内存状态：`free -m` 显示 85% 使用率
- I/O 状态：`iostat -x 1` 显示 %util 达 98%

**2. 应用层排查**
- 应用日志分析：`tail -f /var/log/app.log | grep ERROR`
- 线程状态分析：`jstack` 查看 RUNNABLE / BLOCKED 线程数
- GC 日志分析：排查 Full GC 导致的停顿

**3. 数据库排查**
- 慢查询分析：`SHOW PROCESSLIST` + `information_schema.processlist`
- 锁等待分析：`innodb_lock_waits` 表
- 索引使用情况：`EXPLAIN` 执行计划分析

**4. 根因定位**
- 数据库缺少合适的复合索引，导致全表扫描
- 连接池配置过小（默认值），无法支撑高并发
- 缓存命中率低，大量请求穿透到数据库

**5. 优化方案**
- **数据库索引优化**：添加复合索引 `idx_user_status` 和覆盖索引 `idx_user_time`
- **连接池调优**：HikariCP 参数调整（maximum-pool-size: 50, minimum-idle: 10）
- **多级缓存策略**：L1 本地缓存 + L2 Redis 缓存 + 数据库，逐级回源

### 4.3 优化效果

| 指标 | 优化前 | 优化后 | 提升幅度 |
|------|--------|--------|---------|
| P99 延迟 | 500ms | 50ms | 降低 90% |
| QPS | 5,000 | 50,000 | 提升 10 倍 |
| 错误率 | 5% | 0.1% | 降低 98% |

### 4.4 学习要点

通过本案例，你将掌握：
- Linux 系统性能排查的标准流程（CPU → 内存 → I/O → 网络）
- JVM 层面的性能诊断方法（线程分析、GC 日志解读）
- MySQL 慢查询与锁等待的排查技巧
- 数据库索引优化和连接池调优的具体方法
- 多级缓存架构的设计与实现
- 从问题到根因的系统性排查思维

## 五、两个案例的关联与递进

案例一和案例二并非孤立存在，它们构成了云原生架构从"建设"到"运维"的完整闭环：

```mermaid
graph LR
    subgraph 案例一：建设阶段
        A[集群规划] --> B[应用容器化]
        B --> C[Deployment 编排]
        C --> D[HPA 弹性伸缩]
        D --> E[大促上线]
    end

    subgraph 案例二：运维阶段
        E --> F[性能监控]
        F --> G[问题发现]
        G --> H[根因分析]
        H --> I[优化实施]
        I --> J[效果验证]
    end

    J -.->|经验反馈| A
```

- **案例一**解决了"怎么建"的问题：集群搭建、应用部署、自动扩缩
- **案例二**解决了"怎么养"的问题：监控告警、性能诊断、根因优化
- 两者的共同背景都是电商大促场景，可以对照阅读，理解同一业务在不同阶段面临的不同挑战

## 六、实战学习路径建议

根据你的角色和目标，建议以下学习路径：

### 路径一：全栈工程师（从零到一）

1. **先读理论基础**：微服务架构设计 + 容器化 → 建立基本概念
2. **再读核心技巧**：容器化深入 + 微服务拆分 → 掌握方法论
3. **精读案例一**：跟着 YAML 一步步搭建 → 动手实践
4. **精读案例二**：学习排查流程 → 建立运维思维
5. **动手实验**：用 minikube 或 kind 搭建本地集群，复现案例中的 Deployment 和 HPA

### 路径二：DevOps/SRE（运维视角）

1. **跳过基础理论**：直接看案例一的集群规划、网络方案、存储方案
2. **重点学习 HPA**：多指标自动伸缩的配置技巧
3. **精读案例二**：性能排查的标准流程是 SRE 的核心技能
4. **延伸学习**：Prometheus 告警规则、Grafana Dashboard 设计、PagerDuty 集成

### 路径三：架构师（设计决策）

1. **快速浏览案例一**：重点关注节点池设计、网络选型、存储方案的权衡理由
2. **深度阅读案例二**：理解从监控数据到根因分析的决策链
3. **思考扩展**：如果你的业务不是电商，哪些设计可以复用？哪些需要调整？
4. **对比评估**：Istio 服务网格方案 vs 传统 SDK 集成方案的优劣对比

## 七、从案例到实践：关键提醒

### 7.1 不要照搬，要理解决策逻辑

案例中的每个配置都基于特定的业务场景：order-service 的 HPA maxReplicas=12 是基于 QPS 计算得出的，不是拍脑袋决定的。在你的项目中，需要根据自己的业务特征重新计算。

### 7.2 监控是基础设施，不是锦上添花

案例二中之所以能快速定位问题，前提是已有完善的监控体系。在引入任何分布式架构之前，先建立 Prometheus + Grafana + 链路追踪的可观测性基础。

### 7.3 渐进式迁移，不要一步到位

电商平台从 VM 迁移到 K8s 也是渐进式的：先迁移无状态服务，验证稳定后再迁移有状态服务。不要试图一次性将所有服务云原生化。

### 7.4 成本控制不可忽视

弹性伸缩的核心价值之一是成本优化：日常只运行必要副本，大促时按需扩缩。但要注意：HPA 扩容依赖节点池有足够余量，如果节点池满了，Cluster Autoscaler 会自动补充节点——这会产生额外的云资源费用。

### 7.5 安全内建，事后补救代价巨大

案例一中的非 root 运行、Secret 加密存储、NetworkPolicy 网络隔离等安全措施，都是在设计阶段就内建的。安全不是上线后再考虑的事情。
