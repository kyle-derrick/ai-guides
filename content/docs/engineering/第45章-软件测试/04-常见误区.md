## 常见误区

软件测试领域存在大量根深蒂固的错误观念。这些误区不是初学者的专利——许多有十年经验的资深工程师也深陷其中。更危险的是，这些误区往往被包装成"行业惯例"或"最佳实践"，导致团队在错误的方向上越走越远。

本节逐一拆解最常见的 **15 个测试误区**，分析其产生的原因、带来的危害，并给出经过验证的纠正方法。每个误区都配有真实场景、代码示例和可执行的改进方案。

### 误区一：代码覆盖率 100% = 零缺陷

这是测试领域流传最广的神话。许多团队将代码覆盖率作为质量的核心指标，甚至设置"覆盖率不达标不让合并"的硬性门槛。

**为什么这是错的**

代码覆盖率衡量的是"哪些代码被执行过"，而非"哪些行为被验证过"。两者有本质区别：

```python
# 一个覆盖率 100% 但毫无价值的测试
def test_calculate_discount():
    result = calculate_discount(100, 0.1)
    assert result is not None  # 只验证返回值不为 None

# 这个测试覆盖了 calculate_discount 函数，但完全没有验证正确性
# 即使函数返回 -9999 也能通过

# 真正有价值的测试
def test_calculate_discount_basic():
    assert calculate_discount(100, 0.1) == 90.0

def test_calculate_discount_zero_rate():
    assert calculate_discount(100, 0.0) == 100.0

def test_calculate_discount_full_rate():
    assert calculate_discount(100, 1.0) == 0.0

def test_calculate_discount_exceeds_max():
    assert calculate_discount(100, 1.5) == 0.0  # 折扣率超过100%应返回0

def test_calculate_discount_negative_price():
    assert calculate_discount(-100, 0.1) < 0  # 负价格的行为
```

微软研究院的经典研究发现：**覆盖率从 0% 提升到 60% 时，缺陷检出率显著提升；从 60% 到 80% 提升变缓；从 80% 到 100% 几乎没有额外收益**。更关键的是，Google 的内部数据表明，许多覆盖率极高的模块依然存在严重的生产事故——因为覆盖率没有覆盖到"边界条件""异常路径""并发场景"这些最容易出问题的地方。

**真实案例**

某金融系统达到了 97% 的代码覆盖率，团队信心满满地发布了新版本。上线后第一天就出现了严重的利率计算错误。根因分析发现：所有测试都使用了整数金额，从未测试过浮点精度问题。97% 的覆盖率，一个浮点 bug 就能击穿。

**纠正方法**

| 指标 | 用途 | 局限性 |
|------|------|--------|
| 代码覆盖率 | 发现未测试的代码 | 不衡量测试质量 |
| 分支覆盖率 | 发现未覆盖的逻辑分支 | 不衡量边界条件 |
| 变异测试 | 评估测试的"杀伤力" | 计算成本较高 |
| 缺陷逃逸率 | 衡量测试的真实效果 | 需要生产环境数据 |

实际操作中，建议采用**多维质量指标组合**：

```yaml
# 质量门禁配置示例
quality_gates:
  coverage:
    line: ">= 80%"      # 基本门槛，不是终极目标
    branch: ">= 70%"
  mutation:
    kill_rate: ">= 75%"  # 变异测试存活率——更有说服力
  regression:
    escape_rate: "< 2%"  # 上个季度的缺陷逃逸率
```

### 误区二：测试是测试团队的事

这是组织层面最昂贵的误解。当开发人员认为"写代码是我的活，找 bug 是测试的活"时，质量成本会急剧上升。

**经济学真相**

IBM Systems Sciences Institute 的经典研究数据显示：**需求阶段发现的缺陷修复成本为 1x，设计阶段 5x，编码阶段 10x，测试阶段 15x，生产环境 30-100x**。换句话说，一个在需求评审中花 5 分钟就能发现的问题，到生产环境可能要花 50 万来修复。

当测试团队是唯一负责质量的群体时，问题发现的时间点被推后，修复成本成倍增加。更糟糕的是，开发人员因为"有测试兜底"而降低自测意识，导致大量低质量代码涌入测试环节。

**全链路质量责任模型**

需求阶段：产品经理负责需求可测试性，测试人员参与需求评审
    ↓
设计阶段：架构师负责可测试性设计，测试人员编写测试策略
    ↓
编码阶段：开发者编写单元测试 + 集成测试，执行本地质量门禁
    ↓
测试阶段：测试人员执行系统测试、探索式测试、性能测试
    ↓
发布阶段：SRE 负责线上监控、灰度发布、快速回滚

**实操方案**

在 CI 流水线中强制设置质量关卡，让"质量"成为每个人的职责而非某个团队的专属工作：

```yaml
# .gitlab-ci.yml 示例
stages:
  - build
  - test
  - quality_gate
  - deploy

unit_tests:
  stage: test
  script:
    - pytest tests/unit/ --cov=src --cov-fail-under=80
  # 开发者必须通过此步骤才能进入下一阶段

code_review:
  stage: quality_gate
  script:
    - echo "必须有至少1位非作者的reviewer批准"
  rules:
    - if: $CI_MERGE_REQUEST_APPROVALS_COUNT < 1
      when: never
```

### 误区三：自动化测试可以完全替代手工测试

自动化测试被过度神化。许多团队投入大量资源搭建自动化框架，然后试图将所有测试活动自动化，结果发现投入产出比极低。

**什么适合自动化**

- **重复性高的回归测试**：每次发版都要跑的冒烟测试、核心流程验证
- **数据驱动的参数化测试**：同一流程不同输入组合的验证
- **性能基准测试**：需要精确控制并发数、响应时间阈值的场景
- **API 契约测试**：验证接口契约一致性的持续检查

**什么不适合自动化**

- **探索式测试**：测试人员凭经验和直觉发现非预期行为，这是创造性活动
- **可用性测试**：需要真人用户的视觉判断和交互感受
- **Ad-hoc 测试**：新功能上线前的快速验证，编写自动化脚本的 ROI 太低
- **复杂业务场景验证**：涉及多系统联动、需要人工判断结果正确性的场景

**真实数据**

ThoughtWorks 的行业报告指出：成熟团队的自动化测试通常覆盖总测试工作的 **30%-50%**，而非 100%。那些声称"100% 自动化"的团队，要么测试范围定义过窄，要么在自动化脚本的维护上投入了惊人的成本。

```python
# 一个典型的手工测试更适合的场景：UI 可用性验证
# 如果你试图自动化这个测试，你会陷入维护地狱

# 错误做法：自动化"按钮颜色是否正确"
def test_submit_button_color():
    button = driver.find_element(By.ID, "submit-btn")
    color = button.value_of_css_property("background-color")
    assert color == "rgb(0, 123, 255)"  # 每次 UI 改版都会挂

# 正确做法：自动化核心流程，手工验证视觉设计
def test_submit_order_flow():
    """自动化核心业务流程，UI细节由手工测试覆盖"""
    driver.get("/products")
    driver.find_element(By.ID, "add-to-cart").click()
    driver.find_element(By.ID, "checkout").click()
    driver.find_element(By.ID, "confirm-order").click()
    assert "order-confirmation" in driver.current_url
```

### 误区四：测试用例越多越好

许多团队用测试用例数量来衡量测试工作的"充实度"。测试用例从 500 条增长到 5000 条，团队觉得质量一定更好了。

**真相：冗余测试的代价**

每一条测试用例都有生命周期成本——编写、维护、执行、结果分析。当用例数量膨胀时：

- **执行时间变长**：CI 流水线从 10 分钟变成 2 小时，开发者等不起，开始跳过测试
- **维护成本飙升**：一次 UI 改版导致 300 条用例失效，修改它们需要两天
- **噪声掩盖信号**：5000 条用例中有 2000 条是低价值的重复验证，真正的失败被淹没在噪音中

**有效测试设计的原则**

等价类划分：将输入域划分为有限个等价类，每个类只取一个代表值
    ↓
边界值分析：聚焦边界条件（最大值、最小值、空值、刚好超出）
    ↓
判定表法：多条件组合的逻辑覆盖（尤其适合业务规则复杂的场景）
    ↓
状态转换法：验证系统状态机的每个转换路径
    ↓
错误推测法：基于经验和直觉，针对高风险区域设计攻击性用例

**具体示例**

假设要测试一个密码强度校验功能：

```python
# 低效做法：穷举所有字符组合（数百条用例）
# test_password_a, test_password_b, test_password_c ... 无意义

# 高效做法：等价类 + 边界值，15 条高质量用例覆盖所有场景
class TestPasswordStrength:
    # 等价类：有效密码
    def test_valid_mixed_case_and_digit(self):
        assert check_strength("Abc123") == "strong"

    def test_valid_with_special_char(self):
        assert check_strength("Abc@123") == "strong"

    # 边界值：最短/最长
    def test_minimum_length_boundary(self):
        assert check_strength("A1a") == "medium"  # 刚好 3 字符

    def test_maximum_length_boundary(self):
        assert check_strength("A" * 128 + "1") == "strong"

    # 等价类：无效密码
    def test_empty_password(self):
        assert check_strength("") == "invalid"

    def test_all_lowercase(self):
        assert check_strength("abcdef") == "weak"

    def test_no_digit_no_special(self):
        assert check_strength("Abcdef") == "weak"

    # 边界值：特殊字符组合
    def test_only_special_chars(self):
        assert check_strength("!@#$%^") == "weak"

    def test_unicode_characters(self):
        assert check_strength("密码Ab1") == "medium"

    # 攻击性用例
    def test_sql_injection_attempt(self):
        assert check_strength("' OR 1=1 --") == "invalid"

    def test_buffer_overflow_attempt(self):
        assert check_strength("A" * 10000) == "invalid"
```

15 条精心设计的用例，效果远超 300 条随机编写的用例。

### 误区五：单元测试就够了

测试金字塔（Test Pyramid）告诉我们，单元测试是基础，但不是全部。然而许多团队走到了另一个极端——只写单元测试，忽视集成测试和端到端测试。

**为什么单元测试不够**

单元测试在隔离环境中运行，验证的是单个函数/类的行为。但生产环境中，bug 往往发生在**组件之间的交互处**：

```python
# 单元测试：UserService 和 OrderService 都通过了
class TestUserService:
    def test_create_user(self):
        user = UserService(mock_db).create_user("alice@example.com")
        assert user.email == "alice@example.com"

class TestOrderService:
    def test_place_order(self):
        order = OrderService(mock_repo).place_order(user_id=1, items=[...])
        assert order.status == "confirmed"

# 但集成时出问题了：UserService 返回的 user_id 格式是 UUID
# 而 OrderService 期望的是整数
# 两个服务的单元测试都通过，但集成测试会失败

# 集成测试：发现接口契约不匹配
def test_user_to_order_integration():
    user = UserService(real_db).create_user("alice@example.com")
    # 报错：OrderService 期望 int，得到 UUID 字符串
    order = OrderService(real_repo).place_order(
        user_id=user.id, items=[{"sku": "ABC", "qty": 1}]
    )
    assert order.status == "confirmed"
```

**测试金字塔的合理配比**

          ╱╲
         ╱  ╲         端到端测试（少量）
        ╱ 5% ╲        ——关键业务流程验证
       ╱──────╲
      ╱        ╲       集成测试（适量）
     ╱   20%    ╲      ——组件间交互、API契约、数据库操作
    ╱────────────╲
   ╱              ╲    单元测试（大量）
  ╱      75%       ╲   ——业务逻辑、算法、工具函数
 ╱──────────────────╲

具体配比因项目而异。微服务架构中，集成测试的比例可能需要更高（30%-40%），因为服务间通信是最大的故障源。

### 误区六：Bug 修完就完了

修复 bug 只完成了一半工作。另一半是**根因分析（Root Cause Analysis）**——理解 bug 为什么会产生、如何防止再次出现。

**缺乏根因分析的后果**

同一个类型的 bug 会在不同模块反复出现。团队一直在"修 bug → 发现新 bug → 再修"的循环中消耗精力。

**5-Why 分析法**

对每个重要 bug，追问至少 5 层"为什么"：

Bug: 生产环境出现订单金额计算错误
  ↓ Why 1: 为什么金额错误？
  浮点数精度丢失
  ↓ Why 2: 为什么会精度丢失？
  代码使用 float 而非 Decimal
  ↓ Why 3: 为什么使用 float？
  开发者不了解金融计算的最佳实践
  ↓ Why 4: 为什么不了解？
  团队没有金融领域的编码规范
  ↓ Why 5: 为什么没有规范？
  没有人负责建立和维护编码规范

根因：缺乏领域编码规范
修复：建立金融计算编码规范 + 浮点数替代方案审查清单 + 单元测试模板

**将根因分析转化为预防措施**

```python
# 修复前
def calculate_total(items):
    total = 0
    for item in items:
        total += item.price * item.quantity  # float 精度问题
    return total

# 修复后（基于根因分析的系统性改进）
from decimal import Decimal, ROUND_HALF_UP

# 1. 建立工具函数，所有金额计算必须使用
def money_add(*values: Decimal) -> Decimal:
    """金额加法，统一精度处理"""
    return sum(values).quantize(Decimal("0.01"), rounding=ROUND_HALF_UP)

def money_multiply(price: Decimal, quantity: int) -> Decimal:
    """金额乘法"""
    return (price * quantity).quantize(Decimal("0.01"), rounding=ROUND_HALF_UP)

# 2. 修复原有代码
def calculate_total(items):
    return money_add(*[
        money_multiply(Decimal(str(item.price)), item.quantity)
        for item in items
    ])

# 3. 添加回归测试
def test_calculate_total_precision():
    """防止浮点精度问题再次出现"""
    items = [
        Decimal("0.1"),
        Decimal("0.2"),
        Decimal("0.3"),
    ]
    total = money_add(*items)
    assert total == Decimal("0.60")  # 不是 0.5999999999999999
```

### 误区七：测试环境和生产环境"差不多就行"

"差不多"是最危险的三个字。测试环境与生产环境的差异是导致"测试通过但生产故障"的头号原因。

**常见的环境差异陷阱**

| 差异维度 | 测试环境 | 生产环境 | 后果 |
|----------|----------|----------|------|
| 数据量 | 1000 条记录 | 1000 万条记录 | 查询超时、OOM |
| 并发数 | 5 个开发者同时测试 | 5000 个用户同时访问 | 死锁、连接池耗尽 |
| 网络延迟 | 本机 <1ms | 跨区域 50-200ms | 超时、重试风暴 |
| 操作系统 | macOS 开发机 | Linux 生产集群 | 文件路径、编码差异 |
| 依赖版本 | 测试用的 mock 服务 | 真实第三方服务 | 行为不一致 |
| 配置参数 | 默认配置 | 调优后的配置 | 性能差异巨大 |

**实操建议：生产镜像测试（Production Mirror Testing）**

```yaml
# docker-compose.prod-mirror.yml
# 在 CI 中启动一个与生产环境配置一致的测试环境
version: "3.8"
services:
  app:
    image: ${CI_REGISTRY_IMAGE}:${CI_COMMIT_SHA}
    environment:
      - DB_HOST=postgres
      - DB_POOL_SIZE=50        # 匹配生产配置
      - CACHE_TTL=300          # 匹配生产配置
      - LOG_LEVEL=warn         # 匹配生产配置
    depends_on:
      - postgres
      - redis

  postgres:
    image: postgres:15
    environment:
      POSTGRES_MAX_CONNECTIONS: 200  # 匹配生产
      POSTGRES_SHARED_BUFFERS: 256MB # 匹配生产
    command: >
      postgres
      -c max_connections=200
      -c shared_buffers=256MB
      -c work_mem=16MB
```

### 误区八：回归测试 = 重跑所有用例

随着项目增长，回归测试套件从 100 个用例膨胀到 10000 个。每次回归都要跑 2 小时。团队开始抱怨："每次改一行代码都要等 2 小时，测试还有什么意义？"

**智能回归策略**

不是所有改动都需要全量回归。根据变更范围选择回归策略：

变更类型            →  回归策略
───────────────────────────────────────────
修改单个函数        →  该函数的单元测试 + 相关模块的集成测试
修改 API 接口       →  该 API 的契约测试 + 调用方的集成测试
修改数据库 Schema   →  数据迁移测试 + 受影响的查询测试
修改配置参数        →  该参数影响范围内的所有功能测试
发布新版本          →  冒烟测试 + 核心业务流程 + 高风险模块

**基于代码变更的精准回归**

```python
# 使用 coverage 分析找出变更影响的测试用例
# tools/selective_test_runner.py

import subprocess
import json

def get_changed_files():
    """获取本次变更涉及的文件"""
    result = subprocess.run(
        ["git", "diff", "--name-only", "main"],
        capture_output=True, text=True
    )
    return [f for f in result.stdout.strip().split("\n") if f.endswith(".py")]

def select_relevant_tests(changed_files):
    """基于变更文件选择相关测试"""
    import coverage
    cov = coverage.Coverage(source=["src"])
    cov.load()

    # 获取每个测试文件覆盖的源文件
    test_to_source = {}
    for test_file in get_all_test_files():
        # 运行该测试并记录覆盖的源文件
        source_files = get_coverage_for_test(test_file)
        if any(sf in changed_files for sf in source_files):
            test_to_source[test_file] = source_files

    return list(test_to_source.keys())

# 实际效果：变更 3 个文件，精准选出 47 个相关测试
# 而非跑全部 10000 个测试，执行时间从 2 小时降到 8 分钟
```

### 误区九：忽视非功能测试

功能测试通过 ≠ 系统质量达标。性能、安全、可用性、兼容性这些非功能性维度的缺失，往往是生产事故的真正元凶。

**非功能测试清单**

性能测试
  ├── 负载测试：系统在预期负载下的表现
  ├── 压力测试：系统在超出预期负载时的行为
  ├── 浸泡测试：长时间运行是否内存泄漏
  └── 峰值测试：突发流量的应对能力

安全测试
  ├── 渗透测试：模拟攻击者视角
  ├── 依赖扫描：已知漏洞检测
  ├── 密钥检测：代码中不应包含敏感信息
  └── 权限验证：越权访问检测

兼容性测试
  ├── 浏览器兼容：Chrome/Firefox/Safari/Edge
  ├── 移动端适配：不同屏幕尺寸和操作系统版本
  └── API 版本兼容：向后兼容性验证

可用性测试
  ├── 响应时间 SLA：P95 < 200ms
  ├── 错误率：5xx < 0.1%
  └── 可用性指标：99.9% uptime

**实操：在 CI 中加入安全扫描**


### 误区十：追求零 Bug 发布

"这个版本不能有任何 Bug"——这种目标看似负责，实则有害。

**为什么零 Bug 是个伪目标**

- 测试只能证明 bug 的**存在**，不能证明 bug 的**不存在**（Dijkstra 的经典论断）
- 追求零 Bug 导致无限期的测试循环，错过市场窗口
- 团队为了避免"引入 Bug"而不敢重构，技术债越积越多
- 低优先级 bug 的修复成本可能超过其带来的业务损失

**正确的质量策略：基于风险的质量管理**

高影响 + 高概率 → 必须在发布前修复
高影响 + 低概率 → 制定缓解计划，发布后持续监控
低影响 + 高概率 → 修复，但不阻塞发布
低影响 + 低概率 → 记录到 backlog，按优先级排期

```python
# Bug 优先级评估矩阵
def assess_bug_priority(severity: str, probability: str, business_impact: str) -> dict:
    """基于多维度评估 Bug 优先级，而非简单的'必须全修'"""
    priority_matrix = {
        ("critical", "high", "high"): {"action": "block_release", "sla": "24h"},
        ("critical", "high", "medium"): {"action": "block_release", "sla": "48h"},
        ("critical", "low", "high"): {"action": "fix_before_release", "sla": "72h"},
        ("major", "high", "high"): {"action": "fix_before_release", "sla": "48h"},
        ("major", "low", "medium"): {"action": "fix_in_sprint", "sla": "1_week"},
        ("minor", "high", "low"): {"action": "backlog", "sla": "next_quarter"},
        ("minor", "low", "low"): {"action": "wont_fix", "sla": "none"},
    }
    key = (severity, probability, business_impact)
    return priority_matrix.get(key, {"action": "manual_review", "sla": "depends"})
```

### 误区十一：测试环境数据用真实数据

为了"测试更真实"，很多团队直接把生产数据库复制一份到测试环境。这带来了两个严重问题：

- **数据安全**：生产数据包含用户隐私信息（手机号、身份证、银行卡），测试环境的安全措施通常较弱，一旦泄露就是合规事故
- **数据污染**：测试人员在"生产数据"上执行破坏性操作（删除、修改），可能影响后续测试的准确性

**正确做法：使用合成测试数据（Synthetic Data）**

```python
# 使用 Faker 生成合成数据
from faker import Faker
import random

fake = Faker("zh_CN")

def generate_test_user():
    """生成逼真但完全虚构的测试数据"""
    return {
        "name": fake.name(),
        "phone": fake.phone_number(),           # 格式正确但不存在的号码
        "email": fake.email(),
        "id_number": fake.ssn(),                # 格式正确但虚构的身份证号
        "address": fake.address(),
        "bank_card": fake.credit_card_number(),  # 格式正确但无效的卡号
        "created_at": fake.date_time_between(
            start_date="-2y", end_date="now"
        ),
    }

# 批量生成
test_users = [generate_test_user() for _ in range(10000)]
```

对于需要保留生产数据分布特征的场景（如性能测试），使用**数据脱敏工具**：

```bash
# 使用 Datafaker 进行数据脱敏
java -jar datafaker.jar \
  --input production_dump.sql \
  --output masked_dump.sql \
  --config masking_rules.yaml
```

### 误区十二：Mock/Stub 滥用

Mock 是测试的利器，但过度使用会让测试失去意义。当你的测试中 80% 是 mock 对象时，你测试的其实是"mock 是否按预期返回"，而非系统的真实行为。

**Mock 滥用的典型症状**

```python
# 过度 Mock：这个测试毫无价值
def test_process_order():
    mock_user_service = Mock()
    mock_user_service.get_user.return_value = Mock(name="Alice", credit=1000)

    mock_inventory = Mock()
    mock_inventory.check_stock.return_value = True

    mock_payment = Mock()
    mock_payment.charge.return_value = Mock(status="success")

    mock_notification = Mock()

    # 被测试的代码只有一行调用逻辑
    # 所有依赖都被 mock 了，测试只验证了"调用顺序"
    service = OrderService(mock_user_service, mock_inventory,
                           mock_payment, mock_notification)
    result = service.process_order(user_id=1, item_id="ABC", qty=1)

    mock_user_service.get_user.assert_called_once_with(1)
    mock_inventory.check_stock.assert_called_once_with("ABC", 1)
    # 这不是在测业务逻辑，是在测 mock 的行为
```

**正确的 Mock 使用原则**

```python
# 原则：Mock 外部系统（网络服务、消息队列），保留内部逻辑真实性

# 对于纯业务逻辑：不 mock，直接测试
def test_calculate_discount_eligibility():
    """纯逻辑测试，不需要任何 mock"""
    user = User(level="gold", total_spent=5000)
    assert is_eligible_for_discount(user, min_spend=3000) == True
    assert calculate_discount_rate(user) == Decimal("0.15")

# 对于外部依赖：只 mock 网络调用
def test_process_payment_integration():
    """mock 支付网关（外部HTTP服务），但保留真实的业务处理逻辑"""
    with responses.RequestsMock() as rsps:
        rsps.add(
            rsps.POST,
            "https://payment-gateway.example.com/charge",
            json={"status": "success", "transaction_id": "TXN123"},
            status=200,
        )
        result = PaymentService().charge(order_id="ORD001", amount=Decimal("99.00"))
        assert result.success == True
        assert result.transaction_id == "TXN123"
```

### 误区十三：性能测试只看平均响应时间

"平均响应时间 50ms，性能没问题！"——这是最具误导性的性能报告。

**为什么平均值会骗人**

假设 100 个请求中，99 个在 20ms 内完成，1 个耗时 5000ms。平均值是 `(99*20 + 5000) / 100 = 69.7ms`。看起来不错？但那个等了 5 秒的用户体验极差，而且可能是你的付费客户。

**正确的性能指标体系**

```python
# 使用百分位数而非平均值
import statistics

response_times = [20, 18, 22, 19, 21, 20, 23, 25, 18, 5000]  # 10个样本

avg = statistics.mean(response_times)
p50 = statistics.median(response_times)
p95 = sorted(response_times)[int(len(response_times) * 0.95)]
p99 = sorted(response_times)[int(len(response_times) * 0.99)]

print(f"平均: {avg}ms")    # 516.6ms — 被极端值拉高
print(f"P50:  {p50}ms")    # 20.5ms  — 中位数正常
print(f"P95:  {p95}ms")    # 5000ms  — 95%的用户在25ms内
print(f"P99:  {p99}ms")    # 5000ms  — 1%的用户体验极差
```

**性能测试完整指标**

延迟指标
  ├── P50（中位数）：50%的请求在此时间内完成
  ├── P95：95%的请求在此时间内完成（行业标准）
  ├── P99：99%的请求在此时间内完成（长尾效应）
  ├── P999：99.9%的请求在此时间内完成（极端情况）
  └── Max：最大延迟（关注但不作为核心指标）

吞吐指标
  ├── QPS/RPS：每秒处理的请求数
  ├── TPS：每秒处理的事务数
  └── Throughput：每秒传输的数据量

资源指标
  ├── CPU 使用率：正常 < 70%，峰值 < 90%
  ├── 内存使用率：正常 < 80%，避免 swap
  ├── 磁盘 I/O：IOPS 和带宽
  └── 网络：带宽使用率、丢包率、重传率

稳定性指标
  ├── 错误率：< 0.1%
  ├── 超时率：< 0.01%
  └── 连接池使用率：< 80%

### 误区十四：测试报告 = 通过/失败数字

"本轮测试 500 条用例全部通过"——这条信息几乎没有决策价值。管理者需要知道的是：**这个版本能不能发布？风险在哪里？**

**有价值的测试报告应该包含**

一、质量摘要（30 秒内可决策）
  - 本次测试范围
  - 核心业务流程：通过/阻塞
  - 新增/修复的 Bug 数量
  - 遗留 Bug 清单（含风险评估）
  - 发布建议：建议发布 / 有条件发布 / 建议延期

二、详细数据
  - 用例执行统计：总数/通过/失败/阻塞/跳过
  - Bug 分布：按模块、按严重级别、按类型
  - 性能基线对比：与上一版本的性能差异
  - 覆盖率报告：增量覆盖率

三、风险分析
  - 未覆盖的测试区域及原因
  - 已知限制和规避方案
  - 需要上线后重点监控的模块

**测试报告模板**

```markdown
## 版本测试报告 v2.3.1

### 质量摘要
- 测试时间：2024-01-15 ~ 2024-01-18
- 测试范围：订单模块重构、支付网关升级
- 核心业务流程：✅ 全部通过
- 新增 Bug：3 个（0 个 Blocker，1 个 Major，2 个 Minor）
- 遗留 Bug：2 个 Minor（已确认不影响核心流程）
- 发布建议：✅ 建议发布

### 风险提示
1. 支付网关升级后，高并发场景（>1000 QPS）未充分验证
   - 缓解方案：灰度发布，从 10% 流量开始，逐步扩到 100%
   - 监控重点：支付成功率、响应时间 P99
2. 订单导出功能在数据量 > 10 万条时响应较慢
   - 已知问题，非本次引入，计划下个迭代优化
```

### 误区十五：忽视测试基础设施的维护

测试框架、CI 流水线、测试环境、测试数据——这些"基础设施"往往在项目初期搭建后就无人维护。随着时间推移，它们逐渐腐化：

- 测试用例频繁因环境问题失败（Flaky Tests），团队开始忽略失败结果
- CI 流水线运行时间越来越长，没人清理过时的步骤
- 测试环境的配置与生产环境差距越来越大
- 测试数据陈旧，无法覆盖新业务场景

**测试基础设施维护清单**

每周
  - 检查 Flaky Tests 列表，修复或删除不稳定的用例
  - 清理不再使用的测试数据

每月
  - 审查 CI 流水线运行时间，优化或拆分耗时步骤
  - 更新测试依赖版本
  - 审查测试覆盖率趋势，识别下降的模块

每季度
  - 对齐测试环境与生产环境配置
  - 评估测试工具链，引入更优方案
  - 组织测试策略评审会

持续
  - Flaky Test 必须在 24 小时内修复或标记跳过
  - 新功能必须配套测试，无测试不允许合并
  - 测试代码与产品代码同等对待：Code Review、静态分析

**治理 Flaky Tests 的具体方案**

```python
# 标记已知不稳定的测试
import pytest

@pytest.mark.flaky(reruns=3, reruns_delay=2)
def test_send_email_notification():
    """邮件发送测试依赖外部 SMTP，偶尔超时"""
    result = send_notification(user_email="test@example.com", message="Hello")
    assert result.delivered == True

# 使用 CI 标签自动隔离不稳定测试
# .github/workflows/test.yml
# jobs:
#   stable-tests:
#     script: pytest -m "not flaky"
#   flaky-tests:
#     script: pytest -m "flaky" --reruns 3
#     # 单独运行，不影响主测试结果
```
