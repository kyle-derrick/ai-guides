## 常见误区

在数据一致性的实践中，错误的直觉往往比错误的代码更危险。工程师容易在分布式系统的复杂性面前形成一些看似合理、实则致命的习惯性认知。本节系统梳理数据一致性领域的十大常见误区，每个误区都从"错误认知→问题剖析→正确做法→代码示例"四个层面展开，帮助读者建立正确的分布式思维模型。

## 误区一：用数据库事务包打天下

### 错误认知

"只要开了事务（BEGIN...COMMIT），数据一致性就有了保证。本地事务能解决的问题，为什么还要搞分布式事务？"

### 问题剖析

本地事务确实能保证单机单库的 ACID，但在以下场景中完全失效：

- **微服务拆分后**：订单库和库存库分属不同服务，一个事务无法跨库提交。
- **水平分库分表**：同一张逻辑表被拆到 8 个物理库，一条 UPDATE 可能涉及多个库。
- **异构存储**：MySQL + Redis + Elasticsearch 的组合中，事务只能管住 MySQL。
- **跨网络调用**：调用远程接口后再回写本地数据库，中间任何一步失败都可能导致不一致。

经典案例：电商下单场景中，订单服务写 MySQL 成功，库存服务扣减 Redis 成功，但支付回调写入失败。此时订单和库存已变更，但支付记录缺失——本地事务对此无能为力。

### 正确做法

根据一致性要求选择合适的方案：

| 场景 | 方案 | 一致性级别 | 性能影响 |
|------|------|-----------|---------|
| 单库单服务 | 本地事务 | 强一致 | 极低 |
| 同库跨表 | 本地事务 | 强一致 | 低 |
| 跨库同构 | 2PC / XA | 强一致 | 高（阻塞） |
| 跨服务最终一致 | Saga + 补偿 | 最终一致 | 低 |
| 跨异构存储 | Outbox 模式 | 最终一致 | 中 |

```java
// ❌ 错误：用本地事务管跨服务操作
@Transactional
public void createOrder(OrderDTO dto) {
    orderMapper.insert(dto);          // 本地 MySQL
    inventoryClient.deduct(dto);      // 远程 HTTP 调用 — 事务管不到！
    pointClient.addPoints(dto);       // 远程 HTTP 调用 — 事务管不到！
}

// ✅ 正确：Saga 模式 + 本地消息表
public void createOrder(OrderDTO dto) {
    // 1. 本地事务：写订单 + 写本地消息表（原子操作）
    transactionTemplate.execute(status -> {
        orderMapper.insert(dto);
        outboxMapper.insert(new OutboxEvent(
            "ORDER_CREATED", dto.toJson(), "PENDING"
        ));
        return null;
    });
    // 2. 异步：由消息处理器驱动后续步骤
    // 3. 失败时通过补偿逻辑回滚已执行的步骤
}
```

## 误区二：盲目追求强一致性

### 错误认知

"所有数据都必须实时一致，任何延迟都是 bug。"

### 问题剖析

强一致性（线性一致性）的代价极其高昂：

- **性能开销**：Paxos/Raft 共识需要多数节点确认，每次写入至少 1.5 个 RTT。
- **可用性牺牲**：CAP 定理表明，网络分区时强一致意味着拒绝服务。
- **吞吐瓶颈**：Raft 日志是串行提交的，写入吞吐受限于单 leader。
- **扩展困难**：线性一致性要求全局排序，水平扩展空间有限。

一个真实的数据：某支付系统将所有操作改为线性一致性后，QPS 从 12000 降到 3000，P99 延迟从 5ms 飙升到 45ms，而业务实际上只需要"秒级最终一致"。

### 正确做法

对不同数据按业务需求分级：

┌─────────────────────────────────────────────────────┐
│           数据一致性分级策略                           │
├──────────────┬──────────────┬───────────────────────┤
│   一级：强一致  │   二级：因果一致  │   三级：最终一致       │
├──────────────┼──────────────┼───────────────────────┤
│ • 账户余额     │ • 消息因果顺序   │ • 推荐列表            │
│ • 库存扣减     │ • 用户动态时间线 │ • 点赞/评论计数        │
│ • 支付状态     │ • 文档协作版本   │ • 日志/审计           │
│ • 分布式锁     │ • 库存预留      │ • 缓存数据            │
├──────────────┼──────────────┼───────────────────────┤
│ 方案：Raft/Paxos│ 方案：向量时钟   │ 方案：异步复制          │
│      2PC      │  因果一致性协议   │      CRDT            │
│              │              │   消息队列              │
└──────────────┴──────────────┴───────────────────────┘

```python
# 账户余额：必须强一致
def transfer(from_id, to_id, amount):
    # 使用 SELECT ... FOR UPDATE 加行锁
    with db.begin():
        from_acct = db.query(
            "SELECT balance FROM accounts WHERE id = %s FOR UPDATE",
            from_id
        )
        if from_acct.balance < amount:
            raise InsufficientFunds()
        db.execute(
            "UPDATE accounts SET balance = balance - %s WHERE id = %s",
            amount, from_id
        )
        db.execute(
            "UPDATE accounts SET balance = balance + %s WHERE id = %s",
            amount, to_id
        )

# 推荐列表：最终一致即可
def update_recommendations(user_id):
    # 异步写入，延迟几秒可接受
    async_task_queue.push(
        user_id=user_id,
        action="refresh_recommendations",
        # 允许最多 30 秒延迟
        max_delay=30
    )
```

## 误区三：把"写成功"等同于"写持久化"

### 错误认知

"INSERT 返回了 OK，数据就安全了。"

### 问题剖析

这是一个极其危险的认知偏差。以下场景中"写成功"并不意味着"已持久化"：

- **写缓存未落盘**：MySQL 的 InnoDB 虽然有 redo log，但 `innodb_flush_log_at_trx_commit` 设为 2 时，每秒才刷盘一次。宕机可能丢 1 秒数据。
- **操作系统缓冲**：写文件时不调 fsync，数据可能还在 page cache 中。断电即丢失。
- **副本延迟**：主库写成功后，从库还没同步到。主库此时宕机，数据回退。
- **半同步空洞**：MySQL 半同步复制中，`rpl_semi_sync_master_wait_point = AFTER_SYNC` 之前有窗口期。

真实案例：某公司数据库服务器突然断电重启后，发现有 200 多条订单"消失"。排查发现，应用收到的 INSERT 成功响应是正确的——数据确实进入了 MySQL buffer pool，但 redo log 尚未刷盘就被截断了。

### 正确做法

根据数据重要性选择合适的持久化策略：

```sql
-- 关键金融数据：最严格持久化
SET GLOBAL innodb_flush_log_at_trx_commit = 1;  -- 每次提交都刷盘
SET GLOBAL sync_binlog = 1;                       -- 每次提交都刷 binlog

-- 一般业务数据：折中方案
SET GLOBAL innodb_flush_log_at_trx_commit = 2;    -- 每秒刷盘
SET GLOBAL sync_binlog = 100;                       -- 每 100 次提交刷盘

-- 日志/审计数据：性能优先
SET GLOBAL innodb_flush_log_at_trx_commit = 2;
SET GLOBAL sync_binlog = 0;                        -- 依赖 OS 刷盘
```

```java
// 文件写入时必须显式 fsync
try (FileOutputStream fos = new FileOutputStream(file);
     FileChannel channel = fos.getChannel()) {
    channel.write(ByteBuffer.wrap(data));
    channel.force(true);  // fsync，确保数据落盘
}
// 注意：channel.force(false) 只保证数据到磁盘，不保证元数据
// channel.force(true) 保证数据 + 元数据（文件大小等）都落盘
```

## 误区四：忽略时钟偏差的影响

### 错误认知

"服务器时间是准的，用系统时间戳做排序和比较没有问题。"

### 问题剖析

分布式系统中的时钟偏差比你想象的更严重：

- **NTP 同步精度**：普通 NTP 同步精度在毫秒级，高精度 PTP 在微秒级。对于要求严格排序的场景，毫秒级偏差足以导致数据错乱。
- **闰秒问题**：闰秒调整可能导致系统时间回退 1 秒，基于时间戳的单调递增假设被打破。
- **虚拟机时钟漂移**：VM 的时钟精度取决于宿主机负载，极端情况下偏差可达数十毫秒。
- **逻辑时钟缺失**：仅依赖物理时间戳无法表达事件的因果关系。

```java
// ❌ 错误：用系统时间戳做因果排序
long t1 = System.currentTimeMillis();  // 节点 A 的时间
// ... 网络传输 ...
long t2 = System.currentTimeMillis();  // 节点 B 的时间
if (t1 < t2) {
    // 不能保证 t1 发生的事件先于 t2！
    // 时钟偏差可能让物理时间更晚的事件被误判为更早
}

// ✅ 正确：使用混合逻辑时钟（HLC）
public class HybridLogicalClock {
    private long physical;  // 物理时间部分
    private long logical;   // 逻辑计数器部分

    public HLCEvent now() {
        long pt = System.currentTimeMillis();
        if (pt > physical) {
            physical = pt;
            logical = 0;
        } else {
            logical++;
        }
        return new HLCEvent(physical, logical);
    }

    // 本地事件递增
    public HLCEvent localEvent() {
        return now();
    }

    // 接收远程事件时合并
    public HLCEvent receiveEvent(HLCEvent remote) {
        long pt = System.currentTimeMillis();
        if (pt > physical &amp;&amp; pt > remote.physical) {
            physical = pt;
            logical = 0;
        } else if (remote.physical > physical) {
            physical = remote.physical;
            logical = remote.logical + 1;
        } else if (physical == remote.physical) {
            logical = Math.max(logical, remote.logical) + 1;
        } else {
            logical++;
        }
        return new HLCEvent(physical, logical);
    }
}
```

**实践建议**：

- 同机房内使用 NTP 服务，偏差控制在 1ms 以内。
- 跨机房场景使用 HLC 或向量时钟做因果排序。
- 关键排序逻辑不要单独依赖物理时间戳，至少加上逻辑计数器。
- 监控各节点时钟偏差，设置告警阈值（建议 50ms）。

## 误区五：认为最终一致就是"不用管一致性"

### 错误认知

"既然允许最终一致，那就不需要做任何一致性保障，数据迟早会对齐的。"

### 问题剖析

最终一致性的"最终"是有条件的——它要求系统在没有新写入的情况下，经过足够时间后能收敛到一致状态。以下情况会导致"最终"永远不会到来：

- **补偿逻辑 bug**：重试/补偿代码本身有缺陷，错误会被无限复制。
- **消息丢失**：异步消息队列丢消息且没有重试机制，下游永远收不到更新。
- **网络分区持续**：节点间网络长期不通，数据无法同步。
- **写入覆盖**：两个节点同时写入同一 key，冲突解决策略不当导致数据回退。

常见症状：

```text
症状1: 用户修改了个人资料，刷新后又变回旧值
       → 原因：写入主库成功，读请求路由到了尚未同步的从库

症状2: 两个用户同时编辑文档，最终结果只保留了一个人的修改
       → 原因：CRDT 合并策略实现有误，Last-Write-Wins 导致覆盖

症状3: 订单状态从"已完成"退回到"处理中"
       → 原因：补偿事件晚于正向事件到达，且没有去重机制
```

### 正确做法

最终一致性需要工程保障，而非放任不管：

```java
// 1. 消息可靠性：本地消息表 + 定时重试
public class OutboxProcessor {
    @Scheduled(fixedDelay = 5000)
    public void processPendingEvents() {
        List<OutboxEvent> events = outboxRepo.findPending(100);
        for (OutboxEvent event : events) {
            try {
                mqClient.send(event.getTopic(), event.getPayload());
                event.setStatus("SENT");
            } catch (Exception e) {
                event.incrementRetry();
                if (event.getRetryCount() > MAX_RETRY) {
                    event.setStatus("DEAD");
                    alertService.alert("Dead letter: " + event.getId());
                }
            }
            outboxRepo.save(event);
        }
    }
}

// 2. 冲突检测：版本向量
public boolean updateWithConflictCheck(String key, String value,
                                        VectorClock incomingVC) {
    VectorClock currentVC = store.getVC(key);
    if (incomingVC.isConcurrent(currentVC)) {
        // 并发冲突：需要合并或提示用户
        log.warn("Conflict detected for key={}, incoming={}, current={}",
                 key, incomingVC, currentVC);
        return false;  // 返回给上层做冲突解决
    }
    if (incomingVC.happensBefore(currentVC)) {
        // 过时写入：丢弃
        log.info("Stale write rejected for key={}", key);
        return false;
    }
    // 正常写入
    store.put(key, value, incomingVC.merge(currentVC));
    return true;
}

// 3. 对账机制：定期检查一致性
@Scheduled(cron = "0 0 2 * * ?")  // 每天凌晨 2 点
public void reconciliation() {
    List<ReconciliationMismatch> mismatches = orderService
        .compareOrderVsInventory();
    for (ReconciliationMismatch mismatch : mismatches) {
        // 自动修复或人工介入
        if (mismatch.isAutoFixable()) {
            compensationService.fix(mismatch);
        } else {
            alertService.escalate(mismatch);
        }
    }
}
```

## 误区六：幂等性设计"说起来简单，做起来随便"

### 错误认知

"加个去重表/去重键不就行了，幂等性有什么难的？"

### 问题剖析

幂等性设计的复杂度往往藏在细节中：

- **去重键的选择**：用请求 ID 作为去重键，但客户端重试时生成了新的 ID——去重失效。
- **去重窗口的设置**：去重表只保留 24 小时记录，但一笔支付在 48 小时后被重复回调。
- **业务逻辑非幂等**：INSERT 操作通过唯一索引保证幂等没问题，但"UPDATE balance = balance - 100"执行两次就多扣了 100 元。
- **外部系统交互**：调用第三方接口后本地失败回滚，但第三方已经扣款——再重试又扣一次。

```text
幂等性设计的常见陷阱：

┌─────────────────┬──────────────────────────────────┐
│    操作类型      │         正确的幂等化方法            │
├─────────────────┼──────────────────────────────────┤
│ INSERT          │ 唯一索引 + ON DUPLICATE KEY       │
│ UPDATE（绝对值） │ 天然幂等，直接执行                  │
│ UPDATE（相对值） │ WHERE + 乐观锁版本号               │
│ DELETE          │ 软删除 + 唯一索引                   │
│ 外部 API 调用   │ 幂等键 + 去重表 + 状态机            │
│ 消息消费        │ 消息 ID + 消费记录表                │
│ 文件上传        │ 文件 MD5 + 业务 ID 联合唯一索引      │
└─────────────────┴──────────────────────────────────┘
```

### 正确做法

```sql
-- 1. INSERT 幂等：唯一索引兜底
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    idempotency_key VARCHAR(64) UNIQUE,  -- 幂等键
    amount DECIMAL(10,2),
    status VARCHAR(20)
);

INSERT INTO orders (id, idempotency_key, amount, status)
VALUES (1001, 'req-abc-123', 99.00, 'CREATED')
ON DUPLICATE KEY UPDATE id=id;  -- 重复插入忽略

-- 2. UPDATE 幂等：乐观锁
UPDATE inventory
SET stock = stock - 1,
    version = version + 1
WHERE product_id = 'P001'
  AND version = 5          -- 当前版本号
  AND stock >= 1;          -- 防止负库存
-- 影响行数为 0 时，说明并发冲突或库存不足，需要重试或拒绝

-- 3. 消费记录：防重复消费
CREATE TABLE message_consumed (
    message_id VARCHAR(64) PRIMARY KEY,
    consumer_group VARCHAR(64),
    consumed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

```java
// 带状态机的幂等处理器
public class IdempotentPaymentHandler {

    public PaymentResult handlePayment(PaymentRequest req) {
        // 1. 查询是否已处理
        PaymentRecord existing = paymentRepo
            .findByIdempotencyKey(req.getIdempotencyKey());

        if (existing != null) {
            // 2. 已处理：直接返回之前的结果
            return existing.toResult();
        }

        // 3. 未处理：执行业务 + 记录结果（原子操作）
        return transactionTemplate.execute(status -> {
            try {
                // 先插入记录（标记为 PROCESSING）
                PaymentRecord record = new PaymentRecord(
                    req.getIdempotencyKey(), "PROCESSING");
                paymentRepo.insert(record);

                // 执行支付
                PaymentResult result = gateway.charge(req);

                // 更新状态
                record.setStatus("COMPLETED");
                record.setResult(result);
                paymentRepo.update(record);
                return result;
            } catch (Exception e) {
                // 标记失败，允许重试
                PaymentRecord record = paymentRepo
                    .findByIdempotencyKey(req.getIdempotencyKey());
                record.setStatus("FAILED");
                paymentRepo.update(record);
                throw e;
            }
        });
    }
}
```

## 误区七：缓存和数据库的双写一致性"设个过期时间就行"

### 错误认知

"缓存设了 TTL，过期后会自动从数据库重新加载，不存在一致性问题。"

### 问题剖析

TTL 过期只是"最终一致"的一种弱保障，它带来的问题包括：

- **缓存穿透**：热点 key 过期的瞬间，大量并发请求同时穿透到数据库。
- **脏数据窗口**：TTL 设为 5 分钟，意味着数据最多有 5 分钟的不一致窗口。
- **更新时序**：先更新数据库再删缓存，在删除和重新加载之间存在不一致窗口。
- **雪崩效应**：大量 key 同时过期，数据库瞬间承受所有流量。

```text
典型的缓存不一致时序（先更新 DB，再删缓存）：

时间线：
  T1: 线程A 更新数据库 price=100
  T2: 线程B 读缓存，得到旧值 price=50（缓存还没删）
  T3: 线程A 删除缓存
  T4: 线程B 用旧值覆盖缓存 price=50   ← 不一致！

  在 T2-T4 之间，读请求拿到了脏数据
```

### 正确做法

```java
// ✅ 推荐方案：延迟双删 + 超时兜底
public void updateProduct(String productId, BigDecimal newPrice) {
    // 1. 更新数据库
    productMapper.updatePrice(productId, newPrice);

    // 2. 删除缓存
    cacheClient.delete("product:" + productId);

    // 3. 延迟二次删除（覆盖并发读写造成的脏缓存）
    scheduler.schedule(() -> {
        cacheClient.delete("product:" + productId);
    }, 500, TimeUnit.MILLISECONDS);

    // 4. 数据库变更事件（异步最终兜底）
    eventBus.publish(new ProductPriceChanged(productId, newPrice));
}

// ✅ 消费端：监听 binlog 变更，异步刷新缓存
@KafkaListener(topics = "binlog.product")
public void onProductChange(BinlogEvent event) {
    String productId = event.getFieldValue("id");
    cacheClient.delete("product:" + productId);
    // 删除后，下次读请求会自动从 DB 加载最新值
}
```

**缓存一致性方案对比**：

| 方案 | 一致性 | 复杂度 | 适用场景 |
|------|--------|--------|---------|
| 纯 TTL | 弱 | 低 | 允许短暂不一致的展示数据 |
| 先更新 DB 再删缓存 | 中 | 低 | 一般业务，配合 TTL 兜底 |
| 延迟双删 | 较强 | 中 | 对一致性要求较高的场景 |
| binlog 订阅刷新 | 强 | 高 | 强一致要求的金融/库存场景 |
| 订阅 binlog + 重试 | 最终一致 | 高 | 大规模分布式系统 |

## 误区八：分布式锁"用 Redis SETNX 就够了"

### 错误认知

"Redis SETNX + 过期时间就是分布式锁的全部，简单可靠。"

### 问题剖析

裸 SETNX 分布式锁存在多个严重问题：

- **锁过期但任务未完成**：线程 A 获取锁，GC 暂停 30 秒，锁自动过期，线程 B 获取锁——两个线程同时执行。
- **误删其他线程的锁**：线程 A 删锁时，删的可能是线程 B 刚获取的新锁。
- **Redis 主从切换丢锁**：主库写入锁后宕机，从库提升为主但没有锁信息。
- **脑裂问题**：网络分区时，两个客户端都认为自己持有锁。

### 正确做法

```java
// ✅ 使用 Redisson（带看门狗机制）
RLock lock = redissonClient.getLock("order:process:" + orderId);

try {
    // waitTime=0 不等待，leaseTime=30 秒自动过期
    // Redisson 看门狗会在锁持有期间自动续期
    if (lock.tryLock(0, 30, TimeUnit.SECONDS)) {
        // 业务逻辑
        processOrder(orderId);
    } else {
        throw new OrderProcessingException("获取锁失败");
    }
} finally {
    // 必须判断是自己的锁才删除（Redisson 已内置）
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();
    }
}
```

```python
# 如果必须用原生 Redis，至少做到以下几点
import redis
import uuid
import time

class RedLock:
    def __init__(self, client):
        self.client = client
        self.token = str(uuid.uuid4())  # 唯一标识，防止误删

    def acquire(self, key, ttl=10):
        """获取锁"""
        return self.client.set(
            key, self.token, nx=True, ex=ttl
        )

    def release(self, key):
        """
        释放锁：用 Lua 脚本保证原子性
        必须先验证 token，再删除
        """
        lua_script = """
        if redis.call("get", KEYS[1]) == ARGV[1] then
            return redis.call("del", KEYS[1])
        else
            return 0
        end
        """
        return self.client.eval(lua_script, 1, key, self.token)

    def extend(self, key, ttl=10):
        """续期：同样需要验证 token"""
        lua_script = """
        if redis.call("get", KEYS[1]) == ARGV[1] then
            return redis.call("expire", KEYS[1], ARGV[2])
        else
            return 0
        end
        """
        return self.client.eval(lua_script, 1, key, self.token, ttl)
```

**分布式锁方案选型**：

| 方案 | 一致性 | 性能 | 容灾 | 推荐场景 |
|------|--------|------|------|---------|
| Redis SETNX | 弱 | 高 | 中 | 非关键业务的并发控制 |
| Redisson | 较强 | 高 | 中 | 一般业务的分布式锁 |
| Redlock（多节点） | 强 | 中 | 高 | 高可靠要求的分布式锁 |
| etcd/ZooKeeper | 强 | 中 | 高 | 核心业务的强一致锁 |

## 误区九：重试机制"加个循环就行了"

### 错误认知

"调用失败就重试，多试几次总会成功。加个 for 循环比什么都管用。"

### 问题剖析

无策略的重试是分布式系统中最常见的雪崩触发器：

- **幂等性未保证**：非幂等操作重试 N 次 = 执行 N 次 = 数据翻 N 倍。
- **下游过载**：上游疯狂重试，下游压力翻倍甚至更多，形成恶性循环。
- **资源耗尽**：同步重试阻塞线程，线程池耗尽后整个服务不可用。
- **级联重试**：服务 A 重试调 B，B 重试调 C，每层重试叠加，总延迟 = N1 × N2 × N3 × 单次延迟。

### 正确做法

```java
// ✅ 指数退避 + 抖动 + 熔断
@Retryable(
    value = {ServiceException.class, TimeoutException.class},
    maxAttempts = 3,
    backoff = @Backoff(
        delay = 1000,       // 首次重试延迟 1 秒
        multiplier = 2,      // 每次翻倍：1s → 2s → 4s
        maxDelay = 10000     // 最大延迟 10 秒
    )
)
@CircuitBreaker(
    name = "inventoryService",
    failureRateThreshold = 50,   // 50% 失败率触发熔断
    waitDurationInOpenState = 30 // 熔断恢复等待 30 秒
)
public void deductStock(String productId, int quantity) {
    inventoryClient.deduct(productId, quantity);
}

// 手动实现：指数退避 + 抖动
public <T> T retryWithBackoff(Supplier<T> operation, int maxRetries) {
    for (int attempt = 0; attempt <= maxRetries; attempt++) {
        try {
            return operation.get();
        } catch (Exception e) {
            if (attempt == maxRetries) throw e;

            long delay = (long) (1000 * Math.pow(2, attempt));
            long jitter = ThreadLocalRandom.current()
                .nextLong(delay / 2);  // 0 ~ delay/2 的随机抖动
            delay += jitter;

            log.warn("Retry attempt {} after {}ms: {}",
                     attempt + 1, delay, e.getMessage());
            try { Thread.sleep(delay); }
            catch (InterruptedException ie) {
                Thread.currentThread().interrupt();
                throw new RuntimeException(ie);
            }
        }
    }
    throw new IllegalStateException("Unreachable");
}
```

**重试策略选型**：

| 策略 | 适用场景 | 注意事项 |
|------|---------|---------|
| 立即重试 | 瞬时网络抖动 | 最多 1-2 次，避免雪崩 |
| 固定间隔 | 简单的轮询场景 | 间隔不能太短 |
| 指数退避 | 通用外部调用 | 必须加上限 |
| 指数退避 + 抖动 | 高并发场景 | 防止重试风暴 |
| 自适应退避 | 大规模分布式 | 根据下游负载动态调整 |

## 误区十：过度设计一致性方案

### 错误认知

"用最先进的分布式一致性协议，实现最严格的强一致，才是技术实力的体现。"

### 问题剖析

过度设计在数据一致性领域同样是一个严重的反模式：

- **2PC 用于非关键路径**：一个非核心的统计报表功能用 XA 事务跨三个库，吞吐降低 80%，而报表延迟几秒完全可接受。
- **Paxos 用于配置存储**：用 etcd 的 Raft 协议存储非关键配置，但配置变更频率极低且允许短暂不一致，直接用 Redis + 发布订阅就够了。
- **引入 CRDT 用于简单计数器**：一个简单的点赞计数器用 CRDT 实现，代码复杂度暴增，而直接用 Redis INCR + 异步落库就完全满足需求。
- **全局分布式事务**：为所有操作都加上分布式事务协调器，P99 延迟从 10ms 涨到 200ms，而大部分操作只需要最终一致。

```text
过度设计的信号检查清单：

□ 团队中超过一半的成员无法解释当前一致性方案的原理
□ 为了解决一个偶尔出现的不一致问题，引入了三个新的中间件
□ 一致性方案的代码量超过了业务逻辑本身的代码量
□ 性能测试显示引入一致性方案后 QPS 下降超过 50%
□ 你正在用 Paxos 解决一个 TTL 过期就能搞定的问题
□ 需要专门雇一个人来维护一致性基础设施
```

### 正确做法

坚持"够用就好"原则，逐步演进：

```text
一致性方案的演进路径：

阶段 1（MVP）：本地事务 + 缓存 TTL
    ↓ 数据量增长，单库成为瓶颈
阶段 2：读写分离 + 最终一致（消息队列）
    ↓ 微服务拆分，跨服务一致性问题
阶段 3：Saga 模式 + 补偿逻辑 + 对账
    ↓ 核心交易链路需要强一致
阶段 4：关键链路引入分布式事务（Seata AT/TCC）
    ↓ 超大规模，一致性成为瓶颈
阶段 5：定制化方案（CRDT + 自研协议）

每一步都应该在前一步"不够用"时才升级，而非一开始就跳到最后一步。
```

## 误区总结与自查清单

### 十大误区速查表

| 序号 | 误区 | 核心错误 | 正确原则 |
|------|------|---------|---------|
| 1 | 本地事务包打天下 | 事务无法跨服务/异构存储 | 按场景选方案（Saga/Outbox/TCC） |
| 2 | 盲目追求强一致 | 性能代价过高，可用性牺牲 | 按业务分级，非核心用最终一致 |
| 3 | 写成功 = 已持久化 | 缓冲/副本延迟导致丢失 | 关键数据必须 fsync + 同步复制 |
| 4 | 忽略时钟偏差 | 物理时间戳不能保证因果序 | HLC/向量时钟 + 监控偏差 |
| 5 | 最终一致 = 不用管 | 没有工程保障，一致性永远不到 | 消息可靠性 + 冲突检测 + 对账 |
| 6 | 幂等很简单 | 去重键/窗口/外部调用的陷阱 | 状态机 + 乐观锁 + 去重表 |
| 7 | 缓存 TTL 搞定一致性 | 过期窗口 + 并发穿透 | 延迟双删 / binlog 订阅 |
| 8 | SETNX 就是分布式锁 | 锁过期 / 误删 / 主从丢锁 | Redisson 看门狗 / etcd 强一致锁 |
| 9 | for 循环重试 | 非幂等 + 雪崩 + 级联 | 指数退避 + 抖动 + 熔断 |
| 10 | 过度设计一致性 | 方案复杂度 > 业务复杂度 | 够用就好，逐步演进 |

### 实践自查清单

在设计或审查数据一致性方案时，逐项检查：

- [ ] **业务分级**：哪些数据需要强一致，哪些最终一致即可？
- [ ] **失败模式**：每一步操作失败后会发生什么？有补偿逻辑吗？
- [ ] **幂等保障**：所有可能被重试的操作都有幂等保护吗？
- [ ] **时钟依赖**：是否有排序/过期逻辑强依赖系统时间？
- [ ] **消息可靠性**：异步消息丢失时，有兜底机制（对账/重试）吗？
- [ ] **并发安全**：高并发下的竞态条件都覆盖了吗？
- [ ] **监控告警**：不一致发生时能及时发现吗？
- [ ] **恢复方案**：数据不一致时有修复工具和 SOP 吗？
- [ ] **性能影响**：一致性方案的性能开销在可接受范围内吗？
- [ ] **团队理解**：团队成员能解释当前方案的原理和局限性吗？

## 本节要点

1. **不要用本地事务解决分布式问题**——微服务架构下，事务边界必须重新设计。
2. **强一致有代价**——按业务需求分级，不是所有数据都需要线性一致性。
3. **写成功 ≠ 持久化**——理解缓冲层和副本延迟，关键数据必须显式刷盘。
4. **时钟不可信**——用逻辑时钟补充物理时间，监控时钟偏差。
5. **最终一致需要工程保障**——消息可靠投递 + 冲突检测 + 定期对账缺一不可。
6. **幂等性远比想象复杂**——不只是加个唯一索引，要考虑去重窗口和外部调用。
7. **缓存一致性是系统工程**——TTL 只是兜底，真正的方案要覆盖更新时序和并发场景。
8. **分布式锁不是 SETNX**——考虑锁续期、误删保护、主从切换等边界情况。
9. **无策略的重试是灾难**——必须配合幂等性、退避、抖动、熔断四重保障。
10. **过度设计和设计不足一样有害**——从简单方案开始，按需演进。
