## 6. 编译与链接

理解编译与链接是安全研究的基础能力。当你分析一个二进制漏洞时，你需要知道目标文件是怎么生成的、各段的含义、符号如何解析、动态链接如何工作——这些知识直接决定了你能否构造 ROP 链、覆写 GOT 表、理解 PIE 随机化的边界。本章从源代码出发，逐阶段拆解编译流水线，深入 ELF 文件格式，最后覆盖构建系统和安全编译选项。

### 6.1 编译流水线全景

一段 C 代码从文本到可执行程序，经历四个独立阶段。每个阶段有独立的输入和输出产物，可以用 `gcc` 的阶段选项单独执行：

```bash
# 完整流水线演示（以 hello.c 为例）
# 源文件：hello.c

# 阶段1：预处理（Preprocessing）
# 输入: hello.c → 输出: hello.i
# 展开宏、处理 #include/#ifdef、删除注释
gcc -E hello.c -o hello.i

# 阶段2：编译（Compilation）
# 输入: hello.i → 输出: hello.s
# 词法分析→语法分析→语义分析→中间代码→优化→目标汇编
gcc -S hello.i -o hello.s

# 阶段3：汇编（Assembly）
# 输入: hello.s → 输出: hello.o（可重定位目标文件）
# 汇编指令→机器码，生成符号表和重定位条目
gcc -c hello.s -o hello.o

# 阶段4：链接（Linking）
# 输入: hello.o + 库 → 输出: hello（可执行文件）
# 符号解析、重定位、合并节、解析库依赖
gcc hello.o -o hello

# 一步到位（gcc 内部就是依次执行上面四步）
gcc hello.c -o hello
```

查看各阶段产物是理解编译过程最直接的方式：

```bash
# 查看预处理结果：宏已展开，头文件已内联
# -P 抑制行号标记，-C 保留注释
gcc -E -P -C hello.c -o hello.i
wc -l hello.c hello.i    # hello.i 通常是 hello.c 的几百倍

# 查看汇编输出：AT&T 语法（默认）
gcc -S -masm=att hello.c -o hello_att.s
# 查看汇编输出：Intel 语法（更易读）
gcc -S -masm=intel hello.c -o hello_intel.s

# 查看目标文件内容
objdump -d hello.o          # 反汇编
objdump -t hello.o          # 符号表
readelf -s hello.o          # ELF 符号表（更详细）
readelf -r hello.o          # 重定位条目

# 查看可执行文件
objdump -d hello            # 反汇编
readelf -l hello            # 程序头（段信息）
readelf -S hello            # 节头（节信息）
```

#### 6.1.1 预处理器详解

预处理器是纯文本替换工具，不理解 C 语法。它的输出仍然是合法的 C 代码。

```c
// === 宏展开 ===
#define MAX(a, b) ((a) > (b) ? (a) : (b))
// 注意括号的重要性：
// #define BAD_MAX(a, b) a > b   // BAD: BAD_MAX(1+2, 3) → 1+2>3 → 错误
// #define MAX(a,b) ((a)>(b)?(a):(b))  // GOOD: ((1+2)>(3)?(1+2):(3)) → 正确

// === 条件编译（安全工具开发常用）===
#ifdef DEBUG
    #define LOG(fmt, ...) fprintf(stderr, "[DEBUG] " fmt "\n", ##__VA_ARGS__)
#else
    #define LOG(fmt, ...)   // 空宏，编译器会优化掉
#endif

// 平台检测
#if defined(__linux__)
    #define PLATFORM "Linux"
#elif defined(__APPLE__)
    #define PLATFORM "macOS"
#elif defined(_WIN32)
    #define PLATFORM "Windows"
#endif

// === 防止头文件重复包含 ===
// 传统方式：
#ifndef MY_HEADER_H
#define MY_HEADER_H
// ... 头文件内容 ...
#endif

// 现代方式（GCC/Clang 支持）：
#pragma once
```

```bash
# 查看宏展开过程
gcc -E -dM hello.c          # 列出所有预定义宏
gcc -E -dM - < /dev/null    # 查看编译器内置宏

# 查看包含路径搜索顺序
cpp -v < /dev/null 2>&1 | sed -n '/#include.*search starts/,/End of search/p'
```

#### 6.1.2 编译器内部阶段

编译阶段本身又细分为多个子阶段：

```mermaid
graph LR
    A[源代码] --> B[词法分析<br/>Lexer]
    B --> C[语法分析<br/>Parser]
    C --> D[语义分析<br/>Type Check]
    D --> E[中间表示<br/>IR/AST]
    E --> F[优化 passes]
    F --> G[代码生成]
    G --> H[汇编代码]
    
    style A fill:#1a1a2e,color:#eee
    style H fill:#1a1a2e,color:#eee
```

| 子阶段 | 输入 | 输出 | 安全相关性 |
|--------|------|------|-----------|
| 词法分析 | 源代码字符流 | Token 序列 | 无直接关系 |
| 语法分析 | Token 序列 | 抽象语法树(AST) | 宏展开后的代码结构 |
| 语义分析 | AST | 带类型标注的 AST | 类型检查、隐式转换警告 |
| 中间表示 | AST | LLVM IR / GIMPLE | 反编译器的目标格式 |
| 优化 | IR | 优化后的 IR | 优化可能改变漏洞形态 |
| 代码生成 | IR | 目标汇编 | 调用约定、栈帧布局 |

```bash
# 查看 GCC 中间表示（GIMPLE/RTL）
gcc -fdump-tree-all hello.c       # 输出所有 GIMPLE dump
gcc -fdump-rtl-all hello.c        # 输出所有 RTL dump
# 会生成大量 .XXX 文件，查看关键的几个：
cat hello.c.003t.original         # 原始 AST
cat hello.c.004t.gimple           # GIMPLE IR
cat hello.c.254t.optimized        # 优化后的 GIMPLE

# 查看 LLVM IR（使用 clang）
clang -S -emit-llvm hello.c -o hello.ll
cat hello.ll                      # 人类可读的 LLVM IR

# 优化级别对代码的影响
gcc -O0 -S hello.c -o hello_O0.s  # 无优化：忠实于源码
gcc -O2 -S hello.c -o hello_O2.s  # 激进优化：内联、循环展开、死代码消除
diff hello_O0.s hello_O2.s        # 差异可能非常大
```

**优化对安全分析的影响**：编译器优化会显著改变生成代码的结构。`-O0` 下的栈布局与 `-O2` 完全不同——变量重排、内联展开、尾调用优化都会影响溢出利用。在 CTF 比赛和实际漏洞分析中，必须考虑目标的编译优化级别。

#### 6.1.3 汇编器与目标文件

汇编器将汇编指令翻译为机器码，同时生成两个关键结构：

- **符号表（Symbol Table）**：记录本文件定义和引用的全局符号
- **重定位表（Relocation Table）**：记录需要在链接时修正的地址引用

```bash
# 查看目标文件的符号表
readelf -s hello.o
# 输出示例：
# Num:    Value          Size Type    Bind   Vis      Ndx Name
#   8: 0000000000000000    17 FUNC    GLOBAL DEFAULT    1 main
#   9: 0000000000000000     0 NOTYPE  GLOBAL DEFAULT  UND puts
# 注意 main 的 Ndx=1（.text 节），puts 的 Ndx=UND（未定义，需链接时解析）

# 查看重定位条目
readelf -r hello.o
# 输出示例：
# Offset          Info           Type           Sym. Value    Sym. Name
# 00000000000c  000500000002 R_X86_64_PC32     0000000000000000 puts - 4
# 说明：在 .text 偏移 0xc 处，需要将 puts 的地址以 PC32 方式填入

# 查看节信息
readelf -S hello.o
# 典型的 .o 文件包含：
# .text    代码段
# .data    已初始化数据
# .bss     未初始化数据
# .rodata  只读数据（字符串常量）
# .symtab  符号表（完整，包含局部符号）
# .strtab  字符串表（符号名）
# .rela.text  重定位信息
```

#### 6.1.4 链接器工作原理

链接器的核心任务有两个：

**1. 符号解析（Symbol Resolution）**：将每个符号引用绑定到唯一的符号定义。

```bash
# 多文件编译演示
# utils.h
echo 'void helper(void);' > utils.h

# utils.c
cat > utils.c << 'EOF'
#include <stdio.h>
void helper(void) {
    printf("helper called\n");
}
EOF

# main.c
cat > main.c << 'EOF'
#include "utils.h"
int main(void) {
    helper();
    return 0;
}
EOF

# 分别编译为目标文件
gcc -c main.c -o main.o
gcc -c utils.c -o utils.o

# 链接：链接器解析 main.o 中对 helper 的引用 → 绑定到 utils.o 中的定义
gcc main.o utils.o -o program

# 常见链接错误
# undefined reference to 'xxx'  → 缺少某个 .o 或 -l库
# multiple definition of 'xxx'  → 同一符号在多个文件中定义
```

**2. 重定位（Relocation）**：编译时不知道最终地址，链接器将所有节合并后分配虚拟地址，并修正所有地址引用。

```text
重定位过程示意：

编译时（hello.o）：
  call <占位符>       # puts 的地址未知，先填 0

链接后（hello）：
  call 0x401030       # puts 的实际地址已填入（通过 PLT）
```

链接器支持两种链接方式，对安全分析有根本性影响：

| 特性 | 静态链接 | 动态链接 |
|------|---------|---------|
| 库代码 | 复制进可执行文件 | 运行时从 .so 加载 |
| 文件大小 | 大（几百KB到几MB） | 小（只包含自身代码） |
| 内存占用 | 每个进程一份拷贝 | 多进程共享同一份 .so |
| 更新 | 需重新编译 | 替换 .so 即可 |
| 安全分析 | 所有代码在二进制中 | 需分析 .so 依赖 |
| 利用难度 | libc 函数地址相对固定 | 需考虑 ASLR 偏移 |

```bash
# 静态链接
gcc -static main.c -o main_static
ldd main_static         # "not a dynamic executable"
ls -lh main_static      # 通常几百KB到几MB

# 动态链接（默认）
gcc main.c -o main_dynamic
ldd main_dynamic        # 列出所有依赖的 .so
# 输出示例：
# linux-vdso.so.1 =>  (0x00007ffd...)
# libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6 (0x00007f...)
# /lib64/ld-linux-x86-64.so.2 (0x00007f...)

# 查看动态依赖树
readelf -d main_dynamic         # 动态段信息
objdump -T main_dynamic         # 动态符号表
```

### 6.2 ELF 文件格式深度解析

ELF（Executable and Linkable Format）是 Linux 下可执行文件、目标文件、共享库的统一格式。安全研究中的大部分工作——逆向分析、漏洞利用、恶意代码分析——都以 ELF 为基础。

#### 6.2.1 ELF 整体结构

```text
┌─────────────────────┐
│    ELF Header        │ ← 64字节，魔数 7f 45 4c 46（.ELF）
│                      │   文件类型、机器架构、入口点地址
│                      │   Program Header 偏移和大小
│                      │   Section Header 偏移和大小
├─────────────────────┤
│    Program Header 0  │ ← 描述一个"段"（Segment）
│    Program Header 1  │   段是给内核/加载器看的
│    ...               │   定义哪些节映射到内存的哪个区域
├─────────────────────┤
│    .text             │ ← 可执行代码
│    .init / .fini     │   程序初始化/清理代码
│    .plt              │ ← 过程链接表（跳板到 GOT）
│    .rodata           │ ← 只读数据：字符串常量、switch 跳转表
│    .data             │ ← 已初始化全局/静态变量
│    .got              │ ← 全局偏移表（存放全局变量地址）
│    .got.plt          │ ← PLT 使用的 GOT（存放函数地址）★攻击热点
│    .bss              │ ← 未初始化全局/静态变量（不占文件空间）
│    .symtab           │ ← 符号表（完整，包括局部符号）
│    .dynsym           │ ← 动态符号表（仅导出/导入的符号）
│    .strtab           │ ← 符号名字符串表
│    .dynstr           │ ← 动态符号名字符串表
│    .rel.text/.rela.text  │ ← 重定位信息
│    .debug_*          │ ← 调试信息（-g 选项生成，DWARF 格式）
│    .comment          │ ← 编译器版本信息
│    .shstrtab         │ ← 节名字符串表
├─────────────────────┤
│    Section Header 0  │ ← 描述一个"节"（Section）
│    Section Header 1  │   节是给链接器和分析工具看的
│    ...               │   运行时不需要（可以 strip 掉）
└─────────────────────┘
```

关键区分：**段（Segment）** 是给内核的加载视图，**节（Section）** 是给链接器和分析工具的组织视图。一个段可以包含多个节。

```bash
# 查看 ELF Header
readelf -h /bin/ls
# 重点关注：
# Magic:   7f 45 4c 46 02 01 01 ...   ← ELF 魔数 + 64位 + 小端
# Type:    DYN (Position-Independent Executable)  ← PIE 可执行文件
# Entry point address: 0x1060           ← 程序入口（_start）
# Size of section headers: 64 (bytes)
# Number of section headers: 30

# 查看 Program Headers（段信息，加载器使用）
readelf -l /bin/ls
# 关键段：
# PHDR         程序头表本身
# INTERP       动态链接器路径（/lib64/ld-linux-x86-64.so.2）
# LOAD         需要加载到内存的段（通常有 2-3 个）
#   LOAD  R E   代码段（可读可执行）
#   LOAD  R W   数据段（可读可写）
# DYNAMIC      动态链接信息
# GNU_STACK    栈属性（检查是否有 Exec 标志→判断 NX）
# GNU_RELRO    RELRO 范围

# 查看 Section Headers（节信息）
readelf -S /bin/ls
# 输出中 R/W/A/X 标志含义：
# R=可读 W=可写 A=分配(占内存) X=可执行
```

#### 6.2.2 关键节与安全含义

| 节名 | 内容 | 安全相关性 |
|------|------|-----------|
| `.text` | 机器指令 | ROP gadgets 的来源；代码注入的目标 |
| `.rodata` | 字符串常量 | 信息泄露源；`/bin/sh` 等关键字符串的位置 |
| `.data` | 已初始化全局变量 | 全局变量覆写攻击 |
| `.bss` | 未初始化全局变量 | 大块可写区域；有时可执行 |
| `.plt` | 函数跳转桩 | ret2plt 攻击；泄露函数地址 |
| `.got` | 全局变量地址 | GOT 覆写攻击（Partial RELRO） |
| `.got.plt` | 函数实际地址 | ★ 核心攻击目标：覆写后劫持函数调用 |
| `.init_array` | 初始化函数指针数组 | 覆写可劫持程序启动流程 |
| `.fini_array` | 清理函数指针数组 | 覆写可劫持程序退出流程 |
| `.dynsym` | 动态符号表 | 确定 libc 函数偏移 |
| `.dynamic` | 动态链接元数据 | 确定 .so 依赖链 |

```bash
# 查看特定节的内容
readelf -x .rodata /bin/ls       # 以十六进制显示 .rodata
objdump -s -j .rodata /bin/ls    # 同上，另一种格式
readelf -x .got.plt /bin/ls      # 查看 GOT.PLT 内容

# 查找 /bin/sh 字符串在 libc 中的地址（漏洞利用常用）
strings -t x /lib/x86_64-linux-gnu/libc.so.6 | grep "/bin/sh"
# 输出示例：1b75aa  /bin/sh

# 查看 .init_array 和 .fini_array
readelf -S /bin/ls | grep -E 'init_array|fini_array'
```

#### 6.2.3 GOT/PLT 机制详解

GOT/PLT 是动态链接的核心机制，也是二进制漏洞利用中最常见的攻击面。

```text
函数调用流程（以 printf 为例）：

第一次调用 printf：
  main → call printf@PLT
         → jmp *GOT[printf]      # GOT 中存的是 PLT 第二条指令
         → push 0                # 函数编号
         → jmp PLT[0]            # 跳到 PLT[0]（动态链接器入口）
         → ld.so 解析 printf 真实地址
         → 将真实地址写入 GOT[printf]
         → 调用 printf

后续调用 printf：
  main → call printf@PLT
         → jmp *GOT[printf]      # GOT 中已经是 printf 真实地址
         → 直接进入 printf       # 跳过链接器，零开销

┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│     .plt     │     │   .got.plt   │     │ libc.so      │
│              │     │              │     │              │
│ PLT[0]:      │     │ GOT[0]: .dynamic│  │              │
│  push linkmap│────→│ GOT[1]: link_map│  │              │
│  jmp ld.so   │     │ GOT[2]: _dl_runtime│              │
│              │     │              │     │              │
│ PLT[1]:printf│     │ GOT[3]: printf│    │              │
│  jmp *GOT[3]│────→│  (初始→PLT[1]│    │              │
│  push 0      │     │   解析后→printf)──→│ printf()     │
│  jmp PLT[0]  │     │              │     │              │
└──────────────┘     └──────────────┘     └──────────────┘
```

```bash
# 查看 PLT 表
objdump -d -j .plt /bin/ls
# 每个 PLT 条目格式相同：
# push <reloc_index>
# jmp  PLT[0]

# 查看 GOT.PLT 表
readelf -r /bin/ls | grep R_X86_64_JUMP_SLOT
# 每条记录对应一个通过 PLT 调用的外部函数

# 实际观察 GOT 覆写效果
# 编译一个简单程序
cat > got_demo.c << 'EOF'
#include <stdio.h>
#include <stdlib.h>

void my_puts(const char *s) {
    printf("[HOOKED] %s\n", s);
}

int main(void) {
    puts("Hello");          // 正常调用
    
    // 查看 puts@GOT 地址
    // （实际利用中通过信息泄露获取）
    printf("puts@GOT = %p\n", &puts);  // 概念演示
    
    return 0;
}
EOF
gcc -no-pie got_demo.c -o got_demo
objdump -d -j .plt got_demo | grep -A2 'puts@plt'
```

#### 6.2.4 调试信息（DWARF）

DWARF 是 ELF 文件中调试信息的标准格式，`-g` 选项生成：

```bash
# 带调试信息编译
gcc -g hello.c -o hello_debug

# 不带调试信息编译（默认）
gcc hello.c -o hello_stripped

# 对比文件大小
ls -lh hello_debug hello_stripped

# 查看调试节
readelf -S hello_debug | grep debug
# .debug_info      变量类型、函数参数、源码行号
# .debug_abbrev    缩写表
# .debug_line      行号与地址映射
# .debug_str       调试字符串

# 剥离调试信息
strip hello_debug
# 或编译时指定
gcc -s hello.c -o hello_stripped

# DWARF 信息在安全研究中的价值：
# 1. 恢复函数签名和参数类型 → 逆向分析
# 2. 源码行号映射 → 精确定位漏洞代码
# 3. 局部变量布局 → 构造精确的溢出 payload
# 4. 结构体/类定义 → 理解内存布局
```

#### 6.2.5 符号表与 strip

```bash
# 符号表有三种：
readelf -s program.o           # .symtab：完整符号表（链接用，strip 后消失）
readelf --dyn-syms program     # .dynsym：动态符号表（运行时需要，strip 后保留）
# nm program.o                 # nm 也显示符号表（等价于 readelf -s）

# 符号绑定类型：
# LOCAL   - 文件内部符号（static 函数/变量）
# GLOBAL  - 全局符号（非 static）
# WEAK    - 弱符号（可被同名强符号覆盖）

# strip 的效果
ls -lh hello_debug
strip -s hello_debug           # strip 所有符号
ls -lh hello_debug             # 明显变小

# strip 移除的内容：
# .symtab（完整符号表）
# .strtab（符号名字符串）
# .debug_*（调试信息）
# 不移除：.dynsym（动态符号表，运行时需要）
```

### 6.3 库的创建与使用

#### 6.3.1 静态库（.a）

```bash
# 创建静态库
cat > libmath.c << 'EOF'
int add(int a, int b) { return a + b; }
int mul(int a, int b) { return a * b; }
EOF
gcc -c libmath.c -o libmath.o
ar rcs libmath.a libmath.o

# ar 工具用法：
ar t libmath.a          # 列出 .a 中的 .o 文件
ar x libmath.a          # 提取 .o 文件
ar r libmath.a new.o    # 添加/替换 .o 文件

# 使用静态库
gcc main.c -L. -lmath -o main_static
# -L 指定库搜索路径，-l 指定库名（去掉 lib 前缀和 .a 后缀）

# 静态库本质就是 .o 文件的打包
# 链接器只提取需要的 .o（不是函数级别）
```

#### 6.3.2 动态库（.so）

```bash
# 创建动态库（必须编译为位置无关代码 -fPIC）
gcc -fPIC -c libmath.c -o libmath_pic.o
gcc -shared -o libmath.so libmath_pic.o
# 或一步完成：
gcc -fPIC -shared libmath.c -o libmath.so

# 使用动态库
gcc main.c -L. -lmath -o main_dynamic
# 运行时需要能找到 .so 文件
export LD_LIBRARY_PATH=.:$LD_LIBRARY_PATH
./main_dynamic

# 查看 .so 的导出符号
nm -D libmath.so              # 动态符号
objdump -T libmath.so         # 动态符号表

# 查看 .so 的 SONAME
readelf -d libmath.so | grep SONAME

# 动态库搜索顺序（Linux）：
# 1. DT_RPATH（编译时硬编码路径，已弃用）
# 2. LD_LIBRARY_PATH 环境变量
# 3. /etc/ld.so.cache（ldconfig 缓存）
# 4. /lib, /usr/lib

# LD_PRELOAD：强制加载指定 .so（优先级最高）
# 安全用途：hook libc 函数、恶意提权
LD_PRELOAD=./hook.so ./target_program
```

### 6.4 构建系统

实际项目使用构建系统管理编译过程。

#### 6.4.1 Makefile

```makefile
# 基础 Makefile
CC = gcc
CFLAGS = -Wall -Wextra -g -O2
LDFLAGS = -lm

TARGET = program
SRCS = main.c utils.c math_utils.c
OBJS = $(SRCS:.c=.o)

# 默认目标
all: $(TARGET)

# 链接
$(TARGET): $(OBJS)
	$(CC) $(OBJS) -o $@ $(LDFLAGS)

# 编译：每个 .c → .o
%.o: %.c
	$(CC) $(CFLAGS) -c $< -o $@

# 自动依赖生成
depend: $(SRCS)
	$(CC) -MM $(SRCS) > .depend

-include .depend

# 清理
clean:
	rm -f $(OBJS) $(TARGET)

.PHONY: all clean depend
```

### 6.5 安全编译选项

编译选项直接决定二进制文件的安全防护级别。以下从安全攻防角度逐一解析每个关键选项。

#### 6.5.1 栈保护

```bash
# Stack Canary（栈金丝雀）
gcc -fstack-protector          # 仅对有 char 数组的函数启用
gcc -fstack-protector-all      # 对所有函数启用
gcc -fstack-protector-strong   # 对包含数组、取地址变量的函数启用（推荐）
gcc -fno-stack-protector       # 关闭（CTF/学习用）

# 生成汇编中 canary 的表现：
# 函数入口：mov rax, fs:[0x28]   ; 从 TLS 读取 canary
#           mov [rbp-8], rax     ; 放到栈上
# 函数出口：mov rax, [rbp-8]
#           xor rax, fs:[0x28]   ; 比较
#           jnz __stack_chk_fail ; 不匹配则终止
```

#### 6.5.2 NX（不可执行栈）

```bash
# 控制栈/堆是否可执行
gcc -z execstack vuln.c -o vuln      # 关闭 NX（栈可执行）→ 适合 shellcode
gcc -z noexecstack prog.c -o prog    # 开启 NX（默认）

# 查看 NX 状态
readelf -l program | grep GNU_STACK
# Flags: R W      → NX 开启（无 E）
# Flags: R W E    → NX 关闭（有 E）
```

#### 6.5.3 PIE（地址空间布局随机化）

```bash
# PIE 使程序每次加载到不同地址
gcc -pie -fPIE prog.c -o prog         # 开启 PIE（现代发行版默认）
gcc -no-pie -fno-PIE prog.c -o prog   # 关闭 PIE

# 查看 PIE 状态
file program
# "ELF 64-bit LSB pie executable" → PIE 开启
# "ELF 64-bit LSB executable"     → PIE 关闭

# PIE 关闭时的地址布局：
# .text 总是从固定地址开始（如 0x400000）
# 便于构造 ROP 链（gadgets 地址已知）
```

#### 6.5.4 RELRO（GOT 保护）

```bash
# RELRO 控制 GOT 表的写权限
gcc -z relro prog.c -o prog           # Partial RELRO（默认）
gcc -z relro -z now prog.c -o prog    # Full RELRO

# 查看 RELRO 状态
checksec ./program   # 使用 pwntools 的 checksec
# 或手动查看：
readelf -l program | grep GNU_RELRO
readelf -d program | grep BIND_NOW    # Full RELRO 有此条目

# Partial vs Full RELRO：
# Partial: .got 只读，.got.plt 可写 → GOT 覆写仍可行
# Full:    启动时解析所有符号，.got.plt 设为只读 → GOT 覆写不可行
# 代价：Full RELRO 增加启动时间（所有库函数需预解析）
```

#### 6.5.5 其他安全选项

```bash
# Fortify Source：替换不安全的 libc 函数
gcc -D_FORTIFY_SOURCE=2 -O2 prog.c   # 编译时+运行时检查
# strcpy → __strcpy_chk（检查目标缓冲区大小）
# sprintf → __sprintf_chk
# gets → 编译错误（完全禁止）

# 栈不可执行 + 栈对齐
gcc -mpreferred-stack-boundary=4 prog.c  # 16字节对齐

# 格式化字符串保护
gcc -Wformat -Wformat-security prog.c    # 警告格式化字符串问题
gcc -Werror=format-security prog.c       # 将警告升级为错误

# 控制流完整性（CFI）— Clang
clang -fsanitize=cfi prog.c              # 间接调用/跳转检查

# AddressSanitizer（内存错误检测）
gcc -fsanitize=address -g prog.c         # 检测溢出、UAF、double-free
# 运行时开销约 2x，适合开发/测试环境

# 学习环境：关闭所有保护
gcc -g -fno-stack-protector -no-pie -z execstack -z norelro \
    -D_FORTIFY_SOURCE=0 -fno-pie \
    -o vuln vuln.c

# 生产环境：开启所有保护
gcc -O2 -fstack-protector-strong -pie -fPIE \
    -z relro -z now -D_FORTIFY_SOURCE=2 \
    -Wformat -Wformat-security -Werror=format-security \
    -o safe safe.c
```

#### 6.5.6 checksec 实战

```bash
# 使用 pwntools 的 checksec（Python）
python3 -c "from pwn import *; ELF('/bin/ls').checksec()"

# 使用 checksec 脚本
# https://github.com/slimm609/checksec.sh
./checksec.sh --file=/bin/ls
# 输出示例：
# RELRO           STACK CANARY      NX            PIE
# Full RELRO      Canary found      NX enabled    PIE enabled

# 使用 checksec.sh 分析自编译程序
gcc -g -fno-stack-protector -no-pie vuln.c -o vuln
./checksec.sh --file=vuln
# RELRO           STACK CANARY      NX            PIE
# Partial RELRO   No canary found   NX disabled   No PIE
# → 全部关闭，经典的 CTF 练习环境
```

### 6.6 交叉编译

交叉编译指在一种架构上编译出另一种架构的可执行文件，在 IoT 安全、嵌入式设备漏洞研究中必不可少。

```bash
# 安装交叉编译工具链（Debian/Ubuntu）
apt install gcc-aarch64-linux-gnu      # ARM64
apt install gcc-arm-linux-gnueabihf    # ARM32 (hard-float)
apt install gcc-mips-linux-gnu         # MIPS
apt install gcc-mipsel-linux-gnu       # MIPS (little-endian)
apt install gcc-powerpc-linux-gnu      # PowerPC

# 交叉编译示例
aarch64-linux-gnu-gcc hello.c -o hello_arm64
file hello_arm64
# hello_arm64: ELF 64-bit LSB executable, ARM aarch64...

# 使用 QEMU 模拟运行
apt install qemu-user
qemu-aarch64 ./hello_arm64
# 或指定动态链接器
qemu-aarch64 -L /usr/aarch64-linux-gnu/ ./hello_arm64

# 静态编译避免链接器问题
aarch64-linux-gnu-gcc -static hello.c -o hello_arm64_static
qemu-aarch64 ./hello_arm64_static   # 无需 -L 参数

# 安装目标架构的 libc（用于动态链接分析）
apt install libc6:arm64               # 多架构支持
# 或下载 libc
apt download libc6-arm64-cross
```

### 6.7 常见编译/链接错误与排查

| 错误信息 | 原因 | 解决方法 |
|---------|------|---------|
| `undefined reference to 'func'` | 链接时找不到符号定义 | 检查是否缺少 `.o` 文件或 `-l库名` |
| `multiple definition of 'func'` | 多个编译单元定义了同名全局符号 | 加 `static` 或用头文件声明 |
| `relocation truncated to fit` | 代码段过大，相对跳转超出范围 | 加 `-mcmodel=large` 或拆分代码 |
| `cannot find -lxxx` | 链接器找不到库 | `-L` 指定路径，或 `apt install libxxx-dev` |
| `ld-linux.so: No such file` | 动态链接器路径错误 | 检查 `INTERP` 段，设置 `LD_LIBRARY_PATH` |
| `Segmentation fault` (链接后) | 栈对齐问题或初始化代码崩溃 | 检查 `-mpreferred-stack-boundary` |
| `error: expected ';'` (预处理后) | 宏展开引入语法错误 | 用 `gcc -E` 查看展开结果 |

```bash
# 调试链接问题
gcc -Wl,--verbose main.c -o main 2>&1 | less    # 详细链接过程
gcc -Wl,-M main.c -o main 2>&1 | less           # 链接器内存映射
nm -C main | grep ' U '                          # 列出未定义符号

# 查看链接器搜索路径
ld --verbose 2>&1 | grep SEARCH_DIR | tr ';' '\n'
```

### 6.8 本章要点回顾

```text
编译流水线：
  源代码 → [预处理器] → 纯C代码 → [编译器] → 汇编代码
         → [汇编器] → .o目标文件 → [链接器] → 可执行文件

安全关键结构：
  ELF Header   → 入口点、文件类型（PIE/非PIE）
  .text        → ROP gadgets 来源
  .plt/.got.plt → 动态函数跳转（GOT 覆写攻击目标）
  .bss         → 可写大块内存（有时可执行）
  .dynamic     → 动态链接元数据

安全编译选项对照：
  选项                  开启效果              关闭方法
  Stack Canary          栈溢出检测            -fno-stack-protector
  NX                    栈/堆不可执行         -z execstack
  PIE                   地址随机化            -no-pie
  Full RELRO            GOT 只读              -z norelro
  FORTIFY               不安全函数替换        -D_FORTIFY_SOURCE=0
  ASLR                  系统级地址随机化      echo 0 > /proc/sys/kernel/randomize_va_space

工具链：
  gcc -E/-S/-c       逐步编译
  readelf            ELF 结构分析
  objdump            反汇编
  nm                 符号查看
  strip              剥离符号/调试信息
  checksec           安全选项检查
  ldd                动态依赖查看
  ldconfig           动态链接器缓存管理
```
