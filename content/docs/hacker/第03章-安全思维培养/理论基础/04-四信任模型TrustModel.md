## 四、信任模型（Trust Model）

### 4.1 信任的本质

在安全领域，"信任"不是一种感觉，而是一种**可量化、可审计、可撤销的授权关系**。当系统 A "信任"系统 B 时，意味着 A 将某些决策权或数据访问权委托给了 B，同时承担 B 滥用这些权限的风险。

理解信任模型的核心，在于回答以下四个问题：

| 问题 | 含义 | 安全影响 |
|------|------|----------|
| 信任谁（Who） | 哪些实体被授予了信任 | 攻击面的范围 |
| 信任什么（What） | 被信任的具体内容是什么 | 权限的粒度 |
| 信任依据（Why） | 凭什么给予信任 | 身份验证和认证机制 |
| 信任可否传递（Transitivity） | A 信任 B，B 信任 C，A 是否信任 C | 信任链的长度和风险 |

信任是安全中最危险的假设。历史上几乎每一起重大安全事件，都可以追溯到一个被错误授予或过度延伸的信任关系。SolarWinds 事件中，18,000 个组织信任了被植入后门的软件更新；Heartbleed 漏洞让攻击者可以窃取服务器的私钥，从而冒充任何 HTTPS 网站——这些都源于对信任边界的错误假设。

### 4.2 信任边界的本质

每个系统都有信任边界（Trust Boundary）。信任边界内是被默认认为"可信"的实体和数据，边界外是"不可信"的。**安全问题绝大多数发生在信任边界的交叉点上**——也就是不可信数据被当作可信数据处理的那一刻。

#### 4.2.1 典型信任边界

```text
┌─────────────────────────────────────────────────────┐
│                    互联网（不可信）                    │
│  攻击者、恶意输入、伪造请求、中间人                    │
└──────────────────────┬──────────────────────────────┘
                       │  ← 网络信任边界（防火墙/WAF）
┌──────────────────────▼──────────────────────────────┐
│                   DMZ（半信任）                       │
│  Web 服务器、反向代理、负载均衡                       │
└──────────────────────┬──────────────────────────────┘
                       │  ← 应用信任边界（认证/鉴权）
┌──────────────────────▼──────────────────────────────┐
│               内部网络（受控信任）                     │
│  应用服务器、消息队列、缓存                           │
└──────────────────────┬──────────────────────────────┘
                       │  ← 数据信任边界（加密/ACL）
┌──────────────────────▼──────────────────────────────┐
│               数据层（最小信任）                       │
│  数据库、文件存储、备份系统                           │
└─────────────────────────────────────────────────────┘
```

每一层之间的箭头，都是一个信任边界交叉点。攻击者的目标，就是穿透这些边界。

#### 4.2.2 信任边界的分类

| 边界类型 | 描述 | 典型控制措施 | 常见攻击手法 |
|----------|------|-------------|-------------|
| 网络边界 | 内网与外网之间 | 防火墙、VPN、NAT | 端口扫描、隧道穿透、VPN 凭据窃取 |
| 进程边界 | 不同进程之间的隔离 | 操作系统进程隔离、容器 | 提权、共享内存攻击、竞态条件 |
| 用户边界 | 不同用户的数据和权限 | 认证、鉴权、RBAC | 横向移动、IDOR、会话劫持 |
| 权限边界 | 不同权限级别之间 | UAC、sudo、Capabilities | 提权漏洞、配置错误 |
| 数据边界 | 敏感数据与非敏感数据 | 加密、脱敏、访问控制 | 注入攻击、越权访问、数据泄露 |
| 代码边界 | 可信代码与不可信输入 | 输入验证、沙箱、CSP | 注入攻击、XSS、反序列化 |
| 供应链边界 | 自研代码与第三方组件 | SCA 签名验证、SBOM | 供应链投毒、依赖混淆 |

#### 4.2.3 信任边界在哪里被打破

从攻击者的视角看，突破信任边界的手法可归为三类：

1. **绕过（Bypass）**：完全不经过边界检查。例如：利用未被防火墙规则覆盖的端口、利用 SSRF 从内网发起请求以绕过网络边界。

2. **伪装（Impersonation）**：让不可信数据看起来可信。例如：SQL 注入将恶意 SQL 伪装成合法参数、JWT 签名伪造、ARP 欺骗。

3. **滥用（Abuse）**：利用已被信任的实体去做不该做的事。例如：合法用户越权访问他人数据、已授权的 OAuth Token 被用于未授权的 scope。

### 4.3 信任模型的分类

#### 4.3.1 基于边界的安全模型（Perimeter-Based）

这是最传统的安全模型，核心思想是"城堡与护城河"：网络外部不可信，内部可信。

```text
                    ┌─── 防火墙 ───┐
    不可信（互联网） ─┤              ├─ 可信（内网）
                    └──────────────┘
```

**典型实现**：防火墙 + VPN + 网络准入控制（NAC）

**优点**：
- 架构简单，易于理解和部署
- 对内部应用透明，不需要修改业务代码
- 在网络边界集中实施安全策略，运维成本低

**致命缺陷**：
- 假设内部都是安全的，一旦攻击者突破边界（通过钓鱼、VPN 漏洞、供应链攻击），内网几乎不设防
- 无法应对内部威胁（恶意员工、被入侵的内部主机）
- 边界日益模糊：云服务、远程办公、BYOD 让"内部网络"的定义瓦解
- 横向移动畅通无阻：攻击者进入内网后可以在系统间自由跳转

**历史案例**：2014 年 Sony Pictures 被朝鲜黑客组织入侵。攻击者通过钓鱼邮件获得域管理员凭据后，在内部网络中横向移动数月，最终窃取了约 100TB 数据，包括未发布电影、员工个人信息、高管邮件。传统的边界安全模型完全没有检测到这次入侵，因为攻击者始终在"受信任"的内部网络中操作。

#### 4.3.2 零信任架构（Zero Trust Architecture, ZTA）

零信任的核心哲学可以用一句话概括：**永不信任，始终验证（Never Trust, Always Verify）**。

零信任不是某个具体产品，而是一种**安全架构理念**，由 Forrester Research 的分析师 John Kindervag 于 2010 年提出，后被 NIST 在 SP 800-207 标准中正式定义。

**五大核心原则**：

| 原则 | 说明 | 具体措施 |
|------|------|----------|
| 永不信任，始终验证 | 无论请求来自内网还是外网，都必须经过身份验证和授权 | mTLS、OAuth 2.0、OIDC、持续认证 |
| 最小权限访问 | 每个实体只获得完成当前任务所需的最小权限 | JIT（Just-In-Time）权限、PAM、ABAC |
| 微分段 | 将网络分割成小的安全区域，限制横向移动 | 网络策略、服务网格、mTLS between services |
| 持续验证 | 不是一次验证后就永久信任，而是持续评估 | 行为分析、UEBA、设备健康检查、风险评分 |
| 假设已被入侵 | 系统设计时就假设攻击者已经在内网 | 加密所有通信、最小暴露面、爆炸半径控制 |

**零信任架构的关键组件**：

```text
┌──────────────────────────────────────────────────────────────┐
│                      策略决策点（PDP）                         │
│  ┌─────────────┐  ┌──────────────┐  ┌────────────────────┐  │
│  │  身份引擎    │  │  设备引擎     │  │  环境/风险引擎     │  │
│  │  (IAM)       │  │  (MDM/EDR)   │  │  (SIEM/UEBA)      │  │
│  └──────┬──────┘  └──────┬───────┘  └────────┬───────────┘  │
│         └───────────────┬┘───────────────────┘               │
│                         ▼                                    │
│              ┌─────────────────────┐                         │
│              │   策略引擎（PE）     │                         │
│              │  综合评估并决策      │                         │
│              └─────────┬───────────┘                         │
└────────────────────────┼─────────────────────────────────────┘
                         │ 授权/拒绝
                         ▼
┌──────────────────────────────────────────────────────────────┐
│                      策略执行点（PEP）                         │
│  API 网关 / 服务网格 Sidecar / 微分段网关 / 应用层代理        │
└──────────────────────────────────────────────────────────────┘
```

**零信任的实施路径**：

1. **识别阶段**：梳理所有数据流、用户、设备和应用，绘制完整的资产清单
2. **映射阶段**：绘制业务数据流图，标记每个流经的信任边界
3. **设计阶段**：定义策略引擎的决策逻辑（身份 + 设备状态 + 环境风险 → 允许/拒绝）
4. **部署阶段**：从最关键的数据和应用开始，逐步扩展零信任控制
5. **运维阶段**：持续监控策略有效性，基于实际事件调优规则

**零信任的常见误解**：

- ❌ "零信任 = 不信任任何人" → ✅ 零信任是"不默认信任，但可以通过验证建立信任"
- ❌ "零信任是一个产品" → ✅ 零信任是架构理念，需要多种技术组合实现
- ❌ "部署了 ZTNA 就是零信任" → ✅ ZTNA 只是零信任的一个组件，还需要 IAM、微分段、数据保护等
- ❌ "零信任会让系统变慢" → ✅ 合理设计下，零信任对用户体验的影响可以很小（通过风险自适应认证）

#### 4.3.3 基于角色的信任模型（Role-Based Trust）

RBAC（Role-Based Access Control）是最广泛使用的信任模型之一。信任不是授予个体，而是授予角色。

```text
用户 ──分配──→ 角色 ──授权──→ 权限
张三 ──→ 开发工程师 ──→ 读写代码仓库、提交 PR
李四 ──→ 运维工程师 ──→ 部署生产环境、查看监控
王五 ──→ 安全工程师 ──→ 审计日志、漏洞扫描
```

**优点**：易于管理，权限与人员解耦，适合组织规模较大的场景
**局限**：角色爆炸（Role Explosion）、无法表达上下文相关的权限（如"只在工作时间允许"）

#### 4.3.4 基于属性的信任模型（Attribute-Based Trust）

ABAC（Attribute-Based Access Control）通过评估多个属性来决定是否授予信任。

```text
决策 = f(用户属性, 资源属性, 环境属性, 操作属性)

示例规则：
IF 用户.部门 == "研发" AND 用户.安全等级 >= 3
   AND 资源.分类 == "源代码"
   AND 环境.时间 IN 工作时间
   AND 环境.位置 == 公司网络
THEN 允许读取
```

**优势**：比 RBAC 更灵活，能表达复杂的上下文策略
**挑战**：策略复杂度高，调试困难，需要完善的属性管理基础设施

#### 4.3.5 基于声誉的信任模型（Reputation-Based）

通过历史行为积累信任分数，常用于网络安全和分布式系统。

| 场景 | 声誉来源 | 应用 |
|------|----------|------|
| IP 信誉 | 历史攻击记录、黑名单 | 防火墙、WAF 自动封禁 |
| 用户信誉 | 异常登录频率、操作模式 | 风控系统、反欺诈 |
| 软件信誉 | 签名状态、下载量、漏洞历史 | 应用白名单、EDR |
| 证书信誉 | 证书透明度、吊销状态 | 浏览器证书信任 |

### 4.4 信任链分析

在安全分析中，**追踪信任链（Trust Chain）** 是最关键的技巧之一。信任链描述了信任如何从一个实体传递到另一个实体，攻击者的目标就是找到链条中最薄弱的环节。

#### 4.4.1 典型信任链

**Web 应用信任链**：

```text
用户(浏览器)
  │ 信任依据：TLS 证书 + 域名
  ▼
CDN/WAF
  │ 信任依据：源站 IP 白名单
  ▼
负载均衡器
  │ 信任依据：后端健康检查
  ▼
Web 服务器 (Nginx)
  │ 信任依据：反向代理配置
  ▼
应用服务器 (Spring/Django)
  │ 信任依据：数据库凭据
  ▼
数据库 (MySQL/PostgreSQL)
  │ 信任依据：备份密钥
  ▼
备份系统 (S3/OSS)
```

**软件供应链信任链**：

```text
开发者
  │ 信任依据：GPG 签名 / SSH Key
  ▼
代码仓库 (GitHub)
  │ 信任依据：OAuth Token / Webhook Secret
  ▼
CI/CD 系统 (GitHub Actions)
  │ 信任依据：Runner 注册 Token
  ▼
构建产物
  │ 信任依据：SHA256 校验和
  ▼
镜像仓库 (Docker Hub/ECR)
  │ 信任依据：镜像签名 (Cosign/Notary)
  ▼
部署系统 (Kubernetes)
  │ 信任依据：Admission Controller 策略
  ▼
运行时 (Container Runtime)
```

#### 4.4.2 信任链的攻击分析

每一个信任传递节点，都是一个潜在的攻击面。分析信任链时，需要对每个节点问三个问题：

1. **这个节点的信任依据是否充分？** 仅靠 IP 白名单不如 mTLS + 身份验证可靠
2. **信任被滥用时的影响范围有多大？** 一个节点被攻破后，下游所有节点都受影响
3. **是否有检测机制？** 能否及时发现某个节点的信任被滥用

```text
信任链风险评估矩阵：

节点            信任依据         可靠性   影响范围   检测能力   综合风险
─────────────────────────────────────────────────────────────────────
浏览器→CDN      TLS证书          高       中         高         低
CDN→源站        IP白名单         中       高         中         中
应用→数据库     账号密码         中       极高       中         高
CI/CD→镜像      SHA256           高       极高       低         中
K8s→容器        准入策略         高       极高       高         低
```

#### 4.4.3 信任传递的三种模型

**线性传递（Linear Transitivity）**：

```text
A 信任 B，B 信任 C → A 信任 C
```

这是最危险的信任传递模型。如果 B 被攻破，攻击者可以利用 B 作为跳板攻击 C，而 A 不会察觉。

**条件传递（Conditional Transitivity）**：

```text
A 信任 B（仅限读取），B 信任 C（仅限写入）→ A 不自动信任 C
```

条件传递比线性传递安全，但需要精确的策略定义。

**委托传递（Delegated Transitivity）**：

```text
A 委托 B 代表 A 向 C 请求服务，但 B 不能以 A 的身份做其他事
```

OAuth 2.0 的授权码模式就是典型的委托传递：用户授权 GitHub 代表自己向第三方应用提供身份信息，但第三方应用无法以用户身份做其他操作。

### 4.5 身份与认证中的信任模型

身份认证是建立信任的第一步。不同的认证机制提供不同强度的信任级别。

#### 4.5.1 认证因素与信任级别

| 认证因素 | 示例 | 信任强度 | 易受攻击方式 |
|----------|------|----------|-------------|
| 知识因素（你知道什么） | 密码、PIN、安全问题 | 低 | 暴力破解、钓鱼、数据泄露 |
| 持有因素（你拥有什么） | 手机 OTP、硬件 Token、智能卡 | 中 | SIM Swap、中间人、Token 窃取 |
| 固有因素（你是什么） | 指纹、人脸、虹膜 | 中高 | 传感器欺骗、深度伪造 |
| 行为因素（你怎么做） | 打字节奏、鼠标轨迹、步态 | 中 | 行为模式模仿 |
| 上下文因素（你在哪） | IP、地理位置、时间、设备指纹 | 辅助 | VPN、GPS 伪造、设备克隆 |

信任强度随认证因素的组合而增强：

```text
单因素（密码）     → 信任级别 1（低）   → 访问公开资源
双因素（密码+OTP）  → 信任级别 2（中）   → 访问个人数据
多因素+上下文       → 信任级别 3（高）   → 执行敏感操作
持续认证+行为分析   → 信任级别 4（极高） → 访问核心系统
```

#### 4.5.2 PKI 与证书信任模型

公钥基础设施（PKI）是互联网信任的基石。浏览器如何知道 google.com 的证书是合法的？答案是**证书信任链**。

```text
根证书（Root CA）              ← 操作系统/浏览器预装
  │ 自签名，信任的最终锚点
  ▼
中间证书（Intermediate CA）    ← 根 CA 签发
  │ 分离根 CA 的风险，实际签发终端证书
  ▼
终端证书（End-Entity Cert）    ← 中间 CA 签发
  │ 绑定到具体的域名和服务器
  ▼
TLS 握手（验证身份）            ← 浏览器验证整条链
```

**PKI 信任模型的脆弱性**：

- **CA 被入侵**：2011 年 DigiNotar CA 被入侵，攻击者签发了 google.com 的伪造证书，用于伊朗政府对本国公民的中间人攻击。DigiNotar 最终破产。
- **证书透明度（Certificate Transparency）**：为防止 CA 私下签发恶意证书，所有公开信任的证书都必须记录在公开的 CT 日志中，任何人都可以审计。
- **证书吊销的困境**：CRL 和 OCSP 机制存在实时性和可靠性问题，浏览器往往选择"软失败"（继续连接），这削弱了吊销机制的安全价值。

#### 4.5.3 OAuth 2.0 与委托信任

OAuth 2.0 解决的核心问题是：**用户如何在不暴露密码的情况下，授权第三方应用访问自己的资源**。

```text
┌──────┐  1. 授权请求  ┌──────────┐
│      │──────────────→│          │
│ 用户 │               │ 第三方应用│
│      │←──────────────│          │
└──┬───┘  2. 授权码     └────┬─────┘
   │                         │ 3. 用授权码换 Token
   │    ┌──────────────┐     │
   └───→│  OAuth Provider│←──┘
        │  (GitHub/Google)│
        └───────┬───────┘
                │ 4. 返回 Access Token
                ▼
        ┌──────────────┐
        │   资源服务器   │
        │  (API/数据)    │
        └──────────────┘
```

**OAuth 中的信任边界**：

- 用户信任 OAuth Provider（GitHub/Google 等）
- 用户信任第三方应用不会滥用 Token
- OAuth Provider 验证第三方应用的身份（Client ID + Secret）
- **薄弱点**：用户通常不会仔细检查授权的 scope，过度授权是最常见的风险

### 4.6 信任模型在攻防中的应用

#### 4.6.1 攻击者的信任链利用思维

优秀的攻击者不会硬攻系统，而是**找到信任链中最薄弱的环节**，利用它来获取下游的信任。

**经典攻击模式**：

| 攻击模式 | 描述 | 真实案例 |
|----------|------|----------|
| 供应链投毒 | 攻击信任链上游，自动获得下游所有系统的信任 | SolarWinds（2020）、event-stream（2018） |
| CA 攻击 | 攻击或胁迫 CA，获取任意域名的合法证书 | DigiNotar（2011）、CNNIC（2015） |
| 信任域跳跃 | 利用受信任的内部系统作为跳板 | 大多数 APT 攻击的横向移动阶段 |
| 委托滥用 | 利用过度授权的 OAuth/API Key | GitHub Token 泄露导致仓库被篡改 |
| 身份伪造 | 冒充受信任的身份 | 钓鱼邮件、BGP 劫持、DNS 欺骗 |

#### 4.6.2 防御者的信任链加固思维

防御者需要对信任链的每个环节进行加固：

1. **缩短信任链**：信任链越长，风险越大。能直接验证的不要通过中间人传递。
2. **加强信任依据**：从单因素升级到多因素，从共享密钥升级到 PKI。
3. **实施条件信任**：不是全有或全无，而是根据上下文动态调整信任级别。
4. **监控信任使用**：记录每次信任验证的日志，建立基线，检测异常。
5. **准备信任撤销**：确保在发现信任被滥用时，能够快速撤销（证书吊销、Token 失效、权限回收）。

### 4.7 分布式系统中的信任模型

#### 4.7.1 服务间信任（Service-to-Service Trust）

在微服务架构中，服务之间的信任是一个核心问题。

**三种常见的服务间信任模型**：

| 模型 | 描述 | 适用场景 | 实现方式 |
|------|------|----------|----------|
| 网络边界信任 | 同一网络内的服务默认互信 | 传统内网架构 | 网络隔离 + IP 白名单 |
| 共享密钥信任 | 服务通过共享的 API Key/Secret 互信 | 简单微服务 | 环境变量注入密钥 |
| mTLS 身份信任 | 每个服务有独立证书，通过 TLS 双向认证 | 零信任微服务 | Istio/Linkerd 服务网格 |
| SPIFFE/SPIRE | 标准化的服务身份框架 | 大规模分布式系统 | SPIFFE ID + SVID |

#### 4.7.2 区块链与去中心化信任

区块链试图用**数学共识**替代**中心化信任**。其核心创新是：不需要信任任何单一实体，只需要信任密码学和共识算法。

```text
传统信任模型：  用户 → 信任银行 → 银行记账
区块链模型：    用户 → 信任密码学 → 所有节点共同记账（共识）
```

但这并不意味着区块链消除了信任——它只是将信任从人转移到了代码和共识机制。智能合约的漏洞（如 2016 年 The DAO 攻击导致 6000 万美元损失）证明：**代码就是新的信任边界**。

### 4.8 信任模型的评估框架

如何评估一个系统的信任模型是否合理？以下是一个系统化的评估框架：

#### 4.8.1 信任模型评估清单

```yaml
信任模型评估清单：

身份信任：
  - [ ] 所有用户和服务是否有唯一身份标识
  - [ ] 认证机制是否达到所需信任级别
  - [ ] 是否支持多因素认证
  - [ ] 凭据存储是否安全（哈希加盐、密钥管理）
  - [ ] 身份生命周期管理是否完整（入职/离职/角色变更）

授权信任：
  - [ ] 是否遵循最小权限原则
  - [ ] 是否有定期权限审查机制
  - [ ] 特权操作是否有审批流程
  - [ ] 权限粒度是否足够细

数据信任：
  - [ ] 数据输入是否经过验证（来源、格式、范围）
  - [ ] 数据传输是否加密
  - [ ] 数据存储是否有访问控制
  - [ ] 数据完整性是否有校验

通信信任：
  - [ ] 服务间通信是否加密和认证
  - [ ] 是否防止重放攻击
  - [ ] 是否有消息完整性校验

供应链信任：
  - [ ] 第三方依赖是否有版本锁定和完整性校验
  - [ ] 是否有 SBOM（软件物料清单）
  - [ ] 是否定期扫描依赖漏洞
  - [ ] 构建和部署管道是否有完整性保护

监控与响应：
  - [ ] 信任使用是否有审计日志
  - [ ] 异常信任使用是否能被检测
  - [ ] 信任撤销机制是否可用且经过测试
  - [ ] 是否有信任事件的应急响应流程
```

#### 4.8.2 信任模型成熟度模型

| 等级 | 名称 | 特征 | 典型组织 |
|------|------|------|----------|
| Level 0 | 隐式信任 | 无明确信任模型，默认全信任 | 个人项目、小型初创公司 |
| Level 1 | 边界信任 | 有网络边界，内部默认信任 | 传统企业 |
| Level 2 | 角色信任 | 实施了 RBAC，有基本的权限管理 | 中型企业 |
| Level 3 | 最小信任 | 最小权限 + 审计日志 + 定期审查 | 大型企业 |
| Level 4 | 动态信任 | 基于上下文的动态信任评估 | 云原生企业 |
| Level 5 | 零信任 | 全面零信任架构，持续验证 | Google（BeyondCorp）、Microsoft |

### 4.9 常见误区与纠正

#### 误区一："内网是安全的"

**错误认知**：只要在防火墙后面，系统就是安全的。

**现实**：攻击者通过钓鱼邮件、VPN 漏洞、供应链攻击进入内网后，传统内网几乎是不设防的。根据 Mandiant 的报告，攻击者在内网的平均驻留时间（Dwell Time）在 2023 年仍超过 200 天。

**纠正**：假设攻击者已经在内网中，对所有内部通信实施加密和身份验证。

#### 误区二："零信任意味着完全不信任"

**错误认知**：零信任就是不信任任何人、任何事，会导致业务无法运转。

**现实**：零信任不是消除信任，而是**用验证替代默认信任**。合法用户通过多因素认证后仍然可以获得所需的访问权限。

**纠正**：零信任 = 默认不信任 + 动态验证 + 最小权限 + 持续监控。

#### 误区三："信任是一次性建立的"

**错误认知**：用户通过一次登录验证后，整个会话期间都应被信任。

**现实**：会话劫持、Token 窃取、账户接管等攻击都是在会话建立后发生的。一次验证不足以保证持续安全。

**纠正**：实施持续认证（Continuous Authentication），根据风险级别动态调整信任级别。例如：用户从新设备或异常位置发起敏感操作时，要求重新验证。

#### 误区四："HTTPS 就是安全的"

**错误认知**：使用了 HTTPS 就意味着通信是安全可信的。

**现实**：HTTPS 只保证传输层加密和服务器身份验证，不保证应用层安全。中间人可以通过受信任的 CA 证书（企业 CA）进行 TLS 拦截，应用层漏洞（如 XSS、业务逻辑漏洞）也不受 HTTPS 保护。

**纠正**：HTTPS 是必要条件但非充分条件，还需要应用层安全控制。

#### 误区五："第三方组件是可信的"

**错误认知**：广泛使用的开源组件或商业软件是安全的。

**现实**：event-stream（2018，被恶意维护者接管）、ua-parser-js（2021，被劫持投毒）、SolarWinds（2020，构建管道被植入后门）等事件证明，供应链是最容易被忽视的信任边界。

**纠正**：实施 SBOM、依赖版本锁定、完整性校验、定期漏洞扫描和供应链安全审计。

### 4.10 进阶主题

#### 4.10.1 机密计算与硬件信任根

现代安全架构越来越依赖**硬件信任根（Hardware Root of Trust）** 来建立最初的信任锚点。

```text
硬件信任根（TPM/安全芯片）
  │ 测量并验证固件完整性
  ▼
固件（BIOS/UEFI）
  │ 测量并验证引导加载程序
  ▼
引导加载程序（GRUB/Bootloader）
  │ 测量并验证操作系统内核
  ▼
操作系统内核
  │ 测量并验证用户空间组件
  ▼
应用层
```

这个过程称为**可信启动（Secure Boot / Measured Boot）**，每一层都验证下一层的完整性和身份，形成一条从硬件到应用的完整信任链。

**机密计算（Confidential Computing）** 更进一步：通过 TEE（Trusted Execution Environment，如 Intel SGX、AMD SEV、ARM TrustZone）保护运行中的数据，即使操作系统被入侵，TEE 中的数据也无法被窃取。

#### 4.10.2 后量子时代的信任模型

量子计算对当前基于 PKI 的信任模型构成根本威胁：Shor 算法可以在多项式时间内破解 RSA 和 ECC，这意味着当前的数字签名和密钥交换机制将不再安全。

应对策略：

- **后量子密码学（PQC）**：NIST 已标准化了基于格（Lattice）的 CRYSTALS-Kyber（密钥交换）和 CRYSTALS-Dilithium（数字签名）算法
- **密码敏捷性（Crypto Agility）**：系统设计应支持快速替换密码算法，而不是硬编码特定算法
- **混合方案**：在过渡期同时使用经典和后量子算法，确保即使一种被破解，另一种仍然安全
