# 分布式数据库核心概念

本节是分布式数据库理论体系的起点。在深入分片策略、共识算法、事务协议等具体技术之前，读者必须先建立一套完整的概念框架：什么是分布式数据库、它面临哪些根本挑战、数据如何分布、故障如何建模、关键术语如何定义。这些概念将贯穿后续所有章节，是理解整个分布式数据库领域的基石。

---

## 什么是分布式数据库

### 精确定义

分布式数据库是由多个物理上分离的节点（Node）组成的数据库系统，这些节点通过网络通信协作，对外呈现为一个逻辑上统一的数据库。核心特征可以概括为三点：

1. **数据分布**：数据被分割存储在多个物理节点上，任何单一节点都不持有全部数据
2. **节点自治**：每个节点具有独立的处理能力，能够执行本地计算和存储操作
3. **全局协调**：通过通信协议和一致性机制，多个节点协同完成跨节点的查询、事务和数据管理

需要强调的是，"分布式"不仅仅是"多台机器"。一个简单的主从复制（Master-Slave Replication）架构虽然涉及多个节点，但如果写入始终路由到单一主节点、从节点只负责读取，严格来说更接近"集中式+副本"模型。真正的分布式数据库要求数据的读写可以发生在多个节点上，且系统具备自主的数据管理和故障恢复能力。

### 与相关概念的辨析

很多读者容易混淆分布式数据库与几个相近概念，下面逐一厘清：

| 概念 | 定义 | 与分布式数据库的关系 |
|------|------|---------------------|
| 数据库集群 | 多台服务器组成一个服务单元 | 集群是物理部署方式，分布式是逻辑架构。集群可以是集中式的（如Oracle RAC共享存储） |
| 分布式系统 | 多台计算机协同完成任务 | 分布式数据库是分布式系统的一个子集，专注于数据管理 |
| 分库分表 | 将大库/大表拆分到多个实例 | 分库分表是应用层的垂直/水平拆分，缺乏统一的查询和事务支持 |
| NewSQL | 兼具SQL接口和分布式能力的数据库 | NewSQL是分布式数据库的一个子类，特指兼容SQL且支持ACID的系统 |
| 数据仓库 | 面向分析的OLAP数据库 | 数据仓库可以是分布式的（如Snowflake），但不一定是分布式数据库（可能只是MPP查询引擎） |

### 分布式数据库解决的四个根本问题

分布式数据库之所以存在，是因为单机数据库面临四个无法通过硬件升级解决的根本限制：

**容量墙（Capacity Wall）**：单机磁盘容量有限。一台配备最大容量SSD的服务器，存储上限通常在100TB以内。而现代互联网应用的数据量早已达到PB甚至EB级别。Google在2003年发表GFS论文时就指出，单台廉价磁盘的容量增长速度远不及数据增长速度。分布式数据库通过将数据分散存储在成百上千个节点上，突破了单机的存储上限。

**吞吐量墙（Throughput Wall）**：单机I/O带宽存在物理上限。即使是NVMe SSD，单盘顺序读写也仅约3-7GB/s，随机IOPS在百万级别封顶。当查询QPS达到数十万甚至百万级别时，单机磁盘成为不可逾越的瓶颈。分布式数据库将读写负载分散到多个节点，使总吞吐量随节点数线性增长。

**延迟墙（Latency Wall）**：对于全球用户，将数据放在单一数据中心意味着跨洲际的网络延迟。北京到纽约的RTT约为150-200ms。将数据部署在用户就近的区域可以将读延迟从200ms降低到10ms以内，但这就要求数据在地理上分布存储。

**可用性墙（Availability Wall）**：金融、电商等关键业务要求99.99%甚至99.999%的可用性，全年停机不超过52分钟甚至5分钟。单机数据库无论做多好的备份，硬件故障时的切换时间都难以压缩到秒级。分布式数据库通过多副本和自动故障转移，将单点故障的影响降到最低。

## 分布式系统的核心挑战：FLP不可能性与拜占庭问题

### FLP不可能性定理

1985年，Fischer、Lynch和Paterson证明了一个具有里程碑意义的结论——**FLP不可能性定理**（FLP Impossibility）：

> 在一个异步分布式系统中，只要有一个进程可能崩溃（Crash），就不存在一个确定性算法能够保证在有限时间内达成共识。

这个定理的含义极其深远：

- **异步系统**：消息传递没有时间上限，无法区分"消息延迟"和"节点故障"
- **崩溃故障**：节点突然停止工作（不是发送错误消息）
- **共识问题**：所有正确节点对某个值达成一致

FLP定理告诉我们，在纯理论层面，完美的共识是不可能的。但实际系统通过放宽假设来绕过这个限制：

| 绕过策略 | 代表方案 | 核心思路 |
|---------|---------|---------|
| 随机化 | Ben-Or算法、HoneyBadgerBFT | 引入随机性打破确定性限制，以概率1达成共识 |
| 部分同步假设 | Raft、Paxos | 假设系统最终会进入同步状态（有消息延迟上限） |
| 故障检测器 | Chandra-Toueg | 使用不完美的故障检测器，允许偶尔的误判 |
| 时间假设 | Spanner TrueTime | 使用物理时钟提供时间窗口，缩小不确定性 |

### 拜占庭故障与非拜占庭故障

分布式系统中的故障分为两大类：

**非拜占庭故障（Fail-Stop / Crash Fault）**：
- 节点要么正常工作，要么完全停止
- 不会发送错误消息
- 大多数分布式数据库假设的就是这种故障模型
- 代表系统：Raft、Paxos、2PC

**拜占庭故障（Byzantine Fault）**：
- 节点可能发送任意错误消息（恶意行为或严重bug）
- 通信链路也可能被篡改
- 需要更复杂的共识协议（BFT类）
- 代表系统：PBFT、Tendermint、LibraBFT

分布式数据库几乎都假设非拜占庭故障模型，原因有三：
1. 拜占庭容错需要至少3f+1个节点容忍f个故障节点，而非拜占庭只需要f+1个
2. 拜占庭容错协议的消息复杂度是O(n²)甚至更高，性能代价巨大
3. 数据中心内部的服务器受物理安全保护，恶意行为的可能性很低

### 两大故障模型的对比

┌──────────────┬──────────────────────┬───────────────────────────┐
│ 特征          │ 非拜占庭故障           │ 拜占庭故障                 │
├──────────────┼──────────────────────┼───────────────────────────┤
│ 节点行为      │ 正常工作或完全停止      │ 可能发送任意错误消息         │
│ 容忍阈值      │ f个故障需2f+1节点      │ f个故障需3f+1节点           │
│ 消息复杂度    │ O(n)                 │ O(n²)                     │
│ 典型系统      │ TiDB, Spanner        │ Tendermint, Zilliqa        │
│ 应用场景      │ 企业级数据库           │ 区块链、跨域不可信环境       │
└──────────────┴──────────────────────┴───────────────────────────┘

## 数据分布模型：分片与复制

分布式数据库的数据分布由两个正交维度决定：**分片（Sharding/Partitioning）** 决定数据如何切分到不同节点，**复制（Replication）** 决定每个分片有多少副本。这两个维度的组合构成了分布式数据库的存储骨架。

### 分片策略

分片是将一个逻辑数据集合分割成多个物理子集的过程。主流分片策略有三种：

**范围分片（Range Partitioning）**：按照键值的范围将数据分配到不同节点。例如，用户ID 1-1000000在节点A，1000001-2000000在节点B。

优势：
  ✓ 范围查询高效（只需要扫描连续的分片）
  ✓ 数据分布直观，易于理解和管理
  ✓ 天然支持有序遍历

劣势：
  ✗ 容易产生热点（新注册用户集中在最后一个分片）
  ✗ 范围边界需要精心设计，否则负载不均衡
  ✗ 节点增减时可能需要大量数据迁移

典型系统：Google Bigtable, HBase, CockroachDB

**哈希分片（Hash Partitioning）**：对键值计算哈希函数，将数据均匀分配到多个分片。

优势：
  ✓ 数据分布均匀，几乎不可能出现热点
  ✓ 节点增减时只需迁移部分数据（约1/N）
  ✓ 实现简单，确定性路由

劣势：
  ✗ 范围查询需要扫描所有分片（全分片扫描）
  ✗ 失去了数据的有序性
  ✗ 扩容时需要rehash，可能引起数据大规模迁移（除非使用一致性哈希）

典型系统：Amazon DynamoDB（默认模式）, Cassandra（可选）

**列表分片（List Partitioning）**：按照预定义的分类值将数据分配到不同节点。例如，按地区分片——中国用户在节点A，美国用户在节点B。

优势：
  ✓ 业务语义清晰，数据与节点的对应关系直观
  ✓ 适合有明确分区键的场景（如多租户SaaS）
  ✓ 可以根据业务需求灵活分配

劣势：
  ✗ 各分片负载可能极不均衡
  ✗ 需要预先定义分片规则
  ✗ 扩容时需要人工调整分片规则

典型系统：Cassandra（按region/datacenter分片）, PostgreSQL（声明式分区）

三种分片策略的对比：

| 维度 | 范围分片 | 哈希分片 | 列表分片 |
|------|---------|---------|---------|
| 数据均匀度 | 依赖分片设计 | 天然均匀 | 依赖业务分布 |
| 范围查询 | 高效 | 需全分片扫描 | 需全分片扫描 |
| 点查询 | O(1)分片定位 | O(1)分片定位 | O(1)分片定位 |
| 热点风险 | 较高 | 极低 | 中等 |
| 扩容成本 | 高（大量迁移） | 中（rehash） | 低（人工调整） |
| 适用场景 | 时序数据、日志 | 通用场景 | 多租户、地域分区 |

### 复制策略

复制是为每个数据分片创建多个副本的过程，目的是提高可用性和读取性能。复制策略按同步程度分为：

**同步复制（Synchronous Replication）**：写入操作必须等待所有（或多数）副本确认后才返回成功。

写入流程：
  客户端 → 主节点写入 → 同步发送到副本1 → 同步发送到副本2
         ← 副本1确认  ← 副本2确认
  → 返回客户端"写入成功"

优势：强一致性，无数据丢失风险
劣势：延迟高（取决于最慢的副本），可用性低（任一副本故障都会影响写入）

典型系统：Spanner（默认）、PostgreSQL同步流复制

**半同步复制（Semi-Synchronous Replication）**：写入等待至少一个副本确认后返回，其他副本异步同步。

写入流程：
  客户端 → 主节点写入 → 同步发送到副本1 → 异步发送到副本2, 3...
         ← 副本1确认
  → 返回客户端"写入成功"
  → 后台异步同步到副本2, 3...

优势：在一致性和性能之间取得平衡
劣势：如果主节点故障，未同步的副本可能丢失数据

典型系统：MySQL半同步复制、TiDB的Follower Read

**异步复制（Asynchronous Replication）**：写入本地后立即返回成功，后台异步复制到副本。

写入流程：
  客户端 → 主节点写入 → 立即返回"写入成功"
         → 后台异步复制到副本1, 2, 3...

优势：低延迟、高可用性
劣势：主节点故障时可能丢失少量未同步的数据

典型系统：Cassandra（默认ONE级别）、DynamoDB、MySQL异步复制

### 副本放置策略

副本不仅要复制，还要决定"放在哪里"。放置策略直接影响容灾能力：

副本放置策略层级：

1. 同机架（Same Rack）
   副本都在同一机架的服务器上
   → 不能防范机架级故障（交换机故障、电源故障）
   → 适合开发测试环境

2. 同数据中心跨机架（Same DC, Different Racks）
   副本分散在不同机架上
   → 可以防范单机架故障
   → 适合一般生产环境

3. 跨数据中心（Cross-DC / Geo-Replicated）
   副本分散在不同地理位置的数据中心
   → 可以防范数据中心级故障
   → 适合金融、全球业务等高可用要求场景
   → 代价：跨DC延迟通常在10-200ms

Google Spanner的做法：
  → 每个分片有3个副本
  → 副本分布在不同的Zone（可用区）
  → 使用Paxos协议在副本间同步
  → 通过TrueTime保证跨Zone的全局一致性

## 分布式事务的基本模型

事务是数据库区别于文件系统的核心能力。在分布式数据库中，一个事务可能涉及多个节点上的数据，这就引入了分布式事务的概念。

### 事务的ACID在分布式环境下的挑战

| ACID属性 | 单机数据库 | 分布式数据库的额外挑战 |
|---------|-----------|---------------------|
| 原子性（Atomicity） | 本地WAL保证 | 跨节点的原子提交需要协调协议（2PC/Paxos） |
| 一致性（Consistency） | 约束检查在单节点完成 | 跨分片约束检查代价高昂，可能需要分布式锁 |
| 隔离性（Isolation） | 本地锁/MVCC | 跨节点的隔离需要分布式锁或MVCC协议 |
| 持久性（Durability） | 本地WAL+磁盘刷写 | 多数副本持久化后才算持久（Quorum Write） |

### 两阶段提交（2PC）：分布式事务的经典协议

两阶段提交（Two-Phase Commit, 2PC）是最经典的分布式事务协议，由Jim Gray在1978年提出。

**核心角色**：
- **协调者（Coordinator）**：负责发起和管理事务提交
- **参与者（Participant）**：持有数据的节点，执行本地事务

**协议流程**：

阶段一：准备（Prepare）
  协调者 → 发送 PREPARE 给所有参与者
  参与者 → 执行本地事务，写入redo/undo日志
  参与者 → 回复 VOTE_COMMIT 或 VOTE_ABORT 给协调者

阶段二：提交（Commit）
  如果所有参与者都回复 VOTE_COMMIT：
    协调者 → 发送 GLOBAL_COMMIT 给所有参与者
    参与者 → 提交本地事务，释放锁
  如果任一参与者回复 VOTE_ABORT 或超时：
    协调者 → 发送 GLOBAL_ROLLBACK 给所有参与者
    参与者 → 回滚本地事务，释放锁

**2PC的致命缺陷**：

1. **阻塞问题**：在阶段一和阶段二之间，参与者持有的锁不会释放。如果协调者崩溃，参与者将无限期等待，形成阻塞。这在高并发场景下可能导致大量事务堆积。

2. **单点故障**：协调者是单点。虽然可以通过备份协调者恢复，但在恢复期间所有参与者都处于阻塞状态。

3. **性能开销**：每个事务需要两轮网络往返（Prepare + Commit），在跨数据中心场景下延迟可能翻倍。

### 三阶段提交（3PC）与Paxos提交

三阶段提交（3PC）通过增加一个预提交阶段来减少阻塞，但在网络分区时仍可能出现不一致。因此，现代分布式数据库更倾向于使用基于Paxos/Raft的提交协议：

基于Raft的分布式事务提交（TiDB模式）：

1. 事务在协调者节点开始
2. 协调者将事务的写操作组织为一个Raft日志条目
3. 通过Raft协议将日志复制到多数节点
4. 日志被提交（Commit）后，事务原子生效
5. 如果协调者故障，Raft会选举新领导者继续处理

优势：
  ✓ 无阻塞（Raft保证日志最终会被提交）
  ✓ 高可用（领导者故障时自动切换）
  ✓ 与数据复制共享同一套协议，减少复杂度

## 分布式数据库的关键性能指标

评估一个分布式数据库的性能，需要关注以下几个核心指标，它们之间往往存在此消彼长的权衡关系。

### 核心指标体系

| 指标 | 含义 | 典型范围 | 影响因素 |
|------|------|---------|---------|
| 读延迟（Read Latency） | 从发送读请求到收到响应的时间 | P50: 1-10ms, P99: 10-100ms | 副本数量、一致性级别、网络延迟 |
| 写延迟（Write Latency） | 从发送写请求到收到确认的时间 | P50: 5-50ms, P99: 50-200ms | 同步副本数、事务大小、锁竞争 |
| 读吞吐量（Read Throughput） | 单位时间内处理的读请求数 | 10K-1M QPS/集群 | 节点数、分片策略、缓存命中率 |
| 写吞吐量（Write Throughput） | 单位时间内处理的写请求数 | 5K-500K QPS/集群 | 副本同步开销、WAL性能、网络带宽 |
| 可用性（Availability） | 系统正常服务时间占比 | 99.9%-99.999% | 副本策略、故障检测速度、自动恢复时间 |
| 持久性（Durability） | 数据不丢失的保证级别 | 0数据丢失-允许少量丢失 | 同步副本数、WAL刷写策略 |

### 指标间的权衡关系

这些指标不是孤立的，它们之间存在本质性的权衡：

一致性 ←────────────────→ 延迟/吞吐量
   ↑                           ↑
   │                           │
   │   越强的一致性保证           │   越高的延迟和越低的吞吐量
   │   需要越多的同步操作         │   因为需要等待多数节点确认
   │                           │
   └───────────────────────────┘

具体例子：
  强一致性写入（QUORUM=ALL, 3副本）：
    写延迟 ≈ 本地写入延迟 + max(到所有副本的网络延迟)
    → 如果跨数据中心，延迟可能达到 100ms+

  最终一致性写入（ONE级别, 3副本）：
    写延迟 ≈ 本地写入延迟（微秒级）
    → 副本在后台异步同步

  读取的权衡：
    从主节点读取 → 保证读到最新数据，但主节点负载高
    从Follower读取 → 减轻主节点负载，但可能读到旧数据

### SLA（Service Level Agreement）设计

在实际生产中，性能指标需要与SLA对齐。SLA设计的关键是选择合适的百分位指标：

P50（中位数）：50%的请求在此延迟内完成
  → 适合评估日常用户体验
  → 示例：P50读延迟 = 3ms

P99（第99百分位）：99%的请求在此延迟内完成
  → 适合评估长尾延迟
  → 示例：P99读延迟 = 50ms

P999（第99.9百分位）：99.9%的请求在此延迟内完成
  → 适合金融级SLA
  → 示例：P999读延迟 = 200ms

为什么不用平均值？
  → 平均值会掩盖异常。如果1%的请求延迟10秒，
    平均值可能只有12ms（看起来很好），
    但那1%的用户体验极差。

## 分布式数据库的分类体系

为了帮助读者建立系统认知，下面从四个维度对主流分布式数据库进行分类。

### 按数据模型分类

分布式数据库
├── 关系型（NewSQL）
│   核心特征：完整SQL支持 + ACID事务 + 水平扩展
│   代表系统：TiDB, CockroachDB, YugabyteDB, OceanBase
│   适用场景：OLTP业务，需要强一致性和复杂查询
│
├── 键值型（Key-Value）
│   核心特征：简单的Key-Value接口，极高的读写性能
│   代表系统：DynamoDB, Redis Cluster, etcd, TiKV
│   适用场景：会话存储、缓存、配置管理
│
├── 文档型（Document）
│   核心特征：存储半结构化文档（JSON/BSON），Schema灵活
│   代表系统：MongoDB, CouchDB, Amazon DocumentDB
│   适用场景：内容管理、用户画像、日志存储
│
├── 列族型（Column-Family）
│   核心特征：宽列存储，按列族组织数据，适合稀疏数据
│   代表系统：Apache Cassandra, HBase, Google Bigtable
│   适用场景：时序数据、IoT数据、大规模分析
│
└── 图数据库（Graph）
    核心特征：以节点和边的方式存储关系，支持图遍历查询
    代表系统：Neo4j（Fabric分布式模式）, JanusGraph, Nebula Graph
    适用场景：社交网络、推荐系统、知识图谱

### 按一致性保证分类

| 类型 | 保证 | 代价 | 代表系统 | 适用场景 |
|------|------|------|---------|---------|
| 强一致（CP） | 任何读操作返回最新写入的值 | 写入延迟高，分区时部分不可用 | Spanner, TiDB, CockroachDB | 金融交易、库存管理 |
| 最终一致（AP） | 如果无新写入，所有副本最终一致 | 可能读到旧数据 | Cassandra, DynamoDB, Riak | 社交Feed、日志、IoT |
| 可调一致性 | 应用按需选择一致性级别 | 灵活性带来设计复杂度 | Cassandra（ONE/QUORUM/ALL） | 需要灵活权衡的场景 |

### 按部署拓扑分类

单主（Single-Leader）：
  所有写入通过单一主节点处理，从节点只负责读取
  
  客户端 → 主节点（读写）
              ↓ 异步/同步复制
           从节点1（只读）  从节点2（只读）
  
  优点：一致性模型简单，实现成熟
  缺点：主节点是写入瓶颈和单点故障
  
  代表：MySQL主从复制、PostgreSQL流复制


多主（Multi-Leader）：
  多个节点都可以接受写入，节点间互相复制
  
  客户端A → 主节点1（读写）←→ 主节点2（读写）← 客户端B
                  ↕                    ↕
              从节点1                从节点2
  
  优点：写入可用性高，支持就近写入
  缺点：需要处理写入冲突（Conflict Resolution）
  
  代表：Galera Cluster、CouchDB、Oracle GoldenGate


无主（Leaderless）：
  没有固定主节点，任何节点都可以处理读写
  
  客户端 → 节点1（读写）
  客户端 → 节点2（读写）
  客户端 → 节点3（读写）
           节点间互相复制
  
  优点：无单点故障，高可用性
  缺点：一致性保证较弱，需要Quorum机制
  
  代表：Cassandra、DynamoDB、Riak、Voldemort

## 分布式系统的网络基础

理解分布式数据库，必须先理解它所依赖的网络环境。很多分布式系统的设计决策，本质上是对网络不可靠性的回应。

### 网络的基本特性

**延迟的构成**：

一次网络请求的总延迟 = 传输延迟 + 传播延迟 + 排队延迟 + 处理延迟

传输延迟（Transmission Delay）：
  数据从网卡发送出去的时间
  = 数据大小 / 网络带宽
  → 发送1KB数据到1Gbps网络：约0.008ms

传播延迟（Propagation Delay）：
  信号在物理介质中传播的时间
  = 距离 / 光速
  → 同机房内：0.1-0.5ms
  → 同城跨区：1-3ms
  → 跨国（北京→纽约）：60-80ms（单程）

排队延迟（Queuing Delay）：
  数据在网络设备（交换机、路由器）队列中等待的时间
  → 正常情况：< 0.1ms
  → 网络拥塞时：可能达到数十毫秒

处理延迟（Processing Delay）：
  目标节点处理请求的时间
  → 取决于应用逻辑复杂度

**延迟的实际测量**：

典型网络延迟（RTT，往返时间）：
┌─────────────────────────┬──────────────┐
│ 场景                     │ RTT          │
├─────────────────────────┼──────────────┤
│ 同机架内                 │ 0.1-0.3ms   │
│ 同数据中心不同机架         │ 0.3-1ms     │
│ 同城不同数据中心           │ 1-5ms       │
│ 同国不同城市               │ 5-20ms      │
│ 跨国（亚太↔北美）          │ 120-200ms   │
│ 跨国（亚太↔欧洲）          │ 150-250ms   │
└─────────────────────────┴──────────────┘

对分布式数据库设计的影响：
  同机房部署：可以使用同步复制，延迟代价小
  跨城部署：同步复制的延迟代价显著，需要权衡
  跨国部署：同步复制几乎不可行，必须使用异步复制

### 网络分区与脑裂

**网络分区（Network Partition）**：网络故障导致部分节点之间完全无法通信，但各自内部仍然正常。

正常状态：
  节点A ←──网络──→ 节点B ←──网络──→ 节点C
  
网络分区发生：
  节点A ←──X──→ 节点B ←──X──→ 节点C
  
  A和B无法通信，B和C无法通信
  但A和C可能通过其他路径通信，也可能不能

**脑裂（Split-Brain）**：网络分区时，两个分区各自选出"领导者"，同时接受写入，导致数据不一致。

脑裂场景：
  分区1：节点A（认为自己是主） → 接受写入 x=1
  分区2：节点B（认为自己是主） → 接受写入 x=2
  
  分区恢复后：x的值应该是什么？1还是2？
  
  解决方案：
  1. Quorum投票：多数分区的决定生效
  2. Fencing Token：使用递增的epoch/term号拒绝旧领导者的请求
  3. STONITH（Shoot The Other Node In The Head）：强制关闭旧领导者

## CAP与PACELC：设计权衡的理论框架

CAP定理和PACELC定理是理解分布式数据库设计选择的理论基础。虽然它们在后续章节中会详细展开，但作为核心概念必须在本节给出清晰的框架性介绍。

### CAP定理（快速回顾）

Eric Brewer在2000年提出CAP猜想，2002年Gilbert和Lynch给出形式化证明。核心结论：**在网络分区发生时，系统不可能同时保证一致性和可用性**。

| 选择 | 含义 | 代表系统 |
|------|------|---------|
| CP | 分区时牺牲可用性，保证一致性 | Spanner, TiDB, ZooKeeper |
| AP | 分区时牺牲一致性，保证可用性 | Cassandra, DynamoDB, CouchDB |
| CA | 不容忍分区（仅在单机或无网络分区时成立） | 单机MySQL, PostgreSQL |

关键误解澄清：CAP不是"三选二"。在分布式系统中，P（分区容忍性）是必须的——你无法假设网络永远不出故障。真正的选择是在C和A之间做权衡。

### PACELC定理（超越CAP）

PACELC定理（Daniel Abadi, 2012）弥补了CAP只关注"分区时"行为的不足：

if (Partition) {
    选择 Availability 还是 Consistency    // CAP覆盖的部分
} else {
    选择 Latency 还是 Consistency         // CAP未覆盖的部分
}

即使在没有分区的正常运行时，分布式系统仍然需要在延迟和一致性之间做出权衡。这是因为强一致性需要同步等待多数副本确认，而同步等待会增加延迟。

| 分类 | 分区时选择 | 正常时选择 | 代表系统 |
|------|-----------|-----------|---------|
| PA/EL | 优先可用性 | 优先低延迟 | Cassandra, DynamoDB |
| PC/EC | 优先一致性 | 优先一致性 | Spanner, TiDB, CockroachDB |

## 分布式数据库中的核心术语表

以下是分布式数据库领域的核心术语，后续章节将频繁使用：

| 术语 | 英文 | 定义 | 示例 |
|------|------|------|------|
| 分片 | Shard/Partition | 数据的一个子集，存储在特定节点上 | 用户表按user_id分片为4个分片 |
| 副本 | Replica | 分片的一个完整拷贝 | 每个分片有3个副本分布在不同机架 |
| 副本因子 | Replication Factor | 每个分片的副本数量 | RF=3表示每份数据存3份 |
| 主副本 | Leader/Primary | 负责处理写入的副本 | Raft中的Leader节点 |
| 从副本 | Follower/Secondary | 复制主副本数据的副本 | Raft中的Follower节点 |
| 仲裁 | Quorum | 达成共识所需的最少节点数 | 3节点集群中Quorum=2 |
| 分区键 | Partition Key | 决定数据路由到哪个分片的键 | user_id作为分区键 |
| 一致性哈希 | Consistent Hashing | 节点增减时只需迁移少量数据的哈希策略 | Dynamo、Cassandra使用 |
| 共识算法 | Consensus Algorithm | 分布式节点就某个值达成一致的算法 | Raft、Paxos、ZAB |
| WAL | Write-Ahead Log | 先写日志再修改数据的持久化策略 | 所有关系型数据库都使用 |
| MVCC | Multi-Version Concurrency Control | 多版本并发控制，通过数据快照实现无锁读取 | PostgreSQL、TiDB |
| 两阶段提交 | 2PC | 分布式事务的经典原子提交协议 | 跨分片事务的协调 |
| 故障转移 | Failover | 主节点故障时自动切换到备用节点 | Raft领导者选举 |
| 再平衡 | Rebalance | 节点增减后重新分配数据分片 | TiDB的Region调度 |
| 暗裂 | Split-Brain | 网络分区导致多个节点同时认为自己是主 | 通过Quorum解决 |
| 端到端延迟 | End-to-End Latency | 从客户端发出请求到收到完整响应的时间 | 包含网络+处理+序列化延迟 |

## 本节核心要点总结

| 概念 | 核心结论 | 工程意义 |
|------|---------|---------|
| 分布式数据库定义 | 多节点协作、数据分散、逻辑统一 | 区分真正的分布式与简单的主从复制 |
| FLP不可能性 | 确定性共识在异步系统中不可能 | 实际系统通过随机化/部分同步绕过限制 |
| 拜占庭vs非拜占庭 | 分布式数据库通常假设非拜占庭故障 | 容忍阈值从3f+1降为2f+1 |
| 分片策略 | 范围/哈希/列表各有优劣 | 根据查询模式选择合适的分片策略 |
| 复制策略 | 同步/半同步/异步权衡一致性与延迟 | 根据业务SLA选择合适的复制级别 |
| 分布式事务 | 2PC有阻塞问题，Raft提交更优 | 优先选择基于共识的事务协议 |
| 网络延迟 | 跨数据中心延迟是设计的关键约束 | 部署拓扑直接决定一致性级别选择 |
| CAP/PACELC | 分区时C-A权衡，正常时L-C权衡 | 设计决策需要同时考虑故障时和正常时行为 |
