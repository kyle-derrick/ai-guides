## 故障检测算法

故障检测是分布式系统容错机制的第一道防线。如果系统无法及时、准确地识别故障节点，后续的故障转移、数据恢复、服务降级都无从谈起。本节系统讲解故障检测的核心算法、工程实现与常见陷阱，帮助读者建立从原理到落地的完整认知。

---

### 1. 为什么需要故障检测

在单机系统中，进程崩溃操作系统会立即回收资源，上层几乎无感知。但在分布式系统中，节点之间的故障信息必须通过网络传递——而网络本身是不可靠的。一个节点可能因为硬件故障宕机、因为 GC 暂停假死、因为网络分区而"失联"，甚至因为时钟漂移导致日志时间不一致。

故障检测的核心挑战在于：**如何在有限时间内，以可控的误判率，区分"真的坏了"和"只是暂时慢了"**。

误判的代价极高：
- **漏检（False Negative）**：故障节点未被识别，请求继续发送导致超时、数据不一致、级联故障
- **误判（False Positive）**：健康节点被误判为故障，触发不必要的故障转移，造成资源浪费和系统抖动

| 误判类型 | 英文术语 | 典型后果 | 工程影响 |
|---------|---------|---------|---------|
| 漏检 | False Negative | 请求超时、数据丢失、级联故障 | 故障扩散、雪崩效应 |
| 误判 | False Positive | 不必要的故障转移、资源争抢 | 脑裂、抖动、服务中断 |
| 延迟误判 | Late Detection | 故障已扩散才发现 | 恢复窗口缩小、RTO增大 |

### 2. 故障分类：检测之前先理解故障

不同的故障类型需要不同的检测策略：

**按故障持久性分类：**

| 类型 | 特征 | 检测难度 | 示例 |
|------|------|---------|------|
| 崩溃故障（Crash） | 节点完全停止响应 | 低 | 进程被 kill、硬件宕机 |
| 遗漏故障（Omission） | 部分消息丢失或延迟 | 中 | 网络丢包、队列溢出 |
| 时序故障（Timing） | 响应超出时间约束 | 中 | GC 暂停、磁盘 I/O 飙升 |
| 拜占庭故障（Byzantine） | 节点行为不可预测 | 高 | 软件 bug 输出错误数据 |
| 间歇故障（Intermittent） | 时好时坏 | 高 | 内存翻转、网线松动 |

**按故障范围分类：**
- **单点故障**：单个节点失效，可通过冗余消除
- **网络分区**：节点间通信中断，集群分裂为多个独立子集
- **级联故障**：一个节点故障导致其他节点过载，引发连锁反应

### 3. 核心检测算法

#### 3.1 超时检测（Timeout-based Detection）

最简单也是最基础的检测方式：设定一个超时阈值 T，如果在 T 时间内没有收到目标节点的心跳响应，就判定该节点故障。

检测逻辑：
  1. 每隔 interval 秒向目标节点发送心跳
  2. 启动计时器，等待响应
  3. 若 response_time > T，标记为疑似故障
  4. 连续 miss_count 次超时后，确认故障

**超时阈值的选择**是核心难点。阈值太短，网络波动导致误判；阈值太长，故障发现延迟大。

经验公式：

T = α × RTT_avg + β × RTT_stddev

其中 α 控制灵敏度（通常取 2~5），β 控制对抖动的容忍度（通常取 3~6）。

**局限性：**
- 静态阈值无法适应网络环境变化
- 无法检测"活着但已不可用"的灰度故障（如数据不一致、响应错误）
- 在高延迟网络（跨地域部署）中效果差

#### 3.2 心跳检测（Heartbeat Detection）

心跳检测是超时检测的工程化增强。分为两种模式：

**主动心跳（Active Heartbeat）：**
监控方定期向被监控方发送心跳请求，等待响应。

监控者 ──heartbeat──> 被监控者
监控者 <──ack──────── 被监控者

**被动心跳（Passive Heartbeat）：**
被监控方定期向监控方推送心跳信号，监控方判断是否按时收到。

被监控者 ──heartbeat──> 监控者

| 对比维度 | 主动心跳 | 被动心跳 |
|---------|---------|---------|
| 网络开销 | 额外请求流量 | 额外推送流量 |
| 检测延迟 | 至少 1 个 interval | 可小于 1 个 interval |
| 故障范围感知 | 仅知道目标不可达 | 可携带部分状态信息 |
| 复杂度 | 简单 | 需要推送机制 |
| 适用场景 | 一对多监控 | 多对一上报 |

**心跳风暴问题：** 当监控节点数量 N 增大时，心跳流量为 O(N²)（全互联场景）。解决方案包括树状心跳拓扑（减少为 O(N)）和随机化心跳间隔（避免同步风暴）。

#### 3.3 Phi Accrual 探测器（自适应故障检测）

由 Hayashibara 等人于 2004 年提出，被 Akka、Cassandra 等广泛采用。核心思想：不输出二元的"故障/健康"判断，而是输出一个 **怀疑度（Phi 值）**，由上层应用决定阈值。

**原理：** 基于最近 N 次心跳间隔的统计分布，计算下一次心跳未到达的概率。

Phi = -log10(1 - P(heartbeat_interval))

其中 P 是基于历史心跳间隔拟合的累积分布函数（通常用正态分布近似）。

**优势：**
- 自适应网络延迟波动：网络正常时阈值自动收紧，网络抖动时自动放松
- 无硬阈值：不同服务可设置不同 phi 阈值
- 平滑过渡：从"完全健康"到"完全故障"之间有连续的怀疑度值

**工程实现要点：**
- 滑动窗口大小决定自适应速度：窗口太大反应迟钝，太小容易被噪声干扰
- 需要足够的历史数据积累：冷启动阶段检测精度较低
- 推荐参数：phi 阈值 8~12（Akka 默认值 8），窗口 100~1000 次心跳

#### 3.4 Gossip 协议故障检测

Gossip（流行病传播）协议是一种去中心化的故障检测机制，每个节点都参与检测和传播故障信息。

**工作流程：**

每个 tick:
  1. 随机选择 K 个邻居
  2. 交换彼此维护的故障信息表（failure detector table）
  3. 如果某节点的故障计数超过阈值，标记为故障
  4. 故障信息像病毒一样在网络中传播，最终全网收敛

**关键参数：**

| 参数 | 含义 | 典型值 | 影响 |
|------|------|--------|------|
| Fanout (K) | 每轮传播的邻居数 | 3~5 | 越大收敛越快，开销越大 |
| 判定阈值 | 故障计数达到多少判定故障 | N/2 + 1 | 越大越保守 |
| 衰减因子 | 故障计数随时间衰减 | 0.9~0.99 | 处理间歇故障 |

**代表实现：** Amazon Dynamo、Cassandra、Consul 的 Serf。

**收敛时间分析：** 在 N 个节点、每轮 fanout=K 的网络中，故障信息传播到全网的时间为 O(log(N)/log(K))。例如 1000 个节点、K=5 时，约 5 轮即可覆盖全网。

#### 3.5 基于分布式共识的故障检测

在强一致性系统中，故障检测通常与共识协议结合：

**Raft Leader Election：**
follower 超时 → 发起选举 → 获得多数票 → 成为 leader
                  ↑
          心跳未收到触发

- Election Timeout 通常设为 150~300ms 随机值
- 通过随机化避免活锁（多个节点同时选举）
- 只能检测 leader 是否存活，不能检测 follower 是否异常

**Paxos FLP 定理的限制：**
FLP 不可能定理证明：在一个异步系统中，即使只有一个进程可能崩溃，也不存在一个确定性的共识算法能保证在有限时间内终止。所有实用共识算法都通过引入随机性或部分同步假设来绕过这个限制。

#### 3.6 SWIM 协议（Scalable Weakly-consistent Infection-style Process Group Membership）

SWIM 是一种结合探测和 gossip 的成员组故障检测协议，被 Consul、membership 等系统采用。

**三阶段流程：**

1. 探测阶段（Probe）
   随机选择 1 个目标，发送 ping
   → 如果目标存活，收到 ack
   → 如果目标不回应，进入间接探测

2. 间接探测阶段（Indirect Probe）
   邀请 K 个随机节点向目标发送 ping
   → 任意一个收到 ack，目标判定为存活
   → 全部超时，目标进入怀疑状态

3. 传播阶段（Gossip）
   将检测结果（怀疑/确认故障）通过 gossip 传播
   → 最终全网达成一致

**SWIM 相比传统心跳的优势：**
- 探测开销恒定为 O(1) 每轮（随机选 1 个目标探测）
- 间接探测提高了准确性（减少了因网络分区导致的误判）
- Gossip 传播使故障信息收敛速度快

### 4. 算法对比与选型

| 算法 | 检测延迟 | 误判率 | 网络开销 | 可扩展性 | 适用场景 |
|------|---------|--------|---------|---------|---------|
| 超时检测 | 取决于阈值 | 高 | O(1) | 低 | 小规模、简单场景 |
| 心跳检测 | 固定 interval | 中 | O(N) | 中 | 中小规模集群 |
| Phi Accrual | 自适应 | 低 | O(1) | 中 | 需要精细控制的场景 |
| Gossip | O(log N) 收敛 | 低 | O(K×N) | 高 | 大规模去中心化系统 |
| 共识检测 | 需多数确认 | 极低 | O(N²) | 低 | 强一致性系统 |
| SWIM | O(1) 探测 + 传播 | 低 | O(N) | 高 | 大规模成员组管理 |

**选型决策树：**

节点数量 > 100？
├── 是 → 需要去中心化？
│   ├── 是 → Gossip / SWIM
│   └── 否 → 集中式监控 + Phi Accrual
└── 否 → 需要强一致性？
    ├── 是 → Raft/Paxos 共识检测
    └── 否 → 心跳检测 + 超时

### 5. 工程实践中的关键问题

#### 5.1 脑裂问题

当网络分区发生时，不同分区可能各自选出 leader，形成"脑裂"。

**解决方案：**
- **Fencing Token**：每次选举生成递增 token，旧 leader 的请求被拒绝
- **Quorum 机制**：写操作必须获得多数节点确认
- **STONITH（Shoot The Other Node In The Head）**：通过外部仲裁（如 IPMI、云 API）强制关闭可能的旧节点

#### 5.2 检测与恢复的解耦

故障检测和故障恢复是两个独立的关注点。检测模块只负责发现故障并通知，不应直接执行恢复动作。

故障检测器 ──事件──> 事件总线 ──订阅──> 故障恢复处理器
                   ──订阅──> 告警系统
                   ──订阅──> 流量调度器

这样设计的好处：
- 检测策略和恢复策略可以独立演进
- 多个恢复策略可以同时响应同一个故障事件
- 便于测试：mock 检测器输出即可测试恢复逻辑

#### 5.3 检测粒度选择

| 粒度 | 检测对象 | 适用场景 | 代表系统 |
|------|---------|---------|---------|
| 进程级 | 单个进程是否存活 | 微服务架构 | Kubernetes Liveness Probe |
| 节点级 | 整台机器是否可达 | 有状态服务 | Raft / Paxos |
| 服务级 | 服务整体是否可用 | 流量入口 | Envoy health check |
| 数据级 | 数据副本是否一致 | 存储系统 | Cassandra Read Repair |

#### 5.4 时钟问题

分布式系统中的"超时"依赖时钟，但时钟不可靠：

- **时钟漂移**：不同机器的时钟以不同速率运行，可能差数百毫秒甚至数秒
- **时钟跳变**：NTP 同步可能导致时钟突然前跳或后跳

**应对策略：**
- 使用单调时钟（monotonic clock）而非墙上时钟（wall clock）测量间隔
- 超时值设置要预留足够余量（通常 ≥ 3 倍 RTT 标准差）
- 关键路径上避免依赖绝对时间，用逻辑时钟或向量时钟替代

### 6. 经典实现案例

#### 6.1 ZooKeeper 的故障检测

ZooKeeper 使用 Session Timeout 机制：客户端每 1/3 timeout 周期发送心跳。如果服务端在 1 个 timeout 周期内未收到心跳，判定客户端会话过期。

Client Session Timeout = 30s
Client 心跳间隔 = 10s (timeout/3)
Server 判定超时 = 30s

时间线：
0s     10s    20s    30s    40s
|------|------|------|------|
  hb     hb    hb   (未收到) → 会话过期

#### 6.2 Kubernetes 的 Probe 机制

Kubernetes 提供三种探针，分别检测不同层面的健康状态：

| 探针类型 | 检测目标 | 默认间隔 | 默认超时 | 失败动作 |
|---------|---------|---------|---------|---------|
| livenessProbe | 进程是否存活 | 10s | 1s | 重启容器 |
| readinessProbe | 是否可接受流量 | 10s | 1s | 从 Service 移除 |
| startupProbe | 启动是否完成 | 10s | 1s | 暂停其他探针 |

关键配置建议：
- startupProbe 的 failureThreshold 要足够大，覆盖慢启动场景
- livenessProbe 和 readinessProbe 的 timeoutSeconds 不要设太小，避免瞬时网络抖动导致误判
- 对于 GC 暂停较长的服务，适当增大 periodSeconds

#### 6.3 Consul 的 Serf/SWIM 实现

Consul 使用 Serf（基于 SWIM 协议）进行成员管理：
- 每个 agent 每 200ms 随机选择一个成员进行 ping
- 超时 500ms 触发间接探测（邀请 3 个 peer 协助）
- 总判定时间约 2~3 秒
- 故障信息通过 gossip 传播，5 秒内全集群收敛

### 7. 常见陷阱与最佳实践

#### 陷阱一：超时值一刀切

不同服务的响应时间特征不同，用同一个超时阈值检测所有服务会导致部分服务频繁误判或长期漏检。

**正确做法：** 根据每个服务的历史 RTT 分布动态计算超时值，或使用 Phi Accrual 等自适应算法。

#### 陷阱二：只检测不告警

检测到故障后如果不及时通知运维人员，故障会持续扩散。

**正确做法：** 故障检测事件应同时触发三个动作：自动恢复、告警通知、日志记录。

#### 陷阱三：过度依赖单点检测器

如果监控节点本身故障，所有被它监控的节点都会被误判为故障（级联误判）。

**正确做法：** 采用冗余监控（多主检测）、去中心化检测（Gossip/SWIM），确保检测器自身的高可用。

#### 陷阱四：忽略灰度故障

节点没有完全宕机，但响应变慢、返回错误、数据不一致——这些"灰度故障"是生产环境中最常见也最难检测的。

**正确做法：** 除心跳检测外，增加应用层健康检查（如返回特定接口的响应码和延迟是否在阈值内）。

### 8. 进阶：基于机器学习的故障预测

传统故障检测是"事后检测"——故障已经发生才去发现。前沿方向是"事前预测"——通过监控指标的趋势分析，在故障发生前预警。

**常用方法：**
- **时间序列异常检测**：基于 ARIMA、LSTM 等模型识别指标异常波动
- **因果推断**：构建指标间的因果图，预测一个指标异常可能引发的连锁反应
- **多模态融合**：结合日志（文本）、指标（数值）、追踪（拓扑）等多源数据进行综合判断

**代表性系统：** Google 的 MTS（Monitoring Time Series）、Microsoft 的 Project Natick 预测性维护、Netflix 的 Scryer 预测性容量规划。

这些系统的共同特点：不追求 100% 准确率，而是通过降低误判率来建立运维人员的信任，逐步替代人工告警规则。
