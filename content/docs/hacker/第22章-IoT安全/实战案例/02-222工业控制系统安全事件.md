## 22.2 工业控制系统安全事件

工业控制系统（Industrial Control System, ICS）是国家关键基础设施的"神经中枢"——电力、水务、油气、制造业、交通信号、核电站冷却系统全部依赖它运行。一旦 ICS 被攻破，后果不是数据泄露，而是物理世界的破坏：爆炸、断水、断油、核泄漏。本节通过五个标志性安全事件，从攻击背景、技术细节、防御教训三个维度展开分析，帮助读者建立 ICS 安全的实战认知。

### 22.2.1 ICS 安全基础：攻击面在哪儿

在进入案例之前，先理解 ICS 的架构，否则后续的技术分析会失去锚点。

**Purdue 模型（ISA-95/IEC 62443）**

ICS 网络通常按 Purdue 参考模型分层：

```plaintext
┌─────────────────────────────────────────────┐
│  Level 5: 企业网络（ERP、邮件、互联网）       │
├─────────────────────────────────────────────┤
│  Level 4: 生产调度（MES、排产系统）           │
├─────────────────────────────────────────────┤
│  Level 3: 站点运营（历史数据库、HMI工程站）   │
├─────────────────────────────────────────────┤
│  Level 2: 监控层（HMI、SCADA服务器、工程站）  │
├─────────────────────────────────────────────┤
│  Level 1: 控制层（PLC、RTU、DCS控制器）       │
├─────────────────────────────────────────────┤
│  Level 0: 物理过程（传感器、执行器、阀门）    │
└─────────────────────────────────────────────┘
```

攻击者的目标是穿透层层边界，从 Level 5 一路下沉到 Level 0 操控物理设备。历史上最成功的攻击都实现了这条完整路径。

**IT 与 OT 的关键差异**

| 维度 | IT 网络 | OT 网络（ICS） |
|------|---------|----------------|
| 优先级 | 机密性 > 完整性 > 可用性 | 可用性 > 完整性 > 机密性 |
| 停机容忍 | 分钟级（可计划维护窗口） | 零容忍（停机=生产事故甚至安全事故） |
| 补丁策略 | 定期自动更新 | 极其谨慎，停产窗口才能打补丁 |
| 协议 | TCP/IP、HTTP、TLS | Modbus、OPC、DNP3、IEC 104、EtherNet/IP |
| 设备寿命 | 3-5 年 | 15-30 年 |
| 安全设计假设 | 面向开放网络，需主动防御 | "物理隔离=安全"（已被证伪） |

理解这个差异至关重要——很多 IT 安全人员进入 OT 领域后，习惯性地推动"立即打补丁""全面加密"，却忽略了 OT 环境中一个补丁可能导致整条产线停摆的风险。

**ICS 专用攻击面**

```plaintext
攻击入口                    典型漏洞/弱点
─────────────────────────────────────────────────────
IT-OT 边界                  防火墙配置不当、共享凭证、跳板机
远程维护通道               VPN 未启用 MFA、默认密码、过期凭证
工程站                     Windows XP/7 未打补丁、无 EDR
供应链                     固件后门、恶意更新包、开发环境渗透
物理接口                   USB 口、串口调试、未锁的机柜
协议漏洞                   Modbus 无认证、DNP3 明文、OPC 无加密
无线网络                   工业 WiFi 未加密、蓝牙调试接口
HMI Web 界面               默认凭证、XSS、未授权访问
```

### 22.2.2 案例一：Stuxnet 震网病毒——首个国家级网络武器

**历史背景**

2010年6月，白俄罗斯安全公司 VirusBlokAda 的分析师 Sergey Ulasen 发现了一个异常的 USB 蠕虫。经过 Symantec、Kaspersky 等团队数月逆向，这个蠕虫被确认为人类历史上第一个专门针对工业控制系统的网络武器，代号 Stuxnet。它由美国国家安全局（NSA）和以色列军方联合开发（代号"奥林匹克运动会"行动），目标是破坏伊朗纳坦兹核设施的铀浓缩离心机。

**攻击目标**

- 目标设施：伊朗纳坦兹铀浓缩工厂（Natanz Fuel Enrichment Plant）
- 目标设备：西门子 S7-300/S7-400 系列 PLC，控制 IR-1 型气体离心机
- 目标参数：离心机转速（正常运行约 1,064 Hz，攻击时拉高到 1,410 Hz 再骤降至 2 Hz）
- 攻击窗口：仅当检测到特定变频器配置（频率转换器来自特定两家供应商：Vacon 和 Fararo Paya）时才激活载荷

**传播链（完整 Kill Chain）**

```plaintext
Phase 1: 初始感染（Air-Gap 跨越）
  └─ 受感染的 USB 闪存盘（通过内部人员或供应链引入）
     └─ 利用 Windows Shell LNK 漏洞 (CVE-2010-2568)
        └─ 自动执行，无需用户双击

Phase 2: 本地提权和持久化
  ├─ Win32k.sys 本地提权漏洞 (CVE-2010-3338)
  ├─ Print Spooler 漏洞 (CVE-2010-2729)
  ├─ 任务计划程序漏洞 (CVE-2010-3888)
  └─ 内核级 Rootkit（MrxNet.sys 驱动签名——盗用 Realtek 和 JMicron 的合法数字证书）

Phase 3: 网络传播
  ├─ MS08-067（Windows Server Service 漏洞，与 Conficker 相同）
  ├─ MS10-061（Print Spooler 远程代码执行）
  ├─ 网络共享和弱密码猜测
  └─ WinCC 数据库默认密码（通过西门子硬编码的 WinCC SQL 账户）

Phase 4: 目标识别
  └─ 枚举运行环境，检查：
     ├─ 是否安装 Siemens Step 7 编程软件
     ├─ 是否连接了 S7-315 或 S7-417 PLC
     ├─ 变频器参数是否匹配（6个特征参数同时满足）
     └─ 仅在完全匹配时激活载荷——否则静默潜伏

Phase 5: PLC 代码注入
  ├─ 拦截 OB1（主循环块）和 OB35（定时中断块，100ms 周期）
  ├─ 注入恶意功能块（DB888-DB890），替换正常的转速控制逻辑
  ├─ 在正常操作频率下录制传感器数据并缓存
  └─ 攻击时向 HMI 回放缓存的"正常"数据——操作员看不到异常
```

**关键技术细节**

Stuxnet 的 PLC 代码注入极为精妙。它不是简单地关闭离心机，而是精确操控转速：

```python
# Stuxnet PLC 载荷的工作原理（伪代码还原）
# 基于 Symantec "W32.Stuxnet Dossier" v1.4 (2011)

# 正常运行: 频率转换器输出 1,064 Hz → 离心机正常转速
# 攻击序列:
#   Step 1: 频率拉升至 1,410 Hz（超速 33%） → 持续 15 分钟
#   Step 2: 频率骤降至 2 Hz（近乎停止）     → 持续 50 分钟
#   Step 3: 恢复正常 1,064 Hz               → 持续数周
#   重复 Step 1-3，周期性反复

# 同时，OB35 中断块每 100ms 采样一次正常运行数据
# 攻击期间回放预录制的正常值给监控系统
# 操作员在 HMI 上看到的一直是"一切正常"
```

**为什么 Stuxnet 能跨越物理隔离**

纳坦兹核设施采用了物理隔离（Air-Gap），但 Stuxnet 通过以下方式绕过：

1. **USB 蠕虫机制**：利用 LNK 漏洞实现"插上即感染"，不需要用户执行任何操作
2. **零日漏洞武器库**：同时使用了 4 个 Windows 零日漏洞，确保在各种 Windows 版本上都能成功
3. **合法数字签名**：Rootkit 驱动使用了从 Realtek 和 JMicron 窃取的代码签名证书，绕过驱动签名验证
4. **有限传播 + 精准打击**：设置了传播上限（最多感染 3 台中继主机），避免引起注意；只在检测到目标 PLC 配置时才激活载荷

**实际影响**

- 约 1,000 台 IR-1 离心机被物理损坏（离心机轴承因反复超速/骤停而金属疲劳断裂）
- 伊朗铀浓缩产能下降约 30%
- 伊朗核计划被推迟估计 2-3 年
- 2010年6月被公开分析后，伊朗被迫从纳坦兹移除了约 1,000 台受损离心机

**防御教训**

| 教训 | 具体启示 |
|------|----------|
| 物理隔离不是银弹 | USB 设备、供应链、内部人员都可以跨越 Air-Gap |
| 供应链安全至关重要 | 攻击者盗用了合法驱动签名证书 |
| PLC 程序完整性校验 | 应定期比对 PLC 逻辑的哈希值，检测未授权修改 |
| 最小权限 + 白名单 | USB 端口应禁用或限制为只读；可执行文件白名单 |
| 监控异常行为 | 离心机转速的异常波动应该触发物理层告警 |
| 零日漏洞储备 | 国家级攻击者可以同时使用多个零日漏洞，防御不能依赖单点 |

### 22.2.3 案例二：乌克兰电网攻击——ICS 攻击的实战演进

2015年和2016年乌克兰电网遭受的两次攻击是公开记录中首次导致大面积停电的网络攻击事件，而且第二次攻击展示了 ICS 攻击从"手动操作"到"自动化武器"的质变。

#### 2015年攻击：BlackEnergy 手动攻击

**攻击链完整还原**

```plaintext
时间线: 2015年12月23日

Phase 1: 初始渗透（提前数月）
  ├─ 鱼叉式钓鱼邮件，附件为恶意 Microsoft Office 文档
  ├─ 利用 OLE 对象嵌入 + 宏代码执行
  ├─ 植入 BlackEnergy 3 木马（后门 + 信息窃取）
  └─ 攻击者在 IT 网络中潜伏数月，进行侦察

Phase 2: 横向移动到 OT 网络
  ├─ 利用 IT-OT 共享的 VPN 凭证
  ├─ 访问 SCADA 管理网络
  ├─ 获取 HMI 远程访问权限（通过 VPN + 远程桌面）
  └─ 操作员级别的控制权

Phase 3: 攻击执行（12月23日下午）
  ├─ 通过 HMI 远程打开变电站断路器
  ├─ 同时攻击 3 家电力公司（Prykarpattyaoblenergo、Kyivoblenergo、Chernivtsioblenergo）
  ├─ 覆写串口转以太网转换器固件，切断远程恢复路径
  ├─ 部署 KillDisk 数据擦除工具，破坏 SCADA 服务器和工作站
  └─ 对电力公司呼叫中心发动电话 DoS，阻止用户报修

Phase 4: 恢复受阻
  ├─ 串口转换器固件被覆写，操作员无法远程合闸
  ├─ 必须派人到变电站手动操作物理开关
  └─ 最终在数小时内恢复供电，但影响约 225,000 用户
```

**BlackEnergy 3 的技术特征**

```python
# BlackEnergy 3 核心能力
# 参考: ESET "BlackEnergy &amp; KillDisk" 分析报告 (2016)

capabilities = {
    "initial_access": "Office 文档宏 + OLE 对象执行",
    "persistence": [
        "注册表 Run 键",
        "计划任务",
        "WMI 事件订阅",
        "DLL 劫持（合法白名单程序加载恶意DLL）"
    ],
    "c2_communication": [
        "HTTP/HTTPS 回连",
        "WordPress 站点作为中继",
        "硬编码 C2 域名 + DGA 域名生成"
    ],
    "plugins": [
        "信息窃取插件（系统信息、已安装软件、凭证）",
        "键盘记录插件",
        "屏幕截图插件",
        "网络扫描插件",
        "远程桌面插件"
    ],
    "destruction": "KillDisk 组件——覆写磁盘扇区，MBR 和前 64 个扇区"
}
```

#### 2016年攻击：Industroyer 自动化攻击

**重大升级**

2015年攻击中，攻击者需要手动操作 HMI 来断开断路器。2016年12月17日的攻击发生了质变——攻击者使用了 Industroyer（又名 CrashOverride）恶意软件，这是首个能够直接理解并操控电力系统工业协议的恶意软件。

**Industroyer 的协议攻击模块**

```plaintext
Industroyer 包含 4 个独立的协议攻击模块:

1. IEC 60870-5-101 (IEC 101)
   ├─ 串行通信协议，用于变电站与控制中心
   ├─ Industroyer 实现了完整的 ASDU（应用服务数据单元）操作
   └─ 可发送直接遥控命令（选择-执行序列）

2. IEC 60870-5-104 (IEC 104)
   ├─ IEC 101 的 TCP/IP 版本，广泛用于远程 SCADA
   ├─ 攻击者发送 45 类 ASDU（单点命令）和 46 类 ASDU（步调节命令）
   └─ 操控断路器的开/合状态

3. IEC 61850
   ├─ 变电站自动化通信标准
   ├─ 使用 MMS（制造消息规范）协议
   └─ 操控 GOOSE 报文（通用面向变电站事件对象）快速跳闸

4. OPC Data Access (OPC DA)
   ├─ 工业数据交换标准
   └─ 用于与 SCADA 系统交互

# 每个模块都是可插拔的 DLL
# 攻击者可以针对目标电力公司的具体设备选择对应模块
```

**为什么 Industroyer 如此危险**

- **协议原生支持**：直接实现工业协议栈，不需要借助 HMI 或工程软件
- **绕过安全设备**：传统 IT 安全设备（IDS/IPS）无法解析这些工业协议
- **模块化设计**：可针对不同电力公司的不同设备定制攻击
- **DoS 组件**：包含针对 IEC 104 协议栈的溢出漏洞利用，可以瘫痪串口通信服务器
- **数据擦除组件**：内置 KillDisk 变体，攻击后销毁证据

**2016年攻击结果**

- 基辅部分地区停电约 1 小时
- 电力公司通过手动恢复比 2015 年更快
- 但攻击自动化程度大幅提升，预示着未来攻击的规模化趋势

#### 两次攻击的对比分析

| 维度 | 2015年（BlackEnergy） | 2016年（Industroyer） |
|------|----------------------|----------------------|
| 攻击方式 | 手动操作 HMI | 自动化协议攻击 |
| 攻击速度 | 需要数小时人工操作 | 分钟级自动化执行 |
| 目标范围 | 3 家电力公司 | 1 家电力公司（基辅） |
| 停电规模 | 225,000 用户，数小时 | 基辅部分地区，1 小时 |
| 技术水平 | 利用现有 IT 攻击工具 | 定制 ICS 协议攻击武器 |
| 恢复难度 | 高（固件被覆写） | 中（有备份协议） |
| 演进意义 | 首次 ICS 停电攻击 | 首次自动化 ICS 攻击武器 |

### 22.2.4 案例三：Colonial Pipeline——IT/OT 连锁反应

**事件概要**

2021年5月7日，美国最大的燃油管道运营商 Colonial Pipeline 遭受 DarkSide 勒索软件攻击。管道全长约 8,850 公里，承载美国东海岸约 45% 的燃油供应。攻击发生在 IT 网络，但公司主动关闭了 OT 系统以防止蔓延，导致管道停运 6 天。

**攻击路径详细还原**

```plaintext
时间线:

2021年4月（攻击前数周）
  └─ DarkSide 通过一个已禁用但仍有效的 VPN 账户入侵
     ├─ 该 VPN 账户使用旧的密码，未启用 MFA
     ├─ 密码可能来自此前的数据泄露（凭证填充攻击）
     └─ 没有设备证书验证或网络访问控制（NAC）

2021年5月7日凌晨
  ├─ 攻击者在 IT 网络中部署 DarkSide 勒索软件
  ├─ 加密约 100GB 数据，包括计费系统、ERP、邮件
  ├─ Colonial 的安全团队发现异常，CEO 决定主动关闭管道 OT 系统
  └─ 管道全线停运

2021年5月7-12日
  ├─ 美国东海岸出现燃油恐慌性抢购
  ├─ 多州宣布紧急状态
  ├─ Colonial 向 DarkSide 支付 75 BTC（约 440 万美元）赎金
  ├─ FBI 介入调查
  └─ 5月12日管道逐步恢复运营

后续
  ├─ FBI 追回约 63.7 BTC（约 230 万美元）——通过追踪比特币钱包
  ├─ DarkSide 宣布关闭（基础设施被成员所在国执法机构接管）
  └─ 美国签署行政令加强关键基础设施网络安全
```

**技术分析：DarkSide 勒索软件**

```python
# DarkSide RaaS（勒索软件即服务）架构
# 参考: FireEye/Mandiant 和 CrowdStrike 的分析报告

darkside_features = {
    "deployment_model": "RaaS - 开发者提供软件，附属机构负责攻击",
    "encryption": {
        "algorithm": "Salsa20（对称）+ RSA-1024（密钥封装）",
        "speed": "多线程并行加密，针对大文件优化",
        "selective": "跳过系统关键文件，确保系统可启动以显示勒索信"
    },
    "double_extortion": {
        "layer_1": "加密本地文件，勒索解密密钥",
        "layer_2": "威胁公开窃取的敏感数据（建立泄露站点）"
    },
    "anti_detection": {
        "vm_check": "检测虚拟机/沙箱环境",
        "safe_mode": "在安全模式下执行加密以绕过安全软件",
        "shadow_delete": "删除卷影副本（vssadmin delete shadows /all /quiet）"
    },
    "lateral_movement": [
        "PsExec 远程执行",
        "RDP 横向移动",
        "AD 枚举和凭证窃取",
        "PowerShell Empire / Cobalt Strike"
    ]
}
```

**Colonial Pipeline 的根本失误**

1. **VPN 账户管理失败**：一个已离职员工的 VPN 账户仍然有效，密码未重置，未启用 MFA
2. **IT/OT 未充分隔离**：虽然理论上 IT 和 OT 是分开的，但公司安全团队在紧急情况下选择了全面关闭，说明隔离措施的可信度不足
3. **应急响应决策缺陷**：关闭 OT 系统是过度反应——如果 IT/OT 隔离做得足够好，IT 被加密不应影响 OT 运行
4. **备份恢复能力不足**：如果备份完善且经过测试，无需支付赎金

**防御要点**

```plaintext
针对此事件的防御优先级（按投入产出比排序）:

P0 - 立即执行:
  ├─ 所有远程访问强制 MFA（不仅是 VPN，包括 RDP、SSH、Web 管理）
  ├─ 清理僵尸账户（定期审计 AD/VPN 账户，禁用 90 天未使用的账户）
  └─ 离线备份（3-2-1 原则：3 份副本，2 种介质，1 份离线）

P1 - 短期（1-3 个月）:
  ├─ IT/OT 网络分段验证（红队测试确认无法从 IT 到 OT 横向移动）
  ├─ 部署 EDR（端点检测与响应）覆盖 IT 和 OT 边界
  └─ 制定并演练勒索软件应急响应手册

P2 - 中期（3-6 个月）:
  ├─ Zero Trust 网络架构（BeyondCorp 模型）
  ├─ 特权访问管理（PAM）
  └─ 定期渗透测试 + 红蓝对抗
```

### 22.2.5 案例四：TRITON/TRISIS——攻击安全仪表系统

**为什么这个案例排在最高优先级**

TRITON（又名 TRISIS、HatMan）是 2017 年 12 月发现的针对安全仪表系统（SIS, Safety Instrumented System）的恶意软件。如果说 Stuxnet 攻击的是控制层，TRITON 攻击的是最后的安全防线——当过程失控时，SIS 负责执行紧急停车（ESD），防止爆炸、泄漏、人员伤亡。攻击 SIS 意味着攻击者不仅要破坏过程，还要剥夺操作员紧急停止的能力。

**攻击目标和背景**

```plaintext
目标设施: 沙特阿拉伯某石化厂（公开报道未点名，但安全社区确认为 Petro Rabigh）
目标设备: Triconex SIS 控制器（施耐德电气 Triconex 3008 系列）
安全等级: SIL 3（安全完整性等级 3，最高常见的工业安全等级）
攻击时间: 2017 年 8 月被植入，12 月意外触发

Triconex SIS 的安全设计:
  ├─ 三重模冗余（TMR）：3 个独立处理器投票表决
  ├─ 硬件级别防篡改
  └─ 但通过 TriStation 协议可以进行工程编程（这是攻击入口）
```

**攻击技术细节**

```python
# TRITON 恶意软件架构
# 参考: Mandiant/FireEye "TRITON: Disrupting the Safety Instrumented Systems" (2017)
#        Dragos "TRISIS Malware" 分析 (2017)

# 攻击链
attack_chain = """
1. 初始访问: 攻击者获取了 SIS 工程站的访问权限

   ├─ SIS 工程站运行 Windows 系统，连接 Triconex 控制器
   └─ 工程站上安装了 TriStation 1131 编程软件

   ├─ Python 框架实现了 TriStation 协议的完整通信栈

3. 固件注入:
   ├─ 利用 TriStation 协议的工程功能上传恶意固件

tristation_protocol = """
```

**攻击的意外暴露**

TRITON 的暴露非常偶然——攻击者的代码存在一个 bug，在注入 SIS 固件时触发了 Triconex 控制器的安全故障模式，导致工厂紧急停车。操作员发现 SIS 控制器异常后联系施耐德电气，施耐德在分析中发现了恶意软件。

这个 bug 导致攻击者的目标未能完全实现，但如果没有这个 bug，后果可能是灾难性的——SIS 被禁用后，如果过程失控（如反应器超温超压），将没有自动紧急停车机制，可能导致爆炸和人员伤亡。

**攻击归因**

美国政府和安全社区归因于俄罗斯政府支持的 Turla/APT 组织（也有分析指向与伊朗有关的 Xenotime 组织）。2020 年，美国 DOJ 起诉了三名俄罗斯 GRU 军官。

### 22.2.6 案例五：佛罗里达水处理厂投毒攻击

**事件概要**

2021 年 2 月 5 日，美国佛罗里达州 Oldsmar 市水处理厂的操作员发现有人远程访问了水处理控制系统，将氢氧化钠（NaOH，烧碱）浓度从正常的 100 ppm 修改为 11,100 ppm——足以使饮用水有毒。

**攻击路径**

```plaintext
攻击过程（非常简单，这也是最令人担忧的部分）:

1. 入口: TeamViewer 远程桌面软件
   ├─ 水处理厂使用 TeamViewer 进行远程维护
   ├─ 多名操作员共享同一个密码
   └─ 未启用 MFA

2. 执行:
   ├─ 攻击者通过 TeamViewer 连接到 HMI 工作站
   ├─ 在操作员屏幕上被观察到移动鼠标
   ├─ 将 NaOH 浓度从 100 ppm 修改为 11,100 ppm
   └─ 操作员立即手动恢复了正确值

3. 暴露的根本原因:
   ├─ 远程访问工具（RAT）使用默认/弱密码
   ├─ 无 MFA
   ├─ 共享账户（无法追踪谁在何时访问）
   ├─ Windows 7 系统（已停止安全更新）
   └─ 无网络分段（IT 和 OT 在同一网段）
```

**为什么这个案例重要**

Oldsmar 事件的技术水平很低——没有零日漏洞，没有定制恶意软件，甚至没有复杂的横向移动。它之所以重要，是因为它暴露了全球数以万计的小型水务、电力、废水处理设施的真实安全状态：

```plaintext
小型 ICS 设施的典型安全状态:
  ├─ 远程访问: TeamViewer / VNC / RDP，默认密码，无 MFA
  ├─ 操作系统: Windows 7 / XP（已停止更新）
  ├─ 网络隔离: 无（HMI 直接连接互联网）
  ├─ 安全监控: 无（没有 SIEM、没有告警）
  ├─ 安全预算: 几乎为零
  └─ 安全人员: 无专职 IT 安全人员

# 这些设施不是个例，而是全球普遍现象
```

**防御措施（适用于小型 ICS 设施）**

```plaintext
低成本高效果的防御优先级:

1. 远程访问安全（0 成本）
   ├─ 禁用不必要的远程访问（TeamViewer、VNC、RDP）
   ├─ 必须保留的: 更换强密码 + 启用 MFA
   └─ 使用 VPN 替代直接暴露的远程桌面

2. 网络分段（低成本）
   ├─ 将 HMI/PLC 放在独立 VLAN
   ├─ 只允许特定 IP 访问 OT 网段
   └─ 阻断 OT 到互联网的直接出站连接

3. 物理安全（0 成本）
   ├─ 锁好服务器机柜
   ├─ 禁用未使用的 USB 端口
   └─ 限制物理接触 HMI 的人员

4. 监控（低成本）
   ├─ 记录所有远程连接日志
   ├─ 关键参数变更告警（氢氧化钠浓度突然变化应触发报警）
   └─ 开源 SIEM（如 Wazuh）可以免费部署
```

### 22.2.7 ICS 安全防御体系

**分层防御架构**

```plaintext
                    ┌──────────────────────┐
                    │   人员安全意识培训    │
                    └──────────┬───────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
     ┌────────▼──────┐  ┌─────▼─────┐  ┌───────▼──────┐
     │  IT 安全层     │  │ IT-OT 边界│  │  OT 安全层    │
     │  ├─ EDR       │  │ ├─ 防火墙  │  │ ├─ 工业防火墙 │
     │  ├─ SIEM      │  │ ├─ DMZ     │  │ ├─ 协议深度   │
     │  ├─ 邮件网关  │  │ ├─ 单向    │  │ │  检测(DPI)  │
     │  └─ WAF       │  │ │  光闸    │  │ ├─ 异常检测   │
     └───────────────┘  │ └─ 数据     │  │ └─ PLC 完整   │
                        │    二极管   │  │    性监控     │
                        └────────────┘  └──────────────┘
```

**关键防御技术**

1. **工业协议深度检测（DPI）**：传统 IT IDS 无法解析 Modbus/DNP3/IEC 104 等工业协议。需要部署支持 ICS 协议的 DPI 引擎（如 Suricata + ICS 规则集，或 Claroty、Nozomi、Dragos 等专业产品）。

2. **PLC 程序完整性监控**：定期读取 PLC 逻辑，计算哈希值，与基线比对。任何未授权的修改（如 Stuxnet 注入的恶意功能块）都应触发告警。

3. **单向数据二极管（Unidirectional Security Gateway）**：在 IT 和 OT 之间部署物理单向传输设备，数据只能从 OT 流向 IT（用于监控数据），物理上不可能反向传输。适用于对安全要求极高的场景（核电、化工）。

4. **安全仪表系统（SIS）独立保护**：SIS 必须与基本过程控制系统（BPCS）物理隔离。TRITON 事件证明，如果 SIS 和 BPCS 共享网络，攻击者可以同时瘫痪两者。

**ICS 专用安全工具**

### 22.2.8 常见误区与纠正

| 误区 | 现实 | 纠正措施 |
|------|------|----------|
| "物理隔离就够了" | Stuxnet 证明 USB 可跨越 Air-Gap；实际环境中很多"隔离"存在隐性连接 | 部署数据二极管 + USB 管控 + 定期检查网络连通性 |
| "ICS 不连互联网就安全" | VPN、远程维护、4G 路由器都可能创建隐性互联网连接 | 用 Shodan 和主动扫描确认 OT 设备没有暴露在公网 |
| "OT 协议太老不会被攻击" | Industroyer 直接利用了 IEC 104 等"老旧"协议 | 部署 ICS 协议 DPI，监控异常功能码和异常地址 |
| "打补丁就好" | OT 补丁需要停产窗口，一个补丁可能需要数月才能部署 | 虚拟补丁（IPS 规则）+ 网络分段减少补丁依赖 |
| "安全是 IT 部门的事" | OT 安全需要工艺工程师、控制工程师和 IT 安全人员协作 | 建立跨职能的 OT 安全团队，定期联合演练 |
| "购买安全产品就安全了" | 工具只是辅助，安全流程和人员能力才是关键 | 安全产品 + 安全流程 + 人员培训三位一体 |

**练习 3：PLC 程序完整性检查**

```python
#!/usr/bin/env python3
"""
PLC 程序完整性监控工具（概念演示）
定期读取 PLC 程序并计算哈希值，与基线比对
"""

import hashlib
import json
from datetime import datetime

class PLCIntegrityMonitor:
    def __init__(self, baseline_file="plc_baseline.json"):
        self.baseline_file = baseline_file
        self.baseline = self._load_baseline()

    def _load_baseline(self):
        """加载基线哈希值"""
        try:
            with open(self.baseline_file, 'r') as f:
                return json.load(f)
        except FileNotFoundError:
            return {}

    def _save_baseline(self):
        """保存基线哈希值"""
        with open(self.baseline_file, 'w') as f:
            json.dump(self.baseline, f, indent=2)

    def read_plc_program(self, plc_ip, plc_type="s7"):
        """
        从 PLC 读取程序内容
        实际实现需要使用 python-snap7 (S7) 或 pymodbus (Modbus)
        """
        # 示例: 使用 snap7 读取 S7 PLC 程序块
        # import snap7
        # client = snap7.client.Client()
        # client.connect(plc_ip, 0, 1)
        # program_data = client.upload(block_type=0x38, block_num=1)  # OB1
        # client.disconnect()
        # return program_data
        pass

    def calculate_hash(self, data):
        """计算 SHA-256 哈希"""
        return hashlib.sha256(data).hexdigest()

    def establish_baseline(self, plc_ip, plc_type="s7"):
        """建立初始基线"""
        program_data = self.read_plc_program(plc_ip, plc_type)
        program_hash = self.calculate_hash(program_data)

        self.baseline[plc_ip] = {
            "hash": program_hash,
            "timestamp": datetime.now().isoformat(),
            "plc_type": plc_type,
            "size": len(program_data)
        }
        self._save_baseline()
        print(f"[+] 基线已建立: {plc_ip} -> {program_hash}")

    def check_integrity(self, plc_ip, plc_type="s7"):
        """检查 PLC 程序完整性"""
        if plc_ip not in self.baseline:
            print(f"[!] 无基线数据: {plc_ip}")
            return None

        current_data = self.read_plc_program(plc_ip, plc_type)
        current_hash = self.calculate_hash(current_data)
        baseline_hash = self.baseline[plc_ip]["hash"]

        if current_hash == baseline_hash:
            print(f"[OK] {plc_ip}: 程序未变更")
            return True
        else:
            print(f"[ALERT] {plc_ip}: 程序已被修改!")
            print(f"  基线哈希: {baseline_hash}")
            print(f"  当前哈希: {current_hash}")
            print(f"  基线时间: {self.baseline[plc_ip]['timestamp']}")
            print(f"  → 可能遭受了 Stuxnet 类型的 PLC 代码注入攻击")
            return False
```
