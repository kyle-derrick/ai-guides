## 练习方法

本章围绕事务与并发控制提供了五个递进式练习，从基础概念理解到架构设计实战。每个练习都设有明确的目标、步骤、检查标准和参考答案。建议按顺序完成，前一个练习的成果会成为后续练习的基础。

### 练习一：ACID 特性验证实验（预计40分钟）

**目标**：

通过实际 SQL 操作验证事务的四大特性——原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）、持久性（Durability），建立对事务机制的直觉理解。

**准备工作**：

以 MySQL/PostgreSQL 为例，创建实验用表和初始数据：

```sql
-- 创建银行账户表
CREATE TABLE accounts (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL,
    balance DECIMAL(12,2) NOT NULL CHECK (balance >= 0),
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 插入测试数据
INSERT INTO accounts (name, balance) VALUES
    ('Alice', 10000.00),
    ('Bob',   5000.00),
    ('Carol', 3000.00);

-- 验证约束生效：这条语句应报错
INSERT INTO accounts (name, balance) VALUES ('Dave', -100.00);
```

**步骤**：

1. **验证原子性（10分钟）**

   原子性要求事务中的操作要么全部成功，要么全部回滚，不存在中间状态。

   ```sql
   -- 开启事务
   BEGIN;

   -- Alice 向 Bob 转账 2000 元
   UPDATE accounts SET balance = balance - 2000 WHERE name = 'Alice';
   UPDATE accounts SET balance = balance + 2000 WHERE name = 'Bob';

   -- 模拟中途失败（触发除零错误）
   SELECT 1 / 0;

   -- 即使写了 COMMIT，前面的转账也会被回滚
   COMMIT;
   ```

   验证方法：执行上述语句后查询余额，Alice 和 Bob 的余额应与转账前一致，说明整个事务被回滚。

   ```sql
   SELECT name, balance FROM accounts WHERE name IN ('Alice','Bob');
   -- 期望：Alice=10000, Bob=5000（未变化）
   ```

   **思考题**：如果去掉 BEGIN/COMMIT，直接执行三条语句会发生什么？为什么？

2. **验证一致性（10分钟）**

   一致性要求事务执行前后，数据库的完整性约束始终成立。

   ```sql
   BEGIN;

   -- 尝试让 Alice 转出超过余额的金额
   UPDATE accounts SET balance = balance - 12000 WHERE name = 'Alice';

   COMMIT;  -- 失败：CHECK 约束 (balance >= 0) 被违反
   ```

   验证方法：事务回滚后查询 Alice 余额，应仍为 10000 元。数据库通过约束机制保证了数据一致性。

3. **验证隔离性（10分钟）**

   隔离性要求并发事务之间互不干扰。通过两个并行的终端（终端A 和终端B）来验证：

   终端A（使用默认的 REPEATABLE READ 隔离级别）：

   ```sql
   -- 终端A
   SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
   BEGIN;
   SELECT balance FROM accounts WHERE name = 'Alice';  -- 读到 10000
   ```

   终端B：

   ```sql
   -- 终端B：在终端A未提交时修改 Alice 的余额
   BEGIN;
   UPDATE accounts SET balance = balance - 500 WHERE name = 'Alice';
   COMMIT;
   ```

   终端A 继续查询：

   ```sql
   -- 终端A：再次查询
   SELECT balance FROM accounts WHERE name = 'Alice';
   -- 期望：仍然读到 10000（REPEATABLE READ 下看不到终端B的修改）
   COMMIT;
   ```

   验证方法：对比两次 SELECT 的结果，确认在 REPEATABLE READ 下终端A 读到的是一致性快照。

4. **验证持久性（10分钟）**

   持久性要求已提交的事务对数据库的修改是永久性的，即使系统崩溃也不丢失。

   ```sql
   BEGIN;
   UPDATE accounts SET balance = 99999.99 WHERE name = 'Alice';
   COMMIT;
   ```

   验证方法：
   - 重启 MySQL 服务后查询：`SELECT balance FROM accounts WHERE name = 'Alice';`，结果应为 99999.99
   - 查看 WAL 日志目录（如 PostgreSQL 的 `pg_wal/`）确认 WAL 文件已生成

   ```bash
   # PostgreSQL 查看 WAL 文件
   pg_waldump $(ls -t pg_wal/ | head -1) | grep -i "accounts"
   ```

**检查标准**：

- [ ] 能用实验证明回滚机制（原子性）
- [ ] 能触发并验证 CHECK 约束（一致性）
- [ ] 能在两个终端间验证隔离级别差异（隔离性）
- [ ] 能理解 WAL 日志与持久性的关系（持久性）
- [ ] 能解释为什么每个特性是必要的（思考题）

### 练习二：四种隔离级别对比实验（预计60分钟）

**目标**：

通过并发场景复现脏读、不可重复读、幻读三类问题，并验证不同隔离级别如何逐级解决这些问题，形成对隔离级别选择的判断力。

**准备工作**：

```sql
-- 重置测试数据
UPDATE accounts SET balance = 10000 WHERE name = 'Alice';
UPDATE accounts SET balance = 5000  WHERE name = 'Bob';
DELETE FROM accounts WHERE name NOT IN ('Alice','Bob','Carol');

-- 创建订单表（用于复现幻读）
CREATE TABLE orders (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_name VARCHAR(50),
    amount DECIMAL(10,2),
    status VARCHAR(20) DEFAULT 'pending'
);

INSERT INTO orders (user_name, amount, status) VALUES
    ('Alice', 500.00, 'completed'),
    ('Alice', 300.00, 'pending'),
    ('Bob',   1200.00, 'completed');
```

**步骤**：

1. **复现脏读（READ UNCOMMITTED）（15分钟）**

   ```sql
   -- 终端A：READ UNCOMMITTED 允许读到未提交数据
   SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
   BEGIN;
   SELECT balance FROM accounts WHERE name = 'Alice';  -- 读到 10000
   ```

   ```sql
   -- 终端B：修改但不提交
   BEGIN;
   UPDATE accounts SET balance = 5000 WHERE name = 'Alice';
   -- 注意：这里不执行 COMMIT
   ```

   ```sql
   -- 终端A：读到终端B的"脏数据"
   SELECT balance FROM accounts WHERE name = 'Alice';
   -- 期望：读到 5000（脏读！）
   ```

   ```sql
   -- 终端B：回滚
   ROLLBACK;
   ```

   ```sql
   -- 终端A：再次查询，数据恢复
   SELECT balance FROM accounts WHERE name = 'Alice';
   -- 期望：10000（说明终端A之前读到的是未持久化的中间状态）
   COMMIT;
   ```

   **记录**：将终端A 三次读到的余额值记录在下表中：

   | 读取时机 | 读到的值 | 是否正确 |
   |----------|----------|----------|
   | 终端B修改前 | | |
   | 终端B修改后、回滚前 | | |
   | 终端B回滚后 | | |

2. **复现不可重复读（READ COMMITTED）（15分钟）**

   ```sql
   -- 终端A
   SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
   BEGIN;
   SELECT balance FROM accounts WHERE name = 'Alice';  -- 第一次读
   ```

   ```sql
   -- 终端B：在同一事务期间修改并提交
   BEGIN;
   UPDATE accounts SET balance = 8000 WHERE name = 'Alice';
   COMMIT;
   ```

   ```sql
   -- 终端A：第二次读，结果与第一次不同
   SELECT balance FROM accounts WHERE name = 'Alice';
   -- 期望：8000（不可重复读！同一事务内两次读取结果不同）
   COMMIT;
   ```

   **思考题**：不可重复读在什么业务场景下会造成问题？举一个具体例子。

3. **复现幻读（REPEATABLE READ）（15分钟）**

   ```sql
   -- 终端A
   SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
   BEGIN;
   SELECT COUNT(*) FROM orders WHERE user_name = 'Alice';  -- 第一次统计
   SELECT SUM(amount) FROM orders WHERE user_name = 'Alice';
   ```

   ```sql
   -- 终端B：插入新记录并提交
   BEGIN;
   INSERT INTO orders (user_name, amount, status) VALUES ('Alice', 200.00, 'pending');
   COMMIT;
   ```

   ```sql
   -- 终端A：再次统计，发现行数增加了（幻读）
   SELECT COUNT(*) FROM orders WHERE user_name = 'Alice';
   -- 期望：3（原来2条 + 终端B插入的1条 = 3条）

   -- 注意：MySQL InnoDB 在 REPEATABLE READ 下通过间隙锁避免了部分幻读，
   -- 但 PostgreSQL 的 REPEATABLE READ 仍可能出现幻读
   COMMIT;
   ```

   **注意**：MySQL InnoDB 的 REPEATABLE READ 通过 MVCC + 间隙锁（Gap Lock）在大多数场景下避免了幻读，但以下情况仍可能出现：

   ```sql
   -- MySQL 特殊情况：当前读（SELECT ... FOR UPDATE）会看到新插入的行
   -- 终端A
   BEGIN;
   SELECT * FROM orders WHERE user_name = 'Alice' FOR UPDATE;

   -- 终端B（已在终端A之前提交了新行）
   -- 终端A 再次查看会看到新行
   ```

4. **验证 SERIALIZABLE 的效果（15分钟）**

   ```sql
   -- 终端A
   SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
   BEGIN;
   SELECT COUNT(*) FROM orders WHERE user_name = 'Alice';

   -- 终端B：尝试插入（会被阻塞，直到终端A提交）
   BEGIN;
   INSERT INTO orders (user_name, amount, status) VALUES ('Alice', 100.00, 'pending');
   -- 此语句会挂起，等待终端A释放锁
   ```

   ```sql
   -- 终端A
   COMMIT;  -- 终端B的 INSERT 此时才能执行
   ```

   **记录**：将四种隔离级别的对比结果整理到下表：

   | 隔离级别 | 脏读 | 不可重复读 | 幻读 | 性能影响 |
   |----------|------|-----------|------|----------|
   | READ UNCOMMITTED | | | | |
   | READ COMMITTED | | | | |
   | REPEATABLE READ | | | | |
   | SERIALIZABLE | | | | |

**检查标准**：

- [ ] 能复现脏读现象并解释原因
- [ ] 能复现不可重复读现象并解释原因
- [ ] 能复现幻读现象（含 MySQL 和 PostgreSQL 的差异）
- [ ] 能说明 SERIALIZABLE 如何避免上述所有问题
- [ ] 能填写完整的隔离级别对比表
- [ ] 能针对具体业务场景推荐合适的隔离级别

### 练习三：死锁的产生、检测与解决（预计50分钟）

**目标**：

通过构造死锁场景，掌握死锁的四个必要条件、MySQL InnoDB 的死锁检测机制、死锁日志分析方法，以及死锁的预防策略。

**准备工作**：

```sql
-- 创建两个互相关联的表
CREATE TABLE table_a (
    id INT PRIMARY KEY,
    data VARCHAR(100),
    ref_id INT
);

CREATE TABLE table_b (
    id INT PRIMARY KEY,
    data VARCHAR(100),
    ref_id INT
);

INSERT INTO table_a VALUES (1, 'row_a1', 1), (2, 'row_a2', 2), (3, 'row_a3', 3);
INSERT INTO table_b VALUES (1, 'row_b1', 1), (2, 'row_b2', 2), (3, 'row_b3', 3);
```

**步骤**：

1. **构造并观察死锁（15分钟）**

   使用两个终端模拟两个事务以相反顺序获取锁：

   ```sql
   -- 终端A
   BEGIN;
   UPDATE table_a SET data = 'modified_by_A' WHERE id = 1;
   -- 获得 table_a 第1行的排他锁
   -- 等待2秒后执行下一条语句（给终端B时间获取 table_b 的锁）
   SELECT SLEEP(2);
   UPDATE table_b SET data = 'modified_by_A' WHERE id = 1;
   -- 此处会阻塞，等待终端B释放 table_b 第1行的锁
   ```

   ```sql
   -- 终端B（在终端A执行 SLEEP 期间快速执行）
   BEGIN;
   UPDATE table_b SET data = 'modified_by_B' WHERE id = 1;
   -- 获得 table_b 第1行的排他锁
   UPDATE table_a SET data = 'modified_by_B' WHERE id = 1;
   -- 此处触发死锁！MySQL 自动检测并回滚其中一个事务
   ```

   观察终端B 的输出，应该看到类似 `ERROR 1213 (40001): Deadlock found when trying to get lock` 的错误。

2. **分析死锁日志（15分钟）**

   ```sql
   -- 查看最近一次死锁的详细信息
   SHOW ENGINE INNODB STATUS\G
   ```

   在输出中找到 `LATEST DETECTED DEADLOCK` 部分，重点关注：

   *** (1) TRANSACTION:          -- 事务1的信息
   TRANSACTION ..., ACTIVE X sec
   mysql tables in use 1, locked 1
   LOCK WAIT, ... lock struct(s), ...

   *** (1) WAITING FOR THIS LOCK TO BE GRANTED:  -- 事务1在等待什么锁
   RECORD LOCKS ... space ... page ... n bits ...
   lock_mode X locks rec but not gap waiting

   *** (2) TRANSACTION:          -- 事务2的信息
   ...

   *** (2) WAITING FOR THIS LOCK TO BE GRANTED:  -- 事务2在等待什么锁
   ...

   *** WE ROLL BACK TRANSACTION (1)  -- 被回滚的事务

   **分析要点**：

   | 分析维度 | 事务1 | 事务2 |
   |----------|-------|-------|
   | 已持有的锁 | | |
   | 等待获取的锁 | | |
   | 阻塞的表/行 | | |
   | 被回滚的是哪个 | | |

3. **验证死锁的四个必要条件（10分钟）**

   死锁的四个必要条件（Coffman 条件）：

   | 条件 | 含义 | 在本实验中的体现 |
   |------|------|------------------|
   | 互斥 | 资源一次只能被一个事务持有 | 排他锁（X锁）互斥 |
   | 持有并等待 | 事务持有资源的同时等待其他资源 | 终端A持有 table_a 锁，等待 table_b 锁 |
   | 不可抢占 | 已持有的锁不能被强制释放 | 事务未完成前锁不会释放 |
   | 循环等待 | 事务之间形成等待环路 | A→B→A 形成环路 |

   逐一验证：如果去掉其中任意一个条件，死锁能否发生？

4. **实施死锁预防策略（10分钟）**

   策略一：统一加锁顺序

   ```sql
   -- 终端A
   BEGIN;
   UPDATE table_a SET data = 'safe_A' WHERE id = 1;
   UPDATE table_b SET data = 'safe_A' WHERE id = 1;
   COMMIT;

   -- 终端B（与终端A以相同顺序加锁）
   BEGIN;
   UPDATE table_a SET data = 'safe_B' WHERE id = 1;
   UPDATE table_b SET data = 'safe_B' WHERE id = 1;
   COMMIT;
   ```

   验证：两个事务不再发生死锁。

   策略二：设置锁等待超时

   ```sql
   -- 设置锁等待超时为 5 秒（默认50秒）
   SET innodb_lock_wait_timeout = 5;

   -- 模拟长时间持锁
   BEGIN;
   UPDATE table_a SET data = 'hold_lock' WHERE id = 1;
   SELECT SLEEP(10);  -- 持锁10秒
   COMMIT;
   ```

   在另一个终端尝试获取同一行锁，观察超时后的报错。

   策略三：使用 SELECT ... FOR UPDATE 时按固定顺序

   ```sql
   -- 推荐：先锁 id 较小的行
   BEGIN;
   SELECT * FROM table_a WHERE id = 1 FOR UPDATE;
   SELECT * FROM table_a WHERE id = 2 FOR UPDATE;
   UPDATE table_a SET data = 'reordered' WHERE id = 2;
   COMMIT;
   ```

**检查标准**：

- [ ] 能成功构造并触发死锁
- [ ] 能读懂 `SHOW ENGINE INNODB STATUS` 中的死锁日志
- [ ] 能填写死锁分析表（已持有锁 / 等待锁 / 被回滚事务）
- [ ] 能解释四个必要条件与实验的对应关系
- [ ] 能实施至少一种死锁预防策略并验证有效性

### 练习四：MVCC 原理与实现验证（预计50分钟）

**目标**：

深入理解多版本并发控制（MVCC）的工作原理，通过实验观察 undo log、ReadView 的行为，掌握 MVCC 如何在不加锁的情况下实现读写并发。

**准备工作**：

```sql
-- 确认 InnoDB 的 MVCC 相关配置
SHOW VARIABLES LIKE 'innodb_undo_tablespaces';
SHOW VARIABLES LIKE 'innodb_undo_log_truncate';
SHOW VARIABLES LIKE 'transaction_isolation';

-- 创建测试表
CREATE TABLE products (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100),
    stock INT NOT NULL DEFAULT 100,
    price DECIMAL(10,2)
) ENGINE=InnoDB;

INSERT INTO products (name, stock, price) VALUES
    ('iPhone', 50, 7999.00),
    ('iPad',   30, 3999.00);
```

**步骤**：

1. **观察 MVCC 的版本链（15分钟）**

   MySQL InnoDB 为每行数据隐含了三个字段：

   | 隐含字段 | 含义 |
   |----------|------|
   | `DB_TRX_ID` | 最后修改该行的事务ID |
   | `DB_ROLL_PTR` | 回滚指针，指向 undo log 中该行的上一个版本 |
   | `DB_ROW_ID` | 行ID（当没有显式主键时生成） |

   ```sql
   -- 终端A：查看当前事务ID
   SELECT @@tx_isolation;  -- 确认是 REPEATABLE READ
   BEGIN;
   SELECT txid_current();  -- 记下事务ID，假设为 100

   -- 读取当前版本
   SELECT * FROM products WHERE id = 1;
   -- 期望：name='iPhone', stock=50
   ```

   ```sql
   -- 终端B：修改 stock 值
   BEGIN;
   SELECT txid_current();  -- 记下事务ID，假设为 101
   UPDATE products SET stock = 49 WHERE id = 1;
   COMMIT;
   ```

   ```sql
   -- 终端A：再次读取（应该看到旧版本，因为事务101在终端A开始之后提交）
   SELECT * FROM products WHERE id = 1;
   -- 期望：仍然读到 stock=50（MVCC 快照读）
   COMMIT;
   ```

2. **验证 ReadView 的可见性规则（15分钟）**

   ReadView 包含的关键信息：

   | 字段 | 含义 |
   |------|------|
   | `m_ids` | 创建 ReadView 时活跃的事务ID列表 |
   | `min_trx_id` | 活跃事务ID的最小值 |
   | `max_trx_id` | 下一个将分配的事务ID |
   | `creator_trx_id` | 创建该 ReadView 的事务ID |

   可见性判断规则：

   对于版本链中的某个版本 V（其 trx_id = V.trx_id）：

   if V.trx_id == creator_trx_id:
       可见  // 自己修改的数据

   if V.trx_id < min_trx_id:
       可见  // ReadView 创建前已提交

   if V.trx_id >= max_trx_id:
       不可见  // ReadView 创建后才开始的事务

   if min_trx_id <= V.trx_id < max_trx_id:
       if V.trx_id in m_ids:
           不可见  // ReadView 创建时该事务还未提交
       else:
           可见  // ReadView 创建时该事务已提交

   验证实验：

   ```sql
   -- 终端A
   BEGIN;
   SELECT txid_current();  -- 假设 200

   -- 终端B：修改并提交
   BEGIN;
   UPDATE products SET stock = 48 WHERE id = 1;
   COMMIT;

   -- 终端C：修改并提交
   BEGIN;
   UPDATE products SET stock = 47 WHERE id = 1;
   COMMIT;

   -- 终端A：读取
   SELECT * FROM products WHERE id = 1;
   -- 期望：stock=50（因为200开始时这些事务都未提交或不存在）
   ```

   将结果记录在下表中：

   | 操作 | stock值 | ReadView.m_ids | V.trx_id | 可见性判断 |
   |------|---------|----------------|----------|------------|
   | 初始读取 | | | | |
   | 终端B提交后读取 | | | | |
   | 终端C提交后读取 | | | | |
   | COMMIT后重新读取 | | | | |

3. **验证快照读 vs 当前读的区别（10分钟）**

   ```sql
   -- 终端A（REPEATABLE READ）
   BEGIN;
   SELECT stock FROM products WHERE id = 1;      -- 快照读：看到旧值
   SELECT stock FROM products WHERE id = 1 FOR UPDATE;  -- 当前读：看到最新值

   -- 在这两个查询之间，让另一个终端修改 stock
   COMMIT;
   ```

   快照读与当前读的对比：

   | 读取方式 | SQL 示例 | 是否加锁 | 读取版本 |
   |----------|----------|----------|----------|
   | 快照读 | `SELECT ...` | 不加锁 | MVCC 版本链中的某个版本 |
   | 当前读（加排他锁） | `SELECT ... FOR UPDATE` | 加 X 锁 | 最新已提交版本 |
   | 当前读（加共享锁） | `SELECT ... LOCK IN SHARE MODE` | 加 S 锁 | 最新已提交版本 |
   | 当前读（DML） | `INSERT / UPDATE / DELETE` | 加 X 锁 | 最新已提交版本 |

   **关键结论**：在 REPEATABLE READ 下，`SELECT ... FOR UPDATE` 能看到其他事务已提交的修改，而普通 `SELECT` 不能。这是因为 FOR UPDATE 执行的是当前读，绕过了 MVCC 快照机制。

4. **分析 undo log 的回滚作用（10分钟）**

   ```sql
   -- 终端A
   BEGIN;
   UPDATE products SET stock = 1 WHERE id = 1;
   UPDATE products SET stock = 2 WHERE id = 1;
   UPDATE products SET stock = 3 WHERE id = 1;
   -- 此时 undo log 中保存了 stock 从 50 → 1 → 2 → 3 的完整版本链
   ROLLBACK;
   -- 回滚后 stock 恢复为 50
   SELECT * FROM products WHERE id = 1;  -- stock=50
   ```

   **要点**：undo log 同时服务于两个目的：
   - 事务回滚（原子性保证）
   - MVCC 版本链构建（隔离性保证）

**场景描述**：

电商秒杀场景：100件商品，1000个并发请求同时下单扣减库存。要求：不超卖、高性能、可扩展。

**步骤**：

1. **方案一：悲观锁（SELECT ... FOR UPDATE）（20分钟）**

   ```sql
   -- 创建秒杀商品表
   CREATE TABLE seckill_items (
       id INT PRIMARY KEY AUTO_INCREMENT,
       name VARCHAR(100),
       stock INT NOT NULL DEFAULT 100,
       version INT NOT NULL DEFAULT 0  -- 乐观锁版本号
   ) ENGINE=InnoDB;

   INSERT INTO seckill_items (name, stock) VALUES ('秒杀商品', 100);
   ```

   悲观锁实现：

   ```sql
   BEGIN;
   -- 排他锁锁定目标行，其他事务在此事务提交前无法修改
   SELECT stock FROM seckill_items WHERE id = 1 FOR UPDATE;

   -- 在应用层判断库存是否充足
   -- if stock >= 1 then 扣减
   UPDATE seckill_items SET stock = stock - 1 WHERE id = 1;

   COMMIT;
   ```

   Python 模拟压测脚本：

   ```python
   import threading
   import pymysql
   import time
   from statistics import mean

   CONFIG = {
       'host': 'localhost', 'user': 'root',
       'password': 'your_password', 'database': 'test'
   }

   results = {'success': 0, 'fail': 0, 'latencies': []}
   lock = threading.Lock()

   def worker(worker_id):
       conn = pymysql.connect(**CONFIG)
       try:
           start = time.time()
           with conn.cursor() as cur:
               conn.begin()
               cur.execute(
                   "SELECT stock FROM seckill_items WHERE id=1 FOR UPDATE"
               )
               stock = cur.fetchone()[0]
               if stock > 0:
                   cur.execute(
                       "UPDATE seckill_items SET stock=stock-1 WHERE id=1"
                   )
                   result = 'success'
               else:
                   result = 'fail'
               conn.commit()
           elapsed = time.time() - start
           with lock:
               results[result] += 1
               results['latencies'].append(elapsed)
       except Exception as e:
           conn.rollback()
           with lock:
               results['fail'] += 1
       finally:
           conn.close()

   # 启动1000个并发线程
   threads = [threading.Thread(target=worker, args=(i,)) for i in range(1000)]
   for t in threads:
       t.start()
   for t in threads:
       t.join()

   print(f"成功: {results['success']}, 失败: {results['fail']}")
   print(f"平均延迟: {mean(results['latencies'])*1000:.2f}ms")
   final_stock = get_stock()  # 查询最终库存
   print(f"最终库存: {final_stock}, 扣减数: {100 - final_stock}")
   assert results['success'] == 100 - final_stock  # 验证无超卖
   ```

2. **方案二：乐观锁（CAS 操作）（20分钟）**

   ```sql
   -- 使用版本号实现乐观锁
   BEGIN;

   -- 读取当前版本号和库存
   SELECT stock, version FROM seckill_items WHERE id = 1;
   -- 假设读到 stock=99, version=0

   -- 尝试扣减，同时校验版本号（CAS：Compare-And-Swap）
   UPDATE seckill_items
   SET stock = stock - 1, version = version + 1
   WHERE id = 1 AND version = 0;

   -- 检查受影响行数
   -- rows_affected = 1 → 成功
   -- rows_affected = 0 → 被其他事务修改了版本号，需重试

   COMMIT;
   ```

   带重试的实现：

   ```python
   def optimistic_worker(worker_id, max_retries=3):
       conn = pymysql.connect(**CONFIG)
       for attempt in range(max_retries):
           try:
               with conn.cursor() as cur:
                   cur.execute(
                       "SELECT stock, version FROM seckill_items WHERE id=1"
                   )
                   stock, version = cur.fetchone()
                   if stock <= 0:
                       return 'sold_out'

                   cur.execute("""
                       UPDATE seckill_items
                       SET stock=stock-1, version=version+1
                       WHERE id=1 AND version=%s
                   """, (version,))

                   if cur.rowcount == 1:
                       conn.commit()
                       return 'success'
                   else:
                       conn.rollback()  # 版本冲突，重试
                       time.sleep(0.01 * (attempt + 1))  # 退避等待
           except Exception:
               conn.rollback()
       return 'max_retries_exceeded'
   ```

3. **方案三：原子操作（UPDATE ... 直接判断）（15分钟）**
