## 17.3 可执行文件格式

可执行文件格式（Executable Format）是操作系统加载和运行程序的"蓝图"。它定义了代码、数据、资源在磁盘上的组织方式，以及操作系统如何将这些内容映射到内存中执行。对于逆向工程师而言，理解可执行文件格式是所有分析工作的起点——你必须先理解"容器"的结构，才能正确地提取和解读其中的内容。

### 17.3.1 为什么可执行文件格式对逆向工程至关重要

可执行文件格式不仅仅是"装代码的盒子"，它承载了大量逆向分析所需的关键信息：

| 信息类型 | 说明 | 逆向价值 |
|---------|------|---------|
| 入口点（Entry Point） | 程序开始执行的第一条指令地址 | 定位分析起点 |
| 导入/导出表 | 程序使用的外部API和暴露的接口 | 推断程序功能 |
| 节区属性 | 代码段、数据段的权限（读/写/执行） | 识别代码与数据的边界 |
| 调试信息 | 符号表、行号、源文件路径 | 辅助理解程序逻辑 |
| 重定位信息 | 地址修正表 | 理解内存布局 |
| 资源段 | 图标、字符串、配置数据 | 提取嵌入资源 |
| 数字签名 | 代码签名证书 | 验证文件完整性和来源 |

理解可执行文件格式的深度直接决定了逆向分析的效率。一个经验丰富的逆向工程师可以通过快速浏览文件头，就能判断出目标程序的编译环境、是否加壳、使用了哪些保护技术，从而制定针对性的分析策略。

### 17.3.2 PE格式（Windows）

PE（Portable Executable）是Windows操作系统可执行文件的标准格式，由微软在1993年的Windows NT中引入，沿用至今。所有Windows上的 `.exe`、`.dll`、`.sys`、`.ocx` 等可执行文件都采用PE格式。PE格式脱胎于COFF（Common Object File Format），因此其文件头部分保留了COFF的痕迹。

#### 17.3.2.1 PE文件总体结构

PE文件采用线性布局，从文件头部到尾部依次排列着不同的数据结构：

```text
┌─────────────────────────────┐  偏移 0x0000
│  DOS Header (64 字节)         │  IMAGE_DOS_HEADER
├─────────────────────────────┤
│  DOS Stub (大小可变)          │  DOS时代的残留代码
├─────────────────────────────┤
│  PE Signature (4 字节)        │  "PE\0\0" (0x00004550)
├─────────────────────────────┤
│  COFF File Header (20 字节)   │  IMAGE_FILE_HEADER
├─────────────────────────────┤
│  Optional Header (大小可变)    │  IMAGE_OPTIONAL_HEADER
│  ├─ Standard Fields          │  魔数、链接器版本、入口点
│  ├─ Windows-Specific Fields  │  镜像基址、对齐、栈/堆大小
│  └─ Data Directories (16个)  │  导入表、导出表、资源等
├─────────────────────────────┤
│  Section Headers (40字节×N)  │  各节区的描述信息
├─────────────────────────────┤
│  .text                       │  代码段（可执行，只读）
├─────────────────────────────┤
│  .rdata                      │  只读数据（导入表、字符串常量）
├─────────────────────────────┤
│  .data                       │  已初始化的全局/静态变量
├─────────────────────────────┤
│  .rsrc                       │  资源（图标、对话框、版本信息）
├─────────────────────────────┤
│  .reloc                       │  重定位表（地址修正信息）
└─────────────────────────────┘
```

#### 17.3.2.2 DOS Header 与 DOS Stub

每个PE文件都以DOS Header开头，这是向后兼容DOS系统的遗迹。DOS Header最关键的一个字段是 `e_lfanew`（偏移0x3C处，4字节），它存储了PE签名在文件中的偏移位置。

```c
typedef struct _IMAGE_DOS_HEADER {
    WORD   e_magic;      // 魔数 "MZ" (0x5A4D)
    WORD   e_cblp;       // 最后页的字节数
    WORD   e_cp;         // 文件页数
    // ... 共16个WORD字段（省略）
    LONG   e_lfanew;     // PE签名的文件偏移 (关键字段！)
} IMAGE_DOS_HEADER;
```

DOS Stub是紧跟在DOS Header后面的一小段DOS程序，通常只输出一行提示信息 `"This program cannot be run in DOS mode"`。有些编译器（如Delphi、某些版本的MSVC）会替换默认的DOS Stub，嵌入自定义的代码或数据，这在逆向分析中可以作为编译器识别的线索。

#### 17.3.2.3 COFF File Header

COFF File Header（也称为 `IMAGE_FILE_HEADER`）紧跟在PE签名之后，大小固定为20字节：

```c
typedef struct _IMAGE_FILE_HEADER {
    WORD   Machine;              // 目标CPU架构
    WORD   NumberOfSections;     // 节区数量
    DWORD  TimeDateStamp;        // 编译时间戳（Unix时间）
    DWORD  PointerToSymbolTable; // 符号表偏移（PE中通常为0）
    DWORD  NumberOfSymbols;      // 符号数量（PE中通常为0）
    WORD   SizeOfOptionalHeader; // Optional Header的大小
    WORD   Characteristics;      // 文件属性标志
} IMAGE_FILE_HEADER;
```

**Machine 字段常见值：**

| 值 | 架构 | 说明 |
|----|------|------|
| 0x014C | IMAGE_FILE_MACHINE_I386 | 32位x86 |
| 0x8664 | IMAGE_FILE_MACHINE_AMD64 | 64位x86-64 |
| 0x01C4 | IMAGE_FILE_MACHINE_ARM | ARM 32位 |
| 0xAA64 | IMAGE_FILE_MACHINE_ARM64 | ARM 64位 |
| 0x01C0 | IMAGE_FILE_MACHINE_ARMTHUMB_MIXED | ARM Thumb |

**Characteristics 字段关键标志：**

| 标志位 | 值 | 含义 |
|--------|-----|------|
| IMAGE_FILE_EXECUTABLE_IMAGE | 0x0002 | 文件是可执行的 |
| IMAGE_FILE_DLL | 0x2000 | 文件是DLL |
| IMAGE_FILE_SYSTEM | 0x1000 | 系统文件（驱动） |
| IMAGE_FILE_32BIT_MACHINE | 0x0100 | 32位机器 |
| IMAGE_FILE_LARGE_ADDRESS_AWARE | 0x0020 | 支持大于2GB地址空间 |

#### 17.3.2.4 Optional Header

Optional Header虽然名为"Optional"，但对于可执行文件（.exe/.dll）来说是必需的。它包含程序加载所需的核心信息。PE32和PE32+（64位）的Optional Header结构略有不同。

```c
// PE32+ (64位) 的 Optional Header 结构
typedef struct _IMAGE_OPTIONAL_HEADER64 {
    // === Standard Fields ===
    WORD    Magic;                // 0x020B = PE32+, 0x010B = PE32
    BYTE    MajorLinkerVersion;   // 链接器主版本
    BYTE    MinorLinkerVersion;   // 链接器次版本
    DWORD   SizeOfCode;           // 代码段总大小
    DWORD   SizeOfInitializedData;    // 已初始化数据大小
    DWORD   SizeOfUninitializedData;  // 未初始化数据大小（BSS）
    DWORD   AddressOfEntryPoint;  // 入口点RVA（相对虚拟地址）

    // === Windows-Specific Fields ===
    ULONGLONG ImageBase;          // 镜像首选基址
    DWORD   SectionAlignment;     // 内存中的节对齐粒度（通常0x1000）
    DWORD   FileAlignment;        // 文件中的节对齐粒度（通常0x200）
    WORD    MajorOperatingSystemVersion;
    WORD    MinorOperatingSystemVersion;
    WORD    MajorImageVersion;
    WORD    MinorImageVersion;
    WORD    MajorSubsystemVersion;
    WORD    MinorSubsystemVersion;
    DWORD   Win32VersionValue;    // 保留，通常为0
    DWORD   SizeOfImage;          // 内存中镜像总大小
    DWORD   SizeOfHeaders;        // 所有头部的总大小
    DWORD   CheckSum;             // 文件校验和
    WORD    Subsystem;            // 子系统类型
    WORD    DllCharacteristics;   // DLL特性标志（安全相关）
    ULONGLONG SizeOfStackReserve; // 栈保留大小
    ULONGLONG SizeOfStackCommit;  // 栈提交大小
    ULONGLONG SizeOfHeapReserve;  // 堆保留大小
    ULONGLONG SizeOfHeapCommit;   // 堆提交大小
    DWORD   LoaderFlags;          // 保留
    DWORD   NumberOfRvaAndSizes;  // 数据目录数量（通常16）

    // === Data Directories ===
    IMAGE_DATA_DIRECTORY DataDirectory[16];
} IMAGE_OPTIONAL_HEADER64;
```

**DllCharacteristics 字段中的安全标志（逆向分析重点）：**

| 标志 | 值 | 含义 | 逆向影响 |
|------|-----|------|---------|
| IMAGE_DLLCHARACTERISTICS_DYNAMIC_BASE | 0x0040 | 启用ASLR | 每次加载地址不同，静态分析需注意 |
| IMAGE_DLLCHARACTERISTICS_FORCE_INTEGRITY | 0x0080 | 强制签名验证 | 需要有效签名才能加载 |
| IMAGE_DLLCHARACTERISTICS_NX_COMPAT | 0x0100 | 启用DEP（数据执行保护） | 栈上代码无法执行 |
| IMAGE_DLLCHARACTERISTICS_NO_SEH | 0x0400 | 不使用SEH | 影响异常处理分析 |
| IMAGE_DLLCHARACTERISTICS_GUARD_CF | 0x4000 | 控制流保护（CFG） | 防止间接调用劫持 |

#### 17.3.2.5 数据目录（Data Directory）

Optional Header末尾的DataDirectory数组包含16个条目，每个条目指向文件中特定数据结构的RVA和大小：

```c
typedef struct _IMAGE_DATA_DIRECTORY {
    DWORD   VirtualAddress;  // 数据的RVA
    DWORD   Size;            // 数据的大小
} IMAGE_DATA_DIRECTORY;
```

**16个数据目录的完整列表：**

| 索引 | 名称 | 说明 | 逆向分析用途 |
|------|------|------|-------------|
| 0 | Export Table | 导出函数表 | 分析DLL暴露的API |
| 1 | Import Table | 导入函数表 | 了解程序依赖的外部API |
| 2 | Resource Table | 资源表 | 提取图标、字符串、对话框 |
| 3 | Exception Table | 异常处理表 | 分析SEH/VEH异常处理 |
| 4 | Security Certificate | 数字签名 | 验证文件签名 |
| 5 | Base Relocation | 重定位表 | 理解地址修正机制 |
| 6 | Debug Directory | 调试信息 | 提取PDB路径、源码信息 |
| 7 | Architecture | 保留 | 通常为空 |
| 8 | Global Pointer | 全局指针 | 用于特定架构优化 |
| 9 | TLS Table | 线程局部存储 | 分析TLS回调函数 |
| 10 | Load Config | 加载配置 | SEH处理器表、CFG标志 |
| 11 | Bound Import | 绑定导入 | 预绑定的导入地址 |
| 12 | IAT | 导入地址表 | 运行时填充的API地址 |
| 13 | Delay Import | 延迟导入 | 按需加载的DLL |
| 14 | CLR Runtime | .NET元数据 | .NET程序分析的入口 |
| 15 | Reserved | 保留 | 未使用 |

**导入表（Import Table）的逆向意义：**

导入表是逆向分析中最重要的数据目录之一。它记录了程序调用的所有外部函数，通过分析导入表，逆向工程师可以快速推断程序的功能类别。例如：

- 导入 `CreateRemoteThread`、`VirtualAllocEx`、`WriteProcessMemory` → 可能是进程注入
- 导入 `InternetOpenUrl`、`HttpSendRequest` → 可能有网络通信功能
- 导入 `RegCreateKey`、`RegSetValue` → 可能修改注册表
- 导入 `CryptEncrypt`、`CryptDecrypt` → 可能有加密/解密操作

导入表的工作原理涉及两张关键表：INT（Import Name Table）存储函数名称的RVA，IAT（Import Address Table）在加载前存储函数名称/序号，在加载后由Windows加载器填充为实际的函数地址。程序运行时通过IAT间接调用外部函数。

#### 17.3.2.6 节区（Section）

节区是PE文件中实际存放代码和数据的区域。每个节区在文件头中有对应的节区头（Section Header），大小为40字节：

```c
typedef struct _IMAGE_SECTION_HEADER {
    BYTE    Name[8];              // 节区名（8字节，不一定以\0结尾）
    union {
        DWORD   PhysicalAddress;  // 物理地址
        DWORD   VirtualSize;      // 内存中的实际大小
    } Misc;
    DWORD   VirtualAddress;       // 节区在内存中的RVA
    DWORD   SizeOfRawData;        // 节区在文件中的大小
    DWORD   PointerToRawData;     // 节区在文件中的偏移
    DWORD   PointerToRelocations; // 重定位条目偏移
    DWORD   PointerToLinenumbers; // 行号表偏移
    WORD    NumberOfRelocations;  // 重定位条目数
    WORD    NumberOfLinenumbers;  // 行号条目数
    DWORD   Characteristics;      // 节区属性标志
} IMAGE_SECTION_HEADER;
```

**常见节区及其特性：**

| 节区名 | 用途 | 属性 | 逆向关注点 |
|--------|------|------|-----------|
| .text | 可执行代码 | 代码 | 主要分析目标，所有程序逻辑的载体 |
| .rdata | 只读数据 | 只读 | 导入表、导出表、字符串常量 |
| .data | 已初始化全局数据 | 读写 | 全局变量、配置数据 |
| .bss | 未初始化数据 | 读写 | 运行时分配，文件中不占空间 |
| .rsrc | 资源数据 | 只读 | 图标、字符串表、版本信息 |
| .reloc | 重定位信息 | 只读 | ASLR支持所必需 |
| .edata | 导出数据 | 只读 | DLL的导出函数 |
| .idata | 导入数据 | 只读 | 外部依赖函数 |
| .tls | 线程局部存储 | 读写 | TLS回调函数，常被加壳程序利用 |
| .pdata | 异常信息 | 只读 | x64异常处理必需 |
| .debug | 调试信息 | 只读 | PDB路径、源码映射 |
| .ndata | 嵌入数据 | - | 一些打包工具的压缩数据 |

**节区属性标志（Characteristics）的关键位：**

```text
IMAGE_SCN_CNT_CODE               0x00000020  包含可执行代码
IMAGE_SCN_CNT_INITIALIZED_DATA   0x00000040  包含已初始化数据
IMAGE_SCN_CNT_UNINITIALIZED_DATA 0x00000080  包含未初始化数据
IMAGE_SCN_MEM_EXECUTE            0x20000000  可执行
IMAGE_SCN_MEM_READ               0x40000000  可读
IMAGE_SCN_MEM_WRITE              0x80000000  可写
IMAGE_SCN_MEM_DISCARDABLE        0x02000000  可丢弃（如.reloc）
```

#### 17.3.2.7 RVA 与文件偏移的转换

理解RVA（Relative Virtual Address）与文件偏移（File Offset）之间的转换是PE分析的基本功。

**核心概念：**

- **VA（Virtual Address）**：程序运行时在内存中的绝对地址 = ImageBase + RVA
- **RVA（Relative Virtual Address）**：相对于镜像基址的偏移
- **文件偏移（File Offset）**：数据在磁盘文件中的位置

由于内存对齐（SectionAlignment）和文件对齐（FileAlignment）的粒度不同，同一个数据在内存中的RVA和在文件中的偏移通常是不一样的。

**转换公式：**

```text
文件偏移 = RVA - 节区.RVA + 节区.PointerToRawData
```

**Python实现：**

```python
def rva_to_offset(pe_data, rva):
    """将RVA转换为文件偏移"""
    for section in pe_data.sections:
        if section.VirtualAddress <= rva < section.VirtualAddress + section.Misc_VirtualSize:
            return rva - section.VirtualAddress + section.PointerToRawData
    # RVA在头部区域，直接等于文件偏移
    if rva < pe_data.sections[0].VirtualAddress:
        return rva
    return None

def offset_to_rva(pe_data, offset):
    """将文件偏移转换为RVA"""
    for section in pe_data.sections:
        if section.PointerToRawData <= offset < section.PointerToRawData + section.SizeOfRawData:
            return offset - section.PointerToRawData + section.VirtualAddress
    if offset < pe_data.sections[0].PointerToRawData:
        return offset
    return None
```

#### 17.3.2.8 PE32 与 PE32+ 的差异

PE32（32位）和PE32+（64位）的主要区别：

| 特性 | PE32 | PE32+ |
|------|------|-------|
| Magic值 | 0x010B | 0x020B |
| 指针大小 | 4字节 | 8字节 |
| ImageBase范围 | 0x00400000 ~ 0x7FFFFFFF | 0x0000000000400000 ~ 0x00007FFFFFFF0000 |
| SizeOfStackReserve | 4字节 | 8字节 |
| SizeOfHeapReserve | 4字节 | 8字节 |
| 数据目录数量 | 最多16个 | 最多16个（不变） |
| 入口点格式 | 32位RVA | 32位RVA（不变） |

#### 17.3.2.9 PE文件的逆向分析实战技巧

**快速判断编译器：**

```python
# 通过DOS Stub判断编译器
import struct

with open("target.exe", "rb") as f:
    # 读取DOS Header中的e_lfanew
    f.seek(0x3C)
    pe_offset = struct.unpack("<I", f.read(4))[0]

    # 读取链接器版本
    f.seek(pe_offset + 0x36)  # COFF Header之后的Optional Header
    major, minor = struct.unpack("<BB", f.read(2))

    compiler_map = {
        (14, 0): "Visual Studio 2015+ (MSVC)",
        (12, 0): "Visual Studio 2013 (MSVC)",
        (11, 0): "Visual Studio 2012 (MSVC)",
        (10, 0): "Visual Studio 2010 (MSVC)",
        (2, 24): "GCC (MinGW)",
        (2, 25): "GCC (Cygwin)",
    }
    print(f"Linker version: {major}.{minor}")
    print(f"Likely compiler: {compiler_map.get((major, minor), 'Unknown')}")
```

**批量提取导入函数信息：**

```python
import pefile

def analyze_imports(filepath):
    """分析PE文件的导入函数，按功能分类"""
    pe = pefile.PE(filepath)

    suspicious_apis = {
        "进程操作": ["CreateRemoteThread", "VirtualAllocEx", "WriteProcessMemory",
                     "NtCreateThreadEx", "RtlCreateUserThread"],
        "网络通信": ["InternetOpenUrl", "HttpSendRequest", "WSAStartup",
                     "connect", "send", "recv", "URLDownloadToFile"],
        "注册表":   ["RegCreateKey", "RegSetValue", "RegOpenKey", "RegDeleteValue"],
        "文件操作": ["CreateFile", "WriteFile", "ReadFile", "DeleteFile", "MoveFile"],
        "信息收集": ["GetComputerName", "GetUserName", "GetVersionEx",
                     "GetSystemInfo", "GlobalMemoryStatus"],
        "加密相关": ["CryptEncrypt", "CryptDecrypt", "CryptCreateHash",
                     "CryptDeriveKey", "BCryptEncrypt"],
        "反调试":   ["IsDebuggerPresent", "CheckRemoteDebuggerPresent",
                     "NtQueryInformationProcess", "OutputDebugString"],
        "持久化":   ["CreateService", "RegSetValueEx", "SetWindowsHookEx",
                     "ScheduleCreate"],
    }

    found = {}
    if not hasattr(pe, 'DIRECTORY_ENTRY_IMPORT'):
        return found

    for entry in pe.DIRECTORY_ENTRY_IMPORT:
        dll_name = entry.dll.decode('utf-8', errors='ignore')
        for imp in entry.imports:
            if imp.name:
                func_name = imp.name.decode('utf-8', errors='ignore')
                for category, apis in suspicious_apis.items():
                    if func_name in apis:
                        found.setdefault(category, []).append(
                            f"{dll_name}!{func_name}"
                        )
    return found

# 使用示例
results = analyze_imports("suspicious.exe")
for category, apis in results.items():
    print(f"\n[{category}]")
    for api in apis:
        print(f"  - {api}")
```

**检测加壳（Packing）：**

```python
import pefile

def detect_packing(filepath):
    """检测PE文件是否被加壳"""
    pe = pefile.PE(filepath)
    indicators = []

    # 检查1：代码节熵值（加壳后通常 > 6.5）
    for section in pe.sections:
        entropy = section.get_entropy()
        name = section.Name.decode('utf-8', errors='ignore').strip('\x00')
        if entropy > 6.5 and name in ['.text', '.code']:
            indicators.append(f"高熵值节区 {name}: {entropy:.2f} (可能是压缩/加密)")

    # 检查2：非标准节区名
    standard_sections = {'.text', '.rdata', '.data', '.bss', '.rsrc', '.reloc',
                         '.edata', '.idata', '.tls', '.pdata', '.debug'}
    for section in pe.sections:
        name = section.Name.decode('utf-8', errors='ignore').strip('\x00')
        if name not in standard_sections and name:
            indicators.append(f"非标准节区名: {name}")

    # 检查3：导入函数过少
    import_count = 0
    if hasattr(pe, 'DIRECTORY_ENTRY_IMPORT'):
        for entry in pe.DIRECTORY_ENTRY_IMPORT:
            import_count += len(entry.imports)
    if import_count < 5:
        indicators.append(f"导入函数极少: {import_count}个 (加壳特征)")

    # 检查4：代码节可写
    for section in pe.sections:
        if section.Characteristics & 0x20000000 and section.Characteristics & 0x80000000:
            indicators.append(f"可写+可执行节区 (可能是自修改代码)")

    # 检查5：TLS回调（常用于壳的解密入口）
    if hasattr(pe, 'DIRECTORY_ENTRY_TLS'):
        indicators.append("存在TLS回调 (壳常用技术)")

    return indicators
```

### 17.3.3 ELF格式（Linux/Unix）

ELF（Executable and Linkable Format）是UNIX系统实验室（USL）在1993年发布的标准格式，后被UNIX/Linux社区广泛采纳，成为System V ABI的一部分。Linux、FreeBSD、Solaris等类Unix系统都使用ELF作为默认可执行文件格式。

#### 17.3.3.1 ELF文件类型

ELF文件通过 `e_type` 字段标识文件类型，共定义了四种核心类型：

```text
ET_NONE (0)    未知类型
ET_REL  (1)    可重定位文件（.o目标文件，待链接）
ET_EXEC (2)    可执行文件（静态链接，固定地址）
ET_DYN  (3)    共享目标文件（.so动态库，也可作为PIE可执行文件）
ET_CORE (4)    核心转储文件（进程崩溃时的内存快照）
```

**重要区分：ET_EXEC vs ET_DYN**

在现代Linux系统中，区分可执行文件和共享库并不只是看文件类型字段。Position Independent Executable（PIE）是目前Linux发行版的默认编译选项，PIE可执行文件的类型字段实际上是 `ET_DYN`（与共享库相同），但通过 `PT_INTERP` 程序头（指定动态链接器）和 `e_entry` 入口点来区分。

#### 17.3.3.4 Program Header（程序头）

Program Header描述了操作系统如何创建进程镜像——即哪些部分需要加载到内存、加载到什么地址、具有什么权限。这是运行时加载器的核心数据结构。

```c
typedef struct {
    uint32_t p_type;    // 段类型
    uint32_t p_flags;   // 段标志（权限）
    uint64_t p_offset;  // 段在文件中的偏移
    uint64_t p_vaddr;   // 段在内存中的虚拟地址
    uint64_t p_paddr;   // 段的物理地址（通常忽略）
    uint64_t p_filesz;  // 段在文件中的大小
    uint64_t p_memsz;   // 段在内存中的大小（>= p_filesz，差值用0填充）
    uint64_t p_align;   // 对齐要求
} Elf64_Phdr;
```

**关键段类型（p_type）：**

| 类型 | 值 | 说明 |
|------|-----|------|
| PT_NULL | 0 | 未使用 |
| PT_LOAD | 1 | 可加载段，加载到内存 |
| PT_DYNAMIC | 2 | 动态链接信息 |
| PT_INTERP | 3 | 解释器路径（如/lib64/ld-linux-x86-64.so.2） |
| PT_NOTE | 4 | 附加信息（ABI标记、build-id） |
| PT_PHDR | 6 | Program Header Table自身的位置 |
| PT_TLS | 7 | 线程局部存储模板 |
| PT_GNU_EH_FRAME | 0x6474e550 | 异常处理框架 |
| PT_GNU_STACK | 0x6474e551 | 栈权限（控制NX位） |
| PT_GNU_RELRO | 0x6474e552 | 重定位后只读区域 |

**PT_LOAD 段的逆向分析要点：**

一个典型的ELF可执行文件通常有2-3个PT_LOAD段：

1. **只读代码段**（R-X权限）：包含 `.text`、`.rodata`、`.init`、`.fini` 等

#### 17.3.3.5 Section Header（节区头）

Section Header描述了文件中的各个逻辑分区，主要用于链接阶段。一个ELF文件可以没有Section Header Table（如一些精心构造的shellcode），但不能没有Program Header Table（否则无法加载执行）。

```c

    uint64_t sh_addr;      // 节区在内存中的虚拟地址
```

#### 17.3.3.6 动态链接机制：PLT/GOT

PLT（Procedure Linkage Table，过程链接表）和GOT（Global Offset Table，全局偏移表）是ELF动态链接的核心机制，也是逆向分析和漏洞利用的关键结构。

**PLT/GOT的工作原理：**

```text
程序代码调用 printf()
       │
       ▼
┌─────────────────┐
│  .plt 节区       │  PLT桩代码
│  printf@plt:    │
│    jmp *GOT[N]  │──────────────┐
│    push index   │              │
│    jmp PLT[0]   │              ▼
└─────────────────┘     ┌─────────────────┐
                        │  .got.plt 节区    │  全局偏移表
                        │  GOT[N]:         │
                        │  第一次：→ PLT[0] │  指向解析器
                        │  解析后：→ printf │  指向真实函数
                        └─────────────────┘
```

**延迟绑定（Lazy Binding）流程：**

1. 程序第一次调用 `printf()`，跳转到 `printf@plt`
2. PLT桩代码跳转到GOT中对应的条目，此时GOT条目指向PLT[0]（解析器入口）
3. 解析器通过 `.dynamic` 段找到符号表，解析 `printf` 的真实地址
4. 将解析到的地址写入GOT[N]，后续调用直接通过GOT跳转

**逆向分析中的PLT/GOT利用：**

- **Hooking**：修改GOT条目可以劫持任意动态库函数
- **信息泄露**：GOT表内容在内存中明文存储
- **ROP攻击**：PLT条目可以作为ROP gadget使用
- **延迟加载分析**：观察GOT条目的变化可以追踪函数调用

#### 17.3.3.7 符号表与字符串表

ELF文件包含两套符号表：

| 表名 | 类型 | 内容 | 是否可strip |
|------|------|------|------------|
| .symtab | SHT_SYMTAB | 完整符号表，包含所有符号 | 是（strip会移除） |
| .dynsym | SHT_DYNSYM | 动态符号表，仅包含动态链接所需符号 | 否（保留） |

每个符号条目（Elf64_Sym）包含：

```c

    uint64_t st_size;   // 符号的大小

```

**符号绑定（st_info高4位）：**

| 绑定 | 值 | 说明 |
|------|-----|------|
| STB_LOCAL | 0 | 局部符号，仅本文件可见 |
| STB_GLOBAL | 1 | 全局符号，所有文件可见 |
| STB_WEAK | 2 | 弱符号，可被全局符号覆盖 |

**符号类型（st_info低4位）：**

| 类型 | 值 | 说明 |
|------|-----|------|
| STT_NOTYPE | 0 | 未指定类型 |
| STT_OBJECT | 1 | 数据对象（变量、数组） |
| STT_FUNC | 2 | 函数 |
| STT_SECTION | 3 | 节区符号 |
| STT_FILE | 4 | 源文件名 |
| STT_COMMON | 5 | 公共块 |

#### 17.3.3.8 ELF逆向分析实战

**Python脚本解析ELF结构：**

```python
from elftools.elf.elffile import ELFFile
import io

def analyze_elf(filepath):
    """全面分析ELF文件结构"""
    with open(filepath, 'rb') as f:
        elf = ELFFile(f)

        print(f"=== ELF分析: {filepath} ===")
        print(f"架构: {elf.header['e_machine']}")
        print(f"入口点: 0x{elf.header['e_entry']:x}")
        print(f"类型: {elf.header['e_type']}")

        # 分析程序头
        print(f"\n--- 程序头 ({elf.num_segments()} 个段) ---")
        for segment in elf.iter_segments():
            ptype = segment['p_type']
            flags = segment['p_flags']
            perm = ""
            perm += "R" if flags & 4 else "-"
            perm += "W" if flags & 2 else "-"
            perm += "X" if flags & 1 else "-"
            print(f"  {ptype:<16s} VA=0x{segment['p_vaddr']:012x} "
                  f"Size=0x{segment['p_memsz']:08x} [{perm}]")

            # 特殊处理PT_INTERP
            if ptype == 'PT_INTERP':
                interp = segment.get_interp_name()
                print(f"    解释器: {interp}")

        # 分析节区
        print(f"\n--- 节区 ({elf.num_sections()} 个) ---")
        for section in elf.iter_sections():
            stype = section['sh_type']
            print(f"  {section.name:<20s} Type={stype:<12s} "
                  f"Addr=0x{section['sh_addr']:012x} "
                  f"Size=0x{section['sh_size']:08x}")

        # 分析动态符号
        dynsym = elf.get_section_by_name('.dynsym')
        if dynsym:
            print(f"\n--- 动态符号 ({dynsym.num_symbols()} 个) ---")
            for sym in dynsym.iter_symbols():
                if sym['st_info']['type'] == 'STT_FUNC' and sym['st_value']:
                    print(f"  {sym.name}: 0x{sym['st_value']:012x} "
                          f"size={sym['st_size']}")

        # 分析动态段（依赖库）
        dynamic = elf.get_section_by_name('.dynamic')
        if dynamic:
            print(f"\n--- 动态依赖 ---")
            for tag in dynamic.iter_tags():
                if tag['d_tag'] == 'DT_NEEDED':
                    print(f"  需要: {tag.needed}")

analyze_elf("/usr/bin/ls")
```

### 17.3.4 Mach-O格式（macOS/iOS）

Mach-O（Mach Object）是NeXTSTEP操作系统遗留下来的可执行文件格式，现用于macOS、iOS、iPadOS、watchOS和tvOS。其设计哲学与PE/ELF不同，采用了更灵活的Load Commands机制来描述文件元数据。

#### 17.3.4.1 Mach-O文件总体结构

```text
┌───────────────────────────────┐
│  Mach-O Header (32字节)        │  魔数、CPU类型、文件类型、Load Commands数量
├───────────────────────────────┤
│  Load Commands                │  文件布局和链接信息的描述
│  ├─ LC_SEGMENT_64             │  定义一个内存段及其包含的节区
│  ├─ LC_DYLD_INFO_ONLY        │  动态链接器信息
│  ├─ LC_LOAD_DYLINKER          │  动态链接器路径
│  ├─ LC_SYMTAB                │  符号表位置
│  ├─ LC_DYSYMTAB              │  动态符号表位置
│  ├─ LC_LOAD_DYLIB            │  依赖的动态库
│  ├─ LC_MAIN                  │  主线程入口点
│  ├─ LC_SOURCE_VERSION        │  源码版本
│  ├─ LC_UUID                  │  唯一标识符
│  ├─ LC_CODE_SIGNATURE        │  代码签名
│  ├─ LC_ENCRYPTION_INFO_64    │  加密信息（App Store应用）
│  └─ ...                      │  其他命令
├───────────────────────────────┤
│  __TEXT segment               │
│  ├─ __text section            │  机器码
│  ├─ __stubs section           │  动态库函数桩（类似PLT）
│  ├─ __stub_helper section     │  桩辅助代码
│  ├─ __cstring section         │  C字符串常量
│  ├─ __const section           │  常量数据
│  ├─ __unwind_info section     │  异常展开信息
│  └─ __objc_methname section   │  Objective-C方法名
├───────────────────────────────┤
│  __DATA segment               │
│  ├─ __nl_symbol_ptr           │  非惰性符号指针（类似GOT）
│  ├─ __la_symbol_ptr           │  惰性符号指针（延迟绑定）
│  ├─ __objc_classlist          │  ObjC类列表
│  ├─ __objc_selrefs            │  ObjC选择器引用

├───────────────────────────────┤
```

#### 17.3.4.3 Load Commands

Load Commands是Mach-O格式最独特的设计。与PE/ELF使用固定的数据目录不同，Mach-O将所有元数据以"命令"链表的形式存储，提供了极大的灵活性。

**LC_SEGMENT_64** 是最重要的Load Command，定义了一个可加载的内存段：

```c
typedef struct {
    uint32_t  cmd;          // LC_SEGMENT_64 (0x19)
    uint32_t  cmdsize;      // 命令大小（包含section headers）
    char      segname[16];  // 段名（如 __TEXT, __DATA）
    uint64_t  vmaddr;       // 段的虚拟地址
    uint64_t  vmsize;       // 段的虚拟大小
    uint64_t  fileoff;      // 段在文件中的偏移
    uint64_t  filesize;     // 段在文件中的大小
    vm_prot_t maxprot;      // 最大保护属性
    vm_prot_t initprot;     // 初始保护属性
    uint32_t  nsects;       // 包含的节区数量
    uint32_t  flags;        // 标志
} segment_command_64;
```

**常见段及其用途：**

| 段名 | 权限 | 包含的节区 | 说明 |
|------|------|-----------|------|
| __TEXT | r-x | __text, __stubs, __stub_helper, __cstring, __const, __unwind_info | 代码和只读数据 |
| __DATA | rw- | __nl_symbol_ptr, __la_symbol_ptr, __objc_*, __got | 可读写数据 |
| __DATA_CONST | r-- | __objc_classlist, __objc_const | 常量数据（重定位后只读） |
| __LINKEDIT | r-- | 符号表、字符串表、代码签名等 | 链接器专用数据 |
| __OBJC | r-- | __objc_imageinfo, __objc_methnames | Objective-C元数据 |

#### 17.3.4.4 代码签名与FairPlay

macOS和iOS强制要求代码签名。Mach-O通过 `LC_CODE_SIGNATURE` Load Command指向嵌入的签名数据：

```text
代码签名结构：
┌────────────────────────────┐
│  Super Blob Header          │  魔数 0xFADE0CC0
├────────────────────────────┤
│  Code Directory Blob        │  代码哈希（每页的SHA-256）
│  ├─ 签名者身份              │
│  ├─ 程序标识符              │
│  ├─ 团队标识符              │
│  └─ 每4KB页面的哈希值       │
├────────────────────────────┤
│  CMS Signature Blob         │  PKCS#7签名
├────────────────────────────┤
│  Entitlements Blob          │  权限声明（沙箱权限等）
└────────────────────────────┘
```

**逆向分析代码签名的要点：**

- iOS应用的FairPlay DRM加密层通过 `LC_ENCRYPTION_INFO_64` 命令描述
- `cryptid` 字段为1表示已加密，0表示未加密
- 解密需要在越狱设备上进行，或使用 `frida-ios-dump` 等工具

#### 17.3.4.5 Universal Binary（胖二进制）

Universal Binary（也称Fat Binary）是Apple的多架构打包方案，将多个架构的Mach-O文件合并为一个文件：

```text
┌────────────────────────────┐
│  Fat Header                  │  魔数 0xCAFEBABE
│  ├─ magic: 0xCAFEBABE       │
│  ├─ nfat_arch: 2            │  包含2个架构
│  ├─ Arch 1: arm64           │  偏移/大小/对齐
│  └─ Arch 2: x86_64          │  偏移/大小/对齐
├────────────────────────────┤
│  Mach-O #1 (arm64)          │  ARM64架构的完整Mach-O
├────────────────────────────┤
│  Mach-O #2 (x86_64)         │  x86_64架构的完整Mach-O
└────────────────────────────┘
```

**解析Fat Binary：**

```python
import struct

FAT_MAGIC = 0xCAFEBABE
FAT_MAGIC_64 = 0xCAFEBABF

def parse_fat_header(filepath):
    """解析Fat Binary头部"""
    with open(filepath, 'rb') as f:
        magic = struct.unpack(">I", f.read(4))[0]

        if magic == FAT_MAGIC:
            nfat = struct.unpack(">I", f.read(4))[0]
            print(f"Fat Binary, {nfat}个架构:")
            for i in range(nfat):
                cputype, cpusubtype, offset, size, align = struct.unpack(
                    ">IIIII", f.read(20)
                )
                print(f"  架构 {i+1}: cputype={cputype} "
                      f"offset=0x{offset:x} size=0x{size:x}")
        elif magic in (0xFEEDFACE, 0xFEEDFACF):
            print("单架构Mach-O文件")
        else:
            print("不是Mach-O文件")
```

#### 17.3.4.6 dyld 与动态链接

macOS使用dyld（the dynamic link editor）作为动态链接器，其工作机制与Linux的ld.so有所不同。

**dyld的关键特性：**

- **共享缓存（Shared Cache）**：macOS将系统框架预链接为一个大缓存文件，加速加载
- **环境变量限制**：macOS 10.12+限制了 `DYLD_INSERT_LIBRARIES` 等环境变量的使用
- **interposing**：dyld支持符号插入（symbol interposing），可以在不修改GOT的情况下hook函数
- **链式修复（Chained Fixups）**：iOS 15+引入的新重定位格式，减小二进制大小

**dyld interposing 示例：**

```c
// 通过dyld interposing hook open函数
#include <unistd.h>

// 保存原始函数指针
static int (*original_open)(const char *, int, ...) = NULL;

// 我们的替代函数
int my_open(const char *path, int flags, ...) {
    printf("[HOOK] open(%s, %d)\n", path, flags);
    return original_open(path, flags);
}

// 声明interpose
__attribute__((used))
static struct {
    const void *replacement;
    const void *original;
} _interpose_open
__attribute__((section("__DATA,__interpose"))) = {
    (const void *)my_open,
    (const void *)open
};
```

### 17.3.5 三大格式的横向对比

为了建立全局认知，下面将PE、ELF和Mach-O三种格式进行系统对比：

#### 17.3.5.1 设计哲学对比

| 维度 | PE | ELF | Mach-O |
|------|-----|-----|--------|
| 起源 | COFF（UNIX System V） | UNIX System V ABI | NeXTSTEP/Mach微内核 |
| 首次发布 | 1993 (Windows NT) | 1993 (System V Release 4) | 1989 (NeXTSTEP) |
| 设计目标 | Windows系统兼容 | 跨UNIX可移植性 | Apple生态系统 |
| 头部结构 | 固定位置，数组索引 | 固定位置，节名引用 | 链表式的Load Commands |
| 扩展性 | 有限（固定16个数据目录） | 较好（可自定义节区） | 极佳（可添加任意Load Command） |
| 多架构支持 | Fat Binary（非官方） | 胖ELF（实验性） | Universal Binary（原生支持） |

#### 17.3.5.2 核心数据结构对比

| 结构 | PE | ELF | Mach-O |
|------|-----|-----|--------|
| 文件头 | DOS Header + PE Signature | ELF Header | Mach-O Header |
| 可加载描述 | Section Header | Program Header (PT_LOAD) | LC_SEGMENT_64 |
| 链接描述 | Section Header + Data Directory | Section Header | Load Commands |
| 入口点 | AddressOfEntryPoint (RVA) | e_entry (VA) | LC_MAIN 命令中的 entryoff |
| 动态链接 | Import Table + IAT | PLT + GOT | __stubs + __la_symbol_ptr |
| 重定位 | .reloc节 | .rel/.rela节 | scattered relocations |
| 符号信息 | 无标准符号表（PDB外部） | .symtab + .dynsym | symtab command |
| 调试信息 | Debug Directory → PDB | DWARF (.debug_*) | DWARF + dSYM |
| 代码签名 | Security Directory | .note段（GNU） | LC_CODE_SIGNATURE |

#### 17.3.5.3 安全机制对比

| 安全机制 | PE (Windows) | ELF (Linux) | Mach-O (macOS) |
|---------|-------------|-------------|----------------|
| 地址随机化 | ASLR (DllCharacteristics) | ASLR (PIE) | PIE (MH_PIE) |
| 数据执行保护 | DEP/NX (DllCharacteristics) | NX (PT_GNU_STACK) | DEP (MH_NO_HEAP_EXEC) |
| 栈保护 | /GS (编译器) | Stack Canary (-fstack-protector) | Stack Canary |
| 控制流保护 | CFG (Guard_CF标志) | CFI (-fsanitize=cfi) | Control Flow Integrity |
| 代码签名 | Authenticode | IMA/EVM | 代码签名 + 公证 |
| 沙箱 | AppContainer | seccomp + namespaces | App Sandbox |
| 完整性检查 | 驱动签名强制 | dm-verity (Android) | SIP (系统完整性保护) |

### 17.3.6 高级话题：格式层面的反逆向技术

了解可执行文件格式的攻击者会利用格式结构本身来阻碍逆向分析。以下是常见的格式层面反逆向技术：

#### 17.3.6.1 PE文件的反逆向手段

**1. 节区名混淆：**
将标准节区名（.text、.data）替换为随机字符串或合法节区名（.rsrc），误导分析工具。

**2. 节区属性篡改：**
- 将代码节设为不可执行（清除IMAGE_SCN_MEM_EXECUTE）
- 将数据节设为可写可执行，实际存放自解密代码
- 使用分段属性不一致来混淆反汇编器

**3. 导入表隐藏：**
- 使用延迟导入（Delay Import）代替普通导入
- 运行时通过 `LoadLibrary` + `GetProcAddress` 动态解析API
- 使用序号导入代替名称导入
- 手动清空导入表，运行时重建

**4. PE结构异常：**
- 将入口点指向非标准位置（如资源节或重定位节）
- 使用Overlapping Sections（节区重叠）隐藏数据
- 在DOS Stub中嵌入额外数据
- 增大 `SizeOfHeaders` 以隐藏数据在头部区域

**5. TLS回调隐藏：**
```text
正常执行流程：加载器 → OEP (入口点) → main
TLS流程：    加载器 → TLS回调1 → TLS回调2 → OEP → main
```
壳程序和恶意软件常在TLS回调中执行反调试检查或解密代码，逆向工程师如果只关注OEP会完全错过关键逻辑。

#### 17.3.6.2 ELF文件的反逆向手段

**1. Section Header删除：**
```bash
# strip命令移除节区头和符号表
strip --remove-section=.symtab --remove-section=.strtab binary
# 或者直接截断节区头表
# 这不影响程序运行（加载器使用Program Header），但阻碍静态分析工具
```

**2. 自定义Linker Script：**
通过自定义链接脚本可以创建非标准的节区布局，甚至创建重叠的Program Header。

**3. PT_NOTE利用：**
将shellcode或加密密钥藏在PT_NOTE段中，然后通过代码修改该内存区域的权限并执行。

**4. GOT覆写攻击：**
修改GOT条目使反汇编器对函数调用的分析产生误导。

#### 17.3.6.3 Mach-O的反逆向手段

**1. 代码签名对抗：**
修改LC_CODE_SIGNATURE使其指向无效签名，使静态分析工具报错。

**2. FAT Binary隐藏：**
在FAT头部的arch数组中添加无效架构描述，使工具解析出错，但不影响实际架构的运行。

**3. ObjC元数据混淆：**
Objective-C运行时的类名、方法名存储在Mach-O文件的特殊节区中，对这些元数据进行混淆可以有效对抗基于ObjC反射的分析。

#### 17.3.7.3 使用LIEF进行跨格式统一分析

LIEF（Library to Instrument Executable Formats）是唯一一个用统一API支持PE/ELF/Mach-O三种格式的库：

```python
import lief

def universal_analyze(filepath):
    """使用LIEF进行跨格式统一分析"""
    binary = lief.parse(filepath)

    if binary is None:
        print("无法解析文件")
        return

    print(f"文件: {filepath}")

    # 判断格式
    if isinstance(binary, lief.PE.Binary):
        print("格式: PE (Windows)")
        print(f"入口点: 0x{binary.optional_header.addressof_entrypoint:08x}")
        print(f"子系统: {binary.optional_header.subsystem}")
        if binary.has_imports:
            print(f"\n导入DLL ({len(binary.imports)} 个):")
            for lib in binary.imports:
                print(f"  {lib.name} ({len(lib.entries)} 个函数)")

    elif isinstance(binary, lief.ELF.Binary):
        print("格式: ELF (Linux/Unix)")
        print(f"入口点: 0x{binary.header.entrypoint:016x}")
        print(f"类型: {binary.header.file_type}")
        print(f"\n依赖库:")
        for lib in binary.libraries:
            print(f"  {lib}")

    elif isinstance(binary, lief.MachO.Binary):
        print("格式: Mach-O (macOS/iOS)")
        fat = binary.fat_binary if hasattr(binary, 'fat_binary') else None
        print(f"入口点: 0x{binary.header.entrypoint:016x}")
        print(f"CPU类型: {binary.header.cpu_type}")
        if binary.has_dyld_info:
            print(f"\n依赖库:")
            for lib in binary.libraries:
                print(f"  {lib.name}")

    # 通用分析：列出所有段/节
    print(f"\n--- 段/节区 ---")
    for section in binary.sections:
        print(f"  {section.name:<20s} "
              f"VA=0x{section.virtual_address:012x} "
              f"Size=0x{section.size:08x}")

universal_analyze("/usr/bin/ls")
```

### 17.3.8 常见误区与纠正

| 误区 | 正确理解 |
|------|---------|
| "PE就是Windows，ELF就是Linux" | ELF也被Android、FreeBSD等使用；Windows的WSL1曾使用PE封装ELF；COFF变体广泛用于各种嵌入式系统 |
| "节区和段是同一回事" | 在ELF中，Segment（段）和Section（节区）是不同的概念：段用于加载，节区用于链接。一个段可以包含多个节区 |
| "strip后的ELF无法分析" | strip只移除.symtab/.strtab，.dynsym/.dynstr保留，动态符号仍然可用 |
| "入口点就是main函数" | 入口点是_start，它初始化运行时环境后才调用__libc_start_main，后者才调用main |
| "PIE可执行文件是共享库" | PIE的e_type确实是ET_DYN，但通过PT_INTERP和入口点可以区分；可以正常执行 |
| "导入表为空就是没调用API" | 可能使用动态加载（LoadLibrary/dlopen）或系统调用直接调用 |
| "代码签名保证安全" | 代码签名保证代码未被篡改，但不保证代码本身是安全的 |
| "FAT Binary的每个架构完全独立" | 共享的LC_SYMTAB数据可能被多个架构引用，修改一个可能影响另一个 |
