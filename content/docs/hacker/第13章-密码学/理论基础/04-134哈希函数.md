## 13.4 哈希函数

哈希函数（Hash Function）是密码学中最基础、最广泛应用的原语之一。它将任意长度的输入映射为固定长度的输出（称为哈希值、摘要或指纹），是数字签名、消息认证、密码存储、区块链、完整性校验等安全机制的基石。理解哈希函数的原理、安全性质、攻击手段和正确用法，是每一位安全从业者的必修课。

### 13.4.1 哈希函数的基本性质

哈希函数将任意长度的消息 $m$ 映射为固定长度的摘要 $h = H(m)$。一个密码学安全的哈希函数必须满足以下性质：

| 性质 | 数学定义 | 通俗解释 |
|------|---------|---------|
| 确定性 | $H(m_1) = H(m_2)$ 当且仅当 $m_1 = m_2$（理想情况） | 相同输入永远产生相同输出 |
| 高效性 | 计算 $H(m)$ 的时间复杂度为 $O(|m|)$ | 对任意输入都能快速计算 |
| 抗原像攻击（Preimage Resistance） | 给定 $y$，难以找到 $m$ 使得 $H(m) = y$ | 知道哈希值，无法反推出原文 |
| 抗第二原像攻击（Second Preimage Resistance） | 给定 $m_1$，难以找到 $m_2 \neq m_1$ 使得 $H(m_1) = H(m_2)$ | 给定一个原文，无法伪造另一个相同哈希的原文 |
| 抗碰撞性（Collision Resistance） | 难以找到任意 $m_1 \neq m_2$ 使得 $H(m_1) = H(m_2)$ | 无法找到任意两个不同输入产生相同哈希 |

**抗原像攻击 vs 抗第二原像攻击 vs 抗碰撞性**——这三者的区别是初学者最容易混淆的地方：

- **抗原像攻击**：给你一个哈希值，你找不到任何一个能产生这个哈希值的输入。这是单向性的直接体现。
- **抗第二原像攻击**：给你一个具体的输入 $m_1$，你找不到另一个不同的输入 $m_2$ 使得它们的哈希值相同。攻击者无法对已知消息进行替换。
- **抗碰撞性**：你能找到任意两个不同的输入碰撞即可，不需要指定哪一个。这是最强的要求——因为生日悖论使得寻找碰撞比寻找原像容易得多。

三者的安全强度递减关系：**抗碰撞性 → 抗第二原像攻击 → 抗原像攻击**。如果一个哈希函数满足抗碰撞性，它自动满足抗第二原像攻击（但不自动满足抗原像攻击）。实际上，对于输出长度为 $n$ 位的哈希函数：

- 寻找原像的最优攻击复杂度约为 $2^n$（穷举）
- 寻找第二原像的最优攻击复杂度约为 $2^n$（穷举）
- 寻找碰撞的最优攻击复杂度约为 $2^{n/2}$（生日攻击）

这意味着一个 128 位哈希函数的抗碰撞性只有 64 位安全强度——这也是 MD5（128 位输出）在碰撞攻击面前迅速沦陷的根本原因。

### 13.4.2 哈希函数的内部结构

#### Merkle-Damgård 结构

大多数经典哈希函数（MD5、SHA-1、SHA-2）都基于 Merkle-Damgård（MD）结构。其核心思想是：

1. **消息填充**：将输入消息填充到块大小的整数倍（通常附加长度信息）
2. **分块处理**：将填充后的消息分割为固定大小的块 $m_1, m_2, \dots, m_k$
3. **迭代压缩**：使用压缩函数 $f$ 逐块更新内部状态 $h_i = f(h_{i-1}, m_i)$
4. **输出**：最终状态 $h_k$ 即为哈希值

初始向量（IV）是算法规定的固定常量，作为 $h_0$ 的值。

```text
    IV      m_1      m_2      m_3
     |       |        |        |
     v       v        v        v
     [  f  ]->[  f  ]->[  f  ]--> 输出
```

MD 结构的安全性依赖于一个关键定理：**如果压缩函数 $f$ 是抗碰撞的，那么基于 MD 结构的哈希函数也是抗碰撞的**。

但 MD 结构有一个固有缺陷——**长度扩展攻击**（Length Extension Attack），这是理解许多实际安全漏洞的关键。

#### 海绵结构（Sponge Construction）

SHA-3 采用了完全不同的 Keccak 海绵结构，由两个阶段组成：

1. **吸收阶段（Absorbing）**：消息块依次与内部状态进行 XOR，然后通过置换函数 $f$ 更新状态
2. **挤压阶段（Squeezing）**：从状态中依次提取输出块，每提取一块就再应用一次 $f$

内部状态被分为两部分：**速率**（rate，$r$ 位）和**容量**（capacity，$c$ 位），其中 $r + c$ 是状态总宽度。安全性取决于容量 $c$：抗碰撞强度为 $c/2$ 位。

海绵结构的优势：
- 天然免疫长度扩展攻击
- 可以输出任意长度（可变输出哈希）
- 统一框架支持哈希、MAC、流密码、伪随机数生成等多种功能

### 13.4.3 常见哈希算法详解

#### MD5（Message Digest 5）

| 参数 | 值 |
|------|-----|
| 输出长度 | 128 位（16 字节） |
| 块大小 | 512 位 |
| 轮数 | 4 轮 × 16 步 = 64 步 |
| 结构 | Merkle-Damgård |
| 安全状态 | **已彻底破解** |

MD5 由 Ronald Rivest 于 1991 年设计。2004 年，王小云教授团队展示了 MD5 的实际碰撞攻击，可在数秒内找到碰撞对。2008 年，研究人员利用 MD5 碰撞伪造了合法的 SSL 证书，证明了实际威胁。

**MD5 的致命缺陷**：碰撞攻击的实际可行性意味着攻击者可以构造两个内容不同但哈希值完全相同的文件。在安全场景中，这等同于毁灭性打击。

**MD5 仍可用于**：非安全场景的校验和（如检测文件传输中的意外损坏），但绝不能用于防篡改、数字签名、密码存储等安全用途。

#### SHA-1（Secure Hash Algorithm 1）

| 参数 | 值 |
|------|-----|
| 输出长度 | 160 位（20 字节） |
| 块大小 | 512 位 |
| 轮数 | 80 步 |
| 结构 | Merkle-Damgård |
| 安全状态 | **已不安全** |

SHA-1 由 NSA 设计，1995 年发布。理论上 160 位输出应提供 80 位抗碰撞强度，但密码分析进展远超预期。2017 年，Google 与 CWI Amsterdam 合作完成了 SHAttered 攻击，实际构造了两个不同的 PDF 文件具有相同的 SHA-1 值，总计算量约 $2^{63}$ 次 SHA-1 压缩（远低于理论的 $2^{80}$）。

#### SHA-2 家族

SHA-2 由 NSA 设计，2001 年发布，包含多个变体：

| 变体 | 输出长度 | 块大小 | 安全强度 | 典型应用 |
|------|---------|--------|---------|---------|
| SHA-224 | 224 位 | 512 位 | 112 位 | 资源受限场景 |
| SHA-256 | 256 位 | 512 位 | 128 位 | 比特币、TLS、数字证书 |
| SHA-384 | 384 位 | 1024 位 | 192 位 | 高安全需求 |
| SHA-512 | 512 位 | 1024 位 | 256 位 | 最高安全需求 |
| SHA-512/256 | 256 位 | 1024 位 | 128 位 | SHA-256 的 64 位优化替代 |

SHA-256 是目前使用最广泛的哈希算法。比特币的工作量证明（PoW）要求矿工找到一个 nonce 使得区块头的 SHA-256 双哈希值小于目标值，这一过程消耗的算力构成了比特币的安全基础。

SHA-2 与 SHA-1 内部结构相似但更保守：更多的轮数、更大的消息字、更复杂的轮函数。截至目前，SHA-2 没有任何已知的实际攻击。

#### SHA-3（Keccak）

| 参数 | 值 |
|------|-----|
| 输出长度 | 可变（224/256/384/512 位，或任意长度） |
| 状态宽度 | 1600 位（5 × 5 × 64 位矩阵） |
| 轮数 | 24 轮 |
| 结构 | 海绵结构 |
| 安全状态 | **安全** |

SHA-3 于 2015 年标准化（FIPS 202），源自 Keccak 算法在 NIST 哈希函数竞赛中胜出。它与 SHA-2 设计完全不同，提供了算法多样性——即使 SHA-2 未来被攻破，SHA-3 仍可能安全（因为攻击手法不太可能同时适用于两种完全不同的结构）。

SHA-3 的可扩展输出功能（SHAKE128/SHAKE256）特别有价值，可以生成任意长度的输出，适用于密钥派生、随机数生成等场景。

#### BLAKE2 和 BLAKE3

BLAKE2 是 2012 年发布的哈希函数，源自 SHA-3 竞赛 finalist BLAKE，在保持高安全性的同时，速度比 SHA-256 快约 3 倍，甚至比 MD5 更快。

| 变体 | 输出长度 | 特点 |
|------|---------|------|
| BLAKE2b | 最高 512 位 | 64 位平台优化，通用哈希 |
| BLAKE2s | 最高 256 位 | 32 位平台优化，嵌入式场景 |
| BLAKE2bp | 512 位 | BLAKE2b 的并行版本 |
| BLAKE2sp | 256 位 | BLAKE2s 的并行版本 |

BLAKE3（2020 年）更进一步，采用 Merkle 树结构实现高度并行化，在多核 CPU 上速度极快（比 SHA-256 快 14 倍以上），且支持可变长度输出、密钥哈希和派生密钥功能。

```bash
# BLAKE2 哈希计算（需要 b2sum 工具）
echo -n "Hello, World!" | b2sum
# 或使用 Python
python3 -c "import hashlib; print(hashlib.blake2b(b'Hello, World!').hexdigest())"
```

### 13.4.4 密码哈希函数：专门为密码存储设计

普通哈希函数（SHA-256 等）速度极快——这恰恰是密码存储场景中的致命弱点。攻击者使用 GPU 或 ASIC 可以每秒尝试数十亿次 SHA-256 计算，暴力破解密码。因此，密码存储需要专门设计的**慢速哈希函数**，通过可控的计算成本和内存消耗来抵御暴力破解。

#### bcrypt

```bash
# 使用 htpasswd 生成 bcrypt 密码哈希
htpasswd -nbBC 12 username MyPassword
# 输出: username:$2y$12$...(60 字符)
```

- 基于 Blowfish 密码的密钥扩展
- 成本因子（cost factor）可调，范围 4-31，每增加 1 计算量翻倍
- 自动包含盐值（salt）
- 输出固定 60 字符（含算法标识、成本、盐、哈希）
- 推荐成本因子：12（2024 年基准，约 250ms 计算时间）

#### scrypt

```python
import hashlib
# Python 3.6+ 内置支持
dk = hashlib.scrypt(
    password=b'my_password',
    salt=b'random_salt_16b',
    n=16384,      # CPU/内存成本参数（必须是2的幂）
    r=8,          # 块大小
    p=1,          # 并行度
    dklen=32      # 输出长度
)
```

- 2009 年由 Colin Percival 设计，最初用于 Tarsnap 备份系统
- 核心优势：**内存硬**（memory-hard），需要大量 RAM 才能高效计算
- 这使得 GPU 和 ASIC 攻击的成本大幅提高（GPU 的显存带宽成为瓶颈）
- 参数：$N$（CPU/内存成本）、$r$（块大小）、$p$（并行度）

#### Argon2（推荐）

Argon2 是 2015 年密码哈希竞赛（PHC）的获胜者，当前最推荐的密码哈希算法。

```python
# 安装: pip install argon2-cffi
from argon2 import PasswordHasher

ph = PasswordHasher(
    time_cost=3,        # 迭代次数
    memory_cost=65536,  # 内存使用量（KB），此处为 64MB
    parallelism=4,      # 并行线程数
    hash_len=32,        # 输出长度
    salt_len=16         # 盐长度
)

# 哈希密码
hash = ph.hash("my_secure_password")
# 输出: $argon2id$v=19$m=65536,t=3,p=4$...(完整哈希)

# 验证密码
try:
    ph.verify(hash, "my_secure_password")
    print("密码正确")
except Exception:
    print("密码错误")

# 检查是否需要重新哈希（参数升级后）
if ph.check_needs_rehash(hash):
    hash = ph.hash("my_secure_password")  # 用新参数重新哈希
```

Argon2 有三个变体：

| 变体 | 特点 | 适用场景 |
|------|------|---------|
| Argon2d | 数据依赖内存访问，抗 GPU/ASIC 更强 | 防离线破解（区块链、加密货币） |
| Argon2i | 数据独立内存访问，抗侧信道更强 | 防侧信道攻击（服务器端密码存储） |
| Argon2id | 先 i 后 d 的混合模式 | **通用推荐**（兼顾两者优势） |

**密码哈希函数选择总结**：

| 场景 | 推荐算法 | 参数建议 |
|------|---------|---------|
| Web 应用密码存储 | Argon2id | m=65536, t=3, p=4 |
| 嵌入式/低内存环境 | bcrypt | cost=12 |
| 需要内存硬特性 | scrypt | N=16384, r=8, p=1 |
| 遗留系统兼容 | bcrypt | cost≥10 |

### 13.4.5 消息认证码（MAC）

消息认证码结合了哈希函数（或分组密码）与密钥，用于同时验证消息的**完整性**和**真实性**——不仅消息没有被篡改，而且确实来自持有正确密钥的发送方。

#### HMAC 构造

HMAC（Hash-based MAC）是最广泛使用的 MAC 构造方式，定义于 RFC 2104：

```text
HMAC(K, m) = H((K' ⊕ opad) || H((K' ⊕ ipad) || m))
```

其中：
- $K'$ 是密钥 $K$ 经过处理后的值（短于块大小则填充零，长于块大小则先哈希）
- `ipad` = `0x36` 重复块大小次
- `opad` = `0x5C` 重复块大小次
- $H$ 是底层哈希函数

这种双重嵌套结构的安全性已被严格证明：**只要底层哈希函数满足抗原像攻击（弱于抗碰撞性的要求），HMAC 就是安全的 MAC**。

```python
import hmac
import hashlib

key = b'super_secret_key_32bytes!!!!!'
message = b'Transfer $1000 to Alice'

# 计算 HMAC-SHA256
mac = hmac.new(key, message, hashlib.sha256).hexdigest()
print(f"HMAC: {mac}")

# 验证
expected_mac = hmac.new(key, message, hashlib.sha256).hexdigest()
is_valid = hmac.compare_digest(mac, expected_mac)  # 恒定时间比较！
print(f"验证: {'通过' if is_valid else '失败'}")
```

**关键细节**：验证 HMAC 时必须使用 `hmac.compare_digest()` 而非 `==` 运算符。普通字符串比较会在第一个不匹配字节处提前返回，泄露时间信息（时序攻击）。`compare_digest` 执行恒定时间比较，无论是否匹配都花费相同时间。

#### KMAC（基于 SHA-3 的 MAC）

KMAC（Keccak MAC）是 SHA-3 标准中定义的 MAC 构造，直接基于海绵结构，比 HMAC 更简洁：

```python
# KMAC 需要额外的库支持，如 pysha3
# 原理：将密钥作为海绵结构的输入前缀
```

KMAC 的优势在于可以输出任意长度，且不需要 HMAC 那样的双重嵌套结构。

#### GMAC 和 Poly1305

在需要同时加密和认证的场景中，通常使用 AEAD 模式（如 AES-GCM、ChaCha20-Poly1305），其中 GMAC 和 Poly1305 是常用的 MAC 组件：

- **GMAC**：Galois/Counter Mode 的认证部分，基于伽罗瓦域上的多项式求值
- **Poly1305**：Daniel Bernstein 设计的高速 MAC，基于模运算

### 13.4.6 哈希函数的攻击方法

#### 生日攻击（Birthday Attack）

生日悖论：在 23 人中，至少有两人生日相同的概率超过 50%。推广到哈希函数：对于输出长度为 $n$ 位的哈希函数，约需 $2^{n/2}$ 次哈希计算就能找到碰撞。

```text
所需样本数 ≈ √(π/2 × 2^n) ≈ 1.177 × 2^(n/2)
```

这是哈希函数碰撞攻击的理论下限。实际含义：

| 哈希函数 | 输出长度 | 生日攻击复杂度 | 安全强度评估 |
|---------|---------|--------------|------------|
| MD5 | 128 位 | $2^{64}$ | 已被实际攻破（远低于此复杂度） |
| SHA-1 | 160 位 | $2^{80}$ | 已被实际攻破（$2^{63}$） |
| SHA-256 | 256 位 | $2^{128}$ | 安全 |
| SHA-512 | 512 位 | $2^{256}$ | 安全 |

#### 长度扩展攻击（Length Extension Attack）

基于 Merkle-Damgård 结构的哈希函数都容易受到长度扩展攻击。攻击场景：

假设你有 $H(m)$ 和 $|m|$（消息长度），但不知道 $m$ 的内容。你可以计算 $H(m || padding || m')$，其中 $padding$ 是标准填充，$m'$ 是你选择的附加数据——**无需知道 $m$ 的内容**。

这在以下场景中构成威胁：

```text
# 不安全的 MAC 构造
MAC = H(secret || message)        # 易受长度扩展攻击！

# 攻击者已知 MAC 和 message 长度
# 可以计算 H(secret || message || padding || evil_data)
# 而无需知道 secret
```

**受影响的算法**：MD5、SHA-1、SHA-256、SHA-512（所有 MD 结构的哈希函数）

**不受影响的算法**：SHA-3（海绵结构）、BLAKE2（HAIFA 结构）、HMAC（双重嵌套结构）

```python
# Python 演示长度扩展攻击原理
import struct

def sha256_pad(message_len):
    """计算 SHA-256 标准填充"""
    # 消息长度（位）
    bit_len = message_len * 8
    # 填充: 1 + 若干 0 + 8 字节长度
    padding = b'\x80'
    # 填充到 (message_len + padding) ≡ 448 (mod 512)
    pad_len = (55 - message_len) % 64
    padding += b'\x00' * pad_len
    # 附加 64 位大端长度
    padding += struct.pack('>Q', bit_len)
    return padding

# 攻击者知道:
# - H(secret || original_message) 的值
# - original_message 的长度
# 可以计算 H(secret || original_message || padding || new_data)
# 而不需要知道 secret
```

**防御方法**：
1. 使用 HMAC 而非 `H(key || message)`
2. 使用 SHA-3 或 BLAKE2（不受此攻击影响）
3. 始终验证 MAC 后再处理消息

#### 彩虹表攻击（Rainbow Table Attack）

彩虹表是针对密码哈希的预计算攻击工具。原理：

1. 预先计算大量密码的哈希值，存储为"哈希→密码"的映射表
2. 遇到目标哈希值时，直接查表得到原始密码
3. 通过"缩减链"技术压缩存储空间

| 彩虹表规模 | 覆盖范围 | 存储空间 |
|-----------|---------|---------|
| LM 哈希 | 所有 7 字符 ASCII | ~64 GB |
| NTLM 哈希 | 所有 1-9 字符 ASCII | ~300 GB |
| SHA-1 | 所有 1-10 字符 | ~TB 级 |

**防御彩虹表的核心方法：加盐（Salting）**：

```python
import os
import hashlib

def hash_password_bad(password):
    """不安全：无盐，易受彩虹表攻击"""
    return hashlib.sha256(password.encode()).hexdigest()

def hash_password_good(password):
    """较好：随机盐，每个密码独立"""
    salt = os.urandom(32)  # 256 位随机盐
    hash_val = hashlib.pbkdf2_hmac('sha256', password.encode(), salt, 100000)
    return salt.hex() + ':' + hash_val.hex()

def verify_password(stored, password):
    """验证密码"""
    salt_hex, hash_hex = stored.split(':')
    salt = bytes.fromhex(salt_hex)
    hash_val = hashlib.pbkdf2_hmac('sha256', password.encode(), salt, 100000)
    return hash_val.hex() == hash_hex
```

盐的作用：
- 相同密码产生不同的哈希值（每个用户唯一的盐）
- 彩虹表需要针对每个盐值重新计算，成本等同于暴力破解
- 盐必须是密码学安全的随机数，长度至少 128 位
- 盐不需要保密，可以和哈希值一起存储

#### 多碰撞攻击（Multicollision Attack）

Joux（2004 年）证明，对于基于 MD 结构的 $k$ 轮哈希函数，找到 $2^k$ 个多碰撞（$2^k$ 个消息全部碰撞）的复杂度仅为 $k \times 2^{n/2}$，而非预期的 $2^{(k-1) \times n/2}$。这表明级联多个哈希函数（如 `H1(x) || H2(x)`）并不能成倍增加安全性。

### 13.4.7 哈希函数的实际应用

#### 数字签名

数字签名的标准流程中，哈希函数扮演关键角色：

1. 发送方计算消息的哈希值 $h = H(m)$
2. 用私钥对哈希值签名 $sig = Sign_{sk}(h)$
3. 接收方用公钥验证签名，并独立计算消息哈希值进行比对

为什么签名哈希值而非消息本身？
- **效率**：RSA 签名速度很慢，签名 32 字节的哈希值远快于签名 1GB 的文件
- **安全性**：基于哈希的签名方案不依赖底层签名算法的抗碰撞性
- **兼容性**：RSA 等算法对输入长度有限制

#### 文件完整性校验


#### 区块链与工作量证明

比特币使用双重 SHA-256（SHA256d）：

```python
import hashlib

def sha256d(data):
    """比特币使用的双重 SHA-256"""
    return hashlib.sha256(hashlib.sha256(data).digest()).digest()

# 区块头结构（80 字节）
# version(4) + prev_block(32) + merkle_root(32) + timestamp(4) + bits(4) + nonce(4)
```

Merkle 树是区块链的核心数据结构，利用哈希函数将所有交易组织为二叉树，根哈希存储在区块头中：

```text
         Root Hash
        /         \
    Hash(AB)    Hash(CD)
    /    \      /    \
  H(A)  H(B) H(C)  H(D)
   |     |     |     |
  Tx_A  Tx_B  Tx_C  Tx_D
```

任何一笔交易的修改都会导致根哈希变化，从而被网络检测到。

#### 密钥派生函数（KDF）

从密码或主密钥派生子密钥：

#### 承诺方案（Commitment Scheme）

在零知识证明和安全多方计算中，哈希函数用于构建承诺方案：

```python
import hashlib
import os

def commit(value):
    """生成承诺和解承诺信息"""
    randomness = os.urandom(32)  # 随机数防止暴力枚举
    commitment = hashlib.sha256(value.encode() + randomness).hexdigest()
    return commitment, randomness  # 发送 commitment，保留 randomness

def verify(commitment, value, randomness):
    """验证承诺"""
    expected = hashlib.sha256(value.encode() + randomness).hexdigest()
    return commitment == expected
```

### 13.4.9 常见误区与最佳实践

#### 误区一：用 SHA-256 直接存储密码

```python
# ❌ 极度不安全
password_hash = hashlib.sha256(password.encode()).hexdigest()

# ❌ 加盐但仍然不安全（太快了，GPU 每秒数十亿次）
salt = os.urandom(16)
password_hash = hashlib.sha256(salt + password.encode()).hexdigest()

# ✅ 正确：使用专用密码哈希函数
from argon2 import PasswordHasher
ph = PasswordHasher()
password_hash = ph.hash(password)
```

SHA-256 设计目标是快速计算，这在密码存储场景中是致命缺陷。专用密码哈希函数（Argon2、bcrypt）通过可控的计算成本和内存消耗，将暴力破解的时间和经济成本提高到不可行的程度。

#### 误区二：自己实现哈希函数

```python
# ❌ 永远不要自己实现
def my_hash(data):
    result = 0
    for byte in data:
        result = (result * 31 + byte) % (2**256)
    return result

# ✅ 使用标准库
hashlib.sha256(data).hexdigest()
```

自实现的哈希函数几乎必然存在密码学弱点。即使是专业密码学家设计的算法也需要多年的公开分析才能获得信任。

#### 误区三：MD5/SHA-1 用于安全场景

```python
# ❌ MD5 用于文件完整性校验（防篡改场景）
file_hash = hashlib.md5(file_data).hexdigest()

# ❌ SHA-1 用于数字签名
signature_hash = hashlib.sha1(document).digest()

# ✅ 至少使用 SHA-256
file_hash = hashlib.sha256(file_data).hexdigest()
```

MD5 和 SHA-1 可以在数秒内找到碰撞。在需要防篡改、防伪造的场景中，必须使用 SHA-2 或 SHA-3 家族。

#### 误区四：使用 `==` 比较 MAC/HMAC

```python
# ❌ 时序攻击
if computed_mac == received_mac:
    process_message()

# ✅ 恒定时间比较
if hmac.compare_digest(computed_mac, received_mac):
    process_message()
```

普通字符串比较在第一个不匹配字节处返回，泄露时间信息。攻击者可以通过精确测量响应时间，逐字节猜测正确的 MAC 值。`hmac.compare_digest()` 恒定时间比较消除了这一侧信道。

#### 误区五：盐值过短或复用

```python
# ❌ 全局固定盐
GLOBAL_SALT = b"myapp_salt"
hash = hashlib.sha256(GLOBAL_SALT + password.encode()).hexdigest()

# ❌ 盐值太短（8 字节）
salt = os.urandom(8)

# ✅ 每个用户独立的随机盐，至少 16 字节
salt = os.urandom(16)
```

固定盐等于没有盐——所有相同密码的用户哈希值相同，彩虹表攻击依然有效。盐值太短会增加碰撞概率，导致不同用户共享盐值。

#### 误区六：忽略密钥长度要求

HMAC 的密钥长度不应低于底层哈希函数的输出长度。例如 HMAC-SHA256 的密钥至少 32 字节。过短的密钥直接降低安全强度。

### 13.4.11 进阶话题

#### 可变输出长度哈希（XOF）

SHAKE128 和 SHAKE256 是 SHA-3 标准中的可扩展输出函数（XOF），可以生成任意长度的输出：

```python
import hashlib

# SHAKE256 生成 64 字节输出
shake = hashlib.shake_256(b"input data")
output_64 = shake.hexdigest(64)  # 512 位
output_128 = shake.hexdigest(128)  # 1024 位

# 适用场景：密钥派生、随机数生成、一次性密码本
```

#### 哈希链与时间锁

哈希链通过反复哈希构建时间延迟：

```python
def hash_chain(seed, steps, algorithm='sha256'):
    """生成哈希链：seed -> H(seed) -> H(H(seed)) -> ..."""
    h = hashlib.new(algorithm)
    h.update(seed)
    current = h.digest()
    chain = [seed.hex(), current.hex()]
    for _ in range(steps - 1):
        h = hashlib.new(algorithm)
        h.update(current)
        current = h.digest()
        chain.append(current.hex())
    return chain

# 用途：时间锁谜题（Timelock Puzzle）、VDF（可验证延迟函数）
```

#### Merkle-Damgård 强化

为了解决 MD 结构的长度扩展攻击问题，可以使用 MD 强化（Strengthening）：
- 在消息末尾附加消息长度（MD5/SHA-1/SHA-2 都这样做）
- 但仅防止碰撞攻击，**不能防御长度扩展攻击**
- HMAC 通过双重嵌套结构解决此问题
- SHA-3 通过海绵结构从根本上免疫此问题

### 13.4.12 总结

哈希函数是密码学的基石，理解其安全性质和正确用法至关重要：

1. **选择正确的算法**：安全场景用 SHA-2/SHA-3/BLAKE2；密码存储用 Argon2id/bcrypt/scrypt
2. **理解攻击面**：生日攻击决定了抗碰撞性的安全强度为输出长度的一半；长度扩展攻击影响所有 MD 结构
3. **加盐是必须的**：密码哈希必须使用足够长的随机盐
4. **MAC 必须用 HMAC**：不要用 `H(key || message)` 的朴素构造
5. **恒定时间比较**：验证 MAC/签名时必须使用恒定时间比较函数
6. **弃用已破解算法**：MD5 和 SHA-1 不再用于任何安全用途
