## 常见误区

搜索引擎从搭建到运维，处处隐藏着"看起来合理但实际有害"的做法。本节系统梳理搜索系统全生命周期中的高频误区，每个误区都给出**错误表现 → 根因分析 → 正确做法 → 检测手段**的完整链路，帮助工程师避免踩坑。

---

### 误区一：忽略监控体系

**错误表现**

系统上线后只关注"能不能搜到"，不关注"搜得怎么样"。没有监控的搜索系统就像没有仪表盘的飞机——可能在飞，但不知道何时会坠毁。

常见症状：
- 线上出现慢查询，但无法定位是哪个查询、哪个分片导致
- 索引膨胀到磁盘满才被发现
- 查询延迟从 50ms 悄悄涨到 500ms，无人察觉
- 分片不均衡导致热点节点，直到 OOM 才报警

**根因分析**

许多团队把搜索当作"中间件"，认为只要返回结果就算正常。但实际上搜索引擎是典型的**状态密集型系统**——索引大小、分片分布、缓存命中率、GC 压力等内部状态持续变化，任何一个指标异常都可能引发连锁故障。

**正确做法**

部署 Prometheus + Grafana 构建完整的搜索监控体系，至少覆盖以下四个维度：

| 监控维度 | 关键指标 | 告警阈值参考 | 采集方式 |
|----------|----------|-------------|----------|
| 查询性能 | P50/P95/P99 延迟、QPS、错误率 | P99 > 200ms、错误率 > 1% | Elasticsearch `_stats` API + APM |
| 索引健康 | 索引大小、文档数、segment 数、合并队列 | 索引增速 > 50GB/天、segment > 500 | `_cat/indices`、`_cat/allocation` |
| 集群状态 | 节点数、分片状态、pending tasks、CPU/内存 | 集群状态非 green、pending > 100 | `_cluster/health`、Node Exporter |
| 缓存效率 | query cache 命中率、fielddata 内存占用 | 命中率 < 60%、fielddata > 50% 堆内存 | `_nodes/stats` |

```yaml
  - job_name: 'elasticsearch'
    static_configs:
      - targets: ['es-exporter:9114']

    scrape_interval: 15s
```

```python
# 基于 Python 的简易搜索质量监控脚本
import requests, time, json

ES_HOST = "http://localhost:9200"
ALERT_LATENCY_MS = 200
ALERT_ERROR_RATE = 0.01

def check_search_health():
    """定期执行，检测搜索系统健康状态"""
    # 1. 集群状态
    health = requests.get(f"{ES_HOST}/_cluster/health").json()
    if health["status"] != "green":
        send_alert(f"集群状态异常: {health['status']}")

    # 2. 查询延迟（发送探测查询）
    start = time.time()
    resp = requests.post(f"{ES_HOST}/my_index/_search", json={
        "query": {"match": {"content": "测试查询"}},
        "size": 1
    })
    latency_ms = (time.time() - start) * 1000

    if latency_ms > ALERT_LATENCY_MS:
        send_alert(f"查询延迟过高: {latency_ms:.0f}ms")

    # 3. 错误率
    result = resp.json()
    total = result["hits"]["total"]["value"]
    if total == 0:
        # 空结果也可能是索引问题
        check_index_health()

def check_index_health():
    """检测索引是否正常"""
    stats = requests.get(f"{ES_HOST}/_cat/indices?v&amp;format=json").json()
    for idx in stats:
        if idx["health"] == "red":
            send_alert(f"索引 {idx['index']} 状态为 RED")
        if int(idx["docs.count"]) == 0:
            send_alert(f"索引 {idx['index']} 文档数为 0，可能数据丢失")
```

### 误区二：过度优化（过早优化）

**错误表现**

在系统刚搭建、数据量只有几万条时就开始调优各种参数：手动合并 segment、调整分片数到极致、编写复杂的自定义评分脚本。结果是代码复杂度飙升，但性能提升微乎其微，反而引入了维护负担。

典型错误场景：
- 数据量 10 万条就配置了 100 个分片（每个分片只有 1000 条）
- 为每个字段单独配置 analyzer，导致索引膨胀 5 倍
- 在数据量不足时就引入学习排序（LTR），模型严重过拟合
- 过度使用 `script_score` 自定义评分，查询延迟从 10ms 涨到 500ms

**根因分析**

过早优化违反了软件工程的基本原则。搜索系统的性能瓶颈通常出现在特定的数据规模和负载模式下，没有真实数据和流量就进行优化，等于"在不知道敌人在哪的时候就开枪"。

**正确做法**

遵循**"先运行、再度量、后优化"**的三阶段策略：

┌─────────────────────────────────────────────────────────┐
│                    优化决策流程                          │
│                                                         │
│  ① 功能正确  ──→  ② 基准测试  ──→  ③ 性能分析  ──→    │
│     验证搜索       在目标数据量      用 profiler         │
│     结果相关性     和负载下测速      定位真正瓶颈         │
│                                                         │
│  ④ 针对性优化  ──→  ⑤ 回归验证  ──→  ⑥ 持续监控        │
│     只优化瓶颈      证明优化有效      防止性能回退        │
└─────────────────────────────────────────────────────────┘

**基准测试模板**

```python
import time, statistics
from elasticsearch import Elasticsearch

es = Elasticsearch("http://localhost:9200")

def benchmark_search(index, queries, iterations=100):
    """搜索性能基准测试"""
    latencies = []
    errors = 0

    for i in range(iterations):
        query = queries[i % len(queries)]
        start = time.perf_counter()
        try:
            resp = es.search(index=index, body=query)
            elapsed_ms = (time.perf_counter() - start) * 1000
            latencies.append(elapsed_ms)
        except Exception as e:
            errors += 1
            print(f"迭代 {i} 失败: {e}")

    print(f"总请求数: {iterations}")
    print(f"错误数: {errors}")
    print(f"P50 延迟: {statistics.median(latencies):.1f}ms")
    print(f"P95 延迟: {sorted(latencies)[int(len(latencies)*0.95)]:.1f}ms")
    print(f"P99 延迟: {sorted(latencies)[int(len(latencies)*0.99)]:.1f}ms")

# 使用示例
test_queries = [
    {"query": {"match": {"title": "Python教程"}}},
    {"query": {"match_phrase": {"content": "搜索引擎优化"}}},
    {"query": {"bool": {"must": [{"match": {"tags": "技术"}}], "filter": [{"range": {"date": {"gte": "2025-01-01"}}}]}}}
]
benchmark_search("articles", test_queries)
```

**何时该优化的判断标准**

| 情况 | 建议 |
|------|------|
| 文档数 < 10万，QPS < 100 | 不需要优化，关注功能正确性 |
| 文档数 10-100万，QPS 100-1000 | 可以做基础优化（索引设计、mapping 调整） |
| 文档数 > 100万，QPS > 1000 | 需要系统性优化（分片策略、缓存、查询优化） |
| 任何阶段出现明确瓶颈 | 先用 profiler 定位，再针对性优化 |

### 误区三：索引设计不合理

**错误表现**

索引设计是搜索系统中影响最深远的环节，但也是最容易被忽视的。不合理的索引设计会导致存储膨胀、查询变慢、相关性下降，而且后期修复成本极高。

常见错误：
- **字段类型一刀切**：所有字段都用 `text` 类型，无法精确匹配
- **分片数拍脑袋决定**：上线后发现分片过多导致资源浪费，或过少导致单分片过大
- **mapping 从未更新**：新业务需求上线后才发现字段无法被搜索或聚合
- **忽略字段权重**：标题和正文使用相同权重，搜索结果相关性差

**根因分析**

索引设计需要在"写入性能"、"存储效率"、"查询性能"三者之间取得平衡。很多团队在项目初期用默认配置"快速上线"，后续又缺乏对索引结构的定期审视。

**正确做法**

遵循索引设计的核心原则：

```python
# 正确的 Elasticsearch mapping 设计示例
index_mapping = {
    "settings": {
        "number_of_shards": 5,          # 根据数据量和查询模式计算
        "number_of_replicas": 1,
        "refresh_interval": "5s",       # 非实时场景可适当放宽
        "codec": "best_compression"      # 减少存储占用
    },
    "mappings": {
        "properties": {
            # 标题：text 类型用于全文搜索，keyword 类型用于精确匹配和聚合
            "title": {
                "type": "text",
                "analyzer": "ik_max_word",       # 中文最大分词（索引时）
                "search_analyzer": "ik_smart",   # 中文智能分词（查询时）
                "fields": {
                    "keyword": {
                        "type": "keyword",
                        "ignore_above": 256
                    }
                },
                "boost": 2.0  # 标题权重更高
            },
            # 正文：text 类型，不需要 keyword 子字段（节省存储）
            "content": {
                "type": "text",
                "analyzer": "ik_max_word",
                "search_analyzer": "ik_smart"
            },
            # 标签：keyword 类型，用于过滤和聚合
            "tags": {
                "type": "keyword"
            },
            # 发布时间：date 类型，用于范围查询和排序
            "publish_date": {
                "type": "date",
                "format": "yyyy-MM-dd||epoch_millis"
            },
            # 浏览量：integer 类型，用于排序和数值范围查询
            "view_count": {
                "type": "integer"
            }
        }
    }
}
```

**分片数计算公式**

分片数 = 预估总数据量 / 单分片推荐大小

| 数据量级 | 单分片大小 | 建议分片数 | 说明 |
|----------|-----------|-----------|------|
| < 1GB | 1GB | 1 | 小数据量不要过度分片 |
| 1-100GB | 20-40GB | 3-10 | 分片数 = 节点数的 1-3 倍 |
| 100GB-1TB | 30-50GB | 5-20 | 超过 50GB 单分片会影响查询性能 |
| > 1TB | 30-50GB | 20+ | 考虑索引拆分（按时间/业务） |

### 误区四：配置参数拍脑袋

**错误表现**

搜索引擎有大量可调参数，很多团队要么全用默认值，要么从网上复制一份"最佳配置"直接套用，完全不考虑自身业务特征。

典型错误：
- JVM 堆内存设置为物理内存的 50% 以上（导致 GC 停顿）
- `max_result_window` 保持默认 10000（深分页场景报错）
- `index.translog.durability` 设为 `async`（数据丢失风险）
- `refresh_interval` 设为 1s（高写入场景下 segment 碎片过多）
- 连接池大小、超时时间全部使用默认值

**根因分析**

搜索引擎的配置参数与硬件资源、数据特征、查询模式强相关。没有放之四海而皆准的配置，必须根据实际场景调优。

**正确做法**

核心参数的调优指南：

```yaml
# JVM 配置（elasticsearch.yml）
# 规则：堆内存不超过物理内存的 50%，且不超过 30GB（压缩指针阈值）
-Xms16g
-Xmx16g

# 写入相关配置
index:
  refresh_interval: "5s"              # 默认1s，写多读少场景可放宽
  translog.durability: "request"       # 强一致场景用 request，高吞吐用 async
  translog.sync_interval: "5s"        # async 模式下的同步间隔
  merge.scheduler.max_thread_count: 2 # 合并线程数，通常为 CPU 核数的一半

# 查询相关配置
index.max_result_window: 10000        # 深分页场景可适当放大，但要用 search_after 替代
index.requests.cache.enable: true     # 开启请求缓存

# 内存分配
# fielddata 内存限制（防止单字段耗尽堆内存）
index.fields.fielddata.max_size: "40%"
```

### 误区五：缺乏容错设计

**错误表现**

搜索系统作为关键基础设施，必须具备容错能力。但很多团队的设计假设是"系统永远不会出错"，导致单点故障引发全局搜索不可用。

常见问题：
- ES 集群只有一个节点，宕机后搜索完全不可用
- 没有设置副本分片，单个节点磁盘故障导致数据丢失
- 查询没有超时设置，慢查询阻塞线程池导致雪崩
- 没有降级策略，搜索服务不可用时整个业务页面白屏
- 缺少重试机制，网络抖动导致批量写入失败

**根因分析**

容错设计需要在"可用性"和"一致性"之间做权衡。很多团队只关注功能实现，忽略了系统在异常情况下的行为，而生产环境中的故障是必然发生的。

**正确做法**

构建搜索系统的四层容错体系：

```python
import time
from functools import wraps
from elasticsearch import Elasticsearch
from elasticsearch.exceptions import ConnectionError, NotFoundError

# 第一层：超时保护
class SearchClient:
    def __init__(self, hosts, timeout=10, max_retries=3):
        self.es = Elasticsearch(
            hosts,
            request_timeout=timeout,
            retry_on_timeout=True,
            max_retries=max_retries,
        )

    # 第二层：重试机制（指数退避）
    def search_with_retry(self, index, query, retries=3):
        for attempt in range(retries):
            try:
                return self.es.search(
                    index=index,
                    body=query,
                    timeout="5s",
                    allow_partial_search_results=True
                )
            except ConnectionError as e:
                if attempt == retries - 1:
                    raise
                wait_time = (2 ** attempt) * 0.5  # 0.5s, 1s, 2s
                time.sleep(wait_time)
                print(f"查询失败，{wait_time}s 后重试 ({attempt+1}/{retries})")

    # 第三层：降级策略（返回缓存或空结果）
    def search_with_fallback(self, index, query, cache_client=None):
        try:
            result = self.search_with_retry(index, query)
            # 写入缓存备用
            if cache_client:
                cache_client.set(f"search:{index}:{hash(str(query))}", result, ex=300)
            return result
        except Exception:
            # 降级：尝试从缓存返回上次结果
            if cache_client:
                cached = cache_client.get(f"search:{index}:{hash(str(query))}")
                if cached:
                    return {**cached, "_degraded": True}  # 标记为降级结果
            # 最终降级：返回空结果
            return {"hits": {"hits": [], "total": {"value": 0}}, "_degraded": True}
```

**熔断器模式（防止雪崩）**

```python
import time
from enum import Enum

class CircuitState(Enum):
    CLOSED = "closed"       # 正常状态
    OPEN = "open"           # 熔断状态（拒绝请求）
    HALF_OPEN = "half_open" # 半开状态（试探恢复）

class CircuitBreaker:
    def __init__(self, failure_threshold=5, recovery_timeout=30):
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.failure_count = 0
        self.state = CircuitState.CLOSED
        self.last_failure_time = 0

    def call(self, func, *args, **kwargs):
        if self.state == CircuitState.OPEN:
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = CircuitState.HALF_OPEN
            else:
                raise Exception("搜索服务熔断中，请稍后重试")

        try:
            result = func(*args, **kwargs)
            if self.state == CircuitState.HALF_OPEN:
                self.state = CircuitState.CLOSED
                self.failure_count = 0
            return result
        except Exception as e:
            self.failure_count += 1
            self.last_failure_time = time.time()
            if self.failure_count >= self.failure_threshold:
                self.state = CircuitState.OPEN
            raise

# 使用示例
breaker = CircuitBreaker(failure_threshold=5, recovery_timeout=30)
client = SearchClient("http://localhost:9200")

try:
    result = breaker.call(client.search_with_retry, "my_index", {"query": {"match_all": {}}})
except Exception as e:
    print(f"搜索失败: {e}")
    # 触发降级逻辑
```

**集群高可用部署原则**

| 组件 | 最低要求 | 推荐配置 | 说明 |
|------|----------|----------|------|
| 数据节点 | 3 个 | 5+ 个 | 每个节点一个分片副本 |
| Master 节点 | 3 个（独立） | 3 个 | 避免脑裂，`minimum_master_nodes = n/2+1` |
| 副本数 | 1 | 1-2 | 容忍 1-2 个节点故障 |
| 跨可用区部署 | 否 | 是 | 至少 2 个可用区，防止单机房故障 |

### 误区六：忽视安全防护

**错误表现**

搜索引擎直接暴露在网络中，没有认证、没有授权、没有加密，成为攻击者的"信息宝库"。

常见安全问题：
- ES 默认无认证，任何人都可以读取/删除索引
- 使用默认端口 9200，被扫描器批量发现
- 没有 HTTPS 加密，查询数据在传输中被窃听
- 没有字段级权限控制，内部敏感数据对所有用户可见
- 没有输入过滤，查询注入可导致拒绝服务

**根因分析**

安全往往被视为"性能的对立面"，在追求搜索性能的过程中被优先级排到最后。但数据泄露的代价远大于加解密的性能损耗。

**正确做法**

搜索系统的安全防护应覆盖**网络层、认证层、授权层、数据层**四个维度：

```yaml
# Elasticsearch 安全配置（elasticsearch.yml）

# 1. 网络层：绑定内网地址，禁用公网暴露
network.host: 192.168.1.100
http.port: 9200
transport.port: 9300

# 2. 认证层：启用 X-Pack Security
xpack.security.enabled: true
xpack.security.transport.ssl.enabled: true
xpack.security.http.ssl.enabled: true

# 3. 授权层：基于角色的访问控制
# 使用 API 创建角色和用户
```

```python
# RBAC 角色配置示例
from elasticsearch import Elasticsearch

es = Elasticsearch("https://localhost:9200", basic_auth=("elastic", "password"))

# 创建只读角色（只能查询，不能修改数据）
es.security.put_role(name="search_reader", body={
    "cluster": ["monitor"],
    "indices": [
        {
            "names": ["public_*"],  # 只能访问公共索引
            "privileges": ["read", "view_index_metadata"]
        }
    ]
})

# 创建写入角色（可以写入特定索引）
```

```python
# 输入过滤：防止查询注入
def safe_search(es, index, user_query):
```

### 误区七：查询性能反模式

**错误表现**

写出的查询在小数据量下运行正常，数据量增长后性能急剧恶化。很多反模式在初期不易察觉，但会在数据量达到百万级后暴露。

常见反模式：

**反模式一：深分页**

```python
# 错误：使用 from + size 进行深分页
# 当 from > 10000 时，性能急剧下降
bad_query = {
    "query": {"match_all": {}},
    "from": 50000,   # 需要每个分片返回 50000 + size 条结果
    "size": 20        # 协调节点需要对 50000+ 条结果排序
}

# 正确：使用 search_after 实现深度遍历
good_query = {
    "query": {"match_all": {}},
    "size": 20,
    "sort": [
        {"_score": "desc"},
        {"_id": "asc"}  # 必须加唯一字段作为 tiebreaker
    ],
    "search_after": [0.95, "doc_abc123"]  # 上一页最后一条的排序值
}
```

**反模式二：wildcard 前缀通配**

```python
# 错误：前缀通配会导致全索引扫描
bad_query = {
    "query": {
        "wildcard": {"title": "*python*"}  # * 在开头，无法利用倒排索引
    }
}

# 正确：使用 match 或 match_phrase
good_query = {
    "query": {
        "match_phrase": {"title": "python"}
    }
}

# 如果确实需要部分匹配，使用 ngram 分词器
# 在 mapping 阶段就配置好 ngram analyzer
```

**反模式三：忽略 filter 和 query 的区别**

```python
# 错误：将纯过滤条件放在 query 中（需要计算评分，浪费资源）
bad_query = {
    "query": {
        "bool": {
            "must": [
                {"term": {"status": "published"}},       # 过滤条件不应计算评分
                {"range": {"date": {"gte": "2025-01-01"}}}  # 同上
            ],
            "should": [
                {"match": {"title": "搜索引擎"}}  # 这个才需要评分
            ]
        }
    }
}

# 正确：过滤条件用 filter（不计算评分，且可缓存）
good_query = {
    "query": {
        "bool": {
            "filter": [
                {"term": {"status": "published"}},
                {"range": {"date": {"gte": "2025-01-01"}}}
            ],
            "should": [
                {"match": {"title": "搜索引擎"}}
            ],
            "minimum_should_match": 1
        }
    }
}
```

**反模式四：单次查询返回过多数据**

```python
# 错误：一次返回 10000 条完整文档
bad_query = {
    "query": {"match_all": {}},
    "size": 10000,
    "_source": True  # 返回所有字段
}

# 正确：分批获取 + 只返回需要的字段
def scroll_all_docs(es, index, batch_size=1000):
    """使用 scroll 安全地遍历大量数据"""
    result = es.search(
        index=index,
        body={"query": {"match_all": {}}, "size": batch_size},
        scroll="2m",  # 保持 scroll 上下文 2 分钟
        _source=["id", "title"]  # 只返回必要字段
    )
    scroll_id = result["_scroll_id"]

    while True:
        hits = result["hits"]["hits"]
        if not hits:
            break
        yield from hits
        result = es.scroll(scroll_id=scroll_id, scroll="2m")

    es.clear_scroll(scroll_id=scroll_id)
```

### 误区八：数据同步与一致性问题

**错误表现**

搜索系统通常不是数据的唯一来源，而是从数据库、消息队列等系统同步数据。数据同步的不一致会导致搜索结果与实际数据脱节。

常见问题：
- 数据库已更新，但搜索引擎索引还是旧数据，用户搜索不到新内容
- 批量同步失败后没有重试机制，部分数据永久缺失
- 删除数据时只删了数据库，搜索引擎中的"幽灵数据"依然可搜到
- 并发更新导致搜索引擎中的数据版本混乱

**根因分析**

搜索引擎与业务数据库是两个独立系统，缺乏原生的事务一致性保证。数据同步链路越长、环节越多，出错概率越大。

**正确做法**

采用**"变更数据捕获（CDC）+ 幂等写入 + 对账机制"**的三层保障：

```python
# 基于消息队列的可靠数据同步
import json
from elasticsearch import Elasticsearch

class SearchIndexSync:
    def __init__(self, es_client, index_name):
        self.es = es_client
        self.index = index_name

    def handle_db_change(self, event):
        """
        处理数据库变更事件（由 CDC 捕获）
        event 格式: {"op": "insert/update/delete", "table": "articles", "data": {...}}
        """
        op = event["op"]
        doc_id = str(event["data"]["id"])

        if op in ("insert", "update"):
            # 幂等写入：用 doc_id 作为 ES 的 _id，重复执行不会产生重复数据
            self.es.index(
                index=self.index,
                id=doc_id,
                body=event["data"],
                refresh="wait_for"  # 确保写入后立即可搜索
            )

        elif op == "delete":
            # 删除索引中的文档
            try:
                self.es.delete(index=self.index, id=doc_id)
            except Exception:
                pass  # 文档不存在也不报错（幂等）

    def reconcile(self, db_ids, batch_size=5000):
        """
        定时对账：比较数据库和搜索引擎的数据一致性
        """
        # 获取 ES 中的所有文档 ID
        es_ids = set()
        result = self.es.search(
            index=self.index,
            body={"query": {"match_all": {}}, "_source": ["id"], "size": 0},
            scroll="5m"
        )
        # ... 遍历获取所有 ID ...

        # 找出差异
        missing_in_es = db_ids - es_ids   # 数据库有但 ES 没有
        ghost_in_es = es_ids - db_ids     # ES 有但数据库没有

        # 修复差异
        for doc_id in missing_in_es:
            # 从数据库读取并补录到 ES
            doc = fetch_from_db(doc_id)
            if doc:
                self.es.index(index=self.index, id=str(doc_id), body=doc)

        for doc_id in ghost_in_es:
            # 删除 ES 中的幽灵数据
            self.es.delete(index=self.index, id=str(doc_id))

        return {"missing": len(missing_in_es), "ghost": len(ghost_in_es)}
```

**同步架构选型**

| 方案 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| 双写（应用层同步写） | 简单场景，一致性要求不高 | 实现简单 | 一致性难保证，失败时数据不一致 |
| 消息队列（Kafka/RabbitMQ） | 中高吞吐，需要可靠传递 | 解耦、可重试、有序 | 架构复杂度增加 |
| CDC（Debezium/Canal） | 已有数据库，不想改业务代码 | 侵入性低，实时性好 | 需要额外组件，配置复杂 |
| Logstash/Filebeat | 日志类数据，允许短暂延迟 | 配置简单，生态成熟 | 延迟较高，不适合强一致场景 |

### 误区九：相关性调优缺失

**错误表现**

搜索引擎上线后，"搜得到"但"搜不准"。用户搜索"iPhone"却返回大量不相关结果，搜索"如何退款"却优先展示商品广告。相关性差直接导致用户体验下降和业务转化率降低。

常见问题：
- 搜索结果完全按时间排序，最新的不一定最相关
- 没有对不同字段赋予合理权重
- 同义词、近义词未处理，"手机"搜不到"智能手机"
- 没有利用用户行为数据（点击率、停留时间）优化排序

**根因分析**

相关性调优是一个持续迭代的过程，需要结合业务特点、用户行为和搜索日志进行系统性优化。很多团队认为"配置好 mapping 就够了"，忽略了排序策略的持续优化。

**正确做法**

相关性调优的系统化方法：

```python
# 多字段加权查询
def relevance_search(es, index, query_text):
    """带字段权重的搜索"""
    return es.search(
        index=index,
        body={
            "query": {
                "bool": {
                    "should": [

                        # 标签匹配

                        # 正文匹配（权重最低）

                "query": {

                            "query": {"match_all": {}},
                            "functions": [
                                # 时间衰减：近 30 天的文档加分
                                {
                                    "time_decay": {
                                        "origin": "now",
                                        "scale": "30d",
                                        "decay": 0.5,
                                        "field": "publish_date"
                                    }
                                },
                                # 热度加分：浏览量越高排名越靠前
                                {
                                    "field_value_factor": {
                                        "field": "view_count",
```

**搜索日志分析驱动优化**

```python
# 分析搜索日志，识别相关性问题
import collections

def analyze_search_logs(logs):
    """分析搜索日志，找出需要优化的查询"""
    # 1. 低点击率查询（搜到了但没人点 = 结果不准）
    low_ctr_queries = []
    # 2. 高退出率查询（点进去马上返回 = 结果不对）
    high_bounce_queries = []
    # 3. 零结果查询（完全搜不到 = 需要查询扩展）
    zero_result_queries = []

    query_stats = collections.defaultdict(lambda: {"impressions": 0, "clicks": 0, "bounces": 0, "zeros": 0})

    for log in logs:
        q = log["query"]
        query_stats[q]["impressions"] += 1
        if log.get("clicked"):
            query_stats[q]["clicks"] += 1
        if log.get("bounced"):
            query_stats[q]["bounces"] += 1
        if log.get("result_count") == 0:
            query_stats[q]["zeros"] += 1

    for q, stats in query_stats.items():
        if stats["impressions"] > 10:  # 只分析有足够数据的查询
            ctr = stats["clicks"] / stats["impressions"]
            if ctr < 0.1:  # CTR < 10%
                low_ctr_queries.append((q, ctr))

    return sorted(low_ctr_queries, key=lambda x: x[1])
```

### 误区十：忽视容量规划

**错误表现**

系统上线时配置刚好够用，随着业务增长，索引膨胀、查询变慢，最终不得不在业务高峰期做紧急扩容或数据迁移。

常见问题：
- 磁盘使用率超过 80% 才发现需要扩容
- 没有预估数据增长速度，3 个月后存储翻倍
- 单节点承载过多分片，内存和 CPU 资源耗尽
- 没有数据生命周期管理，历史数据无限堆积

**根因分析**

容量规划需要对业务增长趋势有清晰的预判。搜索引擎的资源消耗与数据量、查询并发量、索引更新频率三者强相关。

**正确做法**

建立容量规划模型：

```python
def capacity_planning(
    current_docs,           # 当前文档数
    avg_doc_size_bytes,     # 平均文档大小
    daily_growth_rate,      # 每日文档增长比例
    peak_qps,               # 峰值 QPS
    days_to_plan            # 规划天数
):
    """搜索系统容量规划计算器"""
    # 1. 存储需求
    future_docs = current_docs * (1 + daily_growth_rate) ** days_to_plan
    raw_storage = future_docs * avg_doc_size_bytes
    # ES 索引存储通常是原始数据的 1.1-1.3 倍（倒排索引+元数据）
    es_storage = raw_storage * 1.2
    # 加上副本
    total_storage = es_storage * 2  # 1 个副本
    print(f"预估存储需求: {total_storage / (1024**3):.1f} GB")

    # 2. 内存需求
    # 每个分片建议不超过 50GB，且需要足够内存缓存热数据
    shard_size_gb = 40  # 目标单分片大小
    num_shards = int(es_storage / (shard_size_gb * 1024**3)) + 1
    # 每个分片需要约 1GB 堆内存
    required_heap_gb = num_shards * 1  # 分片内存
    data_node_heap = max(16, required_heap_gb / 3)  # 假设 3 个数据节点
    print(f"预估分片数: {num_shards}, 每节点堆内存: {data_node_heap:.0f} GB")

    # 3. 查询性能
    # 峰值 QPS 需要分配到多个节点
    qps_per_node = 500  # 单节点承受能力参考值
    data_nodes = max(num_shards, int(peak_qps / qps_per_node) + 1)
    print(f"建议数据节点数: {data_nodes}")

    return {
        "storage_gb": total_storage / (1024**3),
        "num_shards": num_shards,
        "data_nodes": data_nodes,
        "heap_per_node_gb": data_node_heap
    }

# 使用示例
capacity_planning(
    current_docs=10_000_000,       # 1000 万文档
    avg_doc_size_bytes=2000,        # 平均 2KB
    daily_growth_rate=0.02,         # 每天增长 2%
    peak_qps=2000,                  # 峰值 2000 QPS
    days_to_plan=180                # 规划 6 个月
)
```
