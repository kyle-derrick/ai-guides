## 14.17 访问控制测试

访问控制（Access Control）是 Web 应用安全的核心防线，其职责是确保用户只能执行其权限范围内的操作、只能访问其权限范围内的资源。OWASP Top 10 2021 将"失效的访问控制"（A01:2021 – Broken Access Control）列为第一大安全风险，这并非偶然——在真实世界的安全事件中，访问控制缺陷是最常见、影响最广泛、也是最容易被利用的漏洞类型。

访问控制失效的根本原因在于：服务端未能对每个请求进行充分的权限校验，或者校验逻辑存在设计缺陷。攻击者利用这些缺陷，可以越权访问其他用户的数据（水平越权）、执行管理员功能（垂直越权）、甚至完全绕过访问控制机制。

本节将从访问控制模型的理论基础出发，系统讲解各类访问控制缺陷的测试方法、工具使用、实战案例和防御方案。

### 14.17.1 访问控制模型基础

在深入测试之前，必须理解访问控制的基本模型。不同的模型决定了权限校验的逻辑，也决定了可能存在的漏洞类型。

#### 14.17.1.1 四种核心访问控制模型

| 模型 | 全称 | 权限分配方式 | 典型场景 | 常见缺陷 |
|------|------|-------------|---------|---------|
| DAC | 自主访问控制（Discretionary Access Control） | 资源所有者自行决定 | 文件系统权限、共享文档 | 权限传递不受控、用户可自行开放权限 |
| MAC | 强制访问控制（Mandatory Access Control） | 由系统策略强制执行 | 军事系统、安全操作系统 | 策略配置错误导致越级访问 |
| RBAC | 基于角色的访问控制（Role-Based Access Control） | 按角色分配权限 | 企业应用、管理后台 | 角色定义不精确、权限提升 |
| ABAC | 基于属性的访问控制（Attribute-Based Access Control） | 根据属性组合动态判断 | 云平台、微服务 | 属性组合绕过、策略冲突 |

#### 14.17.1.2 访问控制的三个维度

```text
维度一：垂直访问控制（Vertical）
  ├── 控制不同权限级别之间的访问
  ├── 普通用户 → 管理员功能
  └── 典型漏洞：垂直越权

维度二：水平访问控制（Horizontal）
  ├── 控制同级别用户之间的资源访问
  ├── 用户A → 用户B的数据
  └── 典型漏洞：IDOR（不安全的直接对象引用）

维度三：上下文相关访问控制（Context-Dependent）
  ├── 根据应用状态和上下文限制访问
  ├── 流程顺序、时间窗口、地理位置
  └── 典型漏洞：流程绕过、状态机缺陷
```

理解这三个维度至关重要，因为许多测试人员只关注垂直和水平维度，而忽略了上下文相关的访问控制缺陷。例如，一个电商应用可能允许用户跳过支付步骤直接访问订单确认页面，这就是上下文相关的访问控制失效。

### 14.17.2 水平越权与 IDOR 测试

IDOR（Insecure Direct Object Reference，不安全的直接对象引用）是最常见的水平越权漏洞。当应用使用用户可控的输入（如数据库 ID、文件名、订单号）直接引用内部对象，且未校验当前用户是否有权访问该对象时，就会产生 IDOR。

#### 14.17.2.1 IDOR 的基本测试流程

**第一步：识别对象引用点**

系统性地扫描应用中所有包含对象标识符的参数：

```bash
# 使用 Burp Suite 的 Logger 记录所有请求
# 重点关注包含以下模式的参数：
# /api/users/1001          — 路径中的数字ID
# ?order_id=ORD-20240115   — 查询参数中的业务ID
# {"userId": 1001}         — 请求体中的引用
# /files/report_1001.pdf   — 文件名中的标识符
```

**第二步：参数篡改测试**

```text
原始请求：GET /api/v1/users/1001/profile
测试步骤：
  1. 修改为其他用户ID：GET /api/v1/users/1002/profile
  2. 修改为管理员ID：  GET /api/v1/users/1/profile
  3. 使用负数ID：      GET /api/v1/users/-1/profile
  4. 使用零值ID：      GET /api/v1/users/0/profile
  5. 使用超大ID：      GET /api/v1/users/999999999/profile
  6. 使用数组形式：    GET /api/v1/users/[1001,1002]/profile
```

**第三步：判断是否成功越权**

| 判断依据 | 说明 |
|---------|------|
| HTTP 状态码 | 200 而非 403/401 表示越权成功 |
| 响应体大小 | 与原始请求相似大小可能包含数据 |
| 数据内容 | 返回了其他用户的个人信息（姓名、邮箱、手机号等） |
| 响应时间 | 与其他有效请求的响应时间一致 |

#### 14.17.2.2 ID 格式变体测试

应用可能使用不同格式的标识符，测试时需覆盖所有可能的格式：

```bash
# 自增整数ID — 最容易被猜测
/user/1, /user/2, /user/3 ...

# UUID/GUID — 看似安全但可能泄露
/user/a1b2c3d4-e5f6-7890-abcd-ef1234567890

# 编码后的ID — 需要解码后篡改
# Base64: /user/MTAwMQ== (解码为1001)
# Hex: /user/03E9 (1001的十六进制)

# 短哈希 — 可能可预测
/user/X7kM9p (如果是基于自增ID的哈希)

# 组合键 — 多个参数组合
/user?tenant_id=100&user_id=1001
```

**关键测试技巧：** 当遇到 UUID 格式的 ID 时，不要因为"无法猜测"就放弃测试。UUID 可能通过以下途径泄露：

- API 文档或示例中硬编码的 UUID
- 其他 API 端点的响应中包含关联资源的 UUID
- 前端 JavaScript 代码中暴露的 UUID
- 日志、错误信息、调试模式下的 UUID
- GraphQL 查询中通过关联关系获取

#### 14.17.2.3 批量 IDOR 探测脚本

当手动测试发现存在 IDOR 后，可以使用脚本批量验证影响范围：

```python
import requests
import json

def test_idor(base_url, auth_token, id_range, endpoint_template):
    """
    批量IDOR测试脚本
    :param base_url: API基础URL
    :param auth_token: 当前用户的认证令牌
    :param id_range: 要测试的ID范围 (start, end)
    :param endpoint_template: 端点模板，{id}为占位符
    """
    headers = {
        "Authorization": f"Bearer {auth_token}",
        "Content-Type": "application/json"
    }
    
    results = {"accessible": [], "forbidden": [], "errors": []}
    
    for obj_id in range(id_range[0], id_range[1] + 1):
        url = base_url + endpoint_template.format(id=obj_id)
        try:
            resp = requests.get(url, headers=headers, timeout=10)
            if resp.status_code == 200:
                # 记录可访问的资源
                data = resp.json()
                results["accessible"].append({
                    "id": obj_id,
                    "status": resp.status_code,
                    "data_keys": list(data.keys()) if isinstance(data, dict) else "non-dict",
                    "size": len(resp.content)
                })
                print(f"[+] ID {obj_id}: 可访问 (状态码 {resp.status_code}, "
                      f"响应大小 {len(resp.content)} 字节)")
            elif resp.status_code in (403, 401):
                results["forbidden"].append(obj_id)
                print(f"[-] ID {obj_id}: 拒绝访问 ({resp.status_code})")
            else:
                results["errors"].append({"id": obj_id, "status": resp.status_code})
        except requests.RequestException as e:
            results["errors"].append({"id": obj_id, "error": str(e)})
    
    # 输出统计
    print(f"\n[*] 测试完成: {len(results['accessible'])} 可访问, "
          f"{len(results['forbidden'])} 拒绝, {len(results['errors'])} 错误")
    
    return results

# 使用示例
# results = test_idor(
#     base_url="https://target.com",
#     auth_token="eyJhbGciOiJIUzI1NiIs...",
#     id_range=(1000, 1100),
#     endpoint_template="/api/users/{id}/profile"
# )
```

> **警告：** 批量测试必须在授权范围内进行。未经授权的自动化扫描可能构成违法行为。上述脚本仅用于合法渗透测试场景。

#### 14.17.2.4 IDOR 的高级变体

**基于预测的 ID 攻击：** 当 ID 不是简单的自增整数时，需要分析 ID 的生成规律：

```python
import hashlib
import time

def analyze_id_pattern(sample_ids):
    """
    分析ID样本的生成规律
    检查是否为：时间戳、哈希、编码后的序列号等
    """
    print(f"样本数量: {len(sample_ids)}")
    print(f"样本值: {sample_ids}")
    
    # 检查是否为时间戳
    for sid in sample_ids:
        try:
            ts = int(sid)
            if 1600000000 <= ts <= 2000000000:  # 2020-2033时间戳范围
                from datetime import datetime
                print(f"  可能是Unix时间戳: {sid} -> {datetime.fromtimestamp(ts)}")
        except (ValueError, OverflowError):
            pass
    
    # 检查是否为自增ID的MD5/SHA哈希
    for i in range(1, 100):
        md5 = hashlib.md5(str(i).encode()).hexdigest()[:8]
        sha = hashlib.sha256(str(i).encode()).hexdigest()[:8]
        if md5 in [str(s) for s in sample_ids]:
            print(f"  可能是自增ID {i} 的MD5截断: {md5}")
        if sha in [str(s) for s in sample_ids]:
            print(f"  可能是自增ID {i} 的SHA256截断: {sha}")

# 示例：收集多个样本后分析
# analyze_id_pattern(["7a3f9b21", "c8e2d156", "4b7a6e33"])
```

**基于关联关系的 IDOR：** 有时直接枚举 ID 无效，但通过关联资源可以绕过：

```text
场景：订单详情接口 /api/orders/{order_id} 有权限校验
绕过：通过发票接口 /api/invoices/{invoice_id} 获取订单信息
      因为发票接口的权限校验可能遗漏

步骤：
1. 找到一个自己能访问的发票，获取 invoice_id
2. 通过发票接口，尝试使用其他用户的 invoice_id
3. 分析发票响应中是否包含 order_id
4. 利用订单详情接口测试该 order_id 是否可访问
```

### 14.17.3 垂直越权测试

垂直越权（Vertical Privilege Escalation）是指低权限用户获取高权限用户的能力。这通常意味着普通用户能够执行管理员功能，是最危险的访问控制缺陷之一。

#### 14.17.3.1 管理端点发现

**主动发现方法：**

```bash

# 方法2：JavaScript代码分析

grep -rohE '/admin[a-zA-Z0-9/_-]*' static/js/*.js | sort -u
grep -rohE '/manage[a-zA-Z0-9/_-]*' static/js/*.js | sort -u

# 方法3：API文档泄露
curl https://target.com/swagger.json 2>/dev/null | jq '.paths | keys[]' | grep -i admin
curl https://target.com/api-docs 2>/dev/null

# 方法4：Robots.txt和Sitemap
curl https://target.com/robots.txt

```

**被动发现方法：**

**场景一：直接访问管理端点**

```bash

curl -H "Authorization: Bearer <普通用户token>" \
     https://target.com/api/admin/users

# 对比管理员的响应，检查是否返回了相同的数据
```

**场景二：请求参数中的角色篡改**

```json

// 修改个人资料时提升权限

```

**场景三：HTTP 方法绕过**

**场景四：路径遍历绕过**

```bash
# 路径规范化绕过
```

**场景五：多步骤操作的权限绕过**

```text
管理功能可能需要多步操作：
  步骤1: 创建订单 (普通用户可访问)
  步骤2: 审批订单 (仅管理员可访问)
  步骤3: 执行订单 (普通用户可访问)

绕过方法：
  1. 以普通用户身份完成步骤1
  2. 捕获步骤1的响应，获取中间状态数据
  3. 直接构造步骤3的请求，跳过步骤2的审批
```

#### 14.17.3.3 JWT 相关的垂直越权

JWT（JSON Web Token）是现代 Web 应用中最常见的认证机制之一，但也引入了独特的攻击面：

```python
import base64
import json

def decode_jwt_payload(token):
    """解码JWT的payload部分（不验证签名）"""
    parts = token.split('.')
    if len(parts) != 3:
        return None
    
    payload = parts[1]
    # 补齐Base64填充
    payload += '=' * (4 - len(payload) % 4)
    decoded = base64.urlsafe_b64decode(payload)
    return json.loads(decoded)

# JWT攻击向量：
# 1. 算法混淆攻击（alg: none）
#    将算法改为none，移除签名部分
#
# 2. 算法替换（RS256 -> HS256）
#    使用公钥作为HMAC密钥伪造签名
#
# 3. 声明篡改
#    修改 role, sub, iss, aud 等字段
#    {"role": "admin", "sub": "admin_user_id"}

# 示例：篡改JWT中的角色字段
token = "eyJhbGciOiJIUzI1NiJ9.eyJyb2xlIjoiYWRtaW4iLCJzdWIiOiIxMDAxIn0.signature"
payload = decode_jwt_payload(token)
print(json.dumps(payload, indent=2))
```

**JWT 篡改测试清单：**

| 测试项 | 方法 | 工具 |
|-------|------|------|
| 修改 payload 中的 role 字段 | 将 `user` 改为 `admin` | jwt_tool, Burp JWT Editor |
| 算法设置为 none | 修改 header 中的 alg 为 None | jwt_tool |
| RS256 替换为 HS256 | 用公钥作为 HMAC 密钥 | jwt_tool |
| 删除签名 | 移除 JWT 第三部分 | 手动编辑 |
| 重放过期 token | 修改 exp 字段 | jwt_tool |
| kid 参数注入 | 通过 kid 注入路径遍历 | jwt_tool |

```bash
# 使用 jwt_tool 进行自动化JWT测试
python3 jwt_tool.py <token> -X a  # 所有攻击向量
python3 jwt_tool.py <token> -X k  # 密钥混淆
python3 jwt_tool.py <token> -Tam -pc role -pv admin  # 篡改claim
```

#### 14.17.4.2 API 端点权限测试自动化

```python
import requests
from dataclasses import dataclass
from typing import Optional

@dataclass
class Endpoint:
    method: str
    path: str
    data: Optional[dict] = None
    description: str = ""

def test_function_level_access(base_url, tokens, endpoints):
    """
    功能级权限自动化测试
    :param base_url: 基础URL
    :param tokens: dict，键为角色名，值为token
    :param endpoints: 要测试的端点列表
    """
    roles = list(tokens.keys())
    results = []
    
    for ep in endpoints:
        row = {"endpoint": f"{ep.method} {ep.path}", "description": ep.description}
        
        for role in roles:
            headers = {"Authorization": f"Bearer {tokens[role]}"}
            url = base_url + ep.path
            
            try:
                if ep.method == "GET":
                    resp = requests.get(url, headers=headers, timeout=10)
                elif ep.method == "POST":
                    resp = requests.post(url, headers=headers, json=ep.data, timeout=10)
                elif ep.method == "PUT":
                    resp = requests.put(url, headers=headers, json=ep.data, timeout=10)
                elif ep.method == "DELETE":
                    resp = requests.delete(url, headers=headers, timeout=10)
                else:
                    resp = requests.request(ep.method, url, headers=headers, timeout=10)
                
                row[role] = f"{resp.status_code}"
            except Exception as e:
                row[role] = f"ERR: {str(e)[:30]}"
        
        results.append(row)
    
    # 输出表格
    header = f"{'端点':<40} " + " ".join(f"{r:<15}" for r in roles)
    print(header)
    print("-" * len(header))
    for r in results:
        line = f"{r['endpoint']:<40} "
        line += " ".join(f"{r.get(role, 'N/A'):<15}" for role in roles)
        print(line)

# 使用示例
# tokens = {
#     "普通用户": "eyJ...", 
#     "管理员": "eyJ...",
#     "未认证": ""
# }
# endpoints = [
#     Endpoint("GET", "/api/admin/users", description="用户列表"),
#     Endpoint("DELETE", "/api/users/1002", description="删除用户"),
# ]
# test_function_level_access("https://target.com", tokens, endpoints)
```

### 14.17.5 多租户环境下的访问控制测试

多租户（Multi-Tenancy）架构在 SaaS 应用中极为普遍，但也带来了独特的安全挑战。

#### 14.17.5.1 租户隔离测试要点

```bash
# 测试维度：
# 1. 数据隔离 — 租户A能否访问租户B的数据
# 2. 功能隔离 — 租户A能否使用租户B的功能配置
# 3. 管理隔离 — 租户A的管理员能否管理租户B的用户
# 4. 资源隔离 — 租户A能否消耗租户B的配额
```

1. 配置低权限用户的 Cookie/Token
2. 浏览应用时，Autorize 自动用低权限凭证重放每个请求
3. 对比原始响应和低权限响应的差异

### 14.17.8 真实案例分析

#### 案例一：某电商平台订单 IDOR

```text
漏洞描述：
  电商平台的订单查询接口 /api/orders/{order_id} 使用自增整数ID，
  且未校验当前用户是否为订单所有者。

攻击过程：
  1. 用户A下单，获取自己的订单号 ORD-100001
  2. 修改为 ORD-100002，成功获取用户B的订单详情
  3. 订单详情包含：收货地址、手机号、商品信息、支付方式
  4. 批量遍历可获取数万用户的隐私信息

影响：
  - 大规模用户隐私泄露（姓名、地址、手机号）
  - 可用于精准诈骗（知道用户买了什么、在哪、电话多少）
  - 违反个人信息保护法，面临巨额罚款

修复方案：
  1. 使用UUID替代自增ID
  2. 在服务端强制校验 order.owner_id == current_user.id
  3. 添加访问日志审计，监控异常的订单查询行为
```

#### 案例二：SaaS 平台租户隔离失效

```text
漏洞描述：
  某 SaaS CRM 平台的 API 在请求头中接受 X-Tenant-ID，
  服务端直接使用该值进行数据查询，未验证当前用户是否属于该租户。

攻击过程：
  1. 攻击者在租户A注册普通用户账号
  2. 修改 API 请求头 X-Tenant-ID 为其他已知的租户ID
  3. 成功获取租户B的全部客户数据

影响：
  - 跨租户数据泄露，影响所有租户
  - SaaS 平台信任基础崩塌
  - 可能导致平台关停

修复方案：
  1. 租户标识从服务端session/数据库中获取，不信任客户端输入
  2. 所有数据库查询使用 tenant_id = session.tenant_id 强制过滤
  3. 实施行级安全策略（Row-Level Security）
```

#### 案例三：JWT 算法混淆导致管理员权限获取

```text
漏洞描述：
  应用使用 RS256 算法签名JWT，但服务端在验证时未限制算法类型。
  攻击者可将算法改为 HS256，使用公钥作为 HMAC 密钥伪造签名。

攻击过程：
  1. 获取服务端公钥（通常可从 /api/auth/jwks 或 /.well-known/ 获取）
  2. 构造包含 {"role":"admin"} 的JWT payload
  3. 使用 HS256 算法和公钥作为密钥生成签名
  4. 将伪造的JWT发送给服务端
  5. 服务端使用同一公钥进行 HMAC 验证 -> 签名验证通过

影响：
  - 完全绕过认证机制
  - 可伪造任意用户的令牌
  - 获取管理员权限

修复方案：
  1. 服务端硬编码允许的算法列表，不从JWT header中读取算法
  2. RS256和HS256使用完全不同的密钥
  3. 升级到支持JWK的JWT库，自动验证密钥-算法匹配
```

### 14.17.9 常见误区与纠正

| 误区 | 纠正 |
|------|------|
| "用了UUID就不存在IDOR" | UUID可能通过API文档、前端代码、错误信息等途径泄露 |
| "前端隐藏了管理入口就安全了" | 前端隐藏不等于后端有权限校验，必须服务端强制校验 |
| "每个接口都检查了登录状态就够了" | 认证（Authentication）≠ 授权（Authorization），登录不等于有权 |
| "只有管理员API需要权限校验" | 所有涉及敏感数据和操作的接口都需要权限校验 |
| "权限校验放在网关层就够了" | 网关只能做粗粒度校验，业务级权限必须在应用层实现 |
| "角色存在JWT中就安全了" | JWT中的角色信息可以被篡改，必须从数据库验证 |
| "HTTPS能防IDOR" | HTTPS只保护传输安全，不解决应用层的访问控制缺陷 |
| "没有敏感数据就不需要权限校验" | 即使数据不敏感，操作本身的越权（如删除他人评论）也是漏洞 |

### 14.17.11 防御方案

#### 14.17.11.1 服务端权限校验最佳实践

```python
# 错误示例：仅依赖客户端传递的用户ID
def get_order_bad(order_id):
    # 信任了客户端传递的order_id，未校验所有权
    return db.query("SELECT * FROM orders WHERE id = ?", order_id)

# 正确示例：强制从session获取用户身份，并验证资源所有权
def get_order_good(order_id, current_user):
    # Step 1: 从服务端session获取当前用户（不信任客户端）
    user_id = current_user.id
    
    # Step 2: 查询时同时校验资源所有权
    order = db.query(
        "SELECT * FROM orders WHERE id = ? AND owner_id = ?",
        order_id, user_id
    )
    
    # Step 3: 查询无结果时统一返回404（避免泄露资源存在性信息）
    if not order:
        abort(404)  # 而非403，防止攻击者通过状态码判断资源是否存在
    
    return order
```

#### 14.17.11.2 行级安全策略（数据库层面）

```sql
-- PostgreSQL 行级安全策略示例
ALTER TABLE orders ENABLE ROW LEVEL SECURITY;

-- 创建策略：用户只能访问自己的订单
CREATE POLICY user_orders_policy ON orders
    FOR ALL
    USING (owner_id = current_setting('app.current_user_id')::int);

-- 管理员可以访问所有订单
CREATE POLICY admin_orders_policy ON orders
    FOR ALL
    TO admin_role
    USING (true);

-- 通过设置会话变量来标识当前用户
SET app.current_user_id = '1001';
```

#### 14.17.11.3 中间件统一权限校验

```python
# 框架级权限校验中间件（以Flask为例）
from functools import wraps

def require_role(*required_roles):
    """角色校验装饰器"""
    def decorator(f):
        @wraps(f)
        def decorated_function(*args, **kwargs):
            if not current_user.is_authenticated:
                return jsonify({"error": "Authentication required"}), 401
            
            if current_user.role not in required_roles:
                return jsonify({"error": "Insufficient permissions"}), 403
            
            return f(*args, **kwargs)
        return decorated_function
    return decorator

def require_ownership(resource_model, id_param='id'):
    """资源所有权校验装饰器"""
    def decorator(f):
        @wraps(f)
        def decorated_function(*args, **kwargs):
            resource_id = kwargs.get(id_param)
            resource = resource_model.query.get(resource_id)
            
            if not resource:
                return jsonify({"error": "Not found"}), 404
            
            if resource.owner_id != current_user.id and current_user.role != 'admin':
                return jsonify({"error": "Not found"}), 404  # 返回404而非403
            
            kwargs['resource'] = resource
            return f(*args, **kwargs)
        return decorated_function
    return decorator

# 使用示例
@app.route('/api/orders/<int:id>')
@login_required
@require_ownership(Order, 'id')
def get_order(id, resource):
    return jsonify(resource.to_dict())

@app.route('/api/admin/users')
@login_required
@require_role('admin')
def admin_users():
    return jsonify(User.query.all())
```

#### 14.17.11.4 防御设计原则总结

| 原则 | 说明 | 实施要点 |
|------|------|---------|
| 默认拒绝 | 未明确授权的操作一律拒绝 | 使用白名单而非黑名单 |
| 最小权限 | 用户只获得完成工作所需的最小权限 | 角色权限定期审查和回收 |
| 服务端校验 | 所有权限判断在服务端完成 | 客户端校验仅用于UX优化 |
| 纵深防御 | 多层校验，不依赖单一控制点 | 网关+应用+数据库多层防护 |
| 统一框架 | 使用统一的权限校验框架 | 避免各接口自行实现校验逻辑 |
| 审计日志 | 记录所有敏感操作和越权尝试 | 异常行为自动告警 |
| 强制引用 | 资源引用从服务端上下文获取 | 不信任客户端传递的标识符 |
