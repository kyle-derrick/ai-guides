#### 1.3 编写第一个作业：Socket WordCount（15分钟）

这是Flink最经典的入门程序——从Socket文本流中读取单词并实时统计词频。

```java
import org.apache.flink.api.common.functions.FlatMapFunction;
import org.apache.flink.api.java.tuple.Tuple2;
import org.apache.flink.streaming.api.datastream.DataStream;
import org.apache.flink.streaming.api.environment.StreamExecutionEnvironment;
import org.apache.flink.util.Collector;

public class SocketWordCount {
    public static void main(String[] args) throws Exception {
        // 1. 创建执行环境
        StreamExecutionEnvironment env =
            StreamExecutionEnvironment.getExecutionEnvironment();
        env.setParallelism(1); // 练习阶段使用单并行度，方便调试

        // 2. 从Socket读取数据流
        DataStream<String> text = env.socketTextStream("localhost", 9999);

        // 3. 处理逻辑：分词 -> 转换为 (word, 1) -> 按word聚合
        DataStream<Tuple2<String, Integer>> counts = text
            .flatMap(new Tokenizer())
            .keyBy(value -> value.f0)   // 按单词分组
            .sum(1);                     // 对计数求和

        // 4. 输出结果
        counts.print();

        // 5. 触发执行
        env.execute("Socket WordCount");
    }

    public static class Tokenizer
            implements FlatMapFunction<String, Tuple2<String, Integer>> {
        @Override
        public void flatMap(String value, Collector<Tuple2<String, Integer>> out) {
            String[] tokens = value.toLowerCase().split("\\s+");
            for (String token : tokens) {
                if (!token.isEmpty()) {
                    out.collect(new Tuple2<>(token, 1));
                }
            }
        }
    }
}
```

### 练习二：窗口机制实验——从滚动窗口到会话窗口（预计60分钟）

**目标**：通过具体场景理解四种窗口类型的行为差异，掌握窗口函数的使用方法，能够在实际场景中选择合适的窗口策略。

#### 2.1 场景说明

假设你正在为一个电商网站构建实时统计系统，需要统计用户的订单行为。数据源是Kafka中的订单事件流，每个事件包含：用户ID、订单金额、事件时间戳。

#### 2.2 生成模拟数据流（10分钟）

```java
import org.apache.flink.streaming.api.functions.source.SourceFunction;
import java.util.Random;

public class OrderEventSource implements SourceFunction<OrderEvent> {
    private volatile boolean running = true;
    private final Random random = new Random();

    @Override
    public void run(SourceContext<OrderEvent> ctx) throws Exception {
        String[] userIds = {"user_01", "user_02", "user_03", "user_04", "user_05"};
        while (running) {
            String userId = userIds[random.nextInt(userIds.length)];
            double amount = 10 + random.nextDouble() * 990; // 10-1000元
            long eventTime = System.currentTimeMillis()
                           - random.nextInt(5000); // 模拟0-5秒的乱序
            ctx.collect(new OrderEvent(userId, amount, eventTime));
            Thread.sleep(100 + random.nextInt(400)); // 100-500ms间隔
        }
    }

    @Override
    public void cancel() { running = false; }
}

// 订单事件POJO
public class OrderEvent {
    public String userId;
    public double amount;
    public long eventTime;

    public OrderEvent() {}
    public OrderEvent(String userId, double amount, long eventTime) {
        this.userId = userId;
        this.amount = amount;
        this.eventTime = eventTime;
    }
}
```

#### 2.3 四种窗口对比实验（40分钟）

**实验1：滚动窗口（Tumbling Window）**——每5秒统计一次订单总金额

```java
DataStream<OrderEvent> orders = env.addSource(new OrderEventSource())
    .assignTimestampsAndWatermarks(
        WatermarkStrategy.<OrderEvent>forBoundedOutOfOrderness(Duration.ofSeconds(2))
            .withTimestampAssigner((event, timestamp) -> event.eventTime)
    );

// 滚动窗口：固定5秒，无重叠
orders.keyBy(e -> "global")  // 所有用户聚合到一个组
    .window(TumblingEventTimeWindows.of(Time.seconds(5)))
    .reduce((a, b) -> new OrderEvent("total", a.amount + b.amount, a.eventTime))
    .addSink(result -> System.out.println(
        "[Tumbling] 5秒窗口 | 总金额: " + String.format("%.2f", result.amount)
    ));
```

**预期行为**：每5秒输出一次，每个窗口的数据互不重叠。

**实验2：滑动窗口（Sliding Window）**——每2秒输出一次，窗口大小10秒

```java
// 滑动窗口：窗口10秒，每2秒滑动一次
orders.keyBy(e -> "global")
    .window(SlidingEventTimeWindows.of(Time.seconds(10), Time.seconds(2)))
    .reduce((a, b) -> new OrderEvent("total", a.amount + b.amount, a.eventTime))
    .addSink(result -> System.out.println(
        "[Sliding] 10秒窗口/2秒滑动 | 总金额: "
        + String.format("%.2f", result.amount)
    ));
```

**预期行为**：每2秒输出一次，但每次统计的是过去10秒的数据。同一笔订单会被多个窗口包含。

**实验3：会话窗口（Session Window）**——按用户活动间隔分组

```java
// 会话窗口：用户超过5秒无活动则关闭窗口
orders.keyBy(e -> e.userId)
    .window(EventTimeSessionWindows.withGap(Time.seconds(5)))
    .reduce((a, b) -> {
        double totalAmount = a.amount + b.amount;
        return new OrderEvent(a.userId, totalAmount, b.eventTime);
    })
    .addSink(result -> System.out.println(
        "[Session] 用户 " + result.userId
        + " | 会话内总金额: " + String.format("%.2f", result.amount)
    ));
```

**预期行为**：活跃用户（间隔<5秒）的订单被合并到同一个会话窗口；间隔超过5秒则触发输出并开启新窗口。

**实验4：全局窗口（Global Window）**——自定义计数触发器

```java
// 全局窗口 + 自定义触发器：每积累10条数据触发一次
orders.keyBy(e -> "global")
    .window(GlobalWindows.create())
    .trigger(CountTrigger.of(10))  // 每10条触发
    .reduce((a, b) -> new OrderEvent("total", a.amount + b.amount, a.eventTime))
    .addSink(result -> System.out.println(
        "[Global] 每10条触发 | 总金额: " + String.format("%.2f", result.amount)
    ));
```

**预期行为**：不按时间而是按数据条数触发，每累计10条订单事件输出一次。

#### 2.4 窗口行为对比表

运行以上四个实验后，记录并对比输出结果：

| 窗口类型 | 触发频率 | 同一数据被几个窗口包含 | 适用场景 | 典型应用 |
|----------|---------|----------------------|---------|---------|
| 滚动窗口 | 固定间隔 | 恰好1个 | 周期性聚合 | 每分钟PV统计、每小时销售额 |
| 滑动窗口 | 固定间隔 | 窗口大小/滑动步长 | 趋势分析 | 最近5分钟平均值、实时排行榜 |
| 会话窗口 | 数据驱动 | 1个（按活跃度分组） | 用户行为分析 | 浏览会话统计、在线时长 |
| 全局窗口 | 自定义触发器 | 取决于触发器 | 特殊聚合 | 每N条触发、外部事件触发 |

### 练习三：状态管理实验——Keyed State与Checkpoint（预计60分钟）

**目标**：理解Flink状态管理的核心机制，通过实验掌握Keyed State的使用方法，理解Checkpoint如何保证故障恢复后状态的一致性。

#### 3.1 Keyed State基础操作（25分钟）

**场景**：为每个用户维护最近3笔订单的金额，用于计算滑动平均值。

```java
import org.apache.flink.api.common.state.ListState;
import org.apache.flink.api.common.state.ListStateDescriptor;
import org.apache.flink.api.common.typeinfo.Types;
import org.apache.flink.runtime.state.FunctionInitializationContext;
import org.apache.flink.runtime.state.FunctionSnapshotContext;
import org.apache.flink.streaming.api.checkpoint.CheckpointedFunction;
import org.apache.flink.streaming.api.functions.KeyedProcessFunction;
import org.apache.flink.util.Collector;

public class SlidingAverageFunction
        extends KeyedProcessFunction<String, OrderEvent, Tuple3<String, Double, Integer>> {

    // Keyed State：每个用户一个独立的ListState
    private transient ListState<Double> recentAmounts;

    @Override
    public void open(Configuration parameters) {
        // 初始化状态描述符
        ListStateDescriptor descriptor = new ListStateDescriptor<>(
            "recent-amounts",   // 状态名称（用于Checkpoint序列化）
            Types.DOUBLE         // 状态值类型
        );
        recentAmounts = getRuntimeContext().getListState(descriptor);
    }

    @Override
    public void processElement(
            OrderEvent event,
            Context ctx,
            Collector<Tuple3<String, Double, Integer>> out) throws Exception {

        // 1. 将新金额加入状态
        recentAmounts.add(event.amount);

        // 2. 读取所有金额，保留最近3笔
        List<Double> amounts = new ArrayList<>();
        for (Double amount : recentAmounts.get()) {
            amounts.add(amount);
        }

        // 只保留最近3笔
        while (amounts.size() > 3) {
            amounts.remove(0);
        }

        // 更新状态（清除旧数据，写入新数据）
        recentAmounts.update(amounts);

        // 3. 计算滑动平均值
        double average = amounts.stream()
            .mapToDouble(Double::doubleValue)
            .average()
            .orElse(0.0);

        out.collect(Tuple3.of(event.userId, average, amounts.size()));
    }
}
```

### 练习四：CEP模式匹配——构建实时欺诈检测（预计60分钟）

**目标**：掌握Flink CEP（Complex Event Processing）的核心用法，能够设计和实现基于事件模式的实时检测规则。

#### 4.1 场景说明

构建一个信用卡欺诈检测系统，需要检测以下异常模式：

- **规则1**：同一用户在30秒内连续3笔以上交易，每笔金额超过5000元
- **规则2**：同一用户在5分钟内交易金额累计超过50000元
- **规则3**：同一用户在10分钟内先有一笔大额交易（>20000元），紧接着有一笔小额交易（<100元）（可能是盗刷后试探）

#### 4.2 定义事件模式（20分钟）

```java
import org.apache.flink.cep.CEP;
import org.apache.flink.cep.PatternSelectFunction;
import org.apache.flink.cep.PatternStream;
import org.apache.flink.cep.pattern.Pattern;
import org.apache.flink.cep.pattern.conditions.SimpleCondition;

// 交易事件
public class TransactionEvent {
    public String userId;
    public double amount;
    public String location;
    public long eventTime;

    // 构造器、getter省略
}
```

**规则1模式：连续大额交易**

```java
Pattern<TransactionEvent, ?> highValuePattern = Pattern
    .<TransactionEvent>begin("first")  // 第一笔
    .where(new SimpleCondition<TransactionEvent>() {
        @Override
        public boolean filter(TransactionEvent event) {
            return event.amount > 5000;
        }
    })
    .timesOrMore(3)  // 至少连续3笔
    .within(Time.seconds(30));  // 30秒窗口内

PatternStream<TransactionEvent> patternStream = CEP.pattern(
    orders.keyBy(e -> e.userId),
    highValuePattern
);

patternStream.select(new PatternSelectFunction<TransactionEvent, String>() {
    @Override
    public String select(Map<String, List<TransactionEvent>> pattern) {
        List<TransactionEvent> events = pattern.get("first");
        StringBuilder sb = new StringBuilder();
        sb.append("[欺诈告警-规则1] 用户 ")
          .append(events.get(0).userId)
          .append(" 在30秒内连续").append(events.size())
          .append("笔大额交易，总额: ");

        double total = events.stream().mapToDouble(e -> e.amount).sum();
        sb.append(String.format("%.2f", total));
        return sb.toString();
    }
});
```

```java
    .followedBy("small")  // 紧接着（不一定立即，但中间不能有其他大额交易）
    .where(new SimpleCondition<TransactionEvent>() {
        @Override
        public boolean filter(TransactionEvent event) {
            return event.amount < 100;
        }
    })
    .within(Time.minutes(10));

// 输出告警
CEP.pattern(orders.keyBy(e -> e.userId), stolenCardPattern)
    .select((PatternSelectFunction<TransactionEvent, String>) pattern -> {
        TransactionEvent large = pattern.get("large").get(0);
        TransactionEvent small = pattern.get("small").get(0);
        return String.format(
            "[欺诈告警-规则3] 用户 %s 先大额(%.2f)后小额(%.2f)，疑似盗刷",
            large.userId, large.amount, small.amount
        );
    })
    .addSink(alert -> System.out.println(alert));
```

**目标**：理解Flink反压机制的工作原理，能够通过实验观察反压现象，掌握诊断和优化反压问题的方法。

#### 5.1 构造反压场景（25分钟）

**设计思路**：上游Source高速产生数据，中间算子做CPU密集计算（制造速度差），下游Sink慢速写入，形成反压链。

```java
public class BackpressureDemo {
    public static void main(String[] args) throws Exception {
        StreamExecutionEnvironment env =
            StreamExecutionEnvironment.getExecutionEnvironment();
        env.setParallelism(4);

        // Stage 1: 高速Source（每秒产生10000条数据）
        DataStream<String> source = env
            .addSource(new HighThroughputSource(10000)) // 10K events/sec
            .name("HighSpeedSource");

        // Stage 2: CPU密集型算子（故意做耗时计算，制造瓶颈）
        DataStream<String> processed = source
            .map(value -> {
                // 模拟复杂计算：每条数据耗时约2ms
                // 10000条/秒 × 2ms = 20秒/秒 → 远超1秒，产生反压
                heavyComputation(value);
                return value;
            })
            .name("HeavyComputation")
            .setParallelism(2); // 故意降低并行度，加剧瓶颈

        // Stage 3: 慢速Sink（每秒只能写入100条）
        processed.addSink(new SlowSink(100)) // 100 events/sec
            .name("SlowSink");

        env.execute("Backpressure Demo");
    }

    // 模拟CPU密集计算
    static void heavyComputation(String value) {
        double result = 0;
        for (int i = 0; i < 100000; i++) {
            result += Math.sin(i) * Math.cos(i);
        }
    }
}

// 高吞吐Source
class HighThroughputSource extends RichParallelSourceFunction<String> {
    private volatile boolean running = true;
    private final int targetRate;

    public HighThroughputSource(int targetRate) {
        this.targetRate = targetRate;
    }

    @Override
    public void run(SourceContext<String> ctx) throws Exception {
        int subtaskIndex = getRuntimeContext().getIndexOfThisSubtask();
        int totalSubtasks = getRuntimeContext().getNumberOfParallelSubtasks();
        int perSubtaskRate = targetRate / totalSubtasks;

        long intervalUs = 1_000_000 / perSubtaskRate; // 微秒级控制
        while (running) {
            long start = System.nanoTime();
            ctx.collect("event_" + subtaskIndex + "_" + System.currentTimeMillis());
            long elapsed = (System.nanoTime() - start) / 1000;
            if (elapsed < intervalUs) {
                Thread.sleep(0, (int)((intervalUs - elapsed) * 1000));
            }
        }
    }

    @Override
    public void cancel() { running = false; }
}

// 慢速Sink
class SlowSink extends RichSinkFunction<String> {
    private final int maxRate;
    private long count = 0;
    private long windowStart = System.currentTimeMillis();

    public SlowSink(int maxRate) { this.maxRate = maxRate; }

    @Override
    public void invoke(String value, Context context) throws Exception {
        count++;
        long now = System.currentTimeMillis();
        if (now - windowStart >= 1000) {
            // 每秒最多处理maxRate条
            count = 0;
            windowStart = now;
        } else if (count > maxRate) {
            Thread.sleep(10); // 超过速率限制时主动等待
        }
    }
}
```

#### 5.3 反压诊断与优化（35分钟）

**诊断清单**：

| 诊断项 | 检查方法 | 正常值 | 异常值 |
|--------|---------|--------|--------|
| CPU使用率 | `top -p <pid>` 或Dashboard Metrics | <70% | >90%（CPU瓶颈） |
| GC暂停 | GC日志或 `jstat -gcutil <pid> 1000` | <100ms | >500ms（GC瓶颈） |
| 网络带宽 | `iftop` 或 Dashboard Network | <80% | >90%（网络瓶颈） |
| 磁盘IO | `iostat -x 1` | %util<80% | %util>95%（IO瓶颈） |
| 缓冲区使用率 | Dashboard Backpressure页 | <80% | >95%（反压严重） |

**优化策略对照表**：

| 瓶颈类型 | 优化方向 | 具体措施 |
|----------|---------|---------|
| CPU密集型算子 | 增加并行度 / 优化算法 | `setParallelism(8)`；用向量运算替代循环 |
| IO密集型算子 | 异步IO / 批量写入 | 使用 `AsyncDataStream`；Sink端攒批后批量INSERT |
| 数据倾斜 | 加盐分散热点Key | `keyBy(e -> e.userId + "_" + random.nextInt(10))` |
| 网络瓶颈 | 减少Shuffle / 本地聚合 | 开启Mini-Batch；使用 `reduce` 而非 `process` |
| 反序列化开销 | 使用POJO替代Row | 确保字段是public且有无参构造器 |

**实验：优化反压**

```java
// 优化1：增加瓶颈算子的并行度
processed.setParallelism(8); // 从2改为8

// 优化2：开启Mini-Batch（减少网络shuffle次数）
TableConfig config = tableEnv.getConfig();
config.set("table.exec.mini-batch.enabled", "true");
config.set("table.exec.mini-batch.allow-latency", "5s");
config.set("table.exec.mini-batch.size", "5000");

// 优化3：异步IO写入（如果Sink是外部数据库）
AsyncDataStream.unorderedWait(
    processed,
    new AsyncDatabaseSink(),
    30, TimeUnit.SECONDS,  // 超时
    100                     // 最大并发请求数
);
```

#### 常见问题排查

| 问题现象 | 可能原因 | 解决方案 |
|----------|---------|---------|
| 作业提交后立即失败 | JDK版本不匹配 | 确认使用JDK 11+，`java -version` 检查 |
| Checkpoint频繁失败 | 状态后端存储不可用 | 检查HDFS/本地路径权限和空间 |

| 内存溢出（OOM） | StateBackend选择不当 | 小状态用HashMap，大状态切换到RocksDB |
