## 超时与重试

在 RPC（Remote Procedure Call）框架中，超时与重试是最关键的容错机制之一。一个请求从客户端发出，经过网络传输、服务端处理、结果回传，任何一个环节出现延迟或失败，都可能导致调用链的连锁故障。如果没有合理的超时设置和重试策略，单个慢请求就能拖垮整个系统——这就是著名的"尾延迟放大"（Tail Latency Amplification）问题。

本章从理论基础出发，系统讲解 RPC 框架中超时与重试的设计原理、配置策略、常见陷阱和最佳实践。

---

### 1. 为什么超时与重试如此重要

#### 1.1 没有超时的灾难

假设一个 RPC 调用链：`A → B → C → D`，每个环节正常耗时 50ms。如果 D 服务出现故障，响应时间飙升到 30 秒：

正常情况：A(50ms) → B(50ms) → C(50ms) → D(50ms) = 200ms 总耗时
故障情况：A(30s+) → B(30s+) → C(30s+) → D(30s) = 90s+ 总耗时

没有超时的情况下，这个故障会沿着调用链向上蔓延。更可怕的是，当大量请求堆积时：

- 服务端线程池被耗尽，无法处理新请求
- TCP 连接数暴涨，文件描述符耗尽
- 上游服务的线程池也被阻塞，形成**级联故障**（Cascading Failure）
- 最终整个微服务集群雪崩

Netflix 在 2012 年的一次故障中，一个服务的响应变慢导致了数百个下游服务的连锁超时，最终整个推荐系统不可用超过 1 小时。这次事故直接催生了 Hystrix 熔断器的诞生。

#### 1.2 没有重试的脆弱

网络是不可靠的。根据 Google 的公开数据，在大规模集群中，每天都有机器宕机、网络抖动、磁盘故障发生。如果遇到瞬时故障就放弃，可用性将大打折扣：

假设单次调用成功率为 99.9%
无重试：可用性 = 99.9%
1次重试：可用性 = 1 - (1-0.999)² = 99.999%
2次重试：可用性 = 1 - (1-0.999)³ = 99.99999%

但重试不是免费的——无节制的重试会放大故障流量，让本已承压的服务雪上加霜。因此，**超时和重试必须配合设计**，既要保证正常请求的快速失败，又要确保瞬时故障的自动恢复。

#### 1.3 关键概念辨析

| 概念 | 定义 | 作用 |
|------|------|------|
| 超时（Timeout） | 等待响应的最大时间限制 | 防止无限等待，保护调用方资源 |
| 重试（Retry） | 失败后重新发送请求 | 自动恢复瞬时故障 |
| 截止时间（Deadline） | 整个请求链路的绝对截止时间 | 端到端的全局时间约束 |
| 熔断（Circuit Breaker） | 检测到持续故障后快速失败 | 防止故障扩散，保护下游服务 |
| 重试预算（Retry Budget） | 限制重试流量的上限比例 | 防止重试风暴 |

### 2. 超时机制详解

#### 2.1 超时的分层模型

一个 RPC 请求的生命周期中，有多个可以设置超时的层级：

┌─────────────────────────────────────────────────┐
│                   客户端                         │
│  ┌──────────── 调用总超时 ──────────────┐       │
│  │                                       │       │
│  │  ┌─── 连接超时 ───┐                  │       │
│  │  │ TCP 三次握手     │                  │       │
│  │  └─────────────────┘                  │       │
│  │                                       │       │
│  │  ┌───── 等待超时 ─────────────────┐  │       │
│  │  │ 请求已发送，等待响应              │  │       │
│  │  └────────────────────────────────┘  │       │
│  │                                       │       │
│  └───────────────────────────────────────┘       │
│                                                  │
│  ┌─── 读超时 ────┐  ┌─── 写超时 ────┐           │
│  │ 读取响应数据   │  │ 发送请求数据   │           │
│  └───────────────┘  └───────────────┘           │
└─────────────────────────────────────────────────┘

##### 连接超时（Connect Timeout）

客户端与服务端建立 TCP 连接的最大等待时间。通常设置为 1-5 秒。

```python
# gRPC 连接超时设置
import grpc

channel = grpc.insecure_channel(
    'service:50051',
    options=[
        ('grpc.connect_timeout_ms', 3000),   # 3秒连接超时
        ('grpc.keepalive_time_ms', 10000),   # 10秒心跳间隔
    ]
)
```

**为什么连接超时要独立设置？** 因为连接失败通常意味着目标服务完全不可达（宕机、网络分区），此时重试的意义不大（除非目标地址有多个实例），快速失败并切换到其他实例才是正确策略。

##### 读超时 / 写超时（Read/Write Timeout）

- **写超时**：发送请求数据到对端的最大时间，受限于带宽和对端接收能力
- **读超时**：等待并读取响应数据的最大时间，受服务处理时间和网络传输影响

```go
// Go net/http 的超时拆分
server := &amp;http.Server{
    ReadTimeout:    5 * time.Second,   // 读取完整请求的超时
    WriteTimeout:   10 * time.Second,  // 读取请求 + 处理 + 写入响应
    IdleTimeout:    120 * time.Second, // keep-alive 空闲超时
}
```

##### 调用超时 / 总超时（Call Timeout）

从客户端发起调用到收到完整响应（或抛出异常）的总时间限制。这是业务开发者最常配置的超时参数。

```java
// Java gRPC 超时
ManagedChannel channel = ManagedChannelBuilder.forAddress("service", 50051).build();
MyServiceGrpc.MyServiceFutureStub stub = MyServiceGrpc.newFutureStub(channel);

// 设置调用超时为 5 秒
ListenableFuture<Reply> future = stub.withDeadlineAfter(5, TimeUnit.SECONDS)
    .myMethod(request);
```

#### 2.2 Deadline：超越单次调用的全局超时

Deadline（截止时间）是超时机制的高级形态。与单次调用超时不同，Deadline 是**整个调用链路共享的绝对截止时间**。

客户端发起请求，设置 Deadline = 5 秒
  ├── A 服务耗时 1 秒
  ├── B 服务耗时 1 秒（剩余 3 秒）
  ├── C 服务耗时 1 秒（剩余 2 秒）
  └── D 服务收到请求，发现只剩 2 秒，即使正常需要 3 秒也能提前终止

**核心优势：**
- 避免"超时预算浪费"：如果 A 和 B 各自设置 5 秒超时，D 实际只在最后 1 秒才开始执行，但 A 和 B 已经消耗了 2 秒
- 自动传播：上游设置的 Deadline 会自动传递给下游调用，无需手动计算剩余时间

```protobuf
// gRPC 原生支持 Deadline 传播
// 客户端
grpcCtx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
resp, err := client.MyMethod(grpcCtx, req)
// 服务端收到的 context 会自动携带剩余 Deadline
```

**gRPC 的 Deadline 传播机制：** gRPC 通过 `grpc-timeout` 和 `grpc-deadline` 元数据头自动在服务间传递截止时间。当服务 A 调用服务 B 时，gRPC 会自动将剩余时间作为 B 的超时发送过去，无需业务代码介入。

#### 2.3 超时设置的黄金法则

##### 法则一：超时值必须基于真实数据

不要拍脑袋定超时值。通过压测和线上监控数据确定合理的超时值：

```python
# 通过 Prometheus 查询 P99 延迟来设定超时
# 假设某服务 P99 延迟为 200ms，P99.9 为 800ms

# 保守策略：P99.9 的 2 倍
timeout_conservative = 800 * 2  # 1600ms

# 激进策略：P99 的 3 倍
timeout_aggressive = 200 * 3    # 600ms

# 通常选择：P99 延迟的 3-5 倍，或至少覆盖到 P99.9
recommended_timeout = max(p99 * 3, p99_9 * 1.5)
```

##### 法则二：调用链越深，单次超时越短

在多层调用链中，每层的超时应该为下一层留出余量。经验公式：

L(n) = T_total / n × α

其中 `n` 是调用链深度，`α` 是修正系数（通常取 0.6-0.8，给业务处理留空间）。

```python
# 调用链：用户 → 网关(100ms) → 业务(500ms) → 数据库(200ms)
# 总超时 2 秒

gateway_timeout = 500      # 500ms，给下游留 1.5s
business_timeout = 1000    # 1s，给数据库留 500ms
database_timeout = 300     # 300ms，给数据库本身留余量
```

##### 法则三：超时设置要分层，不要"一刀切"

不同类型的操作有不同的合理耗时范围：

| 操作类型 | 典型耗时 | 建议超时 | 说明 |
|----------|---------|---------|------|
| 简单查询（缓存命中） | 1-10ms | 50-100ms | 缓存场景，快速失败 |
| 数据库单表查询 | 10-100ms | 500ms | 考虑索引未命中的情况 |
| 复杂业务计算 | 100-2000ms | 3-5s | 需要为大请求留空间 |
| 文件上传/下载 | 变化大 | 30-60s | 按文件大小动态调整 |
| 跨机房调用 | 50-200ms | 2-3s | 网络延迟波动大 |

### 3. 重试机制详解

#### 3.1 重试的基本策略

##### 简单重试（Fixed Interval）

以固定间隔重试，实现最简单，但有严重缺陷：

```python
import time

def retry_fixed(func, max_retries=3, interval=1.0):
    """简单重试：固定间隔"""
    for attempt in range(max_retries + 1):
        try:
            return func()
        except Exception as e:
            if attempt == max_retries:
                raise
            print(f"第 {attempt+1} 次重试，等待 {interval}s...")
            time.sleep(interval)
```

**缺陷：** 所有客户端在同一时间重试，形成"重试风暴"（Thundering Herd），瞬间放大流量数倍。

##### 指数退避（Exponential Backoff）

每次重试的等待时间指数级增长，是业界标准策略：

```python
import time
import random

def retry_exponential_backoff(func, max_retries=5, base_delay=0.1, max_delay=30):
    """指数退避重试"""
    for attempt in range(max_retries + 1):
        try:
            return func()
        except Exception as e:
            if attempt == max_retries:
                raise
            # 指数增长：0.1s → 0.2s → 0.4s → 0.8s → 1.6s
            delay = min(base_delay * (2 ** attempt), max_delay)
            print(f"第 {attempt+1} 次重试，等待 {delay:.2f}s...")
            time.sleep(delay)
```

重试间隔序列（base=100ms, max=30s）：
第1次：100ms
第2次：200ms
第3次：400ms
第4次：800ms
第5次：1600ms
第6次：3200ms
第7次：6400ms
第8次：12800ms
第9次：25600ms
第10次：30000ms（被 max_delay 截断）

##### 指数退避 + 随机抖动（Exponential Backoff with Jitter）

在指数退避基础上加入随机抖动，是 Google 推荐的最佳实践：

```python
import time
import random

def retry_backoff_jitter(func, max_retries=5, base_delay=0.1, max_delay=30):
    """指数退避 + 随机抖动"""
    for attempt in range(max_retries + 1):
        try:
            return func()
        except Exception as e:
            if attempt == max_retries:
                raise
            # 三种抖动策略：
            # 1. Full Jitter: random(0, delay)  — 范围最大
            # 2. Equal Jitter: delay/2 + random(0, delay/2) — 保底一半
            # 3. Decorrelated: min(max, random(base, prev*3)) — 与前次无关
            
            # Equal Jitter 是最推荐的折中方案
            exponential_delay = min(base_delay * (2 ** attempt), max_delay)
            jitter = random.uniform(0, exponential_delay * 0.5)
            delay = exponential_delay * 0.5 + jitter
            time.sleep(delay)
```

**三种抖动策略对比：**

Full Jitter：     delay = random(0, base × 2^attempt)
                  范围最广，退避最彻底，但可能等太久

Equal Jitter：    delay = base × 2^attempt / 2 + random(0, base × 2^attempt / 2)
                  保底最小延迟，同时有随机性，最推荐

Decorrelated：    delay = min(max_delay, random(base, prev_delay × 3))
                  每次独立计算，不依赖尝试次数，适合长连接场景

```python
def jitter_full(base, attempt, max_delay):
    return random.uniform(0, min(base * (2 ** attempt), max_delay))

def jitter_equal(base, attempt, max_delay):
    exp = min(base * (2 ** attempt), max_delay)
    return exp / 2 + random.uniform(0, exp / 2)

def jitter_decorrelated(base, attempt, max_delay, prev_delay=0):
    return min(max_delay, random.uniform(base, prev_delay * 3))
```

#### 3.2 可重试的错误分类

**不是所有错误都应该重试。** 正确区分错误类型是重试策略的基础：

┌─────────────────────────────────────────────┐
│              错误类型分类                     │
├─────────────┬───────────────┬───────────────┤
│  可重试     │  不可重试      │  条件可重试    │
│ (Retryable) │ (Non-retryable)│ (Conditional) │
├─────────────┼───────────────┼───────────────┤
│ 连接超时     │ 参数错误(400)  │ 服务不可用     │
│ 读超时      │ 认证失败(401)  │  限流(429)    │
│ DNS 解析失败 │ 权限不足(403)  │ 503 服务维护   │
│ 连接被重置   │ 资源不存在(404)│               │
│ 500 内部错误│ 方法不允许(405)│               │
│ 502 网关错误│ 业务逻辑错误   │               │
│ 503 服务不可用│               │               │
│ 504 网关超时 │               │               │
└─────────────┴───────────────┴───────────────┘

**核心原则：**

- **可重试错误**：通常是暂时性的网络或系统故障，重试后大概率成功
- **不可重试错误**：通常是确定性失败（参数错、权限不足），重试多少次结果都一样
- **条件可重试错误**：需要根据响应头（如 `Retry-After`）判断是否重试及重试间隔

```python
from enum import Enum

class RetryPolicy(Enum):
    RETRY = "retry"           # 立即重试
    RETRY_AFTER = "retry_after" # 按响应指示等待后重试
    NO_RETRY = "no_retry"     # 不重试

def classify_error(status_code, exception):
    """错误分类：决定是否重试"""
    # 网络层错误
    if isinstance(exception, (ConnectionError, TimeoutError)):
        return RetryPolicy.RETRY
    
    # HTTP 状态码
    if status_code in (408, 429, 500, 502, 503, 504):
        if status_code == 429:
            return RetryPolicy.RETRY_AFTER  # 限流，需要等待
        return RetryPolicy.RETRY
    
    # 确定性错误，不重试
    if status_code in (400, 401, 403, 404, 405, 422):
        return RetryPolicy.NO_RETRY
    
    # 未知状态码，保守不重试
    return RetryPolicy.NO_RETRY
```

#### 3.3 幂等性：重试的安全前提

**重试的前提是操作是幂等的（Idempotent）。** 幂等意味着同一操作执行一次和执行多次的结果相同。

| 操作类型 | 幂等？ | 说明 |
|----------|--------|------|
| GET 查询 | ✅ 是 | 只读操作，天然幂等 |
| PUT 更新 | ✅ 是 | 整体替换，执行多次结果相同 |
| DELETE 删除 | ✅ 是 | 删除已删除的资源，结果不变 |
| POST 创建 | ❌ 否 | 每次执行可能创建新记录 |
| 增量扣款 | ❌ 否 | 每次执行金额会多扣 |

**非幂等操作的安全重试方案：**

```python
import uuid
import hashlib

class IdempotentClient:
    def __init__(self, rpc_client):
        self.rpc_client = rpc_client
        self.idempotency_cache = {}
    
    def call_with_idempotency(self, method, request):
        """
        通过幂等键保证非幂等操作的安全重试
        
        思路：
        1. 为每次调用生成唯一幂等键（基于请求内容的哈希）
        2. 首次调用时缓存幂等键
        3. 重试时携带同一幂等键
        4. 服务端根据幂等键判断是否重复请求
        """
        # 生成幂等键：请求方法 + 请求体的哈希
        idempotency_key = hashlib.sha256(
            f"{method}:{request.SerializeToString()}".encode()
        ).hexdigest()
        
        # 检查本地缓存：如果之前成功过，直接返回缓存结果
        if idempotency_key in self.idempotency_cache:
            return self.idempotency_cache[idempotency_key]
        
        # 发起调用，携带幂等键作为 metadata
        metadata = {"idempotency-key": idempotency_key}
        result = self.rpc_client.call(method, request, metadata=metadata)
        
        # 成功则缓存结果
        self.idempotency_cache[idempotency_key] = result
        return result
```

**服务端幂等键处理：**

```python
# 服务端实现（以 gRPC 拦截器为例）
class IdempotencyInterceptor:
    def __init__(self):
        self.seen_keys = {}  # 实际生产中用 Redis
    
    def intercept(self, handler, request, context):
        idempotency_key = context.invocation_metadata.get("idempotency-key")
        
        if idempotency_key:
            if idempotency_key in self.seen_keys:
                # 重复请求，直接返回上次的结果
                return self.seen_keys[idempotency_key]
            
            result = handler(request, context)
            self.seen_keys[idempotency_key] = result
            return result
        else:
            return handler(request, context)
```

#### 3.4 重试预算（Retry Budget）

Google SRE 实践中提出的重要概念：限制重试流量不超过正常流量的一定比例（通常 10%）。

**为什么需要重试预算？**

假设系统有 1000 QPS 的正常流量：
- 无重试预算：如果 30% 的请求失败，每次重试 2 次，则重试流量 = 1000 × 30% × 2 = 600 QPS，总流量 = 1600 QPS
- 有重试预算（10%）：重试流量上限 = 1000 × 10% = 100 QPS，超出部分直接失败

```python
import threading
import time

class RetryBudget:
    """重试预算管理器
    
    滑动窗口算法：统计最近 window_size 秒内的重试次数，
    确保重试率不超过 max_retry_ratio
    """
    def __init__(self, window_size=10, max_retry_ratio=0.1):
        self.window_size = window_size       # 滑动窗口大小（秒）
        self.max_retry_ratio = max_retry_ratio  # 最大重试比例（10%）
        self.request_count = 0               # 窗口内正常请求数
        self.retry_count = 0                 # 窗口内重试请求数
        self.lock = threading.Lock()
        self.window_start = time.time()
    
    def can_retry(self):
        """判断当前是否允许重试"""
        with self.lock:
            self._rotate_window()
            total = self.request_count + self.retry_count
            if total == 0:
                return True
            
            current_ratio = self.retry_count / total
            return current_ratio < self.max_retry_ratio
    
    def record_request(self):
        with self.lock:
            self._rotate_window()
            self.request_count += 1
    
    def record_retry(self):
        with self.lock:
            self._rotate_window()
            self.retry_count += 1
    
    def _rotate_window(self):
        now = time.time()
        if now - self.window_start >= self.window_size:
            self.request_count = 0
            self.retry_count = 0
            self.window_start = now
```

### 4. 超时与重试的协作设计

#### 4.1 超时 + 重试的交互模型

请求发出
  │
  ├── 响应成功 → 返回结果
  │
  ├── 超时触发 → 判断错误类型
  │     ├── 可重试 + 预算允许 → 重试（等待退避时间）
  │     ├── 可重试 + 预算耗尽 → 快速失败
  │     └── 不可重试 → 快速失败
  │
  └── 业务错误 → 判断是否重试
        ├── 可重试（如 503）→ 重试
        └── 不可重试（如 400）→ 快速失败

**关键约束：重试必须在总超时内完成。**

```python
import time

def call_with_timeout_retry(func, timeout=5.0, max_retries=3, 
                             base_delay=0.1, retry_budget=None):
    """带超时和重试的 RPC 调用
    
    核心设计：
    1. 使用总截止时间控制全局超时
    2. 每次重试都要检查是否还有剩余时间
    3. 如果退避等待时间超过剩余时间，直接失败
    """
    deadline = time.time() + timeout
    last_exception = None
    
    for attempt in range(max_retries + 1):
        try:
            return func()
        except Exception as e:
            last_exception = e
            
            # 判断是否可重试
            if not is_retryable(e):
                raise
            
            # 检查重试预算
            if retry_budget and not retry_budget.can_retry():
                raise Exception("重试预算耗尽")
            
            # 计算退避时间
            delay = min(base_delay * (2 ** attempt), 30)
            jitter = random.uniform(0, delay * 0.3)
            wait_time = delay * 0.7 + jitter
            
            # 关键：检查退避后是否还有时间
            remaining = deadline - time.time()
            if wait_time >= remaining:
                raise Exception(f"重试等待时间({wait_time:.2f}s)超过剩余时间({remaining:.2f}s)")
            
            time.sleep(wait_time)
    
    raise last_exception
```

#### 5.1 基于 Hedged Request 的延迟优化

Hedged Request（对冲请求）是一种用空间换时间的策略：同时向多个实例发送请求，取最快返回的结果，取消其余。

```python
import asyncio
import time

async def hedged_request(funcs, hedge_delay=50, timeout=5000):
    """
    对冲请求：在主请求发出 hedge_delay 毫秒后，
    如果还没收到响应，则同时向备用实例发送相同请求。
    
    适用场景：
    - 对延迟敏感的读操作
    - 服务实例间性能差异大的场景
    
    注意：
    - 仅适用于幂等操作（读请求）
    - 需要设置合理的对冲延迟（通常为 P99 延迟的 10-30%）
    - 对冲流量有上限，不能无限制发送
    """
    deadline = time.time() + timeout / 1000
    
    async def call_with_retry(func, attempt=0):
        try:
            return await asyncio.wait_for(func(), timeout=timeout/1000)
        except asyncio.TimeoutError:
            if attempt < 2:  # 最多重试 1 次
                return await call_with_retry(func, attempt + 1)
            raise
    
    # 主请求
    primary_task = asyncio.create_task(call_with_retry(funcs[0]))
    
    # 对冲延迟后，如果主请求未完成，发起备用请求
    if len(funcs) > 1:
        await asyncio.sleep(hedge_delay / 1000)
        if not primary_task.done():
            backup_task = asyncio.create_task(call_with_retry(funcs[1]))
            # 等待两个任务，取最先完成的
            done, pending = await asyncio.wait(
                [primary_task, backup_task],
                return_when=asyncio.FIRST_COMPLETED
            )
            for task in pending:
                task.cancel()
            return done.pop().result()
    
    return await primary_task
```

**Hedged Request 的流量开销分析：**

对冲延迟 = P99 延迟的 10%
对冲比例 ≈ 10% 的请求会触发对冲（因为 90% 在 P99 内完成）
总流量增加 ≈ 10%（主请求100% + 对冲请求10%）
延迟改善：P99 从 P99 降至 P50 水平

### 6. 常见陷阱与避坑指南

#### 陷阱一：重试风暴（Thundering Herd）

**场景：** 服务抖动恢复瞬间，所有客户端同时重试，导致服务再次被打挂。

**解决方案：** 在指数退避基础上加入随机抖动（见 3.1 节），并配合重试预算限制总重试流量。

```python
# ❌ 错误：固定间隔重试，所有客户端同步重试
time.sleep(1.0)  # 所有人在 t+1s 同时重试

# ✅ 正确：指数退避 + 随机抖动
delay = base * (2 ** attempt) * random.uniform(0.5, 1.5)
time.sleep(delay)
```

#### 陷阱二：重试非幂等操作

**场景：** 对 POST/PUT 等非幂等操作进行重试，导致重复创建或重复扣款。

**解决方案：**
1. 尽量将非幂等操作转化为幂等操作（使用幂等键）
2. 对写操作默认不重试（`retries: 0`）
3. 必须重试时，使用请求级别的唯一 ID + 服务端去重

```java
// ❌ 错误：对扣款操作重试
@Retryable(maxAttempts = 3)  // 如果第一次成功了但响应丢失，重试会多扣一次
public void deductBalance(Order order) { ... }

// ✅ 正确：使用幂等键
public void deductBalance(Order order, String idempotencyKey) {
    if (deductionExists(idempotencyKey)) {
        return;  // 已执行过，跳过
    }
    doDeduction(order);
    markDeductionComplete(idempotencyKey);
}
```

#### 陷阱三：超时值设置不合理

**场景：** 超时值设得太短，正常请求也被超时；设得太长，故障恢复慢。

**常见错误：**

```python
# ❌ 错误一：所有服务同一个超时值
timeout = 3000  # 查询接口够了，但批量导出接口远远不够

# ❌ 错误二：超时值基于最大可能耗时
timeout = 300000  # 5 分钟，形同虚设

# ❌ 错误三：连接超时和读超时不区分
timeout = 30000  # 连接超时不应该这么长
```

```python
# ✅ 正确：按操作类型分别设置
TIMEOUT_CONFIG = {
    "user.get_profile":    500,    # 简单查询
    "order.create":        3000,   # 创建订单
    "report.generate":     30000,  # 报表生成
    "file.upload":         60000,  # 文件上传
}

# ✅ 连接超时与读超时分离
CONNECT_TIMEOUT = 3000   # 连接超时 3s
READ_TIMEOUT = 5000      # 读超时 5s
```

#### 陷阱四：超时链路断裂

**场景：** 上游设置了 5 秒超时，但下游每层也设置了 5 秒超时，总耗时远超预期。

**解决方案：** 使用 Deadline 传递剩余时间，或手动计算链路超时分配。

❌ 错误配置：
  网关超时 = 5s → 业务层超时 = 5s → 数据层超时 = 5s
  总耗时 = 15s（远超网关的 5s 超时）

✅ 正确配置：
  网关 Deadline = 5s
  业务层 Deadline = 5s - 网关处理时间 ≈ 4s（gRPC 自动传播）
  数据层 Deadline = 4s - 业务处理时间 ≈ 3s（gRPC 自动传播）

#### 陷阱五：重试加剧资源耗尽

**场景：** 服务已经 CPU 飙到 100%，重试请求进一步加剧资源竞争。

**解决方案：** 配合**断路器（Circuit Breaker）**使用：

```python
class CircuitBreaker:
    """
    断路器状态机：
    CLOSED（正常）→ 超过阈值 → OPEN（熔断）→ 等待恢复 → HALF_OPEN（试探）
    
    OPEN 状态下所有请求直接失败，不发送到下游服务
    """
    CLOSED, OPEN, HALF_OPEN = "closed", "open", "half_open"
    
    def __init__(self, failure_threshold=5, recovery_timeout=30, 
                 half_open_max=3):
        self.state = self.CLOSED
        self.failure_count = 0
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.half_open_max = half_open_max
        self.half_open_count = 0
        self.last_failure_time = 0
    
    def can_execute(self):
        if self.state == self.CLOSED:
            return True
        elif self.state == self.OPEN:
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = self.HALF_OPEN
                self.half_open_count = 0
                return True
            return False
        elif self.state == self.HALF_OPEN:
            return self.half_open_count < self.half_open_max
    
    def record_success(self):
        self.failure_count = 0
        self.state = self.CLOSED
    
    def record_failure(self):
        self.failure_count += 1
        self.last_failure_time = time.time()
        if self.failure_count >= self.failure_threshold:
            self.state = self.OPEN
```

```python
# 网关层超时管理：根据下游服务 SLA 动态调整

class GatewayTimeoutManager:
    """
    网关超时策略：
    1. 根据下游服务的历史 P99 延迟动态调整超时值
    2. 设置全局 Deadline 并传播到下游
    3. 监控超时率，自动告警
    """
    
    def __init__(self):
        # 各服务的超时配置（毫秒）
        self.timeout_configs = {
            "user-service":   {"base": 500,  "max": 3000,  "p99_multiplier": 3},
            "order-service":  {"base": 2000, "max": 10000, "p99_multiplier": 2},
            "payment-service":{"base": 3000, "max": 15000, "p99_multiplier": 2},
            "search-service": {"base": 1000, "max": 5000,  "p99_multiplier": 4},
        }
    
    def get_timeout(self, service, current_p99_ms):
        """根据服务 SLA 和实时 P99 计算合理超时"""
        config = self.timeout_configs.get(service, {"base": 3000, "max": 10000})
        
        # 超时值 = max(基础超时, P99 × 倍数)
        calculated = max(
            config["base"],
            current_p99_ms * config["p99_multiplier"]
        )
        
        # 但不超过最大超时值
        return min(calculated, config["max"]) / 1000  # 返回秒
    
    def route_request(self, request, service):
        """路由请求并设置合理的超时"""
        # 获取当前服务的 P99 延迟
        current_p99 = self._get_current_p99(service)
        timeout = self.get_timeout(service, current_p99)
        
        # 设置 Deadline
        ctx = grpc.deadline_context(timeout=timeout)
        
        try:
            return self._forward(request, service, ctx)
        except grpc.DeadlineExceeded:
            # 记录超时事件，触发告警
            self._record_timeout(service, timeout, current_p99)
            raise
```

### 8. 本章小结

超时与重试是 RPC 容错体系的基石。总结核心要点：

1. **超时是第一道防线**：没有超时就没有容错。每个 RPC 调用都必须设置超时。
2. **Deadline 优于单次超时**：在多层调用链中，使用 Deadline 传递全局截止时间。
3. **重试必须有边界**：通过重试预算限制重试流量，防止重试风暴。
4. **幂等是重试的前提**：非幂等操作必须通过幂等键才能安全重试。
5. **错误分类决定重试策略**：区分可重试/不可重试错误，避免无意义重试。
6. **退避 + 抖动是标配**：指数退避配合随机抖动，是重试间隔的最佳实践。
7. **可观测性不可或缺**：超时率、重试率、延迟分布必须有监控和告警。
8. **配合熔断器使用**：断路器防止对故障服务的无效重试，加速系统恢复。

下一章我们将讲解熔断器模式——超时与重试的终极搭档。
