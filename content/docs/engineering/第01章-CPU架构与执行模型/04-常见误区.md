# 第01章 常见误区：CPU架构优化中的十二个认知陷阱

## 引言：为什么误区如此危险

CPU架构优化领域存在一个残酷的悖论：**你的直觉几乎总是错的**。这是因为现代CPU的微架构复杂度已经远远超出了人类的直觉模型——分支预测器、乱序执行引擎、多级缓存层次、缓存一致性协议，这些机制协同工作时产生的性能特征是非线性的、反直觉的。

本节收录了开发者在CPU性能优化中最常犯的十二个认知错误。每个误区都遵循统一的剖析结构：

错误认知 → 根因分析 → 量化对比 → 正确做法

掌握这些误区不仅是为了"避免犯错"，更是为了建立正确的CPU性能心智模型。当你理解了为什么这些直觉是错的，你就能更准确地预测代码的性能行为。

## 误区一：多核一定比单核快

### 错误认知

"我的程序用了多线程，CPU有8核，性能应该提升8倍。"

### 根因分析

多核加速的核心前提是任务可以并行化，且并行化的开销小于收益。Amdahl定律用数学量化了这一限制：

$$加速比 = \frac{1}{(1-P) + P/N}$$

其中P是可并行比例，N是核心数。这个公式的残酷之处在于：**即使N趋近无穷大，加速比的上限也只有 1/(1-P)**。

| 可并行比例P | 4核理论上限 | 8核理论上限 | 16核理论上限 | ∞核理论上限 |
|------------|------------|------------|-------------|------------|
| 50% | 1.8x | 2.3x | 2.7x | 2.0x |
| 75% | 2.9x | 4.0x | 5.3x | 4.0x |
| 90% | 4.0x | 5.7x | 8.2x | 10.0x |
| 95% | 5.2x | 8.0x | 12.4x | 20.0x |
| 99% | 6.6x | 11.0x | 18.5x | 100.0x |

注意：当P=90%时，8核的理论加速比仅为5.7x，远非8倍。而且这还是理论上限。

### 真实性能对比

```c
// 场景：对1000万个元素求和，可并行比例约92%
// 环境：Intel i7-12700K, 8核16线程, Linux 6.x

// 单线程版本
// 编译：gcc -O3 -march=native sum_single.c -o sum_single
// 运行时间：约 28ms

// 4线程版本（OpenMP）
// 编译：gcc -O3 -march=native -fopenmp sum_omp4.c -o sum_omp4
// 运行时间：约 9.5ms（理论加速比3x，实际约2.9x）

// 8线程版本
// 编译：gcc -O3 -march=native -fopenmp sum_omp8.c -o sum_omp8
// 运行时间：约 5.8ms（理论加速比4.8x，实际约4.1x）

// 16线程版本（超线程）
// 运行时间：约 5.2ms（理论加速比5.4x，实际约4.9x）
// 注意：从8线程到16线程，仅获得额外12%的加速
```

实际加速比低于理论值的原因包括：

1. **线程创建与同步开销**：pthread_create本身需要约10-50μs，加上屏障同步
2. **缓存一致性流量**：多核共享数据时，MESI协议的状态转换产生额外延迟
3. **内存带宽饱和**：当所有核心同时读取内存时，带宽成为瓶颈
4. **负载不均衡**：OpenMP的静态划分可能无法完美均分工作量

### 正确做法

1. **先用perf确认瓶颈**：运行`perf stat -e cycles,instructions,cache-misses`确认瓶颈是否真的在CPU计算上，而非I/O或内存
2. **测量并行化前后的实际性能**：不要假设加速比，用`clock_gettime(CLOCK_MONOTONIC)`精确测量
3. **考虑通信开销**：线程间的同步（锁、屏障、原子操作）都有成本
4. **小任务用单线程**：对于运行时间<1ms的任务，线程创建开销可能超过计算本身
5. **用Gustafson定律思考**：如果可以增大问题规模（而非固定问题），增加核心数更有效

## 误区二：忽略缓存行大小

### 错误认知

"一个int才4字节，两个相邻的int变量访问速度是一样的。"

### 根因分析

CPU不按字节或字访问内存，而是按**缓存行（cache line）**访问，典型大小为64字节（Intel自P6架构以来的标准）。更关键的是，多核场景下，缓存一致性协议（MESI/MOESI）以缓存行为粒度维护一致性。

当两个线程修改同一缓存行中的不同变量时，即使它们修改的数据毫无逻辑关联，也会触发**伪共享（false sharing）**——缓存行在两个核心之间来回弹跳，每次弹跳的延迟约为40-70ns（跨核L3延迟）。

伪共享的本质：
  核心0修改变量A（缓存行偏移0-3字节）
  核心1修改变量B（缓存行偏移4-7字节）
  → 整个64字节的缓存行在MESI协议下不断在Modified和Invalid状态间切换
  → 两个核心的实际数据修改量仅为8字节，但一致性流量达到了64字节×N次

### 真实性能对比

```c
// 伪共享基准测试：两个线程各累加10亿次
#include <pthread.h>
#include <stdio.h>
#include <time.h>

struct SharedData {
    long counter_a;  // 偏移0
    long counter_b;  // 偏移8  ← 同一缓存行！
} data;

// 伪共享版本
void* thread_func_false_sharing(void* arg) {
    long id = (long)arg;
    volatile long* target = (id == 0) ? &amp;data.counter_a : &amp;data.counter_b;
    for (long i = 0; i < 1000000000L; i++) {
        (*target)++;
    }
    return NULL;
}

// 修复后的版本（缓存行对齐）
struct FixedData {
    long counter_a;
    char pad_a[56];   // 填充到64字节边界
    long counter_b;
    char pad_b[56];
} fixed_data;
// 性能提升约 10-30 倍
```

在Intel i7-12700K上的实测结果：

| 版本 | 2线程耗时 | 相对性能 | 说明 |
|------|----------|---------|------|
| 伪共享版本 | ~4.2s | 1x | 缓存行频繁弹跳 |
| 缓存行对齐版本 | ~0.18s | 23x | 各自独立缓存行 |

### 正确做法

1. **用perf c2c检测**：`perf c2c record ./program && perf c2c report`，直接定位伪共享的源代码位置
2. **添加缓存行填充**：
   - C/C++：`__attribute__((aligned(64)))`或C++17的`alignas(64)`
   - Go：`[56]byte`填充或`//go:align 64`
   - Java：使用`@Contended`注解（需`-XX:-RestrictContended`）
   - Rust：`#[repr(align(64))]`
3. **重新设计数据布局**：将多线程写入的变量分散到不同结构体中，避免物理邻近
4. **注意编译器可能重排**：`volatile`或`atomic`变量不会被重排，但普通变量可能

## 误区三：认为编译器优化已经足够

### 错误认知

"开了-O3编译器就帮我做好了所有优化，不需要手动优化代码。"

### 根因分析

现代编译器（GCC、Clang、MSVC）的优化能力确实令人印象深刻，但它们有四个根本性限制：

| 限制类型 | 具体表现 | 举例 |
|---------|---------|------|
| 无法改变算法复杂度 | O(n²)的代码再怎么优化也快不过O(n log n) | 冒泡排序→快速排序 |
| 无法消除不必要的内存分配 | new/malloc的位置需要程序员控制 | 频繁的std::vector重新分配 |
| 自动向量化有限制 | 循环体内有复杂控制流时可能无法向量化 | 含break/continue的循环 |
| 无法理解业务语义 | 不知道哪些数据是热数据，哪些路径是热路径 | 数据库的索引查找 vs 全表扫描 |

### 真实性能对比

```c
// 案例：AoS vs SoA（编译器无法自动转换）

// AoS（Array of Structures）—— 默认写法
struct Particle {
    float x, y, z;     // 位置
    float vx, vy, vz;  // 速度
    float mass;         // 质量
};
Particle particles[N];
// 只更新位置的循环：每次加载整个Particle（32字节），只用其中12字节
// 缓存利用率：12/32 = 37.5%

// SoA（Structure of Arrays）—— 手动优化
struct Particles {
    float x[N], y[N], z[N];
    float vx[N], vy[N], vz[N];
    float mass[N];
};
// 只更新x的循环：连续访问x数组，缓存行100%利用
// 性能提升：2-4倍（取决于访问模式）

// 即使开了-O3，编译器也不会自动将AoS转为SoA
// 因为这需要理解数据的访问语义
```

```c
// 案例：编译器能做的 vs 不能做的

// 编译器能做的（自动优化）：
// 1. 循环展开（loop unrolling）
// 2. 常量传播（constant propagation）
// 3. 死代码消除（dead code elimination）
// 4. 公共子表达式消除（CSE）
// 5. 尾递归优化

// 编译器不能做的（需要程序员）：
// 1. 将行优先改为列优先遍历（需要理解数组语义）
// 2. 将AoS改为SoA（需要理解数据访问模式）
// 3. 选择合适的算法（需要理解问题域）
// 4. 循环分块/tiling（需要理解缓存层次）
// 5. 手动预取提示（__builtin_prefetch）
```

### 正确做法

1. **先用-O3编译，再用perf找瓶颈**：不要盲目手写优化，先让编译器做它能做的
2. **分析瓶颈处的算法复杂度**：如果是O(n²)，考虑能否用O(n log n)的算法替代
3. **手动优化数据结构布局**：AoS→SoA、紧凑结构体、消除不必要的字段
4. **循环分块（Tiling）**：将大循环切成适合L1缓存大小的小块，这是编译器几乎不会自动做的
5. **关键循环使用SIMD intrinsics**：当自动向量化失败时，手动使用`_mm256_*`系列函数
6. **查看编译器的优化报告**：GCC用`-fopt-info-vec`，Clang用`-Rpass-analysis=loop-vectorize`

## 误区四：分支预测优化过度

### 错误认知

"所有if-else都应该用位运算消除分支，这样一定更快。"

### 根因分析

现代CPU的分支预测器（尤其是TAGE）非常强大。关键数据：TAGE预测器在典型工作负载上的准确率可达96-97%，对于有规律的分支（如循环计数器、排序后的数据），准确率甚至接近99%。

分支预测命中时的开销接近0——CPU猜测的路径直接进入流水线执行，没有任何额外代价。而无分支代码虽然消除了预测错误的惩罚，但代价是：

1. **指令数量增加**：三元运算符或位运算通常需要更多指令
2. **寄存器压力增大**：需要额外的寄存器保存两个分支的结果
3. **代码可读性下降**：位运算版本难以理解和维护

### 真实性能对比

```c
// 场景1：高度可预测的分支（循环计数器比较）
// 分支预测准确率：~99.9%
for (int i = 0; i < n; i++) {
    if (arr[i] > threshold) {  // 前半部分为true的概率极高
        sum += arr[i];
    }
}
// 分支版本：CPI ≈ 0.3（几乎完美的预测）
// 无分支版本：CPI ≈ 0.4（更多指令，更差的性能）

// 场景2：完全随机的分支
// 分支预测准确率：~50%
for (int i = 0; i < n; i++) {
    if (rand() &amp; 1) {  // 完全随机
        sum += arr[i];
    }
}
// 分支版本：CPI ≈ 1.5（频繁预测错误，每次15-20周期惩罚）
// 无分支版本：CPI ≈ 0.5（没有预测错误，但指令更多）

// 场景3：中等可预测（switch/case分发）
// 分支预测准确率：~70-80%
// 无分支版本（查表法）：通常更快，因为消除了不可预测的跳转
```

用perf测量的对比数据：

| 场景 | 分支预测错误率 | 分支版本性能 | 无分支版本性能 | 推荐 |
|------|-------------|------------|-------------|------|
| 循环计数器 | <1% | 快 | 慢10-20% | 保留分支 |
| 排序后数据 | 2-5% | 快 | 慢5-10% | 保留分支 |
| 状态机分发 | 15-25% | 中等 | 快10-30% | 无分支 |
| 随机数据 | 40-50% | 慢 | 快30-100% | 无分支 |
| 哈希表探测 | 50%+ | 很慢 | 快50-200% | 无分支 |

### 正确做法

1. **先测量，再优化**：用`perf stat -e branch-misses`测量分支预测错误率
2. **只优化错误率>10%的热点分支**：低错误率的分支不值得转换
3. **优先排序数据**：将数据按分支条件排序，使预测器更容易学习模式
4. **无分支代码必须验证**：用perf确认无分支版本确实更快
5. **考虑代码可维护性**：位运算版本难以阅读和调试，性能提升不显著时不要转换

## 误区五：SIMD编程总是更快

### 错误认知

"SIMD一次处理4/8个数据，一定比标量代码快4/8倍。"

### 根因分析

SIMD（Single Instruction Multiple Data）的理论加速比很高，但实际加速受限于多个因素：

| 限制因素 | 具体影响 | 性能损失 |
|---------|---------|---------|
| 数据对齐 | AVX要求32字节对齐，未对齐访问跨缓存行时延迟翻倍 | 2-4x |
| 尾部处理 | 数据量不是向量宽度整数倍时需要标量尾处理 | 5-15% |
| 水平操作 | 归约求和、最大值查找等需要跨lane通信 | 3-8x |
| 内存带宽 | 如果瓶颈是内存带宽而非计算，SIMD无法突破 | 0% |
| 寄存器压力 | 大量使用256/512位寄存器可能导致溢出到L1 | 20-50% |

### 真实性能对比

```c
// 适合SIMD的场景：元素间独立的向量运算
// 例：向量加法 a[i] + b[i] → c[i]
// 标量版本：每次处理1个float
// AVX版本：每次处理8个float（256位/32位）
// 理论加速：8x
// 实际加速：5-6x（受内存带宽限制）

// 不适合SIMD的场景1：链表遍历
// 每个节点的地址依赖前一个节点，无法并行加载
// SIMD版本可能比标量更慢（额外的对齐检查和标量处理）

// 不适合SIMD的场景2：归约求和
// 需要跨lane相加，AVX中需要额外的shuffle+add操作
// 实测加速仅2-3x（远低于8x理论值）

// 不适合SIMD的场景3：条件分支密集的代码
// if (a[i] > 0) sum += a[i];  // 需要mask操作，开销大
// 使用SIMD mask：_mm256_cmpgt_ps + _mm256_and_ps + _mm256_add_ps
// 比标量快2-3x，但远不如纯算术场景的5-6x
```

### 正确做法

1. **先让编译器自动向量化**：用`-O3 -march=native`编译，检查编译器是否成功向量化
2. **只在热点循环手动使用SIMD intrinsics**：手动向量化的维护成本很高
3. **确保数据对齐**：使用`aligned_alloc`或`posix_memalign`分配对齐内存
4. **处理尾部**：确保数据长度是向量宽度的整数倍，或用padding补齐
5. **用perf验证**：对比标量和SIMD版本的实际性能，不要假设加速比
6. **考虑编译器内置函数**：如`__builtin_popcount`、`__builtin_clz`等，编译器可能生成比手写intrinsics更优的代码

## 误区六：缓存越大越好

### 错误认知

"L3缓存从8MB增加到16MB，性能一定提升很多。"

### 根因分析

缓存的收益遵循**边际递减规律**，取决于三个关键因素：

1. **工作集大小**：只有当工作集超过当前缓存容量时，增大缓存才有效
2. **访问模式**：顺序扫描（预取器已足够好）vs 随机访问（缓存关键）
3. **延迟成本**：更大的缓存意味着更高的访问延迟

缓存容量与延迟的关系（Intel典型值）：

| 缓存级别 | 典型容量 | 访问延迟 | 带宽 |
|---------|---------|---------|------|
| L1D | 32-48KB | 1-2ns (4-5 cycles) | ~200 GB/s |
| L2 | 256KB-1.25MB | 3-5ns (12-20 cycles) | ~100 GB/s |
| L3 | 8-32MB | 10-20ns (40-80 cycles) | ~50 GB/s |
| 主存 | 16-64GB | 50-100ns (200-400 cycles) | ~50 GB/s |

### 真实性能对比

```bash
# 场景1：工作集6MB < L3 8MB
# 运行：./program --workset=6MB
# 性能：基线
# 增加L3到16MB：性能提升约 0-2%（几乎无变化）

# 场景2：工作集12MB > L3 8MB
# 运行：./program --workset=12MB
# 性能：L3未命中率约30%
# 增加L3到16MB：性能提升约 30-50%（显著改善）

# 场景3：工作集24MB > L3 16MB
# 运行：./program --workset=24MB
# 性能：L3未命中率约40%
# 增加L3到32MB：性能提升约 25-40%
```

### 正确做法

1. **用perf测量LLC未命中率**：`perf stat -e LLC-load-misses,LLC-loads`，如果未命中率<5%，增大缓存无用
2. **优先优化数据结构和算法**：减小工作集比增大缓存更有效
3. **数据分块（Blocking）**：将大循环切成适合L1/L2大小的块，这是最有效的缓存优化手段
4. **考虑NUMA架构**：在多路服务器上，跨NUMA节点的内存访问延迟比缓存未命中更高
5. **用cachegrind模拟**：`valgrind --tool=cachegrind ./program`，精确分析缓存行为

## 误区七：CPU频率越高性能越好

### 错误认知

"4GHz的CPU一定比3GHz的快33%。"

### 根因分析

实际性能由三个因素共同决定：

$$实际吞吐量 = 频率 \times IPC \times 核心数$$

IPC（Instructions Per Cycle，每周期指令数）受微架构设计影响巨大。不同微架构的IPC差异可达2-3倍。

| CPU | 频率 | IPC | 每秒指令数 | 单核性能 |
|-----|------|-----|-----------|---------|
| Intel Pentium 4 (Prescott) | 3.8GHz | 0.7 | 26.6亿 | 基线 |
| Intel Core 2 (Conroe) | 2.4GHz | 1.8 | 43.2亿 | 1.6x |
| Intel Core i7 (Skylake) | 4.0GHz | 2.5 | 100亿 | 3.8x |
| AMD Zen 4 (Ryzen 7000) | 5.0GHz | 3.0 | 150亿 | 5.6x |

注意Pentium 4的3.8GHz反而比Core 2的2.4GHz更慢——这就是NetBurst架构高频率低IPC的典型案例。

此外，高频CPU的功耗与频率呈超线性关系：

$$功耗 \propto 频率 \times 电压^2$$

更高的频率需要更高的电压，功耗急剧增加，可能导致：
1. 热设计功耗（TDP）限制下的降频
2. 短时间Boost后持续性能下降
3. 多核同时高频时的功耗墙

### 正确做法

1. **关注IPC而非频率**：用`perf stat -e instructions,cycles`计算实际IPC
2. **测量持续性能**：不是看Boost频率，而是看持续负载下的频率
3. **考虑功耗约束**：笔记本和服务器的TDP差异巨大，持续性能完全不同
4. **对于并行任务**：更多核心+中等频率通常优于更少核心+高频率
5. **查看微架构代号**：同一代架构内频率才有直接可比性

## 误区八：超线程总是有益的

### 错误认知

"开启超线程一定提升性能，没有坏处。"

### 根因分析

超线程（SMT，Simultaneous Multi-Threading）让两个逻辑核心共享一个物理核心的全部资源：

物理核心
├── 执行单元（ALU、FPU、SIMD单元）  ← 两个逻辑核心共享
├── L1缓存（32-48KB）                ← 两个逻辑核心共享
├── L2缓存（256KB-1.25MB）           ← 两个逻辑核心共享
├── TLB                             ← 两个逻辑核心共享
└── 分支预测器                       ← 两个逻辑核心共享

当两个线程的工作集都很大时，会发生**缓存驱逐**——线程A的数据被线程B挤出L1/L2，反之亦然，导致两者都频繁缓存未命中。

### 真实性能对比

# 场景1：两个线程的工作集各32KB
# L1D缓存64KB，两个线程各占一半，刚好放下
# 超线程有效，性能提升约 25-30%

# 场景2：两个线程的工作集各128KB
# L1D缓存64KB，两个线程频繁互相驱逐
# 超线程反而降低性能约 10-20%

# 场景3：一个CPU密集型线程 + 一个I/O密集型线程
# I/O线程大部分时间在等待，不占用执行单元
# 超线程非常有效，性能提升约 30-40%

# 场景4：延迟敏感的数据库查询
# 每个查询都需要大量L1/L2缓存
# 关闭超线程可能减少尾延迟（P99）约 15-25%

### 正确做法

1. **对关键应用做A/B测试**：用`echo 0 > /sys/devices/system/cpu/cpu0/thread_siblings_list`关闭超线程
2. **用taskset绑定线程**：`taskset -c 0 ./program`将线程绑定到特定核心
3. **延迟敏感的应用**：考虑关闭超线程以减少P99尾延迟
4. **混合负载**：CPU密集型+I/O密集型混合时，超线程通常有益
5. **用perf测量cache-misses**：对比开启/关闭超线程时的缓存未命中率

## 误区九：汇编代码一定比C快

### 错误认知

"手写汇编比编译器生成的代码快，我应该用汇编写所有性能关键代码。"

### 根因分析

这个误区在30年前可能是对的，但现代编译器的优化能力已经远超大多数人类程序员的手写汇编。原因有三：

1. **编译器理解全局上下文**：可以跨函数内联、跨基本块优化，手写汇编很难做到
2. **编译器了解微架构**：GCC/Clang可以针对特定CPU的执行端口调度指令
3. **编译器不会犯人为错误**：手写汇编容易引入微妙的错误（如错误的标志位使用）

手写汇编可能更快的场景：
- SIMD intrinsics无法表达的特定指令序列
- 利用特定CPU的非公开优化技巧（如Intel的stack engine规避）
- 系统级代码（上下文切换、中断处理）

### 真实性能对比

```c
// 例：字符串长度计算
// 手写x86-64汇编（ naive版）
int strlen_asm(const char* s) {
    int len = 0;
    while (*s++) len++;
    return len;
}

// 编译器生成的代码（-O3 -march=native）
// GCC会自动使用SIMD指令（SSE2/AVX2）
// 一次比较16/32个字节
// 生成的代码比手写循环快5-10倍

// 手写AVX2汇编版本
// 性能与编译器生成的几乎相同（差异<5%）
// 但维护成本高10倍以上

// 结论：除非你对CPU微架构有深入理解，否则让编译器生成更好
```

### 正确做法

1. **用godbolt.org对比**：查看编译器生成的汇编，大多数情况下已经足够好
2. **只在证明更快时手写汇编**：必须用perf对比验证
3. **使用SIMD intrinsics**：比内联汇编更安全、更可维护，编译器可以进一步优化
4. **将手写汇编隔离**：封装在单独的函数中，便于后续替换

## 误区十：用wall-clock时间测量性能就够了

### 错误认知

"我用`time ./program`看了运行时间，性能没问题。"

### 根因分析

`time`命令测量的是wall-clock时间，它包含了所有因素——I/O等待、内存分配、调度延迟、上下文切换。对于CPU性能优化来说，它太粗糙了。

time ./program 输出示例：
real    0m1.234s    ← 包含I/O、调度、上下文切换
user    0m0.987s    ← CPU实际执行时间
sys     0m0.156s    ← 内核态时间

问题：这三个数字无法告诉你：
1. CPU是忙还是在等内存？
2. 分支预测准确率是多少？
3. 缓存未命中率是多少？
4. IPC是多少？

### 正确做法

1. **用perf stat获取硬件计数器**：
   ```bash
   perf stat -e cycles,instructions,cache-misses,branch-misses ./program
   ```
   这告诉你CPU在每个周期做了多少有用的工作

2. **用perf record定位热点**：
   ```bash
   perf record -g ./program &amp;&amp; perf report
   ```
   这告诉你时间花在哪个函数、哪一行

3. **用精确计时器测量微基准**：
   ```c
   struct timespec start, end;
   clock_gettime(CLOCK_MONOTONIC, &amp;start);
   // ... 被测代码 ...
   clock_gettime(CLOCK_MONOTONIC, &amp;end);
   double elapsed = (end.tv_sec - start.tv_sec) +
                    (end.tv_nsec - start.tv_nsec) / 1e9;
   ```

4. **注意测量干扰**：perf本身有约1-3%的开销，微基准测试需要多次运行取中位数

## 误区十一：内存访问顺序是程序员看到的顺序

### 错误认知

"代码里写的内存操作顺序就是CPU执行的顺序，不需要考虑乱序。"

### 根因分析

现代CPU（尤其是x86-64）的乱序执行引擎会重排内存操作以提高性能。虽然x86-64提供了较强的内存排序保证（TSO，Total Store Order），但仍有一些重排是可能的：

x86-64内存排序保证：
  ✓ Load不会与Load重排
  ✓ Store不会与Store重排
  ✗ Store可能与后续Load重排（Store-Load重排）
  ✗ 不同地址的Load可能被提前（但对同一地址的依赖保证有序）

ARM/RISC-V的内存排序（弱排序）：
  ✗ Load和Store都可能被重排
  需要显式的内存屏障（DMB/DSB）来保证顺序

在多线程代码中，内存重排可能导致**可见性问题**：

```c
// 经典的Dekker算法问题
// 线程0：                      // 线程1：
data = 42;                      flag = 1;
flag = 1;                       if (data == 42) {
if (flag == 1) {                    // 保证能执行到
    // 不保证能执行到！             printf("%d\n", data); // 可能打印0！
}                               }
// 即使flag==1为真，data的赋值可能还没对线程1可见
```

### 正确做法

1. **多线程共享数据使用原子操作**：C++11的`std::atomic`带默认的`memory_order_seq_cst`
2. **理解内存序**：`memory_order_relaxed`（无排序保证）、`memory_order_acquire/release`（单向屏障）、`memory_order_seq_cst`（全序）
3. **ARM/RISC-V编程必须使用内存屏障**：`__sync_synchronize()`或内联汇编`dmb`
4. **不要依赖编译器的内存重排**：虽然编译器也可能重排，但`volatile`和`atomic`可以阻止
5. **用ThreadSanitizer检测数据竞争**：`clang -fsanitize=thread`

## 误区十二：优化应该从最慢的代码开始

### 错误认知

"我应该先优化运行时间最长的函数，这样效果最明显。"

### 根因分析

这是对Amdahl定律的错误应用。正确的优化策略应该基于**优化的收益/成本比**，而不仅仅是当前耗时。

优化收益公式：
  收益 = 函数耗时占比 × 优化提升比例 × 调用频率

例：
  函数A：耗时100ms，占比50%，可通过优化提升2倍 → 收益 = 100ms × 50% × 50% = 25ms
  函数B：耗时200ms，占比10%，只能优化10%  → 收益 = 200ms × 10% × 10% = 2ms
  函数C：耗时50ms，占比30%，可通过优化提升10倍 → 收益 = 50ms × 30% × 90% = 13.5ms

  最值得优化的是函数A（收益最大），不是函数B（最慢）。

### 正确做法

1. **用perf record找到真正的热点**：热点函数往往不是你以为的那个
2. **计算优化潜力**：分析每个热点的算法复杂度，评估可优化空间
3. **先消除不必要的工作**：缓存结果、避免重复计算、减少I/O，这些往往比微优化更有效
4. **考虑修改成本**：修改一行配置（如编译选项）vs 重写一个模块，收益/成本比完全不同
