# 数据结构与算法：理论基础

软件工程中，算法是解决问题的逻辑骨架，数据结构是承载数据的容器。两者不是孤立的知识点，而是一个紧密耦合的体系——**选择什么样的数据结构，直接决定了你能用什么样的算法；而算法的效率瓶颈，往往暴露在数据结构的设计缺陷上。**

本章围绕五个核心主题展开，覆盖从内存布局到性能分析的完整理论链路。每个主题都遵循"原理→实现→特性→权衡"的递进逻辑，确保读者不仅知道"怎么用"，更理解"为什么这样设计"以及"什么时候该换方案"。

## 一、哈希表：O(1) 查找背后的工程哲学

### 1.1 核心思想

哈希表的本质是一个**空间换时间**的工程妥协：通过一个哈希函数 `f(key) → index`，将任意键映射到固定大小的数组下标，从而实现平均 O(1) 的查找、插入和删除。

```python
# 哈希表的核心操作伪代码
class HashTable:
    def __init__(self, capacity=16):
        self.capacity = capacity      # 桶数组大小
        self.size = 0                 # 已存元素数
        self.buckets = [[] for _ in range(capacity)]  # 拉链法

    def _hash(self, key):
        return hash(key) % self.capacity

    def put(self, key, value):
        idx = self._hash(key)
        for i, (k, v) in enumerate(self.buckets[idx]):
            if k == key:
                self.buckets[idx][i] = (key, value)  # 更新
                return
        self.buckets[idx].append((key, value))       # 新增
        self.size += 1
        if self.size / self.capacity > 0.75:         # 负载因子阈值
            self._resize(self.capacity * 2)

    def get(self, key):
        idx = self._hash(key)
        for k, v in self.buckets[idx]:
            if k == key:
                return v
        raise KeyError(key)

    def _resize(self, new_capacity):
        old_buckets = self.buckets
        self.capacity = new_capacity
        self.buckets = [[] for _ in range(new_capacity)]
        self.size = 0
        for bucket in old_buckets:
            for k, v in bucket:
                self.put(k, v)
```

### 1.2 冲突处理：工程选择的三岔路口

哈希函数不可能完美——不同的键可能映射到同一个桶，这就是**冲突（Collision）**。处理冲突的策略直接决定了哈希表在最坏情况下的表现。

| 策略 | 原理 | 优点 | 缺点 | 代表实现 |
|------|------|------|------|----------|
| **拉链法（Chaining）** | 每个桶存一个链表/红黑树 | 实现简单，扩容平滑 | 额外指针开销，缓存不友好 | Java HashMap、Go map |
| **开放寻址法（Open Addressing）** | 冲突时按规则探测下一个空位 | 缓存连续，内存紧凑 | 集群效应，删除复杂 | Python dict（早期）、Redis |
| **Cuckoo Hashing** | 用2-3个哈希函数，冲突时踢走原有元素 | 稳定 O(1) 查找 | 插入可能触发连锁踢出，实现复杂 | 用于高性能网络设备 |

**负载因子（Load Factor）** 是冲突概率的核心指标：

负载因子 α = 已存元素数 / 桶数组大小

- α < 0.5：空间浪费严重，但冲突极少
- α = 0.75：工程实践的黄金平衡点（Java HashMap 默认值）
- α > 1.0：冲突频繁，查找退化为 O(n/m)（m 为桶数）

### 1.3 哈希函数的设计原则

好的哈希函数需要满足三个条件：

1. **均匀性**：输出分布尽量均匀，减少聚集。对于整数键，常用 `key % capacity`（需 capacity 为质数）；对于字符串键，常用多项式滚动哈希如 `h = h * 31 + char`。
2. **确定性**：相同的键必须产生相同的哈希值。
3. **高效性**：计算本身必须是 O(1) 的。

**反面教材：** 使用 `key % 10` 作为哈希函数，当键是 10、20、30、40 时全部映射到桶 0，退化为链表查找。

│  查找       │  O(1)    │  O(n)    │  退化时   │

### 1.5 工程陷阱与最佳实践

**陷阱一：哈希DoS攻击**

攻击者精心构造大量冲突键，使 O(1) 退化为 O(n)，耗尽服务器 CPU。防御方案：
- 使用带随机种子的哈希函数（如 Java 8 引入的 `SipHash`）
- 限制输入键的最大长度
- 对容量使用质数或质数附近的2的幂

**陷阱二：可变对象作为键**

用列表（list）或字典（dict）作为哈希表的键会报 `TypeError`，因为可变对象的哈希值会变。解决方案：用 `tuple` 或自定义 `__hash__` + `__eq__`。

**陷阱三：扩容时的性能断崖**

哈希表扩容时需要重新计算所有元素的位置（rehash），这是一个 O(n) 的阻塞操作。Java 8 的优化：当链表长度 > 8 时转红黑树（O(n) → O(log n)）；Go 1.24 引入了渐进式扩容。

### 1.6 变体与演进

- **一致性哈希（Consistent Hashing）**：用于分布式系统（如 Memcached、Cassandra），节点增减时只需迁移少量数据。
- **布隆过滤器（Bloom Filter）**：只支持"可能存在"和"一定不存在"的查询，空间效率极高，用于爬虫去重、缓存穿透防护。
- **跳表（Skip List）**：Redis 的 `Sorted Set` 底层用跳表替代平衡树，实现简单且并发友好，平均 O(log n) 查找。

## 二、平衡二叉搜索树：有序数据的守护者

### 2.1 为什么需要"平衡"

普通二叉搜索树（BST）在有序插入时会退化为链表——每次查找都变成 O(n)。平衡二叉搜索树通过**旋转操作**维持树的高度在 O(log n)，保证所有操作的最坏情况性能。

BST 退化示例（有序插入 1→2→3→4→5）：

        1                   平衡后（AVL 旋转）：
         \                      2
          2                    / \
           \                  1   4
            3                    / \
             \                  3   5
              4
               \
                5
  查找 5 需要 5 步        查找 5 只需 2 步

### 2.2 三大经典平衡树

#### AVL 树

AVL 树是最早被发明的自平衡 BST，严格保证**任意节点的左右子树高度差不超过 1**。

**四种旋转操作：**

- **LL 旋转**（左左）：右旋一次
- **RR 旋转**（右右）：左旋一次
- **LR 旋转**（左右）：先左旋再右旋
- **RL 旋转**（右左）：先右旋再左旋

**核心特性：**
- 严格平衡，高度差 ≤ 1
- 查找性能最优：O(log n)
- 插入/删除代价较高（旋转次数多）
- 适合**读多写少**的场景（如数据库索引的只读副本）

#### 红黑树

红黑树是工程中最广泛使用的平衡 BST，通过给节点着色（红/黑）并维护五条不变式来维持"近似平衡"：

1. 每个节点是红色或黑色
2. 根节点是黑色
3. 所有叶子节点（NIL）是黑色
4. 红色节点的两个子节点必须是黑色（不能有连续红节点）
5. 从任意节点到其所有叶子的路径上，黑色节点数相同

**与 AVL 树的权衡：**

┌──────────────┬─────────────────┬─────────────────┐
│   维度       │   AVL 树        │   红黑树         │
├──────────────┼─────────────────┼─────────────────┤
│  平衡严格度  │  严格（差≤1）    │  近似（高度≤2lg） │
│  查找        │  更快（更矮）    │  稍慢            │
│  插入/删除   │  旋转更多        │  旋转更少        │
│  适用场景    │  查找密集        │  插入删除密集    │
│  实现难度    │  较简单          │  较复杂          │
│  实际应用    │  数据库索引      │  Linux CFS、     │
│              │                 │  Java TreeMap、  │
│              │                 │  C++ std::map    │
└──────────────┴─────────────────┴─────────────────┘

#### B 树与 B+ 树

当数据无法全部放入内存时（如数据库索引、文件系统），需要**多路平衡搜索树**——B 树家族。

**B 树**：每个节点可以有多个子节点（m 阶 B 树最多 m 个子节点），所有数据存储在内部节点和叶子节点中。

**B+ 树**：所有数据只存在叶子节点，内部节点只存索引键；叶子节点通过指针串联成链表。这是 MySQL InnoDB 和大多数关系数据库索引的底层结构。

B+ 树结构示意（3阶）：

              [10 | 20]
             /    |    \
     [3|5|8]  [12|15|18]  [22|25|28]
      ↓          ↓            ↓
     数据链表 ←→ 数据链表 ←→ 数据链表

**B+ 树优于 B 树的三个原因：**
1. 叶子链表支持高效范围查询（`WHERE age BETWEEN 20 AND 50`）
2. 内部节点更小，单次磁盘 IO 能加载更多索引键
3. 查询路径长度一致（都到叶子），性能更可预测

### 2.3 选择决策树

需要有序数据 + O(log n) 操作？
├── 数据在内存中？
│   ├── 查找为主 → AVL 树
│   ├── 增删为主 → 红黑树
│   └── 需要有序遍历 + 范围查询 → 红黑树（TreeMap）
├── 数据在磁盘上？
│   ├── 单点查询为主 → B 树
│   └── 范围查询为主 → B+ 树（数据库索引）
└── 需要近似有序？→ 跳表（Redis Sorted Set）

## 三、堆与优先队列：优先级驱动的数据容器

### 3.1 堆的本质

堆是一种**完全二叉树**，满足堆序性质：

- **最大堆**：每个节点的值 ≥ 其子节点的值（根是最大值）
- **最小堆**：每个节点的值 ≤ 其子节点的值（根是最小值）

堆不需要像 BST 那样保持左小右大的顺序，只需要保证父子关系的偏序——这使得堆的维护成本更低。

### 3.2 数组实现：紧凑的内存布局

堆可以用数组高效存储（利用完全二叉树的性质，无需指针）：

索引关系（0-based）：
  父节点 → 左子：  2*i + 1
  父节点 → 右子： 2*i + 2
  子节点 → 父：  (i - 1) / 2

最大堆的数组表示：
  树形结构：           数组：
       90             [90, 80, 70, 30, 60, 50, 40]
      /  \
    80    70
   / \   / \
  30 60 50  40

```python
class MinHeap:
    def __init__(self):
        self.data = []

    def push(self, val):
        self.data.append(val)
        self._sift_up(len(self.data) - 1)

    def pop(self):
        if not self.data:
            raise IndexError("empty heap")
        top = self.data[0]
        last = self.data.pop()
        if self.data:
            self.data[0] = last
            self._sift_down(0)
        return top

    def _sift_up(self, i):
        while i > 0:
            parent = (i - 1) // 2
            if self.data[i] < self.data[parent]:
                self.data[i], self.data[parent] = self.data[parent], self.data[i]
                i = parent
            else:
                break

    def _sift_down(self, i):
        n = len(self.data)
        while True:
            smallest = i
            left, right = 2*i+1, 2*i+2
            if left < n and self.data[left] < self.data[smallest]:
                smallest = left
            if right < n and self.data[right] < self.data[smallest]:
                smallest = right
            if smallest != i:
                self.data[i], self.data[smallest] = self.data[smallest], self.data[i]
                i = smallest
            else:
                break
```

### 3.3 核心操作的复杂度

| 操作 | 时间复杂度 | 说明 |
|------|-----------|------|
| 查找最大/最小 | O(1) | 直接取根节点 |
| 插入 | O(log n) | 添加到末尾，向上调整 |
| 删除最大/最小 | O(log n) | 取根后用末尾替换，向下调整 |
| 建堆 | O(n) | 从最后一个非叶子节点开始，逐个向下调整 |
| 堆排序 | O(n log n) | 反复取出堆顶 |

### 3.4 堆的经典应用场景

**场景一：Top-K 问题**

从 N 个元素中找最大的 K 个，用最小堆（容量 K）：

```python
import heapq

def top_k(nums, k):
    """用最小堆维护最大的k个元素"""
    min_heap = nums[:k]
    heapq.heapify(min_heap)  # O(k)
    for num in nums[k:]:
        if num > min_heap[0]:
            heapq.heapreplace(min_heap, num)  # O(log k)
    return min_heap  # 未排序
# 总复杂度：O(n log k)，优于排序的 O(n log n)
```

**场景二：合并 K 个有序链表**

用最小堆维护 K 个链表的当前头节点，每次取最小的接到结果链表后：

时间复杂度：O(N log k)，N 为总元素数，k 为链表数
空间复杂度：O(k)（堆的大小）

**场景三：任务调度**

操作系统和消息队列的优先级调度本质就是堆操作。Java 的 `PriorityQueue`、Linux 的 CFS 调度器都基于堆。

**场景四：Dijkstra 最短路径算法**

Dijkstra 的核心是在一个"候选节点集"中反复选取距离最小的节点——这正是最小堆的用武之地，将算法从 O(V²) 优化到 O((V+E) log V)。

### 3.5 堆 vs 有序容器

┌─────────────────┬────────────────┬────────────────┐
│   维度          │   堆           │   红黑树/跳表   │
├─────────────────┼────────────────┼────────────────┤
│  查找最小       │  O(1)          │  O(1)（最左）   │
│  查找任意元素   │  O(n)          │  O(log n)      │
│  有序遍历       │  不支持         │  O(n)          │
│  内存开销       │  极低（数组）   │  较高（指针）   │
│  适用场景       │  Top-K、调度    │  范围查询、排序  │
└─────────────────┴────────────────┴────────────────┘

## 四、图的表示：关系数据的两种面孔

### 4.1 为什么图如此重要

图是表达**实体间关系**的通用模型。社交网络（用户-关注）、交通网络（城市-道路）、依赖关系（模块-引用）——几乎所有的非线性关系都可以建模为图。

### 4.2 两种核心表示法

#### 邻接矩阵

用一个 `n × n` 的二维矩阵 `adj[i][j]` 表示节点 i 和 j 之间是否有边（或边的权重）。

```python
# 无向图的邻接矩阵
# 节点: 0-1-2, 0-2
adj_matrix = [
    [0, 1, 1],  # 节点 0 连接到 1 和 2
    [1, 0, 0],  # 节点 1 只连接到 0
    [1, 0, 0],  # 节点 2 只连接到 0
]
```

**优点：**
- 判断两点是否相邻：O(1)
- 实现简单直观
- 适合稠密图（边数接近 n²）

**缺点：**
- 空间 O(n²)，稀疏图浪费严重
- 遍历某节点的所有邻居：O(n)（需扫描整行）

#### 邻接表

每个节点维护一个列表，只存储它的邻居节点。

```python
# 同一个图的邻接表表示
adj_list = {
    0: [1, 2],
    1: [0],
    2: [0],
}
```

**优点：**
- 空间 O(V + E)，稀疏图友好
- 遍历某节点的所有邻居：O(度数)
- 大多数真实场景（社交网络、网页链接）都是稀疏图

**缺点：**
- 判断两点是否相邻：O(度数)（需遍历列表）
- 实现稍复杂

### 4.3 选择决策

┌──────────────┬─────────────┬─────────────┐
│   场景       │   推荐      │   原因       │
├──────────────┼─────────────┼─────────────┤
│  稠密图      │  邻接矩阵   │  空间差距小  │
│  稀疏图      │  邻接表     │  节省大量空间 │
│  需要快速    │  邻接矩阵   │  O(1) 判断   │
│  判断邻接    │             │             │
│  BFS/DFS     │  邻接表     │  遍历高效    │
│  社交网络    │  邻接表     │  每人好友有限 │
│  全连接      │  邻接矩阵   │  所有边都存在 │
└──────────────┴─────────────┴─────────────┘

### 4.4 特殊图结构

**有向无环图（DAG）**：任务调度（拓扑排序）、编译依赖分析、版本控制（Git commit graph）。

**有权图**：道路距离、网络带宽、通信成本。常见算法：Dijkstra（单源最短路）、Floyd-Warshall（全源最短路）。

**二部图**：节点分为两个集合，边只连接不同集合的节点。匹配问题（婚姻匹配、资源分配）的建模基础。判定方法：BFS/DFS 染色法。

**完全图**：每对节点之间都有边。n 个节点的完全图有 n(n-1)/2 条边。常见于小型密集网络（如 5 节点以内的全互联集群）。

## 五、复杂度速查表：算法效率的度量衡

### 5.1 时间复杂度速查

复杂度分析的核心工具是**大 O 表示法**——描述算法执行时间随输入规模增长的上界。

常见时间复杂度（从快到慢）：

  O(1)        常数          数组下标访问、哈希表查找
  O(log n)    对数          二分查找、平衡树操作、堆操作
  O(n)        线性          遍历数组/链表、计数排序
  O(n log n)  线性对数      归并排序、快速排序（平均）、堆排序
  O(n²)       平方          冒泡排序、选择排序、朴素字符串匹配
  O(2ⁿ)       指数          子集枚举、递归斐波那契
  O(n!)       阶乘          全排列枚举、旅行商暴力解

增长速率对比（n=100）：
  O(1)       = 1
  O(log n)   ≈ 7
  O(n)       = 100
  O(n log n) ≈ 664
  O(n²)      = 10,000
  O(2ⁿ)      ≈ 10³⁰（天文数字）

### 5.2 空间复杂度速查

| 复杂度 | 含义 | 典型场景 |
|--------|------|----------|
| O(1) | 常数额外空间 | 原地排序（快排）、双指针 |
| O(n) | 与输入成正比 | 哈希表、新建数组 |
| O(n²) | 二维数组 | 动态规划表格、邻接矩阵 |
| O(log n) | 递归栈/二分 | 递归深度为 log n 的分治 |
| O(n) | 递归栈 | 递归深度为 n（链表递归） |

### 5.4 复杂度分析实战技巧

**技巧一：关注最内层循环**

```python
# 暴力：O(n²)
for i in range(n):
    for j in range(n):
        matrix[i][j] += 1

# 优化：O(n) — 利用前缀和
prefix = [0] * (n + 1)
for i in range(n):
    prefix[i+1] = prefix[i] + matrix[0][i]
```

**技巧二：递归分析用递推方程**

归并排序：
  T(n) = 2T(n/2) + O(n)
  展开：T(n) = 2[2T(n/4) + O(n/2)] + O(n)
              = 4T(n/4) + 2O(n) + O(n)
              = 8T(n/8) + 3O(n)
              ...
              = O(n log n)    ← 主定理（Master Theorem）Case 2

**技巧三：摊还分析——看长期平均**

动态数组（如 Python list）的 `append` 操作，单次最坏 O(n)（触发扩容时），但均摊每次 O(1)。因为扩容频率随数组大小指数递减，总拷贝量收敛为 O(n)。

### 6.2 工程选型的三个原则

1. **默认选哈希表**：除非有排序、范围查询或关系建模需求，哈希表的 O(1) 性能几乎总是最优选择。
2. **内存敏感选数组**：嵌入式系统、高频交易等场景，数组的缓存友好性和零额外开销至关重要。
3. **数据量决定一切**：小数据（< 1000）用什么都快，中等数据（1k-100 万）选对数据结构很关键，大数据（> 100 万）必须考虑分布式方案。

### 6.3 常见误区

- **误区一：过度设计**。面试中学了红黑树、B+ 树，实际项目中 99% 的场景用哈希表或数组就够了。
- **误区二：只看平均复杂度**。实时系统（自动驾驶、医疗设备）必须考虑最坏情况，不能容忍 O(n) 的尾延迟。
- **误区三：忽略缓存局部性**。数组的 O(n) 遍历在实践中可能比哈希表的 O(1) 查找更快，因为 CPU 缓存命中率高。这就是为什么数据库引擎底层偏爱数组和 B+ 树而非哈希索引。
