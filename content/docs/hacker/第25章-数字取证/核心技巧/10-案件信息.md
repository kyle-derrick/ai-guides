# 案件信息：从填写到管控的全链路实操指南

## 概述

案件信息（Case Information / Case Metadata）是数字取证全流程的**元数据锚点**——它定义了"谁在什么时间、因什么事由、对什么目标、以何种授权进行了取证分析"。没有准确完整的案件信息，所有取证结果在法律上可能沦为"非法证据"，在管理上则变成无法追溯的孤岛数据。

本篇聚焦**实操层面**：如何填写每个字段、如何用模板引擎自动化生成、如何构建案件信息的校验与管控体系。理论基础（法律框架、标准体系、生命周期模型）请参阅[案件信息理论篇](../理论基础/11-案件信息.md)。

> **核心命题**：案件信息不是"填个表格"，而是在为整个取证行为构建法律正当性。每一个字段的缺失或错误，都可能在法庭上成为对方律师攻击取证结论的突破口。

## 一、案件信息字段体系：逐字段实操指南

### 1.1 案件编号：全局唯一标识符

**核心原则**：唯一性、可读性、不可篡改性、可扩展性。

**命名规范设计**

| 编码要素 | 作用 | 设计建议 |
|----------|------|----------|
| 机构标识 | 区分不同组织产生的案件 | 使用官方缩写（如 GA=公安、MOJ=司法、ENT=企业） |
| 案件类型 | 快速分类检索 | IR=事件响应、CR=合规审计、INV=内部调查、EXT=外部协查 |
| 时间戳 | 标记案件创建时间 | 年份必选，月份可选（高频机构建议包含） |
| 流水序号 | 保证唯一性 | 4-6位，按年度重置 |
| 证据源子编码 | 区分同一案件下不同证据 | E01/E02…，与主编码用连字符分隔 |

**实际格式参考**

```text
格式一（通用）：  CASE-2026-00421
格式二（含类型）：DF-IR-202606-003       （数字取证-事件响应-2026年6月-第3号）
格式三（执法）：  GA-2026-0358            （公安-2026年-第358号）
格式四（含子编码）：CASE-2026-00421-E03   （第421号案件的第3个证据源）
格式五（跨国）：  USCA-SDNY-2026-00142   （美国联邦加州南区-2026年-第142号）
```

**最佳实践——二级编码制**

```text
主编码（案件级）：CASE-2026-00421
  ├── 子编码（证据源级）：CASE-2026-00421-E01  （硬盘镜像 #1）
  ├── 子编码（证据源级）：CASE-2026-00421-E02  （内存转储）
  ├── 子编码（证据源级）：CASE-2026-00421-E03  （网络流量抓包）
  └── 子编码（报告级）：CASE-2026-00421-R01   （初步分析报告）
```

这样设计的好处：
- 检索时可以用前缀 `CASE-2026-00421*` 命中该案件所有相关条目
- 子编码可以独立流转（证据 E03 可能被转交给另一位分析师处理）
- 报告编码与证据编码分离，避免编号混乱

**常见错误及规避**

| 错误 | 后果 | 规避方法 |
|------|------|----------|
| 使用日期作为唯一编号 | 同一天多个案件冲突 | 必须包含流水序号 |
| 编号可修改 | 监管链断裂 | 系统锁定，变更走审计日志 |
| 不同系统使用不同编号 | 跨系统无法关联 | 建立编号映射表或统一使用全局编号 |
| 子编码与主编码脱节 | 无法追溯证据归属 | 系统强制要求子编码前缀匹配主编码 |

### 1.2 委托单位：授权来源的完整记录

**必须记录的多层信息**

```text
委托单位信息树
├── 组织信息
│   ├── 单位全称（法律主体名称）
│   ├── 单位简称（日常沟通用）
│   ├── 组织类型（执法机构 / 企业法务 / 律所 / 监管部门 / 个人）
│   └── 统一社会信用代码 / 营业执照号（企业委托时）
├── 联系人信息
│   ├── 姓名（全名，不用花名/昵称）
│   ├── 职务 / 头衔
│   ├── 联系电话（建议加密存储）
│   ├── 电子邮箱
│   └── 备用联系方式
├── 委托关系
│   ├── 委托方案件编号（如有）
│   ├── 委托日期
│   └── 委托范围说明
└── 授权文书
    ├── 文书类型（搜查令 / 调查函 / 内部授权书 / 法院委托书）
    ├── 文书编号
    ├── 签发机关
    ├── 签发日期
    └── 有效期
```

**不同委托类型的关键差异**

| 委托类型 | 授权文书要求 | 见证人要求 | 特殊注意事项 |
|----------|-------------|-----------|-------------|
| 执法机关协查 | 搜查令/立案决定书/协查函 | 必须有见证人 | 涉及搜查的需明确搜查范围，超范围取证无效 |
| 企业内部调查 | 内部调查授权书（法务部签发） | 建议有HR或合规人员在场 | 注意劳动法限制，不得强迫员工配合 |
| 律所委托 | 律师函/法院调查令 | 视司法管辖区而定 | 律师-委托人特权可能限制信息共享范围 |
| 监管合规 | 监管通知/检查令 | 通常不需要 | 注意监管机构的取证权限边界 |
| 跨境协查 | 国际刑事司法协助(MLAT) | 视双边协议而定 | 涉及数据本地化问题，证据出境需审批 |

**脱敏处理规则**

涉及隐私保护的案件中，委托方信息的脱敏策略：

```yaml
# 脱敏等级与字段映射
脱敏规则:
  公开级:
    字段: [全部]  # 所有字段可明文
  内部级:
    字段: [单位名称, 联系人姓名]  # 可明文
    部分脱敏: [联系电话]  # 保留前3后4位
  机密级:
    字段: []  # 无字段可明文
    部分脱敏: [单位名称→"某科技公司", 联系人→"赵某"]
    全脱敏: [联系电话, 邮箱, 身份证号]
  绝密级:
    全部脱敏，使用内部代号替代
    授权文书仅保留文号，正文不纳入案件信息
```

### 1.3 分析人员：谁执行了什么操作

**为什么不能只记录一个名字？**

1. **监管链要求**：每个接触证据的人都必须记录，ISO/IEC 27037:2012 明确要求"证据操作的每一步都应由具体人员负责"
2. **出庭作证**：只有实际执行分析的人员才能作为专家证人（Expert Witness），记录不全将导致无法确定谁有资格作证
3. **质量追溯**：分析出现偏差时，需要快速定位是哪个环节的哪位人员负责

**推荐记录字段**

```yaml
分析人员记录:
  - 姓名: 张三
    角色: 主分析师（Lead Analyst）
    资质: CCFP-2024-0123（注册数字取证专家）
    执行操作: 磁盘镜像获取、文件系统分析、时间线重建
    数字签名: /signatures/zhang_san_20260625.p7s
    签名时间: "2026-06-25T16:30:00+08:00"

  - 姓名: 李四
    角色: 协助分析师
    资质: GCFE-2023-0456（注册取证分析员）
    执行操作: 网络流量分析、日志关联
    数字签名: /signatures/li_si_20260625.p7s

  - 姓名: 王五
    角色: 技术审核（QA Reviewer）
    资质: EnCE-2022-0789
    执行操作: 方法论审查、结果复核
    数字签名: /signatures/wang_wu_20260625.p7s

  - 姓名: 赵六
    角色: 见证人（Witness）——刑事案件专用
    身份: 社区工作人员（非警方/非利益相关方）
    身份证号: [加密存储]
    在场时间: "2026-06-25T10:00:00 ~ 2026-06-25T12:30:00+08:00"
```

**角色分工标准**

| 角色 | 最低资质要求 | 核心职责 | 是否必须 |
|------|-------------|---------|---------|
| 主分析师 | 行业认证（CCFP/GCFE/EnCE）+ 2年以上经验 | 制定分析方案、执行核心分析、撰写报告 | ✅ |
| 协助分析师 | 行业认证或同等培训证明 | 协助执行特定分析任务 | 视案件规模 |
| 技术审核 | 与主分析师不同认证体系（避免同质化） | 独立复核分析方法和结论的合理性 | ✅ |
| 见证人 | 无专业要求，但须为独立第三方 | 在场监督证据操作过程的合法性 | 刑事案件✅ |

> **实战教训**：某企业在内部调查中，技术审核由主分析师的直属上级担任，对方律师以"审核人与分析师存在上下级利益关系，审核独立性存疑"为由，成功挑战了取证结论的可信度。建议审核人应来自不同部门或外部机构。

### 1.4 时间信息：取证行为的时间锚

**时间字段体系**

```yaml
时间线记录:
  案件受理时间: "2026-06-18T09:30:00+08:00"     # 正式接受委托
  证据获取时间:                                       # 逐项记录
    - 证据编号: E01
      操作: 磁盘镜像
      开始: "2026-06-18T10:00:00+08:00"
      结束: "2026-06-18T14:30:00+08:00"
    - 证据编号: E02
      操作: 内存转储
      时间: "2026-06-18T10:05:00+08:00"          # 瞬时操作
  分析时间:
    开始: "2026-06-19T09:00:00+08:00"
    结束: "2026-06-21T17:00:00+08:00"
  报告签发: "2026-06-22T14:00:00+08:00"
```

**时间记录的关键规则**

| 规则 | 原因 | 示例 |
|------|------|------|
| 使用ISO 8601格式 | 国际标准，机器可解析 | `2026-06-25T14:30:00+08:00` |
| 必须标注时区 | 避免跨时区混淆 | `+08:00`（北京）/ `-05:00`（美东） |
| 记录精度至少到分钟 | 法庭需要精确时间线 | 不要只写 `2026-06-25` |
| 区分"操作时间"和"系统时间" | 操作时间是人工记录，系统时间来自设备时钟 | 两者可能不一致 |
| 时间同步证据 | 证明各设备时钟的偏差 | NTP服务器日志、GPS时间戳 |

**时区混淆的真实案例**

```text
场景：跨境网络入侵调查

事件A（中国分析师）：北京时间 2026-06-25 14:00 采集入侵日志
事件B（日志内容）：显示攻击者 IP 在 美国东部时间 2026-06-25 01:00 发起连接

错误计算（忽略时区）：14:00 - 01:00 = 13小时差 → 无法判断攻击发生在采集之前还是之后
正确计算（标注时区）：14:00+08:00 = 06:00 UTC
                     01:00-05:00 = 06:00 UTC
                     → 两个事件在同一时刻（UTC 06:00），攻击与采集同时发生！
```

**时间戳一致性校验**

在分析多源证据时，必须校验各设备的时间是否对齐：

```bash
# 使用 libfaketime 或 NTP 日志校验时钟偏差
# 方法1：从 NTP 服务器日志获取偏差
ntpstat  # 查看本机时钟偏差

# 方法2：对比已知时间源
# 在取证工作站上同步 NTP，记录偏差值
chronyc tracking | grep "System time"

# 方法3：在案件信息中记录
时钟偏差说明: |
  取证工作站时钟经 NTP 校准，偏差 < 0.5秒
  被检设备时钟偏差：+3分12秒（相对取证工作站）
  已在时间线分析中补偿该偏差
```

### 1.5 授权范围：取证行为的法律边界

授权范围决定了取证操作的合法边界，记录不清将导致"超范围取证"的法律风险。

**必须明确的授权要素**

```text
授权范围声明
├── 授权行为类型
│   ├── ✅ 允许：磁盘镜像、日志提取、网络抓包、内存转储
│   ├── ⚠️ 需单独授权：远程取证、云平台数据调取、第三方数据
│   └── ❌ 禁止：删除数据、修改系统配置、植入监控程序
├── 授权对象范围
│   ├── 设备清单（型号、序列号、IP地址）
│   ├── 数据范围（特定时间段 / 全量 / 关键字匹配）
│   └── 人员范围（特定嫌疑人 / 全员）
├── 时间限制
│   ├── 授权有效期（起止日期）
│   └── 操作时间窗口（如只能在工作日 9:00-18:00 操作）
└── 地理限制
    ├── 操作地点（某数据中心机房 / 某办公室）
    └── 数据存储位置（境内 / 境外）
```

**常见授权风险场景及应对**

| 风险场景 | 可能后果 | 预防措施 |
|----------|---------|---------|
| 搜查令只授权"办公电脑"，但同时提取了手机数据 | 手机证据被排除 | 在授权范围字段明确列出每台设备 |
| 内部调查授权未获得员工书面同意 | 劳动仲裁中证据无效 | 授权文书中加入员工知情同意条款 |
| 跨境取证未获数据所在地司法管辖区许可 | 证据在境外法庭无效 | 记录跨境合规审批编号 |
| 授权范围表述模糊（"相关电子数据"） | 被质疑超范围操作 | 使用穷举式描述，列出所有目标设备和数据类型 |

### 1.6 保密等级与保留期限

**保密等级设定指南**

| 等级 | 适用场景 | 访问控制 | 存储要求 | 传输要求 |
|------|---------|---------|---------|---------|
| 公开 | 合规审计的非敏感结果 | 无限制 | 普通存储 | 普通传输 |
| 内部 | 一般企业内部调查 | 仅项目组成员 | 加密存储 | 加密传输 |
| 机密 | 商业秘密泄露调查 | 需审批访问 | 硬件加密 + 物理隔离 | 专用安全通道 |
| 绝密 | 刑事案件/国家安全 | 最小授权原则 | 独立安全屋存储 | 专人物理传递 |

**保留期限建议**

| 案件类型 | 最低保留期限 | 建议保留期限 | 法律依据 |
|----------|-------------|-------------|---------|
| 刑事案件 | 案件审结后20年 | 永久 | 《刑法》追诉时效 |
| 民事案件 | 判决生效后5年 | 10年 | 《民事诉讼法》 |
| 行政处罚 | 处罚决定后3年 | 5年 | 《行政处罚法》 |
| 企业内部调查 | 案件关闭后2年 | 5年 | 企业政策 + 劳动法 |
| 合规审计 | 审计周期+3年 | 7年 | SOX/HIPAA等合规要求 |

## 二、标准化模板体系

### 2.1 通用案件信息模板

以下模板覆盖法律、管理、技术三个维度的需求，可根据实际场景裁剪：

```yaml
# ============================
# 案件信息登记表
# 模板版本: v2.1 | 最后更新: 2026-06
# ============================

# --- 案件标识 ---
案件编号: "{{ case_id }}"
案件名称: "{{ title }}"
案件类型: "{{ case_type }}"  # 网络入侵|数据泄露|内部调查|合规审计|事件响应|刑事协查
优先级: "{{ priority }}"      # P0-紧急 P1-高 P2-中 P3-低 P4-积压
保密等级: "{{ classification }}"  # 公开|内部|机密|绝密

# --- 委托方信息 ---
委托单位:
  全称: "{{ client_full_name }}"
  简称: "{{ client_short_name }}"
  类型: "{{ client_type }}"  # 执法机构|企业|律所|监管部门|个人
  统一社会信用代码: "{{ uscc }}"
联系人:
  姓名: "{{ contact_name }}"

  算法: SHA-256

# --- 取证范围 ---

  允许操作: "{{ allowed_operations }}"

# --- 法律声明 ---
```

```python
"""
案件信息自动化生成器
功能：数据校验 → 模板渲染 → 文件输出 → 审计日志
"""
import re
import hashlib
from datetime import datetime
from dataclasses import dataclass, field
from typing import List, Optional
from enum import Enum

class Priority(Enum):
    P0 = "紧急"
    P1 = "高"
    P2 = "中"
    P3 = "低"
    P4 = "积压"

class Classification(Enum):
    PUBLIC = "公开"
    INTERNAL = "内部"
    CONFIDENTIAL = "机密"
    TOP_SECRET = "绝密"

@dataclass
class Contact:
    name: str
    title: str
    phone: str
    email: str

@dataclass
class Authorization:
    doc_type: str          # 搜查令/调查函/内部授权书
    doc_number: str
    issuer: str
    issue_date: str
    expiry_date: Optional[str] = None

@dataclass
class Analyst:
    name: str
    role: str              # 主分析师/协助/审核/见证
    cert_number: Optional[str] = None
    signature_path: Optional[str] = None

@dataclass
class CaseInfo:
    case_id: str
    title: str
    case_type: str
    priority: Priority
    classification: Classification
    client_name: str
    client_type: str
    contact: Contact
    authorizations: List[Authorization]
    analysts: List[Analyst]
    analysis_start: str
    analysis_end: Optional[str] = None
    report_date: Optional[str] = None
    evidence_count: int = 0
    evidence_hash: str = ""
    scope: str = ""
    jurisdiction: str = "中华人民共和国"
    retention_date: Optional[str] = None
    related_cases: List[dict] = field(default_factory=list)

    def validate(self) -> List[str]:
        """校验案件信息完整性和一致性"""
        errors = []

        # 1. 案件编号格式校验
        if not re.match(r'^[A-Z]+-\d{4}-\d{4,6}(-[A-Z]+\d{2,})?$', self.case_id):
            errors.append(f"案件编号格式不符合规范: {self.case_id}")

        # 2. 必填字段检查
        if not self.client_name:
            errors.append("委托单位名称不能为空")
        if not self.analysts:
            errors.append("至少需要一名分析人员")
        if not self.analysis_start:
            errors.append("分析开始时间不能为空")

        # 3. 人员资质检查
        for analyst in self.analysts:
            if analyst.role == "主分析师" and not analyst.cert_number:
                errors.append(f"主分析师 {analyst.name} 缺少资质编号")
            if analyst.role == "主分析师" and not analyst.signature_path:
                errors.append(f"主分析师 {analyst.name} 缺少数字签名")

        # 4. 时间一致性检查
        if self.analysis_end and self.analysis_start:
            if self.analysis_end < self.analysis_start:
                errors.append("分析结束时间早于开始时间")

        # 5. 保密等级与信息脱敏一致性
        if self.classification in (Classification.CONFIDENTIAL, Classification.TOP_SECRET):
            if self.contact.phone and len(self.contact.phone.replace("-", "")) > 8:
                errors.append("机密及以上案件，联系电话应做脱敏处理（保留前3后4位）")

        # 6. 刑事案件必须有见证人
        if "刑事" in self.case_type:
            witnesses = [a for a in self.analysts if a.role == "见证"]
            if not witnesses:
                errors.append("刑事案件必须有见证人在场")

        # 7. 授权文书检查
        if not self.authorizations:
            errors.append("缺少授权文书记录")

        return errors

    def generate_case_number(self) -> str:
        """自动生成案件编号"""
        now = datetime.now()
        type_prefix = self.case_type[:2].upper()
        return f"{type_prefix}-{now.year}-{now.strftime('%m')}-{hashlib.md5(str(now.timestamp()).encode()).hexdigest()[:4].upper()}"

    def to_template_data(self) -> dict:
        """转换为模板渲染数据"""
        return {
            "case_id": self.case_id,
            "title": self.title,
            "case_type": self.case_type,
            "priority": self.priority.value,
            "classification": self.classification.value,
            "client": self.client_name,
            "client_type": self.client_type,
            "contact": self.contact,
            "authorizations": self.authorizations,
            "analysts": self.analysts,
            "analysis_start": self.analysis_start,
            "analysis_end": self.analysis_end or "进行中",
            "report_date": self.report_date or "未签发",
            "evidence_count": self.evidence_count,
            "evidence_hash": self.evidence_hash,
            "scope": self.scope,
            "jurisdiction": self.jurisdiction,
            "retention_date": self.retention_date,
            "related_cases": self.related_cases,
        }


# 使用示例
case = CaseInfo(
    case_id="DF-2026-0635",
    title="核心代码仓库异常访问事件调查",
    case_type="内部调查",
```

### 3.3 自动化流水线集成

在成熟的取证团队中，案件信息应嵌入端到端的自动化流水线：

```text
┌─────────────┐
│  案件受理    │ ← 委托方提交委托书 / IR告警触发
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 自动生成编号 │ ← 系统根据机构+类型+时间生成唯一编号
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 预填充模板  │ ← 从委托文书OCR提取委托方信息，从通讯录提取分析师信息
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 自动校验    │ ← 必填字段、格式规范、资质有效性、时间逻辑
└──────┬──────┘
       │
       ▼ (校验通过)
┌─────────────┐
│ 分配分析师  │ ← 根据案件类型和分析师负荷自动分配
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 签名锁定    │ ← 主分析师 + 审核人签名，关键字段变为只读
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 证据采集    │ ← 案件信息作为上下文传递到采集工具
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 分析 & 报告 │ ← 报告模板自动引用案件信息，避免手工抄录错误
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 归档 & 保留 │ ← 根据保留期限自动设定归档日期
└─────────────┘
```

**关键设计原则**

1. **不可逆锁定**：案件信息一旦进入取证阶段，关键字段（案件编号、委托方、授权文书、时间戳）应锁定为只读，修改需走审批流程
2. **版本追踪**：允许补充非关键字段（如备注、补充证据），但每次修改需记录：变更人、变更时间、变更前后值、变更原因
3. **签名积木**：各环节的分析人员通过数字签名"积木式"叠加——主分析师签名后锁定分析范围，审核人签名后锁定结论，确保每个接触点都可追溯
4. **变更审计**：任何字段修改都写入审计日志，审计日志本身不可修改

### 4.2 常用查询

```sql
-- 查询1: 某分析师的工作负荷统计
SELECT
    a.analyst_name,
    COUNT(DISTINCT c.case_id) AS total_cases,
    SUM(CASE WHEN a.role = '主分析' THEN 1 ELSE 0 END) AS lead_cases,
    SUM(CASE WHEN c.status IN ('分析中','复核') THEN 1 ELSE 0 END) AS active_cases
FROM case_analysts a
JOIN cases c ON a.case_id = c.case_id
WHERE a.assigned_at BETWEEN '2026-01-01' AND '2026-06-30'
GROUP BY a.analyst_name
ORDER BY total_cases DESC;

-- 查询2: 即将到期的案件（保留期限30天内到期）
SELECT
    case_id, title, retention_date,
    DATE(retention_date, '-30 days') AS warning_date,
    JULIANDAY(retention_date) - JULIANDAY('now') AS days_remaining
FROM cases
WHERE status = '归档'
  AND retention_date BETWEEN DATE('now') AND DATE('now', '+30 days')
ORDER BY retention_date ASC;

-- 查询3: 缺少关键字段的案件（质量巡检）
SELECT
    c.case_id, c.title, c.status,
    GROUP_CONCAT(missing.field_name) AS missing_fields
FROM cases c
CROSS JOIN (
    SELECT '授权文书' AS field_name
    UNION ALL SELECT '主分析师'
    UNION ALL SELECT '时间线'
) missing
LEFT JOIN authorizations auth ON c.case_id = auth.case_id
LEFT JOIN case_analysts ca ON c.case_id = ca.case_id AND ca.role = '主分析'
LEFT JOIN timeline_events te ON c.case_id = te.case_id
WHERE (auth.id IS NULL AND missing.field_name = '授权文书')
   OR (ca.id IS NULL AND missing.field_name = '主分析师')
   OR (te.id IS NULL AND missing.field_name = '时间线')
GROUP BY c.case_id;

-- 查询4: 跨案件关联分析（APT攻击溯源）
SELECT
    cr.case_id,
    c1.title AS current_case,
    cr.related_case_id,
    c2.title AS related_case,
    cr.relation_type,
    cr.notes
FROM case_relations cr
JOIN cases c1 ON cr.case_id = c1.case_id
JOIN cases c2 ON cr.related_case_id = c2.case_id
WHERE cr.case_id IN (
    SELECT case_id FROM cases WHERE title LIKE '%APT%'
)
ORDER BY c1.created_at;

-- 查询5: 审计日志——某案件的完整变更历史
SELECT
    changed_at, changed_by, changed_field,
    old_value, new_value, change_reason
FROM case_audit_log
WHERE case_id = 'CASE-2026-00421'
ORDER BY changed_at ASC;
```

### 4.3 触发器：自动记录变更

```sql
-- SQLite 触发器：案件关键字段变更自动写入审计日志
CREATE TRIGGER audit_case_update
AFTER UPDATE ON cases
WHEN OLD.case_id != NEW.case_id
   OR OLD.client_name != NEW.client_name
   OR OLD.classification != NEW.classification
   OR OLD.status != NEW.status
BEGIN
    INSERT INTO case_audit_log (case_id, changed_by, changed_field, old_value, new_value)
    SELECT
        NEW.case_id,
        'system',  -- 实际应用中应从会话上下文获取操作人
        'status',
        OLD.status,
        NEW.status
    WHERE OLD.status != NEW.status;
END;
```

### 5.1 自动化校验规则引擎

```python
import re
from datetime import datetime, timedelta
from typing import List, Dict, Callable

class CaseValidator:
    def __init__(self):
        self.rules: List[Callable] = []
        self._register_default_rules()

    def _register_default_rules(self):
        """注册默认校验规则"""
        self.rules = [
            self._check_required_fields,
            self._check_case_id_format,
            self._check_date_consistency,
            self._check_analyst_qualification,
            self._check_authorization_existence,
            self._check_classification_consistency,
            self._check_witness_requirement,
            self._check_retention_period,
        ]

    def validate(self, case_data: Dict) -> List[Dict]:
        """执行全部校验规则"""
        results = []
        for rule in self.rules:
            result = rule(case_data)
            if result:
                results.append(result)
        return results

    def _check_required_fields(self, data: Dict) -> Dict:
        required = {
            'case_id': '案件编号',
            'client_name': '委托单位',
            'analyst': '主分析师',
            'analysis_start': '分析开始时间',
            'authorizations': '授权文书',
        }
        missing = [label for field, label in required.items() if not data.get(field)]
        if missing:
            return {'level': 'ERROR', 'rule': '必填字段', 'message': f'缺少: {", ".join(missing)}'}
        return None

    def _check_case_id_format(self, data: Dict) -> Dict:
        pattern = r'^[A-Z]+-\d{4}-\d{4,6}(-[A-Z]+\d{2,})?$'
        case_id = data.get('case_id', '')
        if not re.match(pattern, case_id):
            return {'level': 'ERROR', 'rule': '编号格式', 'message': f'案件编号 {case_id} 不符合规范'}
        return None

    def _check_date_consistency(self, data: Dict) -> Dict:
        start = data.get('analysis_start', '')
        end = data.get('analysis_end', '')
        if start and end:
            try:
                if datetime.fromisoformat(end) < datetime.fromisoformat(start):
                    return {'level': 'ERROR', 'rule': '时间逻辑', 'message': '分析结束时间早于开始时间'}
            except ValueError:
                return {'level': 'ERROR', 'rule': '时间格式', 'message': '日期格式需为ISO 8601'}
        return None

    def _check_analyst_qualification(self, data: Dict) -> Dict:
        analysts = data.get('analyst_list', [])
        unqualified = [a['name'] for a in analysts if a.get('role') == '主分析' and not a.get('cert_number')]
        if unqualified:
            return {'level': 'WARNING', 'rule': '资质检查', 'message': f'主分析师缺少资质编号: {", ".join(unqualified)}'}
        return None

    def _check_authorization_existence(self, data: Dict) -> Dict:
        auths = data.get('authorizations', [])
        if not auths:
            return {'level': 'ERROR', 'rule': '授权文书', 'message': '缺少授权文书记录'}
        for auth in auths:
            if not auth.get('doc_number'):
                return {'level': 'ERROR', 'rule': '授权文书', 'message': f'授权文书 {auth.get("doc_type")} 缺少文号'}
        return None

    def _check_classification_consistency(self, data: Dict) -> Dict:
        level = data.get('classification', '')
        phone = data.get('contact_phone', '')
        if level in ('机密', '绝密') and phone and len(phone.replace('-', '')) > 8:
            return {'level': 'WARNING', 'rule': '脱敏一致性', 'message': '机密级以上案件，联系电话应做脱敏处理'}
        return None

    def _check_witness_requirement(self, data: Dict) -> Dict:
        if '刑事' in data.get('case_type', ''):
            analysts = data.get('analyst_list', [])
            witnesses = [a for a in analysts if a.get('role') == '见证']
            if not witnesses:
                return {'level': 'ERROR', 'rule': '见证人', 'message': '刑事案件必须有见证人'}
        return None

    def _check_retention_period(self, data: Dict) -> Dict:
        retention = data.get('retention_date')
        if retention:
            try:
                ret_date = datetime.fromisoformat(retention)
                if ret_date < datetime.now():
                    return {'level': 'WARNING', 'rule': '保留期限', 'message': '保留期限已过期'}
            except ValueError:
                return {'level': 'ERROR', 'rule': '保留期限', 'message': '保留期限日期格式错误'}
        return None


# 批量校验示例
validator = CaseValidator()
cases = [
    # 案例1：校验通过
    {
        'case_id': 'DF-2026-0635',
        'client_name': 'XX科技有限公司法务部',
        'analyst': '张三',
        'analysis_start': '2026-06-19T09:00:00+08:00',
        'analysis_end': '2026-06-21T17:00:00+08:00',
        'authorizations': [{'doc_type': '内部授权书', 'doc_number': 'AUTH-2026-042'}],
        'analyst_list': [
            {'name': '张三', 'role': '主分析', 'cert_number': 'CCFP-2025-0123'},
        ],
    },
    # 案例2：多项校验失败
    {
        'case_id': 'bad-id',
        'client_name': '',
        'analysis_start': '2026-06-21',
        'analysis_end': '2026-06-19',
        'case_type': '刑事案件协查',
        'authorizations': [],
        'analyst_list': [
            {'name': '匿名', 'role': '主分析'},  # 无资质
        ],
    },
]

for i, case in enumerate(cases, 1):
    print(f"\n=== 案件 {i}: {case.get('case_id', '未编号')} ===")
    results = validator.validate(case)
    if not results:
        print("  ✅ 校验通过")
    else:
        for r in results:
            icon = "❌" if r['level'] == 'ERROR' else "⚠️"
            print(f"  {icon} [{r['rule']}] {r['message']}")
```

### 5.2 质量巡检清单

定期对所有在办案件执行质量巡检：

| 检查项 | 检查方法 | 严重级别 | 修复方式 |
|--------|---------|---------|---------|
| 案件编号格式 | 正则匹配 | ERROR | 重新编号（需审批） |
| 必填字段完整性 | 非空检查 | ERROR | 补充缺失字段 |
| 授权文书有效性 | 对比有效期字段 | ERROR | 获取新授权或结案 |
| 主分析师资质证书 | 证书编号+有效期查询 | WARNING | 更新资质信息 |
| 时间线一致性 | 开始 < 结束检查 | ERROR | 修正时间记录 |
| 保密等级与脱敏 | 交叉检查 | WARNING | 对敏感字段脱敏 |
| 刑事案件见证人 | 角色字段检查 | ERROR | 补充见证人记录 |
| 数字签名完整性 | 签名文件存在性检查 | WARNING | 重新签名 |
| 保留期限合理性 | 对比案件类型最低要求 | WARNING | 延长保留期限 |

### 案例1：企业内部数据泄露调查

**背景**：某互联网公司发现核心代码库被异常访问，怀疑内部员工泄露商业机密。法务部委托数字取证团队进行调查。

1. **审核人独立性**：技术审核人王五来自独立安全部门（非开发团队），避免了利益冲突
2. **证据编号规范**：每个证据都有独立编号（E01-E04）和完整哈希值，可独立追溯
3. **授权范围穷举**：明确列出了"仅限上述4项证据源"，防止后续被质疑超范围取证
4. **时间精确到分钟**：所有时间戳带时区，为后续时间线重建提供可靠基准

4. **证据的物理+逻辑双重编号**：每台设备记录了型号、序列号（物理），同时有证据编号（逻辑）
5. **运输环节**：证据从现场到实验室的运输过程也需要记录（GPS追踪），这是监管链的一部分

## 七、常见误区与实战纠正

### 误区1：案件信息只是"走个流程"

**错误做法**："系统里有就行了，编号随便填一个。"

**为什么这很危险**：
- 法庭上对方律师会要求出示案件信息的原始记录，如果编号与系统不一致，整条证据链的可信度崩塌
- 资质编号造假或缺失，在专家证人质证环节会被直接击溃

**正确做法**：每一个字段都是可验证的事实。编号由系统生成不可手动改，资质编号需核实有效性，日期需与系统日志交叉验证。

### 误区2：一个人包揽所有角色

**错误做法**：小团队资源有限，"反正就我一个人，分析、审核、见证都是我。"

**法律风险**：
- ISO/IEC 27037:2012 明确要求职责分离（Separation of Duties）
- 法庭上对方可以质疑"谁来保证这个人的分析是客观的？"
- 即使结论正确，程序瑕疵也可能导致证据被排除

**最低人员配置**

| 案件规模 | 最少人数 | 角色分配 |
|----------|---------|---------|
| 简单（单一证据源、非刑事案件） | 2人 | 分析 + 审核 |
| 中等（多证据源、内部调查） | 3人 | 分析 + 协助 + 审核 |
| 复杂（刑事案件、跨境） | 4人+ | 分析 + 协助 + 审核 + 见证人 |

### 误区3：时间只写日期，不写时区和具体时间

**错误做法**："分析日期：2026年6月25日"

**具体危害**：
- 无法区分"25日全天分析"还是"25日只分析了1小时"
- 跨时区协作时，时差可能导致事件顺序完全颠倒
- 内存取证等时效性极强的操作，缺少精确时间将使证据时效性无法证明

**正确格式**：`2026-06-25T09:00:00+08:00`（ISO 8601，含时区）

### 误区4：案件信息一旦填写就不再更新

**错误做法**：案件受理时填写了一版，之后有新证据发现、有新人员加入、有授权范围变更，但案件信息没有同步更新。

**后果**：
- 新发现的证据可能缺少合法授权记录
- 新加入的分析人员无法在监管链中被追溯
- 报告引用的案件信息与实际不一致

**正确做法**：建立案件信息"版本日志"，每次变更记录时间、操作人、变更内容。关键字段变更需走审批流程。数据库审计日志自动记录所有修改。

### 误区5：保密信息明文存储在案件信息中

**错误做法**：委托人的身份证号、手机号等敏感信息直接写在案件信息文件中。

**风险**：
- 案件信息文件被泄露时，这些敏感信息也随之暴露
- 不符合《个人信息保护法》的要求

**正确做法**：
- 联系电话、邮箱等使用加密存储（数据库层面AES加密或字段级加密）
- 身份证号只保留后4位，其余用掩码替代（如 `***********1234`）
- 绝密级案件使用内部代号替代真实姓名

## 八、进阶：案件信息与证据清单的衔接

案件信息是**证据清单**的上游数据源。证据清单中的每一条证据都必须能回溯到具体的案件编号。两者的衔接关系如下：

```text
案件信息                    证据清单
┌──────────────┐          ┌──────────────┐
│ 案件编号     │──────────│ 所属案件编号 │
│ 委托方信息   │          │              │
│ 授权范围     │          │ 证据编号     │←── E01, E02, E03
│ 分析团队     │          │ 证据描述     │
│ 时间线       │          │ 获取方法     │
│ 保密等级     │          │ 完整性哈希   │
│ 保留期限     │          │ 存储位置     │
│              │          │ 经手人员     │←── 分析团队子集
│              │          │ 流转记录     │
└──────────────┘          └──────────────┘
```

**衔接要点**：

1. 证据编号的前缀必须与案件编号一致（二级编码制）
2. 证据清单中的"经手人员"必须在案件信息的"分析人员"列表中
3. 证据的"获取时间"必须在案件信息的"授权有效期"范围内
4. 证据的"存储位置"的访问控制应基于案件信息的"保密等级"

详细的证据清单编制方法，请参阅下一节 [证据清单：从发现到呈堂的完整管理链路](11-证据清单.md)。

## 九、总结

案件信息是数字取证的**第一道防线**，也是**最后一根锚绳**。

| 层级 | 核心能力 | 行动建议 |
|------|---------|---------|
| 入门 | 理解每个字段的法律意义和填写要求 | 使用本章模板，不遗漏任何必填字段 |
| 进阶 | 构建自动化校验和模板生成流水线 | 集成模板引擎+校验规则，实现"填写即校验" |
| 精通 | 设计机构级案件信息管理体系 | 数据库Schema+审计日志+质量巡检自动化 |

**实操 Checklist（每次填写案件信息时逐项打勾）**：

```text
□ 案件编号符合机构命名规范，已由系统生成
□ 委托单位记录了全称和联系人
□ 授权文书编号、签发机关、有效期均已记录
□ 分析人员列出了完整角色分工和资质编号
□ 所有时间标注了ISO 8601格式和时区
□ 保密等级设定合理，敏感字段已做脱敏
□ 刑事案件已安排见证人
□ 授权范围明确穷举，无模糊表述
□ 保留期限符合相关法律要求
□ 数字签名已附加
```

**一句话金句**：案件信息填得好，法庭之上少烦恼。每一笔记录，都是在为证据的合法性修筑防线。
