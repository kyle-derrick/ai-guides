## 本章小结

设计模式是软件工程中经过数十年实践验证的设计智慧结晶。本章从设计原则出发，系统讲解了 GoF 23 种经典设计模式、4 种并发模式和 5 种分布式弹性模式，覆盖了从对象创建、结构组合到行为协作，再到并发安全和分布式容错的完整知识体系。以下是本章核心内容的系统回顾与深度提炼。

---

## 一、设计原则：模式的理论根基

设计模式不是凭空产生的，它们根植于一组更基础的设计原则。理解这些原则是理解"为什么需要某个模式"的前提。

### 1.1 SOLID 原则速查

| 原则 | 核心思想 | 违反征兆 | 体现模式 |
|------|----------|----------|----------|
| **SRP**（单一职责） | 一个类只有一个变化的理由 | 类名中出现"And"、方法集明显分组 | Facade、Mediator |
| **OCP**（开闭原则） | 对扩展开放，对修改关闭 | 每次新增功能都要修改已有代码 | Strategy、Template Method、Decorator |
| **LSP**（里氏替换） | 子类必须能替换父类而不影响正确性 | "正方形-长方形"问题 | 组合优于继承 |
| **ISP**（接口隔离） | 客户端不应被迫依赖它不使用的接口 | 实现接口时出现空方法 | 职责拆分 |
| **DIP**（依赖倒置） | 高层和低层都应依赖抽象 | 高层直接 new 低层具体类 | Factory Method、Abstract Factory、DI 容器 |

### 1.2 辅助原则

- **DRY**：每条知识在系统中只有一个权威来源。但要区分"真正的重复"和"偶然的重复"——两段代码恰好现在相同但变化动因不同时，强行抽取反而制造耦合。
- **YAGNI**：不要为假想的未来需求过度设计。设计模式提供的是"当需要扩展时再引入"的能力，不是"提前引入以防万一"的工具。
- **KISS**：最简单的可行方案通常是最好的。引入设计模式应该让系统更简单，而非更复杂。
- **组合优于继承**：继承创建编译时确定的静态关系并暴露实现细节，组合通过持有对象引用创建运行时可变的动态关系。
- **封装变化点**：识别系统中变化的部分，将其封装在稳定的接口之后——这是 GoF 设计模式的核心思想。

## 二、创建型模式：封装对象创建

创建型模式将对象的创建与使用分离，核心目标是提升系统的灵活性。

| 模式 | 意图 | 核心机制 | 典型应用场景 |
|------|------|----------|-------------|
| **Factory Method** | 让子类决定实例化哪个类 | 继承 + 多态 | 框架需要创建对象但不预知具体类型 |
| **Abstract Factory** | 创建一族相关对象 | 接口 + 多个具体工厂 | UI 框架支持多套主题控件 |
| **Builder** | 分步构建复杂对象 | 链式调用 + Director | HTTP 请求构建、SQL 查询构造 |
| **Prototype** | 通过克隆已有对象创建新对象 | 深拷贝 | 创建成本高的对象、需要大量相似对象 |
| **Singleton** | 保证唯一实例 | 私有构造 + 静态访问 | 连接池、配置管理（争议较大，慎用） |

**关键区分**：
- Factory Method vs Abstract Factory：前者创建一个产品，后者创建一族产品
- Builder vs Factory：工厂关注"创建什么"，Builder 关注"如何创建"
- Singleton 的替代方案：依赖注入容器、模块级变量、上下文管理器

## 三、结构型模式：组合对象结构

结构型模式关注如何将类或对象组合成更大的结构。

| 模式 | 意图 | 核心机制 | 典型应用场景 |
|------|------|----------|-------------|
| **Adapter** | 接口转换，让不兼容的类一起工作 | 包装已有类 | 整合第三方库、遗留系统对接 |
| **Bridge** | 抽象与实现分离，独立变化 | 组合（抽象持有实现引用） | 多维度变化（形状×颜色）避免类爆炸 |
| **Composite** | 树形结构的统一处理 | 递归组合 | 文件系统、组织架构、菜单系统 |
| **Decorator** | 动态添加职责 | 包装 + 委托 | Java I/O 流、中间件链 |
| **Facade** | 简化子系统访问 | 提供统一高层接口 | 复杂子系统的简化入口 |
| **Flyweight** | 共享细粒度对象 | 内部状态共享 + 外部状态外传 | 大量相似对象的内存优化 |
| **Proxy** | 控制对象访问 | 代理 + 委托 | 懒加载、权限控制、缓存、远程代理 |

**关键区分**：
- Adapter vs Bridge：Adapter 是事后补救（让不兼容的类工作），Bridge 是事前设计（有意分离抽象和实现）
- Facade vs Mediator：Facade 是单向简化访问，Mediator 是双向协调交互
- Decorator vs Proxy：Decorator 增强行为，Proxy 控制访问

## 四、行为型模式：分配职责与通信

行为型模式关注对象之间的职责分配和通信方式。

| 模式 | 意图 | 核心机制 | 典型应用场景 |
|------|------|----------|-------------|
| **Strategy** | 封装算法族，使其可互换 | 组合 + 多态 | 运行时切换排序算法、支付方式 |
| **Observer** | 一对多依赖通知 | 发布-订阅 | 事件系统、数据绑定 |
| **Command** | 请求的对象化 | 封装请求为对象 | 撤销/重做、命令队列、事务 |
| **Iterator** | 顺序访问聚合元素 | 迭代器协议 | 遍历复杂数据结构 |
| **Mediator** | 对象交互的集中管控 | 中介者协调 | 聊天室、表单验证 |
| **State** | 状态驱动行为变化 | 状态对象 + 转换 | 订单状态机、游戏状态管理 |
| **Template Method** | 算法骨架的延迟步骤 | 继承 + 钩子方法 | 数据挖掘流程、测试框架 |
| **Visitor** | 操作与对象结构分离 | 双重分派 | AST 遍历、编译器优化 |
| **Chain of Responsibility** | 请求的链式传递 | 责任链 | 中间件管道、审批流程 |
| **Memento** | 状态的保存与恢复 | 快照 + 备忘录栈 | 编辑器撤销、游戏存档 |

**关键区分**：
- Strategy vs State：结构相似但意图不同——Strategy 由客户端主动选择，State 由内部状态自动切换
- Strategy vs Template Method：前者用组合，后者用继承
- Visitor 的双刃剑：添加新操作容易（新增 Visitor），但添加新元素困难（需修改所有 Visitor）

## 五、并发模式：解决线程安全与异步协作

| 模式 | 意图 | 核心机制 | 典型应用场景 |
|------|------|----------|-------------|
| **Active Object** | 方法执行与调用分离 | 代理 + 调度器 + Future | 异步任务处理 |
| **Monitor** | 共享数据的互斥访问 | synchronized + wait/notify | 生产者-消费者队列 |
| **Read-Write Lock** | 读并发、写独占 | 读锁计数 + 写锁互斥 | 读多写少的缓存系统 |
| **Future/Promise** | 异步计算结果占位 | 生产者写入 + 消费者查询 | 异步编程、Promise 链 |

## 六、分布式弹性模式：云原生架构的基石

在微服务和云原生架构中，服务间网络通信充满不确定性。分布式弹性模式帮助系统在部分失败时仍能提供可接受的服务。

| 模式 | 意图 | 核心机制 | 典型应用场景 |
|------|------|----------|-------------|
| **Circuit Breaker** | 防止对失败服务的反复调用 | 三态模型：Closed→Open→Half-Open | 服务调用保护、级联故障预防 |
| **Bulkhead** | 故障隔离 | 线程池/信号量/进程隔离 | 微服务间的资源隔离 |
| **Retry** | 自动重试瞬时故障 | 指数退避 + 抖动 | 网络不稳定场景 |
| **Saga** | 分布式事务补偿 | 本地事务 + 补偿操作 | 跨服务业务流程的一致性保证 |
| **Sidecar** | 基础设施能力外挂 | 独立辅助进程 | Service Mesh、日志收集、配置管理 |

**关键原则**：
- Circuit Breaker 的三态转换需要精心调参：故障阈值太低会误触发，太高会延迟保护
- Retry 只对幂等操作使用，必须区分瞬时故障和永久故障
- Saga 的补偿操作必须是幂等的，且要处理补偿本身失败的情况
- Bulkhead 的隔离粒度需要平衡：太粗无法隔离，太细增加管理开销

## 七、模式之间的关系网络

设计模式不是孤立存在的，它们之间存在互补、替代和演进三种关系。

### 7.1 互补组合

| 组合模式 | 协作方式 | 典型应用 |
|----------|----------|----------|
| Composite + Iterator | 遍历树形结构 | 文件系统遍历 |
| Composite + Visitor | 对树结构执行多种操作 | 编译器 AST 处理 |
| Command + Memento | 可撤销操作 | 编辑器 undo/redo |
| Observer + Mediator | 松耦合的组件通信 | 复杂 UI 事件系统 |
| Factory + Strategy | 动态创建和切换算法 | 支付网关选择 |
| Decorator + Factory | 动态组装装饰器链 | HTTP 客户端中间件 |

### 7.2 替代选择

| 场景 | 选项 A | 选项 B | 选择依据 |
|------|--------|--------|----------|
| 算法切换 | Strategy（组合） | Template Method（继承） | Strategy 更灵活，Template Method 更简单 |
| 行为变化 | Strategy（客户端选择） | State（内部自动切换） | Strategy 由外部控制，State 由状态驱动 |
| 接口适配 | Adapter（事后适配） | Bridge（事前设计） | Adapter 解决已有问题，Bridge 预防未来问题 |
| 简化访问 | Facade（单向） | Mediator（双向） | Facade 是简化接口，Mediator 是协调交互 |

### 7.3 演进路径

简单工厂 → 工厂方法 → 抽象工厂（逐步增加抽象层次）
继承 → 组合 → 委托（逐步降低耦合度）
单体 → 分层 → 微服务（逐步增加模块化程度）

## 八、模式选择决策框架

选择设计模式时，遵循"从问题出发，而非从模式出发"的原则。以下是识别问题信号与模式映射的实用对照表：

| 问题信号 | 推荐模式 | 说明 |
|----------|----------|------|
| 频繁修改同一处代码 | Strategy / Template Method | 封装变化点 |
| 大量条件分支（if/else、switch） | Strategy / State / Factory Method | 消除分支，引入多态 |
| 创建逻辑复杂、参数过多 | Builder / Abstract Factory | 分步构建或参数化创建 |
| 对象间紧耦合 | Observer / Mediator / Facade | 降低耦合度 |
| 需要不修改类就添加行为 | Decorator / Visitor | 开闭原则的体现 |
| 需要保存/恢复状态 | Memento / Command | 状态快照与回滚 |
| 树形结构需要统一处理 | Composite | 部分-整体层次 |
| 大量相似对象消耗内存 | Flyweight | 共享内部状态 |

**选择模式的三个判定问题**：
1. 不用模式，这段代码能否在 20 行以内清晰解决？
2. 未来半年内，这段代码的需求是否可能变化？
3. 引入模式后，维护成本是否真正降低？

如果两个以上答案是否定的，就不要引入模式。

## 九、常见误区与纠正

本章详细分析了十大设计模式误区，核心教训如下：

1. **为了模式而模式**（Pattern Fever）：简单问题不要套复杂模式，遵循 YAGNI 原则
2. **Singleton 滥用**：Singleton 本质是全局变量，引入隐式状态和测试困难，优先用依赖注入
3. **过度使用继承**：继承层次超过 2-3 层时考虑组合，LSP 要求子类必须能替换父类
4. **God Object**：按 SRP 原则拆分为职责单一的协作类
5. **Observer 内存泄漏**：使用弱引用（WeakReference）或生命周期绑定自动清理
6. **Strategy 类爆炸**：策略逻辑简单时用函数+注册表替代类
7. **错误选择模式**：参照决策框架，从问题本质出发而非从模式出发
8. **Decorator 层层嵌套**：嵌套超过 3 层时考虑用管道（Pipeline）或中间件模式替代
9. **Visitor 修改困难**：元素稳定时才用 Visitor，频繁新增元素时考虑其他方案
10. **Saga 补偿失败**：补偿操作必须幂等，需要处理补偿本身失败的边界情况

## 十、关键公式与模型

| 概念 | 公式/模型 | 说明 |
|------|-----------|------|
| 吞吐量 | QPS = 并发数 / 平均延迟 | Little 定律，容量规划基础 |
| 可用性 | SLA = 正常时间 / 总时间 | 99.9% = 8.76 小时/年停机 |
| 延迟 | P99 = 排序后第 99 百分位值 | 尾延迟比平均延迟更能反映用户体验 |
| 容量 | QPS × 单次请求资源 = 总资源需求 | 资源规划基础 |
| 线程安全 | 互斥 + 同步 | Monitor 模式的核心 |
| 分布式一致性 | 本地事务 + 补偿 = 最终一致 | Saga 模式的理论基础 |

### 设计阶段

- [ ] 识别系统中的变化点，封装在稳定接口之后
- [ ] 遵循 SOLID 原则，每个类只有一个变化的理由
- [ ] 从问题出发选择模式，而非从模式出发寻找问题
- [ ] 评估模式引入的复杂度是否真正带来收益
- [ ] 设计容错和降级方案（Circuit Breaker、Bulkhead）

## 本章核心要点速记

设计模式 = 反复出现的设计问题的成熟解决方案

原则先行：SOLID + DRY + YAGNI + KISS + 组合优于继承
     ↓
五大创建型：Factory Method / Abstract Factory / Builder / Prototype / Singleton
     ↓
七大结构型：Adapter / Bridge / Composite / Decorator / Facade / Flyweight / Proxy
     ↓
十大行为型：Strategy / Observer / Command / Iterator / Mediator / State / Template Method / Visitor / Chain of Responsibility / Memento
     ↓
四大并发模式：Active Object / Monitor / Read-Write Lock / Future/Promise
     ↓
五大弹性模式：Circuit Breaker / Bulkhead / Retry / Saga / Sidecar

核心理念：识别变化 → 封装变化 → 面向接口 → 组合优于继承
选择标准：问题驱动而非模式驱动，简单优先于优雅
