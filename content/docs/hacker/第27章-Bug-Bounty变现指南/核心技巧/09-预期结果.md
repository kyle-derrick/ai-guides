## 预期结果：漏洞报告中被低估的关键锚点

在Bug Bounty报告的七大核心要素中，"预期结果"（Expected Result）往往是被新手忽略、被老手草率对待的一栏。很多研究者在写报告时，把大量精力投入复现步骤和截图，到了"预期结果"就随手写一句"输入应该被正确转义或过滤"了事。

这种做法的代价是巨大的：厂商安全工程师拿到你的报告后，必须自己去判断"正常情况下应该是什么行为"，然后才能确认你发现的确实是漏洞而非功能特性。这个认知成本直接决定了你的报告是被快速确认还是被标记为"信息不足"退回。

> **一句话原则**：预期结果的职责是建立"正常基线"——告诉审核者在没有漏洞的情况下，系统应该如何行为。有了这个锚点，实际结果中的异常行为才能被一目了然地识别出来。

---

## 一、预期结果的本质与作用

### 1.1 为什么预期结果如此重要

预期结果在漏洞报告中承担着三重角色：

| 角色 | 作用 | 缺失后果 |
|------|------|----------|
| **基线定义者** | 建立正常行为参照，让漏洞"偏差"可被感知 | 审核者无法判断行为是否异常 |
| **攻击面标注器** | 明确指出哪个输入点、哪个参数、哪个端点存在缺陷 | 审核者需要自行定位，增加复现成本 |
| **严重程度支撑** | 预期与实际的差距越大，漏洞越严重 | 严重程度评估缺乏依据，可能被降级 |

### 1.2 预期结果与实际结果的对比关系

预期结果和实际结果是漏洞报告中唯一成对出现的要素，它们的关系类似于科学实验中的"对照组"和"实验组"：

```text
预期结果 = 系统在正常防御状态下的行为（对照组）
实际结果 = 系统在攻击payload作用下的行为（实验组）
漏洞证据 = 实际结果偏离预期结果的程度
```

这个对比越清晰、越具体，审核者确认漏洞的速度就越快。HackerOne的内部数据显示，包含清晰预期/实际结果对比的报告，确认时间平均缩短40%，奖金溢价约15%。

### 1.3 预期结果的四个层次

一个完整的预期结果不是一句话能说清的，它包含四个层次：

```text
┌─────────────────────────────────────────────────────────┐
│                   预期结果四层结构                         │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  第一层：输入处理 —— 系统应如何处理攻击输入                 │
│    → 编码、过滤、拒绝、转义、规范化                        │
│                                                          │
│  第二层：输出行为 —— 处理后的数据应如何呈现                 │
│    → 以纯文本显示、不触发脚本执行、返回错误页               │
│                                                          │
│  第三层：系统状态 —— 漏洞触发后系统状态应如何               │
│    → 无权限变更、无数据泄露、无异常日志                    │
│                                                          │
│  第四层：业务逻辑 —— 从业务角度看应发生什么                 │
│    → 请求被拒绝、操作被记录、用户收到提示                   │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

> **实战建议**：并非所有报告都需要覆盖全部四层。简单漏洞（如反射型XSS）只需写第一层和第二层；复杂漏洞（如权限提升）则需要四层全覆盖。

## 二、按漏洞类型的预期结果写法

不同的漏洞类型，预期结果的写法有本质差异。以下是八大高发漏洞类型的预期结果撰写指南。

### 2.1 XSS（跨站脚本攻击）

XSS的预期结果核心是"输入被安全处理后无害展示"。

**反射型XSS预期结果模板：**
```text
预期结果：
- 输入的payload "<script>alert(1)</script>" 应被HTML编码为 
  &lt;script&gt;alert(1)&lt;/script&gt;
- 编码后的文本应作为纯文本在搜索结果页面显示
- 浏览器不应将编码后的文本解析为可执行脚本
- 应在响应头中设置 Content-Type: text/html; charset=UTF-8
```

**存储型XSS预期结果模板：**
```text
预期结果：
- 提交的payload "<img src=x onerror=alert(1)>" 应被以下方式之一处理：
  a) 完全过滤：移除HTML标签，只保留纯文本内容
  b) 编码输出：对<、>、"、'、&等特殊字符进行HTML实体编码
  c) CSP策略：Content-Security-Policy阻止内联脚本执行
- 该payload在个人资料页面加载时应以纯文本形式展示
- 其他用户访问该资料页面时，不应触发任何JavaScript执行
```

**DOM型XSS预期结果模板：**
```text
预期结果：
- 前端JavaScript代码应使用安全的DOM操作方法：
  - 使用 textContent 代替 innerHTML
  - 使用 URLSearchParams 解析参数而非 location.hash 直接拼接
- 注入的payload应被JavaScript的内置编码函数处理后再插入DOM
- 开发者工具Console中不应出现未捕获的脚本执行痕迹
```

### 2.2 SQL注入

SQL注入的预期结果核心是"用户输入被参数化处理，不参与SQL语法解析"。

```text
预期结果：
- 输入 "admin' OR '1'='1" 应被参数化查询（Prepared Statement）处理，
  作为字符串值传递给数据库，而非拼接到SQL语句中
- 数据库应返回与正常输入 "admin" 相同的结果集
- 即使输入包含SQL关键字（SELECT、UNION、DROP等），
  也不应改变查询逻辑或产生额外的查询行为
- 错误响应不应包含数据库错误详情（如表名、列名、SQL语法错误信息）
```

**时间盲注场景的预期结果：**
```text
预期结果：
- 输入 "test' OR SLEEP(5)--" 时，SLEEP(5) 不应被执行
- 响应时间应与正常输入 "test" 一致（±200ms以内）
- 不应出现明显的响应延迟（超过1秒的时间差异即为异常）
```

### 2.3 SSRF（服务端请求伪造）

```text
预期结果：
- 输入URL "http://169.254.169.254/latest/meta-data/" 时，
  服务端应拒绝发起对该内部地址的请求
- 服务端应返回明确的错误提示（如"不允许访问该地址"）
- 以下URL均应被阻断：
  - 内网地址：http://10.0.0.1, http://172.16.0.1, http://192.168.1.1
  - 本地回环：http://127.0.0.1, http://localhost
  - 云元数据：http://169.254.169.254
  - 文件协议：file:///etc/passwd
- 服务端应仅允许向预定义白名单域名发起请求
```

### 2.4 IDOR（不安全的直接对象引用）

```text
预期结果：
- 使用用户A的Session访问用户B的数据端点 
  GET /api/v1/users/B/profile 时：
- 服务端应校验当前Session与请求资源的归属关系
- 应返回 HTTP 403 Forbidden 或 404 Not Found
- 响应体中不应包含用户B的任何数据
- 通过修改URL中的用户ID、数据库记录ID等参数，
  不应能越权访问其他用户的资源
```

### 2.5 CSRF（跨站请求伪造）

```text
预期结果：
- 服务端应对所有状态变更请求进行CSRF防护：
  a) 验证CSRF Token（每个表单/请求携带唯一Token）
  b) 验证Origin/Referer头（只接受来自可信域名的请求）
  c) 验证自定义请求头（如X-Requested-With）
- 跨域页面发起的POST/PUT/DELETE请求应被拒绝
- 服务端应返回 HTTP 403 并记录安全日志
```

### 2.6 文件上传漏洞

```text
预期结果：
- 上传文件 "shell.php" 时，服务端应拒绝上传或：
  a) 对文件扩展名进行白名单校验（仅允许 .jpg, .png, .pdf 等）
  b) 对文件MIME类型进行校验（验证Content-Type头与实际内容一致）
  c) 对文件内容进行二次渲染（如图片重新编码）
- 上传后的文件不应存放在Web可访问目录中
- 即使文件被上传，访问该文件URL时也不应执行其中的代码
- 服务端应返回明确的错误提示（如"不支持的文件类型"）
```

### 2.7 路径遍历

```text
预期结果：
- 输入 "../../../etc/passwd" 时：
  a) 路径中的 "../" 应被过滤或规范化
  b) 文件读取操作应限制在指定目录内
  c) 不应返回 /etc/passwd 的内容
  d) 应返回错误提示（如"文件不存在"或"无权访问"）
- 经测试，以下遍历payload均应被阻断：
  - %2e%2e%2f（URL编码）
  - ..%252f（双重URL编码）
  - ....//（变异写法）
  - ..\（Windows路径分隔符）
```

### 2.8 命令注入

```text
预期结果：
- 输入 "127.0.0.1; cat /etc/passwd" 时：
  - 服务端应仅将 "127.0.0.1" 作为合法输入处理
  - ";" 及之后的 "cat /etc/passwd" 应被拒绝或忽略
  - 不应执行任何系统命令
  - 响应中不应包含 /etc/passwd 的内容
- 服务端应对所有用户输入使用安全的命令调用方式：
  - 使用参数化调用而非字符串拼接
  - 使用白名单校验输入值
  - 将应用权限降至最低（非root运行）
```

## 三、预期结果的撰写方法论

### 3.1 "如果没有这个漏洞"思考法

这是最核心的撰写方法。在写预期结果之前，问自己：

> "如果这个漏洞不存在，这个功能在收到同样输入时应该怎么做？"

以搜索功能的XSS为例：

| 思考步骤 | 内容 |
|----------|------|
| 第1步：确认功能 | 搜索功能接收用户输入，展示搜索结果 |
| 第2步：确认正常输入 | 用户输入 "hello world" 时，结果显示 "找到与 hello world 相关的结果" |
| 第3步：确认输入处理 | 后端应将 "hello world" 作为纯文本处理，不做HTML解析 |
| 第4步：写出预期 | 攻击payload 应被同样对待——编码后作为纯文本展示，不执行 |

### 3.2 从开发者视角反推预期

换个角度思考：如果你是这个功能的开发者，你会如何设计输入处理逻辑？

```python
# 开发者的设计意图（预期行为）：
def handle_user_input(user_input):
    # 1. 输入验证：只允许特定格式
    if not is_valid_format(user_input):
        return "输入格式不正确"
    
    # 2. 输出编码：HTML编码后展示
    safe_output = html.escape(user_input)
    return render_template("result.html", output=safe_output)
    
# 漏洞产生的原因：
# 开发者跳过了 html.escape() 这一步
# 预期结果就是描述 html.escape() 应该做的工作
```

### 3.3 从安全标准反推预期

安全标准（如OWASP ASVS）为预期结果提供了权威依据：

| 安全标准 | 相关条款 | 预期结果要点 |
|----------|----------|--------------|
| OWASP ASVS V5.1 | 输入验证 | 所有输入应被验证和净化 |
| OWASP ASVS V6.1 | 输出编码 | 输出应根据上下文进行编码 |
| OWASP ASVS V8.1 | 数据保护 | 敏感数据不应泄露到响应中 |
| OWASP ASVS V12.1 | API安全 | API应实施认证和授权检查 |
| CWE-79 | XSS | 系统应对用户提供数据进行编码输出 |
| CWE-89 | SQL注入 | 系统应使用参数化查询处理用户输入 |

## 四、预期结果的常见错误与纠正

### 4.1 六大常见错误

| 错误类型 | 典型错误写法 | 正确写法 | 错误原因 |
|----------|-------------|----------|----------|
| **过于笼统** | "输入应该被过滤" | "输入中的 `<` 应被编码为 `&lt;`，`>` 应被编码为 `&gt;`" | 缺少具体编码方式 |
| **描述意图而非行为** | "开发者应该做输入验证" | "输入中的特殊字符应被移除或编码，响应中不应包含未处理的HTML标签" | 混淆了"应该做什么"和"应该呈现什么" |
| **缺少技术细节** | "页面不应该执行脚本" | "页面加载时，Console中不应有未捕获的脚本执行痕迹；DOM中不应有动态注入的script标签" | 审核者无法据此验证 |
| **遗漏边界条件** | 只写正常输入的预期 | 补充边界输入的预期："空输入应返回错误提示；超长输入应被截断；null值应被拒绝" | 覆盖不完整 |
| **与实际结果混淆** | "输入被编码为纯文本"（这是预期）+ "但实际执行了"（这是实际结果）| 只写预期部分，实际结果放在独立段落 | 混淆了预期和实际 |
| **没有量化标准** | "响应时间不应该太长" | "响应时间应与正常输入一致（±200ms）；注入payload不应导致服务器处理时间增加超过1秒" | 无法客观判定 |

### 4.2 自检清单

写完预期结果后，用以下清单自检：

```text
□ 是否明确指出了输入点（哪个参数、哪个端点）？
□ 是否说明了系统应采用的处理方式（编码/过滤/拒绝）？
□ 是否描述了预期的输出状态（响应内容、HTTP状态码、响应头）？
□ 是否覆盖了关键边界条件（空值、超长输入、特殊字符）？
□ 是否避免了"应该加强安全"之类的空话？
□ 是否与实际结果形成了清晰的对比？
□ 非技术人员能否据此理解"什么是正常行为"？
```

## 五、进阶：不同场景下的预期结果策略

### 5.1 多步骤漏洞的预期结果

对于需要多个步骤触发的漏洞（如存储型XSS → 后台触发），每个步骤都需要独立的预期结果：

```text
步骤1：提交payload（用户操作）
预期结果1：
- payload应被存储到数据库中
- 当前用户查看个人资料时，payload应被编码为纯文本显示

步骤2：管理员查看资料（后台触发）
预期结果2：
- 管理员查看该用户资料时，payload同样应被编码为纯文本
- 管理员的浏览器不应执行payload中的脚本
- 安全日志中不应有脚本执行的痕迹

步骤3：数据外带验证
预期结果3：
- 从攻击者控制的外部服务器角度：
  - 不应收到任何来自目标系统的DNS查询请求
  - 不应收到任何HTTP请求（包括带cookie的请求）
  - 即payload中的外带URL不应被触发
```

### 5.2 条件型漏洞的预期结果

某些漏洞只在特定条件下触发（如特定浏览器、特定配置），预期结果需要包含条件说明：

```text
预期结果（标准配置）：
- 在默认配置下，所有用户输入应被HTML编码后输出
- Chrome/Firefox/Safari 均不应执行注入的脚本

预期结果（宽松CSP配置）：
- 即使Content-Security-Policy允许 'unsafe-inline'，
  输入中的 <script> 标签仍应被服务器端编码处理
- CSP是纵深防御，不应替代输入验证
```

### 5.3 API漏洞的预期结果

API漏洞的预期结果需要额外关注数据格式和错误处理：

```text
预期结果：
- POST /api/v1/users/profile 请求体中的 name 字段：
  - 若包含HTML标签，应返回 400 Bad Request，
    错误信息："name字段包含非法字符"
  - 或将其编码后存储，GET时返回编码后的值
- 响应头中应包含安全头：
  - X-Content-Type-Options: nosniff
  - X-XSS-Protection: 1; mode=block
- 错误响应（4xx/5xx）中不应包含：
  - 数据库错误详情
  - 服务器路径信息
  - 内部IP地址
```

### 5.4 涉及多角色的预期结果

当漏洞涉及不同权限角色时，需要分别说明每个角色的预期行为：

```text
预期结果：
- 普通用户（User A）：
  - 提交payload后，自身查看个人资料时应看到编码后的纯文本
  - 不应触发任何脚本执行

- 管理员（User B）：
  - 查看User A的资料时，同样应看到编码后的纯文本
  - 管理后台的数据预览功能不应解析payload中的HTML/JS

- 匿名用户（未登录）：
  - 访问User A的公开资料页面时，payload应被编码展示
  - 公开页面应有与登录页面相同的输入处理标准
```

## 六、预期结果与报告其他要素的协同

### 6.1 与标题的呼应

标题中承诺的漏洞类型，预期结果必须提供对应的正常行为基线：

```text
标题：[存储型XSS] POST /api/v1/users/profile - 用户Bio字段可执行任意JavaScript

预期结果：（必须覆盖标题中提到的所有要素）
- 端点 POST /api/v1/users/profile 的 bio 参数
- 应对输入进行HTML编码
- 编码后的内容应在用户资料页面以纯文本展示
- 不应有任何脚本执行
```

### 6.2 与复现步骤的衔接

复现步骤中的每一步操作，都应有对应的预期结果说明：

```text
复现步骤                          预期结果
─────────────────────────────    ─────────────────────────────
1. 登录测试账号                    → 应成功进入系统
2. 在bio字段输入payload            → 应返回"保存成功"，payload被编码存储
3. 访问个人资料页面                 → payload以纯文本展示，无脚本执行
4. 其他用户访问该资料页面            → 同样看到编码后的文本
```

### 6.3 与实际结果的对照

预期结果与实际结果的最佳呈现方式是并列对照：

```text
预期结果：
在步骤3中，payload "<script>alert(1)</script>" 应被HTML编码为 
&lt;script&gt;alert(1)&lt;/script&gt;，以纯文本形式在页面上显示。

实际结果：
payload未被编码，浏览器直接解析并执行了<script>alert(1)</script>，
页面弹出了alert对话框，证明存在存储型XSS漏洞。
```

### 6.4 与影响分析的关联

预期结果描述的"本应如此"与实际结果的"实际如此"之间的差距，直接支撑了影响分析中的严重程度判断：

```text
预期结果：payload应被编码，不执行任何脚本
实际结果：payload直接执行，可窃取任意用户的Cookie
↓
影响分析：预期与实际的偏差表明XSS防护完全缺失，
加上无CSP策略兜底，任何用户提交的payload均可直接执行，
影响全部活跃用户（约10万+），建议评级为"高危"。
```

## 七、模板与速查表

## 八、常见误区与进阶建议

### 8.1 误区澄清

**误区1："预期结果越简短越好"**
实际上，过于简短的预期结果会导致审核者需要自行推断"什么才算正常"，增加了复现成本。具体的预期结果虽然篇幅更长，但节省的是双方的时间。

**误区2："预期结果应该和实际结果一样长"**
预期结果的长度取决于漏洞复杂度，不必与实际结果对等。简单的反射型XSS，预期结果可以是两句话；复杂的RCE链，预期结果可能需要分步骤详细说明。

**误区3："预期结果只是走过场，审核者不会细看"**
恰恰相反，HackerOne的triage团队反馈：他们首先看标题和预期结果来判断报告质量。预期结果写得好的报告会被优先处理。

**误区4："预期结果只写成功路径就行"**
边界条件同样重要。审核者需要知道：如果输入为空怎么办？如果输入超长怎么办？如果输入null值怎么办？这些边界场景的预期行为能体现你对漏洞理解的深度。

### 8.2 进阶建议

**策略1：引用安全标准增加权威性**
```text
预期结果：
根据OWASP ASVS v4.0 第5.3.3条，所有用户提供的数据在输出到浏览器前
应根据其上下文进行适当的编码。因此，bio字段的输出应在HTML上下文中
使用HTML实体编码（< → &lt;，> → &gt;，& → &amp;，" → &quot;）。
```

**策略2：提供验证脚本的预期输出**
```bash
预期结果（使用以下验证脚本）：
运行以下命令后，应返回"安全"：
$ curl -s "https://target.com/api/search?q=<script>alert(1)</script>" | grep -o "<script>"
# 预期：无输出（payload被编码）

实际输出：
<script>  ← payload未被编码
```

**策略3：对比同类产品的正常行为**
```text
预期结果：
作为参照，同类产品（如Product B、Product C）在处理相同输入时，
均会对HTML标签进行编码后以纯文本展示。目标系统应采用相同的标准。
```

## 总结

预期结果不是漏洞报告中的"填充项"，而是报告质量的分水岭。它建立的"正常基线"是所有后续分析（实际结果、影响分析、严重程度评估）的逻辑起点。

写好预期结果的核心方法论：
1. **逆向思考**："如果没有漏洞，系统应该怎么做"
2. **具体化**：从编码方式到HTTP状态码，越具体越好
3. **完整覆盖**：边界条件、多角色、多步骤都要考虑
4. **与实际结果形成对比**：让漏洞一目了然

掌握预期结果的撰写，你的报告将从"业余玩家"跃升为"专业研究者"级别——这不仅意味着更快的确认和更高的奖金，更是你安全思维成熟度的体现。
